## Что Вы Изучите

До сих пор вы научились вести беседы с ИИ, эффективно структурировать подсказки и основанывать ответы на ваших документах. Но существует фундаментальное ограничение: языковые модели могут только генерировать текст. Они не могут проверить погоду, выполнить вычисления, запросить базу данных или взаимодействовать с внешними системами.

Инструменты меняют это. Предоставляя модели функции, которые она может вызвать, вы превращаете её из генератора текста в агента, который может предпринимать действия. Модель решает, когда ей нужен инструмент, какой инструмент использовать и какие параметры передать. Ваш код выполняет функцию и возвращает результат. Модель включается этот результат в свой ответ.

## Понимание AI-Агентов с Инструментами

> **📝 Примечание:** Термин «агенты» в этом модуле относится к AI-ассистентам с расширенными возможностями вызова инструментов. Это отличается от паттернов **Agentic AI** (автономные агенты с планированием, памятью и многошаговым рассуждением), которые мы рассмотрим в [Модуле 05: MCP](../05-mcp/README.md).

Без инструментов языковая модель может только генерировать текст на основе обучающих данных. Спросите у неё текущую погоду — и ей придется гадать. Дайте ей инструменты, и она сможет вызвать API погоды, выполнить вычисления или запросить базу данных — затем внедрить эти реальные результаты в ответ.

<img src="../../../translated_images/ru/what-are-tools.724e468fc4de64da.webp" alt="Без инструментов и с инструментами" width="800"/>

*Без инструментов модель может лишь гадать — с инструментами она может вызывать API, выполнять вычисления и возвращать данные в реальном времени.*

AI-агент с инструментами следует паттерну **Reasoning and Acting (ReAct)**. Модель не просто отвечает — она размышляет о том, что ей нужно, действует, вызывая инструмент, наблюдает за результатом, а затем решает, действовать ли дальше или предоставить окончательный ответ:

1. **Рассудить** — Агент анализирует вопрос пользователя и определяет, какая информация нужна
2. **Действовать** — Агент выбирает нужный инструмент, генерирует правильные параметры и вызывает его
3. **Наблюдать** — Агент получает вывод инструмента и оценивает результат
4. **Повторить или Ответить** — Если нужно больше данных, агент возвращается к первому шагу; иначе формирует ответ на естественном языке

<img src="../../../translated_images/ru/react-pattern-detail.96a5efeeb6dd2f61.webp" alt="Паттерн ReAct" width="800"/>

*Цикл ReAct — агент рассуждает, что делать, действует вызовом инструмента, наблюдает результат и повторяет, пока не сформирует окончательный ответ.*

Это происходит автоматически. Вы определяете инструменты и их описания. Модель принимает решения о том, когда и как их использовать.

## Как Работает Вызов Инструментов

### Определения Инструментов

[WeatherTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) | [TemperatureTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/TemperatureTool.java)

Вы определяете функции с четкими описаниями и спецификациями параметров. Модель видит эти описания в своем системном подсказке и понимает, что делает каждый инструмент.

```java
@Component
public class WeatherTool {
    
    @Tool("Get the current weather for a location")
    public String getCurrentWeather(@P("Location name") String location) {
        // Ваша логика поиска погоды
        return "Weather in " + location + ": 22°C, cloudy";
    }
}

@AiService
public interface Assistant {
    String chat(@MemoryId String sessionId, @UserMessage String message);
}

// Ассистент автоматически подключен Spring Boot с помощью:
// - Bean ChatModel
// - Все методы @Tool из классов с @Component
// - ChatMemoryProvider для управления сессиями
```

На диаграмме показан разбор каждой аннотации и как каждый элемент помогает ИИ понять, когда вызывать инструмент и какие аргументы передать:

<img src="../../../translated_images/ru/tool-definitions-anatomy.f6468546037cf28b.webp" alt="Анатомия определений инструментов" width="800"/>

*Анатомия определения инструмента — @Tool говорит ИИ, когда использовать, @P описывает каждый параметр, а @AiService связывает всё вместе при запуске.*

### Принятие Решений

Когда пользователь спрашивает «Как погода в Сиэтле?», модель не выбирает инструмент случайно. Она сравнивает намерение пользователя с каждым описанием инструмента, оценивает релевантность и выбирает лучший вариант. Затем генерирует структурированный вызов функции с нужными параметрами — в данном случае, устанавливая `location` в `"Seattle"`.

Если ни один инструмент не подходит под запрос пользователя, модель отвечает из собственного знания. Если подходит несколько инструментов, выбирается самый конкретный.

<img src="../../../translated_images/ru/decision-making.409cd562e5cecc49.webp" alt="Как ИИ решает, какой инструмент использовать" width="800"/>

*Модель оценивает каждый доступный инструмент по намерению пользователя и выбирает лучший — поэтому важно делать описания инструментов ясными и точными.*

### Архитектура: Авто-подключение Spring Boot

Этот модуль использует интеграцию LangChain4j с Spring Boot через декларативные интерфейсы `@AiService`. При запуске Spring Boot обнаруживает все `@Component` с методами `@Tool`, ваш бин `ChatModel` и `ChatMemoryProvider` — затем связывает их в один интерфейс `Assistant` без какого-либо шаблонного кода.

<img src="../../../translated_images/ru/spring-boot-wiring.151321795988b04e.webp" alt="Архитектура авто-подключения Spring Boot" width="800"/>

*Интерфейс @AiService связывает ChatModel, компоненты инструментов и провайдер памяти — Spring Boot автоматически выполняет все подключения.*

Основные преимущества такого подхода:

- **Авто-подключение Spring Boot** — ChatModel и инструменты автоматически внедряются
- **Паттерн @MemoryId** — Автоматическое управление памятью на основе сессий
- **Одиночный экземпляр** — Assistant создаётся один раз и повторно используется для лучшей производительности
- **Типобезопасное выполнение** — Java-методы вызываются напрямую с конверсией типов
- **Оркестрация многократных ходов** — Автоматически обрабатывает цепочки инструментов
- **Никакого шаблонного кода** — Нет ручных вызовов `AiServices.builder()` или хранения памяти в HashMap

Альтернативные подходы (ручные вызовы `AiServices.builder()`) требуют больше кода и теряют преимущества интеграции Spring Boot.

## Цепочки Инструментов

**Цепочки инструментов** — настоящая сила агентов на основе инструментов проявляется, когда один вопрос требует использования нескольких инструментов. Спросите «Какова погода в Сиэтле по Фаренгейту?» — агент автоматически соединяет два инструмента: сначала вызывается `getCurrentWeather`, чтобы получить температуру в Цельсиях, затем это значение передается в `celsiusToFahrenheit` для конвертации — всё в одном ходе диалога.

<img src="../../../translated_images/ru/tool-chaining-example.538203e73d09dd82.webp" alt="Пример цепочки инструментов" width="800"/>

*Цепочка инструментов в действии — агент сначала вызывает getCurrentWeather, затем передает результат Цельсия в celsiusToFahrenheit и дает объединенный ответ.*

**Обработка сбоев без сбоев** — запросите погоду в городе, которого нет в мок-данных. Инструмент возвращает сообщение об ошибке, и ИИ объясняет, что помочь не может, вместо того чтобы аварийно завершить работу. Инструменты терпят сбои безопасно. Диаграмма ниже показывает два подхода — с правильной обработкой ошибок агент ловит исключение и отвечает полезно, а без него все приложение падает:

<img src="../../../translated_images/ru/error-handling-flow.9a330ffc8ee0475c.webp" alt="Поток обработки ошибок" width="800"/>

*Когда инструмент терпит сбой, агент ловит ошибку и отвечает с полезным объяснением вместо сбоя.*

Это происходит в одном ходе диалога. Агент автономно оркестрирует множественные вызовы инструментов.

## Основные понятия

### Паттерн ReAct (Размышление и Действие)

Агент чередует размышление (решение, что делать) и действие (использование инструментов). Этот паттерн позволяет автономное решение задач, а не просто реагирование на инструкции.

### Описания инструментов имеют значение

Качество описания инструментов напрямую влияет на то, насколько хорошо агент ими пользуется. Чёткие и конкретные описания помогают модели понять, когда и как вызывать каждый инструмент.

### Управление сессиями

Аннотация `@MemoryId` позволяет автоматически управлять памятью на основе сессии. Каждому идентификатору сессии соответствует собственный экземпляр `ChatMemory`, управляемый компонентом `ChatMemoryProvider`, так что несколько пользователей могут взаимодействовать с агентом одновременно без смешения их разговоров. Следующая диаграмма показывает, как разные пользователи направляются в изолированные хранилища памяти на основе ID сессии:

<img src="../../../translated_images/ru/session-management.91ad819c6c89c400.webp" alt="Управление сессиями с помощью @MemoryId" width="800"/>

*Каждому идентификатору сессии соответствует изолированная история разговоров — пользователи не видят сообщения друг друга.*

### Обработка ошибок

Инструменты могут давать сбои — API могут не отвечать, параметры могут быть неверными, внешние сервисы — недоступны. Производственные агенты нуждаются в обработке ошибок, чтобы модель могла объяснять проблемы или пробовать альтернативы, а не вызывать сбой всего приложения. Когда инструмент выбрасывает исключение, LangChain4j его перехватывает и передаёт сообщение об ошибке обратно модели, которая затем может объяснить проблему на естественном языке.
