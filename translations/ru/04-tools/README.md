## Чему вы научитесь

До сих пор вы научились вести диалоги с AI, эффективно структурировать подсказки и основывать ответы на ваших документах. Но остаётся фундаментальное ограничение: языковые модели могут только генерировать текст. Они не могут проверить погоду, выполнять вычисления, запрашивать базы данных или взаимодействовать с внешними системами.

Инструменты меняют это. Предоставляя модели доступ к функциям, которые она может вызвать, вы превращаете её из генератора текста в агента, который может выполнять действия. Модель решает, когда ей нужен инструмент, какой инструмент использовать и какие параметры передать. Ваш код выполняет функцию и возвращает результат. Модель включает этот результат в свой ответ.

## Понимание AI агентов с инструментами

> **📝 Примечание:** Термин «агенты» в этом модуле относится к AI помощникам с возможностями вызова инструментов. Это отличается от паттернов **Agentic AI** (автономных агентов с планированием, памятью и многошаговым рассуждением), которые мы рассмотрим в [модуле 05: MCP](../05-mcp/README.md).

Без инструментов языковая модель может только генерировать текст на основе своих обучающих данных. Если спросить текущую погоду, она будет гадать. Дайте ей инструменты, и она сможет вызвать API погоды, выполнять вычисления или запрашивать базу данных — а затем сшить эти реальные результаты в ответ.

<img src="../../../translated_images/ru/what-are-tools.724e468fc4de64da.webp" alt="Без инструментов vs С инструментами" width="800"/>

*Без инструментов модель только догадывается — с инструментами она может вызывать API, выполнять вычисления и возвращать актуальные данные.*

AI агент с инструментами следует паттерну **Reasoning and Acting (ReAct)**. Модель не просто отвечает — она анализирует, что ей нужно, действует, вызывая инструмент, наблюдает результат и затем решает, нужно ли действовать снова или выдать окончательный ответ:

1. **Размышление** — агент анализирует вопрос пользователя и определяет, какая информация ему нужна
2. **Действие** — агент выбирает правильный инструмент, генерирует корректные параметры и вызывает его
3. **Наблюдение** — агент получает вывод инструмента и оценивает результат
4. **Повтор или ответ** — если нужны дополнительные данные, агент возвращается к началу; иначе формирует ответ на естественном языке

<img src="../../../translated_images/ru/react-pattern-detail.96a5efeeb6dd2f61.webp" alt="Паттерн ReAct" width="800"/>

*Цикл ReAct — агент размышляет, что делать, действует, вызывая инструмент, наблюдает результат и повторяет цикл, пока не сможет выдать окончательный ответ.*

Это происходит автоматически. Вы определяете инструменты и их описания. Модель принимает решения о том, когда и как их использовать.

## Как работает вызов инструментов

### Определения инструментов

[WeatherTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) | [TemperatureTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/TemperatureTool.java)

Вы определяете функции с понятными описаниями и спецификациями параметров. Модель видит эти описания в системном промпте и понимает, что делает каждый инструмент.

```java
@Component
public class WeatherTool {
    
    @Tool("Get the current weather for a location")
    public String getCurrentWeather(@P("Location name") String location) {
        // Ваша логика поиска погоды
        return "Weather in " + location + ": 22°C, cloudy";
    }
}

@AiService
public interface Assistant {
    String chat(@MemoryId String sessionId, @UserMessage String message);
}

// Помощник автоматически настраивается Spring Boot с:
// - Bean модели чата
// - Все методы @Tool из классов @Component
// - ChatMemoryProvider для управления сессиями
```
  
Диаграмма ниже объясняет каждую аннотацию и показывает, как каждый элемент помогает AI понять, когда вызывать инструмент и какие аргументы передавать:

<img src="../../../translated_images/ru/tool-definitions-anatomy.f6468546037cf28b.webp" alt="Анатомия определений инструментов" width="800"/>

*Анатомия определения инструмента — @Tool говорит AI, когда использовать его, @P описывает каждый параметр, а @AiService связывает всё вместе при запуске.*

### Принятие решений

Когда пользователь спрашивает «Какая погода в Сиэтле?», модель не выбирает инструмент случайно. Она сравнивает намерение пользователя с каждым описанием инструмента, оценивает релевантность и выбирает лучший совпадающий. Затем она генерирует структурированный вызов функции с нужными параметрами — в данном случае устанавливая `location` в `"Seattle"`.

Если ни один инструмент не подходит под запрос, модель отвечает исходя из своих знаний. Если несколько инструментов подходят, выбирает самый конкретный.

<img src="../../../translated_images/ru/decision-making.409cd562e5cecc49.webp" alt="Как AI выбирает инструмент" width="800"/>

*Модель оценивает каждый доступный инструмент относительно намерения пользователя и выбирает лучшее совпадение — поэтому важно писать ясные, точные описания инструментов.*

За кулисами полный вызов инструмента проходит через шесть этапов — от вопроса пользователя на естественном языке до ответа тоже на естественном языке:

*Полный поток — пользователь задает вопрос, модель выбирает инструмент, LangChain4j выполняет вызов, и модель вплетает результат в естественный ответ.*

### Архитектура: Авто-связывание Spring Boot

Этот модуль использует интеграцию LangChain4j со Spring Boot с декларативными интерфейсами `@AiService`. При запуске Spring Boot обнаруживает каждый `@Component`, содержащий методы с `@Tool`, ваш бин `ChatModel` и `ChatMemoryProvider` — затем связывает их все в один интерфейс `Assistant` без дополнительного кода.

<img src="../../../translated_images/ru/spring-boot-wiring.151321795988b04e.webp" alt="Архитектура авто-связывания Spring Boot" width="800"/>

*Интерфейс @AiService связывает вместе ChatModel, компоненты инструментов и провайдер памяти — Spring Boot автоматически выполняет всю интеграцию.*

Основные преимущества этого подхода:

- **Авто-связывание Spring Boot** — ChatModel и инструменты автоматически внедряются
- **Паттерн @MemoryId** — автоматическое управление памятью сессий
- **Один экземпляр** — Assistant создаётся один раз и используется повторно для повышения производительности
- **Типобезопасное выполнение** — вызов Java методов напрямую с конверсией типов
- **Оркестрация нескольких шагов** — автоматическая цепочка вызовов инструментов
- **Отсутствие шаблонного кода** — без ручных вызовов `AiServices.builder()` или HashMap для памяти

Альтернативные подходы (ручной `AiServices.builder()`) требуют больше кода и лишены преимуществ интеграции со Spring Boot.

## Цепочка инструментов

**Цепочка инструментов** — настоящая сила агентов на базе инструментов проявляется, когда один вопрос требует нескольких инструментов. Спросите: «Какая погода в Сиэтле по Фаренгейту?» — и агент автоматически связывает два инструмента: сначала вызывает `getCurrentWeather`, чтобы получить температуру в Цельсиях, затем передает это значение в `celsiusToFahrenheit` для конвертации — всё в одном ходе разговора.

<img src="../../../translated_images/ru/tool-chaining-example.538203e73d09dd82.webp" alt="Пример цепочки инструментов" width="800"/>

*Цепочка инструментов в действии — агент сначала вызывает getCurrentWeather, затем передает результат в Цельсиях в celsiusToFahrenheit и выдаёт объединенный ответ.*

Вот как это выглядит в работающем приложении — агент связывает два вызова инструментов в одном ходе разговора:

<a href="images/tool-chaining.png"><img src="../../../translated_images/ru/tool-chaining.3b25af01967d6f7b.webp" alt="Цепочка инструментов" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*Реальный вывод приложения — агент автоматически связывает getCurrentWeather → celsiusToFahrenheit в одном ходе.*

**Плавные сбои** — спросите погоду в городе, которого нет в мок-данных. Инструмент возвращает сообщение об ошибке, и AI объясняет, что не может помочь, вместо того чтобы упасть. Инструменты сбойно обрабатываются безопасно.

<img src="../../../translated_images/ru/error-handling-flow.9a330ffc8ee0475c.webp" alt="Обработка ошибок" width="800"/>

*Когда инструмент выходит из строя, агент ловит ошибку и отвечает полезным объяснением, а не сбойно завершает работу.*

Всё это происходит в одном ходе разговора. Агент самостоятельно оркестрирует несколько вызовов инструментов.

## Ключевые понятия

### Паттерн ReAct (Размышление и действие)

Агент чередует размышления (принятие решения, что делать) и действия (использование инструментов). Этот паттерн позволяет автономно решать задачи, а не просто реагировать на инструкции.

### Описания инструментов имеют значение

Качество описаний ваших инструментов напрямую влияет на то, насколько хорошо агент их использует. Четкие, конкретные описания помогают модели понять, когда и как вызывать каждый инструмент.

### Управление сессиями

Аннотация `@MemoryId` позволяет автоматически управлять памятью на основе сессии. Каждому ID сессии соответствует собственный экземпляр `ChatMemory`, управляемый бином `ChatMemoryProvider`, так что несколько пользователей могут взаимодействовать с агентом одновременно, не мешая друг другу.

<img src="../../../translated_images/ru/session-management.91ad819c6c89c400.webp" alt="Управление сессиями с @MemoryId" width="800"/>

*Каждому ID сессии соответствует изолированная история разговоров — пользователи не видят сообщения друг друга.*

### Обработка ошибок

Инструменты могут давать сбои — API может не ответить вовремя, параметры могут быть неверными, внешние сервисы могут не работать. Для рабочих агентов нужна обработка ошибок, чтобы модель могла объяснить проблему или попробовать альтернативы, вместо того чтобы сбрасывать приложение. Когда инструмент выбрасывает исключение, LangChain4j перехватывает его и передает сообщение об ошибке обратно модели, которая может объяснить проблему естественным языком.

## Когда использовать агентов с инструментами

<img src="../../../translated_images/ru/when-to-use-tools.51d1592d9cbdae9c.webp" alt="Когда использовать инструменты" width="800"/>

*Краткое руководство — инструменты нужны для данных в реальном времени, вычислений и действий; общие знания и творческие задачи этого не требуют.*

**Используйте инструменты, когда:**
- Ответ требует данных в реальном времени (погода, акции, запасы)
- Нужно выполнить вычисления, выходящие за рамки простой арифметики
- Доступ к базам данных или API
- Выполнение действий (отправка писем, создание тикетов, обновление записей)
- Объединение несколько источников данных

**Не используйте инструменты, когда:**
- Вопросы можно ответить на основе общих знаний
- Ответ носит исключительно разговорный характер
- Задержка инструментов сделает взаимодействие слишком медленным

## Инструменты vs RAG

Модули 03 и 04 расширяют возможности ИИ, но принципиально по-разному. RAG дает модели доступ к **знаниям** через извлечение документов. Инструменты дают модели возможность выполнять **действия** через вызов функций.

<img src="../../../translated_images/ru/tools-vs-rag.ad55ce10d7e4da87.webp" alt="Сравнение инструментов и RAG" width="800"/>

*RAG извлекает информацию из статичных документов — Инструменты выполняют действия и получают динамичные, актуальные данные. Во многих продуктивных системах используются обе технологии.*

На практике многие промышленные системы сочетают оба подхода: RAG для обоснования ответов из вашей документации, и инструменты для получения живых данных или выполнения операций.
