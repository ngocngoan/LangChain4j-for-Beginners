## Чему вы научитесь

Вы создали разговорный ИИ, освоили подсказки, связали ответы с документами и создали агентов с инструментами. Но все эти инструменты были специально разработаны для вашего конкретного приложения. А что, если вы могли бы дать вашему ИИ доступ к стандартизированной экосистеме инструментов, которую может создавать и использовать любой? В этом модуле вы узнаете, как именно это сделать с помощью Протокола Контекста Модели (MCP) и агентского модуля LangChain4j. Сначала мы представим простой MCP-читалку файлов, а затем покажем, как её легко интегрировать в продвинутые агентские рабочие процессы с использованием паттерна Агент-наблюдатель.

## Что такое MCP?

Протокол Контекста Модели (MCP) предоставляет именно это — стандартный способ для ИИ-приложений находить и использовать внешние инструменты. Вместо написания индивидуальных интеграций для каждого источника данных или сервиса вы подключаетесь к MCP-серверам, которые показывают свои возможности в едином формате. Ваш агент ИИ после этого может автоматически обнаруживать и использовать эти инструменты.

Схема ниже показывает разницу — без MCP каждая интеграция требует индивидуального точечного подключения; с MCP единый протокол связывает ваше приложение с любым инструментом:

<img src="../../../translated_images/ru/mcp-comparison.9129a881ecf10ff5.webp" alt="MCP Comparison" width="800"/>

*До MCP: сложные точечные интеграции. После MCP: один протокол, бесконечные возможности.*

MCP решает фундаментальную проблему в разработке ИИ: каждая интеграция индивидуальна. Хотите получить доступ к GitHub? Нужен кастомный код. Хотите читать файлы? Кастомный код. Хотите выполнять запросы к базе данных? Кастомный код. И ни одна из этих интеграций не работает с другими ИИ-приложениями.

MCP стандартизирует это. MCP-сервер предоставляет инструменты с чёткими описаниями и схемами параметров. Любой MCP-клиент может подключиться, обнаружить доступные инструменты и использовать их. Создайте один раз — используйте везде.

Схема ниже иллюстрирует эту архитектуру — один MCP-клиент (ваше ИИ-приложение) подключается к нескольким MCP-серверам, каждый из которых предоставляет собственный набор инструментов через стандартный протокол:

<img src="../../../translated_images/ru/mcp-architecture.b3156d787a4ceac9.webp" alt="MCP Architecture" width="800"/>

*Архитектура Протокола Контекста Модели — стандартизированное обнаружение и выполнение инструментов*

## Как работает MCP

Внутри MCP использует многоуровневую архитектуру. Ваше Java-приложение (MCP-клиент) обнаруживает доступные инструменты, отправляет JSON-RPC-запросы через транспортный слой (Stdio или HTTP), а MCP-сервер выполняет операции и возвращает результаты. Следующая схема разбивает протокол по слоям:

<img src="../../../translated_images/ru/mcp-protocol-detail.01204e056f45308b.webp" alt="MCP Protocol Detail" width="800"/>

*Как работает MCP — клиенты обнаруживают инструменты, обмениваются сообщениями JSON-RPC и выполняют операции через транспортный слой.*

**Архитектура сервер-клиент**

MCP основан на модели клиент-сервер. Серверы предоставляют инструменты — чтение файлов, запросы к базам данных, вызов API. Клиенты (ваше ИИ-приложение) подключаются к серверам и используют их инструменты.

Для использования MCP с LangChain4j добавьте следующую зависимость Maven:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```

**Обнаружение инструментов**

Когда ваш клиент подключается к MCP-серверу, он спрашивает: «Какие у тебя есть инструменты?» Сервер отвечает списком инструментов, каждый с описанием и схемой параметров. Ваш агент ИИ потом решает, какие инструменты использовать на основе запросов пользователя. На схеме ниже показано это рукопожатие — клиент отправляет запрос `tools/list`, сервер возвращает доступные инструменты с описаниями и схемами параметров:

<img src="../../../translated_images/ru/tool-discovery.07760a8a301a7832.webp" alt="MCP Tool Discovery" width="800"/>

*ИИ обнаруживает доступные инструменты при запуске — теперь он знает, какие возможности доступны и может решить, какие использовать.*

**Транспортные механизмы**

MCP поддерживает разные транспортные механизмы. Два варианта — Stdio (для локального взаимодействия с подпроцессами) и Streamable HTTP (для удалённых серверов). Этот модуль демонстрирует транспорт Stdio:

<img src="../../../translated_images/ru/transport-mechanisms.2791ba7ee93cf020.webp" alt="Transport Mechanisms" width="800"/>

*Транспортные механизмы MCP: HTTP для удалённых серверов, Stdio для локальных процессов*

**Stdio** — [StdioTransportDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java)

Для локальных процессов. Ваше приложение запускает сервер как подпроцесс и общается через стандартный ввод/вывод. Полезно для работы с файловой системой или инструментами командной строки.

```java
McpTransport stdioTransport = new StdioMcpTransport.Builder()
    .command(List.of(
        npmCmd, "exec",
        "@modelcontextprotocol/server-filesystem@2025.12.18",
        resourcesDir
    ))
    .logEvents(false)
    .build();
```

Сервер `@modelcontextprotocol/server-filesystem` предоставляет следующие инструменты, все ограничены доступом к указанным вами директориям:

| Инструмент | Описание |
|------|-------------|
| `read_file` | Считать содержимое одного файла |
| `read_multiple_files` | Считать несколько файлов за один вызов |
| `write_file` | Создать или перезаписать файл |
| `edit_file` | Выполнить целевую замену текста |
| `list_directory` | Список файлов и каталогов в указанном пути |
| `search_files` | Рекурсивный поиск файлов по шаблону |
| `get_file_info` | Получить метаданные файла (размер, временные метки, права) |
| `create_directory` | Создать директорию (включая родительские каталоги) |
| `move_file` | Переместить или переименовать файл или каталог |

Следующая схема показывает, как транспорт Stdio работает во время выполнения — ваше Java-приложение запускает MCP-сервер как дочерний процесс и они общаются через каналы stdin/stdout, без использования сети или HTTP:

<img src="../../../translated_images/ru/stdio-transport-flow.45eaff4af2d81db4.webp" alt="Stdio Transport Flow" width="800"/>

*Транспорт Stdio в действии — ваше приложение запускает MCP-сервер как дочерний процесс и обменивается данными через stdin/stdout.*

## Агентский модуль

Хотя MCP предоставляет стандартизированные инструменты, агентский модуль LangChain4j предоставляет декларативный способ создавать агентов, которые оркеструют эти инструменты. Аннотация `@Agent` и `AgenticServices` позволяют определять поведение агентов через интерфейсы, а не императивный код.

В этом модуле вы познакомитесь с паттерном **Агент-наблюдатель** — продвинутый агентский подход, при котором агент-наблюдатель динамически решает, каких субагентов вызывать на основе запросов пользователя. Мы объединим обе концепции, предоставив одному из субагентов возможности доступа к файлам через MCP.

Для использования агентского модуля добавьте следующую зависимость Maven:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```
> **Примечание:** Модуль `langchain4j-agentic` использует отдельное свойство версии (`langchain4j.mcp.version`), поскольку выпускается по другому графику, чем основные библиотеки LangChain4j.

> **⚠️ Экспериментально:** Модуль `langchain4j-agentic` **экспериментальный** и может изменяться. Стабильным способом создания ИИ-ассистентов остаётся `langchain4j-core` с кастомными инструментами (Модуль 04).

### Агент-наблюдатель

Паттерн **Агент-наблюдатель** — это **гибкая** форма агентского ИИ. Наблюдатель использует LLM, чтобы автономно решать, каких агентов вызывать на основе запроса пользователя. В следующем примере мы объединяем доступ к файлам через MCP с LLM-агентом для создания рабочего процесса чтения файла → отчёт.

В демо `FileAgent` считывает файл с помощью MCP-инструментов файловой системы, а `ReportAgent` создаёт структурированный отчёт с исполнительным резюме (1 предложение), 3 ключевыми пунктами и рекомендациями. Наблюдатель автоматически оркеструет этот поток:

<img src="../../../translated_images/ru/supervisor-agent-pattern.06275a41ae006ac8.webp" alt="Supervisor Agent Pattern" width="800"/>

*Наблюдатель использует свой LLM, чтобы решить, каких агентов и в каком порядке вызывать — никакой жёстко закодированной маршрутизации.*

Вот конкретный рабочий процесс нашей цепочки файл → отчёт:

<img src="../../../translated_images/ru/file-report-workflow.649bb7a896800de9.webp" alt="File to Report Workflow" width="800"/>

*FileAgent считывает файл через инструменты MCP, затем ReportAgent преобразует сырой контент в структурированный отчёт.*

Каждый агент сохраняет свой вывод в **Agentic Scope** (общая память), что позволяет последующим агентам получать доступ к предыдущим результатам. Это демонстрирует, как инструменты MCP бесшовно интегрируются в агентские рабочие процессы — Наблюдатель не нуждается в знании *как* читаются файлы, важно только, что `FileAgent` может это делать.

#### Как работает Наблюдатель

Перед созданием агентов необходимо подключить MCP-транспорт к клиенту и обернуть его в `ToolProvider`. Так инструменты MCP-сервера становятся доступны вашим агентам:

```java
// Создать MCP клиент из транспорта
McpClient mcpClient = new DefaultMcpClient.Builder()
        .transport(stdioTransport)
        .build();

// Обернуть клиента в ToolProvider — это связывает MCP инструменты с LangChain4j
ToolProvider mcpToolProvider = McpToolProvider.builder()
        .mcpClients(List.of(mcpClient))
        .build();
```

Теперь вы можете внедрить `mcpToolProvider` в любой агент, которому нужны MCP-инструменты:

```java
// Шаг 1: FileAgent читает файлы с помощью инструментов MCP
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // Имеет инструменты MCP для операций с файлами
        .build();

// Шаг 2: ReportAgent генерирует структурированные отчёты
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// Supervisor координирует рабочий процесс файл → отчёт
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // Вернуть итоговый отчёт
        .build();

// Supervisor решает, каких агентов вызывать на основе запроса
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```

#### Стратегии ответа

При конфигурации `SupervisorAgent` вы указываете, как он должен формулировать итоговый ответ пользователю после выполнения субагентов. Схема ниже показывает три доступные стратегии — LAST возвращает напрямую вывод последнего агента, SUMMARY синтезирует все выводы через LLM, а SCORED выбирает тот вариант, который получает более высокий балл по сравнению с исходным запросом:

<img src="../../../translated_images/ru/response-strategies.3d0cea19d096bdf9.webp" alt="Response Strategies" width="800"/>

*Три стратегии формирования финального ответа Наблюдателем — выбирайте исходя из того, хотите ли вы получить вывод последнего агента, синтезированное резюме или лучший по оценке вариант.*

Доступные стратегии:

| Стратегия | Описание |
|----------|-------------|
| **LAST** | Наблюдатель возвращает вывод последнего вызванного субагента или инструмента. Полезно, когда последний агент в рабочем процессе рассчитан на создание полного итогового ответа (например, "Агент-резюме" в исследовательском процессе). |
| **SUMMARY** | Наблюдатель использует внутренний языковой модель (LLM) для синтеза общего резюме всего взаимодействия и результатов субагентов, затем возвращает это резюме как окончательный ответ. Это обеспечивает аккуратный агрегированный ответ пользователю. |
| **SCORED** | Система использует внутренний LLM для оценки как ответа LAST, так и резюме SUMMARY над исходным запросом пользователя, возвращая вариант с более высоким баллом. |
Смотрите [SupervisorAgentDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/SupervisorAgentDemo.java) для полного примера реализации.

#### Понимание вывода

Когда вы запустите демонстрацию, увидите структурированный пройденный путь того, как Supervisor координирует нескольких агентов. Вот что означает каждый раздел:

```
======================================================================
  FILE → REPORT WORKFLOW DEMO
======================================================================

This demo shows a clear 2-step workflow: read a file, then generate a report.
The Supervisor orchestrates the agents automatically based on the request.
```

**Заголовок** вводит концепцию workflow: сфокусированный конвейер от чтения файла до создания отчёта.

```
--- WORKFLOW ---------------------------------------------------------
  ┌─────────────┐      ┌──────────────┐
  │  FileAgent  │ ───▶ │ ReportAgent  │
  │ (MCP tools) │      │  (pure LLM)  │
  └─────────────┘      └──────────────┘
   outputKey:           outputKey:
   'fileContent'        'report'

--- AVAILABLE AGENTS -------------------------------------------------
  [FILE]   FileAgent   - Reads files via MCP → stores in 'fileContent'
  [REPORT] ReportAgent - Generates structured report → stores in 'report'
```

**Диаграмма Workflow** показывает поток данных между агентами. Каждый агент выполняет конкретную роль:
- **FileAgent** читает файлы с помощью инструментов MCP и сохраняет сырой контент в `fileContent`
- **ReportAgent** потребляет этот контент и создаёт структурированный отчёт в `report`

```
--- USER REQUEST -----------------------------------------------------
  "Read the file at .../file.txt and generate a report on its contents"
```

**Запрос пользователя** показывает задачу. Supervisor анализирует его и решает вызвать FileAgent → ReportAgent.

```
--- SUPERVISOR ORCHESTRATION -----------------------------------------
  The Supervisor decides which agents to invoke and passes data between them...

  +-- STEP 1: Supervisor chose -> FileAgent (reading file via MCP)
  |
  |   Input: .../file.txt
  |
  |   Result: LangChain4j is an open-source, provider-agnostic Java framework for building LLM...
  +-- [OK] FileAgent (reading file via MCP) completed

  +-- STEP 2: Supervisor chose -> ReportAgent (generating structured report)
  |
  |   Input: LangChain4j is an open-source, provider-agnostic Java framew...
  |
  |   Result: Executive Summary...
  +-- [OK] ReportAgent (generating structured report) completed
```

**Оркестрация Supervisor** демонстрирует 2-этапный процесс:
1. **FileAgent** читает файл через MCP и сохраняет содержимое
2. **ReportAgent** получает содержимое и генерирует структурированный отчёт

Решения Supervisor были приняты **автономно** на основе запроса пользователя.

```
--- FINAL RESPONSE ---------------------------------------------------
Executive Summary
...

Key Points
...

Recommendations
...

--- AGENTIC SCOPE (Data Flow) ----------------------------------------
  Each agent stores its output for downstream agents to consume:
  * fileContent: LangChain4j is an open-source, provider-agnostic Java framework...
  * report: Executive Summary...
```

#### Объяснение функций модуля Agentic

Пример демонстрирует несколько продвинутых функций модуля agentic. Рассмотрим подробнее Agentic Scope и Agent Listeners.

**Agentic Scope** показывает общую память, где агенты сохраняли результаты с помощью `@Agent(outputKey="...")`. Это позволяет:
- Более поздним агентам получать результаты ранних
- Supervisor синтезировать конечный ответ
- Вам просматривать, что произвел каждый агент

Диаграмма ниже показывает, как Agentic Scope работает как общая память в workflow от файла к отчёту — FileAgent пишет результат под ключом `fileContent`, ReportAgent читает его и записывает свой результат под ключом `report`:

<img src="../../../translated_images/ru/agentic-scope.95ef488b6c1d02ef.webp" alt="Общая память Agentic Scope" width="800"/>

*Agentic Scope действует как общая память — FileAgent записывает `fileContent`, ReportAgent читает и пишет `report`, и ваш код читает итоговый результат.*

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // Исходные данные файла от FileAgent
String report = scope.readState("report");            // Структурированный отчет от ReportAgent
```

**Agent Listeners** позволяют отслеживать и отлаживать выполнение агентов. Пошаговый вывод, который вы видите в демо, получается от AgentListener, который подключается к каждому вызову агента:
- **beforeAgentInvocation** — вызывается, когда Supervisor выбирает агента, чтобы вы видели, какой агент выбран и почему
- **afterAgentInvocation** — вызывается после завершения агента, показывая его результат
- **inheritedBySubagents** — когда true, слушатель отслеживает всех агентов в иерархии

Следующая диаграмма показывает полный жизненный цикл Agent Listener, включая обработку ошибок в `onError` во время выполнения агента:

<img src="../../../translated_images/ru/agent-listeners.784bfc403c80ea13.webp" alt="Жизненный цикл Agent Listeners" width="800"/>

*Agent Listeners подключены к жизненному циклу выполнения — отслеживают запуск, завершение и ошибки агентов.*

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // Распространить на всех подагентов
    }
};
```

Помимо паттерна Supervisor, модуль `langchain4j-agentic` предоставляет несколько мощных паттернов workflow. Диаграмма ниже показывает все пять — от простых последовательных конвейеров до рабочих процессов с участием человека:

<img src="../../../translated_images/ru/workflow-patterns.82b2cc5b0c5edb22.webp" alt="Паттерны Agent Workflow" width="800"/>

*Пять паттернов workflow для организации агентов — от простых последовательных конвейеров до процессов с участием человека.*

| Паттерн | Описание | Сценарий использования |
|---------|-------------|----------|
| **Sequential** | Выполняет агентов по очереди, выход передается следующему | Конвейеры: исследование → анализ → отчет |
| **Parallel** | Запускает агентов одновременно | Независимые задачи: погода + новости + акции |
| **Loop** | Итерации пока не выполнено условие | Оценка качества: уточнение пока балл ≥ 0.8 |
| **Conditional** | Маршрутизация на основе условий | Классификация → отправка к специалисту |
| **Human-in-the-Loop** | Добавляет контрольные точки с участием человека | Рабочие процессы с одобрением, проверка контента |

## Ключевые концепции

Теперь, когда вы изучили MCP и модуль agentic на практике, подведём итоги, когда использовать каждый подход.

Одним из главных преимуществ MCP является растущая экосистема. Диаграмма ниже показывает, как единый универсальный протокол связывает ваше AI-приложение с широким спектром MCP-серверов — от доступа к файловой системе и базам данных до GitHub, email, веб-скрапинга и прочего:

<img src="../../../translated_images/ru/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="Экосистема MCP" width="800"/>

*MCP создает экосистему универсальных протоколов — любой сервер, совместимый с MCP, работает с любым клиентом MCP, позволяя делиться инструментами между приложениями.*

**MCP** идеален, когда вы хотите использовать существующие экосистемы инструментов, создавать инструменты для совместного использования несколькими приложениями, интегрировать сторонние сервисы через стандартные протоколы или менять реализации инструментов без изменения кода.

**Модуль Agentic** лучше подходит, когда нужны декларативные определения агентов с аннотациями `@Agent`, требуется оркестрация workflow (последовательная, цикл, параллельная), предпочтителен интерфейсный дизайн агентов вместо императивного кода или когда объединяются несколько агентов, которые разделяют данные через `outputKey`.

**Паттерн Supervisor Agent** особенно полезен, когда workflow не предсказуем заранее и нужно, чтобы LLM принимал решения, когда есть несколько специализированных агентов для динамической оркестрации, при создании разговорных систем с маршрутизацией по функциям или когда требуется максимально гибкое и адаптивное поведение агента.
