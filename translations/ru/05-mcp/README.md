## Что вы узнаете

Вы уже создали разговорный ИИ, освоили промпты, основали ответы на документах и создали агентов с инструментами. Но все эти инструменты были кастомными, специально разработанными для вашего конкретного приложения. А что если вы могли бы дать вашему ИИ доступ к стандартизированной экосистеме инструментов, которые любой может создавать и делиться ими? В этом модуле вы узнаете, как сделать именно это с помощью Протокола Контекста Модели (MCP) и агентского модуля LangChain4j. Сначала мы показываем простой MCP-читалку файлов, а затем демонстрируем, как она легко интегрируется в продвинутые агентские сценарии с использованием паттерна Супервизорного агента.

## Что такое MCP?

Протокол Контекста Модели (MCP) предоставляет именно это — стандартный способ для ИИ-приложений обнаруживать и использовать внешние инструменты. Вместо написания кастомных интеграций для каждого источника данных или сервиса, вы подключаетесь к MCP-серверам, которые предоставляют свои возможности в едином формате. Ваш ИИ-агент может автоматически находить и использовать эти инструменты.

<img src="../../../translated_images/ru/mcp-comparison.9129a881ecf10ff5.webp" alt="MCP Comparison" width="800"/>

*До MCP: сложные точечные интеграции. После MCP: один протокол, бесконечные возможности.*

MCP решает фундаментальную проблему разработки ИИ: каждая интеграция — уникальная. Хотите получить доступ к GitHub? Кастомный код. Хотите читать файлы? Кастомный код. Хотите опрашивать базу данных? Кастомный код. И ни одна из этих интеграций не работает с другими ИИ-приложениями.

MCP стандартизирует этот процесс. MCP-сервер предоставляет инструменты с чёткими описаниями и схемами. Любой MCP-клиент может подключиться, обнаружить доступные инструменты и использовать их. Создай один раз — используй везде.

<img src="../../../translated_images/ru/mcp-architecture.b3156d787a4ceac9.webp" alt="MCP Architecture" width="800"/>

*Архитектура Протокола Контекста Модели — стандартизованное обнаружение и выполнение инструментов*

## Как работает MCP

<img src="../../../translated_images/ru/mcp-protocol-detail.01204e056f45308b.webp" alt="MCP Protocol Detail" width="800"/>

*Как MCP работает внутри — клиенты обнаруживают инструменты, обмениваются сообщениями JSON-RPC и выполняют операции через транспортный уровень.*

**Архитектура сервер-клиент**

MCP использует модель клиент-сервер. Серверы предлагают инструменты — чтение файлов, запросы к базам данных, вызовы API. Клиенты (ваше ИИ-приложение) подключаются к серверам и используют их инструменты.

Чтобы использовать MCP с LangChain4j, добавьте эту зависимость Maven:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```

**Обнаружение инструментов**

Когда ваш клиент подключается к MCP-серверу, он спрашивает: "Какие у тебя есть инструменты?" Сервер отвечает списком доступных инструментов с описаниями и схемами параметров. Ваш ИИ-агент затем решает, какие инструменты использовать в зависимости от запросов пользователя.

<img src="../../../translated_images/ru/tool-discovery.07760a8a301a7832.webp" alt="MCP Tool Discovery" width="800"/>

*ИИ обнаруживает доступные инструменты при запуске — теперь он знает, какие возможности доступны, и может решить, какие использовать.*

**Транспортные механизмы**

MCP поддерживает разные транспортные механизмы. В этом модуле демонстрируется транспорт Stdio для локальных процессов:

<img src="../../../translated_images/ru/transport-mechanisms.2791ba7ee93cf020.webp" alt="Transport Mechanisms" width="800"/>

*Транспортные механизмы MCP: HTTP для удалённых серверов, Stdio для локальных процессов*

**Stdio** — [StdioTransportDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java)

Для локальных процессов. Ваше приложение запускает сервер как дочерний процесс и общается через стандартный ввод/вывод. Полезно для доступа к файловой системе или командным инструментам.

```java
McpTransport stdioTransport = new StdioMcpTransport.Builder()
    .command(List.of(
        npmCmd, "exec",
        "@modelcontextprotocol/server-filesystem@2025.12.18",
        resourcesDir
    ))
    .logEvents(false)
    .build();
```

<img src="../../../translated_images/ru/stdio-transport-flow.45eaff4af2d81db4.webp" alt="Stdio Transport Flow" width="800"/>

*Транспорт Stdio в действии — ваше приложение запускает MCP-сервер как дочерний процесс и обменивается данными через потоки stdin/stdout.*

## Агентский модуль

Хотя MCP предоставляет стандартизированные инструменты, агентский модуль LangChain4j предлагает декларативный способ создавать агентов, которые оркеструют эти инструменты. Аннотация `@Agent` и `AgenticServices` позволяют определять поведение агента через интерфейсы, а не императивный код.

В этом модуле вы познакомитесь с паттерном **Супервизорного агента** — продвинутым агентским подходом, где "супервизор" динамически решает, каких подагентов запускать на основе запросов пользователя. Мы объединим оба подхода, наделив одного из подагентов возможностями MCP-доступа к файлам.

Чтобы использовать агентский модуль, добавьте эту зависимость Maven:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```

> **⚠️ Экспериментально:** модуль `langchain4j-agentic` является **экспериментальным** и может измениться. Стандартный способ создания ИИ ассистентов по-прежнему остаётся `langchain4j-core` с кастомными инструментами (Модуль 04).

### Супервизор агента

Паттерн **Супервизор агента** — это **гибкая** форма агентского ИИ. Супервизор использует LLM для автономного решения, каких агентов запускать на основе запроса пользователя. В следующем примере мы объединим MCP-поддержку доступа к файлам с LLM-агентом, чтобы создать контролируемый рабочий процесс: чтение файла → составление отчёта.

В демонстрации `FileAgent` читает файл с помощью MCP-инструментов файловой системы, а `ReportAgent` создаёт структурированный отчёт с итоговым резюме (1 предложение), 3 ключевыми пунктами и рекомендациями. Супервизор автоматически управляет этим процессом:

<img src="../../../translated_images/ru/supervisor-agent-pattern.06275a41ae006ac8.webp" alt="Supervisor Agent Pattern" width="800"/>

*Супервизор использует свой LLM, чтобы решать, каких агентов запускать и в каком порядке — без жёстко заданной маршрутизации.*

Вот как выглядит конкретный рабочий процесс для нашей схемы «файл → отчёт»:

<img src="../../../translated_images/ru/file-report-workflow.649bb7a896800de9.webp" alt="File to Report Workflow" width="800"/>

*FileAgent читает файл через MCP-инструменты, затем ReportAgent преобразует сырой контент в структурированный отчёт.*

Каждый агент сохраняет свой вывод в **Agentic Scope** (общая память), позволяя последующим агентам получать доступ к предыдущим результатам. Это демонстрирует, как MCP инструменты плавно интегрируются в агентские цепочки — супервизору не нужно знать *как* читать файлы, достаточно, что `FileAgent` это умеет.

#### Как работает супервизор

```java
// Шаг 1: FileAgent читает файлы с помощью инструментов MCP
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // Имеет инструменты MCP для операций с файлами
        .build();

// Шаг 2: ReportAgent генерирует структурированные отчёты
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// Supervisor организует рабочий процесс от файла к отчёту
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // Вернуть конечный отчёт
        .build();

// Supervisor решает, каких агентов вызывать в зависимости от запроса
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```

#### Стратегии ответа

При настройке `SupervisorAgent` вы указываете, как он должен формулировать окончательный ответ пользователю после того, как подагенты завершат задачи.

<img src="../../../translated_images/ru/response-strategies.3d0cea19d096bdf9.webp" alt="Response Strategies" width="800"/>

*Три стратегии формулировки финального ответа Супервизором — выбирайте в зависимости от того, хотите ли вы взять вывод последнего агента, обобщённое резюме или лучшее по оценке.*

Доступные стратегии:

| Стратегия | Описание |
|----------|-------------|
| **LAST** | супервизор возвращает вывод последнего вызванного подагента или инструмента. Полезно, когда последний агент генерирует полный финальный ответ (например, "Agent сводки" в исследовательском пайплайне). |
| **SUMMARY** | супервизор использует свой внутренний языковой моделью (LLM), чтобы синтезировать резюме всего взаимодействия и всех выводов подагентов, а затем возвращает это резюме как финальный ответ. Это даёт чистый агрегированный ответ пользователю. |
| **SCORED** | система использует внутренний LLM для оценки как ответа LAST, так и резюме SUMMARY относительно оригинального запроса пользователя, возвращая тот вывод, который набрал более высокий балл. |

См. полный пример в [SupervisorAgentDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/SupervisorAgentDemo.java).

#### Объяснение возможностей агентского модуля

Пример демонстрирует несколько продвинутых функций агентского модуля. Рассмотрим подробнее Agentic Scope и Agent Listeners.

**Agentic Scope** показывает общую память, где агенты сохраняют результаты с помощью `@Agent(outputKey="...")`. Это позволяет:
- Позднее доступать к выводам предыдущих агентов
- Супервизору синтезировать финальный ответ
- Вам инспектировать то, что произвёл каждый агент

<img src="../../../translated_images/ru/agentic-scope.95ef488b6c1d02ef.webp" alt="Agentic Scope Shared Memory" width="800"/>

*Agentic Scope выступает общей памятью — FileAgent записывает `fileContent`, ReportAgent читает его и записывает `report`, а ваш код читает финальный результат.*

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // Исходные данные файла от FileAgent
String report = scope.readState("report");            // Структурированный отчет от ReportAgent
```

**Agent Listeners** позволяют отслеживать и отлаживать выполнение агентов. Пошаговый вывод, который вы видите в демонстрации, поступает от AgentListener, который подключается к каждому вызову агента:
- **beforeAgentInvocation** - Вызывается, когда Супервайзер выбирает агента, позволяя увидеть, какой агент был выбран и почему  
- **afterAgentInvocation** - Вызывается после завершения агента, показывая его результат  
- **inheritedBySubagents** - Если true, слушатель отслеживает всех агентов в иерархии  

<img src="../../../translated_images/ru/agent-listeners.784bfc403c80ea13.webp" alt="Жизненный цикл слушателей агентов" width="800"/>

*Слушатели агентов подключаются к жизненному циклу выполнения — отслеживают запуск агентов, их завершение или возникновение ошибок.*

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // Распространить на всех подагентов
    }
};
```
  
Помимо паттерна Супервайзера, модуль `langchain4j-agentic` предоставляет несколько мощных паттернов и функций рабочих процессов:

<img src="../../../translated_images/ru/workflow-patterns.82b2cc5b0c5edb22.webp" alt="Паттерны рабочих процессов агентов" width="800"/>

*Пять паттернов рабочих процессов для оркестровки агентов — от простых последовательных конвейеров до рабочих процессов с участием человека для утверждений.*

| Паттерн | Описание | Пример использования |
|---------|-------------|----------|
| **Последовательный** | Выполнение агентов по порядку, вывод поступает следующему | Конвейеры: исследование → анализ → отчет |
| **Параллельный** | Запуск агентов одновременно | Независимые задачи: погода + новости + акции |
| **Цикл** | Итерация до выполнения условия | Оценка качества: уточнять, пока балл ≥ 0.8 |
| **Условный** | Маршрутизация на основе условий | Классификация → направить к специализированному агенту |
| **Человек в цикле** | Добавление человеческих контрольных точек | Рабочие процессы с утверждением, проверка контента |

## Ключевые понятия

Теперь, когда вы изучили MCP и модуль agentic на практике, подытожим когда использовать каждый из подходов.

<img src="../../../translated_images/ru/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="Экосистема MCP" width="800"/>

*MCP создает универсальную экосистему протоколов — любой сервер, совместимый с MCP, работает с любым клиентом MCP, позволяя делиться инструментами между приложениями.*

**MCP** идеален, если вы хотите использовать существующие экосистемы инструментов, создавать инструменты для нескольких приложений, интегрировать сторонние сервисы через стандартизованные протоколы или менять реализации инструментов без изменения кода.

**Модуль Agentic** лучше подходит, если вы хотите декларативное определение агентов с аннотациями `@Agent`, нуждаетесь в оркестрации рабочих процессов (последовательной, циклической, параллельной), предпочитаете дизайн агентов на основе интерфейсов вместо императивного кода, или объединяете нескольких агентов с общими выходами через `outputKey`.

**Паттерн агента Супервайзера** выделяется, когда рабочий процесс заранее непредсказуем и вы хотите, чтобы LLM принимал решения, когда у вас несколько специализированных агентов, требующих динамической оркестрации, при создании разговорных систем с маршрутизацией по возможностям или когда нужна максимальная гибкость и адаптивность поведения агента.

<img src="../../../translated_images/ru/custom-vs-mcp-tools.c4f9b6b1cb65d8a1.webp" alt="Пользовательские инструменты против инструментов MCP" width="800"/>

*Когда использовать пользовательские методы @Tool и когда инструменты MCP — пользовательские инструменты для логики приложения с полной типовой безопасностью, инструменты MCP для стандартизированных интеграций, работающих между приложениями.*
