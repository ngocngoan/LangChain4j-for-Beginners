## Чему вы научитесь

Вы создали разговорный ИИ, освоили промпты, связали ответы с документами и создали агентов с инструментами. Но все эти инструменты были специально разработаны для вашего приложения. Что если вы могли бы дать вашему ИИ доступ к стандартизированной экосистеме инструментов, которые любой может создавать и делиться ими? В этом модуле вы научитесь именно этому с помощью Протокола Контекста Модели (MCP) и агентского модуля LangChain4j. Сначала мы показываем простой MCP ридер файлов, а затем демонстрируем, как его легко интегрировать в продвинутые агентские рабочие процессы, используя паттерн Супервизор-агент.

## Что такое MCP?

Протокол Контекста Модели (MCP) предоставляет именно это — стандартный способ для ИИ-приложений находить и использовать внешние инструменты. Вместо создания отдельных интеграций для каждого источника данных или сервиса вы подключаетесь к MCP серверам, которые предоставляют свои возможности в едином формате. Ваш ИИ-агент может автоматически обнаруживать и использовать эти инструменты.

На диаграмме ниже показано различие — без MCP каждая интеграция требует индивидуального соединения точка-точка; с MCP один протокол подключает ваше приложение к любому инструменту:

<img src="../../../translated_images/ru/mcp-comparison.9129a881ecf10ff5.webp" alt="MCP Comparison" width="800"/>

*До MCP: сложные интеграции точка-точка. После MCP: один протокол, безграничные возможности.*

MCP решает фундаментальную проблему в разработке ИИ: каждая интеграция индивидуальна. Хотите получить доступ к GitHub? Свой код. Хотите читать файлы? Свой код. Хотите опросить базу данных? Свой код. И ни одна из этих интеграций не работает с другими ИИ-приложениями.

MCP стандартизирует это. MCP сервер предоставляет инструменты с понятными описаниями и схемами. Любой MCP клиент может подключаться, обнаруживать доступные инструменты и использовать их. Создай один раз — используй везде.

Ниже схема этой архитектуры — один MCP клиент (ваше ИИ-приложение) подключается к нескольким MCP серверам, каждый из которых предоставляет свой набор инструментов через стандартный протокол:

<img src="../../../translated_images/ru/mcp-architecture.b3156d787a4ceac9.webp" alt="MCP Architecture" width="800"/>

*Архитектура Протокола Контекста Модели — стандартизированное обнаружение и выполнение инструментов*

## Как работает MCP

Под капотом MCP использует слоистую архитектуру. Ваше Java-приложение (MCP клиент) обнаруживает доступные инструменты, отправляет JSON-RPC запросы через транспортный слой (Stdio или HTTP), а MCP сервер выполняет операции и возвращает результаты. Следующая диаграмма разбирает каждый слой этого протокола:

<img src="../../../translated_images/ru/mcp-protocol-detail.01204e056f45308b.webp" alt="MCP Protocol Detail" width="800"/>

*Как MCP работает под капотом — клиенты обнаруживают инструменты, обмениваются JSON-RPC сообщениями и выполняют операции через транспортный слой.*

**Архитектура клиент-сервер**

MCP использует модель клиент-сервер. Серверы предоставляют инструменты — чтение файлов, запросы к базам данных, вызовы API. Клиенты (ваше ИИ-приложение) подключаются к серверам и используют их инструменты.

Чтобы использовать MCP с LangChain4j, добавьте эту зависимость Maven:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```

**Обнаружение инструментов**

Когда ваш клиент подключается к MCP серверу, он спрашивает: "Какие у тебя инструменты?" Сервер отвечает списком доступных инструментов, каждый с описаниями и схемами параметров. Ваш ИИ-агент затем может решить, какие инструменты использовать в зависимости от запроса пользователя. На диаграмме ниже показано это рукопожатие — клиент отправляет запрос `tools/list`, сервер возвращает свои доступные инструменты с описаниями и схемами параметров:

<img src="../../../translated_images/ru/tool-discovery.07760a8a301a7832.webp" alt="MCP Tool Discovery" width="800"/>

*ИИ обнаруживает доступные инструменты при запуске — теперь оно знает, какие возможности доступны, и может решить, какие использовать.*

**Транспортные механизмы**

MCP поддерживает разные транспортные механизмы. Два варианта — Stdio (для локального взаимодействия с подпроцессами) и Streamable HTTP (для удалённых серверов). Этот модуль демонстрирует транспорт Stdio:

<img src="../../../translated_images/ru/transport-mechanisms.2791ba7ee93cf020.webp" alt="Transport Mechanisms" width="800"/>

*Транспортные механизмы MCP: HTTP для удалённых серверов, Stdio для локальных процессов*

**Stdio** — [StdioTransportDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java)

Для локальных процессов. Ваше приложение запускает сервер как подпроцесс и общается через стандартный ввод/вывод. Полезно для доступа к файловой системе или коммандных инструментов.

```java
McpTransport stdioTransport = new StdioMcpTransport.Builder()
    .command(List.of(
        npmCmd, "exec",
        "@modelcontextprotocol/server-filesystem@2025.12.18",
        resourcesDir
    ))
    .logEvents(false)
    .build();
```

Сервер `@modelcontextprotocol/server-filesystem` предоставляет следующие инструменты, все ограничены указанными вами директориями:

| Инструмент | Описание |
|------|-------------|
| `read_file` | Чтение содержимого одного файла |
| `read_multiple_files` | Чтение нескольких файлов за один вызов |
| `write_file` | Создание или перезапись файла |
| `edit_file` | Целевое нахождение и замена в файле |
| `list_directory` | Список файлов и каталогов по пути |
| `search_files` | Рекурсивный поиск файлов по шаблону |
| `get_file_info` | Получение метаданных файла (размер, временные метки, права) |
| `create_directory` | Создание каталога (включая родительские каталоги) |
| `move_file` | Перемещение или переименование файла или каталога |

<img src="../../../translated_images/ru/stdio-transport-flow.45eaff4af2d81db4.webp" alt="Stdio Transport Flow" width="800"/>

*Транспорт Stdio в действии — приложение запускает MCP сервер как дочерний процесс и общается через каналы stdin/stdout.*

## Агентский Модуль

Хотя MCP предоставляет стандартизированные инструменты, **агентский модуль** LangChain4j даёт декларативный способ строить агентов, которые оркеструют эти инструменты. Аннотация `@Agent` и `AgenticServices` позволяют определить поведение агента через интерфейсы, а не императивный код.

В этом модуле вы изучите паттерн **Супервизор-Агент** — продвинутый агентский ИИ, где "супервизор" динамически решает, каких субагентов вызывать на основе запроса пользователя. Мы объединим оба концепта, предоставив одному из наших субагентов возможности доступа к файлам через MCP.

Чтобы использовать агентский модуль, добавьте эту зависимость Maven:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```
> **Примечание:** Модуль `langchain4j-agentic` использует отдельное свойство версии (`langchain4j.mcp.version`), так как выпускается по другому расписанию, чем основные библиотеки LangChain4j.

> **⚠️ Экспериментально:** Модуль `langchain4j-agentic` является **экспериментальным** и может измениться. Стабильным вариантом создания ИИ ассистентов остаётся `langchain4j-core` с пользовательскими инструментами (Модуль 04).

### Супервизор-Агент

Паттерн **Супервизор-Агент** — это **гибкая** форма агентского ИИ. Супервизор использует LLM для автономного решения, каких агентов вызывать на основе запроса пользователя. В следующем примере мы комбинируем MCP-инструменты доступа к файлам с LLM-агентом, создавая контролируемый рабочий процесс чтения файла → отчёта.

В демо `FileAgent` читает файл с помощью MCP инструментов файловой системы, а `ReportAgent` генерирует структурированный отчёт с кратким резюме (1 предложение), 3 ключевыми моментами и рекомендациями. Супервизор автоматически оркестрирует этот процесс:

<img src="../../../translated_images/ru/supervisor-agent-pattern.06275a41ae006ac8.webp" alt="Supervisor Agent Pattern" width="800"/>

*Супервизор использует свой LLM, чтобы решить, каких агентов вызвать и в каком порядке — жёстко прописанного маршрута нет.*

Вот как выглядит конкретный рабочий процесс для нашего конвейера файл → отчёт:

<img src="../../../translated_images/ru/file-report-workflow.649bb7a896800de9.webp" alt="File to Report Workflow" width="800"/>

*FileAgent читает файл через MCP инструменты, затем ReportAgent превращает сырой контент в структурированный отчёт.*

Ниже диаграмма последовательности, показывающая полный оркестр Супервизора — от запуска MCP сервера через автономный выбор агентов Супервизором, до вызовов инструментов по stdio и итогового отчёта:

<img src="../../../translated_images/ru/supervisor-agent-sequence.1aa389b3bef99956.webp" alt="Supervisor Agent Sequence Diagram" width="800"/>

*Супервизор автономно вызывает FileAgent (который использует MCP сервер через stdio для чтения файла), затем вызывает ReportAgent для генерации структурированного отчёта — каждый агент сохраняет свой вывод в общем Agentic Scope.*

Каждый агент сохраняет свои данные в **Agentic Scope** (общая память), что позволяет последующим агентам получать доступ к предыдущим результатам. Это демонстрирует, как MCP инструменты органично вписываются в агентские рабочие процессы — Супервизору не нужно знать *как* читаются файлы, ему достаточно знать, что `FileAgent` может это сделать.

#### Как работает Супервизор

Перед созданием агентов необходимо подключить MCP транспорт к клиенту и обернуть его как `ToolProvider`. Вот как инструменты MCP сервера становятся доступны вашим агентам:

```java
// Создайте MCP клиент из транспорта
McpClient mcpClient = new DefaultMcpClient.Builder()
        .transport(stdioTransport)
        .build();

// Оберните клиент как ToolProvider — это связывает инструменты MCP с LangChain4j
ToolProvider mcpToolProvider = McpToolProvider.builder()
        .mcpClients(List.of(mcpClient))
        .build();
```

Теперь вы можете внедрять `mcpToolProvider` в любой агент, которому нужны MCP инструменты:

```java
// Шаг 1: FileAgent считывает файлы с помощью инструментов MCP
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // Имеет инструменты MCP для операций с файлами
        .build();

// Шаг 2: ReportAgent генерирует структурированные отчеты
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// Supervisor управляет рабочим процессом файл → отчет
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // Возвращает итоговый отчет
        .build();

// Supervisor решает, каких агентов вызывать, исходя из запроса
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```

#### Как FileAgent обнаруживает MCP инструменты во время выполнения

Вы, возможно, задаётесь вопросом: **как `FileAgent` знает, как использовать npm инструменты файловой системы?** Ответ в том, что он этого не знает — **LLM** вычисляет это во время выполнения, используя схемы инструментов.
Интерфейс `FileAgent` — это просто **определение подсказки**. Он не содержит жестко закодированных знаний о `read_file`, `list_directory` или любых других инструментах MCP. Вот как это работает от начала до конца:

1. **Запуск сервера:** `StdioMcpTransport` запускает npm-пакет `@modelcontextprotocol/server-filesystem` в виде дочернего процесса
2. **Обнаружение инструментов:** `McpClient` отправляет JSON-RPC-запрос `tools/list` серверу, который отвечает именами инструментов, описаниями и схемами параметров (например, `read_file` — *«Прочитать полное содержимое файла»* — `{ path: string }`)
3. **Внедрение схемы:** `McpToolProvider` оборачивает обнаруженные схемы и делает их доступными для LangChain4j
4. **Решение LLM:** Когда вызывается `FileAgent.readFile(path)`, LangChain4j отправляет LLM системное сообщение, сообщение пользователя и **список схем инструментов**. LLM читает описания инструментов и генерирует вызов инструмента (например, `read_file(path="/some/file.txt")`)
5. **Выполнение:** LangChain4j перехватывает вызов инструмента, направляет его через MCP client обратно в дочерний процесс Node.js, получает результат и передает его обратно в LLM

Это тот же самый механизм [Обнаружения Инструментов](../../../05-mcp), описанный выше, но примененный конкретно к работе агента. Аннотации `@SystemMessage` и `@UserMessage` направляют поведение LLM, в то время как внедренный `ToolProvider` предоставляет **возможности** — LLM связывает их во время выполнения.

#### Стратегии ответа

При настройке `SupervisorAgent` вы указываете, как он должен формулировать окончательный ответ пользователю после того, как подагенты завершат задачи. Диаграмма ниже показывает три доступных стратегии — LAST возвращает вывод последнего агента напрямую, SUMMARY синтезирует все выводы через LLM, а SCORED выбирает лучший по оценке ответ относительно исходного запроса:

<img src="../../../translated_images/ru/response-strategies.3d0cea19d096bdf9.webp" alt="Response Strategies" width="800"/>

*Три стратегии формирования итогового ответа Supervisor — выберите, хотите ли вы вывод последнего агента, синтезированное резюме или лучший по оценке вариант.*

Доступные стратегии:

| Стратегия | Описание |
|----------|-------------|
| **LAST** | Супервайзер возвращает результат последнего вызванного подагента или инструмента. Полезно, когда последний агент в цепочке специально создан для выдачи полного окончательного ответа (например, «Агент резюме» в исследовательском пайплайне). |
| **SUMMARY** | Супервайзер использует собственную внутреннюю языковую модель (LLM) для синтеза резюме всего взаимодействия и всех выводов подагентов, затем возвращает это резюме как окончательный ответ. Это обеспечивает чистый и агрегированный ответ пользователю. |
| **SCORED** | Система использует внутренний LLM для оценки и ответа LAST, и резюме SUMMARY относительно исходного запроса пользователя, возвращая тот ответ, который получил более высокую оценку. |

#### Объяснение особенностей модуля Agentic

Пример демонстрирует несколько продвинутых возможностей модуля agentic. Рассмотрим подробнее Agentic Scope и Agent Listeners.

**Agentic Scope** показывает общую память, где агенты сохраняют результаты с помощью `@Agent(outputKey="...")`. Это позволяет:
- Позже агентам получать доступ к выводам предыдущих агентов
- Супервайзеру синтезировать итоговый ответ
- Вам исследовать, что выдал каждый агент

Диаграмма ниже показывает, как Agentic Scope работает как общая память в пайплайне «файл-отчет» — FileAgent записывает вывод под ключом `fileContent`, ReportAgent читает его и записывает свой вывод под `report`:

<img src="../../../translated_images/ru/agentic-scope.95ef488b6c1d02ef.webp" alt="Agentic Scope Shared Memory" width="800"/>

*Agentic Scope выступает общей памятью — FileAgent записывает `fileContent`, ReportAgent читает его и записывает `report`, а ваш код читает итоговый результат.*

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // Исходные данные файла от FileAgent
String report = scope.readState("report");            // Структурированный отчет от ReportAgent
```

**Agent Listeners** позволяют следить за выполнением агентов и отлаживать. Пошаговый вывод, который вы видите в демо, приходит от AgentListener, подключенного к каждому вызову агента:
- **beforeAgentInvocation** — вызывается, когда Супервайзер выбирает агента, показывая, какой агент выбран и почему
- **afterAgentInvocation** — вызывается после завершения агента, показывая результат
- **inheritedBySubagents** — при true слушатель мониторит всех агентов в иерархии

Следующая диаграмма показывает полный жизненный цикл Agent Listener, включая обработку ошибок `onError` при сбое агента:

<img src="../../../translated_images/ru/agent-listeners.784bfc403c80ea13.webp" alt="Agent Listeners Lifecycle" width="800"/>

*Agent Listeners подключаются к жизненному циклу выполнения — отслеживают запуск, завершение или ошибки агентов.*

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // Распространить на всех подагентов
    }
};
```

Помимо паттерна Supervisor, модуль `langchain4j-agentic` предоставляет несколько мощных паттернов рабочих процессов. Диаграмма ниже показывает все пять — от простых последовательных пайплайнов до workflows с участием человека:

<img src="../../../translated_images/ru/workflow-patterns.82b2cc5b0c5edb22.webp" alt="Agent Workflow Patterns" width="800"/>

*Пять паттернов рабочих процессов для оркестрации агентов — от простых последовательных пайплайнов до workflows с участием человека.*

| Паттерн | Описание | Сценарий использования |
|---------|-------------|----------|
| **Sequential** | Выполнять агентов последовательно, вывод идет следующему | Пайплайны: исследование → анализ → отчет |
| **Parallel** | Запуск агентов одновременно | Независимые задачи: погода + новости + акции |
| **Loop** | Итерация до выполнения условия | Оценка качества: уточнять пока оценка ≥ 0.8 |
| **Conditional** | Маршрутизация по условиям | Классификация → отправка к специалисту |
| **Human-in-the-Loop** | Добавление человеческих контрольных точек | Процессы утверждения, проверка контента |

## Ключевые концепции

Теперь, когда вы изучили MCP и модуль agentic на практике, подытожим, когда использовать каждый подход.

Одним из главных преимуществ MCP является его растущая экосистема. Диаграмма ниже показывает, как один универсальный протокол связывает ваше AI-приложение с множеством MCP-серверов — от доступа к файловой системе и базе данных до GitHub, электронной почты, веб-скрейпинга и другого:

<img src="../../../translated_images/ru/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="MCP Ecosystem" width="800"/>

*MCP создает экосистему универсальных протоколов — любой MCP-совместимый сервер работает с любым MCP-совместимым клиентом, обеспечивая обмен инструментами между приложениями.*

**MCP** идеально подходит, если вы хотите использовать существующую экосистему инструментов, создавать инструменты, которые могут использоваться разными приложениями, интегрировать сторонние сервисы через стандартные протоколы или менять реализации инструментов без изменения кода.

**Модуль Agentic** лучше, если вам нужны декларативные определения агентов с аннотациями `@Agent`, требуется оркестрация рабочих процессов (последовательные, циклы, параллели), предпочтительнее дизайн агентов на основе интерфейсов, а не императивного кода, или при объединении нескольких агентов с совместным использованием выводов через `outputKey`.

**Паттерн Supervisor Agent** выходит на первый план, когда рабочий процесс невозможно предсказать заранее, и вы хотите, чтобы LLM принимала решения, когда у вас несколько специализированных агентов, нуждающихся в динамической оркестрации, при создании разговорных систем с маршрутизацией между возможностями или когда требуется наиболее гибкое и адаптивное поведение агентов.

Чтобы помочь вам выбрать между пользовательскими методами `@Tool` из Модуля 04 и инструментами MCP из этого модуля, приведено сравнение ключевых компромиссов — пользовательские инструменты обеспечивают тесную связанность и полную типобезопасность для специфической логики приложений, а инструменты MCP — стандартизированные, многоразовые интеграции:

<img src="../../../translated_images/ru/custom-vs-mcp-tools.c4f9b6b1cb65d8a1.webp" alt="Custom Tools vs MCP Tools" width="800"/>

*Когда использовать пользовательские методы @Tool и когда MCP-инструменты — пользовательские для специфической бизнес-логики с полной типобезопасностью, MCP — для стандартизированных интеграций, работающих во всех приложениях.*
