## Чему вы научитесь

В предыдущем модуле вы узнали, как память позволяет реализовать разговорный ИИ, и использовали модели GitHub для базового взаимодействия. Теперь мы сосредоточимся на том, как задавать вопросы — сами запросы — используя GPT-5.2 от Azure OpenAI. То, как вы структурируете свои запросы, существенно влияет на качество получаемых ответов.

Мы используем GPT-5.2, потому что он вводит управление рассуждениями — вы можете указать модели, сколько размышлять перед ответом. Это делает разные стратегии составления запросов более заметными и помогает понять, когда применять каждый подход. Также мы воспользуемся меньшими ограничениями на частоту запросов для GPT-5.2 в Azure по сравнению с моделями GitHub.

## Понимание инженерии запросов

Инженерия запросов — это проектирование входного текста, который стабильно даёт нужные вам результаты. Это не просто задавание вопросов — это структура запросов так, чтобы модель точно понимала, что вы хотите и как это предоставить.

Подумайте об этом как об инструкции коллеге. «Исправь баг» — расплывчато. «Исправь исключение null pointer в UserService.java строка 45, добавив проверку на null» — конкретно. Языковые модели работают так же — важна конкретика и структура.

## Как это использует LangChain4j

Этот модуль демонстрирует продвинутые паттерны запроса, используя ту же основу LangChain4j, что и в предыдущих модулях, с акцентом на структуру запросов и управление рассуждениями.

<img src="../../../translated_images/ru/langchain4j-flow.48e534666213010b.webp" alt="LangChain4j Flow" width="800"/>

*Как LangChain4j связывает ваши запросы с Azure OpenAI GPT-5.2*

**Зависимости** — Модуль 02 использует следующие зависимости langchain4j, определённые в `pom.xml`:
```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j</artifactId> <!-- Inherited from BOM in root pom.xml -->
</dependency>
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-open-ai-official</artifactId> <!-- Inherited from BOM in root pom.xml -->
</dependency>
```

**Конфигурация OpenAiOfficialChatModel** — [LangChainConfig.java](../../../02-prompt-engineering/src/main/java/com/example/langchain4j/prompts/config/LangChainConfig.java)

Чат-модель вручную конфигурируется как Spring bean с использованием официального клиента OpenAI, который поддерживает конечные точки Azure OpenAI. Главное отличие от модуля 01 — это то, как мы структурируем запросы, отправляемые в `chatModel.chat()`, а не сама настройка модели.

**Системные и пользовательские сообщения** — [Gpt5PromptService.java](../../../02-prompt-engineering/src/main/java/com/example/langchain4j/prompts/service/Gpt5PromptService.java)

LangChain4j разделяет типы сообщений для ясности. `SystemMessage` устанавливает поведение и контекст ИИ (например, «Вы — код-ревьюер»), а `UserMessage` содержит фактический запрос. Это разделение позволяет поддерживать постоянное поведение ИИ при разных пользовательских вопросах.

```java
SystemMessage systemMsg = SystemMessage.from(
    "You are a helpful Java programming expert."
);

UserMessage userMsg = UserMessage.from(
    "Explain what a List is in Java"
);

String response = chatModel.chat(systemMsg, userMsg);
```

<img src="../../../translated_images/ru/message-types.93e0779798a17c9d.webp" alt="Message Types Architecture" width="800"/>

*SystemMessage обеспечивает постоянный контекст, в то время как UserMessages содержат отдельные запросы*

**MessageWindowChatMemory для многоходового диалога** — Для паттерна многоходового общения мы повторно используем `MessageWindowChatMemory` из модуля 01. Каждая сессия получает собственный экземпляр памяти, хранящийся в `Map<String, ChatMemory>`, что позволяет вести несколько параллельных разговоров без смешения контекстов.

**Шаблоны запросов** — Основное внимание здесь уделяется инженерии запросов, а не новым API LangChain4j. Каждый паттерн (низкая готовность, высокая готовность, выполнение задач и др.) использует один и тот же метод `chatModel.chat(prompt)`, но с тщательно структурированными строками запроса. XML-теги, инструкции и форматирование — всё это часть текста запроса, а не особенностей LangChain4j.

**Управление рассуждениями** — Усилия рассуждений GPT-5.2 контролируются через инструкции запроса, такие как «максимум 2 шага рассуждений» или «исследуй подробно». Это техники инженерии запросов, а не настройки LangChain4j. Библиотека просто передаёт ваши запросы модели.

Главный вывод: LangChain4j предоставляет инфраструктуру (связь с моделью через [LangChainConfig.java](../../../02-prompt-engineering/src/main/java/com/example/langchain4j/prompts/config/LangChainConfig.java), память, обработку сообщений через [Gpt5PromptService.java](../../../02-prompt-engineering/src/main/java/com/example/langchain4j/prompts/service/Gpt5PromptService.java)), а этот модуль обучает вас формировать эффективные запросы в рамках этой инфраструктуры.

## Основные паттерны

Не для всех задач подходит один и тот же подход. Некоторые вопросы требуют быстрых ответов, другие — глубокого анализа. Некоторые требуют видимых рассуждений, другие — просто результат. В этом модуле рассматриваются восемь паттернов запросов — каждый оптимизирован для разных сценариев. Вы попробуете их все, чтобы понять, когда какой подход работает лучше.

<img src="../../../translated_images/ru/eight-patterns.fa1ebfdf16f71e9a.webp" alt="Eight Prompting Patterns" width="800"/>

*Обзор восьми паттернов инженерии запросов и их случаев использования*

<img src="../../../translated_images/ru/reasoning-effort.db4a3ba5b8e392c1.webp" alt="Reasoning Effort Comparison" width="800"/>

*Низкая готовность (быстро, прямо) против высокой готовности (тщательно, исследовательно) при рассуждениях*

**Низкая готовность (быстро и сфокусировано)** — Для простых вопросов, где нужны быстрые и прямые ответы. Модель выполняет минимальные рассуждения — максимум 2 шага. Используйте это для вычислений, поисков или простых вопросов.

```java
String prompt = """
    <reasoning_effort>low</reasoning_effort>
    <instruction>maximum 2 reasoning steps</instruction>
    
    What is 15% of 200?
    """;

String response = chatModel.chat(prompt);
```

> 💡 **Эксперимент с GitHub Copilot:** Откройте [`Gpt5PromptService.java`](../../../02-prompt-engineering/src/main/java/com/example/langchain4j/prompts/service/Gpt5PromptService.java) и спросите:
> - «В чём разница между паттернами с низкой и высокой готовностью?»
> - «Как XML-теги в запросах помогают структурировать ответ ИИ?»
> - «Когда стоит использовать паттерны самоанализа, а когда инструкции напрямую?»

**Высокая готовность (глубоко и тщательно)** — Для сложных задач, требующих всестороннего анализа. Модель исследует подробно и показывает детальные рассуждения. Используйте для проектирования систем, архитектурных решений или сложных исследований.

```java
String prompt = """
    <reasoning_effort>high</reasoning_effort>
    <instruction>explore thoroughly, show detailed reasoning</instruction>
    
    Design a caching strategy for a high-traffic REST API.
    """;

String response = chatModel.chat(prompt);
```

**Выполнение задач (пошаговый прогресс)** — Для многошаговых рабочих процессов. Модель предоставляет план, рассказывает о каждом шаге по ходу работы, затем даёт итог. Используйте для миграций, реализации функций или любых процессов с несколькими шагами.

```java
String prompt = """
    <task>Create a REST endpoint for user registration</task>
    <preamble>Provide an upfront plan</preamble>
    <narration>Narrate each step as you work</narration>
    <summary>Summarize what was accomplished</summary>
    """;

String response = chatModel.chat(prompt);
```

Подход Chain-of-Thought явно просит модель показать процесс рассуждений, что повышает точность в сложных задачах. Пошаговое разбиение помогает людям и ИИ понять логику.

> **🤖 Попробуйте с [GitHub Copilot](https://github.com/features/copilot) Chat:** Спросите про этот паттерн:
> - «Как адаптировать паттерн выполнения задач для долгих операций?»
> - «Какие лучшие практики для структурирования предварительных фраз инструментов в продакшен-приложениях?»
> - «Как захватывать и отображать промежуточный прогресс в интерфейсе?»

<img src="../../../translated_images/ru/task-execution-pattern.9da3967750ab5c1e.webp" alt="Task Execution Pattern" width="800"/>

*План → Выполнение → Итоговый отчёт для многошаговых задач*

**Самоанализ кода** — Для генерации кода производственного качества. Модель генерирует код, проверяет его по критериям качества и итеративно улучшает. Используйте при создании новых функций или сервисов.

```java
String prompt = """
    <task>Create an email validation service</task>
    <quality_criteria>
    - Correct logic and error handling
    - Best practices (clean code, proper naming)
    - Performance optimization
    - Security considerations
    </quality_criteria>
    <instruction>Generate code, evaluate against criteria, improve iteratively</instruction>
    """;

String response = chatModel.chat(prompt);
```

<img src="../../../translated_images/ru/self-reflection-cycle.6f71101ca0bd28cc.webp" alt="Self-Reflection Cycle" width="800"/>

*Итерационный цикл улучшения — генерировать, оценивать, выявлять проблемы, улучшать, повторять*

**Структурированный анализ** — Для последовательной оценки. Модель проводит обзор кода по фиксированной схеме (корректность, практики, производительность, безопасность). Используйте для код-ревью или оценки качества.

```java
String prompt = """
    <code>
    public List getUsers() {
        return database.query("SELECT * FROM users");
    }
    </code>
    
    <framework>
    Evaluate using these categories:
    1. Correctness - Logic and functionality
    2. Best Practices - Code quality
    3. Performance - Efficiency concerns
    4. Security - Vulnerabilities
    </framework>
    """;

String response = chatModel.chat(prompt);
```

> **🤖 Попробуйте с [GitHub Copilot](https://github.com/features/copilot) Chat:** Спросите про структурированный анализ:
> - «Как настроить рамки анализа для разных типов код-ревью?»
> - «Как лучше парсить и программно обрабатывать структурированный вывод?»
> - «Как обеспечить единообразие уровней серьёзности в разных сессиях обзора?»

<img src="../../../translated_images/ru/structured-analysis-pattern.0af3b690b60cf2d6.webp" alt="Structured Analysis Pattern" width="800"/>

*Четыре категории для последовательных код-ревью с уровнями серьёзности*

**Многоходовой чат** — Для диалогов, требующих контекста. Модель запоминает предыдущие сообщения и строит на их основе. Используйте для интерактивной помощи или сложных вопросов и ответов.

```java
ChatMemory memory = MessageWindowChatMemory.withMaxMessages(10);

memory.add(UserMessage.from("What is Spring Boot?"));
AiMessage aiMessage1 = chatModel.chat(memory.messages()).aiMessage();
memory.add(aiMessage1);

memory.add(UserMessage.from("Show me an example"));
AiMessage aiMessage2 = chatModel.chat(memory.messages()).aiMessage();
memory.add(aiMessage2);
```

<img src="../../../translated_images/ru/context-memory.dff30ad9fa78832a.webp" alt="Context Memory" width="800"/>

*Как накопление контекста диалога происходит по ходам до достижения лимита токенов*

**Пошаговое рассуждение** — Для задач с видимой логикой. Модель показывает явные рассуждения на каждом шаге. Используйте для математических задач, головоломок или когда нужно понять ход мыслей.

```java
String prompt = """
    <instruction>Show your reasoning step-by-step</instruction>
    
    If a train travels 120 km in 2 hours, then stops for 30 minutes,
    then travels another 90 km in 1.5 hours, what is the average speed
    for the entire journey including the stop?
    """;

String response = chatModel.chat(prompt);
```

<img src="../../../translated_images/ru/step-by-step-pattern.a99ea4ca1c48578c.webp" alt="Step-by-Step Pattern" width="800"/>

*Разбиение задач на явные логические шаги*

**Ограниченный вывод** — Для ответов с конкретными требованиями к формату. Модель строго соблюдает формат и длину. Используйте для резюме или когда нужен точный формат вывода.

```java
String prompt = """
    <constraints>
    - Exactly 100 words
    - Bullet point format
    - Technical terms only
    </constraints>
    
    Summarize the key concepts of machine learning.
    """;

String response = chatModel.chat(prompt);
```

## Изучение паттернов

Веб-интерфейс позволяет экспериментировать с разными стратегиями запросов. Каждый паттерн решает разные задачи — попробуйте их, чтобы увидеть, когда какой подход лучше всего работает.

### Низкая против высокой готовности

Задайте простой вопрос, например «Что такое 15% от 200?» используя Низкую готовность. Вы получите мгновенный, прямой ответ. Теперь задайте что-то сложное, например «Разработай стратегию кэширования для API с высоким трафиком», используя Высокую готовность. Наблюдайте, как модель замедляется и подробно объясняет рассуждения. То же модель, та же структура вопроса — но запрос говорит ей, сколько мыслить.

### Структурированный анализ

Кодревью требует единых фреймворков оценки. Модель анализирует код по фиксированным категориям (корректность, практики, производительность, безопасность) с уровнями серьезности.

## Чему вы на самом деле учитесь

**Усилия в рассуждении меняют всё**

GPT-5.2 позволяет контролировать вычислительные усилия через ваши подсказки. Низкие усилия означают быстрые ответы с минимальным исследованием. Высокие — глубокое осмысление. Вы учитесь подбирать усилия под сложность задачи — не тратить время на простые вопросы, но и не торопить сложные решения.

**Структура управляет поведением**

Заметили XML-теги в подсказках? Это не украшение. Модели надежнее следуют структурированным инструкциям, чем свободному тексту. Если нужны многошаговые процессы или сложная логика — структура помогает модели отслеживать, где она и что дальше.

<img src="../../../translated_images/ru/prompt-structure.a77763d63f4e2f89.webp" alt="Структура подсказки" width="800"/>

*Анатомия хорошо структурированной подсказки с чёткими разделами и организацией в XML-стиле*

**Качество через самооценку**

Шаблоны саморефлексии работают, делая критерии качества явными. Вместо надежды, что модель «сделает правильно», вы говорите ей, что значит «правильно»: корректная логика, обработка ошибок, производительность, безопасность. Модель оценивает результат и улучшает его. Это превращает генерацию кода из лотереи в процесс.

**Контекст ограничен**

Многошаговые разговоры работают, включая историю сообщений в каждый запрос. Но есть предел — у каждой модели максимальное количество токенов. С ростом диалогов нужны стратегии, чтобы сохранять релевантный контекст и не достигать лимита. В этом модуле вы узнаете, как работает память; позже — когда резюмировать, когда забывать и когда доставать.
