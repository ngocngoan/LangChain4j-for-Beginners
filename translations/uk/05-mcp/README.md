## Чому ви навчитеся

Ви створили розмовний ШІ, опанували промпти, прив’язали відповіді до документів і створили агентів із інструментами. Але всі ці інструменти були спеціально розроблені для вашої конкретної програми. А що, якби ви могли надати вашому ШІ доступ до стандартизованої екосистеми інструментів, яку кожен може створювати та ділитися? У цьому модулі ви навчитеся робити саме це за допомогою Протоколу контексту моделі (MCP) та агентного модуля LangChain4j. Спочатку ми покажемо простий MCP рідер файлів, а потім продемонструємо, як він легко інтегрується у складні агентні робочі процеси за допомогою паттерна Супервайзер-агента.

## Що таке MCP?

Протокол контексту моделі (MCP) саме це і забезпечує — стандартний спосіб для ШІ-застосунків виявляти та використовувати зовнішні інструменти. Замість того, щоб писати індивідуальні інтеграції для кожного джерела даних чи сервісу, ви підключаєтесь до MCP серверів, які відкривають свої можливості у послідовному форматі. Ваш ШІ-агент тоді автоматично знайде і використовуватиме ці інструменти.

<img src="../../../translated_images/uk/mcp-comparison.9129a881ecf10ff5.webp" alt="MCP Comparison" width="800"/>

*До MCP: складні точкові інтеграції. Після MCP: один протокол, безмежні можливості.*

MCP вирішує фундаментальну проблему розробки ШІ: кожна інтеграція унікальна. Хочете отримати доступ до GitHub? Потрібен кастомний код. Хочете читати файли? Кастомний код. Хочете запитувати базу даних? Кастомний код. І жодна з цих інтеграцій не працює з іншими ШІ-застосунками.

MCP стандартизує це. MCP сервер відкриває інструменти з чіткими описами та схемами параметрів. Будь-який MCP клієнт може підключитися, дізнатися, які інструменти є, і скористатися ними. Побудуй один раз, використовуй скрізь.

<img src="../../../translated_images/uk/mcp-architecture.b3156d787a4ceac9.webp" alt="MCP Architecture" width="800"/>

*Архітектура Протоколу контексту моделі - стандартизоване виявлення та виконання інструментів*

## Як працює MCP

<img src="../../../translated_images/uk/mcp-protocol-detail.01204e056f45308b.webp" alt="MCP Protocol Detail" width="800"/>

*Як MCP працює під капотом — клієнти знаходять інструменти, обмінюються повідомленнями JSON-RPC і виконують операції через транспортний шар.*

**Архітектура сервер-клієнт**

MCP використовує модель клієнт-сервер. Сервери надають інструменти — читання файлів, запити до баз даних, виклики API. Клієнти (ваша ШІ-застосунок) підключаються до серверів і користуються їх інструментами.

Щоб використовувати MCP з LangChain4j, додайте цю залежність Maven:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```

**Виявлення інструментів**

Коли ваш клієнт підключається до MCP сервера, він запитує: «Які у вас є інструменти?» Сервер відповідає списком доступних інструментів із описами та схемами параметрів. Ваш ШІ-агент може вибрати, які інструменти використовувати залежно від запиту користувача.

<img src="../../../translated_images/uk/tool-discovery.07760a8a301a7832.webp" alt="MCP Tool Discovery" width="800"/>

*ШІ виявляє доступні інструменти при запуску — тепер знає, які можливості є, і може вирішувати, які з них використовувати.*

**Транспортні механізми**

MCP підтримує різні транспортні механізми. У цьому модулі демонструється транспорт Stdio для локальних процесів:

<img src="../../../translated_images/uk/transport-mechanisms.2791ba7ee93cf020.webp" alt="Transport Mechanisms" width="800"/>

*Транспортні механізми MCP: HTTP для віддалених серверів, Stdio для локальних процесів*

**Stdio** - [StdioTransportDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java)

Для локальних процесів. Ваш застосунок запускає сервер як дочірній процес і спілкується через стандартний ввід/вивід. Корисно для доступу до файлової системи або командних утиліт.

```java
McpTransport stdioTransport = new StdioMcpTransport.Builder()
    .command(List.of(
        npmCmd, "exec",
        "@modelcontextprotocol/server-filesystem@2025.12.18",
        resourcesDir
    ))
    .logEvents(false)
    .build();
```

<img src="../../../translated_images/uk/stdio-transport-flow.45eaff4af2d81db4.webp" alt="Stdio Transport Flow" width="800"/>

*Транспорт Stdio в дії — ваш застосунок запускає MCP сервер як дочірній процес і спілкується через конвеєри stdin/stdout.*

## Агентний модуль

У той час як MCP забезпечує стандартизовані інструменти, агентний модуль LangChain4j надає декларативний спосіб будувати агентів, які оркеструють ці інструменти. Анотація `@Agent` та `AgenticServices` дозволяють визначати поведінку агентів через інтерфейси, а не імперативний код.

У цьому модулі ви дослідите паттерн **Супервайзер-агент** — просунутий підхід агентного ШІ, де "супервайзер" динамічно вирішує, які підлеглі агенти викликати, на основі запитів користувача. Ми поєднаємо обидва концепти, надаючи одному з наших підлеглих агентів можливості доступу до файлів через MCP.

Щоб використовувати агентний модуль, додайте цю залежність Maven:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```

> **⚠️ Експериментально:** модуль `langchain4j-agentic` є **експериментальним** і може змінюватися. Стабільним способом створення AI помічників поки залишаються `langchain4j-core` з кастомними інструментами (Модуль 04).

### Супервайзер-агент

Паттерн **Супервайзер-агент** є **гнучкою** формою агентного ШІ. Супервайзер використовує LLM для автономного рішення, які агенти мають бути викликані на основі запиту користувача. У наступному прикладі ми поєднуємо доступ до файлів через MCP з LLM-агентом, щоб створити контрольований робочий процес «читання файлу → звіт».

У демо `FileAgent` читає файл з використанням MCP інструментів файлової системи, а `ReportAgent` генерує структурований звіт з виконавчим підсумком (1 речення), 3 ключовими пунктами та рекомендаціями. Супервайзер автоматично оркеструє цей процес:

<img src="../../../translated_images/uk/supervisor-agent-pattern.06275a41ae006ac8.webp" alt="Supervisor Agent Pattern" width="800"/>

*Супервайзер використовує свій LLM, щоб визначити, які агенти викликати і в якому порядку — жодних жорстких маршрутів.*

Ось як виглядає конкретний робочий процес для нашого pipeline від файлу до звіту:

<img src="../../../translated_images/uk/file-report-workflow.649bb7a896800de9.webp" alt="File to Report Workflow" width="800"/>

*FileAgent читає файл через MCP інструменти, потім ReportAgent перетворює сирий вміст у структурований звіт.*

Кожен агент зберігає свій результат у **Agentic Scope** (спільна пам’ять), що дозволяє наступним агентам отримувати доступ до попередніх результатів. Це демонструє, як MCP інструменти безшовно інтегруються в агентні робочі процеси — Супервайзеру не потрібно знати *як* читаються файли, лише що `FileAgent` вміє це робити.

#### Як працює Супервайзер

```java
// Крок 1: FileAgent читає файли за допомогою інструментів MCP
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // Має інструменти MCP для операцій з файлами
        .build();

// Крок 2: ReportAgent генерує структуровані звіти
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// Supervisor організовує робочий процес файл → звіт
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // Повернути кінцевий звіт
        .build();

// Supervisor вирішує, які агенти викликати на основі запиту
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```

#### Стратегії відповіді

Коли ви налаштовуєте `SupervisorAgent`, ви вказуєте, як він має сформулювати свою фінальну відповідь користувачу після завершення роботи підлеглих агентів.

<img src="../../../translated_images/uk/response-strategies.3d0cea19d096bdf9.webp" alt="Response Strategies" width="800"/>

*Три стратегії, як Супервайзер формує остаточну відповідь — обирайте залежно від того, чи хочете отримати вивід останнього агента, синтезований підсумок, або найкращий за оцінкою варіант.*

Доступні стратегії:

| Стратегія | Опис |
|----------|-------------|
| **LAST** | Супервайзер повертає вивід останнього викликаного підагента або інструменту. Це корисно, коли фінальний агент у робочому процесі спеціально призначений для формування повної, кінцевої відповіді (наприклад, «агент-підсумок» у науковому pipeline). |
| **SUMMARY** | Супервайзер використовує свій внутрішній мовний модель (LLM) для синтезу підсумку всієї взаємодії та виводів підагентів, а потім повертає цей підсумок як остаточну відповідь. Це забезпечує чисту, агреговану відповідь користувачу. |
| **SCORED** | Система використовує внутрішній LLM для оцінки як останньої відповіді (LAST), так і підсумку (SUMMARY) взаємодії за оригінальним запитом користувача, повертаючи той вивід, що отримав вищу оцінку. |

Дивіться [SupervisorAgentDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/SupervisorAgentDemo.java) для повної реалізації.

#### Розуміння виводу

Коли ви запускаєте демо, ви побачите структурований опис того, як Супервайзер керує кількома агентами. Ось що означає кожен розділ:

```
======================================================================
  FILE → REPORT WORKFLOW DEMO
======================================================================

This demo shows a clear 2-step workflow: read a file, then generate a report.
The Supervisor orchestrates the agents automatically based on the request.
```

**Заголовок** представляє концепцію робочого процесу: сфокусований pipeline від читання файлу до створення звіту.

```
--- WORKFLOW ---------------------------------------------------------
  ┌─────────────┐      ┌──────────────┐
  │  FileAgent  │ ───▶ │ ReportAgent  │
  │ (MCP tools) │      │  (pure LLM)  │
  └─────────────┘      └──────────────┘
   outputKey:           outputKey:
   'fileContent'        'report'

--- AVAILABLE AGENTS -------------------------------------------------
  [FILE]   FileAgent   - Reads files via MCP → stores in 'fileContent'
  [REPORT] ReportAgent - Generates structured report → stores in 'report'
```

**Діаграма робочого процесу** показує потік даних між агентами. Кожен агент має конкретну роль:
- **FileAgent** читає файли за допомогою MCP інструментів і зберігає сирий вміст у `fileContent`
- **ReportAgent** використовує цей вміст і створює структурований звіт у `report`

```
--- USER REQUEST -----------------------------------------------------
  "Read the file at .../file.txt and generate a report on its contents"
```

**Запит користувача** показує завдання. Супервайзер аналізує його і вирішує викликати FileAgent → ReportAgent.

```
--- SUPERVISOR ORCHESTRATION -----------------------------------------
  The Supervisor decides which agents to invoke and passes data between them...

  +-- STEP 1: Supervisor chose -> FileAgent (reading file via MCP)
  |
  |   Input: .../file.txt
  |
  |   Result: LangChain4j is an open-source, provider-agnostic Java framework for building LLM...
  +-- [OK] FileAgent (reading file via MCP) completed

  +-- STEP 2: Supervisor chose -> ReportAgent (generating structured report)
  |
  |   Input: LangChain4j is an open-source, provider-agnostic Java framew...
  |
  |   Result: Executive Summary...
  +-- [OK] ReportAgent (generating structured report) completed
```

**Оркестрація Супервайзера** демонструє 2-кроковий потік у дії:
1. **FileAgent** читає файл через MCP і зберігає вміст
2. **ReportAgent** отримує вміст і генерує структурований звіт

Супервайзер прийняв ці рішення **автономно** на основі запиту користувача.

```
--- FINAL RESPONSE ---------------------------------------------------
Executive Summary
...

Key Points
...

Recommendations
...

--- AGENTIC SCOPE (Data Flow) ----------------------------------------
  Each agent stores its output for downstream agents to consume:
  * fileContent: LangChain4j is an open-source, provider-agnostic Java framework...
  * report: Executive Summary...
```

#### Пояснення функцій агентного модуля

Приклад демонструє кілька розширених функцій агентного модуля. Розглянемо ближче Agentic Scope та Agent Listeners.

**Agentic Scope** показує спільну пам’ять, де агенти зберігають свої результати за допомогою `@Agent(outputKey="...")`. Це дозволяє:
- Наступним агентам отримувати доступ до виводів попередніх агентів
- Супервайзеру синтезувати фінальну відповідь
- Вам інспектувати, що створив кожен агент

<img src="../../../translated_images/uk/agentic-scope.95ef488b6c1d02ef.webp" alt="Agentic Scope Shared Memory" width="800"/>

*Agentic Scope діє як спільна пам’ять — FileAgent записує `fileContent`, ReportAgent читає його і записує `report`, а ваш код читає фінальний результат.*

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // Сирі дані файлу від FileAgent
String report = scope.readState("report");            // Структурований звіт від ReportAgent
```

**Agent Listeners** дозволяють моніторити та налагоджувати виконання агентів. Крок за кроком вивід, який ви бачите у демо, надходить від AgentListener, який підключається до кожного виклику агента:
- **beforeAgentInvocation** - Викликається, коли Supervisor обирає агента, дозволяючи побачити, який агент був вибраний і чому  
- **afterAgentInvocation** - Викликається, коли агент завершує роботу, показуючи його результат  
- **inheritedBySubagents** - Якщо true, слухач відстежує всіх агентів у ієрархії  

<img src="../../../translated_images/uk/agent-listeners.784bfc403c80ea13.webp" alt="Agent Listeners Lifecycle" width="800"/>

*Agent Listeners підключаються до життєвого циклу виконання — відстежують, коли агенти починають, завершують або стикаються з помилками.*  

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // Поширити на всі під-агенти
    }
};
```
  
Окрім патерна Supervisor, модуль `langchain4j-agentic` пропонує кілька потужних патернів робочих процесів та функцій:

<img src="../../../translated_images/uk/workflow-patterns.82b2cc5b0c5edb22.webp" alt="Agent Workflow Patterns" width="800"/>

*П’ять патернів робочих процесів для організації агентів — від простих послідовних конвеєрів до робочих процесів з участю людини.*

| Патерн | Опис | Сценарій використання |
|---------|-------------|----------|
| **Sequential** | Виконання агентів по черзі, вивід переходить до наступного | Конвеєри: дослідження → аналіз → звіт |
| **Parallel** | Запуск агентів одночасно | Незалежні завдання: погода + новини + акції |
| **Loop** | Ітерації до виконання умови | Оцінка якості: покращення, поки оцінка ≥ 0.8 |
| **Conditional** | Маршрутизація на основі умов | Класифікація → направлення до спеціаліста |
| **Human-in-the-Loop** | Додавання людських контрольних точок | Робочі процеси погодження, огляд контенту |

## Ключові поняття

Тепер, коли ви ознайомилися з MCP і модулем agentic в дії, підведемо підсумок, коли використовувати кожен підхід.

<img src="../../../translated_images/uk/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="MCP Ecosystem" width="800"/>

*MCP створює універсальну екосистему протоколів — будь-який сервер, сумісний з MCP, працює з будь-яким клієнтом MCP, що дозволяє спільно використовувати інструменти між застосунками.*

**MCP** ідеально підходить, коли потрібно використовувати існуючі екосистеми інструментів, створювати інструменти, якими можуть користуватись кілька застосунків, інтегрувати сторонні сервіси стандартними протоколами або замінювати реалізації інструментів без зміни коду.

**Модуль Agentic** найкраще підходить, коли потрібні декларативні визначення агентів з анотаціями `@Agent`, потрібна оркестрація робочих процесів (послідовна, циклічна, паралельна), віддаєте перевагу дизайну агентів на основі інтерфейсів замість імперативного коду або поєднуєте кілька агентів з спільними виводами через `outputKey`.

**Патерн Supervisor Agent** блищить, коли робочий процес не передбачуваний заздалегідь і ви хочете, щоб LLM приймав рішення, коли є кілька спеціалізованих агентів, які потребують динамічної оркестрації, при створенні розмовних систем, що маршрутизують до різних можливостей, або коли необхідна максимально гнучка, адаптивна поведінка агента.

<img src="../../../translated_images/uk/custom-vs-mcp-tools.c4f9b6b1cb65d8a1.webp" alt="Custom Tools vs MCP Tools" width="800"/>

*Коли використовувати кастомні методи @Tool проти інструментів MCP — кастомні інструменти для логіки застосунку з повною типобезпекою, інструменти MCP для стандартизованих інтеграцій, які працюють між застосунками.*
