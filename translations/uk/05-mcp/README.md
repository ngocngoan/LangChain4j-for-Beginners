## Чому ви навчитеся

Ви створили розмовний ШІ, освоїли підказки, закріпили відповіді в документах і створили агентів із інструментами. Але всі ці інструменти були спеціально розроблені для вашого конкретного застосунку. А якщо б ви могли надати вашому ШІ доступ до стандартизованої екосистеми інструментів, які будь-хто може створювати та поширювати? У цьому модулі ви дізнаєтесь, як саме це зробити за допомогою Протоколу Контексту Моделі (MCP) і агентського модуля LangChain4j. Спочатку ми демонструємо простий MCP читач файлів, а потім показуємо, як легко він інтегрується в розвинені агентські робочі процеси з використанням патерну агента-наглядача.

## Що таке MCP?

Протокол Контексту Моделі (MCP) саме це і забезпечує — стандартний спосіб для ШІ-застосунків виявляти та використовувати зовнішні інструменти. Замість того, щоб писати кастомні інтеграції для кожного джерела даних чи сервісу, ви підключаєтесь до MCP серверів, які відкривають свої можливості у послідовному форматі. Ваш агент ШІ може автоматично знаходити і використовувати ці інструменти.

Діаграма нижче показує різницю — без MCP кожна інтеграція вимагає індивідуального підключення «точка-точка»; з MCP одна система протоколів з’єднує ваш додаток з будь-яким інструментом:

<img src="../../../translated_images/uk/mcp-comparison.9129a881ecf10ff5.webp" alt="MCP Comparison" width="800"/>

*До MCP: Складні точкові інтеграції. Після MCP: Один протокол, нескінченні можливості.*

MCP вирішує фундаментальну проблему у розробці ШІ: кожна інтеграція — унікальна. Потрібен доступ до GitHub? Кастомний код. Потрібно читати файли? Кастомний код. Запит до бази даних? Кастомний код. І жодна з цих інтеграцій не працює з іншими ШІ-застосунками.

MCP стандартизує це. MCP сервер відкриває інструменти з чіткими описами і схемами. Будь-який клієнт MCP може підключитись, виявити наявні інструменти і використовувати їх. Побудував один раз — використовуй у всіх місцях.

Діаграма нижче показує цю архітектуру — один клієнт MCP (ваш ШІ-застосунок) підключається до кількох MCP серверів, кожен з яких відкриває свій набір інструментів через стандартний протокол:

<img src="../../../translated_images/uk/mcp-architecture.b3156d787a4ceac9.webp" alt="MCP Architecture" width="800"/>

*Архітектура Протоколу Контексту Моделі — стандартизоване виявлення та виконання інструментів*

## Як працює MCP

Під капотом MCP використовує багатошарову архітектуру. Ваш Java-застосунок (клієнт MCP) знаходить доступні інструменти, надсилає JSON-RPC запити через транспортний шар (Stdio або HTTP), а MCP сервер виконує операції і повертає результати. Наступна діаграма розбиває кожен шар цього протоколу:

<img src="../../../translated_images/uk/mcp-protocol-detail.01204e056f45308b.webp" alt="MCP Protocol Detail" width="800"/>

*Як MCP працює під капотом — клієнти виявляють інструменти, обмінюються повідомленнями JSON-RPC і виконують операції через транспортний шар.*

**Архітектура Сервер-Клієнт**

MCP використовує модель клієнт-сервер. Сервери надають інструменти — читання файлів, запити до баз, виклики API. Клієнти (ваш ШІ-застосунок) підключаються до серверів і використовують їхні інструменти.

Щоб використовувати MCP з LangChain4j, додайте цю залежність Maven:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```

**Виявлення інструментів**

Коли ваш клієнт підключається до MCP сервера, він питає: «Які у тебе є інструменти?» Сервер відповідає списком доступних інструментів з їхніми описами та схемами параметрів. Ваш агент ШІ тоді вирішує, які інструменти використовувати, залежно від запитів користувача. Діаграма нижче ілюструє цей обмін — клієнт надсилає запит `tools/list`, а сервер повертає перелік доступних інструментів з описами та схемами параметрів:

<img src="../../../translated_images/uk/tool-discovery.07760a8a301a7832.webp" alt="MCP Tool Discovery" width="800"/>

*ШІ відкриває доступні інструменти під час запуску — тепер він знає, які можливості доступні, і може вибрати потрібні.*

**Транспортні механізми**

MCP підтримує різні транспортні механізми. Два варіанти — Stdio (для локальної комунікації з підпроцесом) і Streamable HTTP (для віддалених серверів). Цей модуль демонструє транспорт Stdio:

<img src="../../../translated_images/uk/transport-mechanisms.2791ba7ee93cf020.webp" alt="Transport Mechanisms" width="800"/>

*Транспортні механізми MCP: HTTP для віддалених серверів, Stdio для локальних процесів*

**Stdio** - [StdioTransportDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java)

Для локальних процесів. Ваш застосунок запускає сервер як підпроцес і спілкується через стандартний ввід/вивід. Корисно для доступу до файлової системи або командних інструментів.

```java
McpTransport stdioTransport = new StdioMcpTransport.Builder()
    .command(List.of(
        npmCmd, "exec",
        "@modelcontextprotocol/server-filesystem@2025.12.18",
        resourcesDir
    ))
    .logEvents(false)
    .build();
```

Сервер `@modelcontextprotocol/server-filesystem` відкриває наступні інструменти, усі ізольовані в директоріях, які ви вкажете:

| Інструмент | Опис |
|------|-------------|
| `read_file` | Прочитати вміст одного файлу |
| `read_multiple_files` | Прочитати кілька файлів за одним викликом |
| `write_file` | Створити або перезаписати файл |
| `edit_file` | Виконати цілеспрямовані заміни тексту |
| `list_directory` | Перелічити файли і директорії за шляхом |
| `search_files` | Рекурсивно шукати файли, що відповідають паттерну |
| `get_file_info` | Отримати метадані файлу (розмір, часові позначки, дозволи) |
| `create_directory` | Створити директорію (включно з батьківськими) |
| `move_file` | Перемістити чи перейменувати файл або директорію |

Наступна діаграма показує, як працює транспорт Stdio під час виконання — ваш Java-застосунок запускає MCP сервер як дочірній процес, і вони спілкуються через канали stdin/stdout, без мережі чи HTTP:

<img src="../../../translated_images/uk/stdio-transport-flow.45eaff4af2d81db4.webp" alt="Stdio Transport Flow" width="800"/>

*Транспорт Stdio в дії — ваш застосунок запускає MCP сервер як дочірній процес і спілкується через канали stdin/stdout.*

## Агентський модуль

Хоч MCP і забезпечує стандартизовані інструменти, агентський модуль LangChain4j дає декларативний спосіб створювати агентів, які оркестрирують ці інструменти. Анотація `@Agent` та `AgenticServices` дозволяють визначати поведінку агента через інтерфейси, а не імперативний код.

У цьому модулі ви вивчите патерн **Агент-наглядач** — розвинений підхід агентського ШІ, де «наглядач» автоматично вирішує, яких під-агентів викликати залежно від запиту користувача. Ми поєднаємо обидві концепції, надавши одному з наших під-агентів можливості доступу до файлів через MCP.

Щоб використовувати агентський модуль, додайте цю залежність Maven:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```
> **Примітка:** Модуль `langchain4j-agentic` використовує окрему властивість версії (`langchain4j.mcp.version`), оскільки його реліз відбувається окремо від основних бібліотек LangChain4j.

> **⚠️ Експериментально:** Модуль `langchain4j-agentic` є **експериментальним** і може змінюватись. Стабільний спосіб створення AI-помічників залишається `langchain4j-core` з кастомними інструментами (Модуль 04).

### Агент-наглядач

Патерн **Агент-наглядач** — це **гнучка** форма агентського ШІ. Наглядач використовує LLM, щоб автономно вирішити, яких агентів викликати залежно від запиту користувача. У наступному прикладі ми поєднуємо доступ до файлів через MCP з LLM агентом, створюючи наглянутий робочий процес читання файлу → створення звіту.

У демонстрації `FileAgent` читає файл за допомогою MCP інструментів файлової системи, а `ReportAgent` створює структурований звіт з виконавчим резюме (1 речення), 3 ключовими пунктами та рекомендаціями. Наглядач автоматично керує цим процесом:

<img src="../../../translated_images/uk/supervisor-agent-pattern.06275a41ae006ac8.webp" alt="Supervisor Agent Pattern" width="800"/>

*Наглядач використовує свій LLM, щоб вирішити, яких агентів і в якій послідовності викликати — без жорстко закодованого маршрутизування.*

Ось як виглядає конкретний робочий процес для нашого конвеєра файл-звіт:

<img src="../../../translated_images/uk/file-report-workflow.649bb7a896800de9.webp" alt="File to Report Workflow" width="800"/>

*FileAgent читає файл через інструменти MCP, потім ReportAgent трансформує сирий контент у структурований звіт.*

Наступна послідовна діаграма відстежує повну оркестрацію Наглядача — від запуску MCP сервера, вибору агентів Наглядачем до викликів інструментів через stdio і фінального звіту:

<img src="../../../translated_images/uk/supervisor-agent-sequence.1aa389b3bef99956.webp" alt="Supervisor Agent Sequence Diagram" width="800"/>

*Наглядач автономно викликає FileAgent (який через stdio звертається до MCP сервера, щоб прочитати файл), потім викликає ReportAgent для створення структурованого звіту — кожен агент зберігає свій вивід у спільній Agentic Scope.*

Кожен агент зберігає свій вивід у **Agentic Scope** (спільна пам’ять), що дозволяє наступним агентам отримувати попередні результати. Це демонструє, як інструменти MCP безшовно інтегруються в агентські робочі процеси — Наглядачу не потрібно знати *як* читаються файли, лише те, що `FileAgent` вміє це робити.

#### Як працює агент-наглядач

Перш ніж створювати агентів, потрібно підключити MCP транспорт до клієнта і обгорнути його як `ToolProvider`. Ось як інструменти MCP сервера стають доступними вашим агентам:

```java
// Створити клієнта MCP з транспорту
McpClient mcpClient = new DefaultMcpClient.Builder()
        .transport(stdioTransport)
        .build();

// Обгорнути клієнта як ToolProvider — це з'єднує інструменти MCP із LangChain4j
ToolProvider mcpToolProvider = McpToolProvider.builder()
        .mcpClients(List.of(mcpClient))
        .build();
```

Тепер ви можете інжектувати `mcpToolProvider` у будь-який агент, який потребує MCP інструментів:

```java
// Крок 1: FileAgent читає файли за допомогою інструментів MCP
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // Має інструменти MCP для операцій з файлами
        .build();

// Крок 2: ReportAgent генерує структуровані звіти
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// Supervisor координує робочий процес файл → звіт
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // Повернути кінцевий звіт
        .build();

// Supervisor вирішує, яких агентів викликати на основі запиту
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```

#### Як FileAgent виявляє MCP інструменти під час виконання

Ви могли б запитати: **як `FileAgent` знає, як використовувати npm інструменти файлової системи?** Відповідь — не знає; **LLM** розбирається з цим під час виконання за схемами інструментів.

Інтерфейс `FileAgent` — це лише **визначення підказки**. Він не має жодної жорстко закодованої інформації про `read_file`, `list_directory` чи будь-який інший MCP інструмент. Ось що відбувається повністю:
1. **Запуск сервера:** `StdioMcpTransport` запускає npm-пакет `@modelcontextprotocol/server-filesystem` як дочірній процес  
2. **Виявлення інструментів:** `McpClient` надсилає JSON-RPC-запит `tools/list` серверу, який відповідає назвами інструментів, описами та схемами параметрів (наприклад, `read_file` — *"Прочитати повний вміст файлу"* — `{ path: string }`)  
3. **Ін’єкція схем:** `McpToolProvider` огортає ці знайдені схеми і робить їх доступними для LangChain4j  
4. **Вибір LLM:** Коли викликається `FileAgent.readFile(path)`, LangChain4j надсилає системне повідомлення, повідомлення користувача **і список схем інструментів** до LLM. LLM читає описи інструментів і генерує виклик інструменту (наприклад, `read_file(path="/some/file.txt")`)  
5. **Виконання:** LangChain4j перехоплює виклик інструменту, маршрутизуючи його через MCP-клієнт назад до дочірнього процесу Node.js, отримує результат і передає його назад LLM  

Це той самий механізм [Tool Discovery](../../../05-mcp), описаний вище, але застосований конкретно до робочого процесу агента. Анотації `@SystemMessage` та `@UserMessage` направляють поведінку LLM, тоді як ін’єктований `ToolProvider` дає йому **можливості** — LLM у час виконання поєднує обидва аспекти.

> - "Як цей агент знає, який MCP-інструмент викликати?"
> - "Що буде, якщо я видалю ToolProvider з builder-а агента?"

#### Стратегії відповіді

Коли ви налаштовуєте `SupervisorAgent`, ви визначаєте, як він має сформулювати свою остаточну відповідь користувачу після завершення завдань під агентів. Нижче наведена діаграма трьох доступних стратегій — LAST повертає прямо кінцевий вихід останнього агента, SUMMARY синтезує всі виходи через LLM, а SCORED обирає той, який набрав вищий бал стосовно початкового запиту:

<img src="../../../translated_images/uk/response-strategies.3d0cea19d096bdf9.webp" alt="Response Strategies" width="800"/>

*Три стратегії формування остаточної відповіді Supervisor — оберіть, чи хочете отримати вихід останнього агента, синтезоване резюме або найкращий варіант за оцінкою.*

Доступні стратегії:

| Стратегія | Опис |
|----------|-------------|
| **LAST** | Супервізор повертає результат останнього під-агента або інструменту, який було викликано. Це корисно, коли кінцевий агент у робочому процесі спеціально розроблений для створення повної остаточної відповіді (наприклад, «Summary Agent» у дослідницькому конвеєрі). |
| **SUMMARY** | Супервізор використовує власну внутрішню мовну модель (LLM) для синтезу зведення всієї взаємодії та результатів під-агентів, після чого повертає це резюме як остаточну відповідь. Це дає чисту, агреговану відповідь користувачеві. |
| **SCORED** | Система використовує внутрішню LLM для оцінки як відповіді LAST, так і SUMMARY за оригінальним користувацьким запитом, повертаючи той результат, який отримав вищий бал. |

Дивіться повну реалізацію у [SupervisorAgentDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/SupervisorAgentDemo.java).

#### Розуміння виходу

Коли ви запустите демо, побачите структурований покроковий огляд, як Supervisor організовує роботу кількох агентів. Ось що означає кожен розділ:

```
======================================================================
  FILE → REPORT WORKFLOW DEMO
======================================================================

This demo shows a clear 2-step workflow: read a file, then generate a report.
The Supervisor orchestrates the agents automatically based on the request.
```
  
**Заголовок** представляє концепцію робочого процесу: сфокусований конвеєр від читання файлу до створення звіту.

```
--- WORKFLOW ---------------------------------------------------------
  ┌─────────────┐      ┌──────────────┐
  │  FileAgent  │ ───▶ │ ReportAgent  │
  │ (MCP tools) │      │  (pure LLM)  │
  └─────────────┘      └──────────────┘
   outputKey:           outputKey:
   'fileContent'        'report'

--- AVAILABLE AGENTS -------------------------------------------------
  [FILE]   FileAgent   - Reads files via MCP → stores in 'fileContent'
  [REPORT] ReportAgent - Generates structured report → stores in 'report'
```
  
**Діаграма робочого процесу** показує потік даних між агентами. Кожен агент має конкретну роль:  
- **FileAgent** читає файли за допомогою MCP-інструментів і зберігає необроблений вміст у `fileContent`  
- **ReportAgent** споживає той вміст і генерує структурований звіт у `report`  

```
--- USER REQUEST -----------------------------------------------------
  "Read the file at .../file.txt and generate a report on its contents"
```
  
**Запит користувача** показує завдання. Supervisor парсить його і вирішує викликати FileAgent → ReportAgent.  

```
--- SUPERVISOR ORCHESTRATION -----------------------------------------
  The Supervisor decides which agents to invoke and passes data between them...

  +-- STEP 1: Supervisor chose -> FileAgent (reading file via MCP)
  |
  |   Input: .../file.txt
  |
  |   Result: LangChain4j is an open-source, provider-agnostic Java framework for building LLM...
  +-- [OK] FileAgent (reading file via MCP) completed

  +-- STEP 2: Supervisor chose -> ReportAgent (generating structured report)
  |
  |   Input: LangChain4j is an open-source, provider-agnostic Java framew...
  |
  |   Result: Executive Summary...
  +-- [OK] ReportAgent (generating structured report) completed
```
  
**Оркестрація Supervisor** показує 2-кроковий потік у дії:  
1. **FileAgent** читає файл через MCP і зберігає вміст  
2. **ReportAgent** отримує вміст і генерує структурований звіт  

Supervisor прийняв ці рішення **автономно**, виходячи з запиту користувача.

```
--- FINAL RESPONSE ---------------------------------------------------
Executive Summary
...

Key Points
...

Recommendations
...

--- AGENTIC SCOPE (Data Flow) ----------------------------------------
  Each agent stores its output for downstream agents to consume:
  * fileContent: LangChain4j is an open-source, provider-agnostic Java framework...
  * report: Executive Summary...
```
  
#### Пояснення функцій агентного модуля

Приклад демонструє кілька просунутих особливостей модулю agentic. Розглянемо детальніше Agentic Scope і Agent Listeners.

**Agentic Scope** показує спільну пам’ять, де агенти зберігали свої результати, використовуючи `@Agent(outputKey="...")`. Це дозволяє:  
- Наступним агентам отримувати виходи попередніх  
- Supervisor синтезувати фінальну відповідь  
- Вам інспектувати, що кожен агент виробив  

Нижче діаграма показує, як Agentic Scope працює як спільна пам’ять у робочому процесі від файлу до звіту — FileAgent записує результат під ключем `fileContent`, ReportAgent читає цей ключ і записує власний результат під `report`:

<img src="../../../translated_images/uk/agentic-scope.95ef488b6c1d02ef.webp" alt="Agentic Scope Shared Memory" width="800"/>

*Agentic Scope діє як спільна пам’ять — FileAgent записує `fileContent`, ReportAgent читає його і записує `report`, а ваш код читає фінальний результат.*

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // Сирі дані файлу від FileAgent
String report = scope.readState("report");            // Структурований звіт від ReportAgent
```
  
**Agent Listeners** дають змогу контролювати та налагоджувати виконання агентів. Покроковий вивід, який ви бачите у демо, виходить із AgentListener, що під’єднується до кожного виклику агента:  
- **beforeAgentInvocation** — викликається, коли Supervisor вибирає агента, даючи змогу побачити, який агент і чому був обраний  
- **afterAgentInvocation** — викликається після завершення агента, відображаючи його результат  
- **inheritedBySubagents** — якщо true, лістенер контролює всі агенти у ієрархії  

Наступна діаграма ілюструє весь життєвий цикл Agent Listener, включно з тим, як `onError` опрацьовує помилки під час виконання агентів:

<img src="../../../translated_images/uk/agent-listeners.784bfc403c80ea13.webp" alt="Agent Listeners Lifecycle" width="800"/>

*Agent Listeners підключаються до життєвого циклу виконання — контролюйте початок, завершення або помилки агентів.*

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // Поширити на всі підагенти
    }
};
```
  
Окрім патерна Supervisor, модуль `langchain4j-agentic` пропонує декілька потужних шаблонів робочих процесів. Нижче діаграма показує усі п’ять — від простих послідовних конвеєрів до процесів з участю людини:

<img src="../../../translated_images/uk/workflow-patterns.82b2cc5b0c5edb22.webp" alt="Agent Workflow Patterns" width="800"/>

*П’ять шаблонів робочих процесів для організації агентів — від простих послідовних конвеєрів до процесів затвердження з участю людини.*

| Патерн | Опис | Випадки використання |
|---------|-------------|----------|
| **Sequential** | Виконувати агентів послідовно, вихід передається наступному | Конвеєри: дослідження → аналіз → звіт |
| **Parallel** | Запускати агентів одночасно | Незалежні завдання: погода + новини + акції |
| **Loop** | Ітерувати, доки умова не буде виконана | Оцінювання якості: покращення до оцінки ≥ 0.8 |
| **Conditional** | Маршрутизація за умовами | Класифікувати → направити до спеціаліста |
| **Human-in-the-Loop** | Додати контрольні точки з людиною | Процеси затвердження, перегляд контенту |

## Ключові поняття

Тепер, коли ви ознайомились з MCP та модулем agentic на практиці, підсумуємо, коли застосовувати кожен підхід.

Однією з головних переваг MCP є його зростаюча екосистема. Нижче діаграма показує, як універсальний протокол пов’язує ваш AI-додаток з різноманітними MCP-серверами — від доступу до файлової системи і баз даних до GitHub, електронної пошти, веб-скрапінгу тощо:

<img src="../../../translated_images/uk/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="MCP Ecosystem" width="800"/>

*MCP створює екосистему універсального протоколу — будь-який MCP-сумісний сервер працює з будь-яким MCP-сумісним клієнтом, що дозволяє обмін інструментами між застосунками.*

**MCP** ідеальний, коли ви хочете використати існуючі екосистеми інструментів, створити інструменти, якими можуть користуватись кілька застосунків, інтегрувати сторонні сервіси з використанням стандартних протоколів або замінювати реалізації інструментів без зміни коду.

**Модуль Agentic** найкраще підходить, коли потрібні декларативні визначення агентів з анотаціями `@Agent`, потрібна оркестрація робочих процесів (послідовна, циклична, паралельна), віддаєте перевагу дизайну агентів через інтерфейси замість імперативного коду або поєднуєте кілька агентів, які діляться результатами через `outputKey`.

**Патерн Supervisor Agent** блискуче підходить, коли робочий процес несподіваний і ви хочете, щоб LLM вирішував, коли у вас є кілька спецалізованих агентів з динамічною оркестрацією, коли будуються конверсаційні системи з маршрутизацією до різних можливостей або коли потрібна найбільш гнучка, адаптивна поведінка агента.

Щоб допомогти обрати між власними методами `@Tool` з Модуля 04 і MCP-інструментами з цього модуля, наведено порівняння ключових компромісів — власні інструменти дають щільне зв’язування і повну типобезпечність для логіки конкретного застосунку, тоді як MCP-інструменти пропонують стандартизовану, багаторазову інтеграцію:

<img src="../../../translated_images/uk/custom-vs-mcp-tools.c4f9b6b1cb65d8a1.webp" alt="Custom Tools vs MCP Tools" width="800"/>

*Коли використовувати власні @Tool методи проти MCP-інструментів — власні інструменти для логіки застосунку з повною типобезпечністю, MCP-інструменти — для стандартизованих інтеграцій, що працюють у різних додатках.*
