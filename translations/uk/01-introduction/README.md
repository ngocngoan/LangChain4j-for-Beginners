## Чого ви навчитеся

Якщо ви пройшли швидкий старт, то вже бачили, як надсилати підказки та отримувати відповіді. Це основа, але для реальних застосунків потрібне більше. Цей модуль навчить вас створювати розмовний ШІ, який пам’ятає контекст і підтримує стан — різниця між одноразовим демо і готовим до продакшену додатком.

У цьому посібнику ми будемо користуватися GPT-5.2 від Azure OpenAI, бо його розвинені можливості логічного мислення роблять поведінку різних патернів більш помітною. Додавши пам’ять, ви чітко побачите різницю. Це полегшує розуміння, що кожен компонент дає вашому додатку.

Ви створите один додаток, який демонструє обидва патерни:

**Стейтлес чат** — кожен запит є незалежним. У моделі немає пам’яті про попередні повідомлення. Це той патерн, який ви використали в швидкому старті.

**Стейтфул розмова** — кожен запит включає історію розмови. Модель підтримує контекст між кількома ходами. Саме цього потребують продакшен-додатки.

## Вимоги до підготовки

- Підписка Azure з доступом до Azure OpenAI
- Java 21, Maven 3.9+
- Azure CLI (https://learn.microsoft.com/en-us/cli/azure/install-azure-cli)
- Azure Developer CLI (azd) (https://learn.microsoft.com/en-us/azure/developer/azure-developer-cli/install-azd)

> **Примітка:** Java, Maven, Azure CLI та Azure Developer CLI (azd) попередньо встановлені в наданому devcontainer.

> **Примітка:** Цей модуль використовує GPT-5.2 на Azure OpenAI. Розгортання налаштоване автоматично через `azd up` — не змінюйте ім’я моделі у коді.

## Розуміння основної проблеми

Мовні моделі є безстанними (stateless). Кожен виклик API є незалежним. Якщо ви надішлете "Мене звати Джон", а потім запитаєте "Як мене звати?", модель не знатиме, що ви щойно представилися. Вона розглядає кожен запит, ніби це ваша перша розмова.

Це підходить для простих питань і відповідей, але непридатне для реальних застосунків. Боти служби підтримки мають пам’ятати, що ви їм сказали. Персональні помічники потребують контексту. Будь-яка багатокрокова бесіда вимагає пам’яті.

<img src="../../../translated_images/uk/stateless-vs-stateful.cc4a4765e649c41a.webp" alt="Stateless vs Stateful Conversations" width="800"/>

*Різниця між безстанними (незалежними викликами) та станомими (контекстно-усвідомленими) розмовами*

## Розуміння токенів

Перш ніж заглиблюватися у розмови, важливо зрозуміти токени — базові одиниці тексту, які обробляють мовні моделі:

<img src="../../../translated_images/uk/token-explanation.c39760d8ec650181.webp" alt="Token Explanation" width="800"/>

*Приклад того, як текст розбивається на токени — "I love AI!" стає 4 окремими одиницями обробки*

Токени — це те, як ШІ-моделі вимірюють і обробляють текст. Слова, розділові знаки і навіть пробіли можуть бути токенами. У вашої моделі є обмеження на кількість токенів, які вона може обробити за раз (400,000 у GPT-5.2, із максимум 272,000 вхідних токенів і 128,000 вихідних). Розуміння токенів допомагає керувати довжиною розмови та вартістю.

## Як працює пам’ять

Пам’ять чату вирішує проблему безстанності, зберігаючи історію розмови. Перед відправкою запиту моделі фреймворк додає відповідні попередні повідомлення. Коли ви питаєте "Як мене звати?", система фактично надсилає повну історію розмови, даючи моделі змогу побачити, що ви раніше сказали "Мене звати Джон."

LangChain4j надає реалізації пам’яті, які роблять це автоматично. Ви обираєте, скільки повідомлень зберігати, а фреймворк керує вікном контексту.

<img src="../../../translated_images/uk/memory-window.bbe67f597eadabb3.webp" alt="Memory Window Concept" width="800"/>

*MessageWindowChatMemory підтримує ковзне вікно останніх повідомлень, автоматично видаляючи старі*

## Як це використовує LangChain4j

Цей модуль розширює швидкий старт, інтегруючи Spring Boot та додаючи пам’ять для розмов. Ось як частини працюють разом:

**Залежності** — додайте дві бібліотеки LangChain4j:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j</artifactId> <!-- Inherited from BOM in root pom.xml -->
</dependency>
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-open-ai-official</artifactId> <!-- Inherited from BOM in root pom.xml -->
</dependency>
```

**Модель чату** — налаштуйте Azure OpenAI як Spring bean ([LangChainConfig.java](../../../01-introduction/src/main/java/com/example/langchain4j/config/LangChainConfig.java)):

```java
@Bean
public OpenAiOfficialChatModel openAiOfficialChatModel() {
    return OpenAiOfficialChatModel.builder()
            .baseUrl(azureEndpoint)
            .apiKey(azureApiKey)
            .modelName(deploymentName)
            .timeout(Duration.ofMinutes(5))
            .maxRetries(3)
            .build();
}
```

Будівельник зчитує облікові дані з змінних середовища, які встановлює `azd up`. Вказання `baseUrl` на вашу кінцеву точку Azure дозволяє клієнту OpenAI працювати з Azure OpenAI.

**Пам’ять розмови** — відстежуйте історію чату з MessageWindowChatMemory ([ConversationService.java](../../../01-introduction/src/main/java/com/example/langchain4j/service/ConversationService.java)):

```java
ChatMemory memory = MessageWindowChatMemory.withMaxMessages(10);

memory.add(UserMessage.from("My name is John"));
memory.add(AiMessage.from("Nice to meet you, John!"));

memory.add(UserMessage.from("What's my name?"));
AiMessage aiMessage = chatModel.chat(memory.messages()).aiMessage();
memory.add(aiMessage);
```

Створіть пам’ять з `withMaxMessages(10)`, щоб тримати останні 10 повідомлень. Додавайте повідомлення користувача та ШІ з типізованими обгортками: `UserMessage.from(text)` і `AiMessage.from(text)`. Отримуйте історію через `memory.messages()` і надсилайте її моделі. Сервіс зберігає окремі об’єкти пам’яті для кожного ідентифікатора розмови, дозволяючи одночасно вести чат кільком користувачам.

> **🤖 Спробуйте з [GitHub Copilot](https://github.com/features/copilot) Chat:** Відкрийте [`ConversationService.java`](../../../01-introduction/src/main/java/com/example/langchain4j/service/ConversationService.java) і запитайте:
> - "Як MessageWindowChatMemory вирішує, які повідомлення відкидати, коли вікно повне?"
> - "Чи можна реалізувати власне збереження пам’яті з використанням бази даних замість пам’яті в оперативній пам’яті?"
> - "Як додати зведення тексту, щоб стиснути стару історію розмов?"

Точка доступу стейтлес чату пропускає пам’ять — просто `chatModel.chat(prompt)`, як у швидкому старті. Стейтфул точка додає повідомлення в пам’ять, отримує історію та включає цей контекст у кожен запит. Конфігурація моделі одна, патерни відрізняються.

## Розгортання інфраструктури Azure OpenAI

**Bash:**
```bash
cd 01-introduction
azd up  # Виберіть підписку та розташування (рекомендується eastus2)
```

**PowerShell:**
```powershell
cd 01-introduction
azd up  # Виберіть підписку та розташування (рекомендовано eastus2)
```

> **Примітка:** Якщо виникає помилка тайм-ауту (`RequestConflict: Cannot modify resource ... provisioning state is not terminal`), просто запустіть `azd up` знову. Ресурси Azure можуть все ще розгортатися у фоновому режимі, і повторна спроба дозволяє завершити розгортання, коли ресурси досягнуть кінцевого стану.

Це зробить:
1. Розгортання ресурсу Azure OpenAI з моделями GPT-5.2 і text-embedding-3-small
2. Автоматичне створення файлу `.env` у кореневій теці із обліковими даними
3. Налаштування всіх необхідних змінних середовища

**Проблеми з розгортанням?** Дивіться [Infrastructure README](infra/README.md) для детального усунення неполадок, включно з конфліктами піддоменів, кроками ручного розгортання в Azure Portal та порадами конфігурації моделей.

**Перевірте успішність розгортання:**

**Bash:**
```bash
cat ../.env  # Має показувати AZURE_OPENAI_ENDPOINT, API_KEY тощо.
```

**PowerShell:**
```powershell
Get-Content ..\.env  # Має показувати AZURE_OPENAI_ENDPOINT, API_KEY тощо.
```

> **Примітка:** Команда `azd up` автоматично створює файл `.env`. Якщо потрібно оновити його пізніше, ви можете або відредагувати `.env` вручну, або згенерувати знову, виконавши:
>
> **Bash:**
> ```bash
> cd ..
> bash .azd-env.sh
> ```
>
> **PowerShell:**
> ```powershell
> cd ..
> .\.azd-env.ps1
> ```

## Запуск додатку локально

**Перевірте розгортання:**

Переконайтесь, що файл `.env` існує в кореневій директорії і містить облікові дані Azure:

**Bash:**
```bash
cat ../.env  # Повинен показувати AZURE_OPENAI_ENDPOINT, API_KEY, DEPLOYMENT
```

**PowerShell:**
```powershell
Get-Content ..\.env  # Має показувати AZURE_OPENAI_ENDPOINT, API_KEY, DEPLOYMENT
```

**Запустіть додатки:**

**Опція 1: Використання Spring Boot Dashboard (Рекомендовано для користувачів VS Code)**

Dev контейнер включає розширення Spring Boot Dashboard, яке надає візуальний інтерфейс для керування всіма Spring Boot додатками. Ви знайдете його в Activity Bar зліва у VS Code (значок Spring Boot).

У Spring Boot Dashboard ви можете:
- Переглянути всі доступні Spring Boot додатки у робочій області
- Запускати/зупиняти додатки одним кліком
- Переглядати логи додатків у реальному часі
- Моніторити стан додатків

Просто натисніть кнопку запуску поруч із "introduction", щоб стартувати цей модуль, або запустіть одразу всі модулі.

<img src="../../../translated_images/uk/dashboard.69c7479aef09ff6b.webp" alt="Spring Boot Dashboard" width="400"/>

**Опція 2: Використання shell скриптів**

Запустіть всі веб-застосунки (модулі 01-04):

**Bash:**
```bash
cd ..  # З кореневого каталогу
./start-all.sh
```

**PowerShell:**
```powershell
cd ..  # З кореневого каталогу
.\start-all.ps1
```

Або запустіть тільки цей модуль:

**Bash:**
```bash
cd 01-introduction
./start.sh
```

**PowerShell:**
```powershell
cd 01-introduction
.\start.ps1
```

Обидва скрипти автоматично завантажують змінні середовища з кореневого файлу `.env` і збудують JAR-файли, якщо їх немає.

> **Примітка:** Якщо хочете збудувати всі модулі вручну перед запуском:
>
> **Bash:**
> ```bash
> cd ..  # Go to root directory
> mvn clean package -DskipTests
> ```
>
> **PowerShell:**
> ```powershell
> cd ..  # Go to root directory
> mvn clean package -DskipTests
> ```

Відкрийте http://localhost:8080 у вашому браузері.

**Щоб зупинити:**

**Bash:**
```bash
./stop.sh  # Тільки цей модуль
# Або
cd .. && ./stop-all.sh  # Всі модулі
```

**PowerShell:**
```powershell
.\stop.ps1  # Тільки цей модуль
# Або
cd ..; .\stop-all.ps1  # Всі модулі
```

## Використання додатку

Додаток надає веб-інтерфейс із двома реалізаціями чату поруч.

<img src="../../../translated_images/uk/home-screen.121a03206ab910c0.webp" alt="Application Home Screen" width="800"/>

*Панель керування, що показує опції Простого чату (без пам'яті) та Розмовного чату (з пам'яттю)*

### Стейтлес чат (ліва панель)

Спробуйте спочатку цей. Скажіть "Мене звати Джон", а потім одразу "Як мене звати?" Модель не запам’ятає, бо кожне повідомлення незалежне. Це демонструє основну проблему базової інтеграції мовної моделі — відсутність контексту розмови.

<img src="../../../translated_images/uk/simple-chat-stateless-demo.13aeb3978eab3234.webp" alt="Stateless Chat Demo" width="800"/>

*ШІ не пам’ятає ваше ім’я з попереднього повідомлення*

### Стейтфул чат (права панель)

Тепер спробуйте ту ж послідовність тут. Скажіть "Мене звати Джон", а потім "Як мене звати?" Цього разу модель пам’ятає. Різниця у MessageWindowChatMemory — він зберігає історію розмови і додає її до кожного запиту. Так працює розмовний ШІ у продакшен.

<img src="../../../translated_images/uk/conversational-chat-stateful-demo.e5be9822eb23ff59.webp" alt="Stateful Chat Demo" width="800"/>

*ШІ пам’ятає ваше ім’я з попередньої частини розмови*

Обидві панелі використовують ту саму модель GPT-5.2. Єдина різниця — у пам’яті. Це дозволяє зрозуміти, яке значення має пам’ять для вашого додатку та чому вона потрібна для реальних сценаріїв.
