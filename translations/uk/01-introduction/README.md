## Що ви дізнаєтесь

Якщо ви пройшли швидкий старт, ви бачили, як надсилати підказки і отримувати відповіді. Це основа, але для реальних застосунків потрібно більше. Цей модуль навчить вас створювати розмовний ШІ, який запам’ятовує контекст і підтримує стан — різниця між одноразовим демо та готовим до виробництва застосунком.

Ми будемо використовувати Azure OpenAI GPT-5.2 протягом цього посібника, бо його розвинені можливості логічного мислення роблять поведінку різних патернів більш явною. Додаючи пам’ять, ви чітко побачите різницю. Це полегшує розуміння ролі кожного компонента у вашому застосунку.

Ви створите один застосунок, який демонструватиме обидва патерни:

**Безстанковий чат** — кожен запит незалежний. Модель не пам’ятає попередні повідомлення. Це патерн, який ви використали у швидкому старті.

**Станковий діалог** — кожен запит містить історію розмови. Модель підтримує контекст через кілька кроків. Саме цього потребують виробничі застосунки.

## Вимоги

- Підписка Azure з доступом до Azure OpenAI
- Java 21, Maven 3.9+
- Azure CLI (https://learn.microsoft.com/en-us/cli/azure/install-azure-cli)
- Azure Developer CLI (azd) (https://learn.microsoft.com/en-us/azure/developer/azure-developer-cli/install-azd)

> **Примітка:** Java, Maven, Azure CLI та Azure Developer CLI (azd) встановлені у наданому devcontainer.

> **Примітка:** Цей модуль використовує GPT-5.2 на Azure OpenAI. Розгортання налаштоване автоматично через `azd up` — не змінюйте ім’я моделі у коді.

## Розуміння основної проблеми

Мовні моделі безстанкові. Кожен виклик API незалежний. Якщо ви надішлете «Мене звати Джон», а потім запитаєте «Як мене звати?», модель не знатиме, що ви щойно представилися. Вона сприймає кожен запит так, ніби це ваша перша розмова.

Це підходить для простих питань і відповідей, але непридатне для реальних застосунків. Боти обслуговування клієнтів повинні пам’ятати сказане вами. Особисті асистенти потребують контексту. Будь-який багатокроковий діалог потребує пам’яті.

<img src="../../../translated_images/uk/stateless-vs-stateful.cc4a4765e649c41a.webp" alt="Stateless vs Stateful Conversations" width="800"/>

*Різниця між безстанковими (незалежними) та станковими (що враховують контекст) діалогами*

## Розуміння токенів

Перед тим як поринути у розмови, важливо зрозуміти токени — базові одиниці тексту, які обробляють мовні моделі:

<img src="../../../translated_images/uk/token-explanation.c39760d8ec650181.webp" alt="Token Explanation" width="800"/>

*Приклад, як текст розбивається на токени — "I love AI!" стає 4 окремими одиницями обробки*

Токени — це те, як моделі ШІ вимірюють і обробляють текст. Слова, пунктуація і навіть пробіли можуть бути токенами. Модель має ліміт, скільки токенів може обробити одночасно (400 000 для GPT-5.2, до 272 000 вхідних і 128 000 вихідних токенів). Розуміння токенів допоможе контролювати довжину діалогу і вартість.

## Як працює пам’ять

Пам’ять чату розв’язує проблему безстанковості, підтримуючи історію діалогу. Перед відправленням запиту моделі фреймворк додає релевантні попередні повідомлення. Коли ви питаєте «Як мене звати?», система фактично надсилає всю історію діалогу, дозволяючи моделі бачити, що ви раніше сказали «Мене звати Джон».

LangChain4j надає реалізації пам’яті, які керують цим автоматично. Ви визначаєте, скільки повідомлень зберігати, а фреймворк управляє контекстним вікном.

<img src="../../../translated_images/uk/memory-window.bbe67f597eadabb3.webp" alt="Memory Window Concept" width="800"/>

*MessageWindowChatMemory підтримує ковзне вікно останніх повідомлень, автоматично видаляючи старі*

## Як це використовує LangChain4j

Цей модуль розширює швидкий старт, інтегруючи Spring Boot і додаючи пам’ять розмови. Ось як все працює разом:

**Залежності** — додайте дві бібліотеки LangChain4j:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j</artifactId> <!-- Inherited from BOM in root pom.xml -->
</dependency>
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-open-ai-official</artifactId> <!-- Inherited from BOM in root pom.xml -->
</dependency>
```

**Модель чату** — налаштуйте Azure OpenAI як Spring bean ([LangChainConfig.java](../../../01-introduction/src/main/java/com/example/langchain4j/config/LangChainConfig.java)):

```java
@Bean
public OpenAiOfficialChatModel openAiOfficialChatModel() {
    return OpenAiOfficialChatModel.builder()
            .baseUrl(azureEndpoint)
            .apiKey(azureApiKey)
            .modelName(deploymentName)
            .timeout(Duration.ofMinutes(5))
            .maxRetries(3)
            .build();
}
```

Білдер читає облікові дані з змінних середовища, налаштованих `azd up`. Встановлення `baseUrl` на ваш Azure endpoint дає можливість клієнту OpenAI працювати з Azure OpenAI.

**Пам’ять розмови** — відслідковуйте історію чату через MessageWindowChatMemory ([ConversationService.java](../../../01-introduction/src/main/java/com/example/langchain4j/service/ConversationService.java)):

```java
ChatMemory memory = MessageWindowChatMemory.withMaxMessages(10);

memory.add(UserMessage.from("My name is John"));
memory.add(AiMessage.from("Nice to meet you, John!"));

memory.add(UserMessage.from("What's my name?"));
AiMessage aiMessage = chatModel.chat(memory.messages()).aiMessage();
memory.add(aiMessage);
```

Створіть пам’ять з `withMaxMessages(10)`, щоб зберігати останні 10 повідомлень. Додавайте повідомлення користувача та ШІ за допомогою типізованих обгорток: `UserMessage.from(text)` і `AiMessage.from(text)`. Отримуйте історію через `memory.messages()` і надсилайте її моделі. Сервіс зберігає окремі екземпляри пам’яті для кожного ID розмови, дозволяючи одночасно спілкуватися багатьом користувачам.

> **🤖 Спробуйте з [GitHub Copilot](https://github.com/features/copilot) Chat:** Відкрийте [`ConversationService.java`](../../../01-introduction/src/main/java/com/example/langchain4j/service/ConversationService.java) і запитайте:
> - "Як MessageWindowChatMemory вирішує, які повідомлення видалити, коли вікно заповнене?"
> - "Чи можу я реалізувати власне зберігання пам’яті за допомогою бази даних замість пам’яті в оперативній пам’яті?"
> - "Як додати узагальнення, щоб стискати стару історію розмови?"

Безстанковий чат пропускає пам’ять — просто `chatModel.chat(prompt)`, як у швидкому старті. Станковий чат додає повідомлення до пам’яті, отримує історію і включає цей контекст у кожен запит. Та сама модель, різні патерни.

## Розгортання інфраструктури Azure OpenAI

**Bash:**
```bash
cd 01-introduction
azd up  # Виберіть підписку та розташування (рекомендується eastus2)
```

**PowerShell:**
```powershell
cd 01-introduction
azd up  # Виберіть підписку та розташування (рекомендовано eastus2)
```

> **Примітка:** Якщо ви отримаєте помилку тайм-ауту (`RequestConflict: Cannot modify resource ... provisioning state is not terminal`), просто запустіть `azd up` знову. Ресурси Azure можуть ще продовжувати розгортатись у фоновому режимі, повторна спроба дозволяє завершити розгортання, коли ресурси досягнуть фінального стану.

Це:
1. Розгорне ресурс Azure OpenAI з моделями GPT-5.2 і text-embedding-3-small
2. Автоматично згенерує файл `.env` у корені проекту з обліковими даними
3. Налаштує всі необхідні змінні середовища

**Проблеми розгортання?** Дивіться [Infrastructure README](infra/README.md) для детального усунення проблем, включно з конфліктами імен піддоменів, кроками ручного розгортання через Azure Portal та рекомендаціями щодо налаштування моделей.

**Перевірте, чи розгортання успішне:**

**Bash:**
```bash
cat ../.env  # Має відображати AZURE_OPENAI_ENDPOINT, API_KEY тощо.
```

**PowerShell:**
```powershell
Get-Content ..\.env  # Має показувати AZURE_OPENAI_ENDPOINT, API_KEY тощо.
```

> **Примітка:** Команда `azd up` автоматично створює файл `.env`. Якщо пізніше потрібно оновити цей файл, ви можете або відредагувати `.env` вручну, або згенерувати його заново, виконавши:
>
> **Bash:**
> ```bash
> cd ..
> bash .azd-env.sh
> ```
>
> **PowerShell:**
> ```powershell
> cd ..
> .\.azd-env.ps1
> ```

## Запуск програми локально

**Перевірте розгортання:**

Переконайтеся, що файл `.env` існує у кореневій папці і містить облікові дані Azure:

**Bash:**
```bash
cat ../.env  # Повинен показувати AZURE_OPENAI_ENDPOINT, API_KEY, DEPLOYMENT
```

**PowerShell:**
```powershell
Get-Content ..\.env  # Повинно показувати AZURE_OPENAI_ENDPOINT, API_KEY, DEPLOYMENT
```

**Запустіть застосунки:**

**Опція 1: Використання Spring Boot Dashboard (Рекомендовано для користувачів VS Code)**

Dev container включає розширення Spring Boot Dashboard, яке надає візуальний інтерфейс для керування усіма застосунками Spring Boot. Ви знайдете його на панелі дій ліворуч у VS Code (іконка Spring Boot).

Через Spring Boot Dashboard можна:
- Переглядати всі доступні Spring Boot застосунки у робочій області
- Запускати/зупиняти програми одним кліком
- Переглядати логи застосунку в режимі реального часу
- Моніторити статус застосунку

Просто натисніть кнопку play поруч із "introduction", щоб запустити цей модуль, або запустіть відразу всі модулі.

<img src="../../../translated_images/uk/dashboard.69c7479aef09ff6b.webp" alt="Spring Boot Dashboard" width="400"/>

**Опція 2: Використання shell-скриптів**

Запустіть всі веб-застосунки (модулі 01-04):

**Bash:**
```bash
cd ..  # З кореневого каталогу
./start-all.sh
```

**PowerShell:**
```powershell
cd ..  # З кореневого каталогу
.\start-all.ps1
```

Або запустіть лише цей модуль:

**Bash:**
```bash
cd 01-introduction
./start.sh
```

**PowerShell:**
```powershell
cd 01-introduction
.\start.ps1
```

Обидва скрипти автоматично завантажують змінні середовища з кореневого файлу `.env` та збирають JAR-файли, якщо їх немає.

> **Примітка:** Якщо ви хочете зібрати всі модулі вручну перед запуском:
>
> **Bash:**
> ```bash
> cd ..  # Go to root directory
> mvn clean package -DskipTests
> ```
>
> **PowerShell:**
> ```powershell
> cd ..  # Go to root directory
> mvn clean package -DskipTests
> ```

Відкрийте http://localhost:8080 у браузері.

**Щоб зупинити:**

**Bash:**
```bash
./stop.sh  # Тільки цей модуль
# Або
cd .. && ./stop-all.sh  # Всі модулі
```

**PowerShell:**
```powershell
.\stop.ps1  # Тільки цей модуль
# Або
cd ..; .\stop-all.ps1  # Всі модулі
```

## Використання програми

Програма надає веб-інтерфейс з двома реалізаціями чату поруч.

<img src="../../../translated_images/uk/home-screen.121a03206ab910c0.webp" alt="Application Home Screen" width="800"/>

*Панель з обома варіантами: Простий чат (безстанковий) і Розмовний чат (станковий)*

### Безстанковий чат (ліва панель)

Спробуйте цей спочатку. Скажіть «Мене звати Джон», а потім одразу «Як мене звати?» Модель не запам’ятає, бо кожне повідомлення незалежне. Це демонструє основну проблему при інтеграції базової мовної моделі — відсутність контексту діалогу.

<img src="../../../translated_images/uk/simple-chat-stateless-demo.13aeb3978eab3234.webp" alt="Stateless Chat Demo" width="800"/>

*ШІ не пам’ятає ваше ім’я з попереднього повідомлення*

### Станковий чат (права панель)

Тепер спробуйте ту ж послідовність тут. Скажіть «Мене звати Джон», а потім «Як мене звати?» Цього разу модель пам’ятає. Різниця у MessageWindowChatMemory — вона підтримує історію розмови і додає її до кожного запиту. Так працюють виробничі розмовні ШІ.

<img src="../../../translated_images/uk/conversational-chat-stateful-demo.e5be9822eb23ff59.webp" alt="Stateful Chat Demo" width="800"/>

*ШІ пам’ятає ваше ім’я з попередньої частини розмови*

Обидві панелі використовують ту ж модель GPT-5.2. Єдина різниця — пам’ять. Це чітко показує, що пам’ять дає вашому застосунку і чому вона необхідна для реальних сценаріїв.
