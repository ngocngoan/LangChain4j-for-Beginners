## Чому Ви Навчитеся

Якщо Ви пройшли швидкий старт, Ви бачили, як надсилати підказки і отримувати відповіді. Це основа, але реальні застосунки потребують більше. Цей модуль навчить вас створювати розмовний ШІ, який пам’ятає контекст і підтримує стан — різниця між одноразовою демонстрацією та готовим до виробництва застосунком.

Ми будемо використовувати GPT-5.2 від Azure OpenAI у цьому посібнику, тому що його розвинені здібності до логічного мислення роблять поведінку різних патернів більш очевидною. Коли додається пам’ять, Ви чітко побачите різницю. Це допомагає легше зрозуміти, що кожен компонент приносить у Ваш застосунок.

Ви створите один застосунок із демонстрацією обох патернів:

**Статичний Чат** — Кожен запит незалежний. Модель не пам’ятає попередніх повідомлень. Це патерн, який ви використовували в швидкому старті.

**Динамічна Розмова** — Кожен запит містить історію розмови. Модель підтримує контекст протягом кількох ходів. Саме це потрібно для виробничих застосунків.

## Вимоги

- Підписка Azure з доступом до Azure OpenAI
- Java 21, Maven 3.9+
- Azure CLI (https://learn.microsoft.com/en-us/cli/azure/install-azure-cli)
- Azure Developer CLI (azd) (https://learn.microsoft.com/en-us/azure/developer/azure-developer-cli/install-azd)

> **Примітка:** Java, Maven, Azure CLI і Azure Developer CLI (azd) попередньо встановлені у наданому devcontainer.

> **Примітка:** Цей модуль використовує GPT-5.2 на Azure OpenAI. Розгортання налаштовується автоматично через `azd up` — не змінюйте ім’я моделі в коді.

## Розуміння Основної Проблеми

Мовні моделі безстанкові. Кожен виклик API є незалежним. Якщо ви відправите "Мене звати Джон", а потім запитаєте "Як мене звати?", модель не знає, що ви щойно представились. Вона ставиться до кожного запиту, ніби це перша розмова у вашому житті.

Це підходить для простих питань і відповідей, але марно для реальних застосунків. Боти служби підтримки мають пам’ятати, що ви їм сказали. Особисті помічники потребують контексту. Будь-яка розмова з кількома ходами потребує пам’яті.

<img src="../../../translated_images/uk/stateless-vs-stateful.cc4a4765e649c41a.webp" alt="Статичні та динамічні розмови" width="800"/>

*Різниця між статичними (незалежними викликами) і динамічними (такими, що орієнтуються на контекст) розмовами*

## Розуміння Токенів

Перед тим, як занурюватися в розмови, важливо зрозуміти токени — базові одиниці тексту, які обробляють мовні моделі:

<img src="../../../translated_images/uk/token-explanation.c39760d8ec650181.webp" alt="Пояснення токенів" width="800"/>

*Приклад того, як текст розбивається на токени — "I love AI!" стає 4 окремими одиницями обробки*

Токени — це те, як ШІ-моделі вимірюють і обробляють текст. Слова, розділові знаки й навіть пробіли можуть бути токенами. У вашої моделі є ліміт того, скільки токенів вона може обробити за один раз (400 000 для GPT-5.2, з максимально 272 000 вхідними токенами та 128 000 вихідними). Розуміння токенів допомагає керувати довжиною розмови та витратами.

## Як Працює Пам’ять

Пам’ять чату вирішує проблему безстанковості, підтримуючи історію розмови. Перед відправкою вашого запиту моделі, фреймворк додає відповідні попередні повідомлення. Коли ви запитуєте "Як мене звати?", система фактично надсилає всю історію розмови, дозволяючи моделі побачити, що ви раніше сказали "Мене звати Джон".

LangChain4j надає реалізації пам’яті, які роблять це автоматично. Ви обираєте, скільки повідомлень зберігати, а фреймворк керує вікном контексту.

<img src="../../../translated_images/uk/memory-window.bbe67f597eadabb3.webp" alt="Концепція вікна пам’яті" width="800"/>

*MessageWindowChatMemory підтримує ковзне вікно останніх повідомлень, автоматично видаляючи старі*

## Як Це Використовує LangChain4j

Цей модуль розширює швидкий старт, інтегруючи Spring Boot і додаючи пам’ять розмови. Ось як компоненти з’єднуються:

**Залежності** — Додайте дві бібліотеки LangChain4j:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j</artifactId> <!-- Inherited from BOM in root pom.xml -->
</dependency>
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-open-ai-official</artifactId> <!-- Inherited from BOM in root pom.xml -->
</dependency>
```

**Модель Чату** — Налаштуйте Azure OpenAI як Spring bean ([LangChainConfig.java](../../../01-introduction/src/main/java/com/example/langchain4j/config/LangChainConfig.java)):

```java
@Bean
public OpenAiOfficialChatModel openAiOfficialChatModel() {
    return OpenAiOfficialChatModel.builder()
            .baseUrl(azureEndpoint)
            .apiKey(azureApiKey)
            .modelName(deploymentName)
            .timeout(Duration.ofMinutes(5))
            .maxRetries(3)
            .build();
}
```

Білдер читає облікові дані з змінних середовища, заданих `azd up`. Налаштування `baseUrl` на вашу точку доступу Azure робить клієнт OpenAI сумісним з Azure OpenAI.

**Пам’ять Розмови** — Відслідковуйте історію чату з MessageWindowChatMemory ([ConversationService.java](../../../01-introduction/src/main/java/com/example/langchain4j/service/ConversationService.java)):

```java
ChatMemory memory = MessageWindowChatMemory.withMaxMessages(10);

memory.add(UserMessage.from("My name is John"));
memory.add(AiMessage.from("Nice to meet you, John!"));

memory.add(UserMessage.from("What's my name?"));
AiMessage aiMessage = chatModel.chat(memory.messages()).aiMessage();
memory.add(aiMessage);
```

Створіть пам’ять за допомогою `withMaxMessages(10)`, щоб зберігати останні 10 повідомлень. Додавайте повідомлення користувача й ШІ з типізованими обгортками: `UserMessage.from(text)` та `AiMessage.from(text)`. Отримуйте історію за допомогою `memory.messages()` і надсилайте її моделі. Сервіс зберігає окремі екземпляри пам’яті для кожного ID розмови, що дозволяє кільком користувачам одночасно спілкуватися.

> **🤖 Спробуйте Chat з [GitHub Copilot](https://github.com/features/copilot):** Відкрийте [`ConversationService.java`](../../../01-introduction/src/main/java/com/example/langchain4j/service/ConversationService.java) і запитайте:
> - "Як MessageWindowChatMemory вирішує, які повідомлення видаляти, коли вікно заповнене?"
> - "Чи можу я реалізувати власне сховище пам’яті, використовуючи базу даних замість пам’яті в оперативній пам’яті?"
> - "Як додати підсумовування (summarization), щоб стискати стару історію розмов?"

Ендпоінт статичного чату зовсім пропускає пам’ять — просто `chatModel.chat(prompt)` як у швидкому старті. Ендпоінт динамічного додає повідомлення до пам’яті, отримує історію і включає цей контекст у кожен запит. Конфігурація моделі однакова, патерни різні.

## Розгортання Інфраструктури Azure OpenAI

**Bash:**
```bash
cd 01-introduction
azd up  # Виберіть підписку та розташування (рекомендується eastus2)
```

**PowerShell:**
```powershell
cd 01-introduction
azd up  # Виберіть підписку та розташування (рекомендується eastus2)
```

> **Примітка:** Якщо ви зіткнетеся з помилкою тайм-ауту (`RequestConflict: Cannot modify resource ... provisioning state is not terminal`), просто запустіть `azd up` знову. Ресурси Azure можуть все ще розгортатися у фоновому режимі, і повторна спроба дозволить завершити розгортання, коли ресурси потраплять у кінцевий стан.

Це виконає:
1. Розгортання ресурсу Azure OpenAI з моделями GPT-5.2 та text-embedding-3-small
2. Автоматичне створення файлу `.env` у корені проєкту з обліковими даними
3. Налаштування всіх необхідних змінних середовища

**Проблеми з розгортанням?** Подивіться [Інструкцію по інфраструктурі](infra/README.md) для докладного усунення неполадок, включно з конфліктами імен піддоменів, ручним розгортанням через Azure Portal і налаштуванням моделі.

**Перевірте успішність розгортання:**

**Bash:**
```bash
cat ../.env  # Має показувати AZURE_OPENAI_ENDPOINT, API_KEY тощо.
```

**PowerShell:**
```powershell
Get-Content ..\.env  # Має показувати AZURE_OPENAI_ENDPOINT, API_KEY тощо.
```

> **Примітка:** Команда `azd up` автоматично створює файл `.env`. Якщо потрібно оновити його пізніше, ви можете або редагувати `.env` вручну, або згенерувати його знову, виконавши:
>
> **Bash:**
> ```bash
> cd ..
> bash .azd-env.sh
> ```
>
> **PowerShell:**
> ```powershell
> cd ..
> .\.azd-env.ps1
> ```

## Запуск Застосунку Локально

**Перевірка розгортання:**

Переконайтеся, що файл `.env` існує у кореневій теці і містить облікові дані Azure:

**Bash:**
```bash
cat ../.env  # Має показувати AZURE_OPENAI_ENDPOINT, API_KEY, DEPLOYMENT
```

**PowerShell:**
```powershell
Get-Content ..\.env  # Має показувати AZURE_OPENAI_ENDPOINT, API_KEY, DEPLOYMENT
```

**Запуск застосунків:**

**Варіант 1: Використання Spring Boot Dashboard (Рекомендується для користувачів VS Code)**

Dev контейнер містить розширення Spring Boot Dashboard, яке надає візуальний інтерфейс для керування всіма Spring Boot застосунками. Ви знайдете його на Панелі Активності ліворуч у VS Code (піктограма Spring Boot).

З Spring Boot Dashboard ви можете:
- Переглядати всі доступні Spring Boot застосунки в робочій зоні
- Запускати/зупиняти застосунки одним кліком
- Переглядати логи застосунку в реальному часі
- Моніторити стан застосунку

Просто натисніть кнопку запуску поруч із "introduction", щоб запустити цей модуль, або запустіть усі модулі одночасно.

<img src="../../../translated_images/uk/dashboard.69c7479aef09ff6b.webp" alt="Spring Boot Dashboard" width="400"/>

**Варіант 2: Використання shell-скриптів**

Запустіть усі веб-застосунки (модулі 01-04):

**Bash:**
```bash
cd ..  # З кореневого каталогу
./start-all.sh
```

**PowerShell:**
```powershell
cd ..  # З кореневої директорії
.\start-all.ps1
```

Або запустіть лише цей модуль:

**Bash:**
```bash
cd 01-introduction
./start.sh
```

**PowerShell:**
```powershell
cd 01-introduction
.\start.ps1
```

Обидва скрипти автоматично завантажують змінні середовища з кореневого файлу `.env` і скомпілюють JAR, якщо його немає.

> **Примітка:** Якщо ви хочете вручну скомпілювати всі модулі перед запуском:
>
> **Bash:**
> ```bash
> cd ..  # Go to root directory
> mvn clean package -DskipTests
> ```
>
> **PowerShell:**
> ```powershell
> cd ..  # Go to root directory
> mvn clean package -DskipTests
> ```

Відкрийте http://localhost:8080 у вашому браузері.

**Щоб зупинити:**

**Bash:**
```bash
./stop.sh  # Тільки цей модуль
# Або
cd .. && ./stop-all.sh  # Усі модулі
```

**PowerShell:**
```powershell
.\stop.ps1  # Тільки цей модуль
# Або
cd ..; .\stop-all.ps1  # Усі модулі
```

## Використання Застосунку

Застосунок надає веб-інтерфейс з двома реалізаціями чату поруч.

<img src="../../../translated_images/uk/home-screen.121a03206ab910c0.webp" alt="Головний екран застосунку" width="800"/>

*Панель управління із обома варіантами: Простий Чат (без пам’яті) і Розмовний Чат (з пам’яттю)*

### Статичний Чат (ліва панель)

Спробуйте спершу це. Скажіть "Мене звати Джон", а потім одразу "Як мене звати?" Модель цього не запам’ятає, бо кожне повідомлення незалежне. Це демонструє основну проблему інтеграції базової мовної моделі — відсутність контексту розмови.

<img src="../../../translated_images/uk/simple-chat-stateless-demo.13aeb3978eab3234.webp" alt="Демонстрація статичного чату" width="800"/>

*ШІ не пам’ятає ваше ім’я з попереднього повідомлення*

### Динамічний Чат (права панель)

Тепер спробуйте таку ж послідовність тут. Скажіть "Мене звати Джон", а потім "Як мене звати?" Цього разу він запам’ятає. Різниця — MessageWindowChatMemory — він підтримує історію розмови і включає її в кожен запит. Саме так працює розмовний ШІ у виробництві.

<img src="../../../translated_images/uk/conversational-chat-stateful-demo.e5be9822eb23ff59.webp" alt="Демонстрація динамічного чату" width="800"/>

*ШІ пам’ятає ваше ім’я з ранньої частини розмови*

Обидві панелі використовують ту ж модель GPT-5.2. Єдина відмінність — пам’ять. Це чітко показує, що пам’ять приносить у ваш застосунок і чому вона є необхідною для реальних випадків використання.
