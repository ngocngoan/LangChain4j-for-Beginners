## Чого ви навчитеся

До цього моменту ви навчилися вести розмови з ШІ, ефективно структурувати підказки та базувати відповіді на ваших документах. Але існує фундаментальне обмеження: мовні моделі можуть лише генерувати текст. Вони не можуть перевірити погоду, виконувати обчислення, робити запити в бази даних або взаємодіяти із зовнішніми системами.

Інструменти змінюють це. Надаючи моделі доступ до функцій, які вона може викликати, ви трансформуєте її з генератора тексту в агента, що може здійснювати дії. Модель вирішує, коли їй потрібен інструмент, який інструмент використовувати та які параметри передати. Ваш код виконує функцію і повертає результат. Модель включає цей результат у свою відповідь.

## Розуміння AI-агентів з інструментами

> **📝 Примітка:** Термін "агенти" у цьому модулі означає AI-помічників, доповнених можливістю виклику інструментів. Це відрізняється від патернів **Agentic AI** (автономні агенти з плануванням, пам’яттю та багатоступеневим мисленням), які ми розглянемо у [Модулі 05: MCP](../05-mcp/README.md).

Без інструментів мовна модель може лише генерувати текст з навчальних даних. Запитайте у неї поточну погоду — вона має вгадувати. Наведіть інструменти — і вона зможе викликати API погоди, виконувати обчислення або робити запит у базу — а потім вписати ці реальні результати у відповідь.

<img src="../../../translated_images/uk/what-are-tools.724e468fc4de64da.webp" alt="Без інструментів vs із інструментами" width="800"/>

*Без інструментів модель може лише вгадувати — із інструментами вона може викликати API, виконувати обчислення та повертати дані в реальному часі.*

AI-агент з інструментами слідує патерну **Reasoning and Acting (ReAct)**. Модель не просто відповідає — вона думає, що їй потрібно, діє викликом інструмента, спостерігає результат, а потім вирішує, чи діяти знову або надати остаточну відповідь:

1. **Проаналізувати** — агент аналізує запит користувача й визначає, яка інформація потрібна
2. **Діяти** — агент обирає правильний інструмент, формує коректні параметри і викликає його
3. **Спостерігати** — агент отримує вихідні дані інструмента та оцінює результат
4. **Повторити або відповісти** — якщо потрібно більше даних, агент повертається до початку; інакше — формує відповідь природною мовою

<img src="../../../translated_images/uk/react-pattern-detail.96a5efeeb6dd2f61.webp" alt="Патерн ReAct" width="800"/>

*Цикл ReAct — агент аналізує, що робити, діє викликом інструменту, спостерігає результат і повторює дії, доки не сформує відповідь.*

Цей процес відбувається автоматично. Ви визначаєте інструменти та їхні описи. Модель приймає рішення, коли і як їх використовувати.

## Як працює виклик інструментів

### Визначення інструментів

[WeatherTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) | [TemperatureTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/TemperatureTool.java)

Ви визначаєте функції з чіткими описами та специфікацією параметрів. Модель бачить ці описи у системному підказі й розуміє, що робить кожен інструмент.

```java
@Component
public class WeatherTool {
    
    @Tool("Get the current weather for a location")
    public String getCurrentWeather(@P("Location name") String location) {
        // Ваша логіка пошуку погоди
        return "Weather in " + location + ": 22°C, cloudy";
    }
}

@AiService
public interface Assistant {
    String chat(@MemoryId String sessionId, @UserMessage String message);
}

// Асистент автоматично підключається Spring Boot з:
// - ChatModel bean
// - Усі методи @Tool з класів @Component
// - ChatMemoryProvider для керування сесією
```

Нижче наведена діаграма роз’яснює кожну анотацію та показує, як кожен елемент допомагає ШІ зрозуміти, коли викликати інструмент і які аргументи передати:

<img src="../../../translated_images/uk/tool-definitions-anatomy.f6468546037cf28b.webp" alt="Будова визначення інструментів" width="800"/>

*Будова визначення інструмента — @Tool каже ШІ, коли його використовувати, @P описує кожен параметр, а @AiService об’єднує все при запуску.*

### Прийняття рішень

Коли користувач питає «Яка погода у Сіетлі?», модель не вибирає інструмент випадково. Вона порівнює намір користувача з кожним описом інструмента, оцінює релевантність і вибирає найкращий варіант. Потім створює структурований виклик функції з правильними параметрами — у цьому випадку встановлює `location` в `"Seattle"`.

Якщо жоден інструмент не відповідає запиту, модель дає відповідь зі своїх знань. Якщо підходить кілька інструментів — обирає найконкретніший.

<img src="../../../translated_images/uk/decision-making.409cd562e5cecc49.webp" alt="Як ШІ вибирає інструмент" width="800"/>

*Модель оцінює кожен доступний інструмент згідно з наміром користувача та вибирає найкращий — тому важливо писати чіткі, конкретні описи інструментів.*

### Генерація відповіді

Модель отримує дані про погоду і форматує їх у відповідь природною мовою для користувача.

### Архітектура: Spring Boot Auto-Wiring

Цей модуль використовує інтеграцію LangChain4j зі Spring Boot та декларативні інтерфейси `@AiService`. Під час запуску Spring Boot знаходить усі `@Component` з методами, позначеними `@Tool`, ваш бін `ChatModel` і `ChatMemoryProvider` — а потім автоматично підключає їх у єдиний інтерфейс `Assistant` без зайвого коду.

<img src="../../../translated_images/uk/spring-boot-wiring.151321795988b04e.webp" alt="Архітектура Spring Boot Auto-Wiring" width="800"/>

*Інтерфейс @AiService об’єднує ChatModel, компоненти інструментів і провайдера пам’яті — Spring Boot автоматично здійснює підключення.*

Ось повний життєвий цикл запиту у вигляді діаграми послідовності — від HTTP-запиту через контролер, сервіс і проксі з авто-підключенням до виконання інструменту і назад:

<img src="../../../translated_images/uk/spring-boot-sequence.f83e3d485aa4a3c6.webp" alt="Послідовність виклику інструментів у Spring Boot" width="800"/>

*Повний життєвий цикл запиту у Spring Boot — HTTP-запит проходить через контролер і сервіс до проксі Assistant з авто-підключенням, який автоматично керує LLM і викликами інструментів.*

Ключові переваги такого підходу:

- **Автоматичне підключення у Spring Boot** — ChatModel і інструменти підключаються автоматично
- **Патерн @MemoryId** — автоматичне керування пам’яттю на основі сесії
- **Один екземпляр** — Assistant створюється один раз і повторно використовується для кращої продуктивності
- **Типобезпечне виконання** — Java методи викликаються напряму з конверсією типів
- **Оркестрація багаторазових кроків** — автоматично керує ланцюжками інструментів
- **Немає шаблонного коду** — без ручних викликів `AiServices.builder()` чи hashmap для пам’яті

Альтернативні підходи (ручний `AiServices.builder()`) вимагають більше коду і не мають переваг інтеграції зі Spring Boot.

## Ланцюжки інструментів

**Ланцюжки інструментів** — справжня сила агентів із інструментами виявляється, коли одне питання потребує кількох інструментів. Запитайте «Яка погода у Сіетлі у фаренгейтах?» — агент автоматично з’єднує два інструменти: спочатку викликає `getCurrentWeather` для отримання температури у Цельсіях, потім передає це значення у `celsiusToFahrenheit` для конвертації — усе в одному кроці розмови.

<img src="../../../translated_images/uk/tool-chaining-example.538203e73d09dd82.webp" alt="Приклад ланцюжка інструментів" width="800"/>

*Ланцюжок інструментів в дії — агент спочатку викликає getCurrentWeather, потім передає результат у Celsius у celsiusToFahrenheit і надає комбіновану відповідь.*

**Граційні збої** — запитайте погоду в місті, якого немає у тестових даних. Інструмент поверне повідомлення про помилку, а ШІ пояснить, що не може допомогти, замість того щоб аварійно завершитись. Інструменти безпечно обробляють помилки. Нижче діаграма протиставляє два підходи — з належною обробкою помилок агент ловить виняток і відповідає корисно, без неї весь застосунок аварійно завершується:

<img src="../../../translated_images/uk/error-handling-flow.9a330ffc8ee0475c.webp" alt="Потік обробки помилок" width="800"/>

*Якщо інструмент не вдається, агент ловить помилку і дає корисне пояснення замість краху.*

Це відбувається за один крок розмови. Агент автономно керує кількома викликами інструментів.

### Спробуйте просте використання інструменту

Почніть із простого запиту: "Перетвори 100 градусів за Фаренгейтом у Цельсій". Агент розпізнає, що потрібен інструмент конвертації температури, викликає його з правильними параметрами і повертає результат. Зверніть увагу, як природно це виглядає — вам не потрібно було вказувати, який інструмент використовувати або як його викликати.

### Перевірте послідовне використання інструментів

Тепер спробуйте щось складніше: "Яка погода в Сіетлі і перетвори її у Фаренгейти?" Подивіться, як агент виконає це поетапно. Спочатку він отримує погоду (у Цельсіях), розуміє, що потрібно конвертувати у Фаренгейти, виконує виклик інструменту конвертації і поєднує обидва результати в одну відповідь.

### Дивіться потік розмови

Інтерфейс чату зберігає історію розмови, що дозволяє вам вести багатокрокові взаємодії. Ви можете бачити всі попередні запити та відповіді, що полегшує відстеження розмови і розуміння, як агент формує контекст протягом кількох обмінів.

<a href="images/tools-conversation-demo.png"><img src="../../../translated_images/uk/tools-conversation-demo.89f2ce9676080f59.webp" alt="Conversation with Multiple Tool Calls" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*Багатокрокова розмова, що демонструє прості конвертації, запити погоди та послідовне використання інструментів*

### Експериментуйте з різними запитами

Спробуйте різні поєднання:
- Запити погоди: "Яка погода в Токіо?"
- Конвертації температур: "Скільки буде 25°C в Кельвінах?"
- Поєднані запити: "Перевір погоду в Парижі і скажи, чи більше там 20°C"

Зверніть увагу, як агент інтерпретує природну мову і відображає її у відповідні виклики інструментів.

## Ключові концепції

### Патерн ReAct (Раціонування та Дія)

Агент чергує раціонування (вирішення, що робити) та дії (використання інструментів). Цей патерн дозволяє автономне розв’язання проблем, а не просто реагування на інструкції.

### Опис інструментів має значення

Якість описів інструментів безпосередньо впливає на те, наскільки добре агент їх використовує. Чіткі, конкретні описи допомагають моделі розуміти, коли і як викликати кожен інструмент.

### Керування сесіями

Анотація `@MemoryId` забезпечує автоматичне керування пам’яттю на основі сесій. Кожен ідентифікатор сесії отримує власний екземпляр `ChatMemory`, керований біном `ChatMemoryProvider`, тож кілька користувачів можуть взаємодіяти з агентом одночасно, не перемішуючи свої розмови. Наступна діаграма показує, як кілька користувачів спрямовуються до ізольованих сховищ пам’яті на основі їхніх сесійних ідентифікаторів:

<img src="../../../translated_images/uk/session-management.91ad819c6c89c400.webp" alt="Session Management with @MemoryId" width="800"/>

*Кожен ідентифікатор сесії відповідає ізольованій історії розмов — користувачі ніколи не бачать повідомлення один одного.*

### Обробка помилок

Інструменти можуть збоїти — таймаути API, невірні параметри, відмова зовнішніх сервісів. Промислові агенти потребують обробки помилок, щоб модель могла пояснювати проблеми або пробувати альтернативи, а не ламати весь додаток. Коли інструмент кидає виняток, LangChain4j ловить його і передає повідомлення про помилку моделі, яка потім може пояснити проблему природною мовою.

## Доступні інструменти

Наступна діаграма показує широку екосистему інструментів, які ви можете створити. Цей модуль демонструє інструменти для погоди та температури, але той же патерн `@Tool` працює для будь-якого методу Java — від запитів до баз даних до обробки платежів.

<img src="../../../translated_images/uk/tool-ecosystem.aad3d74eaa14a44f.webp" alt="Tool Ecosystem" width="800"/>

*Будь-який метод Java з анотацією @Tool стає доступним AI — патерн поширюється на бази даних, API, електронну пошту, файлові операції тощо.*

## Коли використовувати агентів на базі інструментів

Не кожен запит потребує інструментів. Рішення залежить від того, чи потрібно AI взаємодіяти із зовнішніми системами, або він може відповідати зі своїх знань. Наступний посібник узагальнює, коли інструменти приносять користь, а коли вони не потрібні:

<img src="../../../translated_images/uk/when-to-use-tools.51d1592d9cbdae9c.webp" alt="When to Use Tools" width="800"/>

*Швидкий посібник для рішення — інструменти потрібні для даних у реальному часі, обчислень і дій; загальні знання та креативні завдання можна виконувати без них.*

## Інструменти проти RAG

Модулі 03 і 04 розширюють можливості AI, але в принципово різний спосіб. RAG надає моделі доступ до **знань** через пошук документів. Інструменти дають моделі можливість виконувати **дії** через виклики функцій. Наступна діаграма порівнює ці два підходи бок о бок — від того, як працює кожен робочий процес до компромісів між ними:

<img src="../../../translated_images/uk/tools-vs-rag.ad55ce10d7e4da87.webp" alt="Tools vs RAG Comparison" width="800"/>

*RAG отримує інформацію зі статичних документів — Інструменти виконують дії і отримують динамічні дані в реальному часі. Багато промислових систем поєднують обидва підходи.*

На практиці багато виробничих систем комбінують обидва підходи: RAG для підкріплення відповідей у документації, та Інструменти для отримання живих даних або виконання операцій.
