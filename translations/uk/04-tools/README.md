## Чому Ви Навчитеся

До цього моменту ви дізналися, як вести розмови з AI, ефективно структурувати підказки та базувати відповіді на ваших документах. Але залишається фундаментальне обмеження: мовні моделі можуть лише генерувати текст. Вони не можуть перевіряти погоду, виконувати обчислення, робити запити до баз даних або взаємодіяти із зовнішніми системами.

Інструменти змінюють цю ситуацію. Даючи моделі доступ до функцій, які вона може викликати, ви трансформуєте її з генератора тексту в агента, який може виконувати дії. Модель вирішує, коли їй потрібен інструмент, який саме інструмент використовувати і які параметри передати. Ваш код виконує функцію і повертає результат. Модель включає цей результат у свою відповідь.

## Розуміння AI Агентів з Інструментами

> **📝 Примітка:** Термін «агенти» у цьому модулі означає AI асистентів, розширених можливістю виклику інструментів. Це відрізняється від патернів **Agentic AI** (автономні агенти з плануванням, пам’яттю та багатокроковим розумуванням), які ми розглянемо в [Модулі 05: MCP](../05-mcp/README.md).

Без інструментів мовна модель може лише генерувати текст на основі свого навчання. Запитайте про поточну погоду — і вона мусить здогадуватися. Дайте їй інструменти — і вона може викликати API погоди, виконувати обчислення або робити запити до бази даних — а потім вплітати ці реальні результати у свою відповідь.

<img src="../../../translated_images/uk/what-are-tools.724e468fc4de64da.webp" alt="Без Інструментів проти З Інструментами" width="800"/>

*Без інструментів модель лише здогадується — з інструментами вона може викликати API, виконувати обчислення і повертати дані в реальному часі.*

AI агент з інструментами слідує патерну **Reasoning and Acting (ReAct)**. Модель не просто відповідає — вона думає, що їй потрібно, діє, викликаючи інструмент, спостерігає результат і потім вирішує, чи діяти ще раз, чи дати остаточну відповідь:

1. **Мислити** — Агент аналізує запит користувача і визначає, яку інформацію йому потрібно
2. **Діяти** — Агент обирає правильний інструмент, формує потрібні параметри і викликає його
3. **Спостерігати** — Агент отримує результат інструмента і оцінює його
4. **Повторити або Відповісти** — Якщо потрібні додаткові дані, агент повертається на початок; інакше формує остаточну відповідь природною мовою

<img src="../../../translated_images/uk/react-pattern-detail.96a5efeeb6dd2f61.webp" alt="Патерн ReAct" width="800"/>

*Цикл ReAct — агент аналізує що зробити, діє, викликаючи інструмент, спостерігає результат і повторює, доки не дасть кінцеву відповідь.*

Це відбувається автоматично. Ви визначаєте інструменти та їх описи. Модель приймає рішення, коли і як їх використовувати.

## Як Працює Виклик Інструментів

### Визначення Інструментів

[WeatherTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) | [TemperatureTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/TemperatureTool.java)

Ви визначаєте функції з чіткими описами та параметрами. Модель бачить ці описи у своєму системному промпті і розуміє, що робить кожен інструмент.

```java
@Component
public class WeatherTool {
    
    @Tool("Get the current weather for a location")
    public String getCurrentWeather(@P("Location name") String location) {
        // Ваша логіка пошуку погоди
        return "Weather in " + location + ": 22°C, cloudy";
    }
}

@AiService
public interface Assistant {
    String chat(@MemoryId String sessionId, @UserMessage String message);
}

// Асистент автоматично підключений за допомогою Spring Boot з:
// - Bean моделі чату
// - Усі методи @Tool з класів @Component
// - ChatMemoryProvider для керування сесіями
```
  
Наведена нижче діаграма розбиває кожну анотацію і показує, як кожна частина допомагає AI зрозуміти, коли викликати інструмент і які аргументи передати:

<img src="../../../translated_images/uk/tool-definitions-anatomy.f6468546037cf28b.webp" alt="Анатомія Визначень Інструментів" width="800"/>

*Анатомія визначення інструмента — @Tool каже AI, коли його використовувати, @P описує кожний параметр, а @AiService з’єднує все разом під час запуску.*

> **🤖 Спробуйте з [GitHub Copilot](https://github.com/features/copilot) Chat:** Відкрийте [`WeatherTool.java`](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) і запитайте:  
> - "Як інтегрувати реальний API погоди, як OpenWeatherMap замість мок-даних?"  
> - "Що робить хороший опис інструмента, який допомагає AI правильно його використовувати?"  
> - "Як обробляти помилки API та обмеження швидкості у реалізації інструментів?"

### Прийняття Рішень

Коли користувач запитує «Яка погода в Сіетлі?», модель не вибирає інструмент випадково. Вона порівнює намір користувача з кожним описом інструментів, які їй доступні, оцінює релевантність і вибирає найкраще співпадіння. Потім генерує структурований виклик функції з правильними параметрами — у цьому випадку встановлює `location` у `"Seattle"`.

Якщо жоден інструмент не підходить під запит користувача, модель дає відповідь з власних знань. Якщо підходить кілька інструментів, вона вибирає найконкретніший.

<img src="../../../translated_images/uk/decision-making.409cd562e5cecc49.webp" alt="Як AI Вирішує, Який Інструмент Використовувати" width="800"/>

*Модель оцінює кожен доступний інструмент щодо наміру користувача і вибирає найкращий варіант — тому важливо писати зрозумілі, конкретні описи інструментів.*

### Генерація Відповіді

Модель отримує дані про погоду і форматує їх у відповідь природною мовою для користувача.

### Архітектура: Spring Boot Автопідключення

Цей модуль використовує інтеграцію LangChain4j зі Spring Boot з декларативними інтерфейсами `@AiService`. Під час запуску Spring Boot знаходить кожен `@Component`, що містить методи `@Tool`, ваш бін `ChatModel` та `ChatMemoryProvider` — і підключає їх усі до одного інтерфейсу `Assistant` без будь-якого шаблонного коду.

<img src="../../../translated_images/uk/spring-boot-wiring.151321795988b04e.webp" alt="Архітектура Spring Boot Автопідключення" width="800"/>

*Інтерфейс @AiService об’єднує ChatModel, компоненти інструментів і провайдера пам’яті — Spring Boot виконує все підключення автоматично.*

Основні переваги цього підходу:

- **Автопідключення Spring Boot** — ChatModel і інструменти автоматично інжектуються  
- **Патерн @MemoryId** — Автоматичне керування сесійною пам’яттю  
- **Один екземпляр** — Assistant створюється один раз і повторно використовується для кращої продуктивності  
- **Типобезпечне виконання** — Виклики Java методів напряму з конвертацією типів  
- **Оркестрація багатокрокових дій** — Автоматично підтримує ланцюжки інструментів  
- **Нуль шаблонного коду** — Немає ручних викликів `AiServices.builder()` або пам’яті через HashMap

Альтернативні підходи (ручний `AiServices.builder()`) потребують більше коду і не мають переваг інтеграції з Spring Boot.

## Ланцюжки Інструментів

**Ланцюжки Інструментів** — Справжня сила агентів на основі інструментів проявляється, коли одне питання вимагає кілька інструментів. Запитайте «Яка погода в Сіетлі у Фаренгейтах?» і агент автоматично зв’язує два інструменти: спочатку викликає `getCurrentWeather`, щоб отримати температуру в Цельсіях, потім передає це значення у `celsiusToFahrenheit` для конвертації — все це в одному кроці розмови.

<img src="../../../translated_images/uk/tool-chaining-example.538203e73d09dd82.webp" alt="Приклад Ланцюжка Інструментів" width="800"/>

*Ланцюжок інструментів у дії — агент спочатку викликає getCurrentWeather, потім передає результат у celsiusToFahrenheit і дає комплексну відповідь.*

**Грамотна Обробка Збоїв** — Запитайте про погоду у місті, якого немає в мок-даних. Інструмент поверне повідомлення про помилку, і AI пояснить, що не може допомогти замість того, щоб аварійно завершитись. Інструменти безпечно падають. Діаграма нижче порівнює два підходи — з правильною обробкою помилок агент ловить виключення і відповідає корисним поясненням, без неї весь додаток крашиться:

<img src="../../../translated_images/uk/error-handling-flow.9a330ffc8ee0475c.webp" alt="Потік Обробки Помилок" width="800"/>

*Коли інструмент падає, агент ловить помилку і дає допоміжне пояснення замість збою.*

Це відбувається в одному кроці розмови. Агент автономно оркеструє кілька викликів інструментів.

## Використання Додатку

Додаток надає веб-інтерфейс, де ви можете взаємодіяти з AI агентом, який має доступ до інструментів погоди та конвертації температури. Ось як виглядає інтерфейс — він містить приклади швидкого старту і чат-панель для відправлення запитів:
<a href="images/tools-homepage.png"><img src="../../../translated_images/uk/tools-homepage.4b4cd8b2717f9621.webp" alt="Інтерфейс інструментів AI Agent" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*Інтерфейс інструментів AI Agent — швидкі приклади та чат для взаємодії з інструментами*

### Спробуйте просте використання інструменту

Почніть з простої запити: «Конвертуйте 100 градусів Фаренгейта у Цельсій». Агент розпізнає, що йому потрібен інструмент для конвертації температури, викликає його з правильними параметрами і повертає результат. Зверніть увагу, як природньо це відчувається — ви не вказували, який інструмент використовувати або як його викликати.

### Тестуйте ланцюжок інструментів

Тепер спробуйте щось складніше: «Яка погода в Сіетлі і конвертуйте її у Фаренгейти?» Спостерігайте, як агент працює крок за кроком. Спочатку він отримує погоду (яка повертається у Цельсіях), розуміє, що потрібно конвертувати у Фаренгейти, викликає інструмент конвертації та об’єднує обидва результати в одну відповідь.

### Подивіться потік розмови

Інтерфейс чату зберігає історію розмов, дозволяючи вести багаторазові діалоги. Ви можете бачити всі попередні запити і відповіді, що полегшує відстеження бесіди та розуміння, як агент створює контекст через кілька обмінів.

<a href="images/tools-conversation-demo.png"><img src="../../../translated_images/uk/tools-conversation-demo.89f2ce9676080f59.webp" alt="Розмова з кількома викликами інструментів" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*Багаторазова розмова, що показує прості конвертації, перевірки погоди та ланцюжок інструментів*

### Експериментуйте з різними запитами

Спробуйте різні комбінації:
- Запити погоди: «Яка погода в Токіо?»
- Конвертації температури: «Що таке 25°C у Кельвінах?»
- Комбіновані запити: «Перевірте погоду в Парижі і скажіть, чи вище там 20°C»

Зверніть увагу, як агент інтерпретує природну мову і відображає її на відповідні виклики інструментів.

## Ключові поняття

### Патерн ReAct (Міркування і дія)

Агент чергує міркування (вирішує, що робити) і дії (використання інструментів). Цей патерн дозволяє автономне розв’язання задач, а не лише відповідь на інструкції.

### Опис інструментів має значення

Якість описів ваших інструментів безпосередньо впливає на те, як добре агент їх використовує. Чіткі, специфічні описи допомагають моделі зрозуміти, коли і як викликати кожен інструмент.

### Управління сесіями

Анотація `@MemoryId` забезпечує автоматичне управління пам’яттю на основі сесій. Кожен ідентифікатор сесії отримує свій власний екземпляр `ChatMemory`, яким керує компонент `ChatMemoryProvider`, тож кілька користувачів можуть одночасно взаємодіяти з агентом, не перетинаючи свої розмови. Наступна діаграма показує, як користувачі направляються до ізольованих сховищ пам’яті на основі своїх ідентифікаторів сесій:

<img src="../../../translated_images/uk/session-management.91ad819c6c89c400.webp" alt="Керування сесіями з @MemoryId" width="800"/>

*Кожен ідентифікатор сесії відповідає ізольованій історії розмов — користувачі ніколи не бачать повідомлення один одного.*

### Обробка помилок

Інструменти можуть давати збій — API можуть не відповідати, параметри можуть бути невалідними, зовнішні сервіси можуть падати. Продакшн-агенти потребують обробки помилок, щоб модель могла пояснювати проблеми або шукати альтернативи замість того, щоб викликати збій усієї програми. Коли інструмент викидає виключення, LangChain4j ловить його і передає повідомлення про помилку назад моделі, яка може пояснити проблему природною мовою.

## Доступні інструменти

Діаграма нижче показує широку екосистему інструментів, які ви можете створювати. Цей модуль демонструє інструменти для погоди та температури, але той же патерн `@Tool` працює для будь-якого Java-методу — від запитів до баз даних до обробки платежів.

<img src="../../../translated_images/uk/tool-ecosystem.aad3d74eaa14a44f.webp" alt="Екосистема інструментів" width="800"/>

*Будь-який Java-метод з анотацією @Tool стає доступним для AI — цей патерн застосовується до баз даних, API, електронної пошти, операцій з файлами та ін.*

## Коли використовувати агентів на основі інструментів

Не всі запити потребують інструментів. Рішення залежить від того, чи потрібно AI взаємодіяти з зовнішніми системами або він може відповісти за власними знаннями. Наступне керівництво узагальнює, коли інструменти додають цінність, а коли вони зайві:

<img src="../../../translated_images/uk/when-to-use-tools.51d1592d9cbdae9c.webp" alt="Коли використовувати інструменти" width="800"/>

*Швидкий довідник — інструменти потрібні для даних у реальному часі, розрахунків та дій; загальні знання та творчі завдання — ні.*

## Інструменти проти RAG

Модулі 03 і 04 істотно розширюють можливості AI, але принципово по-різному. RAG дає моделі доступ до **знань** через вибірку документів. Інструменти дають моделі можливість виконувати **дії** за допомогою викликів функцій. Діаграма нижче порівнює ці два підходи бок о бок — від того, як працює кожен робочий процес до компромісів між ними:

<img src="../../../translated_images/uk/tools-vs-rag.ad55ce10d7e4da87.webp" alt="Порівняння інструментів і RAG" width="800"/>

*RAG отримує інформацію зі статичних документів — Інструменти виконують дії і отримують динамічні, актуальні дані. Багато виробничих систем поєднують обидва.*

На практиці багато виробничих систем поєднують обидва підходи: RAG для підкріплення відповідей документацією, Інструменти — для отримання живих даних або виконання операцій.
