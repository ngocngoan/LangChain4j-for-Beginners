## Чого ви навчитеся

У попередніх модулях ви навчились вести розмови з ШІ та ефективно структурувати запити. Але існує фундаментальне обмеження: мовні моделі знають лише те, що вивчили під час тренування. Вони не можуть відповідати на питання про політики вашої компанії, документацію проєкту чи будь-яку інформацію, якої не було у тренувальних даних.

RAG (Генерація з пошуком інформації) вирішує цю проблему. Замість того, щоб навчати модель вашій інформації (що дорого і недоцільно), ви даєте їй можливість шукати в ваших документах. Коли хтось задає запитання, система знаходить релевантну інформацію і додає її до запиту. Модель відповідає, ґрунтуючись на цьому витягнутому контексті.

Уявіть RAG як надання моделі довідкової бібліотеки. Коли ви задаєте запитання, система:

1. **Запит користувача** — Ви задаєте питання  
2. **Векторне подання** — Перетворює ваше питання у вектор  
3. **Пошук у векторному просторі** — Знаходить схожі частини документів  
4. **Формування контексту** — Додає релевантні частини до запиту  
5. **Відповідь** — LLM генерує відповідь на основі контексту  

Це дає змогу пов’язати відповіді моделі з вашими реальними даними замість того, щоб покладатися лише на знання з тренування або вигадувати відповіді.

## Розуміння RAG

Нижче наведена схема ілюструє основну ідею: замість покладання лише на тренувальні дані моделі, RAG дає їй довідкову бібліотеку ваших документів для консультації перед кожною відповіддю.

<img src="../../../translated_images/uk/what-is-rag.1f9005d44b07f2d8.webp" alt="What is RAG" width="800"/>

Ось як пов’язані частини в кінцевому результаті. Запит користувача проходить через чотири етапи — векторне подання, пошук у векторах, формування контексту та генерація відповіді — кожен з яких базується на попередньому:

<img src="../../../translated_images/uk/rag-architecture.ccb53b71a6ce407f.webp" alt="RAG Architecture" width="800"/>

Решта цього модуля детально розгляне кожен етап із прикладами коду, які ви можете запускати та змінювати.

## Як це працює

### Обробка документів

[DocumentService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/DocumentService.java)

Коли ви завантажуєте документ, система його розбирає (PDF або звичайний текст), додає метадані, як-от ім’я файлу, а потім розбиває на частини — менші фрагменти, які зручно вміщуються у вікно контексту моделі. Ці частини трохи перекриваються, щоб не втратити контекст на межах.

```java
// Розпарсити завантажений файл і обгорнути його в документ LangChain4j
Document document = Document.from(content, metadata);

// Розбити на фрагменти по 300 токенів з перекриттям у 30 токенів
DocumentSplitter splitter = DocumentSplitters
    .recursive(300, 30);

List<TextSegment> segments = splitter.split(document);
```
  
Нижче наведена діаграма, яка показує це візуально. Зверніть увагу, що кожна частина ділить деякі токени з сусідами — перекриття у 30 токенів гарантує, що важливий контекст не загубиться поміж ними:

<img src="../../../translated_images/uk/document-chunking.a5df1dd1383431ed.webp" alt="Document Chunking" width="800"/>

> - "Як працювати з документами кількома мовами або з особливим форматуванням?"

### Створення векторних подань

[LangChainRagConfig.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/config/LangChainRagConfig.java)

Кожна частина перетворюється у числове подання, яке називають embedding — по суті математичний відбиток, що захоплює зміст тексту. Схожі тексти дають схожі embeddings.

```java
@Bean
public EmbeddingModel embeddingModel() {
    return OpenAiOfficialEmbeddingModel.builder()
        .baseUrl(azureOpenAiEndpoint)
        .apiKey(azureOpenAiKey)
        .modelName(azureEmbeddingDeploymentName)
        .build();
}

EmbeddingStore<TextSegment> embeddingStore = 
    new InMemoryEmbeddingStore<>();
```
  
Нижче діаграма класів показує, як ці компоненти LangChain4j взаємодіють. `OpenAiOfficialEmbeddingModel` перетворює текст у вектори, `InMemoryEmbeddingStore` зберігає вектори разом з оригінальними даними `TextSegment`, а `EmbeddingSearchRequest` керує параметрами пошуку, як-от `maxResults` та `minScore`:

<img src="../../../translated_images/uk/rag-langchain4j-classes.bbf3aa9077ab443d.webp" alt="LangChain4j RAG Classes" width="800"/>

Після збереження embedding-и схожий контент природно формує кластери у векторному просторі. Візуалізація нижче показує, як документи на суміжні теми утворюють близькі точки, що робить можливим семантичний пошук:

<img src="../../../translated_images/uk/vector-embeddings.2ef7bdddac79a327.webp" alt="Vector Embeddings Space" width="800"/>

### Семантичний пошук

[RagService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/RagService.java)

Коли ви ставите питання, ваше питання теж перетворюється у embedding. Система порівнює embedding питання з embedding-ами частин документа. Вона знаходить частини з найбільш схожим значенням — не просто за ключовими словами, а за семантичною близькістю.

```java
Embedding queryEmbedding = embeddingModel.embed(question).content();

EmbeddingSearchRequest searchRequest = EmbeddingSearchRequest.builder()
    .queryEmbedding(queryEmbedding)
    .maxResults(5)
    .minScore(0.5)
    .build();

EmbeddingSearchResult<TextSegment> searchResult = embeddingStore.search(searchRequest);
List<EmbeddingMatch<TextSegment>> matches = searchResult.matches();

for (EmbeddingMatch<TextSegment> match : matches) {
    String relevantText = match.embedded().text();
    double score = match.score();
}
```
  
Нижче схема протиставляє семантичний пошук традиційному пошуку за ключовими словами. Пошук за ключовим словом "транспорт" упускає частину про "автомобілі та вантажівки", але семантичний пошук розуміє, що це одне й те саме і показує це як результат з високим балом:

<img src="../../../translated_images/uk/semantic-search.6b790f21c86b849d.webp" alt="Semantic Search" width="800"/>

> **🤖 Спробуйте з [GitHub Copilot](https://github.com/features/copilot) Chat:** Відкрийте [`RagService.java`](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/RagService.java) і запитайте:

> - "Який поріг схожості варто використовувати і як це впливає на результати?"
> - "Що робити, якщо релевантних документів не знайдено?"

### Генерація відповіді

[RagService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/RagService.java)

Найрелевантніші частини збираються у структурований запит, що включає чіткі інструкції, витягнутий контекст і питання користувача. Модель читає лише ці конкретні частини і відповідає на основі цієї інформації — вона може використовувати лише інформацію, що перед нею, що запобігає вигадкам.

```java
String context = matches.stream()
    .map(match -> match.embedded().text())
    .collect(Collectors.joining("\n\n"));

String prompt = String.format("""
    Answer the question based on the following context.
    If the answer cannot be found in the context, say so.

    Context:
    %s

    Question: %s

    Answer:""", context, request.question());

String answer = chatModel.chat(prompt);
```
  
Нижче показано цей процес у дії — найкращі частини з кроку пошуку вставляються у шаблон запиту, і `OpenAiOfficialChatModel` генерує обґрунтовану відповідь:

<img src="../../../translated_images/uk/context-assembly.7e6dd60c31f95978.webp" alt="Context Assembly" width="800"/>

### Завантаження документа

Почніть із завантаження документа — для тестування найкраще підходять TXT-файли. У цій директорії є `sample-document.txt`, що містить інформацію про можливості LangChain4j, впровадження RAG і найкращі практики — ідеально підходить для тестування системи.

Система обробляє ваш документ, розбиває його на частини й створює embedding-и для кожної частини. Це відбувається автоматично під час завантаження.

### Задавання питань

Тепер задавайте конкретні питання про вміст документа. Спробуйте щось фактичне, явно вказане у документі. Система шукає релевантні частини, включає їх у запит і генерує відповідь.

### Експерименти з питаннями

Спробуйте різні типи питань:

## Ключові концепції

### Стратегія розбиття на частини

Документи розбивають на частини по 300 токенів із 30 токенами перекриття. Такий баланс гарантує, що кожна частина має достатній контекст, щоб бути осмисленою, і при цьому є досить невеликою, щоб вмістити кілька частин у запит.

### Оцінки схожості

Кожна витягнута частина супроводжується оцінкою схожості від 0 до 1, яка показує, наскільки тісно вона відповідає питанню користувача. Нижче схема візуалізує діапазони оцінок і як система використовує їх для фільтрації результатів:

<img src="../../../translated_images/uk/similarity-scores.b0716aa911abf7f0.webp" alt="Similarity Scores" width="800"/>

Оцінки від 0 до 1:  
- 0.7-1.0: Дуже релевантно, точний збіг  
- 0.5-0.7: Релевантно, хороший контекст  
- Нижче 0.5: Відфільтровано, занадто низька схожість  

Система отримує тільки частини з оцінками вище мінімального порогу для забезпечення якості.

### Зберігання в пам’яті

Цей модуль використовує зберігання в оперативній пам’яті для простоти. При перезапуску застосунку завантажені документи втрачаються. Промислові системи використовують персистентні векторні бази даних, як Qdrant або Azure AI Search.

### Керування вікном контексту

У кожної моделі є максимальний розмір вікна контексту. Ви не можете включити всі частини великого документа. Система витягує топ N найбільш релевантних частин (за замовчуванням 5), щоб залишатися в межах ліміту, забезпечуючи достатній контекст для точних відповідей.

## Коли RAG є важливим

RAG не завжди є правильним підходом. Наступний посібник допоможе визначити, коли RAG додає цінність, а коли простіші підходи — як включення контенту безпосередньо у запит або покладання на вбудовані знання моделі — є достатніми:

<img src="../../../translated_images/uk/when-to-use-rag.1016223f6fea26bc.webp" alt="When to Use RAG" width="800"/>

**Використовуйте RAG, коли:**
- Відповіді на питання про власницькі документи  
- Інформація часто змінюється (політики, ціни, специфікації)  
- Точність вимагає посилання на джерело  
- Зміст занадто великий, щоб поміститися в один запит  
- Потрібні перевірені, обґрунтовані відповіді  

**Не використовуйте RAG, коли:**  
- Питання вимагають загальних знань, які модель уже має  
- Потрібні дані в реальному часі (RAG працює з завантаженими документами)  
- Вміст досить малий, щоб включити його безпосередньо в запити
