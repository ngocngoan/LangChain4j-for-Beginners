## Чого ви навчитеся

У попередньому модулі ви побачили, як пам’ять дозволяє розмовному ШІ, і використовували GitHub Models для базових взаємодій. Тепер ми зосередимося на тому, як ви ставите питання — самих підказках — використовуючи GPT-5.2 від Azure OpenAI. Те, як ви структуруєте свої підказки, радикально впливає на якість отриманих відповідей.

Ми використаємо GPT-5.2, тому що він вводить контроль міркувань — ви можете вказати моделі, скільки думати перед відповіддю. Це робить різні стратегії підказок більш помітними та допомагає зрозуміти, коли використовувати кожен підхід. Також ми скористаємося меншою кількістю лімітів на запити GPT-5.2 у Azure порівняно з GitHub Models.

## Розуміння проєктування підказок

Проєктування підказок полягає у створенні вхідного тексту, який стабільно дає потрібні вам результати. Це не просто про ставлення питань — це про структурування запитів так, щоб модель чітко розуміла, чого ви хочете й як це надати.

Уявіть, що ви даєте вказівки колезі. «Виправи баг» — це невизначено. «Виправ помилку null pointer exception у UserService.java на рядку 45, додавши перевірку на null» — це конкретно. Мовні моделі працюють так само — важлива конкретність і структура.

## Як це використовує LangChain4j

Цей модуль демонструє просунуті патерни промптингу, використовуючи ту саму основу LangChain4j з попередніх модулів, зосереджуючись на структурі підказок і контролі міркувань.

<img src="../../../translated_images/uk/langchain4j-flow.48e534666213010b.webp" alt="LangChain4j Flow" width="800"/>

*Як LangChain4j підключає ваші підказки до Azure OpenAI GPT-5.2*

**Залежності** — Модуль 02 використовує такі залежності langchain4j, описані в `pom.xml`:
```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j</artifactId> <!-- Inherited from BOM in root pom.xml -->
</dependency>
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-open-ai-official</artifactId> <!-- Inherited from BOM in root pom.xml -->
</dependency>
```

**Конфігурація OpenAiOfficialChatModel** — [LangChainConfig.java](../../../02-prompt-engineering/src/main/java/com/example/langchain4j/prompts/config/LangChainConfig.java)

Чат-модель налаштована вручну як Spring bean, використовуючи офіційний клієнт OpenAI, який підтримує кінцеві точки Azure OpenAI. Ключова різниця від Модуля 01 — як ми структуруємо підказки, що надсилаються в `chatModel.chat()`, а не сама налаштування моделі.

**Повідомлення системи та користувача** — [Gpt5PromptService.java](../../../02-prompt-engineering/src/main/java/com/example/langchain4j/prompts/service/Gpt5PromptService.java)

LangChain4j розділяє типи повідомлень для ясності. `SystemMessage` встановлює поведінку та контекст ШІ (наприклад, «Ви — рецензент коду»), тоді як `UserMessage` містить фактичний запит. Це розділення дозволяє підтримувати послідовну поведінку ШІ для різних користувацьких запитів.

```java
SystemMessage systemMsg = SystemMessage.from(
    "You are a helpful Java programming expert."
);

UserMessage userMsg = UserMessage.from(
    "Explain what a List is in Java"
);

String response = chatModel.chat(systemMsg, userMsg);
```

<img src="../../../translated_images/uk/message-types.93e0779798a17c9d.webp" alt="Message Types Architecture" width="800"/>

*SystemMessage забезпечує постійний контекст, а UserMessages містять індивідуальні запити*

**MessageWindowChatMemory для багатокрокових сесій** — Для патерну багатокрокової бесіди ми повторно використовуємо `MessageWindowChatMemory` з Модуля 01. Кожна сесія отримує власний екземпляр пам’яті, що зберігається у `Map<String, ChatMemory>`, що дозволяє вести кілька одночасних бесід без змішування контекстів.

**Шаблони підказок** — Справжній фокус тут — проєктування підказок, а не нові API LangChain4j. Кожен патерн (низька зацікавленість, висока зацікавленість, виконання завдань тощо) використовує той самий метод `chatModel.chat(prompt)`, але з ретельно структурованими рядками підказок. XML-теги, інструкції та форматування — все це частина тексту підказки, а не функції LangChain4j.

**Контроль міркувань** — Зусилля міркувань GPT-5.2 контролюються через інструкції в підказках, як-от «максимум 2 кроки міркувань» або «досліджуй ретельно». Це техніки проєктування підказок, а не налаштування LangChain4j. Бібліотека просто доставляє ваші підказки моделі.

Головний висновок: LangChain4j забезпечує інфраструктуру (підключення моделі через [LangChainConfig.java](../../../02-prompt-engineering/src/main/java/com/example/langchain4j/prompts/config/LangChainConfig.java), пам’ять, обробку повідомлень через [Gpt5PromptService.java](../../../02-prompt-engineering/src/main/java/com/example/langchain4j/prompts/service/Gpt5PromptService.java)), а цей модуль навчає, як створювати ефективні підказки всередині цієї інфраструктури.

## Основні патерни

Не всі завдання потребують одного підходу. Деякі питання потребують швидких відповідей, інші — глибокого мислення. Дехто потребують помітного міркування, інші — просто результатів. Цей модуль розглядає вісім патернів проєктування підказок — кожен оптимізований для різних сценаріїв. Ви експериментуватимете з усіма, щоб зрозуміти, коли який підхід працює найкраще.

<img src="../../../translated_images/uk/eight-patterns.fa1ebfdf16f71e9a.webp" alt="Eight Prompting Patterns" width="800"/>

*Огляд восьми патернів проєктування підказок та їхніх випадків використання*

<img src="../../../translated_images/uk/reasoning-effort.db4a3ba5b8e392c1.webp" alt="Reasoning Effort Comparison" width="800"/>

*Низька зацікавленість (швидкий, прямий) проти високої зацікавленості (ретельний, дослідницький)*

**Низька зацікавленість (швидко і сфокусовано)** — Для простих питань, де потрібно швидко і прямо відповісти. Модель робить мінімальні міркування — максимум 2 кроки. Використовуйте для обчислень, пошуків або простих питань.

```java
String prompt = """
    <reasoning_effort>low</reasoning_effort>
    <instruction>maximum 2 reasoning steps</instruction>
    
    What is 15% of 200?
    """;

String response = chatModel.chat(prompt);
```

> 💡 **Досліджуйте з GitHub Copilot:** Відкрийте [`Gpt5PromptService.java`](../../../02-prompt-engineering/src/main/java/com/example/langchain4j/prompts/service/Gpt5PromptService.java) і запитайте:
> - «Чим відрізняються патерни низької та високої зацікавленості?»
> - «Як XML-теги в підказках допомагають структуризувати відповідь AI?»
> - «Коли слід використовувати патерни саморефлексії, а коли прямі інструкції?»

**Висока зацікавленість (глибоко і ретельно)** — Для складних задач, де потрібно всебічний аналіз. Модель досліджує ретельно і показує детальне міркування. Використовуйте для проєктування систем, архітектурних рішень або складних досліджень.

```java
String prompt = """
    <reasoning_effort>high</reasoning_effort>
    <instruction>explore thoroughly, show detailed reasoning</instruction>
    
    Design a caching strategy for a high-traffic REST API.
    """;

String response = chatModel.chat(prompt);
```

**Виконання завдань (поступовий прогрес)** — Для багатокрокових робочих процесів. Модель надає попередній план, розповідає про кожен крок у процесі, а потім дає підсумок. Використовуйте для міграцій, впроваджень або будь-яких багатокрокових процесів.

```java
String prompt = """
    <task>Create a REST endpoint for user registration</task>
    <preamble>Provide an upfront plan</preamble>
    <narration>Narrate each step as you work</narration>
    <summary>Summarize what was accomplished</summary>
    """;

String response = chatModel.chat(prompt);
```

Проєктування з ланцюгом міркувань (Chain-of-Thought) явно просить модель показати процес міркувань, підвищуючи точність для складних завдань. Покроковий розбір допомагає і людям, і ШІ зрозуміти логіку.

> **🤖 Спробуйте з чатом [GitHub Copilot](https://github.com/features/copilot):** Запитайте про цей патерн:
> - «Як адаптувати патерн виконання завдань для довготривалих операцій?»
> - «Які кращі практики для структурування преамбул інструментів у промислових додатках?»
> - «Як фіксувати і відображати проміжні оновлення прогресу у UI?»

<img src="../../../translated_images/uk/task-execution-pattern.9da3967750ab5c1e.webp" alt="Task Execution Pattern" width="800"/>

*План → Виконання → Підсумок робочого процесу для багатокрокових завдань*

**Саморефлексивний код** — Для генерації коду виробничої якості. Модель генерує код, перевіряє його за критеріями якості та покращує ітеративно. Використовуйте для створення нових функцій або сервісів.

```java
String prompt = """
    <task>Create an email validation service</task>
    <quality_criteria>
    - Correct logic and error handling
    - Best practices (clean code, proper naming)
    - Performance optimization
    - Security considerations
    </quality_criteria>
    <instruction>Generate code, evaluate against criteria, improve iteratively</instruction>
    """;

String response = chatModel.chat(prompt);
```

<img src="../../../translated_images/uk/self-reflection-cycle.6f71101ca0bd28cc.webp" alt="Self-Reflection Cycle" width="800"/>

*Цикл ітеративного вдосконалення — генеруй, оцінюй, виявляй проблеми, покращуй, повторюй*

**Структурований аналіз** — Для послідовної оцінки. Модель переглядає код, використовуючи фіксовану структуру (коректність, практики, продуктивність, безпека). Використовуйте для рецензій коду або оцінок якості.

```java
String prompt = """
    <code>
    public List getUsers() {
        return database.query("SELECT * FROM users");
    }
    </code>
    
    <framework>
    Evaluate using these categories:
    1. Correctness - Logic and functionality
    2. Best Practices - Code quality
    3. Performance - Efficiency concerns
    4. Security - Vulnerabilities
    </framework>
    """;

String response = chatModel.chat(prompt);
```

> **🤖 Спробуйте з чатом [GitHub Copilot](https://github.com/features/copilot):** Запитайте про структурований аналіз:
> - «Як кастомізувати структуру аналізу для різних типів рецензій коду?»
> - «Який найкращий спосіб програмно розбирати і реагувати на структурований вивід?»
> - «Як забезпечити послідовність рівнів критичності у різних сесіях рецензій?»

<img src="../../../translated_images/uk/structured-analysis-pattern.0af3b690b60cf2d6.webp" alt="Structured Analysis Pattern" width="800"/>

*Рамкова структура з чотирма категоріями для послідовних рецензій коду з рівнями серйозності*

**Багатокроковий чат** — Для розмов, що потребують контексту. Модель пам’ятає попередні повідомлення й базується на них. Використовуйте для інтерактивної допомоги або складних Q&A.

```java
ChatMemory memory = MessageWindowChatMemory.withMaxMessages(10);

memory.add(UserMessage.from("What is Spring Boot?"));
AiMessage aiMessage1 = chatModel.chat(memory.messages()).aiMessage();
memory.add(aiMessage1);

memory.add(UserMessage.from("Show me an example"));
AiMessage aiMessage2 = chatModel.chat(memory.messages()).aiMessage();
memory.add(aiMessage2);
```

<img src="../../../translated_images/uk/context-memory.dff30ad9fa78832a.webp" alt="Context Memory" width="800"/>

*Як контекст розмови акумулюється через кілька кроків до досягнення ліміту токенів*

**Покрокове міркування** — Для задач, що потребують видимої логіки. Модель демонструє явне міркування для кожного кроку. Використовуйте для математичних завдань, логічних головоломок або коли потрібно зрозуміти процес мислення.

```java
String prompt = """
    <instruction>Show your reasoning step-by-step</instruction>
    
    If a train travels 120 km in 2 hours, then stops for 30 minutes,
    then travels another 90 km in 1.5 hours, what is the average speed
    for the entire journey including the stop?
    """;

String response = chatModel.chat(prompt);
```

<img src="../../../translated_images/uk/step-by-step-pattern.a99ea4ca1c48578c.webp" alt="Step-by-Step Pattern" width="800"/>

*Розбиття задач на явні логічні кроки*

**Обмежений вивід** — Для відповідей із конкретними форматними вимогами. Модель суворо дотримується правил формату та довжини. Використовуйте для підсумків або коли потрібна точна структура виводу.

```java
String prompt = """
    <constraints>
    - Exactly 100 words
    - Bullet point format
    - Technical terms only
    </constraints>
    
    Summarize the key concepts of machine learning.
    """;

String response = chatModel.chat(prompt);
```

<img src="../../../translated_images/uk/constrained-output-pattern.0ce39a682a6795c2.webp" alt="Constrained Output Pattern" width="800"/>

*Забезпечення конкретних вимог формату, довжини та структури*

## Що ви справді вивчаєте

**Зусилля з міркування змінює все**

GPT-5.2 дозволяє контролювати обчислювальні зусилля через ваші підказки. Низькі зусилля означають швидкі відповіді з мінімальним дослідженням. Високі зусилля — коли модель думає глибоко і довго. Ви вчитеся співвідносити зусилля з складністю завдання — не витрачайте час на прості запитання, але й не поспішайте з складними рішеннями.

**Структура спрямовує поведінку**

Помітили XML-теги в підказках? Вони не просто для краси. Моделі краще дотримуються структурованих вказівок, ніж вільного тексту. Коли потрібні багатокрокові процеси або складна логіка, структура допомагає моделі відстежувати, де вона і що далі.

<img src="../../../translated_images/uk/prompt-structure.a77763d63f4e2f89.webp" alt="Структура підказки" width="800"/>

*Анатомія добре структурованої підказки з чіткими розділами і організацією в стилі XML*

**Якість через самооцінку**

Шаблони саморефлексії працюють завдяки явному визначенню критеріїв якості. Замість сподівання, що модель «зробить правильно», ви чітко кажете їй, що означає «правильно»: коректна логіка, обробка помилок, продуктивність, безпека. Потім модель оцінює свій вихід і покращується. Це перетворює генерацію коду з лотереї на процес.

**Контекст обмежений**

Багатокрокові розмови працюють за рахунок включення історії повідомлень у кожен запит. Але є межа — у кожної моделі максимальна кількість токенів. Коли розмір розмови збільшується, потрібно стратегії для збереження релевантного контексту, не досягаючи ліміту. Цей модуль показує, як працює пам’ять; пізніше ви дізнаєтесь, коли підсумовувати, коли забувати, а коли витягувати інформацію.
