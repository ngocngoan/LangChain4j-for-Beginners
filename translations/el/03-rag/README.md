## Τι Θα Μάθετε

Στα προηγούμενα modules, μάθατε πώς να έχετε συνομιλίες με το AI και να δομείτε τα prompts σας αποτελεσματικά. Αλλά υπάρχει ένας θεμελιώδης περιορισμός: τα γλωσσικά μοντέλα γνωρίζουν μόνο όσα έμαθαν κατά την εκπαίδευσή τους. Δεν μπορούν να απαντούν σε ερωτήσεις σχετικά με τις πολιτικές της εταιρείας σας, την τεκμηρίωση του έργου σας ή οποιαδήποτε πληροφορία στην οποία δεν εκπαιδεύτηκαν.

Το RAG (Ανάκτηση-Ενισχυμένη Δημιουργία) λύνει αυτό το πρόβλημα. Αντί να προσπαθείτε να διδάξετε στο μοντέλο τις πληροφορίες σας (κάτι που είναι ακριβό και μη πρακτικό), του δίνετε τη δυνατότητα να ψάξει μέσα στα έγγραφά σας. Όταν κάποιος κάνει μια ερώτηση, το σύστημα βρίσκει σχετικές πληροφορίες και τις ενσωματώνει στο prompt. Το μοντέλο απαντά βασισμένο σε αυτό το ανακτημένο συμφραζόμενο.

Σκεφτείτε το RAG σαν να δίνετε στο μοντέλο μια βιβλιοθήκη αναφοράς. Όταν κάνετε μια ερώτηση, το σύστημα:

1. **Ερώτηση Χρήστη** – Κάνετε μια ερώτηση  
2. **Ενσωμάτωση (Embedding)** – Μετατρέπει την ερώτησή σας σε διάνυσμα  
3. **Αναζήτηση Διανύσματος** – Βρίσκει παρόμοια κομμάτια εγγράφων  
4. **Σύνθεση Συμφραζομένων** – Προσθέτει σχετικά κομμάτια στο prompt  
5. **Απάντηση** – Το LLM παράγει απάντηση βάσει του συμφραζομένου  

Αυτό εδραιώνει τις απαντήσεις του μοντέλου στα πραγματικά σας δεδομένα αντί να βασίζεται μόνο στη γνώση εκπαίδευσής του ή να εφευρίσκει απαντήσεις.

### Ποια Προσέγγιση RAG Χρησιμοποιεί Αυτό το Σεμινάριο;

Το LangChain4j προσφέρει τρεις τρόπους υλοποίησης του RAG, καθένας με διαφορετικό επίπεδο αφαίρεσης. Το διάγραμμα παρακάτω τους συγκρίνει δίπλα-δίπλα:

<img src="../../../translated_images/el/rag-approaches.5b97fdcc626f1447.webp" alt="Τρεις Προσεγγίσεις RAG στο LangChain4j" width="800"/>

*Αυτό το διάγραμμα συγκρίνει τις τρεις προσεγγίσεις RAG του LangChain4j — Easy, Native και Advanced — δείχνοντας τα βασικά τους στοιχεία και πότε να χρησιμοποιηθεί κάθε μία.*

| Προσέγγιση | Τι Κάνει | Ανταλλαγή |
|---|---|---|
| **Easy RAG** | Συνδέει τα πάντα αυτόματα μέσω των `AiServices` και `ContentRetriever`. Σημειώνετε ένα interface, επισυνάπτετε έναν retriever, και το LangChain4j χειρίζεται ενσωμάτωση, αναζήτηση και σχηματισμό prompt στο παρασκήνιο. | Ελάχιστος κώδικας, αλλά δεν βλέπετε τι συμβαίνει σε κάθε βήμα. |
| **Native RAG** | Καλείτε εσείς το μοντέλο ενσωμάτωσης, ψάχνετε στην αποθήκη, χτίζετε το prompt και παράγετε την απάντηση — ένα ρητό βήμα τη φορά. | Περισσότερος κώδικας, αλλά κάθε στάδιο είναι ορατό και τροποποιήσιμο. |
| **Advanced RAG** | Χρησιμοποιεί το πλαίσιο `RetrievalAugmentor` με επεκτάσιμους μετασχηματιστές ερωτημάτων, δρομολογητές, επαναταξινομητές και ενθέτες περιεχομένου για παραγωγικές ροές. | Μέγιστη ευελιξία, αλλά σημαντικά πιο πολύπλοκο. |

**Αυτό το σεμινάριο χρησιμοποιεί την Native προσέγγιση.** Κάθε βήμα της ροής RAG — ενσωμάτωση της ερώτησης, αναζήτηση στην αποθήκη διανυσμάτων, σύνθεση συμφραζόμενου και δημιουργία απάντησης — γράφεται ρητά στο [`RagService.java`](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/RagService.java). Αυτό είναι σκόπιμο: ως εκπαιδευτικό εργαλείο, είναι πιο σημαντικό να δείτε και να κατανοήσετε κάθε στάδιο παρά να έχει ο κώδικας τη μικρότερη δυνατή έκταση. Μόλις εξοικειωθείτε με το πώς συνδέονται τα μέρη, μπορείτε να περάσετε σε Easy RAG για γρήγορα πρωτότυπα ή σε Advanced RAG για παραγωγικά συστήματα.

> **💡 Έχετε δει ήδη Easy RAG σε δράση;** Το [Quick Start module](../00-quick-start/README.md) περιλαμβάνει παράδειγμα Document Q&A ([`SimpleReaderDemo.java`](../../../00-quick-start/src/main/java/com/example/langchain4j/quickstart/SimpleReaderDemo.java)) που χρησιμοποιεί την προσέγγιση Easy RAG — το LangChain4j χειρίζεται ενσωμάτωση, αναζήτηση και σύνθεση prompt αυτόματα. Αυτό το module κάνει το επόμενο βήμα ανοίγοντας αυτή τη ροή ώστε να βλέπετε και να ελέγχετε κάθε στάδιο μόνοι σας.

<img src="../../../translated_images/el/easy-rag-pipeline.2e1602e2ad2ded42.webp" alt="Ροή Easy RAG - LangChain4j" width="800"/>

*Αυτό το διάγραμμα δείχνει τη ροή Easy RAG από το `SimpleReaderDemo.java`. Συγκρίνετέ το με την Native προσέγγιση που χρησιμοποιείται σε αυτό το module: η Easy RAG κρύβει την ενσωμάτωση, ανάκτηση και σύνθεση prompt πίσω από `AiServices` και `ContentRetriever` — φορτώνετε ένα έγγραφο, επισυνάπτετε έναν retriever και λαμβάνετε απαντήσεις. Η Native προσέγγιση στο module αυτό ανοίγει την ροή ώστε να καλείτε εσείς καθένα από τα στάδια (ενσωμάτωση, αναζήτηση, σύνθεση συμφραζομένου, παραγωγή) δίνοντάς σας πλήρη ορατότητα και έλεγχο.*

## Πώς Λειτουργεί

Η ροή RAG σε αυτό το module διασπάται σε τέσσερα στάδια που εκτελούνται κατά σειρά κάθε φορά που ένας χρήστης κάνει μια ερώτηση. Πρώτα, το ανεβασμένο έγγραφο **αναλύεται και κομματιάζεται** σε διαχειρίσιμα τμήματα. Αυτά τα κομμάτια μετατρέπονται σε **ενσωματώσεις διανυσμάτων** και αποθηκεύονται ώστε να μπορούν να συγκριθούν μαθηματικά. Όταν φτάνει μια ερώτηση, το σύστημα εκτελεί **σημασιολογική αναζήτηση** για να βρει τα πιο σχετικά κομμάτια και τελικά τα περνά ως συμφραζόμενο στο LLM για **δημιουργία απάντησης**. Οι παρακάτω ενότητες περνούν κάθε στάδιο με τον πραγματικό κώδικα και διαγράμματα. Ας δούμε το πρώτο βήμα.

### Επεξεργασία Εγγράφων

[DocumentService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/DocumentService.java)

Όταν ανεβάζετε ένα έγγραφο, το σύστημα το αναλύει (PDF ή απλό κείμενο), επισυνάπτει μεταδεδομένα όπως το όνομα αρχείου και μετά το χωρίζει σε κομμάτια — μικρότερα τμήματα που χωράνε άνετα στο παράθυρο συμφραζομένων του μοντέλου. Αυτά τα κομμάτια επικαλύπτονται ελαφρώς ώστε να μην χάνεται το συμφραζόμενο στα όρια.

```java
// Αναλύστε το ανεβασμένο αρχείο και τυλίξτε το σε ένα Έγγραφο LangChain4j
Document document = Document.from(content, metadata);

// Διαχωρίστε σε τμήματα των 300 token με επικάλυψη 30 token
DocumentSplitter splitter = DocumentSplitters
    .recursive(300, 30);

List<TextSegment> segments = splitter.split(document);
```

Το διάγραμμα παρακάτω δείχνει πώς λειτουργεί αυτό οπτικά. Παρατηρήστε πώς κάθε κομμάτι μοιράζεται μερικά tokens με τους γείτονες — η επικάλυψη 30 tokens εξασφαλίζει ότι δεν χάνεται σημαντικό συμφραζόμενο ανάμεσα στα κομμάτια:

<img src="../../../translated_images/el/document-chunking.a5df1dd1383431ed.webp" alt="Κομματιασμός Εγγράφου" width="800"/>

*Αυτό το διάγραμμα δείχνει ένα έγγραφο να χωρίζεται σε κομμάτια των 300 tokens με 30 tokens επικάλυψη, διατηρώντας το συμφραζόμενο στα όρια των κομματιών.*

### Δημιουργία Ενσωματώσεων

[LangChainRagConfig.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/config/LangChainRagConfig.java)

Κάθε κομμάτι μετατρέπεται σε έναν αριθμητικό αναπαράστατη που ονομάζεται embedding — ουσιαστικά ένας μετατροπέας νοήματος σε αριθμούς. Το embedding μοντέλο δεν είναι "ευφυές" όπως ένα μοντέλο συνομιλίας· δεν μπορεί να ακολουθήσει οδηγίες, να λογικεύσει ή να απαντήσει σε ερωτήσεις. Αυτό που μπορεί να κάνει είναι να χαρτογραφήσει το κείμενο σε έναν μαθηματικό χώρο όπου παρόμοια νοήματα βρίσκονται κοντά το ένα στο άλλο — "αυτοκίνητο" κοντά σε "όχημα", "πολιτική επιστροφών" κοντά σε "επανάκτηση χρημάτων". Σκεφτείτε το μοντέλο συνομιλίας ως ένα πρόσωπο που μπορείτε να μιλήσετε· το embedding μοντέλο είναι ένα εξαιρετικό σύστημα αρχειοθέτησης.

<img src="../../../translated_images/el/embedding-model-concept.90760790c336a705.webp" alt="Έννοια Μοντέλου Ενσωμάτωσης" width="800"/>

*Αυτό το διάγραμμα δείχνει πώς ένα embedding μοντέλο μετατρέπει κείμενο σε αριθμητικά διανύσματα, τοποθετώντας παρόμοια νοήματα — όπως "αυτοκίνητο" και "όχημα" — κοντά το ένα στο άλλο στον χώρο διανυσμάτων.*

```java
@Bean
public EmbeddingModel embeddingModel() {
    return OpenAiOfficialEmbeddingModel.builder()
        .baseUrl(azureOpenAiEndpoint)
        .apiKey(azureOpenAiKey)
        .modelName(azureEmbeddingDeploymentName)
        .build();
}

EmbeddingStore<TextSegment> embeddingStore = 
    new InMemoryEmbeddingStore<>();
```

Το διάγραμμα κλάσεων παρακάτω δείχνει τις δύο ξεχωριστές ροές σε μια ροή RAG και τις κλάσεις του LangChain4j που τις υλοποιούν. Η **ροή εισαγωγής** (τρέχει μια φορά κατά τη μεταφόρτωση) χωρίζει το έγγραφο, ενσωματώνει τα κομμάτια και τα αποθηκεύει μέσω της `.addAll()`. Η **ροή ερωτήματος** (τρέχει κάθε φορά που κάνει ερώτηση ο χρήστης) ενσωματώνει την ερώτηση, ψάχνει στην αποθήκη μέσω `.search()`, και περνά το ταιριασμένο συμφραζόμενο στο μοντέλο συνομιλίας. Και οι δύο ροές συναντώνται στο κοινό interface `EmbeddingStore<TextSegment>`:

<img src="../../../translated_images/el/rag-langchain4j-classes.bbf3aa9077ab443d.webp" alt="Κλάσεις LangChain4j RAG" width="800"/>

*Αυτό το διάγραμμα δείχνει τις δύο ροές σε μια ροή RAG — εισαγωγή και ερώτημα — και πώς συνδέονται μέσω ενός κοινόχρηστου EmbeddingStore.*

Μόλις οι ενσωματώσεις αποθηκευτούν, παρόμοιο περιεχόμενο φυσικά συγκεντρώνεται στον χώρο διανυσμάτων. Η παρακάτω οπτικοποίηση δείχνει πώς έγγραφα σχετικών θεμάτων εμφανίζονται ως κοντινά σημεία, που καθιστά δυνατή τη σημασιολογική αναζήτηση:

<img src="../../../translated_images/el/vector-embeddings.2ef7bdddac79a327.webp" alt="Χώρος Ενσωματώσεων Διανυσμάτων" width="800"/>

*Αυτή η οπτικοποίηση δείχνει πώς σχετικά έγγραφα ομαδοποιούνται στον τρισδιάστατο χώρο διανυσμάτων, με θεματικές όπως Τεχνική Τεκμηρίωση, Επιχειρησιακοί Κανόνες και Συχνές Ερωτήσεις να σχηματίζουν διακριτές ομάδες.*

Όταν ένας χρήστης ψάχνει, το σύστημα ακολουθεί τέσσερα βήματα: ενσωματώνει τα έγγραφα μία φορά, ενσωματώνει την ερώτηση σε κάθε αναζήτηση, συγκρίνει το διάνυσμα ερωτήματος με όλα τα αποθηκευμένα διανύσματα χρησιμοποιώντας το συνημίτονο της ομοιότητας (cosine similarity) και επιστρέφει τα κορυφαία K κομμάτια με την υψηλότερη βαθμολογία. Το διάγραμμα παρακάτω περνά το κάθε βήμα και τις κλάσεις LangChain4j που εμπλέκονται:

<img src="../../../translated_images/el/embedding-search-steps.f54c907b3c5b4332.webp" alt="Βήματα Αναζήτησης Ενσωμάτωσης" width="800"/>

*Αυτό το διάγραμμα δείχνει τη διαδικασία αναζήτησης ενσωμάτωσης τεσσάρων βημάτων: ενσωμάτωση εγγράφων, ενσωμάτωση ερωτήματος, σύγκριση διανυσμάτων με το cosine similarity και επιστροφή των κορυφαίων αποτελεσμάτων.*

### Σημασιολογική Αναζήτηση

[RagService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/RagService.java)

Όταν κάνετε μια ερώτηση, και η ερώτησή σας μετατρέπεται σε en embedding. Το σύστημα συγκρίνει την ενσωμάτωση της ερώτησής σας με όλα τα embeddings των κομματιών εγγράφων. Βρίσκει τα κομμάτια με τα πιο παρόμοια νοήματα — όχι μόνο με βάση τις λέξεις-κλειδιά αλλά με βάση την πραγματική σημασιολογική ομοιότητα.

```java
Embedding queryEmbedding = embeddingModel.embed(question).content();

EmbeddingSearchRequest searchRequest = EmbeddingSearchRequest.builder()
    .queryEmbedding(queryEmbedding)
    .maxResults(5)
    .minScore(0.5)
    .build();

EmbeddingSearchResult<TextSegment> searchResult = embeddingStore.search(searchRequest);
List<EmbeddingMatch<TextSegment>> matches = searchResult.matches();

for (EmbeddingMatch<TextSegment> match : matches) {
    String relevantText = match.embedded().text();
    double score = match.score();
}
```

Το διάγραμμα παρακάτω αντιπαραβάλλει τη σημασιολογική αναζήτηση με την παραδοσιακή αναζήτηση λέξεων-κλειδιών. Μια αναζήτηση λέξης-κλειδί για "όχημα" αγνοεί ένα κομμάτι για "αυτοκίνητα και φορτηγά", ενώ η σημασιολογική αναζήτηση καταλαβαίνει ότι σημαίνουν το ίδιο και το επιστρέφει ως υψηλά βαθμολογημένο αποτέλεσμα:

<img src="../../../translated_images/el/semantic-search.6b790f21c86b849d.webp" alt="Σημασιολογική Αναζήτηση" width="800"/>

*Αυτό το διάγραμμα συγκρίνει αναζήτηση με λέξεις-κλειδιά και σημασιολογική αναζήτηση, δείχνοντας πώς η σημασιολογική βρίσκει περιεχόμενο που συνδέεται εννοιολογικά ακόμα και όταν οι ακριβείς λέξεις-κλειδιά διαφέρουν.*

Στο παρασκήνιο, η ομοιότητα μετριέται χρησιμοποιώντας το συνημίτονο της γωνίας (cosine similarity) — που ουσιαστικά ρωτάει "δείχνουν αυτά τα δύο βέλη προς την ίδια κατεύθυνση;". Δύο κομμάτια μπορεί να χρησιμοποιούν εντελώς διαφορετικές λέξεις, αλλά αν έχουν το ίδιο νόημα τα διανύσματά τους δείχνουν προς την ίδια κατεύθυνση και η βαθμολογία τους είναι κοντά στο 1.0:

<img src="../../../translated_images/el/cosine-similarity.9baeaf3fc3336abb.webp" alt="Συνημίτονο Ομοιότητας" width="800"/>
*Αυτό το διάγραμμα απεικονίζει την ομοιότητα cosine ως τη γωνία μεταξύ διανυσμάτων ενσωμάτωσης — διανύσματα που ευθυγραμμίζονται περισσότερο έχουν βαθμολογία πιο κοντά στο 1.0, υποδεικνύοντας υψηλότερη σημασιολογική ομοιότητα.*

### Δημιουργία Απάντησης

[RagService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/RagService.java)

Τα πιο σχετικά κομμάτια συναρμολογούνται σε ένα δομημένο prompt που περιλαμβάνει σαφείς οδηγίες, το ανακτηθέν πλαίσιο, και την ερώτηση του χρήστη. Το μοντέλο διαβάζει αυτά τα συγκεκριμένα κομμάτια και απαντά βάσει αυτών — μπορεί να χρησιμοποιήσει μόνο όσα βρίσκονται μπροστά του, αποτρέποντας την παραισθησιογένεση.

```java
String context = matches.stream()
    .map(match -> match.embedded().text())
    .collect(Collectors.joining("\n\n"));

String prompt = String.format("""
    Answer the question based on the following context.
    If the answer cannot be found in the context, say so.

    Context:
    %s

    Question: %s

    Answer:""", context, request.question());

String answer = chatModel.chat(prompt);
```

Το παρακάτω διάγραμμα δείχνει αυτή τη συναρμολόγηση σε δράση — τα κορυφαία κομμάτια από το βήμα αναζήτησης εισάγονται στο πρότυπο προτροπής, και το `OpenAiOfficialChatModel` δημιουργεί μια τεκμηριωμένη απάντηση:

<img src="../../../translated_images/el/context-assembly.7e6dd60c31f95978.webp" alt="Συναρμολόγηση Πλαισίου" width="800"/>

*Αυτό το διάγραμμα δείχνει πώς τα κορυφαία κομμάτια συναρμολογούνται σε ένα δομημένο prompt, επιτρέποντας στο μοντέλο να δημιουργήσει μια τεκμηριωμένη απάντηση από τα δεδομένα σας.*

## Βασικές Έννοιες

### Στρατηγική Διαχωρισμού σε Κομμάτια

Τα έγγραφα χωρίζονται σε κομμάτια των 300 tokens με 30 tokens επικάλυψη. Αυτή η ισορροπία διασφαλίζει ότι κάθε κομμάτι έχει αρκετό πλαίσιο για να είναι ουσιαστικό, ενώ παραμένει αρκετά μικρό για να συμπεριληφθούν πολλαπλά κομμάτια σε ένα prompt.

### Βαθμολογίες Ομοιότητας

Κάθε ανακτηθέν κομμάτι συνοδεύεται από βαθμολογία ομοιότητας μεταξύ 0 και 1 που δείχνει πόσο στενά σχετίζεται με την ερώτηση του χρήστη. Το παρακάτω διάγραμμα οπτικοποιεί τα εύρη βαθμολογίας και πώς το σύστημα τα χρησιμοποιεί για να φιλτράρει τα αποτελέσματα:

<img src="../../../translated_images/el/similarity-scores.b0716aa911abf7f0.webp" alt="Βαθμολογίες Ομοιότητας" width="800"/>

*Αυτό το διάγραμμα δείχνει τα εύρη βαθμολογιών από 0 έως 1, με ελάχιστο όριο 0.5 που φιλτράρει τα μη σχετικά κομμάτια.*

Οι βαθμολογίες κυμαίνονται από 0 έως 1:
- 0.7-1.0: Εξαιρετικά σχετικό, ακριβής αντιστοιχία
- 0.5-0.7: Σχετικό, καλό πλαίσιο
- Κάτω από 0.5: Φιλτράρεται, πολύ διαφορετικό

Το σύστημα ανακτά μόνο κομμάτια πάνω από το ελάχιστο όριο για να διασφαλίσει ποιότητα.

Οι ενσωματώσεις λειτουργούν καλά όταν οι έννοιες ομαδοποιούνται καθαρά, αλλά έχουν τυφλά σημεία. Το παρακάτω διάγραμμα δείχνει τους κοινούς τρόπους αποτυχίας — κομμάτια πολύ μεγάλα παράγουν θολά διανύσματα, κομμάτια πολύ μικρά στερούνται πλαισίου, ασαφείς όροι δείχνουν σε πολλαπλές ομάδες, και ακριβείς αναζητήσεις (IDs, αριθμοί μερών) δεν λειτουργούν καθόλου με ενσωματώσεις:

<img src="../../../translated_images/el/embedding-failure-modes.b2bcb901d8970fc0.webp" alt="Τρόποι Αποτυχίας Ενσωματώσεων" width="800"/>

*Αυτό το διάγραμμα δείχνει κοινές αποτυχίες ενσωματώσεων: κομμάτια πολύ μεγάλα, πολύ μικρά, ασαφείς όροι που δείχνουν σε πολλαπλές ομάδες, και ακριβείς αναζητήσεις όπως IDs.*

### Αποθήκευση στη Μνήμη

Αυτή η μονάδα χρησιμοποιεί αποθήκευση στη μνήμη για απλότητα. Όταν επανεκκινήσετε την εφαρμογή, τα ανεβασμένα έγγραφα χάνονται. Σε συστήματα παραγωγής χρησιμοποιούνται επίμονοι διανυσματικοί αποθηκευτικοί χώροι όπως το Qdrant ή το Azure AI Search.

### Διαχείριση Πλαισίου Κειμένου

Κάθε μοντέλο έχει μέγιστο πλαίσιο κειμένου. Δεν μπορείτε να συμπεριλάβετε κάθε κομμάτι από ένα μεγάλο έγγραφο. Το σύστημα ανακτά τα κορυφαία N πιο σχετικά κομμάτια (προεπιλογή 5) για να παραμείνει εντός ορίων και να παρέχει αρκετό πλαίσιο για ακριβείς απαντήσεις.

## Πότε είναι Σημαντικό το RAG

Το RAG δεν είναι πάντα η σωστή προσέγγιση. Ο οδηγός απόφασης παρακάτω βοηθά να καθορίσετε πότε το RAG προσθέτει αξία και πότε πιο απλές προσεγγίσεις — όπως η συμπερίληψη περιεχομένου απευθείας στο prompt ή η χρήση της ενσωματωμένης γνώσης του μοντέλου — είναι αρκετές:

<img src="../../../translated_images/el/when-to-use-rag.1016223f6fea26bc.webp" alt="Πότε να Χρησιμοποιήσετε RAG" width="800"/>

*Αυτό το διάγραμμα δείχνει έναν οδηγό απόφασης για το πότε το RAG προσθέτει αξία έναντι του πότε οι πιο απλές προσεγγίσεις είναι αρκετές.*

**Χρησιμοποιήστε το RAG όταν:**
- Απαντάτε σε ερωτήσεις για ιδιωτικά έγγραφα
- Η πληροφορία αλλάζει συχνά (πολιτικές, τιμές, προδιαγραφές)
- Ακρίβεια απαιτεί απόδοση πηγής
- Το περιεχόμενο είναι πολύ μεγάλο για ένα prompt
- Χρειάζεστε επαληθεύσιμες, τεκμηριωμένες απαντήσεις

**Μην χρησιμοποιείτε το RAG όταν:**
- Οι ερωτήσεις απαιτούν γενικές γνώσεις που έχει ήδη το μοντέλο
- Χρειάζονται δεδομένα σε πραγματικό χρόνο (το RAG δουλεύει με ανεβασμένα έγγραφα)
- Το περιεχόμενο είναι αρκετά μικρό για να συμπεριληφθεί απευθείας στα prompts
