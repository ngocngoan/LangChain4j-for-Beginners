## Τι Θα Μάθετε

Στα προηγούμενα modules, μάθατε πώς να έχετε συνομιλίες με την τεχνητή νοημοσύνη και να διαμορφώνετε αποτελεσματικά τις ερωτήσεις σας. Αλλά υπάρχει ένας θεμελιώδης περιορισμός: τα γλωσσικά μοντέλα γνωρίζουν μόνο όσα έμαθαν κατά την εκπαίδευσή τους. Δεν μπορούν να απαντήσουν σε ερωτήσεις σχετικά με τις πολιτικές της εταιρείας σας, την τεκμηρίωση του έργου σας ή οποιαδήποτε πληροφορία δεν έχουν εκπαιδευτεί.

Το RAG (Γεννήτρια με Ενίσχυση Ανάκτησης) λύνει αυτό το πρόβλημα. Αντί να προσπαθείτε να διδάξετε στο μοντέλο τις πληροφορίες σας (κάτι που είναι δαπανηρό και μη πρακτικό), του δίνετε τη δυνατότητα να αναζητά μέσα στα έγγραφά σας. Όταν κάποιος θέτει μια ερώτηση, το σύστημα βρίσκει σχετικές πληροφορίες και τις συμπεριλαμβάνει στην προτροπή. Το μοντέλο απαντά βάσει αυτού του ανακτημένου πλαισίου.

Σκεφτείτε το RAG ως ένα σύστημα που παρέχει στο μοντέλο μια βιβλιοθήκη αναφορών. Όταν κάνετε μια ερώτηση, το σύστημα:

1. **Ερώτηση Χρήστη** - Κάνετε μια ερώτηση  
2. **Ενσωμάτωση** - Μετατρέπει την ερώτηση σε διάνυσμα  
3. **Αναζήτηση Διανύσματος** - Βρίσκει παρόμοια τμήματα εγγράφων  
4. **Σύνθεση Πλαισίου** - Προσθέτει σχετικά τμήματα στην προτροπή  
5. **Απάντηση** - Το LLM παράγει απάντηση βασισμένη στο πλαίσιο  

Αυτό καθιστά τις απαντήσεις του μοντέλου εδραιωμένες στα πραγματικά σας δεδομένα αντί να βασίζονται μόνο στη γνώση της εκπαίδευσής του ή να επινοούν απαντήσεις.

### Επεξεργασία Εγγράφων

[DocumentService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/DocumentService.java)

Όταν ανεβάζετε ένα έγγραφο, το σύστημα το αναλύει (PDF ή απλό κείμενο), προσθέτει μεταδεδομένα όπως το όνομα αρχείου και στη συνέχεια το χωρίζει σε τμήματα — μικρότερα κομμάτια που χωρούν άνετα στο πλαίσιο του μοντέλου. Αυτά τα τμήματα επικαλύπτονται λίγο ώστε να μη χάνεται το πλαίσιο στα όρια.

```java
// Ανάλυση του ανεβασμένου αρχείου και το τύλιγμα σε ένα έγγραφο LangChain4j
Document document = Document.from(content, metadata);

// Διαχωρισμός σε κομμάτια των 300 χαρακτήρων με επικάλυψη 30 χαρακτήρων
DocumentSplitter splitter = DocumentSplitters
    .recursive(300, 30);

List<TextSegment> segments = splitter.split(document);
```
  
Το παρακάτω διάγραμμα δείχνει οπτικά πώς λειτουργεί αυτό. Παρατηρήστε πως κάθε τμήμα μοιράζεται κάποιες λέξεις-κλειδιά με τους γείτονές του — η επικάλυψη 30 λέξεων εξασφαλίζει να μην πέφτει κάποιο σημαντικό πλαίσιο μεταξύ:

### Δημιουργία Ενσωματώσεων

[LangChainRagConfig.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/config/LangChainRagConfig.java)

Κάθε τμήμα μετατρέπεται σε αριθμητική αναπαράσταση που ονομάζεται ενσωμάτωση - ουσιαστικά ένα μαθηματικό αποτύπωμα που αποτυπώνει το νόημα του κειμένου. Παρόμοιο κείμενο παράγει παρόμοιες ενσωματώσεις.

```java
@Bean
public EmbeddingModel embeddingModel() {
    return OpenAiOfficialEmbeddingModel.builder()
        .baseUrl(azureOpenAiEndpoint)
        .apiKey(azureOpenAiKey)
        .modelName(azureEmbeddingDeploymentName)
        .build();
}

EmbeddingStore<TextSegment> embeddingStore = 
    new InMemoryEmbeddingStore<>();
```
  
Το διάγραμμα κλάσεων δείχνει πώς συνδέονται αυτά τα στοιχεία του LangChain4j. Το `OpenAiOfficialEmbeddingModel` μετατρέπει το κείμενο σε διανύσματα, το `InMemoryEmbeddingStore` κρατά τα διανύσματα μαζί με τα αρχικά δεδομένα `TextSegment` και το `EmbeddingSearchRequest` ελέγχει παραμέτρους ανάκτησης όπως `maxResults` και `minScore`:

<img src="../../../translated_images/el/rag-langchain4j-classes.bbf3aa9077ab443d.webp" alt="Κλάσεις LangChain4j RAG" width="800"/>

Μόλις αποθηκευτούν οι ενσωματώσεις, παρόμοιο περιεχόμενο δημιουργεί φυσικά συστάδες στον χώρο διανυσμάτων. Η παρακάτω απεικόνιση δείχνει πώς έγγραφα σχετικά με συναφή θέματα καταλήγουν ως κοντινά σημεία, πράγμα που καθιστά δυνατή τη σημασιολογική αναζήτηση:

### Σημασιολογική Αναζήτηση

[RagService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/RagService.java)

Όταν κάνετε μια ερώτηση, και αυτή μετατρέπεται σε ενσωμάτωση. Το σύστημα συγκρίνει την ενσωμάτωση της ερώτησής σας με όλες τις ενσωματώσεις των τμημάτων εγγράφων. Βρίσκει τα τμήματα με τα πιο παρόμοια νοήματα — όχι μόνο με λέξεις-κλειδιά, αλλά πραγματική σημασιολογική ομοιότητα.

```java
Embedding queryEmbedding = embeddingModel.embed(question).content();

EmbeddingSearchRequest searchRequest = EmbeddingSearchRequest.builder()
    .queryEmbedding(queryEmbedding)
    .maxResults(5)
    .minScore(0.5)
    .build();

EmbeddingSearchResult<TextSegment> searchResult = embeddingStore.search(searchRequest);
List<EmbeddingMatch<TextSegment>> matches = searchResult.matches();

for (EmbeddingMatch<TextSegment> match : matches) {
    String relevantText = match.embedded().text();
    double score = match.score();
}
```
  
Το παρακάτω διάγραμμα συγκρίνει τη σημασιολογική αναζήτηση με την παραδοσιακή αναζήτηση λέξεων-κλειδιών. Μια αναζήτηση λέξης-κλειδί για "όχημα" χάνει ένα τμήμα για "αυτοκίνητα και φορτηγά", αλλά η σημασιολογική αναζήτηση καταλαβαίνει ότι σημαίνουν το ίδιο και το επιστρέφει ως υψηλής βαθμολογίας αποτέλεσμα:

<img src="../../../translated_images/el/semantic-search.6b790f21c86b849d.webp" alt="Σημασιολογική Αναζήτηση" width="800"/>

### Παραγωγή Απαντήσεων

[RagService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/RagService.java)

Τα πιο σχετικά τμήματα συντίθενται σε μια δομημένη προτροπή που περιλαμβάνει σαφείς οδηγίες, το ανακτηθέν πλαίσιο και την ερώτηση του χρήστη. Το μοντέλο διαβάζει αυτά τα συγκεκριμένα τμήματα και απαντά βάσει αυτών — μπορεί να χρησιμοποιήσει μόνο όσα είναι μπροστά του, αποτρέποντας την επινόηση.

```java
String context = matches.stream()
    .map(match -> match.embedded().text())
    .collect(Collectors.joining("\n\n"));

String prompt = String.format("""
    Answer the question based on the following context.
    If the answer cannot be found in the context, say so.

    Context:
    %s

    Question: %s

    Answer:""", context, request.question());

String answer = chatModel.chat(prompt);
```
  
Το ακόλουθο διάγραμμα δείχνει αυτή τη σύνθεση σε δράση — τα τμήματα με την υψηλότερη βαθμολογία από το βήμα αναζήτησης εγχύονται στο πρότυπο προτροπής, και το `OpenAiOfficialChatModel` παράγει μια εδραιωμένη απάντηση:

Ξεκινήστε ανεβάζοντας ένα έγγραφο - τα αρχεία TXT λειτουργούν καλύτερα για δοκιμές. Παρέχεται ένα `sample-document.txt` σε αυτόν τον φάκελο που περιέχει πληροφορίες για τα χαρακτηριστικά του LangChain4j, την υλοποίηση RAG, και βέλτιστες πρακτικές - ιδανικό για δοκιμή του συστήματος.

Το σύστημα επεξεργάζεται το έγγραφό σας, το χωρίζει σε τμήματα και δημιουργεί ενσωματώσεις για κάθε τμήμα. Αυτό γίνεται αυτόματα με το ανέβασμα.

Τώρα κάντε συγκεκριμένες ερωτήσεις σχετικά με το περιεχόμενο του εγγράφου. Δοκιμάστε κάτι πραγματικό που αναφέρεται σαφώς στο έγγραφο. Το σύστημα ψάχνει για σχετικά τμήματα, τα ενσωματώνει στην προτροπή και παράγει μια απάντηση.

Οι υψηλότερες βαθμολογίες σημαίνουν καλύτερες αντιστοιχίες. Αυτό σας επιτρέπει να επαληθεύσετε την απάντηση σε σχέση με το πρωτότυπο υλικό.

Παρακολουθήστε πώς αλλάζουν οι βαθμολογίες σχετικότητας ανάλογα με το πόσο καλά ταιριάζει η ερώτησή σας με το περιεχόμενο του εγγράφου.

## Κύριες Έννοιες

### Στρατηγική Τμηματοποίησης

Τα έγγραφα χωρίζονται σε τμήματα των 300 λέξεων με επικάλυψη 30 λέξεων. Αυτή η ισορροπία εξασφαλίζει ότι κάθε τμήμα έχει αρκετό πλαίσιο για να είναι ουσιαστικό, ενώ παραμένει αρκετά μικρό ώστε να χωρούν πολλά τμήματα σε μια προτροπή.

### Βαθμολογίες Ομοιότητας

Κάθε ανακτηθέν τμήμα συνοδεύεται από μια βαθμολογία ομοιότητας μεταξύ 0 και 1 που δείχνει πόσο στενά ταιριάζει με την ερώτηση του χρήστη. Το παρακάτω διάγραμμα απεικονίζει τα εύρη βαθμολογίας και πώς το σύστημα τα χρησιμοποιεί για να φιλτράρει τα αποτελέσματα:

<img src="../../../translated_images/el/similarity-scores.b0716aa911abf7f0.webp" alt="Βαθμολογίες Ομοιότητας" width="800"/>

Οι βαθμολογίες κυμαίνονται από 0 έως 1:  
- 0.7-1.0: Εξαιρετικά σχετικό, ακριβής αντιστοιχία  
- 0.5-0.7: Σχετικό, καλό πλαίσιο  
- Κάτω από 0.5: Φιλτράρεται, πολύ διαφορετικό  

Το σύστημα ανακτά μόνο τμήματα πάνω από το ελάχιστο όριο για να εξασφαλίσει ποιότητα.

### Αποθήκευση στη Μνήμη

Αυτό το module χρησιμοποιεί αποθήκευση στη μνήμη για απλότητα. Όταν επανεκκινήσετε την εφαρμογή, τα ανεβασμένα έγγραφα χάνονται. Τα συστήματα παραγωγής χρησιμοποιούν επίμονες βάσεις δεδομένων διανυσμάτων όπως το Qdrant ή το Azure AI Search.

### Διαχείριση Παραθύρου Πλαισίου

Κάθε μοντέλο έχει ένα μέγιστο παράθυρο πλαισίου. Δεν μπορείτε να συμπεριλάβετε κάθε τμήμα από ένα μεγάλο έγγραφο. Το σύστημα ανακτά τα κορυφαία Ν πιο σχετικά τμήματα (προεπιλογή 5) ώστε να παραμένει εντός ορίων ενώ παρέχει αρκετό πλαίσιο για ακριβείς απαντήσεις.

## Πότε έχει σημασία το RAG

Το RAG δεν είναι πάντα η σωστή προσέγγιση. Ο παρακάτω οδηγός απόφασης σας βοηθά να καθορίσετε πότε το RAG προσθέτει αξία έναντι απλούστερων προσεγγίσεων — όπως η άμεση συμπερίληψη περιεχομένου στην προτροπή ή η χρήση της ενσωματωμένης γνώσης του μοντέλου:

<img src="../../../translated_images/el/when-to-use-rag.1016223f6fea26bc.webp" alt="Πότε να Χρησιμοποιήσετε το RAG" width="800"/>

**Χρησιμοποιήστε το RAG όταν:**
- Απάντηση σε ερωτήσεις σχετικά με ιδιόκτητα έγγραφα  
- Οι πληροφορίες αλλάζουν συχνά (πολιτικές, τιμές, προδιαγραφές)  
- Η ακρίβεια απαιτεί αναφορά πηγής  
- Το περιεχόμενο είναι πολύ μεγάλο για να χωρέσει σε ένα μόνο prompt  
- Χρειάζεστε επαληθεύσιμες, τεκμηριωμένες απαντήσεις  

**Μην χρησιμοποιείτε RAG όταν:**  
- Οι ερωτήσεις απαιτούν γενικές γνώσεις που έχει ήδη το μοντέλο  
- Απαιτούνται δεδομένα σε πραγματικό χρόνο (το RAG λειτουργεί σε ανεβασμένα έγγραφα)  
- Το περιεχόμενο είναι αρκετά μικρό για να συμπεριληφθεί απευθείας στα prompts
