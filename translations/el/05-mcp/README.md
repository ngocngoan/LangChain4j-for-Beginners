## Τι θα μάθετε

Έχετε δημιουργήσει συνομιλητική ΤΝ, έχετε κυριαρχήσει στα prompts, έχετε βασίσει τις απαντήσεις σε έγγραφα και έχετε δημιουργήσει agents με εργαλεία. Αλλά όλα αυτά τα εργαλεία ήταν προσαρμοσμένα για την ειδική εφαρμογή σας. Τι θα γινόταν αν μπορούσατε να δώσετε στην ΤΝ σας πρόσβαση σε ένα τυποποιημένο οικοσύστημα εργαλείων που ο καθένας μπορεί να δημιουργήσει και να μοιραστεί; Σε αυτό το module, θα μάθετε πώς να το κάνετε ακριβώς αυτό με το Πρωτόκολλο Πλαισίου Μοντέλου (MCP) και το agentic module του LangChain4j. Πρώτα παρουσιάζουμε έναν απλό αναγνώστη αρχείων MCP και στη συνέχεια δείχνουμε πώς ενσωματώνεται εύκολα σε προηγμένα agentic workflows χρησιμοποιώντας το μοτίβο Supervisor Agent.

## Τι είναι το MCP;

Το Πρωτόκολλο Πλαισίου Μοντέλου (MCP) παρέχει ακριβώς αυτό - έναν τυποποιημένο τρόπο για εφαρμογές ΤΝ να ανακαλύπτουν και να χρησιμοποιούν εξωτερικά εργαλεία. Αντί να γράφετε προσαρμοσμένες ενσωματώσεις για κάθε πηγή δεδομένων ή υπηρεσία, συνδέεστε σε MCP διακομιστές που εκθέτουν τις δυνατότητές τους με συνεπή μορφή. Ο AI agent σας μπορεί τότε να ανακαλύψει και να χρησιμοποιήσει αυτά τα εργαλεία αυτόματα.

Το παρακάτω διάγραμμα δείχνει τη διαφορά — χωρίς MCP, κάθε ενσωμάτωση απαιτεί προσαρμοσμένη σύνδεση σημείου προς σημείο· με το MCP, ένα μόνο πρωτόκολλο συνδέει την εφαρμογή σας με οποιοδήποτε εργαλείο:

<img src="../../../translated_images/el/mcp-comparison.9129a881ecf10ff5.webp" alt="MCP Comparison" width="800"/>

*Πριν το MCP: Πολύπλοκες ενσωματώσεις σημείου προς σημείο. Μετά το MCP: Ένα πρωτόκολλο, ατελείωτες δυνατότητες.*

Το MCP λύνει ένα θεμελιώδες πρόβλημα στην ανάπτυξη ΤΝ: κάθε ενσωμάτωση είναι προσαρμοσμένη. Θέλετε πρόσβαση στο GitHub; Προσαρμοσμένος κώδικας. Θέλετε να διαβάσετε αρχεία; Προσαρμοσμένος κώδικας. Θέλετε να κάνετε ερώτημα σε βάση δεδομένων; Προσαρμοσμένος κώδικας. Και καμία από αυτές τις ενσωματώσεις δεν λειτουργεί με άλλες εφαρμογές ΤΝ.

Το MCP το τυποποιεί αυτό. Ένας MCP διακομιστής εκθέτει εργαλεία με σαφείς περιγραφές και σχήματα. Οποιοσδήποτε MCP πελάτης μπορεί να συνδεθεί, να ανακαλύψει διαθέσιμα εργαλεία και να τα χρησιμοποιήσει. Φτιάχνετε μία φορά, χρησιμοποιείτε παντού.

Το παρακάτω διάγραμμα απεικονίζει αυτή την αρχιτεκτονική — ένας μόνο MCP πελάτης (η εφαρμογή ΤΝ σας) συνδέεται με πολλούς MCP διακομιστές, καθένας εκθέτοντας το δικό του σύνολο εργαλείων μέσω του τυποποιημένου πρωτοκόλλου:

<img src="../../../translated_images/el/mcp-architecture.b3156d787a4ceac9.webp" alt="MCP Architecture" width="800"/>

*Αρχιτεκτονική Πρωτοκόλλου Πλαισίου Μοντέλου - τυποποιημένη ανακάλυψη και εκτέλεση εργαλείων*

## Πώς λειτουργεί το MCP

Κάτω από την επιφάνεια, το MCP χρησιμοποιεί μια πολυεπίπεδη αρχιτεκτονική. Η εφαρμογή Java σας (ο MCP πελάτης) ανακαλύπτει διαθέσιμα εργαλεία, στέλνει αιτήματα JSON-RPC μέσω ενός επιπέδου μεταφοράς (Stdio ή HTTP), και ο MCP διακομιστής εκτελεί τις ενέργειες και επιστρέφει αποτελέσματα. Το ακόλουθο διάγραμμα διαχωρίζει κάθε επίπεδο αυτού του πρωτοκόλλου:

<img src="../../../translated_images/el/mcp-protocol-detail.01204e056f45308b.webp" alt="MCP Protocol Detail" width="800"/>

*Πώς λειτουργεί το MCP κάτω από την επιφάνεια — οι πελάτες ανακαλύπτουν εργαλεία, ανταλλάσσουν μηνύματα JSON-RPC και εκτελούν ενέργειες μέσω ενός επιπέδου μεταφοράς.*

**Αρχιτεκτονική Server-Client**

Το MCP χρησιμοποιεί το μοντέλο πελάτη-διακομιστή. Οι διακομιστές παρέχουν εργαλεία - ανάγνωση αρχείων, ερωτήσεις βάσεων δεδομένων, κλήσεις API. Οι πελάτες (η εφαρμογή ΤΝ σας) συνδέονται στους διακομιστές και χρησιμοποιούν τα εργαλεία τους.

Για να χρησιμοποιήσετε το MCP με το LangChain4j, προσθέστε αυτή την εξάρτηση Maven:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```

**Ανακάλυψη Εργαλείων**

Όταν ο πελάτης σας συνδέεται σε έναν MCP διακομιστή, ρωτάει "Τι εργαλεία έχετε;" Ο διακομιστής απαντά με μια λίστα διαθέσιμων εργαλείων, κάθε ένα με περιγραφές και σχήματα παραμέτρων. Ο AI agent σας μπορεί τότε να αποφασίσει ποια εργαλεία θα χρησιμοποιήσει βασιζόμενος σε αιτήματα χρηστών. Το παρακάτω διάγραμμα δείχνει αυτό το χέρι-χέρι — ο πελάτης στέλνει ένα αίτημα `tools/list` και ο διακομιστής επιστρέφει τα διαθέσιμα εργαλεία με περιγραφές και σχήματα παραμέτρων:

<img src="../../../translated_images/el/tool-discovery.07760a8a301a7832.webp" alt="MCP Tool Discovery" width="800"/>

*Η ΤΝ ανακαλύπτει διαθέσιμα εργαλεία κατά την εκκίνηση — τώρα γνωρίζει τι δυνατότητες υπάρχουν και μπορεί να αποφασίσει ποια θα χρησιμοποιήσει.*

**Μηχανισμοί μεταφοράς**

Το MCP υποστηρίζει διάφορους μηχανισμούς μεταφοράς. Οι δύο επιλογές είναι το Stdio (για τοπική επικοινωνία με υποδιαδικασίες) και το Streamable HTTP (για απομακρυσμένους διακομιστές). Αυτό το module παρουσιάζει τη μεταφορά Stdio:

<img src="../../../translated_images/el/transport-mechanisms.2791ba7ee93cf020.webp" alt="Transport Mechanisms" width="800"/>

*Μηχανισμοί μεταφοράς MCP: HTTP για απομακρυσμένους διακομιστές, Stdio για τοπικές διεργασίες*

**Stdio** - [StdioTransportDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java)

Για τοπικές διεργασίες. Η εφαρμογή σας δημιουργεί έναν διακομιστή ως υποδιαδικασία και επικοινωνεί μέσω τυπικής εισόδου/εξόδου. Χρήσιμο για πρόσβαση στο σύστημα αρχείων ή εργαλεία γραμμής εντολών.

```java
McpTransport stdioTransport = new StdioMcpTransport.Builder()
    .command(List.of(
        npmCmd, "exec",
        "@modelcontextprotocol/server-filesystem@2025.12.18",
        resourcesDir
    ))
    .logEvents(false)
    .build();
```

Ο διακομιστής `@modelcontextprotocol/server-filesystem` εκθέτει τα ακόλουθα εργαλεία, όλα περιορισμένα στους φακέλους που καθορίζετε:

| Εργαλείο | Περιγραφή |
|------|-------------|
| `read_file` | Ανάγνωση του περιεχομένου ενός μεμονωμένου αρχείου |
| `read_multiple_files` | Ανάγνωση πολλών αρχείων με μία κλήση |
| `write_file` | Δημιουργία ή υπερχείλιση ενός αρχείου |
| `edit_file` | Στοχευμένες διορθώσεις με εύρεση και αντικατάσταση |
| `list_directory` | Ανάκτηση λίστας αρχείων και φακέλων σε μια διαδρομή |
| `search_files` | Αναδρομική αναζήτηση αρχείων που ταιριάζουν με μοτίβο |
| `get_file_info` | Λήψη μεταδεδομένων αρχείου (μέγεθος, χρονικές σφραγίδες, δικαιώματα) |
| `create_directory` | Δημιουργία φακέλου (συμπεριλαμβανομένων των γονικών φακέλων) |
| `move_file` | Μετακίνηση ή μετονομασία αρχείου ή φακέλου |

Το παρακάτω διάγραμμα δείχνει πώς λειτουργεί η μεταφορά Stdio κατά την εκτέλεση — η Java εφαρμογή σας δημιουργεί τον MCP διακομιστή ως υποδιαδικασία και επικοινωνούν μέσω σωλήνων stdin/stdout, χωρίς να εμπλέκεται δίκτυο ή HTTP:

*Μεταφορά Stdio σε δράση — η εφαρμογή σας δημιουργεί τον MCP διακομιστή ως υποδιαδικασία και επικοινωνεί μέσω των σωλήνων stdin/stdout.*

## Το Agentic Module

Ενώ το MCP παρέχει τυποποιημένα εργαλεία, το **agentic module** του LangChain4j παρέχει έναν δηλωτικό τρόπο δημιουργίας agents που οργανώνουν αυτά τα εργαλεία. Η σημείωση `@Agent` και οι `AgenticServices` σας επιτρέπουν να ορίσετε τη συμπεριφορά του agent μέσω διεπαφών αντί για αυστηρό κώδικα.

Σε αυτό το module, θα εξερευνήσετε το μοτίβο **Supervisor Agent** — μια προηγμένη προσέγγιση agentic AI όπου ένας "supervisor" agent αποφασίζει δυναμικά ποιοι υπο-agent θα ενεργοποιηθούν ανάλογα με τα αιτήματα χρηστών. Θα συνδυάσουμε αυτές τις δύο έννοιες δίνοντας σε έναν από τους υπο-agent μας τις δυνατότητες πρόσβασης σε αρχεία μέσω MCP.

Για να χρησιμοποιήσετε το agentic module, προσθέστε αυτή την εξάρτηση Maven:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```
> **Σημείωση:** Το `langchain4j-agentic` module χρησιμοποιεί ξεχωριστή ιδιότητα έκδοσης (`langchain4j.mcp.version`) επειδή κυκλοφορεί με διαφορετικό χρονοδιάγραμμα από τις βασικές βιβλιοθήκες LangChain4j.

> **⚠️ Πειραματικό:** Το `langchain4j-agentic` module είναι **πειραματικό** και υπόκειται σε αλλαγές. Ο σταθερός τρόπος για να δημιουργήσετε AI βοηθούς παραμένει το `langchain4j-core` με προσαρμοσμένα εργαλεία (Module 04).

### Supervisor Agent

Το μοτίβο **Supervisor Agent** είναι μια **ευέλικτη** μορφή agentic AI. Ένας Supervisor χρησιμοποιεί ένα LLM για να αποφασίζει αυτονομώς ποιον agent να καλέσει βάσει του αιτήματος του χρήστη. Στο επόμενο παράδειγμα συνδυάζουμε την πρόσβαση σε αρχεία μέσω MCP με έναν LLM agent για τη δημιουργία ροής εργασίας ανάγνωσης αρχείου → αναφοράς.

Στην επίδειξη, ο `FileAgent` διαβάζει ένα αρχείο χρησιμοποιώντας τα εργαλεία MCP του συστήματος αρχείων, και ο `ReportAgent` δημιουργεί μια δομημένη αναφορά με εκτελεστική περίληψη (1 πρόταση), 3 βασικά σημεία και προτάσεις. Ο Supervisor οργανώνει αυτό το ροή αυτόματα:

<img src="../../../translated_images/el/supervisor-agent-pattern.06275a41ae006ac8.webp" alt="Supervisor Agent Pattern" width="800"/>

*Ο Supervisor χρησιμοποιεί το LLM του για να αποφασίσει ποιοι agents θα κληθούν και με ποια σειρά — χωρίς ανάγκη σκληρού κωδικοποιημένου δρομολογίου.*

Έτσι φαίνεται η συγκεκριμένη ροή εργασίας για τον αγωγό αρχείο-αναφορά:

<img src="../../../translated_images/el/file-report-workflow.649bb7a896800de9.webp" alt="File to Report Workflow" width="800"/>

*Ο FileAgent διαβάζει το αρχείο μέσω εργαλείων MCP, μετά ο ReportAgent μετατρέπει το ακατέργαστο περιεχόμενο σε δομημένη αναφορά.*

Το ακόλουθο διάγραμμα ακολουθίας απεικονίζει την πλήρη οργάνωση του Supervisor — από τη δημιουργία του MCP διακομιστή, μέσα από την αυτόνομη επιλογή agent από τον Supervisor, έως τις κλήσεις εργαλείων μέσω stdio και την τελική αναφορά:

<img src="../../../translated_images/el/supervisor-agent-sequence.1aa389b3bef99956.webp" alt="Supervisor Agent Sequence Diagram" width="800"/>

*Ο Supervisor καλεί αυτόνομα τον FileAgent (που μεταδίδει εντολές στον MCP διακομιστή μέσω stdio για να διαβάσει το αρχείο), έπειτα καλεί τον ReportAgent για να δημιουργήσει δομημένη αναφορά — κάθε agent αποθηκεύει την έξοδό του στον κοινό Agentic Scope.*

Κάθε agent αποθηκεύει την έξοδό του στο **Agentic Scope** (κοινή μνήμη), επιτρέποντας στους agents που ακολουθούν να έχουν πρόσβαση σε προηγούμενα αποτελέσματα. Αυτό δείχνει πώς τα εργαλεία MCP ενσωματώνονται αβίαστα σε agentic workflows — ο Supervisor δεν χρειάζεται να γνωρίζει *πώς* διαβάζονται τα αρχεία, μόνο ότι το `FileAgent` μπορεί να το κάνει.

#### Πώς λειτουργεί ο Supervisor

Πριν δημιουργήσετε agents, πρέπει να συνδέσετε τη μεταφορά MCP με έναν πελάτη και να την τυλίξετε ως `ToolProvider`. Έτσι τα εργαλεία του MCP διακομιστή γίνονται διαθέσιμα στους agents σας:

```java
// Δημιουργήστε έναν πελάτη MCP από το μέσο μεταφοράς
McpClient mcpClient = new DefaultMcpClient.Builder()
        .transport(stdioTransport)
        .build();

// Περιβάλτε τον πελάτη ως ToolProvider — αυτό γεφυρώνει τα εργαλεία MCP με το LangChain4j
ToolProvider mcpToolProvider = McpToolProvider.builder()
        .mcpClients(List.of(mcpClient))
        .build();
```

Τώρα μπορείτε να εισάγετε το `mcpToolProvider` σε οποιονδήποτε agent χρειάζεται MCP εργαλεία:

```java
// Βήμα 1: Το FileAgent διαβάζει αρχεία χρησιμοποιώντας εργαλεία MCP
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // Έχει εργαλεία MCP για λειτουργίες αρχείων
        .build();

// Βήμα 2: Το ReportAgent δημιουργεί δομημένες αναφορές
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// Ο Supervisor συντονίζει τη ροή αρχείου → αναφοράς
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // Επιστρέφει την τελική αναφορά
        .build();

// Ο Supervisor αποφασίζει ποιους πράκτορες θα καλέσει βάσει του αιτήματος
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```

#### Πώς το FileAgent ανακαλύπτει τα MCP εργαλεία κατά την εκτέλεση

Μπορεί να αναρωτιέστε: **πώς ξέρει το `FileAgent` πώς να χρησιμοποιεί τα npm εργαλεία συστήματος αρχείων;** Η απάντηση είναι ότι δεν ξέρει — το **LLM** το ανακαλύπτει κατά την εκτέλεση μέσω των σχημάτων εργαλείων.

Η διεπαφή `FileAgent` είναι απλά μια **ορισμός prompt**. Δεν έχει προκαθορισμένη γνώση του `read_file`, `list_directory` ή οποιουδήποτε άλλου MCP εργαλείου. Δείτε τι συμβαίνει από άκρο σε άκρο:
1. **Εκκινήσεις διακομιστή:** Το `StdioMcpTransport` εκκινεί το πακέτο npm `@modelcontextprotocol/server-filesystem` ως διαδικασία παιδιού  
2. **Ανακάλυψη εργαλείων:** Ο `McpClient` στέλνει ένα αίτημα JSON-RPC `tools/list` στον διακομιστή, ο οποίος απαντά με ονόματα εργαλείων, περιγραφές και σχήματα παραμέτρων (π.χ., `read_file` — *"Διαβάστε το πλήρες περιεχόμενο ενός αρχείου"* — `{ path: string }`)  
3. **Έγχυση σχήματος:** Ο `McpToolProvider` τυλίγει αυτά τα ανακαλυφθέντα σχήματα και τα κάνει διαθέσιμα στο LangChain4j  
4. **Απόφαση LLM:** Όταν καλείται το `FileAgent.readFile(path)`, το LangChain4j στέλνει το σύστημα μηνύματος, το μήνυμα χρήστη, **και τη λίστα των σχημάτων εργαλείων** στο LLM. Το LLM διαβάζει τις περιγραφές εργαλείων και δημιουργεί μία κλήση εργαλείου (π.χ., `read_file(path="/some/file.txt")`)  
5. **Εκτέλεση:** Το LangChain4j παρεμβαίνει στην κλήση εργαλείου, την δρομολογεί μέσω του πελάτη MCP πίσω στη διεργασία Node.js, παίρνει το αποτέλεσμα και το τροφοδοτεί πίσω στο LLM  

Αυτή είναι η ίδια μηχανή [Ανακάλυψη Εργαλείων](../../../05-mcp) που περιγράφτηκε παραπάνω, αλλά εφαρμόζεται συγκεκριμένα στη ροή εργασίας πράκτορα. Οι σχολιασμοί `@SystemMessage` και `@UserMessage` καθοδηγούν τη συμπεριφορά του LLM, ενώ ο εγχυμένος `ToolProvider` του δίνει τις **δυνατότητες** — το LLM γεφυρώνει τα δύο σε χρόνο εκτέλεσης.

#### Στρατηγικές Απάντησης

Όταν ρυθμίζετε έναν `SupervisorAgent`, καθορίζετε πώς θα διαμορφώσει την τελική απάντησή του προς τον χρήστη αφού οι υπο-πράκτορες ολοκληρώσουν τα καθήκοντά τους. Το παρακάτω διάγραμμα δείχνει τις τρεις διαθέσιμες στρατηγικές — το LAST επιστρέφει άμεσα την έξοδο του τελευταίου πράκτορα, το SUMMARY συνθέτει όλες τις εξόδους μέσω ενός LLM, και το SCORED επιλέγει ποια από τις δύο έχει υψηλότερη βαθμολογία έναντι του αρχικού αιτήματος:

<img src="../../../translated_images/el/response-strategies.3d0cea19d096bdf9.webp" alt="Response Strategies" width="800"/>

*Τρεις στρατηγικές για το πώς ο Supervisor διαμορφώνει την τελική απάντηση — επιλέξτε ανάλογα με το αν θέλετε την έξοδο του τελευταίου πράκτορα, μια συνοπτική περίληψη ή την καλύτερη βαθμολογία.*

Οι διαθέσιμες στρατηγικές είναι:

| Στρατηγική | Περιγραφή |
|------------|-----------|
| **LAST** | Ο supervisor επιστρέφει την έξοδο του τελευταίου υπο-πράκτορα ή εργαλείου που κλήθηκε. Αυτό είναι χρήσιμο όταν ο τελικός πράκτορας στη ροή εργασίας έχει σχεδιαστεί ειδικά για να παράγει την πλήρη, τελική απάντηση (π.χ., ένας "Πράκτορας Περίληψης" σε μια ερευνητική διαδικασία). |
| **SUMMARY** | Ο supervisor χρησιμοποιεί το δικό του εσωτερικό Γλωσσικό Μοντέλο (LLM) για να συνθέσει μια περίληψη ολόκληρης της αλληλεπίδρασης και όλων των εξόδων υπο-πρακτόρων, και μετά επιστρέφει αυτή την περίληψη ως τελική απάντηση. Αυτό παρέχει μια καθαρή, συγκεντρωτική απάντηση στον χρήστη. |
| **SCORED** | Το σύστημα χρησιμοποιεί ένα εσωτερικό LLM για να βαθμολογήσει τόσο την απάντηση LAST όσο και την SUMMARY της αλληλεπίδρασης σε σχέση με το αρχικό αίτημα του χρήστη, επιστρέφοντας την έξοδο με την υψηλότερη βαθμολογία. |

#### Κατανόηση της Έξοδου

Όταν εκτελέσετε το demo, θα δείτε μια δομημένη παρουσίαση του πώς ο Supervisor συντονίζει πολλούς πράκτορες. Εδώ τι σημαίνει κάθε ενότητα:

```
======================================================================
  FILE → REPORT WORKFLOW DEMO
======================================================================

This demo shows a clear 2-step workflow: read a file, then generate a report.
The Supervisor orchestrates the agents automatically based on the request.
```
  
**Η κεφαλίδα** εισάγει την έννοια της ροής εργασίας: μια εστιασμένη διαδρομή από την ανάγνωση αρχείων μέχρι τη δημιουργία αναφορών.  

```
--- WORKFLOW ---------------------------------------------------------
  ┌─────────────┐      ┌──────────────┐
  │  FileAgent  │ ───▶ │ ReportAgent  │
  │ (MCP tools) │      │  (pure LLM)  │
  └─────────────┘      └──────────────┘
   outputKey:           outputKey:
   'fileContent'        'report'

--- AVAILABLE AGENTS -------------------------------------------------
  [FILE]   FileAgent   - Reads files via MCP → stores in 'fileContent'
  [REPORT] ReportAgent - Generates structured report → stores in 'report'
```
  
**Το Διάγραμμα Ροής Εργασίας** δείχνει τη ροή δεδομένων μεταξύ των πρακτόρων. Κάθε πράκτορας έχει συγκεκριμένο ρόλο:  
- **FileAgent** διαβάζει αρχεία χρησιμοποιώντας MCP εργαλεία και αποθηκεύει το ακατέργαστο περιεχόμενο στο `fileContent`  
- **ReportAgent** καταναλώνει αυτό το περιεχόμενο και παράγει μια δομημένη αναφορά στο `report`  

```
--- USER REQUEST -----------------------------------------------------
  "Read the file at .../file.txt and generate a report on its contents"
```
  
**Αίτημα Χρήστη** δείχνει το έργο. Ο Supervisor το αναλύει και αποφασίζει να καλέσει FileAgent → ReportAgent.

```

  The Supervisor decides which agents to invoke and passes data between them...

  +-- STEP 1: Supervisor chose -> FileAgent (reading file via MCP)
  |
  |   Input: .../file.txt
  |
  |   Result: LangChain4j is an open-source, provider-agnostic Java framework for building LLM...

  +-- STEP 2: Supervisor chose -> ReportAgent (generating structured report)
  |
  |   Input: LangChain4j is an open-source, provider-agnostic Java framew...
  |
  |   Result: Executive Summary...
  +-- [OK] ReportAgent (generating structured report) completed
```
  
**Συντονισμός Supervisor** δείχνει τη ροή 2 βημάτων σε δράση:  
1. Ο **FileAgent** διαβάζει το αρχείο μέσω MCP και αποθηκεύει το περιεχόμενο  
2. Ο **ReportAgent** λαμβάνει το περιεχόμενο και δημιουργεί μια δομημένη αναφορά  

Ο Supervisor πήρε αυτές τις αποφάσεις **αυτόνομα** βάσει του αιτήματος του χρήστη.

...

...

Recommendations

  Each agent stores its output for downstream agents to consume:

#### Εξήγηση των Δυνατοτήτων του Agentic Module

Το παράδειγμα επιδεικνύει αρκετές προηγμένες δυνατότητες του agentic module. Ας ρίξουμε μια πιο κοντινή ματιά στο Agentic Scope και τους Agent Listeners.

**Agentic Scope** δείχνει τη μοιρασμένη μνήμη όπου οι πράκτορες αποθήκευσαν τα αποτελέσματά τους χρησιμοποιώντας `@Agent(outputKey="...")`. Αυτό επιτρέπει:  
- Σε μεταγενέστερους πράκτορες να έχουν πρόσβαση στις εξόδους προηγούμενων πρακτόρων  
- Στον Supervisor να συνθέσει μια τελική απάντηση  
- Σε εσάς να επιθεωρείτε τι παρήγαγε κάθε πράκτορας  

Το διάγραμμα παρακάτω δείχνει πώς το Agentic Scope λειτουργεί ως κοινή μνήμη στη ροή εργασίας από αρχείο σε αναφορά — ο FileAgent γράφει την έξοδό του υπό το κλειδί `fileContent`, ο ReportAgent το διαβάζει και γράφει τη δική του έξοδο υπό το `report`:

<img src="../../../translated_images/el/agentic-scope.95ef488b6c1d02ef.webp" alt="Agentic Scope Shared Memory" width="800"/>

*Το Agentic Scope λειτουργεί ως κοινή μνήμη — ο FileAgent γράφει το `fileContent`, ο ReportAgent το διαβάζει και γράφει το `report`, και ο κώδικάς σας διαβάζει το τελικό αποτέλεσμα.*

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // Ακατέργαστα δεδομένα αρχείου από το FileAgent
String report = scope.readState("report");            // Δομημένη αναφορά από το ReportAgent
```
  
**Agent Listeners** επιτρέπουν την παρακολούθηση και αποσφαλμάτωση της εκτέλεσης των πρακτόρων. Η βήμα προς βήμα έξοδος που βλέπετε στο demo προέρχεται από έναν AgentListener που συνδέεται σε κάθε κλήση πράκτορα:  
- **beforeAgentInvocation** - Καλείται όταν ο Supervisor επιλέγει έναν πράκτορα, σας επιτρέπει να δείτε ποιος πράκτορας επιλέχθηκε και γιατί  
- **afterAgentInvocation** - Καλείται όταν ένας πράκτορας ολοκληρώνει, εμφανίζοντας το αποτέλεσμα του  
- **inheritedBySubagents** - Όταν είναι αληθές, ο listener παρακολουθεί όλους τους πράκτορες στην ιεραρχία  

Το ακόλουθο διάγραμμα δείχνει όλο τον κύκλο ζωής των Agent Listeners, συμπεριλαμβανομένου του πώς το `onError` χειρίζεται αποτυχίες κατά την εκτέλεση πράκτορα:

<img src="../../../translated_images/el/agent-listeners.784bfc403c80ea13.webp" alt="Agent Listeners Lifecycle" width="800"/>

*Οι Agent Listeners συνδέονται στον κύκλο ζωής εκτέλεσης — παρακολουθούν πότε οι πράκτορες ξεκινούν, ολοκληρώνουν ή συναντούν σφάλματα.*

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // Διάδωση σε όλους τους υπο-πράκτορες
    }
};
```
  
Πέρα από το πρότυπο Supervisor, το module `langchain4j-agentic` παρέχει αρκετά ισχυρά πρότυπα ροής εργασίας. Το διάγραμμα παρακάτω δείχνει και τα πέντε — από απλές διαδοχικές γραμμές παραγωγής μέχρι ροές έγκρισης με ανθρώπινη παρέμβαση:

<img src="../../../translated_images/el/workflow-patterns.82b2cc5b0c5edb22.webp" alt="Agent Workflow Patterns" width="800"/>

*Πέντε πρότυπα ροής εργασίας για το συντονισμό πρακτόρων — από απλές διαδοχικές γραμμές παραγωγής μέχρι ροές έγκρισης με ανθρώπινη παρέμβαση.*

| Πρότυπο | Περιγραφή | Περίπτωση Χρήσης |
|---------|-----------|------------------|
| **Sequential** | Εκτέλεση πρακτόρων με σειρά, η έξοδος ρέει στον επόμενο | Γραμμές παραγωγής: έρευνα → ανάλυση → αναφορά |
| **Parallel** | Εκτέλεση πρακτόρων ταυτόχρονα | Ανεξάρτητα καθήκοντα: καιρός + ειδήσεις + μετοχές |
| **Loop** | Επανάληψη έως ότου ικανοποιηθεί συνθήκη | Βαθμολόγηση ποιότητας: βελτίωση έως ότου σκορ ≥ 0.8 |
| **Conditional** | Δρομολόγηση βάσει συνθηκών | Κατηγοριοποίηση → δρομολόγηση σε εξειδικευμένο πράκτορα |
| **Human-in-the-Loop** | Προσθήκη ανθρώπινων σημείων ελέγχου | Ροές εργασίας έγκρισης, έλεγχος περιεχομένου |

## Βασικές Έννοιες

Τώρα που εξερευνήσατε το MCP και το agentic module σε δράση, ας συνοψίσουμε πότε να χρησιμοποιήσετε κάθε προσέγγιση.

Ένα από τα μεγαλύτερα πλεονεκτήματα του MCP είναι το συνεχώς αναδυόμενο οικοσύστημά του. Το διάγραμμα παρακάτω δείχνει πώς ένα ενιαίο καθολικό πρωτόκολλο συνδέει την εφαρμογή AI σας με ένα ευρύ φάσμα διακομιστών MCP — από πρόσβαση σε σύστημα αρχείων και βάσεις δεδομένων έως GitHub, email, εξαγωγή δεδομένων από web, και άλλα:

<img src="../../../translated_images/el/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="MCP Ecosystem" width="800"/>

*Το MCP δημιουργεί ένα καθολικό οικοσύστημα πρωτοκόλλου — οποιοσδήποτε διακομιστής συμβατός με MCP λειτουργεί με οποιονδήποτε πελάτη συμβατό με MCP, επιτρέποντας την κοινή χρήση εργαλείων μεταξύ εφαρμογών.*

**Το MCP** είναι ιδανικό όταν θέλετε να αξιοποιήσετε υπάρχοντα οικοσυστήματα εργαλείων, να δημιουργήσετε εργαλεία που μπορούν να μοιραστούν πολλαπλές εφαρμογές, να ενσωματώσετε υπηρεσίες τρίτων με τυποποιημένα πρωτόκολλα ή να αντικαταστήσετε υλοποιήσεις εργαλείων χωρίς να αλλάξετε κώδικα.

**Το Agentic Module** λειτουργεί καλύτερα όταν θέλετε δηλωτικούς ορισμούς πρακτόρων με σχολιασμούς `@Agent`, χρειάζεστε ορχήστρωση ροής εργασίας (διαδοχική, βρόχος, παράλληλη), προτιμάτε το σχεδιασμό πράκτορα βασισμένο σε διεπαφή αντί για επιτακτικό κώδικα, ή συνδυάζετε πολλούς πράκτορες που μοιράζονται εξόδους μέσω `outputKey`.

**Το πρότυπο Supervisor Agent** ξεχωρίζει όταν η ροή εργασίας δεν είναι προβλέψιμη εκ των προτέρων και θέλετε να αποφασίσει το LLM, όταν έχετε πολλούς εξειδικευμένους πράκτορες που χρειάζονται δυναμικό συντονισμό, όταν χτίζετε συστήματα διαλόγου που δρομολογούν σε διαφορετικές δυνατότητες, ή όταν θέλετε τη πιο ευέλικτη, προσαρμοστική συμπεριφορά πράκτορα.

Για να σας βοηθήσουμε να αποφασίσετε ανάμεσα στις προσαρμοσμένες μεθόδους `@Tool` από το Module 04 και τα εργαλεία MCP από αυτό το module, η παρακάτω σύγκριση επισημαίνει τα βασικά ανταλλάγματα — τα προσαρμοσμένα εργαλεία προσφέρουν στενή σύνδεση και πλήρη ασφάλεια τύπων για τη λογική ειδική εφαρμογής, ενώ τα εργαλεία MCP προσφέρουν τυποποιημένες, επαναχρησιμοποιήσιμες ενσωματώσεις:

<img src="../../../translated_images/el/custom-vs-mcp-tools.c4f9b6b1cb65d8a1.webp" alt="Custom Tools vs MCP Tools" width="800"/>

*Πότε να χρησιμοποιήσετε προσαρμοσμένες μεθόδους @Tool έναντι εργαλείων MCP — προσαρμοσμένα εργαλεία για λογική εφαρμογής με πλήρη ασφάλεια τύπων, εργαλεία MCP για τυποποιημένες ενσωματώσεις που λειτουργούν σε εφαρμογές.*
