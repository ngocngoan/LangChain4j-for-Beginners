## Τι θα μάθετε

Έχετε δημιουργήσει συνομιλητική τεχνητή νοημοσύνη, έχετε κατακτήσει τα prompts, έχετε στηρίξει απαντήσεις σε έγγραφα και έχετε φτιάξει agents με εργαλεία. Αλλά όλα αυτά τα εργαλεία ήταν προσαρμοσμένα για την συγκεκριμένη εφαρμογή σας. Τι θα γινόταν αν μπορούσατε να δώσετε στην AI σας πρόσβαση σε ένα τυποποιημένο οικοσύστημα εργαλείων που ο καθένας μπορεί να δημιουργήσει και να μοιραστεί; Σε αυτό το module, θα μάθετε πώς να το κάνετε ακριβώς αυτό με το Πρωτόκολλο Πλαισίου Μοντέλου (MCP) και το agentic module του LangChain4j. Αρχικά παρουσιάζουμε έναν απλό αναγνώστη αρχείων MCP και στη συνέχεια δείχνουμε πώς εντάσσεται εύκολα σε προηγμένα agentic workflows με το πρότυπο Supervisor Agent.

## Τι είναι το MCP;

Το Πρωτόκολλο Πλαισίου Μοντέλου (MCP) προσφέρει ακριβώς αυτό - έναν τυποποιημένο τρόπο για εφαρμογές AI να ανακαλύπτουν και να χρησιμοποιούν εξωτερικά εργαλεία. Αντί να γράφετε προσαρμοσμένες ενσωματώσεις για κάθε πηγή δεδομένων ή υπηρεσία, συνδέεστε με servers MCP που εκθέτουν τις δυνατότητές τους σε συνεπή μορφή. Ο AI agent σας μπορεί τότε να ανακαλύψει και να χρησιμοποιήσει αυτά τα εργαλεία αυτόματα.

<img src="../../../translated_images/el/mcp-comparison.9129a881ecf10ff5.webp" alt="MCP Σύγκριση" width="800"/>

*Πριν το MCP: Πολύπλοκες σημείο-προς-σημείο ενσωματώσεις. Μετά το MCP: Ένα πρωτόκολλο, ατελείωτες δυνατότητες.*

Το MCP λύνει ένα θεμελιώδες πρόβλημα στην ανάπτυξη AI: κάθε ενσωμάτωση είναι προσαρμοσμένη. Θέλετε να αποκτήσετε πρόσβαση σε GitHub; Προσαρμοσμένος κώδικας. Θέλετε να διαβάσετε αρχεία; Προσαρμοσμένος κώδικας. Θέλετε να κάνετε ερώτημα σε βάση δεδομένων; Προσαρμοσμένος κώδικας. Και καμία από αυτές τις ενσωματώσεις δεν λειτουργεί με άλλες εφαρμογές AI.

Το MCP το τυποποιεί αυτό. Ένας server MCP εκθέτει εργαλεία με σαφείς περιγραφές και σχήματα. Κάθε client MCP μπορεί να συνδεθεί, να ανακαλύψει διαθέσιμα εργαλεία και να τα χρησιμοποιήσει. Φτιάξτε το μία φορά, χρησιμοποιήστε το παντού.

*Αρχιτεκτονική Πρωτοκόλλου Πλαισίου Μοντέλου - τυποποιημένη ανακάλυψη και εκτέλεση εργαλείων*

## Πώς λειτουργεί το MCP

<img src="../../../translated_images/el/mcp-protocol-detail.01204e056f45308b.webp" alt="Λεπτομέρειες Πρωτοκόλλου MCP" width="800"/>

*Πώς λειτουργεί το MCP "από μέσα" — οι clients ανακαλύπτουν εργαλεία, ανταλλάσσουν μηνύματα JSON-RPC, και εκτελούν ενέργειες μέσω επιπέδου μεταφοράς.*

**Αρχιτεκτονική Server-Client**

Το MCP χρησιμοποιεί μοντέλο client-server. Οι servers παρέχουν εργαλεία - ανάγνωση αρχείων, ερωτήματα βάσεων, κλήσεις API. Οι clients (η εφαρμογή AI σας) συνδέονται σε servers και χρησιμοποιούν τα εργαλεία τους.

Για να χρησιμοποιήσετε το MCP με το LangChain4j, προσθέστε αυτή την Maven εξάρτηση:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```

**Ανακάλυψη Εργαλείων**

Όταν ο client σας συνδέεται σε έναν server MCP, ρωτά "Ποια εργαλεία έχετε;" Ο server απαντά με λίστα διαθέσιμων εργαλείων, κάθε ένα με περιγραφές και σχήματα παραμέτρων. Ο AI agent σας αποφασίζει ποια εργαλεία θα χρησιμοποιήσει βάσει των αιτημάτων των χρηστών.

<img src="../../../translated_images/el/tool-discovery.07760a8a301a7832.webp" alt="Ανακάλυψη Εργαλείων MCP" width="800"/>

*Η AI ανακαλύπτει διαθέσιμα εργαλεία κατά την εκκίνηση — τώρα ξέρει τι δυνατότητες υπάρχουν και μπορεί να αποφασίσει ποια θα χρησιμοποιήσει.*

**Μηχανισμοί Μεταφοράς**

## Το Agentic Module

Ενώ το MCP παρέχει τυποποιημένα εργαλεία, το **agentic module** του LangChain4j παρέχει έναν δηλωτικό τρόπο δημιουργίας agents που ορχηστρώνουν αυτά τα εργαλεία. Το annotation `@Agent` και το `AgenticServices` σας επιτρέπουν να ορίσετε τη συμπεριφορά του agent μέσω διεπαφών αντί για προστακτικό κώδικα.

Σε αυτό το module, θα εξερευνήσετε το πρότυπο **Supervisor Agent** — μια προηγμένη προσεγγιστική agentic AI όπου ένας "επόπτης" agent αποφασίζει δυναμικά ποιον υπο-agent θα καλέσει βάσει των αιτημάτων του χρήστη. Θα συνδυάσουμε και τις δύο έννοιες δίνοντας σε έναν από τους υπο-agents μας δυνατότητες πρόσβασης αρχείων βασισμένες στο MCP.

Για να χρησιμοποιήσετε το agentic module, προσθέστε αυτή την Maven εξάρτηση:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```

> **⚠️ Πειραματικό:** Το `langchain4j-agentic` module είναι **πειραματικό** και υπόκειται σε αλλαγές. Ο σταθερός τρόπος για τη δημιουργία AI βοηθών παραμένει το `langchain4j-core` με προσαρμοσμένα εργαλεία (Module 04).

### Supervisor Agent

Το πρότυπο **Supervisor Agent** είναι μια **ευέλικτη** μορφή agentic AI. Ένας Supervisor χρησιμοποιεί μοντέλο γλώσσας (LLM) για να αποφασίσει αυτόνομα ποια agents θα καλέσει βάσει του αιτήματος του χρήστη. Στο επόμενο παράδειγμα, συνδυάζουμε πρόσβαση αρχείων με MCP με έναν LLM agent για να δημιουργήσουμε μια ροή διαχείρισης ανάγνωσης αρχείων → αναφοράς υπό επίβλεψη.

Στην επίδειξη, ο `FileAgent` διαβάζει ένα αρχείο χρησιμοποιώντας εργαλεία MCP του συστήματος αρχείων, και ο `ReportAgent` παράγει μια δομημένη αναφορά με εκτελεστική περίληψη (1 πρόταση), 3 βασικά σημεία και προτάσεις. Ο Supervisor ορχηστρώνει αυτή τη ροή αυτόματα:

<img src="../../../translated_images/el/supervisor-agent-pattern.06275a41ae006ac8.webp" alt="Πρότυπο Supervisor Agent" width="800"/>

*Ο Supervisor χρησιμοποιεί το LLM του για να αποφασίσει ποιον agent να καλέσει και με ποια σειρά — δεν χρειάζεται σκληροκωδικοποιημένη δρομολόγηση.*

Δείτε πώς ακριβώς είναι η ροή εργασίας για το pipeline αρχείο προς αναφορά:

<img src="../../../translated_images/el/file-report-workflow.649bb7a896800de9.webp" alt="Ροή Αρχείου προς Αναφορά" width="800"/>

*Ο FileAgent διαβάζει το αρχείο μέσω εργαλείων MCP, και ο ReportAgent μετατρέπει το ακατέργαστο περιεχόμενο σε δομημένη αναφορά.*

Κάθε agent αποθηκεύει την έξοδό του στο **Agentic Scope** (κοινόχρηστη μνήμη), επιτρέποντας σε downstream agents να έχουν πρόσβαση σε προηγούμενα αποτελέσματα. Αυτό δείχνει πώς τα εργαλεία MCP εντάσσονται απρόσκοπτα σε agentic workflows — ο Supervisor δεν χρειάζεται να ξέρει *πώς* διαβάζονται τα αρχεία, μόνο ότι ο `FileAgent` μπορεί να το κάνει.

#### Πώς λειτουργεί ο Supervisor

```java
// Βήμα 1: Το FileAgent διαβάζει αρχεία χρησιμοποιώντας εργαλεία MCP
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // Διαθέτει εργαλεία MCP για λειτουργίες αρχείων
        .build();

// Βήμα 2: Το ReportAgent δημιουργεί δομημένες αναφορές
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// Ο Επιβλέπων συντονίζει τη ροή εργασίας αρχείο → αναφορά
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // Επιστρέψτε την τελική αναφορά
        .build();

// Ο Επιβλέπων αποφασίζει ποιους πράκτορες θα ενεργοποιήσει βάσει του αιτήματος
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```

#### Στρατηγικές απόκρισης

Όταν διαμορφώνετε έναν `SupervisorAgent`, καθορίζετε πώς πρέπει να διατυπώσει την τελική του απάντηση προς τον χρήστη αφού οι υπο-agents ολοκληρώσουν τις εργασίες τους.

<img src="../../../translated_images/el/response-strategies.3d0cea19d096bdf9.webp" alt="Στρατηγικές Απόκρισης" width="800"/>

*Τρεις στρατηγικές για το πώς ο Supervisor διαμορφώνει την τελική του απάντηση — επιλέξτε αν θέλετε την έξοδο του τελευταίου agent, μια συνοψισμένη περίληψη ή την επιλογή με τη μεγαλύτερη βαθμολογία.*

Οι διαθέσιμες στρατηγικές είναι:

| Στρατηγική | Περιγραφή |
|----------|-------------|
| **LAST** | Ο supervisor επιστρέφει την έξοδο του τελευταίου υπο-agent ή εργαλείου που κλήθηκε. Αυτό είναι χρήσιμο όταν ο τελικός agent στη ροή εργασίας έχει σχεδιαστεί ειδικά για να παράγει την πλήρη, τελική απάντηση (π.χ., ένας "Agent Περιλήψεων" σε pipeline έρευνας). |
| **SUMMARY** | Ο supervisor χρησιμοποιεί το εσωτερικό του μοντέλο γλώσσας (LLM) για να συνθέσει μια περίληψη ολόκληρης της αλληλεπίδρασης και όλων των εξόδων των υπο-agents, και επιστρέφει αυτήν την περίληψη ως τελική απάντηση. Παρέχει μια καθαρή, συλλεγμένη απάντηση στον χρήστη. |
| **SCORED** | Το σύστημα χρησιμοποιεί ένα εσωτερικό LLM για να βαθμολογήσει τόσο την απάντηση LAST όσο και την περίληψη SUMMARY της αλληλεπίδρασης σε σχέση με το αρχικό αίτημα του χρήστη, επιστρέφοντας αυτή που έχει την υψηλότερη βαθμολογία. |

Δείτε το [SupervisorAgentDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/SupervisorAgentDemo.java) για την πλήρη υλοποίηση.

#### Κατανόηση του Αποτελέσματος

Όταν εκτελέσετε το demo, θα δείτε μια δομημένη παρουσίαση του πώς ο Supervisor ορχηστρώνει πολλαπλούς agents. Δείτε τι σημαίνει κάθε ενότητα:

```
======================================================================
  FILE → REPORT WORKFLOW DEMO
======================================================================

This demo shows a clear 2-step workflow: read a file, then generate a report.
The Supervisor orchestrates the agents automatically based on the request.
```

**Η επικεφαλίδα** εισάγει την έννοια της ροής εργασίας: μια εστιασμένη αλυσίδα από ανάγνωση αρχείου μέχρι παραγωγή αναφοράς.

```
--- WORKFLOW ---------------------------------------------------------
  ┌─────────────┐      ┌──────────────┐
  │  FileAgent  │ ───▶ │ ReportAgent  │
  │ (MCP tools) │      │  (pure LLM)  │
  └─────────────┘      └──────────────┘
   outputKey:           outputKey:
   'fileContent'        'report'

--- AVAILABLE AGENTS -------------------------------------------------
  [FILE]   FileAgent   - Reads files via MCP → stores in 'fileContent'
  [REPORT] ReportAgent - Generates structured report → stores in 'report'
```

**Διάγραμμα Ροής** δείχνει τη ροή δεδομένων μεταξύ agents. Κάθε agent έχει συγκεκριμένο ρόλο:
- **FileAgent** διαβάζει αρχεία χρησιμοποιώντας εργαλεία MCP και αποθηκεύει το ακατέργαστο περιεχόμενο σε `fileContent`
- **ReportAgent** χρησιμοποιεί αυτό το περιεχόμενο και παράγει μια δομημένη αναφορά σε `report`

```
--- USER REQUEST -----------------------------------------------------
  "Read the file at .../file.txt and generate a report on its contents"
```

**Αίτημα Χρήστη** δείχνει την εργασία. Ο Supervisor το αναλύει και αποφασίζει να καλέσει FileAgent → ReportAgent.

```
--- SUPERVISOR ORCHESTRATION -----------------------------------------
  The Supervisor decides which agents to invoke and passes data between them...

  +-- STEP 1: Supervisor chose -> FileAgent (reading file via MCP)
  |
  |   Input: .../file.txt
  |
  |   Result: LangChain4j is an open-source, provider-agnostic Java framework for building LLM...
  +-- [OK] FileAgent (reading file via MCP) completed

  +-- STEP 2: Supervisor chose -> ReportAgent (generating structured report)
  |
  |   Input: LangChain4j is an open-source, provider-agnostic Java framew...
  |
  |   Result: Executive Summary...
  +-- [OK] ReportAgent (generating structured report) completed
```

**Ορχήστρωση Supervisor** δείχνει τη ροή σε 2 βήματα σε δράση:
1. **FileAgent** διαβάζει το αρχείο μέσω MCP και αποθηκεύει το περιεχόμενο
2. **ReportAgent** λαμβάνει το περιεχόμενο και δημιουργεί μια δομημένη αναφορά

Ο Supervisor πήρε αυτές τις αποφάσεις **αυτόνομα** με βάση το αίτημα του χρήστη.

```
--- FINAL RESPONSE ---------------------------------------------------
Executive Summary
...

Key Points
...

Recommendations
...

--- AGENTIC SCOPE (Data Flow) ----------------------------------------
  Each agent stores its output for downstream agents to consume:
  * fileContent: LangChain4j is an open-source, provider-agnostic Java framework...
  * report: Executive Summary...
```

#### Εξήγηση Χαρακτηριστικών του Agentic Module

Το παράδειγμα δείχνει αρκετά προηγμένα χαρακτηριστικά του agentic module. Ας δούμε πιο προσεκτικά το Agentic Scope και τους Agent Listeners.

**Agentic Scope** δείχνει την κοινόχρηστη μνήμη όπου οι agents αποθήκευσαν τα αποτελέσματά τους χρησιμοποιώντας το `@Agent(outputKey="...")`. Αυτό επιτρέπει:
- Μετέπειτα agents να έχουν πρόσβαση στα αποτελέσματα προηγούμενων agents
- Στον Supervisor να συνθέτει μια τελική απάντηση
- Εσάς να ελέγχετε τι παρήγαγε κάθε agent

<img src="../../../translated_images/el/agentic-scope.95ef488b6c1d02ef.webp" alt="Κοινόχρηστη Μνήμη Agentic Scope" width="800"/>

*Το Agentic Scope λειτουργεί ως κοινόχρηστη μνήμη — ο FileAgent γράφει το `fileContent`, ο ReportAgent το διαβάζει και γράφει το `report`, και ο κώδικάς σας διαβάζει το τελικό αποτέλεσμα.*

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // Ακατέργαστα δεδομένα αρχείου από το FileAgent
String report = scope.readState("report");            // Δομημένη αναφορά από το ReportAgent
```

**Agent Listeners** επιτρέπουν την παρακολούθηση και αποσφαλμάτωση της εκτέλεσης των agents. Η βήμα-βήμα έξοδος που βλέπετε στο demo προέρχεται από έναν AgentListener που συνδέεται σε κάθε κλήση agent:
- **beforeAgentInvocation** - Καλείται όταν ο Επόπτης επιλέγει έναν πράκτορα, επιτρέποντάς σας να δείτε ποιος πράκτορας επιλέχθηκε και γιατί  
- **afterAgentInvocation** - Καλείται όταν ένας πράκτορας ολοκληρώνει, εμφανίζοντας το αποτέλεσμα του  
- **inheritedBySubagents** - Όταν είναι αληθές, ο ακροατής παρακολουθεί όλους τους πράκτορες στην ιεραρχία  

<img src="../../../translated_images/el/agent-listeners.784bfc403c80ea13.webp" alt="Κύκλος Ζωής Ακροατών Πρακτόρων" width="800"/>

*Οι ακροατές πρακτόρων συνδέονται με τον κύκλο ζωής εκτέλεσης — παρακολουθούν πότε ξεκινούν, ολοκληρώνουν ή συναντούν σφάλματα οι πράκτορες.*

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // Διαδώστε σε όλους τους υπο-πράκτορες
    }
};
```
  
Πέρα από το πρότυπο Επόπτη, το `langchain4j-agentic` module παρέχει αρκετά ισχυρά πρότυπα ροής εργασίας και λειτουργίες:

<img src="../../../translated_images/el/workflow-patterns.82b2cc5b0c5edb22.webp" alt="Πρότυπα Ροής Εργασίας Πρακτόρων" width="800"/>

*Πέντε πρότυπα ροής εργασίας για τον συντονισμό πρακτόρων — από απλές διαδοχικές διεργασίες μέχρι ροές εργασίας έγκρισης με ανθρώπινη συμμετοχή.*

| Πρότυπο | Περιγραφή | Περίπτωση Χρήσης |
|---------|-------------|----------|
| **Διαδοχικό** | Εκτέλεση πρακτόρων με σειρά, η έξοδος ρέει στον επόμενο | Αλυσίδες επεξεργασίας: έρευνα → ανάλυση → αναφορά |
| **Παράλληλο** | Εκτέλεση πρακτόρων ταυτόχρονα | Ανεξάρτητες εργασίες: καιρός + ειδήσεις + χρηματιστήρια |
| **Βρόχος** | Επανάληψη μέχρι να ικανοποιηθεί η συνθήκη | Βαθμολόγηση ποιότητας: βελτίωση μέχρι βαθμός ≥ 0.8 |
| **Υπό όρους** | Κατεύθυνση βάσει συνθηκών | Κατηγοριοποίηση → κατεύθυνση σε εξειδικευμένο πράκτορα |
| **Άνθρωπος στη ροή** | Προσθήκη ανθρώπινων σημείων ελέγχου | Ροές εργασίας έγκρισης, ανασκόπηση περιεχομένου |

## Βασικές Έννοιες

Τώρα που εξερευνήσατε το MCP και το agentic module στην πράξη, ας συνοψίσουμε πότε να χρησιμοποιήσετε κάθε προσέγγιση.

<img src="../../../translated_images/el/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="Οικοσύστημα MCP" width="800"/>

*Το MCP δημιουργεί ένα καθολικό πρωτόκολλο οικοσύστημα — οποιοσδήποτε συμβατός διακομιστής MCP λειτουργεί με οποιονδήποτε συμβατό πελάτη MCP, επιτρέποντας κοινή χρήση εργαλείων μεταξύ εφαρμογών.*

**Το MCP** είναι ιδανικό όταν θέλετε να αξιοποιήσετε υπάρχοντα οικοσυστήματα εργαλείων, να δημιουργήσετε εργαλεία που μπορούν να μοιραστούν πολλές εφαρμογές, να ενσωματώσετε υπηρεσίες τρίτων με πρότυπα πρωτόκολλα, ή να ανταλλάξετε υλοποιήσεις εργαλείων χωρίς να αλλάζει ο κώδικας.

**Το Agentic Module** λειτουργεί καλύτερα όταν θέλετε δηλωτικούς ορισμούς πρακτόρων με το `@Agent` annotations, χρειάζεστε ορχήστρωση ροής εργασίας (διαδοχική, βρόχος, παράλληλη), προτιμάτε σχεδιασμό πρακτόρων βασισμένο σε διεπαφή αντί για επιτακτικό κώδικα, ή συνδυάζετε πολλαπλούς πράκτορες που μοιράζονται εξόδους μέσω `outputKey`.

**Το πρότυπο Επόπτη Πράκτορα** ξεχωρίζει όταν η ροή εργασίας δεν είναι προβλέψιμη εκ των προτέρων και θέλετε το LLM να αποφασίζει, όταν έχετε πολλούς εξειδικευμένους πράκτορες που χρειάζονται δυναμική ορχήστρωση, όταν δημιουργείτε διαλογικά συστήματα που κατευθύνουν σε διαφορετικές δυνατότητες, ή όταν θέλετε την πιο ευέλικτη, προσαρμοστική συμπεριφορά πράκτορα.

<img src="../../../translated_images/el/custom-vs-mcp-tools.c4f9b6b1cb65d8a1.webp" alt="Προσαρμοσμένα Εργαλεία έναντι Εργαλείων MCP" width="800"/>

*Πότε να χρησιμοποιείτε προσαρμοσμένες μεθόδους @Tool έναντι εργαλείων MCP — προσαρμοσμένα εργαλεία για λογική ειδική εφαρμογής με πλήρη ασφάλεια τύπων, εργαλεία MCP για τυποποιημένες ενσωματώσεις που λειτουργούν μεταξύ εφαρμογών.*
