## Какво е MCP?

Протоколът за контекст на модела (MCP) осигурява точно това – стандартен начин AI приложенията да откриват и използват външни инструменти. Вместо да пишете персонализирани интеграции за всеки източник на данни или услуга, вие се свързвате към MCP сървъри, които излагат своите възможности в консистентен формат. Вашият AI агент може автоматично да открива и използва тези инструменти.

Диаграмата по-долу показва разликата – без MCP, всяка интеграция изисква персонализирано свързване точка до точка; с MCP, един протокол свързва вашето приложение с всеки инструмент:

<img src="../../../translated_images/bg/mcp-comparison.9129a881ecf10ff5.webp" alt="MCP Comparison" width="800"/>

*Преди MCP: Комплексни интеграции точка до точка. След MCP: Един протокол, безкрайни възможности.*

MCP решава фундаментален проблем в разработката на AI: всяка интеграция е персонализирана. Искате да достъпите GitHub? Персонализиран код. Искате да четете файлове? Персонализиран код. Искате да правите заявки към база данни? Персонализиран код. И нито една от тези интеграции не работи с други AI приложения.

MCP стандартизира това. Серверът MCP излага инструменти с ясни описания и схеми. Всеки MCP клиент може да се свърже, да открие наличните инструменти и да ги използва. Създавате веднъж, използвате навсякъде.

Диаграмата по-долу илюстрира тази архитектура – един MCP клиент (вашето AI приложение) се свързва с множество MCP сървъри, всеки от които излага своя набор от инструменти чрез стандартизирания протокол:

<img src="../../../translated_images/bg/mcp-architecture.b3156d787a4ceac9.webp" alt="MCP Architecture" width="800"/>

*Архитектура на Протокола за контекст на модела - стандартизирано откриване и изпълнение на инструменти*

## Как работи MCP

Под капака MCP използва многослойна архитектура. Вашето Java приложение (MCP клиент) открива наличните инструменти, изпраща JSON-RPC заявки чрез транспортен слой (Stdio или HTTP), а MCP сървърът изпълнява операциите и връща резултатите. Следващата диаграма разглежда всеки слой на този протокол:

<img src="../../../translated_images/bg/mcp-protocol-detail.01204e056f45308b.webp" alt="MCP Protocol Detail" width="800"/>

*Как работи MCP под капака — клиентите откриват инструменти, обменят JSON-RPC съобщения и изпълняват операции през транспортен слой.*

**Архитектура клиент-сървър**

MCP използва модел клиент-сървър. Сървърите предоставят инструменти – четене на файлове, заявки към бази данни, извикване на API. Клиентите (вашето AI приложение) се свързват със сървърите и използват техните инструменти.

За да използвате MCP с LangChain4j, добавете тази Maven зависимост:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```

**Откриване на инструменти**

Когато вашият клиент се свърже със MCP сървър, той пита "Какви инструменти имате?" Сървърът отговаря с списък на наличните инструменти, всеки с описание и схеми за параметри. Вашият AI агент може след това да реши кои инструменти да използва на база заявки от потребителя. Диаграмата по-долу показва този ръкостискащ обмен – клиентът изпраща заявка `tools/list`, а сървърът връща наличните си инструменти с описания и схеми за параметри:

<img src="../../../translated_images/bg/tool-discovery.07760a8a301a7832.webp" alt="MCP Tool Discovery" width="800"/>

*AI открива наличните инструменти при стартиране — вече знае какви възможности има и може да реши кои да използва.*

**Транспортни механизми**

MCP поддържа различни транспортни механизми. Двата варианта са Stdio (за локална комуникация с подпроцеси) и Streamable HTTP (за отдалечени сървъри). Този модул демонстрира транспорта Stdio:

<img src="../../../translated_images/bg/transport-mechanisms.2791ba7ee93cf020.webp" alt="Transport Mechanisms" width="800"/>

*Транспортни механизми на MCP: HTTP за отдалечени сървъри, Stdio за локални процеси*

**Stdio** - [StdioTransportDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java)

За локални процеси. Вашето приложение създава сървър като подпроцес и комуникира чрез стандартния вход/изход. Полезно за достъп до файловата система или командни инструменти.

```java
McpTransport stdioTransport = new StdioMcpTransport.Builder()
    .command(List.of(
        npmCmd, "exec",
        "@modelcontextprotocol/server-filesystem@2025.12.18",
        resourcesDir
    ))
    .logEvents(false)
    .build();
```

Сървърът `@modelcontextprotocol/server-filesystem` излага следните инструменти, всички ограничени до директориите, които посочите:

| Инструмент | Описание |
|------|-------------|
| `read_file` | Чете съдържанието на един файл |
| `read_multiple_files` | Чете множество файлове в един вик |
| `write_file` | Създава или презаписва файл |
| `edit_file` | Прави целеви замени чрез намери и замени |
| `list_directory` | Изписва файлове и папки на даден път |
| `search_files` | Рекурсивно търси файлове, отговарящи на модел |
| `get_file_info` | Връща метаданни за файла (размер, времеви отметки, разрешения) |
| `create_directory` | Създава директория (включително родителски директории) |
| `move_file` | Премества или преименува файл или директория |

Следващата диаграма показва как работи транспорта Stdio по време на изпълнение — вашето Java приложение създава MCP сървъра като дете процес и те комуникират през stdin/stdout канали, без мрежа или HTTP:

<img src="../../../translated_images/bg/stdio-transport-flow.45eaff4af2d81db4.webp" alt="Stdio Transport Flow" width="800"/>

*Транспортът Stdio в действие — вашето приложение създава MCP сървър като дете процес и комуникира през stdin/stdout канали.*

## Агенционният модул

Докато MCP осигурява стандартизирани инструменти, **агенционният модул** на LangChain4j предоставя декларативен начин за изграждане на агенти, които оркестрират тези инструменти. Белегът `@Agent` и `AgenticServices` ви позволяват да дефинирате поведението на агента чрез интерфейси, а не с императивен код.

В този модул ще разгледате модела на **Супервизор агента** — усъвършенстван агенционен AI подход, при който "супервизор" агент динамично решава кои подагенти да извика въз основа на заявките на потребителя. Ще комбинираме двата концепта, като дадем на един от нашите подагенти възможности за достъп до файлове, захранван от MCP.

За да използвате агенционния модул, добавете тази Maven зависимост:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```
> **Забележка:** Модулът `langchain4j-agentic` използва отделно свойство за версия (`langchain4j.mcp.version`), защото се пуска по различен график от основните библиотеки LangChain4j.

> **⚠️ Експериментално:** Модулът `langchain4j-agentic` е **експериментален** и може да бъде променен. Стабилният начин за изграждане на AI асистенти остава чрез `langchain4j-core` с персонализирани инструменти (Модул 04).

### Супервизор агент

Моделът на **Супервизор агента** е **гъвкав** вид агенционен AI. Супервизорът използва LLM, за да решава автономно кои агенти да извика на база потребителската заявка. В следващия пример комбинираме MCP-захранен достъп до файлове с LLM агент, за да създадем контролирана работна последователност четене на файл → създаване на доклад.

В демонстрацията `FileAgent` чете файл с MCP инструменти за файловата система, а `ReportAgent` генерира структуриран доклад с изпълнително резюме (1 изречение), 3 ключови точки и препоръки. Супервизорът оркестрира този поток автоматично:

<img src="../../../translated_images/bg/supervisor-agent-pattern.06275a41ae006ac8.webp" alt="Supervisor Agent Pattern" width="800"/>

*Супервизорът използва LLM, за да реши кои агенти да извика и в какъв ред — без нужда от твърдо кодирано маршрутизиране.*

Ето как изглежда конкретният работен поток за нашия pipeline файл→доклад:

<img src="../../../translated_images/bg/file-report-workflow.649bb7a896800de9.webp" alt="File to Report Workflow" width="800"/>

*FileAgent чете файла чрез MCP инструменти, след което ReportAgent преобразува суровото съдържание в структуриран доклад.*

Следващата диаграма последователности проследява пълната оркестрация от страна на Супервизора — от създаването на MCP сървъра, през автономния избор на агенти на Супервизора, до вызовите към инструмента чрез stdio и крайния доклад:

<img src="../../../translated_images/bg/supervisor-agent-sequence.1aa389b3bef99956.webp" alt="Supervisor Agent Sequence Diagram" width="800"/>

*Супервизорът автономно извиква FileAgent (който извиква MCP сървъра чрез stdio, за да прочете файла), после извиква ReportAgent за създаване на структуриран доклад — всеки агент съхранява изхода си в споделената агенционна памет (Agentic Scope).*

Всеки агент съхранява своя изход в **Agentic Scope** (споделена памет), което позволява на следващите агенти да достъпват предишни резултати. Това показва как MCP инструментите се интегрират безпроблемно в агенционни работни потоци — Супервизорът не трябва да знае *как* файловете се четат, а само че `FileAgent` може да го направи.

#### Как работи супервизорът

Преди да изградите агенти, трябва да свържете MCP транспорта към клиент и да го опаковате като `ToolProvider`. Така инструментите на MCP сървъра стават достъпни за вашите агенти:

```java
// Създайте MCP клиент от транспорта
McpClient mcpClient = new DefaultMcpClient.Builder()
        .transport(stdioTransport)
        .build();

// Опаковайте клиента като ToolProvider — това свързва MCP инструментите с LangChain4j
ToolProvider mcpToolProvider = McpToolProvider.builder()
        .mcpClients(List.of(mcpClient))
        .build();
```

Сега можете да инжектирате `mcpToolProvider` в всеки агент, който се нуждае от MCP инструменти:

```java
// Стъпка 1: FileAgent чете файлове с помощта на MCP инструменти
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // Разполага с MCP инструменти за файлови операции
        .build();

// Стъпка 2: ReportAgent генерира структурирани доклади
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// Supervisor координира работния процес файл → доклад
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // Връща окончателния доклад
        .build();

// Supervisor решава кои агенти да бъдат извикани в зависимост от заявката
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```

#### Как FileAgent открива MCP инструментите по време на изпълнение

Може би се питате: **как `FileAgent` знае как да използва инструменти за файловата система от npm?** Отговорът е, че той не знае — **LLM** го изчислява по време на изпълнение чрез схемите на инструментите.
Интерфейсът `FileAgent` е просто **дефинция на подканата**. Той няма вградени знания за `read_file`, `list_directory` или някой друг MCP инструмент. Ето какво се случва от край до край:

1. **Сървърът стартира:** `StdioMcpTransport` пуска npm пакета `@modelcontextprotocol/server-filesystem` като подпроцес
2. **Откриване на инструменти:** `McpClient` изпраща JSON-RPC заявка `tools/list` към сървъра, който отговаря с имена на инструменти, описания и схеми на параметрите (например `read_file` — *"Прочете пълното съдържание на файл"* — `{ path: string }`)
3. **Инжектиране на схеми:** `McpToolProvider` обвива тези открити схеми и ги прави достъпни за LangChain4j
4. **Решение на LLM:** Когато се извика `FileAgent.readFile(path)`, LangChain4j изпраща системното съобщение, потребителското съобщение, **и списъка със схеми на инструментите** към LLM. LLM прочита описанията на инструментите и генерира извикване на инструмент (например `read_file(path="/some/file.txt")`)
5. **Изпълнение:** LangChain4j прихваща извикването на инструмента, маршрутизира го през MCP клиента обратно до Node.js подпроцеса, получава резултата и го подава обратно на LLM

Това е същият механизъм за [Откриване на инструменти](../../../05-mcp), описан по-горе, но прилаган специално към работния процес на агента. Анотациите `@SystemMessage` и `@UserMessage` насочват поведението на LLM, докато инжектирания `ToolProvider` му дава **възможностите** — LLM свързва двете по време на изпълнение.

#### Стратегии за отговор

Когато конфигурирате `SupervisorAgent`, вие посочвате как трябва да формулира крайния си отговор към потребителя след като под-агентите приключат задачите си. Диаграмата по-долу показва наличните три стратегии — LAST връща директно изхода на последния агент, SUMMARY синтезира всички изходи чрез LLM, а SCORED избира този с по-висока оценка спрямо първоначалната заявка:

<img src="../../../translated_images/bg/response-strategies.3d0cea19d096bdf9.webp" alt="Response Strategies" width="800"/>

*Три стратегии за как супервайзорът формулира крайния отговор — изберете според това дали искате изхода на последния агент, синтезиран обобщен отговор или опцията с най-висок резултат.*

Наличните стратегии са:

| Стратегия | Описание |
|----------|-------------|
| **LAST** | Супервайзорът връща изхода на последния извикан под-агент или инструмент. Това е полезно, когато крайният агент в работния процес е специално проектиран да произведе пълен, окончателен отговор (например „Агент за обобщение“ в изследователски процес). |
| **SUMMARY** | Супервайзорът използва собствен вътрешен езиков модел (LLM), за да синтезира обобщение на цялото взаимодействие и всички изходи на под-агентите, след което връща това обобщение като краен отговор. Това осигурява чист и агрегиращ отговор към потребителя. |
| **SCORED** | Системата използва вътрешен LLM, който оценява както отговора LAST, така и обобщението SUMMARY спрямо оригиналната заявка на потребителя, връщайки изхода с по-висока оценка. |

#### Обяснение на функциите на Agentic модула

Примерът демонстрира няколко усъвършенствани функции на agentic модула. Нека разгледаме по-отблизо Agentic Scope и Agent Listeners.

**Agentic Scope** показва споделената памет, където агентите съхраняват резултатите си чрез `@Agent(outputKey="...")`. Това позволява:  
- По-късните агенти да достъпват резултатите на по-ранните агенти  
- Супервайзорът да синтезира краен отговор  
- Вас да инспектирате какво е генерирал всеки агент

Диаграмата по-долу показва как Agentic Scope функционира като споделена памет във workflow-а файл-към-доклад — FileAgent записва изхода си под ключ `fileContent`, ReportAgent чете това и записва своя изход под `report`:

<img src="../../../translated_images/bg/agentic-scope.95ef488b6c1d02ef.webp" alt="Agentic Scope Shared Memory" width="800"/>

*Agentic Scope служи като споделена памет — FileAgent записва `fileContent`, ReportAgent го чете и записва `report`, а вашият код чете крайния резултат.*

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // Сурови файлови данни от FileAgent
String report = scope.readState("report");            // Структуриран доклад от ReportAgent
```
  
**Agent Listeners** позволяват наблюдение и отстраняване на грешки при изпълнението на агентите. Пошаговият изход, който виждате в демото, идва от AgentListener, който се включва във всяко извикване на агент:  
- **beforeAgentInvocation** - Изпълнява се когато Супервайзорът избере агент, позволяващ ви да видите кой агент е избран и защо  
- **afterAgentInvocation** - Изпълнява се когато агентът завърши, показва резултата му  
- **inheritedBySubagents** - Когато е истина, слушателят следи всички агенти в йерархията

Следващата диаграма показва целия жизнен цикъл на Agent Listener, включително как `onError` обработва грешки по време на изпълнение на агент:

<img src="../../../translated_images/bg/agent-listeners.784bfc403c80ea13.webp" alt="Agent Listeners Lifecycle" width="800"/>

*Agent Listeners се включват в жизнения цикъл на изпълнение — следят кога агентите започват, завършват или срещат грешки.*

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // Разпространява към всички подагенти
    }
};
```
  
Освен модела Supervisor, модулът `langchain4j-agentic` предлага няколко мощни варианта за работни потоци. Диаграмата по-долу показва всички пет — от прости последователни канали до одобрение с човешко участие:

<img src="../../../translated_images/bg/workflow-patterns.82b2cc5b0c5edb22.webp" alt="Agent Workflow Patterns" width="800"/>

*Пет модела на работен процес за оркестрация на агенти — от прости последователни канали до одобрение с човешко участие.*

| Модел | Описание | Пример за употреба |
|---------|-------------|----------|
| **Последователен** | Изпълнява агенти по ред, изходът тече към следващия | Канали: изследване → анализ → доклад |
| **Паралелен** | Стартира агенти едновременно | Независими задачи: време + новини + акции |
| **Цикъл** | Итерация докато условието е изпълнено | Оценка на качество: усъвършенстване докато резултат ≥ 0.8 |
| **Условен** | Маршрутизира според условия | Класифицира → маршрутизира към специалист агент |
| **Човек в цикъла** | Добавя човешки контролни точки | Одобрение, преглед на съдържание |

## Ключови концепции

След като разгледахте MCP и agentic модула на практика, нека обобщим кога да използвате всеки подход.

Едно от най-големите предимства на MCP е неговата растяща екосистема. Диаграмата по-долу показва как един универсален протокол свързва вашето AI приложение с разнообразие от MCP сървъри — от достъп до файлови системи и бази данни до GitHub, имейл, уеб скрапинг и други:

<img src="../../../translated_images/bg/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="MCP Ecosystem" width="800"/>

*MCP създава екосистема на универсален протокол — всеки MCP съвместим сървър работи с всеки MCP съвместим клиент, позволявайки споделяне на инструменти между приложения.*

**MCP** е идеален, когато искате да използвате съществуващи екосистеми от инструменти, да изграждате инструменти, които могат да се споделят между приложения, да интегрирате трети услуги със стандартни протоколи или да сменяте реализации на инструменти без промяна на кода.

**Agentic модулът** работи най-добре, когато искате декларативни дефиниции на агенти с анотации `@Agent`, имате нужда от оркестрация на работни потоци (последователна, цикъл, паралелна), предпочитате интерфейсно-базиран дизайн на агенти вместо императивен код или комбинирате няколко агента, които споделят изходи чрез `outputKey`.

**Supervisor Agent моделът** блести, когато работният процес не е предвидим предварително и искате LLM да реши, когато имате няколко специализирани агента с динамична оркестрация, при изграждане на разговорни системи с маршрутизация към различни способности или когато искате най-гъвкавото, адаптивно поведение на агент.

За да ви помогнем да изберете между персонализирани методи `@Tool` от Модул 04 и MCP инструменти от този модул, следната сравнителна таблица подчертава ключовите компромиси — потребителските инструменти предлагат стегнато свързване и пълна типова безопасност за специфична логика, докато MCP инструментите предоставят стандартизирани, преизползваеми интеграции:

<img src="../../../translated_images/bg/custom-vs-mcp-tools.c4f9b6b1cb65d8a1.webp" alt="Custom Tools vs MCP Tools" width="800"/>

*Кога да използваме персонализирани методи @Tool срещу MCP инструменти — персонализирани за специфична логика с пълна типова безопасност, MCP за стандартизирани интеграции между приложения.*
