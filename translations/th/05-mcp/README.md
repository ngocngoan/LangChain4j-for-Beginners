## สิ่งที่คุณจะได้เรียนรู้

คุณได้สร้าง AI แบบสนทนา เรียนรู้การใช้งานพรอมต์อย่างชำนาญ ตอบสนองอย่างมีฐานข้อมูลเอกสาร และสร้างเอเยนต์ที่มีเครื่องมือ แต่เครื่องมือเหล่านั้นล้วนถูกสร้างขึ้นเองสำหรับแอปพลิเคชันเฉพาะของคุณ แล้วถ้าให้ AI ของคุณเข้าถึงระบบนิเวศเครื่องมือมาตรฐานที่ใครก็สามารถสร้างและแชร์ได้ล่ะ? ในโมดูลนี้ คุณจะได้เรียนรู้วิธีทำเช่นนั้นกับโปรโตคอลบริบทของโมเดล (MCP) และโมดูลเอเยนต์ของ LangChain4j เราจะเริ่มด้วยการแสดงตัวอย่างการอ่านไฟล์ MCP อย่างง่าย จากนั้นจะแสดงให้เห็นว่ามันเข้ากันได้ง่ายกับเวิร์กโฟลว์เอเยนต์ขั้นสูงโดยใช้รูปแบบเอเยนต์ผู้ควบคุม

## MCP คืออะไร?

โปรโตคอลบริบทของโมเดล (MCP) ให้สิ่งนั้นอย่างแท้จริง — วิธีมาตรฐานสำหรับแอปพลิเคชัน AI ในการค้นหาและใช้เครื่องมือภายนอก แทนที่จะเขียนการรวมระบบแบบกำหนดเองสำหรับแต่ละแหล่งข้อมูลหรือบริการ คุณจะเชื่อมต่อกับเซิร์ฟเวอร์ MCP ที่เปิดเผยคุณสมบัติในรูปแบบที่สม่ำเสมอ จากนั้นเอเยนต์ AI ของคุณสามารถค้นหาและใช้เครื่องมือเหล่านี้โดยอัตโนมัติ

<img src="../../../translated_images/th/mcp-comparison.9129a881ecf10ff5.webp" alt="MCP Comparison" width="800"/>

*ก่อน MCP: การรวมระบบที่ซับซ้อนแบบจุดต่อจุด หลัง MCP: โปรโตคอลเดียว ความเป็นไปได้ไม่มีที่สิ้นสุด*

MCP แก้ปัญหาพื้นฐานในพัฒนาการ AI: การรวมระบบทุกอย่างเป็นแบบกำหนดเอง ต้องการเข้าถึง GitHub? โค้ดกำหนดเอง ต้องการอ่านไฟล์? โค้ดกำหนดเอง ต้องการถามฐานข้อมูล? โค้ดกำหนดเอง และไม่มีการรวมระบบใดทำงานร่วมกับแอปพลิเคชัน AI อื่นได้

MCP มาตรฐานสิ่งนี้ เซิร์ฟเวอร์ MCP เปิดเผยเครื่องมือพร้อมคำอธิบายที่ชัดเจนและโครงสร้างข้อมูล ลูกค้า MCP ใด ๆ ก็เชื่อมต่อ ค้นหาเครื่องมือที่มีและใช้ได้ สร้างครั้งเดียว ใช้ได้ทุกที่

<img src="../../../translated_images/th/mcp-architecture.b3156d787a4ceac9.webp" alt="MCP Architecture" width="800"/>

*สถาปัตยกรรมโปรโตคอลบริบทของโมเดล — ค้นหาและเรียกใช้เครื่องมือแบบมาตรฐาน*

## MCP ทำงานอย่างไร

<img src="../../../translated_images/th/mcp-protocol-detail.01204e056f45308b.webp" alt="MCP Protocol Detail" width="800"/>

*การทำงานของ MCP ภายใต้พื้นผิว — ลูกค้าค้นหาเครื่องมือ แลกเปลี่ยนข้อความ JSON-RPC และดำเนินการผ่านชั้นการขนส่ง*

**สถาปัตยกรรมแบบเซิร์ฟเวอร์-ลูกค้า**

MCP ใช้โมเดลลูกค้า-เซิร์ฟเวอร์ เซิร์ฟเวอร์ให้บริการเครื่องมือ — อ่านไฟล์ ถามฐานข้อมูล เรียก API ลูกค้า (แอปพลิเคชัน AI ของคุณ) เชื่อมต่อกับเซิร์ฟเวอร์และใช้เครื่องมือเหล่านั้น

ในการใช้ MCP กับ LangChain4j ให้เพิ่มการพึ่งพานี้ใน Maven:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```

**การค้นหาเครื่องมือ**

เมื่อไคลเอนต์ของคุณเชื่อมต่อกับเซิร์ฟเวอร์ MCP จะถามว่า "คุณมีเครื่องมืออะไรบ้าง?" เซิร์ฟเวอร์ตอบด้วยรายการเครื่องมือที่ใช้ได้แต่ละเครื่องพร้อมคำอธิบายและโครงสร้างพารามิเตอร์ เอเยนต์ AI ของคุณจึงตัดสินใจได้ว่าจะใช้เครื่องมือใดตามคำขอของผู้ใช้

<img src="../../../translated_images/th/tool-discovery.07760a8a301a7832.webp" alt="MCP Tool Discovery" width="800"/>

*AI ค้นพบเครื่องมือที่มีอยู่ตอนเริ่มต้น — ตอนนี้รู้ว่ามีความสามารถอะไรบ้างและตัดสินใจได้ว่าจะใช้ตัวไหน*

**กลไกการขนส่ง**

MCP รองรับกลไกการขนส่งที่แตกต่างกัน โมดูลนี้สาธิตการขนส่ง Stdio สำหรับกระบวนการในเครื่อง:

<img src="../../../translated_images/th/transport-mechanisms.2791ba7ee93cf020.webp" alt="Transport Mechanisms" width="800"/>

*กลไกการขนส่ง MCP: HTTP สำหรับเซิร์ฟเวอร์ระยะไกล, Stdio สำหรับกระบวนการในเครื่อง*

**Stdio** - [StdioTransportDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java)

สำหรับกระบวนการในเครื่อง แอปพลิเคชันของคุณจะสปอนเซิร์ฟเวอร์เป็น subprocess และสื่อสารผ่านอินพุต/เอาต์พุตมาตรฐาน เหมาะสำหรับการเข้าถึงระบบไฟล์หรือเครื่องมือคำสั่งบรรทัดคำสั่ง

```java
McpTransport stdioTransport = new StdioMcpTransport.Builder()
    .command(List.of(
        npmCmd, "exec",
        "@modelcontextprotocol/server-filesystem@2025.12.18",
        resourcesDir
    ))
    .logEvents(false)
    .build();
```

*การขนส่ง Stdio ในการทำงาน — แอปพลิเคชันของคุณสปอนเซิร์ฟเวอร์ MCP เป็นกระบวนการลูกและสื่อสารผ่านท่อ stdin/stdout*

## โมดูลเอเยนต์

แม้ว่า MCP จะให้เครื่องมือมาตรฐาน LangChain4j's **โมดูลเอเยนต์** เสนอวิธีการแบบประกาศเพื่อสร้างเอเยนต์ที่ประสานงานเครื่องมือเหล่านั้น การระบุ `@Agent` และ `AgenticServices` ช่วยให้คุณกำหนดพฤติกรรมเอเยนต์ผ่านอินเทอร์เฟซแทนโค้ดคำสั่ง

ในโมดูลนี้ คุณจะสำรวจรูปแบบ **เอเยนต์ผู้ควบคุม** — แนวทาง AI เอเยนต์ขั้นสูงที่เอเยนต์ "ผู้ควบคุม" ตัดสินใจแบบไดนามิกว่าจะเรียกใช้เอเยนต์ย่อยใดตามคำขอผู้ใช้ เราจะรวมแนวคิดทั้งสองโดยให้เอเยนต์ย่อยคนหนึ่งมีความสามารถด้านการเข้าถึงไฟล์ด้วย MCP

ในการใช้โมดูลเอเยนต์ ให้เพิ่มการพึ่งพานี้ใน Maven:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```

> **⚠️ ทดลอง:** โมดูล `langchain4j-agentic` เป็นโมดูล **ทดลอง** และอาจมีการเปลี่ยนแปลง วิธีที่เสถียรในการสร้างผู้ช่วย AI คือ `langchain4j-core` กับเครื่องมือกำหนดเอง (โมดูล 04)

### เอเยนต์ผู้ควบคุม

**รูปแบบเอเยนต์ผู้ควบคุม** เป็นรูปแบบ AI แบบเอเยนต์ที่ **ยืดหยุ่น** ผู้ควบคุมใช้ LLM เพื่อกำหนดเองว่าเอเยนต์ใดจะถูกเรียกใช้ตามคำขอของผู้ใช้ ในตัวอย่างถัดไป เราจะรวมการเข้าถึงไฟล์ด้วย MCP กับเอเยนต์ LLM เพื่อสร้างเวิร์กโฟลว์อ่านไฟล์ → รายงานที่มีการควบคุม

ในเดโม, `FileAgent` อ่านไฟล์โดยใช้เครื่องมือระบบไฟล์ MCP และ `ReportAgent` สร้างรายงานโครงสร้างพร้อมบทสรุปผู้บริหาร (1 ประโยค), 3 ประเด็นสำคัญ, และคำแนะนำ ผู้ควบคุมจัดการเวิร์กโฟลว์นี้โดยอัตโนมัติ:

<img src="../../../translated_images/th/supervisor-agent-pattern.06275a41ae006ac8.webp" alt="Supervisor Agent Pattern" width="800"/>

*ผู้ควบคุมใช้ LLM ของตนเพื่อตัดสินใจว่าเอเยนต์ใดจะถูกเรียกและตามลำดับใด — ไม่ต้องกำหนดเส้นทางแบบตายตัว*

นี่คือเวิร์กโฟลว์ที่ชัดเจนสำหรับสายงานไฟล์สู่รายงานของเรา:

<img src="../../../translated_images/th/file-report-workflow.649bb7a896800de9.webp" alt="File to Report Workflow" width="800"/>

*FileAgent อ่านไฟล์ผ่านเครื่องมือ MCP แล้ว ReportAgent แปลงเนื้อหาดิบเป็นรายงานที่มีโครงสร้าง*

เอเยนต์แต่ละตัวเก็บผลลัพธ์ไว้ใน **Agentic Scope** (หน่วยความจำร่วม) ทำให้เอเยนต์ที่ตามมาติดต่อเข้าถึงผลลัพธ์ก่อนหน้าได้ นี่แสดงให้เห็นว่าเครื่องมือ MCP รวมเข้ากับเวิร์กโฟลว์เอเยนต์ได้อย่างไร — ผู้ควบคุมไม่จำเป็นต้องรู้ *วิธี* อ่านไฟล์ เพียงแค่รู้ว่า `FileAgent` ทำได้

#### การรันเดโม

สคริปต์เริ่มต้นจะโหลดตัวแปรสภาพแวดล้อมจากไฟล์ `.env` ที่อยู่รากโดยอัตโนมัติ:

**Bash:**
```bash
cd 05-mcp
chmod +x start-supervisor.sh
./start-supervisor.sh
```

**PowerShell:**
```powershell
cd 05-mcp
.\start-supervisor.ps1
```

**ใช้ VS Code:** คลิกขวาที่ `SupervisorAgentDemo.java` แล้วเลือก **"Run Java"** (ตรวจสอบว่าไฟล์ `.env` ตั้งค่าแล้ว)

#### การทำงานของผู้ควบคุม

```java
// ขั้นตอนที่ 1: FileAgent อ่านไฟล์โดยใช้เครื่องมือ MCP
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // มีเครื่องมือ MCP สำหรับการดำเนินการไฟล์
        .build();

// ขั้นตอนที่ 2: ReportAgent สร้างรายงานที่มีโครงสร้าง
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// Supervisor ประสานงานกระบวนการทำงานไฟล์ → รายงาน
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // ส่งคืนรายงานสุดท้าย
        .build();

// Supervisor ตัดสินใจเรียกใช้งานเอเจนต์ตามคำขอ
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```

#### กลยุทธ์การตอบกลับ

เมื่อคุณกำหนดค่า `SupervisorAgent` คุณระบุวิธีที่ควรจัดรูปแบบคำตอบสุดท้ายให้ผู้ใช้หลังจากเอเยนต์ย่อยทำงานเสร็จสิ้น

<img src="../../../translated_images/th/response-strategies.3d0cea19d096bdf9.webp" alt="Response Strategies" width="800"/>

*สามกลยุทธ์สำหรับการจัดรูปแบบคำตอบสุดท้ายของผู้ควบคุม — เลือกตามว่าคุณต้องการผลลัพธ์จากเอเยนต์ตัวสุดท้าย, สรุปที่สังเคราะห์, หรือทางเลือกที่ได้คะแนนดีที่สุด*

กลยุทธ์ที่มีได้แก่:

| กลยุทธ์ | คำอธิบาย |
|----------|-------------|
| **LAST** | ผู้ควบคุมส่งกลับผลลัพธ์ของเอเยนต์หรือเครื่องมือย่อยตัวสุดท้าย ซึ่งมีประโยชน์เมื่อเอเยนต์สุดท้ายในเวิร์กโฟลว์ถูกออกแบบมาเฉพาะเพื่อให้คำตอบสุดท้ายที่ครบถ้วน (เช่น "เอเยนต์สรุป" ในสายงานวิจัย) |
| **SUMMARY** | ผู้ควบคุมใช้โมเดลภาษา (LLM) ภายในของตนสังเคราะห์บทสรุปของการโต้ตอบทั้งหมดและผลลัพธ์ของเอเยนต์ย่อยทั้งหมด จากนั้นส่งบทสรุปนั้นเป็นคำตอบสุดท้าย ให้คำตอบที่ชัดเจนและรวบยอดสำหรับผู้ใช้ |
| **SCORED** | ระบบใช้ LLM ภายในเพื่อให้คะแนนทั้งคำตอบสุดท้าย (LAST) และบทสรุป (SUMMARY) ของการโต้ตอบเทียบกับคำขอดั้งเดิมของผู้ใช้ แล้วส่งออกผลลัพธ์ที่ได้คะแนนสูงกว่า |

ดู [SupervisorAgentDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/SupervisorAgentDemo.java) สำหรับการดำเนินการครบถ้วน

> - "ความแตกต่างระหว่างรูปแบบผู้ควบคุมและเวิร์กโฟลว์ต่อเนื่องคืออะไร?"
> - "ฉันจะปรับแต่งพฤติกรรมการวางแผนของผู้ควบคุมได้อย่างไร?"

#### ทำความเข้าใจกับผลลัพธ์

เมื่อรันเดโม คุณจะเห็นการแสดงผลแบบเป็นขั้นตอนของวิธีที่ผู้ควบคุมประสานงานเอเยนต์หลายตัว นี่คือความหมายของแต่ละส่วน:

```
======================================================================
  FILE → REPORT WORKFLOW DEMO
======================================================================

This demo shows a clear 2-step workflow: read a file, then generate a report.
The Supervisor orchestrates the agents automatically based on the request.
```

**ส่วนหัว** แนะนำแนวคิดเวิร์กโฟลว์: สายงานมุ่งเน้นตั้งแต่การอ่านไฟล์ไปจนถึงการสร้างรายงาน

```
--- WORKFLOW ---------------------------------------------------------
  ┌─────────────┐      ┌──────────────┐
  │  FileAgent  │ ───▶ │ ReportAgent  │
  │ (MCP tools) │      │  (pure LLM)  │
  └─────────────┘      └──────────────┘
   outputKey:           outputKey:
   'fileContent'        'report'

--- AVAILABLE AGENTS -------------------------------------------------
  [FILE]   FileAgent   - Reads files via MCP → stores in 'fileContent'
  [REPORT] ReportAgent - Generates structured report → stores in 'report'
```

**แผนผังเวิร์กโฟลว์** แสดงการไหลของข้อมูลระหว่างเอเยนต์ เอเยนต์แต่ละตัวมีบทบาทเฉพาะ:
- **FileAgent** อ่านไฟล์โดยใช้เครื่องมือ MCP และเก็บเนื้อหาดิบใน `fileContent`
- **ReportAgent** ใช้เนื้อหานั้นและสร้างรายงานที่มีโครงสร้างใน `report`

```
--- USER REQUEST -----------------------------------------------------
  "Read the file at .../file.txt and generate a report on its contents"
```

**คำขอของผู้ใช้** แสดงงาน ผู้ควบคุมแปลงคำขอนี้และตัดสินใจเรียกใช้ FileAgent → ReportAgent

```
--- SUPERVISOR ORCHESTRATION -----------------------------------------
  The Supervisor decides which agents to invoke and passes data between them...

  +-- STEP 1: Supervisor chose -> FileAgent (reading file via MCP)
  |
  |   Input: .../file.txt
  |
  |   Result: LangChain4j is an open-source, provider-agnostic Java framework for building LLM...
  +-- [OK] FileAgent (reading file via MCP) completed

  +-- STEP 2: Supervisor chose -> ReportAgent (generating structured report)
  |
  |   Input: LangChain4j is an open-source, provider-agnostic Java framew...
  |
  |   Result: Executive Summary...
  +-- [OK] ReportAgent (generating structured report) completed
```

**การประสานงานของผู้ควบคุม** แสดงการไหลแบบ 2 ขั้นตอน:
1. **FileAgent** อ่านไฟล์ผ่าน MCP และเก็บเนื้อหา
2. **ReportAgent** รับเนื้อหาและสร้างรายงานที่มีโครงสร้าง

ผู้ควบคุมตัดสินใจ **โดยอัตโนมัติ** ตามคำขอของผู้ใช้

```
--- FINAL RESPONSE ---------------------------------------------------
Executive Summary
...

Key Points
...

Recommendations
...

--- AGENTIC SCOPE (Data Flow) ----------------------------------------
  Each agent stores its output for downstream agents to consume:
  * fileContent: LangChain4j is an open-source, provider-agnostic Java framework...
  * report: Executive Summary...
```

#### คำอธิบายฟีเจอร์ของโมดูลเอเยนต์

ตัวอย่างนี้แสดงฟีเจอร์ขั้นสูงของโมดูลเอเยนต์ มาดูใกล้ ๆ ที่ Agentic Scope และ Agent Listeners กัน

**Agentic Scope** แสดงหน่วยความจำร่วมที่เอเยนต์เก็บผลลัพธ์โดยใช้ `@Agent(outputKey="...")` ซึ่งช่วยให้:
- เอเยนต์ภายหลังเข้าถึงผลลัพธ์ของเอเยนต์ก่อนหน้า
- ผู้ควบคุมสังเคราะห์คำตอบสุดท้าย
- คุณตรวจสอบสิ่งที่แต่ละเอเยนต์สร้างขึ้นได้

<img src="../../../translated_images/th/agentic-scope.95ef488b6c1d02ef.webp" alt="Agentic Scope Shared Memory" width="800"/>

*Agentic Scope ทำหน้าที่เป็นหน่วยความจำร่วม — FileAgent เขียน `fileContent`, ReportAgent อ่านแล้วเขียน `report`, โค้ดของคุณอ่านผลลัพธ์สุดท้าย*

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // ข้อมูลไฟล์ดิบจาก FileAgent
String report = scope.readState("report");            // รายงานที่มีโครงสร้างจาก ReportAgent
```

**Agent Listeners** ช่วยติดตามและดีบักการทำงานของเอเยนต์ การแสดงผลทีละขั้นตอนที่คุณเห็นในเดโมมาจาก AgentListener ที่เชื่อมโยงกับการเรียกใช้แต่ละเอเยนต์:
- **beforeAgentInvocation** - ถูกเรียกเมื่อ Supervisor เลือกตัวแทน ทำให้คุณเห็นว่าตัวแทนใดถูกเลือกและเหตุผล
- **afterAgentInvocation** - ถูกเรียกเมื่อมีตัวแทนทำงานเสร็จ แสดงผลลัพธ์ของตัวแทน
- **inheritedBySubagents** - เมื่อเป็นจริง ตัวฟังจะติดตามตัวแทนทั้งหมดในลำดับชั้น

<img src="../../../translated_images/th/agent-listeners.784bfc403c80ea13.webp" alt="วงจรชีวิตของ Agent Listeners" width="800"/>

*Agent Listeners เชื่อมต่อกับวงจรชีวิตการดำเนินการ — ตรวจสอบเมื่อตัวแทนเริ่มทำงาน เสร็จสิ้น หรือพบข้อผิดพลาด*

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // กระจายไปยังตัวแทนย่อยทั้งหมด
    }
};
```

นอกเหนือจากรูปแบบ Supervisor โมดูล `langchain4j-agentic` ยังมีรูปแบบและฟีเจอร์เวิร์กโฟลว์ที่ทรงพลังหลายอย่าง:

<img src="../../../translated_images/th/workflow-patterns.82b2cc5b0c5edb22.webp" alt="รูปแบบเวิร์กโฟลว์ของ Agent" width="800"/>

*ห้ารูปแบบเวิร์กโฟลว์สำหรับจัดการตัวแทน — ตั้งแต่พายป์ไลน์แบบลำดับง่ายๆ จนถึงเวิร์กโฟลว์การอนุมัติแบบมีมนุษย์ควบคุม*

| รูปแบบ | คำอธิบาย | กรณีการใช้งาน |
|---------|-------------|----------|
| **ลำดับ** | ดำเนินการตัวแทนตามลำดับ ผลลัพธ์ส่งต่อไปยังตัวถัดไป | พายป์ไลน์: วิจัย → วิเคราะห์ → รายงาน |
| **ขนาน** | รันตัวแทนพร้อมกัน | งานอิสระ: สภาพอากาศ + ข่าว + หุ้น |
| **ลูป** | ทำซ้ำจนกว่าจะถึงเงื่อนไข | ให้คะแนนคุณภาพ: ปรับปรุงจนคะแนน ≥ 0.8 |
| **ตามเงื่อนไข** | ส่งเส้นทางตามเงื่อนไข | จำแนก → ส่งต่อไปยังตัวแทนผู้เชี่ยวชาญ |
| **มีมนุษย์ควบคุม** | เพิ่มจุดตรวจสอบโดยมนุษย์ | เวิร์กโฟลว์การอนุมัติ, ตรวจสอบเนื้อหา |

## แนวคิดสำคัญ

ตอนนี้ที่คุณได้ศึกษารูปแบบ MCP และโมดูล agentic ในการใช้งานแล้ว มาสรุปกันว่าควรใช้แต่ละวิธีเมื่อไหร่

<img src="../../../translated_images/th/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="ระบบนิเวศ MCP" width="800"/>

*MCP สร้างระบบนิเวศโปรโตคอลสากล — เซิร์ฟเวอร์ที่เข้ากันได้กับ MCP ใดก็ได้สามารถทำงานร่วมกับไคลเอนต์ MCP ใดก็ได้ ช่วยให้เครื่องมือสามารถแชร์ข้ามแอปพลิเคชัน*

**MCP** เหมาะอย่างยิ่งเมื่อคุณต้องการใช้ประโยชน์จากระบบเครื่องมือที่มีอยู่แล้ว สร้างเครื่องมือที่หลายแอปพลิเคชันสามารถใช้ร่วมกัน รวมบริการบุคคลที่สามด้วยโปรโตคอลมาตรฐาน หรือเปลี่ยนการใช้งานเครื่องมือโดยไม่ต้องแก้ไขโค้ด

**โมดูล Agentic** เหมาะเมื่อคุณต้องการการกำหนดตัวแทนแบบประกาศโดยใช้ `@Agent` annotations ต้องการการจัดการเวิร์กโฟลว์ (ลำดับ, ลูป, ขนาน) ชอบการออกแบบตัวแทนแบบอินเทอร์เฟซมากกว่าการเขียนโค้ดคำสั่ง หรือต้องรวมตัวแทนหลายตัวที่แชร์ผลลัพธ์ผ่าน `outputKey`

**รูปแบบ Supervisor Agent** ส่องประกายเมื่องานไม่สามารถคาดเดาได้ล่วงหน้าและคุณต้องการให้ LLM ตัดสินใจ เมื่อคุณมีตัวแทนเฉพาะทางหลายตัวที่ต้องจัดการแบบไดนามิก เมื่อสร้างระบบสนทนาที่ส่งต่อไปยังความสามารถต่างๆ หรือเมื่อคุณต้องการพฤติกรรมตัวแทนที่ยืดหยุ่นและปรับตัวได้ดีที่สุด

<img src="../../../translated_images/th/custom-vs-mcp-tools.c4f9b6b1cb65d8a1.webp" alt="เครื่องมือกำหนดเองเทียบกับเครื่องมือ MCP" width="800"/>

*เมื่อใดควรใช้วิธี @Tool กำหนดเองเทียบกับเครื่องมือ MCP — เครื่องมือกำหนดเองสำหรับตรรกะเฉพาะแอปมีความปลอดภัยประเภทเต็มที่ เครื่องมือ MCP สำหรับการรวมมาตรฐานที่ทำงานข้ามแอปพลิเคชันได้*
