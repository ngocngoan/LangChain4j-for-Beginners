## สิ่งที่คุณจะได้เรียนรู้

คุณได้สร้าง AI ที่แชทได้ เชี่ยวชาญการสร้าง prompt ทำให้การตอบสนองมีข้อมูลอ้างอิงในเอกสาร และสร้างเอเจนท์ที่ใช้เครื่องมือ แต่เครื่องมือเหล่านั้นถูกสร้างขึ้นเฉพาะสำหรับแอปของคุณเท่านั้น แล้วถ้าคุณสามารถให้ AI ของคุณเข้าถึงระบบนิเวศน์ของเครื่องมือที่เป็นมาตรฐานที่ใครก็สามารถสร้างและแชร์ได้ล่ะ? ในโมดูลนี้คุณจะได้เรียนรู้วิธีทำเช่นนั้นกับโปรโตคอลบริบทของโมเดล (MCP) และโมดูลเอเจนติกของ LangChain4j เราจะแสดงตัวอย่างง่ายของโปรแกรมอ่านไฟล์ MCP ก่อน และจากนั้นจะแสดงวิธีการผสานเข้าไปอย่างง่ายดายในเวิร์กโฟลว์เอเจนติกขั้นสูงโดยใช้รูปแบบ Supervisor Agent

## MCP คืออะไร?

โปรโตคอลบริบทของโมเดล (MCP) มอบสิ่งนั้นอย่างแท้จริง – วิธีมาตรฐานสำหรับแอปพลิเคชัน AI ในการค้นหาและใช้เครื่องมือภายนอก แทนการเขียนการเชื่อมต่อแบบกำหนดเองสำหรับแต่ละแหล่งข้อมูลหรือบริการ คุณจะเชื่อมต่อกับเซิร์ฟเวอร์ MCP ที่เปิดเผยความสามารถในรูปแบบที่สม่ำเสมอ จากนั้นเอเจนต์ AI ของคุณสามารถค้นหาและใช้เครื่องมือเหล่านี้โดยอัตโนมัติ

แผนภาพด้านล่างแสดงความแตกต่าง — ไม่มี MCP การเชื่อมต่อแต่ละจุดต้องเขียนการเชื่อมต่อแบบจุดต่อจุดเฉพาะตัว; กับ MCP โปรโตคอลเดียวเชื่อมแอปของคุณกับเครื่องมือใดก็ได้:

<img src="../../../translated_images/th/mcp-comparison.9129a881ecf10ff5.webp" alt="MCP Comparison" width="800"/>

*ก่อน MCP: การเชื่อมต่อแบบจุดต่อจุดซับซ้อน หลัง MCP: โปรโตคอลเดียว โอกาสไม่จำกัด*

MCP แก้ปัญหาพื้นฐานในการพัฒนา AI: ทุกการเชื่อมต่อเป็นแบบกำหนดเอง ต้องการเข้าถึง GitHub? โค้ดกำหนดเอง ต้องการอ่านไฟล์? โค้ดกำหนดเอง ต้องการคิวรีฐานข้อมูล? โค้ดกำหนดเอง และไม่มีการเชื่อมต่อเหล่านี้ทำงานร่วมกับแอป AI อื่นๆ ได้

MCP มาตรฐานสิ่งนี้ เซิร์ฟเวอร์ MCP เปิดเผยเครื่องมือพร้อมคำอธิบายและสคีมาที่ชัดเจน ลูกค้า MCP ใดๆ ก็เชื่อมต่อ ค้นหาเครื่องมือที่มี และใช้ได้ สร้างครั้งเดียว ใช้ได้ทุกที่

แผนภาพด้านล่างแสดงสถาปัตยกรรมนี้ — ลูกค้า MCP เพียงตัวเดียว (แอป AI ของคุณ) เชื่อมต่อกับเซิร์ฟเวอร์ MCP หลายตัว แต่ละตัวเปิดเผยชุดเครื่องมือของตนผ่านโปรโตคอลมาตรฐาน:

<img src="../../../translated_images/th/mcp-architecture.b3156d787a4ceac9.webp" alt="MCP Architecture" width="800"/>

*สถาปัตยกรรมโปรโตคอลบริบทของโมเดล - การค้นหาและใช้งานเครื่องมือแบบมาตรฐาน*

## MCP ทำงานอย่างไร

ภายใต้ระบบ MCP ใช้สถาปัตยกรรมแบบชั้นๆ แอป Java ของคุณ (ลูกค้า MCP) ค้นหาเครื่องมือที่มี ส่งคำขอ JSON-RPC ผ่านเลเยอร์การขนส่ง (Stdio หรือ HTTP) และเซิร์ฟเวอร์ MCP ดำเนินการปฏิบัติการและส่งผลลัพธ์กลับ แผนภาพต่อไปนี้แจกแจงแต่ละชั้นของโปรโตคอลนี้:

<img src="../../../translated_images/th/mcp-protocol-detail.01204e056f45308b.webp" alt="MCP Protocol Detail" width="800"/>

*MCP ทำงานอย่างไรเบื้องหลัง — ลูกค้าค้นหาเครื่องมือ แลกเปลี่ยนข้อความ JSON-RPC และดำเนินการผ่านเลเยอร์ขนส่ง*

**สถาปัตยกรรมเซิร์ฟเวอร์-ลูกค้า**

MCP ใช้โมเดลเซิร์ฟเวอร์-ลูกค้า เซิร์ฟเวอร์จัดหาเครื่องมือ - อ่านไฟล์ คิวรีฐานข้อมูล เรียกใช้งาน API ลูกค้า (แอป AI ของคุณ) เชื่อมต่อกับเซิร์ฟเวอร์และใช้เครื่องมือเหล่านั้น

เพื่อใช้ MCP กับ LangChain4j ให้เพิ่ม dependencies ของ Maven ดังนี้:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```

**การค้นพบเครื่องมือ**

เมื่อไคลเอนต์ของคุณเชื่อมต่อกับเซิร์ฟเวอร์ MCP มันจะถามว่า "คุณมีเครื่องมืออะไรบ้าง?" เซิร์ฟเวอร์ตอบกลับด้วยรายการเครื่องมือที่มี พร้อมคำอธิบายและสคีมาของพารามิเตอร์แต่ละตัว จากนั้นเอเจนต์ AI ของคุณก็สามารถตัดสินใจใช้เครื่องมือใดตามคำขอของผู้ใช้ได้ แผนภาพด้านล่างแสดงการจับมือกันนี้ — ลูกค้าส่งคำขอ `tools/list` และเซิร์ฟเวอร์ส่งคืนเครื่องมือที่มีพร้อมคำอธิบายและสคีมาพารามิเตอร์:

<img src="../../../translated_images/th/tool-discovery.07760a8a301a7832.webp" alt="MCP Tool Discovery" width="800"/>

*AI ค้นพบเครื่องมือที่มีในตอนเริ่มต้น — รู้ความสามารถที่มีและตัดสินใจใช้เครื่องมือไหน*

**กลไกการขนส่ง**

MCP รองรับกลไกการขนส่งหลายชนิด ตัวเลือกสองอย่างคือ Stdio (สำหรับการสื่อสารกับโปรเซสย่อยในเครื่อง) และ Streamable HTTP (สำหรับเซิร์ฟเวอร์ระยะไกล) โมดูลนี้สาธิตการขนส่งแบบ Stdio:

<img src="../../../translated_images/th/transport-mechanisms.2791ba7ee93cf020.webp" alt="Transport Mechanisms" width="800"/>

*กลไกการขนส่งของ MCP: HTTP สำหรับเซิร์ฟเวอร์ระยะไกล, Stdio สำหรับกระบวนการภายในเครื่อง*

**Stdio** - [StdioTransportDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java)

สำหรับกระบวนการภายในเครื่อง แอปของคุณเปิดเซิร์ฟเวอร์เป็นโปรเซสลูก และสื่อสารผ่าน input/output มาตรฐาน เหมาะกับการเข้าถึงระบบไฟล์หรือเครื่องมือคำสั่ง

```java
McpTransport stdioTransport = new StdioMcpTransport.Builder()
    .command(List.of(
        npmCmd, "exec",
        "@modelcontextprotocol/server-filesystem@2025.12.18",
        resourcesDir
    ))
    .logEvents(false)
    .build();
```

เซิร์ฟเวอร์ `@modelcontextprotocol/server-filesystem` เปิดเผยเครื่องมือต่อไปนี้ทั้งหมดใน sandbox ของไดเรกทอรีที่คุณระบุ:

| เครื่องมือ | คำอธิบาย |
|------|-------------|
| `read_file` | อ่านข้อมูลในไฟล์เดียว |
| `read_multiple_files` | อ่านไฟล์หลายไฟล์พร้อมกัน |
| `write_file` | สร้างหรือเขียนทับไฟล์ |
| `edit_file` | แก้ไขด้วยการค้นหาและแทนที่เป้าหมาย |
| `list_directory` | แสดงรายการไฟล์และโฟลเดอร์ในเส้นทาง |
| `search_files` | ค้นหาไฟล์อย่างลึกซึ้งโดยใช้รูปแบบ |
| `get_file_info` | ดึงข้อมูลเมต้าไฟล์ (ขนาด, เวลา, สิทธิ์) |
| `create_directory` | สร้างโฟลเดอร์ (รวมโฟลเดอร์หลัก) |
| `move_file` | ย้ายหรือเปลี่ยนชื่อไฟล์หรือโฟลเดอร์ |

แผนภาพต่อไปนี้แสดงการทำงานของการขนส่งแบบ Stdio ในขณะรัน — แอป Java ของคุณเปิดเซิร์ฟเวอร์ MCP เป็นโปรเซสลูก และสื่อสารผ่านท่อ stdin/stdout โดยไม่มีการใช้เครือข่ายหรือ HTTP:

<img src="../../../translated_images/th/stdio-transport-flow.45eaff4af2d81db4.webp" alt="Stdio Transport Flow" width="800"/>

*การขนส่ง Stdio ทำงาน — แอปของคุณเปิดเซิร์ฟเวอร์ MCP เป็นโปรเซสลูก และสื่อสารผ่านท่อ stdin/stdout*

## โมดูลเอเจนติก

แม้ MCP จะจัดหาเครื่องมือมาตรฐาน แต่โมดูลเอเจนติกของ LangChain4j ให้ทางเลือกการสร้างเอเจนต์แบบประกาศ (declarative) เพื่อจัดการเครื่องมือเหล่านั้น ตัวแอนโนเทชัน `@Agent` และ `AgenticServices` ให้คุณกำหนดพฤติกรรมเอเจนต์ผ่านอินเทอร์เฟซแทนโค้ดแบบมีคำสั่ง

ในโมดูลนี้ คุณจะได้สำรวจรูปแบบ **Supervisor Agent** — วิธีเอเจนต์ AI ขั้นสูงที่ "ผู้ควบคุม" จะตัดสินใจแบบไดนามิกว่าจะเรียกใช้เอเจนท์ใดตามคำร้องขอของผู้ใช้ เราจะรวมสองแนวคิดนี้ด้วยการให้เอเจนต์ย่อยตัวหนึ่งมีความสามารถเข้าถึงไฟล์ด้วยเครื่องมือ MCP

เพื่อใช้โมดูลเอเจนติก ให้เพิ่ม dependency Maven ดังนี้:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```
> **หมายเหตุ:** โมดูล `langchain4j-agentic` ใช้ property เวอร์ชันแยก (`langchain4j.mcp.version`) เพราะออกตามรอบเวลาที่ต่างจาก core LangChain4j

> **⚠️ กำลังทดลอง:** โมดูล `langchain4j-agentic` อยู่ในสถานะทดลองและอาจเปลี่ยนแปลง วิธีที่เสถียรสำหรับสร้างผู้ช่วย AI คือ `langchain4j-core` พร้อมเครื่องมือกำหนดเอง (โมดูล 04)

### การทำงานกับไฟล์ (Stdio)

ตัวอย่างนี้แสดงการใช้เครื่องมือผ่านโปรเซสย่อยท้องถิ่น

**✅ ไม่ต้องติดตั้งอะไรเพิ่ม** - เซิร์ฟเวอร์ MCP จะถูกเปิดขึ้นโดยอัตโนมัติ

**ใช้สคริปต์เริ่มต้น (แนะนำ):**

สคริปต์เหล่านี้โหลดตัวแปรสภาพแวดล้อมจากไฟล์ `.env` รากโดยอัตโนมัติ:

**Bash:**
```bash
cd 05-mcp
chmod +x start-stdio.sh
./start-stdio.sh
```

**PowerShell:**
```powershell
cd 05-mcp
.\start-stdio.ps1
```

**ใช้ VS Code:** คลิกขวาที่ `StdioTransportDemo.java` แล้วเลือก **"Run Java"** (ตรวจสอบว่าไฟล์ `.env` ตั้งค่าแล้ว)

### ตัวแทนผู้ควบคุม

รูปแบบ **Supervisor Agent** คือการสร้าง AI เอเจนต์ที่ **ยืดหยุ่น** ผู้ควบคุมใช้ LLM ตัดสินใจโดยอัตโนมัติว่าเรียกใช้เอเจนต์ใดตามคำร้องขอผู้ใช้ ในตัวอย่างถัดไป เราจะรวมการเข้าถึงไฟล์ด้วย MCP กับเอเจนต์ LLM เพื่อสร้างเวิร์กโฟลว์อ่านไฟล์ → รายงานที่มีผู้ควบคุมดูแล

ในเดโมนี้ `FileAgent` อ่านไฟล์โดยใช้เครื่องมือระบบไฟล์ MCP และ `ReportAgent` สร้างรายงานที่มีโครงสร้าง พร้อมสรุปผู้บริหาร (1 ประโยค) 3 ประเด็นสำคัญ และคำแนะนำ ผู้ควบคุมจัดการการไหลนี้โดยอัตโนมัติ:

#### ตัวผู้ควบคุมทำงานอย่างไร

ก่อนสร้างเอเจนต์ คุณต้องเชื่อมต่อ MCP transport กับไคลเอนต์และห่อหุ้มเป็น `ToolProvider` วิธีนี้เป็นการทำให้เครื่องมือของเซิร์ฟเวอร์ MCP พร้อมใช้กับเอเจนต์ของคุณ:

```java
// สร้างไคลเอนต์ MCP จากการส่งผ่าน
McpClient mcpClient = new DefaultMcpClient.Builder()
        .transport(stdioTransport)
        .build();

// ห่อหุ้มไคลเอนต์เป็น ToolProvider — นี่คือการเชื่อมต่อเครื่องมือ MCP เข้ากับ LangChain4j
ToolProvider mcpToolProvider = McpToolProvider.builder()
        .mcpClients(List.of(mcpClient))
        .build();
```

ตอนนี้คุณสามารถฉีด `mcpToolProvider` เข้าไปในเอเจนต์ใดก็ได้ที่ต้องใช้เครื่องมือ MCP:

```java
// ขั้นตอนที่ 1: FileAgent อ่านไฟล์โดยใช้เครื่องมือ MCP
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // มีเครื่องมือ MCP สำหรับการจัดการไฟล์
        .build();

// ขั้นตอนที่ 2: ReportAgent สร้างรายงานที่มีโครงสร้าง
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// Supervisor ประสานงานเวิร์กโฟลว์จากไฟล์ไปยังรายงาน
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // ส่งคืนรายงานสุดท้าย
        .build();

// Supervisor ตัดสินใจเรียกใช้ตัวแทนตามคำขอ
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```

#### FileAgent ค้นพบเครื่องมือ MCP อย่างไรในขณะรัน

คุณอาจสงสัย: **FileAgent รู้ได้อย่างไรว่าจะใช้เครื่องมือระบบไฟล์ npm ได้อย่างไร?** คำตอบคือมันไม่รู้ — **LLM** จะหาวิธีการนั้นในขณะรันผ่านสคีมาของเครื่องมือ
อินเทอร์เฟซ `FileAgent` คือเพียง **การกำหนด prompt** เท่านั้น ซึ่งไม่มีความรู้ที่เขียนลงโค้ดล่วงหน้าเกี่ยวกับ `read_file`, `list_directory` หรือเครื่องมือ MCP อื่น ๆ นี่คือสิ่งที่เกิดขึ้นตั้งแต่ต้นจนจบ:

1. **เซิร์ฟเวอร์เริ่มทำงาน:** `StdioMcpTransport` เรียกใช้แพ็กเกจ npm `@modelcontextprotocol/server-filesystem` เป็นกระบวนการลูก
2. **การค้นพบเครื่องมือ:** `McpClient` ส่งคำขอ JSON-RPC `tools/list` ไปยังเซิร์ฟเวอร์ ซึ่งตอบกลับด้วยชื่อเครื่องมือ คำอธิบาย และสคีมาพารามิเตอร์ (เช่น `read_file` — *"อ่านเนื้อหาเต็มของไฟล์"* — `{ path: string }`)
3. **การฉีดสคีมา:** `McpToolProvider` ห่อหุ้มสคีมาที่ค้นพบและทำให้พร้อมใช้งานสำหรับ LangChain4j
4. **การตัดสินใจของ LLM:** เมื่อเรียก `FileAgent.readFile(path)` LangChain4j จะส่งข้อความระบบ ข้อความผู้ใช้ **และรายการสคีมาของเครื่องมือ** ไปยัง LLM LLM อ่านคำอธิบายของเครื่องมือและสร้างการเรียกเครื่องมือ (เช่น `read_file(path="/some/file.txt")`)
5. **การดำเนินการ:** LangChain4j ดักจับการเรียกเครื่องมือ ส่งผ่านกลับไปยัง MCP client ให้กับกระบวนการย่อย Node.js รับผลลัพธ์ และส่งกลับไปยัง LLM

นี่คือกลไก [Tool Discovery](../../../05-mcp) เดียวกับที่อธิบายไว้ข้างต้น แต่ใช้เฉพาะในเวิร์กโฟลว์ของเอเยนต์ การทำงานของ `@SystemMessage` และ `@UserMessage` ช่วยชี้นำพฤติกรรมของ LLM ในขณะที่ `ToolProvider` ที่ฉีดเข้าไปมอบ **ความสามารถ** — LLM เชื่อมต่อทั้งสองในเวลารันไทม์

#### กลยุทธ์การตอบกลับ

เมื่อคุณกำหนดค่า `SupervisorAgent` คุณกำหนดวิธีที่มันควรสร้างคำตอบสุดท้ายให้ผู้ใช้หลังจากซับเอเยนต์ทำงานเสร็จ แผนภาพด้านล่างแสดงกลยุทธ์สามแบบที่มีอยู่ — LAST คืนค่าผลลัพธ์เอเยนต์สุดท้ายโดยตรง SUMMARY สังเคราะห์ผลลัพธ์ทั้งหมดผ่าน LLM และ SCORED เลือกผลลัพธ์ที่ได้คะแนนสูงกว่าจากคำขอเดิม:

<img src="../../../translated_images/th/response-strategies.3d0cea19d096bdf9.webp" alt="Response Strategies" width="800"/>

*สามกลยุทธ์ในการสร้างคำตอบสุดท้ายของ Supervisor — เลือกตามต้องการได้ว่าจะใช้ผลลัพธ์เอเยนต์สุดท้าย, สรุปสังเคราะห์, หรือทางเลือกที่ได้คะแนนสูงสุด*

กลยุทธ์ที่มีให้เลือกได้แก่:

| กลยุทธ์ | คำอธิบาย |
|----------|-------------|
| **LAST** | ผู้ควบคุมจะคืนค่าผลลัพธ์ของซับเอเยนต์หรืองานสุดท้ายที่ถูกเรียกใช้ มีประโยชน์เมื่อเอเยนต์สุดท้ายในเวิร์กโฟลว์ออกแบบมาเฉพาะเพื่อผลิตคำตอบสุดท้ายแบบครบถ้วน (เช่น "Summary Agent" ในกระบวนการวิจัย) |
| **SUMMARY** | ผู้ควบคุมใช้โมเดลภาษา (LLM) ภายในตนเองเพื่อสังเคราะห์สรุปจากการโต้ตอบทั้งหมดและผลลัพธ์ของซับเอเยนต์ทั้งหมด จากนั้นส่งคืนสรุปนั้นเป็นคำตอบสุดท้าย ซึ่งให้คำตอบที่ชัดเจนและรวบยอดแก่ผู้ใช้ |
| **SCORED** | ระบบใช้ LLM ภายในเพื่อให้คะแนนทั้งคำตอบ LAST และ SUMMARY ของการโต้ตอบเทียบกับคำขอผู้ใช้ดั้งเดิม จากนั้นคืนค่าผลลัพธ์ที่ได้คะแนนสูงกว่า |

#### ทำความเข้าใจกับผลลัพธ์

เมื่อคุณรันเดโม คุณจะเห็นขั้นตอนที่เป็นโครงสร้างแสดงให้เห็นว่าสุเปอร์ไวเซอร์จัดการหลายเอเยนต์อย่างไร ต่อไปนี้คือความหมายของแต่ละส่วน:

```
======================================================================
  FILE → REPORT WORKFLOW DEMO
======================================================================

This demo shows a clear 2-step workflow: read a file, then generate a report.
The Supervisor orchestrates the agents automatically based on the request.
```

**ส่วนหัว** แนะนำแนวคิดเวิร์กโฟลว์: ท่อทำงานที่เน้นตั้งแต่การอ่านไฟล์จนถึงการสร้างรายงาน

```
--- WORKFLOW ---------------------------------------------------------
  ┌─────────────┐      ┌──────────────┐
  │  FileAgent  │ ───▶ │ ReportAgent  │
  │ (MCP tools) │      │  (pure LLM)  │
  └─────────────┘      └──────────────┘
   outputKey:           outputKey:
   'fileContent'        'report'

--- AVAILABLE AGENTS -------------------------------------------------
  [FILE]   FileAgent   - Reads files via MCP → stores in 'fileContent'
  [REPORT] ReportAgent - Generates structured report → stores in 'report'
```

**แผนภาพเวิร์กโฟลว์** แสดงการไหลของข้อมูลระหว่างเอเยนต์ แต่ละเอเยนต์มีบทบาทเฉพาะ:
- **FileAgent** อ่านไฟล์โดยใช้เครื่องมือ MCP และเก็บเนื้อหาดิบใน `fileContent`
- **ReportAgent** ใช้เนื้อหานั้นและสร้างรายงานโครงสร้างใน `report`

```
--- USER REQUEST -----------------------------------------------------
  "Read the file at .../file.txt and generate a report on its contents"
```

**คำขอของผู้ใช้** แสดงงานที่ต้องทำ Supervisor วิเคราะห์คำขอแล้วตัดสินใจเรียก FileAgent → ReportAgent

```
--- SUPERVISOR ORCHESTRATION -----------------------------------------
  The Supervisor decides which agents to invoke and passes data between them...

  +-- STEP 1: Supervisor chose -> FileAgent (reading file via MCP)
  |
  |   Input: .../file.txt
  |
  |   Result: LangChain4j is an open-source, provider-agnostic Java framework for building LLM...
  +-- [OK] FileAgent (reading file via MCP) completed

  +-- STEP 2: Supervisor chose -> ReportAgent (generating structured report)
  |
  |   Input: LangChain4j is an open-source, provider-agnostic Java framew...
  |
  |   Result: Executive Summary...
  +-- [OK] ReportAgent (generating structured report) completed
```

**การจัดการของ Supervisor** แสดงการเดินทางแบบ 2 ขั้นตอน:
1. **FileAgent** อ่านไฟล์ผ่าน MCP และเก็บเนื้อหา
2. **ReportAgent** รับเนื้อหานั้นและสร้างรายงานโครงสร้าง

Supervisor ตัดสินใจเหล่านี้ได้ **โดยอิสระ** จากคำขอของผู้ใช้

```
--- FINAL RESPONSE ---------------------------------------------------
Executive Summary
...

Key Points
...

Recommendations
...

--- AGENTIC SCOPE (Data Flow) ----------------------------------------
  Each agent stores its output for downstream agents to consume:
  * fileContent: LangChain4j is an open-source, provider-agnostic Java framework...
  * report: Executive Summary...
```

#### คำอธิบายฟีเจอร์โมดูลเอเยนต์

ตัวอย่างนี้แสดงฟีเจอร์ขั้นสูงหลายประการของโมดูล agentic มาดูใกล้ ๆ ที่ Agentic Scope และ Agent Listeners

**Agentic Scope** แสดงหน่วยความจำร่วมที่เอเยนต์เก็บผลลัพธ์โดยใช้ `@Agent(outputKey="...")` ซึ่งช่วยให้:
- เอเยนต์ในลำดับถัดไปเข้าถึงผลลัพธ์จากเอเยนต์ก่อนหน้า
- Supervisor สามารถสังเคราะห์คำตอบสุดท้าย
- คุณตรวจสอบสิ่งที่เอเยนต์แต่ละตัวผลิตได้

แผนภาพด้านล่างแสดงการทำงานของ Agentic Scope เป็นหน่วยความจำร่วมในเวิร์กโฟลว์จากไฟล์ถึงรายงาน — FileAgent เขียนผลลัพธ์ภายใต้คีย์ `fileContent` แล้ว ReportAgent อ่านและเขียนผลลัพธ์ของมันภายใต้ `report`:

<img src="../../../translated_images/th/agentic-scope.95ef488b6c1d02ef.webp" alt="Agentic Scope Shared Memory" width="800"/>

*Agentic Scope ทำหน้าที่เป็นหน่วยความจำร่วม — FileAgent เขียน `fileContent`, ReportAgent อ่านมันและเขียน `report`, และโค้ดของคุณอ่านผลลัพธ์สุดท้าย*

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // ข้อมูลไฟล์ดิบจาก FileAgent
String report = scope.readState("report");            // รายงานที่มีโครงสร้างจาก ReportAgent
```

**Agent Listeners** ช่วยในการติดตามและดีบักการทำงานของเอเยนต์ ข้อมูลผลลัพธ์แบบขั้นตอนที่คุณเห็นในเดโมมาจาก AgentListener ที่เข้าสู่การเรียกใช้เอเยนต์แต่ละครั้ง:
- **beforeAgentInvocation** - เรียกใช้เมื่อ Supervisor เลือกเอเยนต์ ให้คุณเห็นว่าเอเยนต์ใดถูกเลือกและเหตุผล
- **afterAgentInvocation** - เรียกเมื่อเอเยนต์ทำงานเสร็จ แสดงผลลัพธ์ของเอเยนต์
- **inheritedBySubagents** - ถ้าเป็นจริง ตัวฟังนี้ติดตามเอเยนต์ทุกตัวในลำดับชั้น

แผนภาพด้านล่างแสดงวงจรชีวิตเต็มของ Agent Listener รวมถึงวิธีการจัดการ `onError` เมื่อเกิดความผิดพลาดระหว่างการทำงานของเอเยนต์:

<img src="../../../translated_images/th/agent-listeners.784bfc403c80ea13.webp" alt="Agent Listeners Lifecycle" width="800"/>

*Agent Listeners เข้าร่วมวงจรการทำงาน — ตรวจสอบช่วงเริ่มต้น เสร็จสิ้น หรือเกิดข้อผิดพลาดของเอเยนต์*

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // ถ่ายทอดไปยังตัวแทนลูกย่อยทั้งหมด
    }
};
```

นอกเหนือจากรูปแบบ Supervisor แล้ว โมดูล `langchain4j-agentic` ยังมีรูปแบบเวิร์กโฟลว์ทรงพลังอีกหลายแบบ แผนภาพด้านล่างแสดงรูปแบบทั้งห้า — ตั้งแต่ท่อทำงานเรียงลำดับง่าย ๆ จนถึงเวิร์กโฟลว์อนุมัติแบบมนุษย์ร่วม:

<img src="../../../translated_images/th/workflow-patterns.82b2cc5b0c5edb22.webp" alt="Agent Workflow Patterns" width="800"/>

*รูปแบบเวิร์กโฟลว์ห้ารูปแบบสำหรับประสานงานเอเยนต์ — ตั้งแต่ท่อต่อเนื่องง่าย ๆ จนถึงเวิร์กโฟลว์ที่มีมนุษย์ร่วมอนุมัติ*

| รูปแบบ | คำอธิบาย | กรณีใช้งาน |
|---------|-------------|----------|
| **Sequential** | รันเอเยนต์ตามลำดับ ผลลัพธ์ไหลไปยังตัวถัดไป | ท่อทำงาน: วิจัย → วิเคราะห์ → รายงาน |
| **Parallel** | รันเอเยนต์พร้อมกัน | งานอิสระ: สภาพอากาศ + ข่าว + หุ้น |
| **Loop** | ทำซ้ำจนกว่าจะตรงตามเงื่อนไข | การให้คะแนนคุณภาพ: ปรับจนได้คะแนน ≥ 0.8 |
| **Conditional** | เลือกเส้นทางตามเงื่อนไข | จำแนก → ส่งต่อเอเยนต์ผู้เชี่ยวชาญ |
| **Human-in-the-Loop** | เพิ่มจุดตรวจสอบโดยมนุษย์ | เวิร์กโฟลว์อนุมัติ, ตรวจทานเนื้อหา |

## แนวคิดสำคัญ

ตอนนี้ที่คุณได้สำรวจ MCP และโมดูล agentic ในการใช้งานจริงแล้ว มาสรุปเมื่อใดควรใช้แต่ละวิธี

ข้อได้เปรียบสำคัญของ MCP คือระบบนิเวศที่เติบโตขึ้น แผนภาพด้านล่างแสดงโปรโตคอลสากลที่เชื่อมแอป AI ของคุณกับเซิร์ฟเวอร์ MCP หลากหลาย — ทั้งการเข้าถึงไฟล์ระบบและฐานข้อมูล ไปจนถึง GitHub, อีเมล, การดึงข้อมูลจากเว็บ และอื่น ๆ:

<img src="../../../translated_images/th/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="MCP Ecosystem" width="800"/>

*MCP สร้างระบบนิเวศโปรโตคอลสากล — เซิร์ฟเวอร์ MCP ที่เข้ากันได้ทำงานกับไคลเอนต์ MCP ที่เข้ากันได้ ส่งเสริมการแชร์เครื่องมือข้ามแอปพลิเคชัน*

**MCP** เหมาะเมื่อคุณต้องการใช้ระบบนิเวศเครื่องมือที่มีอยู่ สร้างเครื่องมือที่หลายแอปแชร์ได้ ผสานรวมบริการของบุคคลที่สามผ่านโปรโตคอลมาตรฐาน หรือเปลี่ยนการใช้งานเครื่องมือโดยไม่ต้องแก้ไขโค้ด

**โมดูล Agentic** เหมาะอย่างยิ่งเมื่อคุณต้องการกำหนดเอเยนต์แบบประกาศด้วยคำสั่ง `@Agent` ต้องการการประสานงานเวิร์กโฟลว์ (เรียงลำดับ, ลูป, พร้อมกัน) ชอบการออกแบบเอเยนต์ผ่านอินเทอร์เฟซกว่าโค้ดเชิงคำสั่ง หรือรวมหลายเอเยนต์ที่แชร์ผลลัพธ์ผ่าน `outputKey`

**รูปแบบ Supervisor Agent** เหมาะเมื่อเวิร์กโฟลว์ไม่สามารถคาดการณ์ล่วงหน้าและคุณต้องการให้ LLM ตัดสินใจ เมื่อต้องมีหลายเอเยนต์เฉพาะทางที่ต้องประสานงานแบบไดนามิก สร้างระบบสนทนาที่จ่ายงานไปยังความสามารถต่าง ๆ หรือเมื่อต้องการพฤติกรรมเอเยนต์ที่ยืดหยุ่นและปรับตัวได้สูงสุด

เพื่อช่วยคุณตัดสินใจระหว่าง `@Tool` แบบกำหนดเองจากโมดูล 04 และเครื่องมือ MCP จากโมดูลนี้ การเปรียบเทียบด้านล่างแสดงข้อดีข้อเสียหลัก — เครื่องมือกำหนดเองทำให้มีการเชื่อมโยงแน่นและความปลอดภัยด้านชนิดครบถ้วนสำหรับตรรกะเฉพาะแอป ในขณะที่เครื่องมือ MCP เสนอการผสานรวมมาตรฐานซ้ำใช้ได้:

<img src="../../../translated_images/th/custom-vs-mcp-tools.c4f9b6b1cb65d8a1.webp" alt="Custom Tools vs MCP Tools" width="800"/>

*เมื่อใดควรใช้วิธี @Tool แบบกำหนดเองกับเครื่องมือ MCP — เครื่องมือกำหนดเองสำหรับตรรกะแอปโดยเฉพาะพร้อมความปลอดภัยชนิดเต็มที่, เครื่องมือ MCP สำหรับการผสานรวมมาตรฐานที่ทำงานข้ามแอป*
