## สิ่งที่คุณจะได้เรียนรู้

จนถึงตอนนี้ คุณได้เรียนรู้วิธีการสนทนากับ AI การจัดโครงสร้างพรอมต์อย่างมีประสิทธิภาพ และการผลักดันคำตอบให้ยึดโยงกับเอกสารของคุณ แต่ยังมีข้อจำกัดพื้นฐานอย่างหนึ่ง: โมเดลภาษา (language models) สามารถสร้างข้อความเท่านั้น ไม่สามารถตรวจสอบสภาพอากาศ คำนวณข้อมูล สืบค้นฐานข้อมูล หรือโต้ตอบกับระบบภายนอกได้

เครื่องมือเปลี่ยนแปลงสิ่งนี้ ด้วยการให้โมเดลเข้าถึงฟังก์ชันที่สามารถเรียกใช้ได้ คุณจะเปลี่ยนมันจากเครื่องสร้างข้อความเป็นตัวแทนที่สามารถดำเนินการต่าง ๆ ได้ โมเดลจะตัดสินใจว่าเมื่อไหร่ที่ต้องใช้เครื่องมือ เครื่องมือใดที่จะใช้ และพารามิเตอร์อะไรที่จะส่งไป โค้ดของคุณจะดำเนินการฟังก์ชันและส่งผลลัพธ์กลับ โมเดลจะนำผลลัพธ์นั้นเข้าไปในคำตอบของมัน

## ทำความเข้าใจตัวแทน AI พร้อมเครื่องมือ

> **📝 หมายเหตุ:** คำว่า "ตัวแทน" ในโมดูลนี้หมายถึงผู้ช่วย AI ที่เสริมความสามารถด้วยการเรียกใช้เครื่องมือ ซึ่งแตกต่างจากรูปแบบ **Agentic AI** (ตัวแทนอิสระที่มีการวางแผน ความจำ และการให้เหตุผลหลายขั้นตอน) ที่เราจะพูดถึงใน [Module 05: MCP](../05-mcp/README.md)

ถ้าไม่มีเครื่องมือ โมเดลภาษาเพียงแค่สร้างข้อความจากข้อมูลเทรนนิ่ง ถามสภาพอากาศปัจจุบัน โมเดลก็ต้องเดา ให้เครื่องมือกับโมเดล โมเดลสามารถเรียกใช้ API สภาพอากาศ คำนวณ หรือตรวจสอบฐานข้อมูล — จากนั้นนำผลลัพธ์จริงเหล่านั้นมาใส่ในคำตอบ

<img src="../../../translated_images/th/what-are-tools.724e468fc4de64da.webp" alt="ไม่ใช้เครื่องมือ vs ใช้เครื่องมือ" width="800"/>

*ถ้าไม่มีเครื่องมือ โมเดลจะเดา — แต่ถ้ามีเครื่องมือ โมเดลจะเรียก API ทำคำนวณ และส่งข้อมูลแบบเรียลไทม์*

ตัวแทน AI ที่มีเครื่องมือปฏิบัติตามรูปแบบ **Reasoning and Acting (ReAct)** โมเดลไม่ได้เพียงแค่ตอบกลับ — มันคิดว่าต้องการอะไร ลงมือด้วยการเรียกใช้เครื่องมือ สังเกตผลลัพธ์ แล้วตัดสินใจว่าจะลงมือทำอีกหรือส่งคำตอบสุดท้าย:

1. **ให้เหตุผล** — ตัวแทนวิเคราะห์คำถามของผู้ใช้และกำหนดว่าต้องการข้อมูลอะไร
2. **ลงมือทำ** — ตัวแทนเลือกเครื่องมือที่ถูกต้อง สร้างพารามิเตอร์ที่เหมาะสม และเรียกใช้
3. **สังเกต** — ตัวแทนรับผลลัพธ์จากเครื่องมือและประเมินผล
4. **ทำซ้ำหรือส่งคำตอบ** — ถ้าต้องการข้อมูลเพิ่ม ตัวแทนจะวนกลับ; ถ้าไม่ จะสร้างคำตอบเป็นภาษาธรรมชาติ

<img src="../../../translated_images/th/react-pattern-detail.96a5efeeb6dd2f61.webp" alt="รูปแบบ ReAct" width="800"/>

*วงจร ReAct — ตัวแทนคิดว่าจะทำอะไร ลงมือด้วยการเรียกเครื่องมือ สังเกตผลลัพธ์ และวนซ้ำจนกว่าจะสามารถส่งคำตอบสุดท้ายได้*

สิ่งนี้เกิดขึ้นโดยอัตโนมัติ คุณกำหนดเครื่องมือและคำอธิบาย โมเดลจัดการการตัดสินใจว่าจะใช้เมื่อไหร่และอย่างไร

## การทำงานของการเรียกใช้เครื่องมือ

### คำจำกัดความของเครื่องมือ

[WeatherTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) | [TemperatureTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/TemperatureTool.java)

คุณกำหนดฟังก์ชันด้วยคำอธิบายและสเปคพารามิเตอร์ที่ชัดเจน โมเดลเห็นคำอธิบายเหล่านี้ในพรอมต์ระบบและเข้าใจว่าแต่ละเครื่องมือทำอะไร

```java
@Component
public class WeatherTool {
    
    @Tool("Get the current weather for a location")
    public String getCurrentWeather(@P("Location name") String location) {
        // โลจิกการค้นหาสภาพอากาศของคุณ
        return "Weather in " + location + ": 22°C, cloudy";
    }
}

@AiService
public interface Assistant {
    String chat(@MemoryId String sessionId, @UserMessage String message);
}

// ผู้ช่วยถูกเชื่อมต่อโดยอัตโนมัติโดย Spring Boot ด้วย:
// - บีน ChatModel
// - เมธอด @Tool ทั้งหมดจากคลาส @Component
// - ChatMemoryProvider สำหรับการจัดการเซสชัน
```

แผนภาพด้านล่างแสดงการแยกแยะแต่ละคำอธิบายและอธิบายว่าแต่ละส่วนช่วยให้ AI เข้าใจเมื่อใดที่จะเรียกใช้เครื่องมือและส่งอาร์กิวเมนต์ใด:

<img src="../../../translated_images/th/tool-definitions-anatomy.f6468546037cf28b.webp" alt="โครงสร้างคำจำกัดความเครื่องมือ" width="800"/>

*โครงสร้างคำจำกัดความเครื่องมือ — @Tool แจ้ง AI เมื่อใช้, @P อธิบายแต่ละพารามิเตอร์ และ @AiService เชื่อมต่อทุกอย่างที่เริ่มต้น*

> - "คำอธิบายเครื่องมือที่ดีช่วยให้ AI ใช้งานอย่างถูกต้องมีลักษณะอย่างไร?"
> - "ฉันจะจัดการข้อผิดพลาด API และข้อจำกัดอัตราการเรียกใช้ในเครื่องมืออย่างไร?"

### การตัดสินใจ

เมื่อผู้ใช้ถามว่า "สภาพอากาศที่ซีแอตเทิลเป็นอย่างไร?" โมเดลจะไม่สุ่มเลือกเครื่องมือ แต่มันเปรียบเทียบเจตนาของผู้ใช้กับคำอธิบายของเครื่องมือทุกชิ้นที่มี สุ่มคะแนนความเกี่ยวข้องแล้วเลือกที่เหมาะสมที่สุด จากนั้นสร้างการเรียกฟังก์ชันที่มีพารามิเตอร์ถูกต้อง — ในกรณีนี้ตั้งค่า `location` เป็น `"Seattle"`

ถ้าไม่มีเครื่องมือใดตรงกับคำขอของผู้ใช้ โมเดลจะตอบจากความรู้ของตัวเอง หากตรงกันหลายเครื่องมือจะเลือกอันที่เฉพาะเจาะจงที่สุด

<img src="../../../translated_images/th/decision-making.409cd562e5cecc49.webp" alt="การตัดสินใจเลือกเครื่องมือของ AI" width="800"/>

*โมเดลประเมินเครื่องมือทุกชิ้นเทียบกับเจตนาของผู้ใช้และเลือกที่เหมาะสมที่สุด — เหตุผลที่คำอธิบายเครื่องมือชัดเจนและเฉพาะเจาะจงจึงสำคัญ*

### การสร้างคำตอบ

โมเดลได้รับข้อมูลสภาพอากาศและจัดรูปแบบเป็นคำตอบภาษาธรรมชาติให้กับผู้ใช้

### สถาปัตยกรรม: Spring Boot Auto-Wiring

โมดูลนี้ใช้การผสาน LangChain4j กับ Spring Boot ผ่านอินเทอร์เฟซ `@AiService` แบบประกาศ เมื่อเริ่มต้น Spring Boot จะค้นหา `@Component` ทุกอันที่มีเมธอด `@Tool` บีน `ChatModel` ของคุณ และ `ChatMemoryProvider` — จากนั้นเชื่อมต่อเข้าด้วยกันในอินเทอร์เฟซ `Assistant` เดียวโดยไม่มีบัวลอย (boilerplate)

<img src="../../../translated_images/th/spring-boot-wiring.151321795988b04e.webp" alt="สถาปัตยกรรม Spring Boot Auto-Wiring" width="800"/>

*อินเทอร์เฟซ @AiService เชื่อม ChatModel, คอมโพเนนต์เครื่องมือ และโปรไวเดอร์หน่วยความจำ — Spring Boot จะจัดการเชื่อมต่อทั้งหมดโดยอัตโนมัติ*

นี่คือวงจรคำขอเต็มรูปแบบในแผนภาพลำดับ — จากคำขอ HTTP ผ่านคอนโทรลเลอร์ เซอร์วิส และพร็อกซีที่เชื่อมต่ออัตโนมัติ จนถึงการเรียกเครื่องมือและส่งกลับ:

<img src="../../../translated_images/th/spring-boot-sequence.f83e3d485aa4a3c6.webp" alt="แผนภาพลำดับการเรียกเครื่องมือ Spring Boot" width="800"/>

*วงจรคำขอเต็มของ Spring Boot — คำขอ HTTP ไหลผ่านคอนโทรลเลอร์ เซอร์วิส ไปยังพร็อกซี Assistant ที่เชื่อมต่ออัตโนมัติ ซึ่งจัดการ LLM และการเรียกเครื่องมือโดยอัตโนมัติ*

ประโยชน์สำคัญของวิธีนี้:

- **Spring Boot auto-wiring** — ฉีด ChatModel และเครื่องมือโดยอัตโนมัติ
- **รูปแบบ @MemoryId** — จัดการหน่วยความจำแบบเซสชันอัตโนมัติ
- **อินสแตนซ์เดี่ยว** — สร้าง Assistant หนึ่งครั้งและนำกลับมาใช้ใหม่เพื่อประสิทธิภาพที่ดีขึ้น
- **การดำเนินการที่ปลอดภัยชนิดข้อมูล** — เรียกเมธอด Java ตรง ๆ พร้อมแปลงชนิดข้อมูล
- **การจัดการหลายรอบ** — จัดการการเชื่อมต่อเครื่องมืออัตโนมัติ
- **ไม่มีบัวลอย** — ไม่ต้องเรียก `AiServices.builder()` หรือใช้ HashMap จัดการหน่วยความจำด้วยตัวเอง

วิธีทางเลือก (เรียกใช้ `AiServices.builder()` ด้วยตนเอง) ต้องการโค้ดมากกว่าและขาดข้อดีของการผสานกับ Spring Boot

## การเชื่อมต่อเครื่องมือ

**การเชื่อมต่อเครื่องมือ** — พลังจริงของตัวแทนที่ใช้เครื่องมือเกิดขึ้นเมื่อคำถามเดียวต้องใช้เครื่องมือหลายตัว ถามว่า "สภาพอากาศที่ซีแอตเทิลเป็นอย่างไรในฟาเรนไฮต์?" ตัวแทนจะเชื่อมต่อเครื่องมือสองตัวโดยอัตโนมัติ: เรียก `getCurrentWeather` เพื่อรับอุณหภูมิในหน่วยเซลเซียสก่อน แล้วส่งค่านั้นไปที่ `celsiusToFahrenheit` เพื่อแปลงหน่วย — ทั้งหมดในรอบสนทนาเดียว

<img src="../../../translated_images/th/tool-chaining-example.538203e73d09dd82.webp" alt="ตัวอย่างการเชื่อมต่อเครื่องมือ" width="800"/>

*การเชื่อมต่อเครื่องมือ — ตัวแทนเรียก getCurrentWeather ก่อน จากนั้นส่งผลลัพธ์เซลเซียสไปยัง celsiusToFahrenheit และให้คำตอบรวม*

**ความล้มเหลวอย่างนุ่มนวล** — ถามสภาพอากาศในเมืองที่ไม่มีในข้อมูลจำลอง เครื่องมือจะส่งข้อความข้อผิดพลาด และ AI อธิบายว่าช่วยไม่ได้แทนที่จะล่ม เครื่องมือล้มเหลวอย่างปลอดภัย แผนภาพด้านล่างเปรียบเทียบสองแนวทาง — ด้วยการจัดการข้อผิดพลาดที่เหมาะสม ตัวแทนจับข้อยกเว้นและตอบกลับอย่างเป็นประโยชน์ ในขณะที่ถ้าไม่มี ผลแอปพลิเคชันตกทั้งระบบ:

<img src="../../../translated_images/th/error-handling-flow.9a330ffc8ee0475c.webp" alt="การไหลของการจัดการข้อผิดพลาด" width="800"/>

*เมื่อเครื่องมือล้มเหลว ตัวแทนจับข้อผิดพลาดและตอบกลับพร้อมคำอธิบายช่วยเหลือแทนที่จะให้แอปพลิเคชันล่ม*

สิ่งนี้เกิดขึ้นในรอบสนทนาเดียว ตัวแทนจัดการการเรียกเครื่องมือหลายตัวโดยอัตโนมัติ

## การใช้แอปพลิเคชัน

แอปพลิเคชันมีอินเทอร์เฟซเว็บที่คุณสามารถโต้ตอบกับตัวแทน AI ซึ่งเข้าถึงเครื่องมือพยากรณ์อากาศและแปลงอุณหภูมิได้ นี่คือลักษณะของอินเทอร์เฟซ — รวมตัวอย่างเริ่มต้นอย่างรวดเร็วและแผงแชทสำหรับส่งคำร้องขอ:

<a href="images/tools-homepage.png"><img src="../../../translated_images/th/tools-homepage.4b4cd8b2717f9621.webp" alt="AI Agent Tools Interface" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*อินเทอร์เฟซ AI Agent Tools - ตัวอย่างด่วนและอินเทอร์เฟซการแชทสำหรับโต้ตอบกับเครื่องมือ*

### ทดลองใช้เครื่องมือแบบง่าย

เริ่มด้วยคำขอที่ตรงไปตรงมา: "แปลง 100 องศาฟาเรนไฮต์เป็นเซลเซียส" ตัวแทนจะรู้ว่าต้องใช้เครื่องมือแปลงอุณหภูมิ เรียกใช้ด้วยพารามิเตอร์ที่ถูกต้อง และส่งกลับผลลัพธ์ สังเกตความเป็นธรรมชาตินี้ — คุณไม่ได้ระบุว่าจะใช้เครื่องมือไหนหรือต้องเรียกใช้อย่างไร

### ทดสอบการเชื่อมโยงเครื่องมือ

ตอนนี้ลองคำขอที่ซับซ้อนขึ้น: "สภาพอากาศที่ซีแอตเทิลและแปลงเป็นฟาเรนไฮต์หน่อย" ดูตัวแทนทำงานเป็นขั้นตอนก่อน มันดึงข้อมูลสภาพอากาศ (ซึ่งตอบกลับเป็นเซลเซียส) รู้ว่าต้องแปลงเป็นฟาเรนไฮต์ เรียกใช้เครื่องมือแปลง และรวมผลทั้งสองเข้าด้วยกันเป็นคำตอบเดียว

### ดูการไหลของบทสนทนา

อินเทอร์เฟซแชทจะเก็บประวัติการสนทนา ทำให้คุณมีปฏิสัมพันธ์หลายรอบ คุณสามารถดูคำถามและคำตอบก่อนหน้าได้ทั้งหมด ทำให้ติดตามบทสนทนาและเข้าใจว่าเอเจนต์สร้างบริบทจากหลายการแลกเปลี่ยนได้อย่างไร

<a href="images/tools-conversation-demo.png"><img src="../../../translated_images/th/tools-conversation-demo.89f2ce9676080f59.webp" alt="Conversation with Multiple Tool Calls" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*บทสนทนาหลายรอบแสดงการแปลงอุณหภูมิอย่างง่าย การค้นหาสภาพอากาศ และการเชื่อมโยงเครื่องมือ*

### ทดลองคำขอหลากหลาย

ลองชุดคำขอต่างๆ:
- ค้นหาสภาพอากาศ: "สภาพอากาศที่โตเกียวเป็นอย่างไร?"
- แปลงอุณหภูมิ: "25°C เท่ากับกี่เคลวิน?"
- คำถามผสม: "ตรวจสอบสภาพอากาศที่ปารีสและบอกฉันว่ามากกว่า 20°C หรือไม่"

สังเกตว่าเอเจนต์ตีความภาษาธรรมชาติและแมปไปยังการเรียกใช้เครื่องมือได้อย่างเหมาะสม

## แนวคิดสำคัญ

### รูปแบบ ReAct (เหตุผลและการกระทำ)

เอเจนต์สลับกันระหว่างการใช้เหตุผล (ตัดสินใจว่าจะทำอะไร) และการลงมือทำ (ใช้เครื่องมือ) รูปแบบนี้ทำให้แก้ปัญหาได้โดยอิสระไม่ใช่แค่ตอบคำสั่ง

### คำอธิบายเครื่องมือมีความสำคัญ

คุณภาพของคำอธิบายเครื่องมือส่งผลโดยตรงต่อวิธีที่เอเจนต์ใช้งานได้ดีเพียงใด คำอธิบายที่ชัดเจนและเฉพาะเจาะจงช่วยให้โมเดลเข้าใจว่าเมื่อใดและอย่างไรจะเรียกใช้เครื่องมือแต่ละตัว

### การจัดการเซสชัน

คำอธิบาย `@MemoryId` เปิดใช้งานการจัดการหน่วยความจำตามเซสชันโดยอัตโนมัติ แต่ละ session ID จะมี `ChatMemory` ของตัวเองที่จัดการโดย `ChatMemoryProvider` bean ทำให้ผู้ใช้หลายคนโต้ตอบกับเอเจนต์พร้อมกันโดยที่บทสนทนาไม่ปะปนกัน แผนภาพด้านล่างแสดงวิธีการแยกผู้ใช้หลายคนไปยังโฟลเดอร์หน่วยความจำเฉพาะตาม session ID:

<img src="../../../translated_images/th/session-management.91ad819c6c89c400.webp" alt="Session Management with @MemoryId" width="800"/>

*แต่ละ session ID แมปกับประวัติการสนทนาแยกกัน — ผู้ใช้ไม่เห็นข้อความของกันและกัน*

### การจัดการข้อผิดพลาด

เครื่องมืออาจล้มเหลว — API หมดเวลา พารามิเตอร์อาจไม่ถูกต้อง บริการภายนอกล่ม เอเจนต์ในสภาพแวดล้อมจริงต้องจัดการข้อผิดพลาดเพื่อให้โมเดลสามารถอธิบายปัญหาหรือพยายามทางเลือกอื่นแทนการแครชแอปพลิเคชันทั้งหมด เมื่อเครื่องมือขว้างข้อยกเว้น LangChain4j จะจับข้อยกเว้นและส่งข้อความแสดงข้อผิดพลาดกลับไปยังโมเดล ซึ่งสามารถอธิบายปัญหาเป็นภาษาธรรมชาติได้

## เครื่องมือกับ RAG

โมดูล 03 และ 04 ทั้งคู่ขยายความสามารถของ AI แต่มีวิธีพื้นฐานต่างกัน RAG ให้โมเดลเข้าถึง **ความรู้** โดยการดึงเอกสาร เครื่องมือทำให้โมเดลสามารถทำ **การกระทำ** โดยการเรียกใช้ฟังก์ชัน แผนภาพด้านล่างเปรียบเทียบสองวิธีนี้ข้างกัน — ตั้งแต่การทำงานของแต่ละเวิร์กโฟลว์จนถึงข้อแลกเปลี่ยนระหว่างกัน:

*RAG ดึงข้อมูลจากเอกสารคงที่ — เครื่องมือทำงานและดึงข้อมูลเรียลไทม์แบบไดนามิก ระบบจริงหลายระบบใช้ทั้งสองอย่าง*
