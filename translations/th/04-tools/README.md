## สิ่งที่คุณจะได้เรียนรู้

จนถึงตอนนี้ คุณได้เรียนรู้วิธีการสนทนากับ AI, การจัดโครงสร้างคำสั่ง prompt อย่างมีประสิทธิภาพ และการยึดคำตอบให้สอดคล้องกับเอกสารของคุณ แต่ยังมีข้อจำกัดพื้นฐานอยู่ว่า: โมเดลภาษา สามารถสร้างข้อความได้เท่านั้น มันไม่สามารถตรวจสอบสภาพอากาศ, คำนวณ, ค้นหาข้อมูลจากฐานข้อมูล, หรือโต้ตอบกับระบบภายนอกได้

เครื่องมือช่วยเปลี่ยนแปลงแบบนี้ โดยการให้โมเดลเข้าถึงฟังก์ชันที่มันสามารถเรียกใช้ได้ คุณจะเปลี่ยนมันจากเครื่องสร้างข้อความเป็นตัวแทนที่สามารถทำงานได้ โมเดลจะตัดสินใจว่าเมื่อไรที่ต้องใช้เครื่องมือ, เครื่องมือไหนที่ใช้, และส่งพารามิเตอร์อะไร ฟังก์ชันของคุณจะดำเนินการและส่งผลลัพธ์กลับมา โมเดลจะนำผลลัพธ์นั้นไปรวมในการตอบกลับ

## ความเข้าใจตัวแทน AI กับเครื่องมือ

> **📝 หมายเหตุ:** คำว่า "ตัวแทน" ในโมดูลนี้หมายถึงผู้ช่วย AI ที่ได้รับการเสริมความสามารถด้วยการเรียกใช้งานเครื่องมือ ซึ่งแตกต่างจากรูปแบบ **Agentic AI** (ตัวแทนอิสระที่มีกระบวนการวางแผน, ความจำ และการให้เหตุผลหลายขั้นตอน) ซึ่งเราจะพูดถึงใน [Module 05: MCP](../05-mcp/README.md)

ถ้าไม่มีเครื่องมือ โมเดลภาษาจะสร้างข้อความได้จากข้อมูลการฝึกเท่านั้น ถามเกี่ยวกับสภาพอากาศตอนนี้ มันก็ต้องเดา หากให้เครื่องมือ มันสามารถเรียก API สภาพอากาศ, คำนวณ หรือค้นหาจากฐานข้อมูล จากนั้นนำผลลัพธ์จริงเหล่านั้นสอดแทรกในคำตอบ

<img src="../../../translated_images/th/what-are-tools.724e468fc4de64da.webp" alt="ไม่มีเครื่องมือ vs พร้อมเครื่องมือ" width="800"/>

*หากไม่มีเครื่องมือ โมเดลจะเดา — หากมีเครื่องมือจะสามารถเรียก API, คำนวณ และส่งข้อมูลเรียลไทม์*

ตัวแทน AI กับเครื่องมือใช้รูปแบบ **Reasoning and Acting (ReAct)** โมเดลไม่ตอบแค่คำถาม — มันคิดว่าต้องการอะไร, ลงมือโดยเรียกใช้เครื่องมือ, สังเกตผลลัพธ์, แล้วตัดสินใจว่าจะทำซ้ำหรือส่งคำตอบสุดท้าย:

1. **ให้เหตุผล** — ตัวแทนวิเคราะห์คำถามผู้ใช้และกำหนดข้อมูลที่ต้องการ
2. **ลงมือทำ** — ตัวแทนเลือกเครื่องมือที่เหมาะสม, สร้างพารามิเตอร์ถูกต้อง และเรียกใช้เครื่องมือ
3. **สังเกต** — ตัวแทนรับผลลัพธ์จากเครื่องมือและประเมินผล
4. **ทำซ้ำหรือส่งคำตอบ** — หากต้องการข้อมูลเพิ่มเติม ตัวแทนจะวนลูปอีกครั้ง; หากไม่ ตัวแทนจะสร้างคำตอบเป็นภาษาธรรมชาติ

<img src="../../../translated_images/th/react-pattern-detail.96a5efeeb6dd2f61.webp" alt="รูปแบบ ReAct" width="800"/>

*วงจร ReAct — ตัวแทนคิดว่าต้องทำอะไร, ลงมือเรียกใช้เครื่องมือ, สังเกตผลลัพธ์, และวนลูปจนพร้อมให้คำตอบสุดท้าย*

สิ่งนี้เกิดขึ้นโดยอัตโนมัติ คุณกำหนดเครื่องมือและคำอธิบาย โมเดลจะจัดการการตัดสินใจเมื่อไหร่และอย่างไรที่จะใช้เครื่องมือเหล่านั้น

## การทำงานของการเรียกใช้เครื่องมือ

### คำจำกัดความของเครื่องมือ

[WeatherTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) | [TemperatureTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/TemperatureTool.java)

คุณกำหนดฟังก์ชันพร้อมคำอธิบายชัดเจนและข้อกำหนดพารามิเตอร์ โมเดลจะเห็นคำอธิบายเหล่านี้ใน system prompt และเข้าใจว่าแต่ละเครื่องมือทำอะไร

```java
@Component
public class WeatherTool {
    
    @Tool("Get the current weather for a location")
    public String getCurrentWeather(@P("Location name") String location) {
        // ตรรกะการค้นหาสภาพอากาศของคุณ
        return "Weather in " + location + ": 22°C, cloudy";
    }
}

@AiService
public interface Assistant {
    String chat(@MemoryId String sessionId, @UserMessage String message);
}

// ผู้ช่วยเชื่อมต่อโดยอัตโนมัติด้วย Spring Boot กับ:
// - บีน ChatModel
// - เมธอด @Tool ทั้งหมดจากคลาส @Component
// - ChatMemoryProvider สำหรับการจัดการเซสชัน
```

ไดอะแกรมด้านล่างตีความทุกคำอธิบายและแสดงว่าแต่ละส่วนช่วยให้ AI เข้าใจว่าเมื่อใดควรเรียกเครื่องมือและส่งอาร์กิวเมนต์ใด:

<img src="../../../translated_images/th/tool-definitions-anatomy.f6468546037cf28b.webp" alt="โครงสร้างคำจำกัดความของเครื่องมือ" width="800"/>

*โครงสร้างคำจำกัดความเครื่องมือ — @Tool บอก AI ว่าควรใช้เมื่อไร, @P อธิบายพารามิเตอร์แต่ละตัว, และ @AiService เชื่อมต่อส่วนประกอบทั้งหมดตอนเริ่มต้น*

### การตัดสินใจ

เมื่อผู้ใช้ถามว่า "สภาพอากาศใน Seattle เป็นอย่างไร?" โมเดลไม่ได้สุ่มเลือกเครื่องมือ แต่มันจะเปรียบเทียบเจตนาของผู้ใช้กับคำอธิบายเครื่องมือทั้งหมดที่เข้าถึงได้, ให้คะแนนความเกี่ยวข้องแต่ละตัว, และเลือกอันที่เหมาะสมที่สุด จากนั้นสร้างการเรียกใช้งานฟังก์ชันโดยมีพารามิเตอร์ถูกต้อง — ในที่นี้คือกำหนด `location` เป็น `"Seattle"`

ถ้าเครื่องมือไม่มีตัวใดเหมาะสมกับคำขอของผู้ใช้ โมเดลจะตอบจากความรู้ของตัวเองแทน หากมีหลายเครื่องมือเหมาะสม โมเดลจะเลือกอันที่เฉพาะเจาะจงที่สุด

<img src="../../../translated_images/th/decision-making.409cd562e5cecc49.webp" alt="วิธีที่ AI ตัดสินใจเลือกเครื่องมือ" width="800"/>

*โมเดลประเมินเครื่องมือที่มีทั้งหมดกับเจตนาผู้ใช้และเลือกเครื่องมือที่ดีที่สุด — นี่จึงเป็นเหตุผลว่าทำไมการเขียนคำอธิบายเครื่องมือให้ชัดเจนและเฉพาะเจาะจงจึงสำคัญ*

### การสร้างคำตอบ

โมเดลได้รับข้อมูลสภาพอากาศและจัดรูปแบบเป็นคำตอบภาษาธรรมชาติให้ผู้ใช้

### สถาปัตยกรรม: Spring Boot การเชื่อมโยงอัตโนมัติ

โมดูลนี้ใช้การผนวกรวม LangChain4j กับ Spring Boot แบบ interface `@AiService` แบบประกาศ เมื่อสตาร์ท Spring Boot จะค้นหา `@Component` ที่มีเมธอด `@Tool` ทั้งหมด, bean `ChatModel` และ `ChatMemoryProvider` — แล้วเชื่อมทั้งหมดรวมเป็น interface `Assistant` ตัวเดียวโดยไม่มีโค้ดซ้ำ

<img src="../../../translated_images/th/spring-boot-wiring.151321795988b04e.webp" alt="สถาปัตยกรรมการเชื่อมโยงอัตโนมัติของ Spring Boot" width="800"/>

*interface @AiService รวม ChatModel, องค์ประกอบเครื่องมือ และ memory provider — Spring Boot จัดการเชื่อมต่อทั้งหมดให้อัตโนมัติ*

ประโยชน์หลักของวิธีนี้:

- **การเชื่อมโยงอัตโนมัติของ Spring Boot** — ฉีดอัตโนมัติ ChatModel และเครื่องมือ
- **รูปแบบ @MemoryId** — การจัดการหน่วยความจำตามเซสชันอัตโนมัติ
- **อินสแตนซ์เดียว** — สร้าง Assistant ครั้งเดียว ใช้งานซ้ำ เพิ่มประสิทธิภาพ
- **การดำเนินการปลอดภัยแบบ Type-safe** — เรียกใช้เมธอด Java โดยตรงพร้อมแปลงประเภทข้อมูล
- **การจัดการหลายรอบสนทนา** — จัดการการเชื่อมต่อเครื่องมือโดยอัตโนมัติ
- **ไม่ต้องเขียนโค้ดซ้ำ** — ไม่มีการเรียก `AiServices.builder()` หรือ HashMap memory ด้วยตนเอง

วิธีอื่น (เช่น ใช้ `AiServices.builder()` ด้วยมือ) ต้องโค้ดมากกว่าและพลาดประโยชน์ของ Spring Boot

## การเชื่อมต่อเครื่องมือ

**การเชื่อมต่อเครื่องมือ** — พลังของตัวแทนที่ใช้เครื่องมือจริงๆ เกิดขึ้นเมื่อคำถามเดียวต้องใช้หลายเครื่องมือ ถามว่า "สภาพอากาศใน Seattle เป็นฟาเรนไฮต์เท่าไร?" ตัวแทนจะเชื่อมต่อเครื่องมือสองตัว: เรียก `getCurrentWeather` เพื่อรับอุณหภูมิเซลเซียสก่อน แล้วส่งค่าผลลัพธ์ไปที่ `celsiusToFahrenheit` เพื่อแปลง — ทั้งหมดในรอบสนทนาเดียว

<img src="../../../translated_images/th/tool-chaining-example.538203e73d09dd82.webp" alt="ตัวอย่างการเชื่อมเครื่องมือ" width="800"/>

*การเชื่อมต่อเครื่องมือ — ตัวแทนเรียก getCurrentWeather ก่อน แล้วส่งผลลัพธ์ในเซลเซียสเข้า celsiusToFahrenheit แล้วตอบรวมกัน*

**การล้มเหลวอย่างสง่างาม** — ถามหาสภาพอากาศเมืองที่ไม่มีในข้อมูลจำลอง เครื่องมือจะส่งข้อความข้อผิดพลาด และ AI อธิบายว่าช่วยไม่ได้แทนที่จะเกิดขัดข้อง ทูลจะล้มเหลวได้อย่างปลอดภัย ไดอะแกรมด้านล่างเปรียบเทียบสองวิธี — เมื่อจัดการข้อผิดพลาดดี ตัวแทนจะจับข้อผิดพลาดและตอบช่วยเหลือได้ ขณะที่ถ้าไม่ทำ แอปจะล่มทั้งระบบ:

<img src="../../../translated_images/th/error-handling-flow.9a330ffc8ee0475c.webp" alt="กระบวนการจัดการข้อผิดพลาด" width="800"/>

*เมื่อเครื่องมือทำงานล้มเหลว ตัวแทนจับข้อผิดพลาดและตอบคำอธิบายช่วยเหลือแทนที่จะล่ม*

สิ่งนี้เกิดในรอบสนทนาเดียว ตัวแทนจะจัดการเรียกใช้หลายเครื่องมืออย่างอิสระ

## การรันแอปพลิเคชัน

**ตรวจสอบการติดตั้ง:**

ตรวจสอบว่าไฟล์ `.env` อยู่ในโฟลเดอร์หลักพร้อมข้อมูลรับรอง Azure (สร้างตอน Module 01) รันจากไดเรกทอรีโมดูล (`04-tools/`):

**Bash:**
```bash
cat ../.env  # ควรแสดง AZURE_OPENAI_ENDPOINT, API_KEY, DEPLOYMENT
```

**PowerShell:**
```powershell
Get-Content ..\.env  # ควรแสดง AZURE_OPENAI_ENDPOINT, API_KEY, DEPLOYMENT
```

**เริ่มต้นแอปพลิเคชัน:**

> **หมายเหตุ:** หากคุณเริ่มทุกแอปด้วย `./start-all.sh` จากโฟลเดอร์หลัก (ตาม Module 01) โหมดนี้จะรันที่พอร์ต 8084 แล้ว คุณสามารถข้ามคำสั่งเริ่มต้นด้านล่างและไปที่ http://localhost:8084 ได้เลย

**ตัวเลือก 1: ใช้ Spring Boot Dashboard (แนะนำสำหรับผู้ใช้ VS Code)**

dev container รวมส่วนขยาย Spring Boot Dashboard ที่ให้ UI สำหรับจัดการแอป Spring Boot ทั้งหมด คุณสามารถหาได้ที่ Activity Bar ด้านซ้ายของ VS Code (มองหาไอคอน Spring Boot)

จาก Spring Boot Dashboard คุณสามารถ:
- ดูแอป Spring Boot ทั้งหมดใน workspace
- เริ่ม/หยุดแอปด้วยคลิกเดียว
- ดูล็อกแอปเรียลไทม์
- ตรวจสอบสถานะแอป

แค่คลิกปุ่มเล่นข้างชื่อ "tools" เพื่อเริ่มโมดูลนี้ หรือเริ่มทุกโมดูลพร้อมกัน

หน้าตา Spring Boot Dashboard ใน VS Code:

<img src="../../../translated_images/th/dashboard.9b519b1a1bc1b30a.webp" alt="Spring Boot Dashboard" width="400"/>

*Spring Boot Dashboard ใน VS Code — เริ่ม, หยุด และตรวจสอบทุกโมดูลจากที่เดียว*

**ตัวเลือก 2: ใช้สคริปต์เชลล์**

เริ่มแอปเว็บทั้งหมด (โมดูล 01-04):

**Bash:**
```bash
cd ..  # จากไดเรกทอรีราก
./start-all.sh
```

**PowerShell:**
```powershell
cd ..  # จากไดเรกทอรีราก
.\start-all.ps1
```

หรือเริ่มแค่โมดูลนี้:

**Bash:**
```bash
cd 04-tools
./start.sh
```

**PowerShell:**
```powershell
cd 04-tools
.\start.ps1
```

ทั้งสองสคริปต์โหลดตัวแปรสภาพแวดล้อมจากไฟล์ `.env` ในโฟลเดอร์หลักโดยอัตโนมัติ และจะคอมไพล์ JAR หากยังไม่มี

> **หมายเหตุ:** หากคุณต้องการคอมไพล์ทุกโมดูลด้วยตนเองก่อนเริ่ม:
>
> **Bash:**
> ```bash
> cd ..  # Go to root directory
> mvn clean package -DskipTests
> ```
>
> **PowerShell:**
> ```powershell
> cd ..  # Go to root directory
> mvn clean package -DskipTests
> ```

เปิด http://localhost:8084 ในเว็บเบราว์เซอร์ของคุณ

```bash
./stop.sh  # โมดูลนี้เท่านั้น

cd .. && ./stop-all.sh  # ทุกโมดูล
```

**PowerShell:**
```powershell
.\stop.ps1  # โมดูลนี้เท่านั้น

```

## การใช้งานแอปพลิเคชัน

แอปพลิเคชันให้เว็บอินเทอร์เฟซที่คุณโต้ตอบกับตัวแทน AI ที่เข้าถึงเครื่องมือสภาพอากาศและแปลงอุณหภูมิ หน้าตาของอินเทอร์เฟซรวมตัวอย่างเริ่มต้นและแชทเพื่อส่งคำขอ:
<a href="images/tools-homepage.png"><img src="../../../translated_images/th/tools-homepage.4b4cd8b2717f9621.webp" alt="อินเทอร์เฟซเครื่องมือ AI Agent" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*อินเทอร์เฟซเครื่องมือ AI Agent - ตัวอย่างอย่างรวดเร็วและอินเทอร์เฟซแชทสำหรับการโต้ตอบกับเครื่องมือ*

### ลองใช้งานเครื่องมืออย่างง่าย

เริ่มด้วยคำขอที่ตรงไปตรงมา: "แปลง 100 องศาฟาเรนไฮต์เป็นเซลเซียส" ตัวแทนจะรู้ว่าต้องใช้เครื่องมือแปลงอุณหภูมิ เรียกใช้งานด้วยพารามิเตอร์ที่ถูกต้อง และส่งผลลัพธ์กลับมา สังเกตว่ารู้สึกเป็นธรรมชาติอย่างไร - คุณไม่ได้ระบุว่าต้องใช้เครื่องมือใดหรือวิธีการเรียกใช้งาน

### ทดสอบการเชื่อมโยงเครื่องมือ

ตอนนี้ลองสิ่งที่ซับซ้อนขึ้น: "สภาพอากาศที่ซีแอตเทิลเป็นอย่างไรและแปลงเป็นฟาเรนไฮต์?" ดูตัวแทนทำงานผ่านขั้นตอนนี้ก่อน มันดึงข้อมูลสภาพอากาศ (ซึ่งคืนค่าเป็นเซลเซียส) รู้ว่าต้องแปลงเป็นฟาเรนไฮต์ เรียกใช้งานเครื่องมือแปลง และรวมทั้งสองผลลัพธ์เป็นคำตอบเดียว

### ดูการไหลของการสนทนา

อินเทอร์เฟซแชทจะเก็บประวัติการสนทนา ช่วยให้คุณมีการโต้ตอบหลายรอบ คุณจะเห็นคำถามและคำตอบที่ผ่านมา ทั้งหมดสะดวกในการติดตามการสนทนาและเข้าใจว่าเอเจนต์สร้างบริบทอย่างไรในหลายๆ การแลกเปลี่ยน

<a href="images/tools-conversation-demo.png"><img src="../../../translated_images/th/tools-conversation-demo.89f2ce9676080f59.webp" alt="การสนทนาที่ใช้เครื่องมือหลายครั้ง" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*การสนทนาแบบหลายรอบที่แสดงการแปลงค่าอย่างง่าย การค้นหาสภาพอากาศ และการเชื่อมโยงเครื่องมือ*

### ทดลองกับคำขอที่หลากหลาย

ลองผสมผสานต่างๆ:
- ค้นหาสภาพอากาศ: "สภาพอากาศที่โตเกียวเป็นอย่างไร?"
- การแปลงอุณหภูมิ: "25°C เท่ากับเคลวินเท่าไหร่?"
- คำถามรวมกัน: "ตรวจสอบสภาพอากาศที่ปารีสและบอกฉันว่ามากกว่า 20°C หรือไม่"

สังเกตว่าตัวแทนแปลภาษาธรรมชาติและแม็ปไปยังการเรียกใช้งานเครื่องมือที่เหมาะสมอย่างไร

## แนวคิดสำคัญ

### รูปแบบ ReAct (เหตุผลและการกระทำ)

เอเจนต์สลับกันระหว่างการไตร่ตรอง (ตัดสินใจว่าจะทำอะไร) และการกระทำ (ใช้เครื่องมือ) รูปแบบนี้ช่วยให้แก้ปัญหาโดยอัตโนมัติแทนแค่ตอบกลับคำสั่ง

### คำอธิบายเครื่องมือสำคัญ

คุณภาพของคำอธิบายเครื่องมือส่งผลโดยตรงต่อการใช้งานของเอเจนต์ คำอธิบายที่ชัดเจนและเฉพาะเจาะจงช่วยให้โมเดลเข้าใจว่าเมื่อไรและวิธีการเรียกใช้เครื่องมือแต่ละตัว

### การจัดการเซสชัน

คำอธิบาย `@MemoryId` ช่วยให้จัดการหน่วยความจำแบบเซสชันโดยอัตโนมัติ ID เซสชันแต่ละตัวจะได้วัตถุ `ChatMemory` แยกกันที่ดูแลโดยอินสแตนซ์ `ChatMemoryProvider` ซึ่งช่วยให้ผู้ใช้หลายคนสามารถโต้ตอบกับเอเจนต์ได้พร้อมกันโดยไม่มีการปะปนของการสนทนา แผนภาพด้านล่างแสดงการแยกเส้นทางผู้ใช้หลายคนเข้าสู่ที่เก็บประวัติการสนทนาที่แยกกันตาม ID เซสชันของพวกเขา:

<img src="../../../translated_images/th/session-management.91ad819c6c89c400.webp" alt="การจัดการเซสชันด้วย @MemoryId" width="800"/>

*ID เซสชันแต่ละตัวแม็ปไปยังประวัติการสนทนาแยกกัน — ผู้ใช้จะไม่เห็นข้อความของกันและกัน*

### การจัดการข้อผิดพลาด

เครื่องมืออาจล้มเหลว — API หมดเวลา พารามิเตอร์อาจผิด เซอร์วิสภายนอกดาวน์ เอเจนต์สำหรับงานจริงต้องจัดการข้อผิดพลาดเพื่อให้โมเดลสามารถอธิบายปัญหาหรือพยายามทางเลือกอื่นแทนการทำให้แอปพลิเคชันล่ม เมื่อเครื่องมือขว้างข้อยกเว้น LangChain4j จะจับข้อผิดพลาดและส่งข้อความกลับให้โมเดล ซึ่งจะสามารถอธิบายปัญหาเป็นภาษาธรรมชาติได้

## เครื่องมือที่มีให้ใช้

แผนภาพด้านล่างแสดงระบบนิเวศกว้างของเครื่องมือที่คุณสามารถสร้าง โมดูลนี้สาธิตเครื่องมือสภาพอากาศและอุณหภูมิ แต่รูปแบบ `@Tool` เหมาะกับเมธอด Java ใดๆ — ตั้งแต่การสืบค้นฐานข้อมูลไปจนถึงการประมวลผลการชำระเงิน

<img src="../../../translated_images/th/tool-ecosystem.aad3d74eaa14a44f.webp" alt="ระบบนิเวศเครื่องมือ" width="800"/>

*เมธอด Java ใดๆ ที่มี annotation @Tool จะพร้อมใช้งานใน AI — รูปแบบนี้ขยายไปยังฐานข้อมูล, API, อีเมล, การจัดการไฟล์ และอื่นๆ*

## เมื่อไรควรใช้เอเจนต์ที่ใช้เครื่องมือ

ไม่ใช่ทุกคำขอจำเป็นต้องใช้เครื่องมือ การตัดสินใจขึ้นกับว่า AI ต้องโต้ตอบกับระบบภายนอกหรือสามารถตอบจากความรู้ของตัวเอง คู่มือด้านล่างสรุปว่าเมื่อไรที่เครื่องมือช่วยได้และเมื่อไรไม่จำเป็น:

<img src="../../../translated_images/th/when-to-use-tools.51d1592d9cbdae9c.webp" alt="เมื่อไรควรใช้เครื่องมือ" width="800"/>

*คู่มือการตัดสินใจอย่างรวดเร็ว — เครื่องมือใช้สำหรับข้อมูลเรียลไทม์, การคำนวณ และการกระทำ; งานความรู้ทั่วไปและงานสร้างสรรค์ไม่จำเป็นต้องใช้*

## เครื่องมือกับ RAG

โมดูล 03 และ 04 ขยายความสามารถของ AI แต่ด้วยวิธีพื้นฐานที่ต่างกัน RAG ให้โมเดลเข้าถึง **ความรู้** โดยเรียกดูเอกสาร เครื่องมือให้โมเดลมีความสามารถทำ **การกระทำ** โดยเรียกใช้งานฟังก์ชัน แผนภาพด้านล่างเทียบระหว่างสองวิธีนี้ตรงกัน — ตั้งแต่กระบวนการทำงานของแต่ละวิธีไปจนถึงข้อแลกเปลี่ยนระหว่างกัน:

<img src="../../../translated_images/th/tools-vs-rag.ad55ce10d7e4da87.webp" alt="การเปรียบเทียบเครื่องมือกับ RAG" width="800"/>

*RAG ดึงข้อมูลจากเอกสารแบบคงที่ — เครื่องมือทำงานและดึงข้อมูลแบบไดนามิกเรียลไทม์ ระบบผลิตภัณฑ์หลายแห่งรวมทั้งสองแบบนี้*

ในทางปฏิบัติ ระบบผลิตภัณฑ์หลายแห่งผสมผสานทั้งสองแนวทาง: RAG สำหรับให้คำตอบที่อิงกับเอกสารของคุณ และเครื่องมือสำหรับดึงข้อมูลสดหรือทำการดำเนินการต่างๆ
