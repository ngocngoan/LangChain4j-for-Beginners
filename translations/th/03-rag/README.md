## สิ่งที่คุณจะได้เรียนรู้

ในโมดูลก่อนหน้า คุณได้เรียนรู้วิธีการสนทนากับ AI และการจัดโครงสร้างคำสั่งให้มีประสิทธิภาพ แต่มีข้อจำกัดพื้นฐานอย่างหนึ่งคือ: โมเดลภาษาเรียนรู้ได้เฉพาะจากข้อมูลที่ฝึกมาเท่านั้น พวกมันไม่สามารถตอบคำถามเกี่ยวกับนโยบายบริษัทของคุณ เอกสารโครงการ หรือข้อมูลอื่นใดที่ไม่ได้ถูกฝึกไว้

RAG (Retrieval-Augmented Generation) แก้ปัญหานี้ แทนที่จะพยายามสอนโมเดลเกี่ยวกับข้อมูลของคุณ (ซึ่งมีค่าใช้จ่ายสูงและไม่ใช่เรื่องปฏิบัติได้) คุณจะมอบความสามารถให้มันค้นหาข้อมูลในเอกสารของคุณ เมื่อมีคนถามคำถาม ระบบจะค้นหาข้อมูลที่เกี่ยวข้องและนำมาใส่ในคำสั่ง โมเดลจึงตอบกลับตามบริบทที่ได้ค้นพบ

คิดว่า RAG เหมือนการให้โมเดลมีห้องสมุดอ้างอิง เมื่อคุณถามคำถาม ระบบจะทำดังนี้:

1. **คำถามผู้ใช้** - คุณถามคำถาม
2. **Embedding** - แปลงคำถามของคุณเป็นเวกเตอร์
3. **การค้นหาด้วยเวกเตอร์** - ค้นหาชิ้นส่วนเอกสารที่มีความคล้ายคลึง
4. **การรวมบริบท** - เพิ่มชิ้นส่วนที่เกี่ยวข้องลงในคำสั่ง
5. **การตอบคำถาม** - LLM สร้างคำตอบตามบริบท

สิ่งนี้ทำให้คำตอบของโมเดลอิงกับข้อมูลจริงของคุณ แทนที่จะพึ่งพาความรู้จากการฝึกสอนหรือสร้างคำตอบขึ้นมาเอง

### การประมวลผลเอกสาร

[DocumentService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/DocumentService.java)

เมื่อคุณอัปโหลดเอกสาร ระบบจะทำการแยกวิเคราะห์ (PDF หรือข้อความธรรมดา) แนบเมตาดาต้ารวมถึงชื่อไฟล์ จากนั้นทำการแบ่งเอกสารเป็นชิ้นเล็ก ๆ ซึ่งพอดีกับพื้นที่บริบทของโมเดล ชิ้นส่วนเหล่านี้มีการทับซ้อนกันเล็กน้อยเพื่อไม่ให้บริบทบางส่วนหายไปที่ขอบเขต

```java
// วิเคราะห์ไฟล์ที่อัปโหลดแล้วห่อหุ้มในเอกสาร LangChain4j
Document document = Document.from(content, metadata);

// แบ่งออกเป็นชิ้นละ 300 โทเค็นโดยทับซ้อนกัน 30 โทเค็น
DocumentSplitter splitter = DocumentSplitters
    .recursive(300, 30);

List<TextSegment> segments = splitter.split(document);
```
  
แผนภาพด้านล่างแสดงการทำงานเชิงภาพ สังเกตว่าชิ้นแต่ละส่วนจะแบ่งปันโทเค็นบางส่วนกับชิ้นข้างเคียง — ทับซ้อน 30 โทเค็นช่วยให้ไม่มีบริบทสำคัญหลุดรอดไป:

### การสร้าง Embeddings

[LangChainRagConfig.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/config/LangChainRagConfig.java)

ชิ้นส่วนแต่ละชิ้นจะถูกแปลงเป็นการแทนค่าตัวเลขที่เรียกว่า embedding — รอยนิ้วมือทางคณิตศาสตร์ที่เก็บความหมายของข้อความ ข้อความที่เหมือนกันจะมี embedding คล้ายกัน

```java
@Bean
public EmbeddingModel embeddingModel() {
    return OpenAiOfficialEmbeddingModel.builder()
        .baseUrl(azureOpenAiEndpoint)
        .apiKey(azureOpenAiKey)
        .modelName(azureEmbeddingDeploymentName)
        .build();
}

EmbeddingStore<TextSegment> embeddingStore = 
    new InMemoryEmbeddingStore<>();
```
  
แผนผังคลาสด้านล่างแสดงการเชื่อมต่อส่วนประกอบ LangChain4j เหล่านี้ `OpenAiOfficialEmbeddingModel` แปลงข้อความเป็นเวกเตอร์, `InMemoryEmbeddingStore` เก็บเวกเตอร์พร้อมข้อมูล `TextSegment` ดั้งเดิม, และ `EmbeddingSearchRequest` ควบคุมพารามิเตอร์การดึงข้อมูล เช่น `maxResults` และ `minScore`:

<img src="../../../translated_images/th/rag-langchain4j-classes.bbf3aa9077ab443d.webp" alt="LangChain4j RAG Classes" width="800"/>

เมื่อ embeddings ถูกเก็บไว้ เนื้อหาที่คล้ายกันจะกลุ่มใกล้กันในเวกเตอร์สเปซ ภาพด้านล่างแสดงว่าเอกสารเกี่ยวกับหัวข้อที่เกี่ยวข้องจะอยู่ใกล้กันซึ่งทำให้การค้นหาเชิงความหมายเป็นไปได้:

### การค้นหาเชิงความหมาย

[RagService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/RagService.java)

เมื่อคุณถามคำถาม คำถามนั้นจะถูกแปลงเป็น embedding เช่นกัน ระบบเปรียบเทียบ embedding ของคำถามกับ embeddings ของชิ้นส่วนเอกสารทั้งหมด เพื่อค้นหาชิ้นส่วนที่มีความหมายใกล้เคียงที่สุด — ไม่ใช่เพียงคำสำคัญที่เหมือนกัน แต่ความหมายเชิงความสัมพันธ์จริง ๆ

```java
Embedding queryEmbedding = embeddingModel.embed(question).content();

EmbeddingSearchRequest searchRequest = EmbeddingSearchRequest.builder()
    .queryEmbedding(queryEmbedding)
    .maxResults(5)
    .minScore(0.5)
    .build();

EmbeddingSearchResult<TextSegment> searchResult = embeddingStore.search(searchRequest);
List<EmbeddingMatch<TextSegment>> matches = searchResult.matches();

for (EmbeddingMatch<TextSegment> match : matches) {
    String relevantText = match.embedded().text();
    double score = match.score();
}
```
  
แผนภาพด้านล่างเปรียบเทียบการค้นหาเชิงความหมายกับการค้นหาด้วยคำสำคัญ คำค้นคำว่า "vehicle" อาจพลาดชิ้นส่วนเกี่ยวกับ "cars and trucks" แต่การค้นหาเชิงความหมายเข้าใจว่าเป็นเรื่องเดียวกันและแสดงผลลัพธ์ที่มีคะแนนสูง:

### การสร้างคำตอบ

[RagService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/RagService.java)

ชิ้นส่วนที่เกี่ยวข้องมากที่สุดจะถูกรวมเป็นคำสั่งที่มีโครงสร้างโดยบรรจุคำแนะนำชัดเจน, บริบทที่ค้นพบ และคำถามของผู้ใช้ โมเดลจะอ่านชิ้นส่วนนั้นและตอบตามข้อมูลเหล่านั้น — มันใช้ได้แค่ข้อมูลที่อยู่ตรงหน้าเท่านั้น ซึ่งช่วยป้องกันการจินตนาการคำตอบขึ้นมาเอง

```java
String context = matches.stream()
    .map(match -> match.embedded().text())
    .collect(Collectors.joining("\n\n"));

String prompt = String.format("""
    Answer the question based on the following context.
    If the answer cannot be found in the context, say so.

    Context:
    %s

    Question: %s

    Answer:""", context, request.question());

String answer = chatModel.chat(prompt);
```
  
แผนภาพด้านล่างแสดงการรวมคำสั่งนี้ — ชิ้นส่วนที่มีคะแนนสูงสุดจากขั้นตอนค้นหาจะถูกนำเข้าไปในแม่แบบคำสั่ง และ `OpenAiOfficialChatModel` จะสร้างคำตอบที่มีความน่าเชื่อถือ:

### อัปโหลดเอกสาร

เริ่มต้นด้วยการอัปโหลดเอกสาร - ไฟล์ TXT ใช้งานง่ายสุดสำหรับทดสอบ มีไฟล์ `sample-document.txt` ให้ในโฟลเดอร์นี้ ซึ่งมีข้อมูลเกี่ยวกับคุณสมบัติของ LangChain4j, การใช้งาน RAG และแนวปฏิบัติที่ดี เหมาะสำหรับการทดสอบระบบ

ระบบจะประมวลผลเอกสารของคุณ แบ่งเป็นชิ้น และสร้าง embeddings ให้แต่ละชิ้นโดยอัตโนมัติเมื่ออัปโหลด

### ถามคำถาม

ตอนนี้สามารถถามคำถามเฉพาะเจาะจงเกี่ยวกับเนื้อหาเอกสารได้ ลองถามข้อมูลที่ชัดเจนในเอกสาร ระบบจะค้นหาชิ้นส่วนที่เกี่ยวข้อง ใส่ลงในคำสั่ง และสร่างค์คำตอบ

### ตรวจสอบแหล่งข้อมูลอ้างอิง

สังเกตว่าคำตอบแต่ละข้อจะแสดงแหล่งข้อมูลอ้างอิงพร้อมคะแนนความคล้ายคลึง คะแนนเหล่านี้ (0 ถึง 1) แสดงระดับความเกี่ยวข้องของแต่ละชิ้นส่วนต่อคำถาม ยิ่งคะแนนสูงยิ่งแม่นยำ ช่วยให้คุณตรวจสอบคำตอบกับแหล่งที่มาได้

<a href="images/rag-query-results.png"><img src="../../../translated_images/th/rag-query-results.6d69fcec5397f355.webp" alt="RAG Query Results" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*ผลการค้นหาคำถามแสดงคำตอบพร้อมแหล่งที่มาและคะแนนความเกี่ยวข้อง*

### ทดลองถามคำถาม

ลองใช้คำถามหลายรูปแบบ:
- ข้อมูลเฉพาะ: "หัวข้อหลักคืออะไร?"
- การเปรียบเทียบ: "ความแตกต่างระหว่าง X กับ Y คืออะไร?"
- สรุป: "สรุปประเด็นสำคัญเกี่ยวกับ Z"

สังเกตว่าคะแนนความเกี่ยวข้องเปลี่ยนไปตามความสอดคล้องของคำถามกับเนื้อหาเอกสาร

## แนวคิดหลัก

### กลยุทธ์การแบ่งย่อย

เอกสารถูกแบ่งเป็นชิ้นละ 300 โทเค็น โดยทับซ้อน 30 โทเค็น ความสมดุลนี้ช่วยให้แต่ละชิ้นมีบริบทเพียงพอและยังมีขนาดเล็กพอที่จะรวมหลายชิ้นในคำสั่งได้

### คะแนนความคล้ายคลึง

ชิ้นส่วนทุกชิ้นที่ถูกดึงมาจะมีคะแนนความคล้ายคลึงตั้งแต่ 0 ถึง 1 บ่งชี้ว่ามันตรงกับคำถามมากน้อยแค่ไหน แผนภาพด้านล่างแสดงช่วงคะแนนและวิธีการที่ระบบใช้คะแนนเพื่อกรองผลลัพธ์:

<img src="../../../translated_images/th/similarity-scores.b0716aa911abf7f0.webp" alt="Similarity Scores" width="800"/>

คะแนนมีช่วงดังนี้:
- 0.7-1.0: เกี่ยวข้องสูง, ตรงเป๊ะ
- 0.5-0.7: เกี่ยวข้อง, บริบทดี
- ต่ำกว่า 0.5: กรองออกเพราะคลาดเคลื่อนเกินไป

ระบบจะดึงชิ้นส่วนที่มีคะแนนเหนือตัวกรองขั้นต่ำเพื่อให้ผลลัพธ์มีคุณภาพ

### การเก็บข้อมูลในหน่วยความจำ

โมดูลนี้ใช้การเก็บข้อมูลใน memory เพื่อความง่าย เมื่อรีสตาร์ทแอป ข้อมูลเอกสารที่อัปโหลดจะหายไป ในระบบจริงมักใช้ฐานข้อมูลเวกเตอร์ถาวรเช่น Qdrant หรือ Azure AI Search

### การจัดการพื้นที่บริบท

โมเดลแต่ละตัวจำกัดขนาดพื้นที่บริบท คุณไม่สามารถใส่ชิ้นเอกสารทั้งหมดของเอกสารขนาดใหญ่ลงไปได้ ระบบจะดึงเฉพาะชิ้นส่วนที่เกี่ยวข้องสูงสุด N ชิ้น (ค่าปริยายเป็น 5) เพื่อรักษาขนาดบริบทให้อยู่ในขอบเขต พร้อมให้บริบทที่เพียงพอสำหรับคำตอบที่แม่นยำ

## เมื่อไหร่ที่ RAG มีความสำคัญ

RAG ไม่ได้เหมาะสมในทุกสถานการณ์ ไดอะแกรมแนะแนวทางด้านล่างช่วยคุณตัดสินใจว่าเมื่อไหร่ที่ RAG จะเพิ่มคุณค่า และเมื่อไหร่ที่วิธีง่ายกว่า — เช่น ใส่เนื้อหาโดยตรงในคำสั่ง หรือพึ่งพาความรู้ในตัวโมเดล — ก็เพียงพอแล้ว:

<img src="../../../translated_images/th/when-to-use-rag.1016223f6fea26bc.webp" alt="When to Use RAG" width="800"/>

**ใช้ RAG เมื่อ:**
- การตอบคำถามเกี่ยวกับเอกสารลิขสิทธิ์
- ข้อมูลมีการเปลี่ยนแปลงบ่อย (นโยบาย ราคา สเปก)
- ความถูกต้องต้องการแหล่งที่มา
- เนื้อหามีขนาดใหญ่เกินไปที่จะใส่ในคำสั่งเดียว
- คุณต้องการคำตอบที่ตรวจสอบได้และมีพื้นฐาน

**อย่าใช้ RAG เมื่อ:**
- คำถามต้องการความรู้ทั่วไปที่โมเดลมีอยู่แล้ว
- ต้องการข้อมูลแบบเรียลไทม์ (RAG ใช้งานกับเอกสารที่อัปโหลด)
- เนื้อหามีขนาดเล็กพอที่จะใส่ตรงในคำสั่ง
