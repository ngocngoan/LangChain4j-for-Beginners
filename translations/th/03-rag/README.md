## What You'll Learn

ในโมดูลก่อนหน้านี้ คุณได้เรียนรู้วิธีการสนทนากับ AI และจัดโครงสร้างพรอมต์ของคุณอย่างมีประสิทธิภาพ แต่มีข้อจำกัดพื้นฐานว่า: โมเดลภาษาจะรู้แค่สิ่งที่เรียนรู้ระหว่างการฝึกเท่านั้น พวกมันไม่สามารถตอบคำถามเกี่ยวกับนโยบายบริษัทของคุณ เอกสารโครงการของคุณ หรือข้อมูลใดๆ ที่ไม่ได้รับการฝึกสอนมา

RAG (Retrieval-Augmented Generation) แก้ปัญหานี้ แทนที่จะพยายามสอนข้อมูลของคุณให้โมเดล (ซึ่งมีค่าใช้จ่ายสูงและไม่สะดวก) คุณมอบความสามารถให้มันค้นคว้าข้อมูลในเอกสารของคุณ เมื่อ有人ถามคำถาม ระบบจะค้นหาข้อมูลที่เกี่ยวข้องและรวมเข้าไปในพรอมต์ จากนั้นโมเดลจะตอบคำถามโดยอิงจากบริบทที่ถูกดึงมานี้

คิดว่า RAG เหมือนกับการให้โมเดลมีห้องสมุดอ้างอิง เมื่อคุณถามคำถาม ระบบจะ:

1. **คำถามผู้ใช้** - คุณถามคำถาม
2. **Embedding** - แปลงคำถามของคุณเป็นเวกเตอร์
3. **การค้นหาเวกเตอร์** - ค้นหาชิ้นส่วนเอกสารที่คล้ายกัน
4. **การประกอบบริบท** - เพิ่มชิ้นส่วนที่เกี่ยวข้องเข้าไปในพรอมต์
5. **การตอบกลับ** - LLM สร้างคำตอบโดยอิงจากบริบทนั้น

สิ่งนี้ช่วยให้คำตอบของโมเดลมีฐานข้อมูลจริงของคุณแทนที่จะพึ่งพาความรู้จากการฝึกสอนหรือการเดาคำตอบ

## Understanding RAG

แผนภาพด้านล่างแสดงแนวคิดหลัก: แทนที่จะพึ่งพาข้อมูลที่โมเดลได้เรียนรู้เพียงอย่างเดียว RAG จะให้ห้องสมุดอ้างอิงจากเอกสารของคุณเพื่อปรึกษาก่อนจะสร้างคำตอบแต่ละครั้ง

<img src="../../../translated_images/th/what-is-rag.1f9005d44b07f2d8.webp" alt="What is RAG" width="800"/>

*แผนภาพนี้แสดงความแตกต่างระหว่าง LLM ปกติ (ซึ่งเดาจากข้อมูลฝึกสอน) กับ LLM ที่เพิ่ม RAG (ซึ่งปรึกษาเอกสารของคุณก่อน)*

นี่คือวิธีการเชื่อมต่อแต่ละส่วนแบบครบวงจร คำถามของผู้ใช้ผ่านสี่ขั้นตอน — embedding, การค้นหาเวกเตอร์, การประกอบบริบท, และการสร้างคำตอบ — โดยแต่ละขั้นตอนได้สร้างจากขั้นตอนก่อนหน้า:

<img src="../../../translated_images/th/rag-architecture.ccb53b71a6ce407f.webp" alt="RAG Architecture" width="800"/>

*แผนภาพนี้แสดง pipeline RAG ครบวงจร — คำถามของผู้ใช้ผ่าน embedding, การค้นหาเวกเตอร์, การประกอบบริบท และการสร้างคำตอบ*

ส่วนที่เหลือของโมดูลนี้จะอธิบายแต่ละขั้นตอนอย่างละเอียด พร้อมโค้ดที่คุณสามารถใช้งานและแก้ไขได้

### Which RAG Approach Does This Tutorial Use?

LangChain4j มีสามวิธีในการใช้งาน RAG แต่ละแบบมีระดับนามธรรมแตกต่างกัน แผนภาพด้านล่างเปรียบเทียบพวกมันไว้เคียงข้างกัน:

<img src="../../../translated_images/th/rag-approaches.5b97fdcc626f1447.webp" alt="Three RAG Approaches in LangChain4j" width="800"/>

*แผนภาพนี้เปรียบเทียบสามวิธี RAG ใน LangChain4j — Easy, Native, และ Advanced — พร้อมส่วนประกอบสำคัญและเวลาที่ควรใช้แต่ละแบบ*

| วิธีการ | สิ่งที่ทำ | ข้อแลกเปลี่ยน |
|---|---|---|
| **Easy RAG** | เชื่อมต่อทุกอย่างอัตโนมัติผ่าน `AiServices` และ `ContentRetriever` คุณเพียงแค่ระบุอินเทอร์เฟส แนบ retriever และ LangChain4j จะจัดการ embedding, การค้นหา และการประกอบพรอมต์เอง | โค้ดน้อย แต่คุณจะไม่เห็นขั้นตอนที่เกิดขึ้นให้ชัดเจน |
| **Native RAG** | คุณเรียกใช้โมเดล embedding, ค้นหาในสโตร์, สร้างพรอมต์ และสร้างคำตอบเอง — ทีละขั้นตอนอย่างชัดเจน | โค้ดมากขึ้น แต่ทุกขั้นตอนมองเห็นและแก้ไขได้ |
| **Advanced RAG** | ใช้เฟรมเวิร์ก `RetrievalAugmentor` พร้อมตัวแปลงคำถาม, ตัวกระจาย, ตัวจัดลำดับใหม่, และตัวฉีดเนื้อหาสำหรับ pipeline ระดับผลิตจริง | ยืดหยุ่นที่สุด แต่ซับซ้อนสูง |

**บทเรียนนี้ใช้วิธี Native** ทุกขั้นตอนของ pipeline RAG — embedding คำถาม, ค้นหาสตอร์เวกเตอร์, ประกอบบริบท, และสร้างคำตอบ — เขียนอย่างชัดเจนใน [`RagService.java`](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/RagService.java) นี่เป็นเจตนา: ในฐานะแหล่งเรียนรู้ การให้คุณเห็นและเข้าใจทุกขั้นตอนสำคัญกว่าการทำโค้ดให้สั้น เมื่อคุณคุ้นเคยกับวิธีเชื่อมต่อแต่ละส่วนแล้ว คุณอาจก้าวสู่ Easy RAG สำหรับต้นแบบเร็ว หรือ Advanced RAG สำหรับระบบผลิต

> **💡 เคยเห็น Easy RAG แล้วใช่ไหม?** โมดูล [Quick Start](../00-quick-start/README.md) มีตัวอย่างถามตอบเอกสาร ([`SimpleReaderDemo.java`](../../../00-quick-start/src/main/java/com/example/langchain4j/quickstart/SimpleReaderDemo.java)) ที่ใช้ Easy RAG — LangChain4j จัดการ embedding, ค้นหา, และประกอบพรอมต์ให้อัตโนมัติ โมดูลนี้ก้าวไปอีกขั้นด้วยการเปิด pipeline เพื่อให้คุณเห็นและควบคุมแต่ละขั้นตอนเอง

<img src="../../../translated_images/th/easy-rag-pipeline.2e1602e2ad2ded42.webp" alt="Easy RAG Pipeline - LangChain4j" width="800"/>

*แผนภาพนี้แสดง pipeline Easy RAG จาก `SimpleReaderDemo.java` เทียบกับวิธี Native ที่ใช้ในโมดูลนี้: Easy RAG ซ่อน embedding, การดึงข้อมูล และการประกอบพรอมต์ไว้หลัง `AiServices` และ `ContentRetriever` — คุณโหลดเอกสาร แนบ retriever แล้วได้คำตอบ วิธี Native เปิดเผยขั้นตอนเหล่านั้นให้คุณเรียกใช้งานโดยตรงทั้ง embed, ค้นหา, ประกอบบริบท และสร้างคำตอบ พร้อมด้วยการมองเห็นและควบคุมเต็มที่*

## How It Works

Pipeline RAG ในโมดูลนี้แบ่งออกเป็นสี่ขั้นตอนซึ่งทำงานต่อเนื่องกันทุกครั้งที่ผู้ใช้ถามคำถาม ขั้นตอนแรกคือการ **แยกและแบ่งเอกสาร** ที่อัปโหลดเป็นชิ้นเล็กๆ ที่จัดการได้ ชิ้นเหล่านี้จะถูกแปลงเป็น **embedding ในรูปแบบเวกเตอร์** และจัดเก็บเพื่อนำไปเปรียบเทียบเชิงคณิตศาสตร์ เมื่อมีคำถามเข้ามา ระบบจะทำการ **ค้นหาเชิงความหมาย** เพื่อหาชิ้นส่วนที่เกี่ยวข้องมากที่สุด และสุดท้ายใช้ชิ้นเหล่านั้นเป็นบริบทให้ LLM สำหรับ **การสร้างคำตอบ** ส่วนด้านล่างจะพาคุณผ่านแต่ละขั้นตอนพร้อมโค้ดจริงและแผนภาพ มาดูขั้นตอนแรกกันก่อน

### Document Processing

[DocumentService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/DocumentService.java)

เมื่อคุณอัปโหลดเอกสาร ระบบจะแยกเอกสารนั้น (PDF หรือข้อความธรรมดา) แนบข้อมูลเมตา เช่น ชื่อไฟล์ แล้วแบ่งเป็นชิ้นย่อย — ชิ้นเล็กๆ ที่พอดีกับหน้าต่างบริบทของโมเดล ชิ้นเหล่านี้จะทับซ้อนกันเล็กน้อยเพื่อไม่ให้สูญเสียบริบทที่ขอบเขต

```java
// วิเคราะห์ไฟล์ที่อัปโหลดและห่อหุ้มในเอกสาร LangChain4j
Document document = Document.from(content, metadata);

// แบ่งเป็นชิ้นส่วนขนาด 300 โทเคนพร้อมซ้อนทับ 30 โทเคน
DocumentSplitter splitter = DocumentSplitters
    .recursive(300, 30);

List<TextSegment> segments = splitter.split(document);
```


แผนภาพด้านล่างแสดงการทำงานนี้เป็นภาพ มองเห็นว่าชิ้นแต่ละชิ้นแบ่งปันสัญลักษณ์บางส่วนกับชิ้นข้างเคียง — การทับซ้อน 30 โทเค็นช่วยให้ไม่มีบริบทสำคัญตกหล่นไป:

<img src="../../../translated_images/th/document-chunking.a5df1dd1383431ed.webp" alt="Document Chunking" width="800"/>

*แผนภาพนี้แสดงเอกสารถูกแบ่งเป็นชิ้นละ 300 โทเค็นโดยซ้อนทับกัน 30 โทเค็น เพื่อคงบริบทบริเวณขอบชิ้น*

### Creating Embeddings

[LangChainRagConfig.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/config/LangChainRagConfig.java)

ชิ้นแต่ละชิ้นถูกแปลงเป็นการแสดงเลขที่เรียกว่า embedding — เป็นการแปลงความหมายเป็นตัวเลข โมเดล embedding ไม่ได้ “ฉลาด” เหมือนโมเดลแชท; มันไม่สามารถทำตามคำสั่ง ตรรกะ หรือให้คำตอบได้ สิ่งที่มันทำได้คือแมปข้อความเข้าสู่พื้นที่เชิงคณิตศาสตร์ที่ความหมายคล้ายกันจะอยู่ใกล้กัน — เช่น “car” ใกล้ “automobile”, “refund policy” ใกล้ “return my money” คิดว่าโมเดลแชทเป็นคนที่คุณคุยด้วย ส่วนโมเดล embedding คือระบบจัดเก็บเอกสารที่ยอดเยี่ยมมาก

<img src="../../../translated_images/th/embedding-model-concept.90760790c336a705.webp" alt="Embedding Model Concept" width="800"/>

*แผนภาพนี้แสดงว่าโมเดล embedding แปลงข้อความเป็นเวกเตอร์เลข โดยวางความหมายที่คล้ายกัน เช่น “car” และ “automobile” ให้อยู่ใกล้กันในพื้นที่เวกเตอร์*

```java
@Bean
public EmbeddingModel embeddingModel() {
    return OpenAiOfficialEmbeddingModel.builder()
        .baseUrl(azureOpenAiEndpoint)
        .apiKey(azureOpenAiKey)
        .modelName(azureEmbeddingDeploymentName)
        .build();
}

EmbeddingStore<TextSegment> embeddingStore = 
    new InMemoryEmbeddingStore<>();
```


แผนภาพคลาสด้านล่างแสดงสองเส้นทางแยกใน pipeline RAG และคลาส LangChain4j ที่ใช้ พาธ **ingestion** (ทำครั้งเดียวตอนอัปโหลด) แบ่งเอกสาร, embed ชิ้น, และจัดเก็บผ่าน `.addAll()` พาธ **query** (ทำทุกครั้งที่ผู้ใช้ถาม) embed คำถาม, ค้นหาในสโตร์ผ่าน `.search()`, และส่งบริบทที่ตรงกับโมเดลแชท ทั้งสองพาธเชื่อมต่อกันที่อินเทอร์เฟส `EmbeddingStore<TextSegment>` ร่วมกัน:

<img src="../../../translated_images/th/rag-langchain4j-classes.bbf3aa9077ab443d.webp" alt="LangChain4j RAG Classes" width="800"/>

*แผนภาพนี้แสดงสองพาธใน pipeline RAG — ingestion และ query — และวิธีเชื่อมต่อผ่าน EmbeddingStore*

เมื่อ embedding ถูกเก็บไว้ เนื้อหาที่คล้ายกันจะรวมกลุ่มในพื้นที่เวกเตอร์ ภาพจำลองด้านล่างแสดงว่าเอกสารที่เกี่ยวข้องจะกลายเป็นจุดใกล้กัน ซึ่งช่วยให้การค้นหาเชิงความหมายเป็นไปได้:

<img src="../../../translated_images/th/vector-embeddings.2ef7bdddac79a327.webp" alt="Vector Embeddings Space" width="800"/>

*ภาพนี้แสดงการรวมกลุ่มของเอกสารที่เกี่ยวข้องในพื้นที่เวกเตอร์ 3D เช่น เอกสารเทคนิค กฎธุรกิจ และคำถามที่พบบ่อย จัดเป็นกลุ่มแยกกัน*

เมื่อผู้ใช้ค้นหา ระบบจะทำตามสี่ขั้นตอน: embed เอกสารครั้งเดียว, embed คำถามในแต่ละครั้ง, เปรียบเทียบเวกเตอร์ของคำถามกับเวกเตอร์ที่เก็บด้วย cosine similarity, และส่งคืนชิ้นที่คะแนนสูงสุดในจำนวน K ขั้นตอนต่างๆ แสดงในแผนภาพนี้พร้อมคลาส LangChain4j ที่เกี่ยวข้อง:

<img src="../../../translated_images/th/embedding-search-steps.f54c907b3c5b4332.webp" alt="Embedding Search Steps" width="800"/>

*แผนภาพนี้แสดงกระบวนการค้นหา embedding สี่ขั้นตอน: embed เอกสาร, embed คำถาม, เปรียบเทียบเวกเตอร์โดย cosine similarity, และส่งคืนผลลัพธ์สูงสุด*

### Semantic Search

[RagService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/RagService.java)

เมื่อคุณถามคำถาม คำถามของคุณจะถูกแปลงเป็น embedding ระบบเปรียบเทียบ embedding ของคำถามกับ embedding ของชิ้นเอกสารทั้งหมด มันจะหาชิ้นที่มีความหมายใกล้เคียงที่สุด — ไม่ใช่แค่คำหลักที่ตรงกัน แต่เป็นความเหมือนในเชิงความหมายจริงๆ

```java
Embedding queryEmbedding = embeddingModel.embed(question).content();

EmbeddingSearchRequest searchRequest = EmbeddingSearchRequest.builder()
    .queryEmbedding(queryEmbedding)
    .maxResults(5)
    .minScore(0.5)
    .build();

EmbeddingSearchResult<TextSegment> searchResult = embeddingStore.search(searchRequest);
List<EmbeddingMatch<TextSegment>> matches = searchResult.matches();

for (EmbeddingMatch<TextSegment> match : matches) {
    String relevantText = match.embedded().text();
    double score = match.score();
}
```


แผนภาพด้านล่างเปรียบเทียบการค้นหาเชิงความหมายกับการค้นหาคำหลักแบบดั้งเดิม การค้นหาคำหลัก “vehicle” จะพลาดชิ้นที่เกี่ยวกับ “cars and trucks” แต่การค้นหาเชิงความหมายเข้าใจความหมายเดียวกันและส่งคืนเป็นผลลัพธ์ที่มีคะแนนสูง:

<img src="../../../translated_images/th/semantic-search.6b790f21c86b849d.webp" alt="Semantic Search" width="800"/>

*แผนภาพนี้เปรียบเทียบการค้นหาด้วยคำหลักกับการค้นหาเชิงความหมาย แสดงให้เห็นว่าการค้นหาเชิงความหมายจะดึงเนื้อหาที่เกี่ยวข้องแม้ว่าคำหลักจะไม่ตรงกัน*

เบื้องหลัง ความเหมือนถูกวัดโดย cosine similarity — นั่นคือการถามว่า “ลูกศรสองตัวนี้ชี้ไปในทิศทางเดียวกันหรือไม่?” ชิ้นสองชิ้นอาจใช้คำแตกต่างกันโดยสิ้นเชิง แต่ถ้ามีความหมายเดียวกันเวกเตอร์ของพวกมันจะชี้ทิศทางเดียวกันและได้คะแนนใกล้เคียง 1.0:

### คะแนนความคล้ายคลึง

ชิ้นส่วนที่ดึงมาทุกชิ้นจะมีคะแนนความคล้ายคลึงระหว่าง 0 ถึง 1 ซึ่งบ่งบอกว่าเข้ากับคำถามของผู้ใช้อย่างใกล้ชิดแค่ไหน แผนภาพด้านล่างแสดงช่วงคะแนนและการใช้งานเพื่อกรองผลลัพธ์:

<img src="../../../translated_images/th/similarity-scores.b0716aa911abf7f0.webp" alt="Similarity Scores" width="800"/>

*แผนภาพนี้แสดงช่วงคะแนนจาก 0 ถึง 1 โดยมีเกณฑ์ขั้นต่ำที่ 0.5 สำหรับกรองชิ้นส่วนที่ไม่เกี่ยวข้องออก*

คะแนนอยู่ในช่วง 0 ถึง 1:
- 0.7-1.0: เกี่ยวข้องสูง ตรงกันเป๊ะ
- 0.5-0.7: เกี่ยวข้อง บริบทดี
- ต่ำกว่า 0.5: ถูกกรองออก เพราะไม่เหมือนกันพอ

ระบบจะดึงเฉพาะชิ้นส่วนที่คะแนนเกินเกณฑ์ขั้นต่ำเพื่อคุณภาพ

Embeddings ทำงานได้ดีเมื่อตัวหมายความมีการจัดกลุ่มชัดเจน แต่ก็มีข้อจำกัด แผนภาพด้านล่างแสดงรูปแบบความผิดพลาดทั่วไป — ชิ้นส่วนที่ใหญ่มากทำให้เวกเตอร์ไม่ชัด ชิ้นส่วนเล็กเกินไปขาดบริบท คำที่มีความหมายหลายอย่างชี้หลายกลุ่ม และการค้นหาตรง ๆ (ID, หมายเลขชิ้นส่วน) ใช้ embeddings ไม่ได้เลย:

<img src="../../../translated_images/th/embedding-failure-modes.b2bcb901d8970fc0.webp" alt="Embedding Failure Modes" width="800"/>

*แผนภาพนี้แสดงรูปแบบข้อผิดพลาดทั่วไปของ embeddings: ชิ้นส่วนใหญ่เกินไป, ชิ้นส่วนเล็กเกินไป, คำที่กํากว้างชี้ไปยังหลายกลุ่ม และการค้นหาตรง เช่น ID*

## เมื่อไหร่ที่ RAG สำคัญ

RAG ไม่ใช่วิธีที่เหมาะกับทุกสถานการณ์ คู่มือตัดสินใจด้านล่างจะช่วยให้คุณรู้ว่าเมื่อใดควรใช้ RAG และเมื่อใดที่วิธีง่าย ๆ — เช่น ใส่เนื้อหาใน prompt โดยตรง หรือใช้ความรู้ในตัวโมเดล — เหมาะสมกว่า:

<img src="../../../translated_images/th/when-to-use-rag.1016223f6fea26bc.webp" alt="When to Use RAG" width="800"/>

*แผนภาพนี้แสดงคู่มือตัดสินใจว่าควรใช้ RAG หรือใช้วิธีง่ายแทน*

**ใช้ RAG เมื่อ:**
- ตอบคำถามเกี่ยวกับเอกสารเฉพาะ
- ข้อมูลเปลี่ยนแปลงบ่อย (นโยบาย ราคา สเปค)
- ความแม่นยำต้องการการอ้างอิงแหล่งที่มา
- เนื้อหามากเกินกว่าจะใส่ใน prompt เดียว
- ต้องการคำตอบที่ตรวจสอบได้และมีฐานข้อมูล

**อย่าใช้ RAG เมื่อ:**
- คำถามต้องการความรู้ทั่วไปที่โมเดลมีแล้ว
- ต้องการข้อมูลแบบเรียลไทม์ (RAG ทำงานกับเอกสารที่อัปโหลดเท่านั้น)
- เนื้อหามีน้อยพอที่จะใส่ใน prompt ได้โดยตรง
