## آپ کیا سیکھیں گے

اب تک، آپ نے سیکھا ہے کہ کیسے AI کے ساتھ گفتگو کی جاتی ہے، مؤثر انداز میں پرامپٹس بنائے جاتے ہیں، اور اپنے دستاویزات میں جوابات کی بنیاد رکھی جاتی ہے۔ لیکن ایک بنیادی حد ہے: زبان کے ماڈل صرف متن تیار کر سکتے ہیں۔ وہ موسم کا حال چیک نہیں کر سکتے، حساب کتاب نہیں کر سکتے، ڈیٹا بیس کو استفسار نہیں کر سکتے، یا خارجی نظاموں سے تعامل نہیں کر سکتے۔

ٹولز اس کو بدل دیتے ہیں۔ ماڈل کو ایسے فنکشن تک رسائی دے کر جو وہ کال کر سکتا ہے، آپ اسے صرف ایک متن جنریٹر سے ایک ایسے ایجنٹ میں تبدیل کر دیتے ہیں جو کارروائیاں کر سکتا ہے۔ ماڈل فیصلہ کرتا ہے کہ اسے کب ٹول کی ضرورت ہے، کونسا ٹول استعمال کرنا ہے، اور کون سے پیرامیٹرز پاس کرنے ہیں۔ آپ کا کوڈ فنکشن کو چلاتا ہے اور نتیجہ واپس بھیجتا ہے۔ ماڈل اس نتیجے کو اپنے جواب میں شامل کرتا ہے۔

## ٹولز کے ساتھ اے آئی ایجنٹس کو سمجھنا

> **📝 نوٹ:** اس ماڈیول میں "ایجنٹس" سے مراد ایسے AI اسسٹنٹس ہیں جنہیں ٹول کالنگ کے اہل بنایا گیا ہے۔ یہ اس سے مختلف ہے جو ہم [ماڈیول 05: MCP](../05-mcp/README.md) میں کور کریں گے، یعنی **Agentic AI** پیٹرنز (جو خود مختار ایجنٹس ہوتے ہیں جن میں منصوبہ بندی، یادداشت، اور متعدد مرحلوں پر مبنی استدلال ہوتا ہے)۔

بغیر ٹولز کے، زبان کا ماڈل صرف اپنے تربیتی ڈیٹا سے متن تیار کر سکتا ہے۔ اگر آپ موسم حال پوچھیں، تو اسے اندازہ لگانا پڑے گا۔ اگر ٹولز دیں، تو یہ موسم کی API کال کر سکتا ہے، حساب لگا سکتا ہے، یا ڈیٹا بیس کو استفسار کر سکتا ہے — پھر وہ ان حقیقی نتائج کو اپنے جواب میں شامل کر لیتا ہے۔

<img src="../../../translated_images/ur/what-are-tools.724e468fc4de64da.webp" alt="Without Tools vs With Tools" width="800"/>

*بغیر ٹولز کے ماڈل صرف اندازہ لگا سکتا ہے — ٹولز کے ساتھ یہ APIs کال کر سکتا ہے، حساب لگا سکتا ہے، اور حقیقی وقت کا ڈیٹا واپس کر سکتا ہے۔*

ٹولز کے ساتھ ایک AI ایجنٹ ایک **Reasoning and Acting (ReAct)** پیٹرن پر عمل کرتا ہے۔ ماڈل صرف جواب نہیں دیتا — یہ سوچتا ہے کہ اسے کیا چاہیے، ایک ٹول کال کر کے عمل کرتا ہے، نتیجہ مشاہدہ کرتا ہے، اور پھر فیصلہ کرتا ہے کہ دوبارہ عمل کرے یا حتمی جواب دے:

1. **تفکر** — ایجنٹ صارف کے سوال کا تجزیہ کرتا ہے اور معلوم کرتا ہے کہ اسے کونسی معلومات چاہیے
2. **عمل** — ایجنٹ مناسب ٹول منتخب کرتا ہے، درست پیرامیٹرز تیار کرتا ہے، اور کال کرتا ہے
3. **مشاہدہ** — ایجنٹ ٹول کا آؤٹ پٹ وصول کرتا ہے اور نتیجہ کا جائزہ لیتا ہے
4. **دہرائیں یا جواب دیں** — اگر مزید ڈیٹا چاہیے، تو ایجنٹ لوپ میں واپس آتا ہے؛ ورنہ ایک قدرتی زبان کا جواب تیار کرتا ہے

<img src="../../../translated_images/ur/react-pattern-detail.96a5efeeb6dd2f61.webp" alt="ReAct Pattern" width="800"/>

*ری ایکٹ سائیکل — ایجنٹ سوچتا ہے کہ کیا کرنا ہے، ٹول کال کر کے عمل کرتا ہے، نتیجہ دیکھتا ہے، اور پھر لوپ چلاتا ہے جب تک کہ حتمی جواب نہ دے سکے۔*

یہ خودکار ہوتا ہے۔ آپ ٹولز اور ان کی تفصیلات متعین کرتے ہیں۔ ماڈل اس بات کا فیصلہ کرتا ہے کہ کب اور کیسے ان کا استعمال کرنا ہے۔

## ٹول کالنگ کیسے کام کرتی ہے

### ٹول کی تعریفیں

[WeatherTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) | [TemperatureTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/TemperatureTool.java)

آپ واضح تفصیلات اور پیرامیٹر کی وضاحت کے ساتھ فنکشنز تعریف کرتے ہیں۔ ماڈل اپنی سسٹم پرامپٹ میں یہ تفصیلات دیکھتا ہے اور سمجھتا ہے کہ ہر ٹول کیا کرتا ہے۔

```java
@Component
public class WeatherTool {
    
    @Tool("Get the current weather for a location")
    public String getCurrentWeather(@P("Location name") String location) {
        // آپ کی موسمی صورتحال تلاش کرنے کی منطق
        return "Weather in " + location + ": 22°C, cloudy";
    }
}

@AiService
public interface Assistant {
    String chat(@MemoryId String sessionId, @UserMessage String message);
}

// اسسٹنٹ خود بخود Spring Boot کے ذریعے منسلک ہوتا ہے:
// - ChatModel bean
// - تمام @Component کلاسوں کے @Tool طریقے
// - سیشن مینجمنٹ کے لیے ChatMemoryProvider
```

نیچے دیا گیا ڈایاگرام ہر اینوٹیشن کو توڑ کر دکھاتا ہے اور یہ بتاتا ہے کہ ہر حصہ AI کو کس طرح مدد دیتا ہے کہ وہ کب ٹول کال کرے اور کون سے دلائل پاس کرے:

<img src="../../../translated_images/ur/tool-definitions-anatomy.f6468546037cf28b.webp" alt="Anatomy of Tool Definitions" width="800"/>

*ٹول کی تعریف کا تجزیہ — @Tool AI کو بتاتا ہے کہ اسے کب استعمال کرنا ہے، @P ہر پیرامیٹر کو بیان کرتا ہے، اور @AiService اسٹارٹ اپ پر سب کو وائر کرتا ہے۔*

> **🤖 کوشش کریں [GitHub Copilot](https://github.com/features/copilot) چیٹ کے ساتھ:** [`WeatherTool.java`](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) کھولیں اور پوچھیں:
> - "میں موک ڈیٹا کے بجائے حقیقی موسم API جیسے OpenWeatherMap کو کیسے ضم کروں؟"
> - "ایک اچھا ٹول ڈسکرپشن کیا ہوتا ہے جو AI کو اسے صحیح استعمال کرنے میں مدد دیتا ہے؟"
> - "میں API کی غلطیوں اور ریٹ لمٹس کو ٹولز کی امپلیمینٹیشن میں کیسے ہینڈل کروں؟"

### فیصلہ سازی

جب صارف پوچھتا ہے "سیئٹل میں موسم کیسا ہے؟"، تو ماڈل بے ترتیب کسی ٹول کو نہیں چنتا۔ یہ صارف کے ارادے کو ہر ٹول کی تفصیل کے مقابلے میں رکھتا ہے، ہر ایک کی مطابقت کے حساب سے اسکور کرتا ہے، اور بہترین میچ منتخب کرتا ہے۔ پھر یہ ایک مربوط فنکشن کال جنریٹ کرتا ہے جس میں درست پیرامیٹرز ہوتے ہیں— اس معاملے میں `location` کو `"Seattle"` مقرر کرتا ہے۔

اگر کوئی ٹول صارف کی درخواست سے میل نہیں کھاتا، تو ماڈل اپنے علم سے جواب دیتا ہے۔ اگر کئی ٹولز میل کھاتے ہیں، تو سب سے مخصوص کو چنتا ہے۔

<img src="../../../translated_images/ur/decision-making.409cd562e5cecc49.webp" alt="How the AI Decides Which Tool to Use" width="800"/>

*ماڈل ہر دستیاب ٹول کو صارف کے ارادے کے مقابلے میں جانچتا ہے اور بہترین میچ منتخب کرتا ہے — اسی لیے صاف، مخصوص ٹول کی تفصیلات لکھنا اہم ہے۔*

### جواب تیار کرنا

ماڈل موسم کا ڈیٹا وصول کرتا ہے اور صارف کے لیے اسے قدرتی زبان کے جواب کی شکل دیتا ہے۔

### معماری: اسپرنگ بوٹ آٹو-وائرنگ

یہ ماڈیول LangChain4j کی اسپرنگ بوٹ انٹیگریشن استعمال کرتا ہے جس میں اعلان کردہ `@AiService` انٹرفیسز ہوتے ہیں۔ اسٹارٹ اپ پر، اسپرنگ بوٹ ہر `@Component` کو تلاش کرتا ہے جس میں `@Tool` میتھڈز ہوتے ہیں، آپ کا `ChatModel` بین، اور `ChatMemoryProvider` — پھر ان سب کو زیرو بویلر پلیٹ کے ساتھ ایک `Assistant` انٹرفیس میں وائر کرتا ہے۔

<img src="../../../translated_images/ur/spring-boot-wiring.151321795988b04e.webp" alt="Spring Boot Auto-Wiring Architecture" width="800"/>

*@AiService انٹرفیس ChatModel، ٹول کمپونینٹس، اور میموری پروائیڈر کو آپس میں جوڑتا ہے — اسپرنگ بوٹ خودکار طور پر تمام وائرنگ سنبھالتا ہے۔*

یہاں مکمل درخواست کی لائف سائیکل بطور ترتیب وار ڈایاگرام دی گئی ہے — HTTP درخواست سے کنٹرولر، سروس، اور آٹو وائرڈ پراکسی کے ذریعے، ٹول کے عمل درآمد اور واپس تک:

<img src="../../../translated_images/ur/spring-boot-sequence.f83e3d485aa4a3c6.webp" alt="Spring Boot Tool Calling Sequence" width="800"/>

*مکمل اسپرنگ بوٹ درخواست کی لائف سائیکل — HTTP درخواست کنٹرولر اور سروس سے آٹو وائرڈ Assistant پراکسی تک جاتی ہے، جو LLM اور ٹول کالز کو خودکار طور پر منظم کرتا ہے۔*

اس طریقہ کار کے اہم فوائد:

- **اسپرنگ بوٹ آٹو وائرنگ** — ChatModel اور ٹولز خود بخود انجیکٹ کیے جاتے ہیں
- **@MemoryId پیٹرن** — خودکار سیشن کی بنیاد پر میموری مینجمنٹ
- **ایک ہی انسٹانس** — Assistant کو ایک بار تخلیق کیا جاتا ہے اور بہتر کارکردگی کے لیے دوبارہ استعمال کیا جاتا ہے
- **ٹائپ سیف ایکسیکیوشن** — جاوا میتھڈز کو براہ راست کال کیا جاتا ہے ٹائپ کنورژن کے ساتھ
- **ملٹی ٹرن آرسٹراشن** — خودکار طور پر ٹول چیننگ کو ہینڈل کرتا ہے
- **زیرو بویلر پلیٹ** — بغیر کسی دستی `AiServices.builder()` کال یا میموری ہیش میپ کے

متبادل طریقے (دستی `AiServices.builder()`) زیادہ کوڈ چاہیے اور اسپرنگ بوٹ انٹیگریشن کے فوائد سے محروم ہوتے ہیں۔

## ٹول چیننگ

**ٹول چیننگ** — ٹول بیسڈ ایجنٹس کی اصل طاقت اسی وقت ظاہر ہوتی ہے جب ایک سوال کے لیے متعدد ٹولز کی ضرورت ہو۔ پوچھیں "سیئٹل میں فارن ہائیٹ میں موسم کیسا ہے؟" اور ایجنٹ خود بخود دو ٹولز کو جوڑتا ہے: پہلے یہ `getCurrentWeather` کو کال کرتا ہے تاکہ سیلسیس میں درجہ حرارت معلوم کرے، پھر اس ویلیو کو `celsiusToFahrenheit` کو بھیجتا ہے تاکہ کنورژن ہو — یہ سب ایک ہی گفتگو کے چکر میں ہوتا ہے۔

<img src="../../../translated_images/ur/tool-chaining-example.538203e73d09dd82.webp" alt="Tool Chaining Example" width="800"/>

*ٹول چیننگ کا عمل — ایجنٹ پہلے getCurrentWeather کال کرتا ہے، پھر سیلسیس نتیجہ celsiusToFahrenheit میں بھیجتا ہے، اور مشترکہ جواب دیتا ہے۔*

**نرمی سے ناکامیاں** — جب کسی ایسے شہر کا موسم پوچھیں جو موک ڈیٹا میں نہیں ہے۔ ٹول ایک ایرر میسج واپس کرتا ہے، اور AI بتاتا ہے کہ وہ مدد نہیں کر سکتا بجائے اس کے کہ ایپلیکیشن کریش ہو جائے۔ ٹولز محفوظ طریقے سے ناکام ہوتے ہیں۔ نیچے دیا گیا ڈایاگرام دونوں طریقوں کا موازنہ کرتا ہے — مناسب ایرر ہینڈلنگ کے ساتھ، ایجنٹ استثنا کو پکڑتا ہے اور مددگار جواب دیتا ہے، جبکہ بغیر اس کے پورا ایپلیکیشن کریش ہو جاتا ہے:

<img src="../../../translated_images/ur/error-handling-flow.9a330ffc8ee0475c.webp" alt="Error Handling Flow" width="800"/>

*جب ٹول ناکام ہوتا ہے، تو ایجنٹ ایرر کو پکڑتا ہے اور کریش ہونے کی بجائے مددگار وضاحت کے ساتھ جواب دیتا ہے۔*

یہ سب ایک ہی گفتگو کے چکر میں ہوتا ہے۔ ایجنٹ خود بخود متعدد ٹول کالز کا انتظام کرتا ہے۔

## ایپلیکیشن کا استعمال

ایپلیکیشن ایک ویب انٹرفیس فراہم کرتی ہے جہاں آپ ایک AI ایجنٹ کے ساتھ تعامل کر سکتے ہیں جس کے پاس موسم اور درجہ حرارت کی تبدیلی کے آلات تک رسائی ہے۔ انٹرفیس کچھ یوں دکھتا ہے — اس میں فوری شروع کرنے کی مثالیں اور درخواست بھیجنے کے لیے چیٹ پینل شامل ہے:

<a href="images/tools-homepage.png"><img src="../../../translated_images/ur/tools-homepage.4b4cd8b2717f9621.webp" alt="AI Agent Tools Interface" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*AI ایجنٹ ٹولز کا انٹرفیس - آلات کے ساتھ تعامل کے لیے فوری مثالیں اور چیٹ انٹرفیس*

### آسان ٹول استعمال کرنے کی کوشش کریں

ایک سادہ درخواست کے ساتھ شروع کریں: "100 ڈگری فارن ہائیٹ کو سیلسیس میں تبدیل کریں"۔ ایجنٹ پہچانتا ہے کہ اسے درجہ حرارت تبدیلی کے ٹول کی ضرورت ہے، اسے صحیح پیرامیٹرز کے ساتھ کال کرتا ہے، اور نتیجہ واپس کرتا ہے۔ دیکھیں کہ یہ کتنا قدرتی محسوس ہوتا ہے — آپ نے وضاحت نہیں کی کہ کون سا ٹول استعمال کرنا ہے یا اسے کیسے کال کرنا ہے۔

### ٹول چیننگ ٹیسٹ کریں

اب کچھ زیادہ پیچیدہ کوشش کریں: "سیئٹل کا موسم کیا ہے اور اسے فارن ہائیٹ میں تبدیل کریں؟" دیکھیں کہ ایجنٹ اسے مرحلہ در مرحلہ کیسے حل کرتا ہے۔ سب سے پہلے وہ موسم معلوم کرتا ہے (جو سیلسیس واپس کرتا ہے)، پھر اسے معلوم ہوتا ہے کہ اسے فارن ہائیٹ میں تبدیل کرنا ہے، تبدیلی کے ٹول کو کال کرتا ہے، اور دونوں نتائج کو ایک جواب میں جوڑ دیتا ہے۔

### گفتگو کے بہاؤ کو دیکھیں

چیٹ انٹرفیس گفتگو کی تاریخ کو برقرار رکھتا ہے، جو آپ کو متعدد مرحلوں کی بات چیت کرنے کی اجازت دیتا ہے۔ آپ تمام پچھلے سوالات اور جوابات دیکھ سکتے ہیں، جس سے گفتگو کو ٹریک کرنا اور سمجھنا آسان ہو جاتا ہے کہ ایجنٹ کس طرح متعدد تبادلوں کے دوران سیاق و سباق بناتا ہے۔

<a href="images/tools-conversation-demo.png"><img src="../../../translated_images/ur/tools-conversation-demo.89f2ce9676080f59.webp" alt="Conversation with Multiple Tool Calls" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*متعدد مرحلوں والی گفتگو میں سادہ تبدیلیاں، موسم کی جانچ، اور ٹول چیننگ دکھائی گئی ہے*

### مختلف درخواستوں کے ساتھ تجربہ کریں

مختلف امتزاج آزمایں:
- موسمی جانچ: "ٹوکیو میں موسم کیسا ہے؟"
- درجہ حرارت کی تبدیلیاں: "25°C کلوئن میں کیا ہے؟"
- مشترکہ سوالات: "پیرس میں موسم دیکھیں اور بتائیں کہ کیا یہ 20°C سے اوپر ہے"

دیکھیں کہ ایجنٹ قدرتی زبان کو کیسے سمجھتا ہے اور اسے مناسب ٹول کالز میں کیسے تبدیل کرتا ہے۔

## اہم تصورات

### ReAct پیٹرن (منطق اور عمل)

ایجنٹ سوچ بچار (فیصلہ کرنا کہ کیا کرنا ہے) اور عمل (ٹولز کا استعمال) کے درمیان تبدیلی کرتا ہے۔ یہ پیٹرن خود مختار مسئلہ حل کرنے کی اجازت دیتا ہے نہ کہ صرف ہدایات کا جواب دینا۔

### ٹول کے تفصیلات اہم ہیں

آپ کے ٹولز کی وضاحت کی کوالٹی براہ راست متاثر کرتی ہے کہ ایجنٹ انہیں کتنی خوبی سے استعمال کرتا ہے۔ واضح، مخصوص وضاحتیں ماڈل کو سمجھانے میں مدد دیتی ہیں کہ کب اور کیسے ہر ٹول کو کال کرنا ہے۔

### سیشن مینجمنٹ

`@MemoryId` تشریح خودکار سیشن پر مبنی میموری مینجمنٹ کی سہولت دیتی ہے۔ ہر سیشن ID کو `ChatMemory` کا اپنا ایک انسٹنس ملتا ہے جو `ChatMemoryProvider` بین کے ذریعے مینج کیا جاتا ہے، اس طرح متعدد صارفین بیک وقت ایجنٹ کے ساتھ اس طرح بات چیت کر سکتے ہیں کہ ان کی گفتگو ایک دوسرے میں ملا نہیں۔ نیچے دیا گیا خاکہ دکھاتا ہے کہ کس طرح مختلف صارفین کو ان کے سیشن IDs کی بنیاد پر الگ الگ میموری اسٹورز کی طرف بھیجا جاتا ہے:

<img src="../../../translated_images/ur/session-management.91ad819c6c89c400.webp" alt="Session Management with @MemoryId" width="800"/>

*ہر سیشن ID ایک الگ گفتگو کی تاریخ سے منسلک ہے — صارفین ایک دوسرے کے پیغامات کبھی نہیں دیکھتے۔*

### خرابی کا انتظام

ٹولز ناکام ہو سکتے ہیں — APIs ٹائم آؤٹ ہو سکتے ہیں، پیرامیٹر غلط ہو سکتے ہیں، بیرونی سروسز بند ہو سکتی ہیں۔ پروڈکشن ایجنٹس کو خرابی کے انتظام کی ضرورت ہوتی ہے تاکہ ماڈل مسائل کی وضاحت کر سکے یا متبادل آزما سکے بجائے اس کے کہ پورے ایپلیکیشن کریش ہو جائے۔ جب کوئی ٹول ایک استثنا پھینکتا ہے، تو LangChain4j اسے پکڑ لیتا ہے اور غلطی کا پیغام ماڈل کو واپس بھیجتا ہے، جو پھر قدرتی زبان میں مسئلہ بیان کر سکتا ہے۔

نیچے دیا گیا خاکہ ٹولز کے وسیع ماحولیاتی نظام کو دکھاتا ہے جو آپ بنا سکتے ہیں۔ یہ ماڈیول موسم اور درجہ حرارت کے ٹولز کی مثال دیتا ہے، لیکن وہی `@Tool` پیٹرن کسی بھی جاوا میتھڈ کے لیے کام کرتا ہے — چاہے وہ ڈیٹا بیس کی پوچھ گچھ ہو یا ادائیگی کی پراسیسنگ۔
