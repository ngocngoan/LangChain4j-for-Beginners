## آپ کیا سیکھیں گے

آپ نے بات چیت کرنے والی AI بنائی ہے، پرامپٹس میں مہارت حاصل کی ہے، جوابات کو دستاویزات میں جمایا ہے، اور ٹولز کے ساتھ ایجنٹس تیار کیے ہیں۔ لیکن یہ تمام ٹولز آپ کی مخصوص ایپلی کیشن کے لیے مخصوص طور پر بنائے گئے تھے۔ اگر آپ اپنی AI کو ایک معیاری ٹولز کے نظام تک رسائی دے سکیں جو کوئی بھی بنا اور بانٹ سکتا ہے؟ اس ماڈیول میں، آپ Model Context Protocol (MCP) اور LangChain4j کے ایجنٹک ماڈیول کے ذریعے یہ کرنا سیکھیں گے۔ ہم پہلے ایک سادہ MCP فائل ریڈر دکھائیں گے اور پھر دکھائیں گے کہ یہ کیسے آسانی سے ایڈوانسڈ ایجنٹک ورک فلو میں سپروائزر ایجنٹ پیٹرن کے ذریعے ضم ہوتا ہے۔

## MCP کیا ہے؟

Model Context Protocol (MCP) بالکل یہی فراہم کرتا ہے — ایک معیاری طریقہ AI ایپلی کیشنز کو خارجی ٹولز دریافت کرنے اور استعمال کرنے کا۔ ہر ڈیٹا سورس یا سروس کے لیے کسٹم انٹیگریشن لکھنے کے بجائے، آپ MCP سرورز سے جڑتے ہیں جو اپنی صلاحیتیں ایک مستقل فارمیٹ میں پیش کرتے ہیں۔ آپ کا AI ایجنٹ پھر خود بخود ان ٹولز کو دریافت اور استعمال کر سکتا ہے۔

نیچے دیا گیا ڈایاگرام فرق دکھاتا ہے — MCP کے بغیر ہر انٹیگریشن کے لیے کسٹم نقطہ بہ نقطہ وائرنگ کی ضرورت ہوتی ہے؛ MCP کے ساتھ، ایک پروٹوکول آپ کی ایپ کو کسی بھی ٹول سے جوڑ دیتا ہے:

<img src="../../../translated_images/ur/mcp-comparison.9129a881ecf10ff5.webp" alt="MCP کی موازنہ" width="800"/>

*MCP سے پہلے: پیچیدہ نقطہ بہ نقطہ انٹیگریشن۔ MCP کے بعد: ایک پروٹوکول، لا متناہی امکانات۔*

MCP AI ڈیولپمنٹ میں ایک بنیادی مسئلہ حل کرتا ہے: ہر انٹیگریشن کسٹم ہوتی ہے۔ GitHub تک رسائی چاہتے ہیں؟ کسٹم کوڈ۔ فائلیں پڑھنا چاہیئں؟ کسٹم کوڈ۔ ڈیٹا بیس کو کوئری کرنا ہے؟ کسٹم کوڈ۔ اور یہ انٹیگریشنز دوسری AI ایپلی کیشنز کے ساتھ کام نہیں کرتے۔

MCP اسے معیاری بناتا ہے۔ ایک MCP سرور ٹولز کو واضح تفصیلات اور اسکیموں کے ساتھ ظاہر کرتا ہے۔ کوئی بھی MCP کلائنٹ جڑ سکتا ہے، دستیاب ٹولز کو دریافت کر سکتا ہے، اور استعمال کر سکتا ہے۔ ایک بار بنائیں، ہر جگہ استعمال کریں۔

نیچے دیا گیا ڈایاگرام اس فن تعمیر کی تصویر کشی کرتا ہے — ایک MCP کلائنٹ (آپ کی AI ایپلی کیشن) متعدد MCP سرورز سے جڑتا ہے، ہر ایک اپنے ٹولز کو معیاری پروٹوکول کے ذریعے پیش کرتا ہے:

<img src="../../../translated_images/ur/mcp-architecture.b3156d787a4ceac9.webp" alt="MCP فن تعمیر" width="800"/>

## MCP کیسے کام کرتا ہے

اندرونی طور پر، MCP ایک پرت بندی شدہ فن تعمیر استعمال کرتا ہے۔ آپ کی جاوا ایپلی کیشن (MCP کلائنٹ) دستیاب ٹولز کو دریافت کرتی ہے، JSON-RPC درخواستیں ٹرانسپورٹ لیئر (Stdio یا HTTP) کے ذریعے بھیجتی ہے، اور MCP سرور آپریشنز انجام دیتا ہے اور نتائج واپس کرتا ہے۔ ذیل کا ڈایاگرام اس پروٹوکول کی ہر پرت کو واضح کرتا ہے:

<img src="../../../translated_images/ur/mcp-protocol-detail.01204e056f45308b.webp" alt="MCP پروٹوکول کی تفصیل" width="800"/>

*MCP کیسے کام کرتا ہے — کلائنٹس ٹولز دریافت کرتے ہیں، JSON-RPC پیغامات کا تبادلہ کرتے ہیں، اور ٹرانسپورٹ لیئر کے ذریعے آپریشنز انجام دیتے ہیں۔*

**سرور-کلائنٹ فن تعمیر**

MCP کلائنٹ-سرور ماڈل استعمال کرتا ہے۔ سرورز ٹولز فراہم کرتے ہیں — فائلیں پڑھنا، ڈیٹا بیس کو کوئری کرنا، APIs کو کال کرنا۔ کلائنٹس (آپ کی AI ایپ) سرورز سے جڑتے ہیں اور ان کے ٹولز استعمال کرتے ہیں۔

LangChain4j کے ساتھ MCP استعمال کرنے کے لیے، یہ Maven انحصار شامل کریں:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```

**ٹول دریافت**

جب آپ کا کلائنٹ MCP سرور سے جڑتا ہے، تو وہ پوچھتا ہے "آپ کے پاس کون سے ٹولز ہیں؟" سرور دستیاب ٹولز کی فہرست جواب میں دیتا ہے، ہر ایک کے ساتھ تفصیلات اور پیرامیٹر اسکیمے ہوتے ہیں۔ آپ کا AI ایجنٹ پھر صارف کی درخواستوں کی بنیاد پر فیصلہ کر سکتا ہے کہ کون سے ٹولز استعمال کرنے ہیں۔ نیچے دیا گیا ڈایاگرام اس ہینڈشیک کو دکھاتا ہے — کلائنٹ `tools/list` درخواست بھیجتا ہے اور سرور دستیاب ٹولز تفصیل اور اسکیمے کے ساتھ واپس کرتا ہے:

<img src="../../../translated_images/ur/tool-discovery.07760a8a301a7832.webp" alt="MCP ٹول دریافت" width="800"/>

*AI اسٹارٹ اپ پر دستیاب ٹولز دریافت کرتا ہے — اب اسے معلوم ہے کہ کون سی صلاحیتیں دستیاب ہیں اور کون سے استعمال کرنے ہیں۔*

**ٹرانسپورٹ میکانزم**

MCP مختلف ٹرانسپورٹ میکانزمز کی حمایت کرتا ہے۔ دو آپشنز ہیں Stdio (مقامی ذیلی عمل مواصلات کے لیے) اور Streamable HTTP (دور دراز سرورز کے لیے)۔ یہ ماڈیول Stdio ٹرانسپورٹ کا مظاہرہ کرتا ہے:

<img src="../../../translated_images/ur/transport-mechanisms.2791ba7ee93cf020.webp" alt="ٹرانسپورٹ میکانزم" width="800"/>

*MCP ٹرانسپورٹ میکانزمز: HTTP دور دیئے گئے سرورز کے لیے، Stdio مقامی عمل کے لیے*

**Stdio** - [StdioTransportDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java)

مقامی پروسیسز کے لیے۔ آپ کی ایپلیکیشن ایک سرور کو ذیلی عمل کے طور پر شروع کرتی ہے اور معیاری ان پٹ/آؤٹ پٹ کے ذریعے بات چیت کرتی ہے۔ فائل سسٹم تک رسائی یا کمانڈ لائن ٹولز کے لیے مفید۔

```java
McpTransport stdioTransport = new StdioMcpTransport.Builder()
    .command(List.of(
        npmCmd, "exec",
        "@modelcontextprotocol/server-filesystem@2025.12.18",
        resourcesDir
    ))
    .logEvents(false)
    .build();
```

`@modelcontextprotocol/server-filesystem` سرور درج ذیل ٹولز پیش کرتا ہے، تمام آپ کے مخصوص کردہ فولڈرز کے لیے محدود:

| ٹول | وضاحت |
|------|-------------|
| `read_file` | ایک فائل کا مواد پڑھیں |
| `read_multiple_files` | ایک کال میں متعدد فائلیں پڑھیں |
| `write_file` | فائل بنائیں یا اوور رائٹ کریں |
| `edit_file` | مخصوص فائنڈ اینڈ ریپلیس ترمیمات کریں |
| `list_directory` | کسی راستے پر فائلیں اور ڈائریکٹریز کی فہرست بنائیں |
| `search_files` | کسی پیٹرن سے میل کھانے والی فائلوں کی ریڈرسو سرچ کریں |
| `get_file_info` | فائل کے میٹا ڈیٹا (سائز، ٹائم اسٹیمپس، اجازتیں) حاصل کریں |
| `create_directory` | ڈائریکٹری بنائیں (والدین والی ڈائریکٹریز سمیت) |
| `move_file` | فائل یا ڈائریکٹری کو منتقل یا نام تبدیل کریں |

ذیل کا ڈایاگرام بتاتا ہے کہ Stdio ٹرانسپورٹ رن ٹائم پر کیسے کام کرتا ہے — آپ کی جاوا ایپ MCP سرور کو بطور ذیلی عمل شروع کرتی ہے اور وہ stdin/stdout پائپس کے ذریعے بات چیت کرتے ہیں، کوئی نیٹ ورک یا HTTP شامل نہیں:

<img src="../../../translated_images/ur/stdio-transport-flow.45eaff4af2d81db4.webp" alt="Stdio ٹرانسپورٹ فلو" width="800"/>

*Stdio ٹرانسپورٹ عمل میں — آپ کی ایپ MCP سرور کو بچہ عمل کی طرح شروع کرتی ہے اور stdin/stdout پائپس کے ذریعے بات چیت کرتی ہے۔*

> - "Stdio ٹرانسپورٹ کیسے کام کرتا ہے اور اسے HTTP کے مقابلے کب استعمال کرنا چاہیے؟"
> - "LangChain4j MCP سرور کے شروع کیے گئے عملوں کی لائف سائیکل کیسے مینیج کرتا ہے؟"
> - "AI کو فائل سسٹم کی رسائی دینے کے سیکورٹی مسائل کیا ہیں؟"

## ایجنٹک ماڈیول

جبکہ MCP معیاری ٹولز فراہم کرتا ہے، LangChain4j کا **ایجنٹک ماڈیول** ایجنٹس بنانے کا ایک اظہاری طریقہ فراہم کرتا ہے جو ان ٹولز کو مربوط کرتے ہیں۔ `@Agent` انوٹیشن اور `AgenticServices` آپ کو انٹرفیسز کے ذریعے ایجنٹ کے رویے کی تعریف کرنے دیتے ہیں، بجائے اس کے کہ آپ امپیریٹو کوڈ لکھیں۔

اس ماڈیول میں، آپ **سپروائزر ایجنٹ** پیٹرن کو دریافت کریں گے — ایک متقدم ایجنٹک AI اپروچ جہاں "سپروائزر" ایجنٹ صارف کی درخواستوں کی بنیاد پر متحرک طور پر فیصلہ کرتا ہے کہ کون سے سب-ایجنٹس کو فعال کرنا ہے۔ ہم دونوں تصورات کو ملائیں گے اور ایک سب-ایجنٹ کو MCP کی طاقت فراہم کرکے فائل تک رسائی دیں گے۔

ایجنٹک ماڈیول استعمال کرنے کے لیے، یہ Maven انحصار شامل کریں:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```

> **نوٹ:** `langchain4j-agentic` ماڈیول ایک علیحدہ ورژن پراپرٹی (`langchain4j.mcp.version`) استعمال کرتا ہے کیونکہ اسے کور LangChain4j لائبریری سے الگ شیڈول پر جاری کیا جاتا ہے۔

> **⚠️ تجرباتی:** `langchain4j-agentic` ماڈیول **تجرباتی** ہے اور تبدیلی کے تابع ہے۔ AI اسسٹنٹس بنانے کا مستحکم طریقہ `langchain4j-core` کے ساتھ کسٹم ٹولز کا استعمال ہے (ماڈیول 04)۔

### سپروائزر ایجنٹ

**سپروائزر ایجنٹ پیٹرن** ایک **لچکدار** قسم کی ایجنٹک AI ہے۔ سپروائزر LLM استعمال کرتا ہے تاکہ خود کار طور پر فیصلہ کرے کہ صارف کی درخواست کی بنیاد پر کون سے ایجنٹس کو فعال کرنا ہے۔ اگلی مثال میں، ہم MCP سے طاقت یافتہ فائل رسائی کو LLM ایجنٹ کے ساتھ جوڑ کر ایک منظم فائل پڑھنے → رپورٹ ورک فلو تیار کرتے ہیں۔

ڈیمو میں، `FileAgent` MCP فائل سسٹم ٹولز استعمال کر کے فائل پڑھتا ہے، اور `ReportAgent` ایک منظم رپورٹ جنریٹ کرتا ہے جس میں ایک ایگزیکٹو خلاصہ (1 جملہ)، 3 اہم نکات، اور سفارشات شامل ہیں۔ سپروائزر اس فلو کو خود بخود منظم کرتا ہے:

<img src="../../../translated_images/ur/supervisor-agent-pattern.06275a41ae006ac8.webp" alt="سپروائزر ایجنٹ پیٹرن" width="800"/>

*سپروائزر اپنے LLM کے ذریعے فیصلہ کرتا ہے کہ کون سے ایجنٹس کو کس ترتیب میں شروع کرنا ہے — کسی ہارڈ کوڈڈ روٹنگ کی ضرورت نہیں۔*

یہاں ہماری فائل سے رپورٹ پائپ لائن کا ٹھوس ورک فلو دکھایا گیا ہے:

<img src="../../../translated_images/ur/file-report-workflow.649bb7a896800de9.webp" alt="فائل سے رپورٹ ورک فلو" width="800"/>

*FileAgent MCP ٹولز کے ذریعے فائل پڑھتا ہے، پھر ReportAgent خام مواد کو منظم رپورٹ میں تبدیل کرتا ہے۔*

مندرجہ ذیل سیکوینس ڈایاگرام مکمل سپروائزر منظم کاری کی وضاحت کرتا ہے — MCP سرور کی شروعات، سپروائزر کے خود کار ایجنٹ انتخاب، stdio پر ٹول کالز، اور حتمی رپورٹ:

<img src="../../../translated_images/ur/supervisor-agent-sequence.1aa389b3bef99956.webp" alt="سپروائزر ایجنٹ سیکوینس ڈایاگرام" width="800"/>

*سپروائزر خود بخود FileAgent کو فعال کرتا ہے (جو MCP سرور کو stdio کے ذریعے کال کرتا ہے تاکہ فائل پڑھے)، پھر ReportAgent کو ایک منظم رپورٹ تیار کرنے کے لیے۔ ہر ایجنٹ اپنا آؤٹ پٹ مشترکہ Agentic Scope میں محفوظ کرتا ہے۔*

ہر ایجنٹ اپنا آؤٹ پٹ **Agentic Scope** (مشترکہ میموری) میں محفوظ کرتا ہے، جو اگلے ایجنٹس کو پچھلے نتائج تک رسائی فراہم کرتا ہے۔ یہ ظاہر کرتا ہے کہ MCP ٹولز ایجنٹک ورک فلو میں کس طرح آسانی سے ضم ہوتے ہیں — سپروائزر کو یہ جاننے کی ضرورت نہیں ہوتی کہ فائلیں *کیسے* پڑھی جاتی ہیں، صرف یہ معلوم ہے کہ `FileAgent` ایسا کر سکتا ہے۔

#### سپروائزر کیسے کام کرتا ہے

ایجنٹس بنانے سے پہلے، آپ کو MCP ٹرانسپورٹ کو کلائنٹ سے جوڑنا ہوگا اور اسے `ToolProvider` کے طور پر لپیٹنا ہوگا۔ یہ MCP سرور کے ٹولز کو آپ کے ایجنٹس کے لیے دستیاب بناتا ہے:

```java
// ٹرانسپورٹ سے ایک MCP کلائنٹ بنائیں
McpClient mcpClient = new DefaultMcpClient.Builder()
        .transport(stdioTransport)
        .build();

// کلائنٹ کو ایک ToolProvider کے طور پر لپیٹیں — اس طرح MCP ٹولز کو LangChain4j میں جوڑا جاتا ہے
ToolProvider mcpToolProvider = McpToolProvider.builder()
        .mcpClients(List.of(mcpClient))
        .build();
```

اب آپ `mcpToolProvider` کو کسی بھی ایسے ایجنٹ میں شامل کر سکتے ہیں جسے MCP ٹولز کی ضرورت ہو:

```java
// مرحلہ 1: فائل ایجنٹ MCP ٹولز کا استعمال کرتے ہوئے فائلیں پڑھتا ہے
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // فائل کے آپریشنز کے لیے MCP ٹولز موجود ہیں
        .build();

// مرحلہ 2: رپورٹ ایجنٹ منظم شدہ رپورٹیں تیار کرتا ہے
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// سپروائزر فائل → رپورٹ کے ورک فلو کو منظم کرتا ہے
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // آخری رپورٹ واپس کریں
        .build();

// سپروائزر درخواست کی بنیاد پر ہر ایجنٹ کو چلانے کا فیصلہ کرتا ہے
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```

#### FileAgent رن ٹائم پر MCP ٹولز کیسے دریافت کرتا ہے

آپ سوچ سکتے ہیں: **FileAgent کیسے جانتا ہے کہ npm فائل سسٹم ٹولز کو کیسے استعمال کرنا ہے؟** جواب یہ ہے کہ وہ نہیں جانتا — **LLM** رن ٹائم پر ٹول اسکیموں کے ذریعے اسے پتہ لگاتا ہے۔

`FileAgent` انٹرفیس صرف ایک **پرامپٹ تعریف** ہے۔ اسے `read_file`، `list_directory`، یا کسی اور MCP ٹول کا کوئی ہارڈ کوڈڈ علم نہیں ہے۔ یہ ہے پورے عمل کا خلاصہ:
1. **سرور کی شروعات:** `StdioMcpTransport` `@modelcontextprotocol/server-filesystem` npm پیکیج کو ایک چائلڈ پروسیس کے طور پر لانچ کرتا ہے  
2. **ٹول کی دریافت:** `McpClient` سرور کو `tools/list` JSON-RPC درخواست بھیجتا ہے، جو ٹول کے نام، تفصیلات، اور پیرامیٹر اسکیمات کے ساتھ جواب دیتا ہے (مثلاً، `read_file` — *"ایک فائل کا مکمل مواد پڑھیں"* — `{ path: string }`)  
3. **اسکیمہ انجیکشن:** `McpToolProvider` ان دریافت شدہ اسکیمات کو لپیٹ کر LangChain4j کو دستیاب بناتا ہے  
4. **LLM فیصلہ کرتا ہے:** جب `FileAgent.readFile(path)` کال کیا جاتا ہے، LangChain4j سسٹم میسج، یوزر میسج، **اور ٹول اسکیمات کی فہرست** LLM کو بھیجتا ہے۔ LLM ٹول کی تفصیلات پڑھتا ہے اور ٹول کال جنریٹ کرتا ہے (مثلاً، `read_file(path="/some/file.txt")`)  
5. **نفاذ:** LangChain4j ٹول کال کو میان کرتی ہے، اسے MCP کلائنٹ کے ذریعے Node.js سب پروسیس تک بھیجتی ہے، نتیجہ حاصل کرتی ہے، اور اسے واپس LLM کو دیتی ہے  

یہی [ٹول دریافت](../../../05-mcp) کا طریقہ ہے جو اوپر بیان کیا گیا ہے، لیکن خاص طور پر ایجنٹ ورک فلو پر لاگو ہوتا ہے۔ `@SystemMessage` اور `@UserMessage` انوٹیشنز LLM کے رویے کی رہنمائی کرتی ہیں، جبکہ انجیکٹ کیا گیا `ToolProvider` اسے **صلاحیتیں** دیتا ہے — LLM دونوں کو رن ٹائم پر پل کرتا ہے۔

#### جواب دینے کی حکمت عملیاں

جب آپ `SupervisorAgent` کو ترتیب دیتے ہیں، تو آپ وضاحت کرتے ہیں کہ ذیلی ایجنٹس کے کام مکمل ہونے کے بعد صارف کو حتمی جواب کیسے تیار کرنا چاہیے۔ درج ذیل خاکہ تین دستیاب حکمت عملیوں کو دکھاتا ہے — LAST براہِ راست آخری ایجنٹ کا آؤٹ پٹ دیتا ہے، SUMMARY تمام آؤٹ پٹس کو LLM کے ذریعے مکس کرتا ہے، اور SCORED وہ آؤٹ پٹ منتخب کرتا ہے جس کا اسکور اصل درخواست کے مقابلے میں زیادہ ہو:

<img src="../../../translated_images/ur/response-strategies.3d0cea19d096bdf9.webp" alt="Response Strategies" width="800"/>

*Supervisor کے حتمی جواب تیار کرنے کے لیے تین حکمت عملی — اس بات کی بنیاد پر انتخاب کریں کہ آپ آخری ایجنٹ کا آؤٹ پٹ چاہتے ہیں، ایک مجموعی خلاصہ، یا بہترین اسکورنگ آپشن*۔

دستیاب حکمت عملیاں:

| حکمت عملی | وضاحت |
|----------|--------|
| **LAST** | سپروائزر آخری ذیلی ایجنٹ یا کال کیے گئے ٹول کا آؤٹ پٹ دیتا ہے۔ یہ اس وقت مفید ہے جب ورک فلو کا آخری ایجنٹ خاص طور پر مکمل حتمی جواب کے لیے ڈیزائن کیا گیا ہو (مثلاً تحقیق کے پائپ لائن میں "خلاصہ ایجنٹ")۔ |
| **SUMMARY** | سپروائزر اپنا داخلی زبان ماڈل (LLM) استعمال کرتا ہے تاکہ تمام تعامل اور ذیلی ایجنٹس کے آؤٹ پٹس کا خلاصہ تیار کرے، پھر وہ خلاصہ حتمی جواب کے طور پر دیتا ہے۔ یہ صارف کو ایک صاف، مرکب جواب فراہم کرتا ہے۔ |
| **SCORED** | سسٹم اپنا داخلی LLM استعمال کر کے LAST جواب اور SUMMARY کو اصل یوزر درخواست کے خلاف اسکور کرتا ہے، اور جو بھی آؤٹ پٹ زیادہ اسکور حاصل کرے اسے واپس کرتا ہے۔ |

#### آؤٹ پٹ کو سمجھنا

جب آپ ڈیمو چلائیں گے، تو آپ دیکھیں گے کہ سپروائزر کس طرح متعدد ایجنٹس کو منظّم کرتا ہے۔ ہر سیکشن کا مطلب درج ذیل ہے:

```
======================================================================
  FILE → REPORT WORKFLOW DEMO
======================================================================

This demo shows a clear 2-step workflow: read a file, then generate a report.
The Supervisor orchestrates the agents automatically based on the request.
```
  
**سرخی** ورک فلو کے تصور کا تعارف کراتی ہے: فائل پڑھنے سے رپورٹ بنانے تک مرکوز پائپ لائن۔

```
--- WORKFLOW ---------------------------------------------------------
  ┌─────────────┐      ┌──────────────┐
  │  FileAgent  │ ───▶ │ ReportAgent  │
  │ (MCP tools) │      │  (pure LLM)  │
  └─────────────┘      └──────────────┘
   outputKey:           outputKey:
   'fileContent'        'report'

--- AVAILABLE AGENTS -------------------------------------------------
  [FILE]   FileAgent   - Reads files via MCP → stores in 'fileContent'
  [REPORT] ReportAgent - Generates structured report → stores in 'report'
```
  
**ورک فلو خاکہ** ایجنٹس کے درمیان ڈیٹا کے بہاؤ کو ظاہر کرتا ہے۔ ہر ایجنٹ کا مخصوص کردار ہوتا ہے:  
- **FileAgent** MCP ٹولز کے ذریعے فائلیں پڑھتا ہے اور خام مواد `fileContent` میں ذخیرہ کرتا ہے  
- **ReportAgent** وہ مواد استعمال کرتا ہے اور `report` میں ایک منظم رپورٹ تیار کرتا ہے  

```
--- USER REQUEST -----------------------------------------------------
  "Read the file at .../file.txt and generate a report on its contents"
```
  
**یوزر کی درخواست** کام دکھاتی ہے۔ سپروائزر اسے پارس کر کے فیصلہ کرتا ہے کہ FileAgent → ReportAgent کو کال کیا جائے۔

```
--- SUPERVISOR ORCHESTRATION -----------------------------------------
  The Supervisor decides which agents to invoke and passes data between them...

  +-- STEP 1: Supervisor chose -> FileAgent (reading file via MCP)
  |
  |   Input: .../file.txt
  |
  |   Result: LangChain4j is an open-source, provider-agnostic Java framework for building LLM...
  +-- [OK] FileAgent (reading file via MCP) completed

  +-- STEP 2: Supervisor chose -> ReportAgent (generating structured report)
  |
  |   Input: LangChain4j is an open-source, provider-agnostic Java framew...
  |
  |   Result: Executive Summary...
  +-- [OK] ReportAgent (generating structured report) completed
```
  
**سپروائزر کی تنظیم** دو مرحلوں کا بہاؤ دکھاتی ہے:  
1. **FileAgent** MCP کے ذریعے فائل پڑھتا ہے اور مواد ذخیرہ کرتا ہے  
2. **ReportAgent** مواد وصول کرتا ہے اور منظم رپورٹ تیار کرتا ہے  

سپروائزر نے یہ فیصلے **خود مختاری** سے یوزر کی درخواست کی بنیاد پر کیے۔

```
--- FINAL RESPONSE ---------------------------------------------------
Executive Summary
...

Key Points
...

Recommendations
...

--- AGENTIC SCOPE (Data Flow) ----------------------------------------
  Each agent stores its output for downstream agents to consume:
  * fileContent: LangChain4j is an open-source, provider-agnostic Java framework...
  * report: Executive Summary...
```
  
#### ایجنٹک ماڈیول کی خصوصیات کی وضاحت

یہ مثال ایجنٹک ماڈیول کی کئی جدید خصوصیات کو ظاہر کرتی ہے۔ آئیے ایجنٹک اسکوپ اور ایجنٹ لسنرز پر قریب سے نظر ڈالتے ہیں۔

**ایجنٹک اسکوپ** مشترکہ میموری دکھاتا ہے جہاں ایجنٹس نے `@Agent(outputKey="...")` استعمال کرتے ہوئے اپنے نتائج ذخیرہ کیے۔ اس سے یہ ممکن ہوتا ہے:  
- بعد کے ایجنٹس پہلے ایجنٹس کے آؤٹ پٹس تک رسائی حاصل کر سکتے ہیں  
- سپروائزر حتمی جواب مرتب کر سکتا ہے  
- آپ ہر ایجنٹ کے پیدا کردہ مواد کا معائنہ کر سکتے ہیں  

ذیل کا خاکہ دکھاتا ہے کہ ایجنٹک اسکوپ فائل سے رپورٹ ورک فلو میں کس طرح مشترکہ میموری کے طور پر کام کرتا ہے — FileAgent اپنا آؤٹ پٹ `fileContent` کے تحت لکھتا ہے، ReportAgent اسے پڑھتا ہے اور اپنا آؤٹ پٹ `report` کے تحت لکھتا ہے:

<img src="../../../translated_images/ur/agentic-scope.95ef488b6c1d02ef.webp" alt="Agentic Scope Shared Memory" width="800"/>

*ایجنٹک اسکوپ ایک مشترکہ میموری کی طرح کام کرتا ہے — FileAgent `fileContent` لکھتا ہے، ReportAgent اسے پڑھ کر `report` لکھتا ہے، اور آپ کا کوڈ حتمی نتیجہ پڑھتا ہے۔*

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // فائل ایجنٹ سے خام فائل کا ڈیٹا
String report = scope.readState("report");            // رپورٹ ایجنٹ سے ساختہ رپورٹ
```
  
**ایجنٹ لسنرز** ایجنٹ کی عمل کاری کی مانیٹرنگ اور ڈیبگنگ کو فعال کرتے ہیں۔ ڈیمو میں جو قدم بہ قدم آؤٹ پٹ آپ دیکھتے ہیں وہ ایک AgentListener سے آتا ہے جو ہر ایجنٹ کال میں ہک ہوتا ہے:  
- **beforeAgentInvocation** — جب سپروائزر ایک ایجنٹ منتخب کرتا ہے، کال ہوتا ہے تاکہ آپ جان سکیں کون سا ایجنٹ منتخب ہوا اور کیوں  
- **afterAgentInvocation** — جب ایجنٹ مکمل ہوتا ہے، اس کا نتیجہ دکھاتا ہے  
- **inheritedBySubagents** — جب true ہو، تو لسنر تمام ہائیرارکی میں موجود ایجنٹس کو مانیٹر کرتا ہے  

ذیل کا خاکہ Agent Listener کی مکمل لائف سائیکل دکھاتا ہے، بشمول یہ کہ `onError` ایجنٹ کی عمل کاری کے دوران خرابیوں کو کیسے ہینڈل کرتا ہے:

<img src="../../../translated_images/ur/agent-listeners.784bfc403c80ea13.webp" alt="Agent Listeners Lifecycle" width="800"/>

*ایجنٹ لسنرز عمل کاری کی لائف سائیکل میں ہک ہوتے ہیں — ایجنٹس کے شروع، مکمل ہونے، یا غلطیوں کا پتہ چلانے کے لیے مانیٹر کرتے ہیں۔*

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // تمام ذیلی ایجنٹوں تک پھیلائیں
    }
};
```
  
سپروائزر پیٹرن کے علاوہ، `langchain4j-agentic` ماڈیول کئی طاقتور ورک فلو پیٹرنز فراہم کرتا ہے۔ ذیل کا خاکہ ان پانچ میں سے تمام دکھاتا ہے — سادہ سیکونشل پائپ لائنز سے لے کر انسان کی منظوری والے ورک فلو تک:

<img src="../../../translated_images/ur/workflow-patterns.82b2cc5b0c5edb22.webp" alt="Agent Workflow Patterns" width="800"/>

*ایجنٹس کو منظم کرنے کے لیے پانچ ورک فلو پیٹرنز — سادہ ترتیب وار پائپ لائنز سے انسانی منظوری والے ورک فلو تک۔*

| پیٹرن | وضاحت | استعمال کا موقع |
|---------|-------------|----------|
| **Sequential** | ایجنٹس کو ترتیب سے چلائیں، آؤٹ پٹ اگلے کو ملے | پائپ لائنز: تحقیق → تجزیہ → رپورٹ |
| **Parallel** | ایجنٹس کو بیک وقت چلائیں | آزاد کام: موسم + خبریں + اسٹاکس |
| **Loop** | شرط پوری ہونے تک دور چکر لگائیں | معیار کی جانچ: اسکور ≥ 0.8 تک بہتر کریں |
| **Conditional** | شرائط کی بنیاد پر راستہ منتخب کریں | درجہ بندی → ماہر ایجنٹ کو بھیجیں |
| **Human-in-the-Loop** | انسانی چیک پوائنٹس شامل کریں | منظوری کے ورک فلو، مواد کی جانچ |

## کلیدی تصورات

اب جب کہ آپ نے MCP اور ایجنٹک ماڈیول کو عملی طور پر دیکھا ہے، آئیے خلاصہ کرتے ہیں کہ ہر طریقہ کب استعمال کرنا چاہیے۔

MCP کا ایک بڑا فائدہ اس کا بڑھتا ہوا ایکوسسٹم ہے۔ ذیل کا خاکہ دکھاتا ہے کہ ایک یونیورسل پروٹوکول آپ کی AI ایپلیکیشن کو مختلف MCP سرورز سے کیسے جوڑتا ہے — فائل سسٹم اور ڈیٹا بیس تک رسائی سے لے کر GitHub، ای میل، ویب اسکریپنگ، اور مزید:

<img src="../../../translated_images/ur/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="MCP Ecosystem" width="800"/>

*MCP ایک یونیورسل پروٹوکول ایکوسسٹم بناتا ہے — کوئی بھی MCP-مطابق سرور کسی بھی MCP-مطابق کلائنٹ کے ساتھ کام کر سکتا ہے، ایپلیکیشنز کے درمیان ٹول شیئرنگ کو ممکن بناتا ہے۔*

**MCP** اس وقت بہترین ہے جب آپ موجودہ ٹول ایکوسسٹمز سے فائدہ اٹھانا چاہتے ہیں، ایسے ٹولز بنانا چاہتے ہیں جو کئی ایپلیکیشنز شیئر کریں، تیسری پارٹی کی سروسز کو معیاری پروٹوکولز کے ذریعے انٹیگریٹ کرنا چاہتے ہیں، یا بغیر کوڈ بدلے ٹول کی امپلیمینٹیشنز بدلنا چاہتے ہیں۔

**ایجنٹک ماڈیول** اس وقت بہترین کام کرتا ہے جب آپ کو `@Agent` انوٹیشنز کے ساتھ اعلان کردہ ایجنٹ ڈیفینیشنز چاہیے، ورک فلو آرکیسٹریشن (سیکونشل، لوپ، پیریلل) درکار ہو، انٹرفیس بیسڈ ایجنٹ ڈیزائن ترجیح دیں یا متعدد ایجنٹس کو ایک ساتھ `outputKey` کے ذریعے آؤٹ پٹس بانٹنے ہوں۔

**سپروائزر ایجنٹ پیٹرن** اس وقت بہترین ہے جب ورک فلو پہلے سے پیش گوئی نہیں کی جا سکتی اور آپ LLM کو فیصلہ کرنے دینا چاہتے ہیں، جب آپ کے پاس متعدد خاص مہارت والے ایجنٹس ہوں جنہیں متحرک طور پر آرکیسٹریٹ کرنا ہو، جب آپ ایسے بات چیت کے نظام تعمیر کر رہے ہوں جو مختلف صلاحیتوں کی طرف راہنمائی کرتے ہوں، یا جب آپ سب سے زیادہ لچکدار، مطابقت پذیر ایجنٹ رویہ چاہتے ہوں۔

کس طرح ذاتی `@Tool` طریقے (ماڈیول 04 سے) اور MCP ٹولز (اس ماڈیول سے) کے درمیان انتخاب کرنا ہے، اس کے لیے ذیل میں کلیدی موازنہ دیا گیا ہے — کسٹم ٹولز آپ کو سخت کوڈنگ اور مکمل ٹائپ سیفٹی فراہم کرتے ہیں، جبکہ MCP ٹولز معیاری اور قابلِ استعمال انٹیگریشنز دیتے ہیں:

<img src="../../../translated_images/ur/custom-vs-mcp-tools.c4f9b6b1cb65d8a1.webp" alt="Custom Tools vs MCP Tools" width="800"/>

*کسٹم @Tool طریقے کب استعمال کریں اور MCP ٹولز کب — ایپ مخصوص منطق کے لیے کسٹم ٹولز مکمل ٹائپ سیفٹی کے ساتھ، MCP ٹولز معیاری انٹیگریشنز کے لیے جو ایپلیکیشنز میں کام کریں۔*
