## آپ کیا سیکھیں گے

آپ نے مکالماتی AI بنایا ہے، پرامپٹس پر عبور حاصل کیا ہے، جوابات کو دستاویزات کی بنیاد پر بنایا ہے، اور ایجنٹس ٹولز کے ساتھ تخلیق کیے ہیں۔ لیکن وہ تمام ٹولز آپ کی مخصوص ایپلیکیشن کے لیے خود بنائے گئے تھے۔ اگر آپ اپنی AI کو ایک معیاری ٹولز کے ماحولی نظام تک رسائی دے سکیں جو کوئی بھی تخلیق اور شیئر کر سکتا ہو تو کیا ہوگا؟ اس ماڈیول میں، آپ ماڈل کانٹیکسٹ پروٹوکول (MCP) اور LangChain4j کے ایجنٹک ماڈیول کے ساتھ یہ کرنا سیکھیں گے۔ ہم پہلے ایک سادہ MCP فائل ریڈر پیش کریں گے اور پھر دکھائیں گے کہ یہ کس طرح آسانی سے سپروائزر ایجنٹ پیٹرن کے ذریعے جدید ایجنٹک ورک فلو میں شامل ہو جاتا ہے۔

## MCP کیا ہے؟

ماڈل کانٹیکسٹ پروٹوکول (MCP) بالکل یہی فراہم کرتا ہے - AI ایپلیکیشنز کے لیے ایک معیاری طریقہ کہ وہ خارجی ٹولز دریافت اور استعمال کر سکیں۔ ہر ڈیٹا ماخذ یا سروس کے لیے الگ الگ کسٹم انٹیگریشن لکھنے کی بجائے، آپ MCP سرورز سے جڑتے ہیں جو اپنی صلاحیتیں ایک مستقل فارمیٹ میں ظاہر کرتے ہیں۔ آپ کا AI ایجنٹ خودکار طریقے سے یہ ٹولز دریافت اور استعمال کر سکتا ہے۔

<img src="../../../translated_images/ur/mcp-comparison.9129a881ecf10ff5.webp" alt="MCP موازنہ" width="800"/>

*MCP سے پہلے: پیچیدہ نقطہ بہ نقطہ انٹیگریشنز۔ MCP کے بعد: ایک پروٹوکول، لا محدود ممکنات۔*

MCP AI ترقی میں ایک بنیادی مسئلہ حل کرتا ہے: ہر انٹیگریشن کسٹم ہے۔ GitHub تک رسائی چاہیے؟ کسٹم کوڈ۔ فائلیں پڑھنی ہیں؟ کسٹم کوڈ۔ ڈیٹا بیس سے سوال کرنا ہے؟ کسٹم کوڈ۔ اور یہ انٹیگریشنز دوسرے AI ایپلیکیشنز کے ساتھ کام نہیں کرتے۔

MCP اسے معیاری بناتا ہے۔ ایک MCP سرور ٹولز کو واضح تفصیلات اور اسکیما کے ساتھ ظاہر کرتا ہے۔ کوئی بھی MCP کلائنٹ جڑ سکتا ہے، دستیاب ٹولز تلاش کر سکتا ہے اور انہیں استعمال کر سکتا ہے۔ ایک بار بنائیں، ہر جگہ استعمال کریں۔

<img src="../../../translated_images/ur/mcp-architecture.b3156d787a4ceac9.webp" alt="MCP آرکیٹیکچر" width="800"/>

*ماڈل کانٹیکسٹ پروٹوکول کی آرکیٹیکچر - معیاری ٹول دریافت اور عمل درآمد*

## MCP کیسے کام کرتا ہے

<img src="../../../translated_images/ur/mcp-protocol-detail.01204e056f45308b.webp" alt="MCP پروٹوکول تفصیل" width="800"/>

*MCP اندرونی طور پر کیسے کام کرتا ہے — کلائنٹس ٹولز دریافت کرتے ہیں، JSON-RPC پیغامات کا تبادلہ کرتے ہیں، اور ایک ٹرانسپورٹ پرت کے ذریعے آپریشنز انجام دیتے ہیں۔*

**سرور-کلائنٹ آرکیٹیکچر**

MCP کلائنٹ-سرور ماڈل استعمال کرتا ہے۔ سرورز ٹولز فراہم کرتے ہیں - فائلیں پڑھنا، ڈیٹا بیس سے سوال کرنا، APIs کال کرنا۔ کلائنٹس (آپ کا AI ایپلیکیشن) سرورز سے جڑتے ہیں اور ان کے ٹولز استعمال کرتے ہیں۔

LangChain4j کے ساتھ MCP استعمال کرنے کے لیے، اس Maven ڈیپینڈینسی کو شامل کریں:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```

**ٹول دریافت**

جب آپ کا کلائنٹ MCP سرور سے جڑتا ہے، تو وہ پوچھتا ہے "آپ کے پاس کون سے ٹولز ہیں؟" سرور دستیاب ٹولز کی فہرست کے ساتھ جواب دیتا ہے، ہر ایک کی وضاحت اور پیرامیٹر اسکیما کے ساتھ۔ آپ کا AI ایجنٹ پھر صارف کی درخواست کی بنیاد پر فیصلہ کر سکتا ہے کہ کون سے ٹولز استعمال کرنے ہیں۔

<img src="../../../translated_images/ur/tool-discovery.07760a8a301a7832.webp" alt="MCP ٹول دریافت" width="800"/>

*AI ابتدا میں دستیاب ٹولز دریافت کرتا ہے — اب اسے معلوم ہے کہ کون سی صلاحیتیں دستیاب ہیں اور وہ فیصلہ کر سکتا ہے کہ کون سے استعمال کرے۔*

**ٹرانسپورٹ میکانزم**

MCP مختلف ٹرانسپورٹ میکانزمز کو سپورٹ کرتا ہے۔ یہ ماڈیول Stdio ٹرانسپورٹ کو مقامی عملوں کے لیے ظاہر کرتا ہے:

<img src="../../../translated_images/ur/transport-mechanisms.2791ba7ee93cf020.webp" alt="ٹرانسپورٹ میکانزمز" width="800"/>

*MCP ٹرانسپورٹ میکانزمز: ریموٹ سرورز کے لیے HTTP، مقامی عملوں کے لیے Stdio*

**Stdio** - [StdioTransportDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java)

مقامی عملوں کے لیے۔ آپ کی ایپلیکیشن ایک سرور کو ذیلی عمل کے طور پر جنم دیتی ہے اور معیاری ان پٹ/آؤٹ پٹ کے ذریعے بات چیت کرتی ہے۔ فائل سسٹم کی رسائی یا کمانڈ لائن ٹولز کے لیے مفید۔

```java
McpTransport stdioTransport = new StdioMcpTransport.Builder()
    .command(List.of(
        npmCmd, "exec",
        "@modelcontextprotocol/server-filesystem@2025.12.18",
        resourcesDir
    ))
    .logEvents(false)
    .build();
```

## ایجنٹک ماڈیول

جبکہ MCP معیاری ٹولز فراہم کرتا ہے، LangChain4j کا **ایجنٹک ماڈیول** ایسے ایجنٹس بنانے کا ایک اعلانیہ طریقہ فراہم کرتا ہے جو ان ٹولز کو مربوط کرتے ہیں۔ `@Agent` اینوٹیشن اور `AgenticServices` آپ کو ایجنٹ کے رویے کو واجبی کوڈ کے بجائے انٹرفیسز کے ذریعہ تعریف کرنے دیتے ہیں۔

اس ماڈیول میں، آپ **سپروائزر ایجنٹ** پیٹرن کو دریافت کریں گے — ایک جدید ایجنٹک AI طریقہ جہاں "سپروائزر" ایجنٹ صارف کی درخواست کی بنیاد پر متحرک طور پر فیصلہ کرتا ہے کہ کون سے سب ایجنٹس کو کال کرنا ہے۔ ہم دونوں تصورات کو ملائیں گے اور اپنے سب ایجنٹس میں سے ایک کو MCP سے چلنے والی فائل تک رسائی کی صلاحیت دیں گے۔

ایجنٹک ماڈیول استعمال کرنے کے لیے، اس Maven ڈیپینڈینسی کو شامل کریں:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```

> **⚠️ تجرباتی:** `langchain4j-agentic` ماڈیول **تجرباتی** ہے اور تبدیل ہو سکتا ہے۔ AI اسسٹنٹس بنانے کا مستحکم طریقہ `langchain4j-core` ہے جس میں کسٹم ٹولز ہوتے ہیں (ماڈیول 04)۔

### سپروائزر ایجنٹ

**سپروائزر ایجنٹ پیٹرن** ایجنٹک AI کی ایک **لچکدار** صورت ہے۔ ایک سپروائزر LLM استعمال کرتا ہے تاکہ صارف کی درخواست کی بنیاد پر خود مختار طور پر فیصلہ کرے کہ کون سے ایجنٹس کو کال کرنا ہے۔ اگلی مثال میں، ہم MCP سے چلنے والی فائل رسائی کو LLM ایجنٹ کے ساتھ جوڑتے ہیں تاکہ ایک زیر نگرانی فائل پڑھنے → رپورٹ ورک فلو بنایا جا سکے۔

ڈیمو میں، `FileAgent` MCP فائل سسٹم ٹولز استعمال کرتے ہوئے فائل پڑھتا ہے، اور `ReportAgent` ایک ساختہ رپورٹ تیار کرتا ہے جس میں ایک ایگزیکٹو سامری (1 جملہ)، 3 کلیدی نکات، اور تجاویز شامل ہیں۔ سپروائزر خودکار طور پر اس فلو کو منظم کرتا ہے:

<img src="../../../translated_images/ur/supervisor-agent-pattern.06275a41ae006ac8.webp" alt="سپروائزر ایجنٹ پیٹرن" width="800"/>

*سپروائزر اپنے LLM کو استعمال کرتا ہے تاکہ فیصلہ کرے کہ کون سے ایجنٹس کب اور کس ترتیب میں کال کرنا ہے — کوئی ہارڈ کوڈڈ راستہ نہیں چاہیے۔*

ہمارے فائل سے رپورٹ پائپ لائن کے لیے ٹھوس ورک فلو کچھ یوں ہے:

<img src="../../../translated_images/ur/file-report-workflow.649bb7a896800de9.webp" alt="فائل سے رپورٹ ورک فلو" width="800"/>

*FileAgent MCP ٹولز کے ذریعے فائل پڑھتا ہے، پھر ReportAgent خام مواد کو ساختہ رپورٹ میں تبدیل کرتا ہے۔*

ہر ایجنٹ اپنی آؤٹ پٹ **ایجنٹک سکوپ** (مشترکہ میموری) میں ذخیرہ کرتا ہے، جس سے نیچے کے ایجنٹس کو پچھلے نتائج تک رسائی حاصل ہوتی ہے۔ یہ ظاہر کرتا ہے کہ MCP ٹولز ایجنٹک ورک فلو میں بغیر کسی وقفے کے شامل ہو جاتے ہیں — سپروائزر کو جاننے کی ضرورت نہیں کہ فائلیں کیسے پڑھی جاتی ہیں، بس یہ کہ `FileAgent` یہ کر سکتا ہے۔

#### ڈیمو چلانا

اسٹارٹ اسکرپٹس خودکار طور پر روٹ `.env` فائل سے ماحول کی متغیرات لوڈ کرتے ہیں:

**بش:**

**پاور شیل:**

**VS Code استعمال کرتے ہوئے:** `SupervisorAgentDemo.java` پر رائٹ کلک کریں اور **"Run Java"** منتخب کریں (یقینی بنائیں کہ `.env` فائل ترتیب دی گئی ہے)۔

#### سپروائزر کیسے کام کرتا ہے

```java
// مرحلہ 1: فائل ایجنٹ MCP کے ٹولز استعمال کرتے ہوئے فائلیں پڑھتا ہے
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // فائل آپریشنز کے لیے MCP کے ٹولز موجود ہیں
        .build();

// مرحلہ 2: رپورٹ ایجنٹ منظم رپورٹس تیار کرتا ہے
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// سپروائزر فائل → رپورٹ کے ورک فلو کو منظم کرتا ہے
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // حتمی رپورٹ واپس کریں
        .build();

// سپروائزر درخواست کی بنیاد پر ایجنٹس کو متعین کرتا ہے
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```

#### ردعمل کی حکمت عملیاں

جب آپ `SupervisorAgent` کو ترتیب دیتے ہیں، تو آپ بتاتے ہیں کہ اسے سب ایجنٹس کے کام مکمل ہونے کے بعد صارف کو اپنا حتمی جواب کیسے دینا چاہیے۔

<img src="../../../translated_images/ur/response-strategies.3d0cea19d096bdf9.webp" alt="ردعمل کی حکمت عملیاں" width="800"/>

*وہ تین حکمت عملیاں جن سے سپروائزر اپنا حتمی جواب تیار کرتا ہے — انتخاب آپ کی ضرورت کے مطابق کریں کہ آپ آخری ایجنٹ کا آؤٹ پٹ چاہتے ہیں، ایک جامع سمری یا سب سے بہتر اسکور والا آپشن۔*

دستیاب حکمت عملیاں یہ ہیں:

| حکمت عملی | وضاحت |
|----------|--------|
| **LAST** | سپروائزر آخری سب ایجنٹ یا کال کیے گئے ٹول کا آؤٹ پٹ واپس کرتا ہے۔ یہ اس وقت مفید ہے جب ورک فلو کا آخری ایجنٹ خاص طور پر مکمل، حتمی جواب تیار کرنے کے لیے بنایا گیا ہو (مثلاً تحقیقاتی پائپ لائن میں "سمری ایجنٹ")۔ |
| **SUMMARY** | سپروائزر اپنا داخلی لینگوئج ماڈل (LLM) استعمال کر کے ساری بات چیت اور سب ایجنٹس کے آؤٹ پٹس کی ایک سمری تیار کرتا ہے، پھر اسے حتمی جواب کے طور پر دیتا ہے۔ یہ صارف کو صاف، جامع جواب فراہم کرتا ہے۔ |
| **SCORED** | نظام داخلی LLM کا استعمال کرتے ہوئے آخری جواب اور سمری دونوں کو اصل صارف کی درخواست کے مقابلے میں اسکور کرتا ہے، اور جو زیادہ اسکور حاصل کرے اسے واپس کرتا ہے۔ |

مکمل عمل درآمد کے لیے [SupervisorAgentDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/SupervisorAgentDemo.java) دیکھیں۔

> - "سپروائزر کس طرح فیصلہ کرتا ہے کہ کون سے ایجنٹس کو کال کرنا ہے؟"
> - "سپروائزر اور سیquential ورک فلو پیٹرن میں کیا فرق ہے؟"
> - "میں سپروائزر کی منصوبہ بندی کے رویے کو کیسے حسب ضرورت بنا سکتا ہوں؟"

#### آؤٹ پٹ کی سمجھ

جب آپ ڈیمو چلائیں گے، آپ دیکھیں گے کہ سپروائزر کس طرح متعدد ایجنٹس کو منظم کرتا ہے۔ ہر سیکشن کا مطلب یہ ہے:

```
======================================================================
  FILE → REPORT WORKFLOW DEMO
======================================================================

This demo shows a clear 2-step workflow: read a file, then generate a report.
The Supervisor orchestrates the agents automatically based on the request.
```

**ہیڈر** ورک فلو کے تصور کا تعارف کرواتا ہے: فائل پڑھنے سے لے کر رپورٹ بنانے تک ایک مرکوز پائپ لائن۔

```
--- WORKFLOW ---------------------------------------------------------
  ┌─────────────┐      ┌──────────────┐
  │  FileAgent  │ ───▶ │ ReportAgent  │
  │ (MCP tools) │      │  (pure LLM)  │
  └─────────────┘      └──────────────┘
   outputKey:           outputKey:
   'fileContent'        'report'

--- AVAILABLE AGENTS -------------------------------------------------
  [FILE]   FileAgent   - Reads files via MCP → stores in 'fileContent'
  [REPORT] ReportAgent - Generates structured report → stores in 'report'
```

**ورک فلو خاکہ** ایجنٹس کے درمیان ڈیٹا فلو دکھاتا ہے۔ ہر ایجنٹ کا ایک مخصوص کردار ہے:
- **FileAgent** MCP ٹولز کی مدد سے فائلیں پڑھتا ہے اور خام مواد `fileContent` میں ذخیرہ کرتا ہے
- **ReportAgent** اس مواد کو استعمال کرتے ہوئے ساختہ رپورٹ `report` تیار کرتا ہے

```
--- USER REQUEST -----------------------------------------------------
  "Read the file at .../file.txt and generate a report on its contents"
```

**صارف کی درخواست** کام دکھاتا ہے۔ سپروائزر اسے پڑھ کر فیصلہ کرتا ہے کہ FileAgent → ReportAgent کال کرنا ہے۔

```
--- SUPERVISOR ORCHESTRATION -----------------------------------------
  The Supervisor decides which agents to invoke and passes data between them...

  +-- STEP 1: Supervisor chose -> FileAgent (reading file via MCP)
  |
  |   Input: .../file.txt
  |
  |   Result: LangChain4j is an open-source, provider-agnostic Java framework for building LLM...
  +-- [OK] FileAgent (reading file via MCP) completed

  +-- STEP 2: Supervisor chose -> ReportAgent (generating structured report)
  |
  |   Input: LangChain4j is an open-source, provider-agnostic Java framew...
  |
  |   Result: Executive Summary...
  +-- [OK] ReportAgent (generating structured report) completed
```

**سپروائزر کی منظم کاری** 2-مرحلہ عمل کو دکھاتی ہے:
1. **FileAgent** MCP کے ذریعے فائل پڑھتا ہے اور مواد ذخیرہ کرتا ہے
2. **ReportAgent** مواد حاصل کرتا ہے اور ساختہ رپورٹ بناتا ہے

یہ فیصلے سپروائزر نے صارف کی درخواست کی بنیاد پر **خود مختار** طور پر کیے ہیں۔

```
--- FINAL RESPONSE ---------------------------------------------------
Executive Summary
...

Key Points
...

Recommendations
...

--- AGENTIC SCOPE (Data Flow) ----------------------------------------
  Each agent stores its output for downstream agents to consume:
  * fileContent: LangChain4j is an open-source, provider-agnostic Java framework...
  * report: Executive Summary...
```

#### ایجنٹک ماڈیول کی خصوصیات کی وضاحت

یہ مثال ایجنٹک ماڈیول کی کئی جدید خصوصیات کو ظاہر کرتی ہے۔ آئیے ایجنٹک سکوپ اور ایجنٹ لسٹنرز پر غور کریں۔

**ایجنٹک سکوپ** وہ مشترکہ میموری دکھاتا ہے جہاں ایجنٹس نے `@Agent(outputKey="...")` استعمال کر کے اپنے نتائج ذخیرہ کیے۔ اس سے سہولت ہوتی ہے:
- بعد کے ایجنٹس کو پہلے کے ایجنٹس کے آؤٹ پٹس تک رسائی حاصل کرنا
- سپروائزر کو حتمی جواب تیار کرنا
- آپ کو یہ جانچنے کی سہولت کہ ہر ایجنٹ نے کیا تیار کیا

<img src="../../../translated_images/ur/agentic-scope.95ef488b6c1d02ef.webp" alt="ایجنٹک سکوپ مشترکہ میموری" width="800"/>

*ایجنٹک سکوپ مشترکہ میموری کے طور پر کام کرتا ہے — FileAgent `fileContent` لکھتا ہے، ReportAgent اسے پڑھ کر `report` لکھتا ہے، اور آپ کا کوڈ حتمی نتیجہ پڑھتا ہے۔*

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // فائل ایجنٹ سے خام فائل ڈیٹا
String report = scope.readState("report");            // رپورٹ ایجنٹ سے ساختہ رپورٹ
```

**ایجنٹ لسٹنرز** ایجنٹس کی عملدرآمد کی نگرانی اور ڈیبگنگ کو قابل بناتے ہیں۔ ڈیمو میں جو قدم بہ قدم آؤٹ پٹ آپ دیکھتے ہیں، وہ ایک ایجنٹ لسٹنر سے آتا ہے جو ہر ایجنٹ کال میں شامل ہوتا ہے:
- **beforeAgentInvocation** - جب سپروائزر ایک ایجنٹ منتخب کرتا ہے تو کال ہوتا ہے، جس سے آپ کو پتہ چلتا ہے کہ کون سا ایجنٹ منتخب کیا گیا اور کیوں  
- **afterAgentInvocation** - جب ایک ایجنٹ مکمل کرتا ہے تو کال ہوتا ہے، اس کے نتیجہ کو دکھاتا ہے  
- **inheritedBySubagents** - جب صحیح ہو، تو سننے والا تمام ایجنٹس کو ہائیرارکی میں مانیٹر کرتا ہے  

<img src="../../../translated_images/ur/agent-listeners.784bfc403c80ea13.webp" alt="ایجنٹ سننے والوں کا لائف سائیکل" width="800"/>

*ایجنٹ سننے والے عملدرآمد کے لائف سائیکل میں شامل ہوتے ہیں — جب ایجنٹس شروع ہوتے ہیں، مکمل کرتے ہیں، یا خرابی کا سامنا کرتے ہیں تو مانیٹر کرتے ہیں۔*

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // تمام ذیلی ایجنٹس تک پھیلائیں
    }
};
```
  
سپروائزر پیٹرن سے آگے، `langchain4j-agentic` ماڈیول کئی طاقتور ورک فلو پیٹرنز اور خصوصیات فراہم کرتا ہے:

<img src="../../../translated_images/ur/workflow-patterns.82b2cc5b0c5edb22.webp" alt="ایجنٹ ورک فلو پیٹرنز" width="800"/>

*پانچ ورک فلو پیٹرنز ایجنٹوں کی ترتیب کے لیے — سادہ ترتیب وار پائپ لائنز سے لے کر ہیومن ان-دی-لوپ منظوری ورک فلو تک۔*

| پیٹرن | وضاحت | استعمال کا کیس |
|---------|-------------|----------|
| **متسلسل** | ایجنٹس کو ترتیب سے چلائیں، آؤٹ پٹ اگلے کو گزاریں | پائپ لائنز: تحقیق → تجزیہ → رپورٹ |
| **متوازی** | ایجنٹس کو بیک وقت چلائیں | آزاد کام: موسم + خبریں + اسٹاکس |
| **لوپ** | شرط پوری ہونے تک دہرائیں | کوالٹی اسکورنگ: درجہ بندی 0.8 یا اس سے زیادہ تک بہتر بنائیں |
| **مشروط** | شرائط کی بنیاد پر راہنمائی کریں | درجہ بندی → ماہر ایجنٹ کو راؤٹ کریں |
| **ہیومن ان-دی-لوپ** | انسانی نقطہ چیک شامل کریں | منظوری کے ورک فلو، مواد کی جانچ پڑتال |

## اہم تصورات

اب جب کہ آپ نے MCP اور ایجنٹک ماڈیول کو عمل میں دیکھا ہے، آئیے خلاصہ کریں کہ کب کون سا طریقہ استعمال کرنا ہے۔

<img src="../../../translated_images/ur/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="MCP ماحولیاتی نظام" width="800"/>

*MCP ایک یونیورسل پروٹوکول ماحولیاتی نظام بناتا ہے — کوئی بھی MCP-مطابق سرور کوئی بھی MCP-مطابق کلائنٹ کے ساتھ کام کرتا ہے، ایپلیکیشنز کے درمیان ٹول شیئرنگ کو قابل بناتا ہے۔*

**MCP** مثالی ہے جب آپ موجودہ ٹول ماحولیاتی نظام سے فائدہ اٹھانا چاہتے ہیں، ایسے ٹولز بنائیں جو متعدد ایپلیکیشنز شیئر کر سکیں، تیسرے فریق کی خدمات کو معیاری پروٹوکولز کے ساتھ ضم کریں، یا بغیر کوڈ تبدیل کیے ٹولز کی عملی شکل بدلیں۔

**ایجنٹک ماڈیول** بہترین کام کرتا ہے جب آپ کو `@Agent` تشریحات کے ساتھ اعلان شدہ ایجنٹ تعریفیں چاہئیں، ورک فلو کی ترتیب (متسلسل، لوپ، متوازی) کی ضرورت ہو، انٹرفیس پر مبنی ایجنٹ ڈیزائن کو احکامات پر مبنی کوڈ پر ترجیح دیں، یا متعدد ایجنٹوں کو مل کر کام کرتے ہوئے `outputKey` کے ذریعے نتائج کو مشترکہ کرنا ہو۔

**سپروائزر ایجنٹ پیٹرن** اس وقت بہترین ہے جب ورک فلو پیشگی متعین نہ ہو اور آپ چاہتے ہوں کہ LLM فیصلہ کرے، جب آپ کے پاس متعدد مخصوص ایجنٹ ہوں جنہیں متحرک ترتیب دینا ہو، جب آپ بات چیت کے نظام بنائیں جو مختلف صلاحیتوں کی طرف راہنمائی کرتے ہیں، یا جب آپ سب سے زیادہ لچکدار، موافق ایجنٹ رویہ چاہتے ہوں۔

<img src="../../../translated_images/ur/custom-vs-mcp-tools.c4f9b6b1cb65d8a1.webp" alt="حسب ضرورت ٹولز بمقابلہ MCP ٹولز" width="800"/>

*جب حسب ضرورت @Tool طریقے استعمال کریں بمقابلہ MCP ٹولز — حسب ضرورت ٹولز ایپ مخصوص منطق کے لیے مکمل ٹائپ سیفٹی کے ساتھ؛ MCP ٹولز معیاری یکساں انضمام کے لیے جو ایپلیکیشنز میں کام کرتے ہیں۔*
