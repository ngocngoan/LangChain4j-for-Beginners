## သင်ယူမည့်အချက်များ

ယခင် module များတွင် သင်သည် AI နှင့် ဖလှယ်ဆက်သွယ်ပုံနှင့် သင်၏ prompt များကို ထိထိရောက်ရောက် ဖန်တီးနိုင်ပုံကို ရယူခဲ့သည်။ ဤမှာ အခြေခံ ကန့်သတ်ချက်တစ်ခုရှိသည်- ဘာသာစကား မော်ဒယ်များသည် သင်တန်းကာလအတွင်းသာ သင်ယူထားသမျှကိုသာ သိရှိသည်။ ကုမ္ပဏီ၏ မူဝါဒများ၊ စီမံကိန်းစာရွက်စာတမ်းများ သို့မဟုတ် မသင်ကြားခဲ့သော အချက်အလက်များအကြောင်း မေးလျှင် မဖြေနိုင်ပါ။

RAG (Retrieval-Augmented Generation) သည် ဤပြဿနာကို ဖြေရှင်းပေးသည်။ မော်ဒယ်ကို သင်၏သတင်းအချက်အလက်များ သင်ကြားပေးရန် မကြိုးစားဘဲ (ဤသည်မှာ ထိတွေ့ကျလှ နှင့် ရှုပ်ထွေးသည်)၊ သင်၏စာရွက်စာတမ်းများမှ ရှာဖွေရန် စွမ်းရည်ကိုပေးသည်။ မည်သူမဆို မေးခွန်းမေးလျှင် စနစ်သည် သက်ဆိုင်ရာ သတင်းအချက်အလက်များကို ရှာဖွေနိုင်ၿပီး prompt ထဲတွင် ထည့်သွင်းပေးသည်။ ဒီနောက် မော်ဒယ်က ထို ရရှိထားသော အကြောင်းအရာအခြေခံ၍ ဖြေကြားသည်။

RAG ကို မော်ဒယ်အတွက် ကိုးကားစာကြည့်တိုက် တစ်ခုပေးခြင်းဟု ယူဆနိုင်သည်။ မေးခွန်းမေးလျှင် စနစ်မှာ-

1. **အသုံးပြုသူ မေးခွန်း** - သင်မေးမည်
2. **Embedding** - မေးခွန်းကို ဗက်စတာသို့ ပြောင်းသည်
3. **Vector Search** - ဆင်တူသော စာရွက်စာတမ်းအစိတ်အပိုင်းများ ရှာဖွေသည်
4. **Context Assembly** - သက်ဆိုင်ရာ အစိတ်အပိုင်းများကို prompt ထဲ ထည့်သည်
5. **ဖြေဆိုခြင်း** - LLM သည် ရရှိထားသော အကြောင်းအရာအရ ဖြေကြားသည်

ဤနည်းဖြင့် မော်ဒယ်၏ ဖြေဆိုမှုများသည် သင်၏ အချက်အလက်များပေါ် အခြေခံသည့်အတွက် သင်တန်းမှုအသိပညာ နှင့် သြဇာ၍ ဖြေဆိုခြင်း မဟုတ်ပါ။

## RAG ကို နားလည်ခြင်း

အောက်တွင် ရှုမြင်ပုံမှာ အဓိက အကြောင်းအရာကို ဖော်ပြသည်- မော်ဒယ်၏ သင်တန်းမှု ဒေတာတွင် တစ်ဖက်တည်း မေ့လျော့ မော်ဒယ်ရှေ့တွင် ထုတ်ပေးရန် မိမိစာရွက်စာတမ်း ကိုးကားစာကြည့်တိုက် တစ်ခု ပေးသည်။

### စာရွက်စာတမ်း ပြုလုပ်ခြင်း

[DocumentService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/DocumentService.java)

သင်စာရွက်စာတမ်းတင်သွင်းလျှင် စနစ်သည် အဲဒီစာရွက်စာတမ်းကို ပြန်သုံးသပ်၍ (PDF သို့ Plain Text) ဖိုင်နာမည်အစရှိသော မီတာဒေတာများ ပူးတွဲ ထားသည်။ ထို့နောက် စာရွက်စာတမ်းကို မော်ဒယ်၏ context ပြကွက်အတွင်း ထည့်နိုင်မည့် အရွယ်အစား သေးငယ်သော အပိုင်းများသို့ ခွဲထုတ်သည်။ ၎င်း အပိုင်းများသည် မျဉ်းတစ်ကြောင်း ထိပ်ဆုံးတွေ့ရသော ဒေတာများကို မပျောက်သွားစေရန် ကွက်လပ်အနည်းငယ်ကိုမျှဝေထားသည်။

```java
// အပ်လုဒ်လုပ်ထားသောဖိုင်ကို ဖတ်ရှု၍ LangChain4j Document အဖြစ် ထုပ်ပိုးပါ
Document document = Document.from(content, metadata);

// ၃၀-token အလွှာဖြင့် ၃၀၀-token အပိုင်းများသို့ ခွဲထုတ်ပါ
DocumentSplitter splitter = DocumentSplitters
    .recursive(300, 30);

List<TextSegment> segments = splitter.split(document);
```

အောက်တွင် ဤလုပ်ဆောင်မှုကို ရုပ်ပိုင်းဆိုင်ရာ ဖော်ပြချက်ဖြင့် ပြထားသည်။ အပိုင်းတိုင်းသည် ၎င်း၏ အနီးကပ် အပိုင်းများနှင့် အပိုင်းလေးတစ်ခုတည်းနဲ့ Token ရှိခြင်းအား သတိပြုပါ- 30 Token ကိုမျှဝေထားခြင်းဖြင့် အရေးပါတဲ့ context မပျောက်တော့ပါ-

### Embeddings ဖန်တီးခြင်း

[LangChainRagConfig.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/config/LangChainRagConfig.java)

အပိုင်းတိုင်းကို embedding ဟုခေါ်သော ဂဏန်းအရေအတွက် ဖော်ပြချက်တစ်ခုသို့ ပြောင်းလဲသည် - အဓိပ္ပာယ်ကို ဖမ်းဆီးထားသော သင်္ချာ သွင်ပြင်ချက်တစ်ခု အပါအဝင်ဖြစ်သည်။ ဆင်တူသော စာသားများသည် ဆင်တူသော embedding များ ထုတ်ပေးသည်။

```java
@Bean
public EmbeddingModel embeddingModel() {
    return OpenAiOfficialEmbeddingModel.builder()
        .baseUrl(azureOpenAiEndpoint)
        .apiKey(azureOpenAiKey)
        .modelName(azureEmbeddingDeploymentName)
        .build();
}

EmbeddingStore<TextSegment> embeddingStore = 
    new InMemoryEmbeddingStore<>();
```

အောက်တွင် LangChain4j အစိတ်အပိုင်းများကို ဆက်စပ် ပုံမှန် ဖော်ပြထားသည်။ `OpenAiOfficialEmbeddingModel` သည် စာသားကို ဗက်စတာများသို့ ပြောင်းလဲပြီး၊ `InMemoryEmbeddingStore` သည် ဗက်စတာများနှင့် မူရင်း `TextSegment` ထုတ်ပြန်ချက်များကို ထိန်းသိမ်းသည်။ `EmbeddingSearchRequest` သည် ရှာဖွေရေး ပြင်ဆင်ချက်များ `maxResults` နှင့် `minScore` စသဖြင့် ထိန်းချုပ်သည်-

<img src="../../../translated_images/my/rag-langchain4j-classes.bbf3aa9077ab443d.webp" alt="LangChain4j RAG Classes" width="800"/>

embedding များသိမ်းဆည်းပြီးနောက် ဆင်တူ အကြောင်းအရာများသည် ဗက်စတာစွန်းထဲတွင် သီးခြားစုစည်းမှု ဖြစ်သည်။ အောက်တွင် မြင်မြင်သာသာ ဖော်ပြချက်သည် ဆင်တူ အကြောင်းအရာများ အနီးကပ်အမှတ်များအဖြစ် အစိမ်းတောင်တက်ကြောင်းတွက်ချက်ထားသည်။ ဒါကြောင့် semantic ရှာဖွေရေး ဖြစ်နိုင်သည်-

### Semantic ရှာဖွေရေး

[RagService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/RagService.java)

မေးခွန်းမေးသောအခါ မေးခွန်းကို embedding ဖန်တီးသည်။ စနစ်သည် မေးခွန်း embedding ကို စာရွက်စာတမ်း အပိုင်း embedding နှင့် နှိုင်းယှဉ်ပါသည်။ များစွာ ဆင်တူ ပြည့်စုံသော အပိုင်းများရှာတွေ့သည် - ရှာဖွေရေးစကားလုံးနှင့် မတူ၊ ဒါပေမဲ့ အဓိပ္ပာယ် ဆက်စပ်မှုဖြစ်သည်။

```java
Embedding queryEmbedding = embeddingModel.embed(question).content();

EmbeddingSearchRequest searchRequest = EmbeddingSearchRequest.builder()
    .queryEmbedding(queryEmbedding)
    .maxResults(5)
    .minScore(0.5)
    .build();

EmbeddingSearchResult<TextSegment> searchResult = embeddingStore.search(searchRequest);
List<EmbeddingMatch<TextSegment>> matches = searchResult.matches();

for (EmbeddingMatch<TextSegment> match : matches) {
    String relevantText = match.embedded().text();
    double score = match.score();
}
```

အောက်တွင် semantic ရှာဖွေမှု နှင့် ပုံမှန် keyword ရှာဖွေမှုကို နှိုင်းယှဉ်ထားသည်။ "vehicle" ဟု keyword ရှာဖွေရေးသည် "cars and trucks" ပါသော အပိုင်းကို မတွေ့နိုင်ပေမယ့် semantic ရှာဖွေမှုသည် ၎င်းတို့ အဓိပ္ပာယ်တူကြောင်း နားလည်၍ အဆင့်မြင့်နေရာတွင် ပြန်ပေးသည်-

### စာရွက်စာတမ်း တင်ခြင်း

စာရွက်စာတမ်းတင်ခြင်းဖြင့် စတင်ပါ - မစမ်းသပ်ခြင်းတွင် TXT ဖိုင်များ သင့်တော်သည်။ ဒီ directory တွင် စမ်းသပ်ရေးအတွက် LangChain4j ၏ အင်္ဂါရပ်များ၊ RAG ဆောင်ရွက်မှု၊ နှင့် ကောင်းမွန်သောလေ့လာမှုများ ပါဝင်သော `sample-document.txt` ဖိုင် ရှိသည်။

စနစ်သည် စာရွက်စာတမ်းကို ကြည့်ဖြတ်၍ chunk များ ခွဲထုတ်ပြီး chunk တစ်ခုချင်းစီအတွက် embedding များ ဖန်တီးပေးသည်။ ဤလုပ်ဆောင်ချက်သည် တင်သွင်းချိန်အတွင်း အလိုအလျောက် ဖြစ်သည်။

ယခု စာရွက်စာတမ်းအကြောင်းအရာအပေါ် သတ်မှတ်ချက်တိကျသော မေးခွန်းများ မေးပါ။ စနစ်သည် သက်ဆိုင်ရာ chunk များကို ရှာဖွေ ရှာမတွေ့ခဲ့ပါက prompt ထဲတွင် ထည့်သည်။ ထို့နောက် မော်ဒယ်က အဖြေ ထုတ်ပေးပါသည်။

## အဓိကအတွေးအခေါ်များ

### Chunking များလုပ်နည်း

စာရွက်စာတမ်းများကို ၃၀၀ token အရွယ်အစား chunks အဖြစ် ခွဲသည်၊ ၃၀ token ဖြင့် အချင်းချင်းအပေါ် overlapping ဖြစ်သည်။ ဤ စိတ်ချမ်းသာမှုသည် chunk တစ်ခုလောက်မှာ အကြောင်းအရာအသုံးပြုရေးထားဖို့ လုံလောက်စေပြီး prompt တစ်ခုထဲတွင် ဆက်လက်ထည့်သွင်းနိုင်ပါသည်။

### ဆင်တူမှုအဆင့်များ

ရရှိသော chunk တစ်ခုချင်းစီတွင် ၀ မှ ၁ အထိ ဆင်တူမှုအဆင့် တစ်ခုရှိသည်၊ ၎င်းသည် အသုံးပြုသူ၏ မေးခွန်းနှင့် မည်မျှ ပေါင်းသင်းမှု ရှိကြောင်းပြသည်။ အောက်ဖော်ပြထားသော ပုံတွင် အဆင့်များ၏ အကွာအဝေးနှင့် စနစ်၏ ရလဒ် ထုတ်ပေးမှုကို မည်သို့ ထိန်းညှိသလဲ ဖော်ပြပါသည်-

<img src="../../../translated_images/my/similarity-scores.b0716aa911abf7f0.webp" alt="Similarity Scores" width="800"/>

အဆင့်များသည် ၀ မှ ၁ အတွင်းဖြစ်သည်-
- ၀.၇-၁.၀: အလွန်ဆက်စပ်သောအချက်၊ တိကျသော မိုက်ခရိုအရေအတွက်
- ၀.၅-၀.၇: ဆက်စပ်နေသော အကြောင်းအရာကောင်း
- ၀.၅ အောက်: အနှစ်သာရမပါသော အပိုင်းများ ဖြုတ်ပယ်ခဲ့သည်

စနစ်သည် အနိမ့်ဆုံး အဆင့်ထက် မနိမ့်သော chunks များကိုသာ ရွေးချယ် ပေးသည်။

### In-Memory စာအုပ်တိုက်

ဤ module သည် ရိုးရှင်းမှုအတွက် In-memory storage ကို အသုံးပြုသည်။ application ကို ပြန်စတင်လျှင် uploaded စာရွက်စာတမ်းများ ပျောက်ဆုံးသွားမည်။ ထုတ်လုပ်မှု စနစ်များတွင် Qdrant သို့မဟုတ် Azure AI Search ကဲ့သို့ Persistent vector databases များကို အသုံးပြုသည်။

### အကြောင်းအရာ ပြကွက် စီမံခန့်ခွဲမှု

မော်ဒယ်တိုင်းတွင် အများဆုံး context ပြကွက် ရှိသည်။ စာရွက်စာတမ်း ကြီးများမှ chunksအားလုံး မထည့်နိုင်ပါ။ စနစ်သည် အဓိကဆက်စပ်မှုရှိသော chunks အပေါင်း N ခု (ပုံမှန်အားဖြင့် ၅ ခု) ကိုရွေးချယ် သာ လုံလောက်သော context ဖြင့် ပြန်လည် ဖြေကြားရန် ကြိုးစားသည်။

## RAG သုံးဖို့ အရေးပါတဲ့အချိန်

RAG သည် အမြဲတမ်း မှန်ကန်သော နည်းလမ်းမဟုတ်ပါ။ အောက်ဖော်ပြထားတဲ့ ဆုံးဖြတ်ချက်လမ်းညွှန်က RAG အသုံးပြုသင့်သောအခါနှင့် ရိုးရိုးနည်းလမ်းများ (prompt ထဲတွင် အကြောင်းအရာ ထည့်သွင်းခြင်း သို့မဟုတ် မော်ဒယ်၏ အတွင်းစစ် အသိပညာအပေါ် မူတည်မှု) သာ လိုအပ်သောအချိန်တို့ကို သတ်မှတ်ရာတွင် ကူညီပေးသည်-

<img src="../../../translated_images/my/when-to-use-rag.1016223f6fea26bc.webp" alt="When to Use RAG" width="800"/>

**RAG ကိုသုံးပါက:**
- ပုဂ္ဂိုလ်ရေးစာရွက်စာတမ်းများနှင့် ပတ်သက်သောမေးခွန်းများကို ဖြေကြားခြင်း
- သတင်းအချက်အလက်များသည် အမြဲပြောင်းလဲနေခြင်း (မူဝါဒများ၊ စျေးနှုန်းများ၊ ဖော်ပြချက်များ)
- တိကျမှန်ကန်မှုအတွက် ဒေတာအရင်းအမြစ်ကို ကိုးကားရပါမည်
- အကြောင်းအရာသည် တစ်ခုတည်းသော prompt တွင် ထည့်သွင်းရန် မလုံလောက်နိုင်သော ကြီးမားမှုရှိပါသည်
- သင်သည် အတည်ပြုနိုင်သော၊ အခြေခံထားသော အဖြေများ လိုအပ်သည်

**RAG ကို အသုံးမပြုသင့်သောအခါများ:**
- မေးခွန်းများသည် မော်ဒယ်တွင် ရှိပြီးသား ယေဘုယျအသိပညာကို လိုအပ်သောအခါ
- အချိန်နှင့်တပြေးညီ ဒေတာလိုအပ်သောအခါ (RAG သည် တင်သွင်းထားသောစာရွက်စာတမ်းများပေါ်မှာသာအလုပ်လုပ်သည်)
- အကြောင်းအရာသည် prompt များတွင် တိုက်ရိုက် ထည့်သွင်းရန် လုံလောက်လောက် လိုအပ်သောအခါ
