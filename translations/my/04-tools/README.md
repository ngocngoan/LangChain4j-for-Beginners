## What You'll Learn

ယခုအထိ၊ သင်သည် AI နှင့် စကားပြောနည်း၊ ပြောဆိုရန် သင့်တော်သော ပရောမ့်များ ဖန်တီးနည်း နှင့် တုံ့ပြန်ချက်များကို သင့်စာရွက်များတွင် အခြေခံပေးနည်းများကို သင်ယူခဲ့ပါသည်။ သို့သော်မူ၊ အခြေခံကန့်သတ်ချက်တစ်ခုရှိနေသေးသည်- ဘာသာစကားမော်ဒယ်များသည် စာသားအတုထုတ်နိုင်သော်လည်း၊ မိုးလေဝသစစ်ဆေးခြင်း၊ မှတ်ချက်တွက်ချက်မှု၊ ဒေတာဘေ့စ် မေးနိုင်ခြင်း သို့မဟုတ် ပြင်ပစနစ်များနှင့် ဆက်သွယ်နိုင်ခြင်း မပြုလုပ်နိုင်ပါ။

ကိရိယာများက ဤအခြေအနေကို ပြောင်းလဲပေးသည်။ မော်ဒယ်ကို ခေါ်နိုင်သောလုပ်ဆောင်ချက်များထောက်ပံ့ခြင်းအားဖြင့်၊ စာသားထုတ်လုပ်သူမော်ဒယ်မှ လုပ်ငန်းဆောင်ရွက်နိုင်သော ကိုယ်စားလှယ်သို့ ပြောင်းလဲစေသည်။ မော်ဒယ်သည် မည်သည့်အခါကိရိယာလိုအပ်ကြောင်း၊ မည်သည့်ကိရိယာကို အသုံးပြုမည်နည်း၊ နှင့် parameter များကို မည်သည့်သဘောဖြင့် ပေးပို့မည်နည်း ဆုံးဖြတ်သည်။ သင်၏ကုဒ်သည်လုပ်ဆောင်ချက်ဖြစ်အပြီး ယင်းရလဒ်ကို ပြန်လည်ပေးပို့သည်။ မော်ဒယ်မှ ထိုရလဒ်ကို တုံ့ပြန်ချက်တွင် ထည့်သွင်းပေးသည်။

## Understanding AI Agents with Tools

> **📝 မှတ်ချက်။** ဤမော်ဒူးတွင် "agents" ဟု ဆိုသည်မှာ ကိရိယာခေါ်ယူမှု ဆောင်နိုင်သော AI အကူအညီသူများကို ရည်ညွှန်းသည်။ ၎င်းသည် [Module 05: MCP](../05-mcp/README.md) တွင်ဖော်ပြမည့် **Agentic AI** ပုံစံများ (အစီအစဉ်ရေးဆွဲခြင်း၊ မှတ်ဉာဏ် နှင့် မျိုးစုံအဆင့် သုံးသပ်မှုများ ပါဝင်သည့် စက်အမြန်ဆောင် အေးဂျင့်များ) နှင့်ကွဲပြားပါသည်။

ကိရိယာမပါတော့မည်ဆိုလျှင် ဘာသာစကားမော်ဒယ်သည် သင်၏လေ့ကျင့်မှုဒေတာမှ စာသားကိုသာ ထုတ်လွှင့်နိုင်သည်။ မိုးလေဝသအခြေအနေကို မေးလျှင် ခန့်မှန်းရမည် ဖြစ်သည်။ ကိရိယာများပေးလျှင်၊ မိုးလေဝသ API ဖုန်းခေါ်ခြင်း၊ မှတ်ချက်တွက်ချက်ခြင်း သို့မဟုတ် ဒေတာဘေ့စ် မေးခြင်း ပြုလုပ်နိုင်ပြီး ထိုရလဒ်များကို စာတိုင်းထဲတြင်းဆက်ဖောက်မှတ်ထိုးနိုင်သည်။

<img src="../../../translated_images/my/what-are-tools.724e468fc4de64da.webp" alt="Without Tools vs With Tools" width="800"/>

*ကိရိယာမပါက မော်ဒယ်သည် ခန့်မှန်းသာဖြစ် - ကိရိယာရှိလျှင် API များခေါ်၊ တွက်ချက်မှုများဆောင်ရွက်၊ အချိန်နောက်ဆုံးဒေတာများပြန်လည်ရရှိစေသည်။*

ကိရိယာပါရှိသည့် AI ကိုယ်စားလှယ်သည် **Reasoning and Acting (ReAct)** ပုံစံကိုလိုက်နာသည်။ မော်ဒယ်က တုံ့ပြန်ခြင်းတစ်ခုသာမဟုတ်၊ မည်လိုလိုအပ်သလဲစဉ်းစားသည့်အခါ အသုံးပြုစရာအရာကို ခေါ်ယူပြီး ရလဒ်ကိုကြည့်ရှုပြီး ထပ်ဆောင်ရွက်မည်/မည်မဟုတ် ဆုံးဖြတ်သည်-

1. **Reason** — ကိုယ်စားလှယ်သည် အသုံးပြုသူ၏မေးခွန်းကို ဖြန့်ခွဲ၍ မည်သည့်အချက်အလက်လိုအပ်သည် သတ်မှတ်သည်။
2. **Act** — ကိုယ်စားလှယ်သည် သင့်တော်သောကိရိယာကို ရွေးချယ်၊ parameter များမှန်ကန်စွာ ထုတ်လုပ်ပြီး ဦးတည်ခေါ်ယူသည်။
3. **Observe** — ကိုယ်စားလှယ်သည် ကိရိယာ၏ ရလဒ်ကို လက်ခံ၍ အကဲဖြတ်သည်။
4. **Repeat or Respond** — အချက်အလက်ပို၍ လိုအပ်လျှင် ပြန်လည်လုပ်ဆောင်၊ မဟုတ်လျှင် နောက်ဆုံးဖြေဆိုချက်ကို တွဲဖက်ရေးဆွဲသည်။

<img src="../../../translated_images/my/react-pattern-detail.96a5efeeb6dd2f61.webp" alt="ReAct Pattern" width="800"/>

*ReAct စက်ဝိုင်း၊ ကိုယ်စားလှယ်သည် လုပ်ဆောင်ရမည်ကို စဉ်းစားပြီး ကိရိယာခေါ်ယူရေးဆောင်ကိ၊ လုပ်ဆောင်မှုရလဒ်ကို ကြည့်၍ နောက်ဆက်တွဲလုပ်ဆောင်ခြင်း၊ နောက်ဆုံးဖြေဆိုချက် ပေးပို့သည်။*

အဆိုပါလုပ်ငန်းစဉ်သည် အလိုအလျောက်ဖြစ်ပေါ်သည်။ သင်သည် ကိရိယာများနှင့် ၎င်းတို့၏ဖေါ်ပြချက်များကို သတ်မှတ်ပေးသည်။ မော်ဒယ်က မည်အခါ၊ မည်သို့ အသုံးပြုမည်ကို ဆုံးဖြတ်ပေးသည်။

## How Tool Calling Works

### Tool Definitions

[WeatherTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) | [TemperatureTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/TemperatureTool.java)

သင်သည် ဖေါ်ပြချက်များနှင့် parameter သတ်မှတ်ချက်များ ပြတ်သားစွာပါရှိသော လုပ်ဆောင်ချက်များကို သတ်မှတ်ပြုလုပ်သည်။ မော်ဒယ်သည် ၎င်းများကို စနစ် prompt တွင် မြင်နိုင်ပြီး မည်သည့်ကိရိယာသည် မည်သို့ဆောင်ရွက်သည်ကို နားလည်သည်။

```java
@Component
public class WeatherTool {
    
    @Tool("Get the current weather for a location")
    public String getCurrentWeather(@P("Location name") String location) {
        // မိမိရာသီဥတုပြည့်စုံစစ်ဆေးမှု ဆင့်လမ်း
        return "Weather in " + location + ": 22°C, cloudy";
    }
}

@AiService
public interface Assistant {
    String chat(@MemoryId String sessionId, @UserMessage String message);
}

// အကူအညီပေးသည်ကို Spring Boot ဖြင့် အလိုအလျောက် ချိတ်ဆက်ထားသည်၊
// - ChatModel bean
// - @Component အတန်းများမှ စွဲချက် @Tool နည်းလမ်းများအားလုံး
// - အစည်းအဝေး စီမံခန့်ခွဲမှုအတွက် ChatMemoryProvider
```

အောက်ပါ ပုံကြမ်းသည် သတ်မှတ်ချက်များ၏ အပိုင်းအစကိုသွားသင်ပြပြီး မည်သို့ AI သည် ကိရိယာကိုခေါ်ယူရန်အချိန်နှင့် parameter များကို ေပးပို့ရမည်နဲ့ နားလည်အောင်ကူညီသည်ကို ဖော်ပြသည်-

<img src="../../../translated_images/my/tool-definitions-anatomy.f6468546037cf28b.webp" alt="Anatomy of Tool Definitions" width="800"/>

*ကိရိယာအကြောင်းအရာအတွက် - @Tool သည် AI ကို မည်အခါအသုံးပြုရမည်ဟု ပြောကြား၊ @P သည် parameter တစ်ခုချင်း၏ ဖော်ပြချက်ဖြစ်ပြီး @AiService သည် စတင်တပ်ဆင်ချိန်တွင် အားလုံးကို ဆက်သွယ်ပေးသည်။*

> **🤖 GitHub Copilot Chat ဖြင့် စမ်းသပ်ပါ:** [`WeatherTool.java`](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) ဖွင့်ပြီး မေးမြန်းပါ-
> - "မိုးလေဝသ API ကို mock data အစား OpenWeatherMap API ဖော်စပ်ပါက နည်းလမ်းသည် ဘယ်လိုဖြစ်မလဲ?"
> - "AI ကို ရိုးရှင်းတိကျစွာအသုံးပြုရန်ကိရိယာဖေါ်ပြချက်ကောင်းမည်သို့ရေးရမလဲ?"
> - "API error များနှင့် rate limit များကို ကိရိယာဆောင်ရွက်ရာတွင် မည်သို့ ကိုင်တွယ်သင့်သလဲ?"

### Decision Making

အသုံးပြုသူမှ "စီးယတ်တယ်မြို့ရှိ မိုးလေဝသသည် ဘာလဲ?" ဟူ၍ မေးသည့်အခါ၊ မော်ဒယ်က ကိရိယာတစ်ခုခုကို မနှစ်သက်စွာရွေးမထားပဲ၊ အသုံးပြုသူရဲ့ရည်ရွယ်ချက်နှင့် လက်ရှိကိရိယာဖေါ်ပြချက်များအား စိစစ်ဂဏန်းဖြင့် ချိန်ညှိပြီး အကောင်းဆုံးကို ရွေးချယ်သည်။ ထို့နောက် parameter များပါဝင်သည့် ဖော်ပြချက်တစ်ခုဖြင့် function call ဖန်တီးသည်။ ဤအမှုတွင် `location`ကို `"Seattle"` ဟူ၍ သတ်မှတ်သည်။

အသုံးပြုသူတစ်ဦး၏ တောင်းဆိုချက်နှင့် မမှန်ကန်သော ကိရိယာရှိပါက မော်ဒယ်သည် မိမိလေ့လာမှုအရ မိမိ၏ ဗဟုသုတမှ တုံ့ပြန်သည်။ ကိရိယာ များအများကြီးဖြစ်ပါက အသေးစိတ်အကျဉ်းဆုံး ကိရိယာကို ဖြစ်စေရန် ရွေးချယ်သည်။

<img src="../../../translated_images/my/decision-making.409cd562e5cecc49.webp" alt="How the AI Decides Which Tool to Use" width="800"/>

*မော်ဒယ်သည် အသုံးပြုသူရဲ့ရည်ရွယ်ချက်နှင့် ကိရိယာတိုင်းကို သုံးသပ်၍ အကောင်းဆုံးကို ရွေးချယ်သည် - ထိုကြောင့် ရိုးရှင်းတိတိကျကျ ဖေါ်ပြချက်ရေးရန်အရေးကြီးသည်။*

### Architecture: Spring Boot Auto-Wiring

ဤမော်ဒူသည် LangChain4j ၏ Spring Boot ပေါင်းသင်းမှုဖြင့် `@AiService` interface များ အသုံးပြုသည်။ စတင်တပ်ဆင်ချိန်ကို စတင်၍ Spring Boot သည် `@Tool` လုပ်ဆောင်ချက်ပါရှိသည့် `@Component` များအားလုံး၊ သင့် `ChatModel` bean နှင့် `ChatMemoryProvider` ကို ရှာဖွေပြီး အစုလိုက် `Assistant` interface တစ်ခုအဖြစ် ပြုလုပ်ပေးသည်။

<img src="../../../translated_images/my/spring-boot-wiring.151321795988b04e.webp" alt="Spring Boot Auto-Wiring Architecture" width="800"/>

*@AiService interface သည် ChatModel, ကိရိယာပစ္စည်းများ နှင့် memory provider များကို ချိတ်ဆက်ပေးပြီး Spring Boot သည် အလိုအလျောက် ချိတ်ဆက်မှုကို စီမံခန့်ခွဲပေးသည်။*

ဤနည်းလမ်း၏ အဓိကအကျိုးကျေးဇူးများ-

- **Spring Boot auto-wiring** — ChatModel နှင့် ကိရိယာများ အလိုအလျောက် ထည့်သွင်းပေးသည်။
- **@MemoryId ပုံစံ** — session အခြေပြု မှတ်ဉာဏ်စီမံခန့်ခွဲမှု။
- **Single instance** — Assistant တစ်ခုတည်း ဖန်တီးပြီး ထပ်မံအသုံးပြုမှု တိုးတက်စေသည်။
- **Type-safe execution** — Java method များကို တိုက်ရိုက် ကုဒ်အမျိုးအစား ပြောင်းခြင်းဖြင့် ခေါ်ယူသည်။
- **Multi-turn orchestration** — ကိရိယာ Chaining ကို အလိုအလျောက် စီမံပေးသည်။
- **Zero boilerplate** — လက်ကိုင် `AiServices.builder()` ခေါ်ယူမှု လုံးဝ မလိုအပ်ပါ။

အခြား နည်းလမ်းများ (`AiServices.builder()` ကို လက်ဖြင့် ကုဒ်ရေးခြင်း) သည် ကုဒ်ပိုများပြီး Spring Boot ၏ စွမ်းဆောင်ရည် အကျိုးရှိမှု မရပါ။

## Tool Chaining

**Tool Chaining** — ကိရိယာအခြေပြု ကိုယ်စားလှယ်များ၏ အင်အားအပြည့်အစုံကို တစ်မေးခွန်းတည်းတွင် ကိရိယာများစွာ လိုအပ်သောအခါ တွေ့နိုင်သည်။ "စီးယတ်တယ်မြို့ရှိ မိုးလေဝသ Fahrenheit အတိုင်း" ဟုပြောလျှင် ကိုယ်စားလှယ်သည် ကိရိယာ နှစ်ခုချိတ်ဆက်ခြင်းလုပ်ဆောင်သည်- ပထမထုတ်လွှင့်မှုမှာ `getCurrentWeather` သည် Celsius အပူချိန်ရယူပြီး ထိုတန်ဖိုးကို `celsiusToFahrenheit` သို့ ဖြတ်သွား၍ ပြောင်းလဲသည်- အားလုံးကို စကားပြောချိန်တစ်ခုတည်းတွင် ပြီးစီးသည်။

<img src="../../../translated_images/my/tool-chaining-example.538203e73d09dd82.webp" alt="Tool Chaining Example" width="800"/>

*Tool chaining လုပ်ငန်းစဉ် - agent သည် အရင်ဆုံး getCurrentWeather ကို ခေါ်ပြီး Celsius ရလဒ်ကို celsiusToFahrenheit သို့ ပေးပို့ကာ ပေါင်းစပ်ဖြေချာ ပြောကြားသည်။*

**Graceful Failures** — mock data မရှိသောမြို့ için မိုးလေဝသ တောင်းဆိုပါက ကိရိယာသည် အမှားစာတမ်းပြန်ပေးပြီး AI သည် ယခုစာပိုဒ် ဝန်ကြီးမဟုတ်ကြောင်းရှင်းလင်းထားသည်။ ကိရိယာများသည် စိတ်ချရသော ပျက်ကွက်မှုများ ဖြစ်ပေါ်ပါက လျင်မြန်အကာအကွယ် ဆောင်ရွက်ထားသည်။ အောက်ပါ ပုံသည် နှစ်မျိုးဖြစ်ပုံကို နှိုင်းယှဉ်ပြသည်- error handling အကောင်းပြုလုပ်ပါက AI ကိုယ်စားလှယ်သည် အမှားကို ဖမ်းယူပြီး အကူအညီဖြင့် ဖြေကြားသော်လည်း error handling မရှိပါက အပြည့်အစုံ ဝေဖန်ချက်ဖြင့် application ပြိုကွဲသည်။

<img src="../../../translated_images/my/error-handling-flow.9a330ffc8ee0475c.webp" alt="Error Handling Flow" width="800"/>

*ကိရိယာ ပျက်ကွဲစဉ်အမှားကို agent က ဖမ်းယူပြီး အကူအညီယူနိုင်သည့် ရှင်းလင်းချက်ဖြင့်တုံ့ပြန်သည်။ Application ပြိုကွဲခြင်း မဖြစ်စေရန်။*

ဤသို့ တစ်ချက်တည်း ပြောဆိုမှုတွင် ဖြစ်ပေါ်ပါသည်။ agent သည် ကိရိယာ ခေါ်ယူမှုများ နှစ်ပွင့်စိတ်အလိုက် စီမံသည်။

### ရိုးရှင်းသော ကိရိယာအသုံးပြုမှု စမ်းကြည့်ပါ

ရိုးရှင်းတဲ့ တောင်းဆိုချက်နဲ့စတင်ပါ။ "100 ဒီဂရီ ‌ဖာရင်ဟိုက်ကို ဆယ်လ်စီယပ်ပြောင်းပါ"။ တာဝန်ရှိသူက ဒီအပူချိန် ပြောင်းလဲခြင်းကိရိယာ လိုအပ်ကြောင်း သိရှိပြီး၊ မှန်ကန်သော ပါရာမီတာများဖြင့် ခေါ်ယူပြီး ရလဒ်ကို ပြန်ပေးပါသည်။ ဘယ်လောက်သဘာဝမျိုးဖြစ်သနည်း သတိပြုပါ - ဘယ်ကိရိယာကို အသုံးပြုမယ်၊ ဘယ်လိုခေါ်ဆိုမယ် စသည့်အကြောင်း မသတ်မှတ်ပေးခဲ့ဘူး။

### ကိရိယာချိတ်ဆက်မှု စမ်းသပ်ကြည့်ပါ

ယခု နည်းနည်းရှုပ်ထွေးသည့် တောင်းဆိုချက်တစ်ခုကို စမ်းသပ်ပါ။ "စီယာတယ်မြို့တွင် မိုးလေဝသ ဘာလဲ၊ ပြီးတော့ ဖာရင်ဟိုက်သို့ ပြောင်းပါ"။ တာဝန်ရှိသူသည် အဆင့်ဆင့် နည်းလမ်းများဖြင့် အလုပ်လုပ်သည့်အတိုင်း မျှော်လင့်ကြည့်ပါ။ စတင်ပြီး မိုးလေဝသ (ဆယ်လ်စီယပ်ဖြင့် ပြန်လည်ရရှိသည်) ကို ရယူကာ၊ ဖာရင်ဟိုက်သို့ ပြောင်းပေးရန် လိုအပ်သည်ကို သတိပြုခဲ့ပြီး ပြောင်းလဲမှု ကိရိယာကို ခေါ်ယူကာ နှစ်ခုရလဒ်တို့ကို တစ်ခုတည်းဖြင့် ပေါင်းထည့်ပေးသည်။

### စကားပြော လည်ပတ်မှုကို မြင်ပါ

စကားပြော အင်တာဖေ့စ်သည် စကားပြောမှတ်တမ်းကို ထိန်းသိမ်းထားပြီး မျိုးစုံအချိန်အခါများတွင် ဆက်သွယ်ဖလှယ်နိုင်စေသည်။ ယခင်တောင်းဆိုချက်များနှင့် ဖြေကြားချက်များအားလုံးကို မြင်နိုင်ပြီး စကားပြော ထွက်ရှိမှုကို လွယ်ကူစွာ နားလည်နိုင်သည်။

<a href="images/tools-conversation-demo.png"><img src="../../../translated_images/my/tools-conversation-demo.89f2ce9676080f59.webp" alt="Conversation with Multiple Tool Calls" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*ရိုးရှင်းသော ပြောင်းလဲမှုများ၊ မိုးလေဝသ ရှာဖွေမှုများနှင့် ကိရိယာချိတ်ဆက်မှုများပါသော မျိုးစုံ အချိန်ဆက်သော စကားပြော*

### မတူညီသော တောင်းဆိုချက်များနှင့် စမ်းသပ်ကြည့်ပါ

မျိုးစုံ ချိတ်ဆက်မှုများအား စမ်းသပ်ပါ -
- မိုးလေဝသ ရှာဖွေမှုများ - "တိုကျိုမြို့ ရဲ့ မိုးလေဝသ ဘာလဲ?"
- အပူချိန်ပြောင်းလဲမှုများ - "25°C ကို ကယ်လ်ဗင် အဖြစ် ဘယ်လိုရလဲ?"
- ပေါင်းစပ်တောင်းဆိုချက်များ - "ပါရီမြို့ အပူချိန် မိုးလေဝသ ကိုစစ်ဆေးပြီး 20°C ထက် ထိပ်တန်းကောင်လား ပြောပြပါ"

တာဝန်ရှိသူသည် သဘာဝဘာသာစကားကို မည်သို့ ဖော်ပြပြီး သင့်တော်သော ကိရိယာ ခေါ်ဆောင်မှုများသို့ ပြောင်းလဲပေးသည်ကို သတိပြုပါ။

## အဓိက သဘောတရားများ

### ReAct နမူနာ (စဉ်းစားပြီး လုပ်ဆောင်ခြင်း)

တာဝန်ရှိသူသည် စဉ်းစားမှု (ဘာလုပ်မရမလဲ ဆုံးဖြတ်ခြင်း) နှင့် လုပ်ဆောင်မှု (ကိရိယာများသုံးခြင်း) အကြား အစားထိုးမှု လုပ်သည်။ ဒီနမူနာက လမ်းညွှန်ချက်တင်ဖြေကြားခြင်းဖြစ်ခြင်းမဟုတ်ဘဲ စွမ်းဆောင်နိုင်သော ပြဿနာပြေရှင်းမှုကို အခွင့်အရေးပေးသည်။

### ကိရိယာဖော်ပြချက် အရေးကြီးသည်

သင့်ကိရိယာ ဖော်ပြချက်အရည်အသွေးသည် တာဝန်ရှိသူက သိထားသည့် အသုံးပြုနိုင်မှုကို တိုက်ရိုက် သက်ရောက်မှုရှိသည်။ ရှင်းလင်း၊ တိကျသော ဖော်ပြချက်များက မော်ဒယ်ကို မည်သို့ခေါ်ယူရမည်၊ ဘယ်အချိန်တွင်ခေါ်ယူရမည်ကို နားလည်ရန်အထောက်အကူပြုသည်။

### အကြိမ်ရေစီမံခန့်ခွဲမှု

`@MemoryId` မှတ်သားချက်က အလိုအလျောက် အကြိမ်ရေ အခြေခံ အမှတ်တရ ကို စီမံခန့်ခွဲမှု ပေးစွမ်းသည်။ အကြိမ်ရေအလိုက် သီးခြား ChatMemory အမည်ရှိသော ကွန်ပျူတာကို ChatMemoryProvider bean ဖြင့် စီမံခန့်ခွဲ၍ အသုံးပြုသူစွာ ဆက်သွယ်မှုများသည် ဆက်လက်လူးလဲခြင်းမရှိဘဲ ဖြစ်နိုင်သည်။ အောက်ပါ ပုံတော်သည် အသုံးပြုသူ များစွာမှုကို မဟုတ်ဘဲ သီးသန့် အမှတ်တရများသို့ လမ်းညွှန်ပေးသည်ကို ပြသည် -

<img src="../../../translated_images/my/session-management.91ad819c6c89c400.webp" alt="Session Management with @MemoryId" width="800"/>

*အကြိမ်ရေတိုင်းသည် သီးခြားသော စကားပြောမှတ်တမ်းသို့ သွားပြီး အသုံးပြုသူများက တစ်ယောက်ယောက်ရဲ့ စာကို ပြန်ကြည့်လေ့ မရှိပါ။*

### အမှားကို ကိုင်တွယ်ခြင်း

ကိရိယာများ ထွက်ပေါ်နိုင်သည် — API များ အချိန်ကုန်၊ ပါရာမီတာများ မှားယွင်း၊ ပြင်ပ ဝန်ဆောင်မှုများ ပျက်ကွက်နိုင်သည်။ ထုတ်လုပ်ရေးအတွက် တာဝန်ရှိသူများအတွက် Error Handling လိုအပ်သည်။ အမှားပြသောအခါ မော်ဒယ်က ပြဿနာကို ရှင်းပြခြင်း သို့မဟုတ် အခြားနည်းလမ်းများ စမ်းသပ်နိုင်ရန် ဖြစ်သည်။ ကိရိယာမှ Exception တစ်ခု ထုတ်လွှင့်သော် LangChain4j သည် ထိုအမှားစာကို ဖမ်းယူပြီး မော်ဒယ်သို့ ပြန်ပို့ကာ သဘာဝဘာသာဖြင့် ကျပန်းစွာရှင်းပြနိုင်သည်။

## ရနိုင်သော ကိရိယာများ

အောက်ဖော်ပြပါ ပုံတော်က သင်တည်ဆောက်နိုင်သော ကိရိယာ ဆိုက်အီကိုစက်မှုပုံကို ပြသည်။ ဒီမောဂျူးမှာ မိုးလေဝသနှင့် အပူချိန် ကိရိယာများကို ပြသထားပေမယ့် `@Tool` နမူနာသည် Java နည်းလမ်းမည်မျှအတွက်မဆို အသုံးပြုနိုင်သည် — ဒေတာဘေ့စ် စုံစမ်းမှုများမှ ငွေပေးချေမှုလုပ်ငန်းများအထိ။

<img src="../../../translated_images/my/tool-ecosystem.aad3d74eaa14a44f.webp" alt="Tool Ecosystem" width="800"/>

*@Tool ဖြင့် မှတ်သားထားသည့် Java နည်းလမ်း တစ်ခုခြင်းခြင်းသည် AI အတွက် အသုံးပြုနိုင်ရန် ဖြစ်ပါသည် — ဒီနမူနာသည် ဒေတာဘေ့စ်များ၊ API များ၊ အီးမေးလ်၊ ဖိုင်များ အလုပ်များသို့ တိုးချဲ့နေပါသည်။*

## ကိရိယာ အခြေခံ တာဝန်ရှိသူများ ဘယ်အချိန် အသုံးပြုရန်လဲ

အသုံးပြုလိုသော တောင်းဆိုချက်အတွက် မရှိမဖြစ် ကိရိယာလိုအပ်သလားဟူ၍ ဆုံးဖြတ်ချက်ပြုပါ။ AI က ပြင်ပ စနစ်များနှင့် ဆက်သွယ်ရန်လိုအပ်မည်ဟု ဆိုရင် အသုံးပြုပါ၊ မဟုတ်ရင် သိမြင်မှုအပေါ် အဖြေရေးပါ။ အောက်တွင် အကျဉ်းချုပ် ဖော်ပြချက်ဖြင့် တောင့်ပြောထားသည် -

<img src="../../../translated_images/my/when-to-use-tools.51d1592d9cbdae9c.webp" alt="When to Use Tools" width="800"/>

*အချိန်နှင့်တပြေးညီ ဒေတာ၊ တွက်ချက်မှုများ၊ လုပ်ဆောင်မှုများ အတွက် ကိရိယာများအသုံးပြုသည်။ အများပြည်သူ ရှင်းလင်းသော သဘောတရားများနှင့် ဖန်တီးမှု အလုပ်များ မလိုအပ်ပါ။*

## ကိရိယာများ နှင့် RAG

မော်ဂျူး 03 နှင့် 04 နှစ်ခုစလုံး AI ၏ စွမ်းဆောင်ရည်တိုးတက်စေသော်လည်း နည်းလမ်းအစားထိုးမှု ကွာခြားသည်။ RAG သည် မော်ဒယ်အတွက် **အသိပညာ** ရယူစေပြီး စာတမ်းများကို ဆွဲထုတ်ပေးသည်။ ကိရိယာများက မော်ဒယ်ကို **လုပ်ဆောင်ချက်များ** ပြုလုပ်ရေးအတွက် နည်းလမ်းခေါ်ယူနိုင်စေသည်။ အောက်ပါ ပုံတော်သည် နှစ်ခုန် လုပ်ငန်းစဉ်များ၊ နှင့် တန်ဖိုးဆိုင်ရာ ကွာခြားချက်များကို နှိုင်းယှဥ် တင်ပြထားသည် -

<img src="../../../translated_images/my/tools-vs-rag.ad55ce10d7e4da87.webp" alt="Tools vs RAG Comparison" width="800"/>

*RAG သည် ရပ်တည်မှု ရှိသော စာရွက်များမှ သတင်းအချက်အလက်ကို ရယူသည်။ ကိရိယာများသည် လုပ်ဆောင်ချက်များကို အကောင်အထည်ဖော်ရန် နှင့် အချိန်နှင့်တပြေးညီ အချက်အလက်များကို ရယူရန် ဖြစ်သည်။ ထုတ်လုပ်ရေးစနစ်များတွင် နှစ်ခုစလုံး ပေါင်းစပ်အသုံးပြုကြသည်။*

လက်တွေ့မှာ လူကြီးမင်း စံချိန်စနစ်များတွင် နှစ်မျိုးစလုံး (RAG က မိမိစာရွက်စာတမ်းများတွင် မူတည်ကာ ဖြေကြားမှုများအတွက်၊ ကိရိယာများက အသက်တမ်းရှိဒေတာရန် ယူခြင်း သို့မဟုတ် လုပ်ဆောင်ချက်များ ပြုလုပ်ရန်) ပေါင်းစပ်အသုံးပြုကြသည်။
