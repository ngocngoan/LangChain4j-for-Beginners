## ನೀವು ಕಲಿಯುವುದು

ಈವರೆಗೆ, ನೀವು ಎಐ ಜೊತೆ ಸಂವಾದ ನಡೆಸುವುದು, ಪ್ರಾಂಪ್ಟ್‌ಗಳನ್ನು ಪರಿಣಾಮಕಾರಿಯಾಗಿ ರಚಿಸುವುದು ಮತ್ತು ನಿಮ್ಮ ಡಾಕ್ಯುಮೆಂಟ್‌ಗಳಲ್ಲಿ ಪ್ರತಿಕ್ರಿಯೆಗಳನ್ನು ನೆಲೆಮಾಡುವುದು ಹೇಗೆ ಎನ್ನುವುದನ್ನು ಕಲಿಸಿದ್ದಾರೆ. ಆದರೆ ಇನ್ನೂ ಒಂದು ಮೂಲಭೂತ ಸೀಮಿತತೆ ಇದೆ: ಭಾಷಾ ಮಾದರಿಗಳು ಕೇವಲ ಪಠ್ಯವನ್ನು ರಚಿಸುವಷ್ಟೇ ಮಾಡಬಲ್ಲವು. ಅವು ಹವಾಮಾನವನ್ನು ಪರಿಶೀಲಿಸಲು, ಲెక్కಾಚಾರಗಳನ್ನು ಮಾಡಬಲ್ಲವುವು, ಡೇಟಾಬೇಸ್‌ಗಳನ್ನು ವಿಚಾರಿಸಬಹುದಿಲ್ಲ, ಅಥವಾ ಹೊರಗಿನ ವ್ಯವಸ್ಥೆಗಳೊಂದಿಗೆ ಸಂವಹನ ಮಾಡಲಾರವು.

ಟೂಲ್ಗಳು ಇದನ್ನು ಬದಲಾಯಿಸುತ್ತವೆ. ಮಾದರಿಗೂ ಕರೆಮಾಡಬಹುದಾದ ಕಾರ್ಯಗಳಿಗೆ ಪ್ರವೇಶ ನೀಡುವುದರಿಂದ, ನೀವು ಅವರನ್ನು ಪಠ್ಯ ರಚನಕಾರರಾಗಿ ಅಲ್ಲದೆ ಕ್ರಮ ಕೈಗೊಳ್ಳುವ ಏಜೆಂಟ್‌గా ಪರಿವರ್ತಿಸುತ್ತೀರಿ. ಮಾದರಿ ಯಾವಾಗ ಟೂಲ್ ಬೇಕಾಗುತ್ತದೆ, ಯಾವ ಟೂಲ್ ಬಳಸಬೇಕೆಂದು ಮತ್ತು ಯಾವ ಪ್ಯಾರಾಮೀಟರ್‌ಗಳನ್ನು ಪಾಸ್ ಮಾಡಬೇಕೆಂದು ತೀರ್ಮಾನಿಸುತ್ತದೆ. ನಿಮ್ಮ ಕೋಡ್ ಆ ಕಾರ್ಯವನ್ನು ನಿರ್ವಹಿಸಿ ಫಲಿತಾಂಶವನ್ನು ಹಿಂತಿರುಗಿಸುತ್ತದೆ. ಮಾದರಿ ಆ ಫಲಿತಾಂಶವನ್ನು ತನ್ನ ಪ್ರತಿಕ್ರಿಯೆಯಲ್ಲಿ ಸೇರಿಸುತ್ತದೆ.

## ಟೂಲ್ಗಳೊಂದಿಗೆ ಎಐ ಏಜೆಂಟ್ಗಳನ್ನು ಅರ್ಥಮಾಡಿಕೊಳ್ಳುವುದು

> **📝 ಸೂಚನೆ:** ಈ ಮODULEನಲ್ಲಿ "ಏಜೆಂಟ್" ಪದವು ಟೂಲ್-ಕರೆಗೊಳ್ಳುವ ಸಾಮರ್ಥ್ಯಗಳಿಂದ ಸುಧಾರಿಸಿದ ಎಐ ಸಹಾಯಕರಿಗೆ ಸಂಬಂಧಿಸಿದೆ. ಇದು [ಮODULE 05: MCP](../05-mcp/README.md)ನಲ್ಲಿ ಚರ್ಚಿಸುವ **Agentic AI** ಮಾದರಿಗಳಿಂದ (ಸ್ವತಂತ್ರ ಏಜೆಂಟ್ಗಳು ಯೋಜನೆ, ಸ್ಮರಣೆ, ಮತ್ತು ಬಹು ಹಂತದ ಯುಕ್ತಿಯೊಂದಿಗೆ) ಭಿನ್ನವಾಗಿದೆ.

ಟೂಲ್ಗಳು ಇಲ್ಲದಿದ್ದರೆ, ಭಾಷಾ ಮಾದರಿ ತನ್ನ ತರಬೇತಿ ದತ್ತಾಂಶದಿಂದ ಮಾತ್ರ ಪಠ್ಯವನ್ನು ರಚಿಸಬಲ್ಲದು. ಹವಾಮಾನ ಯಾವುವು ಎಂದು ಕೇಳಿದರೆ, ಅದು ಊಹಿಸಬೇಕಾಗುತ್ತದೆ. ಟೂಲ್ಗಳು ನೀಡಿದರೆ, ಅದು ಹವಾಮಾನ API ಕರೆಗೊಳ್ಳಬಹುದು, ಲెక్కಾಚಾರಗಳನ್ನು ನಿರ್ವಹಿಸಬಹುದು ಅಥವಾ ಡೇಟಾಬೇಸ್ ಅನ್ನು ವಿಚಾರಿಸಬಹುದು — ನಂತರ ಆ ನಿಜವಾದ ಫಲಿತಾಂಶಗಳನ್ನು ತನ್ನ ಪ್ರತಿಕ್ರಿಯೆಯಲ್ಲಿ ಜೋಡಿಸುತ್ತದೆ.

<img src="../../../translated_images/kn/what-are-tools.724e468fc4de64da.webp" alt="Without Tools vs With Tools" width="800"/>

*ಟೂಲ್ಗಳಿಲ್ಲದೆ ಮಾದರಿ ಕೇವಲ ಊಹೆ ಮಾಡಬಹುದು — ಟೂಲ್ಗಳೊಂದಿಗೆ ಅದು APIs ಅನ್ನು ಕರೆ ಮಾಡಬಹುದು, ಲెక్కಾಚಾರಗಳನ್ನು ನಡೆಸಬಹುದು ಮತ್ತು ನಿಜಕಾಲದ ಡೇಟಾ ಹಿಂತಿರುಗಿಸಬಹುದು.*

ಟೂಲ್ಗಳೊಂದಿಗೆ AI ಏಜೆಂಟ್ **ಯುಕ್ತಿ ಮತ್ತು ಕ್ರಮ (ReAct)** ಮಾದರಿಯನ್ನು ಅನುಸರಿಸುತ್ತದೆ. ಮಾದರಿ ಕೇವಲ ಪ್ರತಿಕ್ರಿಯಿಸುವುದಿಲ್ಲ — ಇದು ಬೇಕಾದುದನ್ನು ಯೋಚಿಸುತ್ತದೆ, ಟೂಲ್ ಅನ್ನು ಕರೆಮಾಡಿ ಆಕ್ರಮಣ ವಾಗಿದೆ, ಫಲಿತಾಂಶವನ್ನು ಪರಿಶೀಲಿಸುತ್ತದೆ, ನಂತರ ಮತ್ತೊಮ್ಮೆ ಕ್ರಮ ಕೈಗೊಳ್ಳಬೇಕೆ, ಅಂತಿಮ ಉತ್ತರ ಕೊಡುವುದಕ್ಕೆ ಯೋಚಿಸುತ್ತದೆ:

1. **ಯುಕ್ತಿ** — ಬಳಕೆದಾರನ ಪ್ರಶ್ನೆಯನ್ನು ವಿಶ್ಲೇಷಿಸಿ ಮತ್ತು ಬೇಕಾದ ಮಾಹಿತಿಯನ್ನು ನಿರ್ಧರಿಸಿ
2. **ಕ್ರಮ** — ಸರಿಯಾದ ಟೂಲ್ ಆಯ್ಕೆ ಮಾಡಿ, ಸರಿಯಾದ ಪ್ಯಾರಾಮೀಟರ್‌ಗಳನ್ನು ರಚಿಸಿ, ಅದನ್ನು ಕರೆಮಾಡಿ
3. **ನೋಟ** — ಟೂಲ್ ನ ಹೊರಹೊಮ್ಮುವಿಕೆಯನ್ನು ಸ್ವೀಕರಿಸಿ ಮತ್ತು ಫಲಿತಾಂಶವನ್ನು ವಿಮರ್ಶಿಸಿ
4. **ಪುನರಾವರ್ತನೆ ಅಥವಾ ಪ್ರತಿಕ್ರಿಯೆ** — ಹೆಚ್ಚು ಡೇಟಾ ಬೇಕಾದರೆ ಲೂಪ್ ಮಾಡಿ; ಇಲ್ಲದಿದ್ದರೆ ಸಹಜ ಭಾಷೆಯ ಉತ್ತರ ರಚಿಸಿ

<img src="../../../translated_images/kn/react-pattern-detail.96a5efeeb6dd2f61.webp" alt="ReAct Pattern" width="800"/>

*ReAct ಚಕ್ರ — ಏಜೆಂಟ್ ಏನು ಮಾಡಬೇಕೆಂದಿದೆಂದು ಯುಕ್ತಿ ಮಾಡುವುದು, ಟೂಲ್ ಕರೆಮಾಡಿ ಕ್ರಮ ಕೈಗೊಳ್ಳುವುದು, ಫಲಿತಾಂಶವನ್ನು ನೋಡಿ, ಆಗ ತನಕ ಲೂಪ್ ಮಾಡುವುದು ಮತ್ತು ಅಂತಿಮ ಉತ್ತರವನ್ನು ಒದಗಿಸುವುದು.*

ಇದು ಸ್ವಯಂಚಾಲಿತವಾಗಿದು. ನೀವು ಟೂಲ್ಗಳನ್ನು ಮತ್ತು ಅವುಗಳ ವಿವರಣೆಗಳನ್ನು ನಿರ್ಧರಿಸುತ್ತೀರಿ. ಮಾದರಿ ಯಾವಾಗ ಮತ್ತು ಹೇಗೆ ಬಳಸಬೇಕೆಂಬ ನಿರ್ಧಾರವನ್ನು ಕೈಗೊಳ್ಳುತ್ತದೆ.

## ಟೂಲ್ ಕರೆದೊಯ್ಯುವಿಕೆ ಹೇಗೆ ಕೆಲಸಮಾಡುತ್ತದೆ

### ಟೂಲ್ ವ್ಯಾಖ್ಯಾನಗಳು

[WeatherTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) | [TemperatureTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/TemperatureTool.java)

ನೀವು ಸ್ಪಷ್ಟ ವಿವರಣೆಗಳು ಮತ್ತು 参数ವಿವರಣೆಗಳೊಂದಿಗೆ ಕಾರ್ಯಗಳನ್ನು ವ್ಯಾಖ್ಯಾನಿಸುತ್ತೀರಿ. ಮಾದರಿ ಈ ವಿವರಣೆಗಳನ್ನು ತನ್ನ ಸಿಸ್ಟಮ್ ಪ್ರಾಂಪ್ಟ್‌ನಲ್ಲಿ ನೋಡಿ ಪ್ರತಿ ಟೂಲ್ಗೆ ಏನು ಕೆಲಸ ಇದೆ ಎಂದು ಅರ್ಥಮಾಡಿಕೊಳ್ಳುತ್ತದೆ.

```java
@Component
public class WeatherTool {
    
    @Tool("Get the current weather for a location")
    public String getCurrentWeather(@P("Location name") String location) {
        // ನಿಮ್ಮ ಹವಾಮಾನ ಹುಡುಕುವ ಲಾಜಿಕ್
        return "Weather in " + location + ": 22°C, cloudy";
    }
}

@AiService
public interface Assistant {
    String chat(@MemoryId String sessionId, @UserMessage String message);
}

// Spring Boot ಮೂಲಕ ಸಹಾಯಕನು ಸ್ವತಃ ತರುವಾಯ ಮಾಡಲಾಗಿದೆ:
// - ChatModel ಬಿನ್
// - @Component ವರ್ಗಗಳಲ್ಲಿನ ಎಲ್ಲಾ @Tool ವಿಧಾನಗಳು
// - ಸೆಷನ್ ನಿರ್ವಹಣೆಯಿಗಾಗಿ ChatMemoryProvider
```

ಕೆಳಗಿನ ಚಿತ್ರಣವು ಪ್ರತಿ ಅನೋಟೇಶನ್ ಅನ್ನು ವಿಶ್ಲೇಷಿಸುತ್ತದೆ ಮತ್ತು ಯಾವ ಭಾಗ ಎಐಗೆ ಟೂಲನ್ನು ಕರೆಮಾಡಬೇಕೆಂದು ಮತ್ತು ಯಾವವಾದಿ ಪ್ಯಾರಾಮೀಟರ್ ಗಳನ್ನು ಪಾಸ್ ಮಾಡಬೇಕೆಂದು ತಿಳಿಸುತ್ತದೆ:

<img src="../../../translated_images/kn/tool-definitions-anatomy.f6468546037cf28b.webp" alt="Anatomy of Tool Definitions" width="800"/>

*ಟೂಲ್ ವ್ಯಾಖ್ಯಾನದ ರಚನೆ — @Tool ಎಂಬುದು ಎಐಗೆ ಇದನ್ನು ಯಾವಾಗ ಬಳಸಬೇಕು ಎಂದು ತಿಳಿಸುತ್ತದೆ, @P ಪ್ರತಿ 参数ವಿವರಣೆ ನೀಡುತ್ತದೆ, ಮತ್ತು @AiService ಪ್ರಾರಂಭಾವಧಿಯಲ್ಲಿ ಎಲ್ಲವನ್ನೂ ಸಂಪರ್ಕಿಸುತ್ತದೆ.*

> - "ಎಐಗೆ ಸರಿಯಾಗಿ ಬಳಸಲು ಸಹಾಯ ಮಾಡುವ ಉತ್ತಮ ಟೂಲ್ ವಿವರಣೆ ಏನು?"
> - "ಟೂಲ್ ಅನುಷ್ಠಾನದಲ್ಲಿ API ದೋಷಗಳು ಮತ್ತು ರೇಟ್ ಲಿಮಿಟ್‌ಗಳನ್ನು ಹೇಗೆ ನಿರ್ವಹಿಸಬೇಕು?"

### ನಿರ್ಧಾರ ಕೈಗೊಳ್ಳುವುದು

ಬಳಕೆದಾರನು "ಸ್ಯಾಟಲ್‌ನಲ್ಲಿ ಹವಾಮಾನ ಹೇಗಿದೆ?" ಎಂದು ಕೇಳಿದಾಗ, ಮಾದರಿ ಯಾದೃಚ್ಛಿಕವಾಗಿ ಟೂಲನ್ನು ಆಯ್ಕೆಮಾಡುವುದಿಲ್ಲ. ಅದು ಬಳಕೆದಾರರ ಉದ್ದೇಶವನ್ನು ಪ್ರತಿ ಟೂಲ್ ವಿವರಣೆಯೊಂದಿಗೆ ಹೋಲಿಸಿ, ಪ್ರಸ್ತುತತೆಯನ್ನು ಅಸತ್ತೂಡುತ್ತದೆ, ಮತ್ತು ಉಚಿತವಾದ ಯಥಾರ್ಥವನ್ನು ಆಯ್ಕೆಮಾಡುತ್ತದೆ. ನಂತರ ಅದು ಸರಿಯಾದ ಪ್ಯಾರಾಮೀಟರ್‌ಗಳೊಂದಿಗೆ ಸಾಂರಚನಾತ್ಮಕ ಕಾರ್ಯಕರೆಗೊಳ್ಳುವಿಕೆಯನ್ನು ತಯಾರಿಸುತ್ತದೆ — ಈ ಸಂದರ್ಭದಲ್ಲಿ, `location` ಅನ್ನು `"Seattle"` ಎಂದು ಸೆಟ್ ಮಾಡುತ್ತದೆ.

ಯಾವುದೇ ಟೂಲ್ ಬಳಕೆದಾರರ ವಿನಂತಿಗೆ ಸರಿಹೊಬ್ಬದಿದ್ದರೆ, ಮಾದರಿ ತನ್ನದೇ ಜ್ಞಾನದಿಂದ ಉತ್ತರಿಸುತ್ತದೆ. ಹಲವಾರು ಟೂಲ್ಗಳು ಹೊಂದಿದ್ದರೆ, ಅತ್ಯಂತ ಸ್ಪಷ್ಟವಾದುದನ್ನು ಆಯ್ಕೆಮಾಡುತ್ತದೆ.

<img src="../../../translated_images/kn/decision-making.409cd562e5cecc49.webp" alt="How the AI Decides Which Tool to Use" width="800"/>

*ಮಾದರಿ ಪ್ರತಿ ಲಭ್ಯವಿರುವ ಟೂಲ್ನ್ನು ಬಳಕೆದಾರರ ಉದ್ದೇಶದ ವಿರುದ್ಧ ಮೌಲ್ಯಾಂಕನ ಮಾಡಿ ಉತ್ತಮದನ್ನು ಆಯ್ಕೆಮಾಡುತ್ತದೆ — ಆದ್ದರಿಂದ ಸ್ಪಷ್ಟ, ಖಚಿತ ಟೂಲ್ ವಿವರಣೆಗಳನ್ನು ಬರೆಯುವಿಕೆ ಎಷ್ಟು ಮುಖ್ಯವೋ ಅುತ್ತದೆ.*

ಸ್ಕೃತಿ ಬೂಟ್ ಆಟೋ- ಬೈಯರ್ ಡಿಕ್ಲೇರೇಟಿವ್ `@AiService` ಇಂಟರ್ಫೇಸ್ನನ್ನು ಎಲ್ಲಾ ನೋಂದಾಯಿಸಲಾದ ಟೂಲ್ಗಳಿಗೆ ತಾನೆ ಸಂಪರ್ಕಿಸುತ್ತದೆ, ಮತ್ತು LangChain4j ಸ್ವಯಂಚಾಲಿತವಾಗಿ ಟೂಲ್ ಕರೆಗಳನ್ನು ಕಾರ್ಯಗತಗೊಳಿಸುತ್ತದೆ. ಹಿಂದೆ, ಬಳಕೆದಾರರ ಸಹಜ ಭಾಷೆಯ ಪ್ರಶ್ನೆಯಿಂದ ಆರಂಭವಾಗಿ ಸಹಜ ಭಾಷೆಯ ಉತ್ತರಕ್ಕೆ ಹದಿನಾರು ಹಂತಗಳ ಮೂಲಕ ಸಂಪೂರ್ಣ ಟೂಲ್ ಕರೆ ಹರಿವು ಚಾಲನೆಯುತ:

### ಪ್ರತಿಕ್ರಿಯೆ ಪುಟ್‌ಪಡಿಸುವಿಕೆ

ಮಾದರಿ ಹವಾಮಾನ ಡೇಟಾವನ್ನು ಸ್ವೀಕರಿಸಿ ಅದನ್ನು ಬಳಕೆದಾರನಿಗೆ ಸಹಜ ಭಾಷೆಯ ಪ್ರತಿಕ್ರಿಯೆಯಾಗಿ ವಿನ್ಯಾಸಗೊಳ್ಳುತ್ತದೆ.

### ವಾಸ್ತುಶಿಲ್ಪ: ಸ್ಪ್ರಿಂಗ್ ಬೂಟ್ ಆಟೋ- ವೈಯರಿಂಗ್

ಈ ಮODULE LangChain4j ನ ಸ್ಪ್ರಿಂಗ್ ಬೂಟ್ ಏಕೀಕರಣವನ್ನು ಡಿಕ್ಲೇರೇಟಿವ್ `@AiService` ಇಂಟರ್ಫೇಸ್ಗಳೊಂದಿಗೆ ಬಳಸುತ್ತದೆ. ಪ್ರಾರಂಭದಲ್ಲಿ ಸ್ಪ್ರಿಂಗ್ ಬೂಟ್ ಪ್ರತಿಯೊಬ್ಬ `@Component` ಅನ್ನು ಕಂಡುಹಿಡಿಯುತ್ತದೆ, ಅದರಲ್ಲಿ `@Tool` ವಿಧಾನಗಳಿವೆ, ನಿಮ್ಮ `ChatModel` ಬಿನ್, ಮತ್ತು `ChatMemoryProvider` — ನಂತರ ಅವುಗಳೆಲ್ಲವನ್ನು ಶೂನ್ಯ ಬೋಯ್ಲರ್‌ಪ್ಲೇಟ್‌ನೊಂದಿಗೆ ಏಕ `Assistant` ಇಂಟರ್ಫೇಸ್ಗಾಗಿ ವೈಯರ್ ಮಾಡುತ್ತದೆ.

<img src="../../../translated_images/kn/spring-boot-wiring.151321795988b04e.webp" alt="Spring Boot Auto-Wiring Architecture" width="800"/>

*@AiService ಇಂಟರ್ಫೇಸ್ ChatModel, ಟೂಲ್ ಘಟಕಗಳು, ಮತ್ತು ಮೆಮೊರಿ ಒದಗಿಸುವಿಕೆಯನ್ನು ಸೇರಿಸಲು ಸಹಾಯಕವಾಗಿದ್ದು — ಸ್ಪ್ರಿಂಗ್ ಬೂಟ್ ಇವೆಲ್ಲವನ್ನು ಆಟೋಮ್ಯಾಟಿಕ್‌ ಆಗಿ ವೈಯರ್ ಮಾಡುತ್ತದೆ.*

ಇದಲ್ಲದೆ, ಕೆಳಗಿನ ಪೂರ್ಣ ವಿನಂತಿ ಜೀವನಚರ್ತಿಯನ್ನು ಸೀಕ್ವೆನ್ಸ್ ಚಿತ್ತಾರ ರೂಪದಲ್ಲಿ ನೋಡಿ — HTTP ವಿನಂತಿಯಿಂದ ಕಂಡ್ರೋಲ್, ಸರ್ವೀಸ್, ಆಟೋ- ವೈಯರ್ ಪ್ರಾಕ್ಸಿ ಮೂಲಕ, ಟೂಲ್ ಕಾರ್ಯವಾಯುವಿಕೆ ಮತ್ತು ಹಿಂತಿರುಗುವಿಕೆಗೆ至る:

<img src="../../../translated_images/kn/spring-boot-sequence.f83e3d485aa4a3c6.webp" alt="Spring Boot Tool Calling Sequence" width="800"/>

*ಪೂರ್ಣ ಸ್ಪ್ರಿಂಗ್ ಬೂಟ್ ವಿನಂತಿ ಜೀವನಚರ್ತಿ — HTTP ವಿನಂತಿ ಕಂಡ್ರೋಲರ್ ಮತ್ತು ಸರ್ವೀಸ್ ಮೂಲಕ ಆಟೋ- ವೈಯರ್‌ ಆಗಿರುವ Assistant ಪ್ರಾಕ್ಸಿಗೆ ಹರಿದು, ಅದು LLM ಮತ್ತು ಟೂಲ್ ಕರೆಗಳನ್ನು ಸ್ವಯಂಚಾಲಿತವಾಗಿ ಸಂಚಾಲಿಸುತ್ತದೆ.*

ಈ ವಿಧಾನದಿಂದ ಲಭ್ಯವಿರುವ ಪ್ರಮುಖ ಪ್ರಯೋಜನಗಳು:

- **ಸ್ಪ್ರಿಂಗ್ ಬೂಟ್ ಆಟೋ- ವೈಯರಿಂಗ್** — ChatModel ಮತ್ತು ಟೂಲ್ಗಳು ಸ್ವಯಂಚಾಲಿತವಾಗಿ ಇಂಜೆಕ್ಟ್ ಆಗುತ್ತವೆ
- **@MemoryId ಮಾದರಿ** — ಸ್ವಯಂಚಾಲಿತ ಸೆಷನ್ ಆಧಾರಿತ ಮೆಮೊರಿ ನಿರ್ವಹಣೆ
- **ಒಂದು ಉದಾಹರಣೆ** — Assistant ಒಂದೇ ಬಾರಿಗೆ ಸೃಷ್ಟಿಸಿ ಉತ್ತಮ ಕಾರ್ಯಕ್ಷಮತೆಗೆ ಮರುಬಳಕೆ
- **ಟೈಪ್-ಸೇಫ್ ಕಾರ್ಯ ನಿರ್ವಹಣೆ** — ಜಾವಾ ವಿಧಾನಗಳನ್ನು ನೇರವಾಗಿ ಟೈಪ್ ಪರಿವರ್ತನೆಯೊಂದಿಗೆ ಕರೆ ಮಾಡುತ್ತದೆ
- **ಬಹು-ತಿರುವು ಸಂಚಾಲನೆ** — ಟೂಲ್ ಚೈನಿಂಗ್ ಅನ್ನು ಸ್ವಯಂಚಾಲಿತವಾಗಿ ನಿರ್ವಹಿಸುತ್ತದೆ
- **ಸೊನ್ನೆ ಬೋಯ್ಲರ್‌ಪ್ಲೇಟ್** — ಕೈಯಿಂದ `AiServices.builder()` ಕರೆಗಳು ಅಥವಾ ಮೆಮೊರಿ ಹ್ಯಾಷ್‌ಮ್ಯಾಪ್ ಇಲ್ಲ

ಮ್ಯಾನುಯಲ್ `AiServices.builder()` ಬಳಸಿ ಪರ್ಯಾಯ ವಿಧಾನಗಳು ಹೆಚ್ಚುವರಿ ಕೋಡ್ ಅಗತ್ಯವಿರುತ್ತವೆ ಮತ್ತು ಸ್ಪ್ರಿಂಗ್ ಬೂಟ್ ಏಕೀಕರಣ ಲಾಭಗಳನ್ನು ಮಿಸ್ ಮಾಡುತ್ತವೆ.

## ಟೂಲ್ ಚೈನಿಂಗ್

**ಟೂಲ್ ಚೈನಿಂಗ್** — ಸತ್ಯಕ್ಕೂರ್ಾಚಿ ಜಾಗತಿಕ ಶಕ್ತಿ ಟೂಲ್- ಆಧಾರಿತ ಏಜೆಂಟ್‌ಗಳಲ್ಲಿ ಕಂಡುಬರುತ್ತದೆ, ಏಕೆಂದರೆ ಒಂದೇ ಪ್ರಶ್ನೆಗೆ ಬಹು ಟೂಲ್ಗಳ ಅಗತ್ಯವಿರುತ್ತದೆ. "ಸ್ಯಾಟಲ್‌ನ ಹವಾಮಾನ ಫಾರೆನ್‌ಹೀಟ್ ನಲ್ಲಿ ಏನು?" ಎಂದು ಕೇಳಿದರೆ, ಏಜೆಂಟ್ ಸ್ವಯంచಾಲಿತವಾಗಿ ಎರಡು ಟೂಲ್ಗಳನ್ನು ಕಂಡುಹಿಡಿದು ಸಂಪರ್ಕಿಸುತ್ತದೆ: ಮೊದಲು `getCurrentWeather` ಅನ್ನು ಸಿ ಸೆಲ್ಸಿಯಸ್ ನಲ್ಲಿ ತಾಪಮಾನ ಪಡೆಯಲು ಕರೆಮಾಡುವುದು, ನಂತರ ಆ ಬೆಲೆಯನ್ನು `celsiusToFahrenheit` ಗೆ ಪರಿವರ್ತನೆಗಾಗಿ ಪಾಸ್ ಮಾಡುವುದು — ಎಲ್ಲವೂ ಒಂದೇ ಸಂವಾದ ತಿರುವಿನಲ್ಲಿ.

<img src="../../../translated_images/kn/tool-chaining-example.538203e73d09dd82.webp" alt="Tool Chaining Example" width="800"/>

*ಟೂಲ್ ಚೈನಿಂಗ್ ಕಾರ್ಯಾಚರಣೆ — ಏಜೆಂಟ್ ಮೊದಲು getCurrentWeather ಕರೆ ಮಾಡುತ್ತದೆ, ನಂತರ ಸೆಲ್ಸಿಯಸ್ ಫಲಿತಾಂಶವನ್ನು celsiusToFahrenheit ಗೆ ಪೈಪ್ ಮಾಡಿ ಸಂಯುಕ್ತ ಉತ್ತರ ನೀಡುತ್ತದೆ.*

**ಗೌರವಯುತ ವಿಫಲತೆಗಳು** — ನಕಲಿ ಡೇಟಾದಲ್ಲಿಲ್ಲದ ನಗರದಲ್ಲಿ ಹವಾಮಾನ ಕೇಳಿದರೆ, ಟೂಲ್ ದೋಷ ಸಂದೇಶ ನೀಡುತ್ತದೆ ಮತ್ತು ಎಐ ಸಹಾಯಕವಾಗಿ ಸಹಾಯ ಮಾಡಲು ಆಗದಂತಿದೆ ಎಂದು ವಿವರಿಸುತ್ತದೆ, ಸಂಘಟನೆ ಬೀಳಿಸುವುದಿಲ್ಲ. ಟೂಲ್ಗಳು ಸುರಕ್ಷಿತವಾಗಿ ವಿಫಲವಾಗುತ್ತವೆ. ಕೆಳಗಿನ ಚಿತ್ರಣದಲ್ಲಿ ಎರಡು ವಿಧಾನಗಳ ವ್ಯತ್ಯಾಸ ತೋರಿಸಲಾಗಿದೆ — ಸರಿಯಾದ ದೋಷ ನಿರ್ವಹಣೆಯೊಂದಿಗೆ, ಏಜೆಂಟ್ ಪರಿಸ್ಥಿತಿಯನ್ನು ಹಿಡಿದು ಸಹಾಯಕ ಪ್ರತಿಕ್ರಿಯೆ ನೀಡುತ್ತದೆ, ಇಲ್ಲದಿದ್ದರೆ ಸಂಪೂರ್ಣ ಆ್ಯಪ್ಲಿಕೇಶನ್ ಕ್ರ್ಯಾಶ್ ಆಗುತ್ತದೆ:

<img src="../../../translated_images/kn/error-handling-flow.9a330ffc8ee0475c.webp" alt="Error Handling Flow" width="800"/>

*ಟೂಲ್ ವಿಫಲವಾದಾಗ, ಏಜೆಂಟ್ ದೋಷವನ್ನು ಹಿಡಿದು ಕ್ರ್ಯಾಶ್ ಆಗದೆ ಸಹಾಯಕರ ಪ್ರತಿಕ್ರಿಯೆ ನೀಡುತ್ತದೆ.*

ಇದು ಒಂದೇ ಸಂವಾದ ತಿರುವಿನಲ್ಲಿ ನಡೆಯುತ್ತದೆ. ಏಜೆಂಟ್ ಸ್ವತಂತ್ರವಾಗಿ ಬಹು ಟೂಲ್ ಕರೆಗಳನ್ನು ಸಂಚಾಲಿಸುತ್ತದೆ.

## ಅಪ್ಲಿಕೇಶನ್ ಬಳಸುವುದು

ಈ ಅಪ್ಲಿಕೇಶನ್ ಒಂದು ವೆಬ್ ಇಂಟರ್ಫೇಸ್ ಒದಗಿಸುತ್ತದೆ, ಇಲ್ಲಿ ನೀವು ಹವಾಮಾನ ಮತ್ತು ತಾಪಮಾನ ಪರಿವರ್ತನೆ ಉಪಕರಣಗಳಿಗೆ ಪ್ರವೇಶ ಇರುವ AI ഏಜೆಂಟ್ ಜೊತೆಗೆ ಸಂವಹನ ಮಾಡಬಹುದು. ಇಂತಿದಾಗಿದೆ ಇಂಟರ್ಫೇಸ್ — ಇದರಲ್ಲಿ ತ್ವರಿತ ಪ್ರಾರಂಭ ಉದಾಹರಣೆಗಳು ಮತ್ತು ವಿನಂತಿಗಳನ್ನು ಕಳುಹಿಸಲು ಚಾಟ್ ಪ್ಯಾನೆಲ್ ಸೌಲಭ್ಯ ಇದೆ:

<a href="images/tools-homepage.png"><img src="../../../translated_images/kn/tools-homepage.4b4cd8b2717f9621.webp" alt="AI Agent Tools Interface" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*AI ಏಜೆಂಟ್ ಟೂಲ್ಸ್ ಇಂಟರ್ಫೇಸ್ - ತ್ವರಿತ ಉದಾಹರಣೆಗಳು ಮತ್ತು ಸಾಧನಗಳೊಂದಿಗೆ ಸಂವಹನಕ್ಕೆ ಚಾಟ್ ಇಂಟರ್‌ಫೇಸ್*

### ಸರಳ ಉಪಕರಣ ಬಳಕೆಯನ್ನು ಪ್ರಯತ್ನಿಸಿ

ಸರಳ ವಿನಂತಿಯಿಂದ ಪ್ರಾರಂಭಿಸಿ: "100 ಡಿಗ್ರಿ ಫಾರೆನ್‌ಹೈಟ್ ಅನ್ನು ಸೆಲ್ಸಿಯಸ್ ಗೆ ಪರಿವರ್ತಿಸಿ". ಏಜೆಂಟ್ ಇದನ್ನು ತಾಪಮಾನ ಪರಿವರ್ತನೆ ಉಪಕರಣ ಬೇಕಾಗಿದೆ ಎಂದು ಗುರುತಿಸಿ, ಸರಿಯಾದ ಅಂಶಗಳೊಂದಿಗೆ ಕರೆ ಮಾಡಿ ಫಲಿತಾಂಶವನ್ನು ಹಿಂದಿರುಗಿಸುತ್ತದೆ. ನೀವು ಯಾವ ಉಪಕರಣವನ್ನು ಬಳಸಬೇಕೆಂದು ಅಥವಾ ಹೇಗೆ ಕರೆ ಮಾಡಬೇಕೆಂದು ನಿರ್ದಿಷ್ಟಪಡಿಸದೆ ಈ ಸಹಜವಾಗಿದೆ ಎಂದು ಗಮನಿಸಿ.

### ಉಪಕರಣ ಸರಪಳಿ ಪರೀಕ್ಷೆ ಮಾಡಿ

ಇದೀಗ ಮತ್ತಷ್ಟು ಸಂಕೀರ್ಣವನ್ನೊಬ್ಬಿಸಿ: "ಸಿಯಾಟಲ್ ನಲ್ಲಿ ಹವಾಮಾನ ಏನು ಮತ್ತು ಅದನ್ನು ಫಾರೆನ್‌ಹೈಟ್ ಗೆ ಪರಿವರ್ತಿಸು?" ಆಯ್ಕೆ ಪ್ರಕ್ರಿಯೆಯ ಮೂಲಕ ಏಜೆಂಟ್ ಹೇಗೆ ಕೆಲಸ ಮಾಡುತ್ತದೆ ಎಂಬುದನ್ನು ನೋಡಿ. ಮೊದಲನೇದಾಗಿ ಹವಾಮಾನ ಪಡೆಯುತ್ತದೆ (ಅದು ಸೆಲ್ಸಿಯಸ್ ನ್ನು ನೀಡುತ್ತದೆ), ನಂತರ ಫಾರೆನ್‌ಹೈಟ್ ಗೆ ಪರಿವರ್ತನೆ ಬೇಕಾಗಿದೆ ಎಂದು ಗುರುತಿಸಿ, ಪರಿವರ್ತನೆ ಉಪಕರಣವನ್ನು ಕರೆಸಿ, ಎರಡೂ ಫಲಿತಾಂಶಗಳನ್ನು ಒಟ್ಟಾಗಿ ಉತ್ತರವಾಗಿ ಹೊಂದಾಣಿಕೆ ಮಾಡುತ್ತದೆ.

### ಸಂಭಾಷಣೆಯ ಪ್ರವಾಹವನ್ನು ನೋಡಿ

ಚಾಟ್ ಇಂಟರ್ಫೇಸ್ ಸಂಭಾಷಣೆ ಇತಿಹಾಸವನ್ನು ಕಾಪಾಡುತ್ತದೆ, ಇದು ಬಹು ಪ್ರತ್ಯುತ್ತರ ಸಂವಹನಗಳಿಗೂ ಅನುಮತಿಸುತ್ತದೆ. ನೀವು ಎಲ್ಲಾ ಹಳೆಯ ಪ್ರಶ್ನೆಗಳು ಮತ್ತು ಪ್ರತಿಕ್ರಿಯೆಗಳನ್ನು ನೋಡಬಹುದು, ಇದರಿಂದ ಸಂಭಾಷಣೆಯನ್ನು ಅರ್ಥಮಾಡಿಕೊಳ್ಳುವುದು ಮತ್ತು ಏಜೆಂಟ್ ಅನೇಕ ವಿನಿಮಯಗಳ ಮೂಲಕ ಭಾಗವನ್ನು ಹೇಗೆ ನಿರ್ಮಿಸುತ್ತದೆಯೋ ತಿಳಿದುಕೊಳ್ಳುವುದು ಸುಲಭ.

<a href="images/tools-conversation-demo.png"><img src="../../../translated_images/kn/tools-conversation-demo.89f2ce9676080f59.webp" alt="Conversation with Multiple Tool Calls" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*ಬಹು-ತಿರುವಿನ ಸಂಭಾಷಣೆ, ಸರಳ ಪರಿವರ್ತನೆಗಳು, ಹವಾಮಾನ ಪರಿಶೀಲನೆಗಳು ಮತ್ತು ಉಪಕರಣ ಸರಪಳಿ ತೋರಿಸುತ್ತಿದೆ*

### ವಿವಿಧ ವಿನಂತಿಗಳೊಂದಿಗೆ ಪ್ರಯೋಗ ಮಾಡಿ

ವಿವಿಧ ಸಮ್ಮಿಶ್ರಣ ಪ್ರಯತ್ನಿಸಿ:
- ಹವಾಮಾನ ಪರಿಶೀಲನೆಗಳು: "ಟೋಕಿಯೋದಲ್ಲಿ ಹವಾಮಾನ ಹೇಗಿದೆ?"
- ತಾಪಮಾನ ಪರಿವರ್ತನೆಗಳು: "25°C ಅನ್ನು ಕೆಲ್ವಿನ್ ಗೆ ಪರಿವರ್ತಿಸು?"
- ಸಂಯುಕ್ತ ಪ್ರಶ್ನೆಗಳು: "ಪ್ಯಾರಿಸ್ ನಲ್ಲಿ ಹವಾಮಾನ ಪರಿಶೀಲಿಸಿ ಮತ್ತು ಅದು 20°C ಗಿಂತ ಮೇಲಾಗಿದೆಯೇ ನೋಡಿ"

ಎದುರಾಗಿ, ಏಜೆಂಟ್ ಸಹಜ ಭಾಷೆಯನ್ನು ಹೇಗೆ ಅರ್ಥಮಾಡಿಕೊಂಡು ಸೂಕ್ತ ಉಪಕರಣ ಕರೆಗಳಿಗೆ ನಕ್ಷೆ ಮಾಡಿ ಎಂದು ಗಮನಿಸಿ.

## ಪ್ರಮುಖ ಕಲ್ಪನೆಗಳು

### ReAct ಮಾದರಿ (ತರ್ಕ ಮತ್ತು ಕ್ರಿಯೆ)

ಏಜೆಂಟ್ ತರ್ಕ (ಎನ್ನುವುದು ಹೇಗೆ ಎಂಬುದನ್ನು ನಿರ್ಧರಿಸುವುದು) ಮತ್ತು ಕ್ರಿಯೆ (ಉಪಕರಣಗಳನ್ನು ಬಳಸುವುದು) ನಡುವೆ ಬದಲಾಯಿಸುತ್ತಾನೆ. ಈ ಮಾದರಿ ಸ್ವಾಯತ್ತ ಸಮಸ್ಯಾ ಪರಿಹಾರವನ್ನು ಅನುಮತಿಸುತ್ತದೆ, ಕೇವಲ ಸೂಚನೆಗಳಿಗೆ ಪ್ರತಿಕ್ರಿಯಿಸುವುದನ್ನು ಮಾತ್ರವಲ್ಲ.

### ಉಪಕರಣ ವಿವರಣೆಗಳು ಪ್ರಮುಖವಾಗಿವೆ

ನಿಮ್ಮ ಉಪಕರಣಗಳ ವಿವರಣೆಗಳ ಗುಣಮಟ್ಟದಿಂದ ಏಜೆಂಟ್ ಅವುಗಳನ್ನು ಹೇಗೆ ಬಳಸುತ್ತದೆ ಎಂಬುದನ್ನು ನೇರವಾಗಿ ಪ್ರಭಾವಿಸುತ್ತದೆ. ಸ್ಪಷ್ಟ, ವಿಶೇಷ ವಿವರಣೆಗಳು ಮಾದರಿಯನ್ನು ಯಾವಾಗ ಮತ್ತು ಹೇಗೆ ಉಪಕರಣವನ್ನು ಕರೆಮಾಡಬೇಕೆಂದು ಸಹಾಯಮಾಡುತ್ತವೆ.

### ಸೆಷನ್ ನಿರ್ವಹಣೆ

`@MemoryId` ವಿವರಿಸುವಿಕೆ ಸ್ವಯಂಚಾಲಿತ ಸೆಷನ್ ಆಧಾರಿತ ಮೆಮೊರಿ ನಿರ್ವಹಣೆಯನ್ನು ಸಕ್ರಿಯಗೊಳಿಸುತ್ತದೆ. ಪ್ರತಿ ಸೆಷನ್ ID ಒಬ್ಬದೇ ತನ್ನ `ChatMemory` ಉದಾಹರಣೆಯನ್ನು `ChatMemoryProvider` ಬೀನ್ ಮೂಲಕ ನಿರ್ವಹಿಸಲಾಗುತ್ತದೆ, ಆದ್ದರಿಂದ ಬಹು ಬಳಕೆದಾರರು ಏಜೆಂಟ್ ಜೊತೆಗೆ ಒಂದೇ ಸಮಯದಲ್ಲಿ ಸಂವಹನ ನಡೆಸಬಹುದು ಮತ್ತು ಅವರ ಸಂಭಾಷಣೆಗಳು ಮಿಶ್ರಿತವಾಗುವುದಿಲ್ಲ. ಕೆಳಗಿನ ಚಿತ್ರದಲ್ಲಿ, ಬಹು ಬಳಕೆದಾರರು ಅವರ ಸೆಷನ್ ID ಗಳ ಆಧಾರದಲ್ಲಿ ವಿಭಿನ್ನ ಮೆಮೊರಿ ಸಂಗ್ರಹಗಳಿಗೆ ಹೇಗೆ ಹಾದು ಹೋಗುತ್ತಾರೆ ಎಂಬುದು ತೋರಿಸಲಾಗಿದೆ:

<img src="../../../translated_images/kn/session-management.91ad819c6c89c400.webp" alt="Session Management with @MemoryId" width="800"/>

*ಪ್ರತಿ ಸೆಷನ್ ID ಪ್ರತ್ಯೇಕ ಸಂಭಾಷಣಾ ಇತಿಹಾಸಕ್ಕೆ ನಕ್ಷೆ ಮಾಡುತ್ತದೆ — ಬಳಕೆದಾರರು ಪರಸ್ಪರ ಸಂದೇಶಗಳನ್ನು ನೋಡುವುದಿಲ್ಲ.*

### ದೋಷ ನಿರ್ವಹಣೆ

ಉಪಕರಣಗಳು ವಿಫಲವಾಗಬಹುದು — API ಗಳು ಸಮಯ ಮೀರಬಹುದು, ಪರಿಮಾಣಗಳು ತಪ್ಪಾಗಿರಬಹುದು, ಬಾಹ್ಯ ಸೇವೆಗಳು ಬಂದುಕೊಳ್ಳಬಹುದು. ಉತ್ಪಾದನಾ ಏಜೆಂಟ್‌ಗಳು ದೋಷ ನಿರ್ವಹಣೆಯನ್ನು ಹೊಂದಬೇಕು, ώστε ಮಾದರಿ ಸಮಸ್ಯೆಗಳನ್ನು ವಿವರಿಸಬಹುದು ಅಥವಾ ಬದಲಾವಣೆಗಳನ್ನು ಪ್ರಯತ್ನಿಸಬಹುದು, ಸಂಪೂರ್ಣ ಅಪ್ಲಿಕೇಶನ್ ಕ್ರ್ಯಾಶ್ ಆಗುವುದನ್ನು ತಪ್ಪಿಸಲು. ಉಪಕರಣದಲ್ಲಿ ಎಕ್ಸೆಪ್ಷನ್ Throw ಆದಾಗ, LangChain4j ಅದನ್ನು ಹಿಡಿದು ದೋಷ ಸಂದೇಶವನ್ನು ಮತ್ತೆ ಮಾದರಿಗೆ ಕಳುಹಿಸುತ್ತದೆ, ಇದು ಸಮಸ್ಯೆಯನ್ನು ಸಹಜ ಭಾಷೆಯಲ್ಲಿ ವಿವರಿಸಲು ನೆರವಾಗುತ್ತದೆ.

## ಲಭ್ಯವಿರುವ ಉಪಕರಣಗಳು

ಕೆಳಗಿನ ಚಿತ್ರವು ನೀವು ರಚಿಸಬಹುದಾದ ಉಪಕರಣಗಳ ವಿಶಾಲ ಪರಿಸರ ವ್ಯವಸ್ಥೆಯನ್ನು ತೋರಿಸುತ್ತದೆ. ಈ ಮೊಡ್ಯೂಲ್ ಹವಾಮಾನ ಮತ್ತು ತಾಪಮಾನ ಉಪಕರಣಗಳನ್ನು ಪ್ರದರ್ಶಿಸುತ್ತದೆ, ಆದರೆ ಅದೇ `@Tool` ಮಾದರಿ ಯಾವುದೇ Java ವಿಧಾನಕ್ಕೆ ಅನ್ವಯಿಸುತ್ತದೆ — ಡೇಟಾ ಬೇಸ್ ಪ್ರಶ್ನೆಗಳಿಂದ ಪಾವತಿ ಪ್ರಕ್ರಿಯೆಗೆ.

<img src="../../../translated_images/kn/tool-ecosystem.aad3d74eaa14a44f.webp" alt="Tool Ecosystem" width="800"/>

*ಯಾವುದೇ Java ವಿಧಾನ `@Tool` ಅಟ್ಯಾನೋಟೇಶನ್ ಸಹಿತ AI ಉಪಯೋಗಕ್ಕೆ ಲಭ್ಯವಾಗುತ್ತದೆ — ಮಾದರಿ ಡೇಟಾಬೇಸ್ಗಳು, API ಗಳು, ಇಮೇಲ್, ಫೈಲ್ ಕಾರ್ಯಾಚರಣೆಗಳು ಮತ್ತು ಇನ್ನಷ್ಟು ವಿಸ್ತರಿಸಿದೆ.*

## ಉಪಕರಣ ಆಧಾರಿತ ಏಜೆಂಟ್ ಗಳನ್ನು ಯಾವಾಗ ಬಳಸುವುದು

ಎಲ್ಲಾ ವಿನಂತಿಗಳಿಗೆ ಉಪಕರಣಗಳು hoeftಾ ಇಲ್ಲ. ತೀರ್ಮಾನವು ಏನುಂದರೆ AI ಬಾಹ್ಯ ವ್ಯವಸ್ಥೆಗಳೊಡನೆ ಸಂವಹನ ಮಾಡಬೇಕೇ ಅಥವಾ ತನ್ನ ಸ್ವಂತ ಜ್ಞಾನದಿಂದ ಉತ್ತರ ನೀಡಬಹುದೇ ಎಂಬುದು. ಕೆಳಗಿನ ಮಾರ್ಗದರ್ಶಿ ಉಪಕರಣಗಳು ಹಿತಕರವಾಗುವಾಗ ಮತ್ತು ಅವಶ್ಯಕವಿಲ್ಲದಾಗ ಯಾವುದು ಎಂದು ಸಂಕ್ಷಿಪ್ತವಾಗಿ ವಿವರಿಸುತ್ತದೆ:

<img src="../../../translated_images/kn/when-to-use-tools.51d1592d9cbdae9c.webp" alt="When to Use Tools" width="800"/>

*ತ್ವರಿತ ನಿರ್ಧಾರ ಮಾರ್ಗದರ್ಶಿ — ಉಪಕರಣಗಳು ನೈಜ-ಸಮಯ ಡೇಟಾ, ಗಣನೆಗಳು ಮತ್ತು ಕ್ರಿಯೆಗಳಿಗಾಗಿ; ಸಾಮಾನ್ಯ ಜ್ಞಾನ ಮತ್ತು ಸೃಜನಾತ್ಮಕ ಕಾರ್ಯಗಳಿಗೆ ಅವಶ್ಯಕವಿಲ್ಲ.*

## ಉಪಕರಣಗಳು ಮತ್ತು RAG

ಮೊಡ್ಯೂಲ್‌ಗಳು 03 ಮತ್ತು 04 ಎರಡೂ AI ಯ ಶಕ್ತಿಯನ್ನು ವಿಸ್ತಾರಗೊಳಿಸುತ್ತವೆ, ಆದೃಷ್ಠಾಂಶದಲ್ಲಿ ಬೇರೆ ರೀತಿಯಲ್ಲಿ. RAG ಮಾದರಿಗೆ ಡಾಕ್ಯುಮೆಂಟ್‌ಗಳನ್ನು ಪ್ರಾಪ್ತಿಗೊಳಿಸುವ ಮೂಲಕ **ಜ್ಞಾನ** ಕಾರ್ಯನಿರ್ವಹಿಸಲು ಅವಕಾಶ ನೀಡುತ್ತದೆ. ಉಪಕರಣಗಳು ಕಾರ್ಯಗಳನ್ನು ತಾನೆ — ಕಾರ್ಯ ನಿರ್ವಹಣೆಗಾಗಿ ಫಂಕ್ಷನ್‌ಗಳನ್ನು ಕರೆಯುವ ಮೂಲಕ. ಕೆಳಗಿನ ಚಿತ್ರದಲ್ಲಿ ಈ ಎರಡು ವಿಧಾನಗಳನ್ನು ಪಕ್ಕಪಕ್ಕವಾಗಿ ಹೋಲಿಸಲಾಗಿದೆ — ಪ್ರತಿಯೊಂದು ಕೆಲಸ ಹೇಗೆ ನಡೆಯುತ್ತದೆ ಮತ್ತು ಅವುಗಳ ನಡುವಿನ ವ್ಯತ್ಯಾಸಗಳನ್ನು ವಿವರಿಸುತ್ತವೆ:

<img src="../../../translated_images/kn/tools-vs-rag.ad55ce10d7e4da87.webp" alt="Tools vs RAG Comparison" width="800"/>

*RAG ಸ್ಥಿರ ಡಾಕ್ಯುಮೆಂಟ್‌ಗಳಿಂದ ಮಾಹಿತಿ ಪಡೆಯುತ್ತದೆ — ಉಪಕರಣಗಳು ಕ್ರಿಯೆಗಳನ್ನು ನಿರ್ವಹಿಸಿ ನೈಜ-ಸಮಯ ಡೇಟಾ ಪಡೆದುಕೊಳ್ಳುತ್ತವೆ. ಅನೇಕ ಉತ್ಪಾದನಾ ವ್ಯವಸ್ಥೆಗಳು ಎರಡನ್ನೂ ಸೇರಿಸಿ ಬಳಸುತ್ತವೆ.*

ವಾಸ್ತವದಲ್ಲಿ, ಅನೇಕ ಉತ್ಪಾದನಾ ವ್ಯವಸ್ಥೆಗಳು ಎರಡೂ ವಿಧಾನಗಳನ್ನು ಸೇರಿಸಿ ಬಳಸುತ್ತವೆ: RAG ನಿಮ್ಮ ಡಾಕ್ಯುಮೆಂಟೇಷನ್‌ನಲ್ಲಿ ಉತ್ತರಗಳನ್ನು ನೆಲೆಸಲು ಮತ್ತು ಉಪಕರಣಗಳು ನೈಜ-ಸಮಯ ಡೇಟಾ ಪಡೆಯಲು ಅಥವಾ ಕಾರ್ಯಗಳನ್ನು ನಿರ್ವಹಿಸಲು.
