## ನೀವು ಕಲಿಯುವದು ಏನು

ಇದುವರೆಗೆ, ನೀವು AI ಸಹ ಸಂಭಾಷಣೆ ನಡೆಸುವುದು, ಪ್ರಾಂಪ್ಟ್‌ಗಳನ್ನು ಸಮರ್ಥವಾಗಿ ರಚಿಸುವುದು ಮತ್ತು ನಿಮ್ಮ ದಾಖಲೆಗಳಲ್ಲಿ ಪ್ರತಿಕ್ರಿಯೆಗಳನ್ನು ಆಧರಿಸುವುದನ್ನು ಕಲಿತಿದ್ದೀರಿ. ಆದಾಗ್ಯೂ, ಇನ್ನೂ ಒಂದು ಮೂಲ ಮಿತಿ ಇದೆ: ಭಾಷಾ ಮಾದರಿಗಳು ಕೇವಲ ಪಠ್ಯವನ್ನು ಉತ್ಪಾದಿಸುತ್ತವೆ. ಅವು ಹವಾಮಾನವನ್ನು ಪರಿಶೀಲಿಸಲು, ಗಣನೆಗಳನ್ನು ನಿರ್ವಹಿಸಲು, ಡೇಟಾಬೇಸ್‌ಗಳನ್ನು ಪ್ರಶ್ನಿಸಲು ಅಥವಾ ಬಾಹ್ಯ ವ್ಯವಸ್ಥೆಗಳೊಂದಿಗೆ ಸಂವಹನ ಮಾಡಲು ಸಾಧ್ಯವಿಲ್ಲ.

ಸಂಚಾರಗಳು ಇದನ್ನು ಬದಲಿಸುತ್ತವೆ. ಮಾದರಿಗೆ ಕರೆಯಬಹುದಾದ ಕಾರ್ಯಗಳನ್ನು ನೀಡುವುದರಿಂದ, ನೀವು ಅದನ್ನು ಪಠ್ಯ ಉತ್ಪಾದಕನಿಂದ ಕಾರ್ಯಾಚರಣೆ ನಡೆಸುವ ಏಜೆಂಟ್ ಆಗಿ ಪರಿವರ್ತಿಸುತ್ತೀರಿ. ಮಾದರಿ ಯಾವಾಗ ಸಂಚಾರ ಬೇಕು, ಯಾವ ಸಂಚಾರ ಉಪಯೋಗಿಸಬೇಕು ಮತ್ತು ಯಾವ ಪ್ಯಾರಾಮೀಟರ್ಗಳನ್ನು ಪಾಸು ಮಾಡಬೇಕು ಎಂದು ತೀರ್ಮಾನಿಸುತ್ತದೆ. ನಿಮ್ಮ ಕೋಡ್ ಆ ಕಾರ್ಯವನ್ನು ಕಾರ್ಯಗತಗೊಳಿಸಿ ಫಲಿತಾಂಶವನ್ನು ಹಿಂತಿರುಗಿಸುತ್ತದೆ. ಮಾದರಿ ಆ ಫಲಿತಾಂಶವನ್ನು ತನ್ನ ಪ್ರತಿಕ್ರಿಯೆಯಲ್ಲಿ ಸೇರಿಸುತ್ತದೆ.

## ಸಂಚಾರಗಳೊಂದಿಗೆ AI ಏಜಂಟ್ಗಳನ್ನು ಅರ್ಥಮಾಡಿಕೊಳ್ಳುವುದು

> **📝 ಟಿಪ್ಪಣಿ:** ಈ ಮODULE ನಲ್ಲಿ "ಏಜಂಟ್ಗಳು" ಎಂಬುದು ಸಂಚಾರ ಕರೆಗೊಳ್ಳುವ ಸಾಮರ್ಥ್ಯವಿರುವ AI ಸಹಾಯಕರನ್ನು ಸೂಚಿಸುತ್ತದೆ. ಇದು [Module 05: MCP](../05-mcp/README.md) ನಲ್ಲಿ ನಾವು ಪರಿಚಯಿಸುವ **ಏಜೆಂಟಿಕ್ AI** ಮಾದರಿಗಳಿಂದ (ಸ್ವಯಂಚಾಲಿತ ಏಜಂಟ್ಗಳು ಯೋಜನೆ, ಸ್ಮರಣಾ ಮತ್ತು ಬಹು-ಹಂತ ತರ್ಕ ಹೊಂದಿವೆ) ಭಿನ್ನವಾಗಿದೆ.

ಸಂಚಾರಗಳಿಲ್ಲದೆ, ಒಂದು ಭಾಷಾ ಮಾದರಿ ಕೇವಲ ತನ್ನ ತರಬೇತಿ ಡೇಟಾದಿಂದ ಪಠ್ಯವನ್ನು ಸೃಷ್ಟಿಸುತ್ತದೆ. ಸದ್ಯ ಹವಾಮಾನವೇನು ಎಂದು ಕೇಳಿದರೆ, ಅದು ಊಹಿಸುತ್ತದೆ. ಸಂಚಾರಗಳನ್ನು ನೀಡಿದರೆ, ಅದು ಹವಾಮಾನ API ಅನ್ನು ಕರೆದು, ಗಣನೆಗಳನ್ನು ನಡೆಸಿ, ಅಥವಾ ಡೇಟಾಬೇಸ್ queries ಮಾಡಿ — ನಂತರ ಆ ನಿಜವಾದ ಫಲಿತಾಂಶಗಳನ್ನು ತನ್ನ ಪ್ರತಿಕ್ರಿಯೆಯಲ್ಲಿ ಹೊಳೆಸುತ್ತದೆ.

<img src="../../../translated_images/kn/what-are-tools.724e468fc4de64da.webp" alt="ಸಂಚಾರಗಳಿಲ್ಲದೇ ಮತ್ತು ಸಂಚಾರಗಳೊಂದಿಗೆ" width="800"/>

*ಸಂಚಾರಗಳಿಲ್ಲದೆ ಮಾದರಿ ಊಹಿಸಬಹುದು — ಸಂಚಾರಗಳೊಂದಿಗೆ ಇದು APIಗಳನ್ನು ಕರೆದು, ಗಣನೆಗಳನ್ನು ನಡೆಸಿ, ನೈಜ ಸಮಯದ ಡೇಟಾವನ್ನು ಹಿಂತಿರುಗಿಸುತ್ತದೆ.*

ಸಂಚಾರಗಳೊಂದಿಗೆ AI ಏಜೆಂಟ್ ಒಂದು **ತರ್ಕ ಮತ್ತು ಕಾರ್ಯ (ReAct)** ಮಾದರಿಯನ್ನು ಅನುಸರಿಸುತ್ತದೆ. ಮಾದರಿ ಕೇವಲ ಪ್ರತಿಕ್ರಿಯಿಸುವುದಲ್ಲ — ಅದು ಅವಶ್ಯಕತೆಯನ್ನು ಆಲೋಚಿಸಿ, ಸಂಚಾರವನ್ನು ಕರೆದು, ಫಲಿತಾಂಶವನ್ನು ಗಮನಿಸಿ, ನಂತರ ಮತ್ತಷ್ಟು ಕಾರ್ಯಚರಣೆ ಮಾಡಲು ಅಥವಾ ಅಂತಿಮ ಉತ್ತರ ವಿತರಿಸಲು ನಿರ್ಧರಿಸುತ್ತದೆ:

1. **ತರ್ಕಿಸಿ** — ಬಳಕೆದಾರರ ಪ್ರಶ್ನೆಯನ್ನು ವಿಶ್ಲೇಷಿಸಿ ಅವಶ್ಯಕ ಮಾಹಿತಿಯನ್ನು ನಿರ್ಧರಿಸುತ್ತದೆ
2. **ಕಾರ್ಯವನ್ನು ನಿರ್ವಹಿಸಿ** — ಸರಿಯಾದ ಸಂಚಾರ ಆಯ್ಕೆಮಾಡಿ, ಸರಿಯಾದ ಪ್ಯಾರಾಮೀಟರ್ಗಳು ಸೃಷ್ಟಿಸಿ ಕರೆ ಮಾಡುತ್ತದೆ
3. **ಗರ್ತಿಸಿ** — ಸಂಚಾರದ ಫಲಿತಾಂಶವನ್ನು ಸ್ವೀಕರಿಸಿ ಮೌಲ್ಯಮಾಪನ ಮಾಡುತ್ತದೆ
4. **ಮರುಮುಖವಾಗಿ ಅಥವಾ ಪ್ರತಿಕ್ರಿಯಿಸಿ** — ಇನ್ನಷ್ಟು ಡೇಟಾ ಬೇಕಾದರೆ ಹಿಂದಕ್ಕೆ ಹೋಗಿ; ಇಲ್ಲದಿದ್ದರೆ, ಸಹಜ ಭಾಷೆಯ ಉತ್ತರ ರಚಿಸುತ್ತದೆ

<img src="../../../translated_images/kn/react-pattern-detail.96a5efeeb6dd2f61.webp" alt="ReAct ವಿನ್ಯಾಸ" width="800"/>

*ReAct ಚಕ್ರ — ಏಜೆಂಟ್ ಏನು ಮಾಡಬೇಕೆಂದು ತರ್ಕಿಸಿ, ಸಂಚಾರವನ್ನು ಕರೆದು, ಫಲಿತಾಂಶವನ್ನು ಗರ್ತಿಸಿ, ಮತ್ತು ಅಂತಿಮ ಉತ್ತರ ನೀಡುವವರೆಗೆ ಲೂಪ್ ಮಾಡುತ್ತದೆ.*

ಇದು ಸ್ವಯಂಚಾಲಿತವಾಗಿಯೇ ನಡೆಯುತ್ತದೆ. ನೀವು ಸಂಚಾರಗಳನ್ನು ಮತ್ತು ಅವುಗಳ ವಿವರಣೆಗಳನ್ನು ನಿರ್ಧರಿಸುತ್ತೀರಿ. ಮಾದರಿ ಅವುಗಳನ್ನು ಯಾವಾಗ ಮತ್ತು ಹೇಗೆ ಉಪಯೋಗಿಸಬೇಕೆಂದು ತೀರ್ಮಾನಿಸುವುದನ್ನು ನಿರ್ವಹಿಸುತ್ತದೆ.

## ಸಂಚಾರ ಕರೆ ಹೇಗೆ ಕೆಲಸ ಮಾಡುತ್ತದೆ

### ಸಂಚಾರ ವ್ಯಾಖ್ಯಾನಗಳು

[WeatherTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) | [TemperatureTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/TemperatureTool.java)

ನೀವು ಸ್ಪಷ್ಟ ವಿವರಣೆಗಳು ಮತ್ತು ಪ್ಯಾರಾಮೀಟರ್ ನಿರ್ದಿಷ್ಟತೆಗಳೊಂದಿಗೆ ಕಾರ್ಯಗಳನ್ನು ವ್ಯಾಖ್ಯಾನಿಸುತ್ತೀರಿ. ಮಾದರಿ ಈ ವಿವರಣೆಗಳನ್ನು ತನ್ನ ವ್ಯವಸ್ಥಾ ಪ್ರಾಂಪ್ಟ್ ನಲ್ಲಿ ನೋಡಿ ಪ್ರತಿ ಸಂಚಾರ ಏನು ಮಾಡುತ್ತದೆ ಎಂದು ಅರ್ಥಮಾಡಿಕೊಳ್ಳುತ್ತದೆ.

```java
@Component
public class WeatherTool {
    
    @Tool("Get the current weather for a location")
    public String getCurrentWeather(@P("Location name") String location) {
        // ನಿಮ್ಮಹವಾಮಾನ ಹುಡುಕಿಕೊಳ್ಳುವ ತರ್ಕ
        return "Weather in " + location + ": 22°C, cloudy";
    }
}

@AiService
public interface Assistant {
    String chat(@MemoryId String sessionId, @UserMessage String message);
}

// ಸಹಾಯಕನು ಸ್ವಯಂಚಾಲಿತವಾಗಿ ಸ್ಪ್ರಿಂಗ್ ಬೂಟ್ ಮೂಲಕ ಸಂಪರ್ಕಿಸಲಾಗಿದೆ:
// - ಚಾಟ್‌ಮಾ಼ಡೆಲ್ ಬೀನ್
// - ಅಲ್ಲಿ @ಕಂಪೋನಂಟ್ ತರಗತಿಗಳಿಂದ ಎಲ್ಲಾ @ಟೂಲ್ ವಿಧಾನಗಳು
// - ಅಧಿವೇಶನ ನಿರ್ವಹಣೆಗೆ ಚಾಟ್‌ಮೆಮೊರಿ ಪ್ರೊವೈಡರ್
```

ಕೆಳಗಿನ ಚಿತ್ರವು ಪ್ರತಿ ಟಿಪ್ಪಣಿಯನ್ನು ವಿಭಜಿಸಿ ಹೇಗೆ ಪ್ರತಿ ಭಾಗ AIಗೆ ಸಂಚಾರವನ್ನು ಕರೆಸುವ ಸಮಯ ಮತ್ತು ಪ್ಯಾರಾಮೀಟರ್ಗಳನ್ನು ಪಾಸ್ ಮಾಡುವುದನ್ನು ಅರ್ಥಮಾಡಿಕೊಳ್ಳಲು ಸಹಾಯ ಮಾಡುತ್ತದೆ ಎಂಬುದನ್ನು ತೋರಿಸುತ್ತದೆ:

<img src="../../../translated_images/kn/tool-definitions-anatomy.f6468546037cf28b.webp" alt="ಸಂಚಾರ ವ್ಯಾಖ್ಯಾನಗಳ ಅಟಾನಮಿ" width="800"/>

*ಸಂಚಾರ ವ್ಯಾಖ್ಯಾನದ ಅಟಾನಮಿ — @Tool ಸಂಚಾರವನ್ನು ಯಾವಾಗ ಬಳಸಬೇಕು ಎಂದು AIಗೆ ತಿಳಿಸುತ್ತದೆ, @P ಪ್ರತಿಯೊಂದು ಪ್ಯಾರಾಮೀಟರ್ ವಿವರಿಸುತ್ತದೆ, ಮತ್ತು @AiService ಪ್ರಾರಂಭದಲ್ಲಿ ಎಲ್ಲವನ್ನೂ ಕನೆಕ್ಟ್ ಮಾಡುತ್ತದೆ.*

> - "ಸಂಚಾರ ಅನುಷ್ಠಾನಗಳಲ್ಲಿ API ದೋಷಗಳು ಮತ್ತು ದರ ಮಿತಿ ಹೇಗೆ ನಿರ್ವಹಿಸಬೇಕು?"

### ತೀರ್ಮಾನ ಮಾಡಿಕೊಳ್ಳುವಿಕೆ

ಬಳಕೆದಾರರು "ಸಿಯಾಟಲ್ ನಲ್ಲಿ ಹವಾಮಾನ ಏನು?" ಎಂದು ಕೇಳಿದಾಗ, ಮಾದರಿ ಯಾದೃಚ್ಛಿಕವಾಗಿ ಸಂಚಾರ ಆಯ್ಕೆ ಮಾಡದು. ಅದು ಬಳಕೆದಾರರ ಉದ್ದೇಶವನ್ನು ಪ್ರತಿ ಸಂಚಾರ ವಿವರಣೆಗೂ ಹೋಲಿಸಿ, ಸಂಬಂಧಿತತೆಗಾಗಿ ಅಂಕೆ ಹಂಚಿಕೆ ಮಾಡುತ್ತದೆ; ಮತ್ತು ಅತ್ಯುತ್ತಮ ಹೊಂದಿಕೆಯನ್ನು ಆಯ್ಕೆಮಾಡುತ್ತದೆ. ನಂತರ ಸರಿಯಾದ ಪ್ಯಾರಾಮೀಟರ್‌ಗಳೊಂದಿಗೆ ಸಂಗ್ರಹಿತ ಕಾರ್ಯ ಕರೆಗೆ ನಿರ್ಮಾಣ ಮಾಡುತ್ತದೆ — ಈ ಉದಾಹರಣೆಯಲ್ಲಿ, `location` ಅನ್ನು `"Seattle"`ಗೆ ಹೊಂದಿಸಿದೆ.

ಯಾವುದೇ ಸಂಚಾರ ಬಳಕೆದಾರದ ವಿನಂತಿಗೆ ಹೊಂದಿಕೆಯಾಗದಿದ್ದರೆ, ಮಾದರಿ ತನ್ನ ಸ್ವಂತ ಜ್ಞಾನದಿಂದ ಉತ್ತರ ನೀಡಲು ಹಿಂದಕ್ಕೆ ಹೋಗುತ್ತದೆ. ಒಬ್ಬಕ್ಕಿಂತ ಹೆಚ್ಚು ಸಂಚಾರಗಳು ಹೊಂದಿದ್ದರೆ, ಅತ್ಯಂತ ವಿಶಿಷ್ಟವನ್ನು ಆರಿಸುತ್ತದೆ.

<img src="../../../translated_images/kn/decision-making.409cd562e5cecc49.webp" alt="AI ಎಷ್ಟು ಸಂಚಾರವನ್ನು ಆಯ್ಕೆ ಮಾಡುತ್ತದೆ" width="800"/>

*ಮಾದರಿ ಲಭ್ಯವಿರುವ ಪ್ರತಿಯೊಂದು ಸಂಚಾರವನ್ನು ಬಳಕೆದಾರದ ಉದ್ದೇಶದ ವಿರುದ್ಧ ಮೌಲ್ಯಮಾಪನ ಮಾಡುತ್ತದೆ ಮತ್ತು ಅತ್ಯುತ್ತಮ ಹೊಂದಿಕೆಯನ್ನು ಆಯ್ಕೆಮಾಡುತ್ತದೆ — ಅದಕ್ಕಾಗಿ ಸ್ಪಷ್ಟ ಮತ್ತು ನಿರ್ದಿಷ್ಟ ಸಂಚಾರ ವಿವರಣೆಗಳನ್ನು ಬರೆಯುವುದು ಮುಖ್ಯ.*

### ಕಾರ್ಯಗತಗೊಳಿಸುವಿಕೆ

[AgentService.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/service/AgentService.java)

ಸ್ಪ್ರಿಂಗ್ ಬುಟ್ ವಿಶೇಷ @AiService ಸ್ಥೂಲಾಕಾರವನ್ನು ನೋಂದಾಯಿಸಿದ ಎಲ್ಲಾ ಸಂಚಾರಗಳೊಂದಿಗೆ ಆಟೋ-ವೈರಿಂಗ್ ಮಾಡುತ್ತದೆ ಮತ್ತು LangChain4j ಸ್ವಯಂಚಾಲಿತವಾಗಿ ಸಂಚಾರ ಕರೆಗೆ ಕಾರ್ಯಗತಗೊಳಿಸುತ್ತದೆ. ಪರದೆ ಹರ್ಥಿಯಲ್ಲಿ, ಪೂರ್ಣ ಸಂಚಾರ ಕರೆ ಆರು ಹಂತಗಳಿಂದ ಹರಿಯುತ್ತದೆ — ಬಳಕೆದಾರರ ಸಹಜ ಭಾಷೆಯ ಪ್ರಶ್ನೆಯಿಂದ ನೈಸರ್ಗಿಕ ಭಾಷೆ ಉತ್ತರದವರೆಗೆ:

<img src="../../../translated_images/kn/tool-calling-flow.8601941b0ca041e6.webp" alt="ಸಂಚಾರ ಕರೆ ಹರಿವು" width="800"/>

*ಎಂಡ್-ಟು-ಎಂಡ್ ಹರಿವು — ಬಳಕೆದಾರ ಪ್ರಶ್ನೆ ಕೇಳುತ್ತದೆ, ಮಾದರಿ ಸಂಚಾರ ಆಯ್ಕೆ ಮಾಡುತ್ತದೆ, LangChain4j ಅದರ ಕಾರ್ಯಾಚರಣೆ ಮಾಡುತ್ತದೆ, ಮತ್ತು ಫಲಿತಾಂಶವನ್ನು ಸಹಜ ಪ್ರತಿಕ್ರಿಯೆಗೆ ಒಡನಾಡಿಸುತ್ತದೆ.*

> - "ಏಜೆಂಟ್ ಯಾವ ಸಂಚಾರವನ್ನು ಯಾವ ಆದಿಯಲ್ಲಿ ಉಪಯೋಗಿಸಬೇಕೆಂದು ಹೇಗೆ ತೀರ್ಮಾನಿಸುತ್ತದೆ?"
> - "ಸಂಚಾರ ಕಾರ್ಯರಾಹಿತ್ಯ ಸಂಭವಿಸಿದರೆ ಏನು ಆಗುತ್ತದೆ - ದೋಷಗಳನ್ನು ಬಲಿಷ್ಠವಾಗಿ ಹೇಗೆ ನಿರ್ವಹಿಸಬೇಕು?"

### ಪ್ರತಿಕ್ರಿಯೆ ಸೃಷ್ಟಿ

ಮಾದರಿ ಹವಾಮಾನ ಡೇಟಾವನ್ನು ಪಡೆದು ಅದನ್ನು ಬಳಕೆದಾರರಿಗಾಗಿ ಸಹಜ ಭಾಷೆಯ ಪ್ರತಿಕ್ರಿಯೆಯಲ್ಲಿ ತೆರೆಯುತ್ತದೆ.

### ವಾಸ್ತುಶಿಲ್ಪ: ಸ್ಪ್ರಿಂಗ್ ಬುಟ್ ಆಟೋ-ವೈರಿಂಗ್

ಈ ಮODULE LangChain4j ನ ಸ್ಪ್ರಿಂಗ್ ಬುಟ್ ಏಕೀಕರಣವನ್ನು ನಿಯೋಜಿಸಿದೆ, ಘೋಷಣಾತ್ಮಕ @AiService ಸ್ಥೂಲಾಕಾರಗಳೊಂದಿಗೆ. ಪ್ರಾರಂಭದಲ್ಲಿ, ಸ್ಪ್ರಿಂಗ್ ಬುಟ್ ಪ್ರತಿ @Tool ವಿಧಾನ ಹೊಂದಿದ @Component, ನಿಮ್ಮ ChatModel ಬೀನ್ ಮತ್ತು ChatMemoryProvider ಅನ್ನು ಕಂಡು ಹಿಡಿದು — ಅವುಗಳನ್ನು ಎಲ್ಲವನ್ನೂ ಶೂನ್ಯ ಬಾಯ್ಲರ್ ಪ್ಲೇಟ್ ಸಹಿತ ಒಬ್ಬೊಬ್ಬ ಅಸಿಸ್ಟೆಂಟ್ ಇಂಟರ್ಫೇಸ್ ಗೆ ವೈರ್ ಮಾಡುತ್ತದೆ.

<img src="../../../translated_images/kn/spring-boot-wiring.151321795988b04e.webp" alt="ಸ್ಪ್ರಿಂಗ್ ಬುಟ್ ಆಟೋ-ವೈರಿಂಗ್ ವಾಸ್ತುಶಿಲ್ಪ" width="800"/>

*@AiService ಇಂಟರ್ಫೇಸ್ ChatModel, ಸಂಚಾರ ಘಟಕಗಳು ಮತ್ತು ಸ್ಮೃತಿ ಒದಗಿಸುವವರನ್ನುೊಡನೆ ಜೋಡಿಸುತ್ತದೆ — ಸ್ಪ್ರಿಂಗ್ ಬುಟ್ ಎಲ್ಲಾ ವೈರಿಂಗ್ ಸ್ವಯಂಚಾಲಿತವಾಗಿ ನಿರ್ವಹಿಸುತ್ತದೆ.*

ಈ ವಿಧಾನದ ಪ್ರಮುಖ ಲಾಭಗಳು:

- **ಸ್ಪ್ರಿಂಗ್ ಬುಟ್ ಆಟೋ-ವೈರಿಂಗ್** — ChatModel ಮತ್ತು ಸಂಚಾರಗಳು ಸ್ವಯಂಚಾಲಿತವಾಗಿ ಇಂಜೆಕ್ಟ್ ಆಗುತ್ತವೆ
- **@MemoryId ವಿನ್ಯಾಸ** — ಸ್ವಯಂಚಾಲಿತ ಸೆಷನ್ ಆಧಾರಿತ ಸ್ಮೃತಿ ನಿರ್ವಹಣೆ
- **ಏಕೈಕ ಇನ್ಸ್ಟೆನ್ಸ್** — ಅಸಿಸ್ಟೆಂಟ್ ಒಮ್ಮೆ ರಚಿಸಿ ಉತ್ತಮ ಕಾರ್ಯಕ್ಷಮತೆಗೆ ಮರುಬಳಕೆ
- **ಟೈಪ್-ಸೆಫ್ ಕಾರ್ಯಾಚರಣೆ** — ಜವಾ ವಿಧಾನಗಳು ನೇರವಾಗಿ ಟೈಪ್ ರೂಪಾಂತರದೊಂದಿಗೆ ಕರೆ
- **ಬಹು-ತಿರುಗು ನಿರ್ವಹಣೆ** — ಸ್ವಯಂಚಾಲಿತ ಸಂಚಾರ ಸರಪಳಿ ನಿರ್ವಹಣೆ
- **ಶೂನ್ಯ ಬಾಯ್ಲರ್ ಪ್ಲೇಟ್** — ಕೈಯಲ್ಲಿ AiServices.builder() ಕರೆ ಅಥವಾ ಸ್ಮೃತಿ HashMap ಇಲ್ಲ

ವೈಕಲ್ಪಿಕ ವಿಧಾನಗಳು (ಮಾನ್ಯುವಲ್ AiServices.builder()) ಹೆಚ್ಚು ಕೋಡ್ ಗರನ್ನು ಆಗಿಸುತ್ತದೆ ಮತ್ತು ಸ್ಪ್ರಿಂಗ್ ಬುಟ್ ಏಕೀಕರಣದ ಲಾಭವನ್ನು ಕಳೆದುಹೋಗುತ್ತದೆ.

## ಸಂಚಾರ ಸರಪಳಿ

**ಸಂಚಾರ ಸರಪಳಿ** — ಸಂಚಾರ ಆಧಾರಿತ ಏಜೆಂಟ್‍ಗಳ ನಿಜವಾದ ಶಕ್ತಿ ಆಗುವುದು, ಒಂದೇ ಪ್ರಶ್ನೆಗೆ ಬಹು ಸಂಚಾರಗಳು ಬೇಕಾದಾಗ. "ಸಿಯಾಟಲ್ ನಲ್ಲಿ ಫಾರೆನ್‌ಹೀಟ್ ನಲ್ಲಿ ಹವಾಮಾನ ಏನು?" ಎಂದು ಕೇಳಿದರೆ, ಏಜೆಂಟ್ ಸ್ವಯಂಚಾಲಿತವಾಗಿ ಎರಡು ಸಂಚಾರಗಳನ್ನು ಸರಪಳಿ ಮಾಡಿ: ಮೊದಲು `getCurrentWeather` ಅನ್ನು ಕರೆದು ಸೆಲ್ಸಿಯಸ್‌ನಲ್ಲಿ ತಾಪಮಾನ ಪಡೆಯುತ್ತದೆ ಆಮೇಲೆ ಆ ಮೌಲ್ಯವನ್ನು `celsiusToFahrenheit` ಗೆ ಪರಿವರ್ತನೆಗೊಳಿಸಲು ಪಾಸ್ ಮಾಡುತ್ತದೆ — ಎಲ್ಲಾ ಒಂದೇ ಸಂಭಾಷಣೆ ತಿರುಗಿನಲ್ಲಿ.

<img src="../../../translated_images/kn/tool-chaining-example.538203e73d09dd82.webp" alt="ಸಂಚಾರ ಸರಪಳಿಯ ಉದಾಹರಣೆ" width="800"/>

*ಸಂಚಾರ ಸರಪಳಿ ಕ್ರಿಯೆಯಲ್ಲಿ — ಏಜೆಂಟ್ ಮೊದಲು getCurrentWeather ಅನ್ನು ಕರೆದು, ಶಿಕ್ಷಣ ಫಲಿತಾಂಶವನ್ನು celsiusToFahrenheit ಗೆ ಪೈಪ್ ಮಾಡಿ, ಮತ್ತು ಸಂಯುಕ್ತ ಉತ್ತರವನ್ನು ನೀಡುತ್ತದೆ.*

ಇದು ಚಾಲಿತ ಅಪ್ಲಿಕೇಶನಿನಲ್ಲಿ ಹೀಗೆ ಕಾಣುತ್ತದೆ — ಏಜೆಂಟ್ ಒಂದೇ ಸಂಭಾಷಣೆ ತಿರುಗಿನಲ್ಲಿ ಎರಡು ಸಂಚಾರ ಕರೆಗಳನ್ನು ಸರಪಳಿಸುತ್ತಾನೆ:

<a href="images/tool-chaining.png"><img src="../../../translated_images/kn/tool-chaining.3b25af01967d6f7b.webp" alt="ಸಂಚಾರ ಸರಪಳಿ" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*ನಿಜವಾದ ಅಪ್ಲಿಕೇಶನ್ ಔಟ್‌ಪುಟ್ — ಏಜೆಂಟ್ ಸ್ವಯಂಚಾಲಿತವಾಗಿ getCurrentWeather → celsiusToFahrenheit ಸರಪಳಿಯನ್ನು ಒಂದೇ ತಿರುಗಿನಲ್ಲಿ ಮಾಡುತ್ತದೆ.*

**ಸೌಜನ್ಯ ದೋಷಗಳು** — ಮಾರುಕಟ್ಟೆ ಡೇಟಾದಲ್ಲಿ ಇಲ್ಲದ ನಗರದಲ್ಲಿ ಹವಾಮಾನವನ್ನು ಕೇಳಿ. ಸಂಚಾರವು ದೋಷ ಸಂದೇಶವನ್ನು ಹಿಂತಿರುಗಿಸುತ್ತದೆ, ಮತ್ತು AI ಸಹಾಯ ಮಾಡಲಾರದೆಂದು ವಿವರಿಸುತ್ತದೆ ಬದಲು ಕ್ರ್ಯಾಶ್ ಆಗುವುದಿಲ್ಲ. ಸಂಚಾರಗಳು ಸುರಕ್ಷಿತವಾಗಿ ವೈಫಲ್ಯಗೊಳಿಸುತ್ತವೆ.

<img src="../../../translated_images/kn/error-handling-flow.9a330ffc8ee0475c.webp" alt="ದೋಷ ನಿರ್ವಹಣೆ ಹರಿವು" width="800"/>

*ಸಂಚಾರ ವೈಫಲ್ಯ ಆಗುವಾಗ, ಏಜೆಂಟ್ ದೋಷವನ್ನು ಹಿಡಿದು, ಸಹಾಯಕ ವಿವರಣೆಯೊಂದಿಗೆ ಪ್ರತಿಕ್ರಿಯಿಸುತ್ತದೆ ಕ್ರ್ಯಾಶ್ ಆಗುವುದನ್ನು ಬದಲು.*

ಇದು ಒಂದೇ ಸಂಭಾಷಣೆ ತಿರುಗಿನಲ್ಲಿ ನಡೆಯುತ್ತದೆ. ಏಜೆಂಟ್ ಸ್ವಯಂಚಾಲಿತವಾಗಿ ಬಹು ಸಂಚಾರಗಳ ಕರೆಗಳನ್ನು ನಿರ್ವಹಿಸುತ್ತದೆ.

ಡಿವ್ ಕನ್ಟೈನರ್ ಸ್ಪ್ರಿಂಗ್ ಬುಟ್ ಡ್ಯಾಶ್‌ಬೋರ್ಡ್ ವಿಸ್ತರಣೆ ಹೊಂದಿದೆ, ಇದು ಎಲ್ಲಾ ಸ್ಪ್ರಿಂಗ್ ಬುಟ್ ಅಪ್ಲಿಕೇಶನ್‌ಗಳನ್ನು ನಿಭಾಯಿಸಲು ದೃಶ್ಯಮಾನದ ಮೊದಲು ಒದಗಿಸುತ್ತದೆ. ನೀವು ಅದನ್ನು VS Code ಬಲಭಾಗದ ಆಕ್ಟಿವಿಟಿ ಬಾರ್‌ನಲ್ಲಿ (ಸ್ಪ್ರಿಂಗ್ ಬುಟ್ ಐಕಾನ್ ನೋಡಿ) ಹುಡುಕಬಹುದು.

ಸ್ಪ್ರಿಂಗ್ ಬುಟ್ ಡ್ಯಾಶ್‌ಬೋರ್ಡ್‌ನಿಂದ ನೀವು:

- ಕ್ಲಿಕ್ ಒಮ್ಮೆ ಮೂಲಕ ಅಪ್ಲಿಕೇಶನ್ ಪ್ರಾರಂಭ / ನಿಲ್ಲಿಸಬಹುದು
- ಅಪ್ಲಿಕೇಶನ್ ಲಾಗ್‌ಗಳನ್ನು ನೇರ ಸಮಯದಲ್ಲಿ ವೀಕ್ಷಿಸಬಹುದು

## ಅಪ್ಲಿಕೇಶನ್ ಬಳಸುವುದು

ಅಪ್ಲಿಕೇಶನ್ ಹವಾಮಾನ ಮತ್ತು ತಾಪಮಾನ ಪರಿವರ್ತನೆ ಸಂಚಾರಗಳಿಗೆ ಪ್ರವೇಶ ಹೊಂದಿರುವ AI ಏಜೆಂಟ್ ಜೊತೆಗೆ ಸಂಭಾಷಣೆ ನಡೆಸಲು ವೆಬ್ ಅಂತರ್ಮುಖ್ಯವನ್ನು ಒದಗಿಸುತ್ತದೆ.

<a href="images/tools-homepage.png"><img src="../../../translated_images/kn/tools-homepage.4b4cd8b2717f9621.webp" alt="AI ಏಜೆಂಟ್ ಸಂಚಾರಗಳ ಇಂಟರ್ಫೇಸ್" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*AI ಏಜೆಂಟ್ ಸಂಚಾರಗಳ ಇಂಟರ್ಫೇಸ್ - ಸಮೀಪದ ಉದಾಹರಣೆಗಳು ಮತ್ತು ಸಂಚಾರಗಳೊಂದಿಗೆ ಸಂವಹನಕ್ಕಾಗಿ ಚಾಟ್ ಇಂಟರ್ಫೇಸ್*

### ಸರಳ ಸಂಚಾರ ಬಳಕೆ ಪ್ರಯತ್ನಿಸಿ
ಸರಳ ವಿನಂತಿಯಿಂದ ಪ್ರಾರಂಭಿಸಿ: "100 ಡಿಗ್ರಿ ಫಾರೆನ್‌ಹೈಟ್ ಅನ್ನು ಸೆಲ್ಸಿಯಸ್‌ಗೆ ಪರಿವರ್ಥಿಸಿ". ಏಜೆಂಟ್ ಅದಕ್ಕೆ ತಾಪಮಾನ ಪರಿವರ್ತನೆ ಸಾಧನ ಬೇಕಾಗಿರುವುದು ಗುರುತಿಸಿ, ಸರಿಯಾದ ಪರಿಮಾಣಗಳೊಂದಿಗೆ ಅದನ್ನು ಕರೆಸುತ್ತದೆ ಮತ್ತು ಫಲಿತಾಂಶವನ್ನು ನೀಡುತ್ತದೆ. ಇದು ಹೇಗೆ ಸ್ವಾಭಾವಿಕವಾಗಿದೆಯೋ ಗಮನಿಸಿ - ನೀವು ಯಾವ ಸಾಧನವನ್ನು ಬಳಸಬೇಕು ಅಥವಾ ಅದನ್ನು ಹೇಗೆ ಕರೆಸಬೇಕು ಎಂಬುದನ್ನು ಸೂಚಿಸಲಿಲ್ಲ.

### ಸಾಧನ ಕಡ್ಡಾಯ ಸಪೂರಣಾ ಪರೀಕ್ಷೆ

ಈಗ ಸ್ವಲ್ಪ ಕಠಿಣದೊಂದು ಪ್ರಯತ್ನಿಸಿ: "ಸಿಯಾಟಲ್ ನಲ್ಲಿ ಹವಾಮಾನ ಯಾವುದು ಮತ್ತು ಅದನ್ನು ಫಾರೆನ್‌ಹೈಟ್ ಗೆ ಪರಿವರ್ತಿಸಿ?" ಈ ಪ್ರಕ್ರಿಯೆಯಲ್ಲಿ ಏಜೆಂಟ್ ಹಂತ ಹಂತವಾಗಿ ಕೆಲಸ ಮಾಡುವುದನ್ನು ಗಮನಿಸಿ. ಮತ್ತೆ ಮೊದಲಿಗೆ ಹವಾಮಾನ ಪಡೆಯುತ್ತದೆ (ಅದು ಸೆಲ್ಸಿಯಸ್ ನಲ್ಲಿ ನೀಡುತ್ತದೆ), ಅದನ್ನು ಫಾರೆನ್‌ಹೈಟ್ ಗೆ ಪರಿವರ್ತಿಸಲು ಬೇಕಾಗಿರುವುದನ್ನು悟ತ್ತದೆ, ಪರಿವರ್ತನೆ ಸಾಧನವನ್ನು ಕರೆಸುತ್ತದೆ, ಮತ್ತು ಎರಡೂ ಫಲಿತಾಂಶಗಳನ್ನು ಏಕರೂಪದಲ್ಲಿ ಒದಗಿಸುತ್ತದೆ.

### ಸಂವಾದ ಪ್ರವಾಹವನ್ನು ನೋಡಿ

ಚಾಟ್ ಇಂಟರ್ಫೇಸ್ ಸಂವಾದದ ಇತಿಹಾಸವನ್ನು ಇಟ್ಟುಕೊಳ್ಳುತ್ತದೆ, ನಿಮಗೆ ಬಹು-ತಿರುವು ಆಂತರಿಕ್ಷ ಚರ್ಚೆಗಳು ಮಾಡಲು ಅನುಮತಿಸುತ್ತದೆ. ನೀವು ಮುಂಚಿನ ಎಲ್ಲಾ ಪ್ರಶ್ನೆಗಳ ಮತ್ತು ಪ್ರತಿಕ್ರಿಯೆಗಳನ್ನೂ ನೋಡಬಹುದು, ಇದರಿಂದ ಸಂವಾದವನ್ನು ಸುಲಭವಾಗಿ ಹತ್ತಿರದಿಂದ ಗಮನಿಸಬಹುದು ಮತ್ತು ಏಜೆಂಟ್ ಹೇಗೆ ಸನ್ನಿವೇಶವನ್ನು ಕಟ್ಟಿಕೊಳ್ಳುತ್ತದೆಯೋ ಅರ್ಥಮಾಡಿಕೊಳ್ಳಬಹುದು.

<a href="images/tools-conversation-demo.png"><img src="../../../translated_images/kn/tools-conversation-demo.89f2ce9676080f59.webp" alt="ಬಹು ಸಾಧನ ಕರೆಗಳೊಂದಿಗೆ ಸಂವಾದ" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*ಸರಳ ಪರಿವರ್ತನೆಗಳು, ಹವಾಮಾನ ಹುಡುಕುಗಳು, ಮತ್ತು ಸಾಧನ ಕಡ್ಡಾಯ ಸಪೂರಣಗಳನ್ನು ತೋರಿಸುತ್ತಿರುವ ಬಹು-ತಿರುವು ಸಂವಾದ*

### ವಿಭಿನ್ನ ವಿನಂತಿಗಳೊಂದಿಗೆ ಪ್ರಯೋಗ ಮಾಡಿ

ವಿವಿಧ ಸಂಯೋಜನೆಗಳನ್ನು ಪ್ರಯತ್ನಿಸಿ:
- ಹವಾಮಾನ ಹುಡುಕುಗಳು: "ಟೋಕಿಯೋದಲ್ಲಿ ಹವಾಮಾನ ಯಾವುದು?"
- ತಾಪಮಾನ ಪರಿವರ್ತನೆಗಳು: "25°C ಅನ್ನು ಕೆಲ್ವಿನ್‌ನಲ್ಲಿ ಎಷ್ಟು?"
- ಸಂಯೋಜಿತ ಪ್ರಶ್ನೆಗಳು: "ಪ್ಯಾರಿಸ್ ನಲ್ಲಿ ಹವಾಮಾನ ಪರಿಶೀಲಿಸಿ ಮತ್ತು ಅದು 20°C ಉಷ್ಣಕ್ಕಿಂತ ಮೇಲ್ವ್ಗಿದೆಯೇ ತಿಳಿಸಿ"

ನೀತಿ ಏಜೆಂಟ್ प्राकृतिक ಭಾಷೆಯನ್ನು ಹೇಗೆ ಅರ್ಥಮಾಡಿಕೊಳ್ಳುತ್ತದೆ ಮತ್ತು ಸೂಕ್ತ ಸಾಧನ ಕಾಲ್‌ಗಳಿಗೆ ನಕ್ಷೆ ಮಾಡುತ್ತದೆ ಎಂದು ಗಮನಿಸಿ.

## ಪ್ರಮುಖ ತತ್ವಗಳು

### ReAct ಮಾದರಿ (ಚಿಂತನೆ ಮತ್ತು ಕಾರ್ಯಾಚರಣೆ)

ಏಜೆಂಟ್ ವಿಚಾರಿಸುವಿಕೆ (ಮಾಡಬೇಕೆಂದಿದ್ದುದನ್ನು ತೀರ್ಮಾನಿಸುವುದು) ಮತ್ತು ಕಾರ್ಯಾಚರಣೆ (ಸಾಧನಗಳನ್ನು ಉಪಯೋಗಿಸುವುದು) ನಡುವೆ ಬದಲಾವಣೆ ಆಗುತ್ತವೆ. ಈ ಮಾದರಿ ಸ್ವಯಂಪ್ರೇರಿತ ಸಮಸ್ಯೆ ಪರಿಹಾರಕ್ಕೆ ಅನುಕೂಲ ಮಾಡಿಕೊಡುವುದು, ಕೇವಲ ನಿರ್ದೇಶನಗಳಿಗೆ ಪ್ರತಿಕ್ರಿಯಿಸುವುದಕ್ಕೆ ಮಾತ್ರವಲ್ಲ.

### ಸಾಧನ ವಿವರಣೆಗಳು ಮಹತ್ವವುಳ್ಳವು

ನಿಮ್ಮ ಸಾಧನ ವಿವರಣೆಗಳ ಗುಣಮಟ್ಟವೇ ಏಜೆಂಟ್ ಅವುಗಳನ್ನು ಹೇಗೆ ಬಳಸುತ್ತದೆಯೋ ನೇರವಾಗಿ ಪ್ರಭಾವಿಸುತ್ತದೆ. ಸ್ಪಷ್ಟ, ವಿಶೇಷ ವಿವರಣೆಗಳು ಮಾದರಿಯನ್ನು ಯಾವಾಗ ಮತ್ತು ಹೇಗೆ ಸಾಧನವನ್ನು ಕರೆಸಬೇಕು ಎಂದು ಚೆನ್ನಾಗಿ ಅರ್ಥಮಾಡಿಕೊಳ್ಳಲು ಸಹಾಯಮಾಡುತ್ತವೆ.

### ಸೆಷನ್ ನಿರ್ವಹಣೆ

`@MemoryId` ಟ್ಯಾಗ್ ಸ್ವಯಂಚಾಲಿತ ಸೆಷನ್ ಆಧಾರಿತ ಸ್ಮೃತಿ ನಿರ್ವಹಣೆಗೆ ಅವಕಾಶ ನೀಡುತ್ತದೆ. ಪ್ರತಿ ಸೆಷನ್ ಐಡಿಗೆ ತನ್ನದೇ ಆದ `ChatMemory` ಇನ್ಸ್ಟಾನ್ಸ್ ಯೂನಿಕ್ ಆಗಿ `ChatMemoryProvider` ಬೀನ್ ಮೂಲಕ ನಿರ್ವಹಿಸಲಾಗುತ್ತದೆ, ಹೀಗೆ ಬಹುಪ್ಯಾರ್ಮುಲ್ ಬಳಕೆದಾರರು ಏಜೆಂಟ್ ಜೊತೆಗೆ ಕೂಡಲೇ ಸಂವಾದ ಮಾಡಬಹುದು ಮತ್ತು ಅವರ ಸಂವಾದಗಳು ಮಿಶ್ರಣವಾಗುವುದಿಲ್ಲ.

*ಪ್ರತಿ ಸೆಷನ್ ಐಡಿ ಒಂದೇ ಸಂವಾದ ಇತಿಹಾಸವನ್ನು ಹೊಂದಿದೆ — ಬಳಕೆದಾರರು ಪರಸ್ಪರ ಸಂದೇಶಗಳನ್ನು ಕ見るದಲ್ಲ.*

### ದೋಷ ನಿರ್ವಹಣೆ

ಸಾಧನಗಳು ವಿಫಲವಾಗಬಹುದು — API ಗಳು ಟೈಮೌಟ್ ಆಗಬಹುದು, ಪರಿಮಾಣಗಳು ಅಮಾನ್ಯವಾಗಬಹುದು, ಹೊರಗಿನ ಸೇವೆಗಳು ಜಾರಿಮಾಡಬಹುದು. ಉತ್ಪಾದನಾ ಏಜೆಂಟ್‌ಗಳಿಗೆ ದೋಷ ನಿರ್ವಹಣೆ ಅವಶ್ಯಕ, ಹೀಗೆ ಮಾದರಿ ಸಮಸ್ಯೆಗಳನ್ನು ಸಹಜ ಭಾಷೆಯಲ್ಲಿ ವಿವರಿಸಬಹುದು ಅಥವಾ ಪರ್ಯಾಯಗಳನ್ನು ಪ್ರಯತ್ನಿಸಬಹುದು ಮತ್ತು ಒಂದೇ ಬಾರಿ ಅಪ್ಲಿಕೇಶನ್ ಸಕ್ರ್ಯತೆಯನ್ನು ನಾಶಗೊಳಿಸುವುದಿಲ್ಲ. ಸಾಧನಗಳು ಅಷ್ಟೆ ಅದೃಷ್ಟವಶಾತ್ ದೋಷವನ್ನು ಉಂಟುಮಾಡಿದಾಗ, LangChain4j ಅದನ್ನು ಹಿಡಿದುಕೊಳ್ಳುತ್ತದೆ ಮತ್ತು ದೋಷ ಸಂದೇಶವನ್ನು ಮಾದರಿಗೆ ಹಿಂತಿರುಗಿಸುತ್ತದೆ, ಹಾಗಾಗಿ ಮಾದರಿ ಅದನ್ನು ಸಹಜ ಭಾಷೆಯಲ್ಲಿ ವಿವರಿಸಬಹುದು.

## ಲಭ್ಯವಿರುವ ಸಾಧನಗಳು

ಕೆಳಗಿನ ಚಿತ್ರಿನಲ್ಲಿ ನೀವು ನಿರ್ಮಿಸಬಹುದಾದ ವ್ಯಾಪಕ ಸಾಧನ ಪರಿಸರವನ್ನು ತೋರಿಸುತ್ತದೆ. ಈ ಘಟಕ ಹವಾಮಾನ ಮತ್ತು ತಾಪಮಾನ ಸಾಧನಗಳನ್ನು ತೋರಿಸುತ್ತದೆ, ಆದರೆ ಅದೇ `@Tool` ಮಾದರಿ ಯಾವುದೇ ಜಾವಾ ವಿಧಾನಕ್ಕೆ ಅನ್ವಯಿಸುತ್ತದೆ — ಡೇಟಾಬೇಸ್ ಪ್ರಶ್ನೆಗಳಿಂದ ಪಾವತಿ ಪ್ರಕ್ರಿಯೆಗೆ.

<img src="../../../translated_images/kn/tool-ecosystem.aad3d74eaa14a44f.webp" alt="ಸಾಧನ ಪರಿಸರ" width="800"/>

*ಯಾವುದೇ `@Tool` ಟ್ಯಾಗ್ ಹೊಂದಿರುವ ಜಾವಾ ವಿಧಾನ AI ಗೆ ಲಭ್ಯವಾಗುತ್ತದೆ — ಈ ಮಾದರಿ ಡೇಟಾಬೇಸ್‌ಗಳು, API ಗಳು, ಇಮೇಲ್, ಕಡತ ಕಾರ್ಯಾಚರಣೆಗಳು ಇತ್ಯಾದಿ ವರೆಗೆ ವಿಸ್ತಾರಗೊಳ್ಳುತ್ತದೆ.*

## ಸಾಧನ ಆಧಾರಿತ ಏಜೆಂಟ್‌ಗಳನ್ನು ಯಾವಾಗ ಬಳಸುವುದು

<img src="../../../translated_images/kn/when-to-use-tools.51d1592d9cbdae9c.webp" alt="ಸಾಧನಗಳನ್ನು ಯಾವಾಗ ಬಳಸಬೇಕು" width="800"/>

*ಸಣ್ಣ ತೀರ್ಮಾನ ಮಾರ್ಗದರ್ಶಿ — ಸಾಧನಗಳು ನೈಜ ಸಮಯದ ಡೇಟಾ, ಗಣನೆಗಳು, ಮತ್ತು ಕಾರ್ಯಗಳಿಗಾಗಿ; ಸಾಮಾನ್ಯ ಜ್ಞಾನ ಮತ್ತು ಸೃಜನಾತ್ಮಕ ಕಾರ್ಯಗಳಿಗೆ ಅವಶ್ಯಕವಿಲ್ಲ.*

**ಸಾಧನಗಳನ್ನು ಬಳಸಿ, ಯಾವಾಗ:**
- ಉತ್ತರಿಸಲು ನೈಜ ಸಮಯದ ಡೇಟಾ (ಹವಾಮಾನ, ಸ್ಟಾಕ್ ಬೆಲೆಗಳು, ಇನ್‌ವೆಂಟರಿ) ಬೇಕಾಗಿದ್ದರೆ
- ಸರಳ ಗಣಿತದ ಹೊರತಾಗಿ ಗಣನೆಗಳನ್ನು ಮಾಡಬೇಕಾದರೆ
- ಡೇಟಾಬೇಸ್‌ಗಳು ಅಥವಾ API ಗಳ ಪ್ರವೇಶ ಬೇಕಾದರೆ
- ಕಾರ್ಯಗಳನ್ನು ಕೈಗೊಳ್ಳುವಾಗ (ಇಮೇಲ್ ಕಳುಹಿಸುವುದು, ಟಿಕೆಟ್ ರಚಿಸುವುದು, ದಾಖಲೆಗಳನ್ನು ನವೀಕರಿಸುವುದು)
- ಹಲವಾರು ಡೇಟಾ ಮೂಲಗಳನ್ನು ಸಂಯೋಜಿಸುವಾಗ

**ಸಾಧನಗಳನ್ನು ಬಳಸಬಾರದು, ಯಾವಾಗ:**
- ಪ್ರಶ್ನೆಗಳು ಸಾಮಾನ್ಯ ಜ್ಞಾನದಿಂದ ಉತ್ತರಿಸಲಾಗುತ್ತದೆ
- ಪ್ರತಿಕ್ರಿಯೆ ಸಂಪೂರ್ಣ ಸಂವಾದಾತ್ಮಕವಾಗಿದೆ
- ಸಾಧನ ವಿಳಂಬ ಅನುಭವವನ್ನು ತಡಗೊಳಿಸುತ್ತದೆ

## ಸಾಧನಗಳ ಮತ್ತು RAG ರ ನಡುವೆ

ಮಾಡ್ಯೂಲ್‌ಗಳು 03 ಮತ್ತು 04 ಎರಡೂ AI ಯ ಶಕ್ತಿಯನ್ನು ವಿಸ್ತರಿಸುತ್ತವೆ, ಆದರೆ ಮೂಲತಃ ವಿಭಿನ್ನ ರೀತಿಯಲ್ಲಿ. RAG ಮಾದರಿಗೆ **ಜ್ಞಾನ** ದೊರೆಯಲು ಡಾಕ್ಯುಮೆಂಟ್‌ಗಳನ್ನು ಹಿಂಬಾಲಿಸುವುದನ್ನು ನೀಡುತ್ತದೆ. ಸಾಧನಗಳು ಕಾರ್ಯಗಳನ್ನು ಕೈಗೊಳ್ಳಲು ಸಾಧ್ಯಮಾಡುತ್ತವೆ.

<img src="../../../translated_images/kn/tools-vs-rag.ad55ce10d7e4da87.webp" alt="ಸಾಧನಗಳು ಮತ್ತು RAG ಹೋಲಿಕೆ" width="800"/>

*RAG ಸ್ಥಿರ ಡಾಕ್ಯುಮೆಂಟ್‌ಗಳಿಂದ ಮಾಹಿತಿ ಪಡೆಯುತ್ತದೆ — ಸಾಧನಗಳು ಕಾರ್ಯಗಳನ್ನು ನಡಿಸುತ್ತದೆ ಮತ್ತು ಡೈನಾಮಿಕ್, ನೈಜ ಸಮಯದ ಡೇಟಾವನ್ನು ತರುತ್ತವೆ. ಹೆಚ್ಚಿನ ಉತ್ಪಾದನಾ ವ್ಯವಸ್ಥೆಗಳು ಎರಡನ್ನೂ ಕೂಡ ಒಟ್ಟಿಗೆ ಬಳಸುತ್ತವೆ.*

ಪ್ರಾಯೋಗಿಕವಾಗಿ, ಹೆಚ್ಚಿನ ಉತ್ಪಾದನಾ ವ್ಯವಸ್ಥೆಗಳು ಎರಡೂ ವಿಧಾನಗಳ ಸಂಯೋಜನೆ ಮಾಡುತ್ತವೆ: RAG ನಿಮ್ಮ ಡಾಕ್ಯುಮೆಂಟೇಶನ್‌ನಲ್ಲಿ ಉತ್ತರಗಳನ್ನು ನೆಲೆಗೊಳಿಸುವುದಕ್ಕಾಗಿ ಮತ್ತು ಸಾಧನಗಳು ಲೈವ್ ಡೇಟಾ ಪಡೆಯಲು ಅಥವಾ ಕಾರ್ಯಾಚರೆಗಳಿಗಾಗಿ.
