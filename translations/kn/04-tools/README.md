## ನೀವು ಕಲಿಯುವುದು ಏನು

ಇವರೆಗೆ, ನೀವು AI ಜೊತೆ ಸಂವಾದಿಸುವ ವಿಧಾನ, ಪರಿಣಾಮಕಾರಿಯಾಗಿ ಪ್ರಾಂಪ್ಟ್‌ಗಳನ್ನು ರಚಿಸುವುದು ಮತ್ತು ಉತ್ತರಗಳನ್ನು ನಿಮ್ಮ ಡಾಕ್ಯುಮೆಂಟ್ಗಳಲ್ಲಿ ನೆಟ್ಟಗೊಳಿಸುವುದನ್ನು ಕಲಿತಿದ್ದೀರಿ. ಆದರೆ ಅახლಿ ಮಿತಿ ಇದೆ: ಭಾಷಾ ಮಾದರಿಗಳು ಕೇವಲ ಪಠ್ಯವನ್ನು ಉತ್ಪಾದಿಸಬಹುದು. ಅವು ಹವಾಮಾನ ಪರಿಶೀಲನೆ, ಲೆಕ್ಕಾಚಾರ, ಡೇಟಾಬೇಸ್ ವಿಚಾರಣೆ ಅಥವಾ ಹೊರಗಿನ ವ್ಯವಸ್ಥೆಗಳೊಂದಿಗೆ ಸಂವಹನ ಮಾಡಲಾರದವು.

ಉಪಕರಣಗಳು ಇದನ್ನು ಬದಲಾಯಿಸುತ್ತವೆ. ಮಾದರಿಗೆ ಕರೆಮಾಡಬಹುದಾದ ಕಾರ್ಯಕ್ಷಮತೆಗಳನ್ನು ನೀಡಿ, ನೀವು ಅದನ್ನು ಪಠ್ಯ ಉತ್ಪಾದಕರಿಂದ ಏಜೆಂಟ್ ಆಗಿ ಪರಿವರ್ತಿಸುತ್ತೀರಿ, ಇದು ಕ್ರಮಗಳನ್ನು ತೆಗೆದುಕೊಳ್ಳಬಹುದು. ಮಾದರಿ ಯಾವಾಗ ಉಪಕರಣ ಬೇಕು, ಯಾವ ಉಪಕರಣ ಬಳಸಬೇಕು ಮತ್ತು ಯಾವ ಪ್ಯಾರಾಮೀಟರ್‌ಗಳನ್ನು ಪಾಸ್ ಮಾಡಬೇಕು ಎಂದು ನಿರ್ಧರಿಸುತ್ತದೆ. ನಿಮ್ಮ ಕೋಡ್ ಕಾರ್ಯಾಚರಣೆಯನ್ನು ನೆರವೇರಿಸಿ ಫಲಿತಾಂಶವನ್ನು ನೀಡುತ್ತದೆ. ಮಾದರಿ ಆ ಫಲಿತಾಂಶವನ್ನು ತನ್ನ ಪ್ರತಿಕ್ರಿಯಾದಲ್ಲಿ ಸೇರಿಸುತ್ತದೆ.

## ಉಪಕರಣಗಳೊಂದಿಗೆ AI ಏಜೆಂಟ್‌ಗಳನ್ನು ಅರ್ಥಮಾಡಿಕೊಳ್ಳುವುದು

> **📝 ಟಿಪ್ಪಣಿ:** ಈ ಮೊಡ್ಯೂಲ್‌ನಲ್ಲಿ "ಏಜೆಂಟ್‌ಗಳು" ಎಂಬ ಪದವು ಉಪಕರಣ-ಕರೆಮಾಡುವ ಸಾಮರ್ಥ್ಯಗಳನ್ನು ಹೊಂದಿರುವ AI ಸಹಾಯಕಗಳನ್ನು ಸೂಚಿಸುತ್ತದೆ. ಇದು **Agentic AI** ಮಾದರಿಗಳಿಂದ (ಸ್ವಾಯತ್ತ ಏಜೆಂಟ್‌ಗಳು ಯೋಜನೆ, ಸ್ಮರಣೆ ಮತ್ತು ಬಹು ಹಂತ ತರ್ಕದೊಂದಿಗೆ) ಭಿನ್ನವಾಗಿದೆ, ಅವುಗಳನ್ನು ನಾವು [Module 05: MCP](../05-mcp/README.md) ನಲ್ಲಿ ಚರ್ಚಿಸುವೆವು.

ಉಪಕರಣಗಳಿಲ್ಲದೆ, ಭಾಷಾ ಮಾದರಿ ತನ್ನ ತರಬೇತಿ ಡೇಟಾದಿಂದ ಮಾತ್ರ ಪಠ್ಯವನ್ನು ಉತ್ಪಾದಿಸಬಹುದು. ಆಗಿದ್ದಾಗ ಹವಾಮಾನವನ್ನು ಕೇಳಿದರೆ, ಅದು ಊಹಿಸಿಕೊಳ್ಳಲೇಬೇಕು. ಉಪಕರಣಗಳನ್ನು ಕೊಟ್ಟರೆ, ಅದು ಹವಾಮಾನ API ಅನ್ನು ಕರೆಮಾಡಬಹುದು, ಲೆಕ್ಕಾಚಾರಗಳನ್ನು ನಡೆಸಬಹುದು, ಅಥವಾ ಡೇಟಾಬೇಸ್ ವಿಚಾರಿಸಬಹುದು – ನಂತರ ಆ ನಿಜವಾದ ಫಲಿತಾಂಶಗಳನ್ನು ತನ್ನ ಪ್ರತಿಕ್ರಿಯಾದಲ್ಲಿ ನೇರಿಸುತ್ತದೆ.

<img src="../../../translated_images/kn/what-are-tools.724e468fc4de64da.webp" alt="ఉపಕರಣಗಳಿಲ್ಲದ vs ಉಪಕರಣಗಳೊಂದಿಗೆ" width="800"/>

*ಉಪಕರಣಗಳಿಲ್ಲದೆ ಮಾದರಿ ಊಹೆ ಮಾಡಬೇಕಾಗುತ್ತದೆ — ಉಪಕರಣಗಳೊಂದಿಗೆ ಅದು APIಗಳನ್ನು ಕರೆಮಾಡಿ, ಲೆಕ್ಕಾಚಾರಗಳನ್ನು ನಡೆಸಿ, ನಿಜ ಸಮಯ ಡೇಟಾವನ್ನು ನೀಡಬಹುದು.*

ಉಪಕರಣಗಳೊಂದಿಗೆ AI ಏಜೆಂಟ್ ಒಂದು **ತರ್ಕ ಮತ್ತು ಕ್ರಿಯೆ(ReAct)** ಮಾದರಿಯನ್ನು ಅನುಸರಿಸುತ್ತದೆ. ಮಾದರಿ ಕೇವಲ ಉತ್ತರಿಸುವುದಲ್ಲ, ಅದು ಏನು ಬೇಕು ಎಂದು ಚಿಂತಿಸಿ, ಒಂದು ಉಪಕರಣವನ್ನು ಕರೆದು ಕ್ರಮ ಕೈಗೊಳ್ಳುತ್ತದೆ, ಫಲಿತಾಂಶವನ್ನು ಗಮನಿಸಿ, ನಂತರ ಮತ್ತೊಮ್ಮೆ ಕ್ರಮ ತೆಗೆದುಕೊಳ್ಳಬೇಕೇ ಅಥವಾ ಅಂತಿಮ ಉತ್ತರ ನೀಡಬೇಕೇ ಎಂಬುದನ್ನು ನಿರ್ಧರಿಸುತ್ತದೆ:

1. **ತರ್ಕ ಮಾಡುವುದು** — ಬಳಕೆದಾರನ ಪ್ರಶ್ನೆಯನ್ನು ವಿಶ್ಲೇಷಿಸಿ ಯಾವ ಮಾಹಿತಿ ಬೇಕು ಎಂದು ನಿರ್ಧರಿಸುವುದು
2. **ಕ್ರಮ ಕೈಗೊಳ್ಳುವುದು** — ಸರಿಯಾದ ಉಪಕರಣ ಆಯ್ಕೆ ಮಾಡಿ, ಸರಿಯಾದ ಪ್ಯಾರಾಮೀಟರ್ ರಚಿಸಿ ಅದನ್ನು ಕರೆಮಾಡುವುದು
3. **ಗಮನಿಸುವುದು** — ಉಪಕರಣದ ಔಟ್‌ಪುಟ್ ಅನ್ನು ಸ್ವೀಕರಿಸಿ ಫಲಿತಾಂಶವನ್ನು ಮೌಲ್ಯಮಾಪನ ಮಾಡುವುದು
4. **ಉತ್ತರಿಸುವುದು ಅಥವಾ ಪುನರಾವರ್ತಿಸುವುದು** — ಹೆಚ್ಚು ಡೇಟಾ ಬೇಕಾದರೆ ಪ್ರಕ್ರಿಯೆ ಪುನರಾವರ್ತನೆ, ಇಲ್ಲದೆ ಇದ್ದರೆ ಸ್ವಾಭಾವಿಕ ಭಾಷೆಯಲ್ಲಿ ಉತ್ತರ ರಚಿಸುವುದು

<img src="../../../translated_images/kn/react-pattern-detail.96a5efeeb6dd2f61.webp" alt="ReAct ಮಾದರಿ" width="800"/>

*ReAct ಚಕ್ರ — ಏಜೆಂಟ್ ಏನು ಮಾಡಬೇಕೆಂದು ತರ್ಕ ಮಾಡುತ್ತದೆ, ಉಪಕರಣವನ್ನು ಕರೆದು ಕ್ರಮ ಕೈಗೊಳ್ಳುತ್ತದೆ, ಫಲಿತಾಂಶವನ್ನು ಗಮನಿಸುತ್ತದೆ ಮತ್ತು ಅಂತಿಮ ಉತ್ತರವನ್ನು ನೀಡುವವರೆಗೆ ಲೂಪ್ ಮಾಡುತ್ತದೆ.*

ಇದು ಸ್ವಯಂಚಾಲಿತವಾಗಿ ನಡೆಯುತ್ತದೆ. ನೀವು ಉಪಕರಣಗಳನ್ನು ಮತ್ತು ಅವುಗಳ ವಿವರಣೆಗಳನ್ನು ನಿರ್ಧರಿಸುತ್ತೀರಿ. ಮಾದರಿ ನಿರ್ಣಯ ತೆಗೆದುಕೊಳ್ಳುವ ಕಾರ್ಯವನ್ನು ನಿರ್ವಹಿಸುತ್ತದೆ ಯಾವಾಗ ಮತ್ತು ಹೇಗೆ ಉಪಕರಣಗಳನ್ನು ಬಳಸಬೇಕೆಂದು.

## ಉಪಕರಣ ಕರೆ ಮಾಡುವ ವಿಧಾನ

### ಉಪಕರಣ ನಿರ್ದಿಷ್ಟನೆಗಳು

[WeatherTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) | [TemperatureTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/TemperatureTool.java)

ನೀವು ಸ್ಪಷ್ಟ ವಿವರಣೆಗಳು ಮತ್ತು ಪ್ಯಾರಾಮೀಟರ್ ವಿವರಗಳೊಂದಿಗೆ ಕಾರ್ಯಗಳನ್ನು ನಿರ್ಧರಿಸುತ್ತೀರಿ. ಮಾದರಿ ಈ ವಿವರಣೆಗಳನ್ನು ತನ್ನ ಸಿಸ್ಟಂ ಪ್ರಾಂಪ್ಟ್‌ನಲ್ಲಿ ನೋಡುತ್ತದೆ ಮತ್ತು ಯಾವ ಉಪಕರಣ ಏನು ಮಾಡುತ್ತದೆ ಎಂಬುದನ್ನು ಅರ್ಥಮಾಡಿಕೊಳ್ಳುತ್ತದೆ.

```java
@Component
public class WeatherTool {
    
    @Tool("Get the current weather for a location")
    public String getCurrentWeather(@P("Location name") String location) {
        // ನಿಮ್ಮ ಹವಾಮಾನ ಹುಡುಕಾಟ ತರ್ಕ
        return "Weather in " + location + ": 22°C, cloudy";
    }
}

@AiService
public interface Assistant {
    String chat(@MemoryId String sessionId, @UserMessage String message);
}

// ಸಹಾಯಕನು ಸ್ವಯಂಚಾಲಿತವಾಗಿ Spring Boot ನೊಂದಿಗೆ ಸಂಪರ್ಕಗೊಂಡಿರುತ್ತಾನೆ:
// - ChatModel ಬೀನ್
// - @Component ವರ್ಗಗಳಿಂದ ಎಲ್ಲಾ @Tool ವಿಧಾನಗಳು
// - ಅಧಿವೇಶನ ನಿರ್ವಹಣೆಗೆ ChatMemoryProvider
```

ಕೆಳಗಿನ ಚಿತ್ರವು ಪ್ರತಿಯೊಂದು ಅನೋಟೇಶನ್ ಅನ್ನು ವಿವರಿಸಿ, ಯಾವ ಭಾಗವು AI ಯಿಂದ ಉಪಕರಣವನ್ನು ಯಾವಾಗ ಕರೆ ಮಾಡಬೇಕೆಂದು ಮತ್ತು ಯಾವ ಆರ್ಗ್ಯುಮೆಂಟ್ಗಳನ್ನು ಪಾಸ್ ಮಾಡಬೇಕೆಂದು ತಿಳಿಯಲು ಸಹಾಯ ಮಾಡುತ್ತದೆಯೋ ತೋರಿಸುತ್ತದೆ:

<img src="../../../translated_images/kn/tool-definitions-anatomy.f6468546037cf28b.webp" alt="ಉಪಕರಣ ನಿರ್ದಿಷ್ಟನೆಗಳ ರಚನೆ" width="800"/>

*ಉಪಕರಣ ನಿರ್ದಿಷ್ಟನೆಯ ರಚನೆ — @Tool AI ಗೆ ಯಾವಾಗ ಉಪಕರಣ ಬಳಸಬೇಕು ಎಂದು ಹೇಳುತ್ತದೆ, @P ಪ್ರತಿಯೊಂದು ಪ್ಯಾರಾಮೀಟರ್ ವಿವರಣೆಯನ್ನು ನೀಡುತ್ತದೆ, ಮತ್ತು @AiService ಸ್ಟಾರ್ಟ್‌ಅಪ್‌ನಲ್ಲಿ ಎಲ್ಲವನ್ನೂ ಜೋಡಿಸುತ್ತದೆ.*

> **🤖 [GitHub Copilot](https://github.com/features/copilot) ಚಾಟ್ ಜೊತೆಗೆ ಪ್ರಯತ್ನಿಸಿ:** [`WeatherTool.java`](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) ತೆರೆಯಿರಿ ಮತ್ತು ಕೇಳಿ:
> - "ನನಗಾಗಿ ನಿಜವಾದ ಹವಾಮಾನ API OpenWeatherMap ಯನ್ನು ಮೋಕ್ ಡೇಟಾ ಬದಲು ಹೇಗೆ ಸಂಯೋಜಿಸಬಹುದು?"
> - "ಉಪಕರಣವನ್ನು ಸರಿಯಾಗಿ ಬಳಸಲು ನೆರವಾಗುವ ಉತ್ತಮ ಉಪಕರಣ ವಿವರಣೆ ಯಾವುದು?"
> - "ಉಪಕರಣ ಅನುಷ್ಠಾನದ ದೋಷಗಳು ಮತ್ತು ರೇಟ್ ಸೀಮಿತಿಗಳನ್ನು ಹೇಗೆ ನಿರ್ವಹಿಸಬೇಕು?"

### ನಿರ್ಣಯ ತೆಗೆದುಕೊಳ್ಳುವುದು

ಬಳಕೆದಾರ "ಸಿಯಾಟಲಿನ ಹವಾಮಾನ ಹೇಗಿದೆ?" ಎಂದು ಕೇಳಿದಾಗ, ಮಾದರಿ ಯಾದೃಚ್ಛಿಕವಾಗಿ ಒಂದು ಉಪಕರಣ ಆಯ್ಕೆ ಮಾಡದು. ಅದು ಬಳಕೆದಾರದ ಉದ್ದೇಶವನ್ನು ಪ್ರತಿಯೊಂದು ಉಪಕರಣದ ವಿವರಣೆಯೊಂದಿಗೆ ಹೋಲಿಸಿ, ಪ್ರಸ್ತುತತೆಗೆ ಪ್ರತಿ ಉಪಕರಣವನ್ನು ಅಂಕನೀಡುತ್ತದೆ ಮತ್ತು ಅತ್ಯುತ್ತಮ ಹೊಂದಾಣಿಕೆಯನ್ನು ಆರಿಸಿಕೊಂಡು, ಸರಿಯಾದ ಪ್ಯಾರಾಮೀಟರ್‌ಗಳ ಜೊತೆ ವರ್ಣಿತ ಕಾರ್ಯ ಕ್ಲಾಸ್ ಅನ್ನು ರಚಿಸುತ್ತದೆ — ಈ ಸಂದರ್ಭದಲ್ಲಿ, `location` ನ್ನು `"Seattle"` ಗೆ ಹೊಂದಿಸುತ್ತದೆ.

ಯಾವುದೇ ಉಪಕರಣ ಬಳಕೆದಾರನ ವಿನಂತಿಯನ್ನು ಹೊಂದದಿದ್ದರೆ, ಮಾದರಿ ತನ್ನದೇ ಜ್ಞಾನದಿಂದ ಉತ್ತರ ನೀಡುತ್ತದೆ. ಹಲವು ಉಪಕರಣಗಳು ಹೊಂದಿದ್ದರೆ, ಅತ್ಯಂತ ನಿಖರವಾದುದನ್ನು ಆಯ್ಕೆ ಮಾಡುತ್ತದೆ.

<img src="../../../translated_images/kn/decision-making.409cd562e5cecc49.webp" alt="AI ಯಾವ ಉಪಕರಣವನ್ನು ಬಳಸಬೇಕೆಂದು ನಿರ್ಧರಿಸುವುದು ಹೇಗೆ" width="800"/>

*ಮಾದರಿ ಪ್ರತಿ ಲಭ್ಯವಿರುವ ಉಪಕರಣವನ್ನು ಬಳಕೆದಾರನ ಉದ್ದೇಶದ ವಿರುದ್ಧ ಮೌಲ್ಯಮಾಪನ ಮಾಡುತ್ತದೆ ಮತ್ತು ಅತ್ಯುತ್ತಮ ಹೊಂದಾಣಿಕೆಯನ್ನು ಆರಿಸುತ್ತದೆ — ಅದಕ್ಕಾಗಿ ಸ್ಪಷ್ಟ, ನಿಖರ ಉಪಕರಣ ವಿವರಣೆಗಳನ್ನು ಬರೆಯುವುದು ಬಹಳ ಪ್ರಾಮುಖ್ಯ.*

### ನಿರ್ವಹಣೆ

[AgentService.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/service/AgentService.java)

ಸ್ಪ್ರಿಂಗ್ ಬುಟ್ ಸ್ವಯಂಚಾಲಿತವಾಗಿ ಘೋಷಿತ `@AiService` ಇಂಟರ್ಫೇಸ್‌ಗೆ ಲಭ್ಯವಿರುವ ಎಲ್ಲಾ ಉಪಕರಣಗಳನ್ನು ಜೋಡಿಸುತ್ತದೆ ಮತ್ತು LangChain4j ಉಪಕರಣ ಕರೆಗಳನ್ನು ಸ್ವಯಂಚಾಲಿತವಾಗಿ ನಿರ್ವಹಿಸುತ್ತದೆ. ಬಳಕೆದಾರನ ಸಹಜ ಭಾಷೆ ಪ್ರಶ್ನೆಯಿಂದ ಆರಂಭಿಸಿ ಪೂರ್ತಿ ಉಪಕರಣ ಕರೆ ಆರು ಹಂತಗಳ ಮೂಲಕ ಹರಿಯುತ್ತದೆ:

<img src="../../../translated_images/kn/tool-calling-flow.8601941b0ca041e6.webp" alt="ಉಪಕರಣ ಕರೆ ಮಾಡುವ ಪ್ರಕ್ರಿಯೆ" width="800"/>

*ಅಂತಿಮ ಪ್ರವಾಹ — ಬಳಕೆದಾರನು ಪ್ರಶ್ನೆ ಕೇಳುತ್ತಾನೆ, ಮಾದರಿ ಒಂದು ಉಪಕರಣ ಆರಿಸುತ್ತಾನೆ, LangChain4j ಅದನ್ನು ನಿರ್ವಹಿಸುತ್ತದೆ, ಮತ್ತು ಮಾದರಿ ಫಲಿತಾಂಶವನ್ನು ಸಹಜ ಪ್ರತಿಕ್ರિયામાં ಸೇರಿಸುತ್ತದೆ.*

> **🤖 [GitHub Copilot](https://github.com/features/copilot) ಚಾಟ್ ಜೊತೆ ಪ್ರಯತ್ನಿಸಿ:** [`AgentService.java`](../../../04-tools/src/main/java/com/example/langchain4j/agents/service/AgentService.java) ತೆರೆಯಿರಿ ಮತ್ತು ಕೇಳಿ:
> - "ReAct ಮಾದರಿ ಹೇಗೆ ಕಾರ್ಯನಿರ್ವಹಿಸುತ್ತದೆ ಮತ್ತು AI ಏಜೆಂಟ್‌ಗಳಿಗೆ ಅದು ಯಶಸ್ವಿಯಾಗಿದೆ ಯಾಕೆ?"
> - "ಏಜೆಂಟ್ ಉಪಕರಣವನ್ನು ಯಾವ ಕ್ರಮದಲ್ಲಿ ಮತ್ತು ಯಾವ ಉಪಕರಣವನ್ನು ಬಳಸಬೇಕು ಎನ್ನು ಹೇಗೆ ನಿರ್ಧರಿಸುತ್ತದೆ?"
> - "ಉಪಕರಣ ಕಾರ್ಯಾಚರಣೆ ವಿಫಲವಾದರೆ ಏನು ಆಗುತ್ತದೆ - ದೋಷಗಳನ್ನು ಬಲವಂತವಾಗಿ ಹೇಗೆ ನಿರ್ವಹಿಸಬೇಕು?"

### ಪ್ರತಿಕ್ರಿಯಾ ಉತ್ಪಾದನೆ

ಮಾದರಿ ಹವಾಮಾನ ಡೇಟಾವನ್ನು ಸ್ವೀಕರಿಸಿ ಬಳಕೆದಾರನಿಗಾಗಿ ಸಹಜ ಭಾಷೆಯಲ್ಲಿ ಪ್ರತಿಕ್ರಿಯೆಯನ್ನು ರೂಪಿಸುತ್ತದೆ.

### ವಾಸ್ತುಶಿಲ್ಪ: ಸ್ಪ್ರಿಂಗ್ ಬುಟ್ ಸ್ವಯಂಚಾಲಿತ ಜೋಡಣೆ

ಈ ಮೊಡ್ಯೂಲ್ LangChain4j ನ ಸ್ಪ್ರಿಂಗ್ ಬುಟ್ ಸಮನುಸ್ಥಾಪನೆಯೊಂದಿಗೆ ಘೋಷಿತ `@AiService` ಇಂಟರ್ಫೇಸ್‌ಗಳ ಬಳಕೆಯನ್ನು ಮಾಡುತ್ತದೆ. ಸ್ಟಾರ್ಟ್‌ಅಪ್‌ನಲ್ಲಿ ಸ್ಪ್ರಿಂಗ್ ಬುಟ್ ಎಲ್ಲಾ `@Component` ಗಳು, `@Tool` ವಿಧಾನಗಳು, ನಿಮ್ಮ `ChatModel` ಬಿನ್ ಮತ್ತು `ChatMemoryProvider` ಅನ್ನು ಕಂಡುಹಿಡಿದು ಒಂದು `Assistant` ಇಂಟರ್ಫೇಸ್ನಲ್ಲಿ ಎಲ್ಲವನ್ನೂ ಜೋಡಿಸುತ್ತದೆ, ಯಾವುದೇ ಅಗತ್ಯವಿಲ್ಲದೆ:

<img src="../../../translated_images/kn/spring-boot-wiring.151321795988b04e.webp" alt="ಸ್ಪ್ರಿಂಗ್ ಬುಟ್ ಸ್ವಯಂಚಾಲಿತ ಜೋಡಣೆಯ ವಾಸ್ತುಶಿಲ್ಪ" width="800"/>

*@AiService ಇಂಟರ್ಫೇಸ್ ChatModel, ಉಪಕರಣ ಘಟಕಗಳು ಮತ್ತು ಮೆಮೊರಿ ಪ್ರೊವೈಡರ್‌ಗಳನ್ನು ಸರಿಯಾಗಿ ಜೋಡಿಸುತ್ತದೆ — ಸ್ಪ್ರಿಂಗ್ ಬುಟ್ ಸ್ವಯಂಚಾಲಿತವಾಗಿ ಎಲ್ಲಾ ಜೋಡಣೆಗಳನ್ನು ನಿರ್ವಹಿಸುತ್ತದೆ.*

ಈ ವಿಧಾನದಿಂದ ಪ್ರಮುಖ ಲಾಭಗಳು:

- **ಸ್ಪ್ರಿಂಗ್ ಬುಟ್ ಸ್ವಯಂಚಾಲಿತ ಜೋಡಣೆ** — ChatModel ಮತ್ತು ಉಪಕರಣಗಳು ತಾನಾದಂತೆ ಸೇರ್ಪಡೆಯಾಗುತ್ತವೆ
- **@MemoryId ಮಾದರಿ** — ಸ್ವಯಂ ನಿರ್ವಹಿತ ಸೆಷನ್ ಆಧಾರಿತ ಮೆಮೊರಿ ವ್ಯವಸ್ಥೆ
- **ಒಂದು ಪ್ರತಿಯನ್ನು ಸೃಷ್ಟಿ** — ಉತ್ತಮ ಕಾರ್ಯಕ್ಷಮತೆಗಾಗಿ Assistant ಒಂದುನಮ್ಮೆ ಸೃಷ್ಟಿಸಿ ಮರುಬಳಕೆ ಮಾಡುತ್ತದೆ
- **ತರಬೇತಿ ಸುರಕ್ಷಿತ ಕಾರ್ಯಗತಗೊಳಣೆ** — ಜಾವಾ ವಿಧಾನಗಳನ್ನು ನೇರವಾಗಿ_calls ಮಾಡುತ್ತದೆ ಮತ್ತು ತരം ಪರಿವರ್ತನೆ ಮಾಡುತ್ತದೆ
- **ಬಹು ತಿರುವು ಸಂಯೋಜನೆ** — ಉಪಕರಣ ಶೃಂಖಲೆಯನ್ನು ಸ್ವಯಂಚಾಲಿತವಾಗಿ ನಿರ್ವಹಿಸುತ್ತದೆ
- **ಏಕದಂಡ ಕುಕರ್ಮತೆ ಇಲ್ಲ** — ಕೈಯಿಂದ `AiServices.builder()` ಆಹ್ವಾನಿಸುವ ಅಗತ್ಯವಿಲ್ಲ, ಅಥವಾ ಮೆಮೊರಿ HashMap ನಿರ್ವಹಣೆ ಇಲ್ಲ

ಬೇರೆ ವಿಧಾನಗಳು (`AiServices.builder()` ಕೈಯಿಂದ) ಹೆಚ್ಚಿನ ಕೋಡ್ ಅಗತ್ಯವಿದೆ ಮತ್ತು ಸ್ಪ್ರಿಂಗ್ ಬುಟ್ ಏಕೀಕರಣ ಲಾಭಗಳನ್ನು ತಪ್ಪಿಸಲು ಸಾಧ್ಯ.

## ಉಪಕರಣ ಶೃಂಖಲನೆ

**ಉಪಕರಣ ಶೃಂಖಲನೆ** — ಉಪಕರಣಆಧಾರಿತ ಏಜೆಂಟ್‌ಗಳ ನಿಜವಾದ ಶಕ್ತಿ ಒಂದು ಪ್ರಶ್ನೆಗೆ ಬಹು ಉಪಕರಣಗಳು ಬೇಕಾದಾಗ ಪ್ರकटವಾಗುತ್ತದೆ. "ಸಿಯಾಟಲಿನಲ್ಲಿ ಫಾರೆನ್‌ಹೀಟ್‌ನಲ್ಲಿ ಹವಾಮಾನ ಹೇಗೆ ಇದೆ?"ಎಂದು ಕೇಳಿದಾಗ, ಏಜೆಂಟ್ ಸ್ವಯಂಚಾಲಿತವಾಗಿ ಎರಡು ಉಪಕರಣಗಳನ್ನು ಶೃಂಖಲಿಸಲು ಪ್ರಾರಂಭಿಸುತ್ತದೆ: ಮೊದಲಿಗೆ `getCurrentWeather` ಅನ್ನು ಕರೆಸುತ್ತದೆ ಸೆಲ್ಶಿಯಸ್‌ನಲ್ಲಿ ತಾಪಮಾನ ಪಡೆಯಲು, ನಂತರ ಆ ಮೌಲ್ಯವನ್ನು `celsiusToFahrenheit` ಗೆ ಪರಿವರ್ತಿಸಲು ಪಾಸ್ ಮಾಡುತ್ತದೆ — ಎಲ್ಲವೂ ಒಂದೇ ಸಂವಾದ ತಿರುವಿನಲ್ಲಿ.

<img src="../../../translated_images/kn/tool-chaining-example.538203e73d09dd82.webp" alt="ಉಪಕರಣ ಶೃಂಖಲೆ ಉದಾಹರಣೆ" width="800"/>

*ಉಪಕರಣ ಶೃಂಖಲೆ ಚಾಲನೆಯಲ್ಲಿ — ಏಜೆಂಟ್ ಮೊದಲು getCurrentWeather ಅನ್ನು ಕರೆದು, ನಂತರ ಸೆಲ್ಶಿಯಸ್ ಫಲಿತಾಂಶವನ್ನು celsiusToFahrenheit ಗೆ ನೀಡುತ್ತದೆ ಮತ್ತು ಸಂಯುಕ್ತ ಉತ್ತರ ನೀಡುತ್ತದೆ.*

**ಕ್ರಿಪಯಾವಂತ ವಿಫಲತೆಗಳು** — ಮೋಕ್ ಡೇಟಾದಲ್ಲಿಲ್ಲದ ನಗರದಲ್ಲಿ ಹವಾಮಾನವನ್ನು ಕೇಳಿ. ಉಪಕರಣ ದೋಷ ಸಂದೇಶವನ್ನು ನೀಡುತ್ತದೆ ಮತ್ತು AI ಸಹಾಯ ಸಾಧ್ಯವಿಲ್ಲ ಎಂದು ವಿವರಿಸುತ್ತದೆ ಬದಲಿಗೆ ಕ್ರ್ಯಾಶ್ ಆಗುವುದರ. ಉಪಕರಣಗಳು ಸುರಕ್ಷಿತವಾಗಿ ವಿಫಲವಾಗುತ್ತವೆ. ಕೆಳಗಿನ ಚಿತ್ರವು ಎರಡನೇ ವಿಧಾನಗಳನ್ನು ಹೋಲಿಸುತ್ತದೆ — ಸರಿಯಾದ ದೋಷ ನಿರ್ವಹಣೆಯಿಂದ, ಏಜೆಂಟ್ ವಿನಂತಿಯನ್ನು ಹಿಡಿದು ಸಹಾಯಕರ ಪ್ರತಿಕ್ರಿಯೆಯನ್ನು ಕೊಡುತ್ತದೆ, ಅದೇ ಸಂದರ್ಭದಲ್ಲಿ ಇಲ್ಲದಿದ್ದರೆ ಸಂಪೂರ್ಣ ಅಪ್ಲಿಕೇಶನ್ ಚಲನವಲನವಾಗುತ್ತದೆ:

<img src="../../../translated_images/kn/error-handling-flow.9a330ffc8ee0475c.webp" alt="ದೋಷ ನಿರ್ವಹಣಾ ಪ್ರವಾಹ" width="800"/>

*ಉಪಕರಣ ವಿಫಲವಾದಾಗ, ಏಜೆಂಟ್ ದೋಷವನ್ನು ಹಿಡಿದು ಕ್ರ್ಯಾಶ್ ಆಗದೆ ಸಹಾಯಕರ ವಿವರಣೆ ನೀಡುತ್ತದೆ.*

ಇದು ಒಂದೇ ಸಂವಾದ ತಿರುವಿನಲ್ಲಿ ನಡೆಯುತ್ತದೆ. ಏಜೆಂಟ್ ಸ್ವಯಂ ಪ್ರೇರಿತವಾಗಿ ಬಹು ಉಪಕರಣ ಕರೆಗಳನ್ನ ನಿರ್ವಹಿಸುತ್ತದೆ.

## ಅಪ್ಲಿಕೇಶನ್ ಅನ್ನು ಚಾಲನೆ ಮಾಡಿ

**ನಿಯೋಜನೆ ಪರಿಶೀಲಿಸಿ:**

ಅಜುರ ಕ್ರೆಡೆನ್ಶಿಯಲ್ಸ್ ಒಳಗೊಂಡ `.env` ಫೈಲ್ ರುಟ್ ಡೈರೆಕ್ಟರಿಯಲ್ಲಿ ಇದೆಯೇ ಎಂದು ಖಚಿತಪಡಿಸಿಕೊಳ್ಳಿ (Module 01 ಅವಧಿಯಲ್ಲಿ ನಿರ್ಮಿಸಲಾಗಿದೆ). ಈ ಫೋಲ್ಡರಿಂದ ಈ ಕ್ರಿಯಾ ಮಾಡಿ (`04-tools/`):

**ಬ್ಯಾಶ್:**
```bash
cat ../.env  # AZURE_OPENAI_ENDPOINT, API_KEY, DEPLOYMENT ಅನ್ನು ತೋರಿಸಬೇಕು
```

**ಪವರ್‌ಶೆಲ್:**
```powershell
Get-Content ..\.env  # AZURE_OPENAI_ENDPOINT, API_KEY, DEPLOYMENT ಅನ್ನು ತೋರಿಸಬೇಕು
```

**ಅಪ್ಲಿಕೇಶನ್ ಪ್ರಾರಂಭಿಸಿ:**

> **ಗಮನಿಸಿ:** ನೀವು ಈಗಾಗಲೇ రುಟ್ ಡೈರೆಕ್ಟರಿಯಿಂದ `./start-all.sh` ಮೂಲಕ ಎಲ್ಲ ಅಪ್ಲಿಕೇಶನ್ಗಳನ್ನು ಪ್ರಾರಂಭಿಸಿದ್ದರೆ (Module 01 ನಲ್ಲಿ ವಿವರಣೆ ಇದೆ), ಈ ಮೊಡ್ಯೂಲ್ ಈಗ 8084 ಪೋರ್ಟ್ ನಲ್ಲಿ ಚಲಿಸುತ್ತಿದೆ. ಈ ಸ್ಟಾರ್ಟ್ ಕಮಾಂಡ್ಗಳನ್ನು ಬಿಟ್ಟು http://localhost:8084 ಗೆ ನೇರವಾಗಿ ಹೋಗಬಹುದು.

**ಆಯ್ಕೆ 1: ಸ್ಪ್ರಿಂಗ್ ಬುಟ್ ಡ್ಯಾಶ್‌ಬೋರ್ಡ್ ಬಳಸಿ (VS ಕೋಡ್ ಬಳಕೆದಾರರಿಗೆ ಶಿಫಾರಸು)**

ಡೆವ್ ಕಂಟೈನರ್ ಸ್ಪ್ರಿಂಗ್ ಬುಟ್ ಡ್ಯಾಶ್‌ಬೋರ್ಡ್ ವಿಸ್ತರಣೆಯನ್ನು ಸೇರಿಸಲಾಗಿದೆ, ಇದು ಎಲ್ಲಾ ಸ್ಪ್ರಿಂಗ್ ಬುಟ್ ಅಪ್ಲಿಕೇಶನ್ ಗಳನ್ನು ನಿಯಂತ್ರಿಸಲು ದೃಶ್ಯ ಇಂಟರ್ಫೇಸ್ ನೀಡುತ್ತದೆ. ಇದನ್ನು VS ಕೋಡ್‌ನ ಎಕ್ಟಿವಿಟಿ ಬಾರ್ ಎಡಬದಿಯಲ್ಲಿ ಸ್ಪ್ರಿಂಗ್ ಬುಟ್ ಐಕಾನ್ ಬಳಿ ಕಾಣಬಹುದು.

ಸ್ಪ್ರಿಂಗ್ ಬುಟ್ ಡ್ಯಾಶ್‌ಬೋರ್ಡ್‌ನಿಂದ ನೀವು:
- ಕಾರ್‌ಯಕ್ಷೇತ್ರದ ಎಲ್ಲಾ ಲಭ್ಯವಿರುವ ಸ್ಪ್ರಿಂಗ್ ಬುಟ್ ಅಪ್ಲಿಕೇಶನ್‌ಗಳನ್ನು ನೋಡಬಹುದು
- ಸಿಂಗ್‌ಲ್ ಕ್ಲಿಕ್‌ನಿಂದ ಅಪ್ಲಿಕೇಶನ್‌ಗಳನ್ನು ಪ್ರಾರಂಭ / ನಿಲ್ಲಿಸಬಹುದು
- ಅಪ್ಲಿಕೇಶನ್ ಲಾಗ್‌ಗಳನ್ನು ನೈಜ ಸಮಯದಲ್ಲಿ ವೀಕ್ಷಿಸಬಹುದು
- ಅಪ್ಲಿಕೇಶನ್ ಸ್ಥಿತಿಯನ್ನು ಮೇಲ್ವಿಚಾರಣೆ ಮಾಡಬಹುದು

"tools" ಪಕ್ಕದ ಪ್ಲೇ ಬಟನ್ ಕ್ಲಿಕ್ ಮಾಡಿ ಈ ಮೊಡ್ಯೂಲ್ ಪ್ರಾರಂಭಿಸಿ, ಅಥವಾ ಎಲ್ಲಾ ಮೊಡ್ಯೂಲ್‌ಗಳನ್ನು ಒಂದೇ ಬಾರಿ ಪ್ರಾರಂಭಿಸಿ.

VS ಕೋಡ್‌ನಲ್ಲಿ ಸ್ಪ್ರಿಂಗ್ ಬುಟ್ ಡ್ಯಾಶ್‌ಬೋರ್ಡ್ ಇವುಂಥದಾಗಿರುತ್ತದೆ:

<img src="../../../translated_images/kn/dashboard.9b519b1a1bc1b30a.webp" alt="ಸ್ಪ್ರಿಂಗ್ ಬುಟ್ ಡ್ಯಾಶ್‌ಬೋರ್ಡ್" width="400"/>

*VS ಕೋಡ್‌ನ ಸ್ಪ್ರಿಂಗ್ ಬುಟ್ ಡ್ಯಾಶ್‌ಬೋರ್ಡ್ — ಎಲ್ಲ ಮೊಡ್ಯೂಲ್‌ಗಳನ್ನು ಒಂದೇ ಜಾಗದಲ್ಲಿ ಪ್ರಾರಂಭಿಸಿ, ನಿಲ್ಲಿಸಿ ಮತ್ತು ಮೇಲ್ವಿಚಾರಣೆ ಮಾಡಿ*

**ಆಯ್ಕೆ 2: ಶೆಲ್ ಸ್ಕ್ರಿಪ್ಟ್ ಬಳಸಿ**

ಎಲ್ಲಾ ವೆಬ್ ಅಪ್ಲಿಕೇಶನ್‌ಗಳನ್ನು ಪ್ರಾರಂಭಿಸಿ (ಮೊಡ್ಯೂಲ್‌ಗಳು 01-04):

**ಬ್ಯಾಶ್:**
```bash
cd ..  # ರೂಟ್ ಡೈರೆಕ್ಟರಿ ನಿಂದ
./start-all.sh
```

**ಪವರ್‌ಶೆಲ್:**
```powershell
cd ..  # ರೂಟ್ ಡೈರೆಕ್ಟರಿ nಿಂದ
.\start-all.ps1
```

ಅಥವಾ ಈ ಮೊಡ್ಯೂಲ್ ಮಾತ್ರ ಪ್ರಾರಂಭಿಸಿ:

**ಬ್ಯಾಶ್:**
```bash

./start.sh
```

**ಪವರ್‌ಶೆಲ್:**
```powershell
cd 04-tools

```

ಎರಡೂ ಸ್ಕ್ರಿಪ್ಟ್‌ಗಳು ಸ್ವಯಂಚಾಲಿತವಾಗಿ ರುಟ್ `.env` ಫೈಲ್‌ನಿಂದ ಪರಿಸರ ಚರಗಳು ಲೋಡ್ ಮಾಡುತ್ತವೆ ಮತ್ತು ಜಾರ್‌ಗಳು ಇಲ್ಲದೆ ಇದ್ದರೆ ಬಿಲ್ಡ್ ಮಾಡುತ್ತವೆ.

> **ಗಮನಿಸಿ:** ನೀವು ಪ್ರಾರಂಭಿಸುವ ಮುನ್ನ ಎಲ್ಲ ಮೊಡ್ಯೂಲ್‌ಗಳನ್ನು ಕೈಯಿಂದ ನಿರ್ಮಿಸಲು ಇಚ್ಛಿಸಿದರೆ:
>
> **ಬ್ಯಾಶ್:**
> ```bash
> cd ..  # Go to root directory
> mvn clean package -DskipTests
> ```
>
> **ಪವರ್‌ಶೆಲ್:**
> ```powershell
> cd ..  # Go to root directory
> mvn clean package -DskipTests
> ```

ನೀವು ನಿಮ್ಮ ಬ್ರೌಸರ್‌ನಲ್ಲಿ http://localhost:8084 ತೆರೆದಿರಿ.

**ನಿಲ್ಲಿಸಲು:**

**ಬ್ಯಾಶ್:**

## ಅಪ್ಲಿಕೇಶನ್ ಬಳಕೆ ಮಾಡುವುದು

ಅಪ್ಲಿಕೇಶನ್ ಒಂದು ವೆಬ್ ಇಂಟರ್ಫೇಸ್ ಒದಗಿಸುತ್ತದೆ, ಇಲ್ಲಿ ನೀವು ಹವಾಮಾನ ಮತ್ತು ತಾಪಮಾನ ಪರಿವರ್ತನೆ ಉಪಕರಣಗಳಿಗೆ ಪ್ರವೇಶ ಹೊಂದಿರುವ AI ಏಜೆಂಟ್ ಜೊತೆ ಸಂವಹನ ಮಾಡಬಹುದು. ಇವುಂಥದ್ದು ಇಂಟರ್ಫೇಸ್ — ವೇಗವಾಗಿ ಪ್ರಾರಂಭಿಸಲು ಉದಾಹರಣೆಗಳು ಮತ್ತು ವಿನಂತಿಗಳನ್ನು ಕಳುಹಿಸಲು ಚಾಟ್ ಪ್ಯಾನೆಲ್ ಇರುತ್ತದೆ:
<a href="images/tools-homepage.png"><img src="../../../translated_images/kn/tools-homepage.4b4cd8b2717f9621.webp" alt="AI ಏಜೆಂಟ್ ಟೂಲ್ಸ್ ಇಂಟರ್‌ಫೇಸ್" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*AI ಏಜೆಂಟ್ ಟೂಲ್ಸ್ ಇಂಟರ್‌ಫೇಸ್ - ಟೂಲ್ಸ್‌ನೊಂದಿಗೆ ಸಂವಹನಕ್ಕಾಗಿ ತ್ವರಿತ ಉದಾಹರಣೆಗಳು ಮತ್ತು ಚಾಟ್ ಇಂಟರ್‌ಫೇಸ್*

### ಸರಳ ಟೂಲ್ ಉಪಯೋಗವನ್ನು ಪ್ರಯತ್ನಿಸಿ

ಸರಳ ವಿನಂತಿಯಿಂದ ಪ್ರಾರಂಭಿಸಿ: "100 ಡಿಗ್ರಿ ಫಾರೆನ್ಹೀಟ್ ಅನ್ನು ಸೆಲ್ಸಿಯಸ್‌ಗೆ ಪರಿವರ್ತಿಸಿ". ಏಜೆಂಟ್ ಅದಕ್ಕೆ ತಾಪಮಾನ ಪರಿವರ್ತನೆ ಟೂಲ್ ಬೇಕಾಗಿದೆ ಎಂದು ಗುರುತಿಸಿ, ಸರಿಯಾದ ಪ್ಯಾರಾಮೀಟರ್‌ಗಳೊಂದಿಗೆ ಅದನ್ನು ಕರೆಯುತ್ತದೆ, ಮತ್ತು ಫಲಿತಾಂಶವನ್ನು ಹಿಂತಿರುಗಿಸುತ್ತದೆ. ಇದರಿಂದ 얼마나 الطبيعي ಎಂದು ಗಮನಿಸಿ - ನೀವು ಯಾವ ಟೂಲನ್ನು ಉಪಯೋಗಿಸುವುದು ಅಥವಾ ಅದನ್ನು ಹೇಗೆ ಕರೆಸಬೇಕು ಎಂದು ನಿರ್ದಿಷ್ಟಪಡಿಸಿರಲಿಲ್ಲ.

### ಟೂಲ್ ಚೇನಿಂಗ್ ಪರೀಕ್ಷಿಸಿ

ಈಗ ಸ್ವಲ್ಪ ಹೆಚ್ಚು ಸಂಕೀರ್ಣವನ್ನು ಪ್ರಯತ್ನಿಸಿ: "ಸ್ಯಿಯಾಟಲ್ನಲ್ಲಿ ವಾತಾವರಣವೇನು ಮತ್ತು ಅದನ್ನು ಫಾರೆನ್ಹೀಟ್‌ಗೆ ಪರಿವರ್ತಿಸಿ?" ಏಜೆಂಟ್ ಹಂತ ಹಂತವಾಗಿ ಇದನ್ನು ಹೇಗೆ ಕೆಲಸ ಮಾಡುತ್ತದೆಯೋ ನೋಡಿ. ಮೊದಲಿಗೆ ವಾತಾವರಣ ಪಡೆಯುತ್ತದೆ (ಅದು ಸೆಲ್ಸಿಯಸ್ ಅನ್ನು ಹಿಂತಿರುಗಿಸುತ್ತದೆ), ಫಾರೆನ್ಹೀಟ್‌ಗೆ ಪರಿವರ್ತಿಸುವ ಅಗತ್ಯವನ್ನು ಗುರುತಿಸಿ, ಪರಿವರ್ತನೆ ಟೂಲನ್ನು ಕರೆದು, ಎರಡೂ ಫಲಿತಾಂಶಗಳನ್ನು ಒಂದೇ ಪ್ರತಿಕ್ರિયામાં ಸೇರಿಸುತ್ತದೆ.

### ಸಂಭಾಷಣೆ ಪ್ರವಾಹವನ್ನು ನೋಡಿ

ಚಾಟ್ ಇಂಟರ್‌ಫೇಸ್ ಸಂಭಾಷಣೆ ಇತಿಹಾಸವನ್ನು ಕಾಯ್ದಿರಿಸುತ್ತದೆ, ಇದರಿಂದ ನೀವು ಬಹು-ತಿರುವು ಸಂವಹನಗಳನ್ನು ನಡೆಸಬಹುದು. ಪೂರ್ವದ ಎಲ್ಲಾ ಪ್ರಶ್ನೆಗಳು ಮತ್ತು ಪ್ರತಿಕ್ರಿಯೆಗಳನ್ನು ನೀವು ನೋಡಬಹುದು, ಸಂಭಾಷಣೆಯನ್ನು ಹಿಮ್ಮುಖಗೆ ನಯವಾಗಿ ಟ್ರ್ಯಾಕ್ ಮಾಡುವುದು ಮತ್ತು ಏಜೆಂಟ್ ಹಲವಾರು ವಿನಿಮಯಗಳ ಮೂಲಕ ಸಂದರ್ಭವನ್ನು ಹೇಗೆ ನಿರ್ಮಿಸುತ್ತದೆ ಎಂಬುದನ್ನು ಅರಿತುಕೊಳ್ಳುವುದು ಸುಲಭವಾಗುತ್ತದೆ.

<a href="images/tools-conversation-demo.png"><img src="../../../translated_images/kn/tools-conversation-demo.89f2ce9676080f59.webp" alt="ಬಹು-ಟೂಲ್ ಕರೆಗಳೊಂದಿಗೆ ಸಂಭಾಷಣೆ" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*ಸರಳ ಪರಿವರ್ತನೆಗಳು, ವಾತಾವರಣ ಹುಡುಕಾಟಗಳು ಮತ್ತು ಟೂಲ್ ಚೇನಿಂಗ್ ತೋರಿಸುವ ಬಹು-ತಿರುವು ಸಂಭಾಷಣೆ*

### ವಿಭಿನ್ನ ವಿನಂತಿಗಳೊಂದಿಗೆ ಪ್ರಯೋಗ ಮಾಡಿ

ವಿವಿಧ ಸಂಯೋಜನೆಗಳನ್ನು ಪ್ರಯತ್ನಿಸಿ:
- ವಾತಾವರಣ ಹುಡುಕಾಟಗಳು: "ಟೋಕಿಯೋದಲ್ಲಿ ವಾತಾವರಣ ಹೇಗಿದೆ?"
- ತಾಪಮಾನ ಪರಿವರ್ತನೆಗಳು: "25°C ಅನ್ನು ಕೆಲ್ವಿನ್‌ನಲ್ಲಿ ಎಷ್ಟು?"
- ಸಂಯೋಜಿತ ಪ್ರಶ್ನೆಗಳು: "ಪ್ಯಾರಿಸ್‌ನಲ್ಲಿ ವಾತಾವರಣವನ್ನು ಪರಿಶೀಲಿಸಿ ಮತ್ತು ಅದು 20°C ಮೇರಿದಿದೆಯೇ ಎಂದು ತಿಳಿಸಿ"

ಏಜೆಂಟ್ ಪ್ರಕೃತ ಭಾಷೆಯನ್ನು ಹೇಗೆ ಅರ್ಥಮಾಡಿಕೊಂಡು, ವಿಷಯಕ್ಕೆ thích uyğun ಟೂಲ್ ಕರೆಗಳಿಗೆ ನಕ್ಷೆ ಹಾಕುತ್ತದೆ ಎಂಬುದನ್ನು ಗಮನಿಸಿ.

## ಪ್ರಮುಖ ಸಂಪ್ರದಾಯಗಳು

### ರಿಯಾಕ್ಟ್ ಹೂಡಿಕೆ ಮಾದರಿ (ಚಿಂತನೆ ಮತ್ತು ಕಾರ್ಯ)

ಏಜೆಂಟ್ ಚಿಂತನೆ (ಏನು ಮಾಡಬೇಕು ಎಂಬ ನಿರ್ಣಯ) ಮತ್ತು ಕಾರ್ಯಾಚರಣೆ (ಟೂಲ್‌ಗಳನ್ನು ಉಪಯೋಗಿಸುವುದು) ನಡುವೆ ಪರ್ಯಾಯವಾಗಿರುತ್ತದೆ. ಈ ಮಾದರಿ ಸ್ವತಂತ್ರ ಸಮಸ್ಯಾ ಪರಿಹಾರಕ್ಕೆ ಅನುಕೂಲ ಮಾಡಿಕೊಡುತ್ತದೆ, ಕೇವಲ ಸೂಚನೆಗಳಿಗೆ ಪ್ರತಿಕ್ರಿಯಿಸುವ ಬದಲು.

### ಟೂಲ್ ವಿವರಣೆಗಳು ಪ್ರಮುಖವಾಗಿವೆ

ನಿಮ್ಮ ಟೂಲ್ ವಿವರಣೆಗಳ ಗುಣಮಟ್ಟ ನೇರವಾಗಿ ಏಜೆಂಟ್ ಅವುಗಳನ್ನು ಹೇಗೆ ಬಳಸುತ್ತದೆ ಎಂಬುದನ್ನು ಪ್ರಭಾವಿಸುತ್ತದೆ. ಸ್ಪಷ್ಟ, ನಿರ್ದಿಷ್ಟ ವಿವರಣೆಗಳು ಮodelಗೆ ಯಾವಾಗ ಮತ್ತು ಹೇಗೆ ಪ್ರತಿಯೊಂದು ಟೂಲನ್ನು ಕರೆಸಬೇಕೆಂದು ಅರ್ಥಮಾಡಿಕೊಳ್ಳಲು ಸಹಾಯ ಮಾಡುತ್ತದೆ.

### ಸೆಷನ್ ನಿರ್ವಹಣೆ

`@MemoryId` ನಿರೂಪಣೆ ಸ್ವಯಂಚಾಲಿತ ಸೆಷನ್ ಆಧಾರಿತ ಮೆಮೊರಿ ನಿರ್ವಹಣೆಯನ್ನು ಸಕ್ರಿಯಗೊಳಿಸುತ್ತದೆ. ಪ್ರತಿ ಸೆಷನ್ ಐಡಿ ತನ್ನದೇ ಆದ `ChatMemory` ಉದಾಹರಣೆಯನ್ನೂ `ChatMemoryProvider` ಬೀನಿನಿಂದ ನಿರ್ವಹಿಸಲಾಗುತ್ತದೆ,multiple ಬಳಕೆದಾರರು ಏಜೆಂಟ್ ಜೊತೆ ಒಂದೇ ವೇಳೆ ಸಂವಹನ ನಡೆಸಿದರೂ ಅವರ ಸಂಭಾಷಣೆಗಳು ಮಿಶ್ರವಾಗುವುದಿಲ್ಲ. ಕೆಳಗಿನ ಚಿತ್ರೀಕರಣ ಒಂದೇ ಭಾಗದ ಮೆಮೊರಿ ಸಂಗ್ರಹಗಳಿಗೆ ಎಷ್ಟು ಬಳಕೆದಾರರು ತಮ್ಮ ಸೆಷನ್ IDಗಳ ಆಧಾರದಲ್ಲಿ ಮಾರ್ಗದರ್ಶನ ಮಾಡಲ್ಪಡುತ್ತಾರೆ ಎಂಬುದನ್ನು ತೋರಿಸುತ್ತದೆ:

<img src="../../../translated_images/kn/session-management.91ad819c6c89c400.webp" alt="@MemoryId ನೊಂದಿಗೆ ಸೆಷನ್ ನಿರ್ವಹಣೆ" width="800"/>

*ಪ್ರತಿ ಸೆಷನ್ ಐಡಿ ವಿಭಿನ್ನ ಸಂಭಾಷಣೆಯ ಇತಿಹಾಸಕ್ಕೆ ನಕ್ಷೆ ಮಾಡುತ್ತದೆ — ಬಳಕೆದಾರರು ಪರಸ್ಪರ ಸಂದೇಶಗಳನ್ನು ಎಂದಿಗೂ ನೋಡುವುದಿಲ್ಲ.*

### ದೋಷ ನಿರ್ವಹಣೆ

ಟೂಲ್ಸ್ ವಿಫಲವಾಗಬಹುದು — APIs ಸಮಯದೊಳಗೆ ಪ್ರತಿಕ್ರಿಯಿಸದೇ ಇರಬಹುದು, ಪ್ಯಾರಾಮೀಟರ್ಗಳು ಅನೈತಿಕವಾಗಿರಬಹುದು, ಬಾಹ್ಯ ಸೇವೆಗಳು ಡೌನ್ ಆಗಿರಬಹುದು. ಉತ್ಪಾದನಾ ಏಜೆಂಟ್‌ಗಳಿಗೆ ದೋಷ ನಿರ್ವಹಣೆ ಅವಶ್ಯಕ, ಹಾಗಾಗಿ ಮಾದರಿ ಸಮಸ್ಯೆಗಳನ್ನು ವಿವರಿಸಬಹುದು ಅಥವಾ ಅಭಿವೃದ್ಧಿ ಸ್ಥಗಿತಗೊಳ್ಳುವ ಬದಲು ಪರ್ಯಾಯಗಳನ್ನು ಪ್ರಯತ್ನಿಸಬಹುದು. ಟೂಲ್ ಯಾವುದಾದರೂ ಅಪवादವನ್ನು.throw ಮಾಡುತ್ತದಾದರೆ, LangChain4j ಅದನ್ನು ಹಿಡಿದು ಆ ಬಗ್ಗೆ ದೋಷ ಸಂದೇಶವನ್ನು ಮಾದರಿಗೆ ನೀಡುತ್ತದೆ, ಮಾದರಿ ಬnatural ಭಾಷೆಯಲ್ಲಿ ಸಮಸ್ಯೆಯನ್ನು ವಿವರಿಸಬಹುದು.

## ಲಭ್ಯವಿರುವ ಟೂಲ್ಸ್

ಕೆಳಗಿನ ಚಿತ್ರವು ನೀವು ನಿರ್ಮಿಸಬಹುದಾದ ವ್ಯಾಪಕ ಟೂಲ್ಸ್ ಹಬ್ಬವನ್ನು ತೋರಿಸುತ್ತದೆ. ಈ ಮಾಡ್ಯೂಲ್ ವಾತಾವರಣ ಮತ್ತು ತಾಪಮಾನ ಟೂಲ್‌ಗಳನ್ನು ಪ್ರದರ್ಶಿಸುತ್ತದೆ, ಆದರೆ ಅದೇ `@Tool` ಹೂಡುಿಕೆಯು ಯಾವುದೇ ಜಾವಾ ವಿಧಾನಕ್ಕೆ ಅನ್ವಯಿಸುತ್ತದೆ — ಡೇಟಾಬೇಸ್ ವಿಚಾರಣೆಗಳಿಂದ ಶುರುವಾಗಿಸಿ ಪಾವತಿ ಪ್ರಕ್ರಿಯೆಗಳು ವರೆಗೆ.

<img src="../../../translated_images/kn/tool-ecosystem.aad3d74eaa14a44f.webp" alt="ಟೂಲ್ ಹಬ್ಬ" width="800"/>

*ಯಾವುದೇ ಜಾವಾ ವಿಧಾನವನ್ನು @Tool ನೊಂದಿಗೆ ಗುರುತಿಸಿದರೆ ಅದು AIಗೆ ಲಭ್ಯವಾಗುತ್ತದೆ — ಈ ಮಾದರಿ ಡೇಟಾಬೇಸ್‌ಗಳು, APIಗಳು, ಇಮೇಲ್, ಫೈಲ್ ಕಾರ್ಯಚಟುವಟಿಕೆಗಳು ಮತ್ತು ಇನ್ನಷ್ಟು ಕಡೆಗೆ ವಿಸ್ತರಿತವಾಗಿದೆ.*

## ಟೂಲ್ ಆಧಾರಿತ ಏಜೆಂಟ್‌ಗಳನ್ನು ಯಾವಾಗ ಉಪಯೋಗಿಸಬೇಕು

ಪ್ರತಿ ವಿನಂತಿಗೂ ಟೂಲ್ಗಳು ಅಗತ್ಯವಿರುವುದಿಲ್ಲ. ನಿರ್ಧಾರವು AIಗೆ ಹೊರಗಿನ ವ್ಯವಸ್ಥೆಗಳೊಂದಿಗೆ ಸಂವಹನ ಮಾಡಬೇಕೇ ಅಥವಾ ಅದು ತನ್ನ ಸ್ವಂತ ಜ್ಞಾನದಾಧಾರದಿಂದ ಉತ್ತರಿಸಬಹುದೇ ಎಂಬುದರ ಮೇಲೆ ಅವಲಂಬಿತವಾಗಿದೆ. ಕೆಳಗಿನ ಮಾರ್ಗದರ್ಶಕವು ಯಾವಾಗ ಟೂಲ್ಸ್ ಮೌಲ್ಯ ಹೊಂದುತ್ತವೆ ಮತ್ತು ಯಾವಾಗ ಅವು ಅನಗತ್ಯವಾಗುತ್ತವೆಯೆಂಬುದನ್ನು ಸಾರುತ್ತದೆ:

<img src="../../../translated_images/kn/when-to-use-tools.51d1592d9cbdae9c.webp" alt="ಟೂಲ್ಸ್ ಬಳಕೆ ಸಮಯ" width="800"/>

*ತ್ವರಿತ ನಿರ್ಧಾರ ಮಾರ್ಗದರ್ಶಕ — ಟೂಲ್ಸ್ ನೈಜ-ಸಮಯದ ಡೇಟಾ, ಗಣನೆಗಳು ಮತ್ತು ಕ್ರಮಗಳಿಗಾಗಿ; ಸಾಮಾನ್ಯ ಜ್ಞಾನ ಮತ್ತು ಸೃಜನಾತ್ಮಕ ಕಾರ್ಯಗಳಿಗೆ ಅವು ಅಗತ್ಯವಿಲ್ಲ.*

## ಟೂಲ್ಸ್ ಮತ್ತು RAG

ಮಾಡ್ಯೂಲ್‌ಗಳು 03 ಮತ್ತು 04 ಎರಡೂ AIಗೆ ಸಾಮರ್ಥ್ಯವನ್ನು ವಿಸ್ತರಿಸುತ್ತವೆ, ಆದರೆ ಮೂಲಭೂತವಾಗಿ ವಿಭಿನ್ನ ರೀತಿಗಳಲ್ಲಿ. RAG ಡಾಕ್ಯುಮೆಂಟ್‌ಗಳನ್ನು ತರಲು ಮಾದರಿಗೆ ಜ್ಞಾನವನ್ನು ಒದಗಿಸುತ್ತದೆ. ಟೂಲ್ಸ್ ಕಾರ್ಯಗಳನ್ನು ಕೈಗೊಳ್ಳಲು ಫಂಕ್ಷನ್‌ಗಳನ್ನು ಕಾಲ್ ಮಾಡುವ ಮೂಲಕ ಮಾದರಿಗೆ ಸಾಮರ್ಥ್ಯ ನೀಡುತ್ತವೆ. ಕೆಳಗಿನ ಚಿತ್ರವು ಈ ಎರಡು ವಿಧಾನಗಳನ್ನು ಎದುರಾಗಿ ಹೋಲಿಕೆ ಮಾಡುತ್ತದೆ — ಪ್ರತಿ ಕೆಲಸ ಹೇಗೆ ನಡೆದುಕೊಂಡು ಹೋದೀತೋ ಹಾಗೂ ಅವುಗಳ ನಡುವಿನ ವ್ಯವಹಾರಗಳೇನು ಎಂಬುದನ್ನು:

<img src="../../../translated_images/kn/tools-vs-rag.ad55ce10d7e4da87.webp" alt="ಟೂಲ್ಸ್ ಮತ್ತು RAG ಹೋಲಿಕೆ" width="800"/>

*RAG ಸ್ಥಿರ ಡಾಕ್ಯುಮೆಂಟ್‌ಗಳಿಂದ ಮಾಹಿತಿಯನ್ನು ಪಡೆದುಕೊಳ್ಳುತ್ತದೆ — ಟೂಲ್ಸ್ ಕ್ರಮಗಳನ್ನು ನಿರ್ವಹಿಸಿ ಡೈನಾಮಿಕ್, ನೈಜ-ಸಮಯ ಡೇಟಾವನ್ನು ತರಲು ಸಹಾಯ ಮಾಡುತ್ತದೆ. ಹಲವಾರು ಉತ್ಪಾದನಾ ವ್ಯವಸ್ಥೆಗಳು ಎರಡನ್ನೂ ಸಂಯೋಜಿಸುತ್ತವೆ.*

ಪ್ರಾಯೋಗಿಕವಾಗಿ, ಬಹು ಉತ್ಪಾದನಾ ವ್ಯವಸ್ಥೆಗಳು ಎರಡೂ ವಿಧಾನಗಳನ್ನು ಸಂಯೋಜಿಸುತ್ತವೆ: RAG ನಿಮ್ಮ ದಸ್ತಾವೇಜುಗಳಲ್ಲಿ ಉತ್ತರಗಳನ್ನು ನೆಟ್ಟಿಗೊಳಿಸಲು, ಮತ್ತು ಟೂಲ್ಸ್ ನೈಜ-ಸಮಯ ಡೇಟಾ ತರಲು ಅಥವಾ ಕಾರ್ಯಗಳನ್ನು ನಿರ್ವಹಿಸಲು.
