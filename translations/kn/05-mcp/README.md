## ನೀವು ಏನು ಕಲಿತೀರಿ

ನೀವು ಸಂಭಾಷಣಾತ್ಮಕ AI ನಿರ್ಮಿಸಿದ್ದೀರಿ, ಪ್ರಾಂಪ್ಟ್‌ಗಳನ್ನು ನಿಪುಣವಾಗಿ ಬಳಸಿದ್ದೀರಿ, ಉತ್ತರಗಳನ್ನು ಡಾಕ್ಯುಮೆಂಟ್ಗಳಲ್ಲಿ ನೆಲೆಯಿಟ್ಟಿದ್ದೀರಿ ಮತ್ತು ಉಪಕರಣಗಳೊಂದಿಗೆ ಏಜೆಂಟ್‌ಗಳನ್ನು ಸೃಷ್ಟಿಸಿದ್ದೀರಿ. ಆದರೆ ಈ ಎಲ್ಲಾ ಉಪಕರಣಗಳು ನಿಮ್ಮ ನಿರ್ದಿಷ್ಟ ಅನ್ವಯಿಕೆಗೆ ಕಸ್ಟಮ್ ರೀತಿ ನಿರ್ಮಿಸಲ್ಪಟ್ಟಿದ್ದವು. AI ಗೆ ಪ್ರತಿಯೊಬ್ಬರೂ ಸೃಷ್ಟಿಸಬಹುದು ಮತ್ತು ಹಂಚಿಕೊಳ್ಳಬಹುದು ಎಂಬ ಮಾನಕ ಉಪಕರಣಗಳ ಪರಿಕರ ವ್ಯವಸ್ಥೆಗೆ ಪ್ರಾಪ್ತಿಯಾಗುವಂತೆ ಮಾಡಬೇಕಾದರೆ? ಈ ಮಾಯ್ದಲಿನಲ್ಲಿ, ನೀವು ಅದನ್ನು ಮಾಡುವುದು ಹೇಗೆ ಎಂಬುದನ್ನು Model Context Protocol (MCP) ಮತ್ತು LangChain4j ನ agentic module ಮೂಲಕ ಕಲಿತೀರಿ. ಮೊದಲು ಸರಳ MCP ಫೈಲ್ ರೀಡರ್ ಪ್ರದರ್ಶಿಸಿ ನಂತರ ಅದು ಸುಲಭವಾಗಿ ಎಡ್ವಾನ್ಸ್ ಏಜೆಂಟಿಕ್ ಕೆಲಸನಡಿಗೆ ಅಳವಡಿಸುವುದನ್ನು Supervisor Agent ಮಾದರಿಯನ್ನು ಬಳಸಿ ತೋರಿಸಲಾಗುತ್ತದೆ.

## MCP ಎಂದರೇನು?

Model Context Protocol (MCP) ನೀವು ಬೇಕಾದ ವೈಶಿಷ್ಟ್ಯಗಳನ್ನು ನಿಖರವಾಗಿ ಒದಗಿಸುತ್ತದೆ - AI ಅಪ್ಲಿಕೇಶನ್‌ಗಳಿಗೆ ಬಾಹ್ಯ ಉಪಕರಣಗಳನ್ನು ಕಂಡುಹಿಡಿಯಲು ಮತ್ತು ಬಳಸಲು ಸರಳ ಮಾನಕ ವಿಧಾನ. ಪ್ರತಿ ಡೇಟಾ ಮೂಲ ಅಥವಾ ಸೇವೆಗೆ ಕಸ್ಟಮ್ ಇಂಟಿಗ್ರೇಷನ್ ಬದಲಾಗಿ, ನೀವು ನಿರಂತರವಾಗಿ ಸಾಮರ್ಥ್ಯಗಳನ್ನು ನೀಡುವ MCP ಸರ್ವರ್‌ಗಳಿಗೆ ಸಂಪರ್ಕ ಮಾಡುತ್ತೀರಿ. ನಿಮ್ಮ AI ಏಜೆಂಟ್ ನಂತರ ಸ್ವಯಂಚಾಲಿತವಾಗಿ ಈ ಉಪಕರಣಗಳನ್ನು ಕಂಡುಹಿಡಿದು ಬಳಸಬಹುದು.

ಕೆಳಗಿನ ಚಿತ್ರದಲ್ಲಿ ಬೇಧವನ್ನು ತೋರಿಸಲಾಗಿದೆ — MCP ಇಲ್ಲದೆ ಪ್ರತಿ ಇಂಟಿಗ್ರೇಷನ್ ಕಸ್ಟಮ್ ಪಾಯಿಂಟ್-ಟು-ಪಾಯಿಂಟ್ ವೈಯಾ ಕಾರ್ಜುತ್ತದೆ; MCP ಮೂಲಕ, ಒಂದೇ ಪ್ರೋಟೋಕಾಲ್ ನಿಮ್ಮ ಅಪ್ಲಿಕೇಶನ್ ಅನ್ನು ಯಾವುದೇ ಉಪಕರಣಕ್ಕೆ ಸಂಪರ್ಕಿಸುತ್ತದೆ:

<img src="../../../translated_images/kn/mcp-comparison.9129a881ecf10ff5.webp" alt="MCP ಹೋಲಿಕೆ" width="800"/>

*ಮುಂದೆ MCP: ಸಂಕೀರ್ಣ ಪಾಯಿಂಟ್-ಟು-ಪಾಯಿಂಟ್ ಇಂಟಿಗ್ರೇಷನ್ಗಳು. MCP ನಂತರ: ಒಬ್ಬ ಪ್ರೋಟೋಕಾಲ್, ಅನಂತ ಸಾಧ್ಯತೆಗಳು.*

MCP AI ಅಭಿವೃದ್ಧಿಯಲ್ಲಿ ಮೂಲಭೂತ ಸಮಸ್ಯೆಗೆ ಪರಿಹಾರ ನೀಡುತ್ತದೆ: ಪ್ರತಿ ಇಂಟಿಗ್ರೇಷನ್ ಕಸ್ಟಮ್ ಆಗಿರುತ್ತದೆ. GitHub ಪ್ರವೇಶಿಸಲು ಬಯಸುವಿರಾ? ಕಸ್ಟಮ್ ಕೋಡ್ ಅಗತ್ಯ. ಫೈಲ್ ಓದಲು ಬಯಸುವಿರಾ? ಕಸ್ಟಮ್ ಕೋಡ್ ಅಗತ್ಯ. ಡೇಟಾಬೇಸ್ ಪರಿಶೀಲಿಸಲು ಬಯಸುವಿರಾ? ಕಸ್ಟಮ್ ಕೋಡ್ ಅಗತ್ಯ. ಮತ್ತು ಈ ಇಂಟಿಗ್ರೇಷನ್ಗಳು ಇತರ AI ಅಪ್ಲಿಕೇಶನ್‌ಗಳಿಗೆ ಸೂಕ್ತವಾಗಿಲ್ಲ.

MCP ಇದನ್ನು ಮಾನಕಗೊಳಿಸುತ್ತದೆ. MCP ಸರ್ವರ್ ಸ್ಪಷ್ಟ ವಿವರಣೆ ಮತ್ತು ಸ್ಕೀಮಾ‌ಗಳೊಂದಿಗೆ ಉಪಕರಣಗಳನ್ನು ಒದಗಿಸುತ್ತದೆ. ಯಾವುದೇ MCP ಕ್ಲೈಂಟ್ ಸಂಪರ್ಕಿಸಿ, ಲಭ್ಯವಿರುವ ಉಪಕರಣಗಳನ್ನು ಕಂಡುಹಿಡಿದು ಬಳಸಬಹುದು. ಒಂದು ಬಾರಿ ನಿರ್ಮಿಸಿ ಎಲ್ಲೆಡೆ ಬಳಸಿರಿ.

ಕೆಳಗಿನ ಚಿತ್ರ ಇದು ವಿನ್ಯಾಸವನ್ನು ಪ್ರದರ್ಶಿಸುತ್ತದೆ — ಒಂದು MCP ಕ್ಲೈಂಟ್ (ನಿಮ್ಮ AI ಅಪ್ಲಿಕೇಶನ್) ನಾನಾ MCP ಸರ್ವರ್‌ಗಳಿಗೆ ಸಂಪರ್ಕಿಸುತ್ತವೆ, ಪ್ರತಿಯೊಂದು ತಮ್ಮ ಸ್ವಂತ ಉಪಕರಣಗಳ ಸೆಟ್ ಅನ್ನು ಮಾನಕ ಪ್ರೋಟೋಕಾಲ್ ಮೂಲಕ ಒದಗಿಸುತ್ತದೆ:

<img src="../../../translated_images/kn/mcp-architecture.b3156d787a4ceac9.webp" alt="MCP ವಿನ್ಯಾಸ" width="800"/>

*ಮಾದರಿ_Context ಪ್ರೋಟೋಕಾಲ್ ವಿನ್ಯಾಸ - ಮಾನಕಗೊಂಡ ಉಪಕರಣ ಕಂಡುಹಿಡಿಯೋ ಮತ್ತು ಕಾರ್ಯಾಚರಣೆ*

## MCP ಹೇಗೆ ಕಾರ್ಯ ನಿರ್ವಹಿಸುತ್ತದೆ

ಹೆಸರುಕೆಳಗೆ, MCP ಪದರಗಟ್ಟಿದ ವಿನ್ಯಾಸವನ್ನು ಬಳಸುತ್ತದೆ. ನಿಮ್ಮ Java ಅಪ್ಲಿಕೇಶನ್ (MCP ಕ್ಲೈಂಟ್) ಲಭ್ಯವಿರುವ ಉಪಕರಣಗಳನ್ನು ಕಂಡುಹಿಡಿದು, JSON-RPC ಬೇಡಿಕೆಗಳನ್ನು ಟ್ರಾನ್ಸ್‌ಪೋರ್ಟ್ ಪದರ (Stdio ಅಥವಾ HTTP) ಮೂಲಕ ಕಳುಹಿಸುತ್ತದೆ, ಮತ್ತು MCP ಸರ್ವರ್ ಕಾರ್ಯಾಚರಣೆಗಳನ್ನು ಕಾರ್ಯಗತಗೊಳಿಸಿ ಫಲಿತಾಂಶಗಳನ್ನು ತರುತ್ತದೆ. ಕೆಳಗಿನ ಚಿತ್ರವು ಪ್ರೋಟೋಕಾಲ್ ಪ್ರತಿ ಪದರವನ್ನು ವಿವರಿಸುತ್ತದೆ:

<img src="../../../translated_images/kn/mcp-protocol-detail.01204e056f45308b.webp" alt="MCP ಪ್ರೋಟೋಕಾಲ್ ವಿವರ" width="800"/>

*ಹೆಸರುಕೆಳಗೆ MCP ಹೇಗೆ ಕಾರ್ಯ ನಿರ್ವಹಿಸುತ್ತದೆ - ಕ್ಲೈಂಟ್ ಉಪಕರಣಗಳನ್ನು ಕಂಡುಹಿಡಿಯುತ್ತದೆ, JSON-RPC ಸಂದೇಶ ವಿನಿಮಯ ಮಾಡುತ್ತದೆ, ಮತ್ತು ಟ್ರಾನ್ಸ್‌ಪೋರ್ಟ್ ಪದರದಿಂದ ಕಾರ್ಯಗಳನ್ನು ಕಾರ್ಯಗತಗೊಳಿಸುತ್ತದೆ.*

**ಸರ್ವರ್-ಕ್ಲೈಂಟ್ ವಿನ್ಯಾಸ**

MCP ಕ್ಲೈಂಟ್-ಸರ್ವರ್ ಮಾದರಿಯನ್ನು ಉಪಯೋಗಿಸುತ್ತದೆ. ಸರ್ವರ್‌ಗಳು ಉಪಕರಣಗಳನ್ನು ಒದಗಿಸುತ್ತದೆ - ಫೈಲ್ ಓದುವುದು, ಡೇಟಾಬೇಸ್ ವಿಚಾರಣೆ, API ಕರೆಗಳು. ಕ್ಲೈಂಟ್‌ಗಳು (ನಿಮ್ಮ AI ಅಪ್ಲಿಕೇಶನ್) ಸರ್ವರ್‌ಗಳಿಗೆ ಸಂಪರ್ಕ ಮಾಡಿ ಆ ಉಪಕರಣಗಳನ್ನು ಬಳಸುತ್ತಾರೆ.

LangChain4j ನಲ್ಲಿ MCP ಬಳಕೆಗೆ ಈ Maven ಅವಲಂಬನೆಯನ್ನು ಸೇರಿಸಿ:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```

**ಉಪಕರಣ ಕಂಡುಹಿಡಿಯೋಣೆ**

ನಿಮ್ಮ ಕ್ಲೈಂಟ್ MCP ಸರ್ವರ್‌ಗಳಿಗೆ ಸಂಪರ್ಕಿಸಿದಾಗ, ಅದು ಕೇಳುತ್ತದೆ "ನಿಮ್ಮ ಬಳಿ ಯಾವ ಉಪಕರಣಗಳಿವೆ?" ಸರ್ವರ್ ಲಭ್ಯವಿರುವ ಉಪಕರಣಗಳ ಪಟ್ಟಿ, ಆವೃತ್ತಿ ವಿವರಣೆ ಮತ್ತು ಪ್ಯಾರಾಮೀಟರ್ ಸ್ಕೀಮಾ‌ಗಳೊಂದಿಗೆ ಉತ್ತರಿಸುತ್ತದೆ. ನಿಮ್ಮ AI ಏಜೆಂಟ್ ನಂತರ ಬಳಕೆದಾರರ ವಿನಂತಿಗಳ ಆಧಾರದ ಮೇಲೆ ಉಪಕರಣಗಳನ್ನು ಬಳಸಲು ತೀರ್ಮಾನ ಮಾಡಬಹುದು. ಕೆಳಗಿನ ಚಿತ್ರವು ಈ ಹ್ಯಾಂಡ್ಷೇಕ್ ಅನ್ನು ತೋರಿಸುತ್ತದೆ — ಕ್ಲೈಂಟ್ `tools/list` ವಿನಂತಿಯನ್ನು ಕಳುಹಿಸಿ ಸರ್ವರ್ ತನ್ನ ಲಭ್ಯ ಉಪಕರಣಗಳನ್ನು ವಿವರಣೆ ಮತ್ತು ಪ್ಯಾರಾಮೀಟರ್ ಸ್ಕೀಮಾ‌ಗಳೊಂದಿಗೆ ಪ್ರತಿಕ್ರಿಯಿಸುತ್ತದೆ:

<img src="../../../translated_images/kn/tool-discovery.07760a8a301a7832.webp" alt="MCP ಉಪಕರಣ ಕಂಡುಹಿಡಿಯೋಣೆ" width="800"/>

*AI ಕಾರ್ಯಾರಂಭದಲ್ಲಿ ಲಭ್ಯವಿರುವ ಉಪಕರಣಗಳನ್ನು ಕಂಡುಹಿಡಿಯುತ್ತದೆ — ಈಗ ಯಾವ ಸಾಮರ್ಥ್ಯಗಳು ಲಭ್ಯವಿವೆ ಎಂದು ತಿಳಿದು ಅವುಗಳನ್ನು ಬಳಸಲು ತೀರ್ಮಾನಿಸಬಹುದು.*

**ಟ್ರಾನ್ಸ್‌ಪೋರ್ಟ್ ವಿಧಾನಗಳು**

MCP ವಿವಿಧ ಟ್ರಾನ್ಸ್‌ಪೋರ್ಟ್ ವಿಧಾನಗಳನ್ನು ಬೆಂಬಲಿಸುತ್ತದೆ. ಎರಡು ಆಯ್ಕೆಗಳು Stdio (ಸ್ಥಳೀಯ ಉಪಕ್ರಿಯೆಗಳ ಸಂವಹನಕ್ಕಾಗಿ) ಮತ್ತು Streamable HTTP (ದೂರದ ಸರ್ವರ್‌ಗಳು لاءِ). ಈ ಮಾಯ್ದಲಿನಲ್ಲಿ Stdio ಪ್ರಯೋಗ ಮಾಡಲಾಗಿದೆ:

<img src="../../../translated_images/kn/transport-mechanisms.2791ba7ee93cf020.webp" alt="ಟ್ರಾನ್ಸ್‌ಪೋರ್ಟ್ ವಿಧಾನಗಳು" width="800"/>

*MCP ಟ್ರಾನ್ಸ್‌ಪೋರ್ಟ್ ವಿಧಾನಗಳು: ದೂರದ ಸರ್ವರ್‌ಗಳಿಗೆ HTTP, ಸ್ಥಳೀಯ ಪ್ರಕ್ರಿಯೆಗಳಿಗೆ Stdio*

ಸ್ಥಳೀಯ ಪ್ರಕ್ರಿಯೆಗಳಿಗಾಗಿ. ನಿಮ್ಮ ಅಪ್ಲಿಕೇಶನ್ ಸರ್ವರ್ ಅನ್ನು ಉಪಪ್ರಕ್ರಿಯೆಯಾಗಿ ಸ್ಫೂರ್ತಿಮಾಡಿ, ಸಾಮಾನ್ಯ ಇನ್‌ಪುಟ್/ಔಟ್‌ಪುಟ್ ಮೂಲಕ ಸಂವಹನ ನಡೆಸುತ್ತದೆ. ಫೈಲ್‌ಸಿಸ್ಟ್‌ಮ್ ಪ್ರವೇಶ ಅಥವಾ ಕಮಾಂಡ್ ಲೈನ್ ಉಪಕರಣಗಳಿಗೆ ಉಪಯುಕ್ತ.

```java
McpTransport stdioTransport = new StdioMcpTransport.Builder()
    .command(List.of(
        npmCmd, "exec",
        "@modelcontextprotocol/server-filesystem@2025.12.18",
        resourcesDir
    ))
    .logEvents(false)
    .build();
```

`@modelcontextprotocol/server-filesystem` ಸರ್ವರ್ ಕೆಳಗಿನ ಉಪಕರಣಗಳನ್ನು ಒದಗಿಸುತ್ತದೆ, ಎಲ್ಲವು ನೀವು ಸೂಚಿಸಿದ ಡೈರೆಕ್ಟರಿಗಳಲ್ಲಿ ಸ್ಯಾಂಡ್‌ಬಾಕ್ಸ್ ಮಾಡಲ್ಪಟ್ಟಿವೆ:

| ಉಪಕರಣ | ವಿವರಣೆ |
|------|-------------|
| `read_file` | ಒಂದು ಫೈಲ್‌ನ ವಿಷಯವನ್ನು ಓದಿ |
| `read_multiple_files` | ಒಂದೇ ಕಾಲ್‌ನಲ್ಲಿ ಹಲವಾರು ಫೈಲ್‌ಗಳನ್ನು ಓದಿ |
| `write_file` | ಫೈಲ್ ರಚಿಸು ಅಥವಾ ಮರುಬರೆಯು |
| `edit_file` | ಗುರಿ-ನಿಷ್ಠ ಹುಡುಕಿ-ಮರೆಯುವ ಸಂಪಾದನೆ ಮಾಡಿ |
| `list_directory` | ದಾರಿಯಲ್ಲಿ ಫೈಲ್‌ಗಳು ಮತ್ತು ಡೈರೆಕ್ಟರಿಗಳನ್ನು ಪಟ್ಟಿ ಮಾಡು |
| `search_files` | ಮಾದರಿಯನ್ನು ಹೊಂದಿರುವ ಫೈಲ್‌ಗಳನ್ನು ಪುನರಾವರ್ತಿತವಾಗಿ ಹುಡುಕು |
| `get_file_info` | ಫೈಲ್ ಮೆಟಾದತ್ತ (ಆಕಾರ, ಸಮಯಮುದ್ರೆಗಳು, ಅನುಮತಿಗಳು) ಪಡೆಯು |
| `create_directory` | ಒಂದು ಡೈರೆಕ್ಟರಿ ರಚಿಸು (ವರುಣಿ ಡೈರೆಕ್ಟರಿಗಳ ಸಹಿತ) |
| `move_file` | ಫೈಲ್ ಅಥವಾ ಡೈರೆಕ್ಟರಿಯನ್ನು ತೆರವುಗೊಳಿಸು ಅಥವಾ ಹೆಸರಿಸು |

ಕೆಳಗಿನ ಚಿತ್ರವು ರನ್‌ಟೈಮ್‌ನಲ್ಲಿ Stdio ಟ್ರಾನ್ಸ್‌ಪೋರ್ಟ್ ಹೇಗೆ ಕೆಲಸ ಮಾಡುತ್ತದೆ ಎಂಬುದನ್ನು ತೋರುತ್ತದೆ — ನಿಮ್ಮ Java ಅಪ್ಲಿಕೇಶನ್ MCP ಸರ್ವರ್ ಅನ್ನು ಶಿಶು ಪ್ರಕ್ರಿಯೆಯಾಗಿ ಸೃಷ್ಟಿಸಿ stdin/stdout ಪೈಪ್ಗಳ ಮೂಲಕ ಸಂವಹನ ನಡೆಸುತ್ತದೆ, ಯಾವುದೇ ನೆಟ್‌ವರ್ಕ್ ಅಥವಾ HTTP ಭಾಗವಹಿಸುವುದಿಲ್ಲ:

<img src="../../../translated_images/kn/stdio-transport-flow.45eaff4af2d81db4.webp" alt="Stdio ಟ್ರಾನ್ಸ್‌ಪೋರ್ಟ್ ಪ್ರವಾಹ" width="800"/>

*Stdio ಟ್ರಾನ್ಸ್‌ಪೋರ್ಟ್ ಚಾಲನೆಯಲ್ಲಿ — ನಿಮ್ಮ ಅಪ್ಲಿಕೇಶನ್ MCP ಸರ್ವರ್ ಅನ್ನು ಶಿಶು ಪ್ರಕ್ರಿಯೆಯಾಗಿ ಸೃಷ್ಟಿಸುತ್ತದೆ ಮತ್ತು stdin/stdout ಪೈಪ್ಗಳ ಮೂಲಕ ಸಂವಹನ ಮಾಡುತ್ತದೆ.*

> - "LangChain4j MCP ಸರ್ವರ್ ಪ್ರಕ್ರಿಯೆಗಳ ಜೀವನಚರಿತ್ರೆಯನ್ನು ಹೇಗೆ ನಿರ್ವಹಿಸುತ್ತದೆ?"
> - "AI ಗೆ ಫೈಲ್ ಸಿಸ್ಟಂ ಪ್ರವೇಶ ಮಂಜೂರು ಮಾಡುವ ಸಮಯದ ಭದ್ರತಾ ಪರಿಣಾಮಗಳು ಗಳು ಯಾವುವು?"

## ಏಜೆಂಟಿಕ್ Module

MCP ಮಾನಕೃತ ಉಪಕರಣಗಳನ್ನು ಒದಗಿಸುವಾಗ, LangChain4j ನ **agentic module** ಆ ಉಪಕರಣಗಳನ್ನು ಆಯೋಜಿಸುವ ಏಜೆಂಟ್‌ಗಳನ್ನು ಘೋಷಣಾತ್ಮಕ ರೀತಿಯಲ್ಲಿ ನಿರ್ಮಿಸುವ ವಿಧಾನವನ್ನು ಒದಗಿಸುತ್ತದೆ. `@Agent` ವಿಶೇಷಣ ಮತ್ತು `AgenticServices` ಮೂಲಕ ನೀವು ಕ್ರಮಾನುಗತ ಸಂಕೇತ ಬದಲು ಇಂಟರ್ಫೇಸ್ಗಳ ಮೂಲಕ ಏಜೆಂಟ್ ವರ್ತನೆಗಳನ್ನು ವ್ಯಾಖ್ಯಾನ ಮಾಡಬಹುದು.

ಈ ಮಾಯ್ದಲಲ್ಲಿ, ನೀವು **ಸರಪಳೆಯ ನಾಯಕ ಏಜೆಂಟ್** ಮಾದರಿಯನ್ನು ಅನ್ವೇಷಿಸುವಿರಿ — ಇದು ಉನ್ನತತರ ಏಜೆಂಟಿಕ್ AI ವಿಧಾನ, ಇಲ್ಲಿಗೆ “ಸರಪಳೆಯ ನಾಯಕ” ಏಜೆಂಟ್ ಉಪಯೋಗಕರ ವಿನಂತಿಗಳ ಆಧಾರದಲ್ಲಿ ಉಪ-ಏಜೆಂಟ್‌ಗಳನ್ನು ಡೈನಾಮಿಕ್ ಆಗಿ ಕರೆಯಲು ತೀರ್ಮಾನಿಸುತ್ತದೆ. ನಾವು ಎರಡನ್ನೂ ಸಂಯೋಜಿಸಿ, ನಮ್ಮ ಉಪ-ಏಜೆಂಟ್‌ಗಳಲ್ಲಿ ಒಂದಿಗೆ MCP ಚಾಲಿತ ಫೈಲ್ ಪ್ರವೇಶ ಸಾಮರ್ಥ್ಯವನ್ನು ನೀಡುತ್ತೇವೆ.

ಏಜೆಂಟಿಕ್_Module ಬಳಸಲು, ಈ Maven ಅವಲಂಬನೆಯನ್ನು ಸೇರಿಸಿ:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```
> **ಟಿಪ್ಪಣಿ:** `langchain4j-agentic` ಮಾಯ್ದಲವು ಬೇರೆ ವೇಳಾಪಟ್ಟಿಯಲ್ಲಿ ಬಿಡುಗಡೆಯಾಗುವುದರಿಂದ `langchain4j.mcp.version` ಎಂಬ ಪ್ರತ್ಯೇಕ ಆವೃತ್ತಿ ಗುಣಲಕ್ಷಣವನ್ನು ಬಳಸುತ್ತದೆ.

> **⚠️ ಪ್ರಯೋಗಾತ್ಮಕ:** `langchain4j-agentic` ಮಾಯ್ದಲವು **ಪ್ರಯೋಗಾತ್ಮಕ** ಆಗಿದ್ದು ಬದಲಾವಣೆಗಳಿಗೆ ಒಳಪಟ್ಟಿದೆ. AI ಸಹાયકರನ್ನು ನಿರ್ಮಿಸಲು ಸ್ಥಿರ ವಿಧಾನ `langchain4j-core` ಮತ್ತು ಕಸ್ಟಮ್ ಉಪಕರಣಗಳನ್ನು ಬಳಸುವ  (Module 04) ಆಗಿ ಉಳಿದಿದೆ.

### ಸರಪಳೆಯ ನಾಯಕ ಏಜೆಂಟ್

**ಸರಪಳೆಯ ನಾಯಕ ಏಜೆಂಟ್ ಮಾದರಿ** ಒಂದು **ಸ್ಥಿರಸ್ಥಿತಿಯಾಗಿರುವ** ಏಜೆಂಟಿಕ್ AI ರೂಪವಾಗಿದೆ. ಸರಪಳೆಯ ನಾಯಕ ಬಳಕೆದಾರ ವಿನಂತಿಯ ಆಧಾರದ ಮೇಲೆ LLM ಬಳಸಿ ಸ್ವತಃ ತೀರ್ಮಾನಿಸಿ ಯಾವ ಏಜೆಂಟ್‌ಗಳನ್ನು ಕರೆಯಬೇಕೆಂದು ನಿರ್ಧರಿಸುತ್ತಾನೆ. ಮುಂದಿನ ಉದಾಹರಣೆಯಲ್ಲಿ, ನಾವು MCP ಚಾಲಿತ ಫೈಲ್ ಪ್ರವೇಶವನ್ನು LLM ಏಜೆಂಟ್ ಜೊತೆಗೆ ಸಂಯೋಜಿಸಿ, ಮೇಲ್ವಿಚಾರಿತ ಫೈಲ್ ಓದುವುದು → ವರದಿ ಸೃಷ್ಟಿಸುವ ಕಾರ್ಯನಡಿಗೆ ಯನ್ನು ತಯಾರಿಸುತ್ತೇವೆ.

ಡೆಮೋದಲ್ಲಿ, `FileAgent` MCP ಫೈಲ್ ಸಿಸ್ಟಮ್ ಉಪಕರಣಗಳನ್ನು ಬಳಸಿ ಫೈಲ್ ಓದುತ್ತದೆ, ಮತ್ತು `ReportAgent` ನಿರ್ವಹಣಾ ಸಾರಾಂಶ (ಒಂದು ವಾಕ್ಯ), 3 ಮುಖ್ಯ ಅಂಶಗಳು ಮತ್ತು ಶಿಫಾರಸುಗಳೊಂದಿಗೆ ಸಂರಚಿತ ವರದಿಗಳನ್ನು ತಯಾರಿಸುತ್ತದೆ. ಸರಪಳೆಯ ನಾಯಕ ಈ ಕಾರ್ಯ ಸಂಘಟನೆ ಸ್ವಯಂಚಾಲಿತವಾಗಿ ಮಾಡುತ್ತಾನೆ:

<img src="../../../translated_images/kn/supervisor-agent-pattern.06275a41ae006ac8.webp" alt="ಸರಪಳೆಯ ನಾಯಕ ಏಜೆಂಟ್ ಮಾದರಿ" width="800"/>

*ಸರಪಳೆಯ ನಾಯಕ ತನ್ನ LLM ಬಳಸಿ ಯಾವ ಏಜೆಂಟ್‌ಗಳನ್ನು ಯಾವ ಕ್ರಮದಲ್ಲಿ ಕರೆಯಬೇಕೆಂದು ನಿರ್ಧರಿಸುತ್ತದೆ — ನಿಖರ ಮಾರ್ಗದರ್ಶನ ಅಗತ್ಯವಿಲ್ಲ.*

ನಮ್ಮ ಫೈಲ್-ದಿಂದ-ವರದಿ ಕಾರ್ಯನಡಿಗೆಯ ಸ್ಪಷ್ಟ ಚಿತ್ರಣವು ಹೀಗೆ ಕಾಣುತ್ತದೆ:

<img src="../../../translated_images/kn/file-report-workflow.649bb7a896800de9.webp" alt="ಫೈಲ್-ದಿಂದ-ವರದಿ ಕಾರ್ಯನಡಿಗೆ" width="800"/>

*FileAgent MCP ಉಪಕರಣಗಳ ಮೂಲಕ ಫೈಲ್ ಓದುತ್ತದೆ, ನಂತರ ReportAgent ಅಸಂರಚಿತ ವಿಷಯವನ್ನು ಸಂರಚಿತ ವರದಿಯಲ್ಲಿ ಪರಿವರ್ತಿಸುತ್ತದೆ.*

ಕೆಳಗಿನ ಕ್ರಮ ಚಿತ್ತರವು ಪೂರ್ಣ ಸರಪಳೆಯ ನಾಯಕರ ಸಂಯೋಜನೆಯನ್ನು ತೋರಿಸುತ್ತದೆ — MCP ಸರ್ವರ್ ಸ್ಫೂರ್ತಿಗೊಳಿಸುವುದು, ಸ್ವಯಂಚಾಲಿತ ಏಜೆಂಟ್ ಆಯ್ಕೆ ಸರಪಳೆಯ ನಾಯಕನಿಂದ, stdio ಮೂಲಕ ಉಪಕರಣ ಕರೆ ಮತ್ತು ಅಂತಿಮ ವರದಿ:

<img src="../../../translated_images/kn/supervisor-agent-sequence.1aa389b3bef99956.webp" alt="ಸರಪಳೆಯ ನಾಯಕ ಕ್ರಮ ಚಿತ್ತರ" width="800"/>

*ಸರಪಳೆಯ ನಾಯಕ ಸ್ವತಃ FileAgent (ಮಾಧ್ಯಮವಾಗಿ MCP ಸರ್ವರ್‌ಗೆ stdio ಮೂಲಕ ಕರೆ ಮಾಡಿ ಫೈಲ್ ಓದುವುದು) ಅನ್ನು ಕರೆಯುತ್ತಾನೆ, ಬಳಿಕ ReportAgent ಅನ್ನು ಕರೆಮಾಡಿ ಸಂರಚಿತ ವರದಿ ತಯಾರಿಸುತ್ತಾನೆ — ಪ್ರತಿಯೊಬ್ಬ ಏಜೆಂಟ್ ತನ್ನ ಔಟ್‌ಪುಟ್ ಅನ್ನು ಹಂಚಲಾದ Agentic Scope ನಲ್ಲಿ ಸಂಗ್ರಹಿಸುತ್ತದೆ.*

ಪ್ರತಿ ಏಜೆಂಟ್ ತನ್ನ ಔಟ್‌ಪುಟ್ ಅನ್ನು **Agentic Scope** (ಹಂಚಲ್ಪಟ್ಟ ಮೆಮೊರಿ) ಯಲ್ಲಿ ಸಂಗ್ರಹಿಸುತ್ತದೆ, ಇದು ಅನುಕ್ರಮ ಏಜೆಂಟ್‌ಗಳಿಗೆ ಹಿಂದಿನ ಫಲಿತಾಂಶಗಳಿಗೆ ಪ್ರವೇಶ ನೀಡಲು ಅನುಕೂಲವಾಗುತ್ತದೆ. ಇದು MCP ಉಪಕರಣಗಳು ಏಜೆಂಟಿಕ್ ಕೆಲಸನಡಿಗೆ ಸುಗಮವಾಗಿ ಅಳವಡಿಕೆಯಾಗುವುದನ್ನು ತೋರಿಸುತ್ತದೆ — ಸರಪಳೆಯ ನಾಯಕ ಫೈಲ್ ಹೇಗೆ ಓದಲಾಗುತ್ತದೆ ಎಂಬುದನ್ನು ತಿಳಿದುಕೊಳ್ಳಬೇಕಿಲ್ಲ, صرف FileAgent ಇದು ಮಾಡಬಹುದು ಎಂದಷ್ಟೆ ತಿಳಿಯಲು ಸಾಕು.

#### ಸರಪಳೆಯ ನಾಯಕ ಹೇಗೆ ಕೆಲಸ ಮಾಡುತ್ತಾನೆ

ಏಜೆಂಟ್ ನಿರ್ಮಿಸುವ ಮೊದಲು, MCP ಟ್ರಾನ್ಸ್‌ಪೋರ್ಟ್ ಅನ್ನು ಕ್ಲೈಂಟ್‌ಗೆ ಸಂಪರ್ಕಿಸಿ, ಅದನ್ನು `ToolProvider` ಆಗಿ ಲೇಪಿಸಬೇಕು. MCP ಸರ್ವರ್ ಉಪಕರಣಗಳು ನಿಮಗೆ ಲಭ್ಯವಾಗುವ ವಿಧಾನ ಈ ಕೆಳಗಿನಂತಿದೆ:

```java
// ಸಾರಿಗೆಂದಿಂದ MCP ಕ್ಲೈಂಟ್ ಸೃಷ್ಟಿ ಮಾಡಿ
McpClient mcpClient = new DefaultMcpClient.Builder()
        .transport(stdioTransport)
        .build();

// ಕ್ಲೈಂಟನ್ನು ToolProvider ಆಗಿ ತಿರುವುಮಾಡಿ — ಇದು MCP ಉಪಕರಣಗಳನ್ನು LangChain4j ಗೆ ಸಂಪರ್ಕಿಸುತ್ತದೆ
ToolProvider mcpToolProvider = McpToolProvider.builder()
        .mcpClients(List.of(mcpClient))
        .build();
```

ಈಗ ನೀವು ಯಾವುದೇ ಏಜೆಂಟ್‌ಗೆ MCP ಉಪಕರಣಗಳ ಅಗತ್ಯವಿದ್ದರೆ `mcpToolProvider` ಅನ್ನು ಇಂಜೆಕ್ಟ್ ಮಾಡಬಹುದು:

```java
// ಹಂತ 1: FileAgent MCP ಸಾಧನಗಳನ್ನು ಬಳಸಿ ಕಡತಗಳನ್ನು ಓದುತ್ತದೆ
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // ಕಡತ ಕಾರ್ಯಾಚರಣೆಗಳಿಗಾಗಿ MCP ಸಾಧನಗಳಿವೆ
        .build();

// ಹಂತ 2: ReportAgent ರಚನೆಯೊಂದಿಗಿನ ವರದಿಗಳನ್ನು ತಯಾರಿಸುತ್ತದೆ
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// Supervisor ಕಡತ → ವರದಿ ಕೆಲಸಸರಣಿಯನ್ನು ನಿಯಂತ್ರಿಸುತ್ತದೆ
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // ಕೊನೆಗಿನ ವರದಿಯನ್ನು ಮರಳಿ ನೀಡುತ್ತದೆ
        .build();

// ವಿನಂತಿಯ ಆಧಾರದ ಮೇಲೆ ಯಾರನ್ನು ಆವರ್ತಿಸಬೇಕೆಂದು Supervisor ತೀರ್ಮಾನಿಸುವದು
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```

#### FileAgent_runtime_ನಲ್ಲಿ MCP ಉಪಕರಣಗಳನ್ನು ಹೇಗೆ ಕಂಡುಹಿಡಿಯುತ್ತದೆ

ನೀವುಕೇಳಬಹುದು: **`FileAgent` npm ಫೈಲ್‌ಸಿಸ್ಟ್‌ಮ್ ಉಪಕರಣಗಳನ್ನು runtime ನಲ್ಲಿ ಹೇಗೆ ಬಳಸುವುದನ್ನು ತಿಳಿದುಕೊಳ್ಳುತ್ತದೆ?** ಉತ್ತರವೆಂದರೆ ಅದು ತಿಳಿದುಕೊಳ್ಳಲ್ಲ — **LLM** ಉಪಕರಣ ಸ್ಕೀಮಾದ ಮೂಲಕ runtime ನಲ್ಲಿ ಅದನ್ನು ನಿರ್ಧರಿಸುತ್ತದೆ.
`FileAgent` ಇಂಟರ್ಫೇಸ್ ಕೇವಲ ಒಂದು **ಪ್ರಾಂಪ್ಟ್ ವ್ಯಾಖ್ಯಾನ** ಮಾತ್ರ. ಇದರಲ್ಲಿ `read_file`, `list_directory` ಅಥವಾ ಯಾವುದೇ ಇತರ MCP ಯಂತ್ರಗಳ ಬಗ್ಗೆ ಹಾರ್ಡ್ಕೋಡೆಡ್ ಜ್ಞಾನವಿಲ್ಲ. ಇಲ್ಲಿ ಕೊನೆಯವರೆಗೆ ಸಂಭವಿಸುವುದು ಹೀಗಿದೆ:

1. **ಸರ್ವರ್ ಶುರುಮಾಡುವುದು:** `StdioMcpTransport` `@modelcontextprotocol/server-filesystem` npm ಪ್ಯಾಕೇಜ್ ಅನ್ನು ಚಿಂತಕ ಪ್ರಕ್ರಿಯಾ ರೂಪದಲ್ಲಿ ಲಾಂಚ್ ಮಾಡುತ್ತದೆ  
2. **ಯಂತ್ರ ಅನ್ವೇಷಣೆ:** `McpClient` సర్వర్ కు `tools/list` JSON-RPC ವಿನಂತಿಯನ್ನು ಕಳುಹಿಸುತ್ತದೆ, ಅದು ಯಂತ್ರದ ಹೆಸರುಗಳು, ವಿವರಣೆಗಳು ಮತ್ತು ಪ್ಯಾರಾಮೀಟರ್ ಸ್ಕೀಮಾಗಳೊಂದಿಗೆ ಪ್ರತಿಕ್ರಿಯಿಸುತ್ತದೆ (ಉದಾ: `read_file` — *"ಒಂದು ಫೈಲ್‌ನ ಪೂರ್ಣ ವಿಷಯಗಳನ್ನು ಓದಿ"* — `{ path: string }`)  
3. **ಸ್ಕೀಮಾ ಇಂಜೆಕ್ಷನ್:** `McpToolProvider` ಇತ್ತೀಚೆಗೆ ಕಂಡುಹಿಡಿದಿರುವ ಈ ಸ್ಕೀಮಾಗಳನ್ನು ಸುತ್ತುವರಿದಂತೆ ಮಾಡಿದರೆ ಮತ್ತು ಅವುಗಳನ್ನು LangChain4j ಗೆ ಲಭ್ಯಮಾಡಿಸಿ  
4. **LLM ನಿರ್ಧರಿಸುತ್ತದೆ:** `FileAgent.readFile(path)` ಕರೆ ಮಾಡಿದಾಗ, LangChain4j ಸಿಸ್ಟಮ್ ಸಂದೇಶ, ಬಳಕೆದಾರ ಸಂದೇಶ, **ಮತ್ತು ಯಂತ್ರ ಸ್ಕೀಮಾಗಳ ಪಟ್ಟಿ** ಅನ್ನು LLM ಗೆ ಕಳುಹಿಸುತ್ತದೆ. LLM ಯಂತ್ರದ ವಿವರಣೆಗಳನ್ನು ಓದಿ ಯಂತ್ರ ಕರೆ ತಯಾರಿಸುತ್ತದೆ (ಉದಾ: `read_file(path="/some/file.txt")`)  
5. **ಕಾರ್ಯನ್ವಯಿಕೆ:** LangChain4j ಯಂತ್ರ ಕರೆಗಳನ್ನು ಪکڑುತ್ತದೆ, ಮರುಮಾರ್ಗವಾಗಿ MCP ಕ್ಲೈಂಟ್ ಮೂಲಕ Node.js ಉಪಪ್ರಕ್ರಿಯೆಗೆ ಕಳುಹಿಸುತ್ತದೆ, ಫಲಿತಾಂಶ ಪಡೆಯುತ್ತದೆ ಮತ್ತು ಅದನ್ನು LLM ಗೆ ಹಿಂತಿರುಗಿಸುತ್ತದೆ  

ಇದು ಮೇಲ್ಕಾಣಿಸಿದ [ಯಂತ್ರ ಅನ್ವೇಷಣೆ](../../../05-mcp) ಯಂತ್ರಜ್ಞಾನದ ವೇದಿಕೆ ಆಜ್ಞೆಯನ್ನು , ಆದರೆ ಇದನ್ನು ವಿಶೇಷವಾಗಿ ಏಜೆಂಟ್ ಕೆಲಸದ ಸರಣಿಗೆ ಅನ್ವಯಿಸುತ್ತದೆ. `@SystemMessage` ಮತ್ತು `@UserMessage` ಟಿಪ್ಪಣಿಗಳು LLM ನ ವರ್ತನೆಯನ್ನು ಮಾರ್ಗದರ್ಶಿಸುತ್ತವೆ, ಮತ್ತು ಇಂಜೆಕ್ಟ್ ಮಾಡಿದ `ToolProvider` ಅದಕ್ಕೆ **ಸಾಧ್ಯತೆಗಳನ್ನು** ನೀಡುತ್ತದೆ — LLM ಎರಡು ನಡುವಿನ ಸೇತುವೆಯಾಗಿ runtime ನಲ್ಲಿ ಕಾರ್ಯನಿರ್ವಹಿಸುತ್ತದೆ.

#### ಪ್ರತಿಕ್ರಿಯೆ ರಣನೀತಿಗಳು

ನೀವು `SupervisorAgent` ಅನ್ನು ಸಂರಚಿಸುವಾಗ, ಉಪ-ಏಜೆಂಟ್ ಗಳು ಕೆಲಸಗಳನ್ನು ಪೂರ್ಣಗೊಳಿಸಿದ ನಂತರ ಬಳಕೆದಾರಿಗೆ ತನ್ನ ಅಂತಿಮ ಉತ್ತರವನ್ನು ಹೇಗೆ ರೂಪಿಸಬೇಕೆಂದು ನೀವು ನಿರ್ಧರಿಸುತ್ತೀರಿ. ಕೆಳಗಿನ ಚಿತ್ರಾತ್ಮಕವು ಲಭ್ಯವಿರುವ ಮೂರು ತಂತ್ರಗಳನ್ನು ತೋರಿಸುತ್ತದೆ — LAST ನೇರವಾಗಿ ಅಂತಿಮ ಏಜೆಂಟ್ ಔಟ್‍ಪುಟ್‌ನನ್ನೇ ನೀಡುತ್ತದೆ, SUMMARY ಎಲ್ಲಾ ಔಟ್‍ಪುಟ್‍ಗಳನ್ನು LLM ಮೂಲಕ ಸಂಯೋಜಿಸುತ್ತದೆ, ಮತ್ತು SCORED ಮೂಲ ವಿನಂತಿಯ ವಿರುದ್ಧ ಯಾವುದು ಹೆಚ್ಚು ತನ್ನ ಮೌಲ್ಯಮಾಪನ ಸಿಗುತ್ತದೆ ಅದನ್ನು ಆರಿಸುತ್ತದೆ:

<img src="../../../translated_images/kn/response-strategies.3d0cea19d096bdf9.webp" alt="Response Strategies" width="800"/>

*Supervisor ತನ್ನ ಅಂತಿಮ ಪ್ರತಿಕ್ರಿಯೆಯನ್ನು ರೂಪಿಸುವಲ್ಲಿ ಮೂರು ತಂತ್ರಗಳು — ನೀವು ಕೊನೆಯ ಏಜೆಂಟ್ ಔಟ್‍ಪುಟ್, ಸಂಯೋಜಿತ ಸಾರಾಂಶ ಅಥವಾ ಅತ್ಯುತ್ತಮ ಮೌಲ್ಯಮಾಪನ ಆಯ್ಕೆಗಳನ್ನು ಆಯ್ಕೆಮಾಡಬಹುದು.*

| ತಂತ್ರ | ವಿವರಣೆ |
|----------|-------------|
| **LAST** | ಮೇಲ್ವಿಚಾರಕ ಕೊನೆ ಉಪಏಜೆಂಟ್ ಅಥವಾ ಕರೆಯಲಾದ ಯಂತ್ರದ ಔಟ್‍ಪುಟ್ ಅನ್ನು ವಾಪಸು ನೀಡುತ್ತದೆ. ಇದು ಕಾರ್ಯಪಟಲದ ಕೊನೆ ಏಜೆಂಟ್ ಸಂಪೂರ್ಣ, ಅಂತಿಮ ಉತ್ತರವನ್ನು ನೀಡಲು ವಿಶೇಷವಾಗಿ ವಿನ್ಯಾಸಗೊಳ್ಳಿರುವಾಗ ಉಪಯುಕ್ತವಾಗಿದೆ (ಉದಾಹರಣೆ: ಸಂಶೋಧನಾ ಪೈಪ್ಲೈನ್‌ನ "ಸಂಕ್ಷಿಪ್ತ ಏಜೆಂಟ್"). |  
| **SUMMARY** | ಮೇಲ್ವಿಚಾರಕ ತನ್ನ ಸ್ವಂತ ಒಳಗಿನ ಭಾಷಾ ಮಾದರಿಯನ್ನು (LLM) ಉಪಯೋಗಿಸಿಕೊಂಡು ಸಂಪೂರ್ಣ ಸಂವಾದ ಮತ್ತು ಎಲ್ಲಾ ಉಪಎಜೆಂಟ್ ಔಟ್‍ಪುಟ್​ ಗಳ ಸಾರಾಂಶವನ್ನು ಸಂಯೋಜಿಸಿ ಅಂತಿಮ ಪ್ರತಿಕ್ರಿಯೆಯಾಗಿ ತರುವುದು. ಇದು ಬಳಕೆದಾರಿಗೆ ಸ್ವಚ್ಛ, ಸಂಗ್ರಹಿತ ಉತ್ತರವನ್ನು ಒದಗಿಸುತ್ತದೆ. |  
| **SCORED** | ವ್ಯವಸ್ಥೆ ಒಳಗಿನ LLM ಉಪಯೋಗಿಸಿ LAST ಪ್ರತಿಕ್ರಿಯೆ ಮತ್ತು SUMMARY ಅನ್ನು ಮೂಲ ಬಳಕೆದಾರ ವಿನಂತಿಯ ವಿರುದ್ಧ ಮೌಲ್ಯಮಾಪನ ಮಾಡಿ, ಹೆಚ್ಚು ಮೌಲ್ಯ ಪಡೆದ ಔಟ್‍ಪುಟ್ ಅನ್ನು ವಾಪಸು ನೀಡುತ್ತದೆ. |

> - "ಮೇಲ್ವಿಚಾರಕ ಮತ್ತು ಸರಣಿಯಾಗಿರುವ ಕಾರ್ಯವಾಹಿಕೆ ಮಾದರಿಗಳ ನಡುವಿನ ವ್ಯತ್ಯಾಸವೇನು?"  
> - "ನಾನು ಮೇಲ್ವಿಚಾರಕನ ಯೋಜನೆ ವರ್ತನೆಯನ್ನು ಹೇಗೆ ಕಸ್ಟಮೈಸ್ ಮಾಡಬಹುದು?"  

#### ಔಟ್‌ಪುಟ್ ಅನ್ನು ಅರ್ಥಮಾಡಿಕೊಳ್ಳುವುದು

ನೀವು ಡೆಮೊವನ್ನು چلಾಯಿಸಿದಾಗ, ಮೇಲ್ವಿಚಾರಕ ಹಲವಾರು ಏಜೆಂಟ್ ಗಳನ್ನು ಹೇಗೆ ಸಂಯೋಜಿಸುತ್ತಾನೆ ಎಂಬುದರ ಘನವಾದ ನಿರೂಪಣೆಯನ್ನು ನೋಡುತ್ತೀರಿ. ಪ್ರತಿ ವಿಭಾಗದ ಅರ್ಥ ಹೀಗಿದೆ:

```
======================================================================
  FILE → REPORT WORKFLOW DEMO
======================================================================

This demo shows a clear 2-step workflow: read a file, then generate a report.
The Supervisor orchestrates the agents automatically based on the request.
```
  
**ಶೀರ್ಷಿಕೆ** ಕಾರ್ಯವಾಹಿಕೆಯನ್ನು ಪರಿಚಯಿಸುತ್ತದೆ: ಫೈಲ್ ಓದುವುದು ರಿಂದ ವರದಿ ರಚನೆಗೆ ಗಮನ ಕೊಟ್ಟ ಪೈಪ್ಲೈನ್.  

```
--- WORKFLOW ---------------------------------------------------------
  ┌─────────────┐      ┌──────────────┐
  │  FileAgent  │ ───▶ │ ReportAgent  │
  │ (MCP tools) │      │  (pure LLM)  │
  └─────────────┘      └──────────────┘
   outputKey:           outputKey:
   'fileContent'        'report'

--- AVAILABLE AGENTS -------------------------------------------------
  [FILE]   FileAgent   - Reads files via MCP → stores in 'fileContent'
  [REPORT] ReportAgent - Generates structured report → stores in 'report'
```
  
**Workflow Diagram** ಏಜೆಂಟ್ ಗಳ ನಡುವೆ ಡೇಟಾ ಹರಿವು ತೋರಿಸುತ್ತದೆ. ಪ್ರತಿ ಏಜೆಂಟ್ ಗೂ ನಿರ್ದಿಷ್ಟ ಪಾತ್ರವಿದೆ:  
- **FileAgent** MCP ಯಂತ್ರಗಳನ್ನು ಬಳಸಿ ಫೈಲ್ ಅನ್ನು ಓದಿ `fileContent` ನಲ್ಲಿ ಮೂಲ ವಿಷಯವನ್ನು ಸಂಗ್ರಹಿಸುತ್ತದೆ  
- **ReportAgent** ಆ ವಿಷಯವನ್ನು ಬಳಸಿ ರಚನೆಗೊಂಡ ವರದಿಯನ್ನು `report` ಆಗಿ ಉಳಿತಾಯ ಮಾಡುತ್ತದೆ  

```
--- USER REQUEST -----------------------------------------------------
  "Read the file at .../file.txt and generate a report on its contents"
```
  
**ಬಳಕೆದಾರ ವಿನಂತಿ** ಕಾರ್ಯವನ್ನು ತೋರಿಸುತ್ತದೆ. ಮೇಲ್ವಿಚಾರಕ ಇದನ್ನು ಪಾರ್ಸ್ ಮಾಡಿ FileAgent → ReportAgent ಕರೆಯಲು ನಿರ್ಧರಿಸುತ್ತದೆ.  

```
--- SUPERVISOR ORCHESTRATION -----------------------------------------
  The Supervisor decides which agents to invoke and passes data between them...

  +-- STEP 1: Supervisor chose -> FileAgent (reading file via MCP)
  |
  |   Input: .../file.txt
  |
  |   Result: LangChain4j is an open-source, provider-agnostic Java framework for building LLM...
  +-- [OK] FileAgent (reading file via MCP) completed

  +-- STEP 2: Supervisor chose -> ReportAgent (generating structured report)
  |
  |   Input: LangChain4j is an open-source, provider-agnostic Java framew...
  |
  |   Result: Executive Summary...
  +-- [OK] ReportAgent (generating structured report) completed
```
  
**ಮೇಲ್ವಿಚಾರಕ ಸಂಯೋಜನೆ** ಕ್ರಿಯೆಯಲ್ಲಿ 2 ಹಂತಗಳ ಹರಿವನ್ನು ತೋರಿಸುತ್ತದೆ:  
1. **FileAgent** MCP ಮುಖಾಂತರ ಫೈಲ್ ಓದಿ ವಿಷಯವನ್ನು ಸಂಗ್ರಹಿಸುತ್ತದೆ  
2. **ReportAgent** ಆ ವಿಷಯ ಪಡೆದ ಬಳಿಕ ರಚನೆಗೊಂಡ ವರದಿಯನ್ನು ಸೃಷ್ಟಿಸುತ್ತದೆ  

ಮೇಲ್ವಿಚಾರಕ ಈ ನಿರ್ಧಾರಗಳನ್ನು ಬಳಕೆದಾರ ವಿನಂತಿಯ ಆಧಾರದ ಮೇಲೆ **ಸ್ವತಂತ್ರವಾಗಿ** ತೆಗೆದುಕೊಂಡಿದೆ.  

```
--- FINAL RESPONSE ---------------------------------------------------
Executive Summary
...

Key Points
...

Recommendations
...

--- AGENTIC SCOPE (Data Flow) ----------------------------------------
  Each agent stores its output for downstream agents to consume:
  * fileContent: LangChain4j is an open-source, provider-agnostic Java framework...
  * report: Executive Summary...
```
  
#### ಏಜೆಂಟಿಕ್ ಮಿಷನ್ ವೈಶಿಷ್ಟ್ಯಗಳ ವಿವರಣೆ

ಉದಾಹರಣೆ ಏಜೆಂಟಿಕ್ ಮಿಷನ್‌ನ ಹಲವು ಅಭಿವೃದ್ಧಿಶೀಲ ವೈಶಿಷ್ಟ್ಯಗಳನ್ನು ತೋರಿಸುತ್ತದೆ. ಈಗ ಏಜೆಂಟಿಕ್ ವ್ಯಾಪ್ತಿ ಮತ್ತು ಏಜೆಂಟ್ ಶ್ರೋತೃಗಳನ್ನು ಹತ್ತಿರದಿಂದ ನೋಡೋಣ.

**Agentic Scope** `@Agent(outputKey="...")` ಬಳಸಿ ಏಜೆಂಟ್ ಗಳ ಫಲಿತಾಂಶಗಳನ್ನು ಸಂಗ್ರಹಿಸಿದ ಶೇರ್ ಮೆಮೊರಿಯನ್ನು ತೋರಿಸುತ್ತದೆ. ಇದರಿಂದ:  
- ನಂತರದ ಏಜೆಂಟ್ ಗಳು ಮುಂಚಿನ ಏಜೆಂಟ್ ಔಟ್‌ಪುಟ್‌ಗಳನ್ನು ಪ್ರವೇಶಿಸಬಹುದು  
- ಮೇಲ್ವಿಚಾರಕ ಅಂತಿಮ ಪ್ರತಿಕ್ರಿಯೆಯನ್ನು ಸಂಯೋಜಿಸಬಹುದು  
- ನೀವು ಪ್ರತಿ ಏಜೆಂಟ್ ತಯಾರಿಸಿದ ದತವನ್ನು ಪರಿಶೀಲಿಸಬಹುದು  

ಕೆಳಗಿನ ಚಿತ್ರವು agentic scope  ಅನ್ನು ಫೈಲ್-ರಿಂದ-ವರದಿ ಕಾರ್ಯವಾಹಿಕೆಯಲ್ಲಿ ಶೇರ್ ಮೆಮೊರಿ ಆಗಿ ಹೇಗೆ ಕೆಲಸಮಾಡುತ್ತದೆ ಎಂದು ತೋರಿಸುತ್ತದೆ — FileAgent ತನ್ನ ಔಟ್‌ಪುಟ್ ಅನ್ನು `fileContent` ನಲ್ಲಿ ಬರೆಯುತ್ತದೆ, ReportAgent ಅದನ್ನು ಓದಿ `report` ರಚಿಸುತ್ತದೆ:  

<img src="../../../translated_images/kn/agentic-scope.95ef488b6c1d02ef.webp" alt="Agentic Scope Shared Memory" width="800"/>

*Agentic Scope ಶೇರ್ ಮೆಮೊರಿ ರೂಪದಲ್ಲಿ ಕೆಲಸ ಮಾಡುತ್ತದೆ — FileAgent `fileContent` ಬರೆಯುತ್ತದೆ, ReportAgent ಓದಿ `report` ಬರೆಯುತ್ತದೆ, ಮತ್ತು ನಿಮ್ಮ ಕೋಡ್ ಅಂತಿಮ ಫಲಿತಾಂಶವನ್ನು ಓದುತ್ತದೆ.*  

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // ಫೈಲ್ ಏಜೆಂಟ್‌ನಿಂದ ರಾ ಫೈಲ್ ಡೇಟಾ
String report = scope.readState("report");            // ರಿಪೋರ್ಟ್ ಏಜೆಂಟ್‌ನಿಂದ ರಚಿಸಲಾದ ವರದಿ
```
  
**Agent Listeners** ಏಜೆಂಟ್ ಕಾರ್ಯನಿರ್ವಹಣೆಯ ಮೇಲ್ವಿಚಾರಣೆ ಮತ್ತು ಡೀಬಗಿಂಗ್ ಅನ್ನು ಸಾಧ್ಯಮಾಡುತ್ತದೆ. ನೀವು ಡೆಮೊದಲ್ಲಿರುವ ಹಂತ ಹಂತ ಔಟ್‌ಪುಟ್‌ ಅನ್ನು ಪ್ರತಿ ಏಜೆಂಟ್ ಕರೆಗಾಗಿ AgentListener ಇವುಗಳಿಂದ ಪಡೆಯಲಾಗುತ್ತದೆ:  
- **beforeAgentInvocation** - ಮೇಲ್ವಿಚಾರಕ ಏಜೆಂಟ್ ಆಯ್ಕೆಮಾಡಿದಾಗ ಕರೆಮಾಡುತ್ತದೆ, ಯಾವ ಏಜೆಂಟ್ ಆಯ್ಕೆಮಾಡಲ್ಪಟ್ಟಿದೆಯೋ ಮತ್ತು ಏಕೆ ಎಂಬುದನ್ನು ನಿಮಗೆ ತೋರಿಸುತ್ತದೆ  
- **afterAgentInvocation** - ಏಜೆಂಟ್ ಪೂರ್ಣಗೊಂಡಾಗ ಅದರ ಫಲಿತಾಂಶವನ್ನು ತೋರಿಸುತ್ತದೆ  
- **inheritedBySubagents** - ನಿಜವಾಗಿದ್ದರೆ, ಈ ಶ್ರೋತೃHierarchy ಯ ಎಲ್ಲಾ ಏಜೆಂಟ್ ಗಳನ್ನು ಮೇಲ್ವಿಚಾರಣೆ ಮಾಡುತ್ತದೆ  

ಕೆಳಗಿನ ಚಿತ್ರವು ಸಂಪೂರ್ಣ ಏಜೆಂಟ್ ಶ್ರೋತೃ ಜೀವನಚಕ್ರವನ್ನು ತೋರಿಸುತ್ತದೆ, ಜೊತೆಗೆ `onError` ಏಜೆಂಟ್ ಕಾರ್ಯನಿರ್ವಹಣೆಯಲ್ಲಿ ವಿಫಲವಾಗಿ ಆದಾಗ ಯಾವ ರೀತಿ ನಡೆಸಿಕೊಳ್ಳುತ್ತದೆ ಎಂದು ವಿವರಿಸುತ್ತದೆ:  

<img src="../../../translated_images/kn/agent-listeners.784bfc403c80ea13.webp" alt="Agent Listeners Lifecycle" width="800"/>

*Agent Listeners ಕಾರ್ಯನಿರ್ವಹಣೆಯ ಜೀವನಚಕ್ರಕ್ಕೆ ಹೂಕೊಳುತ್ತದೆ — ಏಜೆಂಟ್ ಆರಂಭಿಸಿದಾಗ, ಪೂರ್ಣಗೊಂಡಾಗ ಅಥವಾ ದೋಷ ಎದುರಿಸಿದಾಗ ಮೇಲ್ವಿಚಾರಣೆ ಮಾಡುತ್ತದೆ.*  

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // ಎಲ್ಲಾ ಉಪ-ಏಜೆಂಟ್‌ಗಳಿಗೆ ಪಸರಿಸು
    }
};
```
  
ಮೇಲ್ವಿಚಾರಕ ಮಾದರಿ ಹೊರತು, `langchain4j-agentic` ಮಿಷನ್ ಹಲವು ಶಕ್ತಿಶಾಲಿ ಕಾರ್ಯವಾಹಿಕೆ ಮಾದರಿಗಳನ್ನು ಒದಗಿಸುತ್ತದೆ. ಕೆಳಗಿನ ಚಿತ್ರವು ಎಲ್ಲಾ ಐದು ಮಾದರಿಗಳನ್ನು ತೋರಿಸುತ್ತದೆ — ಸರಳ ಸರಣಿಯಾಗಿರುವ ಪೈಪ್ಲೈನಿನಿಂದ ಮಾನವ-ವ್ಯಾಯಾಮ ವೀಕ್ಷಣೆ ಕೆಲಸದವರೆಗೆ:  

<img src="../../../translated_images/kn/workflow-patterns.82b2cc5b0c5edb22.webp" alt="Agent Workflow Patterns" width="800"/>

* ಏಜೆಂಟ್ ಕಾರ್ಯವಾಹಿಕೆ ಮಾದರಿಗಳ ಐದು ವಿಧಗಳು — ಸರಳ ಸರಣಿಯಾಗಿರುವ ಪೈಪ್ಲೈನಿನಿಂದ ಮಾನವ-ವ್ಯಾಯಾಮ ಅನುಮೋದನೆ ಕಾರ್ಯವರೆಗೆ.*

| ಮಾದರಿ | ವಿವರಣೆ | ಬಳಕೆ ಪ್ರಕರಣ |
|---------|-------------|----------|
| **Sequential** | ಏಜೆಂಟ್ ಗಳನ್ನು ಕ್ರಮವಾಗಿ ಕಾರ್ಯಗತಗೊಳಿಸಿ, ಔಟ್‍ಪುಟ್ ಮುಂದಿನದಕ್ಕೆ ಹರಿಯುತ್ತದೆ | ಪೈಪ್ಲೈನ್ಗಳು: ಸಂಶೋಧನೆ → ವಿಶ್ಲೇಷಣೆ → ವರದಿ |  
| **Parallel** | ಏಜೆಂಟ್ ಗಳನ್ನು ಸಮಕಾಲೀನವಾಗಿ ಕಾರ್ಯಗತಗೊಳಿಸಿ | ಸ್ವತಂತ್ರ ಕರ್ತವ್ಯಗಳು: ಹವಾಮಾನ + ಸುದ್ದಿ + ಸ್ಟಾಕ್ಸ್ |  
| **Loop** | ಶರತ್ತು ಪೂರೈಸುವವರೆಗೆ ಪುನರಾವರ್ತನೆ ಮಾಡಿ | ಗುಣಮಟ್ಟ ಮೌಲ್ಯಮಾಪನ: ಸ್ಕೋರ್ ≥ 0.8 ಆಗುವವರೆಗೆ ಪರಿಶೀಲನೆ |  
| **Conditional** | ಶರತ್ತು ಆಧಾರಿತ ದಾರಿ ನಿರ್ಧಾರ | ವರ್ಗೀಕರಿಸಿ → ತಜ್ಞ ಏಜೆಂಟ್ ಗೆ ಮಾರ್ಗದರ್ಶನ |  
| **Human-in-the-Loop** | ಮಾನವ ಪರಿಶೀಲನಾ ಬಿಂದುಗಳನ್ನು ಸೇರಿಸಿ | ಅನುಮೋದನೆ ಕೆಲಸಗಾರಿಕೆಗಳು, ವಿಷಯ ಪರಿಶೀಲನೆ |  

## ಮುಖ್ಯ ಕಲ್ಪನೆಗಳು

MCP ಮತ್ತು ಏಜೆಂಟಿಕ್ ಮಿಷನ್ ಕಾರ್ಯಾಚರಣೆಯನ್ನು ನೀವು ಅನ್ವೇಷಿಸಿದ ನಂತರ, ಪ್ರತಿ ವಿಧಾನವನ್ನು ಯಾವಾಗ ಬಳಸಬೇಕೆಂದು ಸಂಕಲನ ಮಾಡಿ.

MCP ನ ದೊಡ್ಡ ಒಳ್ಳೆಯದೊಂದು ಅದರ ವೃದ್ಧಿಯಾಗುತ್ತಿರುವ ಪರಂಪರೆ. ಕೆಳಗಿನ ಚಿತ್ರವು ಒಂದೇ ಸಾಮಾನ್ಯ ಪ್ರೋಟೋಕಾಲ್ ನಿಮ್ಮ AI ಅಪ್ಲಿಕೇಶನ್ ಅನ್ನು ವಿಭಿನ್ನ MCP ಸರ್ವರ್ ಗಳುಗಳಿಗೆ ಸಂಪರ್ಕಿಸುವ ವಿಧವನ್ನು ತೋರಿಸುತ್ತದೆ — ಫೈಲ್‌ಸಿಸ್ಟಂ ಮತ್ತು ಡೇಟಾಬೇಸ್ ಪ್ರವೇಶದಿಂದ GitHub, ಇಮೇಲ್, ವೆಬ್ ಸ್ಕ್ರ್ಯಾಪಿಂಗ್ ಮುಂತಾದವರೆಗೆ:  

<img src="../../../translated_images/kn/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="MCP Ecosystem" width="800"/>

*MCP ಒಂದು ಸಾಮಾನ್ಯ ಪ್ರೋಟೋಕಾಲ್ ಪರಂಪರೆಯನ್ನು ಸೃಷ್ಠಿಸುತ್ತದೆ — ಯಾವುದೇ MCP-ಹೊಂದಾಣಿಕೆಯ ಸರ್ವರ್, ಯಾವುದೇ MCP-ಹೊಂದಾಣಿಕೆಯ ಕ್ಲೈಂಟ್ ಜೊತೆಗೆ ಕಾರ್ಯನಿರ್ವಹಿಸುತ್ತದೆ ಹಾಗೂ ಅಪ್ಲಿಕೇಶನ್‌ಗಳಲ್ಲಿ ಯಂತ್ರ ಹಂಚಿಕೆಯನ್ನು ಅನುಮತಿಸುತ್ತದೆ.*  

**MCP** ಆಗಿದ್ದರೆ, ನೀವು ಈಗಾಗಲೇ ಇರುವ ಯಂತ್ರ ಪರಂಪರೆಗಳನ್ನು ಉಪಯೋಗಿಸಲು, ಸಾಕಷ್ಟು ಅನೇಕ ಅಪ್ಲಿಕೇಶನ್ ಗಳಿಗೆ ಹಂಚಿಕೊಳ್ಳಬಹುದಾದ ಯಂತ್ರಗಳನ್ನು ಸಹ ನಿರ್ಮಿಸಲು, ಮೂರನೇ ಪಕ್ಷ ಸೇವೆಗಳನ್ನದೊಂದಿಗೆ ಸಾಮಾನ್ಯ ಪ್ರೋಟೋಕಾಲ್‌ಗಳ ಮೂಲಕ ಸಂಯೋಜಿಸಲು ಅಥವಾ ಕೋಡ್ ಬದಲಾಯಿಸದೆ ಯಂತ್ರಗಳ ಅನುಷ್ಠಾನಗಳನ್ನು ಬದಲಾಯಿಸಲು ಬಯಸುವಾಗ ಇದನ್ನು ಬಳಸಬಹುದು.  

**Agentic Module** ನಿಮ್ಮಿಗೆ `@Agent` ಟಿಪ್ಪಣಿಗಳೊಂದಿಗೆ ಘೋಷಣಾತ್ಮಕ ಏಜೆಂಟ್ ವ್ಯಾಖ್ಯಾನಗಳನ್ನು, ಕಾರ್ಯವಾಹಿಕೆ orchestration (ಕ್ರಮಬದ್ಧ, ಲೂಪ್, ಸಮಕಾಲೀನ), ಅನಿವಾರ್ಯ ಕೋಡ್‌ಗೆ ಬದಲಾಗಿ ಇಂಟರ್ಫೇಸ್ ಆಧಾರಿತ ಏಜೆಂಟ್ ವಿನ್ಯಾಸ, ಅಥವಾ ಹಲವು ಏಜೆಂಟ್ ಗಳ ಔಟ್‌ಪುಟ್ ಶೇರ್ ಮಾಡುವ ಬಗೆಗೆ ಉತ್ತಮವಾಗಿದೆ.  

**Supervisor Agent ಮಾದರಿ** ಮುಂಚಿತವಾಗಿ ಕಾರ್ಯವಾಹಿಕೆ ಪೂರೈಕೆ ನಿರೀಕ್ಷಿತವಾಗದಿರುವ, ಮತ್ತು LLM ತೀರ್ಮಾನ ಮಾಡಬೇಕಾಗಿರುವ, ಹಲವಾರು ವಿಶೇಷ ಏಜೆಂಟ್ ಗಳನ್ನು ಡೈನಾಮಿಕ್ ರೀತಿಯಲ್ಲಿ orchestration ಮಾಡಬೇಕಾಗಿರುವ, ವಿವಿಧ ಸಾಮರ್ಥ್ಯಗಳಿಗೆ ಸಂವಾದ ವ್ಯವಸ್ಥೆಗಳನ್ನು ನಿರ್ಮಿಸುವ ಸಮಸ್ಯೆಗಳು ಇದ್ದಾಗ ಬಹುಮುಖ್ಯವಾಗುತ್ತದೆ; ಹೆಚ್ಚಿನ ಅಡ್ಜಸ್ಟಿವ್ ಏಜೆಂಟ್ ವರ್ತನೆಗೆ.  

Custom `@Tool` ವಿಧಾನಗಳು ಮತ್ತು ಈ ಮಿಷನ್‌ನ MCP ಯಂತ್ರಗಳ ನಡುವಣ ತೀರ್ಮಾನಕ್ಕೆ ಸಹಾಯವಾಗಲು ಕೆಳಗಿನ ಹೋಲಿಕೆ ಪ್ರಮುಖ ವ್ಯತ್ಯಾಸಗಳನ್ನು ಹಿಗ್ಗಿಸುತ್ತದೆ — ಕಸ್ಟಮ್ ಯಂತ್ರಗಳು ಅಪ್ಲಿಕೇಶನ್ ನ ನಿರ್ದಿಷ್ಟ ತರ್ಕಕ್ಕೆ ತುಂಡು ಬದ್ಧತೆ ಮತ್ತು ಸಂಪೂರ್ಣ ಟೈಪ್ ಸುರಕ್ಷತೆ ನೀಡುತ್ತವೆ, MCP ಯಂತ್ರಗಳು ಮಾನಕೀಕೃತ, ಮರುಬಳಕೆಗೋಚಿ ಮಾಡಬಹುದಾದ ಸಂಯೋಜನೆಗಳನ್ನು ಒದಗಿಸುತ್ತವೆ:  

<img src="../../../translated_images/kn/custom-vs-mcp-tools.c4f9b6b1cb65d8a1.webp" alt="Custom Tools vs MCP Tools" width="800"/>

*Custom @Tool ವಿಧಾನಗಳನ್ನು ಮತ್ತು MCP ಯಂತ್ರಗಳನ್ನು ಯಾವಾಗ ಉಪಯೋಗಿಸುವುದು — ಅಡ್ವಾನ್ಸ್ ಅಪ್ಲಿಕೇಶನ್ ತರ್ಕಕ್ಕೆ ಕಸ್ಟಮ್ ಯಂತ್ರಗಳು, ವ್ಯಾಪಕ ಅಪ್ಲಿಕೇಶನ್ಗಳ ನಡುವೆ ಕಾರ್ಯನಿರ್ವಹಿಸುವ ಮಾನಕ ಸಂಯೋಜನೆಗಳಿಗೆ MCP ಯಂತ್ರಗಳು.*  

## ಅಭಿನಂದನೆಗಳು!

ನೀವು LangChain4j for Beginners ಪಾಠಕ್ರಮದ ಐದು ಮಿಷನ್ ಮುಗಿಸಿದ್ದೀರಿ! ಈ ಕೆಳಗಿನ ಚಿತ್ರವು ನೀವು ಪೂರ್ಣಗೊಳಿಸಿದ ಪೂರ್ಣ ಕಲಿಕೆ ಯಾತ್ರಿ ತೋರಿಸುತ್ತದೆ — ಮೂಲಭೂತ ಚಾಟ್ ದಿಂದ MCP-ಪ್ರೇರಿತ ಏಜೆಂಟಿಕ್ ವ್ಯವಸ್ಥೆಗಳವರೆಗೆ:  

<img src="../../../translated_images/kn/course-completion.48cd201f60ac7570.webp" alt="Course Completion" width="800"/>

*ನೀವು ಪೂರೈಸಿದ ಐದು ಮಿಷನ್ ಗಳ ಮೂಲಕ ನಿಮ್ಮ ಕಲಿಕೆ ಪ್ರಯಾಣ.*  

ನೀವು LangChain4j for Beginners ಪಾಠಕ್ರಮವನ್ನು ಪೂರ್ಣಗೊಳಿಸಿದ್ದೀರಿ. ನೀವು ಕಲಿತಿದ್ದೀರಿ:  

- ಮೆಮರಿ ಹೊಂದಿರುವ ಸಂವಾದಾತ್ಮಕ AI ನಿರ್ಮಿಸುವ ವಿಧಾನ (Module 01)  
- ವಿಭಿನ್ನ ಕಾರ್ಯಗಳಿಗೆ ಪ್ರಾಂಪ್ಟ್ ಇಂಜಿನಿಯರಿಂಗ್ ಮಾದರಿಗಳು (Module 02)  
- ನಿಮ್ಮ ದಾಖಲೆಗಳಲ್ಲಿ ಆಧಾರಿತ ಉತ್ತರಗಳನ್ನು RAG ಮೂಲಕ ಸೂಕ್ತಗೊಳಿಸುವುದು (Module 03)  
- ಕಸ್ಟಮ್ ಯಂತ್ರಗಳೊಂದಿಗೆ ಮೂಲ AI ಏಜೆಂಟ್ ಗಳ ರಚನೆ (Module 04)  
- LangChain4j MCP ಮತ್ತು Agentic ಮಿಷನ್‌ಗಳೊಂದಿಗೆ ಮಾನಕ ಯಂತ್ರಗಳ ಸಂಯೋಜನೆ (Module 05)
