## ನೀವು ಕಲಿಯುವುದು

ನೀವು ಸಂಭಾಷಣಾತ್ಮಕ AI ನಿರ್ಮಿಸಿದ್ದೀರಿ, ಪ್ರಾಂಪ್ಟ್ ಗಳನ್ನು ನಿಪುಣರಾದೀರಿ, ಪ್ರತಿಕ್ರಿಯೆಗಳನ್ನು ದಾಖಲೆಗಳ ಮೇಲೆ ಆಧರಿಸಿರುವಿರಿ ಮತ್ತು ಸಾಧನಗಳ ಸಹಿತ ಏಜೆಂಟುಗಳನ್ನು ರಚಿಸಿದ್ದೀರಿ. ಆದರೆ ಆ ಎಲ್ಲ ಸಾಧನಗಳು ನಿಮ್ಮ ನಿರ್ದಿಷ್ಟ ಅಪ್ಲಿಕೇಶನ್ ಗೆ зориулಿ ಕಸ್ಟಮ್ ಆಗಿ ನಿರ್ಮಿಸಲ್ಪಟ್ಟವು. ಯಾರಾದರೂ ಸೃಷ್ಟಿಸಿ ಹಂಚಿಕೊಳ್ಳಬಹುದಾದ ಪ್ರಮಾಣಿತ ಸಾಧನಗಳ ಪರಿಸರಕ್ಕೆ ನಿಮ್ಮ AI ಪ್ರವೇಶ ಹೊಂದಿಸಿದರೆ ಏನಾಗುತ್ತಿತ್ತು? ಈ module ನಲ್ಲಿ ನೀವು Model Context Protocol (MCP) ಮತ್ತು LangChain4j ನ ಏಜೆಂಟಿಕ್ module ನ ಮೂಲಕ ಅದನ್ನೇ ಹೇಗೆ ಮಾಡಬಹುದು ಎಂದು ಕಲಿಯುತ್ತೀರಿ. ಮೊದಲು ಸರಳ MCP ಫೈಲ್ ರೀಡರ್ ನ್ನು ತೋರಿಸುತ್ತೇವೆ ಮತ್ತು ನಂತರ ಅದು ಸುಲಭವಾಗಿ ಸೂಪರ್‌ವೈಸರ್ ಏಜೆಂಟ್ ದ ಶ್ರೀಗಳಾದ ಆಧುನಿಕ ಏಜೆಂಟಿಕ್ ಕಾರ್ಯಪ್ರವಾಹಗಳಿಗೆ ಹೇಗೆ ಸಂಯೋಜಿಸಲ್ಪಟ್ಟಿದೆ ಎಂದು ತೋರಿಸುತ್ತೇವೆ.

## MCP ಎಂದರೇನು?

Model Context Protocol (MCP) ಅದೇನೂ - AI ಅಪ್ಲಿಕೇಶನ್ ಗಾಗಿ ಬಾಹ್ಯ ಸಾಧನಗಳನ್ನು ಕಂಡುಹಿಡಿಯಲು ಮತ್ತು ಉಪಯೋಗಿಸಲು ಪ್ರಮಾಣಿತ ಮಾರ್ಗವನ್ನು ನೀಡುತ್ತದೆ. ಪ್ರತಿ ಡೇಟಾ ಮೂಲ ಅಥವಾ ಸೇವೆಗೆ ಕಸ್ಟಮ್ ಪರಸ್ಪರ ಸಂಪರ್ಕವನ್ನು ಬರೆಯುವ ಬದಲು, ನೀವು MCP ಸರ್ವರ್ ಗಳಿಗೆ ಕೇಳಿ ಅವುಗಳ ಸಾಮರ್ಥ್ಯಗಳನ್ನು ಅಂತಹರूपದಲ್ಲಿ ಬಳಕೆ ಮಾಡಿಕೊಳ್ಳುತ್ತೀರಿ. ನಿಮ್ಮ AI ಏಜೆಂಟ್ ಸ್ವಯಂಚಾಲಿತವಾಗಿ ಈ ಸಾಧನಗಳನ್ನು ಕಂಡುಹಿಡಿದು ಉಪಯೋಗಿಸಬಹುದು.

ಕೆಳಗಿನ ಚಿತ್ರಣವಾದವು MCP ಇಲ್ಲದೆ ಪ್ರತಿಯೊಂದು ಪರಸ್ಪರಸಂಪರ್ಕಕ್ಕೆ ಕಸ್ಟಮ್ ಪಾಯಿಂಟ್-ಟು-ಪಾಯಿಂಟ್ ವೈರ್ ಕಳುಹಿಸುವುದು; MCP ಇದ್ದಲ್ಲಿ ಒಂದು ಪ್ರೋಟೋಕಾಲ್ ನಿಮ್ಮ ಅಪ್ ಅನ್ನು ಯಾವುದೇ ಸಾಧನಕ್ಕೆ ಸಂಪರ್ಕಿಸುತ್ತದೆ:

<img src="../../../translated_images/kn/mcp-comparison.9129a881ecf10ff5.webp" alt="MCP Comparison" width="800"/>

*MCP ಮುಂಚೆ: ಸಂಕೀರ್ಣ ಪಾಯಿಂಟ್-ಟು-ಪಾಯಿಂಟ್ ಇಂಟಿಗ್ರೇಶನ್ಸ್. MCP ನಂತರ: ಒಂದು ಪ್ರೋಟೋಕಾಲ್, ಅನಂತ ಸಾಧ್ಯತೆಗಳು.*

MCP AI ಅಭಿವೃದ್ಧಿಯ ಆಧಾರಭೂತ ಸಮಸ್ಯೆಯನ್ನು ಪರಿಹರಿಸುತ್ತದೆ: ಪ್ರತಿಯೊಂದು ಇಂಟಿಗ್ರೇಶನ್ ಕಸ್ಟಮ್ ಆಗಿದೆ. GitHub ಗೆ ಪ್ರವೇಶಿಸಲು ಬಯಸುವಿರಾ? ಕಸ್ಟಮ್ ಕೋಡ್. ಕಡತಗಳನ್ನು ಓದಲು ಬಯಸುವಿರಾ? ಕಸ್ಟಮ್ ಕೋಡ್. ಡೇಟಾಬೇಸ್ ಅನ್ನು ವಿಚಾರಿಸುವುದಕ್ಕೆ? ಕಸ್ಟಮ್ ಕೋಡ್. ಮತ್ತು ಈ ಎಲ್ಲ ಇಂಟಿಗ್ರೇಶನ್ಗಳು ಇತರ AI ಅಪ್ಲಿಕೇಶನ್ ಗಳೊಂದಿಗೆ ಕೆಲಸ ಮಾಡವುದು ಇಲ್ಲ.

MCP ಇದನ್ನು ಪ್ರಮಾಣೀಕೃತವಾಗಿಯೇ ಮಾಡುತ್ತದೆ. MCP ಸರ್ವರ್ ಸ್ಪಷ್ಟ ವಿವರಣೆಗಳು ಮತ್ತು ಸ್ಕೀಮಾಗಳೊಂದಿಗೆ ಸಾಧನಗಳನ್ನು ತೆರೆಗೆ ಹಾಕುತ್ತದೆ. ಯಾವುದೇ MCP ಕ್ಲೈಂಟ್ ಸಂಪರ್ಕಿಸಿಕೊಂಡು ಲಭ್ಯವಿರುವ ಸಾಧನಗಳನ್ನು ಕಂಡುಹಿಡಿದು ಬಳಸಬಹುದು. ಒಂದೇ ಬಾರಿ ನಿರ್ಮಿಸಿ ಎಲ್ಲೆಡೆ ಬಳಸಿರಿ.

ಕೆಳಗಿನ ಚಿತ್ರಣ ಈ ವಿನ್ಯಾಸವನ್ನು ತೋರಿಸುತ್ತದೆ — ಒಂದೇ MCP ಕ್ಲೈಂಟ್ (ನಿಮ್ಮ AI ಅಪ್ಲಿಕೇಶನ್) ಹಲವು MCP ಸರ್ವರ್ಗಳಿಗೆ ಸಂಪರ್ಕಿಸುತ್ತದೆ, ಪ್ರತಿ ಸರ್ವರ್ ತಮ್ಮ ಸಾಧನಗಳ ಸಮೂಹವನ್ನು ಪ್ರಮಾಣಿತ ಪ್ರೋಟೋಕಾಲ್ ಮೂಲಕ ಬಹಿರಂಗಪಡಿಸುತ್ತದೆ:

<img src="../../../translated_images/kn/mcp-architecture.b3156d787a4ceac9.webp" alt="MCP Architecture" width="800"/>

*Model Context Protocol ವಿನ್ಯಾಸ - ಪ್ರಮಾಣಿತ ಸಾಧನ ಕಂಡುಹಿಡಿತ ಮತ್ತು ಕಾರ್ಯಗತಗೊಳಿಸುವಿಕೆ*

## MCP ಹೇಗೆ ಕಾರ್ಯನಿರ್ವಹಿಸುತ್ತದೆ

ಮೂಳೆಯಲ್ಲಿ, MCP ಪದರರೀತಿಯ ವಿನ್ಯಾಸವನ್ನು ಬಳಸುತ್ತದೆ. ನಿಮ್ಮ Java ಅಪ್ಲಿಕೇಶನ್ (MCP ಕ್ಲೈಂಟ್) ಲಭ್ಯವಿರುವ ಸಾಧನಗಳನ್ನು ಕಂಡುಹಿಡಿದು, JSON-RPC ವಿನಂತಿಗಳನ್ನು ಸಂಚಾರ ಪದರ (Stdio ಅಥವಾ HTTP) ಮೂಲಕ ಕಳುಹಿಸುತ್ತದೆ, ಮತ್ತು MCP ಸರ್ವರ್ ಕಾರ್ಯಾಚರಣೆಗಳನ್ನು ಕಾರ್ಯಗತಗೊಳಿಸಿ ಫಲಿತಾಂಶಗಳನ್ನು ಹಿಂತಿರುಗಿಸುತ್ತದೆ. ಕೆಳಗಿನ ಚಿತ್ರಣ ಈ ಪ್ರೋಟೋಕಾಲ್ ನ ಪ್ರತಿಯೊಂದು ಪದರವನ್ನು ವಿಭಜಿಸುತ್ತದೆ:

<img src="../../../translated_images/kn/mcp-protocol-detail.01204e056f45308b.webp" alt="MCP Protocol Detail" width="800"/>

*ಮೂಳೆಯಲ್ಲಿ MCP ಹೇಗೆ ಕಾರ್ಯನಿರ್ವಹಿಸುತ್ತದೆ — ಕ್ಲೈಂಟ್ಗಳು ಸಾಧನಗಳನ್ನು ಕಂಡುಹಿಡಿದು, JSON-RPC ಸಂದేశಗಳನ್ನು ವಿನಿಮಯಮಾಡಿ, ಮತ್ತು ಸಂಚಾರ ಪದರದಿಂದ ಕಾರ್ಯಗಳನ್ನು ಕಾರ್ಯಗతಗೊಳಿಸುತ್ತವೆ.*

**ಸರ್ವರ್-ಕ್ಲೈಂಟ್ ವಿನ್ಯಾಸ**

MCP ಕ್ಲೈಂಟ್-ಸರ್ವರ್ ಮಾದರಿಯನ್ನು ಬಳಸಿ. ಸರ್ವರ್ಗಳು ಸಾಧನಗಳನ್ನು ಒದಗಿಸುತ್ತವೆ - ಕಡತಗಳನ್ನು ಓದುವುದು, ಡೇಟಾಬೇಸ್ ವಿಚಾರಿಸುವುದು, API ಗಳಿಗೆ ಕಾಲ್ ಮಾಡುವುದು. 클ೈಂಟ್ ಗಳು (ನಿಮ್ಮ AI ಅಪ್ಲಿಕೇಶನ್) ಸರ್ವರ್ಗಳಿಗೆ ಸಂಪರ್ಕಿಸಿ ಅವುಗಳ ಸಾಧನಗಳನ್ನು ಉಪಯೋಗಿಸುತ್ತವೆ.

LangChain4j ಜೊತೆ MCP ಬಳಸಲು, ಈ Maven ಅವಲಂಬನೆಯನ್ನು ಸೇರಿಸಿ:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```


**ಸಾಧನ ಕಂಡುಹಿಡಿತ**

ನಿಮ್ಮ 클ೈಂಟ್ MCP ಸರ್ವರ್ ಗೆ ಸಂಪರ್ಕಿಸಿದಾಗ, ಅದು "ನಿಮ್ಮ ಬಳಿ ಯಾವ ಸಾಧನಗಳಿವೆ?" ಎಂದು ಕೇಳುತ್ತದೆ. ಸರ್ವರ್ ಲಭ್ಯವಿರುವ ಸಾಧನಗಳ ಪಟ್ಟಿಯನ್ನು ವಿವರಣೆ ಹಾಗೂ ಪರಿಮಾಣ ಸ್ಕೀಮಾ ಗಳೊಂದಿಗೆ ಪೂರೈಸುತ್ತದೆ. ನಿಮ್ಮ AI ಏಜೆಂಟ್ ಆ ಬಳಕೆದಾರ ಪ್ರಾರ್ಥನೆ ಆಧಾರವಾಗಿ ಯಾವ ಸಾಧನಗಳನ್ನು ಉಪಯೋಗಿಸಬೇಕೆಂದು ನಿರ್ಧರಿಸಬಹುದು. ಕೆಳಗಿನ ಚಿತ್ರಣ ಈ ವಿವರವನ್ನು ತೋರಿಸುತ್ತದೆ — ಕ್ಲೈಂಟ್ `tools/list` ವಿನಂತಿ ಕಳುಹಿಸುತ್ತದೆ ಮತ್ತು ಸರ್ವರ್ ತನ್ನ ಸಾಧನಗಳು ಹಾಗೂ ವಿವರಣೆ ಮತ್ತು ಪರಿಮಾಣ ಸ್ಕೀಮಾ ಗಳನ್ನು ಹಿಂತಿರುಗಿಸುತ್ತದೆ:

<img src="../../../translated_images/kn/tool-discovery.07760a8a301a7832.webp" alt="MCP Tool Discovery" width="800"/>

*AI ಪ್ರಾರಂಭದಲ್ಲಿ ಲಭ್ಯವಿರುವ ಸಾಧನಗಳನ್ನು ಕಂಡುಕೊಳ್ಳುತ್ತದೆ — ಇವುಗಳ ಸಾಮರ್ಥ್ಯಗಳನ್ನು ಈಗ ಇತ್ತಿಚೆಗೆ ತಿಳಿದುಕೊಳ್ಳುತ್ತದೆಯೆಂದು ನಿರ್ಧರಿಸುತ್ತದೆ.*

**ಸಂವಹನ ವಿಧಾನಗಳು**

MCP ವಿಭಿನ್ನ ಸಂವಹನ ವಿಧಾನಗಳನ್ನು ಬೆಂಬಲಿಸುತ್ತದೆ. ಎರಡು ಆಯ್ಕೆಗಳು Stdio (ಸ್ಥಳೀಯ ಉಪಪ್ರಕ್ರಿಯಾ ಸಂವಹನಕ್ಕೆ) ಮತ್ತು ಸ್ಟ್ರೀಮಬಲ್ HTTP (ದೂರದ ಸರ್ವರ್ಗಳಿಗೆ) ಆಗಿವೆ. ಈ module ನಲ್ಲಿ Stdio ಸಂಚಾರ ಉದಾಹರಿಸುವುದು:

<img src="../../../translated_images/kn/transport-mechanisms.2791ba7ee93cf020.webp" alt="Transport Mechanisms" width="800"/>

*MCP ಸಂಚಾರ ವಿಧಾನಗಳು: ದೂರದ ಸರ್ವರ್ಗಳಿಗೆ HTTP, ಸ್ಥಳೀಯ ಪ್ರಕ್ರಿಯೆಗಳಿಗಾಗಿ Stdio*

**Stdio** - [StdioTransportDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java)

ಸ್ಥಳೀಯ ಪ್ರಕ್ರಿಯೆಗಳಿಗಾಗಿ. ನಿಮ್ಮ ಅಪ್ಲಿಕೇಶನ್ ಸರ್ವರ್ ಅನ್ನು ಉಪಪ್ರಕ್ರಿಯೆಯಾಗಿ ಆರಂಭಿಸಿ ಸ್ಟ್ಯಾಂಡರ್ಡ್ ಮಾಹಿತಿ/ಪ್ರತಿಕ್ರಿಯೆ ಮೂಲಕ ಸಂವಹನ ಮಾಡುತ್ತದೆ. ಕಡತ ವ್ಯವಸ್ಥೆ ಪ್ರವೇಶ ಅಥವಾ ಕಮಾಂಡ್ ಲೈನ್ ಸಾಧನಗಳಿಗೆ ಉಪಯುಕ್ತ.

```java
McpTransport stdioTransport = new StdioMcpTransport.Builder()
    .command(List.of(
        npmCmd, "exec",
        "@modelcontextprotocol/server-filesystem@2025.12.18",
        resourcesDir
    ))
    .logEvents(false)
    .build();
```


`@modelcontextprotocol/server-filesystem` ಸರ್ವರ್ ಕೆಳಗಿನ ಸಾಧನಗಳನ್ನು ಒದಗಿಸುತ್ತದೆ, ನಿಮ್ಮ ಸೂಚಿಸಿದ ಡೈರೆಕ್ಟರಿಗಳಿಗೆ ಕಡು ನಿರ್ಬಂಧಿತ:

| ಸಾಧನ | ವಿವರಣೆ |
|------|-------------|
| `read_file` | ಒಂಬತ್ತು ಒಂದೇ ಕಡತದ ವಿಷಯವನ್ನು ಓದಿ |
| `read_multiple_files` | ಒಂಬತ್ತು ಹಲವಾರು ಕಡತಗಳನ್ನು ಒಂದೇ ಪ್ರಕ್ರಿಯೆಯಲ್ಲಿ ಓದಿ |
| `write_file` | ಕಡತವನ್ನು ರಚಿಸಿ ಅಥವಾ ಮರುಬರೆವು |
| `edit_file` | ಗುರಿಯಾಗಿಸಿದ ಹುಡುಕು-ಬದಲಾವಣೆಗಳನ್ನು ಮಾಡಿ |
| `list_directory` | ಪಥದಲ್ಲಿ ಕಡತಗಳು ಮತ್ತು ಡೈರೆಕ್ಟರಿಗಳನ್ನು ಪಟ್ಟಿ ಮಾಡಿ |
| `search_files` | ಮಾದರಿ ಹೊಂದುವ ಕಡತಗಳನ್ನು ಪುನರಾವರ್ತಿತವಾಗಿ ಹುಡುಕಿ |
| `get_file_info` | ಕಡತ ಮೆಟಾಡೇಟಾ ಪಡೆಯಿರಿ (ಗಾತ್ರ, ಟೈಮ್ಸ್ಟ್ಯಾಂಪ್ ಗಳು, ಅನುಮತಿಗಳು) |
| `create_directory` | ಡೈರೆಕ್ಟರಿ ರಚಿಸಿ (ಪೋಷಕ ಡೈರೆಕ್ಟರಿಗಳನ್ನು ಸೇರಿಸಿ) |
| `move_file` | ಕಡತ ಅಥವಾ ಡೈರೆಕ್ಟರಿಯನ್ನು ಸರಿಸಿ ಅಥವಾ ಮರುಹೆಸರಿಸಿ |

ಕೆಳಗಿನ ಚಿತ್ರಣದಲ್ಲಿ Stdio ಸಂಚಾರ ಕಾರ್ಯಗತಗೊಳ್ಳುತ್ತಿರುವ ರೀತಿಯನ್ನು ತೋರಿಸಲಾಗಿದೆ — ನಿಮ್ಮ Java ಅಪ್ಲಿಕೇಶನ್ MCP ಸರ್ವರ್ ಅನ್ನು ಚೈಲ್ ಪ್ರಕ್ರಿಯೆಯಾಗಿ ಆರಂಭಿಸಿ, stdin/stdout ಪೈಪ್ ಗಳ ಮೂಲಕ ಸಂವಹನ ನಡೆಸುತ್ತವೆ, ಯಾವುದೇ ನೆಟ್‌ವರ್ಕ್ ಅಥವಾ HTTP ಸೇರಿಲ್ಲ:

<img src="../../../translated_images/kn/stdio-transport-flow.45eaff4af2d81db4.webp" alt="Stdio Transport Flow" width="800"/>

*Stdio ಸಂಚಾರ ಕ್ರಿಯೆಯಲ್ಲಿ — ನಿಮ್ಮ ಅಪ್ಲಿಕೇಶನ್ MCP ಸರ್ವರ್ ಅನ್ನು ಚೈಲ್ ಪ್ರಕ್ರಿಯೆಯಾಗಿ ಆರಂಭಿಸಿ stdin/stdout ಪೈಪ್ಗಳ ಮೂಲಕ ಸಂವಹನ ಮಾಡುತ್ತದೆ.*

> **🤖 [GitHub Copilot](https://github.com/features/copilot) ಚಾಟ್ ಜೊತೆಗೆ ಪ್ರಯತ್ನಿಸಿ:** [`StdioTransportDemo.java`](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java) ತೆರೆಯಿರಿ ಮತ್ತು ಕೇಳಿ:
> - "Stdio ಸಂಚಾರ ಹೇಗೆ ಕಾರ್ಯನಿರ್ವಹಿಸುತ್ತದೆ ಮತ್ತು HTTP ಗೆ ಹೋಲಿಸಿದಾಗ ಯಾವಾಗ ಬಳಸಬೇಕು?"
> - "LangChain4j spawned MCP ಸರ್ವರ್ ಪ್ರಕ್ರಿಯೆಗಳ ಜೀವನಚರಿತ್ರೆಯನ್ನು ಹೇಗೆ ನಿರ್ವಹಿಸುತ್ತದೆ?"
> - "AI ಗೆ ಕಡತ ವ್ಯವಸ್ಥೆ ಪ್ರವೇಶ ನೀಡುವ ಭದ್ರತಾ ಪರಿಣಾಮಗಳು ಯಾವುವು?"

## ಏಜೆಂಟಿಕ್ module

MCP ಪ್ರಮಾಣಿತ ಸಾಧನಗಳನ್ನು ಒದಗಿಸಿದರೆ, LangChain4j ನ **ಏಜೆಂಟಿಕ್ module** ಆ ಸಾಧನಗಳನ್ನು ನಿಯಂತ್ರಿಸುವ ಏಜೆಂಟುಗಳನ್ನು ರಚಿಸಲು ಘೋಷಣಾತ್ಮಕ ವಿಧಾನವನ್ನು ಒದಗಿಸುತ್ತದೆ. `@Agent` ಅನ್‌ಟೇಷನ್ ಮತ್ತು `AgenticServices` ನಿಮಗೆ ಏಜೆಂಟ್‌ನ ವರ್ತನೆಯನ್ನು ನಿಯಂತ್ರಣಕೋಡ್ ಬದಲು ಇಂಟರ್ಫೇಸ್ ಮೂಲಕ ನಿರ್ದಿಷ್ಟಪಡಿಸಲು ಅನುಮತಿಸುತ್ತವೆ.

ಈ module ನಲ್ಲಿ ನೀವು **ಸೂಪರ್‌ವೈಸರ್ ಏಜೆಂಟ್** ಮಾದರಿಯನ್ನು ಪರಿಶೀಲಿಸುತ್ತೀರಿ — ಇದೊಂದು ಆಧುನಿಕ ಏಜೆಂಟಿಕ್ AI ನೆಪ, ಇಲ್ಲಿ "ಸೂಪರ್‌ವೈಸರ್" ಏಜೆಂಟ್ ಬಳಕೆದಾರನ ಬೇಡಿಕೆಯ ಆಧಾರದಲ್ಲಿ ನಿರಂತರವಾಗಿ ಯಾವ ಉಪ-ಏಜೆಂಟ್ ಗಳನ್ನು ಕರೆಸಬೇಕು ಎಂದು ನಿರ್ಧರಿಸುತ್ತದೆ. ನಾವು MCP ಚालित ಕಡತ ಪ್ರವೇಶ ಸಾಮರ್ಥ್ಯಗಳನ್ನು ಒದಗಿಸುವ ಒಬ್ಬ ಉಪ-ಏಜೆಂಟ್ ಅನ್ನು ಒಟ್ಟುಗೂಡಿಸುತ್ತೇವೆ.

ಏಜೆಂಟಿಕ್ module ಬಳಸಲು, ಈ Maven ಅವಲಂಬನೆಯನ್ನು ಸೇರಿಸಿ:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```


> **ಗಮನಿಸಿ:** `langchain4j-agentic` module ವಿಭಿನ್ನ ನಿಗದಿಯಾದ ಎಲ್ಲಾ ವೇರೆಸನ್ ಪ್ರಾಪರ್ಟಿ (`langchain4j.mcp.version`) ಬಳಸುತ್ತದೆ ಏಕೆಂದರೆ ಇದು ಮುಖ್ಯ LangChain4j ಗ್ರಂಥಾಲಯಗಳಿಗಿಂತ ವಿಭಿನ್ನ ವೇಳಾಪಟ್ಟಿಯಲ್ಲಿ ಬಿಡುಗಡೆ ಆಗುತ್ತದೆ.

> **⚠️ ಪ್ರಯೋಗಾತ್ಮಕ:** `langchain4j-agentic` module **ಪ್ರಯೋಗಾತ್ಮಕವಾಗಿದ್ದು** ಬದಲಾವಣೆಗೆ ಒಳಪಟ್ಟಿರಬಹುದು. ಸತತವಾಗಿ AI ಸಹಾಯಕೆಯನ್ನು ನಿರ್ಮಿಸುವ ಸಾಮಾನ್ಯ ವಿಧಾನ `langchain4j-core` ಮತ್ತು ಕಸ್ಟಮ್ ಸಾಧನಗಳನ್ನೇ ಬಳಕೆ (Module 04) ಆಗಿದೆ.

**Maven ಉಪಯೋಗಿಸಿ:** ಪರ್ಯಾಯವಾಗಿ, ಕೆಳಗಿನ ಉದಾಹರಣೆಗಳೊಂದಿಗೆ ಕಮಾಂಡ್ ಲೈನ್ ನಿಂದ ಚಾಲನೆ ಮಾಡಬಹುದು.

### ಕಡತ ಕಾರ್ಯಾಚರಣೆಗಳು (Stdio)

ಇದು ಸ್ಥಳೀಯ ಉಪಪ್ರಕ್ರಿಯಾ ಆಧಾರಿತ ಸಾಧನಗಳನ್ನು ತೋರಿಸುತ್ತದೆ.

**✅ ಪೂರ್ವಾಪೇಕ್ಷತೆಗಳ ಅಗತ್ಯವಿಲ್ಲ** - MCP ಸರ್ವರ್ ಸ್ವಯಂಚಾಲಿತವಾಗಿ ಆರಂಭವಾಗುತ್ತದೆ.

**ಸ್ಟಾರ್ಟ್ ಸ್ಕ್ರಿಪ್ಟ್ ಗಳ ಬಳಕೆ (ಏಕೆಂದರೆ ಶಿಫಾರಸು):**

ಸ್ಟಾರ್ಟ್ ಸ್ಕ್ರಿಪ್ಟ್ ಗಳಿಂದ ಮೂಲ `.env` ಫೈಲ್ ನಿಂದ ವಾತಾವರಣ ಚರಗಳನ್ನು ಸ್ವಯಂಚಾಲಿತವಾಗಿ ಲೋಡ್ ಮಾಡಲಾಗುತ್ತದೆ:

```

**PowerShell:**

```

**VS Code ಬಳಕೆ:** `StdioTransportDemo.java` ಮೇಲೆ ರೈಟ್ ಕ್ಲಿಕ್ ಮಾಡಿ ಮತ್ತು **"Run Java"** ಆಯ್ಕೆಮಾಡಿ (ನಿಮ್ಮ `.env` ಫೈಲ್ ಅಝೂರ್ ಪ್ರಮಾಣಪತ್ರಗಳಿಂದ ಹೆಸರಿಸಲಾಗಿದೆ ಎಂದು ಖಚಿತಪಡಿಸಿಕೊಳ್ಳಿ).

ಅಪ್ಲಿಕೇಶನ್ ಸ್ವಯಂಚಾಲಿತವಾಗಿ ಫೈಲ್ ಸಿಸ್ಟಮ್ MCP ಸರ್ವರ್ ಅನ್ನು ಆರಂಭಿಸಿ ಸ್ಥಳೀಯ ಒಂದು ಕಡತ ಓದುತ್ತದೆ. ಉಪಪ್ರಕ್ರಿಯೆ ನಿರ್ವಹಣೆಯನ್ನು ನೀವು ಗಮನಿಸಿ.

```

for integrating Large Language Models (LLMs) into Java applications...
```


### ಸೂಪರ್‌ವೈಸರ್ ಏಜೆಂಟ್

**ಸೂಪರ್‌ವೈಸರ್ ಏಜೆಂಟ್ ಮಾದರಿ** ಒಂದು **ಲವಚಿಕ** ರೂಪದ ಏಜೆಂಟಿಕ್ AI ಆಗಿದೆ. ಸೂಪರ್‌ವೈಸರ್ ಒಬ್ಬ LLM ಬಳಸಿ ಸ್ವತಂತ್ರವಾಗಿ ಬಳಕೆದಾರನ ಬೇಡಿಕೆಯ ಆಧಾರದಲ್ಲಿ ಯಾವ ಏಜೆಂಟ್ ಗಳನ್ನು ಕರೆಸಬೇಕು ಎಂದು ನಿರ್ಧರಿಸುವುದೇ. ಮುಂದಿನ ಉದಾಹರಣೆಯಲ್ಲಿ, ನಾವು MCP ಚಾಲಿತ ಕಡತ ಪ್ರವೇಶವನ್ನು LLM ಏಜೆಂಟ್ ಜೊತೆಗೆ ಸಂಯೋಜಿಸಿ ಆಡಳಿತಾತ್ಮಕ ಕಡತ ಓದುವ → ವರದಿ ನಿರ್ಮಿಸುವ ಕಾರ್ಯಪ್ರವಾಹವನ್ನು ಮಾಡುತ್ತೇವೆ.

ಡೆಮೊದಲ್ಲಿ, `FileAgent` MCP ಕಡತ ವ್ಯವಸ್ಥೆ ಸಾಧನಗಳನ್ನು ಬಳಸಿ ಕಡತ ಓದುತ್ತದೆ, ಮತ್ತು `ReportAgent` ಕಾರ್ಯನಿರ್ವಹಣಾ ಸಂಕ್ಷಿಪ್ತ (ಒಂದು ವಾಕ್ಯ), 3 ಪ್ರಮುಖ ಅಂಶಗಳು, ಮತ್ತು ಶಿಫಾರಸಿನೊಂದಿಗೆ ಸಂರಚಿತ ವರದಿಯನ್ನು ತಯಾರಿಸುತ್ತದೆ. ಸೂಪರ್‌ವೈಸರ್ ಈ ಪ್ರಕ್ರಿಯೆಯನ್ನು ಸ್ವಯಂಚಾಲಿತವಾಗಿ ಪೈಪ್ಲೈನ್ ನಿಯಂತ್ರಿಸುತ್ತದೆ:

<img src="../../../translated_images/kn/supervisor-agent-pattern.06275a41ae006ac8.webp" alt="Supervisor Agent Pattern" width="800"/>

*ಸೂಪರ್‌ವೈಸರ್ ತನ್ನ LLM ಉಪಯೋಗಿಸಿ ಯಾವ ಏಜೆಂಟ್ ಗಳನ್ನು ಯಾವ ಕ್ರಮದಲ್ಲಿ ಕರೆಸಬೇಕೇಂದು ನಿರ್ಧರಿಸುತ್ತದೆ—ಹಾರ್ಡ್ಕೋಡ್ ಮಾಡಿದ ಮಾರ್ಗವಲ್ಲ.*

ನಮ್ಮ ಕಡತದಿಂದ ವರದಿ ನಿರ್ಮಿಸುವ ಕಾರ್ಯಪ್ರವಾಹದ ಸ್ಪಷ್ಟ ದೃಶ್ಯ ಇಲ್ಲಿದೆ:

<img src="../../../translated_images/kn/file-report-workflow.649bb7a896800de9.webp" alt="File to Report Workflow" width="800"/>

*FileAgent MCP ಸಾಧನಗಳ ಮೂಲಕ ಕಡತ ಓದುತ್ತದೆ, ReportAgent ಅಸಂರಚಿತ ವಿಷಯವನ್ನು ಸಂರಚಿತ ವರದಿಯಾಗಿಸುತ್ತಾರೆ.*

ಪ್ರತಿ ಏಜೆಂಟ್ ತನ್ನ ಔಟ್ ಪುಟ್ ಅನ್ನು **ಏಜೆಂಟಿಕ್ ವ್ಯಾಪ್ತು** (ಹಂಚಿಕೊಂಡ ഓർമ്മ) ನಲ್ಲಿ ಸಂಗ್ರಹಿಸುತ್ತದೆ, ಇದರಿಂದ ಮುಂದಿನ ಏಜೆಂಟ್ ಗಳು ಹಿಂದಿನ ಫಲಿತಾಂಶಗಳನ್ನು ಪ್ರವೇಶಿಸಬಹುದಾಗುತ್ತದೆ. ಇದು MCP ಸಾಧನಗಳು ಏಜೆಂಟಿಕ್ ಕಾರ್ಯಪ್ರವಾಹದಲ್ಲಿ ಸುಗಮವಾಗಿ ಏಕತೆಯಾಗಿರುವುದನ್ನು ತೋರಿಸುತ್ತದೆ — ಸೂಪರ್‌ವೈಸರ್ ನಿಗದಿಪಡಿಸುವುದಿಲ್ಲ *ಹೇಗೆ* ಕಡತಗಳನ್ನು ಓದಲಾಗುತ್ತಿದೆ, ಕೇವಲ `FileAgent` ಅದು ಮಾಡಬಹುದು ಎಂದು ತಿಳಿದುಕೊಳ್ಳುವುದು.

#### ಡೆಮೊಗಳನ್ನು ನಡಿಸುವುದು

ಸ್ಟಾರ್ಟ್ ಸ್ಕ್ರಿಪ್ಟ್ ಗಳಿಂದ ಮೂಲ `.env` ಫೈಲ್ ನಿಂದ ವಾತಾವರಣ ಚರಗಳನ್ನು ಸ್ವಯಂಚಾಲಿತವಾಗಿ ಲೋಡ್ ಮಾಡಲಾಗುತ್ತದೆ:

**bash:**
```bash
cd 05-mcp
chmod +x start-supervisor.sh
./start-supervisor.sh
```

**PowerShell:**
```powershell
cd 05-mcp
.\start-supervisor.ps1
```

**VS Code ಬಳಕೆ:** `SupervisorAgentDemo.java` ಮೇಲೆ ರೈಟ್ ಕ್ಲಿಕ್ ಮಾಡಿ ಮತ್ತು **"Run Java"** ಆಯ್ಕೆಮಾಡಿ (ನಿಮ್ಮ `.env` ಫೈಲ್ ಅಝೂರ್ ಪ್ರಮಾಣಪತ್ರಗಳಿಂದ ಹೆಸರಿಸಲಾಗಿದೆ ಎಂದು ಖಚಿತಪಡಿಸಿಕೊಳ್ಳಿ).

#### ಸೂಪರ್‌ವೈಸರ್ ಹೇಗೆ ಕಾರ್ಯನಿರ್ವಹಿಸುತ್ತದೆ

ಏಜೆಂಟ್ ಗಳನ್ನು ನಿರ್ಮಿಸುವ ಮೊದಲು, ನೀವು MCP ಸಂಚಾರವನ್ನು ಕ್ಲೈಂಟ್ ಗೆ ಸಂಪರ್ಕಿಸಿ ಅದನ್ನು `ToolProvider` ರೂಪದಲ್ಲಿ ಮುಟ್ಟಿಸಬೇಕು. ಈ ಮೂಲಕ MCP ಸರ್ವರ್ ಸಾಧನಗಳು ನಿಮ್ಮ ಏಜೆಂಟ್ ಗಳಿಗೆ ಲಭ್ಯವಾಗುತ್ತವೆ:

```java
// ಟ್ರಾನ್ಸ್‌ಪೋರ್ಟ್‌ನಿಂದ MCP ಕ್ಲೈಂಟ್ ಸೃಷ್ಟಿಸಿ
McpClient mcpClient = new DefaultMcpClient.Builder()
        .transport(stdioTransport)
        .build();

// ಗ್ರಾಹಕನನ್ನು ಟೂಲ್‌ಪ್ರೊವೈಡರ್ ಆಗಿ ಚుట్టಿಕೊಳ್ಳಿ — ಇದು MCP ಟೂಲ್‌ಗಳನ್ನು LangChain4j ಗೆ ಸೇರುತ್ತದೆ
ToolProvider mcpToolProvider = McpToolProvider.builder()
        .mcpClients(List.of(mcpClient))
        .build();
```


ಈಗ ನೀವು `mcpToolProvider` ಅನ್ನು ಏಜೆಂಟ್ ಯಾವುದೇ ಅದನ್ನು ಬೇಕಾದಲ್ಲಿ ಎಂಜೆಕ್ತ್ ಮಾಡಬಹುದು:

```java
// ಹಂತ 1: ಫೈಲ್ಏಜಂಟ್ MCP ಉಪಕರಣಗಳನ್ನು ಬಳಸಿಕೊಂಡು ಫೈಲ್‌ಗಳನ್ನು ಓದುತ್ತದೆ
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // ಫೈಲ್ ಕಾರ್ಯಚಟುವಟಿಕೆಗಳಿಗೆ MCP ಉಪಕರಣಗಳನ್ನು ಹೊಂದಿದೆ
        .build();

// ಹಂತ 2: ರಿಪೋರ್ಟ್ ಎಜಂಟ್ ರಚನೆಗೊಳಿಸಿದ ವರದಿಗಳನ್ನು ಸೃಷ್ಟಿಸುತ್ತದೆ
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// ನಿರ್ವಾಹಕ ಫೈಲ್ → ವರದಿ ಕಾರ್ಯವಾಹಿಕೆಯನ್ನು ನಿರ್ವಹಿಸುತ್ತದೆ
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // ಅಂತಿಮ ವರದಿಯನ್ನು ಮರಳಿಸಿ
        .build();

// ವಿನಂತಿಯ ಆಧಾರದ ಮೇಲೆ ಯಾವ ಏಜಂಟುಗಳನ್ನು ಕರೆಸಬೇಕು ಎಂದು ನಿರ್ವಾಹಕ ನಿರ್ಧರಿಸುತ್ತದೆ
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```


#### ಪ್ರತಿಕ್ರಿಯೆ ತಂತ್ರಗಳು

ನೀವು `SupervisorAgent` ನ್ನು ಸಂರಚಿಸುವಾಗ, ಉಪ-ಏಜೆಂಟ್ ಗಳ ಕಾರ್ಯಪೂರ್ತಿಯ ನಂತರ ಬಳಕೆದಾರನಿಗೆ ಅಂತಿಮ ಉತ್ತರವನ್ನು ಹೇಗೆ ರೂಪಿಸಲು ಬೇಕೋಂದು ಸೂಚಿಸಬಹುದು. ಕೆಳಗಿನ ಚಿತ್ರಣವು ಮೂರು ಲಭ್ಯವಿರುವ ತಂತ್ರಗಳನ್ನು ತೋರಿಸುತ್ತದೆ - LAST ಕೊನೆ ಏಜೆಂಟ್ ನ ಔಟ್ ಪುಟ್ ನ್ನು ನೇರವಾಗಿ ಹಿಂತಿರುಗಿಸುತ್ತದೆ, SUMMARY LLM ಮೂಲಕ ಎಲ್ಲಾ ಔಟ್ ಪುಟ್ ಗಳ ಸಂಕ್ಷಿಪ್ತವನ್ನು ಸೃಷ್ಟಿಸುತ್ತದೆ, SCORED ಒಂದು ಅಂತರಂಗ LLM ಮೂಲಕ ಮೂಲ ವಿನಂತಿಗೆ ಗಿಣಿಗಿಣಿಯಿಂದ ಅಂಕಗಳನ್ನು ನೀಡುತ್ತದೆ ಮತ್ತು ಹೆಚ್ಚು ಅಂಕ ಪಡೆದ ಅವರ ಔಟ್ ಪುಟ್ ಅನ್ನು ಆಯ್ಕೆಮಾಡುತ್ತದೆ:

<img src="../../../translated_images/kn/response-strategies.3d0cea19d096bdf9.webp" alt="Response Strategies" width="800"/>

*ಸೂಪರ್‌ವೈಸರ್ ತನ್ನ ಅಂತಿಮ ಪ್ರತಿಕ್ರಿಯೆಯನ್ನು ರೂಪಿಸುವ ಮೂರೂ ತಂತ್ರಗಳು - ಕೊನೆಯ ಏಜೆಂಟ್ ಔಟ್ ಪುಟ್, ಸಂಕ್ಷಿಪ್ತ, ಅಥವಾ ಅತಿ ಉತ್ತಮ ಅಂಕ ಪಡೆದ ಆಯ್ಕೆ.*

ಲಭ್ಯವಿರುವ ತಂತ್ರಗಳು:

| ತಂತ್ರ | ವಿವರಣೆ |
|----------|-------------|
| **LAST** | ಸೂಪರ್‌ವೈಸರ್ ಕೊನೆ ಉಪ-ಏಜೆಂಟ್ ಅಥವಾ ಸಾಧನದ ಔಟ್ ಪುಟ್ ನ್ನು ಹಿಂತಿರುಗಿಸುತ್ತದೆ. ಇದು ಕಾರ್ಯಪ್ರವಾಹದಲ್ಲಿ ಕೊನೆ ಏಜೆಂಟ್ ಸಂಪೂರ್ಣ, ಅಂತಿಮ ಉತ್ತರವನ್ನು ನೀಡುವಂತೆ ವಿನ್ಯಾಸಗೊಳಿಸಿದಾಗ ಉಪಯುಕ್ತ (ಉದಾ. ಸಂಶೋಧನಾ ಪೈಪ್ಲೈನ್ ನ "ಸಂಕ್ಷಿಪ್ತ ಏಜೆಂಟ್"). |
| **SUMMARY** | ಸೂಪರ್‌ವೈಸರ್ ತನ್ನ ಅಂತರಂಗ ಭಾಷಾ ಮಾದರಿಯನ್ನು (LLM) ಬಳಸಿ ಸಂಪೂರ್ಣ ಸಂವಾದ ಮತ್ತು ಎಲ್ಲಾ ಉಪ-ಏಜೆಂಟ್ ಗಳ ಔಟ್ ಪುಟ್ ಗಳ ಸಂಕ್ಷಿಪ್ತ ಸೃಷ್ಟಿ ಮಾಡಿ, ಆ ಸಂಕ್ಷಿಪ್ತವನ್ನು ಅಂತಿಮ ಪ್ರತಿಕ್ರಿಯೆಯಾಗಿ ಹಿಂತಿರುಗಿಸುತ್ತದೆ. ಇದು ಬಳಕೆದಾರಿಗೆ ಸ್ವಚ್ಛ, ಗುಡ್ಡೆಗೂಡಿದ ಉತ್ತರವನ್ನು ಒದಗಿಸುತ್ತದೆ. |
| **SCORED** | ವ್ಯವಸ್ಥೆ ಆಂತರಂಗ LLM ಅನ್ನು ಬಳಸಿ LAST ಪ್ರತಿಕ್ರಿಯೆ ಮತ್ತು SUMMARY ನಡುವೆ ಅಮೂಲ್ಯ ಮೂಲ ಬಳಕೆದಾರ ವಿನಂತಿಗೆ ಅಂಕಗಳನ್ನು ನೀಡುತ್ತದೆ ಮತ್ತು ಹೆಚ್ಚು ಅಂಕ ಪಡೆದ ಔಟ್ ಪುಟ್ ಅನ್ನು ಹಿಂತಿರುಗಿಸುತ್ತದೆ. |

#### ಔಟ್‌ಪುಟ್ ಅನ್ನು ಅರ್ಥಮಾಡಿಕೊಳ್ಳುವುದು

ನೀವು ಡೆಮೊ ಅನ್ನು ಓಡಿಸಿದಾಗ, Supervisor ಹಲವಾರು ಏಜೆಂಟ್‌ಗಳನ್ನು ಹೇಗೆ ಸಂಯೋಜಿಸುತ್ತಾನೆ ಎಂಬುದರ ರಚಿತವಾದ ಹೆಜ್ಜೆಹತ್ತು ವಾಕಿಂಗ್‌ಥ್ರೂ ಕಾಣುತ್ತೀರಿ. ಪ್ರತಿ ವಿಭಾಗವು ಅರ್ಥವತ್ತಾಗಿದೆ:

```
======================================================================
  FILE → REPORT WORKFLOW DEMO
======================================================================

This demo shows a clear 2-step workflow: read a file, then generate a report.
The Supervisor orchestrates the agents automatically based on the request.
```

**ಶೀರ್ಷಿಕೆ** ವರ್ಕ್‌ಫ್ಲೋ ಅವಧಾರಣೆಯನ್ನು ಪರಿಚಯಿಸುತ್ತದೆ: ಫೈಲ್ ಓದಿಕೆಯುಂದಿಂದ ವರದಿ ತಯಾರಿಕೆಗೂ ಕೇಂದ್ರೀಕೃತ ಪೈಪ್ಲೈನ್.

```
--- WORKFLOW ---------------------------------------------------------
  ┌─────────────┐      ┌──────────────┐
  │  FileAgent  │ ───▶ │ ReportAgent  │
  │ (MCP tools) │      │  (pure LLM)  │
  └─────────────┘      └──────────────┘
   outputKey:           outputKey:
   'fileContent'        'report'

--- AVAILABLE AGENTS -------------------------------------------------
  [FILE]   FileAgent   - Reads files via MCP → stores in 'fileContent'
  [REPORT] ReportAgent - Generates structured report → stores in 'report'
```

**ವರ್ಕ್‌ಫ್ಲೋ ಚಿತ್ರ** ಏಜೆಂಟ್‌ಗಳ ನಡುವಿನ ಡೇಟಾ ಹರಿವನ್ನು ತೋರಿಸುತ್ತದೆ. ಪ್ರತಿಯೊಬ್ಬ ಏಜೆಂಟ್‌ಗೆ ವಿಶೇಷ ಪಾತ್ರವಿದೆ:
- **FileAgent** MCP ಸಾಧನಗಳನ್ನು ಬಳಸಿ ಫೈಲ್‌ಗಳನ್ನು ಓದುತ್ತದೆ ಮತ್ತು ಅಸಂಸ್ಕೃತ ವಿಷಯವನ್ನು `fileContent` ನಲ್ಲಿ ಸಂಗ್ರಹಿಸುತ್ತದೆ
- **ReportAgent** ಆ ವಿಷಯವನ್ನು ಉಪಯೋಗಿಸಿ ರಚನಾತ್ಮಕ ವರದಿ `report` ನಲ್ಲಿ ಉತ್ಪಾದಿಸುತ್ತದೆ

```
--- USER REQUEST -----------------------------------------------------
  "Read the file at .../file.txt and generate a report on its contents"
```

**ಬಳಕೆದಾರ ವಿನಂತಿ** ಕಾರ್ಯವನ್ನು ತೋರಿಸುತ್ತದೆ. Supervisor ಇದನ್ನು ವಿಶ್ಲೇಷಿಸಿ FileAgent → ReportAgent ಅನ್ನು ಕರೆಸುವುದಾಗಿ ನಿರ್ಧಾರ ಮಾಡುತ್ತದೆ.

```
--- SUPERVISOR ORCHESTRATION -----------------------------------------
  The Supervisor decides which agents to invoke and passes data between them...

  +-- STEP 1: Supervisor chose -> FileAgent (reading file via MCP)
  |
  |   Input: .../file.txt
  |
  |   Result: LangChain4j is an open-source, provider-agnostic Java framework for building LLM...
  +-- [OK] FileAgent (reading file via MCP) completed

  +-- STEP 2: Supervisor chose -> ReportAgent (generating structured report)
  |
  |   Input: LangChain4j is an open-source, provider-agnostic Java framew...
  |
  |   Result: Executive Summary...
  +-- [OK] ReportAgent (generating structured report) completed
```

**Supervisor ಸಂಯೋಜನೆ** 2 ಹೆಜ್ಜೆಯ ಹರಿವು ಕಾರ್ಯನಿರ್ವಹಣೆಯನ್ನು ತೋರಿಸುತ್ತದೆ:
1. **FileAgent** MCP ಮೂಲಕ ಫೈಲ್ ಅನ್ನು ಓದುತ್ತದೆ ಮತ್ತು ವಿಷಯವನ್ನು ಸಂಗ್ರಹಿಸುತ್ತದೆ
2. **ReportAgent** ಆ ವಿಷಯವನ್ನು ಸ್ವೀಕರಿಸಿ ರಚನಾತ್ಮಕ ವರದಿ ಸೃಷ್ಟಿಸುತೆದು

Supervisor ಈ ನಿರ್ಧಾರಗಳನ್ನು ಬಳಕೆದಾರರ ವಿನಂತಿಯ ಆಧಾರದ ಮೇಲೆ **ಸ್ವಯಂಚಾಲಿತವಾಗಿ** ಮಾಡಿದೆ.

```
--- FINAL RESPONSE ---------------------------------------------------
Executive Summary
...

Key Points
...

Recommendations
...

--- AGENTIC SCOPE (Data Flow) ----------------------------------------
  Each agent stores its output for downstream agents to consume:
  * fileContent: LangChain4j is an open-source, provider-agnostic Java framework...
  * report: Executive Summary...
```

#### ಏಜೆಂಟಿಕ್ ಮಡುಲಿನ ಲಕ್ಷಣಗಳ وضاحت

ಈ ಉದಾಹರಣೆಯು ಏಜೆಂಟಿಕ್ ಮಡುಗಿನ ಹಲವು ಮುಂದುವರೆದ ಲಕ್ಷಣಗಳನ್ನು ತೋರಿಸುತ್ತದೆ. ಏಜೆಂಟಿಕ್ ಸ್ಕೋಪ್ ಮತ್ತು ಏಜೆಂಟ್ ಕೇಳಿಗಾರರನ್ನು ನಿಖರವಾಗಿ ನೋಡಿ.

**ಏಜೆಂಟಿಕ್ ಸ್ಕೋಪ್** @Agent(outputKey="...") ಬಳಸಿ ಏಜೆಂಟ್‌ಗಳು ತಮ್ಮ ಫಲಿತಾಂಶಗಳನ್ನು ಸಂಗ್ರಹಿಸಿದ ಹಂಚಿಕಲ್ಪಟ್ಟ ಮೆಮೊರಿಗೆ ಸೂಚಿಸುತ್ತದೆ. ಇದು ಸಾಧ್ಯಮಾಡುತ್ತದೆ:
- ಮಿಸಲಿನ ಏಜೆಂಟ್‌ಗಳು ಮುಂಚಿನ ಏಜೆಂಟ್ ಔಟ್‌ಪುಟ್‌ಗಳನ್ನು ಪ್ರಾಪ್ತಿಸುವುದು
- Supervisor ಅಂತಿಮ ಪ್ರತಿಕ್ರಿಯೆಯನ್ನು ಸಂಯೋಜಿಸುವುದು
- ನೀವು ಪ್ರತಿಯೊಂದು ಏಜೆಂಟ್ ಉತ್ಪಾದಿಸಿದುದನ್ನು ಪರಿಶೀಲಿಸುವುದು

ಕೆಳಗಿನ ಚಿತ್ರವು ಫೈಲ್-ಇಂದ-ವರದಿ ವರ್ಕ್‌ಫ್ಲೋದಲ್ಲಿ ಏಜೆಂಟಿಕ್ ಸ್ಕೋಪ್ ಹೇಗೆ ಹಂಚಿಕಲ್ಪಟ್ಟ ಮೆಮರಿಗೆ ಕಾರ್ಯನಿರ್ವಹಿಸುತ್ತದೆ ಎಂಬುದನ್ನು ತೋರಿಸುತ್ತದೆ — FileAgent ತನ್ನ ಔಟ್‌ಪುಟ್ `fileContent` ಅಡಿ ಬರೆಯುತ್ತದೆ, ReportAgent ಅದನ್ನು ಓದಿಸಿಕೊಂಡು ತನ್ನ ಔಟ್‌ಪುಟ್‌ನ್ನು `report` ಅಡಿ ಬರೆಯುತ್ತದೆ:

<img src="../../../translated_images/kn/agentic-scope.95ef488b6c1d02ef.webp" alt="ಏಜೆಂಟಿಕ್ ಸ್ಕೋಪ್ ಹಂಚಿಕಲ್ಪಟ್ಟ ಮೆಮೊರಿ" width="800"/>

*ಏಜೆಂಟಿಕ್ ಸ್ಕೋಪ್ ಹಂಚಿಕಲ್ಪಟ್ಟ ಮೆಮೊರಿಯಾಗಿ ಕಾರ್ಯನಿರ್ವಹಿಸುತ್ತದೆ — FileAgent `fileContent` ಬರೆಯುತ್ತದೆ, ReportAgent ಅದನ್ನು ಓದುತ್ತದೆ ಮತ್ತು `report` ಬರೆಯುತ್ತದೆ, ಮತ್ತು ನಿಮ್ಮ ಕೋಡ್ ಅಂತಿಮ ಫಲಿತಾಂಶವನ್ನು ಓದುತ್ತದೆ.*

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // FileAgent‌ನಿಂದ ಕಚ್ಚಾ ಫೈಲ್ ದತ್ತಾಂಶ
String report = scope.readState("report");            // ReportAgentರಿಂದ ಸಂಸ್ಥಿತ ವರದಿ
```

**ಏಜೆಂಟ್ ಕೇಳಿಗಾರರು** ಏಜೆಂಟ್ ಕಾರ್ಯಾಚರಣೆಯ ಕ್ರಮಲಕ್ಷಣ ಮತ್ತು ಡಿಬಗ್ ಮಾಡಲು ಸಹಾಯಕರು. ಡೆಮೊದಲ್ಲಿ ನೀವು ಕಾಣುವ ಹಂತ ಹಂತ ಔಟ್‌ಪುಟ್ ಏಜೆಂಟ್ ಕೇಳಿಗಾರರಿಂದ ಬರುತ್ತದೆ, ಅದು ಪ್ರತಿಯೊಂದು ಏಜೆಂಟ್ ಕರೆಸುವಾಗ ಹೂಕುಡುತ್ತದೆ:
- **beforeAgentInvocation** - Supervisor ಏಜೆಂಟ್ ಆಯ್ಕೆ ಮಾಡಿದಾಗ ಕರೆಸಲ್ಪಡುವುದು, ಆಯ್ದ ಏಜೆಂಟ್ ಮತ್ತು ಕಾರಣವನ್ನು ಕಾಣಲು ಅನುಮತಿಸುವುದು
- **afterAgentInvocation** - ಏಜೆಂಟ್ ಕಾರ್ಯ ಪೂರ್ಣಗೊಂಡಾಗ ಕರೆಯಲಾಗುತ್ತದೆ, ಅದರ ಫಲಿತಾಂಶವನ್ನು ತೋರಿಸುತ್ತದೆ
- **inheritedBySubagents** - ಸತ್ಯವಾದಾಗ, ಕೇಳಿಗಾರನು ವರ್ಗ ಸಾರಣೆಯ ಎಲ್ಲಾ ಏಜೆಂಟ್‌ಗಳನ್ನು ಗಮನಿಸುತ್ತಾನೆ

ಕೆಳಗಿನ ಚಿತ್ರವು ಪೂರ್ಣ ಏಜೆಂಟ್ ಕೇಳಿಗಾರ ಜೀವನಚಕ್ರವನ್ನು ತೋರಿಸುತ್ತದೆ, `onError` ಏಜೆಂಟ್ ನಿರ್ವಹಣೆಯ ವೇಳೆ ವೈಫಲ್ಯಗಳನ್ನು ಹಿಡಿಯುವ ವಿಧಾನವನ್ನು ಒಳಗೊಂಡಂತೆ:

<img src="../../../translated_images/kn/agent-listeners.784bfc403c80ea13.webp" alt="ಏಜೆಂಟ್ ಕೇಳಿಗಾರರ ಜೀವನಚಕ್ರ" width="800"/>

*ಏಜೆಂಟ್ ಕೇಳಿಗಾರರು ಕಾರ್ಯಚರಣೆ ಜೀವನಚಕ್ರದಲ್ಲಿ ಹೂಕುಡುತ್ತಾರೆ — ಏಜೆಂಟ್ ಪ್ರಾರಂಭವಾಗುವಾಗ, ಪೂರ್ಣಗೊಳ್ಳುವಾಗ ಅಥವಾ ದೋಷಗಳಾಗುವಾಗ ಗಮನಿಸುವರು.*

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // ಎಲ್ಲಾ ಉಪ ಏಜೆಂಟ್‌ಗಳಿಗೆ ಹರಡಿಸಿ
    }
};
```

Supervisor ಮಾದರಿಯ ಹೊರತಾಗಿ, `langchain4j-agentic` ಮಡುಗು ಹಲವಾರು ಶಕ್ತಿಶಾಲಿ ವರ್ಕ್‌ಫ್ಲೋ ಮಾದರಿಗಳನ್ನು ಒದಗಿಸುತ್ತದೆ. ಕೆಳಗಿನ ಚಿತ್ರವು ಎಲ್ಲ ಐದು ಮಾದರಿಗಳನ್ನು ತೋರಿಸುತ್ತದೆ — ಸರಳ ಕ್ರಮವಾರಿಯ ಪೈಪ್ಲೈನಿನಿಂದ ಮಾನವ-ಇನ್-ದಿ-ಲೂಪ್ ಅನುಮೋದನೆ ವರ್ಕ್‌ಫ್ಲೋಗಳವರೆಗೆ:

<img src="../../../translated_images/kn/workflow-patterns.82b2cc5b0c5edb22.webp" alt="ಏಜೆಂಟ್ ವರ್ಕ್‌ಫ್ಲೋ ಮಾದರಿಗಳು" width="800"/>

*ಏಜೆಂಟ್‌ಗಳನ್ನು ಸಂಯೋಜಿಸುವ ಐದು ವರ್ಕ್‌ಫ್ಲೋ ಮಾದರಿಗಳು — ಸರಳ ಕ್ರಮವಾರಿಯ ಪೈಪ್ಲೈನ್‌ಗಳಿಂದ ಮಾನವ-ಇನ್-ದಿ-ಲೂಪ್ ಅನುಮೋದನೆ ವರ್ಕ್‌ಫ್ಲೋಗಳವರೆಗೆ.*

| ಮಾದರಿ | ವಿವರಣೆ | ಉಪಯೋಗ |
|---------|-------------|----------|
| **ಕ್ರಮವಾರಿ** | ಏಜೆಂಟ್‌ಗಳನ್ನು ಕ್ರಮವಾಗಿ ಕಾರ್ಯಗತಗೊಳಿಸಿ, ಔಟ್‌ಪುಟ್ ಮುಂದಿನಕ್ಕೆ ಹರಿಯುತ್ತದೆ | ಪೈಪ್ಲೈನ್ಗಳು: ಸಂಶೋಧನೆ → ವಿಶ್ಲೇಷಣೆ → ವರದಿ |
| **ಪ್ಯಾರಲೆಲ್** | ಏಜೆಂಟ್‌ಗಳನ್ನು ಒಂದೇ ವೇಳೆ ನಿರ್ವಹಿಸಿ | ಸ್ವತಂತ್ರ ಕಾರ್ಯಗಳು: ಹವಾಮಾನ + ಸುದ್ದಿಗಳು + ಸ್ಟಾಕ್‌ಗಳು |
| **ಲೂಪ್** | ನಿರ್ದಿಷ್ಟ ಶರತ್ತು ಪೂರ್ತಿಯಾಗುವವರೆಗೆ ಪುನರಾವರ್ತಿಸಿ | ಗುಣಮಟ್ಟ ಅಂಕೆಗಣನೆ: ಅಂಕೆ ≥ 0.8 ಆಗುವವರೆಗೆ ತಿದ್ದುಪಡಿ |
| **ಶರತಾನದ ಮೂಲಕ** | ಶರತ್ತಿನ ಆಧಾರದ ಮೇಲೆ ಮಾರ್ಗವನ್ನೊದಗಿಸಿ | ವರ್ಗೀಕರಣ → ಪರಿಣತಿ ಏಜೆಂಟ್ ಗೆ ಮಾರ್ಗದರ್ಶನ |
| **ಮಾನವ-ಇನ್-ದಿ-ಲೂಪ್** | ಮಾನವ ನಿಯಂತ್ರಣ ಬಿಂದುವುಗಳನ್ನು ಸೇರಿಸಿ | ಅನುಮೋದನೆ ವರ್ಕ್‌ಫ್ಲೋಗಳು, ವಿಷಯ ಪರಿಶೀಲನೆ |

## ಪ್ರಮುಖ ಸಂಜ್ಞೆಗಳು

ನೀವು ಈಗ MCP ಮತ್ತು ಏಜೆಂಟಿಕ್ ಮಡುಲನ್ನು ಕಾರ್ಯದಲ್ಲಿ ನೋಡಿದ್ದೀರಿ, ಪ್ರತಿ ವಿಧಾನವನ್ನು ಯಾವಾಗ ಬಳಸಬೇಕೆಂದು ಸಾರಾಂಶ ಮಾಡೋಣ.

MCP ಯ ಅತ್ಯಂತ ದೊಡ್ಡ ಲಾಭವೆಂದರೆ ಅದರ ಬೆಳೆಯುತ್ತಿರುವ ಪರಿಸರ ವ್ಯವಸ್ಥೆ. ಕೆಳಗಿನ ಚಿತ್ರವು ಒಟ್ಟಾರೆ ವಿಶ್ವವ್ಯಾಪಕ ಪ್ರೋಟೋಕಾಲ್ ಮೂಲಕ ನಿಮ್ಮ AI ಅಪ್ಲಿಕೇಶನ್ ಅನ್ನು ವಿವಿಧ MCP ಸರ್ವರ್‌ಗಳಿಗೆ ಹೇಗೆ ಸಂಪರ್ಕಿಸುತ್ತದೆ ಎಂಬುದನ್ನು ತೋರಿಸುತ್ತದೆ — ಫೈಲ್ ಸಿಸ್ಟಮ್ ಮತ್ತು ಡೇಟಾಬೇಸ್ ಪ್ರವೇಶದಿಂದ GitHub, ಇಮೇಲ್, ವೆಬ್ ಸ್ಕ್ರಾಪಿಂಗ್ ಮತ್ತಿತರವರೆಗೆ:

<img src="../../../translated_images/kn/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="MCP ಪರಿಸರ ವ್ಯವಸ್ಥೆ" width="800"/>

*MCP ಒಂದು ವಿಶ್ವವ್ಯಾಪಿ ಪ್ರೋಟೋಕಾಲ್ ಪರಿಸರ ವ್ಯವಸ್ಥೆಯನ್ನು ಸೃಷ್ಟಿಸುತ್ತದೆ — ಯಾವುದೇ MCP-ಅನುಕೂಲ ಸರ್ವರ್ ಯಾವುದೇ MCP-ಅನುಕೂಲ ಕ್ಲೈಂಟ್ ಜೊತೆಗೆ ಕೆಲಸ ಮಾಡಬಲ್ಲದು, ಹೀಗಾಗಿ ಏಪ್ಲಿಕೇಶನ್ ಗಳು ಸಾಧನಗಳನ್ನು ಹಂಚಿಕೊಳ್ಳಬಹುದು.*

**MCP** ನೀವು ಇತ್ತೀಚಿನ ಸಾಧನ ಪರಿಸರಗಳನ್ನು ಉಪಯೋಗಿಸಲು ಬಯಸುವಾಗ, ಹಲವು ಅಪ್ಲಿಕೇಶನ್ಗಳು ಹಂಚಿಕೊಳ್ಳಬಲ್ಲ ಸಾಧನಗಳನ್ನು ನಿರ್ಮಿಸಲು, ಮೂರನೇ ಪಕ್ಷದ ಸೇವೆಗಳನ್ನು ಮಾನಕರ ಪ್ರೋಟೋಕಾಲ್‌ಗಳೊಂದಿಗೆ ಏಕೀಕರಿಸಲು ಅಥವಾ ಸಾಧನಗಳನ್ನು ಬದಲಿಸಬೇಕಾದಾಗ ಹೇಗೆ ಕಾರ್ಯನಿರ್ವಹಿಸುವುದನ್ನು ಬಯಸುವಾಗ ಸೂಕ್ತ.

**ಏಜೆಂಟಿಕ್ ಮಡುಗು** ನಿಮ್ಮಿ `@Agent` ಟ್ಯಾಗ್‌ಗಳೊಂದಿಗೆ ಘೋಷಣೆಯಾಧಾರಿತ ಏಜೆಂಟ್ ವ್ಯಾಖ್ಯಾನಗಳನ್ನು ಬಯಸುವಾಗ, ವರ್ಕ್‌ಫ್ಲೋ ಸಂಯೋಜನೆ (ಕ್ರಮವಾರಿ, ಲೂಪ್, ಪ್ಯಾರಲೆಲ್) ಅಗತ್ಯವಿದ್ದಾಗ, ಆದೇಶಾತ್ಮಕ ಕೋಡ್ ಬದಲು ಇಂಟರ್ಫೇಸ್ ಆಧಾರಿತ ಏಜೆಂಟ್ ವಿನ್ಯಾಸವನ್ನು ಇಚ್ಛಿಸುವಾಗ, ಅಥವಾ `outputKey` ಮುಖೇನ ಹಂಚಿಕೊಳ್ಳುವ ಹಲವು ಏಜೆಂಟ್‌ಗಳನ್ನು ಸಂಯೋಜಿಸುವಾಗ ಉತ್ತಮ.

**Supervisor ಏಜೆಂಟ್ ಮಾದರಿ** ಮುಂಚಿತವಾಗಿ ನಿರೀಕ್ಷಿಸಲು ಸಾಧ್ಯವಿಲ್ಲದ ವರ್ಕ್‌ಫ್ಲೋಗಳಿಗೆ ಮತ್ತು ನಾನಾ ವಿಶಿಷ್ಟ ಏಜೆಂಟ್‌ಗಳು ಗತಿಯುತ ಸಂಯೋಜನೆ ಅಗತ್ಯವಿದ್ದಾಗ, ವಿಭಿನ್ನ ಸಾಮರ್ಥ್ಯಗಳಿಗೆ ಮಾರ್ಗನಿರ್ದೇಶಿಸುವ ಸಂವಾದಾತ್ಮಕ ವ್ಯವಸ್ಥೆಗಳು ನಿರ್ಮಿಸುವಾಗ ಅಥವಾ ಅತ್ಯಂತ ಲವಚಿಕ, ಅನುಕೂಲಕರ ಏಜೆಂಟ್ ವರ್ತನೆ ಬೇಕಾಗುವಾಗ ಜಿಗಿಯಲು.

ಮಾಡ್ಯೂಲ್ 04 ನ ಕಸ್ಟಮ್ `@Tool` ವಿಧಿಗಳೊಂದಿಗೆ ಈ ಮಾಡ್ಯೂಲ್‌ನ MCP ಸಾಧನಗಳನ್ನು ಹೋಲಿಸುವಾಗ, ಕೆಳಗಿನ ತುಲನೆ ಮುಖ್ಯ ವ್ಯತ್ಯಾಸಗಳನ್ನು ಬೆಳಿಗ್ಗುತ್ತದೆ — ಕಸ್ಟಮ್ ಸಾಧನಗಳು ಅಪ್ಲಿಕೇಶನ್ ನಿರ್ದಿಷ್ಟ ಲಾಜಿಕ್‌ಗೆ ತೀವ್ರ ಜೋಡಣೆ ಮತ್ತು ಸಂಪೂರ್ಣ ಪ್ರಕಾರ ಭದ್ರತೆ ನೀಡುತ್ತವೆ, MCP ಸಾಧನಗಳು ಮಾನಕರ, ಮರುಬಳಕೆ ಯೋಗ್ಯ ಏಕೀಕರಣಗಳನ್ನು ಒದಗಿಸುತ್ತವೆ:

<img src="../../../translated_images/kn/custom-vs-mcp-tools.c4f9b6b1cb65d8a1.webp" alt="ಕಸ್ಟಮ್ ಸಾಧನಗಳು ಮತ್ತು MCP ಸಾಧನಗಳ ಹೋಲಿಕೆ" width="800"/>

*ಯಾವಾಗ ಕಸ್ಟಮ್ @Tool ವಿಧಾನಗಳನ್ನು ಬಳಸುವುದು ಮತ್ತು ಯಾವಾಗ MCP ಸಾಧನಗಳನ್ನು ಬಳಸುವುದು — ಅಪ್ಲಿಕೇಶನ್-ನಿರ್ದಿಷ್ಟ ಲಾಜಿಕ್ ಗೆ ಪೂರಕ ಕಸ್ಟಮ್ ಸಾಧನಗಳು ಸಂಪೂರ್ಣ ಪ್ರಕಾರ ಭದ್ರತೆ ಯೊಂದಿಗೆ, MCP ಸಾಧನಗಳು ವಿವಿಧ ಅಪ್ಲಿಕೇಶನ್‌ಗಳ ನಡುವೆ ಕೆಲಸ ಮಾಡುವ ಮಾನಕರ ಏಕೀಕರಣಗಳು.*

ನೀವು LangChain4j for Beginners ಪಾಠಶಾಲೆಯಲ್ಲಿನ ಕೆಳಗಿನವನ್ನಾಗ್ನ್ನು ಕಲಿತಿದ್ದೀರಿ:

- ಮೆಮೊರಿಯೊಂದಿಗೆ ಸಂವಾದಾತ್ಮಕ AI ನಿರ್ಮಿಸುವುದು (ಮಾಡ್ಯೂಲ್ 01)
- ವಿಭಿನ್ನ ಕಾರ್ಯಗಳಿಗೆ ಪ್ರಂಪ್ಟ್ ಇಂಜಿನಿಯರಿಂಗ್ ಮಾದರಿಗಳು (ಮಾಡ್ಯೂಲ್ 02)
- ನಿಮ್ಮ ಡಾಕ್ಯುಮೆಂಟ್‌ಗಳಲ್ಲಿ ಆಧರಿಸಿಕೊಂಡು ಉತ್ತರಗಳನ್ನು ನೆಲಗೊಳಿಸುವುದು RAG ಮೂಲಕ (ಮಾಡ್ಯೂಲ್ 03)
- ಕಸ್ಟಮ್ ಸಾಧನಗಳೊಂದಿಗೆ ಮೂಲ AI ಏಜೆಂಟ್‌ಗಳು (ಸಹಾಯಕರು) ಸೃಷ್ಟಿಸುವುದು (ಮಾಡ್ಯೂಲ್ 04)
- LangChain4j MCP ಮತ್ತು ಏಜೆಂಟಿಕ್ ಮಾಡ್ಯೂಲ್‌ಗಳೊಂದಿಗೆ ಮಾನಕರ ಸಾಧನಗಳನ್ನು ಅಳವಡಿಸುವುದು (ಮಾಡ್ಯೂಲ್ 05)

### ಮುಂದೇನು?

ಮಾಡ್ಯೂಲ್‌‌ಗಳನ್ನೂ ಮುಗಿಸಿಕೊಂಡ ನಂತರ, LangChain4j ಪರೀಕ್ಷಾ ವ್ಯಾಖ್ಯಾನಗಳನ್ನು ಅನುಭವಿಸಲು [ಟೆಸ್ಟಿಂಗ್ ಗೈಡ್](../docs/TESTING.md) ಅನ್ನು ನೋಡಿ.

**ಅಧಿಕೃತ ಸಂಪನ್ಮೂಲಗಳು:**
- [LangChain4j ಡಾಕ್ಯುಮೆಂಟೇಶನ್](https://docs.langchain4j.dev/) - ಸವಿಸ್ತಾರ ಮಾರ್ಗದರ್ಶಿಗಳು ಮತ್ತು API ಸೂಚಿಕೆ
- [LangChain4j GitHub](https://github.com/langchain4j/langchain4j) - ಮೂಲ ಕೋಡ್ ಮತ್ತು ಉದಾಹರಣೆಗಳು
- [LangChain4j ಟ್ಯುಟೋರಿಯಲ್ಸ್](https://docs.langchain4j.dev/tutorials/) - ವಿವಿಧ ಉಪಯೋಗಗಳಿಗೆ ಹಂತ ಹಂತ ಮಾರ್ಗದರ್ಶನ

ಈ ಪಾಠಶಾಲೆಯನ್ನು ಮುಗಿಸಿದ ನಿಮಗೆ ಧನ್ಯವಾದಗಳು!
