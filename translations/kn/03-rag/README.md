## ನೀವು ಕಲಿಯುವುದು ಏನು

ಹಿಂದಿನ ಮದ್ಯಮಗಳಲ್ಲಿ, ನೀವು AI ಜೊತೆ ಸಂವಾದಗಳನ್ನು ನಡೆಸುವುದು ಮತ್ತು ನಿಮ್ಮ ಪ್ರಾಂಪ್ಟ್‌ಗಳನ್ನು ಪರಿಣಾಮಕಾರಿಯಾಗಿ ರಚಿಸುವುದನ್ನು ಕಲಿದ್ದರು. ಆದರೆ ಮೂಲತಃ ಒಂದು ನಿಯಂತ್ರಣವಿದೆ: ಭಾಷಾ ಮಾದರಿಗಳು ಕೇವಲ ತರಬೇತಿಯ ಸಮಯದಲ್ಲಿ ಸಿಗಿದ ಮಾಹಿತಿಗಳನ್ನು ಮಾತ್ರ ತಿಳಿದುಕೊಳ್ಳುತ್ತವೆ. ಅವು ನಿಮ್ಮ ಸಂಸ್ಥೆಯ ನೀತಿಗಳ ಬಗ್ಗೆ, ನಿಮ್ಮ ಯೋಜನೆಯ ದಾಖಲೆಗಳ ಬಗ್ಗೆ ಅಥವಾ ತರಬೇತಿ ಪಡೆಯದೆ থাকা ಯಾವ ಮಾಹಿತಿ ಬಗ್ಗೆ ಉತ್ತರಿಸಲು ಸಾಧ್ಯವಿಲ್ಲ.

RAG (ರಿಟ್ರೀವಲ್-ಆಗ್ಮೆಂಟೆಡ್ ಜೇನರೇಷನ್) ಈ ಸಮಸ್ಯೆಯನ್ನು ಪರಿಹರಿಸುತ್ತದೆ. ಮಾದರಿಯನ್ನು ನಿಮ್ಮ ಮಾಹಿತಿಯನ್ನು ಕಲಿಸುವ ಪ್ರಯತ್ನ ಮಾಡದೆ (ಅದು ವೆಚ್ಚದಾಯಕ ಮತ್ತು ಅಪ್ರಾಯೋಗಿಕವಾಗಿದೆ), ನೀವು ಅದಕ್ಕೆ ನಿಮ್ಮ ದಾಖಲೆಗಳ ಮೂಲಕ ಹುಡುಕುವ ಸಾಮರ್ಥ್ಯವನ್ನು ನೀಡುತ್ತೀರಿ. ಯಾರಾದರೂ ಪ್ರಶ್ನೆ ಕೇಳಿದಾಗ, ವ್ಯವಸ್ಥೆ ಸಂಬಂಧಿತ ಮಾಹಿತಿಯನ್ನು ಹುಡುಕಿ ಪ್ರಾಂಪ್ಟ್‌ಗೆ ಸೇರಿಸುತ್ತದೆ. ನಂತರ ಮಾದರಿ ಆ ತಂದ ಮಾಹಿತಿಯ ಆಧಾರದಲ್ಲಿ ಉತ್ತರವನ್ನು ನೀಡುತ್ತದೆ.

RAG ಅನ್ನು ಮಾದರಿಗೆ ಒಂದು ಸಂಬಂದ ಪಠ್ಯಗ್ರಂಥಾಲಯ ನೀಡಿದಂತೆ ಊಹಿಸಿ. ನೀವು ಪ್ರಶ್ನೆ ಕೇಳಿದಾಗ, ವ್ಯವಸ್ಥೆ:

1. **ಬಳಕೆದಾರರ ಪ್ರಶ್ನೆ** - ನೀವು ಒಂದು ಪ್ರಶ್ನೆ ಕೇಳುತ್ತೀರಿ  
2. **ಎಂಬೆಡ್ಡಿಂಗ್** - ನಿಮ್ಮ ಪ್ರಶ್ನೆಯನ್ನು ಒಂದು ವೆಕ್ಟರ್‌ಗೆ ಪರಿವರ್ತಿಸುತ್ತದೆ  
3. **ವೆಕ್ಟರ್ ಶೋಧನೆ** - ಸಂಬಂಧಿತ ದಸ್ತಾವೇಜು ಚಂಕ್‌ಗಳನ್ನು ಹುಡುಕುತ್ತದೆ  
4. **ಸಂದರ್ಭ ಸಂಯೋಜನೆ** - ಸಂಬಂಧಿತ ಚಂಕ್‌ಗಳನ್ನು ಪ್ರಾಂಪ್ಟ್‌ಗೆ ಸೇರಿಸುತ್ತದೆ  
5. **ಪ್ರತಿಕ್ರಿಯೆ** - LLM ಆ ಸಂದರ್ಭ ಆಧಾರದಲ್ಲಿ ಉತ್ತರವನ್ನು ರಚಿಸುತ್ತದೆ  

ಇದು ಮಾದರಿಯ ಉತ್ತರಗಳನ್ನು ಅದರ ತರಬೇತಿ ಜ್ಞಾನಕ್ಕೆ ಅವಲಂಬಿಸದೆ ನಿಮ್ಮ ನಿಜವಾದ ಡೇಟಾದ ಮೇಲೆ ನೆಡೆಸುತ್ತದೆ.

## RAG ಅನ್ನು ಅರ್ಥಮಾಡಿಕೊಳ್ಳುವುದು

ಕೆಳಗಿನ ಚಿತ್ರವು ಪ್ರಮುಖ ತತ್ವವನ್ನು ತೋರುತ್ತದೆ: ಮಾದರಿಯ ತರಬೇತಿಯ ಡೇಟಾವನ್ನು ಮಾತ್ರ ಅವಲಂಬಿಸುವುದರ ಬದಲು, RAG ಅದಕ್ಕೆ ಪ್ರತಿಯೊಂದು ಉತ್ತರವನ್ನು ರಚಿಸುವ ಮುನ್ನ ನಿಮ್ಮ ದಸ್ತಾವೇಜುಗಳ ಸಂಬಂದ ಪುಸ್ತಕಾಲಯವನ್ನು ನೀಡುತ್ತದೆ.

<img src="../../../translated_images/kn/what-is-rag.1f9005d44b07f2d8.webp" alt="What is RAG" width="800"/>

ಇಲ್ಲಿ ಭಾಗಗಳು ಹೇಗೆ ಮುಗಿದು ಮುಗಿದಿವೆ ಎಂಬುದನ್ನು ವಿವರಿಸಲಾಗಿದೆ. ಬಳಕೆದಾರರ ಪ್ರಶ್ನೆ ನಾಲ್ಕು ಹಂತಗಳಿಂದ ಹರಿಯುತ್ತದೆ — ಎಂಬೆಡ್ಡಿಂಗ್, ವೆಕ್ಟರ್ ಶೋಧನೆ, ಸಂದರ್ಭ ಸಂಯೋಜನೆ, ಮತ್ತು ಉತ್ತರ ರಚನೆ — ಪ್ರತಿ ಹಂತವು ಹಿಂದಿನದನ್ನೇ ಆಧರಿಸಿದೆ:

<img src="../../../translated_images/kn/rag-architecture.ccb53b71a6ce407f.webp" alt="RAG Architecture" width="800"/>

ಈ ಭಾಗಗಳ ಪ್ರತಿಯೊಂದನ್ನು ವಿವರವಾಗಿ ಇತ್ತೀಚೆಗಿನ ಕೋಡ್ ಜೊತೆಗೆ ನೀವು ನಡಿಸಬಹುದು ಮತ್ತು ಬದಲಾಯಿಸಬಹುದು.

## ಇದು ಹೇಗೆ ಕೆಲಸ ಮಾಡುತ್ತದೆ

### ದಸ್ತಾವೇಜು ಪ್ರಕ್ರಿಯೆ

[DocumentService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/DocumentService.java)

ನೀವು ದಸ್ತಾವೇಜನ್ನು ಅಪ್‌ಲೋಡ್ ಮಾಡಿದಾಗ, ವ್ಯವಸ್ಥೆ ಅದನ್ನು ಪಾರ್ಸ್ ಮಾಡುತ್ತದೆ (PDF ಅಥವಾ ಸರಳ ಪಠ್ಯ), ಫೈಲ್‌ನಾಮ್ ಇತ್ಯಾದಿ ಮೆಟಾಡೇಟಾವನ್ನು ಸೇರಿಸುತ್ತದೆ ಮತ್ತು ನಂತರ ಅದನ್ನು ಚಂಕ್‌ಗಳಾಗಿ ವಿಭಾಗಿಸುತ್ತದೆ — ಮಾದರಿಯ ಕಾಂಟೆಕ್ಸ್ಟ್ ವಿಂಡೋಗೆ ಅನುಕೂಲವಾಗುವಷ್ಟು ಸಣ್ಣ ಭಾಗಗಳು. ಈ ಚಂಕ್‌ಗಳು ಸ್ವಲ್ಪ ತೊಂದರೆಗಳಾಗಿ ಹೊಂದಿದ್ದು, ಸೀಮೆಗಳ ಬಳಿ ಸಂಗತಿಯನ್ನು ಕಳೆದುಕೊಳ್ಳದಂತೆ ನೋಡಿಕೊಳ್ಳುತ್ತದೆ.

```java
// ಅಪ್ಲೋಡ್ ಮಾಡಲಾದ ಫೈಲ್ ಅನ್ನು ವಿಶ್ಲೇಷಿಸಿ ಮತ್ತು ಅದನ್ನು LangChain4j ಡಾಕ್ಯುಮೆಂಟ್‌ನಲ್ಲಿ ಸುತ್ತು ಮಾಡಿ
Document document = Document.from(content, metadata);

// 30-ಟೋಕನ್ ಓವರ್‌ಲೆಪ್‌ ಜೊತೆಗೆ 300-ಟೋಕನ್ ತುಂಡುಗಳಾಗಿ ವಿಭಜಿಸಿ
DocumentSplitter splitter = DocumentSplitters
    .recursive(300, 30);

List<TextSegment> segments = splitter.split(document);
```
  
ಕೆಳಗಿನ ಚಿತ್ರವು ಇದನ್ನು ದೃಶ್ಯಾತ್ಮಕವಾಗಿ ತೋರುತ್ತದೆ. ಪ್ರತಿ ಚಂಕ್ ಸುತ್ತಲೂ ತ್ವರಿತವಾಗಿ 30 ಟೋಕನ್‌ಗಳಷ್ಟು ಹಂಚಿಕೊಳ್ಳುತ್ತದೆ — ಇದು ದೊಡ್ಡ ಭಾಗಗಳಲ್ಲಿ ಸಾಂದರ್ಭಿಕ ಮಾಹಿತಿ ಕಳೆದುಕೊಳ್ಳದಂತೆ ಖಾತ್ರಿ ಪಡಿಸುತ್ತದೆ:

<img src="../../../translated_images/kn/document-chunking.a5df1dd1383431ed.webp" alt="Document Chunking" width="800"/>

> **🤖 [GitHub Copilot](https://github.com/features/copilot) ಚಾಟ್ ಬಳಸಿ ಪ್ರಯತ್ನಿಸಿ:** [`DocumentService.java`](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/DocumentService.java) ಓದಿ ಹಾಗೂ ಕೇಳಿ:  
> - "LangChain4j ಹೇಗೆ ದಸ್ತಾವೇಜುಗಳನ್ನು ಚಂಕ್‌ಗಳಿಗೆ ವಿಭಜಿಸುತ್ತದೆ ಮತ್ತು ಹಂಚಿಕೆಯನ್ನು ಮುಖ್ಯವೆಂದು ಯಾಕೆ ಹೇಳುತ್ತದೆ?"  
> - "ಬೇರೆ ದಸ್ತಾವೇಜು ಪ್ರಕಾರಗಳಿಗೆ ಯಥಾಸ್ಥಿತಿಚಂಕ್ ಗಾತ್ರ ಏನು ಮತ್ತು ಯಾಕೆ?"  
> - "ಬಹುಭಾಷಾ ಅಥವಾ ವಿಶೇಷ ಸ್ವರೂಪದ ದಸ್ತಾವೇಜುಗಳನ್ನು ನಿರ್ವಹಿಸುವ ವಿಧಾನ ಏನು?"

### ಎಂಬೆಡ್ಡಿಂಗ್ಸ್ ಸೃಷ್ಟಿಸುವುದು

[LangChainRagConfig.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/config/LangChainRagConfig.java)

ಪ್ರತಿ ಚಂಕ್ ಅನ್ನು ಸಂಖ್ಯಾತ್ಮಕ ಪ್ರತಿನಿಧಾನವಾಗಿರುವ ಎಂಬೆಡ್ಡಿಂಗ್ ಗೆ ಪರಿವರ್ತಿಸಲಾಗುತ್ತದೆ - ಇದು ಪಠ್ಯದ ಅರ್ಥವನ್ನು ಹಿಡಿದಿಡುವ ಗಣಿತೀಯ ಫಿಂಗರ್‌ಪ್ರಿಂಟ್‌. ಸಮಾನ ಪಠ್ಯ ಸಮಾನ ಎಂಬೆಡ್ಡಿಂಗ್ಸ್ ಉತ್ಪಾದಿಸುತ್ತದೆ.

```java
@Bean
public EmbeddingModel embeddingModel() {
    return OpenAiOfficialEmbeddingModel.builder()
        .baseUrl(azureOpenAiEndpoint)
        .apiKey(azureOpenAiKey)
        .modelName(azureEmbeddingDeploymentName)
        .build();
}

EmbeddingStore<TextSegment> embeddingStore = 
    new InMemoryEmbeddingStore<>();
```
  
ಕೆಳಗಿನ ವರ್ಗ ಚಿತ್ತರದಲ್ಲಿ LangChain4j ಘಟಕಗಳು ಹೇಗೆ ಸಂಪರ್ಕ ಹೊಂದಿವೆ ಎನ್ನುವುದು ತೋರುತ್ತದೆ. `OpenAiOfficialEmbeddingModel` ಪಠ್ಯವನ್ನು ವೆಕ್ಟರ್‌ಗಳಾಗಿ ಪರಿವರ್ತಿಸುತ್ತದೆ, `InMemoryEmbeddingStore` ಅವುಗಳನ್ನು ಮೂಲ `TextSegment` ಡೇಟಾ ಜೊತೆಗೆ ಇಡುತ್ತದೆ, ಮತ್ತು `EmbeddingSearchRequest` `maxResults` ಮತ್ತು `minScore` ಹೀಗೆ ಪುನಃಪ್ರಾಪ್ತಿಯ ನಿಯಂತ್ರಣಗಳನ್ನು ನೋಡಿಕೊಳ್ಳುತ್ತದೆ:

<img src="../../../translated_images/kn/rag-langchain4j-classes.bbf3aa9077ab443d.webp" alt="LangChain4j RAG Classes" width="800"/>

ಒಮ್ಮೆ ಎಂಬೆಡ್ಡಿಂಗ್ಸ್ ಸಂಗ್ರಹಿಸಿದ ಮೇಲೆ, ಸಮಾನ ವಿಷಯಗಳು ಸ್ವಾಭಾವಿಕವಾಗಿ ವೆಕ್ಟರ್ ಸ್ಥಳದಲ್ಲಿ ಕ್ಲಸ್ಟರ್ ಆಗುತ್ತವೆ. ಕೆಳಗಿನ ದೃಶ್ಯವು ಸಂಬಂಧಿತ ವಿಷಯಗಳ ದಸ್ತಾವೇಜುಗಳು ಸಮೀಪದ ಬಿಂದುಗಳಾಗಿ ಬರುವುದನ್ನು ತೋರುತ್ತದೆ, ಇದು ಸಾಮಾಜಿಕ ಶೋಧನೆಯನ್ನು ಸಾಧ್ಯಮಾಡುತ್ತದೆ:

### ಸಾಮಾಜಿಕ ಶೋಧನೆ

[RagService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/RagService.java)

ನೀವು ಪ್ರಶ್ನೆ ಕೇಳಿದಾಗ, ನಿಮ್ಮ ಪ್ರಶ್ನೆಯೂ ಕೂಡ ಎಂಬೆಡ್ಡಿಂಗ್ ಆಗುತ್ತದೆ. ವ್ಯವಸ್ಥೆ ನಿಮ್ಮ ಪ್ರಶ್ನೆಯ ಎಂಬೆಡ್ಡಿಂಗ್ ಅನ್ನು ಎಲ್ಲಾ ದಸ್ತಾವೇಜು ಚಂಕ್ ಗಳ ಎಂಬೆಡ್ಡಿಂಗ್ಸ್ ಜೊತೆ ಹೋಲಿಸುತ್ತದೆ. ಅದು ಅರ್ಥದಲ್ಲಿ ಅತ್ಯಂತ ಸಮಾನವಾದ ಚಂಕ್‌ಗಳನ್ನು ಹುಡುಕುತ್ತದೆ — ಕೇವಲ ಕೀವರ್ಡ್ ಮೀಟಿಂಗ್ ಅಲ್ಲ, ನಿಜವಾದ ಸಾಂದರ್ಭಿಕ ಸಾದೃಶ್ಯ.

```java
Embedding queryEmbedding = embeddingModel.embed(question).content();

EmbeddingSearchRequest searchRequest = EmbeddingSearchRequest.builder()
    .queryEmbedding(queryEmbedding)
    .maxResults(5)
    .minScore(0.5)
    .build();

EmbeddingSearchResult<TextSegment> searchResult = embeddingStore.search(searchRequest);
List<EmbeddingMatch<TextSegment>> matches = searchResult.matches();

for (EmbeddingMatch<TextSegment> match : matches) {
    String relevantText = match.embedded().text();
    double score = match.score();
}
```
  
ಕೆಳಗಿನ ಚಿತ್ರವು ಸಾಮಾಜಿಕ ಶೋಧನೆ ಮತ್ತು ಸಾಂಪ್ರದಾಯಿಕ ಕೀವರ್ಡ್ ಶೋಧನೆಯನ್ನು ಹೋಲಿಸುತ್ತದೆ. "vehicle" ಕೀವರ್ಡ್ ಶೋಧನೆ "cars and trucks" ಕುರಿತು ಚಂಕ್ ಒಂದನ್ನು ಮಿಸ್ ಮಾಡಬಹುದು, ಆದರೆ ಸಾಮಾಜಿಕ ಶೋಧನೆ ಅವು ಒಂದೇ ಅರ್ಥ ಎನ್ನುವ ತಿಳಿವಳಿಕೆ ಹೊಂದಿದ್ದು ಅದನ್ನು ಹೆಚ್ಚು ಅಂಕಗಳೊಂದಿಗೆ ತರುತ್ತದೆ:

<img src="../../../translated_images/kn/semantic-search.6b790f21c86b849d.webp" alt="Semantic Search" width="800"/>

> **🤖 [GitHub Copilot](https://github.com/features/copilot) ಚಾಟ್ ಬಳಸಿ ಪ್ರಯತ್ನಿಸಿ:** [`RagService.java`](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/RagService.java) ಓದಿ ಹಾಗೂ ಕೇಳಿ:  
> - "ಎಂಬೆಡ್ಡಿಂಗ್ಸ್ ಜೊತೆಗೆ ಸಾದೃಶ್ಯ ಶೋಧನೆ ಹೇಗೆ ಕೆಲಸ ಮಾಡುತ್ತದೆ ಮತ್ತು ಅಂಕೆಯನ್ನು ಏನು ನಿರ್ಧರಿಸುತ್ತದೆ?"  
> - "ಯಾವ ಸಾದೃಶ್ಯ ಮಿತಿ ಬಳಸಬೇಕು ಮತ್ತು ಅದು ಫಲಿತಾಂಶಗಳಿಗೆ ಹೇಗೆ ಪ್ರಭಾವ ಬೀರುತ್ತದೆ?"  
> - "ಸಂಬಂಧಿತ ದಸ್ತಾವೇಜುಗಳು ಕಂಡುಬರದ ಸಂದರ್ಭದಲ್ಲಿ ನಾನು ಹೇಗೆ ನಿರ್ವಹಿಸಬೇಕು?"

### ಉತ್ತರ ಉತ್ಪಾದನೆ

[RagService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/RagService.java)

ಸಂಬಂಧಿತ ಅತ್ಯುತ್ತಮ ಚಂಕ್‌ಗಳು ಸಜ್ಜುಗೊಳಿಸಲಾಗುವ ಪ್ರಾಂಪ್ಟ್‌ನೊಂದಿಗೆ ಸೇರಿಸಲಾಗುತ್ತವೆ, ಇದರಲ್ಲಿ ಸ್ಪಷ್ಟ ಸೂಚನೆಗಳು, ಪಡೆದ ಸಂದರ್ಭ ಮತ್ತು ಬಳಕೆದಾರನ ಪ್ರಶ್ನೆ ಸೇರಿವೆ. ಮಾದರಿ ಆ ನಿರ್ದಿಷ್ಟ ಚಂಕ್‌ಗಳನ್ನು ಓದಿ ಆ ಮಾಹಿತಿಯ ಆಧಾರದಲ್ಲಿ ಉತ್ತರ ನೀಡುತ್ತದೆ — ಇದು ದೃಢತೆಯ ರಚನೆ, ಅದೆಷ್ಟೋ ಮನೋವೈಕಲ್ಯವನ್ನು ತಡೆಯುತ್ತದೆ.

```java
String context = matches.stream()
    .map(match -> match.embedded().text())
    .collect(Collectors.joining("\n\n"));

String prompt = String.format("""
    Answer the question based on the following context.
    If the answer cannot be found in the context, say so.

    Context:
    %s

    Question: %s

    Answer:""", context, request.question());

String answer = chatModel.chat(prompt);
```
  
ಕೆಳಗಿನ ಚಿತ್ರವು ಈ ಸಂಯೋಜನೆಯನ್ನು ತೋರುತ್ತದೆ — ಶೋಧನೆಯಿಂದ ಅತ್ಯಧಿಕ ಅಂಕಗಳನ್ನು ಪಡೆದ ಚಂಕ್‌ಗಳನ್ನು ಪ್ರಾಂಪ್ಟ್ ಟೆಂಪ್ಲೇಟಿಗೆ ಸೇರ್ಪಡೆ ಮಾಡಲಾಗುತ್ತದೆ, ಮತ್ತು `OpenAiOfficialChatModel` ನೆಡೆಸಿದ ಉತ್ತರವನ್ನು ರಚಿಸುತ್ತದೆ:

### ದಸ್ತಾವೇಜನ್ನು ಅಪ್‌ಲೋಡ್ ಮಾಡಿ

ದಸ್ತಾವೇಜು ಅಪ್ಲೋಡ್ ಮಾಡುವುದು ಪ್ರಾರಂಭಿಸಿ — ಪರೀಕ್ಷೆಗಾಗಿ TXT ಫೈಲ್‌ಗಳು ಉತ್ತಮ ಕೆಲಸ ಮಾಡುತ್ತವೆ. ಈ ಡೈರೆಕ್ಟರಿಯಲ್ಲಿ `sample-document.txt` ಇದೆ, ಇದರಲ್ಲಿ LangChain4j ವೈಶಿಷ್ಟ್ಯಗಳ, RAG ಅನುಷ್ಠಾನದ ಮತ್ತು ಉತ್ತಮ ಅಭ್ಯಾಸಗಳ ಕುರಿತು ಮಾಹಿತಿಯಿದೆ — ವ್ಯವಸ್ಥೆಯನ್ನು ಪರೀಕ್ಷಿಸಲು ಇದು ಸೂಕ್ತವಾಗಿದೆ.

ವ್ಯವಸ್ಥೆ ನಿಮ್ಮ ದಸ್ತಾವೇಜನ್ನು ಸಂಸ್ಕರಿಸಿ ಅದರನ್ನ ಚಂಕ್‌ಗಳಾಗಿ ವಿಭಜಿಸಿ, ಪ್ರತಿ ಚಂಕ್‌ಗೆ ಎಂಬೆಡ್ಡಿಂಗ್ಸ್ ಸೃಷ್ಟಿಸುತ್ತದೆ. ಇದು ನೀವು ಅಪ್‌ಲೋಡ್ ಮಾಡಿದಾಗ ಸ್ವಯಂಚಾಲಿತವಾಗಿ ನಡೆಯುತ್ತದೆ.

### ಪ್ರಶ್ನೆಗಳ ಕೇಳಿ

ಈಗ ದಸ್ತಾವೇಜಿನ ವಿಷಯದ ಕುರಿತು ನಿರ್ದಿಷ್ಟ ಪ್ರಶ್ನೆಗಳನ್ನು ಕೇಳಿ. ಪಠ್ಯದಲ್ಲಿನ ಸ್ಮಾರಕವಾಗಿ ಸ್ಪಷ್ಟ ಮಾಹಿತಿಯನ್ನು ಕೇಳಿ. ವ್ಯವಸ್ಥೆ ಸಂಬಂಧಿತ ಚಂಕ್‌ಗಳನ್ನು ಹುಡುಕಿ ಅವುಗಳನ್ನು ಪ್ರಾಂಪ್ಟ್‌ನಲ್ಲಿ ಸೇರಿಸಿ, ಉತ್ತರವನ್ನು ರಚಿಸುತ್ತದೆ.

### ಮೂಲ ಉಲ್ಲೇಖಗಳನ್ನು ಪರಿಶೀಲಿಸಿ

ಪ್ರತಿ ಉತ್ತರವು ಮೂಲ ಉಲ್ಲೇಖಗಳನ್ನು ಮತ್ತು ಸಮಯೋಚಿತ ಅಂಕೆಗಳನ್ನು ಹೊಂದಿದೆ. ಈ ಅಂಕೆಗಳು (0 ರಿಂದ 1) ನಿಮ್ಮ ಪ್ರಶ್ನೆಗೆ ಸಂಬಂಧಿತ ಚಂಕ್ ಎಷ್ಟು ಸಂಬಂಧಿತವೋ ತೋರಿಸುತ್ತವೆ. ಹೆಚ್ಚು ಅಂಕೆಗಳು ಉತ್ತಮ ಹೊಂದಾಣಿಕೆಯನ್ನು ಸೂಚಿಸುತ್ತವೆ. ಇದರಿಂದ ನೀವು ಉತ್ತರವನ್ನು ಮೂಲದೊಂದಿಗೆ ಪರಿಶೀಲಿಸಬಹುದು.

### ಪ್ರಶ್ನೆಗಳೊಂದಿಗೆ ಪ್ರಯೋಗ ಮಾಡಿ

ಬೇರೆ ಬೇರೆ ರೀತಿಯ ಪ್ರಶ್ನೆಗಳನ್ನು ಪ್ರಯತ್ನಿಸಿ:  
- ನಿರ್ದಿಷ್ಟ ಸಂಗತಿಗಳು: "ಮುಖ್ಯ ವಿಷಯವೇನು?"  
- ಹೋಲಿಕೆಗಳು: "X ಮತ್ತು Y ನಡುವಿನ ವ್ಯತ್ಯಾಸವೇನು?"  
- ಸಾರಾಂಶಗಳು: "Z ಕುರಿತ ಪ್ರಮುಖ ಅಂಶಗಳನ್ನು ಸಾರಾಂಶ ಮಾಡು"

ನಿಮ್ಮ ಪ್ರಶ್ನೆ ಹಾಗೂ ದಸ್ತಾವೇಜಿನ ವಿಷಯದ ಹೊಂದಾಣಿಕೆಯ ಆಧಾರದಲ್ಲಿ ಅಂಕೆಗಳು ಹೇಗೆ ಬದಲಾಗುತ್ತವೆ ನೋಡಿರಿ.

## ಪ್ರಮುಖ ತತ್ವಗಳು

### ಚಂಕಿಂಗ್ ತಂತ್ರ

ದಸ್ತಾವೇಜುಗಳನ್ನು 300-ಟೋಕನ್ ಚಂಕ್‌ಗಳಾಗಿ ವಿಭಜಿಸಲಾಗುತ್ತದೆ, 30 ಟೋಕನ್ ಜಾಸ್ತಿ ಹೊಂದುವ ಹಂಚಿಕೆ ಇದೆ. ಈ ಸಮತೋಲನವು ಪ್ರತಿ ಚಂಕ್ ಸಾಕಷ್ಟು ಸಂದರ್ಭವಿದ್ದು ಅರ್ಥಪೂರ್ಣವಾಗುವಂತೆ ನಿರ್ವಹಿಸುತ್ತದೆ ಮತ್ತು ಪ್ರಾಂಪ್ಟ್‌ನಲ್ಲಿ ಬಹು ಚಂಕ್‌ಗಳಿಗೆ ಅವಕಾಶ ನೀಡುತ್ತದೆ.

### ಸಾದೃಶ್ಯ ಅಂಕೆಗಳು

ಪ್ರತಿಯೊಬ್ಬ ಸಮೀಕ್ಷಿತ ಚಂಕ್ 0 ರಿಂದ 1 ಮಧ್ಯದ ಸಾದೃಶ್ಯ ಅಂಕೆಯೊಂದನ್ನು ಹೊಂದಿದೆ, ಇದು ಬಳಕೆದಾರನ ಪ್ರಶ್ನೆಗೆ ಎಷ್ಟು ಸಮೀಪವಾಗಿದೆ ಎಂಬುದನ್ನು ಸೂಚಿಸುತ್ತದೆ. ಕೆಳಗಿನ ಚಿತ್ರವು ಅಂಕೆಗಳು ಹಾಗೂ ವ್ಯವಸ್ಥೆ ಅವುಗಳನ್ನು ಫಲಿತಾಂಶಗಳನ್ನ ಫಿಲ್ಟರ್ ಮಾಡಲು ಹೇಗೆ ಬಳಸುತ್ತದೆ ತೋರುತ್ತದೆ:

<img src="../../../translated_images/kn/similarity-scores.b0716aa911abf7f0.webp" alt="Similarity Scores" width="800"/>

ಅಂಕೆಗಳ ಶ್ರೇಣಿ 0 ರಿಂದ 1:  
- 0.7-1.0: ಅತ್ಯಂತ ಸಂಬಂಧಿತ, ಸರಿಯಾದ ಹೊಂದಾಣಿಕೆ  
- 0.5-0.7: ಸಂಬಂಧಿತ, ಉತ್ತಮ ಸಂದರ್ಭ  
- 0.5 ಕೆಳಗೆ: ಫಿಲ್ಟ್ ಆಗಿದ್ದು, ಬಹಳ ಬದಲಾಗಿರುವುದು  

ವ್ಯವಸ್ಥೆ ಕಡಿಮೆ ಮಿತಿ ಮೀರಿ ಚಂಕ್‌ಗಳನ್ನು ಮಾತ್ರ ಪುನಃಪ್ರಾಪ್ತಿಮಾಡುತ್ತದೆ, ಗುಣಮಟ್ಟ ಖಾತರಿಪಡಿಸಲು.

### ಇನ್-ಮೆಮೊರಿ ಸಂಗ್ರಹಣೆ

ಈ ಮಾಡ್ಯೂಲ್ ಸರಳತೆಗೆ ಇನ್-ಮೆಮೊರಿ ಸಂಗ್ರಹಣೆಯನ್ನು ಬಳಸುತ್ತದೆ. ನೀವು ಅಪ್ಲಿಕೇಶನ್ ಅನ್ನು ಪುನರಾರಂಭಿಸಿದಾಗ, ಅಪ್‌ಲೋಡ್ ಮಾಡಲಾದ ದಸ್ತಾವೇಜುಗಳು ಕಳೆದುಹೋಗುತ್ತವೆ. ಉತ್ಪಾದನಾ ವ್ಯವಸ್ಥೆಗಳು Qdrant ಅಥವಾ Azure AI ಶೋಧನಾ ಮಾದರಿಗಳನ್ನು ಬಳಸುತ್ತವೆ.

### ಸಂದರ್ಭ ವಿಂಡೋ ನಿರ್ವಹಣೆ

ಪ್ರತಿ ಮಾದರಿಗೆ ಗರಿಷ್ಠ ಸಂದರ್ಭ ವಿಂಡೋ ಇರುತ್ತದೆ. ದೊಡ್ಡ ದಸ್ತಾವೇಜಿನಿಂದ ಎಲ್ಲಾ ಚಂಕ್‌ಗಳನ್ನು ಸೇರಿಸಲು ಸಾಧ್ಯವಿಲ್ಲ. ವ್ಯವಸ್ಥೆ ಅತಿ ಸಂಬಂಧಿತ ಚಂಕ್‌ಗಳ ಟೋಪ್ N (ಮಾನ್ಯವಾಗಿ 5) ಅನ್ನು ಆಯ್ಕೆಮಾಡಿ, ಮಿತಿಗಳನ್ನು ಪಾಲಿಸುವುದರ ಜೊತೆಗೆ ಸಮರ್ಪಕ ಉತ್ತರಗಳಿಗೆ ಸಾಕಷ್ಟು ಸಂದರ್ಭ ನೀಡುತ್ತದೆ.

## ಯಾವಾಗ RAG ಪ್ರಾಮುಖ್ಯತೆ ಹೊಂದಿದೆ

RAG ಯಾವಾಗಲೂ ಸರಿಯಾದ ವಿಧಾನವಾಗಿರುವುದಿಲ್ಲ. ಕೆಳಗಿನ ನಿರ್ಣಯ ಮಾರ್ಗದರ್ಶಿ ನಿಮಗೆ RAG ಮೌಲ್ಯವನ್ನೇಯೆ ಸೇರಿಸುವಾಗ ಮತ್ತು ಸುಲಭ ವಿಧಾನಗಳು — ಉದಾಹರಣೆಗೆ ಪ್ರಾಂಪ್ಟ್‌ನಲ್ಲಿ ವಿಷಯವನ್ನು ನೇರವಾಗಿ ಸೇರಿಸುವುದು ಅಥವಾ ಮಾದರಿಯ ಒಳನಿವಾರ ಜ್ಞಾನವನ್ನು ಅವಲಂಬಿಸುವುದು — ಸಾಕಾಗುವಾಗವನ್ನು ನಿರ್ಧರಿಸಲು ಸಹಾಯ ಮಾಡುತ್ತದೆ:

<img src="../../../translated_images/kn/when-to-use-rag.1016223f6fea26bc.webp" alt="When to Use RAG" width="800"/>

**RAG ಉಪಯೋಗಿಸಿ:**
- ಸ್ವಂತಿಕೃತ ದಾಖಲೆಗಳ ಬಗ್ಗೆ ಪ್ರಶ್ನೆಗಳಿಗೆ ಉತ್ತರಿಸುವುದು  
- ಮಾಹಿತಿ ಅಧಿನವಾಗಿ ಬದಲಾಗುತ್ತದೆ (ನೀತಿಗಳು, ಬೆಲೆಗಳು, ನಿರ್ದಿಷ್ಟತೆಗಳು)  
- ಸೂಕ್ಷ್ಮತೆಗಾಗಿ ಮೂಲ ಸೂಚನೆ ಅಗತ್ಯವಿದೆ  
- ವಿಷಯವು ಒಂದು ಪ್ರಾಂಪ್ಟ್‌ನಲ್ಲಿ ಹೊಂದಿಸಲು ತುಂಬಾ ದೊಡ್ಡದು  
- ನೀವು ಪರಿಶೀಲನೀಯ, ಆಧಾರಿತ ಉತ್ತರಗಳನ್ನು ಬೇಕಾಗಿರುವುದು  

**RAG ಬಳಸದಿರಬೇಕಾದಾಗ:**  
- ಪ್ರಶ್ನೆಗಳು ಮಾದರಿಯಲ್ಲಿರುವ ಸಾಮಾನ್ಯ ಜ್ಞಾನದ ಅವಶ್ಯಕತೆ ಇಲ್ಲದಿದ್ದಾಗ  
- ನೈಜ-ಸಮಯದ ಡೇಟಾ ಬೇಕಾಗಿದ್ದಾಗ (RAG ಅಪ್ಲೋಡ್ ಮಾಡಲಾದ ದಾಖಲೆಗಳ ಮೇಲೆ ಕೆಲಸಮಾಡುತ್ತದೆ)  
- ವಿಷಯವು ಪ್ರಾಂಪ್ಟ್‌ಗಳಲ್ಲಿ ನೇರವಾಗಿ ಸೇರಿಸಲು ಸಾಕಷ್ಟಾಗಿದ್ದಾಗ
