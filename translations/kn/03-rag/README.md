## ನೀವು ಕಲಿಯುವದ್ದು ಏನು

ಹಿಂದಿನ ಮಾಡ್ಯೂಲ್‌ಗಳಲ್ಲಿ, ನೀವು AI ಜೊತೆ ಸಂಭಾಷಣೆ ನಡೆಸುವುದು ಮತ್ತು ನಿಮ್ಮ ಪ್ರಾಂಪ್ಟ್‌ಗಳನ್ನು ಪರಿಣಾಮಕಾರಿಯಾಗಿ ರಚಿಸುವುದನ್ನು ಕಲಿತಿದ್ದೀರಿ. ಆದರೆ ಮೂಲಭೂತವಾಗಿ ಒಂದು ನಿರ್ಬಂಧವಿದೆ: ಭಾಷಾ ಮಾದರಿಗಳು ತರಬೇತಿಯ ಸಮಯದಲ್ಲಿ ಮಾತ್ರ ತಿಳಿದಿರುವದನ್ನು ಮಾತ್ರ ಅರಿತಿರುತ್ತವೆ. ಅವರು ನಿಮ್ಮ ಕಂಪನಿಯ ನೀತಿಗಳು, ನಿಮ್ಮ ಪ್ರಾಜೆಕ್ಟ್ ಡಾಕ್ಯುಮೆಂಟೇಶನ್ ಅಥವಾ ತರಬೇತಿ ಮಾಡದ ಯಾವುದೇ ಮಾಹಿತಿಯನ್ನು ಪ್ರತ್ಯುತ್ತರಿಸಲು ಸಾಧ್ಯವಿಲ್ಲ.

RAG (Retrieval-Augmented Generation) ಈ ಸಮಸ್ಯೆಯನ್ನು ಪರಿಹರಿಸುತ್ತದೆ. ಮಾದರಿಯನ್ನು ನಿಮ್ಮ ಮಾಹಿತಿಯನ್ನು ಕಲಿಸುವ ಪ್ರಯತ್ನ ಮಾಡುವ ಬದಲು (ಅದು ದುಬಾರಿ ಮತ್ತು ಅಪ್ರಾಯಕ್ತಿಕವಾಗಿದೆ), ನೀವು ಅದನ್ನು ನಿಮ್ಮ ದಾಖಲೆಗಳಲ್ಲಿ ಹುಡುಕಾಟ ಮಾಡಲು ಶಕ್ತಿ ನೀಡುತ್ತೀರಿ. ಯಾರಾದರೂ ಪ್ರಶ್ನೆ ಕೇಳಿದಾಗ, ವ್ಯವಸ್ಥೆ ಸಂಬಂಧಿತ ಮಾಹಿತಿಯನ್ನು ಕಂಡು ಆ ಪ್ರಾಂಪ್ಟ್‌ನಲ್ಲಿ ಸೇರಿಸುತ್ತದೆ. ನಂತರ ಮಾದರಿ ಆ ಪಡೆದುಕೊಂಡ ಸಂದರ್ಭ ಆಧರಿಸಿ ಉತ್ತರಿಸುತ್ತದೆ.

RAG ಅನ್ನು ಆದರ್ಶ ಗ್ರಂಥಾಲಯ ಒದಗಿಸುವಂತೆ ಪರಿಗಣಿಸಿ. ನೀವು ಪ್ರಶ್ನೆ ಕೇಳಿದಾಗ ವ್ಯವಸ್ಥೆ:

1. **ಬಳಕೆದಾರ ಪ್ರಶ್ನೆ** - ನೀವು ಪ್ರಶ್ನೆ ಕೇಳುತ್ತೀರಿ
2. **ಎಂಬೆಡ್ಡಿಂಗ್** - ನಿಮ್ಮ ಪ್ರಶ್ನೆಯನ್ನು ಒಂದು ವೆಕ್ಟರ್ ಆಗಿ ಪರಿವರ್ತಿಸುತ್ತದೆ
3. **ವೆಕ್ಟರ್ ಶೋಧನೆ** - ಸಾದೃಶ್ಯವಿರುವ ದಾಖಲೆ ಚಂಕ್‌ಗಳನ್ನು ಹುಡುಕುತ್ತದೆ
4. **ಸಂದರ್ಭ ಸಂಯೋಜನೆ** - ಸಂಬಂಧಿತ ಚಂಕ್‌ಗಳನ್ನು ಪ್ರಾಂಪ್ಟ್‌ಗೆ ಸೇರಿಸುತ್ತದೆ
5. **ಪ್ರತ್ಯುತ್ತರ** - LLM ಸಂದರ್ಭ ಆಧರಿಸಿ ಉತ್ತರ ಉತ್ಪಾದಿಸುತ್ತದೆ

ಈ ಮೂಲಕ ಮಾದರಿಯ ಪ್ರತಿಕ್ರಿಯೆಗಳು ಅದರ ತರಬೇತಿ ಜ್ಞಾನ ಅಥವಾ ಕற்பನೆಗಳಿಗೆ ಅವಲಂಬಿಸದೇ ನಿಮ್ಮ ವಾಸ್ತವ ಡೇಟಾ ಮೇಲೆ ಪ್ರತಿಷ್ಠಿತವಾಗುತ್ತವೆ.

## RAG ಅನ್ನು ಅರ್ಥಮಾಡಿಕೊಳ್ಳುವುದು

ಕೆಳಗಿನ ಚಿತ್ರವು ಮೂಲ ಕಲ್ಪನೆಯನ್ನು ತೋರಿಸುತ್ತದೆ: ಮಾದರಿಯ ತರಬೇತಿ ಡೇಟಾ ಮೇಲೆ ಮಾತ್ರ ಅವಲಂಬಿಸದೆ, RAG ಅದಕ್ಕೆ ಪ್ರತಿಕ್ರಿಯೆನು ಉತ್ಪಾದಿಸುವ ಮೊದಲು ನಿಮ್ಮ ದಾಖಲೆಗಳ ಸಂಕಲನಾ ಗ್ರಂಥಾಲಯವನ್ನು ಒದಗಿಸುತ್ತದೆ.

<img src="../../../translated_images/kn/what-is-rag.1f9005d44b07f2d8.webp" alt="RAG ಎಂದರೆ ಏನು" width="800"/>

*ಈ ಚಿತ್ರವು ಸಾಮಾನ್ಯ LLM (ತರಬೇತಿಯ ಡೇಟಾದಿಂದ ಊಹಿಸುವ) ಮತ್ತು RAG ಹೆಚ್ಚಿದ LLM (ನಿಮ್ಮ ದಾಖಲೆಗಳನ್ನು ಮೊದಲು ಪರಿಶೀಲಿಸುವ) ನಡುವಿನ ವ್ಯತ್ಯಾಸವನ್ನು ತೋರಿಸಿದೆ.*

ಇದೀಗ ಹೀಗಾಗಿ ಒಳಗುಳಿ ಪ್ರಕ್ರಿಯೆ ಸೂಚಿಸಲಾಗಿದೆ. ಬಳಕೆದಾರರ ಪ್ರಶ್ನೆ ನಾಲ್ಕು ಹಂತಗಳ ಮೂಲಕ ಹರಿಯುತ್ತದೆ — ಎಂಬೆಡ್ಡಿಂಗ್, ವೆಕ್ಟರ್ ಶೋಧನೆ, ಸಂದರ್ಭ ಸಂಯೋಜನೆ, ಮತ್ತು ಉತ್ತರ ಉತ್ಪಾದನೆ — ಪ್ರತಿಯೊಂದು ಹಂತ ಹಿಂದಿನದರ ಮೇಲೆ ಆಧಾರಿತ:

<img src="../../../translated_images/kn/rag-architecture.ccb53b71a6ce407f.webp" alt="RAG ವಾಸುಕಟ್ಟಿಗ념" width="800"/>

*ಈ ಚಿತ್ರವು ಮುಗಿಯುವ RAG ಪೈಪ್‌ಲೈನ್ ತೋರಿಸುತ್ತದೆ — ಬಳಕೆದಾರ ಪ್ರಶ್ನೆ ಎಂಬೆಡ್ಡಿಂಗ್, ವೆಕ್ಟರ್ ಶೋಧನೆ, ಸಂದರ್ಭ ಸಂಯೋಜನೆ ಮತ್ತು ಉತ್ತರ ಉತ್ಪಾದನೆ ಮೂಲಕ ಹರಿಯುತ್ತದೆ.*

ಈ ಮಾಡ್ಯೂಲ್ ಉಳಿದ ಭಾಗವು ಪ್ರತಿ ಹಂತವನ್ನು ಕೇಸ್ ಕೋಡ್ ಮತ್ತು ಚಿತ್ರಗಳೊಂದಿಗೆ ವಿವರವಾಗಿ ಸಾರುತ್ತದೆ.

### ಈ ಟ್ಯುಟೋರಿಯಲ್ ಯಾವ RAG ವಿಧಾನವನ್ನು ಬಳಸುತ್ತದೆ?

LangChain4j ಮೂರು ರೀತಿಯ RAG ಅನುಷ್ಠಾನ ನೀಡುತ್ತದೆ, ಪ್ರತಿ ಒಂದೂ ರೀತಿ ವಿಭಿನ್ನ ಮಟ್ಟದ ಅವಿಭಾಜ್ಯತೆ ಹೊಂದಿದೆ. ಕೆಳಗಿನ ಚಿತ್ರವು ಅವುಗಳ ಹೋಲಿಕೆಯನ್ನು ತೋರಿಸುತ್ತದೆ:

| **Native RAG** | ನೀವು ಎಂದಾಗ embedding ಮಾದರಿಯನ್ನು ಕರೆದು, ಸ್ಟೋರ್ ಅನ್ನು ಹುಡುಕಿ, ಪ್ರಾಂಪ್ಟ್ ರಚಿಸಿ, ಉತ್ತರವನ್ನು ನೆರವೇರಿಸುವುದು—ಪ್ರತಿ ಹಂತವನ್ನು ಸ್ಪಷ್ಟವಾಗಿ ಹಸ್ತಪದಿ ನಿಯಂತ್ರಿಸುವುದು. | ಹೆಚ್ಚು ಕೋಡ್ ಇದೆ, ಆದರೆ ಪ್ರತಿ ಹಂತ ಸ್ಪಷ್ಟವಾಗಿ ಗೋಚರಿಸುತ್ತದೆ ಮತ್ತು ಸವಿದಬಹುದಾಗಿದೆ. |
| **Advanced RAG** | `RetrievalAugmentor` ಫ್ರೇಮ್‌ವರ್ಕ್ ಬಳಸಿ, ಪ್ರಶ್ನೆ ಪರಿವರ್ತಕರು, ರೂಟರ್‌ಗಳು, ಪುನರ್ನಿರ್ದೇಶಕರು ಮತ್ತು ವಿಷಯ ಸೇರ್ಪಡೆದಾರರೊಂದಿಗೆ ಉತ್ಪಾದನಾ ಮಟ್ಟದ ಪೈಪ್‌ಲೈನ್ಗಳಿಗೆ ಮಾರುಕಟ್ಟೆ. | ಅತ್ಯಧಿಕ ಲವಚಿಕತೆ, ಆದರೆ ಬಹಳ ಹೆಚ್ಚು ಜಟಿಲತೆ. |

**ಈ ಟ್ಯುಟೋರಿಯಲ್ ನಲ್ಲಿಇ ನೇಟಿವ್ ವಿಧಾನವನ್ನು ಬಳಸಲಾಗುತ್ತದೆ.** RAG ಪೈಪ್‌ಲೈನ್‌ನ ಪ್ರತಿ ಹಂತ — ಪ್ರಶ್ನೆಯನ್ನು ಎನ್ನಿಸುವುದು, ವೆಕ್ಟರ್ ಸ್ಟೋರ್‌ನಲ್ಲಿ ಹುಡುಕುವುದು, ಸಂಧರ್ಭವನ್ನು ಸಂಯೋಜಿಸುವುದು, ಮತ್ತು ಉತ್ತರವನ್ನು ಇತರಿಸುವುದು — ಎಲ್ಲವು [`RagService.java`](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/RagService.java) ನಲ್ಲಿ ಸ್ಪಷ್ಟವಾಗಿ ಬರೆಯಲಾಗಿದೆ. ಇದು ಜ್ಞಾನಾರ್ಜನೆಗಾಗಿ ಉದ್ದೇಶಿತ: ಪ್ರತಿ ಹಂತವನ್ನು ನೀವು ನೋಡಿ ಅರ್ಥಮಾಡಿಕೊಳ್ಳುವುದು ಅತ್ಯಂತ ಮುಖ್ಯ; ಕೋಡ್ ಕಡಿಮೆ ಮಾಡುವುದು ಅಷ್ಟೂ ಮುಖ್ಯವಲ್ಲ. ಒಂದುಬಾರಿ ನೀವು ತಂತ್ರಗಳನ್ನು ಅರ್ಥ ಮಾಡಿಕೊಂಡು ಅದಕ್ಕೆ ಅನುಗುಣವಾಗಿ ಸುಲಭ RAG ಅಥವಾ ಉನ್ನತ RAG ಗೆ ಮುಂದಾಗಬಹುದು.

> **💡 ನೀವು ಈಗಾಗಲೇ Easy RAG ಅನ್ನು ನೋಡಿದ್ದೀರಾ?** [Quick Start ಮಾಡ್ಯೂಲ್](../00-quick-start/README.md) ಒಂದು ಡಾಕ್ಯುಮೆಂಟ್ ಪ್ರಶ್ನೋತ್ತರ ಉದಾಹರಣೆಯನ್ನು ಒಳಗೊಂಡಿದೆ ([`SimpleReaderDemo.java`](../../../00-quick-start/src/main/java/com/example/langchain4j/quickstart/SimpleReaderDemo.java)) ಇದು Easy RAG ವಿಧಾನವನ್ನು ಬಳಸುತ್ತದೆ — LangChain4j ತನ್ನಲ್ಲೇ ಎಂಬೆಡ್ಡಿಂಗ್, ಶೋಧನೆ ಮತ್ತು ಪ್ರಾಂಪ್ಟ್ ಸಂಯೋಜನೆಯನ್ನು ನಿರ್ವಹಿಸುತ್ತದೆ. ಈ ಮಾಡ್ಯೂಲ್ ಅದೇ ಪೈಪ್‌ಲೈನನ್ನು ತೆರೆಯುತ್ತದೆ ಹಾಗೆಯೇ ಪ್ರತಿ ಹಂತವನ್ನು ನೀವು ನೋಡಿ ನಿಯಂತ್ರಿಸಬಹುದು.

<img src="../../../translated_images/kn/easy-rag-pipeline.2e1602e2ad2ded42.webp" alt="Easy RAG Pipeline - LangChain4j" width="800"/>

*ಈ ಚಿತ್ರವು `SimpleReaderDemo.java` ಯಿಂದ Easy RAG ಪೈಪ್‌ಲೈನನ್ನು ತೋರಿಸುತ್ತದೆ. ಇದನ್ನು ಈ ಮಾಡ್ಯೂಲ್‌ನ ನೇಟಿವ್ ವಿಧಾನದೊಂದಿಗೆ ಹೋಲಿಸಿ: Easy RAG ಎಂಬೆಡ್ಡಿಂಗ್, ಅನುಪ್ರಾಪ್ತಿಯ ಮತ್ತು ಪ್ರಾಂಪ್ಟ್ ಸಂಯೋಜನೆಯನ್ನು `AiServices` ಮತ್ತು `ContentRetriever` પાછળ ಹೊಡೆದಿರುತ್ತದೆ — ನೀವು ಒಂದು ಡಾಕ್ಯುಮೆಂಟ್ ಅಪ್‌ಲೋಡ್ ಮಾಡಿ, ರಿಟ್ರೀವರ್ ಜೋಡಿಸಿ, ಮತ್ತು ಉತ್ತರಗಳನ್ನು ಪಡೆಯಿರಿ. ನೇಟಿವ್ ವಿಧಾನ ಪ್ರತಿ ಹಂತವನ್ನು ತೆರೆಯುತ್ತದೆ ಮತ್ತು ನೀವು ಸ್ಪಷ್ಟವಾಗಿ ಅಳಿಸಲೂ ಮತ್ತು ನಿಯಂತ್ರಿಸಬಹುದಾಗಿದೆ.*

## ಇದು ಹೇಗೆ ಕಾರ್ಯನಿರ್ವಹಿಸುತ್ತದೆ

ಈ ಮಾಡ್ಯೂಲ್‌ನ RAG ಪೈಪ್‌ಲೈನ್ ಪ್ರತಿಯೊಬ್ಬ ಬಳಕೆದಾರ ಪ್ರಶ್ನೆ ಕೇಳುವಾಗ ನಾಲ್ಕು ಹಂತಗಳಲ್ಲಿ ಕ್ರಮವಾಗಿ ನಡೆಯುತ್ತದೆ. ಮೊದಲು, ಅಪ್‌ಲೋಡ್ ಮಾಡಿದ ಡಾಕ್ಯುಮೆಂಟ್ ಅನ್ನು **ವಿಚ್ಛಿನ್ನ ಮಾಡಿ ಚಂಕ್‌ಗಳಾಗಿ** ಕಡಿತಗೊಳಿಸಲಾಗುತ್ತದೆ. ಆ ಚಂಕ್‌ಗಳನ್ನು ನಂತರ **ವೆಕ್ಟರ್.embeddings** ಗೆ ಪರಿವರ್ತಿಸಿ ಸಂಗ್ರಹಿಸಲಾಗುತ್ತದೆ, ಕೂಡಲೇ ಗಣಿತಾತ್ಮಕ ಹೋಲಿಕೆ ಮಾಡಲು. ಪ್ರಶ್ನೆ ಬಂದಾಗ, ವ್ಯವಸ್ಥೆ **ಅರ್ಥಪೂರ್ಣ ಶೋಧನೆಯನ್ನು** ನಡಿಸಿ ಅತ್ಯಂತ ಸಂಬಂಧಿತ ಚಂಕ್‌ಗಳನ್ನು ಹುಡುಕುತ್ತದೆ, ಮತ್ತು ಅವುಗಳನ್ನು LLM ಗೆ **ಉತ್ತರ ಉತ್ಪಾದನೆಗೆ** ಕಳುಹಿಸುತ್ತದೆ. ಕೆಳಗಿನ ಭಾಗಗಳು ಪ್ರತಿ ಹಂತವನ್ನು ಜೋಡಣೆ ಕೋಡ್ ಮತ್ತು ಚಿತ್ರಗಳೊಂದಿಗೆ ವಿವರಿಸುವುವು. ಮೊದಲು ಹಂತವನ್ನು ನೋಡೋಣ.

### ದಾಖಲೆ ಪ್ರಕ್ರಿಯೆ

[DocumentService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/DocumentService.java)

ನೀವು ಒಂದು ಡಾಕ್ಯುಮೆಂಟ್ ಅನ್ನು ಅಪ್‌ಲೋಡ್ ಮಾಡಿದಾಗ, ವ್ಯವಸ್ಥೆ ಅದನ್ನು (PDF ಅಥವಾ ಸರಳ ಪಠ್ಯ) ವಿಶ್ಲೇಷಿಸಿ, ಫೈಲ್ ಹೆಸರಿನಂತಹ ಮೆಟಾಡೇಟಾಗಳನ್ನು ಒದಗಿಸಿ, ನಂತರ ಅದನ್ನು ಚಂಕ್‌ಗಳಾಗಿ ವಿಭಜಿಸುತ್ತದೆ — ಮಾದರಿಯ ಸಂದರ್ಭ ವಿಂಡೋಗೆ ಅನುಕೂಲಕರವಾಗುವ ತುಂಡುಗಳು. ಇವು ಪರಸ್ಪರ ಸ್ವಲ್ಪ ಭಾಗಗಳನ್ನು ಹಂಚಿಕೊಳ್ಳುತ್ತವೆ ताकि ಗಡಿಯಲ್ಲಿ ಯಾವುದೇ ಪ್ರಮುಖ ಸಂದರ್ಭ ಕಳೆದುಕೊಡದು.

```java
// ಅಪ್ಲೋಡ್ ಮಾಡಲಾದ ಫೈಲ್ ಅನ್ನು ವಿಶ್ಲೇಷಿಸಿ ಮತ್ತು ಅದನ್ನು LangChain4j ಡಾಕ್ಯುಮೆಂಟ್‌ ನಲ್ಲಿ ಲಪೇಟಿಸಿ
Document document = Document.from(content, metadata);

// 300 ಟೋಕನ್ ಭಾಗಗಳಾಗಿ ವಿಭಜಿಸಿ 30 ಟೋಕನ್ ಒಹೊვილ್ಪಟ್ಟನೆಯೊಂದಿಗೆ
DocumentSplitter splitter = DocumentSplitters
    .recursive(300, 30);

List<TextSegment> segments = splitter.split(document);
```

ಕೆಳಗಿನ ಚಿತ್ರವು ಇದನ್ನು ದೃಶ್ಯರೂಪದಲ್ಲಿ ತೋರಿಸುತ್ತದೆ. ಪ್ರತಿ ಚಂಕ್ ತನ್ನ ನೆರೆಹೊರೆಯವರೊಂದಿಗೆ ಕೆಲವು ಟೋಕನ್‌ಗಳನ್ನು ಹಂಚಿಕೊಳ್ಳುತ್ತದೆ — 30-ಟೋಕನ್ ಮಿಲನದಿಂದ ಯಾವುದೇ ಮುಖ್ಯ ಸಂದರ್ಭ ಕುಸಿತವಾಗುವುದಿಲ್ಲ:

<img src="../../../translated_images/kn/document-chunking.a5df1dd1383431ed.webp" alt="ದಾಖಲೆ ಚಂಕಿಂಗ್" width="800"/>

*ಈ ಚಿತ್ರವು 300-ಟೋಕನ್ ಚಂಕ್‌ಗಳಿಗೆ 30-ಟೋಕನ್ ಮಿಲನವನ್ನು ಹೊಂದಿಸುವ ಮೂಲಕ, ಗಡಿಯಲ್ಲಿನ ಸಂದರ್ಭದಲ್ಲಿ ಕಳೆದುಕೊಳ್ಳುವಿಕೆಯನ್ನು ತಡೆಯುತ್ತಿರುವಿಕೆ.*

> - "ವಿಭಿನ್ನ ದಸ್ತಾವೇಜು ಪ್ರಕಾರಗಳಿಗಾಗಿ ಅದ್ಭುತ ಚಂಕ್ ಗಾತ್ರ ಏನು ಮತ್ತು ಯಾಕೆ?"
> - "ಬಹುಭವಿಷ್ಯದ ಭಾಷೆಗಳ ದಸ್ತಾವೇಜುಗಳನ್ನು ಅಥವಾ ವಿಶೇಷ ಸ್ವರೂಪಗಳನ್ನು ನಾನು ಹೇಗೆ ನಿರ್ವಹಿಸಬೇಕು?"

### ಎಂಬೆಡ್ಡಿಂಗ್ಸ್ ಸೃಷ್ಟಿಸುವುದು

[LangChainRagConfig.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/config/LangChainRagConfig.java)

ಪ್ರತಿಯೊಂದು ಚಂಕ್ ಅನ್ನು ಸಂಖ್ಯಾತ್ಮಕ ಪ್ರತಿನಿಧಿಸುವEmbeddings ಎಂದು ಕರೆಯಲಾಗುತ್ತದೆ — ಅರ್ಥವನ್ನು ಸಂಖ್ಯೆಗಳಿಗೆ ಪರಿವರ್ತಿಸುವುದಾದದ್ದು. ಎಂಬೆಡ್ಡಿಂಗ್ ಮಾದರಿ "ಚಾಟ್ ಮಾದರಿಯಂತಿಲ್ಲ" — ಅದು ಸೂಚನೆಗಳನ್ನು ಅನುಸರಿಸುವುದು, ನಿರೂಪಣೆ ಮಾಡುವುದು ಅಥವಾ ಪ್ರಶ್ನೆಗಳಿಗೆ ಉತ್ತರಿಸುವುದು ಸಾಧ್ಯವಿಲ್ಲ. ಆದರೆ ಅದು ಪಠ್ಯವನ್ನು ಗಣಿತೀಯ ಸ್ಥಳದಲ್ಲಿ ರಚಿಸುವುದನ್ನು ಸಹಾಯ ಮಾಡುತ್ತದೆ, ಅಲ್ಲಿ ಸಾದೃಶ್ಯಾರ್ಥಗಳು ಹತ್ತಿರದಲ್ಲಿರುತ್ತವೆ — "ಕಾರ್" ಮತ್ತು "ಆಟೋಮೊಬೈಲ್," "ಹಣ ಹಿಂತಿರುಗಿಸುವ ನೀತಿ" ಮತ್ತು "ನನ್ನ ಹಣವನ್ನು ಹಿಂತಿರುಗಿಸು". ಚಾಟ್ ಮಾದರಿಯನ್ನು ನೀವು ಮಾತನಾಡಬಹುದಾದ ವ್ಯಕ್ತಿ ಎಂದು ಪರಿಗಣಿಸಿ; ಎಂಬೆಡ್ಡಿಂಗ್ ಮಾದರಿ ಉತ್ತಮ ದಾಖಲೆ ನಿರ್ವಹಣೆಯ ವ್ಯವಸ್ಥೆ.

<img src="../../../translated_images/kn/embedding-model-concept.90760790c336a705.webp" alt="ಎಂಬೆಡ್ಡಿಂಗ್ ಮಾದರಿ ಕಲ್ಪನೆ" width="800"/>

*ಈ ಚಿತ್ರವು ಎಂಬೆಡ್ಡಿಂಗ್ ಮಾದರಿ ಪಠ್ಯವನ್ನು ಸಂಖ್ಯಾ ವೆಕ್ಟರ್‌ಗಳಾಗಿ ಪರಿವರ್ತಿಸುವ ವಿಧಾನವನ್ನು ತೋರಿಸುತ್ತದೆ, "ಕಾರ್" ಮತ್ತು "ಆಟೋಮೊಬೈಲ್" ಸಾದೃಶ್ಯದ ಅರ್ಥಗಳನ್ನು ಹತ್ತಿರವೊಂದು ಹುದ್ದೆಯಲ್ಲಿಡುತ್ತದೆ.*

```java
@Bean
public EmbeddingModel embeddingModel() {
    return OpenAiOfficialEmbeddingModel.builder()
        .baseUrl(azureOpenAiEndpoint)
        .apiKey(azureOpenAiKey)
        .modelName(azureEmbeddingDeploymentName)
        .build();
}

EmbeddingStore<TextSegment> embeddingStore = 
    new InMemoryEmbeddingStore<>();
```
  
ಕೆಳಗಿನ ವರ್ಗ диаграм್ RAG ಪೈಪ್‌ಲೈನಿನ ಎರಡು ವಿಭಿನ್ನ ಹರಿವುಗಳನ್ನೂ ಹಾಗೂ LangChain4j ವರ್ಗಗಳನ್ನು ತೋರಿಸುತ್ತದೆ. **ಇನ್‌ಜೆಸ್ಟನ್ ಹರಿವು** (ಏಕ ಬಾರಿಗೆ ಅಪ್‌ಲೋಡ್ ಸಮಯದಲ್ಲಿ) ನ್ನು ಡಾಕ್ಯುಮೆಂಟ್ ಅನ್ನು ವಿಭಜಿಸಿ, ಚಂಕ್‌ಗಳನ್ನು ಎಂಬೆಡ್ ಮಾಡಿ, `.addAll()` ಮೂಲಕ ಸಂಗ್ರಹಿಸುತ್ತದೆ. **ಪ್ರಶ್ನೆ ಹರಿವು** (ಪ್ರತಿ ಸಲ ಬಳಕೆದಾರ ಪ್ರಶ್ನೆ ಕೇಳುವಾಗ) ಪ್ರಶ್ನೆಯನ್ನು ಎಂಬೆಡ್ ಮಾಡಿ, `.search()` ಮೂಲಕ ಅಂಗಡಿಯನ್ನು ಹುಡುಕಿ, ಹೊಂದಿದ ಸಂದರ್ಭವನ್ನು ಚಾಟ್ ಮಾದರಿಗೆ ಪಾಸ್ ಮಾಡುತ್ತದೆ. ಎರಡೂ ಹರಿವುಗಳು ಹಂಚಿಕೊಂಡ `EmbeddingStore<TextSegment>` ಇಂಟರ್ಫೇಸ್‌ನಲ್ಲಿ ಸೇರುತ್ತವೆ:

<img src="../../../translated_images/kn/rag-langchain4j-classes.bbf3aa9077ab443d.webp" alt="LangChain4j RAG ವರ್ಗಗಳು" width="800"/>

*ಈ ಚಿತ್ರವು RAG ಪೈಪ್‌ಲೈನಿನ ಎರಡು ಹರಿವುಗಳನ್ನು ತೋರಿಸುತ್ತದೆ — ಇನ್‌ಜೆಸ್ಟನ್ ಮತ್ತು ಪ್ರಶ್ನೆ — ಮತ್ತು ಅವು shared EmbeddingStore ಮೂಲಕ ಹೇಗೆ ಸಂಪರ್ಕಗೊಳ್ಳುತ್ತವೆ.*

ಎಂಬೆಡ್ಡಿಂಗ್ಸ್ ಸಂಗ್ರಹಿಸಿದ ಮೇಲೆ, ಸಾದೃಶ್ಯವಿರುವ ವಿಷಯಗಳು ಸ್ವಯಂಚಾಲಿತವಾಗಿ ಪಕ್ಕದಲ್ಲಿ ಸಮೂಹಗೊಳ್ಳುತ್ತವೆ. ಕೆಳಗಿನ ದೃಶ್ಯವು ಸಂಬಂಧಿತ ವಿಷಯಗಳ ದಾಖಲೆಗಳು ಹತ್ತಿರದ ಅಂಕಿಗಳಾಗಿ 3D ವೆಕ್ಟರ್ ಸ್ಥಳದಲ್ಲಿ ಹೇಗೆ ಗುಚ್ಛವಾಗುತ್ತವೆ ಎಂಬುದನ್ನು ತೋರಿಸುತ್ತದೆ:

<img src="../../../translated_images/kn/vector-embeddings.2ef7bdddac79a327.webp" alt="ವೆಕ್ಟರ್ ಎಂಬೆಡ್ಡಿಂಗ್ಸ್ ಸ್ಥಳ" width="800"/>

*ಈ ದೃಶ್ಯವು 3D ವೆಕ್ಟರ್ ಸ್ಥಳದಲ್ಲಿ ಸಂಬಂಧಿತ ವಿಷಯಗಳ ಗುಚ್ಛವನ್ನು ತೋರಿಸುತ್ತದೆ, ಉದ್ದೇಶಗಳು ಟೇಕ್ನಿಕಲ್ ಡಾಕ್ಸ್, ವ್ಯವಹಾರ ನಿಯಮಗಳು ಮತ್ತು FAQಗಳು ಸ್ಪಷ್ಟ ಗುಂಪುಗಳನ್ನು ರೂಪಿಸುತ್ತವೆ.*

ಬಳಕೆದಾರರು ಶೋಧಿಸಿದಾಗ, ವ್ಯವಸ್ಥೆ ನಾಲ್ಕು ಹಂತಗಳನ್ನು ಅನುಸರಿಸುತ್ತದೆ: ದಾಖಲೆಗಳನ್ನು ಏಕವಾಗಿ ಎಂಬೆಡ್ ಮಾಡುವುದು, ಪ್ರತಿ ಹುಡುಕಾಟಕ್ಕೆ ಪ್ರಶ್ನೆEmbed ಮಾಡುವುದು, ಪ್ರಶ್ನಾ ವೆಕ್ಟರ್ ಅನ್ನು ಎಲ್ಲ ಸಂರಕ್ಷಿತ ವೆಕ್ಟರ್‌ಗಳೊಂದಿಗೆ ಕೋಸೈನ್ ಸಮಾನತೆ ಮೂಲಕ ಹೋಲಿಸಿ, ಮತ್ತು ಗರಿಷ್ಠ K ಅಂಕಿದಾರಿತ ಚಂಕ್‌ಗಳನ್ನು ಪೂರೈಸುವುದು. ಕೆಳಗಿನ ಚಿತ್ರವು ಪ್ರತಿ ಹಂತವನ್ನು ಹಾಗೂ LangChain4j ವರ್ಗಗಳನ್ನು ತೋರಿಸುತ್ತದೆ:

<img src="../../../translated_images/kn/embedding-search-steps.f54c907b3c5b4332.webp" alt="ಎಂಬೆಡ್ಡಿಂಗ್ ಶೋಧ ಹಂತಗಳು" width="800"/>

*ಈ ಚಿತ್ರವು ನಾಲ್ಕು ಹಂತದ ಎಂಬೆಡ್ಡಿಂಗ್ ಶೋಧ ಪ್ರಕ್ರಿಯೆಯನ್ನು ತೋರಿಸುತ್ತದೆ: ದಾಖಲೆಗಳನ್ನು ಎಂಬೆಡ್ ಮಾಡುವುದು, ಪ್ರಶ್ನೆಯನ್ನು ಎಂಬೆಡ್ ಮಾಡುವುದು, ವೆಕ್ಟರ್‌ಗಳನ್ನು ಕೋಸೈನ್ ಸಮಾನತೆ ಮೂಲಕ ಹೋಲಿಸು, ಮತ್ತು ಗರಿಷ್ಠ K ಫಲಿತಾಂಶಗಳನ್ನು ನೀಡುವುದು.*

### ಅರ್ಥಪೂರ್ಣ ಶೋಧನೆ

[RagService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/RagService.java)

ನೀವು ಪ್ರಶ್ನೆ ಕೇಳಿದಾಗ, ನಿಮ್ಮ ಪ್ರಶ್ನೆಯೂ ಒಂದೇ embeddings ಆಗಿ ಪರಿವರ್ತಿಸುತ್ತದೆ. ವ್ಯವಸ್ಥೆ ನಿಮ್ಮ ಪ್ರಶ್ನೆಯ ಎಂಬೆಡ್ಡಿಂಗ್ ಅನ್ನು ಎಲ್ಲ ದಾಖಲೆ ಚಂಕ್ ಎಂಬೆಡ್ಡಿಂಗ್‌ಗಳೊಂದಿಗೆ ಹೋಲಿಸುತ್ತದೆ. ಅದು ಸಾದೃಶ್ಯವಾದ ಅರ್ಥಗಳನ್ನು ಹೊಂದಿರುವ ಚಂಕ್‌ಗಳನ್ನು ಕಂಡುಕೊಳ್ಳುತ್ತದೆ — ಹೀಗೇ ಕೀವರ್ಡ್ ಗಳು ಸರಿಯಾಗಿ ಹೊಂದಿಕೆಯಾಗದಿದ್ದರೂ ಸಹ, ಅರ್ಥಪೂರ್ಣವಾಗಿ ಹೊಂದಿದ ವಿಷಯಗಳನ್ನು ಕಂಡುಹೊರತೆಗೊಳ್ಳುತ್ತದೆ.

```java
Embedding queryEmbedding = embeddingModel.embed(question).content();

EmbeddingSearchRequest searchRequest = EmbeddingSearchRequest.builder()
    .queryEmbedding(queryEmbedding)
    .maxResults(5)
    .minScore(0.5)
    .build();

EmbeddingSearchResult<TextSegment> searchResult = embeddingStore.search(searchRequest);
List<EmbeddingMatch<TextSegment>> matches = searchResult.matches();

for (EmbeddingMatch<TextSegment> match : matches) {
    String relevantText = match.embedded().text();
    double score = match.score();
}
```
  
ಕೆಳಗಿನ ಚಿತ್ರವು ಪರಂಪರাগত ಕೀವರ್ಡ್ ಶೋಧನೆ ಮತ್ತು ಅರ್ಥಪೂರ್ಣ ಶೋಧನೆ ನಡುವಿನ ವ್ಯತ್ಯಾಸವನ್ನು ತೋರಿಸುತ್ತದೆ. "ವಾಹನ" ಎಂಬ ಕೀವರ್ಡ್ ಶೋಧನೆ "ಕಾರ್ ಮತ್ತು ಟ್ರಕ್‌ಗಳ" ಕುರಿತು ಇರುವ ಚಂಕ್ ಅನ್ನು ತಪ್ಪಿಸುತ್ತದೆ, ಆದರೆ ಅರ್ಥಪೂರ್ಣ ಶೋಧನೆ ಅವು ಸಮಾನ ಅರ್ಥ ಹೊಂದಿದುದರಿಂದ ಅದನ್ನು ಮೇಲಾಗುತ್ತಂತೆಯೇ ಪಕ್ಕಕ್ಕೆ ತರುವುದನ್ನು ತೋರಿಸುತ್ತದೆ:

<img src="../../../translated_images/kn/semantic-search.6b790f21c86b849d.webp" alt="ಅರ್ಥಪೂರ್ಣ ಶೋಧನೆ" width="800"/>

*ಈ ಚಿತ್ರವು ಕೀವರ್ಡ್ ಆಧಾರಿತ ಮತ್ತು ಅರ್ಥಪೂರ್ಣ ಶೋಧನೆಯನ್ನು ಹೋಲಿಸುತ್ತದೆ, ಸ್ಪಷ್ಟ ಕೀವರ್ಡ್ ಭಿನ್ನತೆಗಳಿದ್ದರೂ ಸಹ ಸಂಧರ್ಭ ಸಂಬಂಧಿತ ವಿಷಯಗಳನ್ನು ಅರ್ಥಪೂರ್ಣ ಶೋಧನೆ ಹೇಗೆ ಹಿಂತಿರುಗಿಸುತ್ತದೆ ಎಂಬುದನ್ನು ತೋರಿಸುತ್ತದೆ.*

ಮೂಲತಃ, ಸಮಾನತೆ ಪತ್ತೆ ಮಾಡುವುದು ಕೋಸೈನ್ ಸಮಾನತೆ ಮೂಲಕ — ಅಂದರೆ "ಇವು ಎರಡೂ ಕೋಣೆಯಲ್ಲಿ ಏಕರೂಪವಾಗಿ ಇಲ್ಲವೇ?" ಎಂದು ಕೇಳುವಂತೆ. ಎರಡು ಚಂಕ್‌ಗಳು ಸಂಪೂರ್ಣ ವಿವಿಧ ಪದಗಳನ್ನು ಬಳಸಬಹುದು, ಆದರೆ ಅವು ಒಂದೇ ರೀತಿ ಅರ್ಥವಿದ್ದರೆ ಅವುಗಳ ವೆಕ್ಟರ್‌ಗಳು ಒಂದೇ ದಿಕ್ಕಿಗೆ ಸಾಗುತ್ತವೆ ಮತ್ತು 1.0 ಗೆ ಸಮೀಪದ ಅಂಕಿಯನ್ನು ಪಡೆಯುತ್ತವೆ:

<img src="../../../translated_images/kn/cosine-similarity.9baeaf3fc3336abb.webp" alt="ಕೋಸೈನ್ ಸಮಾನತೆ" width="800"/>

*ಈ ಚಿತ್ರವು ಎಂಬೆಡ್ಡಿಂಗ್ ವೆಕ್ಟರ್‌ಗಳ ನಡುವೆ ಕೋನ ರೂಪದಲ್ಲಿ ಕೋಸೈನ್ ಸಮಾನತೆಯ ಅರ್ಥವನ್ನು ತೋರಿಸುತ್ತದೆ — ಹೆಚ್ಚು ಹೊಂದಿರುವ ವೆಕ್ಟರ್‌ಗಳು 1.0 ಗೆ ಸಮೀಪ ಅಂಕಿಯನ್ನು ಪಡೆದುಲ್ಲವು ಹೆಚ್ಚಿನ ಅರ್ಥಪೂರ್ಣ ಸಮಾನತೆಯನ್ನು ಸೂಚಿಸುತ್ತದೆ.*
> **🤖 [GitHub Copilot](https://github.com/features/copilot) ಚಾಟ್‌ನೊಂದಿಗೆ ಪ್ರಯತ್ನಿಸಿ:** [`RagService.java`](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/RagService.java) ತೆರೆಯಿರಿ ಮತ್ತು ಕೇಳಿ:
> - "ಎಂಬೆಡ್ಡಿಂಗ್‌ಗಳೊಂದಿಗೆ ಸೌಂದರ್ಯ ಹುಡುಕಾಟ ಹೇಗೆ ಕೆಲಸ ಮಾಡುತ್ತದೆ ಮತ್ತು ಅಂಕವನ್ನು ಏನು ನಿರ್ಧರಿಸುತ್ತದೆ?"
> - "ನಾನು ಯಾವ ಸೌಂದರ್ಯ ಥ್ರೆಷೋಲ್ಡ್ ಅನ್ನು ಬಳಸಬೇಕು ಮತ್ತು ಅದು ಫಲಿತಾಂಶಗಳಿಗೆ ಹೇಗೆ ಪ್ರಭಾವ ಬೀರುತ್ತದೆ?"
> - "ಸಂಬಂಧಿತ ಡಾಕ್ಯುಮೆಂಟ್‌ಗಳು ಕಂಡುಬರುವುದಿಲ್ಲದ ಸಂದರ್ಭದಲ್ಲಿ ನಾನು ಹೇಗೆ ಪರಿಹರಿಸಬೇಕು?"

### ಉತ್ತರ ರಚನೆ

[RagService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/RagService.java)

ಅತ್ಯಂತ ಸಂಬಂಧಿತ ಟುಕಡಿಗಳನ್ನು ಸ್ಪಷ್ಟ ಸೂಚನೆಗಳು, ಪಡೆದುಕೊಂಡ ಸಂಗತಿಗಳು ಮತ್ತು ಬಳಕೆದಾರರ ಪ್ರಶ್ನೆಯನ್ನು ಒಳಗೊಂಡ ಶ್ರೇಣೀಕೃತ ಪ್ರಾಂಪ್ಟ್‌ಗೆ ಸಂಯೋಜಿಸಲಾಗುತ್ತದೆ. ಮಾದರಿ ಆ ಟುಕಡಿಗಳನ್ನು ಓದಿ ಆ ಮಾಹಿತಿಯ ಆಧಾರದಲ್ಲಿ ಉತ್ತರ ನೀಡುತ್ತದೆ — ಇದು ತನ್ನ ಮುಂದಿನ ವಿಷಯಗಳನ್ನು ಮಾತ್ರ ಬಳಸಬಹುದು, ಇದು ಹುಲಸುವಿಕೆಯನ್ನು ತಡೆಯುತ್ತದೆ.

```java
String context = matches.stream()
    .map(match -> match.embedded().text())
    .collect(Collectors.joining("\n\n"));

String prompt = String.format("""
    Answer the question based on the following context.
    If the answer cannot be found in the context, say so.

    Context:
    %s

    Question: %s

    Answer:""", context, request.question());

String answer = chatModel.chat(prompt);
```

ಕೆಳಗಿನ ಚಿತ್ರದಲ್ಲಿ ಈ ಸಂಯೋಜನೆಯ ಕ್ರಿಯೆಯನ್ನು ತೋರಿಸಲಾಗಿದೆ — ಹುಡುಕಾಟ ಹಂತದಿಂದ ಶ್ರೇಷ್ಠ ಅಂಕಗಳನ್ನು ಪ್ರಾಂಪ್ಟ್ ಟೆಂಪ್ಲೇಟ್‌ನಲ್ಲಿ ಸೇರಿಸಲಾಗುತ್ತದೆ ಮತ್ತು `OpenAiOfficialChatModel` ನೆಲಸಾಯಿತ ಉತ್ತರವನ್ನು ರಚಿಸುತ್ತದೆ:

<img src="../../../translated_images/kn/context-assembly.7e6dd60c31f95978.webp" alt="Context Assembly" width="800"/>

*ಈ ಚಿತ್ರವು ಅತ್ಯಂತ ಅಂಕ ಪಡೆದ ಟುಕಡಿಗಳನ್ನು ಶ್ರೇಣೀಕೃತ ಪ್ರಾಂಪ್ಟ್‌ಗೆ ಹೇಗೆ ಸಂಯೋಜಿಸಲಾಗುತ್ತದೆ ಮತ್ತು ನಿಮ್ಮ ಡೇಟಾದಿಂದ ಮಾದರಿ ನೆಲಸಾಯಿತ ಉತ್ತರವನ್ನು ರಚಿಸಲು ಅವಕಾಶ ನೀಡುತ್ತದೆ ಎಂದು ತೋರಿಸುತ್ತದೆ.*

ಡೆವ್ ಕಂಟೈನರ್‌ನಲ್ಲಿ Spring Boot ಡ್ಯಾಶ್‌ಬೋರ್ಡ್ ಎಕ್ಸ್ಟೆನ್ಶನ್ ಸೇರಿದೆ, ಇದು ಎಲ್ಲಾ Spring Boot ಅಪ್ಲಿಕೇಶನ್‌ಗಳ ನಿರ್ವಹಣೆಗೆ ದೃಶ್ಯಾತ್ಮಕ ಮುಖವಾಡ ಒದಗಿಸುತ್ತದೆ. ನೀವು ಇದನ್ನು VS Code ಬಲಭಾಗದ Activity Bar ನಲ್ಲಿ ಸಿಗಬಹುದು (Spring Boot ಐಕಾನ್ ನೋಡಿ).

Spring Boot ಡ್ಯಾಶ್‌ಬೋರ್ಡ್ ನಿಂದ ನೀವು:
- ವರ್ಕ್‌ಸ್ಪೇಸ್‌ನಲ್ಲಿ ಲಭ್ಯವಿರುವ ಎಲ್ಲಾ Spring Boot ಅಪ್ಲಿಕೇಶನ್‌ಗಳನ್ನು ಕಾಣಬಹುದು
- ಒಂದು ಕ್ಲಿಕ್‌ನಲ್ಲಿ ಅಪ್ಲಿಕೇಶನ್‌ಗಳನ್ನು ಪ್ರಾರಂಭ/ನಿಲ್ಲಿಸಬಹುದು
- ಅಪ್ಲಿಕೇಶನ್ ಲಾಗ್‌ಗಳನ್ನು实时ವಾಗಿ ನೋಡಬಹುದು
- ಅಪ್ಲಿಕೇಶನ್ ಸ್ಥಿತಿಯನ್ನು ಮೇಲ್ವಿಚಾರಣೆ ಮಾಡಬಹುದು

"rag" ಬದಿಯಲ್ಲಿ ಇರುವ ಪ್ಲೇ ಬಟನ್ ಮೇಲೆ ಕ್ಲಿಕ್ ಮಾಡಿ ಈ ಮಾಡ್ಯೂಲ್ ಪ್ರಾರಂಭಿಸಬಹುದು, ಅಥವಾ ಎಲ್ಲ ಮಾಡ್ಯೂಲ್‌ಗಳನ್ನು ಒಟ್ಟಿಗೆ ಪ್ರಾರಂಭಿಸಬಹುದು.

*ಈ ಸ್ಕ್ರೀನ್‌ಶಾಟ್‌ನಲ್ಲಿ VS Code ನಲ್ಲಿ Spring Boot ಡ್ಯಾಶ್‌ಬೋರ್ಡ್ ತೋರಿಸಲಾಗಿದೆ, ಇಲ್ಲಿ ನೀವು ಅಪ್ಲಿಕೇಶನ್‌ಗಳನ್ನು ಪ್ರಾರಂಭ, ನಿಲ್ಲಿಸಿ ಮತ್ತು ದೃಶ್ಯಾತ್ಮಕವಾಗಿ ಮೇಲ್ವಿಚಾರಣೆ ಮಾಡಬಹುದು.*

## ಅಪ್ಲಿಕೇಶನ್ ಬಳಕೆ

ಅಪ್ಲಿಕೇಶನ್ ಡಾಕ್ಯುಮೆಂಟ್ ಅಪ್ಲೋಡ್ ಮತ್ತು ಪ್ರಶ್ನೆಗಳಿಗೆ ವೆಬ್ ಇಂಟರ್‌ಫೇಸ್ ಒದಗಿಸುತ್ತದೆ.

*ಈ ಸ್ಕ್ರೀನ್‌ಶಾಟ್ RAG ಅಪ್ಲಿಕೇಶನ್ ಇಂಟರ್‌ಫೇಸ್ ಅನ್ನು ತೋರಿಸುತ್ತದೆ, ಇಲ್ಲಿ ನೀವು ಡಾಕ್ಯುಮೆಂಟ್ ಅಪ್ಲೋಡ್ ಮಾಡಿ ಪ್ರಶ್ನೆ ಕೇಳುತ್ತೀರಾ.*

### ಡಾಕ್ಯುಮೆಂಟ್ ಅಪ್ಲೋಡ್ ಮಾಡಿ

ಮೊದಲು ಡಾಕ್ಯುಮೆಂಟ್ ಅನ್ನು ಅಪ್ಲೋಡ್ ಮಾಡಿ - ಪರೀಕ್ಷೆಗಾಗಿ TXT ಫೈಲ್‌ಗಳು ಉತ್ತಮ. ಈ ಡೈರೆಕ್ಟರಿಯಲ್ಲಿ `sample-document.txt` ಇದೆ, ಇದು LangChain4j ವೈಶಿಷ್ಟ್ಯಗಳು, RAG ಜಾರಿಗೆ ಮತ್ತು ಉತ್ತಮ ಅಭ್ಯಾಸಗಳು ಬಗ್ಗೆ ಮಾಹಿತಿ ಒಳಗೊಂಡಿದೆ - ಅವುಗಳನ್ನು ಪರೀಕ್ಷಿಸಲು ಉತ್ತಮ.

ಸಿಸ್ಟಂ ನಿಮ್ಮ ಡಾಕ್ಯುಮೆಂಟ್ ಅನ್ನು ಪ್ರಕ್ರಿಯೆಗೊಳಿಸಿ, ಅದನ್ನು ಟುಕಡಿಗಳಾಗಿ ವಿಭಜಿಸಿ ಮತ್ತು ಪ್ರತಿ ಟುಕಡಿಗೆ embeddings ರಚಿಸುತ್ತದೆ. ಇದು ನಿಮ್ಮ ಅಪ್ಲೋಡ್ ಮಾಡಿದಾಗ ಸ್ವಯಂಚಾಲಿತವಾಗಿ ಸಂಭವಿಸುತ್ತದೆ.

### ಪ್ರಶ್ನೆಗಳನ್ನು ಕೇಳಿ

ಇದ್ದ ಡಾಕ್ಯುಮೆಂಟ್ ವಿಷಯಗಳ ಬಗ್ಗೆ స్పಷ್ಟವಾದ ಪ್ರಶ್ನೆಗಳನ್ನು ಕೇಳಿ. ಸಿಸ್ಟಂ ಸಂಬಂಧಿತ ಟುಕಡಿಗಳನ್ನು ಹುಡುಕಿ, ಅವುಗಳನ್ನು ಪ್ರಾಂಪ್ಟ್‌ಗೆ ಸೇರಿಸಿ, ಉತ್ತರವನ್ನು ರಚಿಸುತ್ತದೆ.

### ಮೂಲ ದಾಖಲಾತಿಗಳ ಪರಿಶೀಲನೆ

ಪ್ರತಿ ಉತ್ತರವು ಸಾಮಾನ್ಯವಾಗಿ ಸೌಂದರ್ಯ ಅಂಕಗಳೊಂದಿಗೆ ಮೂಲ ಉಲ್ಲೇಖಗಳನ್ನೂ ಒಳಗೊಂಡಿರುತ್ತದೆ. ಈ ಅಂಕಗಳು (0 ರಿಂದ 1 ವರೆಗೆ) ಪ್ರತಿಯೊಂದು ಟುukao chunk  ನಿಮ್ಮ ಪ್ರಶ್ನೆಗೆ ಎಷ್ಟು ಸಂಬಂಧಿತವಾಗಿದೆ ಎಂದು ತೋರಿಸುತ್ತದೆ. ಹೆಚ್ಚಿನ ಅಂಕಗಳು ಉತ್ತಮ ಹೊಂದಿಕೆಯನ್ನು ಸೂಚಿಸುತ್ತವೆ. ಇದರಿಂದ ನೀವು ಉತ್ತರವನ್ನು ಮೂಲವನ್ನೆಂದರೆ ಪರಿಶೀಲಿಸಬಹುದು.

<a href="images/rag-query-results.png"><img src="../../../translated_images/kn/rag-query-results.6d69fcec5397f355.webp" alt="RAG Query Results" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*ಈ ಸ್ಕ್ರೀನ್‌ಶಾಟ್‌ನಲ್ಲಿ ದೃಷ್ಟಾಂತ ಪ್ರಶ್ನೆಗಳ ಫಲಿತಾಂಶಗಳು, ಉತ್ಪನ್ನ ಉತ್ತರ, ಮೂಲ ಉಲ್ಲೇಖಗಳು ಮತ್ತು ಪ್ರತಿಯೊಂದು ಟುukao chunk ಗೆ ಸಂಬಂಧಿತ ಸೌಂದರ್ಯ ಅಂಕಗಳು ತೋರಿಸಲಾಗಿದೆ.*

### ಪ್ರಶ್ನೆಗಳಲ್ಲಿ ಪ್ರಯೋಗ ಮಾಡಿ

ವಿವಿಧ ರೀತಿಯ ಪ್ರಶ್ನೆಗಳನ್ನು ಪ್ರಯತ್ನಿಸಿ:
- ಸ್ಪಷ್ಟ ವಾಸ್ತವಗಳು: "ಮುಖ್ಯ ವಿಷಯ ಏನು?"
- ಹೋಲಿಕೆಗಳು: "X ಮತ್ತು Y ನಡುವೆ ವ್ಯತ್ಯಾಸವೇನು?"
- ಸಾರಾಂಶಗಳು: "Z ಬಗ್ಗೆ ಮುಖ್ಯ ಅಂಶಗಳನ್ನು ಸಾರಾಂಶಿಸಿ"

ನಿಮ್ಮ ಪ್ರಶ್ನೆ ಡಾಕ್ಯುಮೆಂಟ್ ವಿಷಯಕ್ಕೆ ಹೇಗೆ ಹೊಂದಿಕೆಯಾಗುತ್ತದೆಯೋ ಆಧರಿಸಿ ಸೌಂದರ್ಯ ಅಂಕಗಳು ಹೇಗೆ ಬದಲಾಗುತ್ತವೆ ಅಂತ ಗಮನಿಸಿ.

## ಪ್ರಮುಖ ಪರಿಕಲ್ಪನೆಗಳು

### ಟುukao chunk ಮಾಡುವುದು

ಡಾಕ್ಯುಮೆಂಟ್‌ಗಳು 300-ಟೋಕನ್ ಟುukao chunk ಗಳಾಗಿ ವಿಭಜಿಸಲ್ಪಡುತ್ತವೆ, 30 ಟೋಕನ್‌ಗಳ ಅವಧಿಯೊಂದಿಗೆ. ಇದರಿಂದ ಪ್ರತಿಯೊಂದು chunk ಗೆ ಸಾರ್ಥಕವಾದ ಹಿನ್ನಲೆ ಇರುತ್ತದೆ ಮತ್ತು ಭಾರಿ ಆಗದೆ ಪ್ರಯೋಗದಲ್ಲೂ ಕಾಳಜಿ ಇರುತ್ತದೆ.

### ಸೌಂದರ್ಯ ಅಂಕಗಳು

ಪ್ರತಿ ಪಡೆದ ಟುukao chunk ಗೆ ಬಳಕೆದಾರರ ಪ್ರಶ್ನೆಗೆ ಹೊಂದಾಣಿಕೆಯಾಗುವ ಸೌಂದರ್ಯ ಅಂಕ 0 ಮತ್ತು 1 ನಡುವಣೆಯೇ ಇರುತ್ತದೆ. ಕೆಳಗಿನ ಚಿತ್ರದಲ್ಲಿ ಅಂಕಗಳ ವ್ಯಾಪ್ತಿಯನ್ನು ಮತ್ತು ಸಿಸ್ಟಂವು ಅವುಗಳನ್ನು ಫಲಿತಾಂಶಗಳನ್ನು ಶೋಧಿಸಲು ಹೇಗೆ ಬಳಸುತ್ತದೆಯೋ ತೋರಿಸಲಾಗಿದೆ:

<img src="../../../translated_images/kn/similarity-scores.b0716aa911abf7f0.webp" alt="Similarity Scores" width="800"/>

*ಈ ಚಿತ್ರವು 0 ರಿಂದ 1 ರವರೆಗೆ ಅಂಕಗಳ ವ್ಯಾಪ್ತಿಯನ್ನು ಮತ್ತು ಕನಿಷ್ಟ 0.5 ಥ್ರೆಷೋಲ್ಡ್ ಅನ್ನು ತೋರಿಸುತ್ತದೆ, ಇದು ಅಸಂಬಂಧಿತ chunk ಗಳನ್ನು ತೆಗೆಯುತ್ತದೆ.*

ಅಂಕಗಳ ವ್ಯಾಪ್ತಿಗಳು:
- 0.7-1.0: ಅತ್ಯಂತ ಸಂಬಂಧಿತ, ನಿಖರ ಹೊಂದಿಕೆ
- 0.5-0.7: ಸಂಬಂಧಿತ, ಉತ್ತಮ ಹಿನ್ನಲೆ
- 0.5 ಕೆಳಗೆ: ತೆಗೆಯಲ್ಪಟ್ಟಿವೆ, ಅಸಂಬಂಧಿತ

ಸಿಸ್ಟಂನ್ನುವ ಕನಿಷ್ಟ ಥ್ರೆಷೋಲ್ಡ್ ಮೇಲನಿದ್ದಾಗ ಮಾತ್ರ chunk ಗಳು ಹುಡುಕಲ್ಪಡುತ್ತವೆ ಮತ್ತು ಗುಣಮಟ್ಟವನ್ನು ಖಚಿತಪಡಿಸುತ್ತದೆ.

ಎಂಬೆಡ್ಡಿಂಗ್‌ಗಳು ಅರ್ಥ ಹಂಚಿಕೆಯಾಗುವಾಗ ಉತ್ತಮವಾಗಿ ಕೆಲಸ ಮಾಡುತ್ತವೆ, ಆದರೆ ಇವುಗಳಲ್ಲಿ ತಪ್ಪುಗಳಿದ್ದಾಗ ಜಾಗಗಳನ್ನು ಹೊಂದುತ್ತವೆ. ಕೆಳಗಿನ ಚಿತ್ರವು ಸಾಮಾನ್ಯ ವಿಫಲತೆಗಳನ್ನು ತೋರಿಸುತ್ತದೆ — ತುಂಬಾ ದೊಡ್ಡ chunk ಗಳು ಮೃದುವಾದ ವಿಕೆಟ್‌ಗಳನ್ನು ಉತ್ಪತ್ತಿ ಮಾಡುತ್ತವೆ, ತುಂಬಾ ಸಣ್ಣ chunk ಗಳಿಗೆ ಹಿನ್ನಲೆ ಕೊರते ಇದೆ, ಅಸ್ಪಷ್ಟ ಪದಗಳು ಅನೇಕ ಕ್ಲಸ್ಟರ್ ಗಳಿಗೆ ಸೂಚಿಸುತ್ತವೆ, ಮತ್ತು ನಿಖರ ಹೊಂದಿಕೆಗೆ lookup (IDs, ಭಾಗ ಸಂಖ್ಯೆ) ಗಳಿಗೆ embeddings ಕೆಲಸ ಮಾಡುವುದಿಲ್ಲ:

<img src="../../../translated_images/kn/embedding-failure-modes.b2bcb901d8970fc0.webp" alt="Embedding Failure Modes" width="800"/>

*ಈ ಚಿತ್ರವು ಸಾಮಾನ್ಯ embedding ವಿಫಲತೆಗಳು: ತುಂಬಾ ದೊಡ್ಡ ಅಥವಾ ಸಣ್ಣ chunk ಗಳು, ಅನೇಕ ಕ್ಲಸ್ಟರ್ ಗಳಿಗೆ ಸೂಚಿಸುವ ಅಸ್ಪಷ್ಟ ಪದಗಳು, ಮತ್ತು ನಿಖರ ಹೊಂದಿಕೆ lookup ಗಳುವನ್ನು ತೋರಿಸುತ್ತದೆ.*

### ಮೆಮೊರಿ ಸಂಗ್ರಹಣೆ

ಈ ಮಾಡ್ಯೂಲ್ ಸರಳತೆಗೆ ಮೆಮೊರಿಯಲ್ಲಿ ಸಂಗ್ರಹಣೆಯನ್ನೇ ಬಳಸುತ್ತದೆ. ಅಪ್ಲಿಕೇಶನ್ ನವೀಕರಿಸಿದಾಗ ಅಪ್ಲೋಡ್ ಮಾಡಿದ ಡಾಕ್ಯುಮೆಂಟ್‌ಗಳು ಕಳೆದುಹೋತವೆ. ಉತ್ಪಾದನಾ ವ್ಯವಸ್ಥೆಗಳು Qdrant ಅಥವಾ Azure AI Searchಂತಹ ಸ್ಥಾಯಿ ವೆಕ್ಟರ್ ಡೇಟಾಬೇಸ್‌ಗಳನ್ನು ಬಳಸುತ್ತವೆ.

### ಸನ್ನಿವೇಶ ಕಿಟಕಿಯ ನಿರ್ವಹಣೆ

ಪ್ರತಿ ಮಾದರಿಗಿನ್ನು ಗರಿಷ್ಠ ಸನ್ನಿವೇಶ ಕಿಟಕಿ ಇದ್ದು, ನೀವು ದೊಡ್ಡ ಡಾಕ್ಯುಮೆಂಟ್‌ನ ಪ್ರತಿಯೊಂದು chunk ಅನ್ನು ಸೇರಿಸಲು ಸಾಧ್ಯವಿಲ್ಲ. ಸಿಸ್ಟಂ ಗರಿಷ್ಠ ಸೀಮೆಯೊಳಗೆ ಉಳಿಯಲು ಅತ್ಯಂತ ಸಂಬಂಧಿತ ಟುukao chunk ಗಳನ್ನು (ಡಿ'ಫಾಲ್ಟ್ 5) ಪಡೆಯುತ್ತದೆ ಮತ್ತು ಸರಿಯಾದ ಉತ್ತರಗಳಿಗೆ ಸಾಕಷ್ಟು ಹಿನ್ನಲೆ ಒದಗಿಸುತ್ತದೆ.

## RAG ಯಾಕಾಗ ಮುಖ್ಯ

RAG ಯಾವಾಗಲೂ ಸೂಕ್ತ ವಿಧಾನವಲ್ಲ. ಕೆಳಗಿನ ನಿರ್ಧಾರ ಮಾರ್ಗದರ್ಶಕವು ನೀವು RAG ಉಪಯೋಗಿಸುವುದು ಯಾವಾಗ ಸೂಕ್ತ ಹಾಗೂ ಸರಳ ವಿಧಾನಗಳು ಯಾವಾಗ ಸಾಕಾಗುತ್ತವೆಯೋ ತಿಳಿಯಲು ಸಹಾಯ ಮಾಡುತ್ತದೆ — ಉದಾಹರಣೆಗೆ, ಪ್ರಾಂಪ್ಟ್‌ನಲ್ಲಿ ವಿಷಯವನ್ನು ನೇರ ಸೇರಿಸುವುದು ಅಥವಾ ಮಾದರಿಯ ಸ್ವಂತ ತಿಳುವಳಿಕೆಯನ್ನು ಅವಲಂಬಿಸುವುದು.

<img src="../../../translated_images/kn/when-to-use-rag.1016223f6fea26bc.webp" alt="When to Use RAG" width="800"/>

*ಈ ಚಿತ್ರವು RAG ಬಳಸುವ ಸಮಯ ಮತ್ತು ಸರಳ ವಿಧಾನಗಳು ಸಾಕಾಗುವ ಸಮಯದ ನಿರ್ಧಾರ ಮಾರ್ಗದರ್ಶಕವನ್ನು ತೋರಿಸುತ್ತದೆ.*

**RAG ಬಳಸಿರಿ:**
- ಖಾಸಗಿ ಡಾಕ್ಯುಮೆಂಟ್‌ಗಳ ಕುರಿತು ಪ್ರಶ್ನೆಗಳಿಗೆ ಉತ್ತರಿಸುವಾಗ
- ಮಾಹಿತಿಗಳು ತೀವ್ರವಾಗಿ ಬದಲಾಗುತ್ತಾ ಇದ್ದಾಗ (ನೀತಿ, ಬೆಲೆ, ವಿವರಗಳು)
- ಸರಿಯಾದ ಅತಿಥಿ ಸೂಚನೆ ಅಗತ್ಯವಿದ್ದಾಗ
- ವಿಷಯ ಅತ್ಯಂತ ದೊಡ್ಡದಾಗಿದ್ದು ಒಂದು ಪ್ರಾಂಪ್ಟ್‌ಗೆ ಒಟ್ಟು ಬೇಡಿನಾಗಿದ್ದಾಗ
- ಖಚಿತಪಡಿಸಬಹುದಾದ, ನೆಲಸಾಯಿತ ಉತ್ತರ ಬೇಕಾದಾಗ

**RAG ಉಪಯೋಗಿಸಬೇಡಿ:**
- ಪ್ರಶ್ನೆಗಳು ಸಾಮಾನ್ಯ ತಿಳುವಳಿಕೆಗೆ ಸಂಬಂಧಿಸಿದಾಗ, ಇದು ಮಾದರಿಯಲ್ಲಿದೆ
- ನೈಜ ಸಮಯದ ಡೇಟಾವು ಬೇಕಾದಾಗ (RAG ಅಪ್ಲೋಡ್ ಮಾಡಿದ ಡಾಕ್ಯುಮೆಂಟ್‌ಗಳ ಮೇಲೆ ಕಾರ್ಯನಿರ್ವಹಿಸುತ್ತದೆ)
- ವಿಷಯವು ಸ್ವಲ್ಪ ಏನಾದರೂ ಆಗೇ ಪ್ರಾಂಪ್ಟ್ ಗಳಲ್ಲಿ ನೇರ ಸೇರಿಸಬಹುದಾದಾಗ
