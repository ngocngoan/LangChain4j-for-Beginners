## What You'll Learn

മുൻ മോഡ്യൂളുകളിൽ, നിങ്ങൾ AI-യുമായി സംവദിക്കാൻ എങ്ങനെ പഠിച്ചുവെന്നും നിങ്ങളുടെ പ്രോംപ്റ്റുകൾ എഫക്ടീവായി രൂപപ്പെടുത്തിയെടുക്കാനുള്ള രീതി പഠിച്ചുതന്നിരുന്നു. പക്ഷേ ഒരു അടിസ്ഥാനപരമായ പരിധിയാണ്: ഭാഷ മോഡലുകൾ പരിശീലന സമയത്ത് മാത്രം പഠിച്ചതാണ് അറിയുന്നത്. നിങ്ങളുടെ കമ്പനി നയങ്ങൾ, പ്രോജക്ട് ഡോക്യുമെന്റേഷൻ, അല്ലെങ്കിൽ അവ പരിശീലിപ്പിക്കപ്പെട്ടിട്ടില്ലാത്ത വിവരങ്ങളെക്കുറിച്ച് ചോദ്യങ്ങൾക്ക് അവർ ഉത്തരം നൽകാൻ കഴിയില്ല.

RAG (റെട്രീവൽ-ഓഗ്മെന്റഡ് ജനറേഷൻ) ഈ പ്രശ്നം പരിഹരിക്കുന്നു. മോഡലിന് നിങ്ങളുടെ വിവരം പഠിപ്പിക്കാൻ ശ്രമിക്കുന്നതിന് പകരം (അത് ചെലവേറിയതും പ്രായോഗികവുമല്ല), നിങ്ങൾ അതിനെ നിങ്ങളുടെ ഡോക്യുമെന്റുകൾ തിരയാൻ ശേഷിയുള്ളതായി നൽകുന്നു. ആരാണ് ചോദ്യംചെയ്യുമ്പോൾ, സിസ്റ്റം ബന്ധപ്പെട്ട വിവരങ്ങൾ കണ്ടെത്തി പ്രോംപ്റ്റിൽ ഉൾപ്പെടുത്തുന്നു. മോഡൽ പിന്നീട് ആ നേടപ്പെട്ട സന്ദർഭത്തെ ആശ്രയിച്ച് ഉത്തരം നൽകുന്നു.

RAG-നെ നിങ്ങൾ മോഡലിന് ഒരു റഫറൻസ് ലൈബ്രറി നൽകുന്നതെന്തെന്നു ചിന്തിക്കുക. നിങ്ങൾ ചോദ്യംചെയ്യുമ്പോൾ, സിസ്റ്റം:

1. **ഉപയോക്തൃ ചോദ്യം** - നിങ്ങൾ ചോദ്യം ചോദിക്കുന്നു  
2. **എംബെഡിംഗ്** - നിങ്ങളുടെ ചോദ്യം ഒരു വെക്ടറായി മാറ്റുന്നു  
3. **വെക്ടർ സെർച്ച്** - സമാനമായ ഡോക്യുമെന്റ് ചങ്കുകൾ കണ്ടെത്തുന്നു  
4. **സന്ദർഭ ശേഖരണം** - ബന്ധപ്പെട്ട ചങ്കുകൾ പ്രോംപ്റ്റിൽ ചേർക്കുന്നു  
5. **പ്രതികരണം** - LLM ആ സന്ദർഭത്തെ അടിസ്ഥാനമാക്കി ഉത്തരം സൃഷ്ടിക്കുന്നു  

ഇത് മോഡലിന്റെ ഉത്തരം നിങ്ങളുടെ യഥാർഥ ഡാറ്റയിൽ അധിഷ്ഠിതമാക്കുന്നു, അതിന്റെ പരിശീലന ജ്ഞാനത്തിലോ കൽപ്പിത ഉത്തരം സൃഷ്ടിക്കുന്നതിലോ ആശ്രയിക്കാതെ.

## Understanding RAG

താഴെയുള്ള ഡയഗ്രാം മുഖ്യ ആശയം പ്രതീകം ചെയ്യുന്നു: മോഡലിന്റെ പരിശീലന ഡാറ്റ മാത്രം ആശ്രയിക്കുന്നതിനുപകരം, RAG അതിനെ ഓരോ ഉത്തരം സൃഷ്ടിക്കുന്നതിന് മുമ്പ് നിങ്ങളുടെ ഡോക്യുമെന്റുകളുടെ ഒരു റഫറൻസ് ലൈബ്രറി നൽകുന്നു.

<img src="../../../translated_images/ml/what-is-rag.1f9005d44b07f2d8.webp" alt="What is RAG" width="800"/>

ഉപയോക്താവിന്റെ ചോദ്യം എങ്ങനെ നാല് ഘട്ടങ്ങളിലൂടെ പോകുന്നു എന്ന് കാണിക്കുന്നു — എംബെഡിംഗ്, വെക്ടർ സെർച്ച്, സന്ദർഭ ശേഖരണം, ഉത്തരം സൃഷ്ടിക്കൽ — ഓരോ ഘട്ടവും മുമ്പത്തെ ഘട്ടത്തിലൂടെ നിർമ്മിച്ചത്:

<img src="../../../translated_images/ml/rag-architecture.ccb53b71a6ce407f.webp" alt="RAG Architecture" width="800"/>

ഈ മോഡ്യൂളിന്റെ ശേഷിക്കുന്ന ഭാഗം ഓരോ ഘട്ടവും വിശദമായി വിശദീകരിക്കുന്നു, നിങ്ങൾക്ക് ഇത് പ്രവർത്തിപ്പിക്കുകയും മാറ്റിസ്ഥാപിക്കുകയും ചെയ്യാവുന്ന കോഡുമായി.

## How It Works

### Document Processing

[DocumentService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/DocumentService.java)

താങ്കൾ ഒരു ഡോക്യുമെന്റ് അപ്‌ലോഡ്ചെയ്യുമ്പോൾ, സിസ്റ്റം അതിനെ (PDF അല്ലെങ്കിൽ പ്ലൈൻ ടെക്സ്റ്റ്) പാഴ്സു ചെയ്ത്, ഫയൽനെയിം പോലെയുള്ള മെറ്റാഡേറ്റ ഉൾപ്പെടുത്തിയ ശേഷം, മോഡലിന്റെ സന്ദർഭ വിൻഡോയ്ക്ക് അനുകൂലമായ ചെറിയ ചങ്കുകളായി വിഭജിക്കുന്നു. ഈ ചങ്കുകൾക്കിടയിൽ ചെറിയ ഒരുപോലും ഓവർലാപ്പ് ഉണ്ടാകുന്നു, അതിലൂടെ ബൗണ്ടറിയിൽ നിന്ന് സന്ദർഭം നഷ്ടപ്പെടാതെ ഉറപ്പുവരുത്തുന്നു.

```java
// അപ്ലോഡ് ചെയ്യപ്പെട്ട ഫയൽ പാഴ്സുചെയ്ത് LangChain4j ഡോക്യുമെന്റിൽ പൊതിഞ്ഞു
Document document = Document.from(content, metadata);

// 300-ടോക്കൺ ഭാഗങ്ങളായി വിഭജിച്ച് 30-ടോക്കൺ ഒവർലാപ്പ് ചേർത്ത്
DocumentSplitter splitter = DocumentSplitters
    .recursive(300, 30);

List<TextSegment> segments = splitter.split(document);
```
  
താഴെയുള്ള ഡയഗ്രാം ഇതിനായുള്ള ദൃശ്യവിവരണം നൽകുന്നു. ഓരോ ചങ്കും സമീപവാസികളുമായി കുറച്ച് ടോക്കനുകൾ പങ്കിടുന്നു — 30-ടോക്കൺ ഓവർലാപ്പ് അത്യന്താപേക്ഷിത സന്ദർഭം നഷ്ടപ്പെടാതിരിക്കാനാണ് കാരണമാകുന്നത്:

<img src="../../../translated_images/ml/document-chunking.a5df1dd1383431ed.webp" alt="Document Chunking" width="800"/>

> **🤖 GitHub Copilot ചാറ്റ് ഉപയോഗിച്ച് ശ്രമിക്കുക:** [`DocumentService.java`](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/DocumentService.java) തുറക്കുക, ചോദിക്കുക:  
> - "LangChain4j ഡോക്യുമെന്റുകൾ എങ്ങനെ ചങ്കുകളാക്കി വിഭജിക്കുന്നു, ഓവർലാപ്പിന് എന്ത് പ്രാധാന്യമുണ്ട്?"  
> - "ബിവിധ ഡോക്യുമെന്റ് തരങ്ങൾക്കുള്ള ഏറ്റവും അനുയോജ്യമായ ചങ്ക് വലുപ്പം എന്ത്, എന്തുകൊണ്ട്?"  
> - "பல ഭാഷകളിലുള്ള ഡോക്യുമെന്റുകളും പ്രത്യേക ഫോർമാറ്റിങ് ഉള്ളവയും എങ്ങനെ കൈകാര്യം ചെയ്യും?"

### Creating Embeddings

[LangChainRagConfig.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/config/LangChainRagConfig.java)

ഓരോ ചങ്കും എമ്പെഡിംഗ് എന്ന വിശേഷിപ്പിക്കപ്പെട്ട സംഖ്യാനീത രീതിയിലേക്ക് മാറ്റുന്നു - അടിസ്ഥാനപരമായി ടെക്സ്റ്റിന്റെ അർത്ഥം പിടിച്ചെടുത്തു സൂക്ഷിക്കുന്ന ഗണിതശാസ്ത്ര വിരലുറപ്പ്. സമാന ടെക്സ്റ്റുകൾ സമാനമായ എമ്പെഡിങ്ങുകൾ സൃഷ്ടിക്കും.

```java
@Bean
public EmbeddingModel embeddingModel() {
    return OpenAiOfficialEmbeddingModel.builder()
        .baseUrl(azureOpenAiEndpoint)
        .apiKey(azureOpenAiKey)
        .modelName(azureEmbeddingDeploymentName)
        .build();
}

EmbeddingStore<TextSegment> embeddingStore = 
    new InMemoryEmbeddingStore<>();
```
  
താഴെയുള്ള ക്ലാസ് ഡയഗ്രാം എങ്ങനെ LangChain4j ഘടകങ്ങൾ ബന്ധപ്പെട്ടിരിക്കുന്നുവെന്നു കാണിക്കുന്നു. `OpenAiOfficialEmbeddingModel` ടെക്സ്റ്റിനെ വെക്ടറുകളായി മാറ്റുന്നു, `InMemoryEmbeddingStore` ആ വെക്ടറുകളെ അവരുടെ യഥാർത്ഥ `TextSegment` ഡാറ്റയോടൊപ്പം സൂക്ഷിക്കുന്നു, `EmbeddingSearchRequest`-ൽ `maxResults` എന്നത്, `minScore` പോലുള്ള പുന:വിശകലന പരാമീറ്ററുകള് നിയന്ത്രിക്കുന്നു:

<img src="../../../translated_images/ml/rag-langchain4j-classes.bbf3aa9077ab443d.webp" alt="LangChain4j RAG Classes" width="800"/>

എംബെഡിങ്ങുകൾ സൂക്ഷിച്ചതിന് ശേഷം, സമാന ഉള്ളടക്കം സ്വാഭാവികമായി വെക്ടർ സ്‌പേസിൽ ഒത്തുകൂടുന്നു. താഴെയുള്ള ദൃശ്യീകരണം ബന്ധപ്പെട്ട വിഷയങ്ങളുടെ ഡോക്യുമെന്റുകൾ അടുത്തിടെയുള്ള പോയിന്റുകളായി എത്തുന്നതിന്റെ ഉദാഹരണമാണ്, അവയാണ് സെമാന്റിക് സെർച്ച് സാദ്ധ്യമാക്കുന്നത്:

നിങ്ങൾ ചോദ്യമുചെയ്യുമ്പോൾ, നിങ്ങളുടെ ചോദ്യം സ്വയം ഒരു എംബെഡിംഗായി മാറുന്നു. സിസ്റ്റം നിങ്ങളുടെ ചോദ്യത്തിന്റെയും എല്ലാ ഡോക്യുമെന്റ് ചങ്കുകളുടെയും എम्बെഡിങ്ങുകൾ താരതമ്യം ചെയ്യുന്നു. ഏറ്റവും സമാനമായ അർത്ഥം ഉള്ള ചങ്കുകൾ കണ്ടെത്തുന്നു — വെറും കീ വേഡ് മാച്ചിംഗ് മാത്രമല്ല, യഥാർത്ഥ സെമാന്റിക് സമാനതയും പരിശോധിക്കുന്നു.

```java
Embedding queryEmbedding = embeddingModel.embed(question).content();

EmbeddingSearchRequest searchRequest = EmbeddingSearchRequest.builder()
    .queryEmbedding(queryEmbedding)
    .maxResults(5)
    .minScore(0.5)
    .build();

EmbeddingSearchResult<TextSegment> searchResult = embeddingStore.search(searchRequest);
List<EmbeddingMatch<TextSegment>> matches = searchResult.matches();

for (EmbeddingMatch<TextSegment> match : matches) {
    String relevantText = match.embedded().text();
    double score = match.score();
}
```
  
താഴെയുള്ള ഡയഗ്രാം സെമാന്റിക് സെർച്ചിനെ പരമ്പരാഗത കീവേഡ് സെർച്ചുമായി താരതമ്യം ചെയ്യുന്നു. "vehicle" എന്ന കീവേഡ് സെർച്ച് "cars and trucks" എന്ന അർത്ഥമുള്ള ചങ്ക് കണ്ടുപിടിക്കാനല്ലാത്ത പക്ഷം, സെമാന്റിക് സെർച്ച് അവ രണ്ടും സമാനമാണ് എന്നും തിരിച്ചറിയുകയും ഉയർന്ന സ്‌കോർ നൽകിയ മാച്ച് ആയി തിരിച്ചറിയ്ക്കുകയും ചെയ്യുന്നു:

### Answer Generation

ഏറ്റവും പ്രസക്തമായ ചങ്കുകൾ ഒരു ഘടനയുള്ള പ്രോംപ്റ്റിൽ സമാഹരിക്കും, അതിൽ വ്യക്തമായ നിര്‍ദ്ദേശങ്ങളും, പുനഃപ്രാപിച്ചത് ഉള്‍കൊള്ളും, ഉപയോക്താവിന്റെ ചോദ്യവും ഉണ്ടായിരിക്കും. മോഡൽ ആ ചങ്കുകൾ വായിച്ച് അതിന്റെ അടിസ്ഥാനത്തിൽ ഉത്തരം നൽകും — അത് മുൻവശത്ത് ഉള്ളത് മാത്രമേ ഉപയോഗിക്കാനാകൂ, ഇതുവഴി ഹല്യൂസിനേഷൻ ഒഴിവാക്കാം.

```java
String context = matches.stream()
    .map(match -> match.embedded().text())
    .collect(Collectors.joining("\n\n"));

String prompt = String.format("""
    Answer the question based on the following context.
    If the answer cannot be found in the context, say so.

    Context:
    %s

    Question: %s

    Answer:""", context, request.question());

String answer = chatModel.chat(prompt);
```
  
താഴെയുള്ള ഡയഗ്രാം ഈ പ്രക്രിയ സൂചിപ്പിക്കുന്നു — സെർച്ച് ഘട്ടത്തിലെ ടോപ്പ്-സ്കോറിംഗ് ചങ്കുകൾ പ്രോംപ്റ്റ് ടെംപ്ലേറ്റിലേക്ക് ചെരിഞ്ഞ്, `OpenAiOfficialChatModel` ഇതിനെ അടിസ്ഥാനമാക്കി ഒരു നിർബന്ധിത ഉത്തരം സൃഷ്ടിക്കുന്നു:

- താരതമ്യങ്ങൾ: "X-നും Y-നും ഉള്ള വ്യത്യാസം എന്ത്?"  
- സംഗ്രഹം: "Z-ന്‍റെ പ്രത്യയകോർട്ടുകൾ സംഗ്രഹിക്കൂ"  

നിങ്ങളുടെ ചോദ്യത്തിന്റെ പ്രാസ്താവികത ഡോക്യുമെന്റ് ഉള്ളടക്കത്തിനോടുള്ള പൊരുത്തത്തിന്റെ അടിസ്ഥാനത്തിൽ ഇങ്ങനെ സ്കോറുകൾ എങ്ങനെ മാറുന്നു എന്ന് ശ്രദ്ധിക്കുക.

## Key Concepts

### Chunking Strategy

ഡോക്യുമെന്റുകൾ 300-ടോക്കൺ ചങ്കുകളായി വിഭജിക്കപ്പെടുന്നു, അവയിൽ ഓരോ ചങ്കിനും 30 ടോക്കണുകൾ ഓവർലാപ്പ്. ഈ തുല്യത അതോറ്റ ഒറ്റ ചങ്കിൽ മതിയായ സന്ദർഭം ഉറപ്പുവരുത്തുന്നു എങ്കിലും ഒരു പ്രോംപ്റ്റിൽ പല ചങ്കുകളും ഉൾപ്പെടുത്താൻ കഴിയുന്ന വലിപ്പം സൂക്ഷിക്കുന്നു.

### Similarity Scores

ആയിരപ്പൗ മടക്കം ലഭിച്ച ഓരോ ചങ്കിനും 0 മുതൽ 1 വരെയുള്ള സമാനത സ്‌കോർ ഉണ്ടാകുന്നു, ഇത് ആ ചങ്ക് ഉപയോക്താവിന്റെ ചോദ്യം എത്രത്തോളം കൃത്യമായി പൊരുത്തപ്പെടുന്നു എന്നത് സൂചിപ്പിക്കുന്നു. താഴെ ഡയഗ്രാം സ്കോർ പരിധി കാണിക്കുന്നു, സിസ്റ്റത്തിൽ ഫില്ടറിംഗിന് ഇത് എങ്ങനെ സഹായിക്കുന്നു:

<img src="../../../translated_images/ml/similarity-scores.b0716aa911abf7f0.webp" alt="Similarity Scores" width="800"/>

സ്കോർ പരിധികൾ:  
- 0.7-1.0: വളരെ പ്രസക്തം, കൃത്യമായ പൊരുത്തം  
- 0.5-0.7: പ്രസക്തമായ, നല്ല സന്ദർഭം  
- 0.5-യുടെ താഴ്‌വിൽ: ഫിൽട്ടർ ചെയ്തത്, വളരെ വ്യത്യസ്തം  

ന്യുനാംശ പരിധിക്ക് മുകളിലുള്ള ചങ്കുകൾ മാത്രമേ സിസ്റ്റം തിരഞ്ഞെടുത്ത് കാണിക്കൂ, നല്ല ഗുണമേന്മ ഉറപ്പാക്കാൻ.

### In-Memory Storage

ഈ മോഡ്യൂളിൽ എളുപ്പത്തിന് ഇൻ-മെമ്മറി സംഭരണം ഉപയോഗിക്കുന്നു. ആപ്ലിക്കേഷൻ പുനരാരംഭിച്ചാൽ അപ്‌ലോഡ് ചെയ്ത ഡോക്യുമെന്റുകൾ നഷ്ടപ്പെടും. പ്രൊഡക്ഷൻ സിസ്റ്റങ്ങൾ സ്ഥിരമായ വെക്ടർ ഡാറ്റാബേസുകൾ (Qdrant, Azure AI Search) ഉപയോഗിക്കുന്നുണ്ട്.

### Context Window Management

ഓരോ മോഡലിനും പരമാവധി സന്ദർഭ വിൻഡോ ഉണ്ട്. വലിയ ഡോക്യുമെന്റിൽ നിന്ന് എല്ലാ ചങ്കുകളും ഉൾപ്പെടുത്താൻ കഴിയില്ല. സിസ്റ്റം പരമാവധി 5 ഏറ്റവും പ്രസക്തമായ ചങ്കുകൾ തിരഞ്ഞെടുത്തു പരിധിയുള്ള വിൻഡോയിൽ വെച്ച് ശരിയായ ഉത്തരം നൽകാൻ ആവശ്യമായ സന്ദർഭം ഉറപ്പാക്കുന്നു.

## When RAG Matters

എപ്പോഴാണ് RAG ശരിയായ മാർഗം എന്നു തീരുമാനിക്കാൻ താഴെയുള്ള മാർഗനിർദ്ദേശം സഹായിക്കും - RAG മൂല്യം കൂട്ടുന്ന സമയവും, നേരിട്ടുള്ള പ്രോംപ്റ്റിൽ ഉള്ളടക്കം ഉൾപ്പെടുത്തുകയോ മോഡലിന്റെ ഉൾനിലജ്ഞാനത്തിൽ ആശ്രയിക്കുകയോ 하는 ലളിതമായ രീതികൾ മതിയായ സമയവും:

<img src="../../../translated_images/ml/when-to-use-rag.1016223f6fea26bc.webp" alt="When to Use RAG" width="800"/>

**RAG ഉപയോഗിക്കുക,**
- സ്വന്തമായ രേഖകളുമായി ബന്ധപ്പെട്ട ചോദ്യങ്ങൾക്ക് ഉത്തരം നൽകുക  
- വിവരങ്ങൾ നിരന്തരം മാറുന്നു (നയം, വിലകൾ, പ്രത്യേകതകൾ)  
- കൃത്യത ഉറപ്പാക്കാൻ സ്രോതസ്സുകൾ പറയേണ്ടിവരും  
- ഉള്ളടക്കം ഒരു പ്രాంప്റ്റിൽ_fit_ചെയ്യേണ്ടതിൽ വലുതാണ്  
- നിങ്ങൾക്ക് പരിശോധന സ്വഭാവമുള്ള, അടിത്തറയുള്ള മറുപടികൾ വേണം  

**പിൻഷേം ഉപയോഗിക്കരുതെന്ന്:**  
- ചോദ്യങ്ങൾ മോഡലിന് ഇതിനകം ഉള്ള സാധാരണ വിജ്ഞാനം ആവശ്യപ്പെട്ടാൽ  
- യഥാർത്ഥ സമയ ഡേറ്റ വേണെങ്കിൽ (പിന്‍ഷേം അപ്ലോഡ് ചെയ്ത രേഖകളിൽ പ്രവർത്തിക്കുന്നു)  
- ഉള്ളടക്കം വളരെ ചെറിയതാണ്, പ്രോമ്പ്റ്റുകളിൽ നേരിട്ട് ഉൾപ്പെടുത്താനാകുമ്പോൾ
