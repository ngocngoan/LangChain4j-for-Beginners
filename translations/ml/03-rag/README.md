## What You'll Learn

കഴിഞ്ഞ മോഡ്യൂളുകളിൽ, നിങ്ങൾ എങ്ങനെ AI യുമായി സംഭാഷണം നടത്താം, പ്രോംപ്റ്റുകൾ ഫലപ്രദമായി ഘടിപ്പിക്കാം എന്നിവ പഠിച്ചിരുന്നു. പക്ഷേ ഒരു അടിസ്ഥാന നിയന്ത്രണം ഉണ്ട്: ഭാഷ മോഡലുകൾക്ക് മാത്രം അവരെ പരിശീലിപ്പിച്ചുകൊടുത്തവ തന്നെ അറിയാം. നിങ്ങളുടെ കമ്പനിയുടെ നയങ്ങൾ, നിങ്ങളുടെ പ്രോജക്ട് രേഖകൾ, അല്ലെങ്കിൽ അവരെ പരിശീലിപ്പിക്കാതിരുന്ന ഏതെങ്കിലും വിവരങ്ങൾക്കു മറുപടി നൽകാൻ അവയ്ക്ക് കഴിയില്ല.

RAG (റീട്രീവൽ-ഓഗ്മെന്റഡ് ജനറേഷൻ) ഈ പ്രശ്നം പരിഹരിക്കുന്നു. മോഡലിന് നിങ്ങളുടെ വിവരങ്ങൾ പഠിപ്പിക്കാനായുള്ള ശ്രമം (ഇത് ചെലവേറിയതും പ്രായോഗികവുമായില്ല) ചെയ്യുന്നതിനുപകരം, നിങ്ങൾ അതിന് നിങ്ങളുടെ രേഖകളിലൂടെ തിരയാൻ കഴിയുന്ന കഴിവ് നൽകുന്നു. ഒരാൾ ചോദ്യം ചോദിക്കുമ്പോഴാണ് സിസ്റ്റം ബന്ധപ്പെട്ട വിവരങ്ങൾ കണ്ടെത്തി പ്രോംപ്റ്റിൽ ഉൾപ്പെടുത്തുന്നത്. മോഡൽ പിന്നെ അവ റിട്രീവ് ചെയ്ത കണ്ടക്സ്റ്റ് അടിസ്ഥാനമാക്കി മറുപടി നൽകുന്നു.

RAG മോഡലിന് ഒരു റഫറൻസ് ലൈബ്രറി നൽകുന്നതുപോലെയാണ്. നിങ്ങൾ ചോദ്യംചെയ്യുമ്പോൾ, സിസ്റ്റം:

1. **ഉപയോക്തൃ ചോദ്യം** - നിങ്ങൾ ചോദ്യംചെയ്യുന്നു
2. **എംബെഡിങ്** - നിങ്ങളുടെ ചോദ്യം ഒരു വെക്ടറാക്കി മാറ്റുന്നു
3. **വെക്ടർ തിരച്ചിൽ** - സമാനമായ ഡോക്യുമെന്റ് ചങ്കുകൾ കണ്ടെത്തുന്നു
4. **കണ്ടക്സ്റ്റ് അസംബ്ലി** - പ്രസക്തമായ ചങ്കുകൾ പ്രോംപ്റ്റിൽ ചേർക്കുന്നു
5. **പ്രതികരണം** - LLM കണ്ടക്സ്റ്റ് അടിസ്ഥാനമാക്കി ഉത്തരം സൃഷ്ടിക്കുന്നു

ഇത് മോഡലിന്റെ മറുപടികൾ നിങ്ങളുടെ യഥാർത്ഥ ഡാറ്റയിൽ ആധാരമാക്കുന്നു, പരിശീലന ഡാറ്റയിൽ ആശ്രയിക്കുകയോ മറുപടികൾ കരുതുകയോ ചെയ്യാതെ.

## Understanding RAG

താഴെയുള്ള ചിത്രീകരണം මുഖ്യ ആശയം വിശദീകരിക്കുന്നു: മോഡലിന്റെ പരിശീലന ഡാറ്റയിൽ മാത്രം ആശ്രയിക്കുന്നതിനുപകരം, RAG ഓരോ മറുപടി സൃഷ്ടിക്കുന്നതിനുമുമ്പും പരിശോധിക്കാനുള്ള നിങ്ങളുടെ രേഖകളുടെ റഫറൻസ് ലൈബ്രറി നൽകുന്നു.

<img src="../../../translated_images/ml/what-is-rag.1f9005d44b07f2d8.webp" alt="What is RAG" width="800"/>

*ഈ ചിത്രീകരണം ഒരു സ്റ്റാൻഡർഡ് LLM (പഠന ഡാറ്റയിൽ നിന്ന് കരുതുന്നു) കൂടാതെ ഒരു RAG-ഉയർത്തിയ LLM (നിങ്ങളുടെ രേഖകൾ ആദ്യം പരിശോധിക്കുന്നു) തമ്മിലുള്ള വ്യത്യാസം കാണിക്കുന്നു.*

ഇവിടെയാണ് ഭാഗങ്ങൾ അവസാനത്തേയ്ക്കെത്തുന്നത്: ഒരു ഉപയോക്താവിന്റെ ചോദ്യം നാല് ഘട്ടങ്ങളിലൂടെ ഒഴുകുന്നു — എംബെഡിംഗ്, വെക്ടർ തിരച്ചിൽ, കണ്ടക്സ്റ്റ് അസംബ്ലി, മറുപടി സൃഷ്ടി — ഓരോ ഘട്ടവും മുൻതുറക്കുന്ന ഘട്ടത്തിന്‍റെ അടിസ്ഥാനത്തിൽ രൂപപ്പെടുന്നു:

<img src="../../../translated_images/ml/rag-architecture.ccb53b71a6ce407f.webp" alt="RAG Architecture" width="800"/>

*ഈ ചിത്രീകരണം എൻഡ്-ടു-എൻഡ് RAG പൈപ്പ്‌లൈൻ കാണിക്കുന്നു — ഉപയോക്താവിന്റെ ചോദ്യം എംബെഡിംഗ്, വെക്ടർ തിരച്ചിൽ, കണ്ടക്സ്റ്റ് അസംബ്ലി, മറുപടി സൃഷ്ടി എന്നിവ വഴി ഒഴുകുന്നു.*

മറ്റ് ഭാഗം ഈ മോഡ്യൂൾ ഓരോ ഘട്ടവും വിശദമായി ചുവടുവയറ്റുന്നു, നിങ്ങൾക്കു റൺ ചെയ്‌തും മാതൃകകളും മാറ്റിയും ഉപയോഗിക്കാൻ സാധിക്കും.

### Which RAG Approach Does This Tutorial Use?

LangChain4j RAG നടപ്പിലാക്കാന് മൂന്ന് രീതികൾ നൽകുന്നു, ഓരോന്നിന്റെയും വ്യത്യസ്ത ലെവൽ abstractions ഉണ്ട്. താഴെയുള്ള ചിത്രത്തിൽ അവ പക്കൽനെ പൊരുത്തപ്പെടുത്തുന്നു:

<img src="../../../translated_images/ml/rag-approaches.5b97fdcc626f1447.webp" alt="Three RAG Approaches in LangChain4j" width="800"/>

*ഈ ചിത്രത്തിൽ Easy, Native, Advanced എന്നിങ്ങനെ മൂന്ന് LangChain4j RAG രീതികൾ അവരുടെ പ്രധാന ഘടകങ്ങൾയും ഉപയോഗിക്കേണ്ട സമയവും പ്രകടിപ്പിക്കുന്നു.*

| രീതി | അത് ചെയ്യുക | തുല്യസ്ഥിതി |
|---|---|---|
| **Easy RAG** | `AiServices` ഉം `ContentRetriever` ഉം വഴി എല്ലാ ഒട്ടുമിക്കവുമും സ്വയം वायरിംഗ് ചെയ്യും. നിങ്ങൾ ഒരു ഇന്റർഫേസ് അനോട്ടേറ്റ് ചെയ്ത്, റിട്രീവർ ചേർക്കുകയാണു്; LangChain4j എംബെഡിങ്‌, തിരച്ചിൽ, പ്രോംപ്റ്റ് അസംബ്ലി എല്ലാം പശ്ചാത്തലത്തിൽ കൈകാര്യം ചെയ്യുന്നു. | കുറഞ്ഞ കോഡ്, പക്ഷേ ഓരോ ഘട്ടത്തിലും എന്തുണ്ടെന്ന് കാണാനാകില്ല. |
| **Native RAG** | നിങ്ങൾക്കു സഹായിക്കുന്നത് — embedding മോഡൽ വിളിക്കുക, സ്റ്റോർ തിരയുക, പ്രോംപ്റ്റ് നിർമ്മിക്കുക, മറുപടി സൃഷ്ടിക്കുവ് — ഓരോ ഘട്ടവും വ്യക്തമായി നിർവഹിക്കുന്നു. | കൂടുതൽ കോഡ്, പക്ഷേ ഓരോ ഘട്ടവും കാണാനും മാറ്റാനും കഴിയും. |
| **Advanced RAG** | പ്രൊഡക്ഷൻ-ഗ്രേഡ് പൈപ്പ്‌ലൈനുകൾക്ക് pluggable ക്വറി ട്രാൻസ്ഫോർമറുകൾ, റൂട്ടറുകൾ, റീ-റാങ്കറുകൾ, കത്തന്റിവസ്തുക്കൾ എന്നിവ ഉപയോഗിച്ച് `RetrievalAugmentor` ഫ്രെയിംവർക്കുകൾ ഉപയോഗിക്കുന്നു. | പരമാവധി ലളിതമായ സൗകര്യം, പക്ഷേ വളരെ അധികം സങ്കീർണ്ണത. |

**ഈ ട്യൂട്ടോറിയൽ Native രീതി ഉപയോഗിക്കുന്നു.** RAG പൈപ്പ്‌ലൈനിലെ ഓരോ ഘട്ടവും – ക്വറി എംബെഡിംഗ്, വെക്ടർ സ്റ്റോർ തിരച്ചിൽ, കണ്ടക്സ്റ്റ് അസംബ്ലി, മറുപടി സൃഷ്ടി – [RagService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/RagService.java) -ല് വ്യക്തമായി എഴുതിയതാണ്. ഇത് ഉദ്ദേശപൂർവ്വമാണ്: പഠന വിഭവമായി, ഓരോ ഘട്ടവും കാണാനും മനസിലാക്കാനും കഴിയുന്നത് കൂടുതൽ പ്രധാനം; കോഡ് കുറഞ്ഞുകൊണ്ടിരിക്കുമ്പോൾ അല്ല. ഭാഗങ്ങൾ എങ്ങനെ ചേർക്കപെടുന്നു എന്ന് നിങ്ങൾ അറിയുമ്പോൾ, എളുപ്പ RAG യിലേക്കോ പ്രൊഡക്ഷൻ സിസ്റ്റങ്ങളിലേക്ക് Advanced RAG യിലേക്കോ നീങ്ങാം.

> **💡 എളുപ്പ RAG യെ നിങ്ങൾ ഇതിനകം കണ്ടിട്ടുണ്ടോ?** [Quick Start module](../00-quick-start/README.md) Document Q&A ഉദാഹരണമുള്‍പ്പെടുന്നു ([`SimpleReaderDemo.java`](../../../00-quick-start/src/main/java/com/example/langchain4j/quickstart/SimpleReaderDemo.java)) എളുപ്പ RAG രീതി ഉപയോഗിച്ച് - LangChain4j എംബെഡിങ്, തിരച്ചിൽ, പ്രോംപ്റ്റ് അസംബ്ലി സ്വയം കൈകാര്യം ചെയ്യുന്നു. ഈ മോഡ്യൂള് ഒന്ന് മുന്നോട്ട് പോവുന്നു, മൊത്തം പൈപ്പ്‌ലൈനിനെ തുറന്ന് നിങ്ങൾക്ക് ഓരോ ഘട്ടവും കാണാനും നിയന്ത്രിക്കാനും സാധിക്കുന്നു.

*ഈ ചിത്രത്തിൽ `SimpleReaderDemo.java` -മുന്‍ നിന്ന Easy RAG പൈപ്പ്‌ലൈൻ കാണിക്കുന്നു. ഈ മോഡ്യൂളിൽ ഉപയോഗിക്കുന്ന ന്യൂനരൂപ Native രീതിയുമായി താരതമ്യം ചെയ്യുക: Easy RAG എംബെഡിങ്, റെട്രീവൽ, പ്രോംപ്റ്റ് അസംബ്ലി `AiServices` ഉം `ContentRetriever` ഉം മറച്ചു വച്ചിരിക്കുന്നു — നിങ്ങൾ ഡോക്യുമെന്റ് ലോഡ് ചെയ്യും, റെട്രീവർ ചേർക്കും, മറുപടികൾ പ്രാപിക്കും. ഈ മോഡ്യൂളിലെ Native രീതി ആ പൈപ്പ്‌ലൈനിൽ നിന്നും ഘട്ടങ്ങൾ മുഴുവൻ തുറക്കുന്നു, നിങ്ങൾ ഓരോ ഘട്ടവും (എംബെഡ്, തിരയുക, കണ്ടക്സ്റ്റ് അസംബ്ലി, ജനറേറ്റ്) വിളിച്ച് പൂര്‍ണ കാഴ്ചയും നിയന്ത്രണവും നൽകുന്നു.*

## How It Works

ഈ മോഡ്യൂളിലെ RAG പൈപ്പ്‌ലൈനിൽ ഓരോ തവണയും യوزർ ചോദ്യം ചോദിക്കുമ്പോൾ നാലു ഘട്ടം അനുക്രമമായി നടക്കുന്നു. ആദ്യം, അപ്‌ലോഡ് ചെയ്ത ഡോക്യുമെന്റ് **പാഴ്സിംഗ് ചെയ്യുകയും ചെറുതായി വിഭജിക്കുകയും** ചെയ്യുന്നു. ഈ ചങ്കുകൾ പിന്നീട് **വെക്ടർ എംബെഡിങ്ങുകളായി** രൂപപ്പെടുത്തുകയും സ്റ്റോർ ചെയ്യുകയും ചെയ്യും. ചോദ്യം വന്നാൽ, സിസ്റ്റം **സെമാന്റിക് തിരച്ചിൽ** നടത്തുന്നു ഏറ്റവും പ്രസക്തമായ ചങ്കുകൾ കണ്ടെത്താൻ, ഒടുവിൽ അവ LLM എടുക്കാതെ മറുപടി സൃഷ്ടിക്കേണ്ട കണ്ടക്സ്റ്റായി നൽകുന്നു. താഴെ ഓരോ ഘട്ടവും കൃത്യമായ കോഡും ചിത്രം സഹിതം വിശദീകരിക്കുന്നു. ആദ്യം ഘട്ടം നോക്കാം.

### Document Processing

[DocumentService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/DocumentService.java)

നിങ്ങൾ ഒരു ഡോക്യുമെന്റ് അപ്‌ലോഡ് ചെയ്‌യുമ്പോൾ, സിസ്റ്റം അത് പാഴ്സു ചെയ്യുന്നു (PDF അല്ലെങ്കിൽ പ്ലെയിൻ ടെക്സ്റ്റ്), ഫയൽനെയിം പോലുള്ള മെറ്റാഡേറ്റ കൂട്ടിച്ചേർക്കുന്നു, ഒപ്പം അത് ചങ്കുകളായി തകർക്കുന്നു — മോഡലിന്റെ കണ്ടക്സ്റ്റ് വിൻഡോയിലായി സൗകര്യപ്രദമായ ചെറുതായി. ഈ ചങ്കുകൾ പരസ്പരം ചെറിയ തോതിൽ കൂട്ടിയിടുന്നു അതായത് അതിരുകൾക്കിടയിലെ കണ്ടക്സ്റ്റ് നഷ്ടപ്പെടുന്നില്ല.

```java
// അപ്‌ലോഡ് ചെയ്‍ത ഫയൽ വിശകലനം ചെയ്ത് LangChain4j ഡോക്യുമെന്റായി മടക്കി ഏർക്കുക
Document document = Document.from(content, metadata);

// 300-ടോകൺ ഘടകങ്ങളായി വിഭജിക്കുക, 30-ടോകൺ ഓവർലാപ്പോടുകൂടെ
DocumentSplitter splitter = DocumentSplitters
    .recursive(300, 30);

List<TextSegment> segments = splitter.split(document);
```
  
താഴെയുള്ള ചിത്രത്തിൽ ഇത് ദൃശ്യമായി കാണിക്കുന്നു. ഓരോ ചങ്കും അടുത്തുള്ള ചില ടോക്കണുകൾ പങ്കിടുന്നതു ശ്രദ്ധിക്കുക — 30 ടോക്കൺ ഒവർലാപ്പ് പ്രധാനമായ കണ്ടക്സ്റ്റ് തിരുവിളിക്കുന്നില്ലെന്ന് ഉറപ്പാക്കുന്നു:

<img src="../../../translated_images/ml/document-chunking.a5df1dd1383431ed.webp" alt="Document Chunking" width="800"/>

*ഈ ചിത്രത്തില് ഡോക്യുമെന്റ് 300 ടോക്കൺ ചങ്കുകള് ആയി 30 ടോക്കൺ ഒവർലാപ്പിൽ ആയാണ് വിഭജിക്കുന്നത്, ചങ്ക് അതിരുകളിൽ കണ്ടക്സ്റ്റ് സംരക്ഷിക്കുന്നു.*

### Creating Embeddings

[LangChainRagConfig.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/config/LangChainRagConfig.java)

ഓരോ ചങ്കും ഒരു സാംഖ്യാത്മക പ്രതിനിധാനമായ എംബെഡിങ്ങിലേക്ക് മാറ്റുന്നു — അതായത് അർത്ഥം-നമ്ബറുകളായി മാറ്റുന്ന ഒരു പരിണാമം. എംബെഡിങ് മോഡൽ ഒരു ചാറ്റ് മോഡലുപോലെ "ബുദ്ധിമാനല്ല"; നിർദ്ദേശങ്ങൾ പാലിക്കുകയോ, തർക്കം ചെയ്യുകയോ, ചോദ്യം മറുപടി നൽകുകയോ സാധിക്കില്ല. എന്തെങ്കിലും ടെക്സ്റ്റിൽ നിന്ന് സമാനമായ അർത്ഥങ്ങൾ അടുത്ത് വയ്ക്കുന്ന ഗണിതീയ സ്പേസിലേക്ക് വിതയ്ക്കുകയാണ് സാധ്യമായത് — "കാർ" എന്നത് "ഓട്ടോമൊബൈൽ" നിന് അടുത്ത്, "റിഫണ്ട് നയം" "പണം തിരികെ നൽകൽ" ന് അടുത്ത്. ഒരു ചാറ്റ് മോഡൽ നിങ്ങളെ പോലെ സംസാരിക്കാൻ കഴിയും; എംബെഡിങ് മോഡൽ അതിന്റെ ഫയലിങ് സിസ്റ്റം പോലെയാണ്.

<img src="../../../translated_images/ml/embedding-model-concept.90760790c336a705.webp" alt="Embedding Model Concept" width="800"/>

*ഈ ചിത്രത്തിൽ എങ്ങനെ എംബെഡിങ് മോഡൽ ടെക്സ്റ്റ് സംഖ്യാ-വെക്ടറുകളായി മാറ്റുന്നുവെന്നും സമാന അർത്ഥങ്ങൾ - "കാർ" എന്നും "ഓട്ടോമൊബൈൽ" എന്നും - വെക്ടർ ഖണ്ഡത്തിലെ അടുത്തു വയ്ക്കുന്നതുമാണ് കാണിക്കുന്നത്.*

```java
@Bean
public EmbeddingModel embeddingModel() {
    return OpenAiOfficialEmbeddingModel.builder()
        .baseUrl(azureOpenAiEndpoint)
        .apiKey(azureOpenAiKey)
        .modelName(azureEmbeddingDeploymentName)
        .build();
}

EmbeddingStore<TextSegment> embeddingStore = 
    new InMemoryEmbeddingStore<>();
```
  
താഴെയുള്ള ക്ലാസ് ചിത്രം RAG പൈപ്പ്‌ലൈനിലെ രണ്ട് പ്രധാനം പ്രവാഹങ്ങളും LangChain4j ക്ലാസ്സുകളും കാണിക്കുന്നു. **ഇംജെഷൻ ഫ്ലോ** (അപ്‌ലോഡ് സമയത്ത് ഒരിക്കൽ ഓടുന്നു) ഡോക്യുമെന്റ് വിഭജിച്ച്, ചങ്കുകൾ എംബെഡ് ചെയ്ത്, `.addAll()` വഴിയാണ് സ്റ്റോർ ചെയ്യുന്നത്. **ക്വറി ഫ്ലോ** (പ്രതിക്കോട്ടു ഉപയോക്താവ് ചോദിക്കുമ്പോൾ ഓടുന്നു) ചോദ്യം എംബെഡ് ചെയ്ത്, സ്റ്റോർ `.search()` വഴി തിരയുകയും, കണ്ടെത്തിയ കണ്ടക്സ്റ്റ് ചാറ്റ് മോഡലിന് നൽകുകയും ചെയ്യുന്നു. ഇരുവരും പങ്കിടുന്ന `EmbeddingStore<TextSegment>` ഇന്റർഫേസിൽ കാണിക്കുന്നു:  

<img src="../../../translated_images/ml/rag-langchain4j-classes.bbf3aa9077ab443d.webp" alt="LangChain4j RAG Classes" width="800"/>

*ഈ ചിത്രത്തിൽ RAG പൈപ്പ്‌ലൈനിലെ രണ്ട് പ്രവാഹങ്ങൾ — ഇംജെഷൻ, ക്വറി — അവ എങ്ങനെ EmbeddingStore വഴി ബന്ധിപ്പിക്കപ്പെടുന്നുവെന്ന് കാണിക്കുന്നു.*

### Semantic Search

[RagService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/RagService.java)

നിങ്ങൾ ചോദ്യം ചോദിക്കുമ്പോൾ, നിങ്ങളുടെ ചോദ്യം ഒരു എംബെഡിങ്ങുംആകുന്നു. സിസ്റ്റം നിങ്ങളുടെ ചോദ്യം എംബെഡിങ്ങും, ഡോക്യുമെന്റ് ചങ്കുകളുടെ എംബെഡിങ്ങുകളുമായും താരതമ്യം ചെയ്യുന്നു. അതായത് ഏറ്റവും സമാനമായ അർത്ഥമുള്ള ചങ്കുകൾ കണ്ടെത്തുന്നു - ഇതിൽ കീവേഡ് മാത്രം പൊരുത്തപ്പെടുന്നത് അല്ല, വാസ്തവ സെമാന്റിക് സമാനത ആണ്.

```java
Embedding queryEmbedding = embeddingModel.embed(question).content();

EmbeddingSearchRequest searchRequest = EmbeddingSearchRequest.builder()
    .queryEmbedding(queryEmbedding)
    .maxResults(5)
    .minScore(0.5)
    .build();

EmbeddingSearchResult<TextSegment> searchResult = embeddingStore.search(searchRequest);
List<EmbeddingMatch<TextSegment>> matches = searchResult.matches();

for (EmbeddingMatch<TextSegment> match : matches) {
    String relevantText = match.embedded().text();
    double score = match.score();
}
```
  
താഴെയുള്ള ചിത്രം പ്രദർശിപ്പിക്കുന്നത് സെമാന്റിക് തിരച്ചില് പരമ്പരാഗത കീവേഡ് തിരച്ചിലുമായി എങ്ങനെ വ്യത്യാസപ്പെടുന്നു എന്നതാണ്. "വാഹനം" എന്ന കീവേഡുള്ള തിരച്ചിൽ "കാറുകളും ട്രക്കുകളും" എന്നൊരു ചങ്ക് പാഴ്‌സുചെയ്യാൻമുട്ടുന്നില്ല, പക്ഷേ സെമാന്റിക് തിരച്ചിൽ അവയ്ക്ക് സമാനം എന്ന് മനസ്സിലാക്കി ഉയര്‍ന്ന സ്കോർ നൽകുന്നു:

<img src="../../../translated_images/ml/semantic-search.6b790f21c86b849d.webp" alt="Semantic Search" width="800"/>

*ഈ ചിത്രത്തില് കീവേഡ് അടിസ്ഥാനമായ തിരച്ചിൽ സെമാന്റിക് തിരച്ചിലുമായി താരതമ്യം ചെയ്യുന്നു, എങ്ങനെ സെമാന്റിക് തിരച്ചിൽ കീവേഡുകൾ പൊരുത്തപ്പെടാതെ ഉള്ള ആശയവുമായുള്ള ഉള്ളടക്കം കണ്ടെത്തുന്നു എന്ന് കാണിക്കുന്നു.*

അടിസ്ഥാനത്തിൽ, സമാനത്‌വം കോസൈൻ സമാനത ഉപയോഗിച്ച് അളക്കുന്നു — അതായത് "ഈ രണ്ട് സൂചികകള് ഒരേ ദിശയിൽ സൂചിപ്പിക്കുന്നുണ്ടോ?" എന്ന് ചോദിക്കുന്നത്. രണ്ട് ചങ്കുകൾ പൂർണ്ണമായും വ്യത്യസ്ത വാക്കുകൾ ഉപയോഗിക്കാം, പക്ഷേ അവ സമാന അർത്ഥം കൈകാര്യം ചെയ്യുന്നുവെങ്കിൽ അവയുടെ വെക്ടറുകൾ ഒരേ ദിശയിൽ സൂചിപ്പിക്കുന്നു, സ്കോർ 1.0-ന് അടുത്തതാണ്:

<img src="../../../translated_images/ml/cosine-similarity.9baeaf3fc3336abb.webp" alt="Cosine Similarity" width="800"/>

*ഈ ചിത്രത്തിൽ എംബർഡിങ്ങ് വെക്ടറുകൾ തമ്മിലെ കോസൈൻ സമാനത എന്ന ആംഗിള്‍ ചിത്രീകരിക്കുന്നു — കൂടുതൽ അരുമുറുക്കപ്പെട്ട വെക്ടറുകൾ 1.0-ന് അടുത്തു സ്കോർ നേടുന്നു, അത് ഉയർന്ന സെമാന്റിക് സമാനത സൂചിപ്പിക്കുന്നു.*

### ഉത്തര നിർമ്മാണം

[RagService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/RagService.java)

വളരെ പ്രാധാന്യമുള്ള ഭാഗങ്ങൾ നിർദ്ദിഷ്ട നിർദ്ദേശങ്ങൾ, തിരിച്ചെടുത്ത സംദർഭം, ഉപയോക്താവിന്റെ ചോദ്യം എന്നിവ ഉൾപ്പെടുന്ന ഘടനാപരമായ പ്രോമ്പ്റ്റിലേക്ക് ഒടുക്കപ്പെടുന്നു. മോഡൽ ആ പ്രത്യേക ഭാഗങ്ങൾ ಓദ്യോഗിച്ച് ആ വിവരങ്ങളെ അടിസ്ഥാനമാക്കിയുള്ള ഉത്തരങ്ങൾ നൽകുന്നു — ഇത് മുന്നിൽ ഉള്ളതിനെ മാത്രം ഉപയോഗിക്കാനാകും, അതുവഴി ഹാൽസിനേഷൻ ഒഴിവാക്കുന്നു.

```java
String context = matches.stream()
    .map(match -> match.embedded().text())
    .collect(Collectors.joining("\n\n"));

String prompt = String.format("""
    Answer the question based on the following context.
    If the answer cannot be found in the context, say so.

    Context:
    %s

    Question: %s

    Answer:""", context, request.question());

String answer = chatModel.chat(prompt);
```

താഴെയുള്ള ചിത്രരേഖ ഈ കൂട്ടിച്ചേർക്കൽ പ്രവർത്തനത്തെ കാണിക്കുന്നു — തിരച്ചിൽ ഘട്ടത്തിൽ ഏറ്റവും ഉയർന്ന സ്കോറുള്ള ഭാഗങ്ങൾ പ്രോമ്പ്റ്റ് ടെംപ്ലേറ്റിൽ ചേർക്കപ്പെടുന്നുണ്ട്, `OpenAiOfficialChatModel` ആധാരമാക്കിയ ഉത്തരങ്ങൾ സൃഷ്ടിക്കുന്നു:

<img src="../../../translated_images/ml/context-assembly.7e6dd60c31f95978.webp" alt="Context Assembly" width="800"/>

*ഈ ചിത്രരേഖയിൽ, ഏറ്റവും ഉയർന്ന സ്കോറുള്ള ഭാഗങ്ങൾ ഘടനാപരമായ പ്രോമ്പ്റ്റായി ഒടുക്കപ്പെടുന്നുവെന്ന് കാണിക്കുന്നു, ഇതിലൂടെ മോഡൽ നിങ്ങളുടെ ഡാറ്റയിൽ നിന്നുള്ള അടിസ്ഥാനമുള്ള ഒരു ഉത്തരമുണ്ടാക്കുന്നു.*

### ഒരു ഡോക്യുമെന്റ് അപ്‌ലോഡ് ചെയ്യുക

ഒരു ഡോക്യുമെന്റ് അപ്‌ലോഡ് ചെയ്ത് തുടങ്ങൂ - പരീക്ഷണത്തിനായി TXT ഫയലുകൾ ഉത്തമമാണ്. ഈ ഡയറക്ടറിയിൽ ഒരു `sample-document.txt` ഉണ്ടായിരിക്കുന്നു, LangChain4j ഫീച്ചറുകൾ, RAG നടപ്പാക്കല്‍, മികച്ച പ്രാക്ടിസുകൾ എന്നിവയെക്കുറിച്ച് വിവരങ്ങൾ ഉൾപ്പെടുന്നു — പരീക്ഷണത്തിന് അനുയോജ്യം.

സിസ്റ്റം നിങ്ങളുടെ ഡോക്യുമെന്റ് പ്രക്രിയ ചെയ്ത്, അതിനെ ഒതുങ്ങിയ ഭാഗങ്ങളായി പിളർക്കും, ഓരോ ഭാഗത്തിനും എംബെഡിംഗുകൾ സൃഷ്ടിക്കും. ഇത് നിങ്ങളുടെ അപ്ലോഡ് ചെയ്തപ്പോൾ തന്നെ ഓട്ടോമാറ്റിക്കായി സംഭവിക്കും.

### ചോദ്യങ്ങൾ ചോദിക്കുക

ഇപ്പോൾ ഡോക്യുമെന്റ് ഉള്ളടക്കത്തെക്കുറിച്ചുള്ള പ്രത്യേക ചോദ്യങ്ങൾ ചോദിക്കൂ. വ്യക്തമല്ലെങ്കിൽ ഇതിൽ നിന്നുള്ള ഫർക്കെട്ടു സൂചിപ്പിക്ക്ത്തുണ്ടാകണം. സിസ്റ്റം ബന്ധപ്പെട്ട ഭാഗങ്ങൾ തിരഞ്ഞെടുത്ത് പ്രോമ്പ്റ്റിൽ ഉൾപ്പെടുത്തി അവതരണമുണ്ടാക്കും.

### ഉറവിടം റഫറൻസുകൾ പരിശോധിക്കുക

ഓരോ ഉത്തരത്തിലും ഉറവിട റഫറൻസുകളും സാമ്യതാ സ്കോറുകളുമുണ്ട്. ഈ സ്കോറുകൾ (0 മുതൽ 1 വരെ) നിങ്ങളുടെ ചോദ്യത്തിനു ബന്ധപ്പെട്ട ഭാഗങ്ങളോ സമ്മാനിച്ചവയോ എത്രത്തോളം ചേർന്നുവാനാണെന്ന് കാണിക്കുന്നു. മുകളിൽ ഉള്ള സ്കോർ കൂടുതലെങ്കിൽ നല്ല താരതമ്യം ഉള്ളതാണ്. ഇത് ഉത്തരത്തെ ഉറവിട വസ്തുക്കളുടെ അടിസ്ഥാനത്തിൽ സ്ഥിരീകരിക്കാൻ സഹായിക്കും.

<a href="images/rag-query-results.png"><img src="../../../translated_images/ml/rag-query-results.6d69fcec5397f355.webp" alt="RAG Query Results" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*ഈ സ്ക്രീൻഷോട്ട് ചോദคำലംഫലം കാണിക്കുന്നു, ഇതിൽ സൃഷ്ടിച്ച ഉത്തരവും ഉറവിട റഫറൻസുകളും വിജയിച്ച ഓരോ ഭാഗത്തിന്റെയും പ്രാധാന്യം സ്കോറുകളുമുണ്ട്.*

### ചോദ്യങ്ങളുമായി പരീക്ഷണം നടത്തുക

വിവിധ തരം ചോദ്യങ്ങൾ പരീക്ഷിക്കൂ:
- പ്രത്യേക വാസ്തവങ്ങൾ: "പ്രധാനം വിഷയം എന്ത്?"
- താരതമ്യങ്ങൾ: "Xനും Yനും തമ്മിൽ വ്യത്യാസമാത്രം എന്ത്?"
- സംഗ്രഹങ്ങൾ: "Z-യുടെ പ്രധാനപ്പെട്ട ചിന്തകൾ സംഗ്രഹിക്കൂ"

ചോദ്യങ്ങളോട് അനുയോജ്യമായ റീലവൻസ് സ്കോറുകൾ എങ്ങനെ മാറുന്നുവെന്ന് ശ്രദ്ധിക്കുക.

## പ്രധാന ആശയങ്ങൾ

### തകർച്ച തന്ത്രം

ഡോക്യുമെന്റുകൾ 300-ടോക്കൺ അടുക്കളകളായി പിളർക്കുന്നു, 30 ടോക്കൺ ഒവറ്ലാപ്പ് ഉള്ളത്. ഇതിലൂടെ ഓരോ ഭാഗവും മതിയായ സംദർഭം നൽകുന്നത്ര വലിയതായാകും, കൂടാതെ പരിധിക്കുള്ളിൽ നിറയ്ക്കാൻ ധാരാളം ഭാഗങ്ങൾ പ്രോമ്പ്റ്റിൽ ഉൾപ്പെടുത്താം.

### സാമ്യത സ്കോറുകൾ

തിരിച്ചെടുത്ത ഓരോ ഭാഗത്തിനും ഉപയോക്തൃ ചോദ്യവുമായി ചേർന്നിരിക്കുന്നതിനെ സൂചിപ്പിക്കുന്ന 0 മുതൽ 1 വരെ സാമ്യത സ്കോർ ഉണ്ടാകും. താഴെയുള്ള ചിത്രരേഖ ഈ സ്കോർ പരിധികൾ വിശദീകരിക്കുന്നു, കൂടാതെ സിസ്റ്റം ഫലങ്ങൾ എങ്ങനെ വേർതിരിക്കുന്നു എന്നതും കാണിക്കുന്നു:

<img src="../../../translated_images/ml/similarity-scores.b0716aa911abf7f0.webp" alt="Similarity Scores" width="800"/>

*ഈ ചിത്രരേഖ 0 മുതൽ 1 വരെ സ്കോർ പരിധികൾ കാണിക്കുന്നു, കുറഞ്ഞ പറ്റിപ്പ് 0.5-നു താഴെ അസംബന്ധ ഭാഗങ്ങൾ ഒഴിവാക്കുന്നു.*

സ്കോറുകളുടെ പരിധി:
- 0.7-1.0: വളരെ പ്രാധാന്യമുള്ള, കൃത്യമായ പശ്ചാത്തലം
- 0.5-0.7: സാരമായ, നല്ല സംദർഭം
- 0.5-ന് താഴെ: ഫിൽട്ടർ ചെയ്തവ, വളരെ വ്യത്യസ്തം

സിസ്റ്റം കുറഞ്ഞ പറ്റിപ്പ് മികവിനായി മാത്രം ഭാഗങ്ങൾ തിരയുന്നു.

എംബെഡിംഗുകൾ അർത്ഥം ആകൃതമായ വർഗ്ഗീകരണത്തിലായപ്പോഴാണ് നല്ലത്, എന്നാൽ അവയിൽ അന്ധസ്ഥലങ്ങൾ ഉണ്ട്. താഴെയുള്ള ചിത്രരേഖ സാധാരണ പരാജയ മോഡുകൾ കാണിക്കുന്നു — ഭാഗങ്ങൾ വളരെ വലുതെങ്കിൽ തെറ്റായ വെക്ടറുകൾ ഉണ്ടാകുന്നു, വളരെ ചെറുതായാൽ യോഗ്യതയില്ല, දෙർത്ഥാത്മക വാക്കുകൾ ഒട്ടേറെ ഗ്രൂപ്പുകളിലേക്ക് സൂചിപ്പിക്കുന്നു, കൃത്യമായ മിശ്രിതം (ഐഡികൾ, ഭാഗ നമ്പറുകൾ) എംബെഡിംഗുകളിൽ പ്രവർത്തിക്കാറില്ല:

<img src="../../../translated_images/ml/embedding-failure-modes.b2bcb901d8970fc0.webp" alt="Embedding Failure Modes" width="800"/>

*ഈ ചിത്രരേഖ സാധാരണ എംബെഡിംഗ് പരാജയ മോഡുകൾ കാണിക്കുന്നു: ഭാഗങ്ങൾ വളരെ വലുത്, വളരെ ചെറുത്, സഹാർഥകം ഉള്ള വാക്കുകൾ പല ഗ്രൂപ്പുകളിലേക്കും സൂചിപ്പിക്കുന്നു, കൃത്യമായ മിശ്രിതം പോലുള്ള ഐഡികൾ.*

### ഇൻ-മെമ്മറി സ്രോതസ്സ്

സൗകര്യത്തിനായി ഈ മോഡ്യൂൾ ഇൻ-മെമ്മറി സംഭരണമാണ് ഉപയോഗിക്കുന്നത്. അപ്ലിക്കേഷൻ വീണ്ടും ആരംഭിക്കുമ്പോൾ, അപ്‌ലോഡ് ചെയ്ത ഡോക്യുമെന്റുകൾ നഷ്ടപ്പെടും. യാഥാർത്ഥ്യത്തിൽ Qdrant പോലുള്ള സ്ഥിരതയുള്ള ഫീച്ചർ വേക്റ്റർ ഡാറ്റാബേസുകളും Azure AI Search പോലുള്ള സംവിധാനങ്ങളും ഉപയോഗിക്കപ്പെടുന്നു.

### സംദർഭ വിൻഡോ മാനേജ്‌മെന്റ്

ഓരോ മോഡലിനും പരമാവധി സംദർഭ വിൻഡോയുടെ പരിധി ഉണ്ട്. വലിയ ഡോകയുമെന്റിലെ എല്ലാ ഭാഗങ്ങളും ഉൾപ്പെടുത്താനാകില്ല. സിസ്റ്റം പരിമിതികൾക്കനുസരിച്ച് ഏറ്റവും പ്രസക്തിയായ ടോപ്പ് N (ഡീഫോൾട്ടായി 5) ഭാഗങ്ങൾ മാത്രം തിരഞ്ഞെടുത്ത് ഉത്തരം നൽകുന്നത്ര സംദർഭം നൽകുന്നു.

- കൃത്യത ഉറപ്പാക്കാൻ ഉറവിട സംവരണം ആവശ്യമായപ്പോൾ
- ഉള്ളടക്കം ഒരേസമയം പ്രോമ്പ്റ്റിൽ നിറയ്ക്കാൻ കഴിയാത്തത്ര വലുതായപ്പോൾ
- പരിശോധിക്കാവുന്ന, അടിസ്ഥാനമുള്ള മറുപടികൾ ആവശ്യമുള്ളപ്പോൾ

**RAG ഉപയോഗിക്കരുത്:**
- ചോദ്യങ്ങൾ മോഡലിന്റെ പൊതുവായ അറിവിൽ ഉൾപ്പെട്ടവയായിരിക്കും
- യാഥാർത്ഥകാല വിവരങ്ങൾ (RAG അപ്‌ലോഡ് ചെയ്ത ഡോക്യുമെന്റുകളിൽ മാത്രം പ്രവർത്തിക്കും)
- ഉള്ളടക്കം ചെറുതായിരിക്കും അതിവേഗം പ്രോമ്പ്റ്റിൽ ഉൾപ്പെടുത്താൻ സാധിക്കുന്നതിന്
