## What You'll Learn

മുന്‍പത്തെ മോഡ്യൂളുകളിൽ, AIയുമായി സംഭാഷണം നടത്താനും നിങ്ങളുടെ പ്രോംപ്റ്റുകൾ ഫലപ്രദമായി ഘടിപ്പിക്കാനും പഠിച്ചിരുന്നു. പക്ഷേ ഒരു അടിസ്ഥാനപരമായ പരിമിതിയാണ്: ഭാഷ മോഡലുകൾ ട്രെയിനിംഗിന്റെ സമയത്ത് കേവലം പഠിച്ചത് മാത്രമേ അറിയൂ. നിങ്ങളുടെ കമ്പനിയുടെ നയങ്ങളെക്കുറിച്ചോ, നിങ്ങളുടെ പ്രൊജക്ട് ഡോക്യുമെന്റേഷനോ, അല്ലെങ്കിൽ അത് ട്രെയിൻ ചെയ്തിട്ടില്ലാത്ത എന്തെങ്കിലും വിവരങ്ങളെക്കുറിച്ചോ അവര്‍ക്ക് ഉത്തരം പറയാനാകില്ല.

RAG (Retrieval-Augmented Generation) ഈ പ്രശ്നം പരിഹരിക്കുന്നു. മോഡലിന് നിങ്ങളുടെ വിവരങ്ങൾ പഠിപ്പിക്കാൻ ശ്രമിക്കുന്നതിന് പകരം (ഇത് ചെലവേറിയതും പ്രായോഗികമല്ലാത്തതുമാണ്), നിങ്ങള്‍ അതിന് നിങ്ങളുടെ ഡോക്യുമെന്റുകളിൽ തിരയാനുള്ള കഴിവ് നൽകുന്നു. ആരെങ്കിലും ഒരു ചോദ്യം ചോദിച്ചാൽ, സിസ്റ്റം ബന്ധപ്പെട്ട വിവരങ്ങൾ കണ്ടെത്തി പ്രോംപ്റ്റിൽ ഉൾപ്പെടുത്തുന്നു. പിന്നീട് മോഡൽ തിരഞ്ഞെടുത്ത ഉള്ളടക്കത്തിന്റെ അടിസ്ഥാനത്തിൽ ഉത്തരം നൽകുന്നു.

RAG നെ മോഡലിന് ഒരു റഫറൻസ് ലൈബ്രറി നല്‍കുന്നതായി കണക്കാക്കുക. നിങ്ങൾ ഒരു ചോദ്യം ചോദിക്കുമ്പോൾ, സിസ്റ്റം:

1. **ഉപയോക്തൃ ചോദ്യം** - നിങ്ങൾ ഒരു ചോദ്യം ചോദിക്കുന്നു
2. **എംബെഡിങ്** - നിങ്ങളുടെ ചോദ്യം വെക്ടറാക്കി മാറ്റുന്നു
3. **വെക്ടർ സേർച്ച്** - സമാനമായ ഡോക്യുമെന്റ് ചങ്കുകളും സമാനമായ ഭാഗങ്ങൾ കണ്ടെത്തുന്നു
4. **Context Assembly** - ബന്ധപ്പെട്ട ചങ്കുകൾ പ്രോംപ്റ്റിലേക്ക് ചേർക്കുന്നു
5. **പ്രതികരണം** - LLM 해당 context അടിസ്ഥാനമാക്കി ഒരു ഉത്തരം സൃഷ്ടിക്കുന്നു

ഇതാണ് മോഡലിന്റെ പ്രതികരണങ്ങളെ അവിന്റെ പരിശീലന അറിവിലല്ല, നിങ്ങളുടെ യഥാർത്ഥ ഡാറ്റയിൽ ആധാരമാക്കുന്നത്.

## Understanding RAG

താഴെയുള്ള സംരൂപം മുഖ്യ ആശയം വ്യക്തമാക്കുന്നു: മോഡലിന്റെ ട്രെയിനിംഗിൽ മാത്രം ആശ്രയിക്കാതെ, RAG ഓരോ ഉത്തരം സൃഷ്ടിക്കുന്നതിന് മുൻപ് നിങ്ങളുടെ ഡോക്യുമെന്റുകളുടെ റഫറൻസ് ലൈബ്രറി നൽകുന്നു.

<img src="../../../translated_images/ml/what-is-rag.1f9005d44b07f2d8.webp" alt="What is RAG" width="800"/>

*ഈ ഡയഗ്രാം സാധാരണ LLM (ട്രെയ്‌നിംഗ് ഡേറ്റയിൽ നിന്നുള്ള ഗസ്സ് ചെയ്യുന്നത്) നും RAG-ഉള്ള LLM (ആദ്യമായി നിങ്ങളുടെ ഡോക്യുമെന്റുകൾ പരിശോധിക്കുന്നത്) തമ്മിലുള്ള വ്യത്യാസം കാണിക്കുന്നു.*

ഉപയോക്താവിന്റെ ചോദ്യം നാലു ഘട്ടങ്ങളിലൂടെ ഒഴുകുന്നു — എംബെഡിങ്, വെക്ടർ തിരയൽ, context ഘടന, ഉത്തരം കുറിക്കല് — ഓരോത് മുൻപ് കൈകാര്യം ചെയ്ത ഘട്ടം അടിസ്ഥാനമാക്കിയുള്ളതാണ്:

<img src="../../../translated_images/ml/rag-architecture.ccb53b71a6ce407f.webp" alt="RAG Architecture" width="800"/>

*ഈ ഡയഗ്രാം മുഴുവൻ RAG പൈപ്പ്‌ലൈൻ കാണിക്കുന്നു — ഉപയോക്തൃ ചോദ്യം എംബെഡിങ്, വെക്ടർ തിരയൽ, context ഘടന, ഉത്തരം സൃഷ്ടിക്കൽ വഴി പോയി.*

ഈ മോഡ്യൂളിലെ വൈകാരിക കോഡ് നിങ്ങൾ ഓടിക്കാനും മാറിപ്പറയാനും സാധിക്കുന്ന വിധത്തിൽ ഓരോ ഘട്ടവും വിശദമായി സൂക്ഷിക്കും.

### Which RAG Approach Does This Tutorial Use?

LangChain4j RAG നടപ്പാക്കാൻ മൂന്ന് മാർഗങ്ങൾ നൽകുന്നു, ഓരോന്നിന്റെയും വ്യത്യസ്ത Abstraction അളവുകളുണ്ട്. താഴെ ഡയഗ്രാം അവ ചേർന്നിരിക്കുന്നു:

<img src="../../../translated_images/ml/rag-approaches.5b97fdcc626f1447.webp" alt="Three RAG Approaches in LangChain4j" width="800"/>

*ഈ ഡയഗ്രാം LangChain4jയിലെ മൂന്ന് RAG മാർഗങ്ങൾ - Easy, Native, Advanced - തമ്മിലുള്ള വ്യത്യാസവും ഓരോന്നും ഉപയോഗിക്കുന്ന സമയവും കാണിക്കുന്നു.*

| മാർഗ് | എന്ത് ചെയ്യുന്നു | ട്രേഡ്-ഓഫ് |
|---|---|---|
| **Easy RAG** | എല്ലാ(ele) പ്രവർത്തനങ്ങളും `AiServices` and `ContentRetriever` വഴി സ്വയം ഓട്ടോമാറ്റിക്കായി ബന്ധിപ്പിക്കുന്നു. ഒരു ഇന്റർഫേസ് അനോട്ടേറ്റ് ചെയ്യുക, ഒരു റിട്രീവർ ചേർക്കുക, LangChain4j ആണ് എംബെഡിങ്, തിരയൽ, പ്രോംപ്റ്റ് ഘടന പരിപാലിക്കുന്നത്. | കുറഞ്ഞ കോഡ്, പക്ഷേ ഓരോ ഘട്ടവും നടക്കുന്നത് കാണാനാകില്ല. |
| **Native RAG** | നിങ്ങൾ സ്വയം എംബെഡിങ് മോഡൽ വിളിച്ചുപറയുക, സോർച്ചിംഗ് പ്രവർത്തിക്കുക, പ്രോംപ്റ്റ് നിര്‍മ്മിക്കുകയും ഉത്തരം സൃഷ്ടിക്കുകയും ചെയ്യുക - ഒന്നൊന്ന് ദൃശ്യവും മാറ്റാനും കഴിയുന്ന രീതിയിലുള്ള ഘട്ടങ്ങൾ. | കൂടുതൽ കോഡ്, എന്നാൽ ഓരോ ഘട്ടവും ദൃശ്യവും തിരുത്തുകയും ചെയ്യാമെന്നും. |
| **Advanced RAG** | `RetrievalAugmentor` ഫ്രെയിംവർക്ക് ഉപയോഗിച്ച് ക്വറിയ്പര്‍വർത്തനങ്ങൾ, റൂട്ടറുകൾ, റീ-റാങ്കറുകൾ, ഉള്ളടക്കം ഇഞ്ചക്ടറുകൾ എന്നിവ ചേർത്തുനിൽക്കുന്ന പ്രൊഡക്ഷൻ-ഗ്രേഡ് പൈപ്പ്‌ലൈൻസ്. | പരമമായ സൗകര്യം, എന്നാൽ വളരെ കൂടുതൽ സങ്കീർണത. |

**ഈ ട്യൂട്ടോറിയൽ Native മാർഗമാണ് ഉപയോഗിക്കുന്നത്.** RAG പൈപ്പ്ലൈനിലെ ഓരോ ഘട്ടവും — ക്വറി എംബെഡിങ്, വെക്ടർ സ്റ്റോർ തിരയൽ, context ഘടന, ഉത്തരം സൃഷ്ടിക്കൽ — ഈ [RagService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/RagService.java) ലെ നിർദ്ദിഷ്ടമായി എഴുതപ്പെട്ടിരിക്കുന്നു. പഠനങ്ങൾക്ക് സൗകര്യമുള്ളതിനാൽ, കോഡ് കുറഞ്ഞത് കണക്കാക്കുമ്പോള്‍ ഓരോ ഘടവും കാണുകയും മനസിലാക്കുകയും ചെയ്യുന്നത് കൂടുതൽ പ്രധാനമാണ്. ഘടങ്ങൾ എങ്ങിനെയാണ് ചേർന്നിരിക്കുന്നത് മനസ്സിലായ ശേഷം, വേഗത്തിലുള്ള പ്രോട്ടോട്ടൈപ്പുകൾക്കായി Easy RAG ലേക്ക് അല്ലെങ്കിൽ ഉല്‍പാദനത്തുരുള്‍ Advanced RAG ലേക്ക് ഉയരാൻ കഴിയും.

## How It Works

ഈ മോഡ്യൂളിലെ RAG പൈപ്പ്‌ലൈൻ ഓരോ തവണയും ഒരു ഉപയോക്താവ് ചോദ്യം ചോദിക്കുന്നപ്പോഴും പ്രവർത്തിക്കുന്ന നാല് ഘട്ടങ്ങളായി വിഭജിക്കുന്നു. ആദ്യം, അപ്‌ലോഡ് ചെയ്തത് ഒരു ഡോക്യുമെന്റ് **പരിഭാഷിക്കുകയും ചെറുതായി വിഭജിക്കുകയും** ചെയ്യും. ആ ചെറുതായി വിഭജിച്ച ഭാഗങ്ങൾ പിന്നീട് **വെക്ടർ എംബെഡിങ്** ആയി മാറ്റിവയ്ക്കും, അതുകൊണ്ട് അവ ഗണിതപരമായി താരതമ്യം ചെയ്യാവുന്നതും ആയിരിക്കും. ഒരു ചോദ്യം വരുമ്പോൾ, സിസ്റ്റം ഏറ്റവും പ്രസക്തമായ ഭാഗങ്ങൾ കണ്ടെത്താൻ **സെമാന്റിക് തിരയൽ** നടത്തും, അവസാനം അവ LLM-ന് **ഉത്തരം സൃഷ്ടിക്കുന്നതിന്** context ആയി കടത്തി നൽകും. താഴെയുള്ള വിഭാഗങ്ങൾ ഓരോ ഘട്ടവും യാഥാർത്ഥ്യ കോഡും ഡയഗ്രാമുകളുമായി പരിശോധിക്കുന്നു. ആദ്യഘട്ടം നോക്കാം.

### Document Processing

[DocumentService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/DocumentService.java)

നിങ്ങൾ ഒരു ഡോക്യുമെന്റ് അപ്‌ലോഡ് ചെയുമ്പോൾ, സിസ്റ്റം അതിനെ (PDF അല്ലെങ്കിൽ സാദാ ടെക്സ്റ്റ് ഫയൽ) പരിഭാഷ ചെയ്ത്, ഫയൽനാമം പോലുള്ള മെറ്റഡേറ്റ അടയ്ക്കുകയും പിന്നീട് ചെറു ഭാഗങ്ങളായി വിഭജിക്കുകയും ചെയ്യും — മോഡലിന്റെ context വിൻഡോയിൽ മനസ്സിലാകുന്ന ചെറുതായി. ഈ ഭാഗങ്ങൾ തമ്മിൽ അല്പം ഓവർലാപ്പ് ഉള്ളതായി കണക്കാക്കുന്നു, അതിനാൽ പരിധികളിൽ പ്രധാനപ്പെട്ട context നഷ്ടപ്പെടുകയില്ല.

```java
// അപ്‌ലോഡ് ചെയ്‌ത ഫയൽ പാഴ്സുചെയ്യുകയും LangChain4j ഡോക്യുമെന്റായി മടക്കുകയും ചെയ്യുക
Document document = Document.from(content, metadata);

// 30-ടോക്കൺ ഓവർലാപുമായി 300-ടോക്കൺ ചങ്കുകളായി വിഭജിക്കുക
DocumentSplitter splitter = DocumentSplitters
    .recursive(300, 30);

List<TextSegment> segments = splitter.split(document);
```

ചുവടെയുള്ള ഡയഗ്രാം ഇത് ദൃശ്യമായി കാണിക്കുന്നു. ഓരോ ഭാഗവും സമീപവാസികളുമായ 30-ടോക്കൺ ഓവർലാപ്പ് പങ്കിടുന്നത് ശ്രദ്ധിക്കുക — ഇതു ensures context പരിധികളിൽ നഷ്ടപ്പെടാതിരിക്കുന്നു:

<img src="../../../translated_images/ml/document-chunking.a5df1dd1383431ed.webp" alt="Document Chunking" width="800"/>

*ഈ ഡയഗ്രാം 30-ടോക്കൺ ഓവർലാപ്പുള്ള 300-ടോക്കൺ ചങ്കുകളായി ഒരു ഡോക്യുമെന്റ് വിഭജിക്കുന്നത് കാണിക്കുന്നു, context പരിധികളിൽ സംരക്ഷിക്കുന്നു.*

> - "വിവിധ ഡോക്യുമെന്റ് തരംകൾക്കായി ഏറ്റവും അനുയോജ്യമായ ചങ്ക് വലിപ്പം എത്രയാണ്, എന്തുകൊണ്ടാണ്?"
> - "വിവിധ ഭാഷകളിലുള്ള ഡോക്യുമെന്റുകളും പ്രത്യേക ഫോർമാറ്റിംഗ് ഉള്ള ഡോക്യുമെന്റുകളും എങ്ങനെ കൈകാര്യം ചെയ്യണം?"

### Creating Embeddings

[LangChainRagConfig.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/config/LangChainRagConfig.java)

ഓരോ ചങ്കും ഒരു സംഖ്യാത്മക പ്രതിനിധാനമായ എംബെഡിംഗ് ആയി മാറുന്നു — സാരാംശം-ടു-നമ്പറ്സ് മാറ്റത്തിനൊപ്പം. എംബെഡിങ് മോഡൽ ഒരു ചാറ്റ് മോഡൽ പോലെ "ബുദ്ധിമുട്ടുള്ള" അല്ല; ഇത് നിർദ്ദേശങ്ങൾ പിന്തുടരാനോ, തർക്കിക്കാൻ അല്ലെങ്കിൽ ചോദ്യം ഉത്തരം നൽകാനോ കഴിയുന്നില്ല. എങ്കിലും, പദങ്ങൾ സമാനമായ അർത്ഥങ്ങൾ വരുന്ന സ്ഥലം ഗണിതപരമായി മാപ്പ് ചെയ്യുന്നു — "കാർ" എന്നത് "ഓട്ടോമൊബൈൽ" കാണുന്നതുപോലെ, "റിഫണ്ട് നയം" "എന്റെ പണം തിരിച്ചിടുക" എന്നതിനു സമീപം. ഒരു ചാറ്റ് മോഡൽ നിങ്ങൾ സംസാരിക്കാൻ കഴിയുന്ന ഒരു വ്യക്തി പോലെ, ഒരു ഇംബെഡിങ് മോഡൽ ഉത്തമമായ ഒരു ഫയൽ ചെയ്യാനുള്ള സംവിധാനം പോലെയാണ്.

<img src="../../../translated_images/ml/embedding-model-concept.90760790c336a705.webp" alt="Embedding Model Concept" width="800"/>

*ഈ ഡയഗ്രാം ഒരു എംബെഡിങ് മോഡൽ പദങ്ങൾ സംഖ്യാത്മക വെക്ടറുകളായി മാറിക്കുന്നതും സമാനമായ അർത്ഥങ്ങൾ vector space-ലിൽ അടുത്തായി നിലനിൽക്കുന്നതും കാണിക്കുന്നു.*

```java
@Bean
public EmbeddingModel embeddingModel() {
    return OpenAiOfficialEmbeddingModel.builder()
        .baseUrl(azureOpenAiEndpoint)
        .apiKey(azureOpenAiKey)
        .modelName(azureEmbeddingDeploymentName)
        .build();
}

EmbeddingStore<TextSegment> embeddingStore = 
    new InMemoryEmbeddingStore<>();
```

താഴെയുള്ള ക്ലാസ് ഡയഗ്രാം RAG പൈപ്പ്‌ലൈന്റെ രണ്ട് സ്വതന്ത്ര ഫ്ലോകളും അവ നടപ്പാക്കുന്ന LangChain4j ക്ലാസ്സുകളും കാണിക്കുന്നു. **ഇൻജെക്ഷൻ ഫ്ലോ** (ഒരു പ്രാവശ്യം അപ്‌ലോഡ് സമയത്ത്) ഡോക്യുമെന്റ് വിഭജിച്ച്, ചങ്കുകൾ എംബെഡ് ചെയ്ത് `.addAll()` വഴിയുള്ള സംഭരണം നടത്തുന്നു. **ക്വറി ഫ്ലോ** (ഓരോ പ്രാവശ്യം ഉപയോക്താവ് ചോദിച്ചുകൊണ്ടിരിക്കുമ്പോൾ) ചോദ്യവുമായി ബന്ധപ്പെട്ട എംബെഡിങ് നിർവ്വഹിക്കുകയും `.search()` വഴി സ്റ്റോർ തിരയുകയും ചാറ്റ് മോഡലിലേക്ക് പൊരുത്തപ്പെട്ട context നൽകുകയും ചെയ്യുന്നു. രണ്ടു ഫ്ലോകളും ഒരു പൊതുവായ `EmbeddingStore<TextSegment>` ഇന്റർഫേസ് വഴി ബന്ധിക്കുന്നു:

<img src="../../../translated_images/ml/rag-langchain4j-classes.bbf3aa9077ab443d.webp" alt="LangChain4j RAG Classes" width="800"/>

*ഈ ഡയഗ്രാം RAG പൈപ്പ്ലൈനിലെ രണ്ട് ഫ്ലോകൾ - ഇൻജെക്ഷനും ക്വറിയും - കാണിക്കുകയും അത് EmbeddingStore വഴി എങ്ങനെ ബന്ധപ്പെടുന്നതുമാണ്.*

എംബെഡിങുകൾ സംഭരിച്ചതിന് ശേഷം, സമാന ഉള്ളടക്കം സ്വാഭാവികമായി വീക്ഷണ സ്ഥലത്ത് കൂട്ടമായി കിടക്കുന്നു. താഴെ കാണിക്കുന്ന ദൃശ്യവൽക്കരണം ബന്ധപ്പെട്ട വിഷയങ്ങൾ അടങ്ങിയ ഡോക്യുമെന്റുകൾ അടുത്തുള്ള പോയിന്റുകളായി മാറുന്നതായി കാണിക്കുന്നു, ഇത് സെമാന്റിക് സേർഷിന്റെ സാദ്ധ്യത നൽകുന്നു:

<img src="../../../translated_images/ml/vector-embeddings.2ef7bdddac79a327.webp" alt="Vector Embeddings Space" width="800"/>

*ഈ ദൃശ്യവൽക്കരണം ബന്ധപ്പെട്ട ഡോക്യുമെന്റുകൾ 3D വെക്ടർ സ്‌പെയ്സിൽ കൂട്ടമായി നീട്ടുന്നു, ടെക്‌നിക്കൽ ഡോക്സ്, ബിസിനസ് നിയമങ്ങൾ, ഫ്രീക്വന്റലി ആസ്ക്ഡ് ക്വഷൻസ് തുടങ്ങിയ പ്രത്യേക ഗ്രൂപ്പുകൾ രൂപീകരിക്കുന്നു.*

ഉപയോക്താവ് തിരയുമ്പോൾ, സിസ്റ്റം നാല് ഘട്ടങ്ങൾ പിന്തുടരുന്നു: ഡോക്യുമെന്റുകൾ ഒരുമിച്ച് എംബെഡ് ചെയ്യുക, ഓരോ തിരയലിലും ക്വറി എംബെഡ് ചെയ്യുക, കോസൈൻ സമാനത ഉപയോഗിച്ച് എല്ലാ സംഭരിച്ച വെക്ടറുകളുമായും താരതമ്യം ചെയ്യുക, ഉന്നത സ്കോർ ലഭിച്ച Top-K ചങ്കുകൾ തിരിച്ചുകിട്ടിക്കുക. ചുവടെയുള്ള ഡയഗ്രാം ഓരോ ഘട്ടവും LangChain4j ക്ലാസ്സുകളുമായി വിശദീകരിക്കുന്നു:

<img src="../../../translated_images/ml/embedding-search-steps.f54c907b3c5b4332.webp" alt="Embedding Search Steps" width="800"/>

*ഈ ഡയഗ്രാം നാല് ഘട്ടമെമ്പെഡിങ് സേർഷ് പ്രോസസ്സ് കാണിക്കുന്നു: ഡോക്യുമെന്റുകൾ എംബെഡ് ചെയ്യുക, ക്വറി എംബെഡ് ചെയ്യുക, കോസൈൻ സമാനതയോടെ വെക്ടറുകൾ താരതമ്യം ചെയ്യുക, Top-K ഫലങ്ങൾ നൽകുക.*

### Semantic Search

[RagService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/RagService.java)

നിങ്ങൾ ഒരു ചോദ്യം ചോദിക്കുമ്പോൾ, നിങ്ങളുടെ ചോദ്യം కూడా ഒരു എംബെഡിങ്ങായി മാറുന്നു. സിസ്റ്റം നിങ്ങളുടെ ചോദ്യം എംബെഡിങ് എല്ലാ ഡോക്യുമെന്റ് ചങ്കുകളുടെ എംബെഡിങുമായി താരതമ്യം ചെയ്യുന്നു. ഇത് ഏറ്റവും ഒത്ത അർത്ഥമുള്ള ചങ്കുകൾ കണ്ടെത്തുന്നു — കുറവ് മാത്രം കീവേഡുകളെ പൊരുത്തപ്പെടുത്തുന്നതല്ല, യാഥാർത്ഥ്യത്തിൽ സാമാന്യ അർത്ഥം വിലയിരുത്തുന്നതുമാണ്.

```java
Embedding queryEmbedding = embeddingModel.embed(question).content();

EmbeddingSearchRequest searchRequest = EmbeddingSearchRequest.builder()
    .queryEmbedding(queryEmbedding)
    .maxResults(5)
    .minScore(0.5)
    .build();

EmbeddingSearchResult<TextSegment> searchResult = embeddingStore.search(searchRequest);
List<EmbeddingMatch<TextSegment>> matches = searchResult.matches();

for (EmbeddingMatch<TextSegment> match : matches) {
    String relevantText = match.embedded().text();
    double score = match.score();
}
```

താഴെയുള്ള ഡയഗ്രാം സെമാന്റിക് സേർഷ് പരമ്പരാഗത കീവേഡ് സേർഷുമായി താരതമ്യം ചെയ്യുന്നു. "വാഹനം" എന്ന കീവേഡ് സേർഷ് "കാറുകളും ട്രക്കുകളും" എന്ന വിഷയത്തെ ഒഴിവാക്കും, പക്ഷേ സെമാന്റിക് സേർഷ് അവയ്ക്ക് ഒരേ അർത്ഥമുണ്ടെന്ന് മനസിലാക്കി അത് ഉയർന്ന സ്കോർഡുള്ള ഫലമായി നൽകുന്നു:

<img src="../../../translated_images/ml/semantic-search.6b790f21c86b849d.webp" alt="Semantic Search" width="800"/>

*ഈ ഡയഗ്രാം കീവേഡ് അടിസ്ഥാനത്തിലുള്ള തിരയലും സെമാന്റിക് തിരയലും തമ്മിലുള്ള വ്യത്യാസം കാണിക്കുന്നു, സെമാന്റിക് സേർഷ് അൽപ്പം വ്യത്യസ്തമായ പദങ്ങൾ ഉപയോഗിച്ചിരുന്നാലും ആശയ ബന്ധപ്പെട്ട ഉള്ളടക്കം തിരിച്ചറിഞ്ഞ് നൽകുന്നു.*

പ്രതി അന്തർീക്ഷത്തിൽ, സാമ്യമത്രയും കോസൈൻ സമാനത ഉപയോഗിച്ച് അളക്കുന്നു — അടിസ്ഥാനത്തിൽ "ഈ രണ്ട് അമ്പുകൾ ഒരേ ദിശയിൽ കാണിക്കുന്നുണ്ടോ?" എന്ന ചോദ്യം അല്പം വ്യത്യസ്തമാണ്. രണ്ട് ചങ്കുകൾ പൂർണ്ണമായി വ്യത്യസ്തമായ വാക്കുകൾ ഉപയോഗിച്ചാലും, അവ ഒരേ അർത്ഥമുണ്ടെങ്കിൽ അവരുടെ മൊത്തം വെക്ടറുകൾ ഒരേ ദിശയിലേക്ക് സൂചിപ്പിക്കുകയും 1.0-ക്കടുത്ത് സ്കോർ ചെയ്യുകയും ചെയ്യും:

<img src="../../../translated_images/ml/cosine-similarity.9baeaf3fc3336abb.webp" alt="Cosine Similarity" width="800"/>
*ഈ രേഖയിൽ കോസൈൻ സാദൃശ്യത്തെ embedding വെക്റ്ററുകളിലെ മИАലമായ കോണായി കാണിക്കുന്നു — കൂടുതൽ പൊരുത്തമുള്ള വെക്റ്ററുകൾ 1.0-നോട് അടുക്കെ സ്കോർ ചെയ്യുന്നു, ഇത് ഉയർന്ന വിവക്ഷാമാനാർത്ഥകതയെ സൂചിപ്പിക്കുന്നു.*

> - "എന്താണ് ഉപയോഗിക്കേണ്ട സാദൃശ്യ പരിധി, അത് ഫലങ്ങളെ എങ്ങനെ ബാധിക്കുന്നു?"

### ഉത്തരമുണ്ടാക്കൽ

[RagService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/RagService.java)

ഏറ്റവും പ്രസക്തമായ ചങ്കുകൾ വ്യക്തമായ നിർദ്ദേശങ്ങൾ, തിരികെ കണ്ടെത്തിയ പാരിസ്ഥിതികം, ഉപയോക്താവിന്റെ ചോദ്യം എന്നിവ ഉൾകൊള്ളുന്ന ഘടിത പ്രോംപ്റ്റിലേക്ക് ഒന്നിപ്പിക്കുന്നു. മോഡൽ ആ പ്രത്യേക ചങ്കുകൾ വായിച്ച് അവയുടെ അടിസ്ഥാനത്തിൽ ഉത്തരം നൽകുന്നു — അത് മുമ്പിലുള്ള വിവരങ്ങൾ മാത്രമേ ഉപയോഗിക്കുവാൻ കഴിയൂ, ഇത് ഹള്ളുസിനേഷനെ തടയുന്നു.

```java
String context = matches.stream()
    .map(match -> match.embedded().text())
    .collect(Collectors.joining("\n\n"));

String prompt = String.format("""
    Answer the question based on the following context.
    If the answer cannot be found in the context, say so.

    Context:
    %s

    Question: %s

    Answer:""", context, request.question());

String answer = chatModel.chat(prompt);
```

താഴെയുള്ള രേഖാചിത്രം ഈ ഘടന പ്രവർത്തനത്തിൽ കാണിക്കുന്നു — തിരച്ചിൽ ഘട്ടത്തിൽ ഉയർന്ന സ്കോർ നേടിയ ചങ്കുകൾ പ്രോംപ്റ്റ് ടെംപ്ലേറ്റിൽ ചേർത്ത്, `OpenAiOfficialChatModel` ഒരു അടിസ്ഥാനപ്പെട്ട ഉത്തരം സൃഷ്ടിക്കുന്നു:

<img src="../../../translated_images/ml/context-assembly.7e6dd60c31f95978.webp" alt="Context Assembly" width="800"/>

*ഈ രേഖാചിത്രം കാണിക്കുന്നത് എങ്ങനെ ഉയർന്ന സ്കോർ നേടിയ ചങ്കുകൾ ഘടിത പ്രോംപ്റ്റായി ചേർത്ത്, മോഡലിന് നിങ്ങളുടെ ഡേറ്റയിൽ നിന്ന് അടിസ്ഥാനപ്പെട്ട ഉത്തരമുണ്ടാക്കാൻ സഹായിക്കുന്നു എന്നതാണ്.*

### ഒരു ഡോക്യുമെന്റ് അപ്‌ലോഡ് ചെയ്യുക

ഡോക്യുമെന്റ് അപ്‌ലോഡ് ചെയ്ത് ആരംഭിക്കുക - ടെസ്റ്റ് ചെയ്യാൻ TXT ഫയലുകൾ മികച്ചതാണ്. ഈ ഡയറക്ടറിയിൽ `sample-document.txt` ആണ് നൽകിയിരിക്കുന്നത്, LangChain4j ഫീച്ചറുകൾ, RAG നടപ്പാക്കൽ, മികച്ച ചടുലങ്ങൾ എന്നിവയെക്കുറിച്ചുള്ള വിവരങ്ങൾ അടങ്ങിയതാണ് — സിസ്റ്റം പരിശോധിക്കാൻ ഏറ്റവും ഉചിതം.

സിസ്റ്റം നിങ്ങളുടെ ഡോക്യുമെന്റ് പ്രോസസ് ചെയ്ത്, ചങ്കുകളായി വിഭജിച്ച്, ഓരോ ചങ്കിനും എംബെഡിംഗുകൾ നിർമ്മിക്കുന്നു. ഇത് സ്വയമേവ അപ്‌ലോഡ് ചെയ്യുമ്പോൾ സംഭവിക്കുന്നു.

### ചോദ്യങ്ങൾ ചോദിക്കുക

ഇപ്പോൾ ഡോക്യുമെന്റ് ഉള്ളടക്കത്തെക്കുറിച്ച് പ്രത്യേക ചോദ്യങ്ങൾ ചോദിക്കുക. ഡോക്യുമെന്റിൽ വ്യക്തമായി പറയപ്പെട്ട യാഥാർത്ഥ്യങ്ങളെ അടയാളപ്പെടുത്തുക. സിസ്റ്റം പ്രസക്തമായ ചങ്കുകൾ തിരയുകയും അവ പ്രോംപ്റ്റിൽ ഉൾപ്പെടുത്തിയ വായിച്ച് ഉത്തരം സൃഷ്ടിക്കുകയും ചെയ്യും.

### സ്രോതസ്സ് റഫറൻസുകൾ പരിശോധിക്കുക

ഓരോ ഉത്തരത്തിനും പ്രസക്തി സ്കോറുകളോട് കൂടിയ സ്രോതസ്സ് റഫറൻസുകൾ ഉൾക്കൊള്ളുന്നു. ഈ സ്കോറുകൾ (0 മുതൽ 1 വരെ) ഓരോ ചങ്കും നിങ്ങളുടെ ചോദ്യത്തിന് എത്രത്തോളം അനുയോജ്യമായിരുന്നോ കാണിക്കുന്നു. ഉയർന്ന സ്കോർ മികച്ച പൊരുത്തം സൂചിപ്പിക്കുന്നു. ഇതിലൂടെ നിങ്ങൾ ഉത്തരം സ്രോതസ്സ് ദ്രശ്യവസ്തുക്കൾക്കെതിരെ പരിശോധിക്കാം.

<a href="images/rag-query-results.png"><img src="../../../translated_images/ml/rag-query-results.6d69fcec5397f355.webp" alt="RAG Query Results" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*ഈ സ്ക്രീൻഷോട്ട് ക്വറി ഫലങ്ങൾ കാണിക്കുന്നു, ജെനറേറ്റഡ് ഉത്തരം, സ്രോതസ്സ് റഫറൻസുകൾ, ഓരോ തിരികെ കണ്ടെത്തിയ ചങ്കിന്റെയും പ്രസക്തി സ്കോറുകൾ ഉൾപ്പെടുത്തിയാണ്.*

### ചോദ്യങ്ങളുമായി പരീക്ഷണം ചെയ്യുക

വ്യത്യസ്ത തരം ചോദ്യങ്ങൾ പരീക്ഷിക്കൂ:
- പ്രത്യേക യാഥാർത്ഥ്യങ്ങൾ: "മുഖ്യ വിഷയം എന്ത്?"
- താരതമ്യം: "X ഉം Y ഉം തമ്മിലുള്ള വ്യത്യാസം എന്താണ്?"
- സാരാംശം: "Z-യെക്കുറിച്ച് പ്രധാന ബിന്ദുക്കൾ സാരാംശമാക്കൂ"

നിങ്ങളുടെ ചോദ്യവും ഡോക്യുമെന്റ് ഉള്ളടക്കവും എങ്ങനെ പൊരുത്തപ്പെടുന്നുവെന്ന് ആശ്രയിച്ച് പ്രസക്തി സ്കോറുകൾ എങ്ങനെ മാറുന്നു എന്ന് ശ്രദ്ധിക്കുക.

## പ്രധാന ആശയങ്ങൾ

### ചങ്ക് කිරීමയുടെ രീതി

ഡോക്യുമെന്റുകൾ 300 ടോക്കൺ ചങ്കുകളായി വിഭജിച്ച് 30 ടോക്കൺ ഒവർലാപ്പ് ഉണ്ട്. ഈ സമതുലനം ഓരോ ചങ്കും പ്രസക്തമായ സാങ്കേതിക പശ്ചാത്തലത്തോടെ സൌകര്യമുള്ളതാക്കുകയും, ഒരു പ്രോംപ്റ്റിൽ പല ചങ്കുകളും ഉൾക്കൊള്ളാൻ കഴിയുന്ന തരത്തിലാക്കുകയും ചെയ്യുന്നു.

### സാദൃശ്യ സ്കോറുകൾ

തിരികെ കണ്ടെത്തിയ ഓരോ ചങ്കിനും ഉപയോക്താവിന്റെ ചോദ്യത്തിനോട് എത്രത്തോളം പണം പൊരുത്തമാണ് എന്നതിന് 0 മുതൽ 1 വരെ ഒരു സാദൃശ്യ സ്കോർ ഉണ്ടാകും. താഴെയുള്ള രേഖാചിത്രം സ്കോർ പരിധികളും സിസ്റ്റം ഫലങ്ങളിൽ അവയെ എങ്ങനെ ഉപയോഗിക്കുന്നു എന്നതും കാണിക്കുന്നു:

<img src="../../../translated_images/ml/similarity-scores.b0716aa911abf7f0.webp" alt="Similarity Scores" width="800"/>

*ഈ രേഖാചിത്രം 0 മുതൽ 1 വരെ സ്കോർ പരിധികൾ കാണിക്കുന്നു, 0.5 എന്ന കുറഞ്ഞ പരിധി പ്രസക്തമല്ലാത്ത ചങ്കുകൾ ഒഴിവാക്കാൻ ഉപയോഗിക്കുന്നു.*

സ്കോറുകൾ പരിധി:
- 0.7-1.0: അത്യന്തം പ്രസക്തം, കൃത്യമായ പൊരുത്തം
- 0.5-0.7: പ്രസക്തം, നല്ല సందర్భം
- 0.5-ന് താഴെ: ഒഴിവാക്കിയിരിക്കുന്നു, വളരെ വ്യത്യാസമുള്ളത്

താഴെയുള്ള പരിധിയ്ക്ക് മുകളിൽ മാത്രമേ സിസ്റ്റം ചങ്കുകൾ തിരികെ കൊണ്ടുവരൂ — ഇത് ഗുണമേന്മ ഉറപ്പാക്കുന്നു.

എംബെഡിംഗുകൾ അർത്ഥ ഗ്രൂപ്പുകൾ ശുദ്ധമായി വേർതിരിക്കുന്നത് ഞാൻ ചെയ്യുമ്പോൾ നല്ലതാണ്, പക്ഷേ അവക്ക് ബലയുമുണ്ട്. താഴെയുള്ള രേഖാചിത്രം സാധാരണ പരാജയ മോഡുകൾ കാണിക്കുന്നു — വളരെ വലിയ ചങ്കുകൾ മങ്ങിയ വെക്റ്ററുകൾ ഉണ്ടാക്കുന്നു, ചെറുതായ ചങ്കുകൾക്ക് പശ്ചാത്തലം കുറവാണ്, പ്രസക്തമല്ലാത്ത പദങ്ങൾ പല ഗ്രൂപ്പുകളിലേക്കും സൂചിപ്പിക്കുന്നു, കൃത്യമായ പൊരുത്തമുള്ള ലുക്കപ്പുകൾ (ഐഡികൾ, ഭാഗ നമ്പറുകൾ) എംബെഡിംഗുകൾ വർഗ്ഗത്തിൽ പ്രവർത്തിക്കാറില്ല:

<img src="../../../translated_images/ml/embedding-failure-modes.b2bcb901d8970fc0.webp" alt="Embedding Failure Modes" width="800"/>

*ഈ രേഖാചിത്രം സാധാരണ এম്ബെഡിങ്ങ് പരാജയ മോഡുകൾ കാണിക്കുന്നു: വളരെ വലിയ, വളരെ ചെറിയ ചങ്കുകൾ, ബഹുദൂത പദങ്ങൾ വിവിധ ഗ്രൂപ്പുകളിലേക്കുള്ള സൂചനകൾ, കൃത്യമായ പൊരുത്തമുള്ള ഐഡികൾ ലുക്കപ്പുകൾ തുടങ്ങിയവ.*

### ഇൻ-മെമ്മറി സ്റ്റോറേജ്

ഈ മൊഡ്യൂൾ സൌകര്യത്തിനായി ഇൻ-മെമ്മറി സ്റ്റോറേജ് ഉപയോഗിക്കുന്നു. അപ്ലിക്കേഷൻ പുനരാരംഭിക്കുമ്പോൾ, അപ്‌ലോഡ് ചെയ്ത രേഖകൾ നഷ്ടപ്പെടും. പ്രൊഡക്ഷൻ സിസ്റ്റങ്ങൾ Qdrant അല്ലെങ്കിൽ Azure AI Search പോലുള്ള സ്ഥിരമായ വെക്റ്റർ ഡാറ്റാബേസുകൾ ഉപയോഗിക്കുന്നു.

### കോൺടെക്‌സ്‌റ്റ് വിൻഡോ മാനേജ്‌മെന്റ്

ഓരോ മോഡലിനും പരമാവധി കുറവ് കോൺടെക്‌സ്‌റ്റ് വിൻഡോ ഉണ്ട്. വലിയ ഡോക്യുമെന്റിലെ എല്ലാ ചങ്കുകളും ഉൾപ്പെടുത്താൻ കഴിയില്ല. സിസ്റ്റം പരിമിതികൾ പാലിക്കാൻ പരമാവധി പ്രസക്തി ഉള്ള ടോപ് N (ഡീഫോൾട്ട് 5) ചങ്കുകൾ തിരികെ കൊണ്ടുവരുന്നു, നന്നായി ഉച്ചവെക്കുന്ന ഉത്തരങ്ങൾ നൽകാൻ മതിയായ പശ്ചാത്തലം ഉണ്ട്.

## RAG പ്രാധാന്യം

RAG എന്നും അനുയോജ്യമായ സമീപനമല്ല. താഴെയുള്ള തീരുമാന മാർഗ്ഗദർശി RAGയുടെ മൂല്യം ഉൾപ്പെടുത്തുമ്പോഴും എപ്പോൾ ലളിതമായ സമീപനങ്ങൾ — പ്രോംപ്റ്റിൽ നേരിട്ടുള്ള ഉള്ളടക്കം ചേർക്കൽ അല്ലെങ്കിൽ മോഡലിന്റെ സ്വന്തം അറിവിൽ ആശ്രയിക്കൽ — മതിയാകും എന്നും കാണിക്കുന്നു:

<img src="../../../translated_images/ml/when-to-use-rag.1016223f6fea26bc.webp" alt="When to Use RAG" width="800"/>

*ഈ രേഖാചിത്രം എപ്പോൾ RAG മൂല്യമേടും, എപ്പോൾ ലളിതമായ സമീപനങ്ങൾ മതിയാകും എന്നതിന്റെ തീരുമാന മാർഗ്ഗദർശി ആണ്.*

**RAG ഉപയോഗിക്കുക:**
- സ്വകാര്യ രേഖകളെക്കുറിച്ച് ചോദ്യങ്ങൾക്ക് ഉത്തരം പറയുമ്പോൾ
- വിവരങ്ങൾ അധികമായി മാറുമ്പോൾ (നയങ്ങൾ, വിലകൾ, സവിശേഷതകൾ)
- കൃത്യതയ്ക്ക് സ്രോതസ്സ് സൂചനം ആവശ്യമായപ്പോൾ
- ഉള്ളടക്കം ഒരിടത്തേക്ക് ഒത്തു ചേരാനാവാതെ വളരെയധികം വലിയതായിരിക്കുമ്പോൾ
- തെളിയിക്കാവുന്ന, അടിസ്ഥാനപ്പെട്ട മറുപടികൾ ആവശ്യമുള്ളപ്പോൾ

**RAG ഉപയോഗിക്കരുത്:**
- മോഡലിന് മുൻകൂട്ടി പ്രാപ്തിയുള്ള പൊതുവായ അറിവ് ആവശ്യമായ ചോദ്യങ്ങൾക്കായി
- റിയൽ-ടൈം ഡാറ്റ ആവശ്യമാണ് (RAG അപ്‌ലോഡ് ചെയ്ത രേഖകളിൽ പ്രവർത്തിക്കുന്നു)
- ഉള്ളടക്കം നേരിട്ട് പ്രോംപ്റ്റിൽ ഉൾക്കൊള്ളാത്തതിനൊരേറെ ചെറിയതാണ്
