## നിങ്ങൾ പഠിക്കാനുള്ളത്

ഇപ്പോൾവരെ, നിങ്ങൾ എങ്ങനെ AI-യുമായി സംവദിക്കാം, പ്രോമ്പ്റ്റുകൾ ഫലപ്രദമായി ഘടിപ്പിക്കാം, നിങ്ങളുടെ ഡോക്യുമെന്റുകളിൽ മറുപടികളെ അടിസ്ഥാനമാക്കി നിൽക്കാം എന്നോപരം പഠിച്ചിട്ടുണ്ട്. എന്നാൽ ഒരു അടിസ്ഥാനപരമായ പരിമിതിയുണ്ട്: ഭാഷാ മാതൃകകൾ ടെക്സ്റ്റ് മാത്രമേ സൃഷ്ടിക്കുകയുള്ളു. അവ കാലാവസ്ഥ പരിശോധിക്കാൻ, കണക്കുകൾ നിർവ്വഹിക്കാൻ, ഡാറ്റാബേസുകൾ ക്വറി ചെയ്യാൻ അല്ലെങ്കിൽ ബാഹ്യ സിസ്റ്റങ്ങളുമായി ഇടപഴകാൻ കഴിയില്ല.

ടൂളുകൾ ഇതിനെ മാറ്റുന്നു. മാതൃകയ്ക്ക് കോളർ ചെയ്യാവുന്ന ഫംഗ്ഷനുകളിലേക്കുള്ള ആക്സസ് നൽകുമ്പോൾ, അത് ഒരു ടെക്സ്റ്റ് ജനറേറ്ററിൽ നിന്ന് പ്രവർത്തനങ്ങൾ നടത്താവുന്ന ഏജന്റായി മാറുന്നു. മാതൃകയ്ക്ക് ടൂൾ എപ്പോൾ വേണമെന്നും, ഏത് ടൂൾ ഉപയോഗിക്കണമെന്നും, എന്ത് പാരാമീറ്ററുകൾ പാസ്സ് ചെയ്യണമെന്നും തീരുമാനിക്കുന്നു. നിങ്ങളുടെ കോഡ് ഫംഗ്ഷൻ നിർവ്വഹിച്ച് ഫലം തിരിച്ചുകൽക്കുകയും, മാതൃക അതു തന്റെ മറുപടിയിലേക്ക് ഉൾപ്പെടുത്തുകയും ചെയ്യുന്നു.

## ടൂളുകളുള്ള AI ഏജൻറികളെ മനസ്സിലാക്കുക

> **📝 കുറിപ്പ്:** ഈ മോഡ്യൂളിലുള്ള “ഏജൻറുകൾ” എന്ന പദം ടൂൾ-കോൾ ചെയ്യൽ കഴിവുള്ള AI അസിസ്റ്റന്റുകളെ സൂചിപ്പിക്കുന്നു. ഇത് നാം [മോഡ്യൂൾ 05: MCP](../05-mcp/README.md)യിൽ പരാമർശിക്കുന്ന **Agentic AI** മാതൃകകൾ (സ്വയം നിർദ്ദേശം, ഓർമ്മ, ബഹു-ഘട്ട തർക്കനിർവഹണം എന്നിവയുള്ള സ്വതന്ത്ര ഏജൻറുകൾ) നിന്നു വ്യത്യസ്തമാണ്.

ടൂളുകളില്ലാതെ, ഒരു ഭാഷാ മാതൃക തന്റെ പരിശീലന ഡാറ്റയിൽ നിന്നുള്ള ടെക്സ്റ്റ് മാത്രമേ സൃഷ്ടിക്കാതെ കഴിയൂ. പ്രస్తుత കാലാവസ്ഥ അറിയാൻ ചോദിച്ചാൽ അത് ഗ്യസ് ചെയ്യണം. അതിന് ടൂളുകൾ നൽകിയാൽ, അത് കാലാവസ്ഥ APIനെ വിളിക്കാൻ, കണക്കുകൾ ചെയ്യാൻ അല്ലെങ്കിൽ ഡാറ്റാബേസ് ക്വറി ചെയ്യാൻ കഴിയും — പിന്നെ ആ യാഥാർഥ്യ ഫലങ്ങൾ മറുപടിയിലേക്ക് ചേർക്കുന്നു.

<img src="../../../translated_images/ml/what-are-tools.724e468fc4de64da.webp" alt="ടൂളുകളില്ലാത്തത് vs ടൂളുകളുള്ളത്" width="800"/>

*ടൂളുകളില്ലാതെ മാതൃക ഗ്യസ് ചെയ്യുന്നത് മാത്രമാണ് ചെയ്യുന്നത് — ടൂളുകൾ ഉപയോഗിച്ചാൽ APIകൾ വിളിക്കാൻ, കണക്കുകൾ നടത്താൻ, റിയൽ-ടൈം ഡാറ്റ വാപസ്സ് നൽകാൻ കഴിയും.*

ടൂളുകളുള്ള AI ഏജൻറ് **താര്ക്കികവും പ്രവർത്തനവുമായ (ReAct)** മാതൃക അനുസരിക്കുന്നു. മാതൃക ഞൊടിയിടെ മറുപടി നൽകുന്നതല്ല — അത് വേണ്ടതെന്തെന്ന് അനാലിസ് ചെയ്ത്, ടൂൾ വിളിച്ച് പ്രവർത്തിച്ച്, ഫലം ആസൂത്രണം ചെയ്ത്, പിന്നെ വീണ്ടും പ്രവർത്തിക്കണമോ അന്തിമ മറുപടി നൽകണമോ തീരുമാനിക്കുന്നു:

1. **അവലംബിക്കുക** — ഉപഭോക്താവിന്റെ ചോദ്യത്തെ വിശകലനം ചെയ്ത് ആവശ്യമായ വിവരം നിർദ്ദേശിക്കുക
2. **പ്രവർത്തിക്കുക** — ശരിയായ ടൂൾ തിരഞ്ഞെടുക്കുക, ശരിയായ പാരാമീറ്ററുകൾ നിർമിച്ച് അത് വിളിക്കുക
3. **കാണുക** — ടൂളിന്റെ ഔട്ട്പുട്ട് സ്വീകരിച്ച് ഫലം വിലയിരുത്തുക
4. **പുനരാവൃത്തി അല്ലെങ്കിൽ മറുപടി** — കൂടുതൽ ഡാറ്റ ആവശ്യമെങ്കിൽ വീണ്ടും തുടക്കത്തിലേക്ക് പോകുക; അല്ലെങ്കിൽ സ്വാഭാവിക ഭാഷയിൽ മറുപടി നൽകുക

<img src="../../../translated_images/ml/react-pattern-detail.96a5efeeb6dd2f61.webp" alt="ReAct മാതൃക" width="800"/>

*ReAct ചക്രം — ഏജൻറ് ചെയ്യേണ്ടതു എന്തെന്നു പരിശോധിച്ച് പ്രവർത്തനം ആരംഭിച്ച് ഫലം നിരീക്ഷിച്ച് ആവശ്യമുള്ളത്രവരെ ആവർത്തിച്ച് അന്തിമ മറുപടി നൽകുന്നു.*

ഇത് സ്വയമെനേജ്മെന്റ് ആയി നടക്കുന്നു. നിങ്ങൾ ടൂളുകൾ, അവയുടെ വിവരണങ്ങൾ നിർവചിക്കുന്നു. മാതൃക അവ ഉപയോഗിക്കേണ്ട സമയവും മാർഗവും തീരുമാനിക്കുന്നു.

## ടൂൾ കോളിംഗ് എങ്ങനെ പ്രവർത്തിക്കുന്നു

### ടൂൾ നിർവചനങ്ങൾ

[WeatherTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) | [TemperatureTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/TemperatureTool.java)

നീங்கள் ഫംഗ്ഷനുകൾ വ്യക്തമായ വിവരണത്തോടും പാരാമീറ്റർ പ്രത്യേകതകളോടും നിർവചിക്കുന്നു. മാതൃക ഈ വിവരണങ്ങൾ സിസ്റ്റം പ്രോമ്പ്റ്റിൽ കാണുകയും ഓരോ ടൂൾ എന്ത് ചെയ്യുമെന്നു മനസ്സിലാക്കുകയും ചെയ്യുന്നു.

```java
@Component
public class WeatherTool {
    
    @Tool("Get the current weather for a location")
    public String getCurrentWeather(@P("Location name") String location) {
        // നിങ്ങളുടെ കാലാവസ്ഥ_lookup ലളിതവിവരം
        return "Weather in " + location + ": 22°C, cloudy";
    }
}

@AiService
public interface Assistant {
    String chat(@MemoryId String sessionId, @UserMessage String message);
}

// അസിസ്റ്റന്റ് സ്വയം Spring Boot വഴി കോർത്തിണക്കിയിരിക്കുന്നു:
// - ChatModel ബീൻ
// - @Component ക്ലാസ്സുകളിൽ നിന്നുള്ള എല്ലാ @Tool മേത്തഡുകൾ
// - സെഷൻ മാനേജ്മെന്റിനുള്ള ChatMemoryProvider
```

അടിസ്ഥാനരീതി ചിഹ്നങ്ങൾ എല്ലാം വിശദീകരിച്ചിരിക്കുന്ന ചിത്രവും എങ്ങനെ ഓരോ ഭാഗവും AI-യെ ടൂൾ എപ്പോൾ വിളിക്കണം, എന്ത് പാരാമീറ്ററുകൾ നൽകണം എന്നു മനസ്സിലാക്കാൻ സഹായിക്കുന്നു:

<img src="../../../translated_images/ml/tool-definitions-anatomy.f6468546037cf28b.webp" alt="ടൂൾ നിർവചനങ്ങളുടെ ഘടന" width="800"/>

*ടൂൾ നിർവചനത്തിന്റെ ഘടന — @Tool ടൂൾ എപ്പോഴാണ് ഉപയോഗിക്കേണ്ടതെന്ന് AI-യ്ക്ക് ഉദ്ദേശിക്കുന്നു, @P ഓരോ പാരാമീറ്റർ വിവരിക്കുന്നു, @AiService സ്റ്റാർട്ടപ്പിൽ എല്ലാം ബന്ധിപ്പിക്കുന്നു.*

> **🤖 [GitHub Copilot](https://github.com/features/copilot) ചാറ്റിൻറെ സഹായത്തോടെ പരീക്ഷിക്കാം:** [`WeatherTool.java`](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) തുറന്ന് ചോദിക്കുക:
> - "മോക്ക് ഡേറ്റിനപ്പുറം ഒരു യഥാർത്ഥ വേദറAPI(OpenWeatherMap പോലുള്ളത്) എങ്ങനെ ഇന്റഗ്രേറ്റുചെയ്യാം?"
> - "AI അത് ശരിയായി ഉപയോഗിക്കാൻ സഹായിക്കുന്ന നല്ല ടൂൾ വിവരണം എങ്ങനെ എഴുതണം?"
> - "API പിശകുകളും റേറ്റ് പരിധികളും ടൂൾ ഉപയോഗത്തിൽ എങ്ങനെ കൈകാര്യം ചെയ്യണം?"

### തീരുമാനമെടുക്കൽ

ഉപഭോക്താവ് "സിയാട്ടിളിലെ കാലാവസ്ഥ എന്ത്?" ചോദിക്കുമ്പോൾ, മാതൃക യാതൊരു ടൂൾ കൊണ്ട് യാദൃച്ഛികമായി തിരഞ്ഞെടുക്കുകയില്ല. ഉപയോക്താവിന്റെ ഉദ്ദേശം എല്ലാ ടൂൾ വിവരണങ്ങളോട് താരതമ്യം ചെയ്ത്, പ്രസക്തതാ സ്കോർ നൽകി ഏറ്റവും അനുയോജ്യത്തെ തിരഞ്ഞെടുക്കും. ശേഷം അതിന് ശരിയായ പാരാമീറ്ററുകളുള്ള ഘടനാപരമായ ഫംഗ്ഷൻ കോളിംഗ് സൃഷ്ടിക്കും — ഇവിടെ `location` `"Seattle"` ആക്കി സെറ്റ് ചെയ്യുന്നു.

ഉപഭോക്താവിന്റെ അഭ്യർത്ഥനയോട് പൊരുത്തപ്പെടുന്ന ടൂൾ കണ്ടില്ലെങ്കില്‍, മാതൃക തന്റെ അറിവിൽ നിന്നുള്ള മറുപടി നൽകുന്നു. ഒന്ന് കൂടുതൽ ടൂൾ പൊരുത്തം കാണിച്ചാൽ, ഏറ്റവും വിശദമായതാകും തിരഞ്ഞെടുക്കുന്നത്.

<img src="../../../translated_images/ml/decision-making.409cd562e5cecc49.webp" alt="AI ഒരു ടൂൾ തിരഞ്ഞെടുക്കുന്നത് എങ്ങനെ" width="800"/>

*മാതൃക ഉപയോക്താവിന്റെ ലക്ഷ്യവുമായി എല്ലാ ലഭ്യമായ ടൂളുകളും വിലയിരുത്തി ഏറ്റവും ഉയർന്ന പൊരുത്തം കാണിക്കുന്നതാണ് തിരഞ്ഞെടുക്കുന്നത് — അതുകൊണ്ടുതന്നെ വ്യക്തമായ, പ്രത്യേക ടൂൾ വിവരണങ്ങൾ എഴുതുന്നത് അത്യന്താപേക്ഷിതമാണ്.*

### നിർവഹണം

[AgentService.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/service/AgentService.java)

സ്പ്രിംഗ് ബൂട്ട് പ്രഖ്യാപനാത്മകമായ `@AiService` ഇന്റർഫേസുമായി എല്ലാ രജിസ്റ്റർ ചെയ്ത ടൂളുകളെയും ഓട്ടോ-വയറിങ് ചെയ്യുന്നു, LangChain4j ടൂൾ കോളുകൾ സ്വയം എക്സിക്യൂട്ട് ചെയ്യുന്നു. പിന്നിൽ ഉപഭോക്താവിന്റെ സ്വാഭാവിക ഭാഷയിലുള്ള ചോദ്യത്തിൽ നിന്നും സ്വാഭാവിക ഭാഷയിലെ മറുപടിയിലേക്ക് ആറു ഘട്ടങ്ങൾ വഴി പൂർണ്ണ ടൂൾ കോളിംഗ് പ്രവാഹം നടക്കുന്നു:

<img src="../../../translated_images/ml/tool-calling-flow.8601941b0ca041e6.webp" alt="ടൂൾ കോളിംഗ് പ്രവാഹം" width="800"/>

*ആകെ പ്രവാഹം — ഉപയോക്താവ് ചോദിക്കുന്നു, മാതൃക ഒരു ടൂൾ തിരഞ്ഞെടുക്കുന്നു, LangChain4j അത് എക്സിക്യൂട്ട് ചെയ്യുന്നു, മാതൃക ഫലം പ്രകൃതിഭാഷയിലേക്ക് തയ്യാറാക്കുന്നു.*

> **🤖 [GitHub Copilot](https://github.com/features/copilot) ചാറ്റിൻറെ സഹായത്തോടെ പരീക്ഷിക്കാം:** [`AgentService.java`](../../../04-tools/src/main/java/com/example/langchain4j/agents/service/AgentService.java) തുറന്ന് ചോദിക്കുക:
> - "ReAct പാറ്റേൺ എങ്ങനെ പ്രവർത്തിക്കുന്നു, AI ഏജൻറുകൾക്കു ഇത് എങ്ങനെ ഫലപ്രദമാണ്?"
> - "ഏജൻറ് ഏത് ടൂൾ എപ്പോഴും ഉപയോഗിക്കണമെന്നു എങ്ങനെ തീരുമാനിക്കുന്നു?"
> - "ഒരു ടൂൾ നിർവഹണം പരാജയപ്പെട്ടാൽ എന്ത് സംഭവിക്കും - എററുകൾ എങ്ങനെ ഭദ്രമായി കൈകാര്യം ചെയ്യണം?"

### പ്രതികരണം സൃഷ്ടിക്കൽ

മാതൃക കാലാവസ്ഥ ഡാറ്റ സ്വീകരിച്ച് ഉപഭോക്താവിന് സ്വാഭാവിക ഭാഷാ മറുപടിയായി രൂപപ്പെടുത്തുന്നു.

### ആർക്കിടെക്ചർ: സ്പ്രിംഗ് ബൂട്ട് ഓട്ടോ-വയറിങ്

ഈ മോഡ്യൂളിൽ LangChain4jന്റെ സ്പ്രിംഗ് ബൂട്ട് ഇന്റഗ്രേഷൻ ഉപയോഗിച്ച് പ്രഖ്യാപനാത്മക `@AiService` ഇന്റർഫേസുകൾ ഉപയോക്തൃനിർമിതമാണ്. സ്‌റ്റാർട്ടപ്പിൽ സ്പ്രിംഗ് ബൂട്ട് ഓരോ `@Component`യിലും ഉള്ള `@Tool` മെതഡ്, നിങ്ങളുടെ `ChatModel` ബീൻ, `ChatMemoryProvider` എന്നിവ കണ്ടെത്തുന്നു — പിന്നെ എല്ലാം ഒരു `Assistant` ഇന്റർഫേസിലേക്ക് ബന്ധിപ്പിക്കുന്നു, ബോയിലർപ്ലേറ്റുകൾ ഇല്ലാതെ.

<img src="../../../translated_images/ml/spring-boot-wiring.151321795988b04e.webp" alt="സ്പ്രിംഗ് ബൂട്ട് ഓട്ടോ-വയറിങ് ആർക്കിടെക്ചർ" width="800"/>

*@AiService ഇന്റർഫേസ് ChatModel, ടൂൾ ഘടകങ്ങൾ, മെമ്മറി പ്രൊവൈഡർ എന്നിവ ഒരുമിച്ചു ബന്ധിപ്പിക്കുന്നു — സ്പ്രിംഗ് ബൂട്ട് ഓട്ടോമാറ്റിക് ആയി വൈറിങ് കൈകാര്യം ചെയ്യുന്നു.*

ഈ സമീപനത്തിന് പ്രധാന ഗുണങ്ങൾ:

- **സ്പ്രിംഗ് ബൂട്ട് ഓട്ടോ-വയറിങ്** — ChatModel, ടൂളുകൾ സ്വയം ഇൻജെക്‌ഷൻ ചെയ്യുന്നു
- **@MemoryId പാറ്റേൺ** — സ്വയം സെഷൻ അടിസ്ഥാനമേയിലുള്ള മെമ്മറി മാനേജ്മെന്റ്
- **ഒറ്റ സിംഗിൾ ഇൻസ്റ്റൻസ്** — ഏജൻറ് ഒരിക്കൽ സൃഷ്ടിച്ച് നല്ല പെർഫോമൻസ് ലക്ഷ്യമിട്ട് പുനരുപയോഗിക്കുന്നു
- **ടൈപ്പ്-സേഫ് എക്സിക്യൂഷൻ** — ജാവ മെതഡുകൾ നേരിട്ട് ടൈപ്പ് കൺവെർഷനോടെ വിളിക്കുന്നു
- **പഴുതോടുള്ള ഓർക്കസ്ട്രേഷൻ** — ടൂൾ ചെയിനിംഗ് സ്വയം കൈകാര്യം ചെയ്യുന്നു
- **ബോയിലർപ്ലേറ്റ് ഇല്ലാത്തത്** — കൈകാര്യം ചെയ്യാവുന്ന `AiServices.builder()` വിളികൾ അല്ല

മാറ്റായ സമീപനങ്ങൾ (മാനുവൽ `AiServices.builder()`) കൂടുതൽ കോഡ് ആവശ്യമാണ്, സ്പ്രിംഗ് ബൂട്ട് ഇന്റഗ്രേഷൻ ഗുണങ്ങൾ ലഭിക്കുന്നില്ല.

## ടൂൾ ചെയിനിംഗ്

**ടൂൾ ചെയിനിംഗ്** — ടൂൾ അടിസ്ഥാന ഏജൻറുകളുടെ യഥാർത്ഥ ശക്തി ഒരു ചോദ്യത്തിന് ഒരധികം ടൂളുകൾ ആവശ്യമായപ്പോൾ വ്യക്തമാകും. "സിയാറ്റിളിലെ കാലാവസ്ഥ ഫാരൻഹീട്ടിൽ എന്ത്?" ചോദിക്കുമ്പോൾ, ഏജൻറ് സ്വയം രണ്ട് ടൂളുകൾ കൈകടന്ന് പ്രവർത്തിക്കുന്നു: ആദ്യം `getCurrentWeather` വിളിച്ച് സെൽഷ്യസ് താപനില കിട്ടി, പിന്നീട് അത് `celsiusToFahrenheit` വഴിയായി പരിവർത്തനം ചെയ്യുന്നു — എല്ലാം ഒരു സംഭാഷണ ടേണിൽ.

<img src="../../../translated_images/ml/tool-chaining-example.538203e73d09dd82.webp" alt="ടൂൾ ചെയിനിംഗ് ഉദാഹരണം" width="800"/>

*ടൂൾ ചെയിനിംഗ് പ്രവർത്തനത്തിൽ — ഏജന്റ് ആദ്യം getCurrentWeather വിളിക്കുന്നു, പിന്നെ സെൽഷ്യസ് ഫലവും celsiusToFahrenheit-ലേക്ക് പാസ്സ് ചെയ്ത് സംയുക്ത മറുപടി നൽകുന്നു.*

ഇത് പ്രവർത്തിക്കുന്ന അപ്ലിക്കേഷനിൽ ഞൊഴുക്കുന്നത് ഇങ്ങനെ ആണ് — ഏജൻറ് ഒരു സംഭാഷണ ടേണിൽ രണ്ട് ടൂൾ കോളുകൾ ചെയിൻ ചെയ്ത് നടത്തുന്നു:

<a href="images/tool-chaining.png"><img src="../../../translated_images/ml/tool-chaining.3b25af01967d6f7b.webp" alt="ടൂൾ ചെയിനിംഗ്" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*വാസ്തവ അപ്ലിക്കേഷൻ ഔട്ട്പുട്ട് — ഏജൻറ് സ്വയം getCurrentWeather → celsiusToFahrenheit ടൂളുകളുമായി ഒരു ടേണിൽ ചെയിൻ ചെയ്യുന്നു.*

**ക്രമസമത്വമുള്ള പരാജയങ്ങൾ** — മോക്ക് ഡാറ്റയിലുള്ളില്ലാത്ത നഗരത്തിലെ കാലാവസ്ഥ ചോദിക്കുമ്പോൾ, ടൂൾ പിശക് സന്ദേശം തിരികെ നൽകുന്നു, AI സഹായം നൽകാൻ കഴിയില്ലെന്ന് വിശദമാക്കുന്നു, ക്രാഷ്ജില്ലാതെ. ടൂളുകൾ സുരക്ഷിതമായി പരാജയപ്പെടുന്നു.

<img src="../../../translated_images/ml/error-handling-flow.9a330ffc8ee0475c.webp" alt="തപ്പുകൾ കൈകാര്യം ചെയ്യുന്ന പ്രവാഹം" width="800"/>

*ടൂൾ പരാജയപ്പെട്ടാൽ, ഏജൻറ് പിശക് പിടിച്ച് തകർക്കാതെ സഹായകരമായ വിശദീകരണം നൽകുന്നു.*

ഇത് ഒരു സംഭാഷണ ടേണിൽ നടക്കുന്നു. ഏജൻറ് സ്വതന്ത്രമായി നിരവധി ടൂൾ കോളുകൾ കാര്യക്ഷമമായി സംഘടിപ്പിക്കുന്നു.

സ്പ്രിംഗ് ബൂട്ട് ഡാഷ്ബോർഡിൽ നിങ്ങൾക്ക് സാധിക്കും:
- വർക്ക്സ്പേസിലുള്ള എല്ലാ സ്പ്രിംഗ് ബൂട്ട് അപ്ലിക്കേഷനുകളും കാണുക
- ഒറ്റ ക്ലിക്കിൽ അപ്ലിക്കേഷനുകൾ ആരംഭിക്കാനും നിർത്താനും
- അപ്ലിക്കേഷൻ ലോഗുകൾ റെതുതായി നോക്കുക
- അപ്ലിക്കേഷൻ നില നിരീക്ഷിക്കുക

"tools" എന്നതിന് സമീപമുള്ള പ്ലേ ബട്ടൺ ക്ലിക്കുചെയ്‌തു ഈ മോഡ്യൂൾ സ്റ്റാർട്ട് ചെയ്യൂ, അല്ലെങ്കിൽ മുഴുവൻ മോഡ്യൂളുകളും ഒരുമിച്ച് തുടങ്ങാവുന്നതാണ്.

## അപ്ലിക്കേഷൻ ഉപയോഗിക്കൽ

അപ്ലിക്കേഷൻ ഒരു വെബ് ഇന്റർഫേസ് നൽകുന്നു, ഇവിടെ നിങ്ങൾ കാലാവസ്ഥയും താപനില പരിവർത്തന ഉപകരണങ്ങളും ഉൾപ്പെടുന്ന ഒരു AI ഏജൻറുമായി ഇടപഴകാം.

<a href="images/tools-homepage.png"><img src="../../../translated_images/ml/tools-homepage.4b4cd8b2717f9621.webp" alt="AI ഏജൻറ് ടൂൾസ് ഇന്റർഫേസ്" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*AI ഏജൻറ് ടൂൾസ് ഇന്റർഫേസ് - ടൂൾസ് ഉപയോഗിച്ച് സംവദിക്കുന്നതിനുള്ള വേഗത്തിലുള്ള ഉദാഹരണങ്ങളും ചാറ്റ് ഇന്റർഫേസും*

### സാധാരണ ടൂൾ ഉപയോഗം പരീക്ഷിക്കുക
100 ഡിഗ്രി ഫാരൻഹീറ്റ് സെൽഷ്യസിലേക്ക് മാറ്റുക. ഏജന്റ് താപനില പരിവർത്തന ടൂൾ വേണ്ടതാണെന്ന് തിരിച്ചറിയുന്നു, ശരിയായ പാരാമീറ്ററുകളോടെ അതിന്റെ കോള്യും ഫലം തിരിച്ചറിയുന്നു. നിങ്ങൾ ഏത് ടൂൾ ഉപയോഗിക്കണമെന്ന് അല്ലെങ്കിൽ എങ്ങനെ കോളുചെയ്യണമെന്ന് വ്യക്തമാക്കിയില്ലെങ്കിലും ഇത് എത്ര സ്വാഭാവികമായാണ് തോന്നുന്നത് എന്ന് ശ്രദ്ധിക്കുക.

### ടൂൾ ചെയിനിംഗ് പരീക്ഷിക്കുക

ഇപ്പോള്‍ കുറച്ചു ബുദ്ധിമുട്ടുള്ള ഒരു കാര്യം ശ്രമിക്കൂ: "സിയാറ്റിലിലെ കാലാവസ്ഥ എന്താണ്, അതിനെ ഫാരൻഹീറ്റിലേക്ക് മാറ്റൂ?" ഏജന്റ് ഇതിനായി ഘട്ടങ്ങളിലൂടെ പ്രവർത്തിക്കുന്നു. ആദ്യം കാലാവസ്ഥ ലഭിക്കുന്നു (സെൽഷ്യസിൽ ഫലം), പിന്നീട് ഫാരൻഹീറ്റിലേക്ക് മാറ്റേണ്ടതുണ്ടെന്ന് തിരിച്ചറിയുന്നു, പരിവർത്തന ടൂൾ കോളുചെയ്യുന്നു, ഇരുണ്ട ഫലങ്ങളെയും ഒറ്റ മറുപടിയാക്കി ചേർക്കുന്നു.

### സംഭാഷണ പ്രവാഹം കാണുക

ചാറ്റ് ഇന്റർഫേസ് സംഭാഷണ ചരിത്രം സൂക്ഷിക്കുന്നു, അതുകൊണ്ട് പല തവണ സംവദിക്കാൻ സാധിക്കുന്നു. നിങ്ങൾക്ക് മുമ്പത്തെ എല്ലാ ചോദനകളും ജവാബുകളും കാണാനാകും, ഇത് സംഭാഷണം എങ്ങനെ ക്രമീകരിച്ചതാണെന്നും എങ്ങനെ ഏജന്റ് സങ്കേതം നിർമ്മിക്കുന്നതാണെന്നും മനസ്സിലാക്കാൻ സഹായിക്കും.

<a href="images/tools-conversation-demo.png"><img src="../../../translated_images/ml/tools-conversation-demo.89f2ce9676080f59.webp" alt="Conversation with Multiple Tool Calls" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*സിമ്പിൾ പരിവർത്തനങ്ങൾ, കാലാവസ്ഥ അന്വേഷിക്കൽ, ടൂൾ ചെയിനിംഗ് എന്നിവയുമായിുള്ള മൾട്ടി-ടേൺ സംഭാഷണം*

### വ്യത്യസ്ത അഭ്യർത്ഥനകളുമായി പരീക്ഷിക്കുക

പல்வിട്ട ചേരുവകൾ ശ്രമിക്കുക:
- കാലാവസ്ഥ പരിശോധിക്കൽ: "ടോക്യോയിലെ കാലാവസ്ഥ എന്താണ്?"
- താപനില പരിവർത്തനങ്ങൾ: "25°C കെൽവിനിൽ എന്താണു?"
- സംയുക്ത ചോദനകൾ: "പാരിസിലെ കാലാവസ്ഥ പരിശോധിക്കുക, 20°C നു മുകളിൽ ആണോ എന്ന് അറിയിക്കുക"

ഏജന്റ് സ്വാഭാവിക ഭാഷയുടെ അർത്ഥം തിരിച്ചറിയുകയും യോജിച്ച ടൂൾ കോളുകളിലേക്ക് മാപ്പു നിർവഹിക്കുകയും ചെയ്യുന്നു.

## പ്രധാന ആശയങ്ങൾ

### ReAct പാറ്റേൺ (തർക്കം ചെയ്യലും പ്രവർത്തനവും)

എജന്റ് തർക്കം (എന്തുചെയ്യണം എന്ന് തീരുമാനിക്കൽ)നും പ്രവർത്തനം (ടൂളുകൾ ഉപയോഗിക്കൽ)നും ഇടയിൽ മാറി പ്രവർത്തിക്കുന്നു. ഇത് നിർദ്ദേശങ്ങൾക്ക് മറുപടി മാത്രമായി ഉപയോഗിക്കുന്നതിന്റെ പകരം സ്വയം പ്രശ്നപരിഹാരം നടത്താൻ ക്ഷമിക്കും.

### ടൂൾ വിവരണങ്ങൾ പ്രധാനമാണ്

നിങ്ങളുടെ ടൂൾ വിവരണങ്ങളുടെ ഗുണമേന്മ എത്രത്തോളം ഏജന്റ് അവ ഉപയോഗിക്കുന്നു എന്നതിന് നേരിട്ട് സ്വാധീനം ചെലുത്തുന്നു. വ്യക്തവും സ്‌പഷ്ടവുമായ വിവരണങ്ങൾ മോഡലിന് ടൂൾ 언제 사용해야 하는지 이해하는 데 도움이 됩니다.

### സെഷൻ മാനേജ്മെന്റ്

`@MemoryId` ആനോട്ടേഷൻ ഓട്ടോമാറ്റിക് സെഷൻ അടിസ്ഥാനമായ മെമ്മറി മാനേജ്മെന്റ് സജ്ജമാക്കുന്നു. ഓരോ സെഷൻ ഐഡിയ്ക്കും സ്വന്തം `ChatMemory` ഘടകം `ChatMemoryProvider`Bean വഴി കൈകാര്യം ചെയ്യുന്നു, അതുകൊണ്ട് നിരവധി ഉപയോക്താക്കൾ ഒരേസമയം ഏജന്റുമായി സംവദിച്ചാലും അവരുടെ സംഭാഷണങ്ങൾ പരസ്പരം കലരാത്ത വിധം ഉപകരിക്കുന്നു.

<img src="../../../translated_images/ml/session-management.91ad819c6c89c400.webp" alt="Session Management with @MemoryId" width="800"/>

*ഓരോ സെഷൻ ഐഡിയും ഒരു വേറിട്ട സംഭാഷണ ചരിത്രത്തിൽ ആക്കി മാപ്പ് ചെയ്യുന്നു — ഉപയോക്താക്കൾ ഒരുത്തരുടെ സന്ദേശങ്ങൾ കാണാറില്ല.*

### പിശക് കൈകാര്യംൽ

ടൂളുകൾ പരാജയപ്പെടാം — APIകൾ സമയ പരിധി überschreiten ചെയ്യാം, പാരാമീറ്ററുകൾ തെറ്റായിരിക്കാം, ബാഹ്യ സേവനങ്ങൾ പ്രവർത്തിക്കാതിരിക്കുക. ഉൽപാദന ഏജന്റുകൾക്ക് പിശക് കൈകാര്യംൽ ആവശ്യമാണ്, അത് മോഡലിന് പ്രശ്നങ്ങൾ വിശദീകരിക്കാനും പകരം മറ്റു മാർഗങ്ങൾ ശ്രമിക്കാനും അനുഗ്രഹിക്കുന്നു, സമ്പൂർണ്ണ ആപ്പ്ലിക്കേഷൻ ക്രാഷ് ആകാതിരിക്കാനുള്ള നടപടിയാണ്. ഒരു ടൂൾ എക്സപ്ഷൻ നൽകുമ്പോൾ, LangChain4j അത് പിടിച്ച് പിശക് സന്ദേശം മോഡലിലേക്ക് തിരിച്ചയക്കുന്നു, മോഡൽ പ്രകൃതി ഭാഷയിൽ പ്രശ്നം വിശദീകരിക്കാം.

## ലഭ്യമായ ടൂളുകൾ

താഴെയുള്ള ഡയഗ്രാം നിങ്ങൾ നിർമ്മിക്കാൻ സാധിക്കുന്ന വിപുലമായ ടൂൾ പരിസരം കാണിക്കുന്നു. ഈ മാഡ്യൂൾ കാലാവസ്ഥയും താപനില പരിവർത്തന ടൂളുകളും കാണിക്കുന്നു, പക്ഷേ സമാനമായ `@Tool` പാറ്റേൺ ഏതൊരു ജാവ മേധോഡിനും പ്രയോഗിക്കാവുന്നതാണ് — ഡാറ്റാബേസ് ക്വെറികളിൽ നിന്നും പേയ്‌മെന്റ് പ്രോസസ്സിങ് വരെ.

<img src="../../../translated_images/ml/tool-ecosystem.aad3d74eaa14a44f.webp" alt="Tool Ecosystem" width="800"/>

*@Tool ആനോട്ടേഷൻ ഉപയോഗിച്ചുള്ള ഏതൊരു ജാവ മേധോഡ് AI ഓട ഉപയോഗിക്കാൻ ലഭ്യമാകുന്നു — പാറ്റേൺ ഡാറ്റാബേസുകൾ, APIകൾ, ഇമെയിൽ, ഫയൽ ഓപ്പറേഷനുകൾ മുതലായവയിലേക്ക് വ്യാപിക്കുന്നു.*

## ടൂൾ അടിസ്ഥാനമാക്കിയ ഏജന്റുകൾ എപ്പോഴുഉപയോഗിക്കണം

<img src="../../../translated_images/ml/when-to-use-tools.51d1592d9cbdae9c.webp" alt="When to Use Tools" width="800"/>

*ക്ഷണിക ഡാറ്റ, കണക്കുകൂട്ടലുകൾ, പ്രവർത്തനങ്ങൾക്കായി ടൂളുകൾ ഉപയോക്താക്കൾക്കൊപ്പം ജോലി ചെയ്യാൻ സഹായിക്കുന്നു; പൊതുവായ അറിവിനും സൃഷ്ടിപരമായ പണികൾക്കും അവ വേണ്ടതല്ല.*

**നിങ്ങൾ ടൂളുകൾ ഉപയോഗിക്കേണ്ടത്:**
- നേരത്തെ ഉള്ള ഡാറ്റ ആവശ്യമായപ്പോൾ (കാലാവസ്ഥ, സ്റ്റോക്ക് വിലകൾ, ഇൻവെന്ററി)
- ലളിതമായ ഗണിതം കടന്ന് കണക്കുകൾ നടത്തേണ്ടത്
- ഡാറ്റാബേസുകൾക്ക് അല്ലെങ്കിൽ APIകൾക്ക് ആക്സസ്
- പ്രവർത്തനങ്ങൾ (ഇമെയിൽ അയക്കൽ, ടിക്കറ്റ് സൃഷ്ടിക്കൽ, റെക്കോർഡുകൾ അപ്ഡേറ്റ് ചെയ്യൽ)
- പല ഡാറ്റാ സ്രോതസ്സുകളും സംയോജിപ്പിക്കൽ

**ടൂളുകൾ ഉപയോക്താക്കയില്ലാത്തത്:**
- സാധാരണ അറിവിൽ നിന്നുള്ള ചോദനകൾക്ക് ഉത്തരം നൽകുമ്പോൾ
- മറുപടി ശുദ്ധമായ സംഭാഷണ സ്വഭാവമുള്ളപ്പോൾ
- ടൂൾ ലേറ്റൻസി അനുഭവം അതിൽ വേഗത കുറയ്ക്കുംപോൾ

## ടൂളുകളും RAG യും തമ്മിലുള്ള വ്യത്യാസം

മാഡ്യൂൾ 03യും 04യും എഐ ചെയ്യാൻ കഴിയുന്ന കാര്യങ്ങൾ വിപുലീകരിക്കുന്നു, എന്നാൽ അടിസ്ഥാനപരമായി വ്യത്യസ്ത മാർഗങ്ങൾ സ്വീകരിക്കുന്നു. RAG ആകുന്നു **അറിവ്** ലഭിക്കുന്നതിന് ഡോക്യുമെന്റുകൾ ഉടൻ തിരയൽ നൽകുന്നത്. ടൂളുകൾ **പ്രവർത്തനങ്ങൾ** നടത്താനുള്ള ശേഷി ഏര്‍പ്പെടുത്തും, പ്രവർത്തനങ്ങൾ കോളു ചയ്യുക.

<img src="../../../translated_images/ml/tools-vs-rag.ad55ce10d7e4da87.webp" alt="Tools vs RAG Comparison" width="800"/>

*RAG സ്റ്റാറ്റിക് ഡോക്യൂമെന്റുകളിൽ നിന്ന് വിവരങ്ങൾ പുനരുദ്ധരിക്കുന്നു — ടൂളുകൾ പ്രവർത്തനങ്ങൾ നിർവ്വഹിച്ചു ഡൈനാമിക്, ഏകദേശം സജീവ ഡാറ്റ നേടുന്നു. ബഹുസംഖ്യ ഉൽപ്പാദന സംവിധാനങ്ങൾ രണ്ടു രീതികളും ചേർത്ത് ഉപയോഗിക്കുന്നു.*

പ്രായോഗികമായി, ധാരാളം ഉൽപ്പാദന സംവിധാനങ്ങൾ ഇരുവിനെയും സംയോജിപ്പിക്കുന്നു: RAG നിങ്ങളുടെ ഡോക്യുമെന്റേഷൻ അടിസ്ഥാനമാക്കിയുള്ള ഉത്തരങ്ങൾക്കായി, ടൂളുകൾ ജീവിച്ചിരിക്കുന്ന ഡാറ്റ നേടാനോ പ്രവർത്തനങ്ങൾ നടത്താനോ.
