## നിങ്ങൾ പഠിക്കുക

നീங்கள் സംഭാഷണ AI നിർമ്മിച്ചു, പ്രോംപ്റ്റുകളിൽ പാടഡുപ്പ് നേടി, രേഖകളിൽ അടിസ്ഥാനമാക്കിയുള്ള പ്രതികരണങ്ങൾ സൃഷ്ടിച്ചു, ഉപകരണങ്ങളോടൊപ്പം ഏജന്റുമാർ രൂപപ്പെടുത്തിയിട്ടുണ്ട്. പക്ഷേ എല്ലാ ഉപകരണങ്ങളും നിങ്ങളുടെ പ്രത്യേക അപ്ലിക്കേഷനായി പ്രത്യേകമായി നിർമ്മിച്ചവയാണ്. ഏതു ആളും സൃഷ്ടിച്ച് പങ്കുവെക്കാൻ കഴിയുന്ന ഒരു സ്റ്റാൻഡേഡ് ഉപകരണ പരിസ്ഥിതിക്ക് നിങ്ങളുടെ AI-ന് ആക്സസ് നൽകാൻ നിങ്ങൾക്ക് സാധിക്കുമോ? ഈ മോഡ്യൂളിൽ, നിങ്ങൾ മോഡൽ കോൺടെക്‌സ് പ്രോട്ടോകോൾ (MCP)യും LangChain4j-ന്റെ എജന്റിക് മോഡ്യൂളും ഉപയോഗിച്ച് അത് എങ്ങനെ ചെയ്യാമെന്നു പഠിക്കും. ആദ്യം ഒരു ലളിതമായ MCP ഫയൽ റീഡർ കാണിക്കുകയും, പിന്നീട് അത് എളുപ്പത്തിൽ സൂപ്പർവൈസർ എജന്റ് പാറ്റേൺ ഉപയോഗിച്ച് ആധുനിക എജന്റിക് വർക്ക്‌ഫ്ലോകളിൽ എങ്ങനെ ഇന്റഗ്രേറ്റ് ചെയ്യും എന്നത് കാണിക്കും.

## MCP എന്താണ്?

മോഡൽ കോൺടെക്‌സ് പ്രോട്ടോകോൾ (MCP) അതെല്ലാം നൽകുന്നു - AI ആപ്പ്ലിക്കേഷനുകൾക്ക് ബാഹ്യ ഉപകരണങ്ങൾ കണ്ടെത്താനും ഉപയോഗിക്കാനും ഒരു സ്റ്റാൻഡേഡ് മാർഗം. ഓരോ ഡേറ്റാ സോഴ്സ് അല്ലെങ്കിൽ സർവിസിനും കസ്റ്റം ഇന്റഗ്രേഷൻ എഴുതുന്നതിനിടെ, MCP സർവറുകൾ അവരുടെ കഴിവുകൾ ഒരു ഏകരൂപ ഫോർമാറ്റിലൂടെ പ്രദർശിപ്പിക്കുന്നു. നിങ്ങളുടെ AI ഏജന്റ് ഈ ഉപകരണങ്ങൾ സ്വയം കണ്ടെത്തുകയും ഉപയോഗിക്കുകയും ചെയ്യും.

താഴെ കാണുന്ന ചാർട്ടിൽ വ്യത്യാസം കാണാം — MCP ഇല്ലാതെ ഓരോ ഇന്റഗ്രേഷനും പ്രത്യേക കസ്റ്റം വയറിംഗ് ആവശ്യമാണ്; MCP ഉപയോഗിച്ചാൽ, ഒരു പ്രോട്ടോക്കോൾ നിങ്ങളുടെ അപ്ലിക്കേഷനെയും ഏത് ഉപകരണത്തെയും ബന്ധിപ്പിക്കുന്നു:

<img src="../../../translated_images/ml/mcp-comparison.9129a881ecf10ff5.webp" alt="MCP Comparison" width="800"/>

*മുൻപ് MCP: സങ്കീർണമുള്ള പോയിന്റ്-ടു-പോയിന്റ് ഇന്റഗ്രേഷനുകൾ. പിന്നീട് MCP: ഒരു പ്രോട്ടോക്കോൾ, എതരുമായും ബന്ധിപ്പിക്കാൻ അന്ത്യഹീന സാധ്യതകൾ.*

MCP AI വികസനത്തിലെ ഒരു മ ൡ ംശപ്രദമായ പ്രശ്നം പരിഹരിക്കുന്നു: ഓരോ ഇന്റഗ്രേഷനും കസ്റ്റം ആണ്. GitHub ആക്സസ് ചെയ്യണോ? കസ്റ്റം കോഡ്. ഫയലുകൾ വായിക്കണോ? കസ്റ്റം കോഡ്. ഡാറ്റാബേസ് ക്വറി ചെയ്യണോ? കസ്റ്റം കോഡ്. ഈ ഇന്റഗ്രേഷനുകളിൽ ഒന്നും മറ്റേതൊരു AI ആപ്പ്ലിക്കേഷനുമായി പ്രവർത്തിക്കാറില്ല.

MCP ഇത് സ്റ്റാൻഡേർഡൈസ് ചെയ്യുന്നു. MCP സർവർ ഉപകരണങ്ങൾ വ്യക്തമായ വിവരണങ്ങൾക്കും സ്കീമകൾക്കും കൂടി തുറന്നിടുന്നു. ഏതെങ്കിലും MCP ക്ലയന്റ് കണക്ട് ചെയ്ത് ലഭ്യമായ ഉപകരണങ്ങൾ കണ്ടെത്തും, അവ ഉപയോഗിക്കും. ഒരിക്കൽ നിർമ്മിച്ചു, എല്ലായിടത്തും ഉപയോഗിക്കുക.

താഴെയുള്ള ചിത്രത്തിൽ ഈ ആർക്കിടെക്ചർ കാണിക്കുന്നു — ഒരു MCP ക്ലയന്റ് (നിങ്ങളുടെ AI ആപ്പ്ലിക്കേഷൻ) നിരവധി MCP സർവറുകളുമായി കണക്ട് ചെയ്യുന്നു, ഓരോ MCP സർവും അവരുടെ സ്വന്തം ഉപകരണങ്ങൾ സ്റ്റാൻഡേർഡ് പ്രോട്ടോക്കോൾ വഴിയാണ് തുറന്നിടുന്നത്:

<img src="../../../translated_images/ml/mcp-architecture.b3156d787a4ceac9.webp" alt="MCP Architecture" width="800"/>

*മോഡൽ കോൺടെക്‌സ് പ്രോട്ടോകോൾ ആർക്കിടെക്ചർ - സ്റ്റാൻഡേർഡൈസ്ഡ് ഉപകരണ കണ്ടെത്തലും പ്രവർത്തനവും*

## MCP എങ്ങനെ പ്രവർത്തിക്കുന്നു

അതിന്റെ ആന്തരത്തിൽ, MCP ഒരു പാളിപ്പാർശ്വ ആർക്കിടെക്ചർ ഉപയോഗിക്കുന്നു. നിങ്ങളുടെ ജാവ അപ്ലിക്കേഷൻ (MCP ക്ലയന്റ്) ലഭ്യമായ ഉപകരണങ്ങൾ കണ്ടെത്തുന്നു, JSON-RPC അഭ്യർത്ഥനകൾ ഒരു ട്രാൻസ്പോർട്ട് ലെയർ (Stdio അല്ലെങ്കിൽ HTTP) വഴി അയയ്ക്കുന്നു, MCP സർവർ പ്രവർത്തനങ്ങൾ സ്വയം നടപ്പിലാക്കി ഫലങ്ങൾ തിരികെ നൽകുന്നു. ഈ പ്രോട്ടോക്കോൾ ഓരോ പാളിയും താഴെ കാണുന്നു:

<img src="../../../translated_images/ml/mcp-protocol-detail.01204e056f45308b.webp" alt="MCP Protocol Detail" width="800"/>

*MCP-യുടെ അടിയിലെ പ്രവർത്തനം — ക്ലയന്റുകൾ ഉപകരണങ്ങൾ കണ്ടെത്തുന്നു, JSON-RPC സന്ദേശങ്ങൾ എക്സ്ചേഞ്ച് ചെയ്യുന്നു, ട്രാൻസ്പോർട്ട് ലെയർ വഴി പ്രവർത്തനങ്ങൾ നടപ്പിലാക്കുന്നു.*

**സർവർ-ക്ലയന്റ് ആർക്കിടെക്ചർ**

MCP ഒരു ക്ലയന്റ്-സർവർ മോഡൽ ആണ്. സർവറുകൾ ഉപകരണങ്ങൾ നൽകുന്നു - ഫയലുകൾ വായിക്കൽ, ഡാറ്റാബേസുകൾ ക്വറി ചെയ്യൽ, APIs കോളിങ്ങ്. ക്ലയന്റുകൾ (നിങ്ങളുടെ AI ആപ്പ്) സർവറുമായി കണക്ട് ചെയ്യുകയും സ്വയം ഉപകരണങ്ങൾ ഉപയോഗിക്കുകയും ചെയ്യുന്നു.

LangChain4j ഉപയോഗിച്ച് MCP ഉപയോഗിക്കാൻ, Maven ഡിപെൻഡൻസി ചേര്ക്കുക:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```

**ഉപകരണ കണ്ടെത്തൽ**

നിങ്ങളുടെ ക്ലയന്റ് MCP സർവറുമായി കണക്ട് ചെയുമ്പോൾ, "നിങ്ങൾക്ക് എന്തെല്ലാം ഉപകരണങ്ങളുണ്ട്?" എന്ന് ചോദിക്കും. സർവർ ലഭ്യമായ ഉപകരണങ്ങളുടെ ലിസ്റ്റും അവയുടെ വിവരണങ്ങളും പാരാമീറ്റർ സ്കീമകളും നൽകുന്നു. നിങ്ങളുടെ AI ഏജന്റ് ഉപഭോക്തൃ അഭ്യർത്ഥനകളുടെ അടിസ്ഥാനത്തിൽ ഏതെല്ലാം ഉപകരണങ്ങൾ ഉപയോഗിക്കാമെന്ന് തീരുമാനിക്കും. താഴെയുള്ള ചിത്രത്തിൽ ഈ ഹാൻഡ്‌ഷേക്ക് കാണാം — ക്ലയന്റ് `tools/list` അഭ്യർത്ഥന അയയ്ക്കുന്നു, സർവർ ലഭ്യമായ ഉപകരണങ്ങൾ പാരാമീറ്ററുകളും വിവരണങ്ങളുമായി അയക്കുന്നു:

<img src="../../../translated_images/ml/tool-discovery.07760a8a301a7832.webp" alt="MCP Tool Discovery" width="800"/>

*AI ലഭ്യമായ ഉപകരണങ്ങൾ സ്റ്റാർട്ടപ്പിൽ കണ്ടെത്തുന്നു — അത് ഇപ്പോൾ എന്ത് കഴിവുകൾ ഉണ്ട് എന്ന് അറിയുകയും ഏതെല്ലാം ഉപകരണങ്ങൾ ഉപയോഗിക്കാമെന്ന് തീരുമാനിക്കുകയും ചെയ്യുന്നു.*

**ട്രാൻസ്പോർട്ട് മെക്കാനിസങ്ങൾ**

MCP വ്യത്യസ്ത ട്രാൻസ്പോർട്ട് മെക്കാനിസങ്ങൾ പിന്തുണയ്ക്കുന്നു. രണ്ട് ഓപ്ഷനുകൾ ഉണ്ട്: Stdio (ലോകൽ subprocess കമ്മ്യൂണിക്കേഷനുകൾക്കും) Streamable HTTP (റിമോട്ട് സർവർകൾക്കുമായി). ഈ മോഡ്യൂളിൽ Stdio ട്രാൻസ്പോർട്ട് പ്രദർശിപ്പിക്കുന്നു:

<img src="../../../translated_images/ml/transport-mechanisms.2791ba7ee93cf020.webp" alt="Transport Mechanisms" width="800"/>

*MCP ട്രാൻസ്പോർട്ട് മെക്കാനിസങ്ങൾ: റിമോട്ട് സർവർക്കായി HTTP, ലോക്കൽ പ്രോസസ്സുകൾക്കായി Stdio*

**Stdio** - [StdioTransportDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java)

ലോക്കൽ പ്രോസസ്സുകൾക്കായി. നിങ്ങളുടെ അപ്ലിക്കേഷൻ MCP സർവർ ഒരു subprocess ആയി തുടങ്ങുകയും സ്റ്റാൻഡേർഡ് ഇൻപുട്ട്/ഔട്ട്‌പുട്ട് ഇടപെടൽ വഴി കമ്മ്യൂണിക്കേറ്റ് ചെയ്യുകയും ചെയ്യും. ഫയൽ സിസ്റ്റം ആക്സസിനും കമാൻഡ് ലൈൻ ടൂളുകൾക്കുമായി ഉപയോഗപ്രദമാണ്.

```java
McpTransport stdioTransport = new StdioMcpTransport.Builder()
    .command(List.of(
        npmCmd, "exec",
        "@modelcontextprotocol/server-filesystem@2025.12.18",
        resourcesDir
    ))
    .logEvents(false)
    .build();
```

`@modelcontextprotocol/server-filesystem` സർവർ താഴെപ്പറഞ്ഞ ഉപകരണങ്ങൾ സാൻഡ്‌ബോക്സ് ചെയ്ത രൂപത്തിൽ നിങ്ങൾ നൽകുന്ന ഡയറക്ടറികൾക്ക് ഉപയോഗിച്ചു തുറന്നിടുന്നു:

| ഉപകരണം | വിവരണം |
|------|-------------|
| `read_file` | ഒരു ഫയലിന്റെ ഉള്ളടക്കം വായിക്കുക |
| `read_multiple_files` | ഒരേസമയം ഒന്നിലധികം ഫയലുകൾ വായിക്കുക |
| `write_file` | ഒരു ഫയൽ സൃഷ്ടിക്കുക അല്ലെങ്കിൽ overwrite ചെയ്യുക |
| `edit_file` | ലക്ഷ്യമിട്ടുള്ള കണ്ടെത്തൽ-മാറ്റം എഡിറ്റുകൾ ചെയ്യുക |
| `list_directory` | ഒരു പാതയിലെ ഫയലുകളും ഡയറക്ടറികളും പട്ടികപ്പെടുത്തുക |
| `search_files` | പാറ്റേണുമായി പൊരുത്തമുള്ള ഫയലുകൾ റികർഷൻ ആയി അന്വേഷിക്കുക |
| `get_file_info` | ഫയൽ മെറ്റാഡേറ്റ (വലിപ്പം, ടൈംസ്റാമ്പ്, അനുമതികൾ) നേടുക |
| `create_directory` | ഒരു ഡയറക്ടറി സൃഷ്ടിക്കുക (പാരണ്ട് ഡയറക്ടറി ഉൾപ്പെടെ) |
| `move_file` | ഒരു ഫയൽ അല്ലെങ്കിൽ ഡയറക്ടറി മാറ്റുക അല്ലെങ്കിൽ പേര് മാറുക |

താഴെയുള്ള ചിത്രത്തിൽ Stdio ട്രാൻസ്പോർട്ട് ഫ്ലോ Runtime-ൽ എങ്ങനെ പ്രവർത്തിക്കുന്നു എന്ന് കാണിക്കുന്നു — നിങ്ങളുടെ ജാവ അപ്ലിക്കേഷൻ MCP സർവർ subprocess ആയി സൃഷ്ടിക്കുകയും stdin/stdout പൈപ്പുകളിലൂടെ കമ്മ്യൂണിക്കേറ്റ് ചെയ്യുകയും ചെയ്യുന്നു, നെറ്റ്‌വർക്കിന്റെയും HTTP യുടെയും ഇടപെടൽ ഇല്ലാതെ:

<img src="../../../translated_images/ml/stdio-transport-flow.45eaff4af2d81db4.webp" alt="Stdio Transport Flow" width="800"/>

*Stdio ട്രാൻസ്പോർട്ട് പ്രവർത്തനത്തിൽ — നിങ്ങളുടെ അപ്ലിക്കേഷൻ MCP സർവർ subprocess ആയി ആരംഭിക്കുകയും stdin/stdout പൈപ്പുകളിലൂടെ കമ്മ്യൂണിക്കേറ്റ് ചെയ്യുകയും ചെയ്യുന്നു.*

> **🤖 [GitHub Copilot](https://github.com/features/copilot) ചാർട്ടിന്റെ സഹായത്തോടെ ശ്രമിക്കുക:** `StdioTransportDemo.java` തുറന്ന് ചോദിക്കുക:
> - "Stdio transport എങ്ങനെ പ്രവർത്തിക്കുന്നു, HTTP ൽ നിന്ന് എപ്പോൾ ഇത് ഉപയോഗിക്കണം?"
> - "LangChain4j എങ്ങനെ MCP സർവർ പ്രോസസ്സുകളുടെ ലൈഫ്‌സൈകിൾ മാനേജർ ചെയ്യുന്നു?"
> - "AI-ന് ഫയൽ സിസ്റ്റത്തിലേക്ക് ആക്‌സസ് നൽകുന്നതിന്റെ സുരക്ഷാ ചിന്തനകൾ എന്തൊക്കെയാണ്?"

## എജന്റിക് മോഡ്യൂൾ

MCP സ്റ്റാൻഡേർഡ്ഡ് ഉപകരണങ്ങൾ നൽകുമ്പോൾ, LangChain4j-ന്റെ **എജന്റിക് മോഡ്യൂൾ** അവ ഉപകരണങ്ങൾ നിയന്ത്രിക്കുന്ന ഏജന്റുകൾ രൂപകൽപ്പന ചെയ്യാൻ ഡിക്ലറേറ്റീവ് മാർഗം നൽകുന്നു. `@Agent` ആനൊട്ടേഷൻ, `AgenticServices` എന്നിവ ഇന്റർഫേസുകൾ വഴിയുള്ള ഏജന്റ് പെരുമാറ്റം നിർവചിക്കാൻ സഹായിക്കുന്നു, അനുഭവപരിചയം നിയന്ത്രണ കോഡിനു പകരം.

ഈ മോഡ്യൂളിൽ, നിങ്ങൾ **സൂപ്പർവൈസർ ഏജന്റ്** പാറ്റേൺ പഠിക്കും — ഇത് ഒരു ആധുനിക എജന്റിക് AI സമീപനമാണ്, ഇവിടെ "സൂപ്പർവൈസർ" ഏജന്റ് ഉപയോക്തൃ ആവശ്യകതകളിന്റെ അടിസ്ഥാനത്തിൽ ഏത് ഉപ-ഏജന്റുകളെ വിളിക്കാമെന്ന് സ്വയം തീരുമാനിക്കുന്നു. MCP-പവർഡ് ഫയൽ ആക്‌സസ് സാങ്കേതികത നമ്മുടെ ഒരു ഉപ-ഏജന്റിൽ ചേർക്കാനും നാം രണ്ട് ആശയങ്ങളും സംയോജിപ്പിക്കും.

എജന്റിക് മോഡ്യൂൾ ഉപയോഗിക്കാൻ, Maven ഡിപെൻഡൻസി ചേർക്കുക:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```
> **കുറിപ്പ്:** `langchain4j-agentic` മോഡ്യൂൾക്ക് വേറിട്ട വേർഷൻ പ്രോപ്പർട്ടി (`langchain4j.mcp.version`) ഉണ്ട്, കാരണം ഇത് കോർ LangChain4j ലൈബ്രറി രീതികളിൽ നിന്ന് വ്യത്യസ്ത ഷെഡ്യൂളിൽ റിലീസ് ചെയ്യുന്നു.

> **⚠️ പരീക്ഷണഘട്ടം:** `langchain4j-agentic` മോഡ്യൂൾ **പരീക്ഷണ ഘട്ടത്തിലാണ്** ഇതിൽ മാറ്റങ്ങൾ ഉണ്ടാകാം. കസ്റ്റം ടൂളുകളോടെയുള്ള `langchain4j-core` ആണ് ഇപ്പോഴും AI സഹായികൾ നിർമ്മിക്കുന്ന സ്ഥിരമായ മാർഗം (Module 04).

## വേഗം തുടങ്ങിയ്ക്കൽ

**Maven ഉപയോഗിക്കുക:** അല്ലെങ്കിൽ താഴെ കാണുന്ന കമാൻഡുകൾ കൊണ്ട് കമാൻഡ് ലൈൻ വഴിയും ഓടിക്കാൻ കഴിയും.

### ഫയൽ പ്രവർത്തനങ്ങൾ (Stdio)

ഇത് ലോക്കൽ subprocess അടിസ്ഥാനമാക്കിയുള്ള ഉപകരണങ്ങൾ കാണിക്കുന്നു.

**✅ മുൻകൂർതയ്യാറുകൾ ആവശ്യമാണ്** - MCP സർവർ സ്വയം ആരംഭിക്കും.

**സ്റ്റാർട്ട് സ്‌ക്രിപ്റ്റുകൾ ഉപയോഗിക്കുക (പരിശോധന നിർദ്ദേശം):**

സ്റ്റാർട്ടപ്പ് സ്‌ക്രിപ്റ്റുകൾ സ്വയം റൂട്ട് `.env` ഫയൽ നിന്നുള്ള എൻവയൺമെന്റ് വേരിയബിൾസ് ലോഡ് ചെയ്യും:

**ബാഷ്:**
```bash
cd 05-mcp
chmod +x start-stdio.sh
./start-stdio.sh
```

**PowerShell:**
```powershell
cd 05-mcp
.\start-stdio.ps1
```

**VS Code ഉപയോഗിച്ച്:** `StdioTransportDemo.java`-യിൽ റൈറ്റ് ക്ലിക്ക് ചെയ്ത് **"Run Java"** തിരഞ്ഞെടുക്കുക (`.env` ഫയൽ ക്രമീകരിച്ചിട്ടുണ്ടെന്ന് ഉറപ്പാക്കുക).

അപ്ലിക്കേഷൻ ഫയൽ സിസ്റ്റം MCP സർവർ സ്വയം ആരംഭിക്കുകയും ഒരു ലോക്കൽ ഫയൽ വായിക്കുകയും ചെയ്യും. subprocess മാനേജ്മെന്റ് എങ്ങനെ കൈകാര്യം ചെയ്യുന്നുവെന്ന് ശ്രദ്ധിക്കുക.

**പ്രതീക്ഷിക്കാത്ത ഫലം:**
```
Assistant response: The file provides an overview of LangChain4j, an open-source Java library
for integrating Large Language Models (LLMs) into Java applications...
```

### സൂപ്പർവൈസർ എജന്റ്

**സൂപ്പർവൈസർ എജന്റ് പാറ്റേൺ** ഒരു **സ്വീലമായ** ഏജന്റിക് AI രൂപമാണ്. ഒരു സൂപ്പർവൈസർ LLM ഉപയോഗിച്ച് ഉപയോക്തൃ അഭ്യർത്ഥനയുടെ അടിസ്ഥാനത്തിൽ ഏജന്റുമാർക്കിടയിൽ ആരെ വിളിക്കാമെന്നു സ്വയം തീരുമാനിക്കുന്നു. അടുത്ത ഉദാഹരണത്തിൽ, MCP-പവർഡ് ഫയൽ ആക്സസും LLM ഏജന്റും ചേർന്ന് സൂപ്പർവൈസ്ഡ് ഫയൽ വായനം → റിപ്പോർട്ട് പ്രവണത സൃഷ്ടിക്കുന്നു.

ഡെമോയിൽ, `FileAgent` MCP ഫയൽസിസ്റ്റം ഉപകരണങ്ങൾ ഉപയോഗിച്ച് ഫയൽ വായിക്കുകയും, `ReportAgent` എക്സിക്യൂട്ടീവ് സംക്ഷേപം (1 വാക്ക്), 3 പ്രധാന പോയിന്റുകൾ, ശുപാർശകൾ അടങ്ങിയ ഘടിത റിപ്പോർത്തൊരുക്കുകയും ചെയ്യുന്നു. സൂപ്പർവൈസർ ഈ പ്രവാഹം സ്വകാര്യം നിയന്ത്രിക്കുന്നു:

<img src="../../../translated_images/ml/supervisor-agent-pattern.06275a41ae006ac8.webp" alt="Supervisor Agent Pattern" width="800"/>

*സൂപ്പർവൈസർ അതിന്റെ LLM ഉപയോഗിച്ച് ഏജന്റുമാരെ ആരെ വിളിക്കണമെന്ന്, ഏതു ക്രമത്തിലാണ് വിളിക്കുന്നത് എന്ന് തീരുമാനിക്കുന്നു — ഹാർഡ്‌കോഡ് റോouting ആവശ്യമില്ല.*

ഇവിടെ ഫോഡ് ടു റിപ്പോർട്ട് പൈപ്പ്ലൈൻ സർവ്വ്വിന്റ എങ്ങനെ പ്രവർത്തിക്കുന്നു എന്നത് കാണാം:

<img src="../../../translated_images/ml/file-report-workflow.649bb7a896800de9.webp" alt="File to Report Workflow" width="800"/>

*FileAgent MCP ഉപകരണങ്ങൾ ഉപയോഗിച്ച് ഫയൽ വായിക്കുന്നു, തുടർന്ന് ReportAgent പ്രഭാഷണം ഘടിത റിപ്പോർട്ടായി പരിഷ്കരിക്കുന്നു.*

ഓരോ ഏജന്റും അവരുടെ ഔട്ട്പുട്ട് **Agentic Scope**-ലെ (പങ്കുവെച്ചു വരുന്ന മെമ്മറി) സ്റ്റോർ ചെയ്യുന്നു, അതുവഴി പിന്നിലേക്കുള്ള ഏജന്റുകൾ മുൻ ഫലങ്ങളിൽ ആക്സസ് നേടുന്നു. ഇത് MCP ഉപകരണങ്ങൾ എങ്ങനെ എജന്റിക് വർക്ക്‌ഫ്ലോകളിലേക്ക് സ്മൂത്തിൽ ഇന്റഗ്രേറ്റ് ചെയ്യുന്നത് കാണിക്കുന്നു — സൂപ്പർവൈസർ എങ്ങനെ ഫയലുകൾ വായിക്കാനാണെന്ന് അറിയേണ്ടതില്ല, മാത്രമേ `FileAgent` അത് നടത്തുക എന്നറിയണം.

#### ഡെമോ പ്രവർത്തിപ്പിക്കൽ

സ്റ്റാർട്ട് സ്‌ക്രിപ്റ്റുകൾ സ്വയം റൂട്ടിലെ `.env` ഫയൽ നിന്നുള്ള എൻവയൺമെന്റ് വേരിയബിൾസ് ലോഡ് ചെയ്യുന്നു:

**ബാഷ്:**
```bash
cd 05-mcp
chmod +x start-supervisor.sh
./start-supervisor.sh
```

**PowerShell:**
```powershell
cd 05-mcp
.\start-supervisor.ps1
```

**VS Code ഉപയോഗിച്ച്:** `SupervisorAgentDemo.java` റൈറ്റ് ക്ലിക്ക് ചെയ്ത് **"Run Java"** തിരഞ്ഞെടുക്കുക (`.env` ഫയൽ ക്രമീകരിച്ചിട്ടുണ്ടെന്ന് ഉറപ്പാക്കുക).

#### സൂപ്പർവൈസർ എങ്ങനെ പ്രവർത്തിക്കുന്നു

ഏജന്റുകൾ നിർമ്മിക്കുന്ന മുമ്പ്, MCP ട്രാൻസ്പോർട്ടിനെ ഒരു ക്ലയന്റിനൊപ്പം കണക്ട് ചെയ്യുകയും അത് ഒരു `ToolProvider` ആയി മൊറപ്പെടുത്തുകയും വേണം. MCP സർവർ ഉപകരണങ്ങൾ ഏജന്റുകൾക്ക് ലഭ്യമാകാനുള്ള വഴി ഇതാണ്:

```java
// ട്രാൻസ്‌പോർട്ടിൽ നിന്ന് ഒരു MCP ക്ലയന്റ് സൃഷ്ടിക്കുക
McpClient mcpClient = new DefaultMcpClient.Builder()
        .transport(stdioTransport)
        .build();

// ക്ലയന്റിനെ ToolProvider ആയി മുറിക്കുക — ഇത് MCP ടൂളുകളെ LangChain4j-ലേക്ക് ബന്ധിപ്പിക്കുന്നു
ToolProvider mcpToolProvider = McpToolProvider.builder()
        .mcpClients(List.of(mcpClient))
        .build();
```

ഇപ്പോൾ MCP ഉപകരണങ്ങൾ ആവശ്യമായ ഏജന്റുകൾ ഏതെവിടെങ്കിലും `mcpToolProvider` injection ചെയ്യാനാകും:

```java
// ചുവടു 1: ഫയൽഎജന്റ് MCP ഉപകരണങ്ങൾ ഉപയോഗിച്ച് ഫയലുകൾ വായിക്കുന്നു
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // ഫയൽ പ്രവർത്തനങ്ങൾക്ക് MCP ഉപകരണങ്ങൾ ഉണ്ട്
        .build();

// ചുവടു 2: റിപ്പോർട്ട് എജന്റ് ഘടനാപരമായ റിപ്പോർട്ടുകൾ സൃഷ്ടിക്കുന്നു
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// സൂപ്പർവൈസർ ഫയൽ → റിപ്പോർട്ട് പ്രവൃത്തി പ്രവാഹം ഓർക്കസ്ട്രേറ്റ് ചെയ്യുന്നു
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // അന്തിമ റിപ്പോർട്ട് തിരികെ നൽകുക
        .build();

// അഭ്യർത്ഥനയുടെ അടിസ്ഥാനത്തിൽ ഏതു ഏജന്റുകളെ വിളിക്കാമെന്ന് സൂപ്പർവൈസർ തീരുമാനം എടുക്കുന്നു
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```

#### പ്രതികരണ തന്ത്രങ്ങൾ

`SupervisorAgent` കോൺഫിഗർ ചെയ്യുമ്പോൾ, ഉപ-ഏജന്റുകൾ അവരുടെ ജോലികൾ പൂർത്തിയാക്കിയ ശേഷം ഉപയോക്താവിന് എങ്ങനെ അവസാന ഉത്തരം നൽകണം എന്ന് നിശ്ചയിക്കുന്നു. താഴെയുള്ള ചിത്രത്തിൽ മൂന്ന് ലഭ്യമായ തന്ത്രങ്ങൾ കാണിക്കുന്നു — LAST അവസാന ഏജന്റിന്റെ ഉയർത്തുന്ന ഔട്ട്പുട്ട് നേരിട്ട് നൽകുന്നു, SUMMARY എല്ലാ ഔട്ട്പുട്ടുകളും LLM വഴി സമാഹരിച്ച് സിന്തസൈസ് ചെയ്യുന്നു, SCORED ആദ്യ അഭ്യർത്ഥനയോട് പൂർത്തിയായ,久久ും പ്രധാനമായ ഉത്തരം ഏത് ആണെന്ന് LLM സ്കോർ ചെയ്ത് തിരികെ നൽകുന്നു:

<img src="../../../translated_images/ml/response-strategies.3d0cea19d096bdf9.webp" alt="Response Strategies" width="800"/>

*സൂപ്പർവൈസർ അവന്റെ അവസാന പ്രതികരണം എങ്ങനെ നിർവചിക്കുന്നുവെന്ന് മൂന്നാം തന്ത്രങ്ങൾ — അവസാന ഏജന്റിന്റെ ഔട്ട്പുട്ട്‌, സിന്തസൈസ് ചെയ്ത സംക്ഷേപം, മികച്ച സ്കോർ ലഭിച്ച ഓപ്ഷൻ എന്നിവയിൽ നിന്ന് തിരഞ്ഞെടുക്കുക.*

ലഭ്യമായ തന്ത്രങ്ങൾ:

| തന്ത്രം | വിവരണം |
|----------|-------------|
| **LAST** | സൂപ്പർവൈസർ അവസാനം കോളായ ഉപ-ഏജന്റിന്റെ അല്ലെങ്കിൽ ടൂളിന്റെ ഔട്ട്പുട്ട് തിരിച്ചുകൊടുക്കുന്നു. അവസാന ഏജന്റ് പ്രത്യേകിച്ച് സമ്പൂർണ്ണ ഉത്തരം നൽകാൻ രൂപകൽപന ചെയ്തിരിക്കുന്നപ്പോൾ ഇത് പ്രയോജനകരമാണ് (ഉദാഹരണത്തിന്, റിസർച്ച് പൈപ്പ്‌ലൈനിലെ "സംക്ഷേപ ഏജന്റ്"). |
| **SUMMARY** | സൂപ്പർവൈസർ തന്റെ ആഭ്യന്തര ഭാഷാ മോഡൽ (LLM) ഉപയോഗിച്ച് ലൈനിലുള്ള എല്ലാ സൗജന്യങ്ങൾക്കും ഉപ-ഏജന്റുകളുടെ ഔട്ട്പുട്ടുകൾക്കും സംക്ഷേപം സൃഷ്ടിക്കുകയും അതിനെ അന്തിമ പ്രതികരണമായി നൽകുകയും ചെയ്യുന്നു. ഇത് ഒരു ക്രമീകരിച്ച, കൂട്ടിച്ചേർത്ത ഉത്തരം നൽകുന്നു. |
| **SCORED** | സിസ്റ്റം ആഭ്യന്തര LLM ഉപയോഗിച്ച് അവസാനുള്ള (LAST) പ്രതികരണ കണ്ടെത്തിയും ഒപ്പം ആശയവിനിമയയുടെ SUMMARY യും ഉപയോഗിച്ച് ഉപയോക്തൃ അഭ്യർത്ഥനയോട് താരതമ്യം ചെയ്യുകയും കൂടുതൽ സ്കോർ ലഭിച്ച ഔട്ട്പുട്ട് തിരികെ നൽകുകയും ചെയ്യുന്നു. |

#### ഔട്ട്‌പുട്ട് മനസ്സിലാക്കുക

ഡെമോ റൺ ചെയ്യുമ്പോൾ, Supervisor എങ്ങനെ അനേകം ഏജന്റുകൾ ഏകീകരിക്കുന്നുവെന്ന് ഘടിതമായ ഒരു വി​വരണം കാണാൻ കൂടും. ഓരോ സെക്ഷനും എന്താണെന്ന് താഴെ കൊടുത്തിട്ടുണ്ട്:

```
======================================================================
  FILE → REPORT WORKFLOW DEMO
======================================================================

This demo shows a clear 2-step workflow: read a file, then generate a report.
The Supervisor orchestrates the agents automatically based on the request.
```

**ഹെഡർ** workflow ആശയം പരിചയപ്പെടുത്തുന്നു: ഫയൽ വായനയിലേക്കും റിപ്പോർട്ട് ജനറേഷനിലേക്കുള്ള ശ്രദ്ധയുള്ള പൈപ്പ്‌ലൈൻ.

```
--- WORKFLOW ---------------------------------------------------------
  ┌─────────────┐      ┌──────────────┐
  │  FileAgent  │ ───▶ │ ReportAgent  │
  │ (MCP tools) │      │  (pure LLM)  │
  └─────────────┘      └──────────────┘
   outputKey:           outputKey:
   'fileContent'        'report'

--- AVAILABLE AGENTS -------------------------------------------------
  [FILE]   FileAgent   - Reads files via MCP → stores in 'fileContent'
  [REPORT] ReportAgent - Generates structured report → stores in 'report'
```

**Workflow ഡയഗ്രാം** ഏജന്റുകൾ തമ്മിലുള്ള ഡാറ്റാ ഫ്ലോ കാണിക്കുന്നു. ഓരോ ഏജന്റിനും ഒരു പ്രത്യേക പണി ഉണ്ട്:
- **FileAgent** MCP ടൂളുകൾ ഉപയോഗിച്ച് ഫയലുകൾ വായിച്ച് `fileContent`ൽ രാസ് ഉള്ളടക്കം സൂക്ഷിക്കുന്നു
- **ReportAgent** ആ ഉള്ളടക്കം ഉപയോഗിച്ച് `report` എന്ന ഘടിത റിപ്പോർട്ട് ഉല്പാദിപ്പിക്കുന്നു

```
--- USER REQUEST -----------------------------------------------------
  "Read the file at .../file.txt and generate a report on its contents"
```

**വ്യവസ്ഥയുടെ അഭ്യർത്ഥന** ടാസ്‌ക് കാണിക്കുന്നു. Supervisor ഇത് പാഴ്‌സ് ചെയ്ത് FileAgent → ReportAgent എന്ന ക്രമത്തിൽ ഏജന്റുകൾ വിളിക്കാനാണ് തീരുമാനിക്കുന്നത്.

```
--- SUPERVISOR ORCHESTRATION -----------------------------------------
  The Supervisor decides which agents to invoke and passes data between them...

  +-- STEP 1: Supervisor chose -> FileAgent (reading file via MCP)
  |
  |   Input: .../file.txt
  |
  |   Result: LangChain4j is an open-source, provider-agnostic Java framework for building LLM...
  +-- [OK] FileAgent (reading file via MCP) completed

  +-- STEP 2: Supervisor chose -> ReportAgent (generating structured report)
  |
  |   Input: LangChain4j is an open-source, provider-agnostic Java framew...
  |
  |   Result: Executive Summary...
  +-- [OK] ReportAgent (generating structured report) completed
```

**Supervisor ഓർക്കെസ്‌ട്രേഷൻ** 2-പടി ഫ്ലോ പ്രവർത്തി കാണിക്കുന്നു:
1. **FileAgent** MCP വഴി ഫയൽ വായിച്ച് ഉള്ളടക്കം സൂക്ഷിക്കുന്നു
2. **ReportAgent** ഉള്ളടക്കം സ്വീകരിച്ച് ഘടിത റിപ്പോർട്ട് നിർമ്മിക്കുന്നു

Supervisor ഉപയോക്താവിന്റെ അഭ്യർത്ഥന അടിസ്ഥാനമാക്കി സ്വയം ഈ തീരുമാനം എടുത്തിരിക്കുന്നു.

```
--- FINAL RESPONSE ---------------------------------------------------
Executive Summary
...

Key Points
...

Recommendations
...

--- AGENTIC SCOPE (Data Flow) ----------------------------------------
  Each agent stores its output for downstream agents to consume:
  * fileContent: LangChain4j is an open-source, provider-agnostic Java framework...
  * report: Executive Summary...
```

#### ഏജന്റിക് മഡ്യൂൾ ഫീച്ചറുകളുടെ وضاحت

ഉദാഹരണം ഏജന്റിക് മഡ്യൂളിന്റെ ചില അതീത ഗുണങ്ങൾ കാണിക്കുന്നു. Agentic Scope, Agent Listeners എന്നിവയെ കൂടുതൽ വിശദമായി നോക്കാം.

**Agentic Scope** ഉപയോഗിച്ചുള്ള പങ്കുവെച്ച മെമ്മറി കാണിക്കുന്നു, ഏജന്റുകൾ `@Agent(outputKey="...")` ഉപയോഗിച്ച് അവരുടെ ഫലങ്ങൾ ഇവിടെ നിക്ഷേപിച്ചു. ഇതിലൂടെ:
- പിന്ഗാമികൾ മുൻ ഏജന്റുകളുടെ ഔട്ട്ഡോട്ടുകൾ ആക്‌സസ് ചെയ്യുന്നതിന്
- Supervisor അവസാന പ്രതികരണം സംയോജിപ്പിക്കാൻ
- നിങ്ങൾ ഓരോ ഏജന്റും എന്ത് ഉല്പാദിപ്പിച്ചു എന്ന് പരിശോധിക്കാനായി

താഴെ ഡയഗ്രാം Agentic Scope ഫയൽ മുതൽ റിപ്പോർട്ട് workflow ഇടയിലെ പങ്കുവെച്ച മെമ്മറിയെന്ന നിലയിൽ എങ്ങനെ പ്രവർത്തിക്കുന്നു എന്ന് കാണിക്കുന്നു — FileAgent തന്റെ ഔട്ട്ഡട്ട് `fileContent` കീയുടെ കീഴിൽ എഴുതുന്നു, ReportAgent അത് വായിച്ച് സ്വന്തം ഔട്ട്പുട്ട് `report` കീയിൽ എഴുതി:

<img src="../../../translated_images/ml/agentic-scope.95ef488b6c1d02ef.webp" alt="Agentic Scope Shared Memory" width="800"/>

*Agentic Scope പങ്കുവെച്ച മെമ്മറിയായി പ്രവർത്തിക്കുന്നു — FileAgent `fileContent` എഴുതുന്നു, ReportAgent അത് വായിച്ച് `report` എഴുതുന്നു, നിങ്ങളുടെ കോഡ് അവസുതി ഫലം വായിക്കുന്നു.*

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // ഫയല്‍ ഏജന്റ് നിന്ന് റാവ് ഫയല്‍ ഡാറ്റ
String report = scope.readState("report");            // റിപ്പോര്ട്ട് ഏജന്റ് ല് നിന്നുള്ള ഘടനാപരമായ റിപ്പോര്ട്ട്
```

**Agent Listeners** ഏജന്റ് പ്രവർത്തനം നിരീക്ഷിക്കാനും ഡീബഗ് ചെയ്യാനും സഹായിക്കുന്നു. ഡെമോയിലുള്ള ഘട്ടംപ്രകാരം ഔട്ട്‌പുട്ട് ഓരോ ഏജന്റ് വിളിപ്പിച്ചുള്ള listener-ൽ നിന്നും പാകം ചെയ്യുന്നു:
- **beforeAgentInvocation** - Supervisor ഒരു ഏജന്റ് തെരഞ്ഞെടുത്തപ്പോൾ വിളിക്കും, ഏജന്റ് ഏതാണ് തെരഞ്ഞെടുത്തതെന്ന് നിങ്ങൾക്ക് കാണാൻ സഹായിക്കും
- **afterAgentInvocation** - ഏജന്റ് പൂർത്തിയാക്കിയ ശേഷം വിളിച്ചു, ഫലം കാണിക്കുന്നു
- **inheritedBySubagents** - യഥാർത്ഥമായാൽ, listener ഹൈക്കൂട്ടിലുള്ള എല്ലാ ഏജന്റുകളെയും നിരീക്ഷിക്കും

താഴെഡയഗ്രാം മുഴുവൻ Agent Listener ജീവിതചക്രം കാണിക്കുന്നു, ഏജന്റ് പ്രവർത്തനത്തിനിടെ `onError` എങ്ങനെ പരാജയം കൈകാര്യം ചെയ്യുന്നു എന്നതും ഉൾപ്പെടെ:

<img src="../../../translated_images/ml/agent-listeners.784bfc403c80ea13.webp" alt="Agent Listeners Lifecycle" width="800"/>

*Agent Listeners പ്രവർത്തന ജീവിതചക്രത്തിൽ ഹുക്കു ചെയ്യുന്നു — ഏജന്റ് ആരംഭിക്കുമ്പോൾ, പൂർത്തിയാകുമ്പോൾ, പിശക് നേരിടുമ്പോൾ നിരീക്ഷണം.*

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // എല്ലാ സബ്-എജന്റുകളിലേക്കും പരത്തുക
    }
};
```

Supervisor പാറ്റേണിനു മുകളില്‍, `langchain4j-agentic` മഡ്യൂള്‍ പല ശക്തിയുള്ള workflow പാറ്റേണുകൾ നൽകുന്നു. താഴെയുള്ള ഡയഗ്രാം എവിടെനിന്നും എവിടെനിലവരെ അഞ്ച് workflow പാറ്റേണുകൾ കാണിക്കുന്നു — ലളിതമായ ക്രമവിന്യാസ പൈപ്പ്‌ലൈൻമുതൽ മനുഷ്യൻ ഉൾപ്പെടുന്ന അനുമതി പ്രവർത്തിപാടുകൾവരെ:

<img src="../../../translated_images/ml/workflow-patterns.82b2cc5b0c5edb22.webp" alt="Agent Workflow Patterns" width="800"/>

*ഏജന്റ് നിയന്ത്രിക്കുന്ന workflow പാറ്റേണുകൾ അഞ്ചാം തരത്തിൽ — ലളിതമായ ശ്രേണി പൈപ്പ്‌ലൈനുകളും മനുഷ്യനു ഉൾപ്പെട്ട അനുമതി പ്രവൃത്തിപടികളും.*

| പാറ്റേണിൽ | വിവരണം | ഉപയോഗ കേസ് |
|---------|-------------|----------|
| **Sequential** | ഏജന്റുകൾ ക്രമത്തിൽ നടപ്പിലാക്കുക, ഫലം അടുത്തതിൽ ഒഴുകുന്നു | പൈപ്പ്‌ലൈനുകൾ: ഗവേഷണ→വിശകലനം→റിപ്പോർട്ട് |
| **Parallel** | ഏജന്റുകൾ ഒരേ സമയം ഓടിക്കുക | സ്വതന്ത്ര ജോലികൾ: കാലാവസ്ഥ+വാർത്ത+സ്റ്റോക്ക് |
| **Loop** | നിബന്ധന പാലിക്കുമ്പോൾ വരെയാണിതരണം | ഗുണനിലവാര സ്കോറിങ്: സ്കോർ ≥ 0.8 വരെയുള്ള പരിഷ്കരണം |
| **Conditional** | നിബന്ധനകൾ അടിസ്ഥാനമാക്കി മാർഗ്ഗനിർദ്ദേശം | ταξീകരണം→വിശിഷ്ട ഏജന്റിലേക്കുള്ള മാർഗ് |
| **Human-in-the-Loop** | മനുഷ്യ പരിശോധന പോയിന്റുകൾ ചേർക്കുക | അനുമതി പ്രവൃത്തിപാട്, ഉള്ളടക്ക അവലോകനം |

## പ്രധാന ആശയങ്ങൾ

MCPയും ഏജന്റിക് മഡ്യൂളും പ്രായോഗികമായി നോക്കിയ കഴിഞ്ഞാൽ, ഓരോ സമീപനം എപ്പോഴാണ് ഉപയോഗിക്കുന്നതെന്ന് ചുരുക്കാം.

MCPയുടെ ഏറ്റവും വലിയ അനുകൂല്യം വളരുന്ന ഇക്കോസിസ്റ്റം ആണ്. താഴെയുള്ള ഡയഗ്രാം ഒരു ഏകസാർവ്വത്രിക പ്രോട്ടോക്കോൾ വഴി നിങ്ങളുടെ AI അപ്ലിക്കേഷൻ MCP സെർവറുകളുമായി എങ്ങനെ ബന്ധിപ്പിക്കുന്നുവെന്ന് കാണിക്കുന്നു — ഫയൽസിസ്റ്റം, ഡാറ്റാബേസ് ആക്‌സസ്, GitHub, ഇമെയിൽ, വെബ് സ്ക്രാപ്പിങ് തുടങ്ങിയവவരെയുള്ള:

<img src="../../../translated_images/ml/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="MCP Ecosystem" width="800"/>

*MCP ഏകസാർവ്വത്രിക പ്രോട്ടോക്കോൾ ഇക്കോസിസ്റ്റം സൃഷ്ടിക്കുന്നു — MCP-സെർവർ MCP-ക്ലയന്റ് ഒപ്പം പ്രവർത്തിക്കുന്നു, ടൂൾ പങ്കിടൽ സൗകര്യം.*

**MCP** നിലവിലുള്ള ടൂൾ ഇക്കോസിസ്റ്റങ്ങൾ പ്രയോജനപ്പെടുത്തുമ്പോൾ, ഒന്നിലധികം ആപ്ലിക്കേഷനുകൾ പങ്കുവെക്കാവുന്ന ടൂളുകൾ നിർമ്മിക്കുമ്പോൾ, മൂന്നാംകക്ഷി സേവനങ്ങൾ സ്റ്റാൻഡേർഡ് പ്രോട്ടോക്കോളൊടെ കൂട്ടിച്ചേർക്കുമ്പോൾ, കോഡ് മാറ്റാതെ ടൂൾ നടപ്പിലാക്കലുകൾ മാറുമ്പോൾ അനുയോജ്യമാണ്.

**Agentic Module** `@Agent` അനോട്ടേഷനുകൾ ഉപയോഗിച്ച് പ്രഖ്യാപനാത്മക ഏജന്റ് നിർവചനങ്ങൾ, workflow ഓർക്കെസ്‌ട്രേഷൻ (ക്രമം, ലൂപ്പ്, സമാന്തരം), ഇൻറർഫേസ് അടിസ്ഥാന ആകൃതിയിലുള്ള ഏജന്റ് ഡിസൈൻ, outputKey ഉപയോഗിച്ച് പങ്കുവെച്ച ഫലങ്ങൾ ചേർക്കുമ്പോൾ ആദർശമാണ്.

**Supervisor Agent പാറ്റേൺ** പ്രവചനീയമല്ലാത്ത workflowകളിൽ LLM തീരുമാനമെടുക്കണം, അനേകം പ്രത്യേകതകളുള്ള ഏജന്റുകൾക്ക് ഡൈനമിക് ഒർക്കസ്ട്രേഷൻ വേണം, സംഭാഷണ സംവിധാനങ്ങൾ വ്യത്യസ്ത കഴിവുകൾക്കായി റൂട്ടുചെയ്യണം, ഏറ്റവും നവീനവും അനായാസവും ആയ ഏജന്റ് പെരുമാറ്റം ആവശ്യപ്പെടുമ്പോൾ നന്നാണ്.

Module 04-ലെ കസ്റ്റം `@Tool` മാർഗങ്ങളും MCP ടൂളുകളും തമ്മിലുള്ള മെച്ചപ്പെട്ട വ്യത്യാസം മനസ്സിലാക്കാൻ താഴെയുള്ള തുലന കാഴ്‌ച സഹായകരമാണ് — കസ്റ്റം ടൂളുകൾ ആപ്പ്-നിർദ്ദിഷ്ട ലൊജിക്കിന് പൂർണ്ണ ടൈപ്പ് സുരക്ഷയും കർശന ബന്ധവും നൽകുന്നു, MCP ടൂളുകൾ സാങ്കേതികമായി സ്റ്റാൻഡേർഡായി വീണ്ടും ഉപയോഗയോഗ്യമായ ഇന്റഗ്രേഷനുകളാണ്:

<img src="../../../translated_images/ml/custom-vs-mcp-tools.c4f9b6b1cb65d8a1.webp" alt="Custom Tools vs MCP Tools" width="800"/>

*കസ്റ്റം @Tool മാർഗുകൾ എപ്പോൾ ഉപയോഗിക്കണം, MCP ടൂളുകൾ എപ്പോൾ ഉപയോഗിക്കണം — കസ്റ്റം ടൂളുകൾ ആപ്പ്-നിർദ്ദിഷ്ട ലൊജിക്കിനും ടൈപ്പ് സുരക്ഷക്കും, MCP ടൂളുകൾ ശക്തമായ സ്റ്റാൻഡേർഡ് ഇന്റഗ്രേഷനുകൾക്കായി, ആപ്പുകൾക്കിടയിൽ പ്രവർത്തനക്ഷമം.*
