## നിങ്ങൾ പഠിക്കുന്നതെ എന്തൊക്കെയാണ്

നിങ്ങൾ സംഭാഷണ AI നിർമ്മിച്ച്, പ്രോംപ്റ്റുകൾ സുലഭമാക്കി, ഡോക്യുമെന്റുകളിൽ നിന്നുള്ള മറുപടികൾ ഉറപ്പാക്കി, ടൂളുകളുള്ള ഏജന്റുകൾ സൃഷ്ടിച്ചിട്ടുണ്ട്. പക്ഷെ ആ എല്ലാ ടൂളുകളും നിങ്ങളുടെ പ്രത്യേക പ്രയോഗത്തിന് വേണ്ടി കസ്റ്റം-ബിൽറ്റ് ചെയ്തവയാണ്. നിങ്ങൾക്ക് നിങ്ങളുടെ AIക്ക് ഒരു സ്റ്റാൻഡേർഡൈസ്ഡ് ടൂൾ ഇക്കോസിസ്റ്റത്തിലുള്ള പ്രവേശനം നൽകാൻ കഴിയും എങ്കിൽ? ഈ മൊഡ്യൂളിൽ, നിങ്ങൾ അത് എങ്ങനെ ചെയ്യാമെന്ന് കാണിക്കും - മോഡൽ കണ്ടക്റ്റ് പ്രോട്ടോകോൾ (MCP)യും LangChain4jയുടെ ഏജന്റിക് മോഡ്യൂളും ഉപയോഗിച്ച്. ആദ്യം ഒരു ലളിതമായ MCP ഫയൽ റീഡർ കാണിക്കും, പിന്നെ അത് എങ്ങനെ എളുപ്പത്തിൽ പാർപ്പിക്കൽ ഏജന്റിക് വർکش്ഫ്ലോയിൽ ഉൾപ്പെടുത്താമെന്ന് സൂപ്പർവൈസർ ഏജന്റ് പാറ്റേണിലൂടെ കാണിക്കും.

## MCP എന്നത് എന്താണ്?

മോഡൽ കണ്ടക്റ്റ് പ്രോട്ടോകോൾ (MCP) AI പ്രയോഗങ്ങൾക്ക് ബഹിരാകാശ ടൂളുകൾ കണ്ടെത്താനും ഉപയോഗിക്കാനും സാധിക്കുന്ന ഒരു സ്റ്റാൻഡർഡ് മാർഗമാണ്. ഓരോ ഡാറ്റാ സ്രോതസ്സിനും സർവീസിനും കസ്റ്റം ഇന്റഗ്രേഷനുകൾ എഴുതുന്നതിനുപകരം, MCP സർവറുകൾ സ്റ്റാൻഡേർഡ് ഫോർമാറ്റിൽ ശേഷവുള്ള ശേഷികൾ പ്രദർശിപ്പിക്കുന്നു. നിങ്ങളുടെ AI ഏജന്റ് ആ ടൂളുകൾ സ്വയം കണ്ടെത്തി ഉപയോഗിക്കാം.

താഴെയുള്ള ചിത്രത്തിലൂടെ MCP ഇല്ലാത്തപ്പോൾ ഓരോ ഇന്റഗ്രേഷനും കസ്റ്റം പോയിന്റ്-ടു-പോയിന്റ് വയറിങ് ആവശ്യമാണ് എങ്കിൽ MCP ഉപയോഗിച്ചാൽ ഒരു പ്രോട്ടോകോൾ നിങ്ങളുടെ ആപ്പ് ഒരു ടൂളിലേക്ക് ബന്ധിപ്പിക്കുന്നു എന്ന് കാണിക്കുന്നു:

<img src="../../../translated_images/ml/mcp-comparison.9129a881ecf10ff5.webp" alt="MCP Comparison" width="800"/>

*എം‌സിപി മുമ്പ്: സങ്കീർണ്ണമായ പോയിന്റ്-ടു-പോയിന്റ് ഇന്റഗ്രേഷനുകൾ. MCP ശേഷം: ഒറ്റ പ്രോട്ടോകോൾ, അതിമനോഹരമായ സാദ്ധ്യതകൾ.*

MCP AI വികസനത്തിൽ ഒരു അടിസ്ഥാന പ്രശ്നം പരിഹരിക്കുന്നു: ഓരോ ഇന്റഗ്രേഷനും കസ്റ്റം ആണ്. GitHub ആക്സസ് ചെയ്യണമോ? കസ്റ്റം കോഡ്. ഫയലുകൾ വായിക്കണമോ? കസ്റ്റം കോഡ്. ഡാറ്റാബേസ് ക്വറി ചെയ്യണമോ? കസ്റ്റം കോഡ്. ഈ കളിൽ ഒന്ന് പോലും മറ്റേതും AI പ്രയോഗങ്ങളുമായി പൊരുത്തപ്പെടുന്നില്ല.

MCP ഇത് സ്റ്റാൻഡർഡൈസ് ചെയ്യുന്നു. MCP സർവർ ടൂളുകൾ വ്യക്തമായ വിവരണങ്ങളോടും സ്കീമകളോടും സഹ വഹിക്കുന്നു. ഏതെങ്കിലും MCP ക്ലയന്റ് ബന്ധിപ്പിച്ച് ലഭ്യമായ ടൂളുകൾ കണ്ടെത്തുകയും ഉപയോഗിക്കുകയും ചെയ്യാം. ഒറ്റ തവണ നിർമ്മിച്ച് എല്ലായിടത്തും ഉപയോഗിക്കുക.

തുടർന്ന് ചിത്രത്തിൽ ഈ ആർക്കിടെക്ചർ കാണിക്കുന്നു — ഒരു MCP ക്ലയന്റ് (നിങ്ങളുടെ AI പ്രയോഗം) ഇതിലൊരു സ്റ്റാൻഡേർഡ് പ്രോട്ടോക്കോൾ വഴി പല MCP സർവറുകളോടും ബന്ധിപ്പിക്കുന്നു, ഓരോതും തങ്ങളുടെ ടൂളുകളുടെ സെറ്റ് പുറപ്പെടുവിക്കുന്നു:

<img src="../../../translated_images/ml/mcp-architecture.b3156d787a4ceac9.webp" alt="MCP Architecture" width="800"/>

*മോഡൽ കണ്ടക്റ്റ് പ്രോട്ടോകോൾ ആർക്കിടെക്ചർ - സ്റ്റാണ്ടർഡൈസ്ഡ് ടൂൾ കണ്ടെത്തൽ, നിർവ്വഹണം*

## MCP എങ്ങനെ പ്രവർത്തിക്കുന്നു

അടിയിൽ MCP ഒരു ലെയേർഡ് ആർക്കിടെക്ചർ ഉപയോഗിക്കുന്നു. നിങ്ങളുടെ ജാവ ആപ്പ്ലിക്കേഷൻ (MCP ക്ലയന്റ്) ലഭ്യമായ ടൂളുകൾ കണ്ടെത്തി JSON-RPC അഭ്യർത്ഥനകൾ ട്രാൻസ്പോർട്ട് ലെയർ (Stdio അല്ലെങ്കിൽ HTTP) വഴി അയയ്ക്കുന്നു, MCP സർവർ ഓപ്പറേഷനുകൾ നിർവ്വഹിച്ച് ഫലം നൽകി. ഈ പ്രോട്ടോക്കോൾ ഓരോ ലെയേറും താഴെ കാണാം:

<img src="../../../translated_images/ml/mcp-protocol-detail.01204e056f45308b.webp" alt="MCP Protocol Detail" width="800"/>

*MCP അടിയിൽ എങ്ങനെ പ്രവർത്തിക്കുന്നു — ക്ലയന്റുകൾ ടൂളുകൾ കണ്ടെത്തുന്നു, JSON-RPC സന്ദേശങ്ങൾ കൈമാറ്റം ചെയ്യുന്നു, ട്രാൻസ്പോർട്ട് ലെയർ വഴി ഓപ്പറേഷനുകൾ നിർവ്വഹിക്കുന്നു.*

**സർവർ-ക്ലയന്റ് ആർക്കിടെക്ചർ**

MCP ക്ലയന്റ്-സർവർ മോഡൽ ഉപയോഗിക്കുന്നു. സർവർ ടൂളുകൾ നൽകുന്നു - ഫയലുകൾ വായിക്കൽ, ഡാറ്റാബേസ് ക്വറി, API കോളുകൾ. ക്ലയന്റ് (നിങ്ങളുടെ AI) സർവറുകളോട് ബന്ധിച്ചു അവരുടെ ടൂൾസ് ഉപയോഗിക്കുന്നു.

LangChain4j-യുമായി MCP ഉപയോഗിക്കാൻ ഈ Maven ഡിപെൻഡൻസി ചേർക്കുക:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```

**ടൂൾ കണ്ടെത്തൽ**

നിങ്ങളുടെ ക്ലയന്റ് MCP സർവറുമായി കണക്റ്റ് ചെയ്യുമ്പോൾ, "ഇവിടെ എത്ര ടൂളുകളുണ്ട്?" എന്ന് ചോദിക്കുന്നു. സർവർ ലഭ്യമായ ടൂളുകളുടെ ലിസ്റ്റ് വിവരണങ്ങളോടും പാരാമീറ്റർ സ്കീമകളോടും നൽകുന്നു. നിങ്ങളുടെ AI ഏജന്റ് ഉപയോഗിക്കുന്ന ടൂൾസ് തീരുമാനിക്കും. താഴെയുള്ള ചിത്രത്തിൽ ഹെൻഡ്ഷേക്ക് കാണിക്കുന്നു — ക്ലയന്റ് `tools/list` അഭ്യർത്ഥന അയയ്ക്കുന്നു, സർവർ ടൂളുകൾ വിശദീകരണ സഹിതം നൽകുന്നു:

<img src="../../../translated_images/ml/tool-discovery.07760a8a301a7832.webp" alt="MCP Tool Discovery" width="800"/>

*AI ലഭ്യമായ ടൂളുകൾ സ്റ്റാർട്ടപ്പ്-ൽ കണ്ടെത്തുന്നു — ഇപ്പോള്‍ ലഭ്യമായ ശേഷികളറിയാം, ഉപയോഗിക്കാൻ എതു തിരഞ്ഞെടുത്ത് തീരുമാനിക്കാം.*

**ട്രാൻസ്പോർട്ട് മെക്കാനിസങ്ങൾ**

MCP വ്യത്യസ്ത ട്രാൻസ്പോർട്ട് മാർഗങ്ങൾ പിന്തുണയ്ക്കുന്നു. രണ്ട് ഓപ്ഷനുകൾ: Stdio (ലോകൽ subprocess കമ്മ്യൂണിക്കേഷൻ) & Streamable HTTP (റിമോട്ട് സർവറുകൾ). ഈ മോഡ്യൂളിൽ Stdio ട്രാൻസ്പോർട്ട് കാണിക്കുന്നു:

<img src="../../../translated_images/ml/transport-mechanisms.2791ba7ee93cf020.webp" alt="Transport Mechanisms" width="800"/>

*MCP ട്രാൻസ്പോർട്ട് മാർഗങ്ങൾ: റിമോട്ട് സർവറുകൾക്ക് HTTP, ലോക്കൽ പ്രോസസുകൾക്ക് Stdio*

**സ്റ്റ്ഡിയോ** - [StdioTransportDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java)

ലോക്കൽ പ്രോസസ്സുകൾക്കായി. നിങ്ങളുടെ ആപ്പ് ഒരു സർവർ subprocess ആയി സഞ്ചിയ്ക്കുകയും സ്റ്റാൻഡേർഡ് ഇൻപ്പുട്ട്/ഔട്ട്പുട്ട് വഴി ആശയവിനിമയം നടത്തുകയും ചെയ്യുന്നു. ഫയൽസിസ്റ്റം ആക്സസ് ചെയ്യാൻ അല്ലെങ്കിൽ കമാൻഡ് ലൈൻ ടൂളുകൾക്ക് ഉപയോഗപ്രദം.

```java
McpTransport stdioTransport = new StdioMcpTransport.Builder()
    .command(List.of(
        npmCmd, "exec",
        "@modelcontextprotocol/server-filesystem@2025.12.18",
        resourcesDir
    ))
    .logEvents(false)
    .build();
```

`@modelcontextprotocol/server-filesystem` സർവർ നിങ്ങൾ നൽകിയ ഡയറക്ടറികളിൽ മാത്രം പ്രവർത്തിക്കുന്ന ടൂളുകൾ ഇവിടെ കാണിക്കുന്നു:

| ടൂൾ | വിവരണം |
|------|---------|
| `read_file` | ഒറ്റ ഫയലിന്റെ ഉള്ളടക്കം വായിക്കുക |
| `read_multiple_files` | ഒരുമിച്ച് പല ഫയലുകളും വായിക്കുക |
| `write_file` | ഫയൽ സൃഷ്ടിക്കുക അല്ലെങ്കിൽ മായ്ക്കുക |
| `edit_file` | ലക്ഷ്യബദ്ധമായ ഫൈൻഡ് ആൻഡ് റെപ്ലേസ് എഡിറ്റുകൾ നടത്തുക |
| `list_directory` | ഒരു പാതയിലെ ഫയലുകളും ഡയറക്ടറികളും ലിസ്റ്റ് ചെയ്യുക |
| `search_files` | ഒരു പാറ്റേൺ പൊരുത്തപ്പെടുന്ന ഫയലുകൾ റികർശൻ ആയി തിരയുക |
| `get_file_info` | ഫയൽ മെടാദത്ത ( വലിപ്പം, ടൈംസ്റ്റാമ്പുകൾ, അനുമതികൾ) ലഭിക്കുക |
| `create_directory` | ഡയറക്ടറി സൃഷ്ടിക്കുക (പാരന്റ് ഡയറക്ടറുകൾ ഉൾപ്പെടെ) |
| `move_file` | ഫയൽ അല്ലെങ്കിൽ ഡയറക്ടറി മാറ്റുക അല്ലെങ്കിൽ പേരുമാറ്റം നടത്തുക |

താഴെയുള്ള ചിത്രം കാണിക്കുന്നു സ്റ്റ്ഡിയോ ട്രാൻസ്പോർട്ട് എങ്ങനെ എക്സിക്യൂഷൻ സമയത്ത് പ്രവർത്തിക്കുന്നു — നിങ്ങളുടെ ജാവ ആപ്പ് MCP സർവർ subprocess ആയി സ്പോൺ ചെയ്യുകയും stdin/stdout പൈപ്പിലൂടെ ആശയവിനിമയം നടത്തുകയും ചെയ്യുന്നു, നെറ്റ്വർക് അല്ലെങ്കിൽ HTTP ഇല്ല:

<img src="../../../translated_images/ml/stdio-transport-flow.45eaff4af2d81db4.webp" alt="Stdio Transport Flow" width="800"/>

*സ്റ്റ്ഡിയോ ട്രാൻസ്പോർട്ട് പ്രവർത്തനത്തിൽ — നിങ്ങളുടെ ആപ്പ് MCP സർവർ subprocess ആയി സ്പോൺ ചെയ്ത് stdin/stdout പൈപ്പിലൂടെ ആശയവിനിമയം നടത്തുന്നു.*

> - "സ്റ്റ്ഡിയോ ട്രാൻസ്പോർട്ട് എങ്ങനെ പ്രവർത്തിക്കുന്നു, HTTP യുമായി എപ്പോൾ ഉപയോഗിക്കണം?"
> - "LangChain4j എങ്ങനെ MCP സർവർ പ്രോസസുകളുടെ ലൈഫ് സൈക്കിൾ മാനജ് ചെയ്യുന്നു?"
> - "AI-ക്ക് ഫയൽ സിസ്റ്റം ആക്സസ് നൽകുന്നതിന്റെ സുരക്ഷാ ഫലങ്ങൾ എന്തെല്ലാം?"

## എജന്റിക് മോഡ്യൂൾ

MCP സ്റ്റാൻഡേർഡൈസ് ചെയ്‌ത ടൂളുകൾ നൽകുമ്പോൾ, LangChain4jയുടെ **Agentic മോഡ്യൂൾ** ടൂളുകൾ നിയന്ത്രിക്കുന്ന ഏജന്റുകൾ നിർമാണത്തിനൊരു ഡിക്ലറേറ്റീവ് മാർഗം നൽകുന്നു. `@Agent` അനോട്ടേഷൻ, `AgenticServices` ഉപയോഗിച്ച് ഏജന്റ് പെരുമാറ്റം ഇന്റർഫേസുകൾ വഴിയാണ് നിർവ്വചിക്കുന്നത്, ആവശ്യമായ നിർദ്ദേശം എഴുതാതെ.

ഈ മോഡ്യൂളിൽ, നിങ്ങൾ **സൂപ്പർവൈസർ ഏജന്റ്** പാറ്റേൺ പഠിക്കുമെന്ന് ഉൾക്കൊള്ളുക — ഇത് ഒരു ആഡ്‌വാൻസ്ഡ് ഏജന്റിക് AI സമീപനം ആണ്, ഒരു "സൂപ്പർവൈസർ" ഏജന്റ്, ഉപ-ഏജന്റുകൾ ഒരുപാട് സ്വതന്ത്രമായി തിരഞ്ഞെടുക്കുന്നു ഉപഭോക്തൃ ആവശ്യകത പ്രകാരം. നാം രണ്ട് ആശയങ്ങളും സംയോജിപ്പിക്കും, ഒരു ഉപ-ഏജന്റിന് MCP-സാമർത്ഥ്യമുള്ള ഫയൽ ആക്‌സസ് കഴിവ് നൽകി.

എജന്റിക് മോഡ്യൂൾ ഉപയോഗിക്കാൻ ഈ Maven ഡിപെൻഡൻസി ചേർക്കുക:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```
> **ശ്രദ്ധ:** `langchain4j-agentic` മൊഡ്യൂളിന് `langchain4j.mcp.version` എന്ന വകതിരിവ് വേർഷൻ പ്രോപർട്ടി വേറിട്ടാണ് ഉപയോഗിക്കുന്നത്, കാരണം കോർ LangChain4j ലൈബ്രറികളിൽ നിന്നും വേറിട്ട റെലീസ് ഷെഡ്യൂളിലാണ്.

> **⚠️ പരീക്ഷണ ഘട്ടം:** `langchain4j-agentic` മൊഡ്യൂൾ **പരീക്ഷണ ഘട്ടത്തിലാണ്**. AI അസിസ്റ്റൻറുകൾ നിർമ്മിക്കുന്ന സ്ഥിരം മാർഗം ഇപ്പോഴും `langchain4j-core` കസ്റ്റം ടൂളുകളെന്നവയാണ് (Module 04).

## ക്വിക്ക് സ്റ്റാർട്ട്

**VS കോഡ് ഉപയോഗിച്ച്:** എക്സ്പ്ലോററിൽ ഏതെങ്കിലും ഡെമോ ഫയലിൽ റൈറ്റ് ക്ലിക്ക് ചെയ്ത് **"Run Java"** തിരഞ്ഞെടുക്കൂ, അല്ലെങ്കിൽ റൺ ആൻഡ് ഡീബഗ് പാനലിൽ നിന്നുള്ള ലോഞ്ച് കോൺഫിഗറേഷനുകൾ ഉപയോഗിക്കൂ (പ്രഥമമായി `.env` ഫയൽ ഏസ്യൂർ ക്രെഡൻഷ്യലുകളോടെ ക്രമീകരിച്ചിരിക്കണം).

**മേവൻ ഉപയോഗിച്ച്:** അല്ലെങ്കിൽ, താഴെയുള്ള ഉദാഹരണങ്ങളോടെ കമാൻഡ് ലൈൻ വഴി പ്രവർത്തിപ്പിക്കാം.

### ഫയൽ പ്രവർത്തനങ്ങൾ (സ്റ്റ്ഡിയോ)

ലോക്കൽ subprocess അടിസ്ഥാനത്തിലുള്ള ടൂളുകൾ കാണിക്കുന്നു.

**✅ മുന്‌പരിക്ക് വേണ്ട:** MCP സർവർ സ്വയം സ്പോൺ ചെയ്യുന്നു.

**സ്റ്റാർട്ട് സ്ക്രിപ്റ്റുകൾ ഉപയോഗിച്ച് (ശുപാർശ):**

സ്റ്റാർട്ട് സ്ക്രിപ്റ്റുകൾ റൂട്ട് `.env` ഫയലിൽ നിന്ന് പരിസ്ഥിതി ചോദ്യങ്ങൾ സ്വയം ലോഡ് ചെയ്യുന്നു:

**Bash:**
```bash
cd 05-mcp
chmod +x start-stdio.sh
./start-stdio.sh
```

**PowerShell:**
```powershell
cd 05-mcp
.\start-stdio.ps1
```

**VS കോഡ്:** `StdioTransportDemo.java` ഫയലിൽ റൈറ്റ് ക്ലിക്ക് ചെയ്ത് **"Run Java"** തിരഞ്ഞെടുക്കുക (`.env` ക്രമീകരിച്ചിട്ടുണ്ടെന്ന് ഉറപ്പാക്കുക).

ആപ്പ് സ്വയം ഫയൽസിസ്റ്റം MCP സർവർ സ്പോൺ ചെയ്ത്, ലോക്കൽ ഫയൽ വായിക്കും. subprocess മാനേജ്‌മെന്റ് എങ്ങനെ കൈകാര്യം ചെയ്യുന്നു ശ്രദ്ധിക്കൂ.

**പ്രതീക്ഷിക്കാവുന്ന ഔട്ട്പുട്ട്:**
```
Assistant response: The file provides an overview of LangChain4j, an open-source Java library
for integrating Large Language Models (LLMs) into Java applications...
```

### സൂപ്പർവൈസർ ഏജന്റ്

**സൂപ്പർവൈസർ ഏജന്റ് പാറ്റേൺ** ഒരു **പ്രവർത്തനക്ഷമമായ** ഏജന്റിക് AI രൂപമാണ്. ഒരു സൂപ്പർവൈസർ LLM ഉപയോഗിച്ച് ഉപഭോക്താവിന്റെ അഭ്യർത്ഥനയുടെ അടിസ്ഥാനത്തിൽ ഏജന്റുകൾ സ്വയം തിരഞ്ഞെടുക്കുന്നു. അടുത്ത ഉദാഹരണത്തിൽ, MCP-സഹായത്തോടെ ഫയൽ ഡാറ്റ ആക്‌സസ് ചെയ്യുന്നതും LLM ഏജന്റ് കൊണ്ട് രചിച്ച റിപ്പോർട്ട് നിർമ്മാണപ്രക്രിയയുമടങ്ങുന്ന സൂപ്പർവൈസർ ഫയൽ റീഡ് → റിപ്പോർട്ട് വര്‍ക്ക്ഫ്‌ളോ കാണിക്കും.

ഡെമോയിൽ, `FileAgent` MCP ഫയൽസിസ്റ്റം ടൂളുകൾ ഉപയോഗിച്ച് ഫയൽ വായിക്കുകയും, `ReportAgent` ഒരു ഘടനയോടുകൂടിയ റിപ്പോർട്ട് (1 വാക്യ എക്സിക്യൂട്ടീവ് സംഗ്രഹം, 3 പ്രധാന കുറിപ്പുകൾ, ഉപദേശങ്ങൾ) സൃഷ്ടിക്കുകയും ചെയ്യുന്നു. സൂപ്പർവൈസർ ഈ പ്രവാഹം സ്വയം നിയന്ത്രിക്കുന്നു:

<img src="../../../translated_images/ml/supervisor-agent-pattern.06275a41ae006ac8.webp" alt="Supervisor Agent Pattern" width="800"/>

*സൂപ്പർവൈസർ LLM ഉപയോഗിച്ച് ഏജന്റുകൾ തിരഞ്ഞെടുക്കുന്നു, നിർദ്ദേശിക്കപ്പെടുന്ന അനുസൃതമോ ചെയ്‌തുതന്നെ — ഹാർഡ്‌കോഡ് ചെയ്ത റൂട്ടിംഗ് ആവശ്യമില്ല.*

ഫയൽ മുതൽ റിപ്പോർട്ട് വരെ പൈപ്പ്ലൈനിന്റെ കങ്ക്രീറ്റ് വർക്ഫ്ലോ ഇങ്ങനെ കാണാം:

<img src="../../../translated_images/ml/file-report-workflow.649bb7a896800de9.webp" alt="File to Report Workflow" width="800"/>

*FileAgent MCP ടൂളുകൾ വഴി ഫയൽ വായിക്കുകയും, ReportAgent അതിന്റെ ഉണരുന്ന ഉള്ളടക്കം ഘടനയുള്ള റിപ്പോർട്ടായി മാറ്റുകയും ചെയ്യുന്നു.*

താഴെയുള്ള സീക്വൻസ് ഡയഗ്രാം മുഴുവൻ സൂപ്പർവൈസർ ഓർക്കസ്ട്രേഷന്റെ പാത കാണിക്കുന്നു — MCP സർവർ സ്പോൺ, അനേകമുള്ള സ്വതന്ത്ര ഏജന്റുകളുടെ തിരഞ്ഞെടുപ്പ്, സ്റ്റ്ഡിയോ വഴി ടൂൾ കോളുകൾ, അന്തിമ റിപ്പോർട്ട്:

<img src="../../../translated_images/ml/supervisor-agent-sequence.1aa389b3bef99956.webp" alt="Supervisor Agent Sequence Diagram" width="800"/>

*സൂപ്പർവൈസർ സ്വയം FileAgent (MCP സർവറെ സ്റ്റ്ഡിയോ വഴി വിളിച്ച് ഫയൽ വായിക്കുന്നു), പിന്നെ ReportAgent (ഘടനയുള്ള റിപ്പോർട്ട് സൃഷ്ടിക്കുന്നു) പ്രവർത്തനത്തിനായി വിളിക്കുന്നു — ഓരോ ഏജന്റും ഔട്ട്പുട്ട് Agentic സ്വേച്ഛാസ്മൃതിയിലേയ്ക്ക് സംഭരിച്ചു.*

ഓരോ ഏജന്റ് ഔട്ട്പുട്ട് **Agentic Scope** എന്ന (പങ്കിട്ട ഓർമ്മ) സംഭരിക്കുന്നു, ഇത് അടുത്തുള്ള ഏജന്റുകൾക്ക് മുൻവത് ഫലങ്ങൾ ഉപയോഗിക്കാൻ സഹായിക്കുന്നു. MCP ടൂളുകൾ എജന്റിക് വർക്ക്‌ഫ്ലോകളിൽ എങ്ങനെ സംഗീതപൂർവ്വം ചേർക്കുന്നു എന്നതിന്റെ ഉദാഹരണമാണ് — സൂപ്പർവൈസർ എങ്ങനെ ഫയലുകൾ എങ്ങനെ വായിക്കപ്പെടും എന്ന് അറിയേണ്ടതില്ല, പകരംFileAgent ചെയ്യുമെന്ന് മതിയാകും.

#### ഡെമോ പ്രവർത്തിപ്പിക്കൽ

സ്റ്റാർട്ട് സ്ക്രിപ്റ്റുകൾ സ്വയം റൂട്ട് `.env` ഫയലിൽ നിന്ന് പരിസ്ഥിതി ചോദ്യങ്ങൾ ലോഡ് ചെയ്യുന്നു:

**Bash:**
```bash
cd 05-mcp
chmod +x start-supervisor.sh
./start-supervisor.sh
```

**PowerShell:**
```powershell
cd 05-mcp
.\start-supervisor.ps1
```

**VS കോഡ്:** `SupervisorAgentDemo.java` ൽ റൈറ്റ് ക്ലിക്ക് ചെയ്ത് **"Run Java"** തിരഞ്ഞെടുക്കുക (`.env` ക്രമീകരിച്ചിട്ടുണ്ടെന്ന് ഉറപ്പാക്കുക).

#### സൂപ്പർവൈസർ എങ്ങനെ പ്രവർത്തിക്കുന്നു

ഏജന്റുകൾ നിർമ്മിക്കുന്നതിന് മുമ്പ്, MCP ട്രാൻസ്പോർട്ട് ഒരു ക്ലയന്റ് ആയി ബന്ധിപ്പിച്ച് `ToolProvider` ആയി ഘടിപ്പിക്കണം. ഇതാണ് MCP സർവറിലെ ടൂളുകൾ ഏജന്റ് ഉപയോഗത്തിന് ലഭ്യമാകുന്നത്:

```java
// ട്രാൻസ്പോർട്ട് നിന്ന് MCP ക്ലയന്റ് സൃഷ്ടിക്കുക
McpClient mcpClient = new DefaultMcpClient.Builder()
        .transport(stdioTransport)
        .build();

// ക്ലയന്റ് ഒരു ടൂൾപ്രൊവൈഡറായി അരപ്പിക്കുക — ഇത് MCP ടൂളുകൾ LangChain4j-ലേക്ക് ബന്ധിപ്പിക്കുന്നു
ToolProvider mcpToolProvider = McpToolProvider.builder()
        .mcpClients(List.of(mcpClient))
        .build();
```

ഇപ്പോൾ നിങ്ങൾക്ക് `mcpToolProvider` ഏജന്റുകളിൽ എവിടെയും ഇഞ്ചെക്ട് ചെയ്യാം:

```java
// പടി 1: ഫയൽ ഏജന്റ് MCP ഉപകരണങ്ങൾ ഉപയോഗിച്ച് ഫയലുകൾ വായിക്കുന്നു
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // ഫയൽ പ്രവർത്തനങ്ങൾക്ക് MCP ഉപകരണങ്ങൾ ഉണ്ട്
        .build();

// പടി 2: റിപ്പോർട്ട് ഏജന്റ് ഘടനാപരമായ റിപ്പോർട്ടുകൾ സൃഷ്ടിക്കുന്നു
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// സൂപർവൈസർ ഫയൽ → റിപ്പോർട്ട് പ്രവാഹം ഏകോപിപ്പിക്കുന്നു
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // അന്തിമ റിപ്പോർട്ട് തിരികെ നൽകുക
        .build();

// അഭ്യർത്ഥനയുടെ അടിസ്ഥാനത്തിൽ ഏജന്റുമാരെ സൂപർവൈസർ തീരുമാനിക്കുന്നു
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```

#### FileAgent എങ്ങനെ MCP ടൂളുകൾ റൺടൈമിൽ കണ്ടെത്തുന്നു

നിങ്ങൾ ആശ്ചര്യപ്പെടാൻ സാധ്യതയുണ്ട്: **`FileAgent` npm ഫയൽസിസ്റ്റം ടൂളുകൾ എങ്ങനെ ഉപയോഗിക്കുമെന്ന് എങ്ങനെ അറിയുന്നു?** ഉത്തരമാണ് ഇത് അറിയില്ല — **LLM** ടൂൾ സ്കീമുകൾ വഴി റൺടൈമിൽ കണ്ടെത്തുന്നു.

`FileAgent` ഇന്റർഫേസ് ഒരു **പ്രോംപ്റ്റ് നിർവചനമേ ആണ്**. അതിന് `read_file`, `list_directory`, അല്ലെങ്കിൽ മറ്റേതെങ്കിലും MCP ടൂളിന് സ്‌പെസിഫിക് അറിവ് ഇല്ല. സംഭവിക്കുന്നത് എന്തെന്നാൽ:
1. **സർവർ സ്‌പോൺസ്:** `StdioMcpTransport` ബാല പ്രക്രിയയായി `@modelcontextprotocol/server-filesystem` npm പാക്കേജ് ആരംഭിക്കുന്നു  
2. **ടൂൾ കണ്ടെത്തൽ:** `McpClient` സർവറിലേക്ക് `tools/list` JSON-RPC അഭ്യർത്ഥന അയയ്ക്കുന്നു, സർവർ ടൂളുകളുടെ പേരുകൾ, വിവരണങ്ങൾ, പാരാമീറ്റർ സ്കീമകൾ (ഉദാഹരണം, `read_file` — *"ഒരു ഫയലിന്റെ മുഴുവൻ ഉള്ളടക്കം വായിക്കുക"* — `{ path: string }`) നൽകുന്നു  
3. **സ്കീമ ഇൻജക്ഷൻ:** `McpToolProvider` കണ്ടെത്തിയ ഈ സ്കീമകൾ പൊതുവാക്കുകയും LangChain4j-യിൽ ലഭ്യമാക്കുകയും ചെയ്യുന്നു  
4. **LLM തീരുമാനിക്കുന്നു:** `FileAgent.readFile(path)` കോളായിപ്പിക്കുമ്പോൾ LangChain4j സിസ്റ്റം സന്ദേശവും, ഉപയോക്തൃ സന്ദേശവും, **ടൂൾ സ്കീമകളുടെ പട്ടികയും** LLM-നിന്നും അയയ്ക്കുന്നു. LLM ടൂൾ വിവരണങ്ങൾ വായിച്ച് ടൂൾ കോൾ (ഉദാഹരണം, `read_file(path="/some/file.txt")`) സൃഷ്ടിക്കുന്നു  
5. **നിറവേറ്റി:** LangChain4j ടൂൾ കോൾ തടഞ്ഞ് MCP ക്ലയന്റിലൂടെ നോഡ്.js subprocess-ലേക്ക് അയച്ചുതരുന്നു, ഫലം ലഭിച്ച് LLM-ന് തിരിച്ചു നൽകുന്നു  

ഇതു മുകളിലെ [ടൂൾ കണ്ടെത്തൽ](../../../05-mcp) മെക്കാനിസത്തിന്റെ സമാനമാണ്, എന്നാൽ ഏജന്റ് വർക്ഫ്ലോയ്‌ക്കായി പ്രത്യേകിച്ച് പ്രയോഗിക്കുന്നു. `@SystemMessage` ഉം `@UserMessage` ഉം എൻഡിഎൽഎം പെരുമാറ്റം മാർഗ്ഗനിർദ്ദേശം ചെയ്യുവാനും, ഇഞ്ചക്റ്റ് ചെയ്ത `ToolProvider` എൽഎൽഎമ്മിന് **ശേഷതകൾ** നൽകുവാനും സഹായിക്കുന്നു — എൽഎല്ലഎം റൺടൈമിൽ ഇരുവേറെ മികവുറ്റും ചേർത്ത് പ്രവർത്തിക്കുന്നു.

#### പ്രതികരണ തന്ത്രങ്ങൾ  

`SupervisorAgent` കോൺഫിഗർ ചെയ്യുമ്പോൾ, ഉപയോക്താവിന് വിശേഷിപ്പിച്ച ഉപ ഏജന്റുകൾ അവരുടെ പ്രവർത്തനം പൂർത്തിയാക്കിയതിന് ശേഷം അവസാന ഉത്തരമുപയോഗിച്ച് എങ്ങനെ ഫോർമുലേറ്റ് നടത്തണമെന്ന് നിർദ്ദേശിക്കുന്നു. താഴെ കാണുന്ന ചിത്രം മൂന്ന് ലഭ്യമായ തന്ത്രങ്ങൾ കാണിക്കുന്നു — LAST അവസാന ഏജന്റിന്റെ ഔട്ട്‌പുട്ട് നേരിട്ട് തിരികെ തരുന്നു, SUMMARY എല്ലാ ഔട്ട്‌പുട്ടുകളും LLM ഉപയോഗിച്ച് സംക്ഷേപിക്കുന്നു, SCORED മുകളിൽ വാലിലേക്ക് സ്കോർ കിട്ടിയതു തിരഞ്ഞെടുത്ത് തിരികെ നൽകുന്നു:  

<img src="../../../translated_images/ml/response-strategies.3d0cea19d096bdf9.webp" alt="Response Strategies" width="800"/>  

*സൂപ്പർവൈസർ അവസാന പ്രതികരണം എങ്ങനെ നിർമ്മിക്കുന്നതിന്റെ മൂന്ന് തന്ത്രങ്ങൾ — അവസാന ഏജന്റിന്റെ ഔട്ട്‌പുട്ടോ, സംക്ഷേപത്തോ, ഉത്തമ സ്കോർ ലഭിച്ചതോ തിരഞ്ഞെടുക്കുക.*  

ലഭ്യമായ തന്ത്രങ്ങൾ:  

| തന്ത്രം | വിവരണം |  
|----------|-------------|  
| **LAST** | സൂപ്പർവൈസർ അവസാന ഉപ ഏജന്റ് അല്ലെങ്കിൽ ടൂൾ കോൾ ചെയ്ത് ലഭ്യമായ ഫലം തിരികെ നൽകുന്നു. പ്രവർത്തിയുടെ അവസാന ഏജന്റ് പ്രത്യേകിച്ച് പൂർണ്ണ, അന്തിമ ഉത്തരം നൽകാൻ രൂപകൽപ്പന ചെയ്തിരിക്കുമ്പോൾ ഇത് പ്രയോജനപ്രദമാണ് (ഉദാഹരണം, ഗവേഷണ പൈപ്ലൈനിലുള്ള "സംക്ഷേപ ഏജന്റ്"). |  
| **SUMMARY** | സൂപ്പർവൈസറിന് സ്വന്തം പ്രത്യന്തരഭാഷാനുബന്ധ മോഡൽ (LLM) ഉപയോഗിച്ച് മുഴുവൻ ഇടപെടലിന്റെയും ഉപ ഏജന്റുകളുടെ ഔട്ട്‌പുട്ടുകളുടെ സംക്ഷേപം സൃഷ്ടിക്കുകയും ആ സംക്ഷേപം അന്തിമ പ്രതികരണമായി തിരികെ നൽകുകയും ചെയ്യുന്നു. ഇത് ഉപയോക്താവിന് ശുദ്ധമായ, സമാഹരിച്ച ഉത്തരം നൽകുന്നു. |  
| **SCORED** | സംവിധാനത്തിന് ഉള്ളിലെ LLM ഉപയോഗിച്ച് LAST മറുപടിയും SUMMARY ഉം ഉപയോക്തൃ അഭ്യർത്ഥനയ്ക്കായുള്ള സ്കോറുമായി താരതമ്യം ചെയ്ത് ഉയർന്ന സ്കോർ ലഭിച്ചതിനെ തിരികെ നൽകുന്നു. |

#### ഔട്ട്‌പുട്ട് മനസിലാക്കൽ  

ഡെമോ പ്രവർത്തിപ്പിക്കുമ്പോൾ, സൂപ്പർവൈസർ എങ്ങനെ നിരവധി ഏജന്റുകൾ ഏകോപിപ്പിക്കുന്നു എന്നതിനൊരു ഘടനാപരമായ വഴി കാണിക്കും. ഓരോ ഭാഗവും അർത്ഥം:  

```
======================================================================
  FILE → REPORT WORKFLOW DEMO
======================================================================

This demo shows a clear 2-step workflow: read a file, then generate a report.
The Supervisor orchestrates the agents automatically based on the request.
```
  
**ഹെഡർ** വർക്ക്‌ഫ്ലോ ആശയം പരിചയപ്പെടുത്തുന്നു: ഫയൽ റീഡിംഗിൽ നിന്ന് റിപ്പോർട്ട് തയാറാക്കലിലേക്ക് കേന്ദ്രീകൃത പൈപ്ലൈൻ.  

```
--- WORKFLOW ---------------------------------------------------------
  ┌─────────────┐      ┌──────────────┐
  │  FileAgent  │ ───▶ │ ReportAgent  │
  │ (MCP tools) │      │  (pure LLM)  │
  └─────────────┘      └──────────────┘
   outputKey:           outputKey:
   'fileContent'        'report'

--- AVAILABLE AGENTS -------------------------------------------------
  [FILE]   FileAgent   - Reads files via MCP → stores in 'fileContent'
  [REPORT] ReportAgent - Generates structured report → stores in 'report'
```
  
**വർക്ക്‌ഫ്ലോ ഡിസൈൻ** ഏജന്റുകൾ തമ്മിലുള്ള ഡാറ്റ പ്രവാഹം കാണിക്കുന്നു. ഓരോ ഏജന്റിനും പ്രത്യേക പങ്കുണ്ട്:  
- **FileAgent** MCP ഉപകരണങ്ങൾ ഉപയോഗിച്ച് ഫയലുകൾ വായിച്ച് `fileContent`-ൽ الخام ഉള്ളടക്കം സംഭരിക്കുന്നു  
- **ReportAgent** ആ ഉള്ളടക്കം ഉപയോഗിച്ച് `report`-ൽ ഘടനാപരമായ റിപ്പോർട്ട് നിർമ്മിക്കുന്നു  

```
--- USER REQUEST -----------------------------------------------------
  "Read the file at .../file.txt and generate a report on its contents"
```
  
**ഉപയോക്തൃ അഭ്യർത്ഥന** ടൊസ്ക് കാണിക്കുന്നു. സൂപ്പർവൈസർ ഇത് വിശകലനം ചെയ്ത് FileAgent → ReportAgent പ്രവർത്തനം തീരുമാനിക്കുന്നു.  

```
--- SUPERVISOR ORCHESTRATION -----------------------------------------
  The Supervisor decides which agents to invoke and passes data between them...

  +-- STEP 1: Supervisor chose -> FileAgent (reading file via MCP)
  |
  |   Input: .../file.txt
  |
  |   Result: LangChain4j is an open-source, provider-agnostic Java framework for building LLM...
  +-- [OK] FileAgent (reading file via MCP) completed

  +-- STEP 2: Supervisor chose -> ReportAgent (generating structured report)
  |
  |   Input: LangChain4j is an open-source, provider-agnostic Java framew...
  |
  |   Result: Executive Summary...
  +-- [OK] ReportAgent (generating structured report) completed
```
  
**സൂപ്പർവൈസർ ഏകോപനം** രണ്ട് ഘട്ട പ്രവാഹം കാണിക്കുന്നു:  
1. **FileAgent** MCP വഴി ഫയൽ വായിച്ച് ഉള്ളടക്കം സംഭരിക്കുന്നു  
2. **ReportAgent** ഉള്ളടക്കം സ്വീകരിച്ച് ഘടനാപരമായ റിപ്പോർട്ട് സൃഷ്ടിക്കുന്നു  

ഉപയോക്തൃ അഭ്യർത്ഥനയുടെ അടിസ്ഥാനത്തിൽ സൂപ്പർവൈസർ സ്വതന്ത്രമായി ഈ തീരുമാനങ്ങൾ എടുത്തു.  

```
--- FINAL RESPONSE ---------------------------------------------------
Executive Summary
...

Key Points
...

Recommendations
...

--- AGENTIC SCOPE (Data Flow) ----------------------------------------
  Each agent stores its output for downstream agents to consume:
  * fileContent: LangChain4j is an open-source, provider-agnostic Java framework...
  * report: Executive Summary...
```
  
#### ഏജന്റിക് മോഡ്യൂളിന്റെ ഫീച്ചറുകളുടെ വ്യാഖ്യാനം  

ഉദാഹരണം ഏജന്റിക് മോഡ്യൂളിന്റെ നിരവധി മുൻനിര ഫീച്ചറുകൾ കാണിക്കുന്നു. ഏജന്റിക് സ്‌കോപ്പ്, ഏജന്റ് ലിസനേഴ്‌സ് അടുത്ത് കണ്ടാൽ:  

**Agentic Scope** `@Agent(outputKey="...")` ഉപയോഗിച്ച് ഏജന്റുകൾ അവരുടെ ഫലങ്ങൾ സംഭരിച്ച പങ്കുവെച്ച മെമ്മറി കാണിക്കുന്നു. ഇതിലൂടെ:  
- പിന്നീട് വരുന്ന ഏജന്റുകൾ മുൻവരെയുള്ള ഔട്ട്‌പുട്ടുകൾ ആക്‌സസ് ചെയ്യുന്നു  
- സൂപ്പർവൈസർ അവസാന മറുപടിക്ക് ഇതിനെ സംക്ഷേപിക്കുന്നു  
- നിങ്ങൾ ഓരോ ഏജന്റും ഉൽപ്പാദിപ്പിച്ചത് പരിശോധിക്കാനാകും  

താഴെയുള്ള ചിത്രം ഫയൽ-റിപ്പോർട്ട് വർക്ക്‌ഫ്ലോയിൽ Agentic Scope പങ്കുവെച്ച മെമ്മറിയായി എങ്ങനെ പ്രവർത്തിക്കുന്നു കാണിക്കുന്നു — FileAgent `fileContent` കീയിൽ എഴുതുന്നു, ReportAgent അത് വായിച്ച് `report` കീയിൽ എഴുതുന്നു:  

<img src="../../../translated_images/ml/agentic-scope.95ef488b6c1d02ef.webp" alt="Agentic Scope Shared Memory" width="800"/>  

*Agentic Scope പങ്കുവെച്ച മെമ്മറിയായി പ്രവർത്തിക്കുന്നു — FileAgent `fileContent` എഴുതുന്നു, ReportAgent അത് വായിച്ച് `report` എഴുതുന്നു. നിങ്ങളുടെ കോഡ് അന്തിമ ഫലം വായിക്കുന്നു.*  

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // ഫയൽ ഏജന്റിൽ നിന്ന് അസംസ്കൃത ഫയൽ ഡാറ്റ
String report = scope.readState("report");            // റിപ്പോർട്ട് ഏജന്റിൽ നിന്നുള്ള ഘടനാപരമായ റിപ്പോർട്ട്
```
  
**Agent Listeners** ഏജന്റ് പ്രവർത്തനം നിരീക്ഷിക്കുകയും ഡീബഗിംഗ് സഹായിക്കുകയും ചെയ്യുന്നു. ഡെമോയിലുള്ള ഘട്ടേനഘട്ടം ഔട്ട്‌പുട്ട് ഏജന്റ് ലിസണറിന്റെ സഹായത്താൽ സാധ്യമാകുന്നു:  
- **beforeAgentInvocation** - സൂപ്പർവൈസർ ഏജന്റ് തിരഞ്ഞെടുക്കുമ്പോൾ വിളിക്കപ്പെടുന്നു, ഏജന്റ് തിരഞ്ഞെടുക്കപ്പെട്ടതും കാരണം കാണിക്കുന്നു  
- **afterAgentInvocation** - ഏജന്റ് പൂർത്തിയായപ്പോൾ വിളിക്കുന്നു, ഫലം കാണിക്കുന്നു  
- **inheritedBySubagents** - true ആണെങ്കിൽ, ശ്രേണിയിൽ എല്ലാ ഏജന്റുകളുടെയും പ്രവർത്തനം നിരീക്ഷിക്കുന്നു  

താഴെയുള്ള ചിത്രം മുഴുവൻ ഏജന്റ് ലിസണർ ലൈഫ്സൈക്കിൾ കാണിക്കുന്നു, കൂടാതെ `onError` ഏജന്റ് പ്രവർത്തനത്തിലെ പിഴവുകൾ കൈകാര്യം ചെയ്യുന്നത്:  

<img src="../../../translated_images/ml/agent-listeners.784bfc403c80ea13.webp" alt="Agent Listeners Lifecycle" width="800"/>  

*ഏജന്റ് ലിസണറുകൾ പ്രവർത്തന ലൈഫ്സൈക്കിൾ ഹുക്ക് ചെയ്യുന്നു — ഏജന്റുകൾ തുടങ്ങുമ്പോൾ, പൂർത്തിയായപ്പോൾ, പിശകുകൾ കണ്ടുപിടിക്കുമ്പോൾ നിരീക്ഷിക്കുന്നു.*  

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // എല്ലാ സബ്-ഏജന്റുകളിലേക്ക് പ്രചരിപ്പിക്കുക
    }
};
```
  
സൂപ്പർവൈസർ പാറ്റേണിന് പുറമേ, `langchain4j-agentic` മോഡ്യൂൾ ശക്തമായ workflow പാറ്റേണുകൾ നൽകുന്നു. താഴെയുള്ള ചിത്രം എല്ലാ അഞ്ച് പാറ്റേണുകളും കാണിക്കുന്നു — ലളിതമായ ക്രമരഹിത പൈപ്ലൈനുകളിൽ നിന്ന് മനുഷ്യപര്യവേഷണ അംഗീകൃത വർക്ക്‌ഫ്ലോയ് വരെ:  

<img src="../../../translated_images/ml/workflow-patterns.82b2cc5b0c5edb22.webp" alt="Agent Workflow Patterns" width="800"/>  

*ഏജന്റുകൾ ഏകോപിപ്പിക്കുന്ന അഞ്ച് വർക്ക്‌ഫ്ലോ പാറ്റേണുകൾ — ലളിതമായ ക്രമരഹിത പൈപ്ലൈനുകളിൽ നിന്ന് മനുഷ്യപങ്കാളിത്ത അംഗീകാര വർക്ക്‌ഫ്ലോടെ.*  

| പാറ്റേണ | വിവരണം | ഉപയോഗം |  
|---------|-------------|----------|  
| **Sequential** | ഏജന്റുകൾ ക്രമത്തിൽ പ്രവർത്തിപ്പിക്കുക, ഔട്ട്‌പുട്ട് അടുത്തയ്ക്ക് പോകുന്നു | പൈപ്ലൈനുകൾ: ഗവേഷണം → വിശകലനം → റിപ്പോർട്ട് |  
| **Parallel** | ഏജന്റുകൾ ഒരേസമയം പ്രവർത്തിക്കുക | സ്വതന്ത്ര ജോലികൾ: കാലാവസ്ഥ + വാർത്തകൾ + സ്റ്റോക്ക്‌സ് |  
| **Loop** | നിബന്ധന പൂർത്തിയാകുന്നത് വരെ ആവൃത്തിയാകുക | ഗുണനിലവാര സ്കോറിംഗ്: സ്കോർ ≥ 0.8 വരെയുള്ള ഒത്തുചേരൽ |  
| **Conditional** | നിബന്ധനകളുടെ അടിസ്ഥാനത്തിൽ റൂട്ടുചെയ്യുക | വർഗ്ഗീകരണം → വിദഗ്ധ ഏജന്റിലേക്ക് റൂട്ടിംഗ് |  
| **Human-in-the-Loop** | മനുഷ്യ പരിശോധന അളവുകൾ ചേർക്കുക | അംഗീകാരം വർക്ക്‌ഫ്ലോ, ഉള്ളടക്കം അവലോകനം |  

## അടിസ്ഥാന ആശയങ്ങൾ  

ഇനി നിങ്ങൾ MCPയും ഏജന്റിക് മോഡ്യൂൾ ഉപയോഗവും അന്വേഷിച്ചതിനുശേഷം, ഓരോ സമീപനവും എപ്പോൾ ഉപയോഗിക്കാമെന്ന് സംഗ്രഹിക്കാം.  

MCP-യുടെ വലിയ ഗുണങ്ങൾ ഒന്നാണ് അതിന്റെ വളരുന്ന ഇക്കോസിസ്റ്റം. താഴെയുള്ള ചിത്രം MCP സർവർ-ക്ലയന്റ് തമ്മിലുള്ള സർവങ്ങളുടെ സർവത്ര പ്രോട്ടോക്കോളിന്റെ സാന്നിധ്യം കാണിക്കുന്നു — ഫയല്സിസ്റ്റം, ഡേറ്റാബേസ് ആക്‌സസ് മുതൽ GitHub, ഇമെയിൽ, വെബ് സ്ക്രാപ്പിംഗ് വരെ:  

<img src="../../../translated_images/ml/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="MCP Ecosystem" width="800"/>  

*MCP സർവീസ് പായനി എങ്കിൽ MCP-അനുകൂല സർവർ-ക്ലയന്റ് ഏത് ആപ്ലിക്കേഷനിലും ടൂൾ പങ്കുവെക്കാൻ ഇടവരുത്തുന്നു.*  

**MCP** നിലവിലുള്ള ടൂൾ ഇക്കോസിസ്റ്റം പ്രയോജനം ചെയ്യാൻ, പലയിടങ്ങളിൽ പങ്കുവെക്കാവുന്ന ടൂളുകൾ നിർമ്മിക്കുവാൻ, മാനദണ്ഡപ്രകാരം മൂന്നാംകക്ഷി സേവനങ്ങൾ സംയോജിപ്പിക്കാൻ, കോഡ് മാറ്റാതെ ടൂൾ ഇംപ്ലിമെന്റേഷൻ മാറ്റാൻ അനുയോജ്യമാണ്.  

**ആജന്റിക് മോഡ്യൂൾ** `@Agent` അനോട്ടേഷൻ ആവിഷ്കരിക്കുന്ന പ്രഖ്യാപനാത്മക ഏജന്റ് നിർവചനത്തിന്, വർക്ക്‌ഫ്ലോ ഏകോപനത്തിന് (ക്രമരഹിതം, ലൂപ്പ്, ഏകകാലികം), നിർദ്ദേശപരമായ കോഡിനുപകരം ഇന്റർഫേസ് ആധാരിത ഏജന്റ് ഡിസൈനിന്, `outputKey` വഴി പങ്കുവെക്കുന്ന ഔട്ട്‌പുട്ടുകൾ ഉപയോഗിക്കുന്ന മൾട്ടി ഏജന്റ് സംയോജിപ്പിക്കുമ്പോൾ മികച്ചതാണ്.  

**സൂപ്പർവൈസർ ഏജന്റ് പാറ്റേൺ** പൂർവ്വനിർദ്ദേശം കഴിയാത്ത വർക്ക്‌ഫ്ലോയ് പോലെ LLM തീരുമാനിക്കാൻ വേണ്ടിയുള്ളതിനും, നിരവധി പ്രത്യേക ഏജന്റുകൾ ഡയനാമിക് ഏകോപനം വേണ്ടതെങ്കിൽ, വിവിധ ശേഷികളിലേക്ക് റൂട്ടുചെയ്യുന്ന സംഭാഷണ സംവിധാനം നിർമ്മിക്കുമ്പോഴും, ഏറ്റവും ഫ്ലെക്സിബിൾ, അതിനുസരിച്ച് ഡൈനമിക് പെരുമാറ്റം ആവശ്യമായപ്പോൾ ഉത്തമമാണ്.  

Module 04-ൽ കസ്റ്റം `@Tool` മെത്തഡുകളും ഈ മോഡ്യൂളിലുള്ള MCP ടൂളുകളും തിരഞ്ഞെടുക്കുന്നതിനുള്ള പ്രധാന വ്യത്യാസങ്ങൾ ചുവടെ കാണിക്കുന്നു — കസ്റ്റം ടൂളുകൾ ടൈപ്പ് സുരക്ഷയും കൃത്യമായ ബന്ധവും നൽകുന്നു; MCP ടൂളുകൾ സുവ്യവസ്ഥിതവും പുനരുപയോഗയോഗ്യവുമാണ്.  

<img src="../../../translated_images/ml/custom-vs-mcp-tools.c4f9b6b1cb65d8a1.webp" alt="Custom Tools vs MCP Tools" width="800"/>  

*കസ്റ്റം @Tool മെത്തഡുകൾ ഉപയോഗിക്കേണ്ടപ്പോൾ vs MCP ടൂളുകൾ — ആപ്പ് നിർദിഷ്ടലാജിക്ക് കസ്റ്റം ടൂൾ, നിയമവിരുദ്ധം ഒഴിവാക്കി നൽകുന്നു MCP ടൂൾ സ്ഥിരമായ സംയോജിപ്പുകൾ എല്ലാ ആപ്ലിക്കേഷനുകളിലും പ്രവർത്തിക്കുന്നു.*
