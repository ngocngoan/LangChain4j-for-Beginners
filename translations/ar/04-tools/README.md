## ما الذي ستتعلمه

حتى الآن، تعلمت كيفية إجراء محادثات مع الذكاء الاصطناعي، وتنظيم الطلبات بفعالية، وربط الردود بوثائقك. لكن هناك قيد أساسي: نماذج اللغة يمكنها فقط توليد النصوص. فهي لا تستطيع التحقق من الطقس، أو إجراء الحسابات، أو الاستعلام في قواعد البيانات، أو التفاعل مع الأنظمة الخارجية.

الأدوات تغير ذلك. من خلال منح النموذج إمكانية الوصول إلى الدوال التي يمكنه استدعاؤها، تحوّله من مولد نصوص إلى وكيل يمكنه اتخاذ الإجراءات. يقرر النموذج متى يحتاج إلى أداة، وأي أداة يستخدم، وما المعطيات التي يمررها. يقوم رمزك بتنفيذ الدالة وإرجاع النتيجة. يدمج النموذج تلك النتيجة في رده.

## فهم وكلاء الذكاء الاصطناعي مع الأدوات

> **📝 ملاحظة:** مصطلح "الوكلاء" في هذه الوحدة يشير إلى مساعدي الذكاء الاصطناعي المعززين بقدرات استدعاء الأدوات. وهذا يختلف عن أنماط **Agentic AI** (وكلاء مستقلون مع تخطيط وذاكرة وتفكير متعدد الخطوات) والتي سنغطيها في [الوحدة 05: MCP](../05-mcp/README.md).

بدون أدوات، يمكن لنموذج اللغة فقط توليد نص من بيانات تدريبه. اسأله عن الطقس الحالي، فيضطر للتخمين. امنحه أدوات، فيمكنه استدعاء API الطقس، إجراء الحسابات، أو الاستعلام في قاعدة بيانات — ثم يدمج تلك النتائج الحقيقية في رده.

<img src="../../../translated_images/ar/what-are-tools.724e468fc4de64da.webp" alt="بدون أدوات مقابل مع أدوات" width="800"/>

*بدون أدوات النموذج يخمن فقط — مع الأدوات يمكنه استدعاء APIs، إجراء الحسابات، وإرجاع بيانات في الوقت الحقيقي.*

وكيل الذكاء الاصطناعي مع الأدوات يتبع نمط **التفكير والعمل (ReAct)**. النموذج لا يرد فقط — بل يفكر فيما يحتاجه، يتصرف باستدعاء أداة، يلاحظ النتيجة، ثم يقرر إن كان سيتصرف مجددًا أو يقدم الإجابة النهائية:

1. **يفكر** — يحلل الوكيل سؤال المستخدم ويحدد المعلومات المطلوبة
2. **يتصرف** — يختار الأداة الصحيحة، يولد المعطيات المناسبة، ويستدعيها
3. **يراقب** — يستقبل المخرجات من الأداة ويقيّم النتيجة
4. **يعيد أو يرد** — إذا كانت هناك حاجة لمزيد من البيانات، يعيد التكرار؛ وإلا يؤلف ردًا بلغة طبيعية

<img src="../../../translated_images/ar/react-pattern-detail.96a5efeeb6dd2f61.webp" alt="نمط ReAct" width="800"/>

*دورة ReAct — الوكيل يفكر فيما يجب فعله، يتصرف باستدعاء أداة، يلاحظ النتيجة، ويكرر حتى يتمكن من تقديم الإجابة النهائية.*

يحدث هذا تلقائيًا. أنت تعرف الأدوات ووصفها. النموذج يتولى قرار متى وكيف يستخدمها.

## كيف تعمل استدعاءات الأدوات

### تعريفات الأدوات

[WeatherTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) | [TemperatureTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/TemperatureTool.java)

تعرف دوالًا مع أوصاف واضحة ومواصفات المعطيات. يرى النموذج هذه الأوصاف في مطالبه النظامية ويفهم ما تفعل كل أداة.

```java
@Component
public class WeatherTool {
    
    @Tool("Get the current weather for a location")
    public String getCurrentWeather(@P("Location name") String location) {
        // منطق البحث عن الطقس الخاص بك
        return "Weather in " + location + ": 22°C, cloudy";
    }
}

@AiService
public interface Assistant {
    String chat(@MemoryId String sessionId, @UserMessage String message);
}

// يتم توصيل المساعد تلقائيًا بواسطة Spring Boot مع:
// - مكون ChatModel
// - جميع طرق @Tool من فئات @Component
// - مزود ذاكرة الدردشة لإدارة الجلسة
```

يبين المخطط أدناه كل توضيح ويشرح كيف يساعد كل جزء الذكاء الاصطناعي على فهم متى يستدعي الأداة وأي المعطيات يمرر:

<img src="../../../translated_images/ar/tool-definitions-anatomy.f6468546037cf28b.webp" alt="تشريح تعريفات الأدوات" width="800"/>

*تشريح تعريف الأداة — @Tool تخبر الذكاء الاصطناعي متى يستخدمها، @P تصف كل معلم، و@AiService توصل كل شيء تلقائيًا عند التشغيل.*

> **🤖 جرب مع [GitHub Copilot](https://github.com/features/copilot) Chat:** افتح [`WeatherTool.java`](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) واسأل:
> - "كيف يمكنني دمج API طقس حقيقي مثل OpenWeatherMap بدلاً من بيانات وهمية؟"
> - "ما الذي يجعل وصف الأداة جيدًا ويساعد الذكاء الاصطناعي على استخدامها بشكل صحيح؟"
> - "كيف أتعامل مع أخطاء API وحدود المعدل في تنفيذ الأدوات؟"

### اتخاذ القرار

عندما يسأل المستخدم "ما هو الطقس في سياتل؟"، لا يختار النموذج أداة عشوائيًا. يقارن نية المستخدم مع كل وصف أداة لديه، ويعطي درجات ذات صلة، ويختار الأنسب. ثم يولّد استدعاء دالة منظم مع المعطيات الصحيحة — في هذه الحالة يعين `location` إلى `"Seattle"`.

إذا لم تتطابق أي أداة مع طلب المستخدم، يعود النموذج للإجابة من معرفته الخاصة. إذا تطابقت أدوات متعددة، يختار الأدق.

<img src="../../../translated_images/ar/decision-making.409cd562e5cecc49.webp" alt="كيف يقرر الذكاء الاصطناعي أي أداة يستخدم" width="800"/>

*النموذج يقيم كل أداة متاحة مقابل نية المستخدم ويختار الأفضل — لهذا السبب يهم كتابة أوصاف أدوات واضحة ومحددة.*

### توليد الاستجابة

يستقبل النموذج بيانات الطقس ويصيغها في رد بلغة طبيعية للمستخدم.

### البنية: التوصيل التلقائي في Spring Boot

تستخدم هذه الوحدة تكامل LangChain4j مع Spring Boot وواجهات `@AiService` الإعلانية. عند التشغيل، يكتشف Spring Boot كل `@Component` الذي يحتوي على طرق `@Tool`، وحبة `ChatModel`، ومقدم `ChatMemoryProvider` — ثم يربطهم جميعًا في واجهة `Assistant` واحدة بدون كود روتيني.

<img src="../../../translated_images/ar/spring-boot-wiring.151321795988b04e.webp" alt="بنية التوصيل التلقائي في Spring Boot" width="800"/>

*واجهة @AiService تربط بين ChatModel، مكونات الأدوات، ومزود الذاكرة — Spring Boot يتولى كل التوصيل تلقائيًا.*

الفوائد الرئيسية لهذا الأسلوب:

- **التوصيل التلقائي في Spring Boot** — يتم حقن ChatModel والأدوات تلقائيًا
- **نمط @MemoryId** — إدارة الذاكرة معتمدة على الجلسة أوتوماتيكيًا
- **حالة مفردة** — يتم إنشاء المساعد مرة واحدة ويعاد استخدامه لتحسين الأداء
- **تنفيذ آمن نوعيًا** — استدعاء دوال جافا مباشرة مع تحويل الأنواع
- **تنسيق متعدد الأدوار** — يدير سلاسل الأدوات تلقائيًا
- **بدون تكرار للكود** — لا حاجة لاستدعاءات يدوية لـ AiServices.builder() أو خرائط الذاكرة

الطرق البديلة (مثل الاستدعاء اليدوي لـ AiServices.builder()) تتطلب مزيدًا من التعليمات البرمجية وتفتقد لفوائد تكامل Spring Boot.

## سلسلة الأدوات

**سلسلة الأدوات** — القوة الحقيقية للوكلاء المعتمدين على الأدوات تظهر عندما يتطلب سؤال واحد عدة أدوات. اسأل "ما هو الطقس في سياتل بالفهرنهايت؟" فيسلسل الوكيل أداتين تلقائيًا: أولًا يستدعي `getCurrentWeather` للحصول على درجة الحرارة بالسيلسيوس، ثم يمرر تلك القيمة إلى `celsiusToFahrenheit` للتحويل — كل ذلك في دورة محادثة واحدة.

<img src="../../../translated_images/ar/tool-chaining-example.538203e73d09dd82.webp" alt="مثال على سلسلة الأدوات" width="800"/>

*سلسلة الأدوات في العمل — يستدعي الوكيل getCurrentWeather أولاً، ثم يمرر النتيجة بالسيلسيوس إلى celsiusToFahrenheit، ويقدم إجابة مجمعة.*

**الفشل الرشيد** — اطلب الطقس في مدينة غير موجودة ضمن البيانات الوهمية. تعيد الأداة رسالة خطأ، ويشرح الذكاء الاصطناعي أنه لا يمكنه المساعدة بدلاً من التعطل. الأدوات تفشل بأمان. يوضح المخطط أدناه الفرق بين الطريقتين — مع معالجة الأخطاء المناسبة، يلتقط الوكيل الاستثناء ويرد بطريقة مساعدة، وبدونها يتعطل التطبيق كله:

<img src="../../../translated_images/ar/error-handling-flow.9a330ffc8ee0475c.webp" alt="تدفق معالجة الأخطاء" width="800"/>

*عندما تفشل أداة، يلتقط الوكيل الخطأ ويرد بتفسير مفيد بدلاً من التعطل.*

يحدث هذا في دورة محادثة واحدة. الوكيل ينسق استدعاءات متعددة للأدوات بشكل مستقل.

## استخدام التطبيق

<a href="images/tools-homepage.png"><img src="../../../translated_images/ar/tools-homepage.4b4cd8b2717f9621.webp" alt="واجهة أدوات وكيل الذكاء الاصطناعي" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*واجهة أدوات وكيل الذكاء الاصطناعي - أمثلة سريعة وواجهة دردشة للتفاعل مع الأدوات*

### جرب استخدام أداة بسيطة

ابدأ بطلب بسيط: "حوّل 100 درجة فهرنهايت إلى سلسيوس". يتعرف الوكيل على أنه يحتاج إلى أداة تحويل درجات الحرارة، ويناديها بالمعلمات الصحيحة، ويعيد النتيجة. لاحظ مدى طبيعية هذا الإجراء - لم تحدد أداة معينة للاستخدام أو كيفية استدعائها.

### اختبر ربط الأدوات

جرّب الآن شيئًا أكثر تعقيدًا: "ما حالة الطقس في سياتل وحوّلها إلى فهرنهايت؟" شاهد الوكيل يعمل على ذلك على مراحل. أولاً يحصل على الطقس (الذي يعيد النتيجة بالسلسيوس)، ويتعرف على أنه يحتاج إلى التحويل إلى فهرنهايت، وينادي أداة التحويل، ويجمع كلا النتيجتين في رد واحد.

### شاهد تدفق المحادثة

تحافظ واجهة الدردشة على سجل المحادثة، مما يتيح لك إجراء تفاعلات متعددة التناوب. يمكنك رؤية كل الاستفسارات والردود السابقة، مما يسهل تتبع المحادثة وفهم كيف يبني الوكيل السياق عبر تبادلات متعددة.

<a href="images/tools-conversation-demo.png"><img src="../../../translated_images/ar/tools-conversation-demo.89f2ce9676080f59.webp" alt="محادثة مع استدعاءات متعددة للأدوات" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*محادثة متعددة التناوب تظهر تحويلات بسيطة، واستعلامات الطقس، وربط الأدوات*

### جرب طلبات مختلفة

جرّب مجموعات متنوعة:
- استعلامات الطقس: "ما حالة الطقس في طوكيو؟"
- تحويلات درجات الحرارة: "كم تعادل 25°س بالكلفن؟"
- استفسارات مركبة: "تحقق من الطقس في باريس وأخبرني إذا كان فوق 20°س"

لاحظ كيف يفسر الوكيل اللغة الطبيعية ويربطها بالاستدعاءات المناسبة للأدوات.

## المفاهيم الأساسية

### نمط ReAct (التفكير والفعل)

يتناوب الوكيل بين التفكير (تقرير ما يجب القيام به) والفعل (استخدام الأدوات). يتيح هذا النمط حل المشكلات بشكل مستقل بدلاً من مجرد الرد على التعليمات.

### أهمية وصف الأدوات

جودة وصف الأدوات يؤثر مباشرة على كيفية استخدامها من قبل الوكيل. تساعد الأوصاف الواضحة والمحددة النموذج على فهم متى وكيف يستدعي كل أداة.

### إدارة الجلسات

تُتيح التعليمة `@MemoryId` إدارة الذاكرة المبنية على الجلسة بشكل تلقائي. يحصل كل معرف جلسة على مثيل `ChatMemory` يتم إدارته بواسطة مكون `ChatMemoryProvider`، مما يسمح بتفاعل عدة مستخدمين مع الوكيل في نفس الوقت دون تداخل محادثاتهم. يوضح الشكل التالي كيف يتم توجيه المستخدمين إلى مخازن ذاكرة معزولة بناءً على معرفات الجلسة الخاصة بهم:

<img src="../../../translated_images/ar/session-management.91ad819c6c89c400.webp" alt="إدارة الجلسة باستخدام @MemoryId" width="800"/>

*يرتبط كل معرف جلسة بتاريخ محادثة معزول — لا يرى المستخدمون رسائل بعضهم البعض أبداً.*

### التعامل مع الأخطاء

يمكن أن تفشل الأدوات — قد تنفد مهلة واجهات برمجة التطبيقات، أو تكون المعلمات غير صالحة، أو تتوقف الخدمات الخارجية. يحتاج الوكلاء في الإنتاج إلى معالجة الأخطاء لتمكين النموذج من شرح المشاكل أو محاولة بدائل بدلاً من تحطم التطبيق بالكامل. عند رمي أداة استثناء، يقوم LangChain4j بالتقاطه ويرسل رسالة الخطأ إلى النموذج، الذي يمكنه حينها شرح المشكلة بلغة طبيعية.

## الأدوات المتاحة

يوضح المخطط أدناه النظام البيئي الواسع للأدوات التي يمكنك بناؤها. يعرض هذا الموديول أدوات الطقس ودرجات الحرارة، لكن نمط `@Tool` نفسه يعمل مع أي طريقة جافا — من استعلامات قواعد البيانات إلى معالجة المدفوعات.

<img src="../../../translated_images/ar/tool-ecosystem.aad3d74eaa14a44f.webp" alt="نظام أدوات" width="800"/>

*أي طريقة جافا معنونة بـ @Tool تصبح متاحة للذكاء الاصطناعي — يمد النمط ليشمل قواعد البيانات، الواجهات البرمجية، البريد الإلكتروني، عمليات الملفات، وأكثر.*

## متى تستخدم وكلاء معتمدين على الأدوات

ليس كل طلب يحتاج إلى أدوات. يتوقف القرار على ما إذا كان الذكاء الاصطناعي يحتاج للتفاعل مع أنظمة خارجية أو يمكنه الإجابة من معرفته الخاصة. يلخص الدليل التالي متى تضيف الأدوات قيمة ومتى تكون غير ضرورية:

<img src="../../../translated_images/ar/when-to-use-tools.51d1592d9cbdae9c.webp" alt="متى تستخدم الأدوات" width="800"/>

*دليل قرار سريع — الأدوات مخصصة للبيانات الحية، الحسابات، والإجراءات؛ المعرفة العامة والمهام الإبداعية لا تحتاجها.*

## الأدوات مقابل RAG

يعمل الموديولان 03 و04 على توسيع إمكانيات الذكاء الاصطناعي، لكن بطرق جوهرية مختلفة. يتيح RAG للنموذج الوصول إلى **المعرفة** عبر استرجاع الوثائق. توفر الأدوات للنموذج القدرة على اتخاذ **الإجراءات** من خلال استدعاء الدوال. يقارن المخطط التالي النهجين جنباً إلى جنب — من كيفية عمل كل تدفق عمل إلى المقايضات بينهما:

<img src="../../../translated_images/ar/tools-vs-rag.ad55ce10d7e4da87.webp" alt="مقارنة الأدوات مقابل RAG" width="800"/>

*يسترجع RAG المعلومات من الوثائق الثابتة — تنفذ الأدوات الإجراءات وتجلب البيانات الحية والديناميكية. كثير من أنظمة الإنتاج تجمع بين النهجين.*

في الممارسة، تجمع العديد من أنظمة الإنتاج بين النهجين: RAG لتأسيس الإجابات على مستنداتك، والأدوات لجلب البيانات الحية أو تنفيذ العمليات.
