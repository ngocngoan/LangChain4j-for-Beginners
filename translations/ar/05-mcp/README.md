## ما الذي ستتعلمه

لقد قمت ببناء ذكاء اصطناعي محادثي، وإتقان المطالبات، وتأسيس الاستجابات على الوثائق، وإنشاء وكلاء مزودين بأدوات. ولكن كل تلك الأدوات كانت مخصصة لتطبيقك الخاص. ماذا لو كان بإمكانك منح ذكاءك الاصطناعي إمكانية الوصول إلى نظام معياري من الأدوات يمكن لأي شخص إنشاؤها ومشاركتها؟ في هذه الوحدة، ستتعلم كيف تفعل ذلك تمامًا باستخدام بروتوكول سياق النموذج (MCP) ووحدة الوكالة في LangChain4j. نعرض أولاً قارئ ملفات MCP بسيط ثم نظهر كيف يندمج بسهولة في تدفقات العمل الوكلائية المتقدمة باستخدام نمط وكيل المشرف.

## ما هو MCP؟

يوفر بروتوكول سياق النموذج (MCP) بالضبط هذا - طريقة معيارية لتطبيقات الذكاء الاصطناعي لاكتشاف واستخدام الأدوات الخارجية. بدلاً من كتابة تكاملات مخصصة لكل مصدر بيانات أو خدمة، تقوم بالاتصال بخوادم MCP التي تعرض قدراتها بتنسيق متسق. يمكن لوكيل الذكاء الاصطناعي الخاص بك بعد ذلك اكتشاف هذه الأدوات واستخدامها تلقائيًا.

يوضح الرسم البياني التالي الفرق — بدون MCP، يتطلب كل تكامل وصلات مخصصة من نقطة إلى نقطة؛ مع MCP، يربط بروتوكول واحد تطبيقك بأي أداة:

<img src="../../../translated_images/ar/mcp-comparison.9129a881ecf10ff5.webp" alt="مقارنة MCP" width="800"/>

*قبل MCP: تكاملات معقدة نقطة إلى نقطة. بعد MCP: بروتوكول واحد، إمكانيات لا نهائية.*

تحل MCP مشكلة أساسية في تطوير الذكاء الاصطناعي: كل تكامل مخصص. هل تريد الوصول إلى GitHub؟ كود مخصص. هل تريد قراءة ملفات؟ كود مخصص. هل تريد الاستعلام من قاعدة بيانات؟ كود مخصص. ولا يعمل أي من هذه التكاملات مع تطبيقات الذكاء الاصطناعي الأخرى.

MCP يقوم بتوحيد هذا. يكشف خادم MCP عن الأدوات مع أوصاف واضحة ومخططات. يمكن لأي عميل MCP الاتصال واكتشاف الأدوات المتاحة واستخدامها. تبني مرة واحدة، تستخدم في جميع الأماكن.

يوضح الرسم البياني التالي هذه البنية — عميل MCP واحد (تطبيق الذكاء الاصطناعي الخاص بك) يتصل بعدة خوادم MCP، كل منها يكشف مجموعته الخاصة من الأدوات عبر البروتوكول المعياري:

<img src="../../../translated_images/ar/mcp-architecture.b3156d787a4ceac9.webp" alt="بنية MCP" width="800"/>

*بنية بروتوكول سياق النموذج - اكتشاف الأدوات وتنفيذها بشكل معياري*

## كيف يعمل MCP

تستخدم MCP في الداخل بنية طبقية. يكتشف تطبيق Java الخاص بك (عميل MCP) الأدوات المتاحة، يرسل طلبات JSON-RPC عبر طبقة نقل (Stdio أو HTTP)، ويقوم خادم MCP بتنفيذ العمليات وإرجاع النتائج. يفصل الرسم البياني التالي كل طبقة من هذا البروتوكول:

<img src="../../../translated_images/ar/mcp-protocol-detail.01204e056f45308b.webp" alt="تفصيل بروتوكول MCP" width="800"/>

*كيف يعمل MCP في الداخل — يكتشف العملاء الأدوات، يتبادلون رسائل JSON-RPC، وينفذون العمليات عبر طبقة نقل.*

**بنية خادم-عميل**

يستخدم MCP نموذج خادم-عميل. توفر الخوادم الأدوات - قراءة الملفات، استعلام قواعد البيانات، استدعاء واجهات برمجة التطبيقات. يتصل العملاء (تطبيق الذكاء الاصطناعي الخاص بك) بالخوادم ويستخدمون أدواتها.

لاستخدام MCP مع LangChain4j، أضف تبعية Maven هذه:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```

**اكتشاف الأدوات**

عندما يتصل العميل الخاص بك بخادم MCP، يسأل "ما الأدوات التي لديك؟" يرد الخادم بقائمة الأدوات المتاحة، كل واحدة بوصفها ومخططات المعلمات. يمكن لوكيل الذكاء الاصطناعي الخاص بك حينها أن يقرر أي الأدوات يستخدم بناءً على طلبات المستخدم. يوضح الرسم البياني التالي هذه المصافحة — يرسل العميل طلب `tools/list` ويرد الخادم بأدواته المتاحة مع الأوصاف ومخططات المعلمات:

<img src="../../../translated_images/ar/tool-discovery.07760a8a301a7832.webp" alt="اكتشاف أدوات MCP" width="800"/>

*يكتشف الذكاء الاصطناعي الأدوات المتاحة عند بدء التشغيل — يعرف الآن القدرات المتاحة ويمكنه تحديد الأدوات التي يستخدمها.*

**آليات النقل**

يدعم MCP آليات نقل مختلفة. الخياران هما Stdio (للتواصل مع العمليات الفرعية المحلية) وStreamable HTTP (لخوادم بعيدة). توضح هذه الوحدة النقل عبر Stdio:

<img src="../../../translated_images/ar/transport-mechanisms.2791ba7ee93cf020.webp" alt="آليات النقل" width="800"/>

*آليات نقل MCP: HTTP للخوادم البعيدة، Stdio للعمليات المحلية*

**Stdio** - [StdioTransportDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java)

للعمليات المحلية. يقوم تطبيقك بتشغيل خادم كعملية فرعية ويتواصل من خلال الإدخال/الإخراج القياسي. مفيد للوصول إلى نظام الملفات أو الأدوات من سطر الأوامر.

```java
McpTransport stdioTransport = new StdioMcpTransport.Builder()
    .command(List.of(
        npmCmd, "exec",
        "@modelcontextprotocol/server-filesystem@2025.12.18",
        resourcesDir
    ))
    .logEvents(false)
    .build();
```

يكشف خادم `@modelcontextprotocol/server-filesystem` عن الأدوات التالية، جميعها معزولة بالمجلدات التي تحددها:

| الأداة | الوصف |
|------|-------------|
| `read_file` | قراءة محتويات ملف واحد |
| `read_multiple_files` | قراءة ملفات متعددة في استدعاء واحد |
| `write_file` | إنشاء ملف أو استبداله |
| `edit_file` | إجراء تعديلات استبدال مستهدفة |
| `list_directory` | سرد الملفات والمجلدات في المسار |
| `search_files` | البحث المتكرر عن ملفات تطابق نمط |
| `get_file_info` | الحصول على بيانات وصفية للملف (الحجم، الطوابع الزمنية، الأذونات) |
| `create_directory` | إنشاء مجلد (بما في ذلك المجلدات الأصل) |
| `move_file` | نقل أو إعادة تسمية ملف أو مجلد |

يُظهر الرسم البياني التالي كيف يعمل نقل Stdio أثناء التشغيل — يقوم تطبيق Java الخاص بك بتشغيل خادم MCP كعملية فرعية ويتواصلان عبر أنابيب stdin/stdout، دون أي شبكات أو HTTP متضمنة:

<img src="../../../translated_images/ar/stdio-transport-flow.45eaff4af2d81db4.webp" alt="تدفق نقل Stdio" width="800"/>

*نقل Stdio أثناء العمل — يقوم التطبيق بتشغيل خادم MCP كعملية فرعية ويتواصلان من خلال أنابيب stdin/stdout.*

## الوحدة الوكلائية

بينما يقدم MCP أدوات معيارية، توفر وحدة الوكالة في LangChain4j طريقة إعلانية لبناء وكلاء ينظمون تلك الأدوات. تسمح لك التعليمة `@Agent` و `AgenticServices` بتعريف سلوك الوكيل من خلال واجهات بدلاً من كود إجرائي.

في هذه الوحدة، ستستكشف نمط **وكيل المشرف** — نهج وكيل AI متقدم حيث يقرر "المشرف" وكيلًا فرعيًا ليتم استدعاؤه ديناميكيًا بناءً على طلب المستخدم. سندمج كلا المفهومين بمنح أحد وكلائنا الفرعيين قدرات وصول إلى الملفات مدعومة بـ MCP.

لاستخدام وحدة الوكالة، أضف تبعية Maven هذه:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```
> **ملاحظة:** تستخدم وحدة `langchain4j-agentic` خاصية إصدار منفصلة (`langchain4j.mcp.version`) لأنها تُصدر بجدول زمني مختلف عن مكتبات LangChain4j الأساسية.

> **⚠️ تجريبية:** وحدة `langchain4j-agentic` **تجريبية** وقد تتغير. الطريقة المستقرة لبناء مساعدين ذكاء اصطناعي تبقى عبر `langchain4j-core` مع أدوات مخصصة (الوحدة 04).

### وكيل المشرف

نمط **وكيل المشرف** هو شكل **مرن** من الذكاء الاصطناعي الوكلي. يستخدم المشرف نموذج لغوي كبير ليقرر بشكل مستقل أي الوكلاء يستدعي بناءً على طلب المستخدم. في المثال التالي، نجمع وصول الملف المدعوم بـ MCP مع وكيل LLM لإنشاء سير عمل قراءة ملف → تقرير بإشراف.

في العرض التوضيحي، يقرأ `FileAgent` ملفًا باستخدام أدوات نظام ملفات MCP، وينشئ `ReportAgent` تقريرًا منظمًا يتضمن ملخصًا تنفيذيًا (جملة واحدة)، 3 نقاط رئيسية، وتوصيات. يدير المشرف هذا التدفق تلقائيًا:

<img src="../../../translated_images/ar/supervisor-agent-pattern.06275a41ae006ac8.webp" alt="نمط وكيل المشرف" width="800"/>

*يستخدم المشرف النموذج اللغوي الكبير لتقرير الوكلاء الذين يستدعيهم وبأي ترتيب — لا حاجة لتوجيه مبرمج.*

إليك كيف يبدو سير العمل العملي لأنابيب الملف إلى التقرير:

<img src="../../../translated_images/ar/file-report-workflow.649bb7a896800de9.webp" alt="سير عمل ملف لتقرير" width="800"/>

*يقرأ FileAgent الملف عبر أدوات MCP، ثم يحول ReportAgent المحتوى الخام إلى تقرير منظم.*

يرصد مخطط التسلسل التالي الإخراج الكامل للمشرف — من تشغيل خادم MCP، عبر اختيار الوكلاء المستقل، إلى استدعاءات الأدوات عبر stdio والتقرير النهائي:

<img src="../../../translated_images/ar/supervisor-agent-sequence.1aa389b3bef99956.webp" alt="مخطط تسلسل وكيل المشرف" width="800"/>

*يقوم المشرف بتمويل استدعاء FileAgent (الذي يستدعي خادم MCP عبر stdio لقراءة الملف)، ثم يستدعي ReportAgent لتوليد تقرير منظم — كل وكيل يخزن مخرجاته في النطاق الوكلي المشترك.*

يخزن كل وكيل مخرجاته في **النطاق الوكلي** (ذاكرة مشتركة)، مما يسمح للوكلاء التاليين بالوصول إلى النتائج السابقة. يوضح هذا كيف تندمج أدوات MCP بسلاسة في تدفقات العمل الوكلائية — لا يحتاج المشرف إلى معرفة *كيفية* قراءة الملفات، فقط أن `FileAgent` قادر على ذلك.

#### كيف يعمل المشرف

قبل بناء الوكلاء، تحتاج إلى ربط نقل MCP إلى عميل وتغليفه كـ `ToolProvider`. هكذا تصبح أدوات خادم MCP متاحة لوكلائك:

```java
// إنشاء عميل MCP من النقل
McpClient mcpClient = new DefaultMcpClient.Builder()
        .transport(stdioTransport)
        .build();

// لف العميل كباحث أدوات — هذا يجسر أدوات MCP إلى LangChain4j
ToolProvider mcpToolProvider = McpToolProvider.builder()
        .mcpClients(List.of(mcpClient))
        .build();
```

يمكنك الآن حقن `mcpToolProvider` في أي وكيل يحتاج إلى أدوات MCP:

```java
// الخطوة 1: يقوم FileAgent بقراءة الملفات باستخدام أدوات MCP
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // يمتلك أدوات MCP لعمليات الملفات
        .build();

// الخطوة 2: يقوم ReportAgent بإنشاء تقارير منظمة
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// يقوم المشرف بتنظيم سير العمل من الملف إلى التقرير
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // إعادة التقرير النهائي
        .build();

// يقرر المشرف الوكلاء الذين سيتم استدعائهم بناءً على الطلب
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```

#### كيف يكتشف FileAgent أدوات MCP أثناء التشغيل

قد تتساءل: **كيف يعرف `FileAgent` كيفية استخدام أدوات نظام ملفات npm؟** الجواب أنه لا يعرف — **النموذج اللغوي الكبير** يكتشف ذلك أثناء التشغيل عبر مخططات الأدوات.

واجهة `FileAgent` هي فقط **تعريف مطالبة**. لا تملك معرفة مبرمجة مسبقًا بـ `read_file`، `list_directory`، أو أي أداة MCP أخرى. إليك ما يحدث بداية إلى نهاية:
1. **تشغيل الخادم:** يقوم `StdioMcpTransport` بإطلاق حزمة npm `@modelcontextprotocol/server-filesystem` كعملية فرعية  
2. **اكتشاف الأداة:** يرسل `McpClient` طلب JSON-RPC من نوع `tools/list` إلى الخادم، الذي يستجيب بأسماء الأدوات، الأوصاف، ومخططات المعلمات (على سبيل المثال، `read_file` — *"قراءة المحتويات الكاملة لملف"* — `{ path: string }`)  
3. **حقن المخطط:** يقوم `McpToolProvider` بتغليف هذه المخططات المكتشفة ويجعلها متاحة لـ LangChain4j  
4. **قرار نموذج اللغة الكبير:** عند استدعاء `FileAgent.readFile(path)`، يرسل LangChain4j رسالة النظام، رسالة المستخدم، **وقائمة مخططات الأدوات** إلى نموذج اللغة الكبير. يقرأ نموذج اللغة الكبير أوصاف الأدوات ويولد استدعاء أداة (مثلًا، `read_file(path="/some/file.txt")`)  
5. **التنفيذ:** يعترض LangChain4j استدعاء الأداة، يمرره عبر عميل MCP مرة أخرى إلى العملية الفرعية Node.js، يحصل على النتيجة، ويرسلها مرة أخرى إلى نموذج اللغة الكبير  

هذه هي نفس آلية [اكتشاف الأداة](../../../05-mcp) الموضحة أعلاه، لكن يتم تطبيقها خصيصًا على سير عمل الوكلاء. تُرشد تعليقات `@SystemMessage` و `@UserMessage` سلوك نموذج اللغة الكبير، بينما تزوده أداة `ToolProvider` المحقونة بـ**القدرات** — حيث يقوم نموذج اللغة الكبير بربط الاثنين أثناء وقت التشغيل.

#### استراتيجيات الرد

عند تكوين `SupervisorAgent`، تحدد كيفية صياغة جوابه النهائي للمستخدم بعد انتهاء الوكلاء الفرعيين من مهامهم. يوضح الرسم البياني أدناه الاستراتيجيات الثلاث المتاحة — `LAST` يعيد مخرج الوكيل النهائي مباشرة، `SUMMARY` يلخص كل المخرجات من خلال نموذج اللغة الكبير، و`SCORED` يختار المخرج الأعلى تقييمًا مقابل الطلب الأصلي:

<img src="../../../translated_images/ar/response-strategies.3d0cea19d096bdf9.webp" alt="استراتيجيات الرد" width="800"/>

*ثلاث استراتيجيات لكيفية صياغة المشرف لرده النهائي — اختر بناءً على ما إذا كنت تريد مخرج الوكيل الأخير، ملخصًا مركبًا، أو الخيار الأعلى تقييمًا.*

الاستراتيجيات المتاحة هي:

| الاستراتيجية | الوصف |
|----------|-------------|
| **LAST** | يعيد المشرف مخرجات آخر وكيل فرعي أو أداة استدعيه. هذا مفيد عندما يكون الوكيل النهائي في سير العمل مصممًا خصيصًا لإنتاج الإجابة النهائية الكاملة (مثل "وكيل الملخص" في خط بحث). |
| **SUMMARY** | يستخدم المشرف نموذج اللغة الكبير الداخلي الخاص به لتوليف ملخص لكامل التفاعل وجميع إخراجات الوكلاء الفرعيين، ثم يعيد ذلك الملخص كاستجابة نهائية. يوفر هذا إجابة مجمعة نظيفة للمستخدم. |
| **SCORED** | يستخدم النظام نموذج لغة كبير داخلي لتقييم كل من رد LAST وملخص التفاعل مقابل طلب المستخدم الأصلي، ويعيد المخرج الذي يحصل على تقييم أعلى. |

#### فهم المخرجات

عند تشغيل العرض التوضيحي، سترى شرحًا منظمًا لكيفية تنسيق المشرف لعدة وكلاء. إليك ما يعنيه كل قسم:

```
======================================================================
  FILE → REPORT WORKFLOW DEMO
======================================================================

This demo shows a clear 2-step workflow: read a file, then generate a report.
The Supervisor orchestrates the agents automatically based on the request.
```
  
**العنوان** يقدم مفهوم سير العمل: خط أنابيب مركز من قراءة الملفات إلى إنشاء التقرير.

```
--- WORKFLOW ---------------------------------------------------------
  ┌─────────────┐      ┌──────────────┐
  │  FileAgent  │ ───▶ │ ReportAgent  │
  │ (MCP tools) │      │  (pure LLM)  │
  └─────────────┘      └──────────────┘
   outputKey:           outputKey:
   'fileContent'        'report'

--- AVAILABLE AGENTS -------------------------------------------------
  [FILE]   FileAgent   - Reads files via MCP → stores in 'fileContent'
  [REPORT] ReportAgent - Generates structured report → stores in 'report'
```
  
**مخطط سير العمل** يظهر تدفق البيانات بين الوكلاء. لكل وكيل دور محدد:  
- **FileAgent** يقرأ الملفات باستخدام أدوات MCP ويخزن المحتوى الخام في `fileContent`  
- **ReportAgent** يستهلك ذلك المحتوى وينتج تقريرًا منظمًا في `report`  

```
--- USER REQUEST -----------------------------------------------------
  "Read the file at .../file.txt and generate a report on its contents"
```
  
**طلب المستخدم** يُظهر المهمة. يقوم المشرف بتحليل هذا ويقرر استدعاء FileAgent → ReportAgent.

```
--- SUPERVISOR ORCHESTRATION -----------------------------------------
  The Supervisor decides which agents to invoke and passes data between them...

  +-- STEP 1: Supervisor chose -> FileAgent (reading file via MCP)
  |
  |   Input: .../file.txt
  |
  |   Result: LangChain4j is an open-source, provider-agnostic Java framework for building LLM...
  +-- [OK] FileAgent (reading file via MCP) completed

  +-- STEP 2: Supervisor chose -> ReportAgent (generating structured report)
  |
  |   Input: LangChain4j is an open-source, provider-agnostic Java framew...
  |
  |   Result: Executive Summary...
  +-- [OK] ReportAgent (generating structured report) completed
```
  
**تنسيق المشرف** يظهر تدفق الخطوتين قيد التنفيذ:  
1. **FileAgent** يقرأ الملف عبر MCP ويخزن المحتوى  
2. **ReportAgent** يستلم المحتوى وينشئ تقريرًا منظمًا  

قام المشرف باتخاذ هذه القرارات **بصورة مستقلة** بناءً على طلب المستخدم.

```
--- FINAL RESPONSE ---------------------------------------------------
Executive Summary
...

Key Points
...

Recommendations
...

--- AGENTIC SCOPE (Data Flow) ----------------------------------------
  Each agent stores its output for downstream agents to consume:
  * fileContent: LangChain4j is an open-source, provider-agnostic Java framework...
  * report: Executive Summary...
```
  
#### شرح ميزات الوحدة الوكيلية

يُظهر المثال عدة ميزات متقدمة لوحدة الوكلاء. دعونا نلقي نظرة أقرب على نطاق الوكيل ومستمعي الوكيل.

**نطاق الوكيل** يُظهر الذاكرة المشتركة حيث خزن الوكلاء نتائجهم باستخدام `@Agent(outputKey="...")`. يسمح ذلك بـ:  
- للوكلاء اللاحقين بالوصول إلى مخرجات الوكلاء السابقين  
- للمشرف بتوليف استجابة نهائية  
- لك بفحص ما أنتجه كل وكيل  

يُظهر الرسم البياني أدناه كيف يعمل نطاق الوكيل كذاكرة مشتركة في سير العمل من ملف إلى تقرير — يكتب FileAgent مخرجاته تحت المفتاح `fileContent`، يقرأ ReportAgent هذا ويكتب مخرجاته تحت `report`:

<img src="../../../translated_images/ar/agentic-scope.95ef488b6c1d02ef.webp" alt="ذاكرة مشتركة لنطاق الوكيل" width="800"/>

*يعمل نطاق الوكيل كذاكرة مشتركة — يكتب FileAgent `fileContent`، يقرأ ReportAgent ذلك ويكتب `report`، ويقرأ كودك النتيجة النهائية.*

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // بيانات ملف خام من FileAgent
String report = scope.readState("report");            // تقرير منظم من ReportAgent
```
  
**مستمعو الوكلاء** يمكّنون من مراقبة وتصحيح تنفيذ الوكلاء. المخرجات خطوة بخطوة التي تراها في العرض التوضيحي تأتي من مستمع وكيل يتصل بكل استدعاء وكيل:  
- **beforeAgentInvocation** - يُستدعى عند اختيار المشرف لوكيل، مما يسمح لك برؤية الوكيل المختار ولماذا  
- **afterAgentInvocation** - يُستدعى عند إكمال وكيل، ويعرض نتيجته  
- **inheritedBySubagents** - عند تعيينه true، يراقب المستمع جميع الوكلاء في التسلسل الهرمي  

يوضح الرسم البياني التالي دورة حياة مستمع الوكيل الكاملة، بما في ذلك كيفية تعامل `onError` مع الأخطاء أثناء تنفيذ الوكيل:

<img src="../../../translated_images/ar/agent-listeners.784bfc403c80ea13.webp" alt="دورة حياة مستمعي الوكلاء" width="800"/>

*مستمعو الوكلاء يتصلون بدورة حياة التنفيذ — يراقبون متى يبدأ الوكلاء، ينهون، أو يواجهون أخطاء.*

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // نشر إلى جميع الوكلاء الفرعيين
    }
};
```
  
بعيدًا عن نمط المشرف، توفر وحدة `langchain4j-agentic` عدة أنماط سير عمل قوية. يوضح الرسم البياني أدناه الأنماط الخمسة كلها — من خطوط أنابيب متسلسلة بسيطة إلى سير عمل بموافقة الإنسان في الحلقة:

<img src="../../../translated_images/ar/workflow-patterns.82b2cc5b0c5edb22.webp" alt="أنماط سير عمل الوكلاء" width="800"/>

*خمسة أنماط سير عمل لتنظيم الوكلاء — من خطوط أنابيب متسلسلة بسيطة إلى سير عمل موافقة الإنسان في الحلقة.*

| النمط | الوصف | حالة الاستخدام |
|---------|-------------|----------|
| **Sequential** | تنفيذ الوكلاء بالترتيب، يمر الإخراج إلى التالي | خطوط أنابيب: بحث → تحليل → تقرير |
| **Parallel** | تشغيل الوكلاء بالتوازي | مهام مستقلة: الطقس + الأخبار + الأسهم |
| **Loop** | تكرار حتى استيفاء شرط | تقييم الجودة: التكرار حتى يصبح التقييم ≥ 0.8 |
| **Conditional** | التوجيه بناءً على شروط | تصنيف → التوجيه إلى وكيل متخصص |
| **Human-in-the-Loop** | إضافة نقاط تحقق بشرية | سير عمل الموافقة، مراجعة المحتوى |

## المفاهيم الرئيسية

الآن بعد أن استكشفت MCP ووحدة الوكلاء في العمل، لنعرض ملخصًا متى تستخدم كل نهج.

أحد أكبر مزايا MCP هو نظامه البيئي المتنامي. يبين الرسم البياني أدناه كيف يربط بروتوكول عالمي تطبيق الذكاء الاصطناعي الخاص بك بمجموعة متنوعة من خوادم MCP — من الوصول إلى نظام الملفات وقواعد البيانات إلى GitHub، البريد الإلكتروني، استخراج الويب، والمزيد:

<img src="../../../translated_images/ar/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="نظام MCP البيئي" width="800"/>

*يخلق MCP نظامًا بيئيًا لبروتوكول عالمي — يعمل أي خادم متوافق مع MCP مع أي عميل متوافق مع MCP، مما يمكّن مشاركة الأدوات عبر التطبيقات.*

**MCP** مثالي عندما تريد الاستفادة من نظم الأدوات القائمة، بناء أدوات يمكن للعديد من التطبيقات مشاركتها، دمج خدمات الطرف الثالث ببروتوكولات قياسية، أو استبدال تنفيذات الأدوات دون تغيير الكود.

**وحدة الوكيل** تعمل بشكل أفضل عندما تريد تعريفات وكلاء إعلانية مع تعليقات `@Agent`، تحتاج إلى تنسيق سير العمل (متسلسل، حلقة، متوازي)، تفضل تصميم الوكيل القائم على الواجهات بدلاً من الكود الإجرائي، أو تجمع بين وكلاء متعددين يشاركون المخرجات عبر `outputKey`.

**نمط وكيل المشرف** يتألق عندما يكون سير العمل غير متوقع مسبقًا وتريد من نموذج اللغة الكبير أن يقرر، عندما لديك وكلاء متخصصين متعددين يحتاجون تنسيقًا ديناميكيًا، عند بناء أنظمة محادثة توجه إلى قدرات مختلفة، أو عندما تريد السلوك الوكلي الأكثر مرونة وتكيفًا.

لمساعدتك في الاختيار بين طرق `@Tool` المخصصة من الوحدة 04 وأدوات MCP من هذه الوحدة، يبرز المقارنة التالية التنازلات الرئيسية — الأدوات المخصصة تمنحك تقاربًا وثيقًا وأمان أنواع كامل للمنطق الخاص بالتطبيق، بينما تقدم أدوات MCP تكاملات معيارية قابلة لإعادة الاستخدام:

<img src="../../../translated_images/ar/custom-vs-mcp-tools.c4f9b6b1cb65d8a1.webp" alt="الأدوات المخصصة مقابل أدوات MCP" width="800"/>

*متى تستخدم طرق @Tool مخصصة مقابل أدوات MCP — أدوات مخصصة للمنطق الخاص بالتطبيق مع أمان نوع كامل، أدوات MCP للتكاملات المعيارية التي تعمل عبر التطبيقات.*
