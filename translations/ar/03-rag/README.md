## ما ستتعلمه

في الوحدات السابقة، تعلمت كيف تجري محادثات مع الذكاء الاصطناعي وتنظم مطالباتك بفعالية. لكن هناك قيد أساسي: نماذج اللغة تعرف فقط ما تعلمته أثناء التدريب. لا يمكنها الإجابة عن أسئلة تتعلق بسياسات شركتك، أو توثيق مشروعك، أو أي معلومات لم يتم تدريبها عليها.

يحل RAG (التوليد المعزز بالاستخراج) هذه المشكلة. بدلاً من محاولة تعليم النموذج معلوماتك (وهو أمر مكلف وغير عملي)، تمنحه القدرة على البحث في مستنداتك. عندما يطرح شخص سؤالاً، يجد النظام المعلومات ذات الصلة ويُدرجها في المطالبة. ثم يجيب النموذج بناءً على ذلك السياق المسترجع.

فكر في RAG كأنه يعطي النموذج مكتبة مرجعية. عندما تطرح سؤالاً، يقوم النظام بـ:

1. **طلب المستخدم** - تطرح سؤالاً  
2. **التضمين** - يحول سؤالك إلى متجه  
3. **البحث في المتجهات** - يجد أجزاء من المستندات مشابهة  
4. **تجميع السياق** - يضيف الأجزاء ذات الصلة إلى المطالبة  
5. **الاستجابة** - يولد النموذج إجابة بناءً على السياق  

هذا يجعل ردود النموذج تستند إلى بياناتك الفعلية بدلاً من الاعتماد على معرفته التدريبية أو اختلاق الإجابات.

## كيف يعمل

### معالجة المستندات

[DocumentService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/DocumentService.java)

عندما ترفع مستندًا، يقوم النظام بتحليله (PDF أو نص عادي)، ويرفق بيانات وصفية مثل اسم الملف، ثم يقسمه إلى أجزاء — قطع أصغر تناسب نافذة السياق للنموذج. هذه الأجزاء تتداخل قليلاً حتى لا تفقد السياق عند الحواف.

```java
// تحليل الملف المحمّل ولفّه في وثيقة LangChain4j
Document document = Document.from(content, metadata);

// تقسيم إلى أجزاء بطول 300 رمز مع تداخل 30 رمزًا
DocumentSplitter splitter = DocumentSplitters
    .recursive(300, 30);

List<TextSegment> segments = splitter.split(document);
```
  
يوضح الرسم البياني أدناه كيفية عمل ذلك بصريًا. لاحظ كيف يشارك كل جزء بعض الرموز مع جيرانه — التداخل بحدود 30 رمزًا يضمن عدم فقدان أي سياق مهم بين الأجزاء:

<img src="../../../translated_images/ar/document-chunking.a5df1dd1383431ed.webp" alt="تجزئة المستندات" width="800"/>

### إنشاء التضمينات

[LangChainRagConfig.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/config/LangChainRagConfig.java)

يتم تحويل كل جزء إلى تمثيل رقمي يسمى تضمين - وهو أساسًا بصمة رياضية تلتقط معنى النص. النصوص المشابهة تنتج تضمينات مشابهة.

```java
@Bean
public EmbeddingModel embeddingModel() {
    return OpenAiOfficialEmbeddingModel.builder()
        .baseUrl(azureOpenAiEndpoint)
        .apiKey(azureOpenAiKey)
        .modelName(azureEmbeddingDeploymentName)
        .build();
}

EmbeddingStore<TextSegment> embeddingStore = 
    new InMemoryEmbeddingStore<>();
```
  
يوضح مخطط الطبقات أدناه كيف ترتبط هذه مكونات LangChain4j. `OpenAiOfficialEmbeddingModel` يحول النصوص إلى متجهات، و`InMemoryEmbeddingStore` يحتفظ بهذه المتجهات مع بيانات `TextSegment` الأصلية، و`EmbeddingSearchRequest` يتحكم في معايير الاسترجاع مثل `maxResults` و`minScore`:

<img src="../../../translated_images/ar/rag-langchain4j-classes.bbf3aa9077ab443d.webp" alt="فئات LangChain4j RAG" width="800"/>

بمجرد تخزين التضمينات، يتجمع المحتوى المشابه طبيعيًا في فضاء المتجهات. يوضح التصور التالي كيف ينتهي المستندات المتعلقة بمواضيع متقاربة إلى نقاط قريبة، وهذا ما يجعل البحث الدلالي ممكنًا:

<img src="../../../translated_images/ar/vector-embeddings.2ef7bdddac79a327.webp" alt="فضاء تضمينات المتجهات" width="800"/>

### البحث الدلالي

[RagService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/RagService.java)

عندما تطرح سؤالاً، يتحول سؤالك أيضًا إلى تضمين. يقارن النظام تضمين السؤال مع جميع تضمينات أجزاء المستندات. يجد الأجزاء التي تحمل معاني مشابهة - ليس فقط مطابقات كلمات رئيسية، بل تشابه دلالي فعلي.

```java
Embedding queryEmbedding = embeddingModel.embed(question).content();

EmbeddingSearchRequest searchRequest = EmbeddingSearchRequest.builder()
    .queryEmbedding(queryEmbedding)
    .maxResults(5)
    .minScore(0.5)
    .build();

EmbeddingSearchResult<TextSegment> searchResult = embeddingStore.search(searchRequest);
List<EmbeddingMatch<TextSegment>> matches = searchResult.matches();

for (EmbeddingMatch<TextSegment> match : matches) {
    String relevantText = match.embedded().text();
    double score = match.score();
}
```
  
يوضح الرسم البياني أدناه الفرق بين البحث الدلالي والبحث التقليدي بالكلمات المفتاحية. بحث بالكلمة المفتاحية "vehicle" يفوت جزءًا عن "cars and trucks"، بينما يفهم البحث الدلالي أنهما يعنيان نفس الشيء ويعيده كنتيجة ذات درجة عالية:

<img src="../../../translated_images/ar/semantic-search.6b790f21c86b849d.webp" alt="البحث الدلالي" width="800"/>

> **🤖 جرب مع [GitHub Copilot](https://github.com/features/copilot) Chat:** افتح [`RagService.java`](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/RagService.java) واسأل:

> - "كيف أتعامل مع الحالات التي لا توجد فيها مستندات ذات صلة؟"

### توليد الإجابات

[RagService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/RagService.java)

يتم تجميع أكثر الأجزاء صلة في مطالبة منظمة تتضمن تعليمات صريحة، والسياق المسترجع، وسؤال المستخدم. يقرأ النموذج هذه الأجزاء المحددة ويجيب بناءً على تلك المعلومات — يمكنه استخدام ما هو أمامه فقط، مما يمنع التخيلات.

```java
String context = matches.stream()
    .map(match -> match.embedded().text())
    .collect(Collectors.joining("\n\n"));

String prompt = String.format("""
    Answer the question based on the following context.
    If the answer cannot be found in the context, say so.

    Context:
    %s

    Question: %s

    Answer:""", context, request.question());

String answer = chatModel.chat(prompt);
```
  
يوضح الرسم البياني أدناه هذا التجميع في العمل — يتم حقن الأجزاء الأعلى تصنيفًا من خطوة البحث في قالب المطالبة، و`OpenAiOfficialChatModel` يولد إجابة مؤسسية:

<img src="../../../translated_images/ar/context-assembly.7e6dd60c31f95978.webp" alt="تجميع السياق" width="800"/>

## استخدام التطبيق

التطبيق يوفر واجهة ويب لتحميل الوثائق وطرح الأسئلة.

<a href="images/rag-homepage.png"><img src="../../../translated_images/ar/rag-homepage.d90eb5ce1b3caa94.webp" alt="واجهة تطبيق RAG" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*واجهة تطبيق RAG - ارفع المستندات واطرح الأسئلة*

### رفع مستند

ابدأ برفع مستند - ملفات TXT هي الأفضل للاختبار. يتوفر ملف `sample-document.txt` في هذا الدليل يحتوي على معلومات حول ميزات LangChain4j، وتنفيذ RAG، وأفضل الممارسات - مثالي لاختبار النظام.

يقوم النظام بمعالجة مستندك، ويقسمه إلى قطع، وينشئ تضمينات لكل قطعة. يحدث هذا تلقائيًا عند الرفع.

### طرح الأسئلة

اطرح الآن أسئلة محددة عن محتوى المستند. جرب شيئًا حقائقيًا واضحًا في المستند. يبحث النظام عن الأجزاء ذات الصلة، ويُدرجها في المطالبة، ويولد إجابة.

### التحقق من المراجع المصدرية

لاحظ أن كل إجابة تتضمن مراجع مصادر مع درجات تشابه. هذه الدرجات (من 0 إلى 1) تُظهر مدى صلة كل جزء بسؤالك. الدرجات الأعلى تعني تطابقًا أفضل. هذا يسمح لك بالتحقق من الإجابة مقابل المادة المصدرية.

<a href="images/rag-query-results.png"><img src="../../../translated_images/ar/rag-query-results.6d69fcec5397f355.webp" alt="نتائج استعلام RAG" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*نتائج الاستعلام تُظهر الإجابة مع المراجع المصدرية ودرجات الصلة*

### التجربة مع الأسئلة

جرب أنواعًا مختلفة من الأسئلة:  
- حقائق محددة: "ما هو الموضوع الرئيسي؟"  
- المقارنات: "ما الفرق بين X و Y؟"  
- الملخصات: "لخص النقاط الرئيسية حول Z"

شاهد كيف تتغير درجات الصلة بناءً على مدى تطابق سؤالك مع محتوى المستند.

## المفاهيم الرئيسية

### استراتيجية التجزئة

يتم تقسيم المستندات إلى قطع مكونة من 300 رمز مع تداخل 30 رمزًا. هذا التوازن يضمن أن كل قطعة تحتوي على سياق كافٍ لتكون ذات معنى مع البقاء صغيرة بما يكفي لتضمين عدة قطع في المطالبة.

### درجات التشابه

كل جزء مسترجع يأتي مع درجة تشابه بين 0 و 1 تشير إلى مدى قرب تطابقه مع سؤال المستخدم. يوضح الرسم البياني أدناه نطاق الدرجات وكيفية استخدام النظام لها لتصفية النتائج:

<img src="../../../translated_images/ar/similarity-scores.b0716aa911abf7f0.webp" alt="درجات التشابه" width="800"/>

الدرجات تتراوح من 0 إلى 1:  
- 0.7-1.0: ذو صلة عالية، تطابق دقيق  
- 0.5-0.7: ذو صلة، سياق جيد  
- أقل من 0.5: مفلترة، غير مشابهة جداً  

النظام يسترجع فقط القطع التي تفوق الحد الأدنى لضمان الجودة.

### التخزين في الذاكرة

تستخدم هذه الوحدة التخزين في الذاكرة للبساطة. عند إعادة تشغيل التطبيق، يتم فقدان المستندات المرفوعة. تستخدم أنظمة الإنتاج قواعد بيانات متجهات دائمة مثل Qdrant أو Azure AI Search.

### إدارة نافذة السياق

لكل نموذج نافذة سياق قصوى. لا يمكنك تضمين كل قطعة من مستند كبير. يسترجع النظام أعلى N قطع ذات صلة (افتراضياً 5) للبقاء ضمن الحدود مع توفير سياق كافٍ للحصول على إجابات دقيقة.

## متى يكون RAG مهماً

ليس RAG دائماً النهج الصحيح. تساعدك دليل القرار أدناه في تحديد متى يضيف RAG قيمة مقابل متى تكون الطرق الأبسط — مثل تضمين المحتوى مباشرة في المطالبة أو الاعتماد على المعرفة المدمجة في النموذج — كافية:

<img src="../../../translated_images/ar/when-to-use-rag.1016223f6fea26bc.webp" alt="متى تستخدم RAG" width="800"/>

**استخدم RAG عندما:**
- الإجابة على الأسئلة حول المستندات الملكية
- المعلومات تتغير بشكل متكرر (السياسات، الأسعار، المواصفات)
- تتطلب الدقة نسب المصدر
- المحتوى كبير جدًا بحيث لا يمكن وضعه في مطالبة واحدة
- تحتاج إلى إجابات موثوقة وقائمة على مصادر

**لا تستخدم RAG عندما:**
- تتطلب الأسئلة معرفة عامة يمتلكها النموذج بالفعل
- الحاجة إلى بيانات في الوقت الحقيقي (يعمل RAG على المستندات المرفوعة)
- المحتوى صغير بما يكفي ليُدرج مباشرة في المطالبات
