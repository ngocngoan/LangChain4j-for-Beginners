## आपण काय शिकाल

आत्तापर्यंत, आपण AI सोबत संवाद साधण्याबाबत, प्रॉम्प्ट्स प्रभावी रित्या कशी तयार करावी व आपल्या दस्तऐवजा मध्ये उत्तर कसे स्थिर करावे हे शिकलात. पण एक मूलभूत मर्यादा अजूनही आहे: भाषा मॉडेल्स फक्त मजकूर तयार करू शकतात. ते हवामान तपासू शकत नाहीत, गणित करू शकत नाहीत, डेटाबेस क्वेरी करू शकत नाहीत किंवा बाह्य प्रणालींसाठी संवाद करू शकत नाहीत.

साधने हे बदलतात. मॉडेलला कॉल करू शकणाऱ्या फंक्शन्समधून प्रवेश दिल्यामुळे, आपण त्याला फक्त मजकूर निर्माते नव्हे तर कृती करणारा एजंट बनवता. मॉडेल ठरवते की कोणत्या वेळी कोणते साधन वापरायचे आणि कोणते पॅरामीटर्स द्यायचे. आपला कोड त्या फंक्शनला अंमलात आणतो आणि परिणाम परत करतो. मॉडेल त्या परिणामाला त्याच्या उत्तरात समाविष्ट करते.

## साधनांसह AI एजंट्स समजून घेणे

> **📝 टीप:** ह्या मोड्यूलमधील "एजंट्स" हा शब्द साधन कॉलिंग क्षमता असलेल्या AI सहाय्यकांसाठी आहे. हे **Agentic AI** नमुन्यांपेक्षा वेगळे आहे (स्वायत्त एजंट्स जे नियोजन, आठवण व बहु-टप्पीय कारणशक्ती वापरतात) जे आपण [Module 05: MCP](../05-mcp/README.md) मध्ये पाहणार आहोत.

साधने नसल्यास, भाषा मॉडेल फक्त त्याच्या प्रशिक्षण डेटावरून मजकूर तयार करू शकते. जर त्याला सध्याचे हवामान विचारले तर त्याला अंदाज लावावा लागतो. साधने दिली तर ते हवामान API कॉल करू शकते, गणित करू शकते, किंवा डेटाबेस क्वेरी करू शकते — नंतर ते खरे निकाल त्याच्या उत्तरात गुंफते.

<img src="../../../translated_images/mr/what-are-tools.724e468fc4de64da.webp" alt="साधने नसल्यास बनाम साधने असताना" width="800"/>

*साधने नसल्यास मॉडेल फक्त अंदाज लावते — साधने असल्यास ते API कॉल, गणना, आणि रिअल-टाइम डेटा परत करू शकते.*

साधनांसह AI एजंट एक **ReAct (Reasoning and Acting)** नमुन्याचा अवलंब करतो. मॉडेल फक्त उत्तर देत नाही — ते काय हवे आहे याचा विचार करते, साधन कॉल करून कृती करते, परिणाम पाहते, आणि मग पुन्हा क्रिया करायची की अंतिम उत्तर द्यायचे निर्णय घेते:

1. **विचार करा** — एजंट वापरकर्त्याच्या प्रश्नाचा अन्वेषण करतो व त्याला कोणती माहिती हवी आहे ते ठरवतो
2. **कृती करा** — एजंट योग्य साधन निवडतो, योग्य पॅरामीटर्स निर्माण करतो आणि कॉल करतो
3. **पाहणी करा** — एजंट साधनाचा आउटपुट घेतो आणि परिणाम मूल्यांकन करतो
4. **पुन्हा करा किंवा उत्तर द्या** — अधिक डेटा आवश्यक असल्यास एजंट लूप मध्ये जातो; अन्यथा, नैसर्गिक भाषेतील उत्तरे तयार करतो

<img src="../../../translated_images/mr/react-pattern-detail.96a5efeeb6dd2f61.webp" alt="ReAct नमुना" width="800"/>

*ReAct चक्र — एजंट काय करायचे आहे याचा विचार करतो, साधन कॉल करून कृती करतो, परिणामी पाहतो, आणि अंतिम उत्तरासाठी लूप करतो.*

हे स्वयंचलितपणे होते. आपण साधने आणि त्यांची वर्णने परिभाषित करता. मॉडेल कधी आणि कसे वापरायचे हे निर्णय घेतो.

## साधन कॉल कसे कार्य करते

### साधन व्याख्या

[WeatherTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) | [TemperatureTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/TemperatureTool.java)

आपण फंक्शन्स स्पष्ट वर्णन आणि पॅरामीटर विशिष्टता सह परिभाषित करता. मॉडेल सिस्टम प्रॉम्प्टमध्ये ही वर्णने पाहते आणि प्रत्येक साधन काय करते ते समजते.

```java
@Component
public class WeatherTool {
    
    @Tool("Get the current weather for a location")
    public String getCurrentWeather(@P("Location name") String location) {
        // तुमची हवामान शोध तंत्र
        return "Weather in " + location + ": 22°C, cloudy";
    }
}

@AiService
public interface Assistant {
    String chat(@MemoryId String sessionId, @UserMessage String message);
}

// सहाय्यक स्प्रिंग बूटद्वारे स्वयंचलितपणे वायर केला जातो:
// - ChatModel बीन
// - @Component वर्गातील सर्व @Tool पद्धती
// - सत्र व्यवस्थापनासाठी ChatMemoryProvider
```
  
खालील आकृती प्रत्येक अॅनोटेशन कसे कार्य करते आणि AI ला कधी साधन कॉल करायचे आणि कोणते आर्ग्युमेंट्स द्यायचे हे कसे समजते हे दाखवते:

<img src="../../../translated_images/mr/tool-definitions-anatomy.f6468546037cf28b.webp" alt="साधन व्याख्या रचना" width="800"/>

*साधन व्याख्येची रचना — @Tool AI ला कधी वापरायचे ते सांगते, @P प्रत्येक पॅरामीटरचे वर्णन करतो, आणि @AiService स्टार्टअपला सर्व एकत्र करतो.*

> **🤖 [GitHub Copilot](https://github.com/features/copilot) चा वापर करून पाहा:** [`WeatherTool.java`](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) उघडा आणि विचारा:
> - "मी mock डेटा ऐवजी खर्या OpenWeatherMap API कसे समाकलित करू?"
> - "AI ला योग्य वापरासाठी मदत करणारी चांगली साधन वर्णन कशी लिहावी?"
> - "API त्रुटी आणि रेट लिमिट कसे हाताळावे साधन अंमलबजावणींमध्ये?"

### निर्णय घेणे

जेव्हा वापरकर्ता "सिएटलमध्ये हवामान काय आहे?" विचारतो, तेव्हा मॉडेल यादृच्छिकपणे कोणतेही साधन निवत नाही. ते वापरकर्त्याच्या हेतूची तुलना प्रत्येक साधन वर्णनाशी करते, प्रत्येकाला सुसंगततेचा गुणांकन देते, आणि सर्वोत्तम जुळवणी निवडते. मग योग्य पॅरामीटर्ससह संरचित फंक्शन कॉल तयार करते — या प्रकरणात `location` ला `"Seattle"` सेट करते.

जर कोणतेही साधन वापरकर्त्याच्या विनंतीशी जुळत नसेल, तर मॉडेल त्याच्या स्वतःच्या ज्ञानावरून उत्तर देते. जर अनेक साधने जुळली, तर ते सर्वात विशिष्ट साधन निवडते.

<img src="../../../translated_images/mr/decision-making.409cd562e5cecc49.webp" alt="AI कोणते साधन वापरावे याचा निर्णय कसा घेतो" width="800"/>

*मॉडेल प्रत्येक उपलब्ध साधन वापरकर्त्याच्या हेतूशी तुलनात्मक मूल्यांकन करते आणि सर्वोत्तम जुळवणी निवडते — त्यामुळे स्पष्ट, विशिष्ट साधन वर्णन लिहिणे महत्त्वाचे आहे.*

### प्रतिसाद निर्मिती

मॉडेल हवामान डेटा प्राप्त करते आणि वापरकर्त्यासाठी नैसर्गिक भाषेत उत्तर स्वरूपित करते.

### आर्किटेक्चर: Spring Boot ऑटो-वायरिंग

हा मोड्यूल LangChain4j च्या Spring Boot इंटिग्रेशनचा वापर करतो डिक्लेरेटिव्ह `@AiService` इंटरफेसेससह. स्टार्टअप वेळी Spring Boot प्रत्येक `@Component` शोधतो ज्यात `@Tool` मेथड्स आहेत, आपला `ChatModel` बीन, आणि `ChatMemoryProvider` — मग सर्व एकाच `Assistant` इंटरफेसमध्ये वायर करतो शून्य बोईलरप्लेटसह.

<img src="../../../translated_images/mr/spring-boot-wiring.151321795988b04e.webp" alt="Spring Boot ऑटो-वायरिंग आर्किटेक्चर" width="800"/>

*@AiService इंटरफेस ChatModel, साधन घटक, आणि मेमरी प्रोवाइडर एकत्र लिंक करतो — Spring Boot सर्व वायरिंग आपोआप हाताळतो.*

या पद्धतीचे मुख्य फायदे:

- **Spring Boot ऑटो-वायरिंग** — ChatModel व साधने आपोआप इंजेक्ट होतात
- **@MemoryId नमुना** — सेशन-आधारित मेमरी व्यवस्थापन स्वयंचलित
- **सिंगल इंस्टन्स** — Assistant एकदाच तयार करतो आणि पुनः वापरतो ज्यामुळे कामगिरी सुधारते
- **टाइप-सेफ अंमलबजावणी** — जावा मेथड्स थेट कॉल करून टाइप रूपांतरणासह
- **मल्टी-टर्न ऑर्केस्ट्रेशन** — साधन साखळी आपोआप हाताळतो
- **शून्य बोईलरप्लेट** — कोणतेही मॅन्युअल `AiServices.builder()` कॉल किंवा मेमरी HashMap नाही

पर्यायी पद्धती (मॅन्युअल `AiServices.builder()`) अधिक कोडची गरज आहे आणि Spring Boot इंटिग्रेशन फायदे गहाळ पडतात.

## साधन साखळी बांधणी

**साधन साखळी बांधणी** — साधन-आधारित एजंट्सची खरी ताकद तेव्हा दिसून येते जेव्हा एकाच प्रश्नासाठी एकाहून अधिक साधनांची गरज असते. प्रश्न विचाराः "सिएटलमध्ये हवामान फॅरेनहाइटमध्ये काय आहे?" आणि एजंट आपोआप दोन साधने साखळी करतो: प्रथम `getCurrentWeather` कॉल करून सेल्सियस तापमान घेतो, नंतर तो मूल्य `celsiusToFahrenheit` कडे रूपांतरणासाठी पाठवतो — सर्व एका संवाद टप्प्यात.

<img src="../../../translated_images/mr/tool-chaining-example.538203e73d09dd82.webp" alt="साधन साखळी उदाहरण" width="800"/>

*साधन साखळी कार्यरत — एजंट प्रथम getCurrentWeather कॉल करतो, नंतर सेल्सियस परिणाम celsiusToFahrenheit कडे वळवतो, आणि एकत्रित उत्तर देतो.*

**सुंदर अपयश हाताळणी** — mock डेटामध्ये नसलेल्या शहरात हवामान विचारल्यास साधन त्रुटी संदेश देते, आणि AI सांगते की मदत करू शकत नाही, संपूर्ण ऍप क्रॅश होत नाही. साधने सुरक्षितपणे अपयशी होतात. खालील चित्र दोन पद्धती दर्शवते — योग्य त्रुटी हाताळणीने, एजंट अपवाद पकडतो आणि मदत करणारे उत्तर देतो, अन्यथा संपूर्ण ऍप क्रॅश होते:

<img src="../../../translated_images/mr/error-handling-flow.9a330ffc8ee0475c.webp" alt="त्रुटी हाताळणी प्रवाह" width="800"/>

*साधन अपयशी झाल्यावर, एजंट त्रुटी पकडतो आणि क्रॅश वगळून उपयुक्त स्पष्टीकरण देतो.*

हे एका संवाद टप्प्यात होते. एजंट स्वयंचलित पद्धतीने अनेक साधन कॉल ऑर्केस्ट्रेट करतो.

## अर्ज वापरणे

अर्ज वेब इंटरफेस प्रदान करतो जिथे आपण अशा AI एजंटशी संवाद साधू शकता ज्याला हवामान आणि तापमान रूपांतरण साधनांची प्रवेश आहे. इंटरफेस कसा दिसतो ते खाली दिलेले आहे — यात त्वरित सुरूवातीसाठी उदाहरणे आणि विनंत्या पाठवण्यासाठी चॅट पॅनेल समाविष्ट आहे:
<a href="images/tools-homepage.png"><img src="../../../translated_images/mr/tools-homepage.4b4cd8b2717f9621.webp" alt="AI एजंट उपकरण इंटरफेस" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*AI एजंट उपकरण इंटरफेस - जलद उदाहरणे आणि उपकरणांशी संवादासाठी चॅट इंटरफेस*

### सोप्या उपकरण वापराचा प्रयत्न करा

सरळ विनंतीने प्रारंभ करा: "100 डिग्री फॅरेनहाइट ते सेलसिअसमध्ये रूपांतर करा". एजंटला तापमान रूपांतर उपकरणाची गरज आहे हे ओळखते, योग्य पॅरामीटर्ससह त्याला कॉल करते आणि निकाल परत करते. हे किती नैसर्गिक वाटते याकडे लक्ष द्या - तुम्ही कोणते उपकरण वापरायचे आहे किंवा ते कसे कॉल करायचे आहे हे निर्दिष्ट केलेले नाही.

### उपकरण साखळी चाचणी करा

आता काहीतरी अधिक क्लिष्ट प्रयत्न करा: "सिएटलमध्ये हवामान काय आहे आणि ते फॅरेनहाइट मध्ये रूपांतर करा?" पाहा कसे एजंट हे टप्प्याटप्प्याने पार पाडतो. तो प्रथम हवामान मिळवतो (जे सेलसिअसमध्ये परत करते), नंतर फॅरेनहाइटमध्ये रूपांतर करायची गरज ओळखतो, रूपांतर उपकरणाला कॉल करतो, आणि दोन्ही निकाल एकत्र करून एक उत्तर तयार करतो.

### संभाषण प्रवाह पाहा

चॅट इंटरफेस संभाषण इतिहास राखतो, ज्यामुळे तुम्हाला बहु-टर्न संवाद शक्य होतो. तुम्ही सर्व मागील प्रश्न आणि उत्तरे पाहू शकता, ज्यामुळे संभाषणाचा मागोवा घेणे आणि एजंट कसे अनेक विनिमयांमध्ये संदर्भ तयार करतो हे समजणे सोपे होते.

<a href="images/tools-conversation-demo.png"><img src="../../../translated_images/mr/tools-conversation-demo.89f2ce9676080f59.webp" alt="अनेक उपकरण कॉलसह संभाषण" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*बहु-टर्न संभाषण ज्यात सोपे रूपांतरण, हवामान शोध आणि उपकरण साखळी दाखवली आहे*

### विविध विनंत्यांसह प्रयोग करा

विविध संयोजने वापरून पहा:
- हवामान शोध: "टोकिओमध्ये हवामान काय आहे?"
- तापमान रूपांतरण: "25°C हे केल्विन मध्ये किती आहे?"
- संयुक्त प्रश्न: "पॅरिसमधील हवामान पहा आणि मला सांगा की ते 20°C पेक्षा जास्त आहे का"

लक्षात ठेवा कसे एजंट नैसर्गिक भाषा समजतो आणि तशीच योग्य उपकरण कॉलमध्ये अनुवाद करतो.

## मुख्य संकल्पना

### ReAct पॅटर्न (विचार करणे आणि क्रिया करणे)

एजंट विचार (काय करायचे ते ठरवणे) आणि कृती (उपकरणे वापरणे) यामध्ये फेरफार करतो. हा पॅटर्न फक्त सूचनांना प्रतिसाद देण्याऐवजी स्वयंचलित समस्या सोडवण्यास सक्षम करतो.

### उपकरण वर्णन महत्त्वाची असते

तुमच्या उपकरण वर्णनांची गुणवत्ता थेट एजंट कसे त्या उपकरणांचा वापर करतो यावर परिणाम करते. स्पष्ट, विशिष्ट वर्णने मॉडेलला कधी आणि कसे उपकरणाला कॉल करावे हे समजून घेण्यास मदत करतात.

### सेशन व्यवस्थापन

`@MemoryId` चिन्हांकन स्वयंचलित सेशन-आधारित स्मृती व्यवस्थापन सक्षम करते. प्रत्येक सेशन आयडीला स्वतःची `ChatMemory` उदाहरण देण्यात येते जी `ChatMemoryProvider` बीनने व्यवस्थापित केली जाते, त्यामुळे अनेक वापरकर्ते वेगवेगळ्या संभाषणांमध्ये एकत्र मिसळले जात नाहीत. खालील आकृतीमध्ये कसे अनेक वापरकर्ते त्यांच्या सेशन आयडींवर आधारित वेगळ्या स्मृती स्टोअरसाठी मार्गदर्शित होतात ते दाखवले आहे:

<img src="../../../translated_images/mr/session-management.91ad819c6c89c400.webp" alt="@MemoryId सह सेशन व्यवस्थापन" width="800"/>

*प्रत्येक सेशन आयडी वेगळ्या संभाषण इतिहासाशी नकाशित होतो — वापरकर्त्यांना एकमेकांचे संदेश कधीही दिसत नाहीत.*

### त्रुटी हाताळणी

उपकरणे अयशस्वी होऊ शकतात — API वेळ संपणे, पॅरामीटर्स अवैध असणे, बाह्य सेवा बंद होणे. उत्पादन एजंट्सना त्रुटी हाताळणीची गरज असते जेणेकरून मॉडेल समस्यांचे स्पष्टीकरण देऊ शकेल किंवा संपूर्ण अनुप्रयोग क्रॅश होण्याऐवजी पर्यायांची चाचणी करू शकेल. जेव्हा उपकरण अपवाद टाकते, तेव्हा LangChain4j ते पकडतो आणि त्रुटी संदेश मॉडेलला परत देतो, जेणेकरून ते नैसर्गिक भाषेत समस्या स्पष्ट करू शकतात.

## उपलब्ध उपकरणे

खालील आकृतीमध्ये तुम्ही तयार करू शकणाऱ्या उपकरणांच्या विस्तृत परिसंस्थेचे दर्शन होते. हा मोड्यूल हवामान आणि तापमान उपकरणे दाखवतो, परंतु त्याच `@Tool` पॅटर्नचा वापर कोणत्याही जावा पद्धतीसाठी केला जाऊ शकतो — डेटाबेस क्वेरीजपासून ते पेमेंट प्रक्रिया पर्यंत.

<img src="../../../translated_images/mr/tool-ecosystem.aad3d74eaa14a44f.webp" alt="उपकरण परिसंस्था" width="800"/>

*कोणतीही जावा पद्धत ज्यावर @Toolचे अॅनोटेशन आहे ती AI साठी उपलब्ध होते — हा पॅटर्न डेटाबेस, API, ईमेल, फाइल ऑपरेशन्स आणि बरेच काही विस्तारतो.*

## कधी टूल-आधारित एजंट वापरायचे

प्रत्येक विनंतीस उपकरणांची गरज नसते. निर्णय येतो की AI ला कोणत्याही बाह्य प्रणालीशी संपर्क साधायचा आहे का किंवा तो स्वतःच्या ज्ञानावरून उत्तर देऊ शकतो का. खालील मार्गदर्शक सारांश करते की कधी उपकरणे मूल्य वाढवतात आणि कधी त्यांची गरज नसते:

<img src="../../../translated_images/mr/when-to-use-tools.51d1592d9cbdae9c.webp" alt="कधी उपकरणे वापरायची" width="800"/>

*जलद निर्णय मार्गदर्शक — उपकरणे रिअल-टाइम डेटा, गणना आणि क्रियांसाठी आहेत; सामान्य ज्ञान आणि सर्जनशील कामांसाठी त्यांची गरज नसते.*

## उपकरणे विरुद्ध RAG

मोड्यूल 03 आणि 04 दोन्ही AI काय करू शकतो हा विस्तार करतात, पण मूलतत्वतः भिन्न मार्गाने. RAG मॉडेलला **ज्ञान** मिळवण्यासाठी दस्तऐवज मिळवण्याची परवानगी देतो. उपकरणे मॉडेलला कार्ये करण्याची क्षमता देतात (फंक्शन्स कॉल करून). खालील आकृती या दोन पद्धतींची बाजूने तुलना करते — प्रत्येक वर्कफ्लो कसा चालतो ते आणि त्यामधील समायोजने:

<img src="../../../translated_images/mr/tools-vs-rag.ad55ce10d7e4da87.webp" alt="उपकरणे विरुद्ध RAG तुलना" width="800"/>

*RAG स्थिर दस्तऐवजांमधून माहिती मिळवतो — उपकरणे क्रियांचे कार्यान्वयन करतात आणि डायनॅमिक, रिअल-टाइम डेटा घेतात. अनेक उत्पादन प्रणाली दोन्ही एकत्र करतात.*

प्रत्यक्षात, अनेक उत्पादन प्रणाली दोन्ही पद्धतींचा संगम करतात: RAG तुमच्या दस्तऐवजीकरणात आधार देण्यासाठी, आणि उपकरणे लाईव्ह डेटा मिळवण्यासाठी किंवा ऑपरेशन्स करण्यासाठी.
