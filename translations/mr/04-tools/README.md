## तुम्ही काय शिकाल

आत्तापर्यंत, तुम्ही AI सोबत संवाद कसा करायचा, प्रभावीपणे प्रॉम्प्ट कसा तयार करायचा, आणि प्रतिसाद कसे तुमच्या कागदपत्रांमध्ये आधारलेले असावेत हे शिकले आहे. पण अजूनही एक मूलभूत मर्यादा आहे: भाषा मॉडेल्स फक्त मजकूर तयार करू शकतात. ते हवामान तपासू शकत नाहीत, गणित करू शकत नाहीत, डेटाबेस क्वेरी करू शकत नाहीत किंवा बाह्य प्रणालींशी संवाद साधू शकत नाहीत.

साधने याला बदलतात. मॉडेलला त्याच्याकडे कॉल करू शकणाऱ्या फंक्शन्सची प्रवेश दिल्याने, तुम्ही त्याला एक मजकूर उत्पन्न करणारे मॉडेल नव्हे तर कार्यात्मक एजंट मध्ये बदलता. मॉडेल ठरवते की कधी साधन हवे आहे, कोणते साधन वापरायचे, आणि कोणते पारामीटर्स द्यायचे. तुमचा कोड फंक्शन चालवतो आणि निकाल परत करतो. मॉडेल त्याच्या प्रतिसादात तो निकाल समाविष्ट करते.

## साधनांसह AI एजंट समजून घेणे

> **📝 नोट:** या मॉड्यूलमधील "एजंट" हा शब्द उपकरण-कलिंग क्षमतांसह सुधारित AI सहाय्यकांना सूचित करतो. हे [मॉड्यूल ०५: MCP](../05-mcp/README.md) मध्ये आपण पाहणार असलेल्या **Agentic AI** नमुन्यांपासून (स्वायत्त एजंट जे नियोजन, स्मृती, आणि बहु-चरण तर्क करतात) वेगळे आहे.

साधने नसेल तर भाषा मॉडेल फक्त त्याच्या प्रशिक्षण डेटामधील मजकूर तयार करू शकते. वर्तमान हवामान विचारले तर ते अंदाज लावते. साधने दिली की ते हवामान API कॉल करू शकते, गणित करू शकते, किंवा डेटाबेस क्वेरी करू शकते — मग त्या वास्तविक निकालांना आपल्या प्रतिसादात वाचवते.

<img src="../../../translated_images/mr/what-are-tools.724e468fc4de64da.webp" alt="टूलशिवाय vs टूलसह" width="800"/>

*साधनांशिवाय मॉडेल फक्त अंदाज लावते — साधनांसह ते API कॉल करू शकते, गणना करू शकते, आणि रिअल-टाइम डेटा परत करू शकते.*

साधनांसह AI एजंट एक **तर्क आणि कृती (ReAct)** नमुना अनुसरतो. मॉडेल फक्त प्रतिसाद देत नाही — ते काय हवे आहे यावर विचार करते, साधन कॉल करून कृती करते, निकाल पाहते, मग पुन्हा कृती करेल का की अंतिम उत्तर देईल हे ठरवते:

1. **तर्क करा** — एजंट वापरकर्त्याचा प्रश्न विश्लेषण करते आणि काय माहिती हवी ते ठरवते
2. **कृती करा** — एजंट योग्य साधन निवडते, बरोबरचे पारामीटर्स तयार करते, आणि कॉल करते
3. **निकाल पाहा** — एजंट साधनाचा आउटपुट प्राप्त करतो आणि निकालाचे मूल्यांकन करतो
4. **पुन्हा करा किंवा उत्तर द्या** — अधिक माहिती हवी असल्यास एजंट लूप मध्ये परततो; अन्यथा, नैसर्गिक भाषेत उत्तर तयार करतो

<img src="../../../translated_images/mr/react-pattern-detail.96a5efeeb6dd2f61.webp" alt="ReAct नमुना" width="800"/>

*ReAct चक्र — एजंट काय करायचे आहे याचा विचार करतो, साधन कॉल करून कृती करतो, निकाल पाहतो, आणि तोपर्यंत लूप करतो जोपर्यंत अंतिम उत्तर देऊ शकत नाही.*

हे स्वयंचलितपणे घडते. तुम्ही साधने आणि त्यांची वर्णने परिभाषित करता. मॉडेल ठरवते कधी आणि कसे ती वापरायची.

## साधन कॉलिंग कसे कार्य करते

### साधन व्याख्या

[WeatherTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) | [TemperatureTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/TemperatureTool.java)

तुम्ही स्पष्ट वर्णने आणि पारामीटर तपशिलांसहित फंक्शन्स परिभाषित करता. मॉडेलला त्याच्या सिस्टम प्रॉम्प्टमध्ये ही वर्णने दिसतात आणि ते प्रत्येक साधन काय करते हे समजते.

```java
@Component
public class WeatherTool {
    
    @Tool("Get the current weather for a location")
    public String getCurrentWeather(@P("Location name") String location) {
        // तुमची हवामान शोधण्याची लॉजिक
        return "Weather in " + location + ": 22°C, cloudy";
    }
}

@AiService
public interface Assistant {
    String chat(@MemoryId String sessionId, @UserMessage String message);
}

// सहाय्यक स्वयंचलितपणे Spring Boot ने जोडलेला आहे:
// - ChatModel बीन
// - @Component वर्गांमधील सर्व @Tool पद्धती
// - सत्र व्यवस्थापनासाठी ChatMemoryProvider
```

खालील आकृती प्रत्येक अ‍ॅनोटेशनचे तुटवडा दाखवते आणि कसे प्रत्येक भाग AI ला साधन कॉल कधी करायचे आणि कोणते आर्ग्युमेंट्स द्यायचे हे समजावून देतो:

<img src="../../../translated_images/mr/tool-definitions-anatomy.f6468546037cf28b.webp" alt="साधन व्याख्यांचे अवयव" width="800"/>

*साधन व्याख्येचे अवयव — @Tool AI ला सांगते की कधी वापरायचे, @P प्रत्येक पारामीटरचे वर्णन करते, आणि @AiService सर्व काही सुरुवातीला वायर करते.*

> - "साधन अंमलबजावणीत API त्रुटी आणि दर मर्यादा कशा हाताळाल?"

### निर्णय घेणे

जेव्हा वापरकर्ता "सिएटलमधले हवामान काय आहे?" विचारतो, तेव्हा मॉडेल यादृच्छिकपणे साधन निवडत नाही. ते वापरकर्त्याच्या उद्देशाशी प्रत्येक साधन वर्णनाचे तुलना करते, प्रत्येकाचे सुसंगततेसाठी स्कोअर करते, व उत्तम जुळणी निवडते. नंतर योग्य पारामीटर्ससह संरचित फंक्शन कॉल तयार करते — या प्रकरणात, `location` ला `"Seattle"` सेट करते.

जर वापरकर्त्याच्या विनंतीला कोणतेही साधन जुळले नाही, तर मॉडेल आपल्या ज्ञानावर आधारित उत्तर देते. जर अनेक साधने जुळली तर सर्वात विशिष्ट साधन निवडते.

<img src="../../../translated_images/mr/decision-making.409cd562e5cecc49.webp" alt="AI कोणते साधन वापरायचे ठरवते कसे" width="800"/>

*मॉडेल प्रत्येक उपलब्ध साधन वापरकर्त्याच्या हेतूशी तुलना करते आणि सर्वोत्तम जुळणी निवडते — म्हणून स्पष्ट आणि विशिष्ट साधन वर्णने लिहिणे आवश्यक आहे.*

### प्रतिसाद निर्मिती

मॉडेल हवामान डेटा प्राप्त करते आणि वापरकर्त्यासाठी त्याला नैसर्गिक भाषेत उत्तर स्वरूपात सादर करते.

### आर्किटेक्चर: स्प्रिंग बूट ऑटो-वायरिंग

हा मॉड्यूल LangChain4j चा स्प्रिंग बूट एकत्रीकरण वापरतो ज्यात `@AiService` घोषणात्मक इंटरफेस वापरले आहेत. स्टार्टअपवर स्प्रिंग बूट प्रत्येक `@Component` ज्यात `@Tool` पद्धती आहेत, तुमचा `ChatModel` बीन, आणि `ChatMemoryProvider` शोधते — मग त्यांना सगळ्यांना एका `Assistant` इंटरफेसमध्ये शून्य बोईलरप्लेटसह जोडते.

<img src="../../../translated_images/mr/spring-boot-wiring.151321795988b04e.webp" alt="स्प्रिंग बूट ऑटो-वायरिंग आर्किटेक्चर" width="800"/>

*@AiService इंटरफेस ChatModel, साधन घटक, आणि मेमरी प्रदात्याला जोडतो — स्प्रिंग बूट सगळे वायरिंग स्वयंचलितरीत्या हाताळते.*

या पद्धतीचे महत्त्वाचे फायदे:

- **स्प्रिंग बूट ऑटो-वायरिंग** — ChatModel आणि साधने आपोआप इंजेक्ट होतात
- **@MemoryId पॅटर्न** — स्वयंचलित सेशन-आधारित मेमरी व्यवस्थापन
- **एकल उदाहरण** — Assistant एकदाच तयार होणारा आणि चांगल्या कार्यक्षमतेसाठी पुन्हा वापरला जातो
- **टाइप-सुरक्षित अंमलबजावणी** — Java पद्धती थेट टाइप रूपांतरणासह कॉल केल्या जातात
- **बहु-टर्न ऑर्केस्ट्रेशन** — साधन साखळी स्वयंचलित हाताळते
- **शून्य बोईलरप्लेट** — कोणतेही मॅन्युअल `AiServices.builder()` कॉल किंवा मेमरी HashMap नाहीत

पर्यायी पद्धती (मॅन्युअल `AiServices.builder()`) अधिक कोडची गरज भासवतात आणि स्प्रिंग बूट एकत्रीकरण फायदे गमावतात.

## साधन साखळी

**साधन साखळी** — साधन-आधारित एजंटची खरी ताकद दिसते जेव्हा एका प्रश्नाला अनेक साधने लागतात. "सिएटलमधली हवामान फॅरेनहाइटमध्ये काय आहे?" असा प्रश्न विचारला तर एजंट स्वतःहून दोन साधने साखळीसारखी वापरतो: प्रथम `getCurrentWeather` कॉल करून सेल्सिअस मधील तापमान मिळवतो, मग त्या मूल्याचा `celsiusToFahrenheit` मध्ये रूपांतर करतो — हे सगळे एका संवाद टर्नमध्ये.

<img src="../../../translated_images/mr/tool-chaining-example.538203e73d09dd82.webp" alt="साधन साखळीचे उदाहरण" width="800"/>

*साधन साखळी क्रियेत — एजंट प्रथम getCurrentWeather कॉल करतो, मग सेल्सिअस निकाल celsiusToFahrenheit मध्ये देतो, आणि एकत्रित उत्तर पाठवतो.*

रनिंग अ‍ॅप्लिकेशनमध्ये हे कसे दिसते — एजंट एका संवाद टर्नमध्ये दोन साधन कॉल्स साखळीसारखे चालवतो:

<a href="images/tool-chaining.png"><img src="../../../translated_images/mr/tool-chaining.3b25af01967d6f7b.webp" alt="साधन साखळ्या" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*खऱ्या अ‍ॅप्लिकेशन आउटपुट — एजंट आपोआप getCurrentWeather → celsiusToFahrenheit एक टर्न मध्ये साखळीसारखे वापरतो.*

**सर्वसन्माननीय अपयश** — मॉक डेटामध्ये नसलेल्या शहराचे हवामान विचारले तर साधन त्रुटी संदेश परत करते, आणि AI सांगते की ते मदत करू शकत नाही पण क्रॅश होत नाही. साधने सुरक्षितपणे अपयशी ठरतात.

<img src="../../../translated_images/mr/error-handling-flow.9a330ffc8ee0475c.webp" alt="त्रुटी हाताळणी प्रवाह" width="800"/>

*जेव्हा साधन अपयशी ठरते, एजंट त्रुटी पकडतो आणि क्रॅश होण्याऐवजी मदत करणारे स्पष्टीकरण देतो.*

हे एकाच संवाद टर्नमध्ये होते. एजंट स्वायत्तपणे अनेक साधन कॉल्स समन्वयित करतो.

## अ‍ॅप्लिकेशन वापरणे

हे अ‍ॅप्लिकेशन वेब इंटरफेस देते जिथे तुम्ही हवामान आणि तापमान रूपांतरण साधनांसह AI एजंटशी संवाद साधू शकता.

<a href="images/tools-homepage.png"><img src="../../../translated_images/mr/tools-homepage.4b4cd8b2717f9621.webp" alt="AI एजंट टूल्स इंटरफेस" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*AI एजंट टूल्स इंटरफेस - तत्काळ उदाहरणे आणि साधने वापरण्यासाठी चॅट इंटरफेस*

### साधे साधन वापरून पहा
सरळ विनंतीने प्रारंभ करा: "100 डिग्री फॅरेनहाइट Celsius मध्ये रूपांतरित करा". एजंटला तापमान रूपांतरण साधनाची गरज आहे हे ओळखले, योग्य पॅरामीटर्ससह त्याला कॉल केले आणि निकाल परत केला. हे कसे नैसर्गिक वाटते हे लक्षात घ्या - तुम्ही कोणते साधन वापरायचे आहे किंवा कसे कॉल करायचे आहे हे सांगितले नाही.

### साधने साखळी तपासा

आता काही अधिक क्लिष्ट प्रयत्न करा: "Seattle मध्ये हवामान काय आहे आणि ते Fahrenheit मध्ये रूपांतरित करा?" एजंट ह्याचे पायऱ्याटपणे काम कसे करते ते बघा. तो प्रथम हवामान मिळवतो (जे Celsius मध्ये परत करते), त्याला Fahrenheit मध्ये रूपांतर करायचे आहे हे ओळखतो, रूपांतरण साधनाला कॉल करतो आणि दोन्ही निकाल एकत्र करून एक उत्तर तयार करतो.

### संभाषण प्रवाह पहा

चॅट इंटरफेस संभाषण इतिहास राखतो, ज्यामुळे तुम्ही बहु-पट्टे संवाद साधू शकता. तुम्हाला सर्व मागील प्रश्न आणि उत्तरे दिसतात, ज्यामुळे संभाषणाचा मागोवा घेणे सोपे होते आणि एजंट कसे अनेक संवादांवर संदर्भ तयार करतो ते समजते.

<a href="images/tools-conversation-demo.png"><img src="../../../translated_images/mr/tools-conversation-demo.89f2ce9676080f59.webp" alt="एकाधिक साधन कॉलसह संभाषण" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*सामान्य रूपांतरे, हवामान शोध आणि साधन साखळी दर्शवणारे बहु-पट्टे संभाषण*

### विविध विनंत्यांसह प्रयोग करा

वेगवेगळ्या संयोगांचा प्रयत्न करा:
- हवामान तपासणी: "Tokyo मध्ये हवामान काय आहे?"
- तापमान रूपांतरे: "25°C हे Kelvin मध्ये किती आहे?"
- संयुक्त प्रश्न: "Paris मधील हवामान तपासा आणि मला सांगा ते 20°C पेक्षा जास्त आहे का"

एजंट कसा नैसर्गिक भाषा समजतो आणि योग्य साधन कॉलशी जुळवतो हे लक्षात घ्या.

## मुख्य संकल्पना

### ReAct नमुना (तर्क व क्रिया)

एजंट विचार-विनिमय (काय करायचे ठरवणे) आणि क्रिया (साधने वापरणे) यात बदल करतो. हा नमुना फक्त सूचना देण्याऐवजी स्वयंचलित समस्या सोडवण्यास मदत करतो.

### साधन वर्णन महत्त्वाचे

तुमच्या साधनांच्या वर्णनांची गुणवत्ता थेट एजंट कसे त्यांचा वापर करतो यावर परिणाम करते. स्पष्ट, विशिष्ट वर्णने मॉडेलला कधी आणि कसे कसे साधन कॉल करायचे हे समजण्यास मदत करतात.

### सत्र व्यवस्थापन

`@MemoryId` अनुक्रमणिका आपोआप सत्र आधारित स्मृती व्यवस्थापन सक्षम करते. प्रत्येक सत्र आयडीसाठी `ChatMemoryProvider` बीनद्वारे स्वतंत्र `ChatMemory` उदाहरण तयार होते, ज्यामुळे असंख्य वापरकर्ते समान एजंटशी संपर्क साधताना त्यांचे संभाषण मिक्स होत नाही.

<img src="../../../translated_images/mr/session-management.91ad819c6c89c400.webp" alt="@MemoryId सह सत्र व्यवस्थापन" width="800"/>

*प्रत्येक सत्र आयडी स्वतंत्र संभाषण इतिहासाशी जोडलेला आहे — वापरकर्ते एकमेकांच्या संदेश कधीही पाहत नाहीत.*

### त्रुटी हाताळणी

साधने अपयशी ठरू शकतात — API चे टाइमआउट होते, पॅरामीटर्स अवैध असू शकतात, बाह्य सेवा स्तब्ध होतात. उत्पादन एजंट्समध्ये त्रुटी हाताळणी आवश्यक असते ज्यामुळे मॉडेल समस्यांचे स्पष्टीकरण देऊ शकते किंवा पर्याय शोधू शकते, पूर्ण अॅप्लिकेशन क्रॅश न होता. जेव्हा साधन अपवाद टाकते, तेव्हा LangChain4j तो पकडते आणि त्रुटी संदेश मॉडेलला देतो, जे नंतर नैसर्गिक भाषेमध्ये समस्या समजावते.

## उपलब्ध साधने

खालील आकृती तुम्ही तयार करू शकता अशा साधनांची व्यापक परिसंस्था दर्शवते. हा मॉड्यूल हवामान आणि तापमान साधने दाखवतो, मात्र त्याच `@Tool` नमुन्याचा वापर कोणत्याही Java पद्धतीसाठी होऊ शकतो — डेटाबेस चौकशींपासून पेमेंट प्रक्रियेसारखे.

<img src="../../../translated_images/mr/tool-ecosystem.aad3d74eaa14a44f.webp" alt="साधन परिसंस्था" width="800"/>

*@Tool शी अ‍ॅनोटेट केलेली कोणतीही Java पद्धत AI साठी उपलब्ध होते — हा नमुना डेटाबेस, API, ईमेल, फाइल ऑपरेशन्स आणि अधिकांपर्यंत विस्तारित होतो.*

## कधी साधनाधारित एजंट वापरावेत

<img src="../../../translated_images/mr/when-to-use-tools.51d1592d9cbdae9c.webp" alt="कधी साधने वापरावीत" width="800"/>

*एक जलद निर्णय मार्गदर्शक — साधने रिअल-टाइम डेटा, गणना, आणि क्रियांसाठी; सामान्य ज्ञान व सर्जनशील कार्यांसाठी त्यांची गरज नाही.*

**साधने वापरा जेव्हा:**
- उत्तरांसाठी रिअल-टाइम डेटा आवश्यक आहे (हवामान, स्टॉक किंमती, सूची)
- सोप्या गणनांव्यतिरिक्त गणना करायची आहे
- डेटाबेस किंवा API वापरायचे असतील
- क्रिया करायच्या (ईमेल पाठवणे, तिकीट तयार करणे, नोंदी अद्ययावत करणे)
- अनेक डेटा स्रोतांचे संयोजन करायचे असेल

**साधने वापरू नका जेव्हा:**
- प्रश्न सामान्य ज्ञानातून उत्तर देता येते
- प्रतिसाद फक्त संभाषणात्मक आहे
- साधनांच्या विलंबामुळे अनुभव खालचा होईल

## साधने वि आरएजी

मॉड्यूल 03 आणि 04 दोन्ही AI च्या क्षमता वाढवतात, पण भिन्न प्रकारे. आरएजी मॉडेलला **ज्ञान** मिळवतो दस्तऐवजांद्वारे, तर साधने मॉडेलला **क्रिया** करण्याची क्षमता देतात फंक्शन्स कॉल करून.

<img src="../../../translated_images/mr/tools-vs-rag.ad55ce10d7e4da87.webp" alt="साधने वि आरएजी तुलना" width="800"/>
