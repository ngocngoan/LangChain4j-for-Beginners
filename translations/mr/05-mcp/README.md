## तुम्ही काय शिकाल

तुम्ही संभाषणात्मक AI तयार केला आहे, प्रॉम्प्ट्समध्ये पारंगत आहात, दस्तऐवजांमध्ये आधारित प्रतिसाद दिले आहेत, आणि तंत्रज्ञानांसह एजंट्स तयार केले आहेत. पण ते सर्व उपकरणे तुमच्या विशिष्ट अनुप्रयोगासाठी सानुकूल बनवली गेली होती. जर तुम्ही तुमच्या AI ला साधनांचे एक प्रमाणित परिसंस्था प्रदान करू शकले तर जे कोणत्याही व्यक्तीस तयार आणि सामायिक करता येईल? या मॉड्यूलमध्ये तुम्ही Model Context Protocol (MCP) आणि LangChain4j च्या एजेंटिक मॉड्यूलसह ते कसे करायचे ते शिकाल. आम्ही प्रथम एक सोपा MCP फाइल वाचक दर्शवितो आणि नंतर तो सुपरिव्हायझर एजंट पॅटर्न वापरून प्रगत एजेंटिक वर्कफ्लोमध्ये कसा सहजपणे समाकलित होतो ते दाखवितो.

## MCP म्हणजे काय?

Model Context Protocol (MCP) नेमके तेच प्रदान करते - AI अनुप्रयोगांसाठी बाह्य साधने शोधण्याचा आणि वापरण्याचा एक प्रमाणित मार्ग. प्रत्येक डेटा स्त्रोत किंवा सेवा साठी सानुकूल समाकलने लिहिण्याऐवजी, तुम्ही MCP सर्व्हर्सशी जोडता जे त्यांची क्षमता एकसंध स्वरूपात प्रदर्शित करतात. तुमचा AI एजंट नंतर या साधनांचा आपोआप शोध घेऊ शकतो आणि वापरू शकतो.

खालील आकृती फरक दाखवते — MCP शिवाय, प्रत्येक समाकलनासाठी सानुकूल पॉइंट-टू-पॉइंट वायरिंग आवश्यक; MCP सह, एकच प्रोटोकॉल तुमचा ऍप कोणत्याही साधनाशी जोडतो:

<img src="../../../translated_images/mr/mcp-comparison.9129a881ecf10ff5.webp" alt="MCP Comparison" width="800"/>

*MCP आधी: गुंतागुंतीची पॉइंट-टू-पॉइंट समाकलने. MCP नंतर: एकच प्रोटोकॉल, अनंत शक्यता.*

MCP AI विकासातील एक मूलभूत समस्या सोडवते: प्रत्येक समाकलन सानुकूल असते. GitHub वापरायचे? सानुकूल कोड. फाइल्स वाचायच्या? सानुकूल कोड. डेटाबेस क्वेरी करायचा? सानुकूल कोड. आणि ही सर्व समाकलने अन्य AI अनुप्रयोगांसोबत काम करत नाहीत.

MCP हे प्रमाणित करते. एक MCP सर्व्हर स्पष्ट वर्णने आणि योजनेसह साधने उघड करतो. कोणताही MCP क्लायंट कनेक्ट होऊन उपलब्ध साधने शोधू शकतो आणि वापरू शकतो. एकदाच तयार करा, सर्वत्र वापरा.

खालील आकृती या आर्किटेक्चरचे चित्रण करते — एक MCP क्लायंट (तुमचा AI अनुप्रयोग) अनेक MCP सर्व्हरशी कनेक्ट होतो, प्रत्येक स्वतःच्या साधनांचा संच प्रमाणित प्रोटोकॉलद्वारे उघड करतो:

<img src="../../../translated_images/mr/mcp-architecture.b3156d787a4ceac9.webp" alt="MCP Architecture" width="800"/>

*Model Context Protocol आर्किटेक्चर - प्रमाणित साधन शोध आणि अंमलबजावणी*

## MCP कसे कार्य करते

खालच्या स्तरावर, MCP स्तरित आर्किटेक्चर वापरते. तुमचा Java अनुप्रयोग (MCP क्लायंट) उपलब्ध साधने शोधतो, ट्रान्सपोर्ट स्तराद्वारे (Stdio किंवा HTTP) JSON-RPC विनंत्या पाठवतो, आणि MCP सर्व्हर ऑपरेशन्स पार पाडतो आणि निकाल परत पाठवतो. खालील आकृती या प्रोटोकॉलच्या प्रत्येक स्तराचे विहंगावलोकन करते:

<img src="../../../translated_images/mr/mcp-protocol-detail.01204e056f45308b.webp" alt="MCP Protocol Detail" width="800"/>

*खालच्या स्तरावर MCP कसे कार्य करते — क्लायंट्स साधने शोधतात, JSON-RPC संदेशांची देवाणघेवाण करतात, आणि ट्रान्सपोर्ट स्तराद्वारे ऑपरेशन्स पार पाडतात.*

**सर्व्हर-क्लायंट आर्किटेक्चर**

MCP क्लायंट-सर्व्हर मॉडेल वापरते. सर्व्हर्स साधने पुरवतात - फाइल्स वाचन, डेटाबेस क्वेरी, API कॉलिंग. क्लायंट्स (तुमचा AI अनुप्रयोग) सर्व्हर्सशी कनेक्ट होऊन त्यांच्या साधनांचा वापर करतात.

LangChain4j सह MCP वापरण्यासाठी, हा Maven अवलंबित्व जोडा:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```

**साधन शोध**

तुमचा क्लायंट जेव्हा MCP सर्व्हरशी कनेक्ट होतो, तेव्हा तो विचारतो "तुमच्याकडे कोणती साधने आहेत?" सर्व्हर उपलब्ध साधनांची यादी वर्णनांसह आणि पॅरामीटर स्कीमांसह परत पाठवतो. तुमचा AI एजंट नंतर वापरकर्त्याच्या विनंतीनुसार कोणती साधने वापरायची हे ठरवू शकतो. खालील आकृती ही हातमिळवणी दाखवते — क्लायंट `tools/list` विनंती पाठवतो आणि सर्व्हर त्याची उपलब्ध साधने वर्णनांसह आणि पॅरामीटर स्कीमांसह परत देतो:

<img src="../../../translated_images/mr/tool-discovery.07760a8a301a7832.webp" alt="MCP Tool Discovery" width="800"/>

*AI स्टार्टअपवर उपलब्ध साधने शोधतो — आता त्याला काय क्षमता उपलब्ध आहेत हे माहिती आहे आणि कोणती वापरावीत हे ठरवू शकतो.*

**ट्रान्सपोर्ट मेकॅनिझम्स**

MCP विविध ट्रान्सपोर्ट मेकॅनिझम्सना समर्थन देते. दोन पर्याय आहेत Stdio (स्थानिक subprocess संवादासाठी) आणि Streamable HTTP (दूरस्थ सर्व्हरसाठी). या मॉड्यूलमध्ये Stdio ट्रान्सपोर्ट दाखविला आहे:

<img src="../../../translated_images/mr/transport-mechanisms.2791ba7ee93cf020.webp" alt="Transport Mechanisms" width="800"/>

*MCP ट्रान्सपोर्ट मेकॅनिझम्स: HTTP दूरस्थ सर्व्हरसाठी, Stdio स्थानिक प्रक्रिया साठी*

**Stdio** - [StdioTransportDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java)

स्थानिक प्रक्रियांसाठी. तुमचे अनुप्रयोग एक सर्व्हर बालप्रक्रियेसारखे तयार करते आणि मानक इनपुट/आउटपुटद्वारे संवाद साधतो. फायली सिस्टम प्रवेश किंवा कमांड-लाइन साधनांसाठी उपयुक्त.

```java
McpTransport stdioTransport = new StdioMcpTransport.Builder()
    .command(List.of(
        npmCmd, "exec",
        "@modelcontextprotocol/server-filesystem@2025.12.18",
        resourcesDir
    ))
    .logEvents(false)
    .build();
```

`@modelcontextprotocol/server-filesystem` सर्व्हर खालील साधने उघड करतो, सर्व निर्दिष्ट निर्देशिकांसाठी सुरक्षित:

| साधन | वर्णन |
|------|-------------|
| `read_file` | एका फाईलची सामग्री वाचा |
| `read_multiple_files` | एका कॉलमध्ये एकाधिक फाइल्स वाचा |
| `write_file` | एक फाइल तयार किंवा ओव्हरराइट करा |
| `edit_file` | लक्ष केंद्रित करून शोध आणि बदला संपादने करा |
| `list_directory` | कोणत्याही पाथवरील फाइल्स आणि निर्देशिका सूचीबद्ध करा |
| `search_files` | पॅटर्नशी जुळणाऱ्या फाइल्स पुनरावृत्तीने शोधा |
| `get_file_info` | फाइल मेटाडेटा मिळवा (आकार, टाइमस्टॅम्प, अनुमती) |
| `create_directory` | निर्देशिका तयार करा (पालक निर्देशिका समाविष्ट) |
| `move_file` | फाइल किंवा निर्देशिका हलवा किंवा नाव बदला |

खालील आकृती Stdio ट्रान्सपोर्ट रनटाइमवर कसा कार्य करतो हे दाखवते — तुमचा Java अनुप्रयोग MCP सर्व्हरला बालप्रक्रिया म्हणून तयार करतो आणि ते stdin/stdout पाइप्सद्वारे संवाद साधतात, यामध्ये कोणताही नेटवर्क किंवा HTTP समाविष्ट नाही:

<img src="../../../translated_images/mr/stdio-transport-flow.45eaff4af2d81db4.webp" alt="Stdio Transport Flow" width="800"/>

*Stdio ट्रान्सपोर्ट क्रियेत — तुमचे अनुप्रयोग MCP सर्व्हरला बालप्रक्रिया म्हणून तयार करतो आणि stdin/stdout पाइप्सद्वारे संवाद साधतो.*

> **🤖 [GitHub Copilot](https://github.com/features/copilot) Chat सह प्रयत्न करा:** [`StdioTransportDemo.java`](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java) उघडा आणि विचारा:
> - "Stdio ट्रान्सपोर्ट कसा कार्य करतो आणि मला HTTP विरुद्ध कधी वापरायचा?"
> - "LangChain4j कसे MCP सर्व्हर प्रक्रियेच्या जीवनचक्राचे व्यवस्थापन करते?"
> - "फाइल सिस्टममध्ये AI ऍक्सेस देण्याचे सुरक्षा परिणाम कोणते?"

## एजेंटिक मॉड्यूल

जरी MCP प्रमाणित साधने प्रदान करतो, LangChain4j चे **एजेंटिक मॉड्यूल** एजंट्स बांधण्यासाठी घोषणात्मक मार्ग देते जे त्या साधनांचे नियोजन करतात. `@Agent` नोंदी आणि `AgenticServices` तुम्हाला आकृतीद्वारे एजंट वर्तन परिभाषित करण्याची परवानगी देतात, जबाबदार कोडऐवजी.

या मॉड्यूलमध्ये, तुम्ही **सुपरिव्हायझर एजंट** पॅटर्न पहाल — एक प्रगत एजेंटिक AI दृष्टिकोन जेथे "सुपरिव्हायझर" एजंट वापरकर्त्याच्या विनंतीनुसार कोणते उप-एजंट्स सक्रिय करायचे ते गतिमानपणे ठरवतो. आम्ही दोन्ही संकल्पना संयोजित करू, आमच्या एका उप-एजंटला MCP-शक्तीत फाइल प्रवेश क्षमता देऊन.

एजेंटिक मॉड्यूल वापरण्यासाठी, हा Maven अवलंबित्व जोडा:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```
> **टीप:** `langchain4j-agentic` मॉड्यूल वेगळ्या वेळापत्रकावर जारी केल्या जाणाऱ्या `langchain4j.mcp.version` वापरून वेगळे आवृत्ती प्रॉपर्टी वापरते कारण ते मुख्य LangChain4j लायब्ररीपेक्षा वेगळ्या वेळापत्रकावर जारी केलेले आहे.

> **⚠️ प्रयोगशील:** `langchain4j-agentic` मॉड्यूल **प्रयोगशील** आहे आणि बदलला जाऊ शकतो. AI सहाय्यक तयार करण्यासाठी स्थिर मार्ग म्हणजे `langchain4j-core` आणि सानुकूल साधने (मॉड्यूल 04).

### फाइल ऑपरेशन्स (Stdio)

हे स्थानिक subprocess आधारित साधने दर्शविते.

**✅ कोणत्याही पूर्वअटी आवश्यक नाहीत** - MCP सर्व्हर आपोआप तयार होतो.

**प्रारंभ स्क्रिप्ट वापरणे (शिफारस केलेले):**

स्टार्ट स्क्रिप्ट्स रूट `.env` फाइलमधील वातावरणीय चल आपोआप लोड करतात:

**Bash:**
```bash
cd 05-mcp
chmod +x start-stdio.sh
./start-stdio.sh
```

**PowerShell:**
```powershell
cd 05-mcp
.\start-stdio.ps1
```

### सुपरिव्हायझर एजंट

**सुपरिव्हायझर एजंट पॅटर्न** एक **लवचिक** एजेंटिक AI प्रकार आहे. सुपरिव्हायझर वापरकर्ता विनंतीनुसार कोणते एजंट चालवायचे ते स्वयंचलितपणे ठरवण्यासाठी LLM वापरतो. पुढील उदाहरणात, आम्ही MCP-शक्तीत फाइल प्रवेश आणि LLM एजंट एकत्र करून सुपरिव्हायझर फाइल वाचन → अहवाल निर्मिती वर्कफ्लो तयार करतो.

डेमो मध्ये, `FileAgent` MCP फायलीसिस्टम साधनांचा वापर करून फाइल वाचतो, आणि `ReportAgent` कार्यकारी सारांश (1 वाक्य), 3 मुख्य मुद्दे आणि शिफारशी सह संरचित अहवाल तयार करतो. सुपरिव्हायझर हा प्रवाह आपोआप नियंत्रित करतो:

<img src="../../../translated_images/mr/supervisor-agent-pattern.06275a41ae006ac8.webp" alt="Supervisor Agent Pattern" width="800"/>

*सुपरिव्हायझर त्याचा LLM वापरून ठरवतो कोणते एजंट्स कोणत्या क्रमाने सक्रिय करायचे — कोणतीही हार्डकोडेड रूटिंग आवश्यक नाही.*

आमच्या फाइल-टू-रिपोर्ट पाइपलाइनसाठी खालील ठोस वर्कफ्लो दिसते:

<img src="../../../translated_images/mr/file-report-workflow.649bb7a896800de9.webp" alt="File to Report Workflow" width="800"/>

*FileAgent MCP साधनांद्वारे फाइल वाचतो, नंतर ReportAgent मूळ सामग्री संरचित अहवालात रूपांतरित करतो.*

प्रत्येक एजंट त्याचे आउटपुट **Agentic Scope** (सामायिक स्मृती) मध्ये साठवतो, ज्यामुळे पुढील एजंट्स पूर्वीच्या निकालांपर्यंत प्रवेश मिळवू शकतात. हे दाखवते की MCP साधने एजेंटिक वर्कफ्लोमध्ये कशी अखंडपणे समाकलित होतात — सुपरिव्हायझरला फाइल कशी वाचली जाते हे माहित असण्याची गरज नाही, फक्त `FileAgent` ते करू शकतो.

#### सुपरिव्हायझर कसा कार्य करतो

एजंट्स तयार करण्यापूर्वी, तुम्हाला MCP ट्रान्सपोर्ट क्लायंटशी कनेक्ट करायचा आहे आणि ते `ToolProvider` म्हणून वापरायचे आहे. हा मार्ग आहे ज्याद्वारे MCP सर्व्हरची साधने तुमच्या एजंट्ससाठी उपलब्ध होतात:

```java
// ट्रान्स्पोर्टमधून MCP क्लायंट तयार करा
McpClient mcpClient = new DefaultMcpClient.Builder()
        .transport(stdioTransport)
        .build();

// क्लायंटला ToolProvider म्हणून वेढा — हे MCP साधने LangChain4j मध्ये जोडते
ToolProvider mcpToolProvider = McpToolProvider.builder()
        .mcpClients(List.of(mcpClient))
        .build();
```

आता तुम्ही `mcpToolProvider` कोणत्याही एजंटमध्ये इंजेक्ट करू शकता ज्याला MCP साधने आवश्यक आहेत:

```java
// पायरी 1: FileAgent MCP साधने वापरून फाईल्स वाचतो
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // फाईलसंबंधी ऑपरेशन्ससाठी MCP साधने आहेत
        .build();

// पायरी 2: ReportAgent संरचित अहवाल तयार करतो
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// Supervisor फाईल → अहवाल कार्यप्रवाहाचे नियोजन करतो
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // अंतिम अहवाल परत करा
        .build();

// विनंतीनुसार कोणत्या एजंट्सना कॉल करायचे ते Supervisor ठरवतो
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```

#### प्रतिसाद धोरणे

जेव्हा तुम्ही `SupervisorAgent` कॉन्फिगर करता, तेव्हा तुम्हाला सांगायचे असते की उप-एजंट्सनी आपले काम पूर्ण केल्यानंतर तो वापरकर्त्याला अंतिम उत्तर कसे मांडेल. खालील आकृती तीन उपलब्ध धोरणे दाखवते — LAST शेवटच्या एजंटचा आउटपुट थेट परत करतो, SUMMARY सर्व आउटपुट्स LLM मार्गे एकत्रित करतो, आणि SCORED मूळ विनंतीच्या विरोधात जास्त गुणांकन मिळालेल्या आउटपुटला परत करतो:

<img src="../../../translated_images/mr/response-strategies.3d0cea19d096bdf9.webp" alt="Response Strategies" width="800"/>

*सुपरिव्हायझर कसा अंतिम प्रतिसाद तयार करतो यासाठी तीन धोरणे — तुम्ही अंतिम एजंटचा आउटपुट, सुव्यवस्थित सारांश किंवा सर्वोच्च गुणांक मिळालेला पर्याय निवडू शकता.*

उपलब्ध धोरणे:

| धोरण | वर्णन |
|----------|-------------|
| **LAST** | सुपरिव्हायझर शेवटच्या उप-एजंट किंवा साधनाने दिलेला आउटपुट परत करतो. हे उपयुक्त आहे जेव्हा वर्कफ्लोमधला अंतिम एजंट पूर्ण, अंतिम उत्तर तयार करण्यासाठी विशेष डिझाइन केलेला असतो (उदा. संशोधन पाइपलाइनमधला "सारांश एजंट"). |
| **SUMMARY** | सुपरिव्हायझर त्याचा अंतर्गत भाषा मॉडेल (LLM) वापरून संपूर्ण संवाद आणि सर्व उप-एजंट आउटपुटचा सारांश तयार करतो आणि हा सारांश अंतिम प्रतिसाद म्हणून परत करतो. हे वापरकर्त्यास स्वच्छ, संक्षिप्त उत्तर प्रदान करते. |
| **SCORED** | प्रणाली अंतर्गत LLM वापरून LAST प्रतिसाद आणि संवादाचा सारांश दोघांनाही मूळ वापरकर्ता विनंतीच्या विरोधात गुणांकन करते आणि ज्याला अधिक गुण मिळतात तो आउटपुट परत करते. |

जर तुम्ही डेमो चालवलात तर तुम्हाला Supervisor मल्टिपल एजंट्स कसे व्यवस्थापित करतो याचा रचनेतून फेरफटका पाहायला मिळेल. प्रत्येक विभाग काय अर्थ लावतो ते खालीलप्रमाणे:

- **FileAgent** MCP टूल्सने फाइल वाचतो आणि कच्चा मजकूर `fileContent` मध्ये साठवतो

1. **FileAgent** MCP द्वारे फाइल वाचतो आणि मजकूर साठवतो

#### Agentic Module वैशिष्ट्यांचे स्पष्टीकरण

हे उदाहरण agentic मॉड्यूलची काही प्रगत वैशिष्ट्ये दाखवते. चला Agentic Scope आणि Agent Listeners वर लक्ष देऊया.

**Agentic Scope** हे समान स्मृती दाखवते जिथे एजंट्सनी `@Agent(outputKey="...")` वापरून त्यांचे निकाल साठवले. यामुळे:  
- नंतरचे एजंट्स आधीच्या एजंट्सचे आउटपुट वापरू शकतात  
- Supervisor अंतिम उत्तर तयार करू शकतो  
- तुम्ही प्रत्येक एजंटने काय तयार केले ते तपासू शकता

खालील आकृती दाखवते की Agentic Scope फाइल-ते-रिपोर्ट वर्कफ्लोमध्ये समान स्मृती म्हणून कशी कार्य करते — FileAgent त्याचा आउटपुट `fileContent` कीखाली लिहितो, ReportAgent त्याचा वाचन करतो आणि स्वतःचा आउटपुट `report` कीखाली लिहितो:

<img src="../../../translated_images/mr/agentic-scope.95ef488b6c1d02ef.webp" alt="Agentic Scope Shared Memory" width="800"/>

*Agentic Scope ही समान स्मृती म्हणून कार्य करते — FileAgent `fileContent` लिहितो, ReportAgent वाचतो आणि `report` लिहितो, आणि तुमचा कोड अंतिम निकाल वाचतो.*

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // FileAgent कडून कच्चा फाइल डेटा
String report = scope.readState("report");            // ReportAgent कडून संरचित अहवाल
```
  
**Agent Listeners** एजंट कार्यान्वयनाचे निरीक्षण आणि डिबगिंग सक्षम करतात. तुम्ही डेमो मध्ये बघणारा टप्प्याटप्प्याने आउटपुट एका AgentListener कडून येतो जो प्रत्येक एजंट कॉलमध्ये हुक करतो:  
- **beforeAgentInvocation** - जेव्हा Supervisor एजंट निवडतो तेव्हा कॉल होतो, तुम्हाला कोणता एजंट का निवडला ते दाखवतो  
- **afterAgentInvocation** - एजंट पूर्ण झाल्यावर कॉल होतो, त्याचा निकाल दाखवतो  
- **inheritedBySubagents** - जर true असेल तर, हा लिस्नर संपूर्ण एजंट हायार्कीमध्ये सर्व एजंट्सचे निरीक्षण करतो

खालील आकृती पूर्ण Agent Listener जीवनचक्र दाखवते, ज्यात `onError` एजंट कार्यान्वयनातील अडचणी हाताळण्याचे वर्णन करते:

<img src="../../../translated_images/mr/agent-listeners.784bfc403c80ea13.webp" alt="Agent Listeners Lifecycle" width="800"/>

*Agent Listeners कार्यान्वयन जीवनचक्रात हुक होतात — एजंट्स सुरू, पूर्ण किंवा त्रुटी आल्यावर निरीक्षण करा.*

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // सर्व उपएजंटपर्यंत प्रसारित करा
    }
};
```
  
Supervisor पॅटर्नशिवाय, `langchain4j-agentic` मॉड्यूल अनेक शक्तिशाली वर्कफ्लो पॅटर्न प्रदान करते. खालील आकृती सर्व पाच दाखवते — सोप्या क्रमाने चालणाऱ्या पाइपलाइनपासून मानवी-सहभाग निर्मितीपर्यंत:

<img src="../../../translated_images/mr/workflow-patterns.82b2cc5b0c5edb22.webp" alt="Agent Workflow Patterns" width="800"/>

*पाच वर्कफ्लो पॅटर्न एजंट्सचे आयोजन करण्यासाठी — सोप्या अनुक्रमिक पाइपलाइनपासून मानवी-सहभाग मंजुरी वर्कफ्लोपर्यंत.*

| पॅटर्न | वर्णन | वापर प्रकरण |
|---------|-------------|----------|
| **Sequential** | एजंट्स अनुक्रमे कार्यान्वित करणे, आउटपुट पुढच्या एजंटकडे प्रवाहित होतो | पाइपलाइन: संशोधन → विश्लेषण → अहवाल |
| **Parallel** | एजंट्स एकत्र चालवणे | स्वतंत्र कामे: हवामान + बातम्या + स्टॉक्स |
| **Loop** | स्थिती पूर्ण होईपर्यंत पुनरावृत्ती करणे | गुणवत्ता स्कोअरिंग: स्कोअर ≥ 0.8 होईपर्यंत सुधारणा |
| **Conditional** | स्थितीवर आधारित मार्गदर्शन | वर्गीकरण → विशेषज्ञ एजंटकडे मार्गदर्शन |
| **Human-in-the-Loop** | मानवी चाचण्या जोडणे | मंजुरी वर्कफ्लो, सामग्री पुनरावलोकन |

## मुख्य संकल्पना

आता जेव्हा तुम्ही MCP आणि agentic मॉड्यूल एका कृतीत पाहिले आहे, तेव्हा चला सारांश करूया की कोणती पद्धत केव्हा वापरावी.

MCP चा एक मोठा फायदा म्हणजे त्याचा वाढणारा पर्यावरण तंत्र. खालील आकृती दाखवते की कसे एक सार्वत्रिक प्रोटोकॉल तुमची AI अनुप्रयोग MCP सर्व्हर्सच्या विविध प्रकारांशी जोडतो — फाइलसिस्टम आणि डेटाबेसपासून GitHub, ईमेल, वेब स्क्रॅपिंग आणि अधिक:

<img src="../../../translated_images/mr/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="MCP Ecosystem" width="800"/>

*MCP एक सार्वत्रिक प्रोटोकॉल पर्यावरण तयार करतो — कोणताही MCP-अनुकूल सर्व्हर कोणत्याही MCP-अनुकूल क्लायंटसह काम करतो, ज्यामुळे अनुप्रयोगांतर्गत құралं शेअर करू शकतात.*

**MCP** सर्वोत्कृष्ट आहे जेव्हा तुम्हाला अस्तित्वातील टूल्सचा फायदा घ्यायचा असतो, असे टूल्स तयार करायचे असतात जे अनेक अनुप्रयोग सामायिक करू शकतात, मानक प्रोटोकॉलसह तृतीय पक्ष सेवा एकत्रित करायच्या असतात, किंवा कोड न बदलता टूल कार्यान्वयन बदलायचे असतात.

**Agentic Module** सर्वोत्तम कार्य करतो जेव्हा तुम्हाला `@Agent` अ‍ॅनोटेशनसह घोषणात्मक एजंट परिभाषा हव्या असतात, वर्कफ्लो ऑर्केस्ट्रेशन (साधारण, लूप, पॅरालेल) आवश्यक असतो, आदेशात्मक कोडच्या ऐवजी इंटरफेस-आधारित एजंट डिझाइन पसंत करायचा असतो, किंवा अनेक एजंट्स जे `outputKey` द्वारे आउटपुट सामायिक करतात ते एकत्र वापरायचे असतात.

**Supervisor Agent पॅटर्न** चमकत असतो जेव्हा वर्कफ्लो पूर्वनिर्धारित नसतो आणि तुम्हाला LLM कडून निर्णय पाहिजे असतो, जेव्हा तुमच्याकडे अनेक विशेषज्ञ एजंट्स असतात जे गतिशील ऑर्केस्ट्रेशनला गरज असते, संवादात्मक प्रणाली तयार करताना जे भिन्न क्षमतांकडे मार्गदर्शन करतात, किंवा जेव्हा तुम्हाला सर्वाधिक लवचिक, अनुकूल एजंट वर्तन हवे असते.

Module 04 मधील कस्टम `@Tool` पद्धती आणि या मॉड्यूलमधील MCP टूल्स यामध्ये निवड करताना खालील तुलना मुख्य फरक दर्शवते — कस्टम टूल्स अ‍ॅप-विशिष्ट लॉजिकसाठी घट्ट जोडणी आणि पूर्ण प्रकार सुरक्षितता देतात, तर MCP टूल्स मानकीकृत, पुनर्वापरयोग्य एकत्रीकरणे देतात:

<img src="../../../translated_images/mr/custom-vs-mcp-tools.c4f9b6b1cb65d8a1.webp" alt="Custom Tools vs MCP Tools" width="800"/>

*कस्टम @Tool पद्धती आणि MCP टूल्स कधी वापरायच्या — कस्टम टूल्स अ‍ॅप-विशिष्ट लॉजिकसाठी पूर्ण प्रकार सुरक्षितता, MCP टूल्स मानकीकृत एकत्रीकरणांसाठी जे अनुप्रयोगांमध्ये काम करतात.*
