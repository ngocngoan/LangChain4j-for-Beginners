## तुम्हाला काय शिकायला मिळेल

तुम्ही संभाषणात्मक AI तयार केले आहे, प्रॉम्प्ट्समध्ये निपुणता मिळवली आहे, उत्तरं दस्तऐवजांमध्ये आधारित केली आहेत, आणि टूल्ससह एजंट्स तयार केले आहेत. पण त्या सर्व टूल्स तुमच्या विशिष्ट अनुप्रयोगासाठी खास तयार केलेल्या होत्या. जर तुम्ही तुमच्या AI ला अशी एक प्रमाणित टूल्सची प्रणाली दिली तर ज्याला कोणीही तयार करून सामायिक करू शकते? या मॉड्यूलमध्ये, तुम्ही Model Context Protocol (MCP) आणि LangChain4j च्या एजेंटिक मॉड्यूलसह हे कसे करायचे ते शिकणार आहात. आम्ही प्रथम एक सोपा MCP फाइल रीडर दाखवतो आणि नंतर तो कसा सुलभपणे उच्च स्तरीय एजेंटिक कार्यप्रवाहांमध्ये सुपरवायझर एजंट पद्धतीने एकत्रित होतो ते दाखवतो.

## MCP म्हणजे काय?

Model Context Protocol (MCP) हे अगदी तशीच सुविधा देते - AI अनुप्रयोगांना बाह्य टूल्स शोधण्यासाठी आणि वापरण्यासाठी एक प्रमाणित मार्ग. प्रत्येक डेटा स्रोत किंवा सेवेकरिता सानुकूल समाकलने लिहिण्याऐवजी, तुम्ही MCP सर्व्हरशी संपर्क साधता ज्यात त्यांच्या क्षमतांचे सुसंगत स्वरूप असते. तुमचा AI एजंट नंतर या टूल्स आपोआप शोधून वापरू शकतो.

खालिल आकृती दाखवते फरक — MCP नसेल तर प्रत्येक समाकलनाला सानुकूल विनियोजन आवश्यक असते; MCP सह, एकच प्रोटोकॉल तुमच्या अॅपला कोणत्याही टूलशी जोडते:

<img src="../../../translated_images/mr/mcp-comparison.9129a881ecf10ff5.webp" alt="MCP Comparison" width="800"/>

*MCP आधी: गुंतागुंतीचे बिंदू-ते-बिंदू समाकलने. MCP नंतर: एक प्रोटोकॉल, अखंड शक्यता.*

MCP AI विकासातील मूलभूत समस्या सोडवते: प्रत्येक समाकलन हे सानुकूल असते. GitHub ला प्रवेश करायचा आहे का? सानुकूल कोड. फाइल वाचायची आहे का? सानुकूल कोड. डेटाबेस क्वेरी करायची आहे का? सानुकूल कोड. आणि ह्या पैकी एकही समाकलन इतर AI अनुप्रयोगांसोबत काम करत नाही.

MCP याला प्रमाणित करते. MCP सर्व्हर टूल्स स्पष्ट वर्णने आणि योजना (स्कीमा) सह प्रकट करतो. कोणताही MCP क्लायंट कनेक्ट होऊ शकतो, उपलब्ध टूल्स शोधू शकतो, आणि वापरू शकतो. एकदा तयार करा, सर्वत्र वापरा.

खालची आकृती दर्शवते या स्थापत्यशास्त्राची — एकच MCP क्लायंट (तुमचा AI अनुप्रयोग) अनेक MCP सर्व्हरशी कनेक्ट होतो, प्रत्येक त्यांच्या प्रमाणित प्रोटोकॉलद्वारे टूल्स सेट दर्शवितो:

<img src="../../../translated_images/mr/mcp-architecture.b3156d787a4ceac9.webp" alt="MCP Architecture" width="800"/>

*Model Context Protocol स्थापत्य - प्रमाणित टूल शोध आणि अंमलबजावणी*

## MCP कसे कार्य करते

आंतर्गत, MCP स्तरित स्थापत्य वापरते. तुमचा Java अनुप्रयोग (MCP क्लायंट) उपलब्ध टूल्स शोधतो, एक ट्रान्सपोर्ट स्तर (Stdio किंवा HTTP) द्वारे JSON-RPC विनंत्या पाठवतो, आणि MCP सर्व्हर ऑपरेशन्स चालवतो व निकाल परत करतो. खालील आकृती या प्रोटोकॉलमधील प्रत्येक स्तर समजावून सांगते:

<img src="../../../translated_images/mr/mcp-protocol-detail.01204e056f45308b.webp" alt="MCP Protocol Detail" width="800"/>

*MCP कसे अंतर्गत कार्य करते — क्लायंट टूल्स शोधतो, JSON-RPC संदेशांची देवाण-घेवाण करतो, आणि ट्रान्सपोर्ट स्तरावरील ऑपरेशन्स पार पाडतो.*

**सर्व्हर-क्लायंट स्थापत्य**

MCP क्लायंट-सर्व्हर मॉडेल वापरतो. सर्व्हर टूल्स पुरवतो - फाइल्स वाचन, डेटाबेस क्वेरी, API कॉल. क्लायंट (तुमचा AI अनुप्रयोग) सर्व्हरशी जोडतो आणि टूल्स वापरतो.

LangChain4j सह MCP वापरण्यासाठी, हा Maven अवलंबित्व जोडा:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```

**टूल शोध**

तुमचा क्लायंट जेव्हा MCP सर्व्हरशी कनेक्ट होतो, तेव्हा तो विचारतो "तुमच्याकडे कोणते टूल्स आहेत?" सर्व्हर उपलब्ध टूल्सची यादी, प्रत्येक टूलचे वर्णन आणि पॅरामीटर योजना (स्कीमा) सह पाठवतो. तुमचा AI एजंट नंतर वापरकर्त्यांच्या विनंत्यांवर आधारित कोणते टूल्स वापरायचे ते ठरवू शकतो. खालील आकृती ही आपापसातील देवाण-घेवाण दाखवते — क्लायंट `tools/list` विनंती पाठवतो आणि सर्व्हर त्याचे टूल्स वर्णने व स्कीमांसह परत पाठवतो:

<img src="../../../translated_images/mr/tool-discovery.07760a8a301a7832.webp" alt="MCP Tool Discovery" width="800"/>

*AI सुरुवातीला उपलब्ध टूल्स शोधते — त्याला आता काय क्षमता आहेत हे माहित असते आणि कोणते वापरायचे ते ठरवू शकते.*

**ट्रान्सपोर्ट यंत्रणा**

MCP विविध ट्रान्सपोर्ट यंत्रणांना सपोर्ट करतो. दोन विकल्प आहेत Stdio (स्थानिक subprocess संवादासाठी) आणि Streamable HTTP (दूरस्थ सर्व्हरसाठी). हे मॉड्यूल Stdio ट्रान्सपोर्ट दाखवते:

<img src="../../../translated_images/mr/transport-mechanisms.2791ba7ee93cf020.webp" alt="Transport Mechanisms" width="800"/>

*MCP ट्रान्सपोर्ट यंत्रणा: HTTP दूरस्थ सर्व्हरसाठी, Stdio स्थानिक प्रक्रिया साठी*

**Stdio** - [StdioTransportDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java)

स्थानिक प्रक्रियांसाठी. तुमचा अनुप्रयोग एक सर्व्हर subprocess म्हणून सुरू करतो आणि मानक इनपुट/आउटपुटद्वारे संवाद करतो. फाइलसिस्टम प्रवेश किंवा कमांड-लाइन टूल्ससाठी उपयुक्त.

```java
McpTransport stdioTransport = new StdioMcpTransport.Builder()
    .command(List.of(
        npmCmd, "exec",
        "@modelcontextprotocol/server-filesystem@2025.12.18",
        resourcesDir
    ))
    .logEvents(false)
    .build();
```

`@modelcontextprotocol/server-filesystem` सर्व्हर खालील टूल्स पुरवतो, सगळे तुम्ही निर्दिष्ट केलेल्या निर्देशिकांमध्ये सॅंडबॉक्स केलेले आहेत:

| टूल | वर्णन |
|------|-------------|
| `read_file` | एका फाईलची सामग्री वाचा |
| `read_multiple_files` | एका कॉलमध्ये अनेक फाइल्स वाचा |
| `write_file` | एक फाइल तयार करा किंवा अधिलेखित करा |
| `edit_file` | लक्षित शोध व बदला संपादन करा |
| `list_directory` | एका मार्गावरील फाइल्स आणि निर्देशिका सूचीबद्ध करा |
| `search_files` | एखाद्या नमुन्यानुसार फाइल्स पुनरावृत्तीने शोधा |
| `get_file_info` | फाइल मेटाडेटा मिळवा (आकार, वेळांच्या टप्प्यांचे वेळापत्रक, परवानग्या) |
| `create_directory` | निर्देशिका तयार करा (पालक निर्देशिका सहित) |
| `move_file` | फाइल किंवा निर्देशिका हलवा किंवा नाव बदला |

खालची आकृती Stdio ट्रान्सपोर्ट कसा रनटाइमवर कार्य करतो ते दर्शवते — तुमचा Java अनुप्रयोग MCP सर्व्हरला चाइल्ड प्रक्रिया म्हणून सुरुवात करतो आणि ते stdin/stdout पाईप्सद्वारे संवाद करतात, नेटवर्क किंवा HTTP वापरले जात नाही:

<img src="../../../translated_images/mr/stdio-transport-flow.45eaff4af2d81db4.webp" alt="Stdio Transport Flow" width="800"/>

*Stdio ट्रान्सपोर्ट क्रियाशील — तुमचा अनुप्रयोग MCP सर्व्हरला एक चाइल्ड प्रक्रिया म्हणून सुरू करतो आणिstdin/stdout पाईप्सद्वारे संवाद साधतो.*

> **🤖 [GitHub Copilot](https://github.com/features/copilot) चॅटसह प्रयत्न करा:** [`StdioTransportDemo.java`](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java) उघडा आणि विचारा:
> - "Stdio ट्रान्सपोर्ट कसा कार्य करतो आणि मला HTTP पेक्षा कधी वापरावा?"
> - "LangChain4j कस प्रकारे MCP सर्व्हर प्रक्रियेचे जीवनचक्र व्यवस्थापित करते?"
> - "फाइल सिस्टमला AI प्रवेश देण्याचे सुरक्षा परिणाम काय आहेत?"

## एजेंटिक मॉड्यूल

MCP प्रमाणित टूल्स पुरवते, तर LangChain4j चे **एजेंटिक मॉड्यूल** त्या टूल्सचे संयोजन करणारे एजंट तयार करण्याचा घोषवाक्यात्मक मार्ग देते. `@Agent` अॅनोटेशन आणि `AgenticServices` तुम्हाला इंटरफेसद्वारे एजंटचे वर्तन परिभाषित करण्याची परवानगी देतात, जबाबदारी कोडऐवजी.

या मॉड्यूलमध्ये, तुम्ही **सुपरवायझर एजंट** पॅटर्न समजून घेणार आहात — एक प्रगत एजेंटिक AI दृष्टीकोन जिथे "सुपरवायझर" एजंट वापरकर्त्याच्या विनंतीनुसार कोणते सब-एजंट कॉल करायचे ते गतिशीलरित्या ठरवतो. आम्ही दोन्ही संकल्पना एकत्र करून आमच्या सब-एजंटना MCP-शक्त टूल्ससह फाइल प्रवेश क्षमता देऊ.

एजेंटिक मॉड्यूल वापरण्यासाठी, हा Maven अवलंबित्व जोडा:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```
> **टीप:** `langchain4j-agentic` मॉड्यूल वेगळ्या आवृत्ती गुणधर्म (`langchain4j.mcp.version`) वापरतो कारण ते मुख्य LangChain4j लायब्रऱ्यांच्या वेळापत्रकापेक्षा वेगळ्या वेळापत्रकावर प्रकाशित केले जाते.

> **⚠️ प्रायोगिक:** `langchain4j-agentic` मॉड्यूल **प्रायोगिक** आहे आणि बदल होऊ शकतात. स्थिर AI सहाय्यक निर्मितीचा मार्ग अजूनही `langchain4j-core` आणि सानुकूल टूल्स (Module 04) आहे.

### फाइल संचालन (Stdio)

हे स्थानिक subprocess-आधारित टूल्स दाखवते.

**✅ कोणतीही पूर्वतयारी आवश्यक नाही** - MCP सर्व्हर आपोआप सुरू होतो.

**प्रारंभ स्क्रिप्ट्स वापरणे (शिफारस केलेले):**

प्रारंभ स्क्रिप्ट्स रूट `.env` फाइलमधून पर्यावरणीय चल आपोआप लोड करतात:

**Bash:**
```bash
cd 05-mcp
chmod +x start-stdio.sh
./start-stdio.sh
```

**PowerShell:**
```powershell
cd 05-mcp
.\start-stdio.ps1
```

**VS Code वापरताना:** `StdioTransportDemo.java` वर उजवी क्लिक करा आणि **"Run Java"** निवडा (तुमची `.env` फाइल कॉन्फिगर आहे याची खात्री करा).

अनुप्रयोग आपोआप फाइलसिस्टम MCP सर्व्हर सुरू करतो आणि स्थानिक फाइल वाचतो. subprocess व्यवस्थापन कसे होते ते लक्षात घ्या.

**अपेक्षित आउटपुट:**

```
for integrating Large Language Models (LLMs) into Java applications...
```

### सुपरवायझर एजंट

**सुपरवायझर एजंट पॅटर्न** हा एक **लवचिक** एजेंटिक AI प्रकार आहे. सुपरवायझर वापरकर्ता विनंतीच्या आधारावर कोणते एजंट्स कॉल करायचे ते स्वयंपूर्णपणे ठरवतो. पुढील उदाहरणात, आम्ही MCP-शक्त फाइल प्रवेश LLM एजंटसह एकत्र करून नियंत्रणाखालील फाइल वाचन → अहवाल कार्यप्रवाह तयार करणार आहोत.

डेमोमध्ये, `FileAgent` MCP फाइलसिस्टम टूल्स वापरून फाइल वाचतो, आणि `ReportAgent` संचिप्त अहवाल तयार करतो ज्यामध्ये एक कार्यकारी सारांश (1 वाक्य), 3 मुख्य मुद्दे आणि शिफारशी असतात. सुपरवायझर हा प्रवाह आपोआप नियंत्रीत करतो:

<img src="../../../translated_images/mr/supervisor-agent-pattern.06275a41ae006ac8.webp" alt="Supervisor Agent Pattern" width="800"/>

*सुपरवायझर त्याचा LLM वापरून ठरवतो कोणते एजंट कोणत्या क्रमाने कॉल करायचे — कोणतीही हार्डकोडेड राउटिंग आवश्यक नाही.*

आमच्या फाइल-टू-रिपोर्ट पाईपलाइनसाठी ठोस कार्यप्रवाह असा दिसतो:

<img src="../../../translated_images/mr/file-report-workflow.649bb7a896800de9.webp" alt="File to Report Workflow" width="800"/>

*FileAgent MCP टूल्सद्वारे फाइल वाचतो, नंतर ReportAgent कच्चा मजकूर संरचित अहवालात रूपांतरित करतो.*

खालील अनुक्रमणिका चित्रकथेने संपूर्ण सुपरवायझर नियोजन दाखवते — MCP सर्व्हर स्पॉन करण्यापासून, सुपरवायझरच्या स्वायत्त एजंट निवडीपर्यंत, stdio ट्रान्सपोर्टवर टूल कॉल्सपर्यंत आणि अंतिम अहवालापर्यंत:

<img src="../../../translated_images/mr/supervisor-agent-sequence.1aa389b3bef99956.webp" alt="Supervisor Agent Sequence Diagram" width="800"/>

*सुपरवायझर स्वायत्तपणे FileAgent कॉल करतो (जो MCP सर्व्हरशी stdio द्वारे संवाद साधतो फाइल वाचण्यासाठी), नंतर ReportAgent कडे पुढे वळवतो संरचित अहवाल तयार करण्यासाठी — प्रत्येक एजंट आपले आउटपुट सामायिक Agentic Scope मध्ये साठवतो.*

प्रत्येक एजंट आपले आउटपुट **Agentic Scope** (सामायिक स्मृती) मध्ये साठवतो, ज्यामुळे नंतरचे एजंट मागील निकालांवर प्रवेश करू शकतात. हे दाखवते की MCP टूल्स एजेंटिक कार्यप्रवाहांमध्ये किती सहज एकत्रित होतात — सुपरवायझरला फाईल्स कशा वाचल्या जातात ते माहित असण्याचा प्रश्नच नाही, फक्त `FileAgent` हे करू शकतो हे माहित आहे.

#### सुपरवायझर कसे कार्य करते

एजंट तयार करण्यापूर्वी, तुम्हाला MCP ट्रान्सपोर्ट क्लायंटशी कनेक्ट करून त्याला `ToolProvider` म्हणून रॅप करणे आवश्यक आहे. हे कसे MCP सर्व्हरचे टूल्स तुमच्या एजंटसाठी उपलब्ध होतात:

```java
// ट्रान्सपोर्टमधून MCP क्लायंट तयार करा
McpClient mcpClient = new DefaultMcpClient.Builder()
        .transport(stdioTransport)
        .build();

// क्लायंटला ToolProvider म्हणून वेढा — यामुळे MCP साधने LangChain4j मध्ये जोडली जातात
ToolProvider mcpToolProvider = McpToolProvider.builder()
        .mcpClients(List.of(mcpClient))
        .build();
```

आता तुम्ही `mcpToolProvider` कोणत्याही एजंटमध्ये इंजेक्ट करू शकता ज्याला MCP टूल्स आवश्यक आहेत:

```java
// टप्पा 1: FileAgent MCP साधने वापरून फाइल्स वाचतो
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // फाइल ऑपरेशन्ससाठी MCP साधने आहेत
        .build();

// टप्पा 2: ReportAgent संरचित अहवाल तयार करतो
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// Supervisor फाइल → अहवाल वर्कफ्लोचे संचालन करतो
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // अंतिम अहवाल परत करा
        .build();

// Supervisor विनंतीनुसार कोणते एजंट्स कॉल करायचे ते ठरवतो
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```

#### FileAgent रनटाइमवर MCP टूल्स कसे शोधतो

तुम्हाला आश्चर्य वाटू शकते: **FileAgent ला npm फाइलसिस्टम टूल्स कसे वापरायचे माहित आहे का?** उत्तर आहे की त्याला नाही — **LLM** रनटाइमवर टूल स्कीमाद्वारे हे ठरवते.

`FileAgent` इंटरफेस फक्त **प्रॉम्प्ट व्याख्या** आहे. त्याला `read_file`, `list_directory`, किंवा इतर कोणत्याही MCP टूल्सबद्दल सानुकूल ज्ञान नाही. संपूर्ण प्रक्रिया अशा प्रकारे होते:
1. **सर्व्हर स्पॉन:** `StdioMcpTransport` `@modelcontextprotocol/server-filesystem` npm पॅकेजला एक चाइल्ड प्रोसेस म्हणून लॉन्च करतो  
2. **टूल शोध:** `McpClient` सर्व्हरकडे `tools/list` JSON-RPC विनंती पाठवतो, जो टूल नावे, वर्णने आणि पॅरामीटर स्कीमा (उदा., `read_file` — *"फाईलची पूर्ण माहिती वाचा"* — `{ path: string }`) प्रत्युत्तरात पाठवतो  
3. **स्कीमा इन्जेक्शन:** `McpToolProvider` या शोधलेल्या स्कीमांना व्रॅप करतो आणि LangChain4j कडे उपलब्ध करून देतो  
4. **LLM निर्णय:** जेव्हा `FileAgent.readFile(path)` कॉल केला जातो, तेव्हा LangChain4j सिस्टम मेसेज, वापरकर्ता मेसेज, **आणि टूल स्कीमांची यादी** LLM कडे पाठवतो. LLM टूल वर्णने वाचतो आणि टूल कॉल तयार करतो (उदा., `read_file(path="/some/file.txt")`)  
5. **अंमलबजावणी:** LangChain4j टूल कॉल इंटरसेप्ट करतो, ते MCP क्लायंटद्वारे नोड.जेएस सबप्रोसेसमध्ये मार्गदर्शित करतो, निकाल प्राप्त करतो आणि LLM ला परत देतो  

हेच [Tool Discovery](../../../05-mcp) यंत्रणा वरील वर्णन केलेल्या प्रमाणेच आहे, पण एजंट वर्कफ्लो मध्ये खास लागू केलेली आहे. `@SystemMessage` आणि `@UserMessage` एनोटेशन्स LLM चे वर्तन मार्गदर्शन करतात, तर इंजेक्ट केलेला `ToolProvider` त्याला **सामर्थ्ये** देतो — LLM हा रनटाइमला दोन्ही जोडतो.

#### प्रतिसाद धोरणे

जेव्हा तुम्ही `SupervisorAgent` कॉन्फिगर करता, तेव्हा तुम्ही सॉब-एजंट्स त्यांच्या कामे पूर्ण केल्यावर वापरकर्त्याला अंतिम उत्तर कसे द्यायचे ते निर्दिष्ट करता. खालील आकृतीत तीन उपलब्ध धोरणे दाखवले आहेत — LAST थेट अंतिम एजंटचा आउटपुट परत करते, SUMMARY सर्व आउटपुट्सची LLM द्वारे संक्षिप्त प्रतिक्रिया तयार करते, आणि SCORED मूळ विनंतीच्या विरुद्ध जास्त गुण आलेले आउटपुट निवडते:

<img src="../../../translated_images/mr/response-strategies.3d0cea19d096bdf9.webp" alt="Response Strategies" width="800"/>

*Supervisor अंतिम उत्तर तयार करण्यासाठी तीन धोरणे — तुम्हाला शेवटचा आउटपुट, सारांश किंवा सर्वोत्तम गुण मिळवलेले पर्याय हवा असेल त्यानुसार निवडा.*

उपलब्ध धोरणे:

| धोरण | वर्णन |
|-------|--------|
| **LAST** | सुपरवायझर शेवटच्या सॉब-एजंट किंवा कॉल केलेल्या टूलचा आउटपुट परत करतो. हे तिथे उपयोगी आहे जिथे वर्कफ्लोमधील अंतिम एजंट पूर्ण, अंतिम उत्तर निर्माण करण्यासाठी विशेष डिझाइन केलेले असते (उदा., संशोधन प्रक्रियेमध्ये "Summary Agent"). |
| **SUMMARY** | सुपरवायझर स्वतःच्या अंतर्गत भाषा मॉडेलचा (LLM) वापर करून संपूर्ण संभाषण व सर्व सॉब-एजंट आउटपुट समेटून एक सारांश तयार करतो आणि तो अंतिम उत्तर म्हणून देतो. याने वापरकर्त्यास एक स्वच्छ, संक्षिप्त उत्तर मिळते. |
| **SCORED** | सिस्टम अंतर्गत LLM वापरून LAST प्रतिसाद व संपूर्ण संभाषणाचा SUMMARY मूळ विनंतीशी गुणांकन करते, आणि जास्त गुण मिळालेल्या आउटपुट परत करते. |

#### आउटपुट समजून घेणे

जेव्हा तुम्ही डेमो चालवाल, तुम्हाला सुपरवायझर कसे अनेक एजंट्सचे समन्वय साधतो हे संरचित रूपात दिसेल. प्रत्येक विभागाचे अर्थ:

```
======================================================================
  FILE → REPORT WORKFLOW DEMO
======================================================================

This demo shows a clear 2-step workflow: read a file, then generate a report.
The Supervisor orchestrates the agents automatically based on the request.
```
  
**शीर्षक भाग** वर्कफ्लो संकल्पना परिचय देते: फाईल वाचनापासून रिपोर्ट जनरेशनपर्यंत एक केंद्रित पाइपलाइन.

```
--- WORKFLOW ---------------------------------------------------------
  ┌─────────────┐      ┌──────────────┐
  │  FileAgent  │ ───▶ │ ReportAgent  │
  │ (MCP tools) │      │  (pure LLM)  │
  └─────────────┘      └──────────────┘
   outputKey:           outputKey:
   'fileContent'        'report'

--- AVAILABLE AGENTS -------------------------------------------------
  [FILE]   FileAgent   - Reads files via MCP → stores in 'fileContent'
  [REPORT] ReportAgent - Generates structured report → stores in 'report'
```
  
**वर्कफ्लो आकृती** एजंट्समधील डेटा प्रवाह दाखवते. प्रत्येक एजंटची विशिष्ट भूमिका:  
- **FileAgent** MCP टूल्स वापरून फाईल वाचतो आणि कच्चा मजकूर `fileContent` मध्ये साठवतो  
- **ReportAgent** त्या मजकुराचा वापर करून `report` मध्ये संरचित अहवाल तयार करतो

```
--- USER REQUEST -----------------------------------------------------
  "Read the file at .../file.txt and generate a report on its contents"
```
  
**वापरकर्ता विनंती** कार्य दर्शवते. सुपरवायझर याला पार्स करतो आणि FileAgent → ReportAgent invoke करण्याचा निर्णय घेतो.

```
--- SUPERVISOR ORCHESTRATION -----------------------------------------
  The Supervisor decides which agents to invoke and passes data between them...

  +-- STEP 1: Supervisor chose -> FileAgent (reading file via MCP)
  |
  |   Input: .../file.txt
  |
  |   Result: LangChain4j is an open-source, provider-agnostic Java framework for building LLM...
  +-- [OK] FileAgent (reading file via MCP) completed

  +-- STEP 2: Supervisor chose -> ReportAgent (generating structured report)
  |
  |   Input: LangChain4j is an open-source, provider-agnostic Java framew...
  |
  |   Result: Executive Summary...
  +-- [OK] ReportAgent (generating structured report) completed
```
  
**सुपरवायझर समन्वय** 2 टप्प्यांतील प्रवाह दाखवतो:  
1. **FileAgent** MCP द्वारे फाईल वाचतो आणि मजकूर साठवतो  
2. **ReportAgent** मजकूर प्राप्त करतो व संरचित अहवाल तयार करतो  

सुपरवायझरने वापरकर्त्याच्या विनंतीवर **स्वतंत्रपणे** निर्णय घेतला.

```
--- FINAL RESPONSE ---------------------------------------------------
Executive Summary
...

Key Points
...

Recommendations
...

--- AGENTIC SCOPE (Data Flow) ----------------------------------------
  Each agent stores its output for downstream agents to consume:
  * fileContent: LangChain4j is an open-source, provider-agnostic Java framework...
  * report: Executive Summary...
```
  
#### एजंट मॉड्यूल वैशिष्ट्ये स्पष्टीकरण

उदाहरणाने एजंट मॉड्यूलची काही प्रगत वैशिष्ट्ये दाखवली आहेत. चला Agentic Scope आणि Agent Listeners जवळून पाहूया.

**Agentic Scope** हे सामायिक मेमरी दाखवते जिथे एजंट्सनी `@Agent(outputKey="...")` वापरून त्यांचे निकाल साठवले. यामुळे:  
- नंतरचे एजंट्स आधीच्या एजंट्सचे आउटपुट वापरू शकतात  
- सुपरवायझर अंतिम उत्तर तयार करू शकतो  
- तुम्ही काय उत्पादन झाले ते तपासू शकता  

खालील आकृती फाईल-टू-रिपोर्ट वर्कफ्लोमधील Agentic Scope चा सामायिक मेमरी म्हणून उपयोग दाखवते — FileAgent आपले आउटपुट `fileContent` कीखाली लिहितो, ReportAgent ते वाचतो आणि आपले आउटपुट `report` अंतर्गत लिहितो:

<img src="../../../translated_images/mr/agentic-scope.95ef488b6c1d02ef.webp" alt="Agentic Scope Shared Memory" width="800"/>

*Agentic Scope सामायिक मेमरी म्हणून काम करते — FileAgent `fileContent` लिहितो, ReportAgent ते वाचतो आणि `report` लिहितो, आणि तुमचा कोड अंतिम निकाल वाचतो.*

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // FileAgent कडून कच्चा फाइल डेटा
String report = scope.readState("report");            // ReportAgent कडून रचलेली अहवाल
```
  
**Agent Listeners** एजंट चालवण्याच्या मॉनिटरिंग व डीबगिंगसाठी सक्षम करतात. डेमोमध्ये तुम्हाला दिसणारा स्टेप-बाय-स्टेप आउटपुट प्रत्येक एजंट कॉलवर हुक होणाऱ्या AgentListener कडून येतो:  
- **beforeAgentInvocation** - सुपरवायझर एजंट निवडताच कॉल होतो, कोणता एजंट का निवडला ते दिसते  
- **afterAgentInvocation** - एजंट पूर्ण झाल्यावर कॉल होतो, निकाल दाखवतो  
- **inheritedBySubagents** - जर खरे असेल तर, लिसनर सगळ्या एजंट्सवर देखरेख करतो

खालील आकृती पूर्ण Agent Listener लाइफसायकल दाखवते, ज्यात `onError` एजंट अंमलबजावणी दरम्यान अपयश हाताळतो:

<img src="../../../translated_images/mr/agent-listeners.784bfc403c80ea13.webp" alt="Agent Listeners Lifecycle" width="800"/>

*Agent Listeners अंमलबजावणी दरम्यान हुक करतात — एजंट सुरू, संपूर्ण किंवा त्रुटी आल्यावर मॉनिटर करा.*

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // सर्व उप-एजंट्सना प्रसारित करा
    }
};
```
  
सुपरवायझर पॅटर्न व्यतिरिक्त, `langchain4j-agentic` मॉड्यूल अनेक सामर्थ्यवान वर्कफ्लो पॅटर्न्स देतो. खालील आकृती सर्व पाच दाखवते — सोप्या सिक्वेन्शियल पाइपलाइनपासून माणसाच्या सहभागाने मान्यतेपर्यंत:

<img src="../../../translated_images/mr/workflow-patterns.82b2cc5b0c5edb22.webp" alt="Agent Workflow Patterns" width="800"/>

*एजंट्सचे समन्वयासाठी पाच वर्कफ्लो पॅटर्न्स — सोप्या सिक्वेन्शियल पाइपलाइनपासून मानवी सहभाग मान्यता वर्कफ्लोपर्यंत.*

| पॅटर्न | वर्णन | वापर केस |
|---------|---------|------------|
| **Sequential** | एजंट्स क्रमाने चालवा, आउटपुट पुढील एजंटला जाईल | पाइपलाइन: संशोधन → विश्लेषण → अहवाल |
| **Parallel** | एजंट्स एकाच वेळी चालवा | स्वतंत्र कामे: हवामान + बातम्या + शेअर्स |
| **Loop** | जोपर्यंत अट पूर्ण होत नाही तोपर्यंत पुनरावृत्ती करा | गुणवत्ता स्कोअरिंग: स्कोअर ≥ 0.8 पर्यंत सुधारणा |
| **Conditional** | अटींवर आधारित मार्गदर्शन करा | वर्गीकरण → तज्ञ एजंटकडे पाठवा |
| **Human-in-the-Loop** | मानवी तपासणी बिंदू जोडा | मंजुरी वर्कफ्लो, सामग्री परीक्षण |

## मुख्य संकल्पना

आता तुम्ही MCP आणि एजंट मॉड्यूल वापरले आहेत, चला संक्षिप्त करूया की कोणता पर्याय केव्हा वापरायचा.

MCP चे मोठे फायदे म्हणजे त्याचा वाढता परिसंस्था. खाली आकृती एका सार्वत्रिक प्रोटोकॉलने तुमचे AI अॅप्लिकेशन विविध MCP सर्व्हर्सशी कसे जोडते ते दाखवते — फाईलसिस्टम, डेटाबेस, GitHub, ईमेल, वेब स्क्रॅपिंग आणि बरेच काही:

<img src="../../../translated_images/mr/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="MCP Ecosystem" width="800"/>

*MCP एक सार्वत्रिक प्रोटोकॉल परिसंस्था तयार करते — कोणताही MCP-सुसंगत सर्व्हर कोणत्याही MCP-सुसंगत क्लायंटशी काम करतो, अॅप्लिकेशन्समध्ये टूल शेअरिंगची परवानगी देते.*

**MCP** तेव्हा उत्तम जेव्हा तुम्हाला विद्यमान टूल परिसंस्था वापरायची आहे, असे टूल्स तयार करायचे आहेत जे अनेक अॅप्लिकेशन्स शेअर करू शकतात, तृतीय-पक्ष सेवा इंटीग्रेट करायच्या आहेत, किंवा कोड न बदलता टूल इंप्लिमेंटेशन बदलायचे आहेत.

**एजंट मॉड्यूल** तेव्हा उपयुक्त जेव्हा तुम्हाला `@Agent` एनोटेशन्ससह घोषणात्मक एजंट डिफिनिशन्स पाहिजे, वर्कफ्लो समन्वय (सिक्वेन्शियल, लूप, पॅरलल) हवा, इंटरफेस-आधारित एजंट डिझाइन आवडते, किंवा अनेक एजंट्सना `outputKey` द्वारे आउटपुट शेअर करायचे आहे.

**सुपरवायझर एजंट पॅटर्न** तेव्हा उपयुक्त जेव्हा वर्कफ्लो आधी निश्चित नसतो आणि तुम्हाला LLM कडून निर्णय हवा, जेव्हा तुम्हाला अनेक विशेष एजंट्सची डायनॅमिक समन्वय हवा, जेव्हा तुम्ही संवादात्मक सिस्टीम तयार करत आहात ज्या वेगवेगळ्या क्षमतांवर मार्गदर्शन करतात, किंवा जेव्हा सर्वात लवचिक, अनुकूली एजंट वर्तन पाहिजे.

कस्टम `@Tool` पद्धती (Module 04) आणि MCP टूल्स (Module 05) यांच्यात निवड करण्यासाठी, खाली तुलना महत्त्वाच्या फायद्यांवर प्रकाश टाकते — कस्टम टूल्स अॅप-विशिष्ट लॉजिकसाठी टाईट कपलिंग आणि पूर्ण टाइप सेफ्टी देतात, तर MCP टूल्स मानकीकृत, पुनर्नवीनीकरणीय एकत्रीकरण देतात:

<img src="../../../translated_images/mr/custom-vs-mcp-tools.c4f9b6b1cb65d8a1.webp" alt="Custom Tools vs MCP Tools" width="800"/>

*कधी कस्टम @Tool पद्धती वापराल आणि कधी MCP टूल्स — कस्टम टूल्स अॅप-विशिष्ट लॉजिकसाठी, MCP टूल्स अॅप्लिकेशन्समध्ये मानकीकृत एकत्रीकरणासाठी.*
