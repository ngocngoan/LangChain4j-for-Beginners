## ਤੁਸੀਂ ਕੀ ਸਿਖੋਗੇ

ਹੁਣ ਤੱਕ, ਤੁਸੀਂ ਸਿੱਖਿਆ ਹੈ ਕਿ AI ਨਾਲ ਗੱਲ-ਬਾਤ ਕਿਵੇਂ ਕਰਨੀ ਹੈ, ਪ੍ਰੌਂਪਟਸ ਨੂੰ ਪ੍ਰਭਾਵਸ਼ਾਲੀ ਢੰਗ ਨਾਲ ਬਣਾਉਣਾ ਹੈ, ਅਤੇ ਜਵਾਬਾਂ ਨੂੰ ਆਪਣੇ ਦਸਤਾਵੇਜ਼ਾਂ ਵਿੱਚ ਅਧਾਰਿਤ ਕਰਨਾ ਹੈ। ਪਰ ਇੱਕ ਮੂਲ ਸੀਮਾ ਹੈ: ਭਾਸ਼ਾ ਮਾਡਲ ਸਿਰਫ ਟੈਕਸਟ ਹੀ ਉਤਪੰਨ ਕਰ ਸਕਦੇ ਹਨ। ਉਹ ਮੌਸਮ ਚੈੱਕ ਨਹੀਂ ਕਰ ਸਕਦੇ, ਗਣਤਰੀ ਨਹੀਂ ਕਰ ਸਕਦੇ, ਡਾਟਾਬੇਸਕੂਇਰੀ ਨਹੀਂ ਕਰ ਸਕਦੇ ਤੇ ਬਾਹਰੀ ਪ੍ਰਣਾਲੀਆਂ ਨਾਲ ਸਬੰਧਤ ਨਹੀਂ ਹੋ ਸਕਦੇ।

ਟੂਲ ਇਹ ਬਦਲ ਦਿੰਦੇ ਹਨ। ਮਾਡਲ ਨੂੰ ਫੰਕਸ਼ਨਾਂ ਦੀ ਪਹੁੰਚ ਦੇ ਕੇ ਜੋ ਉਹ ਕਾਲ ਕਰ ਸਕਦਾ ਹੈ, ਤੁਸੀਂ ਇਸ ਨੂੰ ਇੱਕ ਟੈਕਸਟ ਜਨਰੇਟਰ ਤੋਂ ਇੱਕ ਏਜੰਟ ਵਿੱਚ ਬਦਲ ਦਿੰਦੇ ਹੋ ਜੋ ਕਾਰਵਾਈਆਂ ਕਰ ਸਕਦਾ ਹੈ। ਮਾਡਲ ਫੈਸਲਾ ਕਰਦਾ ਹੈ ਕਿ ਉਸਨੂੰ ਕਦੋਂ ਟੂਲ ਦੀ ਲੋੜ ਹੈ, ਕਿਹੜਾ ਟੂਲ ਵਰਤਣਾ ਹੈ, ਅਤੇ ਕਿਹੜੇ ਪੈਰਾਮੀਟਰ ਭੇਜੇ ਜਾਣ। ਤੁਹਾਡਾ ਕੋਡ ਫੰਕਸ਼ਨ ਚਲਾਉਂਦਾ ਹੈ ਅਤੇ ਨਤੀਜਾ ਵਾਪਸ ਕਰਦਾ ਹੈ। ਮਾਡਲ ਇਸ ਫਲਨੂੰ ਆਪਣੇ ਜਵਾਬ ਵਿੱਚ ਸ਼ਾਮਲ ਕਰਦਾ ਹੈ।

## ਟੂਲਜ਼ ਨਾਲ AI ਏਜੰਟ ਨੂੰ ਸਮਝਣਾ

> **📝 ਨੋਟ:** ਇਸ ਮੋਡੀਊਲ ਵਿੱਚ "ਏਜੰਟ" ਦਾ ਅਰਥ ਹੈ ਉਹ AI ਸਹਾਇਕ ਜੋ ਟੂਲ ਕਾਲਿੰਗ ਸਮਰੱਥਾ ਨਾਲ ਵਧਾਏ ਗਏ ਹਨ। ਇਹ **Agentic AI** ਪੈਟਰਨਜ਼ (ਆਤਮਨਿਰਭਰ ਏਜੰਟ ਜੋ ਯੋਜਨਾ, ਯਾਦاشت ਅਤੇ ਬਹੁ-ਕਦਮੀ ਸੋਚ ਵਾਲੇ ਹਨ) ਤੋਂ ਵੱਖਰੇ ਹਨ, ਜਿਨ੍ਹਾਂ ਨੂੰ ਅਸੀਂ [ਮੋਡੀਊਲ 05: MCP](../05-mcp/README.md) ਵਿੱਚ ਕਵਰ ਕਰਾਂਗੇ।

ਟੂਲਾਂ ਤੋਂ ਬਿਨਾਂ, ਭਾਸ਼ਾ ਮਾਡਲ ਸਿਰਫ ਆਪਣੇ ਸਿੱਖਿਆ ਡੇਟਾ ਵਿੱਚੋਂ ਟੈਕਸਟ ਜਨਰੇਟ ਕਰ ਸਕਦਾ ਹੈ। ਮੌਜੂਦਾ ਮੌਸਮ ਪੁੱਛੋ, ਉਹ ਅਨੁਮਾਨ ਲਗਾਉਂਦਾ ਹੈ। ਟੂਲ ਦਿਓ, ਇਹ ਮੌਸਮ API ਕਾਲ ਕਰ ਸਕਦਾ ਹੈ, ਗਣਨਾ ਕਰ ਸਕਦਾ ਹੈ ਜਾਂ ਡਾਟਾਬੇਸ ਵਿੱਚ ਪੁੱਛਗਿੱਛ ਕਰ ਸਕਦਾ ਹੈ — ਫਿਰ ਉਹਨਾਂ ਅਸਲ ਨਤੀਜਿਆਂ ਨੂੰ ਆਪਣੇ ਜਵਾਬ ਵਿੱਚ ਸ਼ਾਮਲ ਕਰਦਾ ਹੈ।

<img src="../../../translated_images/pa/what-are-tools.724e468fc4de64da.webp" alt="Without Tools vs With Tools" width="800"/>

*ਟੂਲਾਂ ਬਿਨਾਂ ਮਾਡਲ ਸਿਰਫ ਅਨੁਮਾਨ ਕਰਦਾ ਹੈ — ਟੂਲਾਂ ਨਾਲ ਇਹ API ਕਾਲ ਕਰ ਸਕਦਾ ਹੈ, ਗਣਨਾ ਚਲਾ ਸਕਦਾ ਹੈ ਅਤੇ ਅਸਲ ਸਮੇਂ ਦਾ ਡੇਟਾ ਵਾਪਸ ਕਰ ਸਕਦਾ ਹੈ।*

ਟੂਲਾਂ ਵਾਲਾ AI ਏਜੰਟ **Reasoning and Acting (ReAct)** ਪੈਟਰਨ ਅਨੁਸਾਰ ਕੰਮ ਕਰਦਾ ਹੈ। ਮਾਡਲ ਸਿਰਫ ਜਵਾਬ ਨਹੀਂ ਦਿੰਦਾ — ਇਹ ਸੋਚਦਾ ਹੈ ਕਿ ਉਸਨੂੰ ਕੀ ਚਾਹੀਦਾ ਹੈ, ਟੂਲ ਕੱਲ ਦਿੰਦਾ ਹੈ, ਨਤੀਜਾ ਵੇਖਦਾ ਹੈ, ਅਤੇ ਫੇਰ ਫੈਸਲਾ ਕਰਦਾ ਹੈ ਕਿ ਮੁੜ ਕਾਰਵਾਈ ਕਰਨੀ ਹੈ ਜਾਂ ਅਖੀਰਲਾ ਜਵਾਬ ਦੇਣਾ ਹੈ:

1. **ਵਿਚਾਰ ਕਰਨਾ** — ਏਜੰਟ ਉਪਭੋਗਤਾ ਦੇ ਸਵਾਲ ਦਾ ਵਿਸ਼ਲੇਸ਼ਣ ਕਰਦਾ ਹੈ ਤੇ ਇਹ ਨਿਰਧਾਰਿਤ ਕਰਦਾ ਹੈ ਕਿ ਉਸਨੂੰ ਕੀ ਜਾਣਕਾਰੀ ਚਾਹੀਦੀ ਹੈ
2. **ਕਾਰਵਾਈ ਕਰਨਾ** — ਏਜੰਟ ਸਹੀ ਟੂਲ ਚੁਣਦਾ ਹੈ, ਸਹੀ ਪੈਰਾਮੀਟਰ ਤਿਆਰ ਕਰਦਾ ਹੈ, ਤੇ ਕਾਲ ਕਰਦਾ ਹੈ
3. **ਦੇਖਣਾ** — ਏਜੰਟ ਟੂਲ ਦਾ ਨਤੀਜਾ ਪ੍ਰਾਪਤ ਕਰਦਾ ਹੈ ਅਤੇ ਮੁਲਾਂਕਣ ਕਰਦਾ ਹੈ
4. **ਮੁੜ ਕਰਨਾ ਜਾਂ ਜਵਾਬ ਦੇਣਾ** — ਜੇ ਹੋਰ ਡੇਟਾ ਲੋੜੀਂਦਾ ਹੈ, ਏਜੰਟ ਮੁੜ ਲੂਪ ਕਰਦਾ ਹੈ; ਨਹੀਂ ਤਾਂ ਕੁਦਰਤੀ ਭਾਸ਼ਾ ਵਿੱਚ ਜਵਾਬ ਬਣਾਉਂਦਾ ਹੈ

<img src="../../../translated_images/pa/react-pattern-detail.96a5efeeb6dd2f61.webp" alt="ReAct Pattern" width="800"/>

*ReAct ਚੱਕਰ — ਏਜੰਟ ਸੋਚਦਾ ਹੈ ਕਿ ਕੀ ਕਰਨਾ ਹੈ, ਟੂਲ ਕਾਲ ਕਰਦਾ ਹੈ, ਨਤੀਜਾ ਵੇਖਦਾ ਹੈ, ਅਤੇ ਜਦ ਤੱਕ ਅਖੀਰਲਾ ਜਵਾਬ ਨਾ ਦੇਵੇ, ਮੁੜਦਾ ਹੈ।*

ਇਹ ਸਾਰਾ ਪ੍ਰਕਿਰਿਆ ਆਟੋਮੈਟਿਕ ਹੁੰਦੀ ਹੈ। ਤੁਸੀਂ ਟੂਲ ਅਤੇ ਉਹਨਾਂ ਦੇ ਵੇਰਵੇ ਪਰਿਭਾਸ਼ਿਤ ਕਰਦੇ ਹੋ। ਮਾਡਲ ਤਾਂ ਇਹ ਫੈਸਲਾ ਕਰਦਾ ਹੈ ਕਿ ਕਦੋਂ ਅਤੇ ਕਿਵੇਂ ਇਹਨਾਂ ਨੂੰ ਵਰਤਣਾ ਹੈ।

## ਟੂਲ ਕਾਲਿੰਗ ਕਿਵੇਂ ਕੰਮ ਕਰਦੀ ਹੈ

### ਟੂਲ ਪਰਿਭਾਸ਼ਾਵਾਂ

[WeatherTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) | [TemperatureTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/TemperatureTool.java)

ਤੁਸੀਂ ਫੰਕਸ਼ਨਾਂ ਨੂੰ ਸਪਸ਼ਟ ਵੇਰਵਿਆਂ ਅਤੇ ਪੈਰਾਮੀਟਰ ਵੀਰਵਿਆਂ ਨਾਲ ਪਰਿਭਾਸ਼ਿਤ ਕਰਦੇ ਹੋ। ਮਾਡਲ ਆਪਣੇ ਸਿਸਟਮ ਪ੍ਰੌਂਪਟ ਵਿੱਚ ਇਹ ਵੇਰਵੇ ਵੇਖਦਾ ਹੈ ਅਤੇ ਸਮਝਦਾ ਹੈ ਕਿ ਹਰ ਟੂਲ ਕੀ ਕਰਦਾ ਹੈ।

```java
@Component
public class WeatherTool {
    
    @Tool("Get the current weather for a location")
    public String getCurrentWeather(@P("Location name") String location) {
        // ਤੁਹਾਡਾ ਮੌਸਮ ਲੁੱਕਅਪ ਲੋਜਿਕ
        return "Weather in " + location + ": 22°C, cloudy";
    }
}

@AiService
public interface Assistant {
    String chat(@MemoryId String sessionId, @UserMessage String message);
}

// ਅਸਿਸਟੈਂਟ ਆਪਣੇ ਆਪ Spring Boot ਨਾਲ ਵਾਇਰਡ ਹੈ:
// - ChatModel ਬੀਨ
// - ਸਾਰੇ @Component ਕਲਾਸਾਂ ਤੋਂ @Tool ਮੈਥਡ
// - ਸੈਸ਼ਨ ਪ੍ਰਬੰਧਨ ਲਈ ChatMemoryProvider
```

ਹੇਠਾਂ ਦਿੱਤਾ ਡਾਇਗ੍ਰਾਮ ਹਰ ਐਨੋਟੇਸ਼ਨ ਨੂੰ ਤੋੜਦਾ ਹੈ ਅਤੇ ਦਿਖਾਉਂਦਾ ਹੈ ਕਿ ਹਰ ਹਿੱਸਾ AI ਨੂੰ ਕਿਵੇਂ ਸਮਝਣ ਵਿੱਚ ਮਦਦ ਕਰਦਾ ਹੈ ਕਿ ਕਦੋਂ ਟੂਲ ਕਾਲ ਕਰਨਾ ਹੈ ਅਤੇ ਕਿਹੜੇ ਆਰਗੁਮੈਂਟ ਭੇਜਣੇ ਹਨ:

<img src="../../../translated_images/pa/tool-definitions-anatomy.f6468546037cf28b.webp" alt="Anatomy of Tool Definitions" width="800"/>

*ਟੂਲ ਪਰਿਭਾਸ਼ਾ ਦਾ ਗਠਜੋੜ — @Tool AI ਨੂੰ ਦੱਸਦਾ ਹੈ ਕਦੋਂ ਇਸਨੂੰ ਵਰਤਣਾ ਹੈ, @P ਹਰ ਪੈਰਾਮੀਟਰ ਨੂੰ ਵਰਣਨ ਕਰਦਾ ਹੈ, ਅਤੇ @AiService ਸਟਾਰਟਅਪ 'ਤੇ ਸਭ ਕੁਝ ਜੁੜਦਾ ਹੈ।*

> **🤖 [GitHub Copilot](https://github.com/features/copilot) ਚੈਟ ਨਾਲ ਕੋਸ਼ਿਸ਼ ਕਰੋ:** [`WeatherTool.java`](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) ਖੋਲ੍ਹੋ ਅਤੇ ਪੁੱਛੋ:
> - "ਮੈਂ mock ਡਾਟਾ ਦੀ ਜਗ੍ਹਾ ਇੱਕ ਅਸਲ Weather API ਜਿਵੇਂ OpenWeatherMap ਨੂੰ ਕਿਵੇਂ ਜੋੜ ਸਕਦਾ ਹਾਂ?"
> - "ਇੱਕ ਚੰਗੀ ਟੂਲ ਵਰਣਨ ਕੀ ਹੁੰਦੀ ਹੈ ਜੋ AI ਨੂੰ ਸਹੀ ਤਰ੍ਹਾਂ ਵਰਤਣ ਵਿੱਚ ਮਦਦ ਕਰਦੀ ਹੈ?"
> - "ਮੈਂ ਟੂਲ ਅਮਲ ਵਿੱਚ API ਤ੍ਰੁੱਟੀਆਂ ਅਤੇ ਰੇਟ ਸੀਮਾਵਾਂ ਨੂੰ ਕਿਵੇਂ ਸੰਭਾਲਾਂ?"

### ਫੈਸਲਾ ਲੈਣਾ

ਜਦੋਂ ਉਪਭੋਗਤਾ ਪੁੱਛਦਾ ਹੈ "ਸੀਏਟਲ ਵਿੱਚ ਮੌਸਮ ਕੀ ਹੈ?", ਮਾਡਲ ਬੇਤਰਤੀਬੀ ਨਾਲ ਕਿਸੇ ਟੂਲ ਨੂੰ ਨਹੀਂ ਚੁਣਦਾ। ਇਹ ਉਪਭੋਗਤਾ ਦੀ ਇੱਛਾ ਨੂੰ ਹਰ ਟੂਲ ਦੇ ਵੇਰਵੇ ਨਾਲ ਮਿਲਾ ਕੇ, ਹਰ ਇੱਕ ਨੂੰ ਵਿਸ਼ੇਸ਼ਤਾ ਅਨੁਸਾਰ ਸਕੋਰ ਕਰਦਾ ਹੈ ਅਤੇ ਸਭ ਤੋਂ ਵਧੀਆ ਚੁਣਦਾ ਹੈ। ਫਿਰ ਇਹ ਸਹੀ ਪੈਰਾਮੀਟਰ ਦੇ ਨਾਲ ਇੱਕ ਸੰਰਚਿਤ ਫੰਕਸ਼ਨ ਕਾਲ ਬਣਾਉਂਦਾ ਹੈ — ਇਸ ਮਾਮਲੇ ਵਿੱਚ, `location` ਨੂੰ `"Seattle"` ਸੈੱਟ ਕਰਦਾ ਹੈ।

ਜੇ ਕੋਈ ਟੂਲ ਉਪਭੋਗਤਾ ਦੀ ਬੇਨਤੀ ਨਾਲ ਨਹੀਂ ਮਿਲਦਾ, ਤਾਂ ਮਾਡਲ ਆਪਣੇ ਗਿਆਨ ਵਿੱਚੋਂ ਜਵਾਬ ਦਿੰਦਾ ਹੈ। ਜੇ ਕਈ ਟੂਲ ਮੈਚ ਕਰਦੇ ਹਨ, ਤਾਂ ਸਭ ਤੋਂ ਖਾਸ ਚੁਣਦਾ ਹੈ।

<img src="../../../translated_images/pa/decision-making.409cd562e5cecc49.webp" alt="How the AI Decides Which Tool to Use" width="800"/>

*ਮਾਡਲ ਹਰ ਉਪਲੱਬਧ ਟੂਲ ਨੂੰ ਉਪਭੋਗਤਾ ਦੀ ਇੱਛਾ ਨਾਲ ਮੈਚ ਕਰਦਾ ਹੈ ਅਤੇ ਸਭ ਤੋਂ ਵਧੀਆ ਚੁਣਦਾ ਹੈ — ਇਸ ਲਈ ਸਪਸ਼ਟ, ਵਿਸ਼ੇਸ਼ ਟੂਲ ਵੇਰਨਾਂ ਲਿਖਣਾ ਮਹੱਤਵਪੂਰਣ ਹੈ।*

### ਵਿਥਕਲਾ: ਸਪ੍ਰਿੰਗ ਬੂਟ ਆਟੋ-ਵਾਇਰਿੰਗ

ਇਹ ਮੋਡੀਊਲ LangChain4j ਦੀ ਸਪ੍ਰਿੰਗ ਬੂਟ ਇੰਟੀਗ੍ਰੇਸ਼ਨ ਵਰਤਦਾ ਹੈ ਜੋ ਡਿਕਲੈਰੇਟਿਵ `@AiService` ਇੰਟਰਫੇਸ ਨਾਲ ਹੁੰਦਾ ਹੈ। ਸਟਾਰਟਅਪ 'ਤੇ ਸਪ੍ਰਿੰਗ ਬੂਟ ਹਰ `@Component` ਖੋਜਦਾ ਹੈ ਜਿਸ ਵਿੱਚ `@Tool` ਮੈਥਡ ਹੁੰਦੇ ਹਨ, ਤੁਹਾਡਾ `ChatModel` ਬੀਨ, ਅਤੇ `ChatMemoryProvider` — ਫਿਰ ਇਹ ਸਭ ਨੂੰ ਇੱਕ `Assistant` ਇੰਟਰਫੇਸ ਵਿੱਚ ਬਿਨਾ ਕਿਸੇ ਬੁਆਇਲਰਪਲੇਟ ਦੇ ਜੋੜਦਾ ਹੈ।

<img src="../../../translated_images/pa/spring-boot-wiring.151321795988b04e.webp" alt="Spring Boot Auto-Wiring Architecture" width="800"/>

*@AiService ਇੰਟਰਫੇਸ ChatModel, ਟੂਲ ਕੰਪੋਨੇੰਟ ਅਤੇ ਮੈਮੋਰੀ ਪ੍ਰੋਵਾਇਡਰ ਨੂੰ ਜੋੜਦਾ ਹੈ — ਸਪ੍ਰਿੰਗ ਬੂਟ ਸਾਰੇ ਵਾਇਰਿੰਗ ਆਟੋਮੈਟਿਕ ਸੰਭਾਲਦਾ ਹੈ।*

ਇਸ ਢੰਗ ਦੇ ਕੁੰਜੀ ਫਾਇਦੇ:

- **ਸਪ੍ਰਿੰਗ ਬੂਟ ਆਟੋ-ਵਾਇਰਿੰਗ** — ChatModel ਅਤੇ ਟੂਲਾਂ ਆਟੋਮੈਟਿਕ ਇੰਜੈਕਟ
- **@MemoryId ਪੈਟਰਨ** — ਸੈਸ਼ਨ ਅਧਾਰਿਤ ਮੈਮੋਰੀ ਪ੍ਰਬੰਧਨ ਆਟੋਮੈਟਿਕ
- **ਇੱਕ ਹੀ ਉਦਾਹਰਨ** — Assistant ਸਿਰਫ ਇੱਕ ਵਾਰੀ ਬਣਾਓ ਅਤੇ ਬਿਹਤਰ ਪ੍ਰਦਰਸ਼ਨ ਲਈ ਦੁਬਾਰਾ ਵਰਤੋਂ
- **ਟਾਈਪ-ਸੇਫ ਕਾਰਜਨਵਾਇ** — ਜਾਵਾ ਮੈਥਡ ਸਿੱਧੇ ਕਾਲ ਕੀਤੇ ਜਾਂਦੇ ਹਨ ਟਾਈਪ ਕਨਵਰਜ਼ਨ ਨਾਲ
- **ਬਹੁ-ਟਰਨ ਨਿਰਦੇਸ਼ਨ** — ਟੂਲ ਚੇਨਿੰਗ ਨੂੰ ਆਟੋਮੈਟਿਕ ਸੰਭਾਲਦਾ ਹੈ
- **ਜੀਰੋ ਬੁਆਇਲਰਪਲੇਟ** — ਕੋਈ ਮੈਨੂਅਲ `AiServices.builder()` ਕਾਲ ਨਹੀਂ ਜਾਂ ਮੈਮੋਰੀ ਹੈਸ਼ਮੈਪ

ਵਿਕਲਪਿਕ ਢੰਗ (ਮੈਨੂਅਲ `AiServices.builder()`) ਜ਼ਿਆਦਾ ਕੋਡ ਲੈਂਦੇ ਹਨ ਅਤੇ ਸਪ੍ਰਿੰਗ ਬੂਟ ਇੰਟੀਗ੍ਰੇਸ਼ਨ ਫਾਇਦੇ ਨਹੀਂ ਦਿੰਦੇ।

## ਟੂਲ ਚੇਨਿੰਗ

**ਟੂਲ ਚੇਨਿੰਗ** — ਟੂਲ-ਆਧਾਰਿਤ ਏਜੰਟਾਂ ਦੀ ਅਸਲੀ ਤਾਕਤ ਇਸ ਵੇਲੇ ਵੇਖਾਈ ਦਿੰਦੀ ਹੈ ਜਦ ਇੱਕ ਸਵਾਲ ਲਈ ਕਈ ਟੂਲਾਂ ਦੀ ਲੋੜ ਹੁੰਦੀ ਹੈ। ਪੁੱਛੋ "ਸੀਏਟਲ ਵਿੱਚ ਮੌਸਮ ਫੈਰਨਹਾਈਟ ਵਿੱਚ ਕੀ ਹੈ?" ਅਤੇ ਏਜੰਟ ਆਪ ਹੀ ਦੋ ਟੂਲ ਚੇਨ ਕਰਦਾ ਹੈ: ਪਹਿਲਾਂ ਇਹ `getCurrentWeather` ਨੂੰ ਕਾਲ ਕਰਦਾ ਹੈ ਜਿਹੜਾ ਤਾਪਮਾਨ ਸੈਲਸੀਅਸ ਵਿੱਚ ਦਿੰਦਾ ਹੈ, ਫਿਰ ਇਹ ਮੁੱਲ `celsiusToFahrenheit` ਨੂੰ ਪ੍ਰਦਾਨ ਕਰਦਾ ਹੈ ਬਦਲਣ ਲਈ — ਸਾਰਾ ਕੁਝ ਇੱਕ ਗੱਲਬਾਤ ਦੇ ਚੱਕਰ ਵਿੱਚ।

<img src="../../../translated_images/pa/tool-chaining-example.538203e73d09dd82.webp" alt="Tool Chaining Example" width="800"/>

*ਟੂਲ ਚੇਨਿੰਗ ਕਾਰਵਾਈ ਵਿੱਚ — ਏਜੰਟ ਪਹਿਲਾਂ getCurrentWeather ਕਾਲ ਕਰਦਾ ਹੈ, ਫਿਰ ਸੈਲਸੀਅਸ ਨਤੀਜੇ ਨੂੰ celsiusToFahrenheit ਵਿੱਚ ਪਾਇਪ ਕਰਦਾ ਹੈ, ਅਤੇ ਮਿਲੇ-ਜੁਲੇ ਜਵਾਬ ਦਿੰਦਾ ਹੈ।*

ਇਹ ਕੁਝ ਇਸ ਤਰ੍ਹਾਂ ਦਿਖਾਈ ਦਿੰਦਾ ਹੈ ਡੌਕੂਮੈਂਟ ਐਪਲੀਕੇਸ਼ਨ 'ਚ — ਏਜੰਟ ਇਕ ਗੱਲਬਾਤ ਦੇ ਟਰਨ ਵਿੱਚ ਦੋ ਟੂਲ ਕਾਲ ਚੇਨ ਕਰਦਾ ਹੈ:

<a href="images/tool-chaining.png"><img src="../../../translated_images/pa/tool-chaining.3b25af01967d6f7b.webp" alt="Tool Chaining" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*ਅਸਲੀ ਐਪਲੀਕੇਸ਼ਨ ਆਉਟਪੁਟ — ਏਜੰਟ ਆਪਮੈਟਿਕ ਤੌਰ 'ਤੇ getCurrentWeather → celsiusToFahrenheit ਇੱਕ ਟਰਨ ਵਿੱਚ ਚੇਨ ਕਰਦਾ ਹੈ।*

**ਸੁੰਦਰ ਅਸਫਲਤਾਵਾਂ** — ਕਿਸੇ ਅਜਿਹੇ ਸ਼ਹਿਰ ਦਾ ਮੌਸਮ ਪੁੱਛੋ ਜੋ mock ਡੇਟਾ ਵਿੱਚ ਨਹੀਂ ਹੈ। ਟੂਲ ਤ੍ਰੁੱਟੀ ਸੁਨੇਹਾ ਦਿੰਦਾ ਹੈ, ਅਤੇ AI ਵਿਆਖਿਆ ਕਰਦਾ ਹੈ ਕਿ ਇਹ ਮਦਦ ਨਹੀਂ ਕਰ ਸਕਦਾ ਬਜਾਏ ਕਿ ਕਰੈਸ਼ ਕਰਨ ਦੇ। ਟੂਲ ਸੁਰੱਖਿਅਤ ਤਰੀਕੇ ਨਾਲ ਅਸਫਲ ਹੁੰਦੇ ਹਨ।

<img src="../../../translated_images/pa/error-handling-flow.9a330ffc8ee0475c.webp" alt="Error Handling Flow" width="800"/>

*ਜਦ ਕੋਈ ਟੂਲ ਅਸਫਲ ਹੁੰਦਾ ਹੈ, ਤਾਂ ਏਜੰਟ ਤ੍ਰੁੱਟੀ ਨੂੰ ਕੈਚ ਕਰਦਾ ਹੈ ਅਤੇ ਕਰੈਸ਼ ਕਰਨ ਦੀ ਬਜਾਏ ਸਹਾਇਕ ਵਿਆਖਿਆ ਨਾਲ ਜਵਾਬ ਦਿੰਦਾ ਹੈ।*

ਇਹ ਸਾਰਾ ਕੁਝ ਇੱਕ ਗੱਲਬਾਤ ਟਰਨ ਵਿੱਚ ਹੁੰਦਾ ਹੈ। ਏਜੰਟ ਕਈ ਟੂਲ ਕਾਲਾਂ ਨੂੰ ਸੁਤੰਤਰ ਤੌਰ 'ਤੇ ਮਿਲਾ ਕੇ ਚਲਾਉਂਦਾ ਹੈ।

## ਐਪਲੀਕੇਸ਼ਨ ਚਲਾਉਣਾ

**ਤੈਨਾਤੀ ਦੀ ਜਾਂਚ ਕਰੋ:**

ਪੱਕਾ ਕਰੋ ਕਿ ਰੂਟ ਡਾਇਰੈਕਟਰੀ ਵਿੱਚ `.env` ਫਾਇਲ ਹੈ ਜਿਸ ਵਿੱਚ Azure ਪ੍ਰਮਾਣ ਪੱਤਰ ਹਨ (ਮੋਡੀਊਲ 01 ਦੌਰਾਨ ਬਣਾਈ ਗਈ):
```bash
cat ../.env  # ਵਰਤੋਂਕਾਰ ਨੂੰ AZURE_OPENAI_ENDPOINT, API_KEY, DEPLOYMENT ਦਿਖਾਉਣੇ ਚਾਹੀਦੇ ਹਨ
```

**ਐਪਲੀਕੇਸ਼ਨ ਸ਼ੁਰੂ ਕਰੋ:**

> **ਨੋਟ:** ਜੇ ਤੁਸੀਂ ਪਹਿਲਾਂ ਹੀ ਮੋਡੀਊਲ 01 ਤੋਂ `./start-all.sh` ਨਾਲ ਸਾਰੇ ਐਪਲੀਕੇਸ਼ਨ ਚਲਾ ਚੁੱਕੇ ਹੋ, ਇਹ ਮੋਡੀਊਲ ਪਹਿਲਾਂ ਹੀ ਪੋਰਟ 8084 'ਤੇ ਚਲ ਰਿਹਾ ਹੈ। ਤੁਸੀਂ ਹੇਠਾਂ ਦਿੱਤੇ ਸਟਾਰਟ ਕਮਾਂਡ ਛੱਡ ਕੇ ਸਿੱਧਾ http://localhost:8084 ਤੇ ਜਾ ਸਕਦੇ ਹੋ।

**ਵਿਕਲਪ 1: ਸਪ੍ਰਿੰਗ ਬੂਟ ਡੈਸ਼ਬੋਰਡ ਦੀ ਵਰਤੋਂ (VS ਕੋਡ ਉਪਭੋਗਤਾਵਾਂ ਲਈ ਸਿਫ਼ਾਰਸ਼ ਕੀਤੀ)**

ਡਿਵ ਕਨਟੇਨਰ ਵਿੱਚ ਸਪ੍ਰਿੰਗ ਬੂਟ ਡੈਸ਼ਬੋਰਡ ਐਕਸਟੈਂਸ਼ਨ ਸ਼ਾਮਿਲ ਹੈ, ਜੋ ਸਾਰੇ ਸਪ੍ਰਿੰਗ ਬੂਟ ਐਪਲੀਕੇਸ਼ਨ ਨੂੰ ਮੈਨੇਜ ਕਰਨ ਲਈ ਦ੍ਰਿਸ਼ਟੀਗਤ ਇੰਟਰਫੇਸ ਪ੍ਰਦਾਨ ਕਰਦਾ ਹੈ। ਤੁਸੀਂ ਇਸਨੂੰ VS ਕੋਡ ਦੇ ਖੱਬੇ ਪਾਸੇ ਐਕਟਿਵਿਟੀ ਬਾਰ ਵਿੱਚ ਡੂੰਘਾ ਕਰੋ (ਸਪ੍ਰਿੰਗ ਬੂਟ ਆਈਕਨ ਵੇਖੋ)।

ਸਪ੍ਰਿੰਗ ਬੂਟ ਡੈਸ਼ਬੋਰਡ ਤੋਂ, ਤੁਸੀਂ:
- ਕੰਮ ਦੇ ਸਥਾਨ ਵਿੱਚ ਸਾਰੇ ਉਪਲੱਬਧ ਸਪ੍ਰਿੰਗ ਬੂਟ ਐਪਲੀਕੇਸ਼ਨ ਦੇਖ ਸਕਦੇ ਹੋ
- ਇੱਕ ਕਲਿੱਕ ਨਾਲ ਐਪਲੀਕੇਸ਼ਨ ਚਾਲੂ/ਰੋਕ ਸਕਦੇ ਹੋ
- ਰੀਅਲ-ਟਾਈਮ ਵਿੱਚ ਲਾਗ ਵੇਖ ਸਕਦੇ ਹੋ
- ਐਪਲੀਕੇਸ਼ਨ ਸਥਿਤੀ ਦੀ ਨਿਗਰਾਨੀ ਕਰ ਸਕਦੇ ਹੋ

ਸਿਰਫ "tools" ਦੇ ਨਾਲ ਖੇਡ ਬਟਨ 'ਤੇ ਕਲਿੱਕ ਕਰੋ ਇਸ ਮੋਡੀਊਲ ਨੂੰ ਚਲਾਉਣ ਲਈ, ਜਾਂ ਸਾਰਿਆਂ ਮੋਡੀਊਲ ਇੱਕ ਵਾਰੀ ਚਲਾਓ।

**ਵਿਕਲਪ 2: ਸ਼ੈੱਲ ਸਕ੍ਰਿਪਟਾਂ ਦੀ ਵਰਤੋਂ**

ਸਾਰੇ ਵੈੱਬ ਐਪਲੀਕੇਸ਼ਨ ਚਲਾਓ (ਮੋਡੀਊਲ 01-04):

### ਸਧਾਰਣ ਟੂਲ ਵਰਤੋਂ ਦੀ ਕੋਸ਼ਿਸ਼ ਕਰੋ
ਸਿੱਧਾ ਬੇਨਤੀ ਨਾਲ ਸ਼ੁਰੂ ਕਰੋ: "100 ਡਿਗਰੀ ਫੈਰਨਹਾਈਟ ਨੂੰ ਸੈਲਸੀਅਸ ਵਿੱਚ ਬਦਲੋ"। ਏਜੰਟ ਸਮਝਦਾ ਹੈ ਕਿ ਉਸਨੂੰ ਤਾਪਮਾਨ ਬਦਲਣ ਵਾਲੇ ਸੰਦ ਦੀ ਲੋੜ ਹੈ, ਉਹ ਇਸਨੂੰ ਸਹੀ ਪੈਰਾਮੀਟਰਾਂ ਨਾਲ ਕਾਲ ਕਰਦਾ ਹੈ ਅਤੇ ਨਤੀਜਾ ਵਾਪਸ ਕਰਦਾ ਹੈ। ਧਿਆਨ ਦਿਓ ਕਿ ਇਹ ਕਿੰਨਾ ਕੁਸਲਤਾਪੂਰਕ ਲੱਗਦਾ ਹੈ - ਤੁਸੀਂ ਇਹ ਨਹੀਂ ਦੱਸਿਆ ਕਿ ਕਿਹੜਾ ਸੰਦ ਵਰਤਣਾ ਹੈ ਜਾਂ ਕਿਵੇਂ ਕਾਲ ਕਰਨੀ ਹੈ।

### ਟੂਲ ਚੇਨਿੰਗ ਦੀ ਜਾਂਚ ਕਰੋ

ਹੁਣ ਕੁਝ ਹੋਰ ਜਟਿਲ ਕੋਸ਼ਿਸ਼ ਕਰੋ: "ਸੀਏਟਲ ਵਿੱਚ ਮੌਸਮ ਕਿਵੇਂ ਹੈ ਅਤੇ ਇਸਨੂੰ ਫੈਰਨਹਾਈਟ ਵਿੱਚ ਬਦਲੋ?" ਧਿਆਨ ਨਾਲ ਦੇਖੋ ਕਿ ਏਜੰਟ ਕਿਵੇਂ ਕਦਮ ਦਰ ਕਦਮ ਇਸ ਕੰਮ ਨੂੰ ਕਰਦਾ ਹੈ। ਪਹਿਲਾਂ ਉਹ ਮੌਸਮ ਲੈਂਦਾ ਹੈ (ਜੋ ਸੈਲਸੀਅਸ ਵਿੱਚ ਹੁੰਦਾ ਹੈ), ਫਿਰ ਸਮਝਦਾ ਹੈ ਕਿ ਫੈਰਨਹਾਈਟ ਵਿੱਚ ਬਦਲਣਾ ਹੈ, ਬਦਲੀ ਵਾਲੇ ਸੰਦ ਨੂੰ ਕਾਲ ਕਰਦਾ ਹੈ, ਅਤੇ ਦੋਨਾਂ ਨਤੀਜੇ ਇਕੱਠੇ ਕਰ ਕੇ ਇੱਕ ਜਵਾਬ ਦਿੰਦਾ ਹੈ।

### ਗੱਲਬਾਤ ਦਾ ਪ੍ਰਵਾਹ ਵੇਖੋ

ਚੈਟ ਇੰਟਰਫੇਸ ਗੱਲਬਾਤ ਇਤਿਹਾਸ ਨੂੰ ਸੰਭਾਲਦਾ ਹੈ, ਜਿਸ ਨਾਲ ਤੁਸੀਂ ਇੱਕ ਤੋਂ ਵੱਧ ਵਾਰੀ ਗੱਲਬਾਤ ਕਰ ਸਕਦੇ ਹੋ। ਤੁਸੀਂ ਸਾਰੇ ਪਿਛਲੇ ਪ੍ਰਸ਼ਨਾਂ ਅਤੇ ਜਵਾਬਾਂ ਨੂੰ ਦੇਖ ਸਕਦੇ ਹੋ, ਜਿਸ ਨਾਲ ਗੱਲਬਾਤ ਨੂੰ ਟ੍ਰੈਕ ਕਰਨਾ ਅਤੇ ਸਮਝਣਾ ਆਸਾਨ ਹੁੰਦਾ ਹੈ ਕਿ ਏਜੰਟ ਕਿਵੇਂ ਕਈ ਵਾਰੀ ਜਾਂਚ ਕਰਕੇ ਸੰਦਰਭ ਬਣਾਉਂਦਾ ਹੈ।

## ਮੁੱਖ ਧਾਰਨਾਵਾਂ

### ReAct ਪੈਟਰਨ (ਚਿੰਤਨ ਅਤੇ ਕਿਰਿਆ)

ਏਜੰਟ ਸੋਚਣ (ਫੈਸਲਾ ਕਰਨ) ਅਤੇ ਕੰਮ ਕਰਨ (ਟੂਲਾਂ ਦੀ ਵਰਤੋਂ) ਵਿਚ ਬਦਲਦਾ ਰਹਿੰਦਾ ਹੈ। ਇਹ ਪੈਟਰਨ ਆਪਣੇ ਆਪ ਸਮੱਸਿਆ ਹੱਲ ਕਰਨ ਯੋਗ ਬਣਾਉਂਦਾ ਹੈ ਨਾ ਕਿ ਸਿਰਫ ਹੁਕਮਾਂ ਦਾ ਉੱਤਰ ਦੇਣ ਲਈ।

### ਟੂਲ ਵਰਣਨ ਮਹੱਤਵਪੂਰਣ ਹਨ

ਤੁਹਾਡੇ ਟੂਲ ਵਰਣਨਾਂ ਦੀ ਗੁਣਵੱਤਾ ਸਿੱਧੀ ਤਰ੍ਹਾਂ ਪ੍ਰਭਾਵਿਤ ਕਰਦੀ ਹੈ ਕਿ ਏਜੰਟ ਕਿਵੇਂ ਉਹਨਾਂ ਨੂੰ ਵਰਤਦਾ ਹੈ। ਸਾਫ਼ ਅਤੇ ਖਾਸ ਵਰਣਨ ਮਾਡਲ ਨੂੰ ਇਹ ਸਮਝਾਉਂਦੇ ਹਨ ਕਿ ਕਦੋਂ ਅਤੇ ਕਿਵੇਂ ਹਰ ਟੂਲ ਨੂੰ ਕਾਲ ਕਰਨਾ ਹੈ।

### ਸੈਸ਼ਨ ਪ੍ਰਬੰਧਨ

`@MemoryId` ਐਨੋਟੇਸ਼ਨ ਆਟੋਮੈਟਿਕ ਸੈਸ਼ਨ ਅਧਾਰਿਤ ਯਾਦਦਾਸ਼ਤ ਪ੍ਰਬੰਧਨ ਲਈ ਹੁੰਦੈ। ਹਰ ਸੈਸ਼ਨ ID ਨੂੰ ਇੱਕ ਖ਼ਾਸ `ChatMemory` ਇੰਸਟੈਂਸ ਮਿਲਦਾ ਹੈ ਜੋ `ChatMemoryProvider` ਬੀਨ ਦੁਆਰਾ ਸੰਭਾਲਿਆ ਜਾਂਦਾ ਹੈ, ਤਾਂ ਜੋ ਕਈ ਯੂਜ਼ਰ ਏਕੱਠੇ ਏਜੰਟ ਨਾਲ ਬਿਨਾਂ ਗੱਲਬਾਤ ਬਿਲਕੁਲ ਮਿਲੇ ਜੁਲੇ ਬਿਨਾਂ ਇੰਟਰੈਕਟ ਕਰ ਸਕਣ।

<img src="../../../translated_images/pa/session-management.91ad819c6c89c400.webp" alt="Session Management with @MemoryId" width="800"/>

*ਹਰ ਸੈਸ਼ਨ ID ਇੱਕ ਅਲੱਗ ਗੱਲਬਾਤ ਇਤਿਹਾਸ ਨਾਲ ਜੁੜਦਾ ਹੈ — ਯੂਜ਼ਰ ਆਪਣੇ-ਆਪਣੇ ਸੁਨੇਹੇ ਨਹੀਂ ਦੇਖਦੇ।*

### ਗਲਤੀ ਸਾਂਭਣਾ

ਟੂਲ ਫੇਲ ਹੋ ਸਕਦੇ ਹਨ — API ਸਮਾਂ ਖ਼ਤਮ ਹੋ ਸਕਦਾ ਹੈ, ਪੈਰਾਮੀਟਰ ਗਲਤ ਹੋ ਸਕਦੇ ਹਨ, ਬਾਹਰੀ ਸੇਵਾਵਾਂ ਡਾਊਨ ਹੋ ਸਕਦੀਆਂ ਹਨ। ਪ੍ਰੋਡਕਸ਼ਨ ਏਜੰਟਾਂ ਨੂੰ ਗਲਤੀ ਸਾਂਭਣ ਦੀ ਲੋੜ ਹੁੰਦੀ ਹੈ ਤਾਂ ਜੋ ਮਾਡਲ ਦੋਸ਼ ਸਮਝਾ ਸਕੇ ਜਾਂ ਵਿਕਲਪ ਕੋਸ਼ਿਸ਼ ਕਰ ਸਕੇ, ਨਾ ਕਿ ਪੂਰੀ ਐਪਲੀਕੇਸ਼ਨ ਡਿੱਗ ਜਾਵੇ। ਜਦੋਂ ਕਦੇ ਟੂਲ ਵਿੱਚ ਅਪਵਾਦ ਪੈਦਾ ਹੁੰਦਾ ਹੈ, ਤਾਂ LangChain4j ਉਹਨੂੰ ਫੜਦਾ ਹੈ ਅਤੇ ਗਲਤੀ ਦਾ ਸੁਨੇਹਾ ਮਾਡਲ ਨੇ ਵਾਪਸ ਦਿੰਦਾ ਹੈ, ਜੋ ਕੁਦਰਤੀ ਭਾਸ਼ਾ ਵਿੱਚ ਸਮੱਸਿਆ ਸਮਝਾ ਸਕਦਾ ਹੈ।

*RAG ਸਥਿਰ ਦਸਤਾਵੇਜ਼ਾਂ ਵਿਚੋਂ ਜਾਣਕਾਰੀ ਲੱਭਦਾ ਹੈ — ਟੂਲ ਕਾਰਵਾਈਆਂ ਕਰਦੇ ਹਨ ਅਤੇ ਗਤੀਸ਼ੀਲ, ਅਸਲੀ ਸਮੇਂ ਡਾਟਾ ਲੈਂਦੇ ਹਨ। ਕਈ ਪੇਸ਼ੇਵਰ ਪ੍ਰਣਾਲੀਆਂ ਦੋਹਾਂ ਨੂੰ ਮਿਲਾ ਕੇ ਵਰਤਦੀਆਂ ਹਨ।*

ਅਮਲ ਵਿੱਚ, ਕਈ ਪ੍ਰੋਡਕਸ਼ਨ ਸਿਸਟਮ ਦੋਹਾਂ ਤਕਨੀਕਾਂ ਦਾ ਮਿਸ਼ਰਣ ਕਰਦੇ ਹਨ: RAG ਤੁਹਾਡੇ ਡਾਕਯੂਮੈਂਟਰੇਸ਼ਨ ਲਈ ਅਤੇ ਟੂਲਜ਼ ਜਿਉਂਦੇ ਡਾਟਾ ਲੈਣ ਜਾਂ ਕੰਮ ਕਰਨ ਲਈ।
