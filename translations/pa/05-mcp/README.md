## ਤੁਸੀਂ ਕੀ ਸਿੱਖੋਗੇ

ਤੁਸੀਂ ਗੱਲਬਾਤੀ AI ਬਣਾਈ ਹੈ, ਪ੍ਰਾਂਪਟਾਂ ਵਿੱਚ ਮਹਾਰਤ ਹਾਸਲ ਕੀਤੀ ਹੈ, ਦਸਤਾਵੇਜ਼ਾਂ ਵਿੱਚ ਜਵਾਬਾਂ ਨੂੰ ਮੂਲ ਬੁਨਿਆਦ ਦਿੱਤੀ ਹੈ, ਅਤੇ ਟੂਲਾਂ ਨਾਲ ਏਜੰਟ ਬਣਾਏ ਹਨ। ਪਰ ਸਾਰੇ ਉਹ ਟੂਲ ਤੁਹਾਡੇ ਖਾਸ ਐਪਲੀਕੇਸ਼ਨ ਲਈ ਕਸਟਮ ਬਣਾਏ ਗਏ ਸਨ। ਜੇ ਤੁਸੀਂ ਆਪਣੇ AI ਨੂੰ ਇੱਕ ਮਿਆਰੀਕ੍ਰਿਤ ਟੂਲ ਇਕੋ-ਸਿਸਟਮ ਤੱਕ ਪਹੁੰਚ ਦੇ ਸਕੋ ਜੋ ਕਿਸੇ ਵੀ ਬੰਦੇ ਦੁਆਰਾ ਬਣਾਇਆ ਅਤੇ ਸਾਂਝਾ ਕੀਤਾ ਜਾ ਸਕਦਾ ਹੋਵੇ ਤਾਂ? ਇਸ ਮਾਡਿਊਲ ਵਿੱਚ, ਤੁਸੀਂ ਸਿੱਖੋਗੇ ਕਿ ਮਾਡਲ ਸੰਦਰਭ ਪ੍ਰੋਟੋਕੋਲ (MCP) ਅਤੇ LangChain4j ਦੇ ਏਜੰਟਿਕ ਮਾਡਿਊਲ ਨਾਲ ਇਹ ਕਿਵੇਂ ਕੀਤਾ ਜਾਵੇ। ਅਸੀਂ ਪਹਿਲਾਂ ਇੱਕ ਸਧਾਰਣ MCP ਫਾਈਲ ਰੀਡਰ ਦਿਖਾਉਂਦੇ ਹਾਂ ਅਤੇ ਫਿਰ ਦਿਖਾਉਂਦੇ ਹਾਂ ਕਿ ਇਹ ਕਿਵੇਂ ਸੁਪਰਵਾਇਜ਼ਰ ਏਜੰਟ ਪੈਟਰਨ ਦੀ ਵਰਤੋਂ ਕਰਕੇ ਉੱਚ-ਪੱਧਰੀ ਏਜੰਟਿਕ ਵਰਕਫਲੋਜ਼ ਵਿੱਚ ਆਸਾਨੀ ਨਾਲ ਇਕਠੇ ਹੋ ਜਾਂਦਾ ਹੈ।

## MCP ਕੀ ਹੈ?

ਮਾਡਲ ਸੰਦਰਭ ਪ੍ਰੋਟੋਕੋਲ (MCP) ਓਹੀ ਚੀਜ਼ ਪ੍ਰਦਾਨ ਕਰਦਾ ਹੈ - AI ਐਪਲੀਕੇਸ਼ਨਾਂ ਲਈ ਬਾਹਰੀ ਟੂਲਾਂ ਖੋਜਣ ਅਤੇ ਵਰਤਣ ਦਾ ਇੱਕ ਮਿਆਰੀ ਤਰੀਕਾ। ਹਰ ਡੇਟਾ ਸਰੋਤ ਜਾਂ ਸਰਵਿਸ ਲਈ ਕਸਟਮ ਇੰਟੀਗ੍ਰੇਸ਼ਨ ਲਿਖਣ ਦੀ ਜਗ੍ਹਾ, ਤੁਸੀਂ MCP ਸਰਵਰਜ਼ ਨਾਲ ਜੁੜਦੇ ਹੋ ਜੋ ਆਪਣੀਆਂ ਯੋਗਤਾਵਾਂ ਨੂੰ ਇਕਸਾਰ ਫਾਰਮੈਟ ਵਿੱਚ ਪ੍ਰਗਟਾਉਂਦੇ ਹਨ। ਤੁਹਾਡਾ AI ਏਜੰਟ ਫਿਰ ਆਟੋਮੈਟਿਕ ਵਿਚਾਰ ਕਰ ਸਕਦਾ ਹੈ ਅਤੇ ਇਹਨਾਂ ਟੂਲਾਂ ਨੂੰ ਵਰਤ ਸਕਦਾ ਹੈ।

ਹੇਠਾਂ ਦਿੱਤਾ ਚਿੱਤਰ ਫਰਕ ਦਿਖਾਉਂਦਾ ਹੈ — MCP ਦੇ ਬਿਨਾ, ਹਰ ਇਕ ਇੰਟੀਗ੍ਰੇਸ਼ਨ ਲਈ ਕਸਟਮ ਪੌਇੰਟ-ਟੂ-ਪੌਇੰਟ ਵਾਇਰਿੰਗ ਦੀ ਲੋੜ ਹੈ; MCP ਨਾਲ, ਇੱਕ ਪ੍ਰੋਟੋਕੋਲ ਤੁਹਾਡੇ ਐਪ ਨੂੰ ਕਿਸੇ ਵੀ ਟੂਲ ਨਾਲ ਜੋੜਦਾ ਹੈ:

<img src="../../../translated_images/pa/mcp-comparison.9129a881ecf10ff5.webp" alt="MCP Comparison" width="800"/>

*ਪਹਿਲਾਂ MCP: ਜਟਿਲ ਪੌਇੰਟ-ਟੂ-ਪੌਇੰਟ ਇੰਟੀਗ੍ਰੇਸ਼ਨ। MCP ਤੋਂ ਬਾਅਦ: ਇੱਕ ਪ੍ਰੋਟੋਕੋਲ, ਅੰਤਹੀਣ ਸੰਭਾਵਨਾਵਾਂ।*

MCP AI ਵਿਕਾਸ ਵਿੱਚ ਇੱਕ ਮੂਲ ਸਮੱਸਿਆ ਨੂੰ ਹੱਲ ਕਰਦਾ ਹੈ: ਹਰ ਇੰਟੀਗ੍ਰੇਸ਼ਨ ਕਸਟਮ ਹੁੰਦੀ ਹੈ। GitHub ਤੱਕ ਪਹੁੰਚ ਚਾਹੁੰਦੇ ਹੋ? ਕਸਟਮ ਕੋਡ। ਫਾਈਲਾਂ ਪੜ੍ਹਨੀ ਹਨ? ਕਸਟਮ ਕੋਡ। ਡੇਟਾਬੇਸ ਨੂੰ ਪ੍ਰਸ਼ਨ ਪੁੱਛਣਾ ਹੈ? ਕਸਟਮ ਕੋਡ। ਤੇ ਇਹਨਾਂ ਵਿੱਚੋਂ ਕੋਈ ਵੀ ਇੰਟੀਗ੍ਰੇਸ਼ਨ ਹੋਰ AI ਐਪਲੀਕੇਸ਼ਨਾਂ ਨਾਲ ਕੰਮ ਨਹੀਂ ਕਰਦਾ।

MCP ਇਸਨੂੰ ਮਿਆਰੀਕ੍ਰਿਤ ਕਰਦਾ ਹੈ। ਇੱਕ MCP ਸਰਵਰ ਸਾਫ਼ ਵੇਰਵਾ ਅਤੇ ਸਕੀਮਾਂ ਨਾਲ ਟੂਲ ਪ੍ਰਗਟਾਉਂਦਾ ਹੈ। ਕੋਈ ਵੀ MCP ਕਲਾਈਟ ਕੰਨੈਕਟ ਕਰ ਸਕਦਾ ਹੈ, ਉਪਲਬਧ ਟੂਲ ਖੋਜ ਸਕਦਾ ਹੈ, ਅਤੇ ਉਨ੍ਹਾਂ ਨੂੰ ਵਰਤ ਸਕਦਾ ਹੈ। ਇੱਕ ਵਾਰੀ ਬਣਾਉ, ਹਰੇਕ ਜਗ੍ਹਾ ਵਰਤੋਂ।

ਹੇਠਾਂ ਦਿੱਤਾ ਚਿੱਤਰ ਇਹ ਵਿਆਖਿਆ ਕਰਦਾ ਹੈ — ਇੱਕ ਇਕੱਲਾ MCP ਕਲਾਈਟ (ਤੁਹਾਡੀ AI ਐਪ) ਕਈ MCP ਸਰਵਰਜ਼ ਨਾਲ ਜੁੜਦਾ ਹੈ, ਜਿਹੜੇ ਆਪਣਾ ਟੂਲ ਸੈੱਟ ਮਿਆਰੀ ਪ੍ਰੋਟੋਕੋਲ ਰਾਹੀਂ ਪ੍ਰਗਟਾਉਂਦੇ ਹਨ:

<img src="../../../translated_images/pa/mcp-architecture.b3156d787a4ceac9.webp" alt="MCP Architecture" width="800"/>

*ਮਾਡਲ ਸੰਦਰਭ ਪ੍ਰੋਟੋਕੋਲ ਦਾ ਆਰਕੀਟੈਕਚਰ - ਮਿਆਰੀਕ੍ਰਿਤ ਟੂਲ ਖੋਜ ਅਤੇ ਕਾਰਜ*

## MCP ਕਿਵੇਂ ਕੰਮ ਕਰਦਾ ਹੈ

ਅੰਦਰ, MCP ਇੱਕ ਪਰਤਦਾਰ ਆਰਕੀਟੈਕਚਰ ਦੀ ਵਰਤੋਂ ਕਰਦਾ ਹੈ। ਤੁਹਾਡੀ ਜਾਵਾ ਐਪਲੀਕੇਸ਼ਨ (MCP ਕਲਾਈਟ) ਉਪਲਬਧ ਟੂਲ ਖੋਜਦੀ ਹੈ, ਇੱਕ ਟਰਾਂਸਪੋਰਟ ਪਰਤ (Stdio ਜਾਂ HTTP) ਰਾਹੀਂ JSON-RPC ਬੇਨਤੀਆਂ ਭੇਜਦੀ ਹੈ, ਅਤੇ MCP ਸਰਵਰ ਕਾਰਜਕਾਰੀ ਕਰਦਾ ਹੈ ਅਤੇ ਨਤੀਜੇ ਵਾਪਸ ਕਰਦਾ ਹੈ। ਹੇਠਾਂ ਦਿੱਤਾ ਚਿੱਤਰ ਇਸ ਪ੍ਰੋਟੋਕੋਲ ਦੇ ਹਰ ਪਰਤ ਦਾ ਵਿਸਥਾਰ ਦਿਖਾਉਂਦਾ ਹੈ:

<img src="../../../translated_images/pa/mcp-protocol-detail.01204e056f45308b.webp" alt="MCP Protocol Detail" width="800"/>

*MCP ਅੰਦਰ ਕਿਵੇਂ ਕੰਮ ਕਰਦਾ ਹੈ — ਕਲਾਈਟਟੂਲ ਖੋਜਦੇ ਹਨ, JSON-RPC ਸੁਨੇਹੇ ਬਦਲਦੇ ਹਨ, ਅਤੇ ਟਰਾਂਸਪੋਰਟ ਪਰਤ ਰਾਹੀਂ ਕਾਰਜਾਂ ਨੂੰ ਚਲਾਉਂਦੇ ਹਨ।*

**ਸਰਵਰ-ਕਲਾਈਟ ਆਰਕੀਟੈਕਚਰ**

MCP client-server ਮਾਡਲ ਵਰਤਦਾ ਹੈ। ਸਰਵਰ ਟੂਲ ਦਿੰਦਾ ਹੈ - ਫਾਈਲਾਂ ਪੜ੍ਹਨਾ, ਡੇਟਾਬੇਸ ਪ੍ਰਸ਼ਨ, APIs ਕਾਲ ਕਰਨਾ। ਕਲਾਈਟ (ਤੁਹਾਡੀ AI ਐਪ) ਸਰਵਰ ਨਾਲ ਜੁੜਦਾ ਹੈ ਅਤੇ ਉਨ੍ਹਾਂ ਟੂਲਾਂ ਨੂੰ ਵਰਤਦਾ ਹੈ।

LangChain4j ਨਾਲ MCP ਵਰਤਣ ਲਈ, ਇਸ Maven ਡੀਪੈਂਡੇੰਸੀ ਨੂੰ ਸ਼ਾਮਲ ਕਰੋ:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```

**ਟੂਲ ਖੋਜ**

ਜਦੋਂ ਤੁਹਾਡਾ ਕਲਾਈਟ MCP ਸਰਵਰ ਨਾਲ ਜੁੜਦਾ ਹੈ, ਇਹ ਪੁੱਛਦਾ ਹੈ "ਤੁਹਾਡੇ ਕੋਲ ਕਿਹੜੇ ਟੂਲ ਹਨ?" ਸਰਵਰ ਉਪਲਬਧ ਟੂਲਾਂ ਦੀ ਸੂਚੀ ਨਾਲ ਜਵਾਬ ਦਿੰਦਾ ਹੈ, ਹਰ ਇੱਕ ਦੇ ਵੇਰਵਿਆਂ ਅਤੇ ਪੈਰਾਮੀਟਰ ਸਕੀਮਾਂ ਸਮੇਤ। ਤੁਹਾਡਾ AI ਏਜੰਟ ਫਿਰ ਵਰਤੋਂਕਾਰ ਦੀਆਂ ਬੇਨਤੀਆਂ ਅਨੁਸਾਰ ਟੂਲਾਂ ਦੀ ਚੋਣ ਕਰ ਸਕਦਾ ਹੈ। ਹੇਠਾਂ ਇਸ ਹੈਂਡਸ਼ੇਕ ਨੂੰ ਦਿਖਾਇਆ ਗਿਆ ਹੈ — ਕਲਾਈਟ `tools/list` ਬੇਨਤੀ ਭੇਜਦਾ ਹੈ ਅਤੇ ਸਰਵਰ ਆਪਣੇ ਉਪਲਬਧ ਟੂਲਾਂ ਨੂੰ ਵੇਰਵਾ ਅਤੇ ਸਕੀਮਾਂ ਨਾਲ ਵਾਪਸ ਕਰਦਾ ਹੈ:

<img src="../../../translated_images/pa/tool-discovery.07760a8a301a7832.webp" alt="MCP Tool Discovery" width="800"/>

*AI ਮੁਲਾਕਾਤ 'ਤੇ ਉਪਲਬਧ ਟੂਲਾਂ ਦੀ ਖੋਜ ਕਰਦਾ ਹੈ — ਹੁਣ ਇਹ ਜਾਣਦਾ ਹੈ ਕਿ ਕਿਹੜੀਆਂ ਯੋਗਤਾਵਾਂ ਉਪਲਬਧ ਹਨ ਅਤੇ ਕਿਹੜੀਆਂ ਵਰਤਣੀਆਂ ਹਨ।*

**ਟਰਾਂਸਪੋਰਟ ਮਕੈਨਿਜ਼ਮ**

MCP ਵੱਖ-ਵੱਖ ਟਰਾਂਸਪੋਰਟ ਮਕੈਨਿਜ਼ਮਾਂ ਦਾ ਸਮਰਥਨ ਕਰਦਾ ਹੈ। ਦੋ ਵਿਕਲਪ ਹਨ Stdio (ਸਥਾਨਕ ਸਬਪ੍ਰੋਸੈਸ ਸੰਚਾਰ ਲਈ) ਅਤੇ Streamable HTTP (ਦੂਰੀ ਸਰਵਰਾਂ ਲਈ)। ਇਹ ਮਾਡਿਊਲ Stdio ਟਰਾਂਸਪੋਰਟ ਦਿਖਾਉਂਦਾ ਹੈ:

<img src="../../../translated_images/pa/transport-mechanisms.2791ba7ee93cf020.webp" alt="Transport Mechanisms" width="800"/>

*MCP ਟਰਾਂਸਪੋਰਟ ਮਕੈਨਿਜ਼ਮ: ਦੂਰ ਦੇ ਸਰਵਰਾਂ ਲਈ HTTP, ਸਥਾਨਕ ਪ੍ਰਕਿਰਿਆਵਾਂ ਲਈ Stdio*

**Stdio** - [StdioTransportDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java)

ਸਥਾਨਕ ਪ੍ਰਕਿਰਿਆਵਾਂ ਲਈ। ਤੁਹਾਡੀ ਐਪਲੀਕੇਸ਼ਨ ਇੱਕ ਸਬਪ੍ਰੋਸੈਸ ਵਜੋਂ ਸਰਵਰ ਨੂੰ ਸਪੌਨ ਕਰਦੀ ਹੈ ਅਤੇ ਸਟੈਂਡਰਡ ਇੰਪੁੱਟ/ਆਉਟਪੁੱਟ ਰਾਹੀਂ ਸੰਚਾਰ ਕਰਦੀ ਹੈ। ਫਾਈਲਸਿਸਟਮ ਦੀ ਪਹੁੰਚ ਜਾਂ ਕਮਾਂਡ-ਲਾਈਨ ਟੂਲਾਂ ਲਈ ਲਾਭਦਾਇਕ।

```java
McpTransport stdioTransport = new StdioMcpTransport.Builder()
    .command(List.of(
        npmCmd, "exec",
        "@modelcontextprotocol/server-filesystem@2025.12.18",
        resourcesDir
    ))
    .logEvents(false)
    .build();
```

`@modelcontextprotocol/server-filesystem` ਸਰਵਰ ਹੇਠ ਲਿਖੇ ਟੂਲ ਪ੍ਰਗਟਾਉਂਦਾ ਹੈ, ਸਭ ਤੁਹਾਡੇ ਦਿੱਤੇ ਡਾਇਰੈਕਟਰੀਆਂ ਵਿੱਚ ਸੈਂਡਬਾਕਸ ਕੀਤੇ ਗਏ:

| ਟੂਲ | ਵੇਰਵਾ |
|------|--------|
| `read_file` | ਇੱਕ ਇਕੱਲੀ ਫਾਈਲ ਦਾ ਸਮੱਗਰੀ ਪੜ੍ਹਨਾ |
| `read_multiple_files` | ਇੱਕ ਕਾਲ ਵਿੱਚ ਕਈ ਫਾਈਲਾਂ ਪੜ੍ਹਨਾ |
| `write_file` | ਇੱਕ ਫਾਈਲ ਬਣਾਉਣਾ ਜਾਂ ਓਵਰਰਾਈਟ ਕਰਨਾ |
| `edit_file` | ਲਕੜੀ ਖੋਜ ਕੇ ਬਦਲਾਅ ਕਰਨਾ |
| `list_directory` | ਕਿਸੇ ਪਾਥ ਤੇ ਫਾਈਲਾਂ ਅਤੇ ਡਾਇਰੈਕਟਰੀਆਂ ਦੀ ਸੂਚੀ ਬਣਾਉਣਾ |
| `search_files` | ਕਿਸੇ ਨਮੂਨੇ ਨਾਲ ਮੇਲ ਖਾਂਦੇ ਫਾਈਲਾਂ ਦੀ ਰਿਕਰਸੀਵ ਤਲਾਸ਼ |
| `get_file_info` | ਫਾਈਲ ਮੈਟਾਡੇਟਾ ਪ੍ਰਾਪਤ ਕਰਨਾ (ਸਾਈਜ਼, ਟਾਈਮਸਟੈਂਪ, ਅਧਿਕਾਰ) |
| `create_directory` | ਇੱਕ ਡਾਇਰੈਕਟਰੀ ਬਣਾਉਣਾ (ਮਾਪੇ ਡਾਇਰੈਕਟਰੀਆ ਸਮੇਤ) |
| `move_file` | ਇੱਕ ਫਾਈਲ ਜਾਂ ਡਾਇਰੈਕਟਰੀ ਨੂੰ ਹਿਲਾਉਣਾ ਜਾਂ ਨਾਮ ਬਦਲਣਾ |

ਹੇਠਾਂ ਦਿੱਤਾ ਚਿੱਤਰ ਦਿਖਾਉਂਦਾ ਹੈ ਕਿ Stdio ਟਰਾਂਸਪੋਰਟ ਰਨਟਾਈਮ ਵਿੱਚ ਕਿਵੇਂ ਕੰਮ ਕਰਦਾ ਹੈ — ਤੁਹਾਡੀ ਜਾਵਾ ਐਪ MCP ਸਰਵਰ ਨੂੰ ਇੱਕ ਚਾਈਲਡ ਪ੍ਰਕਿਰਿਆ ਵਜੋਂ ਸਪੌਨ ਕਰਦੀ ਹੈ ਅਤੇ ਉਹ ਸਟਡੀਨ/ਸਟਡਾਊਟ ਪਾਈਪ ਰਾਹੀਂ ਗੱਲਬਾਤ ਕਰਦੇ ਹਨ, ਕਿਸੇ ਨੈੱਟਵਰਕ ਜਾਂ HTTP ਦੀ ਲੋੜ ਨਹੀਂ:

<img src="../../../translated_images/pa/stdio-transport-flow.45eaff4af2d81db4.webp" alt="Stdio Transport Flow" width="800"/>

*Stdio ਟਰਾਂਸਪੋਰਟ ਕਾਰਵਾਈ ਵਿੱਚ — ਤੁਹਾਡੀ ਐਪ MCP ਸਰਵਰ ਨੂੰ ਚਾਈਲਡ ਪ੍ਰਕਿਰਿਆ ਵਜੋਂ ਸਪੌਨ ਕਰਦੀ ਹੈ ਅਤੇ ਸਟਡੀਨ/ਸਟਡਾਊਟ ਪਾਈਪਾਂ ਰਾਹੀਂ ਸੰਚਾਰ ਕਰਦੀ ਹੈ।*

> - "LangChain4j spawned MCP ਸਰਵਰ ਪ੍ਰਕਿਰਿਆਵਾਂ ਦਾ ਜੀਵਨ ਚੱਕਰ ਕਿਵੇਂ ਸੰਭਾਲਦਾ ਹੈ?"

## ਏਜੰਟਿਕ ਮਾਡਿਊਲ

ਜਦੋਂ ਕਿ MCP ਮਿਆਰੀਕ੍ਰਿਤ ਟੂਲ ਪ੍ਰਦਾਨ ਕਰਦਾ ਹੈ, LangChain4j ਦਾ **ਏਜੰਟਿਕ ਮਾਡਿਊਲ** ਉਹ ਏਜੰਟ ਬਣਾਉਣ ਲਈ ਘੋਸ਼ਣਾਤਮਕ ਤਰੀਕਾ ਦਿੰਦਾ ਹੈ ਜੋ ਉਹਨਾਂ ਟੂਲਾਂ ਨੂੰ ਸਮਰਥਿਤ ਕਰਦਾ ਹੈ। `@Agent` ਐਨੋਟੇਸ਼ਨ ਅਤੇ `AgenticServices` ਤੁਹਾਨੂੰ ਇੰਟਰਫੇਸਾਂ ਰਾਹੀਂ ਏਜੰਟ ਦਾ ਵਿਹਾਰ ਪਰਿਭਾਸ਼ਿਤ ਕਰਨ ਦਿੰਦੇ ਹਨ, ਨਿਰੀਕਸ਼ਨਾਤਮਕ ਕੋਡ ਦੀ ਜਗ੍ਹਾ।

ਇਸ ਮਾਡਿਊਲ ਵਿੱਚ, ਤੁਸੀਂ **ਸੂਪਰਵਾਇਜ਼ਰ ਏਜੰਟ** ਪੈਟਰਨ ਦੀ ਖੋਜ ਕਰੋਗੇ — ਇੱਕ ਉन्नਤ ਏਜੰਟਿਕ AI ਦ੍ਰਿਸ਼ਟੀਕੋਣ ਜਿੱਥੇ ਇੱਕ "ਸੂਪਰਵਾਇਜ਼ਰ" ਏਜੰਟ ਵਰਤੋਂਕਾਰ ਦੀਆਂ ਬੇਨਤੀਆਂ ਅਨੁਸਾਰ ਡਾਇਨਾਮਿਕ ਤਰੀਕੇ ਨਾਲ ਲੋੜੀਂਦੇ ਉਪ-ਏਜੰਟਾਂ ਦਾ ਚੋਣ ਕਰਦਾ ਹੈ। ਅਸੀਂ ਦੋਵਾਂ ਵਿਚਾਰਾਂ ਨੂੰ ਮਿਲਾਉਂਦੇ ਹਾਂ ਜਿਸ ਵਿੱਚ ਸਾਡੇ ਇੱਕ ਉਪ-ਏਜੰਟ ਨੂੰ MCP ਤਾਕਤਵਰ ਫਾਈਲ ਪਹੁੰਚ ਯੋਗਤਾਵਾਂ ਮਿਲਦੀਆਂ ਹਨ।

ਏਜੰਟਿਕ ਮਾਡਿਊਲ ਵਰਤਣ ਲਈ, ਇਹ Maven ਡੀਪੈਂਡੇੰਸੀ ਸ਼ਾਮਲ ਕਰੋ:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```
> **ਨੋਟ:** `langchain4j-agentic` ਮਾਡਿਊਲ ਇੱਕ ਵੱਖਰੀ ਸੰਗ੍ਰਹਿਤ ਸੰਸਕਰਣ ਪ੍ਰਾਪਰਟੀ (`langchain4j.mcp.version`) ਵਰਤਦਾ ਹੈ ਕਿਉਂਕਿ ਇਹ ਮੁੱਖ LangChain4j ਲਾਇਬ੍ਰੇਰੀਆਂ ਤੋਂ ਵੱਖਰੇ ਸਮੇਂ 'ਤੇ ਰਿਲੀਜ਼ ਹੁੰਦਾ ਹੈ।

> **⚠️ ਪ੍ਰਯੋਗਾਤਮਕ:** `langchain4j-agentic` ਮਾਡਿਊਲ **ਪ੍ਰਯੋਗਾਤਮਕ** ਹੈ ਅਤੇ ਇਸ ਵਿੱਚ ਬਦਲਾਅ ਹੋ ਸਕਦੇ ਹਨ। AI ਸਹਾਇਕ ਬਣਾਉਣ ਦਾ ਥੋਸ ਤਰੀਕਾ ਪੂਰਾ ਰਹਿੰਦਾ ਹੈ `langchain4j-core` ਅਤੇ ਕਸਟਮ ਟੂਲਾਂ ਨਾਲ (ਮਾਡਿਊਲ 04)।

### ਸੂਪਰਵਾਇਜ਼ਰ ਏਜੰਟ

**ਸੂਪਰਵਾਇਜ਼ਰ ਏਜੰਟ ਪੈਟਰਨ** ਇੱਕ **ਲਚਕੀਲਾ** ਏਜੰਟਿਕ AI ਰੂਪ ਹੈ। ਇੱਕ ਸੂਪਰਵਾਇਜ਼ਰ ਇੱਕ LLM ਨੂੰ ਵਰਤਦਾ ਹੈ ਜੋ ਵਰਤੋਂਕਾਰ ਦੀ ਬੇਨਤੀ ਦੇ ਆਧਾਰ ਤੇ ਸੁਤੰਤਰ ਤੌਰ 'ਤੇ ਫੈਸਲਾ ਕਰਦਾ ਹੈ ਕਿ ਕਿਹੜੇ ਏਜੰਟਾਂ ਨੂੰ ਕਾਲ ਕਰਨਾ ਹੈ। ਅਗਲੇ ਉਦਾਹਰਣ ਵਿੱਚ, ਅਸੀਂ MCP-ਸੰਚਾਲਿਤ ਫਾਈਲ ਪਹੁੰਚ ਨੂੰ ਇੱਕ LLM ਏਜੰਟ ਨਾਲ ਮਿਲਾ ਕੇ ਇੱਕ ਸੂਪਰਵਾਈਜ਼ਡ ਫਾਈਲ ਰੀਡ → ਰਿਪੋਰਟ ਵਰਕਫਲੋ ਬਣਾਉਂਦੇ ਹਾਂ।

ਡੈਮੋ ਵਿੱਚ, `FileAgent` MCP ਫਾਈਲਸਿਸਟਮ ਟੂਲਾਂ ਦੀ ਵਰਤੋਂ ਕਰਕੇ ਫਾਈਲ ਪੜ੍ਹਦਾ ਹੈ, ਅਤੇ `ReportAgent` ਇੱਕ ਢਾਂਚাগত ਰਿਪੋਰਟ ਬਣਾਉਂਦਾ ਹੈ ਜਿਸ ਵਿੱਚ ਇੱਕ ਕਾਰਜਨਿਰਦੇਸ਼ ਜਮ੍ਹਾਂ (1 ਵਾਕ), 3 ਮੁੱਖ ਬਿੰਦੂ ਅਤੇ ਸੁਝਾਅ ਹਨ। ਸੂਪਰਵਾਇਜ਼ਰ ਇਸ ਸਰੋਤ ਨੂੰ ਆਪਣੇ ਆਪ ਸੰਚਾਲਿਤ ਕਰਦਾ ਹੈ:

<img src="../../../translated_images/pa/supervisor-agent-pattern.06275a41ae006ac8.webp" alt="Supervisor Agent Pattern" width="800"/>

*ਸੂਪਰਵਾਇਜ਼ਰ ਆਪਣੇ LLM ਨਾਲ ਫੈਸਲਾ ਕਰਦਾ ਹੈ ਕਿ ਕਿਹੜੇ ਏਜੰਟਾਂ ਨੂੰ ਕਾਲ ਕਰਨਾ ਹੈ ਅਤੇ ਕਿਹੜੇ ਕ੍ਰਮ ਵਿੱਚ — ਕੋਈ ਮੁਸ਼ਕਲ ਰੂਟਿੰਗ ਲੋੜੀਂਦੀ ਨਹੀਂ।*

ਸਾਡੇ ਫਾਈਲ-ਤੋਂ-ਰਿਪੋਰਟ ਪਾਈਪਲਾਈਨ ਲਈ ਵਿਸਥਾਰਤ ਵਰਕਫਲੋ ਇਹੋ ਜਿਹਾ ਹੈ:

<img src="../../../translated_images/pa/file-report-workflow.649bb7a896800de9.webp" alt="File to Report Workflow" width="800"/>

*FileAgent MCP ਟੂਲਾਂ ਰਾਹੀਂ ਫਾਈਲ ਪੜ੍ਹਦਾ ਹੈ, ਫਿਰ ReportAgent ਅਸਲ ਸਮੱਗਰੀ ਨੂੰ ਢਾਂਚਾਗਤ ਰਿਪੋਰਟ ਵਿੱਚ ਬਦਲਦਾ ਹੈ।*

ਹੇਠਾਂ ਦਿੱਤੀਆਂ ਕ੍ਰਮਬੱਧ ਤਸਵੀਰਾਂ ਪੂਰੇ ਸੂਪਰਵਾਇਜ਼ਰ ਸੰਚਾਰ ਦਾ ਪਤਾ ਲਗਾਉਂਦੀਆਂ ਹਨ — MCP ਸਰਵਰ ਸਪੌਨ ਕਰਨਾ, ਸੂਪਰਵਾਇਜ਼ਰ ਦਾ ਸੁਤੰਤਰ ਏਜੰਟ ਚੋਣ, stdio ਰਾਹੀਂ ਟੂਲ ਕਾਲਾਂ ਅਤੇ ਅੰਤਿਮ ਰਿਪੋਰਟ:

<img src="../../../translated_images/pa/supervisor-agent-sequence.1aa389b3bef99956.webp" alt="Supervisor Agent Sequence Diagram" width="800"/>

*ਸੂਪਰਵਾਇਜ਼ਰ FileAgent ਨੂੰ ਸੁਤੰਤਰ ਤੌਰ 'ਤੇ ਕਾਲ ਕਰਦਾ ਹੈ (ਜੋ MCP ਸਰਵਰ ਨੂੰ stdio ਰਾਹੀਂ ਫਾਈਲ ਪੜ੍ਹਨ ਲਈ ਕਾਲ ਕਰਦਾ ਹੈ), ਫਿਰ ReportAgent ਨੂੰ ਢਾਂਚাগত ਰਿਪੋਰਟ ਬਣਾਉਣ ਲਈ ਕਾਲ ਕਰਦਾ ਹੈ — ਹਰ ਏਜੰਟ ਆਪਣੇ ਨਤੀਜੇ ਨੂੰ ਸਾਂਝੇ Agentic Scope ਵਿੱਚ ਸਟੋਰ ਕਰਦਾ ਹੈ।*

ਹਰ ਏਜੰਟ ਆਪਣਾ ਨਤੀਜਾ **Agentic Scope** (ਸਾਂਝੀ ਯਾਦਦਾਸ਼ਤ) ਵਿੱਚ ਸਟੋਰ ਕਰਦਾ ਹੈ, ਜੋ ਹੇਠਲੇ ਏਜੰਟਾਂ ਨੂੰ ਪਿਛਲੇ ਨਤੀਜਿਆਂ ਤੱਕ ਪਹੁੰਚਣ ਦਿੰਦਾ ਹੈ। ਇਹ ਦਰਸਾਉਂਦਾ ਹੈ ਕਿ MCP ਟੂਲਾਂ ਕਿਵੇਂ ਨਿਰੰਤਰ ਏਜੰਟਿਕ ਵਰਕਫਲੋਜ਼ ਵਿੱਚ ਸ਼ਾਮਲ ਹੋ ਜਾਂਦੇ ਹਨ — ਸੂਪਰਵਾਇਜ਼ਰ ਨੂੰ ਇਹ ਜਾਣਨ ਦੀ ਲੋੜ ਨਹੀਂ ਕਿ ਫਾਈਲਾਂ ਕਿਵੇਂ ਪੜ੍ਹੀਆਂ ਜਾਂਦੀਆਂ ਹਨ, ਸਿਰਫ਼ ਇਹ ਕਿ `FileAgent` ਇਹ ਕਰ ਸਕਦਾ ਹੈ।

#### ਡੈਮੋ ਚਲਾਉਣਾ

ਸਕ੍ਰਿਪਟ ਆਪਣੇ ਆਪ ਸਾਰੀਆਂ ਵਾਤਾਵਰਨ ਬਦਲੀਆਂ ਰੂਟ `.env` ਫਾਈਲ ਤੋਂ ਲੋਡ ਕਰਦੇ ਹਨ:

**Bash:**
```bash
cd 05-mcp
chmod +x start-supervisor.sh
```

#### ਸੂਪਰਵਾਇਜ਼ਰ ਕਿਵੇਂ ਕੰਮ ਕਰਦਾ ਹੈ

ਏਜੰਟ ਬਣਾਉਣ ਤੋਂ ਪਹਿਲਾਂ, ਤੁਹਾਨੂੰ MCP ਟਰਾਂਸਪੋਰਟ ਨੂੰ ਇੱਕ ਕਲਾਈਟ ਨਾਲ ਜੋੜਨਾ ਅਤੇ ਉਸਨੂੰ `ToolProvider` ਵਜੋਂ ਲਪੇਟਣਾ ਪੈਂਦਾ ਹੈ। ਇਸ ਤਰ੍ਹਾਂ MCP ਸਰਵਰ ਦੇ ਟੂਲ ਤੁਹਾਡੇ ਏਜੰਟਾਂ ਲਈ ਉਪਲਬਧ ਹੋ ਜਾਂਦੇ ਹਨ:

```java
// ਟਰਾਂਸਪੋਰਟ ਤੋਂ ਇੱਕ MCP ਕਲਾਇੰਟ ਬਣਾਓ
McpClient mcpClient = new DefaultMcpClient.Builder()
        .transport(stdioTransport)
        .build();

// ਕਲਾਇੰਟ ਨੂੰ ਇੱਕ ToolProvider ਵਜੋਂ ਲਪੇਟੋ — ਇਹ MCP ਟੂਲਾਂ ਨੂੰ LangChain4j ਵਿੱਚ ਜੋੜਦਾ ਹੈ
ToolProvider mcpToolProvider = McpToolProvider.builder()
        .mcpClients(List.of(mcpClient))
        .build();
```

ਹੁਣ ਤੁਸੀਂ `mcpToolProvider` ਨੂੰ ਕਿਸੇ ਵੀ ਏਜੰਟ ਵਿੱਚ inject ਕਰ ਸਕਦੇ ਹੋ ਜਿਸਨੂੰ MCP ਟੂਲਾਂ ਦੀ ਲੋੜ ਹੈ:

```java
// ਕਦਮ 1: FileAgent MCP ਟੂਲਾਂ ਦੀ ਵਰਤੋਂ ਕਰਕੇ ਫਾਈਲਾਂ ਨੂੰ ਪੜ੍ਹਦਾ ਹੈ
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // ਫਾਈਲ ਕਾਰਵਾਈ ਲਈ MCP ਟੂਲ ਹਨ
        .build();

// ਕਦਮ 2: ReportAgent ਸੰਰਚਿਤ ਰਿਪੋਰਟਾਂ ਬਣਾਉਂਦਾ ਹੈ
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// Supervisor ਫਾਈਲ → ਰਿਪੋਰਟ ਵਰਕਫਲੋ ਨੂੰ ਸੰਗਠਿਤ ਕਰਦਾ ਹੈ
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // ਅੰਤਿਮ ਰਿਪੋਰਟ ਵਾਪਸ ਕਰੋ
        .build();

// Supervisor ਬੇਨਤੀ ਦੇ ਆਧਾਰ 'ਤੇ ਕਿਹੜੇ ਏਜੰਟਾਂ ਨੂੰ ਸੱਦਣਾ ਹੈ ਫੈਸਲਾ ਕਰਦਾ ਹੈ
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```

#### FileAgent MCP ਟੂਲ-runtime 'ਚ ਕਿਵੇਂ ਖੋਜਦਾ ਹੈ

ਤੁਸੀਂ ਸੋਚ ਰਹੇ ਹੋਵੋਗੇ: **FileAgent ਨੂੰ ਇਹ ਕਿਵੇਂ ਪਤਾ ਲੱਗਦਾ ਹੈ ਕਿ npm ਫਾਈਲਸਿਸਟਮ ਟੂਲਾਂ ਨੂੰ ਕਿਵੇਂ ਵਰਤਣਾ ਹੈ?** ਜਵਾਬ ਹੈ ਕਿ ਇਹ ਨਹੀਂ ਜਾਣਦਾ — **LLM** ਟੂਲ ਸਕੀਮਾਂ ਰਾਹੀਂ runtime 'ਚ ਇਸਨੂੰ ਸਮਝਦਾ ਹੈ।

`FileAgent` ਇੰਟਰਫੇਸ ਸਿਰਫ਼ ਇੱਕ **ਪ੍ਰਾਂਪਟ ਪਰਿਭਾਸ਼ਾ** ਹੈ। ਇਹਦੇ ਕੋਲ `read_file`, `list_directory`, ਜਾਂ ਹੋਰ ਕੋਈ MCP ਟੂਲ ਦੀ ਕੋਈ ਹਾਰਡਕੋਡਿਡ ਜਾਣਕਾਰੀ ਨਹੀਂ ਹੁੰਦੀ। ਇਹ ਹਨ ਜੋ ਪੂਰੀ ਪ੍ਰਕਿਰਿਆ ਵਿੱਚ ਹੁੰਦਾ ਹੈ:
1. **ਸਰਵਰ ਸ਼ੁਰੂ ਹੋਣਾ:** `StdioMcpTransport` `@modelcontextprotocol/server-filesystem` npm ਪੈਕੇਜ ਨੂੰ ਇੱਕ ਚਾਈਲਡ ਪ੍ਰੋਸੈਸ ਵਜੋਂ ਸ਼ੁਰੂ ਕਰਦਾ ਹੈ  
2. **ਟੂਲ ਖੋਜ:** `McpClient` ਸਰਵਰ ਨੂੰ `tools/list` JSON-RPC ਬੇਨਤੀ ਭੇਜਦਾ ਹੈ, ਜੋ ਟੂਲਾਂ ਦੇ ਨਾਮ, ਵਰਣਨ ਅਤੇ ਪੈਰਾਮੀਟਰ ਸਕੀਮਾ ਜਿਵੇਂ ਕਿ (ਉਦਾਹਰਨ ਲਈ, `read_file` — *"ਇੱਕ ਫਾਇਲ ਦਾ ਪੂਰਾ ਸਮੱਗਰੀ ਪੜ੍ਹੋ"* — `{ path: string }`) ਦਾ ਜਵਾਬ ਦਿੰਦਾ ਹੈ  
3. **ਸਕੀਮਾ ਇੰਜੈਕਸ਼ਨ:** `McpToolProvider` ਇਹ ਖੋਜੇ ਹੋਏ ਸਕੀਮਾ ਲੈ ਕੇ ਉਹਨਾਂ ਨੂੰ LangChain4j ਨੂੰ ਉਪਲੱਬਧ ਕਰਵਾਉਂਦਾ ਹੈ  
4. **LLM ਫ਼ੈਸਲਾ ਕਰਦਾ ਹੈ:** ਜਦੋਂ `FileAgent.readFile(path)` ਕਾਲ ਹੁੰਦਾ ਹੈ, LangChain4j ਸਿਸਟਮ ਸੁਨੇਹਾ, ਯੂਜ਼ਰ ਸੁਨੇਹਾ, **ਅਤੇ ਟੂਲ ਸਕੀਮਾ ਦੀ ਸੂਚੀ** LLM ਨੂੰ ਭੇਜਦਾ ਹੈ। LLM ਟੂਲ ਦੇ ਵਰਣਨਾਂ ਨੂੰ ਪੜ੍ਹਦਾ ਹੈ ਅਤੇ ਟੂਲ ਕਾਲ ਬਣਾਉਂਦਾ ਹੈ (ਉਦਾਹਰਨ ਲਈ, `read_file(path="/some/file.txt")`)  
5. **ਕਾਰਵਾਈ:** LangChain4j ਟੂਲ ਕਾਲ ਨੂੰ ਰੋਕਦਾ ਹੈ, ਇਸਨੂੰ MCP ਕਲਾਇੰਟ ਰਾਹੀਂ Node.js ਸਬ-ਪ੍ਰੋਸੈਸ ਨੂੰ ਰੀਚ ਕਰਦਾ ਹੈ, ਨਤੀਜਾ ਪ੍ਰਾਪਤ ਕਰਦਾ ਹੈ ਅਤੇ ਇਸਨੂੰ ਵਾਪਸ LLM ਨੂੰ ਫੀਡ ਕਰਦਾ ਹੈ  

ਇਹ ਵੋਹੀ [ਟੂਲ ਖੋਜ](../../../05-mcp) ਤਰੀਕਾ ਹੈ ਜੋ ਉਪਰ ਵਰਣਿਤ ਹੈ, ਪਰ ਖਾਸ ਕਰਕੇ ਏਜੰਟ ਵਰਕਫਲੋ ਲਈ ਲਾਗੂ ਕੀਤਾ ਗਿਆ ਹੈ। `@SystemMessage` ਅਤੇ `@UserMessage` ਐਨੋਟੇਸ਼ਨਾਂ LLM ਦੇ ਵਿਹਾਰ ਨੂੰ ਗਾਈਡ ਕਰਦੀਆਂ ਹਨ, ਜਦਕਿ ਇੰਜੈਕਟ ਕੀਤੇ ਹੋਏ `ToolProvider` ਉਸਨੂੰ **ਸਮਰੱਥਾ** ਦਿੰਦਾ ਹੈ — LLM ਦੋਹਾਂ ਨੂੰ ਰਨਟਾਈਮ ਵਿੱਚ ਜੋੜਦਾ ਹੈ।

#### ਨਤੀਜੇ ਨੂੰ ਸਮਝਣਾ  

ਜਦ ਤੁਸੀਂ ਡੈਮੋ ਚਲਾਉਂਦੇ ਹੋ, ਤਾਂ ਤੁਹਾਨੂੰ ਦੇਖਣ ਨੂੰ ਮਿਲੇਗਾ ਕਿ ਸੁਪਰਵਾਈਜ਼ਰ ਕਿਵੇਂ ਕਈ ਏਜੰਟਾਂ ਦੀ ਸੰਯੋਜਨਾ ਕਰ ਰਿਹਾ ਹੈ। ਹੇਠਾਂ ਦਿੱਤੇ ਭਾਗਾਂ ਵਿੱਚ ਇਹ ਲਿਖਿਆ ਹੈ ਕਿ ਹਰ ਹਿੱਸਾ ਕੀ ਦਰਸਾਉਂਦਾ ਹੈ:  

```
======================================================================
  FILE → REPORT WORKFLOW DEMO
======================================================================

This demo shows a clear 2-step workflow: read a file, then generate a report.
The Supervisor orchestrates the agents automatically based on the request.
```
  
**ਸਿਰਲੇਖ** ਵਰਕਫਲੋ ਸੰਕਲਪ ਦਾ ਪਰਿਚਯ ਕਰਾਉਂਦਾ ਹੈ: ਫਾਇਲ ਪੜ੍ਹਨ ਤੋਂ ਰਿਪੋਰਟ ਤਿਆਰ ਕਰਨ ਤੱਕ ਕੇਂਦਰਿਤ ਪਾਈਪਲਾਈਨ।  

```
--- WORKFLOW ---------------------------------------------------------
  ┌─────────────┐      ┌──────────────┐
  │  FileAgent  │ ───▶ │ ReportAgent  │
  │ (MCP tools) │      │  (pure LLM)  │
  └─────────────┘      └──────────────┘
   outputKey:           outputKey:
   'fileContent'        'report'

--- AVAILABLE AGENTS -------------------------------------------------
  [FILE]   FileAgent   - Reads files via MCP → stores in 'fileContent'
  [REPORT] ReportAgent - Generates structured report → stores in 'report'
```
  
**ਵਰਕਫਲੋ ਡਾਇਗ੍ਰਾਮ** ਏਜੰਟਾਂ ਵਿਚਕਾਰ ਡੇਟਾ ਦਾ ਪ੍ਰਭਾਹ ਦਿਖਾਉਂਦਾ ਹੈ। ਹਰ ਏਜੰਟ ਦਾ ਖਾਸ ਭੂਮਿਕਾ ਹੈ:  
- **FileAgent** MCP ਟੂਲਾਂ ਦੀ ਵਰਤੋਂ ਕਰਕੇ ਫਾਇਲਾਂ ਪੜ੍ਹਦਾ ਹੈ ਅਤੇ ਕੱਚਾ ਸਮੱਗਰੀ `fileContent` ਵਿੱਚ ਸਟੋਰ ਕਰਦਾ ਹੈ  
- **ReportAgent** ਉਹ ਸਮੱਗਰੀ ਲੈਂਦਾ ਹੈ ਅਤੇ ਢਾਂਚਾਗਤ ਰਿਪੋਰਟ `report` ਪੈਦਾ ਕਰਦਾ ਹੈ  

```
--- USER REQUEST -----------------------------------------------------
  "Read the file at .../file.txt and generate a report on its contents"
```
  
**ਯੂਜ਼ਰ ਬੇਨਤੀ** ਕਾਰਜ ਦਿਖਾਉਂਦੀ ਹੈ। ਸੁਪਰਵਾਈਜ਼ਰ ਇਸਨੂੰ ਵਿਸ਼ਲੇਸ਼ਣ ਕਰਦਾ ਹੈ ਅਤੇ FileAgent → ReportAgent ਕਾਲ ਕਰਨ ਦਾ ਫੈਸਲਾ ਕਰਦਾ ਹੈ।  

```
--- SUPERVISOR ORCHESTRATION -----------------------------------------
  The Supervisor decides which agents to invoke and passes data between them...

  +-- STEP 1: Supervisor chose -> FileAgent (reading file via MCP)
  |
  |   Input: .../file.txt
  |
  |   Result: LangChain4j is an open-source, provider-agnostic Java framework for building LLM...
  +-- [OK] FileAgent (reading file via MCP) completed

  +-- STEP 2: Supervisor chose -> ReportAgent (generating structured report)
  |
  |   Input: LangChain4j is an open-source, provider-agnostic Java framew...
  |
  |   Result: Executive Summary...
  +-- [OK] ReportAgent (generating structured report) completed
```
  
**ਸੁਪਰਵਾਈਜ਼ਰ ਸੰਯੋਜਨਾ** 2-ਕਦਮੀ ਪ੍ਰਕਿਰਿਆ ਦਿਖਾਉਂਦਾ ਹੈ:  
1. **FileAgent** MCP ਰਾਹੀਂ ਫਾਇਲ ਪੜ੍ਹਦਾ ਹੈ ਅਤੇ ਸਮੱਗਰੀ ਸਟੋਰ ਕਰਦਾ ਹੈ  
2. **ReportAgent** ਸਮੱਗਰੀ ਪ੍ਰਾਪਤ ਕਰਦਾ ਹੈ ਅਤੇ ਢਾਂਚਾਗਤ ਰਿਪੋਰਟ ਤਿਆਰ ਕਰਦਾ ਹੈ  

ਸੁਪਰਵਾਈਜ਼ਰ ਨੇ ਯੂਜ਼ਰ ਦੀ ਬੇਨਤੀ ਦੇ ਅਧਾਰ 'ਤੇ ਇਹ ਫੈਸਲੇ **ਆਤਮਨਿਰਭਰਤਾ ਨਾਲ** ਲਏ।  

```
--- FINAL RESPONSE ---------------------------------------------------
Executive Summary
...

Key Points
...

Recommendations
...

--- AGENTIC SCOPE (Data Flow) ----------------------------------------
  Each agent stores its output for downstream agents to consume:
  * fileContent: LangChain4j is an open-source, provider-agnostic Java framework...
  * report: Executive Summary...
```
  
#### ਏਜੰਟਿਕ ਮੋਡੀਊਲ ਫੀਚਰਾਂ ਦੀ ਵਿਆਖਿਆ  

ਉਦਾਹਰਨ ਵਿੱਚ ਏਜੰਟਿਕ ਮੋਡੀਊਲ ਦੀ ਕੁਝ ਉੱਨਤ ਫੀਚਰਾਂ ਨੂੰ ਦਰਸਾਇਆ ਗਿਆ ਹੈ। ਆਓ ਏਜੰਟਿਕ ਸਕੋਪ ਅਤੇ ਏਜੰਟ ਲਿਸਨਰਾਂ ‘ਤੇ ਨਜ਼ਰ ਮਾਰਦੇ ਹਾਂ।  

**ਏਜੰਟਿਕ ਸਕੋਪ** ਸਾਂਝਾ ਮੈਮੋਰੀ ਦਿਖਾਉਂਦਾ ਹੈ ਜਿੱਥੇ ਏਜੰਟਾਂ ਨੇ ਆਪਣੇ ਨਤੀਜੇ `@Agent(outputKey="...")` ਦੀ ਵਰਤੋਂ ਕਰਕੇ ਸਟੋਰ ਕੀਤੇ ਹਨ। ਇਹ ਸੁਵਿਧਾ ਦਿੰਦਾ ਹੈ:  
- ਬਾਅਦ ਦੇ ਏਜੰਟ ਪਹਿਲੇ ਏਜੰਟਾਂ ਦੇ ਨਤੀਜੇ ਪ੍ਰਾਪਤ ਕਰ ਸਕਦੇ ਹਨ  
- ਸੁਪਰਵਾਈਜ਼ਰ ਅਖੀਰਲਾ ਜਵਾਬ ਸੰਸ਼ਲੇਸ਼ਿਤ ਕਰ ਸਕਦਾ ਹੈ  
- ਤੁਸੀਂ ਵੇਖ ਸਕਦੇ ਹੋ ਕਿ ਹਰ ਏਜੰਟ ਨੇ ਕੀ ਪੈਦਾ ਕੀਤਾ ਹੈ  

ਹੇਠਾਂ ਡਾਇਗ੍ਰਾਮ ਦਿਖਾਉਂਦਾ ਹੈ ਕਿਵੇਂ Agentic Scope ਸਾਂਝਾ ਮੈਮੋਰੀ ਵਜੋਂ ਕੰਮ ਕਰਦਾ ਹੈ ਫਾਇਲ-ਤੋਂ-ਰਿਪੋਰਟ ਵਰਕਫਲੋ ਵਿੱਚ — FileAgent ਆਪਣਾ ਆਉਟਪੁੱਟ `fileContent` ਕੀ ਤੇ ਲਿਖਦਾ ਹੈ, ReportAgent ਇਸਨੂੰ ਪੜ੍ਹਦਾ ਹੈ ਅਤੇ ਆਪਣਾ ਨਤੀਜਾ `report` ਕੀ ਹੇਠਾਂ ਲਿਖਦਾ ਹੈ:  

<img src="../../../translated_images/pa/agentic-scope.95ef488b6c1d02ef.webp" alt="Agentic Scope Shared Memory" width="800"/>  

*Agentic Scope ਸਾਂਝੀ ਮੈਮੋਰੀ ਵਜੋਂ ਕੰਮ ਕਰਦਾ ਹੈ — FileAgent `fileContent` ਲਿਖਦਾ ਹੈ, ReportAgent ਪੜ੍ਹਦਾ ਹੈ ਅਤੇ `report` ਲਿਖਦਾ ਹੈ, ਅਤੇ ਤੁਹਾਡੇ ਕੋਡ ਅਖੀਰਲਾ ਨਤੀਜਾ ਪੜ੍ਹਦਾ ਹੈ।*  

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // ਫਾਇਲਏਜੰਟ ਤੋਂ ਕੱਚਾ ਫਾਇਲ ਡੇਟਾ
String report = scope.readState("report");            // ਰਿਪੋਰਟਏਜੰਟ ਤੋਂ ਬਣਿਆ ਹੋਇਆ ਰਿਪੋਰਟ
```
  
**ਏਜੰਟ ਲਿਸਨਰ** ਏਜੰਟ ਦੇ ਕਾਰਜਨਵਾਈ ਦੀ ਨਿਗਰਾਨੀ ਅਤੇ ਡਿਗਗਿੰਗ ਲਈ ਸਹਾਇਤ ਕਰਦੇ ਹਨ। ਡੈਮੋ ਵਿੱਚ ਦਿੱਤੀ ਗਈ ਕਦਮ-ਦਰ-কਦਮ ਆਉਟਪੁੱਟ ਇੱਕ AgentListener ਤੋਂ ਆ ਰਹੀ ਹੈ ਜੋ ਹਰ ਏਜੰਟ ਕਾਲ ਵਿੱਚ ਹਿੱਸਾ ਲੈਂਦਾ ਹੈ:  
- **beforeAgentInvocation** - ਜਦੋਂ ਸੁਪਰਵਾਈਜ਼ਰ ਕਿਸੇ ਏਜੰਟ ਨੂੰ ਚੁਣਦਾ ਹੈ, ਇਹ ਕਾਲ ਹੁੰਦਾ ਹੈ, ਤੁਹਾਨੂੰ ਦਿਖਾਉਂਦਾ ਹੈ ਕਿ ਕਿਹੜਾ ਏਜੰਟ ਚੁਣਿਆ ਗਿਆ ਅਤੇ ਕਿਉਂ  
- **afterAgentInvocation** - ਜਦੋਂ ਏਜੰਟ ਮੁਕੰਮਲ ਹੁੰਦਾ ਹੈ, ਇਸਦਾ ਨਤੀਜਾ ਦਿਖਾਉਂਦਾ ਹੈ  
- **inheritedBySubagents** - ਜੇ ਸੱਚ ਹੈ, ਤਾਂ ਇਹ ਲਿਸਨਰ ਸਾਰੇ ਹੇਠਲੇ ਏਜੰਟਾਂ ਤੇ ਨਿਗਰਾਨੀ ਕਰਦਾ ਹੈ  

ਹੇਠਾਂ ਡਾਇਗ੍ਰਾਮ ਪੂਰੇ Agent Listener ਜੀਵਨਚੱਕਰ ਨੂੰ ਵਿਖਾਉਂਦਾ ਹੈ, ਜਿਸ ਵਿੱਚ `onError` ਵੀ ਹੈ ਜੋ ਏਜੰਟ ਕਾਰਜਨਵਾਈ ਦੌਰਾਨ ਆਏ ਫੇਲਿਆਂ ਨੂੰ ਹੈਂਡਲ ਕਰਦਾ ਹੈ:  

<img src="../../../translated_images/pa/agent-listeners.784bfc403c80ea13.webp" alt="Agent Listeners Lifecycle" width="800"/>  

*ਏਜੰਟ ਲਿਸਨਰ ਕਾਰਜਨਵਾਈ ਦੇ ਜੀਵਨਚੱਕਰ ਵਿੱਚ ਜੁੜਦੇ ਹਨ — ਜਦੋਂ ਏਜੰਟ ਸ਼ੁਰੂ ਹੁੰਦਾ ਹੈ, ਮੁਕੰਮਲ ਹੁੰਦਾ ਹੈ, ਜਾਂ ਗਲਤੀ ਆਉਂਦੀ ਹੈ, ਇਸਦੀ ਨਿਗਰਾਨੀ ਕਰਦੇ ਹਨ।*  

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // ਸਾਰੇ ਸਹਾਇਕ ਏਜੰਟਾਂ ਤੱਕ ਫੈਲਾਓ
    }
};
```
  
ਸੁਪਰਵਾਈਜ਼ਰ ਪੈਟਰਨ ਤੋਂ ਇਲਾਵਾ, `langchain4j-agentic` ਮੋਡੀਊਲ ਕਈ ਸ਼ਕਤਿਸ਼ਾਲੀ ਵਰਕਫਲੋ ਪੈਟਰਨ ਪ੍ਰਦਾਨ ਕਰਦਾ ਹੈ। ਹੇਠਾਂ ਡਾਇਗ੍ਰਾਮ ਪੰਜੋਂ ਪੈਟਰਨ ਦਿਖਾਉਂਦਾ ਹੈ — ਸਧਾਰਨ ਲੜੀਵਾਰ ਪਾਈਪਲਾਈਨਾਂ ਤੋਂ ਲੈ ਕੇ ਮਨੁੱਖੀ-ਜਾਂਚ ਪ੍ਰਵਾਹ ਤੱਕ:  

<img src="../../../translated_images/pa/workflow-patterns.82b2cc5b0c5edb22.webp" alt="Agent Workflow Patterns" width="800"/>  

*ਏਜੰਟ ਸੰਚਾਲਨ ਲਈ ਪੰਜ ਵਰਕਫਲੋ ਪੈਟਰਨ — ਸਧਾਰਨ ਲੜੀਵਾਰ, ਸਮਾਂਨਤ ਪੈਰਲਲ, ਲੂਪ, ਸ਼ਰਤ ਬੱਧ, ਅਤੇ ਮਨੁੱਖੀ-ਜਾਂਚ ਪ੍ਰਵਾਹ।*  

| ਪੈਟਰਨ | ਵਰਣਨ | ਵਰਤੋਂ ਦਾ ਮਾਮਲਾ |  
|---------|-------------|----------|  
| **Sequential** | ਏਜੰਟਾਂ ਨੂੰ ਕ੍ਰਮ ਵਿੱਚ ਚਲਾਓ, ਆਉਟਪੁੱਟ ਅਗਲੇ ਨੂੰ ਦਿਓ | ਪਾਈਪਲਾਈਨ: ਖੋਜ → ਵਿਸ਼ਲੇਸ਼ਣ → ਰਿਪੋਰਟ |  
| **Parallel** | ਏਜੰਟਾਂ ਨੂੰ ਇੱਕਸਾਰ੍ਹ ਚਲਾਓ | ਸੁਤੰਤ੍ਰ ਕਾਰਜ: ਮੌਸਮ + ਖ਼ਬਰਾਂ + ਸਟਾਕਸ |  
| **Loop** | ਜੇ ਤੱਕ ਸ਼ਰਤ ਪੂਰੀ ਨਾ ਹੋਵੇ ਲਈ ਮੁੜ ਜਾਓ | ਗੁਣਵੱਤਾ ਮੁਲਾਂਕਣ: ਸਕੋਰ ≥ 0.8 ਤੱਕ ਸੁਧਾਰੋ |  
| **Conditional** | ਸ਼ਰਤਾਂ ਅਨੁਸਾਰ ਰਾਹ ਦਿਓ | ਵਰਗੀਕਰਨ → ਤਕਨੀਕੀ ਖਾਸ ਏਜੰਟ ਨੂੰ ਭੇਜੋ |  
| **Human-in-the-Loop** | ਮਨੁੱਖੀ ਜਾਂਚਦਾਰ ਸ਼ਾਮਲ ਕਰੋ | ਮਨਜ਼ੂਰੀ ਪ੍ਰਵਾਹ, ਸਮੱਗਰੀ ਸੰਸੋਧਨ |  

## ਮੁੱਖ ਸੰਕਲਪ  

ਹੁਣ ਜਦ ਤੁਸੀਂ MCP ਅਤੇ ਏਜੰਟਿਕ ਮੋਡੀਊਲ ਨੂੰ ਵਰਤਦੇ ਵੇਖ ਲਈ ਹੈ, ਚਲੋ ਇੱਕ ਸੰਖੇਪ ਕਰੀਏ ਕਿ ਕਦੋਂ ਕਿਹੜਾ ਤਰੀਕਾ ਵਰਤਣਾ ਹੈ।  

MCP ਦਾ ਸਭ ਤੋਂ ਵੱਡਾ ਫਾਇਦਾ ਇਸਦਾ ਵਧਦਾ ਹੋਇਆ экੋਸਿਸਟਮ ਹੈ। ਹੇਠਾਂ ਡਾਇਗ੍ਰਾਮ ਵਿੱਚ ਦਿਖਾਇਆ ਗਿਆ ਹੈ ਕਿ ਇੱਕ ਇਕਾਈ ਵਰਲਡ ਪ੍ਰੋਟੋਕੋਲ ਤੁਹਾਡੇ AI ਐਪਲੀਕੇਸ਼ਨ ਨੂੰ ਵੱਖ-ਵੱਖ MCP ਸਰਵਰਾਂ ਨਾਲ ਜੋੜਦਾ ਹੈ — ਜਿਵੇਂ ਕਿ ਫਾਇਲ ਸਿਸਟਮ ਅਤੇ ਡੇਟਾਬੇਸ ਪਹੁੰਚ ਤੋਂ ਲੈ ਕੇ GitHub, ਈਮੇਲ, ਵੈੱਬ ਸਕ੍ਰੈਪਿੰਗ ਤੱਕ:  

<img src="../../../translated_images/pa/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="MCP Ecosystem" width="800"/>  

*MCP ਇਕ ਵਰਲਡ-ਪ੍ਰੋਟੋਕੋਲ ਐਕੋਸਿਸਟਮ ਬਣਾਉਂਦਾ ਹੈ — ਕੋਈ ਵੀ MCP-ਅਨੁਕੂਲ ਸਰਵਰ ਕਿਸੇ ਵੀ MCP-ਅਨੁਕੂਲ ਕਲਾਇੰਟ ਨਾਲ ਕੰਮ ਕਰਦਾ ਹੈ, ਐਪਲੀਕੇਸ਼ਨਾਂ ਵਿਚਕਾਰ ਟੂਲ ਸਾਂਝਾ ਕਰਨ ਯੋਗ।*  

**MCP** ਵਧੀਆ ਹੈ ਜਦੋਂ ਤੁਸੀਂ ਮੌਜੂਦਾ ਟੂਲ ਇਕੋਸਿਸਟਮਾਂ ਤੋਂ ਲਾਭ ਲੈਣਾ ਚਾਹੁੰਦੇ ਹੋ, ਅਜਿਹੇ ਟੂਲ ਬਣਾਉਣੇ ਜੋ ਕਈ ਐਪਲੀਕੇਸ਼ਨ ਸਾਂਝਾ ਕਰ ਸਕਦੀਆਂ, ਮਿਆਰੀ ਪ੍ਰੋਟੋਕੋਲਾਂ ਨਾਲ ਤੀਜੇ-ਪੱਖ ਸਰਵਿਸਜ਼ ਨੂੰ ਜੋੜਨਾ, ਜਾਂ ਕੋਡ ਬਦਲੇ ਬਿਨਾਂ ਟੂਲ ਦੀ ਅਮਲੀਕਰਨ ਬਦਲਣਾ ਚਾਹੁੰਦੇ ਹੋ।  

**ਏਜੰਟਿਕ ਮੋਡੀਊਲ** ਸਭ ਤੋਂ ਵਧੀਆ ਹੈ ਜਦੋਂ ਤੁਹਾਨੂੰ `@Agent` ਐਨੋਟੇਸ਼ਨਾਂ ਵਾਲੇ ਘੋਸ਼ਿਤ ਏਜੰਟ ਪਰਿਭਾਸ਼ਾਵਾਂ ਦੀ ਲੋੜ ਹੋਵੇ, ਵਰਕਫਲੋ ਸੰਚਾਲਨ (ਲੜੀਵਾਰ, ਲੂਪ, ਸਮਾਂਨਤ) ਚਾਹੀਦਾ ਹੋਵੇ, ਨਿਰਦੇਸ਼ੀਕ ਕੋਡ ਦੀ ਬਜਾਏ ਇੰਟਰਫੇਸ-ਆਧਾਰਿਤ ਏਜੰਟ ਡਿਜ਼ਾਈਨ ਪਸੰਦ ਹੋਵੇ, ਜਾਂ ਕਈ ਏਜੰਟਾਂ ਨੂੰ `outputKey` ਰਾਹੀਂ ਆਉਟਪੁੱਟ ਸਾਂਝਾ ਕਰਨੇ ਹੋਣ।  

**ਸੁਪਰਵਾਈਜ਼ਰ ਏਜੰਟ ਪੈਟਰਨ** ਤਦ ਚਮਕਦਾ ਹੈ ਜਦ ਵਰਕਫਲੋ ਪਹਿਲਾਂ ਤੋਂ ਪੂਰੀ ਤਰ੍ਹਾਂ ਪਤਾ ਨਹੀਂ ਹੁੰਦਾ ਅਤੇ ਤੁਹਾਨੂੰ LLM ਦੇ ਫੈਸਲੇ ਚਾਹੀਦੇ ਹਨ, ਜਦ ਤੁਹਾਡੇ ਕੋਲ ਵੱਖ-ਵੱਖ ਖਾਸ ਏਜੰਟ ਹੁੰਦੇ ਹਨ ਜੋ ਗਤੀਸ਼ੀਲ ਸੰਚਾਲਨ ਦੀ ਲੋੜ ਰੱਖਦੇ ਹਨ, ਗੱਲਬਾਤੀ ਪ੍ਰਣਾਲੀਆਂ ਬਨਾਉਣੀ ਹੁੰਦੀ ਹੈ ਜੋ ਵੱਖ-ਵੱਖ ਸਮਰੱਥਾਵਾਂ ਨੂੰ ਰੂਟ ਕਰਦੀਆਂ ਹਨ, ਜਾਂ ਜਦ ਤੁਹਾਨੂੰ ਸਭ ਤੋਂ ਲਚਕੀਲਾ ਅਤੇ ਅਨੁਕੂਲਿਤ ਏਜੰਟ ਵਿਹਾਰ ਚਾਹੀਦਾ ਹੈ।  

ਆਪਣੇ ਫੈਸਲੇ ਲਈ ਕਸਟਮ `@Tool` ਮੈਥਡਸ ਜਿਵੇਂ ਮੋਡੀਊਲ 04 ਵਿੱਚ ਹਨ ਅਤੇ MCP ਟੂਲਾਂ ਵਿਚਕਾਰ ਤੁਲਨਾ ਇਹ ਹੈ — ਕਸਟਮ ਟੂਲ ਤੁਹਾਨੂੰ ਘਣਿਸ਼ਠ ਜੋੜ ਅਤੇ ਪੂਰੀ ਟਾਈਪ ਸੁਰੱਖਿਆ ਦਿੰਦੇ ਹਨ ਐਪ-ਵਿਸ਼ੇਸ਼ ਲੋਜਿਕ ਲਈ, ਜਦ MCP ਟੂਲ ਮਿਆਰੀਕ੍ਰਿਤ, ਮੁੜ ਵਰਤਣਯੋਗ ਇੰਟੀਗ੍ਰੇਸ਼ਨ ਪ੍ਰਦਾਨ ਕਰਦੇ ਹਨ:  

<img src="../../../translated_images/pa/custom-vs-mcp-tools.c4f9b6b1cb65d8a1.webp" alt="Custom Tools vs MCP Tools" width="800"/>  

*ਜਦ ਕਸਟਮ @Tool ਮੈਥਡ ਅਤੇ MCP ਟੂਲ ਵਰਤਣ ਜ਼ਰੂਰੀ ਹੋਵੇ — ਕਸਟਮ ਟੂਲ ਐਪ ਵਾਤਾਵਰਣ ਲਈ ਪੂਰੀ ਟਾਈਪ ਸੁਰੱਖਿਆ ਦਿੰਦੇ ਹਨ, MCP ਟੂਲ ਮਿਆਰੀਕ੍ਰਿਤ ਇੰਟੀਗ੍ਰੇਸ਼ਨ ਹਨ ਜੋ ਐਪਲੀਕੇਸ਼ਨਾਂ ਵਿੱਚ ਕੰਮ ਕਰਦੇ ਹਨ।*

<img src="../../../translated_images/pa/course-completion.48cd201f60ac7570.webp" alt="Course Completion" width="800"/>  

*ਤੁਹਾਡੇ ਸਿੱਖਣ ਦੀ ਯਾਤਰਾ ਪੰਜ ਮੋਡੀਊਲ ਤੋਂ ਲੈ ਕੇ — ਬੁਨਿਆਦੀ ਚੈਟ ਤੋਂ MCP-ਸੰਚਾਲਿਤ ਏਜੰਟਿਕ ਪ੍ਰਣਾਲੀਆਂ ਤੱਕ।*  

ਤੁਸੀਂ LangChain4j for Beginners ਕੋਰਸ ਪੂਰਾ ਕੀਤਾ ਹੈ। ਤੁਸੀਂ ਸਿੱਖਿਆ:  

- ਮੇਮੋਰੀ ਨਾਲ ਗੱਲਬਾਤਾਈ AI ਕਿਵੇਂ ਬਣਾਈਏ (ਮੋਡੀਊਲ 01)  
- ਵੱਖ-ਵੱਖ ਕਾਰਜਾਂ ਲਈ ਪ੍ਰੰਪਟ ਇੰਜੀਨੀਅਰਿੰਗ ਪੈਟਰਨ (ਮੋਡੀਊਲ 02)  
- ਆਪਣੇ ਦਸਤਾਵੇਜ਼ਾਂ ਵਿੱਚੋਂ ਉਤਰ ਪ੍ਰਾਪਤ ਕਰਨਾ RAG ਨਾਲ (ਮੋਡੀਊਲ 03)  
- ਕਸਟਮ ਟੂਲਾਂ ਨਾਲ ਮੂਲ AI ਏਜੰਟ ਤਿਆਰ ਕਰਨਾ (ਮੋਡੀਊਲ 04)  
- LangChain4j MCP ਅਤੇ Agentic ਮੋਡੀਊਲਾਂ ਨਾਲ ਮਿਆਰੀਕ੍ਰਿਤ ਟੂਲਾਂ ਦਾ ਇੰਟੀਗ੍ਰੇਸ਼ਨ (ਮੋਡੀਊਲ 05)
