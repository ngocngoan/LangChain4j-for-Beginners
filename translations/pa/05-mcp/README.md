## ਤੁਸੀਂ ਕੀ ਸਿੱਖੋਗੇ

ਤੁਸੀਂ ਗੱਲਬਾਤੀ AI ਬਣਾਇਆ ਹੈ, ਪ੍ਰਾਂਪਟਾਂ ਵਿੱਚ ਮਾਹਿਰ ਹੋ, ਦਸਤਾਵੇਜ਼ਾਂ ਵਿੱਚ ਜਵਾਬਾਂ ਨੂੰ ਅਧਾਰਤ ਕੀਤਾ ਹੈ, ਅਤੇ ਟੂਲਾਂ ਨਾਲ ਏਜੰਟ ਬਣਾਏ ਹਨ। ਪਰ ਉਹ ਸਾਰੇ ਟੂਲ ਤੁਹਾਡੇ ਵਿਸ਼ੇਸ਼ ਐਪਲੀਕੇਸ਼ਨ ਲਈ ਕਸਟਮ ਬਣਾਏ ਗਏ ਸਨ। ਫਿਰ ਕੀ ਹੁੰਦਾ ਜੇ ਤੁਸੀਂ ਆਪਣੀ AI ਨੂੰ ਇੱਕ ਮਿਆਰਬੱਧ ਟੂਲ ਪਰਿਵਾਰ ਦੀ ਪਹੁੰਚ ਦੇ ਸਕਦੇ ਜੋ ਕੋਈ ਵੀ ਬਣਾ ਸਕਦਾ ਅਤੇ ਸਾਂਝਾ ਕਰ ਸਕਦਾ ਹੈ? ਇਸ ਮੌਡੀਊਲ ਵਿੱਚ, ਤੁਸੀਂ ਮਾਡਲ ਪ੍ਰਸੰਗ ਪ੍ਰੋਟੋਕੋਲ (MCP) ਅਤੇ LangChain4j ਦੇ ਏਜੈਂਟਿਕ ਮੌਡੀਊਲ ਨਾਲ ਇਹ ਕੀਮਤੀ ਕਾਰਜ ਕਰਨਾ ਸਿੱਖੋਗੇ। ਪਹਿਲਾਂ ਅਸੀਂ ਸਧਾਰਨ MCP ਫਾਇਲ ਰੀਡਰ ਦਿਖਾਉਂਦੇ ਹਾਂ ਫਿਰ ਦਿਖਾਉਂਦੇ ਹਾਂ ਕਿ ਕਿਵੇਂ ਇਹ ਉੱਚਤਮ ਏਜੈਂਟਿਕ ਵਰਕਫਲੋਜ਼ ਵਿੱਚ ਸੁਪਰਵਾਈਜ਼ਰ ਏਜੰਟ ਪੈਟਰਨ ਦੀ ਵਰਤੋਂ ਨਾਲ ਆਸਾਨੀ ਨਾਲ ਮਿਲਦਾ ਹੈ।

## MCP ਕੀ ਹੈ?

ਮਾਡਲ ਪ੍ਰਸੰਗ ਪ੍ਰੋਟੋਕੋਲ (MCP) ਠੀਕ ਇਹੀ ਪ੍ਰਦਾਨ ਕਰਦਾ ਹੈ - AI ਐਪਲੀਕੇਸ਼ਨਾਂ ਨੂੰ ਬਾਹਰੀ ਟੂਲਾਂ ਨੂੰ ਖੋਜਣ ਅਤੇ ਵਰਤਣ ਦਾ ਇੱਕ ਮਿਆਰੀ ਤਰੀਕਾ। ਹਰ ਡਾਟਾ ਸਰੋਤ ਜਾਂ ਸੇਵਾ ਲਈ ਕਸਟਮ ਇੰਟੀਗਰੇਸ਼ਨ ਲਿਖਣ ਦੀ ਬਜਾਏ, ਤੁਸੀਂ ਉਹ MCP ਸਰਵਰਾਂ ਨਾਲ ਜੁੜਦੇ ਹੋ ਜੋ ਉਹਨਾਂ ਦੀਆਂ ਯੋਗਤਾਵਾਂ ਨੂੰ ਇੱਕ ਸੰਗਠਿਤ ਫਾਰਮੈਟ ਵਿੱਚ ਪ੍ਰਗਟ ਕਰਦੇ ਹਨ। ਤੁਹਾਡਾ AI ਏਜੰਟ ਫਿਰ ਆਪਣੇ ਆਪ ਇਹ ਟੂਲ ਲੱਭ ਅਤੇ ਵਰਤ ਸਕਦਾ ਹੈ।

ਹੇਠਾਂ ਦਿੱਤਾ ਚਿੱਤਰ ਫਰਕ ਨੂੰ ਦਰਸਾਉਂਦਾ ਹੈ — MCP ਤੋਂ ਬਿਨਾਂ, ਹਰ ਇੰਟੀਗਰੇਸ਼ਨ ਲਈ ਕਸਟਮ ਪੁਆਇੰਟ-ਟੁ-ਪੁਆਇੰਟ ਵਾਇਰਿੰਗ ਜ਼ਰੂਰੀ ਹੈ; MCP ਨਾਲ, ਇੱਕ ਪ੍ਰੋਟੋਕੋਲ ਤੁਹਾਡੀ ਐਪ ਨੂੰ ਕਿਸੇ ਵੀ ਟੂਲ ਨਾਲ ਜੋੜਦਾ ਹੈ:

<img src="../../../translated_images/pa/mcp-comparison.9129a881ecf10ff5.webp" alt="MCP ਤુલਨਾ" width="800"/>

*MCP ਤੋਂ ਪਹਿਲਾਂ: ਜਟਿਲ ਪੁਆਇੰਟ-ਟੁ-ਪੁਆਇੰਟ ਇੰਟੀਗਰੇਸ਼ਨ। MCP ਤੋਂ ਬਾਅਦ: ਇੱਕ ਪ੍ਰੋਟੋਕੋਲ, ਅਨੰਤ ਸੰਭਾਵਨਾਵਾਂ।*

MCP AI ਵਿਕਾਸ ਵਿੱਚ ਇੱਕ ਮੂਲ ਸਮੱਸਿਆ ਦਾ ਹੱਲ ਕਰਦਾ ਹੈ: ਹਰ ਇੰਟੀਗਰੇਸ਼ਨ ਕਸਟਮ ਹੁੰਦੀ ਹੈ। ਗਿਟਹੱਬ ਤਕ ਪਹੁੰਚ ਕਰਨੀ ਹੈ? ਕਸਟਮ ਕੋਡ। ਫਾਇਲਾਂ ਪੜ੍ਹਣੀਆਂ ਹਨ? ਕਸਟਮ ਕੋਡ। ਡੇਟਾਬੇਸ ਨੂੰ ਕੈਸਰੀ ਕਰਨੀ ਹੈ? ਕਸਟਮ ਕੋਡ। ਅਤੇ ਇਹਨਾਂ ਇੰਟੀਗਰੇਸ਼ਨਾਂ ਵਿੱਚੋਂ ਕੋਈ ਵੀ ਹੋਰ AI ਐਪਲੀਕੇਸ਼ਨਾਂ ਨਾਲ ਕੰਮ ਨਹੀਂ ਕਰਦਾ।

MCP ਇਸ ਨੂੰ ਸਾਮਾਨ ਕਰਦਾ ਹੈ। ਇੱਕ MCP ਸਰਵਰ ਟੂਲ ਬਿਆਨ ਅਤੇ ਸਕੀਮਾ ਨਾਲ ਸਾਫ਼ ਚੀਜ਼ਾਂ ਪ੍ਰਦਾਨ ਕਰਦਾ ਹੈ। ਕੋਈ ਵੀ MCP ਕਲਾਇੰਟ ਜੁੜ ਸਕਦਾ ਹੈ, ਉਪਲਬਧ ਟੂਲ ਲੱਭ ਸਕਦਾ ਹੈ ਅਤੇ ਉਹਨਾਂ ਨੂੰ ਵਰਤ ਸਕਦਾ ਹੈ। ਇੱਕ ਵਾਰੀ ਬਣਾਓ, ਹਰ ਜਗ੍ਹਾ ਵਰਤੋ।

ਹੇਠਾਂ ਦਿੱਤਾ ਚਿੱਤਰ ਇਸ ਵਿਆਰਚੀਟਿਕ ਨੂੰ ਦਰਸਾਉਂਦਾ ਹੈ — ਇੱਕ MCP ਕਲਾਇੰਟ (ਤੁਹਾਡੀ AI ਐਪਲੀਕੇਸ਼ਨ) ਇੱਕ ਜਾਂ ਇੱਕ ਤੋਂ ਵੱਧ MCP ਸਰਵਰਾਂ ਨਾਲ ਜੁੜਦਾ ਹੈ, ਹਰ ਇੱਕ ਆਪਣੇ ਟੂਲਾਂ ਨੂੰ ਮਿਆਰਿਤ ਪ੍ਰੋਟੋਕੋਲ ਰਾਹੀਂ ਪ੍ਰਗਟ ਕਰਦਾ ਹੈ:

<img src="../../../translated_images/pa/mcp-architecture.b3156d787a4ceac9.webp" alt="MCP ਵਿਆਰਚੀਟਿਕ" width="800"/>

*ਮਾਡਲ ਪ੍ਰਸੰਗ ਪ੍ਰੋਟੋਕੋਲ ਵਿਆਰਚੀਟਿਕ - ਸੰਗਠਿਤ ਟੂਲਾਂ ਦੀ ਖੋਜ ਅਤੇ ਕਾਰਜਕਾਰੀ*

## MCP ਕਿਵੇਂ ਕੰਮ ਕਰਦਾ ਹੈ

ਅੰਦਰੂਨੀ ਤੌਰ 'ਤੇ, MCP ਇੱਕ ਸਤਰਬੱਧ ਵਿਆਰਚੀਟਿਕ ਦੀ ਵਰਤੋਂ ਕਰਦਾ ਹੈ। ਤੁਹਾਡੀ ਜਾਵਾ ਐਪਲੀਕੇਸ਼ਨ (MCP ਕਲਾਇੰਟ) ਉਪਲਬਧ ਟੂਲ ਲੱਭਦਾ ਹੈ, ਟ੍ਰਾਂਸਪੋਰਟ ਲੇਅਰ (Stdio ਜਾਂ HTTP) ਜ਼ਰੀਏ JSON-RPC ਬੇਨਤੀਆਂ ਭੇਜਦਾ ਹੈ, ਅਤੇ MCP ਸਰਵਰ ਓਪਰੇਸ਼ਨਾਂ ਨੂੰ ਚਲਾਉਂਦਾ ਅਤੇ ਨਤੀਜੇ ਵਾਪਸ ਕਰਦਾ ਹੈ। ਹੇਠਾਂ ਦਿੱਤੇ ਚਿੱਤਰ ਵਿੱਚ ਇਸ ਪ੍ਰੋਟੋਕੋਲ ਦੇ ਹਰ ਸਤਰ ਨੂੰ ਵਿਸਥਾਰ ਨਾਲ ਦਿਖਾਇਆ ਗਿਆ ਹੈ:

<img src="../../../translated_images/pa/mcp-protocol-detail.01204e056f45308b.webp" alt="MCP ਪ੍ਰੋਟੋਕੋਲ ਵਿਸਥਾਰ" width="800"/>

*MCP ਅੰਦਰ ਕਿਵੇਂ ਕੰਮ ਕਰਦਾ ਹੈ — ਕਲਾਇੰਟ ਟੂਲ ਲੱਭਦੇ ਹਨ, JSON-RPC ਸੁਨੇਹੇ ਵਟਾਂਦਰੇ ਕਰਦੇ ਹਨ, ਅਤੇ ਓਪਰੇਸ਼ਨ ਇੱਕ ਟ੍ਰਾਂਸਪੋਰਟ ਲੇਅਰ ਰਾਹੀਂ ਚਲਾਉਂਦੇ ਹਨ।*

**ਸਰਵਰ-ਕਲਾਇੰਟ ਵਿਆਰਚੀਟਿਕ**

MCP ਇੱਕ ਕਲਾਇੰਟ-ਸਰਵਰ ਮਾਡਲ ਵਰਤਦਾ ਹੈ। ਸਰਵਰ ਟੂਲ ਪ੍ਰਦਾਨ ਕਰਦੇ ਹਨ - ਫਾਇਲਾਂ ਪੜ੍ਹਨਾ, ਡੇਟਾਬੇਸ ਨੂੰ ਕੈਸਰੀ ਕਰਨਾ, ਏਪੀਆਈ ਕਾਲ ਕਰਨ। ਕਲਾਇੰਟ (ਤੁਹਾਡੀ AI ਐਪਲੀਕੇਸ਼ਨ) ਸਰਵਰਾਂ ਨਾਲ ਜੁੜਦੇ ਅਤੇ ਉਹਨਾਂ ਦੇ ਟੂਲ ਵਰਤਦੇ ਹਨ।

LangChain4j ਨਾਲ MCP ਵਰਤਣ ਲਈ, ਇਹ ਮੈਵਨ ਨਿਰਭਰਤਾ ਸ਼ਾਮਿਲ ਕਰੋ:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```
  
**ਟੂਲ ਖੋਜ**

ਜਦੋਂ ਤੁਹਾਡਾ ਕਲਾਇੰਟ MCP ਸਰਵਰ ਨਾਲ ਜੁੜਦਾ ਹੈ, ਤਾਂ ਉਹ ਪੁੱਛਦਾ ਹੈ "ਤੁਹਾਡੇ ਕੋਲ ਕਿਹੜੇ ਟੂਲ ਹਨ?" ਸਰਵਰ ਉਪਲਬਧ ਟੂਲਾਂ ਦੀ ਇੱਕ ਸੂਚੀ ਨਾਲ ਉਤਰਦਾ ਹੈ, ਹਰ ਇੱਕ ਨਾਲ ਵਿਆਖਿਆਵਾਂ ਅਤੇ ਪੈਰਾਮੀਟਰ ਸਕੀਮਾ ਦੇ ਨਾਲ। ਤੁਹਾਡਾ AI ਏਜੰਟ ਫਿਰ ਯੂਜ਼ਰ ਬੇਨਤੀਆਂ ਦੇ ਆਧਾਰ 'ਤੇ ਇਹ ਤੈਅ ਕਰ ਸਕਦਾ ਹੈ ਕਿ ਕਿਹੜੇ ਟੂਲ ਵਰਤਣੇ ਹਨ। ਹੇਠਾਂ ਦਿੱਤਾ ਚਿੱਤਰ ਇਸ ਹੈਂਡਸ਼ੇਕ ਨੂੰ ਦਿਖਾਉਂਦਾ ਹੈ — ਕਲਾਇੰਟ `tools/list` ਬੇਨਤੀ ਭੇਜਦਾ ਹੈ ਅਤੇ ਸਰਵਰ ਆਪਣਾ ਉਪਲਬਧ ਟੂਲਾਂ ਦੀ ਸੂਚੀ ਵਿਆਖਿਆਵਾਂ ਅਤੇ ਪੈਰਾਮੀਟਰ ਸਕੀਮਾਂ ਨਾਲ ਵਾਪਸ ਕਰਦਾ ਹੈ:

<img src="../../../translated_images/pa/tool-discovery.07760a8a301a7832.webp" alt="MCP ਟੂਲ ਖੋਜ" width="800"/>

*AI ਸ਼ੁਰੂਆਤ ਵਿੱਚ ਉਪਲਬਧ ਟੂਲ ਖੋਜਦਾ ਹੈ — ਹੁਣ ਉਹ ਜਾਣਦਾ ਹੈ ਕਿ ਕੀ ਸਮਰੱਥਾਵਾਂ ਹਨ ਅਤੇ ਕਿਹੜੇ ਵਰਤਣੇ ਹਨ।*

**ਟ੍ਰਾਂਸਪੋਰਟ ਤਰੀਕੇ**

MCP ਵੱਖ-ਵੱਖ ਟ੍ਰਾਂਸਪੋਰਟ ਤਰੀਕੇ ਸਮਰਥਿਤ ਕਰਦਾ ਹੈ। ਦੋ ਵਿਕਲਪ Stdio (ਸਥਾਨਕ ਤਤਕਾਲ ਸਰਵਰ ਕਮਿਊਨਿਕੇਸ਼ਨ ਲਈ) ਅਤੇ ਸਟ੍ਰੀਮਯੋਗ HTTP (ਦੂਰ-ਦਰਾਜ ਦੇ ਸਰਵਰਾਂ ਲਈ) ਹਨ। ਇਹ ਮੌਡੀਊਲ Stdio ਟ੍ਰਾਂਸਪੋਰਟ ਵੇਖਾਉਂਦਾ ਹੈ:

<img src="../../../translated_images/pa/transport-mechanisms.2791ba7ee93cf020.webp" alt="ਟ੍ਰਾਂਸਪੋਰਟ ਤਰੀਕੇ" width="800"/>

*MCP ਟ੍ਰਾਂਸਪੋਰਟ ਤਰੀਕੇ: HTTP ਦੂਰ-ਸਰਵਰਾਂ ਲਈ, Stdio ਸਥਾਨਕ ਪ੍ਰਕਿਰਿਆਵਾਂ ਲਈ*

**Stdio** - [StdioTransportDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java)

ਸਥਾਨਕ ਪ੍ਰਕਿਰਿਆਵਾਂ ਲਈ। ਤੁਹਾਡੀ ਐਪਲੀਕੇਸ਼ਨ ਇੱਕ ਸਰਵਰ ਦੂਜੇ ਪ੍ਰਕਿਰਿਆ ਵਜੋਂ ਖੜੀ ਕਰਦੀ ਹੈ ਅਤੇ ਸਧਾਰਣ ਇਨਪੁੱਟ/ਆਉਟਪੁੱਟ ਰਾਹੀਂ ਸੰਚਾਰ ਕਰਦੀ ਹੈ। ਫਾਇਲਸਿਸਟਮ ਪਹੁੰਚ ਜਾਂ ਕਮਾਂਡ ਲਾਈਨ ਟੂਲਾਂ ਲਈ ਉਪਯੋਗੀ।

```java
McpTransport stdioTransport = new StdioMcpTransport.Builder()
    .command(List.of(
        npmCmd, "exec",
        "@modelcontextprotocol/server-filesystem@2025.12.18",
        resourcesDir
    ))
    .logEvents(false)
    .build();
```
  
`@modelcontextprotocol/server-filesystem` ਸਰਵਰ ਹੇਠਾਂ ਦਿੱਤੇ ਟੂਲ ਪ੍ਰਦਾਨ ਕਰਦਾ ਹੈ, ਜੋ ਤੁਹਾਡੇ ਦਿਓ ਗਏ ਡਾਇਰੈਕਟਰੀਆਂ ਤੱਕ ਸੀਮਿਤ ਹਨ:

| ਟੂਲ | ਵਰਣਨ |
|------|-------|
| `read_file` | ਇੱਕ ਫਾਇਲ ਦਾ ਸਮੱਗਰੀ ਪੜ੍ਹੋ |
| `read_multiple_files` | ਇੱਕ ਵਾਰੀ ਵਿੱਚ ਕਈ ਫਾਇਲਾਂ ਪੜ੍ਹੋ |
| `write_file` | ਫਾਇਲ ਬਣਾਓ ਜਾਂ ਓਵਰਰਾਈਟ ਕਰੋ |
| `edit_file` | ਨਿਸ਼ਚਿਤ ਤਬਦੀਲੀਆਂ ਕਰੋ (ਢੂੰਢੋ ਤੇ ਬਦਲੋ) |
| `list_directory` | ਕਿਸੇ ਰਸਤੇ 'ਤੇ ਫਾਇਲਾਂ ਅਤੇ ਡਾਇਰੈਕਟਰੀਆਂ ਦੀ ਗਿਣਤੀ ਕਰੋ |
| `search_files` | ਰੀਕਰਸਿਵ ਢੰਗ ਨਾਲ ਫਾਇਲਾਂ ਖੋਜੋ ਜੋ ਕਿਸੇ ਨਿਰਧਾਰਿਤ ਪੈਟਰਨ ਨਾਲ ਮਿਲਦੀਆਂ ਹੋਣ |
| `get_file_info` | ਫਾਇਲ ਮੈਟਾਡੇਟਾ ਪ੍ਰਾਪਤ ਕਰੋ (ਆਕਾਰ, ਸਮਾਂ-ਟੈਗ, ਅਧਿਕਾਰ) |
| `create_directory` | ਡਾਇਰੈਕਟਰੀ ਬਣਾਓ (ਮਾਪੜ ਡਾਇਰੈਕਟਰੀਆਂ ਸਮੇਤ) |
| `move_file` | ਫਾਇਲ ਜਾਂ ਡਾਇਰੈਕਟਰੀ ਨੂੰ ਖਸਕਾਓ ਜਾਂ ਨਾਮ ਬਦਲੋ |

ਹੇਠਾਂ ਦਿੱਤਾ ਚਿੱਤਰ ਦਿਖਾਉਂਦਾ ਹੈ ਕਿ Stdio ਟ੍ਰਾਂਸਪੋਰਟ ਚਲਦੇ ਸਮੇਂ ਕਿਵੇਂ ਕੰਮ ਕਰਦਾ ਹੈ — ਤੁਹਾਡੀ ਜਾਵਾ ਐਪਲੀਕੇਸ਼ਨ MCP ਸਰਵਰ ਨੂੰ ਇੱਕ ਬੱਚੇ ਦੀ ਪ੍ਰਕਿਰਿਆ ਵਜੋਂ ਖੜਾ ਕਰਦੀ ਹੈ ਅਤੇ ਉਹ stdin/stdout ਪਾਈਪਾਂ ਰਾਹੀਂ ਸੰਚਾਰ ਕਰਦੇ ਹਨ, ਕੋਈ ਨੈੱਟਵਰਕ ਜਾਂ HTTP ਨਹੀਂ:

<img src="../../../translated_images/pa/stdio-transport-flow.45eaff4af2d81db4.webp" alt="Stdio ਟ੍ਰਾਂਸਪੋਰਟ ਫਲੋ" width="800"/>

*ਕੰਮ ਵਿੱਚ Stdio ਟ੍ਰਾਂਸਪੋਰਟ — ਤੁਹਾਡੀ ਐਪ MCP ਸਰਵਰ ਨੂੰ ਬੱਚੇ ਦੀ ਪ੍ਰਕਿਰਿਆ ਵਜੋਂ ਖੜਾ ਕਰਦੀ ਹੈ ਅਤੇ stdin/stdout ਪਾਈਪਾਂ ਰਾਹੀਂ ਸੰਚਾਰ ਕਰਦੀ ਹੈ।*

> - "LangChain4j MCP ਸਰਵਰ ਪ੍ਰਕਿਰਿਆਵਾਂ ਦੀ ਲਾਈਫਸਾਈਕਲ ਕਿਵੇਂ ਸੰਭਾਲਦਾ ਹੈ?"

## ਏਜੈਂਟਿਕ ਮੌਡੀਊਲ

ਜਦੋਂ ਕਿ MCP ਮਿਆਰਿਤ ਟੂਲ ਪ੍ਰਦਾਨ ਕਰਦਾ ਹੈ, LangChain4j ਦਾ **ਏਜੈਂਟਿਕ ਮੌਡੀਊਲ** ਉਹ ਟੂਲਾਂ ਦਾ ਕੰਮ ਕਰਨ ਲਈ ਏਜੰਟ ਬਣਾਉਣ ਦਾ ਇੱਕ ਵਿਘਟਤ ਢੰਗ ਦਿੰਦਾ ਹੈ। `@Agent` ਐਨੋਟੇਸ਼ਨ ਅਤੇ `AgenticServices` ਤੁਹਾਨੂੰ ਇੰਟਰਫੇਸ ਰਾਹੀਂ ਏਜੰਟ ਵਿਹਾਰ ਨੂੰ ਨਿਰਧਾਰਿਤ ਕਰਨ ਦੇ ਯੋਗ ਬਣਾਉਂਦੇ ਹਨ ਨਾ ਕਿ ਜਬਰਦਸਤੀ ਕੋਡ ਰਾਹੀਂ।

ਇਸ ਮੌਡੀਊਲ ਵਿੱਚ, ਤੁਸੀਂ **ਸੁਪਰਵਾਈਜ਼ਰ ਏਜੰਟ** ਪੈਟਰਨ ਦੀ ਜਾਂਚ ਕਰੋਗੇ — ਇੱਕ ਅਡਵਾਂਸ ਏਜੈਂਟਿਕ AI ਵਿਧੀ ਜਿੱਥੇ “ਸੁਪਰਵਾਈਜ਼ਰ” ਏਜੰਟ ਗਾਹਕ ਦੀ ਬੇਨਤੀ ਅਨੁਸਾਰ ਕਿਸ ਸਬ-ਏਜੰਟ ਨੂੰ ਕਦੋਂ ਕਾਲ ਕਰਨਾ ਹੈ ਫੈਸਲਾ ਕਰਦਾ ਹੈ। ਅਸੀਂ ਦੋਨਾਂ ਧਾਰਨਾਵਾਂ ਨੂੰ ਜੋੜਾਂਗੇ ਜਿੱਥੇ ਸਾਡੇ ਇੱਕ ਸਬ-ਏਜੰਟ ਨੂੰ MCP-ਸ਼ਕਤੀਸਾਲੀ ਫਾਇਲ ਪਹੁੰਚ ਮੁਹੱਈਆ ਕਰਵਾਈ ਜਾਵੇਗੀ।

ਏਜੈਂਟਿਕ ਮੌਡੀਊਲ ਵਰਤਣ ਲਈ ਇਹ ਮੈਵਨ ਨਿਰਭਰਤਾ ਸ਼ਾਮਿਲ ਕਰੋ:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```


> **ਨੋਟ:** `langchain4j-agentic` ਮੌਡੀਊਲ ਇੱਕ ਵੱਖਰੀ ਰਿਲੀਜ਼ ਤਾਰੀਖ ਅਤੇ ਵਰਜ਼ਿਨ ਪ੍ਰਾਪਰਟੀ (`langchain4j.mcp.version`) ਨਾਲ ਹੈ ਜੋ ਕੋਰ LangChain4j ਲਾਇਬ੍ਰੇਰੀਜ਼ ਤੋਂ ਅਲੱਗ ਹੈ।

> **⚠️ ਪ੍ਰਯੋਗਸ਼ਾਲਾ ਦੀ:** `langchain4j-agentic` ਮੌਡੀਊਲ ਪ੍ਰਯੋਗਸ਼ਾਲਾ ਦੌਰ 'ਤੇ ਹੈ ਅਤੇ ਬਦਲ ਸਕਦਾ ਹੈ। AI ਸਹਾਇਕ ਬਣਾਉਣ ਦਾ ਟਿਕਾਊ ਤਰੀਕਾ `langchain4j-core` ਹੈ ਕਸਟਮ ਟੂਲਾਂ (ਮੌਡੀਊਲ 04) ਨਾਲ।

ਐਪਲੀਕੇਸ਼ਨ ਆਟੋਮੈਟਿਕ MCP ਫਾਇਲਸਿਸਟਮ ਸਰਵਰ ਨੂੰ ਬਣਾਉਂਦਾ ਹੈ ਅਤੇ ਇੱਕ ਸਥਾਨਕ ਫਾਇਲ ਪੜ੍ਹਦਾ ਹੈ। ਧਿਆਨ ਦਿਓ ਕਿ ਕਿਵੇਂ ਸਬ-ਪਰਕਿਰਿਆ ਪ੍ਰਬੰਧਨ ਤੁਹਾਡੇ ਲਈ ਸੰਭਾਲਿਆ ਜਾਂਦਾ ਹੈ।

  
### ਸੁਪਰਵਾਈਜ਼ਰ ਏਜੰਟ

**ਸੁਪਰਵਾਈਜ਼ਰ ਏਜੰਟ ਪੈਟਰਨ** ਏਜੈਂਟਿਕ AI ਦਾ ਇੱਕ **ਲਚਕੀਲਾ** ਰੂਪ ਹੈ। ਇੱਕ ਸੁਪਰਵਾਈਜ਼ਰ LLM ਦੀ ਵਰਤੋਂ ਕਰਦਾ ਹੈ ਖੁਦਮੁਖਤਿਆਰ ਤੌਰ ਤੇ ਫੈਸਲਾ ਕਰਨ ਲਈ ਕਿ ਕਿਸੇ ਉਪ-ਏਜੰਟ ਨੂੰ ਕਦੋਂ ਕਿਵੇਂ ਕਾਲ ਕਰਨਾ ਹੈ ਗਾਹਕ ਦੀ ਬੇਨਤੀ ਅਨੁਸਾਰ। ਅਗਲੇ ਉਦਾਹਰਨ ਵਿੱਚ, ਅਸੀਂ MCP-ਸ਼ਕਤੀਵਾਨ ਫਾਇਲ ਪਹੁੰਚ ਨੂੰ ਇੱਕ LLM ਏਜੰਟ ਨਾਲ ਜੋੜ ਕੇ ਇੱਕ ਸੁਪਰਵਾਈਜ਼ਡ ਫਾਇਲ ਪੜ੍ਹਨਾ → ਰਿਪੋਰਟ ਵਰਕਫਲੋ ਬਣਾਂਗੇ।

ਡੈਮੋ ਵਿੱਚ, `FileAgent` MCP ਫਾਇਲਸਿਸਟਮ ਟੂਲ ਵਰਤ ਕੇ ਇੱਕ ਫਾਇਲ ਪੜ੍ਹਦਾ ਹੈ, ਅਤੇ `ReportAgent` ਇੱਕ ਸੰਰਚਿਤ ਰਿਪੋਰਟ ਤਿਆਰ ਕਰਦਾ ਹੈ ਜਿਸ ਵਿੱਚ ਪ੍ਰਧਾਨ ਸੰਖੇਪ (1 ਵਾਕ), 3 ਮੁੱਖ ਬਿੰਦੂ, ਅਤੇ ਸਿਫਾਰਸ਼ਾਂ ਹੁੰਦੀਆਂ ਹਨ। ਸੁਪਰਵਾਈਜ਼ਰ ਇਸ ਸੰਚਾਲਨ ਨੂੰ ਆਟੋਮੈਟਿਕਲੇ ਚਲਾਉਂਦਾ ਹੈ:

<img src="../../../translated_images/pa/supervisor-agent-pattern.06275a41ae006ac8.webp" alt="ਸੁਪਰਵਾਈਜ਼ਰ ਏਜੰਟ ਪੈਟਰਨ" width="800"/>

*ਸੁਪਰਵਾਈਜ਼ਰ ਆਪਣੇ LLM ਨਾਲ ਫੈਸਲਾ ਕਰਦਾ ਹੈ ਕਿਹੜੇ ਏਜੰਟ ਕਾਲ ਕਰਨ ਹਨ ਅਤੇ ਕਿਹੜੇ ਕ੍ਰਮ ਵਿੱਚ — ਕੋਈ ਸਖ਼ਤ ਰੂਟਿੰਗ ਦੀ ਲੋੜ ਨਹੀਂ।*

ਨੀਵੀਂ ਕੰਰੀਟ ਵਰਕਫਲੋ ਅਸੀਂ ਫਾਇਲ ਤੋਂ ਰਿਪੋਰਟ ਪਾਈਪਲਾਈਨ ਲਈ ਦਿਲਚਸਪੀ ਨਾਲ ਵੇਖਦੇ ਹਾਂ:

<img src="../../../translated_images/pa/file-report-workflow.649bb7a896800de9.webp" alt="ਫਾਇਲ ਤੋਂ ਰਿਪੋਰਟ ਵਰਕਫਲੋ" width="800"/>

*FileAgent MCP ਟੂਲਾਂ ਰਾਹੀਂ ਫਾਇਲ ਪੜ੍ਹਦਾ ਹੈ, ਫਿਰ ReportAgent ਕੱਚਾ ਸਮੱਗਰੀ ਸੰਰਚਿਤ ਰਿਪੋਰਟ ਵਿੱਚ ਬਦਲਦਾ ਹੈ।*

ਹਰ ਏਜੰਟ ਆਪਣਾ ਆਉਟਪੁੱਟ **Agentic Scope** (ਸਾਂਝੀ ਮੈਮੋਰੀ) ਵਿੱਚ ਸਾਂਭਦਾ ਹੈ, ਜੋ ਹੇਠਾਂ ਵਾਲੇ ਏਜੰਟਾਂ ਨੂੰ ਪਹਿਲਾਂ ਦੇ ਨਤੀਜੇ ਪ੍ਰਾਪਤ ਕਰਨ ਲਈ ਆਸਾਨੀ ਦਿੰਦਾ ਹੈ। ਇਹ ਦਰਸਾਉਂਦਾ ਹੈ ਕਿ MCP ਟੂਲ ਕਿਵੇਂ ਸਹਿਜਤਾ ਨਾਲ ਏਜੈਂਟਿਕ ਵਰਕਫਲੋਜ਼ ਵਿੱਚ ਮਿਲਦੇ ਹਨ—ਸੁਪਰਵਾਈਜ਼ਰ ਨੂੰ ਇਹ ਜਾਣਨ ਦੀ ਲੋੜ ਨਹੀਂ ਹੁੰਦੀ ਕਿ ਫਾਇਲ ਕਿਵੇਂ ਪੜ੍ਹੀ ਜਾਂਦੀ ਹੈ, ਸਿਰਫ ਇਹ ਕਿ `FileAgent` ਇਸ ਨੂੰ ਕਰ ਸਕਦਾ ਹੈ।

**VS ਕੋਡ ਵਰਤਦੇ ਹੋਏ:** `SupervisorAgentDemo.java` 'ਤੇ ਰਾਈਟ ਕਲਿੱਕ ਕਰੋ ਅਤੇ **"Run Java"** ਚੁਣੋ (ਯਕੀਨ ਕਰੋ ਕਿ `.env` ਫਾਇਲ ਸੈੱਟ ਹੈ)।

#### ਸੁਪਰਵਾਈਜ਼ਰ ਕਿਵੇਂ ਕੰਮ ਕਰਦਾ ਹੈ

ਏਜੰਟ ਬਣਾਉਣ ਤੋਂ ਪਹਿਲਾਂ, ਤੁਸੀਂ MCP ਟ੍ਰਾਂਸਪੋਰਟ ਨੂੰ ਕਲਾਇੰਟ ਨਾਲ ਜੋੜਕੇ `ToolProvider` ਵਜੋਂ ਲਪੇਟਣਾ ਹੋਵੇਗਾ। ਇਹ ਇਸ ਤਰੀਕੇ ਨਾਲ MCP ਸਰਵਰ ਦੇ ਟੂਲ ਤੁਹਾਡੇ ਏਜੰਟਾਂ ਲਈ ਉਪਲਬਧ ਹੋ ਜਾਂਦੇ ਹਨ:

```java
// ਟਰਾਂਸਪੋਰਟ ਤੋਂ MCP ਕਲਾਇੰਟ ਬਣਾਓ
McpClient mcpClient = new DefaultMcpClient.Builder()
        .transport(stdioTransport)
        .build();

// ਕਲਾਇੰਟ ਨੂੰ ToolProvider ਵਜੋਂ ਲਪੇਟੋ — ਇਹ MCP ਟੂਲਾਂ ਨੂੰ LangChain4j ਵਿੱਚ ਜੋੜਦਾ ਹੈ
ToolProvider mcpToolProvider = McpToolProvider.builder()
        .mcpClients(List.of(mcpClient))
        .build();
```
  
ਹੁਣ ਤੁਸੀਂ ਕਿਸੇ ਵੀ ਏਜੰਟ ਵਿੱਚ `mcpToolProvider` ਨੂੰ ਇੰਜੈਕਟ ਕਰ ਸਕਦੇ ਹੋ ਜਿਹੜੇ MCP ਟੂਲਾਂ ਦੀ ਲੋੜ ਹੈ:

```java
// ਕਦਮ 1: FileAgent MCP ਸੰਦਾਂ ਦੀ ਵਰਤੋਂ ਕਰਕੇ ਫਾਈਲਾਂ ਪੜ੍ਹਦਾ ਹੈ
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // ਫਾਈਲ ਓਪਰੇਸ਼ਨਾਂ ਲਈ MCP ਸੰਦ ਹਨ
        .build();

// ਕਦਮ 2: ReportAgent ਸਾਂਰਚਿਤ ਰਿਪੋਰਟਾਂ ਤਿਆਰ ਕਰਦਾ ਹੈ
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// ਸੂਪਰਵਾਈਜ਼ਰ ਫਾਈਲ → ਰਿਪੋਰਟ ਵਰਕਫਲੋਅ ਦਾ ਸਮੰਵਯ ਕਰਦਾ ਹੈ
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // ਆਖਰੀ ਰਿਪੋਰਟ ਵਾਪਸ ਕਰੋ
        .build();

// ਸੂਪਰਵਾਈਜ਼ਰ ਬੇਨਤੀ ਦੇ ਆਧਾਰ 'ਤੇ ਕਿਹੜੇ ਏਜੰਟ ਕਾਲ ਕਰਨੇ ਹਨ ਫੈਸਲਾ ਕਰਦਾ ਹੈ
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```
  
#### ਜਵਾਬ ਦੇਣ ਦੀਆਂ ਰਣਨੀਤੀਆਂ

ਜਦੋਂ ਤੁਸੀਂ `SupervisorAgent` ਕਨਫਿਗਰ ਕਰਦੇ ਹੋ, ਤਾਂ ਤੁਸੀਂ ਇਸ ਗੱਲ ਦਾ ਨਿਰਣਾ ਕਰਦੇ ਹੋ ਕਿ ਜਦੋਂ ਸਬ-ਏਜੰਟਾਂ ਆਪਣੇ ਕੰਮ ਮੁਕੰਮਲ ਕਰ ਲੈਂਦੇ ਹਨ ਤਾਂ ਇਹ ਯੂਜ਼ਰ ਨੂੰ ਆਪਣੇ ਫਾਈਨਲ ਜਵਾਬ ਕਿਵੇਂ ਦਿੰਦਾ ਹੈ। ਹੇਠਾਂ ਚਿੱਤਰ ਵਿੱਚ ਤਿੰਨ ਉਪਲਬਧ ਰਣਨੀਤੀਆਂ ਦਿਖਾਈ ਗਈਆਂ ਹਨ — LAST ਅੰਤਿਮ ਏਜੰਟ ਦਾ ਆਉਟਪੁੱਟ ਸਿੱਧਾ ਵਾਪਸ ਕਰਦਾ ਹੈ, SUMMARY ਸਾਰੇ ਆਉਟਪੁੱਟਾਂ ਨੂੰ LLM ਰਾਹੀਂ ਸੰਸ਼ਲੇਸ਼ਿਤ ਕਰਦਾ ਹੈ, ਅਤੇ SCORED ਵਾਲਾ ਜੇਹੜਾ ਵੀ ਮੁਲਾਂਕਣ ਉੱਚਾ ਲੱਗੇ ਉਸ ਨੂੰ ਚੁਣਦਾ ਹੈ:

<img src="../../../translated_images/pa/response-strategies.3d0cea19d096bdf9.webp" alt="ਜਵਾਬ ਦੇਣ ਦੀਆਂ ਰਣਨੀਤੀਆਂ" width="800"/>

*ਤਿੰਨ ਰਣਨੀਤੀਆਂ ਕਿ ਕੇਵੇਂ ਸੁਪਰਵਾਈਜ਼ਰ ਆਪਣੀ ਆਖ਼ਰੀ ਜਵਾਬ ਨੂੰ ਤਿਆਰ ਕਰਦਾ ਹੈ — ਚੁਣੋ ਕਿ ਤੁਸੀਂ ਆਖ਼ਰੀ ਏਜੰਟ ਦਾ ਜਵਾਬ, ਇਕ ਸੰਸ਼ਲੇਸ਼ਿਤ ਸਾਰ, ਜਾਂ ਸਭ ਤੋਂ ਵਧੀਆ ਸਕੋਰ ਵਾਲਾ ਵਿਕਲਪ ਚਾਹੁੰਦੇ ਹੋ।*

ਉਪਲਬਧ ਰਣਨੀਤੀਆਂ ਹਨ:

| ਰਣਨੀਤੀ | ਵਰਣਨ |
|----------|-------|
| **LAST** | ਸੁਪਰਵਾਈਜ਼ਰ ਆਖ਼ਰੀ ਉਪ-ਏਜੰਟ ਜਾਂ ਟੂਲ ਦੇ ਆਉਟਪੁੱਟ ਨੂੰ ਵਾਪਸ ਕਰਦਾ ਹੈ। ਇਹ ਫਾਇਦਾਮੰਦ ਹੁੰਦਾ ਹੈ ਜਦ ਆਖ਼ਰੀ ਏਜੰਟ ਵਿਸ਼ੇਸ਼ ਤੌਰ 'ਤੇ ਪੂਰਾ, ਅੰਤਿਮ ਜਵਾਬ ਦੇਣ ਲਈ ਡਿਜ਼ਾਈਨ ਕੀਤਾ ਗਿਆ ਹੋਵੇ (ਜਿਵੇਂ ਕਿ ਖੋਜ ਪਾਈਪਲਾਈਨ ਵਿੱਚ "ਸੰਖੇਪ ਏਜੰਟ")। |
| **SUMMARY** | ਸੁਪਰਵਾਈਜ਼ਰ ਆਪਣਾ ਅੰਦਰੂਨੀ ਭਾਸ਼ਾ ਮਾਡਲ (LLM) ਵਰਤਦਾ ਹੈ ਤਾਂ ਜੋ ਪੂਰੇ ਇੰਟਰੈਕਸ਼ਨ ਅਤੇ ਸਾਰੇ ਉਪ-ਏਜੰਟ ਆਉਟਪੁੱਟਾਂ ਦਾ ਸੰਸ਼ੇਪ ਤਿਆਰ ਕਰ ਸਕੇ ਅਤੇ ਫਿਰ ਇਸ ਸਾਰ ਨੂੰ ਅੰਤਿਮ ਜਵਾਬ ਵਜੋਂ ਵਾਪਸ ਕਰਦਾ ਹੈ। ਇਹ ਯੂਜ਼ਰ ਲਈ ਇੱਕ ਸਾਫ ਅਤੇ ਸੰਘੜਿਆ ਜਵਾਬ ਪ੍ਰਦਾਨ ਕਰਦਾ ਹੈ। |
| **SCORED** | ਪਰਣਾਲੀ ਇੱਕ ਅੰਦਰੂਨੀ LLM ਵਰਤਦੀ ਹੈ ਤਾਂ ਜੋ LAST ਜਵਾਬ ਅਤੇ SUMMARY ਦੀ ਮੁਲਾਂਕਣ ਕਰੇ ਅਤੇ ਉਹੀ ਆਉਟਪੁੱਟ ਵਾਪਸ ਕਰੇ ਜਿਸ ਨੂੰ ਅਸਲੀ ਯੂਜ਼ਰ ਬੇਨਤੀ ਦੇ ਖਿਲਾਫ਼ ਵਧੀਆ ਸਕੋਰ ਮਿਲੇ। |

#### ਆਉਟਪੁੱਟ ਦੀ ਸਮਝ

ਜਦੋਂ ਤੁਸੀਂ ਡੈਮੋ ਚਲਾਉਂਦੇ ਹੋ, ਤਦ ਤੁਹਾਨੂੰ ਇਹ ਵੇਖਣ ਨੂੰ ਮਿਲੇਗਾ ਕਿ Supervisor ਕਿਵੇਂ ਕਈ ਏਜੰਟਾਂ ਨੂੰ ਸਮਨ्वਯਿਤ ਕਰਦਾ ਹੈ। ਹੇਠਾਂ ਹਰ ਹਿੱਸੇ ਦਾ ਅਰਥ ਦਿੱਤਾ ਗਿਆ ਹੈ:

```
======================================================================
  FILE → REPORT WORKFLOW DEMO
======================================================================

This demo shows a clear 2-step workflow: read a file, then generate a report.
The Supervisor orchestrates the agents automatically based on the request.
```

**ਹੈਡਰ** ਵਰਕਫਲੋ ਧਾਰਨਾ ਦਾ ਪਰਚਾਰ ਕਰਦਾ ਹੈ: ਫਾਇਲ ਪੜ੍ਹਨ ਤੋਂ ਰਿਪੋਰਟ ਤਿਆਰ ਕਰਨ ਦੀ ਇੱਕ ਕੇਂਦ੍ਰਿਤ ਟੋਟੀ।

```
--- WORKFLOW ---------------------------------------------------------
  ┌─────────────┐      ┌──────────────┐
  │  FileAgent  │ ───▶ │ ReportAgent  │
  │ (MCP tools) │      │  (pure LLM)  │
  └─────────────┘      └──────────────┘
   outputKey:           outputKey:
   'fileContent'        'report'

--- AVAILABLE AGENTS -------------------------------------------------
  [FILE]   FileAgent   - Reads files via MCP → stores in 'fileContent'
  [REPORT] ReportAgent - Generates structured report → stores in 'report'
```

**ਵਰਕਫਲੋ ਡਾਇਗ੍ਰਾਮ** ਏਜੰਟਾਂ ਦੇ ਵਿਚਕਾਰ ਡੇਟਾ ਦੇ ਪ੍ਰਬਾਹ ਨੂੰ ਦਰਸਾਉਂਦਾ ਹੈ। ਹਰ ਏਜੰਟ ਦੀ ਇੱਕ ਵਿਸ਼ੇਸ਼ ਭੂਮਿਕਾ ਹੁੰਦੀ ਹੈ:
- **FileAgent** MCP ਟੂਲਜ਼ ਦੀ ਵਰਤੋਂ ਕਰਕੇ ਫਾਇਲਾਂ ਪੜ੍ਹਦਾ ਹੈ ਅਤੇ ਰਾ ਸਮੱਗਰੀ `fileContent` ਵਿਚ ਸਟੋਰ ਕਰਦਾ ਹੈ
- **ReportAgent** ਉਸ ਸਮੱਗਰੀ ਨੂੰ ਖਪਤ ਕਰਦਾ ਹੈ ਅਤੇ `report` ਵਿੱਚ ਇਕ ਸਧਾਰਿਤ ਰਿਪੋਰਟ ਬਣਾਉਂਦਾ ਹੈ

```
--- USER REQUEST -----------------------------------------------------
  "Read the file at .../file.txt and generate a report on its contents"
```

**ਯੂਜ਼ਰ ਦੀ ਬੇਨਤੀ** ਕੰਮ ਦਿਖਾਉਂਦੀ ਹੈ। Supervisor ਇਸ ਦਾ ਵਿਸ਼ਲੇਸ਼ਣ ਕਰਦਾ ਹੈ ਅਤੇ FileAgent → ReportAgent ਕਿਰਿਆਸ਼ੀਲ ਕਰਨ ਦਾ ਫੈਸਲਾ ਕਰਦਾ ਹੈ।

```
--- SUPERVISOR ORCHESTRATION -----------------------------------------
  The Supervisor decides which agents to invoke and passes data between them...

  +-- STEP 1: Supervisor chose -> FileAgent (reading file via MCP)
  |
  |   Input: .../file.txt
  |
  |   Result: LangChain4j is an open-source, provider-agnostic Java framework for building LLM...
  +-- [OK] FileAgent (reading file via MCP) completed

  +-- STEP 2: Supervisor chose -> ReportAgent (generating structured report)
  |
  |   Input: LangChain4j is an open-source, provider-agnostic Java framew...
  |
  |   Result: Executive Summary...
  +-- [OK] ReportAgent (generating structured report) completed
```

**Supervisor ਦਾ ਸਮਨਵਯ** 2-ਕਦਮ ਪ੍ਰਕਿਰਿਆ ਦਿਖਾਉਂਦਾ ਹੈ:
1. **FileAgent** MCP ਰਾਹੀਂ ਫਾਇਲ ਪੜ੍ਹਦਾ ਹੈ ਅਤੇ ਸਮੱਗਰੀ ਸਟੋਰ ਕਰਦਾ ਹੈ
2. **ReportAgent** ਉਸ ਸਮੱਗਰੀ ਨੂੰ ਲੈਂਦਾ ਹੈ ਅਤੇ ਸਧਾਰਿਤ ਰਿਪੋਰਟ ਤਿਆਰ ਕਰਦਾ ਹੈ

Supervisor ਨੇ ਇਹ ਫੈਸਲੇ **ਆਪਣੇ ਆਪ** ਯੂਜ਼ਰ ਦੀ ਬੇਨਤੀ ਦੇ ਆਧਾਰ 'ਤੇ ਕੀਤੇ।

```
--- FINAL RESPONSE ---------------------------------------------------
Executive Summary
...

Key Points
...

Recommendations
...

--- AGENTIC SCOPE (Data Flow) ----------------------------------------
  Each agent stores its output for downstream agents to consume:
  * fileContent: LangChain4j is an open-source, provider-agnostic Java framework...
  * report: Executive Summary...
```

#### Agentic ਮੌਡਿਊਲ ਦੀਆਂ ਵਿਸ਼ੇਸ਼ਤਾਵਾਂ ਦੀ ਵਿਆਖਿਆ

ਇਸ ਉਦਾਹਰਨ ਵਿੱਚ agentic ਮੌਡਿਊਲ ਦੀਆਂ ਕਈ ਉੱਚ-ਪੱਧਰੀਆਂ ਵਿਸ਼ੇਸ਼ਤਾਵਾਂ ਦਰਸਾਈਆਂ ਗਈਆਂ ਹਨ। ਆਓ Agentic ਸਕੋਪ ਅਤੇ Agent Listeners ਨੂੰ ਨੇੜੇ ਤੋਂ ਦੇਖੀਏ।

**Agentic ਸਕੋਪ** ਸਾਂਝੀ ਮੈਮੋਰੀ ਨੂੰ ਦਰਸਾਉਂਦਾ ਹੈ ਜਿੱਥੇ ਏਜੰਟ `@Agent(outputKey="...")` ਦੀ ਵਰਤੋਂ ਕਰਕੇ ਆਪਣੇ ਨਤੀਜੇ ਸਟੋਰ ਕਰਦੇ ਹਨ। ਇਸ ਨਾਲ:
- ਪਿਛਲੇ ਏਜੰਟਾਂ ਦੇ ਆਉਟਪੁੱਟ ਤੱਕ ਪਿੱਛਲੇ ਏਜੰਟ ਆਸਾਨੀ ਨਾਲ ਪਹੁੰਚ ਸਕਦੇ ਹਨ
- Supervisor ਅੰਤਿਮ ਜਵਾਬ ਤਿਆਰ ਕਰ ਸਕਦਾ ਹੈ
- ਤੁਸੀਂ ਦੇਖ ਸਕਦੇ ਹੋ ਕਿ ਹਰ ਏਜੰਟ ਨੇ ਕੀ ਤਿਆਰ ਕੀਤਾ

ਹੇਠਾਂ ਡਾਇਗ੍ਰਾਮ ਵਿੱਚ ਦਿਖਾਇਆ ਗਿਆ ਹੈ ਕਿ Agentic ਸਕੋਪ ਫਾਇਲ ਤੋਂ ਰਿਪੋਰਟ ਵਰਕਫਲੋ ਵਿੱਚ ਸਾਂਝੀ ਯਾਦਸ਼ ਤਰ੍ਹਾਂ ਕਿਵੇਂ ਕੰਮ ਕਰਦਾ ਹੈ — FileAgent ਆਪਣਾ ਆਉਟਪੁੱਟ `fileContent` ਕੀ ਅਧੀਨ ਲਿਖਦਾ ਹੈ, ReportAgent ਉਹ ਪੜ੍ਹਦਾ ਹੈ ਅਤੇ ਆਪਣਾ ਆਉਟਪੁੱਟ `report` ਵਿੱਚ ਲਿਖਦਾ ਹੈ:

<img src="../../../translated_images/pa/agentic-scope.95ef488b6c1d02ef.webp" alt="Agentic Scope Shared Memory" width="800"/>

*Agentic ਸਕੋਪ ਸਾਂਝੀ ਯਾਦਸ਼ ਵਾਂਗ ਕੰਮ ਕਰਦਾ ਹੈ — FileAgent `fileContent` ਲਿਖਦਾ ਹੈ, ReportAgent ਇਸ ਨੂੰ ਪੜ੍ਹਦਾ ਹੈ ਅਤੇ `report` ਲਿਖਦਾ ਹੈ, ਅਤੇ ਤੁਹਾਡਾ ਕੋਡ ਅੰਤਿਮ ਨਤੀਜੇ ਨੂੰ ਪੜ੍ਹਦਾ ਹੈ।*

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // ਫਾਇਲਐਜੰਟ ਤੋਂ ਕੱਚਾ ਫਾਇਲ ਡੇਟਾ
String report = scope.readState("report");            // ਰਿਪੋਰਟਐਜੰਟ ਤੋਂ ਸਾਂਚਾਬੱਧ ਰਿਪੋਰਟ
```

**Agent Listeners** ਏਜੰਟ ਦੀ ਕਾਰਕਿਰਦਗੀ ਤੇ ਨਜ਼ਰ ਰੱਖਣ ਅਤੇ ਡੀਬੱਗਿੰਗ ਨੂੰ ਯੋਗ ਬਣਾਉਂਦੇ ਹਨ। ਡੈਮੋ ਵਿੱਚ ਜੋ ਕਦਮ-ਬਾਈ-ਕਦਮ ਆਉਟਪੁੱਟ ਤੁਸੀਂ ਵੇਖਦੇ ਹੋ, ਉਹ ਹਰ ਏਜੰਟ ਦੀ ਕਿਰਿਆਸ਼ੀਲਤਾ ਨਾਲ ਜੁੜੇ ਏਜੰਟ ਲਿਸਨਰ ਤੋਂ ਆਉਂਦਾ ਹੈ:
- **beforeAgentInvocation** - ਜਦੋਂ Supervisor ਕਿਸੇ ਏਜੰਟ ਨੂੰ ਚੁਣਦਾ ਹੈ ਤਾਂ ਕਾਲ ਕੀਤਾ ਜਾਂਦਾ ਹੈ, ਇਸ ਨਾਲ ਤੁਹਾਨੂੰ ਪਤਾ ਚਲਦਾ ਹੈ ਕਿ ਕਿਹੜਾ ਏਜੰਟ ਕਿਉਂ ਚੁਣਿਆ ਗਿਆ
- **afterAgentInvocation** - ਜਦੋਂ ਕੋਈ ਏਜੰਟ ਪੂਰਾ ਹੁੰਦਾ ਹੈ ਤਾਂ ਕਾਲ ਕੀਤਾ ਜਾਂਦਾ ਹੈ, ਅਤੇ ਇਸ ਦਾ ਨਤੀਜਾ ਦਿਖਾਉਂਦਾ ਹੈ
- **inheritedBySubagents** - ਜਦੋਂ ਇਹ ਸੱਚ ਹੁੰਦਾ ਹੈ, ਤਾਂ ਲਿਸਨਰ ਹੇਰਾਰਕੀ ਦੇ ਸਾਰੇ ਏਜੰਟਾਂ 'ਤੇ ਨਜ਼ਰ ਰੱਖਦਾ ਹੈ

ਹੇਠਾਂ ਡਾਇਗ੍ਰਾਮ ਪੂਰੇ Agent Listener ਲਾਈਫਸਾਈਕਲ ਨੂੰ ਦਰਸਾਉਂਦਾ ਹੈ, ਜਿਸ ਵਿੱਚ ਇਹ ਵੀ ਦਿਖਾਇਆ ਗਿਆ ਹੈ ਕਿ `onError` ਕਿਵੇਂ ਏਜੰਟ ਕਾਰਜਕਾਰੀ ਦੌਰਾਨ ਗਲਤੀਆਂ ਨੂੰ ਸੰਭਾਲਦਾ ਹੈ:

<img src="../../../translated_images/pa/agent-listeners.784bfc403c80ea13.webp" alt="Agent Listeners Lifecycle" width="800"/>

*Agent Listeners ਕਾਰਜਕਾਰੀ ਲਾਈਫਸਾਈਕਲ ਵਿੱਚ ਜਾ ਕੇ ਕੰਮ ਕਰਦੇ ਹਨ — ਜਦੋਂ ਏਜੰਟ ਸ਼ੁਰੂ ਹੁੰਦੇ ਹਨ, ਮੁਕੰਮਲ ਹੋ ਜਾਂਦੇ ਹਨ ਜਾਂ ਗਲਤੀਆਂ ਦਾ ਸਾਹਮਣਾ ਕਰਦੇ ਹਨ ਉਸ ਨੂੰ ਨਜ਼ਰ ਵਿੱਚ ਰੱਖਦੇ ਹਨ।*

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // ਸਾਰੇ ਸਬ-ਏਜੰਟਾਂ ਵੱਲ ਵਧਾਓ
    }
};
```

Supervisor ਪੈਟਰਨ ਤੋਂ ਬਿਨ੍ਹਾਂ, `langchain4j-agentic` ਮੌਡਿਊਲ ਕਈ ਸ਼ਕਤੀਸ਼ਾਲੀ ਵਰਕਫਲੋ ਪੈਟਰਨ ਪ੍ਰਦਾਨ ਕਰਦਾ ਹੈ। ਹੇਠਾਂ ਡਾਇਗ੍ਰਾਮ ਪੰਜੋ ਪੈਟਰਨਜ਼ ਦਿਖਾਉਂਦਾ ਹੈ — ਸਧਾਰਨ ਕ੍ਰਮਵਾਰ ਪਾਈਪਲਾਈਨਾਂ ਤੋਂ ਲੈ ਕੇ ਮਨੁੱਖ-ਇਨ-ਦ-ਲੂਪ ਮਨਜ਼ੂਰੀ ਵਰਕਫਲੋ ਤੱਕ:

<img src="../../../translated_images/pa/workflow-patterns.82b2cc5b0c5edb22.webp" alt="Agent Workflow Patterns" width="800"/>

*ਏਜੰਟਾਂ ਨੂੰ ਸਮਨ्वਿਤ ਕਰਨ ਲਈ ਪੰਜ ਵਰਕਫਲੋ ਪੈਟਰਨ — ਸਧਾਰਨ ਕ੍ਰਮਵਾਰ ਤੋਂ ਲੈ ਕੇ ਮਨੁੱਖ-ਇਨ-ਦ-ਲੂਪ ਮਨਜ਼ੂਰੀ ਵਰਕਫਲੋ ਤੱਕ।*

| ਪੈਟਰਨ | ਵਰਣਨ | ਇਸਤੇਮਾਲ ਕੇਸ |
|---------|-------------|----------|
| **Sequential** | ਏਜੰਟਾਂ ਨੂੰ ਕ੍ਰਮ ਵਿੱਚ ਚਲਾਓ, ਆਉਟਪੁੱਟ ਅਗਲੇ ਨੂੰ ਜਾਵੇ | ਪਾਈਪਲਾਈਨਾਂ: ਖੋਜ → ਵਿਸ਼ਲੇਸ਼ਣ → ਰਿਪੋਰਟ |
| **Parallel** | ਏਜੰਟਾਂ ਨੂੰ ਇਕੱਠੇ ਚਲਾਓ | ਸੁਤੰਤਰ ਕੰਮ: ਮੌਸਮ + ਖ਼ਬਰਾਂ + ਸਟਾਕ |
| **Loop** | ਹਾਲਤ ਪੂਰੀ ਹੋਣ ਤੱਕ ਦੁਹਰਾਓ | ਗੁਣਵੱਤਾ ਮਾਪ: ਸਕੋਰ ≥ 0.8 ਹੋਣ ਤੱਕ ਸੁਧਾਰ |
| **Conditional** | ਸ਼ਰਤਾਂ ਦੇ ਆਧਾਰ 'ਤੇ ਰੂਟ ਕਰਨਾ | ਵਰਗੀਕਰਨ → ਵਿਸ਼ੇਸ਼ਜ્ઞ ਏਜੰਟ ਨੂੰ ਰੂਟਿੰਗ |
| **Human-in-the-Loop** | ਮਨੁੱਖੀ ਚੈਕਪੌਇੰਟ ਸ਼ਾਮਿਲ ਕਰੋ | ਮਨਜ਼ੂਰੀ ਵਰਕਫਲੋ, ਸਮੱਗਰੀ ਸਮੀਖਿਆ |

## ਮੁੱਖ ਧਾਰਨਾਵਾਂ

ਹੁਣ ਜਦੋਂ ਤੁਸੀਂ MCP ਅਤੇ agentic ਮੌਡਿਊਲ ਨੂੰ ਕਾਰਜ ਵਿਚ ਵੇਖ ਲਿਆ ਹੈ, ਆਓ ਹਰ ਪਹੁੰਚ ਦਾ ਹਮੇ ਸ਼ਨਾਖਤ ਕਰੀਏ।

MCP ਦਾ ਸਭ ਤੋਂ ਵੱਡਾ ਫਾਇਦਾ ਇਹ ਹੈ ਕਿ ਇਸ ਦੀ ਸਮੱਗਰੀ ਪ੍ਰਣਾਲੀ ਤੇਜ਼ੀ ਨਾਲ ਵਧ ਰਹੀ ਹੈ। ਹੇਠਾਂ ਡਾਇਗ੍ਰਾਮ ਦਿਖਾਉਂਦਾ ਹੈ ਕਿ ਇਕ ਸਿੰਗਲ ਯੂਨੀਵਰਸਲ ਪ੍ਰੋਟੋਕੋਲ ਕਿਵੇਂ ਤੁਹਾਡੇ AI ਐਪਲੀਕੇਸ਼ਨ ਨੂੰ ਵੱਖ-ਵੱਖ MCP ਸਰਵਰਾਂ ਨਾਲ ਜੋੜਦਾ ਹੈ — ਫਾਇਲਸਿਸਟਮ ਅਤੇ ਡੇਟਾਬੇਸ ਪਹੁੰਚ ਤੋਂ ਲੈ ਕੇ GitHub, ਈਮੇਲ, ਵੈੱਬ ਸਕ੍ਰੈਪਿੰਗ ਅਤੇ ਹੋਰ:

<img src="../../../translated_images/pa/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="MCP Ecosystem" width="800"/>

*MCP ਇੱਕ ਯੂਨੀਵਰਸਲ ਪ੍ਰੋਟੋਕੋਲ ਸਮੱਗਰੀ ਪ੍ਰਣਾਲੀ ਬਣਾਉਂਦਾ ਹੈ — ਕੋਈ ਵੀ MCP-ਕੰਪੈਟਿਬਲ ਸਰਵਰ ਕਿਸੇ ਵੀ MCP-ਕੰਪੈਟਿਬਲ ਕਲਾਇੰਟ ਨਾਲ ਕੰਮ ਕਰਦਾ ਹੈ, ਐਪਲੀਕੇਸ਼ਨਾਂ ਦਰਮਿਆਨ ਟੂਲ ਸਾਂਝਾ ਕਰਨ ਦੇ ਯੋਗ।*

**MCP** ਉਦੋਂ ਬਿਹਤਰ ਹੈ ਜਦੋਂ ਤੁਸੀਂ ਮੌਜੂਦਾ ਟੂਲ ecosystem ਦੀ ਵਰਤੋਂ ਕਰਨੀ ਹੋਵੇ, ਉਹ ਟੂਲ ਬਣਾਉਣੇ ਜੋ ਕਈ ਐਪਲੀਕੇਸ਼ਨਾਂ ਵਿੱਚ ਸਾਂਝੇ ਕਰ ਸਕਦੇ ਹਨ, ਤੀਸਰੇ ਪੱਖ ਦੀਆਂ ਸੇਵਾਵਾਂ ਨੂੰ ਮਿਆਰੀ ਪ੍ਰੋਟੋਕੋਲ ਨਾਲ ਜੋੜਨਾ ਜਾਂ ਕੋਡ ਨੂੰ ਬਦਲੇ ਬਿਨਾਂ ਟੂਲਾਂ ਨੂੰ ਬਦਲਣਾ ਚਾਹੁੰਦੇ ਹੋ।

**Agentic Module** ਬਿਹਤਰ ਹੁੰਦਾ ਹੈ ਜਦੋਂ ਤੁਸੀਂ `@Agent` ਐਨੋਟੇਸ਼ਨਾਂ ਨਾਲ ਘੋਸ਼ਣਾ-ਮੁੱਖ ਏਜੰਟ ਪਰਿਭਾਸ਼ਾਵਾਂ ਚਾਹੁੰਦੇ ਹੋ, ਵਰਕਫਲੋ ਸਮਨਵਯ (ਕ੍ਰਮਵਾਰ, ਲੂਪ, ਪੈਰਾਲਲ) ਦੀ ਲੋੜ ਹੋਵੇ, ਕੋਡ ਦੇ ਇੰਟੇਰਫੇਸ-ਅਧਾਰਿਤ ਏਜੰਟ ਡਿਜ਼ਾਈਨ ਨੂੰ ਤਰਜੀਹ ਦਿੰਦੇ ਹੋ ਜਾਂ ਕਈ ਏਜੰਟਾਂ ਦੇ ਆਉਟਪੁੱਟ `outputKey` ਰਾਹੀਂ ਸਾਂਝੇ ਕਰਨ ਵਾਲੇ ਸੰਯੋਜਨਾਂ ਨੂੰ ਮਿਲਾ ਰਹੇ ਹੋ।

**Supervisor Agent ਪੈਟਰਨ** ਉੱਭਰਦਾ ਹੈ ਜਦੋਂ ਵਰਕਫਲੋ ਐਡਵਾਂਸ ਤੋਂ ਪਤਾ ਨਹੀਂ ਹੁੰਦਾ ਅਤੇ ਤੁਹਾਨੂੰ LLM ਨੂੰ ਫੈਸਲਾ ਲੈਣਾ ਚਾਹੀਦਾ ਹੈ, ਜਦੋਂ ਤੁਹਾਡੇ ਕੋਲ ਕਈ ਵਿਸ਼ੇਸ਼ ਏਜੰਟ ਹਨ ਜੋ ਗਤੀਸ਼ੀਲ ਸਮਨਵਯ ਦੀ ਲੋੜ ਰੱਖਦੇ ਹਨ, ਗੱਲ-ਬਾਤੀ ਪ੍ਰਣਾਲੀਆਂ ਬਣਾਉਂਦੇ ਹੋ ਜੋ ਵੱਖ-ਵੱਖ ਸਮਰੱਥਾਵਾਂ ਵੱਲ ਰੁੱਟ ਕਰਦੀਆਂ ਹਨ, ਜਾਂ ਜਦੋਂ ਤੁਸੀਂ ਸਭ ਤੋਂ ਲਚਕੀਲਾ, ਅਨੁਕੂਲ ਏਜੰਟ ਵਿਹਾਰ ਚਾਹੁੰਦੇ ਹੋ।

ਆਪਣੇ Module 04 ਦੇ ਕਸਟਮ `@Tool` ਤਰੀਕਿਆਂ ਅਤੇ ਇਸ ਮੌਡਿਊਲ ਦੇ MCP ਟੂਲਾਂ ਵਿਚਕਾਰ ਚੋਣ ਕਰਨ ਲਈ, ਹੇਠਾਂ ਤੁਲਨਾ ਕੀਤੀ ਗਈ ਹੈ — ਕਸਟਮ ਟੂਲ ਤੁਹਾਡੇ ਐਪ-ਵਿਸ਼ੇਸ਼ ਲਾਜਿਕ ਲਈ ਸਖਤ ਜੋੜ ਅਤੇ ਪੂਰਨ ਟਾਈਪ ਸੇਫਟੀ ਦਿੰਦੇ ਹਨ, ਜਦਕਿ MCP ਟੂਲ ਮਿਆਰੀਕ੍ਰਿਤ, ਦੁਹਰਾਏ ਜਾ ਸਕਣ ਵਾਲੇ ਇੰਟਿਗ੍ਰੇਸ਼ਨ ਪ੍ਰਦਾਨ ਕਰਦੇ ਹਨ:

<img src="../../../translated_images/pa/custom-vs-mcp-tools.c4f9b6b1cb65d8a1.webp" alt="Custom Tools vs MCP Tools" width="800"/>

*ਕਦੋਂ ਕਸਟਮ @Tool ਤਰੀਕੇ ਵਰਤੋਂ ਅਤੇ ਕਦੋਂ MCP ਟੂਲਾਂ ਦੀ ਵਰਤੋਂ — ਕਸਟਮ ਟੂਲ ਐਪ-ਵਿਸ਼ੇਸ਼ ਲਾਜਿਕ ਲਈ ਪੂਰੀ ਟਾਈਪ ਸੇਫਟੀ ਨਾਲ, MCP ਟੂਲ ਮਿਆਰੀ ਇੰਟਿਗ੍ਰੇਸ਼ਨਾਂ ਲਈ ਜੋ ਕਈ ਐਪਲੀਕੇਸ਼ਨਾਂ ਵਿਚ ਕੰਮ ਕਰਦੇ ਹਨ।*
