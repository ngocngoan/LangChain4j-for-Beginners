## நீங்கள் கற்றுக்கொள்ளப்போகும் விஷயங்கள்

முந்தைய தொகுதியில், நினைவகம் உரையாடல் செயற்கை நுண்ணறிவை எவ்வாறு சாத்தியப்படுத்துகிறது என்பதைப் பார்த்தீர்கள் மற்றும் அடிப்படையான தொடர்புகளுக்காக GitHub மாதிரிகளை பயன்படுத்தினீர்கள். இப்போது நாம் கவனம் செலுத்துவோம் நீங்கள் கேள்விகளை எப்படி கேட்கிறீர்கள் - ப்ராம்ட் கள் - Azure OpenAI இன் GPT-5.2 மூலம். உங்கள் ப்ராம்டின் அமைப்பு பெறும் பதில்களின் தரத்தைக் குறிப்பிடத்தக்க ලෙස பாதிக்கிறது.

நாம் GPT-5.2 ஐ பயன்படுத்துவோம் ஏனெனில் அது காரணவியல் கட்டுப்பாட்டை அறிமுகப்படுத்துகிறது - பதில் அளிப்பதற்கு முன் எவ்வளவு யோசிக்க வேண்டும் என்பதை நீங்கள் மாதிரியிடம் சொல்ல முடியும். இது வேறு ப்ராம்டிங் நுட்பங்களை தெளிவுபடுத்துகிறது மற்றும் எந்த முறையை எப்போது பயன்படுத்த வேண்டும் என்பதைக் காட்டுகிறது. மேலும் GPT-5.2க்கு Azure இன் குறைந்த வீத வரம்புகள் GitHub மாதிரிகளுக்கு ஒப்பிடுகையில் நமக்கு நன்மை தரும்.

## ப்ராம்ட் இன்ஜினியரிங் புரிதல்

ப்ராம்ட் இன்ஜினியரிங் என்பது நீங்கள் தேவைப்படும் முடிவுகளை தொடர்ந்து பெறும் உள்ளீட்டுப் பதிவை வடிவமைப்பது ஆகும். இது வெறும் கேள்வி கேட்டு முடிவதல்ல - இது மாதிரி நீங்கள் என்ன வேண்டுமோ அதையும் எப்படி வழங்க வேண்டும் என்பதையும் சரியாக புரிந்து கொள்வதற்கான கோரிக்கைகளை உருவாக்குவதே ஆகும்.

இதுபோல் நினைத்துக் கொள்ளுங்கள்: ஒரு உறவு வல்லுநருக்கு அறிவுறுத்தல்கள் வழங்குவது போல. "பிழையை சரி செய்" என்பதற்கு சுட்டுமொழி குறைவாகும். "UserService.java வரி 45 இல் null pointer exception ஐ null சரிபார்ப்புடன் சரி செய்" என்பது தெளிவானதாகும். மொழி மாதிரிகள் இதையே பின்பற்றுகின்றன - தெளிவுத்தன்மை மற்றும் கட்டமைப்பு முக்கியம்.

## இது எப்படி LangChain4j-ஐ பயன்படுத்துகிறது

இந்த தொகுதி முன்பு உள்ள தொகுதிகளில் இருந்து அதே LangChain4j அடித்தளத்தைப் பயன்படுத்தி மேம்பட்ட ப்ராம்டிங் மாதிரிகளை காட்டுகிறது, ப்ராம்ட் கட்டமைப்பு மற்றும் காரணவியல் கட்டுப்பாட்டில் கவனம் செலுத்துகிறது.

<img src="../../../translated_images/ta/langchain4j-flow.48e534666213010b.webp" alt="LangChain4j Flow" width="800"/>

*LangChain4j எப்படி உங்கள் ப்ராம்டுகளை Azure OpenAI GPT-5.2-க்கு இணைக்கிறது*

**உருப்படிகள்** - Module 02 கீழ்காணும் langchain4j உறவைங்களைக் பயன்படுத்துகிறது `pom.xml`ல் குறிப்பிடப்பட்டவை:
```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j</artifactId> <!-- Inherited from BOM in root pom.xml -->
</dependency>
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-open-ai-official</artifactId> <!-- Inherited from BOM in root pom.xml -->
</dependency>
```

**OpenAiOfficialChatModel அமைப்பு** - [LangChainConfig.java](../../../02-prompt-engineering/src/main/java/com/example/langchain4j/prompts/config/LangChainConfig.java)

அந்த உரையாடல் மாதிரி OpenAI அதிகாரப்பூர்வ கிளையண்ட் பயன்படுத்தியும், Azure OpenAI முடிவுகளை ஆதரவுசெய்யும் Spring பீன் மூலம் கைப்பற்றப்பட்டு அமைக்கப்பட்டுள்ளது. Module 01-இல் இருந்து முக்கிய வேறுபாடு `chatModel.chat()` க்கு அனுப்பப்படும் ப்ராம்டுகளின் கட்டமைப்பில் உள்ளது, மாதிரி அமைப்பில் அல்ல.

**கணினி மற்றும் பயனர் செய்திகள்** - [Gpt5PromptService.java](../../../02-prompt-engineering/src/main/java/com/example/langchain4j/prompts/service/Gpt5PromptService.java)

LangChain4j தெளிவுக்காக செய்தி வகைகளை பிரிக்கிறது. `SystemMessage` செயற்கை நுண்ணறிவின் நடத்தை மற்றும் சூழலை அமைக்கிறது (உதாரணமாக "நீங்கள் ஒரு குறியீடு மதிப்பாய்வாளர்" என்று), மற்றும் `UserMessage` உண்மையான கோரிக்கையை கொண்டுள்ளது. இது பல்வேறு பயனர் கேள்விகளுக்கு ஒரு மாதிரிக் கருத்தாமையை பராமரிக்க உதவுகிறது.

```java
SystemMessage systemMsg = SystemMessage.from(
    "You are a helpful Java programming expert."
);

UserMessage userMsg = UserMessage.from(
    "Explain what a List is in Java"
);

String response = chatModel.chat(systemMsg, userMsg);
```

<img src="../../../translated_images/ta/message-types.93e0779798a17c9d.webp" alt="Message Types Architecture" width="800"/>

*SystemMessage நிலையான சூழலை வழங்குகிறது, UserMessages தனிப்பட்ட கோரிக்கைகளை உள்ளடக்குகிறது*

**MessageWindowChatMemory பல முறை உரையாடலுக்கு** - பல முறை உரையாடல் மாதிரி `MessageWindowChatMemory` Module 01 இலிருந்து மீண்டும் பயன்படுத்தப்படுகிறது. ஒவ்வொரு அமர்வுக்கும் தனித்த நினைவக எடுத்துக்காட்டு `Map<String, ChatMemory>` இல் சேமிக்கப்படுகிறது, இது பல இணைந்த உரையாடல்களை சூழல் கலக்கும் இல்லாமல் செயல்படுத்த உதவுகிறது.

**ப்ராம்ட் வார்ப்புருக்கள்** - இங்கு உண்மையான கவனம் ப்ராம்ட் இன்ஜினியரிங்கில் உள்ளது, புதிய LangChain4j API கள் இல்லை. ஒவ்வொரு மாதிரியிலும் (குறைந்த ஆர்வம், அதிக ஆர்வம், பணி செயலாக்கம், போன்றவை) அதே `chatModel.chat(prompt)` முறையைப் பயன்படுத்துகிறது ஆனால் கவனமாக கட்டமைக்கப்பட்ட ப்ராம்ட் சரங்களை உட்படுத்துகிறது. XML குறிச்சொற்கள், வழிமுறைகள் மற்றும் வடிவமைப்புகள் அனைத்தும் ப்ராம்ட் உரையின் பகுதியாக இருக்கும், LangChain4j பயன்பாட்டுகள் அல்ல.

**காரணவியல் கட்டுப்பாடு** - GPT-5.2 இன் காரணவியல் முயற்சி "அதிகபட்சம் 2 காரணவியல் படிகள்" அல்லது "முழுமையாக ஆராய்க" போன்ற ப்ராம்ட் வழிமுறைகள் மூலம் கட்டுப்படுத்தப்படுகிறது. இவை ப்ராம்ட் இன்ஜினியரிங் நுட்பங்கள், LangChain4j அமைப்புகள் அல்ல. நூலகம் உங்கள் ப்ராம்டுகளை மாதிரிக்கு வழங்குகிறது.

முக்கியமான வார்த்தை: LangChain4j கட்டமைப்பை வழங்குகிறது ([LangChainConfig.java](../../../02-prompt-engineering/src/main/java/com/example/langchain4j/prompts/config/LangChainConfig.java) மூலம் மாதிரி இணைப்பு, நினைவகம், [Gpt5PromptService.java](../../../02-prompt-engineering/src/main/java/com/example/langchain4j/prompts/service/Gpt5PromptService.java) மூலம் செய்தி கையாளல்), ஆனால் இந்த தொகுதி அந்த கட்டமைப்புக்குள் பயனுள்ள ப்ராம்டுகளை எவ்வாறு உருவாக்குவது என்பதை கற்பிக்கிறது.

## முக்கிய மாதிரிகள்

அனைத்து பிரச்சனைகளுக்கும் ஒரே அணுகுமுறை தேவையில்லை. சில கேள்விகளுக்கு விரைவான பதில்கள் வேண்டும், பிறவற்றுக்கு ஆழமான சிந்தனை தேவை. சில தோற்றத்தில் காரணவியலும் தேவை, சிலக்கு வெறும் முடிவுகள் வேண்டும். இந்த தொகுதி எட்டுப் ப்ராம்ட் இன்ஜினியரிங் மாதிரிகளை உள்ளடக்குகிறது - ஒவ்வொன்றும் வேறு சூழ்நிலைக்கு உயர்தரமாக வடிவமைக்கப்பட்டது. நீங்கள் அனைத்தையும் பரிசோதித்து, எப்போது எது சிறந்தது என்று அறிந்து கொள்வீர்கள்.

<img src="../../../translated_images/ta/eight-patterns.fa1ebfdf16f71e9a.webp" alt="Eight Prompting Patterns" width="800"/>

*எட்டுப் ப்ராம்ட் இன்ஜினியரிங் மாதிரிகள் மற்றும் அவற்றின் பயன்பாடுகள் கண்காட்சி*

<img src="../../../translated_images/ta/reasoning-effort.db4a3ba5b8e392c1.webp" alt="Reasoning Effort Comparison" width="800"/>

*குறைந்த ஆர்வம் (விரைவு, நேரடி) vs அதிக ஆர்வம் (ஆழமான, ஆராய்ச்சிப்பூர்வமான) காரணவியல் அணுகுமுறைகள்*

**குறைந்த ஆர்வம் (விரைவு மற்றும் கவனமுடையது)** - எளிய கேள்விகளுக்கு விரைவில் நேரடி பதில்களை வேண்டுமெனில். மாதிரி குறைந்ததாக காரணவியலைச் செய்கிறது - அதிகபட்சம் 2 படிகள். கணக்கீடு, தேடல் அல்லது நேரடி கேள்விகளுக்கு இதைப் பயன்படுத்தவும்.

```java
String prompt = """
    <reasoning_effort>low</reasoning_effort>
    <instruction>maximum 2 reasoning steps</instruction>
    
    What is 15% of 200?
    """;

String response = chatModel.chat(prompt);
```

> - "குறைந்த ஆர்வம் மற்றும் அதிக ஆர்வம் ப்ராம்டிங் மாதிரிகளுக்கு என்ன வேறுபாடு?"
> - "ப்ராம்ட்களில் XML குறிச்சொற்கள் AI பதிலை எப்படி அமைக்க உதவுகின்றன?"
> - "சுய-பரிசீலனை மாதிரிகளை நேரடி அறிவுரைக்கு மேல் எப்போது பயன்படுத்தவேண்டும்?"

**அதிக ஆர்வம் (ஆழமான மற்றும் முழுமையானது)** - சிக்கலான பிரச்சனைகளுக்கு முழுமையான பகுப்பாய்வை வேண்டுமெனில். மாதிரி முழுமையாக ஆராய்கிறது மற்றும் விரிவான காரணவியலைக் காண்பிக்கிறது. அமைப்பு வடிவமைப்பு, கட்டமைப்பு முடிவுகள் அல்லது கடுமையான ஆராய்ச்சிக்குப் பயன்படுத்தவும்.

```java
String prompt = """
    <reasoning_effort>high</reasoning_effort>
    <instruction>explore thoroughly, show detailed reasoning</instruction>
    
    Design a caching strategy for a high-traffic REST API.
    """;

String response = chatModel.chat(prompt);
```

**பணி செயலாக்கம் (படி படியான முன்னேற்றம்)** - பல படி பணிகளுக்குத். மாதிரி முன்னதாக திட்டமிடுகிறது, ஒவ்வொரு படியையும் செயல்படுத்தும் போது விளக்குகிறது, பின்னால் சுருக்கம் தருகிறது. பரிமாற்றங்கள், செயலாக்கங்கள் அல்லது பல படி செயல்முறைகளுக்கு இதைப் பயன்படுத்தவும்.

```java
String prompt = """
    <task>Create a REST endpoint for user registration</task>
    <preamble>Provide an upfront plan</preamble>
    <narration>Narrate each step as you work</narration>
    <summary>Summarize what was accomplished</summary>
    """;

String response = chatModel.chat(prompt);
```

சிந்தனைத் தொடர்(p) ப்ராம்டிங் மாதிரி மாதிரியிடம் அதன் காரணவியல் செயல்முறையை வெளிக்காட்டு கேட்கிறது, இதனால் கடுமையான பணிகளின் துல்லியம் மேம்படுகிறது. படி படியான உடைச்சல் மனிதரும் AIயும் தர்க்கத்தை புரிய உதவுகிறது.

> **🤖 [GitHub Copilot](https://github.com/features/copilot) Chat உடன் முயற்சிக்கவும்:** இந்த மாதிரியைக் கேளுங்கள்:
> - "நீண்ட கால இயங்கும் பணிகளுக்கு பணி செயலாக்க மாதிரியை எப்படி மாற்றுவேன்?"
> - "உற்பத்தி பயன்பாடுகளில் கருவி முன்னுரையோடு கட்டமைப்பது சிறந்த நடைமுறைகள் என்ன?"
> - "UIவில் நடுநிலை முன்னேற்றத்தை எப்படி பிடிக்கவும், காண்பிக்கவும் செய்கிறார்?"

<img src="../../../translated_images/ta/task-execution-pattern.9da3967750ab5c1e.webp" alt="Task Execution Pattern" width="800"/>

*திட்டமிடு → செயல்படு → சுருக்கு பணிக்கான வேலைப்பாடு*

**சுய-பரிசீலனை குறியீடு** - உற்பத்தி தரமான குறியீடு உருவாக்கிக்கொள்வதற்கு. மாதிரி குறியீடு உருவாக்குகிறது, தர நிபந்தனைகளுடன் ஒப்பிடுகிறது, மற்றும் தொடர்ந்து மேம்படுத்துகிறது. புதிய அம்சங்கள் அல்லது சேவைகளை உருவாக்கும் போது இதைப் பயன்படுத்தவும்.

```java
String prompt = """
    <task>Create an email validation service</task>
    <quality_criteria>
    - Correct logic and error handling
    - Best practices (clean code, proper naming)
    - Performance optimization
    - Security considerations
    </quality_criteria>
    <instruction>Generate code, evaluate against criteria, improve iteratively</instruction>
    """;

String response = chatModel.chat(prompt);
```

<img src="../../../translated_images/ta/self-reflection-cycle.6f71101ca0bd28cc.webp" alt="Self-Reflection Cycle" width="800"/>

*தொடர் மேம்பாட்டு சுற்று - உருவாக்கு, மதிப்பாய்வு செய், பிரச்சனைகளை கண்டறி, மேம்படு, மீண்டும் செய்*

**கட்டமைக்கப்பட்ட பகுப்பாய்வு** - ஒழுங்கான மதிப்பீட்டிற்கு. மாதிரி குறியீட்டை நிரந்தர அமைப்பை (சரியானது, நடைமுறை, செயல்திறன், பாதுகாப்பு) பயன்படுத்தி மதிப்பாய்வு செய்கிறது. குறியீடு மதிப்பாய்வுகள் அல்லது தர அங்கீகாரங்களுக்கு இதைப் பயன்படுத்தவும்.

```java
String prompt = """
    <code>
    public List getUsers() {
        return database.query("SELECT * FROM users");
    }
    </code>
    
    <framework>
    Evaluate using these categories:
    1. Correctness - Logic and functionality
    2. Best Practices - Code quality
    3. Performance - Efficiency concerns
    4. Security - Vulnerabilities
    </framework>
    """;

String response = chatModel.chat(prompt);
```

> **🤖 [GitHub Copilot](https://github.com/features/copilot) Chat உடன் முயற்சிக்கவும்:** கட்டமைக்கப்பட்ட பகுப்பாய்வைப் பற்றி கேளுங்கள்:
> - "வெவ்வேறு வகை குறியீடு மதிப்பாய்வுகளுக்கு பகுப்பாய்வு அமைப்பை எப்படி தனிப்பயனாக்க முடியும்?"
> - "கட்டமைக்கப்பட்ட வெளியீட்டை நிரல்பூர்வமாக சொடுக்கவும் செயல்படுத்தவும் சிறந்தவாறு எப்படி?"
> - "விவித மதிப்பாய்வு அமர்வுகளில் ஒரேமாதிரி தீவிரத்தன்மை நிலைகளை எவ்வாறு உறுதி செய்வது?"

<img src="../../../translated_images/ta/structured-analysis-pattern.0af3b690b60cf2d6.webp" alt="Structured Analysis Pattern" width="800"/>

*தீவிரத்தன்மை நிலைகளுடன் ஒரே மாதிரி குறியீடு மதிப்பாய்வு நான்கு வகுப்பு அமைப்பு*

**பல்வேறு முறை உரையாடல்** - சூழலைத் தேவைப்படுத்தும் உரையாடல்களுக்கு. மாதிரி முந்தைய செய்திகள் நினைவில் வைக்கப்பட்டு கூட்டிக் கொள்ளப்படுகிறது. தொடர்பு உதவி அமர்வுகள் அல்லது கடுமையான கேள்வி & பதில்களுக்கு இதைப் பயன்படுத்தவும்.

```java
ChatMemory memory = MessageWindowChatMemory.withMaxMessages(10);

memory.add(UserMessage.from("What is Spring Boot?"));
AiMessage aiMessage1 = chatModel.chat(memory.messages()).aiMessage();
memory.add(aiMessage1);

memory.add(UserMessage.from("Show me an example"));
AiMessage aiMessage2 = chatModel.chat(memory.messages()).aiMessage();
memory.add(aiMessage2);
```

*ஒரு பல்வேறு முறைகளின் உரையாடல் சூழல் சீர் வரையில் சேர்க்கப்படுகிறது*

**படி படியாக காரணதீர்வு** - துவக்க உதவி பொருளாக்கமும் தெரிவிக்கிறது. மாதிரி ஒவ்வொரு படியினதும் தெளிவான காரணவியலைக் காட்டுகிறது. கணிதக் கேள்விகள், தர்க்க புதிர்கள் அல்லது சிந்தனை செயல்முறையை புரிந்து கொள்ள வேண்டிய கேள்விகளுக்கு இதைப் பயன்படுத்தவும்.

```java
String prompt = """
    <instruction>Show your reasoning step-by-step</instruction>
    
    If a train travels 120 km in 2 hours, then stops for 30 minutes,
    then travels another 90 km in 1.5 hours, what is the average speed
    for the entire journey including the stop?
    """;

String response = chatModel.chat(prompt);
```

*பிரச்சனைகளை தெளிவான தர்க்க படிகளாக உடைத்தல்*

**கட்டுப்படுத்தப்பட்ட வெளியீடு** - குறிப்பிட்ட வடிவமைப்புடனான பதில்களுக்கு. மாதிரி வடிவம் மற்றும் நீளம் விதிகளை கடுமையாக பின்பற்றுகிறது. சுருக்கங்கள் அல்லது கணக்கு வடிவமைப்புக்கு இதைப் பயன்படுத்தவும்.

```java
String prompt = """

    - Exactly 100 words
    - Bullet point format

    Summarize the key concepts of machine learning.
    """;

String response = chatModel.chat(prompt);
```

**விருப்பம் 1: Spring Boot டாஷ்போர்டு பயன்படுத்துதல் (VS Code பயனர்களுக்கு பரிந்துரைக்கபடும்)**

டெவ் 컨்டெய்னர் Spring Boot டாஷ்போர்டு விரிவாக்கத்தை உட்படுத்தி உள்ளது, இது அனைத்து Spring Boot பயன்பாடுகளையும்視覺த் தெரியுமாறு நிர்வகிக்க உதவுகிறது. இது VS Code இன் இடது Activity பார்(பாரில்) உள்ளது (Spring Boot ஐகானைப் பாருங்கள்).

Spring Boot டாஷ்போர்டு மூலம் நீங்கள்:
- பணியிடத்தில் உள்ள அனைத்து Spring Boot பயன்பாடுகளையும் காணலாம்
- ஒரு கிளிக்கில் பயன்பாடுகளை துவக்க/நிறுத்தலாம்
- ஆப்டிம் நேர பயன்பாட்டு பதிவுகளைக் காணலாம்
- பயன்பாடு நிலையை கண்காணிக்கலாம்

"prompt-engineering" ஐ அருகில் உள்ள பிளே பொத்தானை அழுத்துவதன் மூலம் இந்த தொகுதியை துவக்கலாம், அல்லது அனைத்து தொகுதிகளையும் ஒரே நேரத்தில் துவக்கலாம்.

*முக்கிய டாஷ்போர்டு, எட்டு ப்ராம்ட் இன்ஜினியரிங் மாதிரிகளையும் அவற்றின் பண்புகள் மற்றும் பயன்பாடுகளுடன் காண்பிக்கும்*

## மாதிரிகளை ஆராய்தல்

இணைய இடைமுகம் வேறுபட்ட ப்ராம்டிங் நுட்பங்களுடன் நீங்கள் பரிசோதனை செய்ய அனுமதிக்கிறது. ஒவ்வொரு மாதிரியும் வேறுபட்ட பிரச்சனைகளைத் தீர்க்கிறது - எப்போது எது சிறந்தது என்பதைக் கண்டறிய அவற்றைப் பயன்படுத்திக் கண்டு பிடியுங்கள்.

### குறைந்த vs அதிக ஆர்வம்

"200ன் 15% என்ன?" போன்ற எளிய கேள்வியை குறைந்த ஆர்வம் ப்ராம்டிங் மூலம் கேளுங்கள். நீங்கள் உடனடி நேரடி பதிலைப் பெறுவீர்கள். இப்போது "ஒரு உயர் போக்குவரத்து APIக்காக கேச்சிங் நெறிமுறையை வடிவமைக்க" என்ற சிக்கலான கேள்வியை அதிக ஆர்வம் ப்ராம்டிங் மூலம் கேளுங்கள். மாதிரி எப்படி மெதுவாக செயல்பட்டு விரிவான காரணவியலை வழங்குகிறது என்பதை கவனியுங்கள். ஒரே மாதிரி, ஒரே கேள்வி கட்டமைப்பு - ஆனால் ப்ராம்ட் எவ்வளவு சிந்தனை செய்யவேண்டும் என்பதை சொல்லுகிறது.
<img src="../../../translated_images/ta/low-eagerness-demo.898894591fb23aa0.webp" alt="குறைந்த உற்சாகம் காட்சி" width="800"/>

*குறைந்த காரணங்களை கொண்டு விரைவான கணக்கீடு*

<img src="../../../translated_images/ta/high-eagerness-demo.4ac93e7786c5a376.webp" alt="அதிக உற்சாகம் காட்சி" width="800"/>

*முழுமையான சேமிப்பு रणनीதி (2.8MB)*

### பணித் திறன் (கருவி முன்னுரை)

பன்மடங்கு படிகள் கொண்ட பணிகள் முன்கூட்டியே திட்டமிடல் மற்றும் முன்னேற்ற விளக்கத்தின் மூலம் பயன் பெறுகின்றன. மாடல் என்ன செய்யப்போகிறது என்பதையும், ஒவ்வொரு படியையும் விளக்கியும், பிறகு முடிவுகளை சுருக்கமாக கூறியும் செயல்படுகிறது.

<img src="../../../translated_images/ta/tool-preambles-demo.3ca4881e417f2e28.webp" alt="பணித் திறன் காட்சி" width="800"/>

*படி படியாக விளக்கத்துடன் REST முட்டியை உருவாக்குதல் (3.9MB)*

### சுய பரிசீலனை குறியீடு

"ஒரு மின்னஞ்சல் சரிபார்ப்பு சேவையை உருவாக்கு" என முயற்சிக்கவும். குறியீட்டை உருவாக்கி நிறுத்துவதைவிட, மாடல் உருவாக்கி, தரக் குறிகள் அடிப்படையில் மதிப்பாய்வு செய்து, பலவீனங்களை கண்டறிந்து மேம்படுத்தும். குறியீடு உற்பத்தி தரத்துடன் சேர்ந்தது வரை மீண்டும் மீண்டும் செயல்படுவதை நீங்கள் காணலாம்.

<img src="../../../translated_images/ta/self-reflecting-code-demo.851ee05c988e743f.webp" alt="சுய பரிசீலனை குறியீடு காட்சி" width="800"/>

*முழுமையான மின்னஞ்சல் சரிபார்ப்பு சேவை (5.2MB)*

### கட்டமைக்கப்பட்ட பகுப்பாய்வு

குறியீடு திருத்தங்களுக்கு ஒரே மாதிரியாக மதிப்பீட்டு கட்டமைப்புகள் அவசியம். மாடல் குறியீட்டை நிலைமை (தவறற்ற தன்மை, நடைமுறைகள், செயல்திறன், பாதுகாப்பு) போன்ற நிலையான வகைப் பிரிவுகளால் ஆராய்கிறது.

<img src="../../../translated_images/ta/structured-analysis-demo.9ef892194cd23bc8.webp" alt="கட்டமைக்கப்பட்ட பகுப்பாய்வு காட்சி" width="800"/>

*கட்டமைப்பின் அடிப்படையிலான குறியீடு விமர்சனம்*

### பன்மடங்குச் சந்திப்பு

"Spring Boot எது?" என்று கேட்டு மறுபடியும் "ஒரு எடுத்துக்காட்டை காட்டு" என கேட்கவும். மாடல் உங்கள் முதல் கேள்வியை நினைவில் வைத்துக்கொண்டு குறிப்பிட்ட Spring Boot எடுத்துக்காட்டை தரும். நினைவிறக்கம் இல்லாமல் இரண்டாவது கேள்வி அவ்வளவு தெளிவாகாது.

<img src="../../../translated_images/ta/multi-turn-chat-demo.0d2d9b9a86a12b4b.webp" alt="பன்மடங்கு சந்திப்பு காட்சி" width="800"/>

*கேள்விகள் இடையே சூழல் பாதுகாப்பு*

### படி படியாக காரணாலோசனை

ஒரு கணிதப் பிரச்சனையை எடுத்து, படி படியாக காரணாலோசனையும் குறைந்த உற்சாகத்தையும் பயன்படுத்தி முயற்சியுங்கள். குறைந்த உற்சாகம் முடிவை வேகமாக தரும் ஆனால் வெளிப்படையாக இல்லை. படிநிலை உண்மைகள் மற்றும் கணக்கீடுகளை முழுமையாக காட்டுகிறது.

<img src="../../../translated_images/ta/step-by-step-reasoning-demo.12139513356faecd.webp" alt="படி படியாக காரணாலோசனை காட்சி" width="800"/>

*தெரிவு செய்யப்பட்ட படிகளுடன் கணிதப் பிரச்சனை*

### கட்டுப்படுத்தப்பட்ட வெளியீடு

குறிப்பிட்ட வடிவங்கள் அல்லது வார்த்தை எண்ணிக்கை தேவைப்பட்டால், இந்த மாதிரி கடுமையான கட்டுப்பாட்டை இயற்றும். துல்லியமாக 100 வார்த்தைகளைக் கொண்ட புள்ளி வடிவ சுருக்கம் உருவாக்க முயற்சியுங்கள்.

<img src="../../../translated_images/ta/constrained-output-demo.567cc45b75da1633.webp" alt="கட்டுப்படுத்தப்பட்ட வெளியீடு காட்சி" width="800"/>

*வடிவ கட்டுப்பாட்டுடன் இயந்திரக் கற்றல் சுருக்கம்*

## நீங்கள் உண்மையில் கற்பது என்ன

**காரணாலோசனை முயற்சி அனைத்தையும் மாற்றுவோம்**

GPT-5.2 மூலம் உங்கள் கேட்டல்களால் கணக்கிடும் முயற்சியை நீங்கள் கட்டுப்படுத்தலாம். குறைந்த முயற்சி என்றால் குறைந்த ஆராய்ச்சியுடன் வேகமான பதில்கள். அதிக முயற்சி என்றால் மாடல் ஆழமாக யோசிக்க நேரம் எடுத்துக் கொள்கிறது. பணிக்கான சிக்கலுக்கு ஏற்ப முயற்சியை பொருத்திக் கற்றுக்கொள்கிறீர்கள் - எளிய கேள்விகளுக்கு நேரத்தை வீணடிக்காதீர்கள்; ஆனால் சிக்கலான முடிவுகளுக்கு விரைபடாதீர்கள்.

**கட்டமைப்பு நடத்தை வழிநடத்துகிறது**

பிராம்ப்டுகளின் XML குறியீடுகளை கவனியுங்கள்? அவை அலங்காரமாக அல்ல. மாடல்கள் கட்டமைக்கப்பட்ட வழிமுறைகளை சுதந்திர உரையின் விட அதிக நம்பகமாக பின்பற்றுகின்றன. பன்மடங்கு செயலிகள் அல்லது சிக்கலான நியமங்களுக்காக, கட்டமைப்பு மாடலை தற்போதைய இடம் மற்றும் அடுத்து என்ன ஆகிறது என்பதைப் பதிவுசெய்ய உதவுகிறது.

<img src="../../../translated_images/ta/prompt-structure.a77763d63f4e2f89.webp" alt="பிராம்ப்ட் கட்டமைப்பு" width="800"/>

*தெளிவான பிரிவுகள் மற்றும் XML-பாணி அமைப்புடன் நன்கு கட்டமைக்கப்பட்ட பிராம்ப்ட் உள்ளமைவு*

**தனிப்பரிசீலனையின் மூலம் தரம் உயர்வு**

சுய பரிசீலனை முறைவைகள் தரக் குறிகளை தெளிவுபடுத்துகின்றன. "சரியாக செய்கிறது" என்று நம்பாமல், நீங்கள் "சரி" என்பதற்கான பொருள்: சரியான தர்க்கம், பிழை கையாளல், செயல்திறன், பாதுகாப்பு என்பனவற்றைத் தெளிவுபடுத்துகிறீர்கள். பின்னர் மாடல் தனது வெளியீட்டை மதிப்பாய்வு செய்து மேம்படுத்தக்கூடியது. இது குறியீடு உருவாக்கத்தை பொருட்செய்யும் ஒரு செயலாக்கமாக மாற்றுகிறது.

**சூழல் அவசரமில்லை**

பன்மடங்கு உரையாடல்கள் ஒவ்வொரு கோரிக்கையுடனும் செய்தி வரலாற்றை கொண்டுள்ளது. ஆனால் எல்லைக்கு வரம்பு உள்ளது – ஒவ்வொரு மாடலுக்கும் அதிகபட்ச டோக்கன் எண்ணிக்கை. உரையாடல்கள் வளர்ந்தபோது, அந்த எல்லையை கடக்காமல் தொடர்புடைய சூழல் informஃபர்மேஷனை வைத்திருக்க நடைமுறைகள் தேவைப்படும். இந்த மாட்யூல் நினைவாற்றல் எப்படி வேலை செய்கிறது என்பதை காட்டுகிறது; பின்னர் எப்போது சுருக்கம் செய்யலாம், எப்போது மறக்கலாம், எப்போது மீண்டும் பெறலாம் என்பதைக் கற்றுக்கொள்வீர்கள்.
