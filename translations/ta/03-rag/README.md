## நீங்கள் கற்கப்போகும் விஷயங்கள்

முந்தைய மாடியூல்களில், நீங்கள் AI உடன் உரையாடல் செய்வதையும், உங்கள் வினாடிகளை சரியாக அமைப்பதையும் கற்றுக் கொண்டீர்கள். ஆனால் ஒரு அடிப்படையானக் குறைவு உண்டு: மொழி மாதிரிகள் பயிற்சியின் போது בלבד கற்றுக் கொண்டவை மட்டுமே தெரியும். அவர்கள் உங்கள் நிறுவனக் கொள்கைகள், உங்கள் திட்ட ஆவணங்கள் அல்லது பயிற்சி பெறாத பிற தகவல்களைப் பற்றி கேள்விகளுக்கு பதில் அளிக்க முடியாது.

RAG (வருட்டுதல்-ஆய்வுசெய்தல் உருவாக்கல்) இந்தப் பிரச்சனையை தீர்க்கிறது. மாதிரிக்கு உங்கள் தகவல்களைச் சொல்ல முயற்சிப்பதை விட (அது செலவானதும் நடைமறுவும்), நீங்கள் அதற்கு உங்கள் ஆவணங்களைத் தேடுவதற்கு திறன் அளிக்கிறீர்கள். ஒருவர் கேள்வி கேட்டால், அமைப்பு பொருத்தமான தகவலைக் கண்டறிந்து அதனைப் ப்ராமிடில் சேர்க்கிறது. பின்னர் மாதிரி அந்த ஆவணப்படுத்தப்பட்ட சூழலைப் பயன்படுத்தி பதிலளிக்கிறது.

RAG-ஐ மாதிரிக்கு ஒரு குறிப்ப-reference நூலகத்தை வழங்குவதாக நினைத்துக்கொள்ளுங்கள். நீங்கள் கேள்வி கேட்கும் போது, அமைப்பு:

1. **பயனர் வினா** - நீங்கள் ஒரு கேள்வி கேட்கின்றீர்கள்
2. **எம்பெடிங்** - உங்கள் கேள்வியை வெக்டர் ஆக மாற்றுவது
3. **வெக்டர் தேடல்** - ஒத்த ஆவண துண்டுகளை கண்டறிதல்
4. **சூழல் சேர்க்கை** - பொருத்தமான துண்டுகளை ப்ராமிடிற்கு சேர்க்கிறது
5. **பதில்** - LLM அந்த சூழலை அடிப்படையாகக் கொண்டு பதிலளிக்கிறது

இதனால் மாதிரியின் பதில்கள் அதன் பயிற்சி அறிவில் அல்லது பதில்களை உருவாக்குவதில் இருந்து விலகி நிஜ தரவிலிருந்து மூலம் பெறலாம்.

## RAG ஐ புரிந்துகொள்வது

கீழே காட்டப்படும் விளக்கப்படம் அடிப்படையான கருத்தை விளக்குகிறது: மாதிரியின் பயிற்சி தரவுக்கு மட்டுமல்ல, RAG உங்கள் ஆவணங்களின் குறிப்ப-reference நூலகத்தை அளித்து ஒவ்வொரு பதிலுக்கும் முன்பு அணுகுகிறது.

<img src="../../../translated_images/ta/what-is-rag.1f9005d44b07f2d8.webp" alt="RAG என்றால் என்ன" width="800"/>

பயனர் வினாவுக்கான நான்கு கட்டங்கள் — எம்பெடிங், வெக்டர் தேடல், சூழல் சேர்க்கை, பதில் உருவாக்கல் — ஒருங்கிணைந்தவாறு எப்படி செயல்படுகின்றன என்பதைக் காட்டுகிறது:

<img src="../../../translated_images/ta/rag-architecture.ccb53b71a6ce407f.webp" alt="RAG கட்டமைப்பு" width="800"/>

இந்த மாடியூலில் அடுத்த படியலில் ஒவ்வொன்றும் விரிவாகவும், நீங்கள் இயக்கி மாற்றக்கூடிய குறியீட்டுடன் செல்லப்படுகிறது.

## இது எப்படி வேலை செய்கிறது

### ஆவணம் செயலாக்கம்

[DocumentService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/DocumentService.java)

நீங்கள் ஆவணத்தை பதிவேற்றம் செய்தால், அமைப்பு அதனை (PDF அல்லது தெளிவான உரை) பகுத்து, கோப்பு பெயர் போன்ற மெட்டாடேட்டாவைச் சேர்க்கிறது, பின்னர் அதை சிறு துண்டுகளாகப் பிரிக்கிறது — மாதிரியின் சூழல் ஜன்னலில் சிறப்பாக இடம் பெறும் சிறிய துண்டுகள். இந்த துண்டுகள் சிறிது இருப்பு பகிர்ச்சியுடன் மாறாற் கிடைக்கின்றன, அதனால் எல்லைகளில் உள்ள சூழல் இழக்கப்படாது.

```java
// பதிவேற்றப்பட்ட கோப்பை பகுப்பாய்வு செய்து அதை LangChain4j ஆவணமாக மூடியுங்கள்
Document document = Document.from(content, metadata);

// 30-டோக்கன் மூலமாக இணைக்கப்பட்ட 300-டோக்கன் துண்டுகளாக பிரிக்கவும்
DocumentSplitter splitter = DocumentSplitters
    .recursive(300, 30);

List<TextSegment> segments = splitter.split(document);
```

கீழே காட்டப்படும் அட்டவணை இதைப் பார்வையிடுவதை விவரிக்கிறது. ஒவ்வொரு துண்டும் அதன் அக்கரையில் சில டோக்கன்களை பகிர்ந்து கொண்டிருக்கிறது — 30 டோக்கன் இடைவெளி எந்த முக்கியமான சூழலும் முறையடையாமல் உறுதி செய்கிறது:

<img src="../../../translated_images/ta/document-chunking.a5df1dd1383431ed.webp" alt="ஆவண துண்டாக்கல்" width="800"/>

> - "வேறு ஆவண வகைகளுக்கு சிறந்த துண்டு அளவு என்ன மற்றும் ஏன்?"

### எம்பெடிங்களை உருவாக்குதல்

[LangChainRagConfig.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/config/LangChainRagConfig.java)

ஒவ்வொரு துண்டும் எம்பெடிங் எனப்படும் எண் வடிவ ம representationல் மாற்றப்படுகிறது — உரையின் பொருளைப் பதிவு செய்வதற்கான கணித அடையாளம். ஒத்த உரைகள் ஒத்த எம்பெடிங்களை உண்டாக்கும்.

```java
@Bean
public EmbeddingModel embeddingModel() {
    return OpenAiOfficialEmbeddingModel.builder()
        .baseUrl(azureOpenAiEndpoint)
        .apiKey(azureOpenAiKey)
        .modelName(azureEmbeddingDeploymentName)
        .build();
}

EmbeddingStore<TextSegment> embeddingStore = 
    new InMemoryEmbeddingStore<>();
```

கீழே LangChain4j கூறுகள் எப்படி இணைக்கப்படும் என்பதைக் காட்டும் வகுப்புக் கட்டமைப்பு காணலாம். `OpenAiOfficialEmbeddingModel` உரையை வெக்டர்களாக மாற்றுகிறது, `InMemoryEmbeddingStore` வெக்டர்களை அவற்றின் இயல்பான `TextSegment` தரவுடன் வைத்திருக்கிறது, மற்றும் `EmbeddingSearchRequest` `maxResults` மற்றும் `minScore` போன்ற மீட்டெடுக்கல் கட்டுப்பாடுகளை கட்டுப்படுத்துகிறது:

<img src="../../../translated_images/ta/rag-langchain4j-classes.bbf3aa9077ab443d.webp" alt="LangChain4j RAG வகுப்புகள்" width="800"/>

எம்பெடிங்கள் சேமிக்கப்பட்டதும், ஒத்த உள்ளடக்கம் வெக்டர் இடத்தில் இயல்பாக கூடிக்கொள்கிறது. கீழே காட்டப்பட்டுள்ளது போல தொடர்புடைய தலைப்புக்கள் அருகிய புள்ளிகளாக உருவாக்கப்படுவதால் அர்த்ததேடல் சாத்தியமாகிறது:

### அர்த்தம் அடிப்படையிலான தேடல்

[RagService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/RagService.java)

நீங்கள் கேள்வி கேட்டபோது, அது ஒருங்குடன் பெற்று எம்பெடிங்காக மாற்றப்படுகிறது. அமைப்பு உங்கள் வினாவின் எம்பெடிங்கை அனைத்து ஆவண துண்டுகளின் எம்பெடிங்களுடன் ஒப்பிடுகிறது. அதற்கு மிகவும் பொருந்தும் துண்டுகளை தேடுகிறது — வெறும் முக்கிய வார்த்தைகள் மட்டுமன்றி, உண்மையான அர்த்த ஒத்திசைவைப் பார்க்கிறது.

```java
Embedding queryEmbedding = embeddingModel.embed(question).content();

EmbeddingSearchRequest searchRequest = EmbeddingSearchRequest.builder()
    .queryEmbedding(queryEmbedding)
    .maxResults(5)
    .minScore(0.5)
    .build();

EmbeddingSearchResult<TextSegment> searchResult = embeddingStore.search(searchRequest);
List<EmbeddingMatch<TextSegment>> matches = searchResult.matches();

for (EmbeddingMatch<TextSegment> match : matches) {
    String relevantText = match.embedded().text();
    double score = match.score();
}
```

கீழே விளக்கப்படத்தில் அர்த்ததேடல் மற்றும் பழைய முக்கிய வார்த்தை தேடலை ஒப்பிடுகிறது. "வாகனம்" என்ற வார்த்தை தேடுதல் "கார்கள் மற்றும் வளரிகள்" பற்றிய துண்டை தவறவிடும், ஆனாலும் அர்த்ததேடல் அதே பொருளை புரிந்து அதை மிக அதிக மதிப்பெண் பெற்று காட்டுகிறது:

## விண்ணப்பத்தை பயன்படுத்துதல்

விண்ணப்பம் ஆவண பதிவேற்றம் மற்றும் கேள்வி கேட்க இணைய முகப்பை வழங்குகிறது.

*RAG விண்ணப்ப இடைமுகம் - ஆவணங்களை பதிவேற்றி கேள்விகள் கேட்கவும்*

### ஆவணத்தை பதிவேற்றம் செய்யவும்

ஆரம்பமாக ஒரு ஆவணத்தை பதிவேற்றவும் - TXT கோப்புகள் சோதனைக்கு சிறந்தவை. இந்த அடைவிலே `sample-document.txt` நடுத்தர அளவில் LangChain4j அம்சங்கள், RAG நிறைவேற்றல் மற்றும் சிறந்த நடைமுறைகள் பற்றி உள்ளடக்கத்தை வழங்குகிறது - அமைப்பைப் சோதனை செய்ய சிறந்தது.

சாதனம் உங்கள் ஆவணத்தை செயலாக்கி அதை துண்டுகளாகப் பிரித்து, ஒவ்வொரு துண்டுக்கும் எம்பெடிங்களை உருவாக்கும். இது பதிவேற்றுவதுடன் தானாக நிகழும்.

### கேள்விகள் கேட்கவும்

இப்போது ஆவண உள்ளடக்கத்திற்கான சிறப்பான கேள்விகளை கேளுங்கள். ஆவணத்தில் தெளிவாகக் கூறப்பட்ட பொருத்தமான தரவுகளை நோக்கி அமைப்பு தேடல் செய்கிறது, அவற்றை ப்ராமிட்-இல் சேர்க்கிறது மற்றும் பதிலை உருவாக்குகிறது.

### மூல குறிப்பிடுக்களை சரிபார்க்கவும்

ஒவ்வொரு பதிலும் ஒத்திசைவு மதிப்பெண்களை கொண்ட மூலங்கள் குறிப்பிடப்படுகின்றன. இந்த மதிப்பெண்கள் (0 முதல் 1 வரை) உங்கள் கேள்விக்குப் பொருந்தினவை எவ்வளவு என்பதைக் காட்டுகின்றன. அதிக மதிப்பெண்கள் சிறந்த பொருத்தத்தை குறித்து குறிப்பிடுகிறது. இதனால் பதிலை மூல உத்தரவாதத்துடன் சரிபார்க்க முடியும்.

<a href="images/rag-query-results.png"><img src="../../../translated_images/ta/rag-query-results.6d69fcec5397f355.webp" alt="RAG வினா முடிவுகள்" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*மூல குறிப்பிடுக்களுடன் மற்றும் பொருத்த மதிப்பெண்களுடன் பதில் காட்டுகிறது*

### கேள்விகளுடன் சோதனை செய்யவும்

பல்வேறு வகை கேள்விகளை முயற்சிக்கவும்:
- தெளிவான உண்மைகள்: "முக்கிய தலைப்பு என்ன?"
- ஒப்பீடுகள்: "X மற்றும் Y-இல் என்ன வேறுபாடு?"
- சுருக்கங்கள்: "Z பற்றிய முக்கிய அம்சங்களை சுருக்குங்கள்"

உங்கள் கேள்வி ஆவண உள்ளடக்கத்துடன் எவ்வாறு பொருந்துகிறது என்பதன்படி பொருத்த மதிப்பெண்களில் ஏற்படும் மாற்றத்தை கவனிக்கவும்.

ஒவ்வொரு மாதிரிக்கும் மிக அதிக சூழல் ஜன்னல் உள்ளது. பெரிய ஆவணத்திலிருந்து அனைத்து துண்டுக்களையும் சேர்க்க முடியாது. அமைப்பு பொதுவாக முக்கியமான 5 துண்டுகளை மீட்டெடுத்து வரம்புகளுக்குள் இருக்கவும் சரியான பதிலுக்குப் போதுமான சூழலை வழங்கவும் செய்கிறது.

## எப்போது RAG முக்கியம்

RAG எப்பொழுது சரியான வழி என்பதை எப்பொழுது பயன்படுத்த வேண்டும் என்பது கீழ்காணும் முடிவு வழிகாட்டியால் உங்கள் தீர்மானத்திற்கு உதவும். நேரடியாக ப்ராமிடில் உள்ளடக்குவதல்லாமல், மாதிரியின் உள்ளமைவான அறிவுக்கு நம்பிக்கை வைப்பதற்கும் மாறாக RAG பயன்படும் நேரம்:

<img src="../../../translated_images/ta/when-to-use-rag.1016223f6fea26bc.webp" alt="எப்பொழுது RAG பயன்படுத்துவது" width="800"/>

**RAG-ஐ பயன்படுத்தவும்:**
- பணியாளர் ஆவணங்கள் குறித்த கேள்விகளுக்கு பதில் அளித்தல்  
- தகவல்கள் அடிக்கடி மாறுபடும் (கூறுகள், விலைப்பட்டியல், தொழில்நுட்ப விவரக்குறிப்புகள்)  
- துல்லியத்துக்கு மூலத் தகவல் குறிப்பிடுதல் அவசியம்  
- உள்ளடக்கம் ஒரு சிங்கிள் ப்ராமில் பொருந்தாமை  
- சரிபார்க்கக்கூடிய, அடிப்படையிலான பதில்கள் தேவை  

**RAG ஐ பயன்படுத்த வேண்டாம் when:**  
- கேள்விகள் மாதிரி ஏற்கனவே கொண்டுள்ள பொது அறிவைத் தேவைப்படுத்தும் போது  
- நேரடி தரவு தேவைப்படும் போது (RAG பதிவேற்றப்பட்ட ஆவணங்களில் செயல்படும்)  
- உள்ளடக்கம் நேரடியாக ப்ராம்களில் சேர்க்கத்தக்க அளவில باشد
