## நீங்கள் கற்கப்போகும் விஷயங்கள்

இவ்வழி வரை, நீங்கள் AI உடன் உரையாடல் நடத்துவது, சிக்கலான முன்களவாக்கியங்களை அமைத்தல், மற்றும் பதில்களை உங்கள் ஆவணங்களில் அடிப்படையாக்கொள்ளுதல் பற்றி கற்றுக்கொண்டுள்ளீர்கள். ஆனால் இதுவரை ஒரு அடிப்படையான வரம்பு உள்ளதுஅதாவது: மொழி மாதிரிகள் வெறும் உரை தயாரிப்புதான் செய்ய முடியும். அவை வானிலை பார்வையிட முடியாது, கணக்கீடுகள் செய்ய முடியாது, தரவுத்தளங்களை விசாரிக்க முடியாது, அல்லது வெளி அமைப்புகளுடன் தொடர்பு கொள்ள முடியாது.

கருவிகள் இதை மாற்றுகின்றன. மாதிரிக்கு உயிரியியல் செயற்கைத் தொடர்பு வழங்கி, அது வெறும் உரை உருவாக்கியாளராக இருந்து செயற்பாட்டாளராக மாறும். மாதிரி எப்போது கருவி தேவைப்படுகிறது என்பதை முடிவு செய்கிறது, எந்த கருவியை பயன்படுத்துவது என்று தேர்ந்தெடுக்கிறது, மற்றும் என்ன அளவுருக்களுடன் அழைக்க வேண்டும் என்பதையும் தீர்மானிக்கிறது. உங்கள் குறியீடு அந்த செயற்கைகூற்றை இயக்கி முடிவுகளை திருப்பி அளிக்கிறது. மாதிரி அந்த முடிவுகளை தனது பதிலில் இணக்குகிறது.

## கருவி கொண்ட AI முகவர்களை புரிந்து கொள்ளுதல்

> **📝 குறிப்பு:** இந்த தொகுதியில் "முகவர்கள்" என்றால் கருவி அழைப்புத் திறன்களைக் கொண்ட AI உதவியாளர்கள் ஆகும். இது [Module 05: MCP](../05-mcp/README.md) இல் காணும் **Agentic AI** மாடல்கள் (தன்னாட்சி முகவர்கள் திட்டமிடல், நினைவகம் மற்றும் பல քայլ வல்லுநர்ச்சியுடன்) என்ற விதத்திலிருந்து வேறுபடுகின்றது.

கருவிகள் இல்லாமல், ஒரு மொழி மாதிரி தனது பயிற்சி தரவிலிருந்து மட்டுமே உரை உருவாக்கும். தற்போதைய வானிலை யார் என்று கேட்கும்போது இது ஊகிக்கவேண்டும். கருவிகள் கிடைத்தால், அது வானிலை API, கணக்கீடுகள் அல்லது தரவுத்தளம் விசாரணை ஆகியவற்றை அழைக்க முடியும் — பின்னர் அந்த உண்மை முடிவுகளை பதிலில் இணைக்க முடியும்.

<img src="../../../translated_images/ta/what-are-tools.724e468fc4de64da.webp" alt="கருவிகள் இல்லாமல் மற்றும் கருவிகள் உடன் வித்தியாசம்" width="800"/>

*கருவிகள் இல்லாமல் மாதிரி ஊகிக்கிறது — கருவிகளுடன் API-களை அழைக்க, கணக்கெடுப்புகளை இயக்க மற்றும் நேரடி தரவை திருப்ப முடியும்.*

கருவி கொண்ட AI முகவர் **கருத்து மற்றும் செயல் (ReAct)** முறை பின்பற்றுகின்றது. மாதிரி வெறும் பதில் தருவதில்லை — அது தேவையானதை யோசிக்கிறது, கருவியை அழைக்கும் மூலம் செயல் படுகிறது, முடிவை கவனிக்கிறது, பின்னர் மீண்டும் செயல் பட வேண்டுமா அல்லது இறுதிப் பதிலாக தர வேண்டுமா என்று தீர்மானிக்கிறது:

1. **கருத்து** — பயனர் கேள்வியை பகுப்பாய்வு செய்து தேவையான தகவலைத் தீர்மானிக்கிறது
2. **செயல்** — சரியான கருவியை தேர்ந்தெடுத்து, அவசர அளவுருக்களை உருவாக்கி அதை அழைக்கிறது
3. **கவனிப்பு** — கருவியின் வெளியீட்டை பெறுகிறது மற்றும் முடிவை மதிப்பாய்வு செய்கிறது
4. **மீண்டும் செயல் படுதல் அல்லது பதிலளித்தல்** — கூடுதல் தகவல் தேவைப்பட்டால் மீண்டும் தொடங்குகிறது; இல்லையெனில் இயல்பான மொழி பதிலை அமைக்கிறது

<img src="../../../translated_images/ta/react-pattern-detail.96a5efeeb6dd2f61.webp" alt="ReAct முறை" width="800"/>

*ReAct சுற்று — முகவர் செய்யவேண்டியதை யோசித்து கருவி அழைக்கிறது, முடிவை கவனிக்கிறது, பின்னர் முடிவு தரும் வரை திரும்பி செயல் படுகிறது.*

இது தானாக நடக்கிறது. நீங்கள் கருவிகளையும் அவற்றின் விளக்கங்களையும் வரையறுக்கின்றீர்கள். மாதிரி அவற்றைப் பயன்படுத்த எப்போது, எப்படி என்பதை தீர்மானிக்கிறது.

## கருவி அழைப்பது எப்படி வேலை செய்கிறது

### கருவி வரையறைகள்

[WeatherTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) | [TemperatureTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/TemperatureTool.java)

நீங்கள் தெளிவான விளக்கங்களும் அளவுரு விவரங்களும் கொண்ட செயல்பாடுகளை வரையறுக்கின்றீர்கள். மாதிரி இவைகளை தனது கணினி முன் அச்சாணலில் பார்க்கிறது மற்றும் ஒவ்வொரு கருவி என்ன செய்கிறது என்பதை புரிந்துகொள்ளிறது.

```java
@Component
public class WeatherTool {
    
    @Tool("Get the current weather for a location")
    public String getCurrentWeather(@P("Location name") String location) {
        // உங்கள் வானிலை தேடல் தர்க்கம்
        return "Weather in " + location + ": 22°C, cloudy";
    }
}

@AiService
public interface Assistant {
    String chat(@MemoryId String sessionId, @UserMessage String message);
}

// Spring Boot மூலம் உதவியாளர் தானாக இணைக்கப்பட்டுள்ளார்:
// - ChatModel பீன்
// - @Component வகுப்புகளிலிருந்து அனைத்து @Tool முறைகள்
// - அமர்வு மேலாண்மைக்கான ChatMemoryProvider
```

கீழுள்ள வரைபடம் ஒவ்வொரு குறிப்பு குறிக்கப்பட்ட பாகத்தையும் உடைத்து மாதிரிக்கு அது எப்போது கருவியை அழைக்க வேண்டும் மற்றும் எந்த பராமிதிகளை வழங்க வேண்டும் என்பதில் எவ்வாறு உதவுகின்றது என்பதைக் காட்டுகிறது:

<img src="../../../translated_images/ta/tool-definitions-anatomy.f6468546037cf28b.webp" alt="கருவி வரையறைகள் அமைப்பு" width="800"/>

*கருவி வரையறையின் அமைப்பு — @Tool AIக்கு அது எப்போது பயன்படுத்தவேண்டும் என்பதை சொல்கிறது, @P ஒவ்வொரு பராமிதியையும் விவரிக்கிறது, @AiService துவக்கத்தில் அனைத்து ஒன்றுகூடலை இணைக்கிறது.*

### தீர்மானம் எடுக்கும் நிகழ்வு

பயனர் "Seattle இல் வானிலை என்ன?" என்று கேட்டால், மாதிரி கருவியை எதுவும் தற்செயலாக தேர்வு செய்யாது. அது பயனர் நோக்கத்தை நண்பாகக் கொண்டு இருக்கும் ஒவ்வொரு கருவி விளக்கத்துடனும் ஒப்பிட்டு ஒவ்வொன்றுக்கும் பொருத்தத்தை மதிப்பீடு செய்து சிறந்த பொருத்தத்தை தேர்ந்தெடுக்கிறது. பின்னர் சரியான அளவுருக்களுடன் கூறமைப்பு செய்யப்பட்ட செயற்கைக்குழு அழைப்பை உருவாக்குகிறது — இந்த உதாரணத்தில், `location` இற்கு `"Seattle"` என்று ஒதுக்கப்படுகின்றது.

பயனர் கோரிக்கைக்கு பொருந்தும் கருவி இல்லையெனில், மாதிரி தனது உள்ளுணர்வு அடிப்படையில் பதிலளிக்கிறது. பல கருவிகள் பொருந்தும்போது, மிகவும் குறிப்பிட்ட iray தேர்வு செய்யப்படுகிறது.

*மாதிரி ஒவ்வொரு கிடைக்கும் கருவியுடனும் பயனர் நோக்கத்தை மதிப்பீடு செய்கிறது மற்றும் சிறந்த பொருத்தத்தை தேர்வு செய்கிறது — அதனால் தெளிவான, குறிப்பிட்ட கருவி விளக்கங்கள் எழுதுவது அவசியம்.*

### கட்டமைப்பு: Spring Boot தானாக இணைத்தல்

இந்த தொகுதி LangChain4j இன் Spring Boot ஒன்றிணைப்பை `@AiService` அறிவித்த இடைமுகங்களுடன் பயன்படுத்துகிறது. துவக்கத்தில் Spring Boot ஒவ்வொரு `@Component` உள்ள `@Tool` முறைகள் கொண்டவை, உங்கள் `ChatModel` பீன் மற்றும் `ChatMemoryProvider` காணப்படுகிறது — பின்னர் இவை அனைத்தையும் ஒரு `Assistant` இடைமுகத்துடன் ஒருங்கிணைத்து பூச்சியமாக இணைக்கிறது.

<img src="../../../translated_images/ta/spring-boot-wiring.151321795988b04e.webp" alt="Spring Boot தானாக இணைக்கும் கட்டமைப்பு" width="800"/>

*@AiService இடைமுகம் ChatModel, கருவி கூறுகள் மற்றும் நினைவு வழங்குநரை இணைக்கிறது — Spring Boot தானாகவே அனைத்து இணைப்பு பணியையும் செய்கிறது.*

இதோ முழு கோரிக்கை வாழ்க்கைச் சுற்று தொடர்ச்சி வரைபடம் — HTTP கோரிக்கை முதல் கட்டுப்படுத்தி, சேவை, தானாக இணைக்கப்பட்ட பிரதிநிதி, கருவி செயல்பாடு மற்றும் திரும்புதல் வரை:

<img src="../../../translated_images/ta/spring-boot-sequence.f83e3d485aa4a3c6.webp" alt="Spring Boot கருவி அழைக்கும் தொடர்ச்சி" width="800"/>

*முழுமையான Spring Boot கோரிக்கை வாழ்க்கைச் சுற்று — HTTP கோரிக்கை கட்டுப்படுத்தி மற்றும் சேவையில் வழிநடத்து தானாக இணைக்கப்பட்ட Assistant பிரதிநிதியிடേക്കுச் செல்கிறது, இது தானாக LLM மற்றும் கருவி அழைப்புகளை ஒழுங்கு செய்கிறது.*

இந்த அணுகுமுறையின் முக்கிய பயன்பாடுகள்:

- **Spring Boot தானாக இணைத்தல்** — ChatModel மற்றும் கருவிகள் தானாக இணைக்கப்படுகின்றன
- **@MemoryId முறை** — தானாக அமைவுக்கால நினைவகம் மேலாண்மை
- **ஒரே நிகழ்வு** — Assistant ஒருமுறை உருவாக்கப்பட்டு மீண்டும் பயன்படுத்தப்படுகிறது, செயல்திறன் மேம்பாடு
- **வகை-பாதுகாப்பான நடைமுறை** — Java முறைகள் நேரடியாக வகை மாற்றத்துடன் அழைக்கப்படுகின்றன
- **பல சுற்றுகளின் ஒருங்கமைப்பு** — கருவி சங்கிலிதலை தானாக கையாள்கிறது
- **பூச்சியமற்ற boilerplate** — கைமுறை `AiServices.builder()` அழைப்புகள் இல்லை அல்லது நினைவகம் HashMap

மாற்று முறைகள் (`AiServices.builder()` கைமுறை) அதிக குறியீடு தேவை மற்றும் Spring Boot ஒருங்கமைப்பு பயன்பாடுகளை இழக்கின்றன.

## கருவி சங்கிலித்தல்

**கருவி சங்கிலித்தல்** — ஒரே கேள்விக்கு பல கருவிகள் தேவைப்படும் போது கருவி-ஆதார முகவர்களின் உண்மையான சக்தி வெளிப்படுகிறது. "Seattle இல் வானிலை என்ன, Fahrenheit இல்?" என்று கேட்டால் முகவர் தானாக இரண்டு கருவிகளையும் சங்கிலி செய்வார்: முதலில் `getCurrentWeather` அழைக்கப் பெறுகிறது, அது செல்சியஸில் வெப்பநிலையை தருகிறது, பின்னர் அந்த மதிப்பை மாற்ற `celsiusToFahrenheit` அழைக்கப்படுகின்றது — அனைத்தும் ஒரே உரையாடல் சுற்றில்.

<img src="../../../translated_images/ta/tool-chaining-example.538203e73d09dd82.webp" alt="கருவி சங்கிலித்தல் எடுத்துக்காட்டு" width="800"/>

*கருவி சங்கிலித்தல் செயல்பாட்டில் — முகவர் முதலில் getCurrentWeather அழைக்கிறார், பிறகு செல்சியஸ் முடிவை celsiusToFahrenheit க்கு அனுப்பி இணைந்த பதிலை வழங்குகிறார்.*

**தவிர்க்கும் தோல்விகள்** — mock தரவில் இல்லாத நகரத்தின் வானிலை கேட்கவும். கருவி பிழை செய்தியுடன் திருப்புகிறது, மற்றும் AI உதவ முடியாது என்று விளக்குகிறது, முறிவு இருப்பதை தவிர்க்கிறது. கருவிகள் பாதுகாப்பாக தோல்வியடைகின்றன. கீழே உள்ள வரைபடம் இரண்டு அணுகுமுறைகளின் வேறுபாட்டை காட்டுகிறது — சரியான பிழை கையாளுதலில், முகவர் தவறுகளை பிடித்து உதவியாக பதிலளிக்கிறார்; இல்லை என்றால் முழு பயன்பாடு முறிவு அடைகிறது:

<img src="../../../translated_images/ta/error-handling-flow.9a330ffc8ee0475c.webp" alt="பிழை கையாளுதல் ஓட்டம்" width="800"/>

*ஒரு கருவி தோல்வியடைந்தால், முகவர் பிழையை பிடித்து முறிவு பதிலாக உதவிக்குறிப்புடன் பதிலளிக்கிறார்.*

இது ஒரே உரையாடல் சுற்றில் நடக்கிறது. முகவர் பல கருவி அழைப்புகளை தானாக ஒருங்கிணைக்கிறார்.

- பண்தளத்தில் கிடைக்கும் அனைத்து Spring Boot பயன்பாடுகளையும் காணலாம்

- நேரடி பயன்பாட்டு பதிவு விவரங்களை பார்க்கலாம்
- பயன்பாட்டு நிலையை கண்காணிக்கலாம்

## பயன்பாட்டைப் பயன்படுத்துதல்

அப்பிளிகேஷன் ஒரு வலை இடைமுகத்தை வழங்குகிறது, அங்கு நீங்கள் வானிலை மற்றும் வெப்பநிலை மாற்ற கருவிகளுக்கு அணுகல் கொண்ட AI முகவர் ஒருடன் தொடர்பு கொள்ளலாம். இந்த இடைமுகம் இதுபோல காணப்படுகிறது — இதில் துவக்க உதாரணங்கள் மற்றும் கோரிக்கைகளை அனுப்ப ஒரு அரட்டை குழு உள்ளது:

<a href="images/tools-homepage.png"><img src="../../../translated_images/ta/tools-homepage.4b4cd8b2717f9621.webp" alt="AI Agent Tools Interface" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*AI முகவர் கருவிகள் இடைமுகம் - இரு உதாரணங்கள் மற்றும் கருவிகளுடன் தொடர்பு கொள்ள அரட்டை இடைமுகம்*

### எளிய கருவி பயன்பாட்டைப் பயன்படுத்தி பார்த்து புரிந்து கொள்ளவும்

ஏற்றுக்கொள்ளும் கோரிக்கை: "100 டிகிரி ஃபாரன்ஹீட்டைப் செல்வியஸுக்கு மாற்றவும்" என்பதுடன் தொடங்கவும். முகவர் வெப்பநிலை மாற்ற கருவி தேவை என்பதை கண்டறிந்து, சரியான அளவுகோலுடன் அதை அழைத்து முடிவை திருப்பிக் கொள்கிறது. இது இயல்பானது என்று கவனியுங்கள் - நீங்கள் எந்த கருவியை பயன்படுத்த வேண்டும் அல்லது எப்படி அழைக்க வேண்டும் என்று குறிப்பிடவில்லை.

### கருவி சங்கிலி மூலம் பரிசோதனை செய்தல்

இப்போது அதிக சிக்கலான ஒன்றை முயற்சித்துப் பாருங்கள்: "சியாட்லில் வானிலை என்ன மற்றும் அதை ஃபாரன்ஹீட்டுக்கு மாற்றவும்?" முகவர் இதை படிகள் மூலம் ஆராய்கிறது. முதலில் வானிலை பெறுகிறது (செல்சியஸ் அளவில் வந்து), பின்பு ஃபாரன்ஹீட்டுக்கு மாற்ற வேண்டியதுண்டென அடையாளம் காண்கிறார், மாற்ற கருவியை அழைக்கிறார், மற்றும் இரு முடிவுகளை தொகுத்து ஒன்றாக பதிலளிக்கிறார்.

### உரையாடல் ஓட்டத்தைப் பார்க்கவும்

அரட்டை இடைமுகம் உரையாடல் வரலாற்றை பராமரிக்கிறது, இது பல முறைகளான தொடர்புகளை எளிதாக்குகிறது. நீங்கள் முன் செய்த அனைத்து கேள்விகளையும் பதில்களையும் பார்க்கலாம், இது உரையாடலை கண்காணிக்கவும், முகவர் பல முறை தொடர்பில் எப்படி சூழலை உருவாக்குகிறதென்று புரிந்து கொள்வதற்கு உதவுகிறது.

<a href="images/tools-conversation-demo.png"><img src="../../../translated_images/ta/tools-conversation-demo.89f2ce9676080f59.webp" alt="Conversation with Multiple Tool Calls" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*பல முறை உரையாடல் - எளிய மாற்றங்கள், வானிலை காண்டல்கள் மற்றும் கருவி சங்கிலி காட்டுகிறது*

### வெவ்வேறு கோரிக்கைகளுடன் முயற்சி செய்யவும்

விவிதக் கலவைகளை முயற்சிக்கவும்:
- வானிலை பார்க்க: "டோக்கியோவில் வானிலை என்ன?"
- வெப்பநிலை மாற்றங்கள்: "25°C என்பது கேல்வினில் எவ்வளவு?"
- இணைந்த கேள்விகள்: "பாரிஸில் வானிலை பார்க்கவும் மற்றும் அது 20°Cக்கு மேல் என்றால் சொல்லவும்"

முகவர் இயல்பு மொழியை எவ்வாறு புரிந்து கொண்டு சரியான கருவி அழைப்புகளுக்கு பொருந்துகிறது என்பதைக் கவனியுங்கள்.

## முக்கியக் கருத்துக்கள்

### ReAct மாதிரி (கருத்து வைக்கும் மற்றும் நடவடிக்கை எடுக்கும்)

முகவர் செயல்பாடுகளை (கருத்து வைக்கவும்) மற்றும் கருவிகள் பயன்படுத்துவதிலிருந்து (நடவடிக்கை எடுப்பதில்) மாறாக செயல்படுகிறார். இந்த முறை தானாக பிரச்சனைகளை தீர்க்க உதவுகிறது, கட்டளைகளை மட்டும் பின்பற்றுவதில்லை.

### கருவி விளக்கங்கள் முக்கியம்

உங்கள் கருவி விளக்கங்களின் தரம் முகவர் அவற்றைப் பயன்படுத்துவதைக் குறிப்பது எப்படி என்பதில் நேரடி தாக்கத்தை ஏற்படுத்தும். தெளிவான, குறிப்பிட்ட விளக்கங்கள் வடிவமைப்பிற்கு எப்பொழுது மற்றும் எப்படி ஒவ்வொரு கருவியையும் அழைக்க வேண்டும் என்பதில் உதவுகின்றன.

### 세ஷன் மேலாண்மை

`@MemoryId` குறிப்பு தானாக 세ஷன் முறை அடிப்படையிலான நினைவக மேலாண்மையை செயல்படுத்துகிறது. ஒவ்வொரு 세ஷன் ஐடிக்கும் தனி `ChatMemory` உருவாக்கப்படுகிறது, இது `ChatMemoryProvider` பெனில் மூலம் பராமரிக்கப்படுகிறது, இதனால் பல பயனர்கள் ஒரே நேரத்தில் முகவருடன் தொடர்பு கொண்டு உரையாடல்கள் கலக்காமல் திரட்டப்படுகின்றன. கீழ்க்கண்ட வரைபடம் பல பயனர்களை அவர்களின் 세ஷன் ஐடிகளின் அடிப்படையில் தனிப்பட்ட நினைவக சேமிப்புகளுக்கு வழிநடத்துகிறது:

<img src="../../../translated_images/ta/session-management.91ad819c6c89c400.webp" alt="Session Management with @MemoryId" width="800"/>

*ஒவ்வொரு 세ஷன் ஐடியும் தனித்துவமான உரையாடல் வரலாறு குறிக்கும் — பயனர்கள் ஒருவரின் செய்திகளை மற்றவர்களுக்கு காண்பிக்கமாட்டார்கள்.*

### பிழை கையாளுதல்

கருவிகள் தோல்வியடைந்தால் — APIகள் நேரம் முடியும், அளவுகோல்கள் தவறாக இருக்கலாம், வெளிப்புற சேவைகள் செயலிழக்கலாம். உற்பத்தி முகவர்கள் பிழை கையாளுதலைத் தேவைப்படுத்துகின்றன, இதனால் மாடல் சிக்கல்களை விளக்கக்கூடும் அல்லது மாற்றுத் தேர்வுகளை முயற்சிக்கக்கூடும், முழு பயன்பாட்டை பாதிக்காமல். கருவி தவறை தூக்கினால், LangChain4j அதை பிடித்து பிழை செய்தி மாடலைத் திருப்பி வழங்குகிறது, அது இயல்பு மொழியில் பிரச்சனையை விளக்கக் கூடும்.

ஒவ்வொரு கோரிக்கைக்கும் கருவிகள் தேவையில்லை. இது AI வெளிப்புற அமைப்புகளுடன் தொடர்பு கொள்ள வேண்டுமா அல்லது தன் அறிவாராய்ச்சி பதிலளிக்குமா என்பதைக் கண்டறிவதில் உள்ளது. கீழ்காணும் வழிகாட்டி எப்போது கருவிகள் மதிப்பை உயர்த்தும் மற்றும் எப்போது அவை தேவையற்றவை என்பதை சுருக்குகிறது:

<img src="../../../translated_images/ta/when-to-use-tools.51d1592d9cbdae9c.webp" alt="When to Use Tools" width="800"/>

*சுருக்கமான முடிவு வழிகாட்டி — கருவிகள் நேரடி தரவு, கணக்கீடுகள் மற்றும் செயல்பாடுகளுக்குப் பயனுள்ளதாகும்; பொதுவான அறிவு மற்றும் படைப்பாற்றல் பணிகள் அவசியமில்லை.*

## கருவிகள் புரிந்துணர்வு மற்றும் RAG

மொட்யூல்கள் 03 மற்றும் 04 இரண்டுமே AI செய்யக்கூடியவற்றை விரிவாக்குகின்றன, ஆனால் அடிப்படையில் வேறுபட்ட முறையில். RAG மாடலை **அறிவு** பெறுவதற்காக ஆவணங்களை மீட்டெடுக்க வழிவகுக்கிறது. கருவிகள் மாடலுக்கு **நடவடிக்கைகள்** எடுக்கும் வாய்ப்பை வழங்குகின்றன, அத்துடன் செயல்பாடுகளை அழைக்கிறது. கீழ்கண்ட வரைபடம் இந்த இரண்டு அணுகுமுறைகளை ஒப்பிடுகிறது — ஒவ்வொரு பணியின் ஓட்டத்திலிருந்து அவற்றின் நகர்வுகள் மற்றும் வர்த்தக தள்ளுபடிகள் வரை:

<img src="../../../translated_images/ta/tools-vs-rag.ad55ce10d7e4da87.webp" alt="Tools vs RAG Comparison" width="800"/>

*RAG நிலையான ஆவணங்களிலிருந்து தகவலை மீட்டெடுக்கிறது — கருவிகள் செயல்களை இயக்கி, நேரடி, உயிருடை நிகர தரவை பெறுகின்றன. பல உற்பத்தி அமைப்புகள் இரண்டும் சேர்த்து பயன்படுத்துகின்றன.*

விண்ணப்பத்தில், பல உற்பத்தி அமைப்புகள் இரு அணுகுமுறைகளையும் ஒருங்கிணைக்கின்றன: RAG உங்கள் ஆவணங்களில் பதில்களை நிலைநிறுத்த, கருவிகள் நேரடி தரவு பெற அல்லது செயல்பாடுகளை நடாத்த.
