## நீங்கள் கற்றுக்கொள்ளவிருப்பது

நீங்கள் உரையாடல் செயற்கை நுண்ணறிவு, வினாக்களை கையாளுதல், ஆவணங்களில் பதில்களை நிலைநிறுத்துதல் மற்றும் கருவிகளுடன் ஏஜென்ட்களை உருவாக்கியுள்ளீர்கள். ஆனால் அந்த அனைத்து கருவிகளும் உங்கள் தனிப்பட்ட பயன்பாட்டுக்காக தனிப்பயனாக்கப்பட்டிருந்தன. நீங்கள் உங்கள் AI க்கு படைப்பாயரான கருவிகளின் தரிமான சூழலை அணுகும் வாய்ப்பைத் தர முடியுமானால்? இந்த மோடியூலில், நீங்கள் அதைப் பற்றி Model Context Protocol (MCP) மற்றும் LangChain4jஐயின் ஏஜென்டிக் மோடியூலை பயன்படுத்தி எப்படி செய்வதென்று கற்றுக்கொள்ள இருப்பீர்கள். முதலில் எளிய MCP கோப்பு வாசிப்பாளரை காண்பித்து, பின்னர் அது மேம்பட்ட ஏஜென்டிக் வேலைப்பாடுகளில் எப்படி எளிதாக இணைக்கப்படுகிறதென்றும் Supervisor Agent மாதிரியைப் பயன்படுத்தி காண்பிப்போம்.

## MCP என 무엇?

Model Context Protocol (MCP) செயற்கை நுண்ணறிவு பயன்பாடுகள் வெளியூா் கருவிகளை கண்டுபிடித்து பயன்படுத்த ஒரு தரநிலையான வழியை வழங்குகிறது. ஒவ்வொரு தரவுத் திறனுக்கும் அல்லது சேவைக்கும் தனிப்பயன் ஒருங்கிணைப்புகளை எழுதாமல், MCP சேவையகங்களுடன் இணைக்க, அவற்றின் திறன்களை ஒரே மாதிரியான வடிவத்தில் வெளிப்படுத்துகிறது. உங்கள் AI ஏஜென்ட் தானாகவே அந்த கருவிகளை கண்டுபிடித்து பயன்படுத்த முடியும்.

கீழே வரைபடம் வித்தியாசத்தை காட்சிப்படுத்துகிறது — MCP இல்லாமல் ஒவ்வொரு ஒருங்கிணைப்பும் தனித்தனியான இணைப்பை தேவைப்படுத்தும்; MCP உடன், ஒரு அணி உங்கள் பயன்பாட்டை எந்த கருவியுடன் வேண்டுமானாலும் இணைக்கும்:

<img src="../../../translated_images/ta/mcp-comparison.9129a881ecf10ff5.webp" alt="MCP ஒப்பீடு" width="800"/>

*MCP முன்பே: சிக்கலான பாயிண்ட்-டூ-பாயிண்ட் ஒருங்கிணைப்புகள். MCP பிறகு: ஒரு நடைமுறை, முடிவில்லா வாய்ப்புகள்.*

MCP AI அபிவிருத்தியில் ஒரு அடிப்படைக் குறையை தீர்கிறது: ஒவ்வொரு ஒருங்கிணைப்பு தனிப்பயனாக்கப்பட்டுள்ளது. GitHub அணுக வேண்டுமா? தனிப்பயன் குறியீடு. கோப்புகளை வாசிக்க வேண்டுமா? தனிப்பயன் குறியீடு. தரவுத்தளக் கேள்வி செய்ய வேண்டுமா? தனிப்பயன் குறியீடு. மற்ற AI பயன்பாடுகளோடு இவை வேலை செய்யாது.

MCP இதை ஒருங்கிணைக்கிறது. MCP சேவையகம் கருவிகளை தெளிவான விளக்கங்கள் மற்றும் வடிவமைப்புகளுடன் வெளிப்படுத்துகிறது. எந்த MCP கிளையன்டும் இணைத்து, கிடைக்கும் கருவிகளை கண்டுபிடித்து பயன்படுத்த முடியும். ஒருமுறை கட்டி, எங்கும் பயன்படுத்தலாம்.

கீழே உள்ள வரைபடம் இந்த கட்டமைப்பை விளக்குகிறது — ஒரு MCP கிளையன்ட் (உங்கள் AI பயன்பாடு) பல MCP சேவையகங்களுக்கு இணைக்கிறது, ஒவ்வொன்றும் தங்கள் கருவிகளை நடைமுறை வழிமுறையால் வெளிப்படுத்துகிறது:

<img src="../../../translated_images/ta/mcp-architecture.b3156d787a4ceac9.webp" alt="MCP கட்டமைப்பு" width="800"/>

*Model Context Protocol கட்டமைப்பு - நிலையான கருவி கண்டுபிடிப்பு மற்றும் செயலாக்கம்*

## MCP எப்படி செயல்படுகிறது

உள்நோக்கில், MCP பல அடுக்குகளைக் கொண்ட கட்டமைப்பைக் கொண்டுள்ளது. உங்கள் Java பயன்பாடு (MCP கிளையன்ட்) கிடைக்கக்கூடிய கருவிகளை கண்டுபிடித்து, JSON-RPC கோரிக்கைகளை தொடர்பு அடுக்கு (Stdio அல்லது HTTP) வழியாக அனுப்புகிறது, MCP சேவையகம் செயல்பாடுகளை இயக்கி முடிவுகளை வழங்குகிறது. கீழ்காணும் வரைபடம் இந்த நடைமுறையின் ஒவ்வொரு அடுக்கையும் விவரிக்கிறது:

<img src="../../../translated_images/ta/mcp-protocol-detail.01204e056f45308b.webp" alt="MCP நடைமுறை விரிவான விளக்கம்" width="800"/>

*MCP உள் இயங்கும் விதம் — கிளையன்ட்கள் கருவிகளை கண்டுபிடித்து, JSON-RPC செய்திகளை பரிமாறி, தொடர்பு அடுக்கின் மூலம் செயல்பாடுகளை இயக்குகின்றன.*

**சேவைதி-கிளையன்ட் கட்டமைப்பு**

MCP ஒரு கிளையன்ட்-சேவைதி மாதிரியைப் பயன்படுத்துகிறது. சேவைதிகள் கருவிகள் வழங்குகின்றன - கோப்புகளை வாசித்தல், தரவுத்தளங்களை கேட்குதல், APIகளை அழைத்தல். கிளையன்ட்கள் (உங்கள் AI பயன்பாடு) சேவைதிகளுடன் இணைந்து அவர்களின் கருவிகளை பயன்படுத்துகின்றன.

LangChain4j உடன் MCP ஐ பயன்படுத்த, இதை Maven சார்பு சேர்க்கவும்:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```

**கருவி கண்டுபிடிப்பு**

உங்கள் கிளையன்ட் MCP சேவையகத்துடன் இணைக்கும்போது, அது கேட்கும் "உங்களிடம் என்ன கருவிகள் உள்ளன?" என்று. சேவையகம் கிடைக்கக்கூடிய கருவிகளின் பட்டியலை, விளக்கங்கள் மற்றும் அளவுரு வடிவங்கள் உடன் கல்வி அளிக்கும். உங்கள் AI ஏஜென்ட் பயனர் கோரிக்கைகளின் அடிப்படையில் எந்த கருவிகளை பயன்படுத்துவது என்று தீர்மானிக்க முடியும். கீழ் அடக்கம் இந்த கையெழுத்தைப் பார்க்கிறது — கிளையன்ட் `tools/list` கோரிக்கையை அனுப்பி, சேவையகம் அதை எடுத்துக் காட்டுகின்ற கருவிகளின் விவரங்களை அளிக்கிறது:

<img src="../../../translated_images/ta/tool-discovery.07760a8a301a7832.webp" alt="MCP கருவி கண்டுபிடிப்பு" width="800"/>

*AI துவக்கம் நேரத்தில் கிடைக்கும் கருவிகளை கண்டுபிடிக்கிறது — அது இப்போது எந்த திறன்கள் கிடைக்கும் என்பதைக் கற்றுக்கொள்ளக்கூடியது மற்றும் எந்தவற்றைப் பயன்படுத்துவது என முடிவு செய்யலாம்.*

**போக்குவரத்து முறைமைகள்**

MCP பல போக்குவரத்து முறைகளை ஆதரிக்கிறது. இரண்டு விருப்பங்கள் உள்ளன: Stdio (ஏற்றுமதி செயலிகளுக்கான உள்ளூர் தொடர்பு) மற்றும் Streamable HTTP (தூர சேவைதிகளுக்காக). இந்த மோடியூல் Stdio போக்குவரத்தை காட்டுகிறது:

<img src="../../../translated_images/ta/transport-mechanisms.2791ba7ee93cf020.webp" alt="போக்குவரத்து முறைமைகள்" width="800"/>

*MCP போக்குவரத்து முறைமைகள்: தூர சேவைதிகளுக்கான HTTP, உள்ளூர் செயலிகளுக்கான Stdio*

**Stdio** - [StdioTransportDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java)

உள்ளூர் செயலிகளுக்கு. உங்கள் பயன்பாடு ஒரு சேவையகத்தை துணைசெயலியாக உருவாக்கி, நிலையான உள்ளீடு/வெளியீடு வழியாக தொடர்பு கொள்கிறது. கோப்பமைப்பு அணுகுதல் அல்லது கட்டளை வரி கருவிகளுக்கு பயன்படும்.

```java
McpTransport stdioTransport = new StdioMcpTransport.Builder()
    .command(List.of(
        npmCmd, "exec",
        "@modelcontextprotocol/server-filesystem@2025.12.18",
        resourcesDir
    ))
    .logEvents(false)
    .build();
```

`@modelcontextprotocol/server-filesystem` சேவையகம் கீழ்க்காணும் கருவிகளை வெளிப்படுத்துகிறது, அனைத்தும் நீங்கள் குறிப்பிடும் அடைவுகளில் அனுமதிக்கப்பட்டவை:

| கருவி | விளக்கம் |
|-------|----------|
| `read_file` | ஒரு தனி கோப்பின் உள்ளடக்கங்களை வாசி |
| `read_multiple_files` | ஒரே அழைப்பில் பல கோப்புகளை வாசி |
| `write_file` | கோப்பை உருவாக்கு அல்லது மீண்டும் எழுதுக |
| `edit_file` | குறிப்பிட்ட தேடு-மாற்றியல் திருத்தங்களை செய் |
| `list_directory` | ஒரு பாதையில் கோப்புகள் மற்றும் அடைவுகளின் பட்டியலைப் பார் |
| `search_files` | ஒரு மாதிரிக்கு பொருத்தமான கோப்புகளை மீளிச் சோதி |
| `get_file_info` | கோப்பு பண்புகளை (எடை, நேரதினச்சுட்டிகள், அனுமதிகள்) பெறுக |
| `create_directory` | ஒரு அடையை உருவாக்கு (பெற்றோர் அடைவுகளுடன்) |
| `move_file` | ஒரு கோப்பை அல்லது அடையை நகர்த்து அல்லது பெயர் மாற்று |

கீழே வரைபடம் Stdio போக்குவரத்து செயல்படுவதைக் காட்டுகிறது — உங்கள் Java பயன்பாடு MCP சேவையகத்தை குழந்தை செயலியாக உருவாக்கி, அவர்கள் stdin/stdout குழாய்கள் மூலம் தொடர்பு கொள்கிறார்கள், எந்தப் பிணையமும் HTTPவும் இதில் பயன்படுத்தப்படவில்லை:

<img src="../../../translated_images/ta/stdio-transport-flow.45eaff4af2d81db4.webp" alt="Stdio போக்குவரத்து ஓட்டம்" width="800"/>

*Stdio போக்குவரத்து செயலில் — உங்கள் பயன்பாடு MCP சேவையகத்தை குழந்தை செயலியாக உருவாக்கி stdin/stdout குழாய்கள் மூலம் தொடர்பு கொள்கிறது.*

> - "LangChain4j எப்படி MCP சேவையக துணைசெயலிகள் வாழ்க்கைச் சுழற்சியை நிர்வகிக்கிறது?"
> - "AI க்கு கோப்பமைப்பை அணுக அனுமதி வழங்குவதன் பாதுகாப்பு விளைவுகள் என்ன?"

## ஏஜென்டிக் மோடியூல்

MCP நிலையான கருவிகளை வழங்கினாலும்கூட, LangChain4j இன் **ஏஜென்டிக் மோடியூல்** அந்த கருவிகளை ஒழுங்குபடுத்தும் ஏஜென்ட்களை உருவாக்க ஒரு விளக்கமான வழியை வழங்குகிறது. `@Agent` குறிச்சொல் மற்றும் `AgenticServices` உங்களுக்கு ஆணைக்கோல் குறியீடு இல்லாமல் மேற்பார்வைப் பண்புகளை வரையறுக்க உதவுகின்றன.

இந்த மோடியூலில், நீங்கள் **மேற்பார்வையாளர் ஏஜென்ட்** மாதிரியை ஆராய்வீர்கள் — இது "மேற்பார்வையாளர்" ஏஜென்ட் பயனர் கோரிக்கைகள் அடிப்படையில் எந்த துணை ஏஜென்ட் செயல்படுத்தப்பட வேண்டும் என்று தானாகத் தீர்மானிக்கிறது. நாம் இந்த இரு கருத்துக்களையும் ஒருங்கிணைத்து, ஒரு துணை ஏஜென்ட் MCP சக்தியுள்ள கோப்பு அணுகல் திறன்களைக் கொண்டதாக மாற்றப்போகிறோம்.

ஏஜென்டிக் மோடியூலை பயன்படுத்த, இதை Maven சார்பு சேர்க்கவும்:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```
> **குறிப்பு:** `langchain4j-agentic` மோடியூல் தனித்த பதிப்பு பண்பை (`langchain4j.mcp.version`) பயன்படுத்துகிறது ஏனெனில் இது LangChain4j மூல நூலகங்களிடமிருந்து வேறுபட்ட வெளியீட்டு அட்டவணையில் உள்ளது.

> **⚠️ பரிசோதனை நிலை:** `langchain4j-agentic` மோடியூல் **பரிசோதனை நிலை** மற்றும் மாற்றம் அடிக்கடி இருக்கலாம். நிலைத்த AI உதவியாளர்கள் கட்டமைப்புக்கான வழி `langchain4j-core` மற்றும் தனிப்பயன் கருவிகள் (Module 04) ஆகவே உள்ளது.

### மேற்பார்வையாளர் ஏஜென்ட்

**மேற்பார்வையாளர் ஏஜென்ட் மாதிரி** என்பது **நெகிழ்வான** ஏஜென்டிக் AI ஆகும். ஒரு மேற்பார்வையாளர் LLM ஐ பயன்படுத்தி பயனர் கோரிக்கையின் அடிப்படையில் எந்த ஏஜென்ட்களை செயல்படுத்த வேண்டும் என்று தானாகத் தீர்மானிக்கிறது. அடுத்த உதாரணத்தில், MCP சக்தியுள்ள கோப்பு அணுகலை LLM ஏஜென்டுடன் சேர்த்து மேற்பார்வை கேள்வி → அறிக்கை வேலைப்பாட்டைப் படைக்கிறோம்.

டெமோவில், `FileAgent` MCP கோப்பமைப்புக் கருவிகளைப் பயன்படுத்தி கோப்பை வாசிக்கிறது; `ReportAgent` நிர்வாக சுருக்கம் (ஒரு வாக்கியம்), 3 முக்கிய அம்சங்கள் மற்றும் பரிந்துரைகளை கொண்ட கட்டமைக்கப்பட்ட அறிக்கையை உருவாக்குகிறது. மேற்பார்வையாளர் இந்த ஓட்டத்தை தானாக அமைக்கிறது:

<img src="../../../translated_images/ta/supervisor-agent-pattern.06275a41ae006ac8.webp" alt="மேற்பார்வையாளர் ஏஜென்ட் மாதிரி" width="800"/>

*மேற்பார்வையாளர் தனது LLM ஐப் பயன்படுத்தி எந்த ஏஜென்ட்களை எந்த வரிசையில் செயல்படுத்த வேண்டும் என்று முடிவெடுக்கிறார் — கடினமாக நிரலிடப்பட்ட வழி தேவை இல்லை.*

நமது கோப்பு-முடிவுஆக அறிவிக்கல் கடவுச்சூழலை மகிழ்விக்க கீழ்காணும் விசேஷமான வேலைப்பாடு:

*FileAgent MCP கருவிகள் மூலம் கோப்பை வாசிக்கிறது, பின்னர் ReportAgent அசல் உள்ளடக்கத்தை கட்டமைக்கும் அறிக்கையாக மாற்றுகிறது.*

ஒவ்வொரு ஏஜென்ட் தங்களுடைய வெளியீட்டை **Agentic Scope** (பகிர்ந்து கொள்ளப்பட்ட நினைவக இழை) இல் சேமிக்கின்றனர், இது பின்னணி ஏஜென்ட்கள் முன்னைய முடிவுகளை அணுக அனுமதிக்கிறது. இது MCP கருவிகள் ஏஜென்டிக் வேலைப் பணியில் எவ்வாறு ஒருங்கிணைக்கப்படுகின்றன என்பதை விளக்குகிறது — மேற்பார்வையாளர் *எப்படி* கோப்புகள் வாசிக்கப்படுகின்றன என்பதைக் காட்ட வேண்டியதில்லை, `FileAgent` அதை செய்ய முடியும் என்ற முனைப்பே போதும்.

#### மேற்பார்வையாளர் எப்படி செயல்படுகிறது

ஏஜென்ட்களை கட்டமைப்பதற்கு முன், MCP போக்குவரத்தை உள்ள கிளையன்டுடன் இணைத்து அதை `ToolProvider` ஆக மோதிக்கொள்ள வேண்டும். இதுதான் MCP சேவையக கருவிகள் உங்கள் ஏஜென்ட்களுக்கு கிடைக்கச் செய்வது:

```java
// போக்குவரத்து இருந்து ஒரு MCP கிளையண்டை உருவாக்கு
McpClient mcpClient = new DefaultMcpClient.Builder()
        .transport(stdioTransport)
        .build();

// கிளையண்டை ஒரு ToolProvider ஆக மடக்கு — இது MCP கருவிகளை LangChain4j உடன் இணைக்க உதவுகிறது
ToolProvider mcpToolProvider = McpToolProvider.builder()
        .mcpClients(List.of(mcpClient))
        .build();
```

இப்போது ஏதேனும் ஏஜென்டுக்கு MCP கருவிகள் தேவைப்பட்டால் `mcpToolProvider` ஐ 注入 (inject) செய்யலாம்:

```java
// படி 1: FileAgent MCP கருவிகளைப் பயன்படுத்தி கோப்புகளை வாசிக்கிறது
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // கோப்பு செயல்பாடுகளுக்காக MCP கருவிகள் உள்ளன
        .build();

// படி 2: ReportAgent கட்டமைக்கப்பட்ட அறிக்கைகளை உருவாக்குகிறது
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// மேலாளி கோப்பு → அறிக்கை பணிவழியை ஒருங்கிணைக்கிறார்
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // இறுதி அறிக்கையை திரும்பச் செய்க
        .build();

// கோரிக்கையைப் பொறுத்து மேலாளி எந்த முகவர்களை அழைக்க தீர்மானிக்கிறார்
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```

#### பதில் கொடுக்கும் தந்திரங்கள்

`SupervisorAgent` ஐ அமைக்கும் போது, துணை ஏஜென்ட்கள் பணிப்பதிவுகளை முடித்ததும் பயனருக்கான இறுதி பதிலை எப்படி உருவாக்கும் என்பதைக் குறிப்பிடவும். கீழே உள்ள வரைபடம் மூன்று கிடைக்கும் தந்திரங்களை காட்டுகிறது — LAST இறுதி ஏஜென்டின் வெளியீட்டை நேரடியாக கொடுக்கும், SUMMARY அனைத்து வெளியீடுகளையும் LLM மூலம் தொகுத்து வழங்கும், SCORED அது முதல் கிடைக்கும் ஒருவர் ஆதார கோரிக்கையுடன் பொருத்து அதிக மதிப்பெண் பெறுபவரைத் தேர்வு செய்கிறது:

<img src="../../../translated_images/ta/response-strategies.3d0cea19d096bdf9.webp" alt="பதில் கொடுக்கும் தந்திரங்கள்" width="800"/>

*மேற்பார்வையாளர் இறுதி பதிலை எவ்வாறு உருவாக்குவார் என்பதற்கான மூன்று தந்திரங்கள் — கடைசி ஏஜென்டின் வெளியீடு, தொகுக்கப்பட்ட சுருக்கம் அல்லது சிறந்த மதிப்பெண் பெற்ற விருப்பத்தைத் தேர்ந்தெடுக்கவும்.*

கிடைக்கும் தந்திரங்கள்:

| தந்திரம் | விளக்கம் |
|----------|------------|
| **LAST** | மேற்பார்வையாளர் கடைசியாக அழைக்கப்பட்ட துணை ஏஜென்ட் அல்லது கருவியின் வெளியீட்டைத் திருப்பி அளிக்கும். இது வேலைப்பாட்டின் இறுதி ஏஜென்ட் முழுமையான பதிலை உருவாக்குவதற்கு ஏற்றது (எ.கா., ஆராய்ச்சி குழாயிலுள்ள "சுருக்கம் ஏஜென்ட்"). |
| **SUMMARY** | மேற்பார்வையாளர் தன் உள்ளக மொழி மாதிரியைப் பயன்படுத்தி முழு உரையாடல் மற்றும் துணை ஏஜென்ட் வெளியீடுகளின் சுருக்கத்தைக் தொகுத்து, அதை இறுதி பதிலாக வழங்கும். இது பயனருக்கு தெளிவான, கூட்டு பதிலை வழங்குகிறது. |
| **SCORED** | מערכת內部 LLM கேள்வி மற்றும் LAST பதிலை மதிப்பெண் அளித்து, அதிக மதிப்பீடு பெற்ற பதிலை திருப்பும். |

#### வெளியீட்டை புரிந்துகொள்ளுதல்

நீங்கள் டெமோவை இயக்கும் பொழுது, Supervisor பல முகவர்களை எவ்வாறு ஒருங்கிணைக்கின்றது என்பதற்கான கட்டமைக்கப்பட்ட நடைமுறையை காண்பீர்கள். ஒவ்வொரு பகுதியில் உள்ள பொருள் இதுவாகும்:

```
======================================================================
  FILE → REPORT WORKFLOW DEMO
======================================================================

This demo shows a clear 2-step workflow: read a file, then generate a report.
The Supervisor orchestrates the agents automatically based on the request.
```

**தலைப்பு** வேலைவடிவ கான்செப்டை அறிமுகப்படுத்துகிறது: கோப்புப் படிப்பிலிருந்து அறிக்கையாக்கம் வரை கவனமுள்ள பாதை.

```
--- WORKFLOW ---------------------------------------------------------
  ┌─────────────┐      ┌──────────────┐
  │  FileAgent  │ ───▶ │ ReportAgent  │
  │ (MCP tools) │      │  (pure LLM)  │
  └─────────────┘      └──────────────┘
   outputKey:           outputKey:
   'fileContent'        'report'

--- AVAILABLE AGENTS -------------------------------------------------
  [FILE]   FileAgent   - Reads files via MCP → stores in 'fileContent'
  [REPORT] ReportAgent - Generates structured report → stores in 'report'
```

**வேலைவடிவ வரைபடம்** முகவர்களுக்கு இடையேயான தரவு ஓட்டத்தை காண்பிக்கிறது. ஒவ்வொரு முகவருக்கும் குறிப்பிட்ட பங்கு உள்ளது:
- **FileAgent** MCP கருவிகள் மூலம் கோப்புகளை படிக்கிறான் மற்றும் அசல் உள்ளடக்கத்தை `fileContent` ல் சேமிக்கிறான்
- **ReportAgent** அந்த உள்ளடக்கத்தை எடுத்துக் கொண்டு `report` இல் கட்டமைக்கப்பட்ட அறிக்கையை உருவாக்குகிறது

```
--- USER REQUEST -----------------------------------------------------
  "Read the file at .../file.txt and generate a report on its contents"
```

**பயனர் கோரிக்கை** பணியை காட்டுகிறது. Supervisor இதைப் பகுப்பாய்வு செய்து FileAgent → ReportAgent அழைப்பதை முடிவு செய்கிறது.

```
--- SUPERVISOR ORCHESTRATION -----------------------------------------
  The Supervisor decides which agents to invoke and passes data between them...

  +-- STEP 1: Supervisor chose -> FileAgent (reading file via MCP)
  |
  |   Input: .../file.txt
  |
  |   Result: LangChain4j is an open-source, provider-agnostic Java framework for building LLM...
  +-- [OK] FileAgent (reading file via MCP) completed

  +-- STEP 2: Supervisor chose -> ReportAgent (generating structured report)
  |
  |   Input: LangChain4j is an open-source, provider-agnostic Java framew...
  |
  |   Result: Executive Summary...
  +-- [OK] ReportAgent (generating structured report) completed
```

**Supervisor ஒருங்கிணைப்பு** 2-படி ஓட்டத்தை செயலில் காண்பிக்கிறது:
1. **FileAgent** MCP மூலம் கோப்பை படித்து உள்ளடக்கத்தை சேமிக்கிறான்
2. **ReportAgent** அந்த உள்ளடக்கத்தை பெற்று கட்டமைக்கப்பட்ட அறிக்கையை உருவாக்குகிறான்

பயனர் கோரிக்கையின் அடிப்படையில் Supervisor இவை அனைத்தையும் **தானாகவே** முடிவு செய்தது.

```
--- FINAL RESPONSE ---------------------------------------------------

...

Key Points
...

Recommendations
...

--- AGENTIC SCOPE (Data Flow) ----------------------------------------
  Each agent stores its output for downstream agents to consume:
  * fileContent: LangChain4j is an open-source, provider-agnostic Java framework...
  * report: Executive Summary...
```

#### முகவரியியல் தொகுதியின் பண்புகளின் விளக்கம்

இந்த உதாரணம் முகவரியியல் தொகுதியின் பல மேம்பட்ட பண்புகளை காட்டுகிறது. Agentic Scope மற்றும் Agent Listeners ஐ நிகர்ந்து பார்ப்போம்.

**Agentic Scope** பொது நினைவகம் ஆகும், இது முகவர்கள் `@Agent(outputKey="...")` பயன்படுத்தி தங்கள் முடிவுகளை சேமித்த இடம். இது:
- பிந்தைய முகவர்கள் முன்னைய முகவர்களின் வெளியீட்டில் அணுகல் பெற உதவும்
- Supervisor இறுதி பதிலை இணைக்க உதவும்
- நீங்கள் ஒவ்வொரு முகவரும் உருவாக்கியதை பரிசீலிக்க முடியும்

கீழே உள்ள வரைபடம் Agentic Scope எப்படி கோப்பிலிருந்து அறிக்கைக்கு வேலைபாட்டு பொது நினைவகமாக செயல்படுகிறது என்பதைக் காட்டுகிறது — FileAgent `fileContent` எனும் முக்கியத்தில் வெளியீட்டை எழுதுகிறான், ReportAgent அதைப் படித்து `report` எனும் வெளியீட்டை எழுதுகிறான்:

<img src="../../../translated_images/ta/agentic-scope.95ef488b6c1d02ef.webp" alt="Agentic Scope Shared Memory" width="800"/>

*Agentic Scope பொது நினைவகமாக செயல்படுகிறது — FileAgent `fileContent` எழுதல், ReportAgent படித்து `report` எழுதுதல், உங்கள் குறியீடு இறுதி முடிவை வாசித்தல்.*

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // FileAgent இல் இருந்து மூல கோப்பு தரவு
String report = scope.readState("report");            // ReportAgent இல் இருந்து கட்டமைக்கப்பட்ட அறிக்கை
```

**Agent Listeners** முகவர் செயல்பாட்டை கண்காணித்து பிழைத்திருத்த உதவுகின்றன. டெமோவில் நீங்கள் செயற்கை முறையாகப் பார்த்த படிகள் AgentListener மூலம் ஒவ்வொரு முகவர் அழைப்பிலும் செருகப்பட்டவை:
- **beforeAgentInvocation** - Supervisor ஒரு முகவரைக் தேர்ந்தெடுத்துவிட்டது என்பதைக் காட்டுகிறது, எந்த முகவர்தான் தேர்ந்தெடுக்கப்பட்டது மற்றும் ஏன் என்பதைப் பார்வையிட உதவும்
- **afterAgentInvocation** - ஒரு முகவர் முடிந்ததும் அதன் முடிவை காட்டுகிறது
- **inheritedBySubagents** - உண்மையானால், அந்த நூலகம் முழு முகவர் மரபின் எல்லா முகவர்களையும் கண்காணிக்கும்

அடுத்த வரைபடம் முழுமையான Agent Listener வாழ்க்கைசுழற்சியையும், முகவர் செயல்பாட்டின் போது பிழைகள் நேர்ந்தால் `onError` எவ்வாறு கையாள்கின்றது என்பதையும் காட்டுகிறது:

<img src="../../../translated_images/ta/agent-listeners.784bfc403c80ea13.webp" alt="Agent Listeners Lifecycle" width="800"/>

*Agent Listeners செயல்பாட்டு வாழ்க்கைசுற்றில் செருகல் செய்கின்றன — முகவர்கள் தொடங்குவதையும், முடிப்பதையும், பிழைகள் நேருவதையும் கண்காணிக்கின்றனர்.*

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // அனைத்து துணை முகவர்களுக்கும் பரப்பவும்
    }
};
```

Supervisor முறைமையின் அப்பால், `langchain4j-agentic` தொகுதி பல மின்சார வேலைபாடு சலுகைகளை வழங்குகிறது. கீழே உள்ள வரைபடம் ஐந்து முறைமைகளையும் காண்பிக்கிறது — எளிய வரிசை தவணை செயல்முறைகள் முதல் மனித நுழைவு அங்கீகார வேலைபாடுகள் வரை:

<img src="../../../translated_images/ta/workflow-patterns.82b2cc5b0c5edb22.webp" alt="Agent Workflow Patterns" width="800"/>

*ஐந்து முகவர் வேலைபாடு சலுகைகள் — எளிய வரிசை தவணை செயல்முறைகள் முதல் மனித நுழைவு அங்கீகார வேலைபாடுகள் வரை.*

| முறைமை | விளக்கம் | பயன்படுத்தும் சூழல் |
|---------|-------------|----------|
| **Sequential** | முகவர்களை வரிசைப்படி இயக்கும், வெளியீடு அடுத்து ஓடும் | தவணை: ஆய்வு → பகுப்பாய்வு → அறிக்கை |
| **Parallel** | முகவர்களை ஒரே நேரத்தில் இயக்கும் | சுயாதீன பணிகள்: வானிலை + செய்தி + பங்கு சந்தை |
| **Loop** | நிலைமையை பூர்த்தி செய்யும் வரை 반복ம் | தர மதிப்பீடு: மதிப்பெண் ≥ 0.8 ஆகும் வரை தடவை |
| **Conditional** | நிபந்தனை அடிப்படையில் வழி நடத்துதல் | வகைப்படுத்தி → நிபுணர் முகவருக்கு வழி |
| **Human-in-the-Loop** | மனித இடைசெருகல் கூடுதல் | அங்கீகார வேலைபாடுகள், உள்ளடக்க மதிப்பாய்வு |

## முக்கியக் கருத்துக்கள்

MCP மற்றும் முகவரியியல் தொகுதியைச் செயல்படுத்தி பார்த்த பிறகு, ஒவ்வொரு அணுகுதலை எப்போது பயன்படுத்த வேண்டும் என்பதை சுருக்கமாக பார்ப்போம்.

MCP இன் மிகப்பெரிய நன்மை அதன் வளவளரி உயரும் சூழல்சார் நெட்வொர்க். கீழே உள்ள வரைபடம் ஒரு ஒரே உலகளாவிய நெறிமுறையை காட்டுகிறது; இது உங்கள் AI பயன்பாட்டை மிகப்பெரிய வகையான MCP சேவையகங்களோடு இணைக்கிறது — கோப்பு அமைப்பு மற்றும் தரவுத்தளம் அணுகல் முதல் GitHub, மின்னஞ்சல், வலை ஊதியம் மற்றும் மேலும்:

<img src="../../../translated_images/ta/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="MCP Ecosystem" width="800"/>

*MCP உலகளாவிய நெறிமுறை சூழலை உருவாக்குகிறது — எந்த MCP-இல் இணக்கமான சேவையகம் எந்த MCP-இல் இணக்கமான கிளையண்டுடன் பணியிடுவான், இதனால் கருவி பகிர்வு பயனாளி கணினிகளில் பெறப்படுகிறது.*

**MCP** ஏற்கனவே உள்ள கருவி சூழல்களை பயன்படுத்த விரும்பும் போது, பல பயன்பாடுகள் பகிரக்கூடிய கருவிகளை உருவாக்க விரும்பும் போது, மூன்றாம் பக்க சேவைகளை தரநிலைப்பூர்வ நெறிமுறைகளுடன் இணைப்பதில், அல்லது குறியீடு மாற்றாமல் கருவி அமல்படுத்தல்களை மாற்ற விரும்பும் போது சிறந்தது.

**The Agentic Module** `@Agent` குறியீடுகளுடன் குறிப்பாய்ப் முகவரி வரைவுகளுக்கு சிறந்தது, வேலைபாடு ஒருங்கிணைப்பு (வரிசை, 반복, அலைமோதல்) தேவைப்பட்டால், கட்டளையியல் குறியீட்டிற்கு பதிலாக இடைமுக அடிப்படையிலான முகவர் வடிவமைப்பை விரும்பும் போது, அல்லது பல முகவர்கள் `outputKey` மூலம் வெளியீடுகளை பகிரும் போது சிறந்தது.

**Supervisor Agent முறைமை** வேலைபாடு முன்கூட்டியே கணிக்க முடியாத போது, LLM தீர்மானிக்க வேண்டும் என்பதால், பல சிறப்பு முகவர்கள் செறிவாக ஒருங்கிணைக்கப்படவேண்டும் என்பதால், வீச்சு வழிகாட்டும் அமைப்புகளை உருவாக்கும் போது, அல்லது மிக அசைக்கத் தேவையான, ஏற்றுக்கொள்ளக்கூடிய முகவர் நடத்தையை விரும்பும்போது சிறந்தது.

Module 04 இலிருந்து தனிப்பயன் `@Tool` முறைகள் மற்றும் இந்த தொகுதியில் MCP கருவிகளுக்கு இடையில் தேர்வு செய்ய உதவும் முக்கிய வேறுபாடுகளை கீழ் காட்டும் ஒப்பீடு:

<img src="../../../translated_images/ta/custom-vs-mcp-tools.c4f9b6b1cb65d8a1.webp" alt="Custom Tools vs MCP Tools" width="800"/>

*தனிப்பயன் @Tool முறைகள் மற்றும் MCP கருவிகள் எப்போது பயன்படுத்துவது — தனிப்பயன் கருவிகள் பயன்பாட்டுக்கு உகந்த கட்டுப்பாட்டையும் முழு வகை பாதுகாப்பையும் தருகின்றன, MCP கருவிகள் தரநிலைப்பட்ட ஒருங்கிணைப்புகளுடன் பல பயன்பாடுகளில் செயல்படக்கூடியவை.*

## வாழ்த்துகள்!

LangChain4j ஆரம்பநிலை பயிற்சி பாடநெறியின் ஐந்து தொகுதிகளை முழுமையாக கடந்துள்ளீர்கள்! உங்கள் சம്പூர்ண கற்றல் பயணத்தை இங்கு காணலாம் — அடிப்படையிலான உரையாடல் இருந்து MCP-ஆல் இயக்கப்படும் முகவரியியல் அமைப்புகள் வரை:

<img src="../../../translated_images/ta/course-completion.48cd201f60ac7570.webp" alt="Course Completion" width="800"/>

*உங்கள் ஐந்து தொகுதிகள் முழுவதும் கற்றல் பயணம் — அடிப்படையிலான உரையாடலில் இருந்து MCP இயக்கப்படும் முகவரியியல் அமைப்புகள் வரை.*

நீங்கள் LangChain4j for Beginners பாடநெறியை முடித்துள்ளீர்கள். நீங்கள் கற்றுக் கொண்டவை:

- நினைவகத்துடன் உரையாடல் AI கட்டமைப்பது (Module 01)
- வெவ்வேறு பணிகளுக்கான முன் வடிவமைப்பு முறைமை (Module 02)
- உங்கள் ஆவணங்களில் பதில்களை அடிப்படையாக்க RAG பயன்படுத்துதல் (Module 03)
- தனிப்பயன் கருவிகளுடன் அடிப்படையிலான AI முகவர்களை உருவாக்குதல் (Module 04)
- LangChain4j MCP மற்றும் Agentic தொகுதிகள் மூலம் தரநிலைப்பட்ட கருவிகளை ஒருங்கிணைத்தல் (Module 05)
