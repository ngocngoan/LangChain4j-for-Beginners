## நீங்கள் கற்கப்போகும் விஷயம்

நீங்கள் உரையாடல் AI-யை கட்டியமைத்துள்ளீர்கள், ஊக்கங்கள் அதிரடியானவையாக உள்ளன, பதில்களை ஆவணங்களில் அடித்தளப்படுத்தியுள்ளீர்கள், மற்றும் உபகரணங்களுடன் கூடிய எஜென்ட்களை உருவாக்கியுள்ளீர்கள். ஆனால் அந்த அனைத்து உபகரணங்களும் உங்கள் தனிப்பட்ட பயன்பாட்டுக்கு தனிப்பயனாக உருவாக்கப்பட்டவை. யாரும் உருவாக்கி பகிரக்கூடிய தரமான பண்படுத்தப்பட்ட உபகரணங்கள் கொண்ட ஒரு சூழலுக்குக் உங்கள் AI அணுகல்களை தரவாகக் கொடுக்க முடியுமா? இந்த முறைவியலில், நீங்கள் அதைப் எப்படி செய்யலாம் என்பதை Model Context Protocol (MCP) மற்றும் LangChain4j-இன் எஜென்டிக் முறைவியலுடன் கற்கப்போகிறீர்கள். முதலில் எளிய MCP கோப்பு வாசிப்பாளரை காண்பித்து, அதன் பிறகு அதனை மேலாளர் எஜென்ட் முறைப்பாட்டுடன் எளிதில் இணைக்கப்படுவதை காட்டுகிறோம்.

## MCP என்றால் என்ன?

Model Context Protocol (MCP) AI பயன்பாடுகளுக்கு வெளிப்புற உபகரணங்களை கண்டறிந்து பயன்படுத்த ஒரு தரநிலையான வழியை அளிக்கிறது. ஒவ்வொரு தரவுத்தள ஆதாரம் அல்லது சேவைக்கான தனிப்பயன் ஒருங்கிணைப்புகளை எழுதுவதை தவிர்த்து, நீங்கள் உள்ளடக்கங்களைக் காட்டும் MCP சேவையகங்களை இணைக்கிறீர்கள். உங்கள் AI எஜென்ட் பின்னர் இந்த உபகரணங்களை தானாகவே கண்டுபிடித்து பயன்படுத்தலாம்.

கீழ்காணும் வரைபடம் வேறுபாட்டை காட்டுகிறது — MCP இல்லாமல் ஒவ்வொரு ஒருங்கிணைக்கும் தனிப்பயன் நேரடி இணைப்புகள் தேவை; MCP உடன், ஒரே நெறிமுறை உங்கள் பயன்பாட்டை எந்த உபகரணத்துடனும் இணைக்கும்:

<img src="../../../translated_images/ta/mcp-comparison.9129a881ecf10ff5.webp" alt="MCP ஒப்பீடு" width="800"/>

*ஈ MCP முன்: சிக்கலான நேரடி இணைப்புகள். MCP பிறகு: ஒரு நெறிமுறை, முடிவற்ற வாய்ப்புகள்.*

MCP AI மேம்பாட்டில் அடிப்படைக் குழப்பத்தை தீர்க்கிறது: ஒவ்வொரு ஒருங்கிணைக்கும் தனிப்பயன். GitHub அணுக விருப்பமா? தனிப்பயன் குறியீடு. கோப்புகளை படிக்க விருப்பமா? தனிப்பயன் குறியீடு. தரவுத்தளத்தை விசாரிக்க விருப்பமா? தனிப்பயன் குறியீடு. மேலும் இந்த ஒருங்கிணைப்புகள் மற்ற AI பயன்பாடுகளுடன் வேலை செய்யாது.

MCP இதை தரநிலைப்படுத்துகிறது. MCP சேவையகம் தெளிவான விளக்கங்களும் வடிவமைப்புகளும் கொண்ட உபகரணங்களை வெளிப்படுத்துகிறது. எந்த MCP கிளையண்டும் இணைக்கலாம், கிடைக்கும் உபகரணங்களை கண்டறியலாம் மற்றும் பயன்படுத்தலாம். ஒருமுறை உருவாக்கி, எங்கும் பயன்படுத்துங்கள்.

கீழ்காணும் வரைபடம் இந்த கட்டமைப்பை விளக்குகிறது — ஒரு MCP கிளையண்ட் (உங்கள் AI பயன்பாடு) பல MCP சேவையகங்களுடன் இணைகிறது, ஒவ்வொன்றும் தங்களது உபகரணங்களுடன் தரநிலையான நெறிமுறையைப் பயன்படுத்துகிறது:

<img src="../../../translated_images/ta/mcp-architecture.b3156d787a4ceac9.webp" alt="MCP கட்டமைப்பு" width="800"/>

*மாடல் சூழல் நெறிமுறை கட்டமைப்பு - தரநிலையான உபகரண திறந்தறிதல் மற்றும் செயலாக்கம்*

## MCP எப்படி செயல்படுகிறது

உள்தடத்தில், MCP பல அடுக்குகளைக் கொண்ட கட்டமைப்பைக் கொண்டது. உங்கள் Java பயன்பாடு (MCP கிளையண்ட்) கிடைக்கும் உபகரணங்களை கண்டறிந்து, JSON-RPC கோரிக்கைகளை Stdio அல்லது HTTP போன்ற பரிமாற்ற அடுப்பின் வழியாக அனுப்புகிறது, MCP சேவையகம் செயற்பாடுகளை இயற்றிக் கொள்ளும் மற்றும் முடிவுகளை திருப்பி அனுப்பும். கீழ்காணும் வரைபடம் இந்த நெறிமுறையின் ஒவ்வொரு அடுப்பையும் உடைக்கிறது:

<img src="../../../translated_images/ta/mcp-protocol-detail.01204e056f45308b.webp" alt="MCP நெறிமுறை விவரம்" width="800"/>

*MCP உள்பகுதியில் எப்படி செயல்படுகிறது — கிளையண்டுகள் உபகரணங்களை கண்டறிந்து, JSON-RPC செய்திகளை பரிமாற்று, செயல்பாடுகளை பரிமாற்ற அடுப்பில் செலுத்துகின்றன.*

**சேவை-கிளையண்ட் கட்டமைப்பு**

MCP ஒரு கிளையண்ட்-சேவையகம் மாதிரியைப் பயன்படுத்துகிறது. சேவையகங்கள் உபகரணங்களை வழங்குகின்றன — கோப்புகளை வாசித்தல், தரவுத்தளங்களை விசாரித்தல், API-களை அழைத்தல். கிளையண்டுகள் (உங்கள் AI பயன்பாடு) சேவையகத்துடன் இணைந்து, அவர்களின் உபகரணங்களைப் பயன்படுத்துகின்றன.

LangChain4j உடன் MCP பயன்படுத்த, Maven சார்ந்த இன்றியமையாத சார்பைச் சேர்க்கவும்:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```

**உபகரண கண்டறிதல்**

உங்கள் கிளையண்ட் ஒரு MCP சேவையகத்துடன் இணையும் போது, அது "உங்களிடம் எந்த உபகரணங்கள் உள்ளன?" என கேட்கிறது. சேவையகம் கிடைக்கும் உபகரணங்களின் பட்டியலை விளக்கங்களும் அளவுரு வடிவங்களும் உடன் விடைளிக்கிறது. உங்கள் AI எஜென்ட் பயனரின் கோரிக்கைகளைப் பொருத்து எந்த உபகரணங்களைப் பயன்படுத்துவது என்பதைத் தீர்மானிக்கலாம். கீழ்காணும் வரைபடம் இந்த ஒப்பந்தத்தை காட்டுகிறது — கிளையண்ட் `tools/list` கோரிக்கையை அனுப்பி, சேவையகம் அதன் கிடைக்கும் உபகரணங்களை விளக்கங்களும் அளவுரு வடிவங்களுடன் திருப்புகிறது:

<img src="../../../translated_images/ta/tool-discovery.07760a8a301a7832.webp" alt="MCP உபகரண கண்டறிதல்" width="800"/>

*AI தொடக்கத்தில் கிடைக்கும் உபகரணங்களை கண்டறிகிறது — இப்போது அது எந்த திறன்கள் உள்ளன என்பதைப் புரிந்துகொண்டு, எந்த உபகரணங்களைப் பயன்படுத்துவது என்பதற்கு முடிவெடுக்கலாம்.*

**பரிமாற்ற இயந்திரங்கள்**

MCP பல பரிமாற்ற இயந்திரங்களை ஆதரிக்கிறது. இரண்டு விருப்பங்கள் உள்ளன: Stdio (உள்ளூர் துணை செயல்பாட்டுக்கு) மற்றும் Streamable HTTP (தூர சேவையகங்களுக்கு). இந்த முறைவியல் Stdio பரிமாற்றத்தை விளக்குகிறது:

<img src="../../../translated_images/ta/transport-mechanisms.2791ba7ee93cf020.webp" alt="பரிமாற்ற இயந்திரங்கள்" width="800"/>

*MCP பரிமாற்ற இயந்திரங்கள்: தூர சேவையகங்களுக்கு HTTP, உள்ளூர் செயலிகளுக்கு Stdio*

**Stdio** - [StdioTransportDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java)

உள்ளூர் செயலிகளுக்கு. உங்கள் பயன்பாடு சேவையகத்தை துணை செயலியாக உருவாக்கி, ஸ்டாண்டர்டு உள்ளீடு/வெளியேற்றத்தின் மூலம் தொடர்புகொள்கிறது. கோப்பு முறையமைப்புத் அணுகல் அல்லது கட்டளை வரி உபகரணங்களுக்கு பயனுள்ளதாகும்.

```java
McpTransport stdioTransport = new StdioMcpTransport.Builder()
    .command(List.of(
        npmCmd, "exec",
        "@modelcontextprotocol/server-filesystem@2025.12.18",
        resourcesDir
    ))
    .logEvents(false)
    .build();
```

`@modelcontextprotocol/server-filesystem` சேவையகம் கீழ்காணும் உபகரணங்களை வெளிப்படுத்துகிறது, அனைத்தும் நீங்கள் குறிப்பிடும் கோப்புறைகளுக்குsandbox செய்யப்பட்டுள்ளது:

| உபகரணம் | விளக்கம் |
|------|-------------|
| `read_file` | ஒரு கோப்பின் உள்ளடக்கத்தை படிக்கவும் |
| `read_multiple_files` | ஒரே அழைப்பில் பல கோப்புகளை படிக்கவும் |
| `write_file` | ஒரு கோப்பை உருவாக்கவும் அல்லது மீள்பதிவேற்றவும் |
| `edit_file` | குறிக்கோளான தேடல் மற்றும் மாற்றத்தைச் செய்யவும் |
| `list_directory` | ஒரு பாதையில் கோப்புகள் மற்றும் கோப்புறைகளை பட்டியலிடவும் |
| `search_files` | ஒரு மாதிரிக்கேற்றக் கோப்புகளை மீள்மீடியாய் தேடவும் |
| `get_file_info` | கோப்பின் தரவு (அளவு, காலச்ச் சிட்டுக்கள், அனுமதிகள்) பெறவும் |
| `create_directory` | ஒரு கோப்புறை (மூத்த கோப்புறைகளையும் சேர்த்து) உருவாக்கவும் |
| `move_file` | கோப்பு அல்லது கோப்புறையை நகர்த்தவோ பெயரை மாற்றவோ செய்யவும் |

கீழ்காணும் வரைபடம் Stdio பரிமாற்றம் இயங்கும் முறையை காட்டுகிறது — உங்கள் Java பயன்பாடு MCP சேவையகத்தை துணை செயலியாக உருவாக்கி, stdin/stdout குழாய்களின் மூலம் தொடர்பு கொள்கிறது, எந்த நெட்வொர்க் அல்லது HTTP ஈடுபாடும் இல்லாமல்:

<img src="../../../translated_images/ta/stdio-transport-flow.45eaff4af2d81db4.webp" alt="Stdio பரிமாற்ற ஓட்டம்" width="800"/>

*Stdio பரிமாற்றம் செயல்படும் போது — உங்கள் பயன்பாடு MCP சேவையகத்தை துணை செயலியாக உருவாக்கி, stdin/stdout குழாய்களைப் பயன்படுத்தி தொடர்புகொள்கிறது.*

> - "LangChain4j MCP சேவையக துணை செயல்முறைகளை எப்படி நிர்வகிக்கிறது?"
> - "AI-க்கு கோப்பு அமைப்புக்கான அணுகலை கொடுப்பதில் பாதுகாப்பு விளைவுகள் என்னவாகும்?"

## எஜென்டிக் முறைவியல்

MCP தரநிலைப்படுத்தப்பட்ட உபகரணங்களை வழங்குகிறது, LangChain4j-இன் **எஜென்டிக் முறைவியல்** அந்த உபகரணங்களை ஒழுங்குபடுத்தும் எஜென்ட்களை உருவாக்க ஒரு அறிக்கையிடிய வழியைக் கொடுக்கிறது. `@Agent` குறிக்கோள் மற்றும் `AgenticServices` உங்கள் எஜென்டின் நடத்தையை காண்ணிதவைகள் கூற வரையறுக்க உதவுகின்றன.

இந்த முறைவியலில், நீங்கள் **மேலாளர் எஜென்ட்** முறைப்பாட்டை ஆராயப்போகிறீர்கள் — பயனர் கோரிக்கைகளைப் பொருத்து எந்த துணை எஜென்ட்களை அழைக்க வேண்டும் என்று டைனமிக் முறையில் தீர்மானிக்கும் முன்னேற்றமான எஜென்டிக் AI அணுகல். இரண்டு கருத்துகளையும் ஒருங்கிணைத்து, ஒரு துணை எஜென்டுக்கு MCP-ஆற்றப்பட்ட கோப்பு அணுகல் திறன்களை வழங்குவோம்.

எஜென்டிக் முறைவியலை பயன்படுத்த, Maven சார்ந்த இன்றியமையாத சார்பைச் சேர்க்கவும்:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```
> **குறிப்பு:** `langchain4j-agentic` முறைவியல் `langchain4j.mcp.version` என்னும் தனியாக நிர்ணயிக்கப்பட்ட பதிப்பு பண்பை பயன்படுத்துகிறது, ஏனெனில் இது முக்கிய LangChain4j நூலகங்களுடன் வேறுபட்ட அட்டவணையில் வெளியிடப்படுகிறது.

> **⚠️ பரிசோதனை செயல்பாடுகள்:** `langchain4j-agentic` முறைவியல் பரிசோதனை நிலை மற்றும் மாறக்கூடிய நிலையில் உள்ளது. நிலைத்தமான AI உதவியாளர்களை கட்ட அமைப்பதற்கு `langchain4j-core` மற்றும் தனிப்பயன் உபகரணங்கள் (முறைவி 04) பயன்படுத்தப்படுகிறது.

### மேலாளர் எஜென்ட்

**மேலாளர் எஜென்ட் முறைப்பாடு** என்பது **நேர்த்தியான** எஜென்டிக் AI வடிவமாகும். மேலாளர் LLM-ஐ பயன்படுத்தி பயனர் கோரிக்கைகளின் அடிப்படையில் எந்த எஜென்ட்களை அழைக்க வேண்டும் என்பதை தன்னியக்கமாக தீர்மானிக்கிறது. அடுத்த உதாரணத்தில், MCP-ஆற்றப்பட்ட கோப்பு அணுகலை LLM எஜென்ட் உடன் சேர்த்து மேற்பார்வைக் கோப்பு வாசிப்பு → அறிக்கை பணி உருவாக்குகிறோம்.

டெமோவில், `FileAgent` MCP கோப்பு அமைப்பு உபகரணங்களைப் பயன்படுத்தி கோப்பை வாசிக்கிறது, மற்றும் `ReportAgent` ஒரு கட்டமைக்கப்பட்ட அறிக்கையை உருவாக்குகிறது — நிர்வாக சுருக்கம் (1 வாக்கியம்), 3 முக்கிய புள்ளிகள் மற்றும் பரிந்துரைகள் கொண்டது. மேலாளர் இந்த ஓட்டத்தை தன்னியக்கமாக ஒருங்கிணைக்கிறது:

<img src="../../../translated_images/ta/supervisor-agent-pattern.06275a41ae006ac8.webp" alt="மேலாளர் எஜென்ட் முறைப்பாடு" width="800"/>

*மேலாளர் தனது LLM-ஐ பயன்படுத்தி எந்த எஜென்ட்களை மற்றும் எந்த வரிசையில் அழைக்க வேண்டும் என்பதை தீர்மானிக்கிறது — கடின நிரலாக்க வழிமுறை தேவையில்லை.*

நமது கோப்பு-அறிக்கைக் குழாய்தொகுதிக்கான சரியான பணிச் செய்முறை இவ்வாறு உள்ளது:

<img src="../../../translated_images/ta/file-report-workflow.649bb7a896800de9.webp" alt="கோப்பு முதல் அறிக்கை பணிதொடர்" width="800"/>

*FileAgent MCP உபகரணங்களின் மூலம் கோப்பை வாசிக்கிறது, பின்னர் ReportAgent அசல் உள்ளடக்கத்தை கட்டமைக்கப்பட்ட அறிக்கையாக மாற்றுகிறது.*

கீழ்காணும் தொடர் வரைபடம் முழு மேலாளர் ஒருங்கிணைப்பைக் காண்பிக்கிறது — MCP சேவையகத்தை உருவாக்குதல், மேலாளரின் தன்னியக்க எஜென்ட் தேர்வு, stdio மேல் உபகரண அழைப்புகள் மற்றும் இறுதி அறிக்கை வரை:

<img src="../../../translated_images/ta/supervisor-agent-sequence.1aa389b3bef99956.webp" alt="மேலாளர் எஜென்ட் தொடர்கதை வரைபடம்" width="800"/>

*மேலாளர் தன்னியக்கமாக FileAgent-ஐ அழைக்கிறது (MCP சேவையகத்தை stdio மூலமாக கோப்பை வாசிக்க), பிறகு ReportAgent-ஐ அழைத்து கட்டமைக்கப்பட்ட அறிக்கையை உருவாக்குகிறது — ஒவ்வொரு எஜென்டும் அதன் வெளியீட்டை பகிரப்பட்ட எஜென்டிக் பரப்பில் சேமிக்கிறது.*

ஒவ்வொரு எஜென்டும் அதன் வெளியீட்டை **எஜென்டிக் பரப்பு** (பகிரப்பட்ட நினைவகம்) எனும் இடத்தில் சேமிக்கிறது, இது பிற எஜென்ட்கள் முந்தய முடிவுகளை அணுக உதவுகிறது. இது MCP உபகரணங்கள் எவ்வாறு எஜென்டிக் பணிநெறிகளில் ஒருங்கிணைக்கப்படுகின்றன என்பதைக் காட்டுகிறது — மேலாளர் எவ்வாறு கோப்புகள் வாசிக்கப்படுகின்றன என்பதைப் பற்றி அறிய தேவையில்லை, `FileAgent` அதை செய்ய முடியும் என்றதையே அறிகிறது.

#### மேலாளர் எஜென்ட் எப்படி செயல்படுகிறது

எஜென்ட்களை கட்டுவதற்கு முன்பு, MCP பரிமாற்றத்தை கிளையண்டுடன் இணைத்து, அதனை `ToolProvider` ஆக மடக்க வேண்டும். இதுவே MCP சேவையக உபகரணங்கள் உங்கள் எஜென்ட்களுக்கு கிடைக்க உதவுகிறது:

```java
// பாதுகாப்பிலிருந்து MCP கிளையினை உருவாக்கவும்
McpClient mcpClient = new DefaultMcpClient.Builder()
        .transport(stdioTransport)
        .build();

// கிளையினை ஒரு ToolProvider ஆக மூடியிடுங்கள் — இது MCP கருவிகளை LangChain4j இற்கு மீதங்கள் இணைக்கிறது
ToolProvider mcpToolProvider = McpToolProvider.builder()
        .mcpClients(List.of(mcpClient))
        .build();
```

பின்னர், `mcpToolProvider`-ஐ MCP உபகரணங்கள் தேவைப்படும் எந்த எஜென்ட்டிலும் செலுத்தலாம்:

```java
// படி 1: FileAgent MCP கருவிகள் மூலம் கோப்புகளை வாசிக்கிறது
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // கோப்பு செயல்களுக்கு MCP கருவிகள் உள்ளன
        .build();

// படி 2: ReportAgent கட்டமைக்கப்பட்ட அறிக்கைகளை உருவாக்குகிறது
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// கண்காணிப்பாளர் கோப்பு → அ அறிக்கை பணிவேலைமுறைமைக்கு ஒழுங்குபடுத்துகிறார்
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // இறுதி அறிக்கையை திருப்பி அளிக்கவும்
        .build();

// கோரிக்கையின் அடிப்படையில் எந்த முகவர்களை அழைக்க வேண்டும் என்று கண்காணிப்பாளர் முடிவு செய்தார்
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```

#### FileAgent இயக்க நேரத்தில் MCP உபகரணங்களை எப்படி கண்டறிகிறது

நீங்கள் சிந்திக்கலாம்: **FileAgent npm கோப்பு அமைப்பு உபகரணங்களை எப்படி பயன்படுத்துவது என்று அறிவது எப்படி?** பதில் என்னவெனில் அது தெரியாது — **LLM** இயக்க நேரத்தில் உபகரண வடிவங்களைப் பயன்படுத்தி அதை கண்டுபிடிக்கிறது.

`FileAgent` இடைமுகம் என்பது **ஒரு ஊக்கத்துக்கான வரையறை மட்டுமே**. அதில் `read_file`, `list_directory` அல்லது பிற MCP உபகரணங்களுக்கு கடினமாக நிரலாக்கப்பட்ட அறிவு இல்லை. இதோ முழுமையான செயல்முறை:
1. **சேவையகம் தொடங்குகிறது:** `StdioMcpTransport` `@modelcontextprotocol/server-filesystem` npm பேக்கேஜை குழந்தை செயலாக துவக்குகிறது  
2. **கருவி கண்டுபிடித்தல்:** `McpClient` சேவையகத்துக்கு `tools/list` JSON-RPC கோரிக்கையை அனுப்புகிறது, சேவையகம் கருவி பெயர்கள், விளக்கங்கள் மற்றும் அளவுரு செம்மைகள் (எ.கா., `read_file` — *"ஒரு கோப்பின் முழு உள்ளடக்கத்தை படிக்க"* — `{ path: string }`) கொண்டு பதிலளிக்கிறது  
3. **செம்மை இறுத்தல்:** `McpToolProvider` கண்டறியப்பட்ட செம்மைகளைக் கட்டி LangChain4j-க்கு வழங்குகிறது  
4. **LLM முடிவு செய்கிறது:** `FileAgent.readFile(path)` அழைக்கப்படும் போது, LangChain4j சிஸ்டம் செய்தி, பயனர் செய்தி மற்றும் **கருவி செம்மைகளின் பட்டியல்** ஐ LLM-க்கு அனுப்புகிறது. LLM கருவி விளக்கங்களைப் படித்து கருவி அழைப்பை உருவாக்குகிறது (எ.கா., `read_file(path="/some/file.txt")`)  
5. **செயலாக்கம்:** LangChain4j கருவி அழைப்பை இடையூறாக்கி, அதை MCP கிளையெண்ட் வழியாக Node.js துணைப்பிரிவுக்கு அனுப்பி, முடிவைப் பெற்று அதை LLM-க்கு மீண்டும் வழங்குகிறது  

மேலே விவரிக்கப்பட்ட [கருவி கண்டுபிடித்தல்](../../../05-mcp) இயந்திரம் இதேபோன்று செயல்படுகிறது, ஆனாலும் இது குறிப்பாக முகவர் வேலையிடல் முறைக்கு பயன்படுத்தப்படுகிறது. `@SystemMessage` மற்றும் `@UserMessage` குறித்துகள் LLM நடத்தை வழிநடத்துகின்றன, இறுத்தப்பட்ட `ToolProvider` அதற்கு **சாதனங்களைக்** கொடுத்து தான் — LLM அவற்றை இயக்க நேரத்தில் இணைக்கிறது.

#### பதில் கொடுப்புக் கொள்கைகள்

`SupervisorAgent`-ஐ கட்டமைக்கும் போது, துணை முகவர்கள் பணிகளை முடித்த பிறகு பயனருக்கான இறுதி பதிலை உருவாக்குவது எப்படி என்று உறுதி செய்கிறீர்கள். கீழே உள்ள வரைபடம் மூன்று கிடைக்கும் கொள்கைகளை காட்டுகிறது — LAST நேரடியாக இறுதி முகவர் வெளியீட்டை தருகிறது, SUMMARY அனைத்து முடிவுகளையும் LLM வழியாக ஒருங்கிணைக்கிறது, SCORED முதல் கோரிக்கைக்கு எந்த பதிலை அதிக மதிப்பீடு செய்தால் அதைக் கொடுக்கிறது:  

<img src="../../../translated_images/ta/response-strategies.3d0cea19d096bdf9.webp" alt="Response Strategies" width="800"/>  

*Supervisor எப்படி இறுதி பதிலை அமைக்கிறது என்று மூன்று கொள்கைகள் — கடைசிப் பதிவை, ஒருங்கிணைந்த சுருக்கத்தை அல்லது உயர்ந்த மதிப்பீட்டை தேர்வு செய்யலாம்.*  

கிடைக்கும் கொள்கைகள்:  

| கொள்கை | விளக்கம் |  
|----------|-------------|  
| **LAST** | முதன்மை துணை முகவர் அல்லது கருவி அழைப்பின் முடிவை திருப்புகிறது. இது அடுத்தடுத்த பணியில் இறுதி முகவர் முழுமையான பதிலை உருவாக்க இருக்கும் போது பயனுள்ளது (எ.கா., ஆராய்ச்சி குழாய் "சுருக்க முகவர்"). |  
| **SUMMARY** | தனிப்பட்ட LLM மூலம் அனைத்து தொடர்புகளின் மற்றும் துணை முகவர் முடிவுகளின் சுருக்கத்தை உருவாக்கி, அதை இறுதி பதிலாக வழங்குகிறது. இது பயனருக்கு சுத்தமான, ஒருங்கிணைக்கப்பட்ட பதிலை தருகிறது. |  
| **SCORED** | உள்ளமைந்த LLM FIRST மற்றும் SUMMARY பதில்களுக்கு மதிப்பெண்களை அளித்து, அதிக மதிப்பெண் பெற்ற பதிலை திருப்புகிறது. |

#### வெளியீட்டை புரிந்து கொள்ளுதல்

டெமோ ஓட்டும்போது, Supervisor பல முகவர்களை ஒருங்கிணைப்பது எப்படி என்பதை கட்டமைப்போடு காண்பீர்கள். ஒவ்வொரு பகுதியும் இதைப் பொருள்படுத்துகிறது:  

```
======================================================================
  FILE → REPORT WORKFLOW DEMO
======================================================================

This demo shows a clear 2-step workflow: read a file, then generate a report.
The Supervisor orchestrates the agents automatically based on the request.
```
  
**தலைப்பு** படிக்கும் பணியில் இருந்து அறிக்கை உருவாக்கும் வேலை நெடுவரிசையை அறிமுகப்படுத்துகிறது.  

```
--- WORKFLOW ---------------------------------------------------------
  ┌─────────────┐      ┌──────────────┐
  │  FileAgent  │ ───▶ │ ReportAgent  │
  │ (MCP tools) │      │  (pure LLM)  │
  └─────────────┘      └──────────────┘
   outputKey:           outputKey:
   'fileContent'        'report'

--- AVAILABLE AGENTS -------------------------------------------------
  [FILE]   FileAgent   - Reads files via MCP → stores in 'fileContent'
  [REPORT] ReportAgent - Generates structured report → stores in 'report'
```
  
**வேலை சுருக்க வரைபடம்** முகவர்களுக்கு இடையேயான தரவு பணியை காட்டுகிறது. ஒவ்வொரு முகவருக்கும் தனித்தனி பங்கு உள்ளது:  
- **FileAgent** MCP கருவிகளைப் பயன்படுத்தி கோப்புகளை வாசித்து `fileContent`-ல் மொத்த உள்ளடக்கத்தை சேமிக்கிறது  
- **ReportAgent** அந்த உள்ளடக்கத்தை பயன்படுத்தி கட்டமைக்கப்பட்ட அறிக்கையை `report`-ல் உருவாக்குகிறது  

```
--- USER REQUEST -----------------------------------------------------
  "Read the file at .../file.txt and generate a report on its contents"
```
  
**பயனரின் கோரிக்கை** பணியை காட்டுகிறது. Supervisor இதைப் பார்வை செய்து FileAgent → ReportAgent-ஐ அழைக்க தீர்மானிக்கிறது.  

```
--- SUPERVISOR ORCHESTRATION -----------------------------------------
  The Supervisor decides which agents to invoke and passes data between them...

  +-- STEP 1: Supervisor chose -> FileAgent (reading file via MCP)
  |
  |   Input: .../file.txt
  |
  |   Result: LangChain4j is an open-source, provider-agnostic Java framework for building LLM...
  +-- [OK] FileAgent (reading file via MCP) completed

  +-- STEP 2: Supervisor chose -> ReportAgent (generating structured report)
  |
  |   Input: LangChain4j is an open-source, provider-agnostic Java framew...
  |
  |   Result: Executive Summary...
  +-- [OK] ReportAgent (generating structured report) completed
```
  
**Supervisor ஒருங்கிணைப்பு** 2 படி செயன்முறையை காண்பிக்கிறது:  
1. **FileAgent** MCP வழியாக கோப்பை வாசித்து உள்ளடக்கத்தை சேமிக்கிறது  
2. **ReportAgent** அந்த உள்ளடக்கத்தை பெற்று கட்டமைக்கப்பட்ட அறிக்கையை உருவாக்குகிறது  

Supervisor இந்த தீர்மானங்களை பயனர் கோரிக்கை அடிப்படையில் **தானாக** எடுத்தது.  

```
--- FINAL RESPONSE ---------------------------------------------------
Executive Summary
...

Key Points
...

Recommendations
...

--- AGENTIC SCOPE (Data Flow) ----------------------------------------
  Each agent stores its output for downstream agents to consume:
  * fileContent: LangChain4j is an open-source, provider-agnostic Java framework...
  * report: Executive Summary...
```
  
#### முகவர் தொகுதியின் அம்ச விளக்கம்  

எடுத்துக்காட்டு முகவர் தொகுதியில் சில முன்னேற்ற அம்சங்களை காட்டுகிறது. Agentic Scope மற்றும் Agent Listeners-ஐ நன்கு பார்ப்போம்.  

**Agentic Scope** `@Agent(outputKey="...")` மூலம் முகவர்கள் வெளியீடுகளை பகிர்ந்த நினைவகமாக வேலை செய்கிறது. இது:  
- பின்னர் வந்த முகவர்கள் முந்தையவர்கள் வெளியீடுகளை அணுக உதவும்  
- Supervisor இறுதி பதிலை ஒருங்கிணைக்க முடியும்  
- ஒவ்வொரு முகவரின் வெளியீடுகளை தொடர்ந்து பார் முடியும்  

கீழே உள்ள வரைபடம் agentic scope பகிரப்பட்ட நினைவகமாக, file-to-report வேலைப்பாட்டிலேயே FileAgent `fileContent` என்ற விசையின் கீழ் எழுதுகிறது, ReportAgent அதை வாசித்து `report` என்ற விசையில் எழுதுகிறது:  

<img src="../../../translated_images/ta/agentic-scope.95ef488b6c1d02ef.webp" alt="Agentic Scope Shared Memory" width="800"/>  

*Agentic Scope பகிரப்பட்ட நினைவகம் ஆக செயற்படுகிறது — FileAgent `fileContent` எழுதுகிறது, ReportAgent அதை வாசித்து `report` எழுதுகிறது, கடைசி முடிவை உங்கள் குறியீடு வாசிக்கிறது.*  

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // FileAgent இலிருந்து மூல கோப்பு தரவு
String report = scope.readState("report");            // ReportAgent இலிருந்து அமைக்கப்பட்ட அறிக்கை
```
  
**Agent Listeners** முகவர் செயற்பாட்டை கண்காணித்து பிழைகளை கண்டறிய உதவுகின்றன. டெமோவில் பிக்சல் வெளியீடு ஒரு AgentListener-இல் இருந்து வருகிறது, இது ஒவ்வொரு முகவர் அழைப்பிலும் இணைகிறது:  
- **beforeAgentInvocation** - Supervisor முகவரைக் தேர்ந்தெடுக்கும் போது அழைக்கப்படுகிறது, எந்த முகவர் தேர்ந்தெடுக்கப்பட்டார் மற்றும் ஏன் என்பதை காண உதவும்  
- **afterAgentInvocation** - முகவர் முடிந்ததும் அழைக்கப்படுகிறது, அதன் முடிவை காட்டுகிறது  
- **inheritedBySubagents** - உண்மையானால், இந்த நுகர்வோர் அனைத்து துணை முகவர்களையும் கண்காணிக்கும்  

கீழே உள்ள வரைபடம் Agent Listener வாழ்நாள் முழுவதையும், `onError` முகவர் செயல்பாட்டின்போது தோல்விகள் ஏற்படுச்சொல்லுகிறது:  

<img src="../../../translated_images/ta/agent-listeners.784bfc403c80ea13.webp" alt="Agent Listeners Lifecycle" width="800"/>  

*Agent Listeners செயல்பாட்டு வாழ்நாளில் இணைக்கப்படுகிறார்கள் — முகவர்கள் துவங்கும், முடியும் அல்லது பிழைகள் ஏற்படும் நேரங்களை கண்காணிக்கலாம்.*  

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // அனைத்து துணை-செயலாளர்களுக்கும் பரவச் செய்க
    }
};
```
  
Supervisor முறையின் போது, `langchain4j-agentic` தொகுதி பல சக்திவாய்ந்த வேலைநெடுவரிசை மாதிரிகளை வழங்குகிறது. கீழே உள்ளது அனைத்தையும் காட்டும் வரைபடம் — எளிய தொடர் குழாய்ச் செயல்முறைகளிலிருந்து மனித முகவரி ஒப்புதல்கள் வேலைநெடுவரிசைகளுக்கு வரை:  

<img src="../../../translated_images/ta/workflow-patterns.82b2cc5b0c5edb22.webp" alt="Agent Workflow Patterns" width="800"/>  

*ஐந்து முகவர் வேலைநெடுவரிசை மாதிரிகள் — தொடர் குழாய்கள் முதல் மனித இணைப்பு ஒப்புதல் வேலைநெடுவரிசைகள் வரை.*  

| மாதிரி | விளக்கம் | பயன்பாடு |  
|---------|-------------|----------|  
| **Sequential** | முகவர்களை வரிசைப்படுத்தி இயக்கு, வெளியீடு அடுத்துக்கு செல்லும் | குழாய்கள்: ஆராய்ச்சி → பகுப்பு → அறிக்கை |  
| **Parallel** | முகவர்களை ஒரே நேரத்தில் இயக்கு | சுயாதீன பணிகள்: வானிலை + செய்திகள் + பங்குகள் |  
| **Loop** | நிபந்தனை பூர்த்தி ஆகும் வரை முறையாக நடத்தல் | தர மதிப்பீடு: மதிப்பெண் ≥ 0.8 வரை சீரமைத்தல் |  
| **Conditional** | நிபந்தனைகளுக்கு ஏற்ப வழி மாற்றல் | வகைப்படுத்தல் → நிபுணர் முகவருக்கு அனுப்பு |  
| **Human-in-the-Loop** | மனித ஆய்வு புள்ளிகளை சேர் | ஒப்புதல் வேலைநெடுவரிசைகள், உள்ளடக்க விமர்சனம் |  

## முக்கியக் கருத்துகள்

MCP மற்றும் agentic தொகுதியின் நடைமுறையை ஆராய்ந்த பிறகு, எப்போது எந்த முறையை பயன்படுத்தவேண்டும் என்பதைக் காணலாம்.  

MCP-யின் மிகப்பெரிய நன்மை அதன் வளர்கிற சூழல் அமைப்பில் உள்ளது. கீழே ஒரு ஒரே பொதுவான நெறிமுறை உங்கள் AI செயலியை வெவ்வேறு MCP சேவையகங்களுடன் இணைக்கிறது — கோப்பு முறைமை மற்றும் தரவுத்தள அணுகல், GitHub, மின்னஞ்சல், வலை ஸ்க்ராப்பிங் மற்றும் மேலும் பல:  

<img src="../../../translated_images/ta/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="MCP Ecosystem" width="800"/>  

*MCP ஒரு ஒருங்கிணைந்த நெறிமுறை சூழல் எழுதுகிறது — எந்த MCP பொருத்தமான சேவையகமும் எந்த MCP பொருத்தமான கிளையெண்ட் மூலம் இயங்குகிறது, செயலிகள் இடையே கருவி பகிர்வுக்கு உதவுகிறது.*  

**MCP** தற்போதைய கருவிப் சூழல்களைப் பயன்படுத்த விரும்பும் போது, பன்முகப் பயன்பாடுகளுக்குப் பகிரக்கூடிய கருவிகளை உருவாக்கும் போது, மூன்றாம் பக்கம் சேவைகளுடன் பொதுவான நெறிமுறைகளைப் பயன்படுத்தும் போது, அல்லது குறியீட்டை மாற்றாமல் கருவி செயல்பாடுகளை மாற்ற விரும்பும் போது சிறந்தது.  

**Agentic தொகுதி** அறிவுறுத்தப்பட்ட முகவர் விளக்கங்களுடன் (`@Agent` குறித்துகள்), வேலைநெடுவரிசை ஒருங்கிணைப்புடன் (தொடர், சுழற்சி, ஒத்துழைப்பு), கட்டளைகளுக்குப் பதிலாக இடைமுக அடிப்படையிலான முகவர் வடிவமைப்பை விரும்பும் போது, அல்லது கலந்த பல முகவர்கள் வெளியீடுகளை `outputKey` மூலம் பகிரும் போது சிறந்தது.  

**Supervisor Agent முறை** முன்னறிவாடல் தெரியாத, LLM முடிவெடுக்க வேண்டும் என்று விரும்பும் போது, பல நிபுண முகவர்கள் மற்றும் ஓட்டத்துடன் இருக்கும் போது, உரையாடல் முறைமைகளை கட்டமைக்கும் போது மற்றும் மிகவும் நெகிழ்வான, உகந்த முகவர் நடத்தை வேண்டும் எனும் நிலையில் சிறந்தது.  

Module 04-இல் இருந்து உள்ள தனிப்பயன் `@Tool` முறைகளுக்கும் MCP கருவிகளுக்கும் இடையேயான வியாபாரங்களை கீழே காணலாம் — தனிப்பயன் கருவிகள் பயன்பாட்டுக்கு இழைக்கப்பட்ட கூப்பு மற்றும் முழுமையான வகை பாதுகாப்பை தருகின்றன, MCP கருவிகள் இயல்பான, மறுபயன்பாட்டுக்கேற்ப இணைப்புக்களை தருகின்றன:  

<img src="../../../translated_images/ta/custom-vs-mcp-tools.c4f9b6b1cb65d8a1.webp" alt="Custom Tools vs MCP Tools" width="800"/>  

*தனிப்பயன் @Tool முறைகளை MCP கருவிகளுடன் எப்போது பயன்படுத்துவது — தனிப்பயன் கருவிகள் பயன்பாட்டின் தீர்விற்கு, MCP கருவிகள் நிலையான இணைப்புகளுக்கு.*
