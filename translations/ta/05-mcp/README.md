## நீங்கள் கற்பது என்ன

நீங்கள் உரையாடல் AI ஐ உருவாக்கி, முன்னோட்டங்களை கைப்பற்றியுள்ளீர்கள், ஆவணங்களில் பதில்களை நிலைநாட்டி, கருவிகளுடன் ஏஜென்டுகளை உருவாக்கியுள்ளீர்கள். ஆனால் எல்லா கருவிகளும் உங்கள் தனிப்பட்ட பயன்பாட்டுக்காக தனிப்பயனாக்கப்பட்டவை. எந்த ஒருவனும் உருவாக்கி பகிரக்கூடிய ஒரு நிலையான கருவி சுற்றுப்புறத்துக்கான அணுகலை உங்கள் AI க்கு வழங்க முடிந்தால் எப்படி இருக்கும்? இந்த மொத்தத்தில், மாதிரி சூழல் நெறிமுறை (MCP) மற்றும் LangChain4j இன் ஏஜென்டிக் மொத்தத்துடன் அதைச் செய்வதைக் கற்றுக்கொள்ளப்போகிறீர்கள். முதலில் எளிய MCP கோப்பு வாசிப்பாளரை காண்பித்து, பின்னர் அது எப்படி மேம்பட்ட ஏஜென்டிக் பணியாட்களில் மேற்பார்வையாளர் ஏஜென்ட் மாதிரியைப் பயன்படுத்தி எளிதாக ஒருங்கிணைக்கப்படுகிறதென காட்டுகிறோம்.

## MCP என்றால் என்ன?

மாதிரி சூழல் நெறிமுறை (MCP) இதுவே தருகிறது - AI பயன்பாடுகள் வெளிப்புற கருவிகளை கண்டறிந்து பயன்படுத்துவதற்கான ஒரு நிலையான வழி. ஒவ்வொரு தரவு மூலத்துக்கும் அல்லது சேவைக்கும் தனிப்பயன் ஒருங்கிணைப்புகளை எழுதுவதற்கு பதிலாக, உங்கள் AI ஏஜென்ட் தம்முடைய திறன்களை ஒருங்கிணைந்த வடிவில் வெளிப்படுத்தும் MCP சேவையகங்களுக்கு இணைகிறது. பிறகு உங்கள் AI ஏஜென்ட் அந்த கருவிகளை தானாக கண்டுபிடித்து பயன்படுத்தலாம்.

<img src="../../../translated_images/ta/mcp-comparison.9129a881ecf10ff5.webp" alt="MCP ஒப்பீடு" width="800"/>

*MCP க்கு முன்: சிக்கலான புள்ளி-புள்ளி ஒருங்கிணைப்புகள். MCP க்கு பிறகு: ஒரு நெறிமுறை, முடிவற்ற வாய்ப்புகள்.*

MCP AI அபிவிருத்தியில் ஒரு அடிப்படைக் பிரச்சினையை தீர்க்கிறது: ஒவ்வொரு ஒருங்கிணைப்பு தனிப்பயனாக இருக்கும். GitHub க்கு அணுக விரும்புகிறீர்களா? தனிப்பயன் குறியீடு. கோப்புகளை வாசிக்க விரும்புகிறீர்களா? தனிப்பயன் குறியீடு. தரவுத்தளத்தை வினவ விரும்புகிறீர்களா? தனிப்பயன் குறியீடு. இந்த ஒருங்கிணைப்புகள் மற்ற AI பயன்பாடுகளுடன் பணியாற்றுவதில்லை.

MCP இதை நிலைநாட்டுகிறது. MCP சேவையகம் தெளிவான விளக்கங்களும் வடிவமைப்புகளும் கொண்ட கருவிகளை வெளிப்படுத்துகிறது. எந்த MCP கிளையன்டும் இணைத்து, கிடைக்கும் கருவிகள் அனைத்தையும் கண்டறிந்து, பயன்படுத்தி கொள்ள முடியும். ஒருமுறை கட்டி, எங்கும் பயன்படுத்துங்கள்.

<img src="../../../translated_images/ta/mcp-architecture.b3156d787a4ceac9.webp" alt="MCP கட்டமைப்பு" width="800"/>

*மாதிரி சூழல் நெறிமுறை கட்டமைப்பு - நிலையான கருவி கண்டறிதல் மற்றும் செயல்படுத்தல்*

## MCP எப்படி செயல்படுகிறது

<img src="../../../translated_images/ta/mcp-protocol-detail.01204e056f45308b.webp" alt="MCP நெறிமுறை விவரம்" width="800"/>

*MCP எப்படி வேலை செய்கிறது — கிளையன்டுகள் கருவிகளை கண்டுபிடித்து, JSON-RPC செய்திகளை பரிமாறி, ஒரு டிரான்ஸ்போர்ட் பரப்பு மூலமாக செயல்படுத்துகிறது.*

**சேவையகம்-கிளையன்ட் கட்டமைப்பு**

MCP ஒரு கிளையன்ட்-சேவையக மாதிரியைப் பயன்படுத்துகிறது. சேவையகங்கள் கருவிகள் வழங்குகின்றன - கோப்புகளை வாசித்தல், தரவுத்தளங்கள் வினவுதல், API களை அழைத்தல். கிளையன்டுகள் (உங்கள் AI பயன்பாடு) சேவையகங்களுக்கு இணைந்து அவர்களின் கருவிகளைப் பயன்படுத்துகின்றன.

LangChain4j உடன் MCP ஐ பயன்படுத்த, இதோ Maven சார்ந்த அமைப்பு சேர்க்கவும்:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```

**கருவி கண்டறிதல்**

உங்கள் கிளையன்ட் MCP சேவையகத்துடன் இணைகையில், "உங்களுக்கு எந்த கருவிகள் உள்ளன?" என்று கேட்கிறது. சேவையகம் கிடைக்கும் கருவிகளின் பட்டியலுடன் ஒவ்வொன்றின் விளக்கங்களும் அளவுரு வடிவமைப்புகளும் அனுப்புகிறது. உங்கள் AI ஏஜென்ட் பிறகு பயனர் கோரிக்கைகளின்படி எந்த கருவிகளைப் பயன்படுத்துவது என்பதைக் தீர்மானிக்கிறது.

<img src="../../../translated_images/ta/tool-discovery.07760a8a301a7832.webp" alt="MCP கருவி கண்டறிதல்" width="800"/>

*AI துவக்கத்தில் கிடைக்கும் கருவிகளைக் கண்டறிகிறது — இப்போது எவை கிடைக்கும் என்று அறிவது மற்றும் எவை பயன்படுத்த வேண்டும் என்று முடிவெடுக்கிறது.*

**டிரான்ஸ்போர்ட் செயல்முறைகள்**

MCP வெவ்வேறு டிரான்ஸ்போர்ட் செயல்முறைகளை ஆதரிக்கிறது. இந்த மொத்தத்தில் உள்ளூர் விடயங்களுக்கு Stdio டிரான்ஸ்போர்டை காட்டுகிறோம்:

<img src="../../../translated_images/ta/transport-mechanisms.2791ba7ee93cf020.webp" alt="டிரான்ஸ்போர்ட் செயல்முறைகள்" width="800"/>

*MCP டிரான்ஸ்போர்ட் செயல்முறைகள்: தொலைநிலை சேவையகங்களுக்கு HTTP, உள்ளூரான செயல்முறைகளுக்கு Stdio*

**Stdio** - [StdioTransportDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java)

உள்ளூர் செயல்முறைகளுக்காக. உங்கள் பயன்பாடு ஒரு சேவையகத்தை துணைசெயல்முறையாக தொடங்கி, பராமரிப்பு உள்ளீட்டும்/வெளியீட்டும் வழியாக தொடர்பு கொள்கிறது. கோப்பு முறை அணுகல் அல்லது கட்டளை வரிசை கருவிகளுக்கு பயனுள்ளது.

```java
McpTransport stdioTransport = new StdioMcpTransport.Builder()
    .command(List.of(
        npmCmd, "exec",
        "@modelcontextprotocol/server-filesystem@2025.12.18",
        resourcesDir
    ))
    .logEvents(false)
    .build();
```

<img src="../../../translated_images/ta/stdio-transport-flow.45eaff4af2d81db4.webp" alt="Stdio டிரான்ஸ்போர்ட் ஓட்டம்" width="800"/>

*Stdio டிரான்ஸ்போர்ட் செயலில் — உங்கள் பயன்பாடு MCP சேவையகத்தை ஒரு பிள்ளை செயல்முறையாக தொடங்கி stdin/stdout குழாய்கள் மூலம் தொடர்பு கொள்கிறது.*

> **🤖 [GitHub Copilot](https://github.com/features/copilot) அட்டப்பேசி மூலம் முயற்சிக்கவும்:** [`StdioTransportDemo.java`](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java) திறந்து கேளுங்கள்:
> - "Stdio டிரான்ஸ்போர்ட் எப்படி செயல்படுகிறது மற்றும் HTTP வுடன் எப்போது பயன்படுத்த வேண்டும்?"
> - "LangChain4j எப்படி MCP சேவையக செயல்முறைகளின் லைஃப்சைகிளை நிர்வகிக்கிறது?"
> - "கோப்புத்தொகுப்பு பதிவுகளுக்கு AI அணுகலை தருவதன் பாதுகாப்பு விளைவுகள் என்ன?"

## ஏஜென்டிக் மொத்தம்

MCP நிலையான கருவிகளை வழங்கும் போது, LangChain4j இன் **ஏஜென்டிக் மொத்தம்** அந்த கருவிகளை ஒழுங்கமைக்கும் ஏஜென்டுகளை உருவாக்க ஒரு அறிவித்தமை முறையை வழங்குகிறது. `@Agent` குறிப்பு மற்றும் `AgenticServices` உங்களை நிர்வாக குறியீடு இல்லாமல் இடைமுகங்களின் மூலமாக ஏஜென்ட் நெறியினை வரைய அனுமதிக்கின்றன.

இந்த மொத்தத்தில், நீங்கள் **மேற்பார்வையாளர் ஏஜென்ட்** மாதிரியை ஆராய்வீர்கள் — ஒரு மேம்பட்ட ஏஜென்டிக் AI அணுகல், இதில் "மேற்பார்வையாளர்" ஏஜென்ட் பயனர் கோரிக்கையின் அடிப்படையில் கிளை ஏஜென்டுகளை எப்போது அழைக்க முடிவு செய்கிறது. நாங்கள் இரு கருத்துக்களையும் இணைத்து, எங்கள் கிளை ஏஜென்டுகளில் ஒருவருக்கு MCP இயக்கமான கோப்பு அணுகல் திறன்களை வழங்கும்.

ஏஜென்டிக் மொத்தத்தை பயன்படுத்த, Maven சார்ந்த இணைப்பை சேர்க்கவும்:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```

> **⚠️ பரிசோதனையாக உள்ளது:** `langchain4j-agentic` மொத்தம் பரிசோதனைக் கால போன்றது, மாற்றங்களுக்கு உட்படுகிறது. நிலையான AI உதவியாளர்களை கட்டுவதற்கான வழி `langchain4j-core` மற்றும் தனிப்பயன் கருவிகளுடன் (Module 04).

## உதாரணங்களை இயக்குவது

### முன்பரிசீலனைகள்

- Java 21+, Maven 3.9+
- Node.js 16+ மற்றும் npm (MCP சேவையகங்களுக்கு)

  - `AZURE_OPENAI_ENDPOINT`, `AZURE_OPENAI_API_KEY`, `AZURE_OPENAI_DEPLOYMENT` (Modules 01-04 போல்)

### மேற்பார்வையாளர் ஏஜென்ட்

**மேற்பார்வையாளர் ஏஜென்ட் மாதிரி** என்பது **இடையூறு** வகையான ஏஜென்டிக் AI ஆகும். ஒரு மேற்பார்வையாளர் பயனர் கோரிக்கைக்கு அடிப்படையில் எந்த ஏஜென்டுகளை அழைக்கவேண்டும் என்று தன்னிச்சையாக LLM பயன்படுத்தி முடிவு செய்கிறது. அடுத்த உதாரணத்தில், நாங்கள் MCP இயக்கமான கோப்பு அணுகலை LLM ஏஜென்டுடன் இணைத்து மேற்பார்வையாளர் மூலம் கோப்பை வாசித்து அறிக்கை உருவாக்கும் பணியாட்டை செய்கின்றோம்.

டெமோவில், `FileAgent` MCP கோப்பு முறை கருவிகளைக் கொண்டு கோப்பைப் படிக்கிறது, `ReportAgent` ஒரு கட்டமைக்கப்பட்ட அறிக்கையை உருவாக்கி அதில் ஒரு நிர்வாக சுருக்கம் (ஒரு வாக்கியம்), 3 முக்கிய புள்ளிகள் மற்றும் பரிந்துரைகள் உள்ளன. மேற்பார்வையாளர் இந்த பணியாட்டை தானாக ஒழுங்குபடுத்துகிறது:

<img src="../../../translated_images/ta/supervisor-agent-pattern.06275a41ae006ac8.webp" alt="மேற்பார்வையாளர் ஏஜென்ட் மாதிரி" width="800"/>

*மேற்பார்வையாளர் தன் LLM மூலம் எந்த ஏஜென்ட்களை எப்போது அழைக்கவேண்டும் என்று முடிவு செய்கிறது — கடுமையான வழிசெலுத்தல் தேவையில்லை.*

எங்கள் கோப்பு-அறிக்கை பணியாட்டின் சுழற்சி இங்கே:

<img src="../../../translated_images/ta/file-report-workflow.649bb7a896800de9.webp" alt="கோப்பு-அறிக்கை பணியாட்டம்" width="800"/>

*FileAgent MCP கருவிகள் மூலம் கோப்பை வாசிக்கிறது, பின்னர் ReportAgent அதனுடைய அரிதான உள்ளடக்கத்தை கட்டமைக்கப்பட்ட அறிக்கையாக மாற்றுகிறது.*

ஒவ்வொரு ஏஜென்டும் தன் வெளியீட்டை **Agentic Scope** (பகிர்ந்த நினைவகம்) இல் சேமிக்கிறது, இது பின்னர் வரும் ஏஜென்டுகளுக்கு முந்தைய முடிவுகளை அணுக அனுமதிக்கிறது. இது MCP கருவிகள் ஏஜென்டிக் பணியாட்களில் எளிதாக இணைகின்றன என்பதை எடுத்துக்காட்டுகிறது — மேற்பார்வையாளர் *எப்படி* கோப்புகள் வாசிக்கப்படுகிறதென அறிய தேவையில்லை, FileAgent அதை செய்ய முடியும் என்பது போதும்.

**PowerShell:**

**VS Code பயன்படுத்தும் போது:** `SupervisorAgentDemo.java` ஐ வலது கிளிக்கவும், **"Run Java"** என்பதை தேர்வு செய்யவும் (உங்கள் `.env` கோப்பு அமைந்துள்ளதை உறுதி செய்யவும்).

#### மேற்பார்வையாளர் எப்படி செயல்படுகிறது

```java
// படி 1: FileAgent MCP கருவிகளைக் கொண்டு கோப்புகளை வாசிக்கிறது
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // கோப்பு செயலிகளுக்கான MCP கருவிகள் உள்ளன
        .build();

// படி 2: ReportAgent கட்டமைக்கப்பட்ட அறிக்கைகளை உருவாக்குகிறது
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// மேற்பார்வையாளர் கோப்பு → அறிக்கை பணிவழிமுறையை ஒருங்கிணைக்கிறார்
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // இறுதி அறிக்கையை திருப்பவும்
        .build();

// கோரிக்கையின் அடிப்படையில் எதை அழைக்க வேண்டும் என்பதை மேற்பார்வையாளர் தீர்மானிக்கிறார்
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```

#### பதிலளிக்கும் கொள்கைகள்

`SupervisorAgent` ஐ அமைக்கும் போது, அது துணை ஏஜென்ட்கள் பணிகளை முடித்ததும் பயனருக்கு இறுதியான பதிலை எப்படி உருவாக்க வேண்டும் என்பதைக் குறிப்பிடுகிறீர்கள்.

<img src="../../../translated_images/ta/response-strategies.3d0cea19d096bdf9.webp" alt="பதில் கொள்கைகள்" width="800"/>

*மேற்பார்வையாளர் இறுதியான பதிலை உருவாக்கும் 3 கொள்கைகள் — கடைசி ஏஜென்டின் வெளியீடு வேண்டும், சுருக்கமான தொகுப்பு வேண்டும் அல்லது சிறந்த மதிப்பெண் பெற்ற விருப்பம் வேண்டும் என்ற அடிப்படையில் தேர்வு செய்யவும்.*

கிடைக்கும் கொள்கைகள்:

| கொள்கை | விளக்கம் |
|----------|-------------|
| **LAST** | மேற்பார்வையாளர் கடைசி துணை ஏஜென்ட் அல்லது கருவி வழங்கிய வெளியீட்டை திரும்ப அளிக்கிறது. இது பணியாட்டின் இறுதி ஏஜென்ட் முழுமையான இறுதி பதிலை உருவாக்கும் போது பயனுள்ளது (எ.கா., ஒரு "சுருக்கம் ஏஜென்ட்" ஆராய்ச்சி பணியாட்டில்). |
| **SUMMARY** | மேற்பார்வையாளர் தன் உள்ளக மொழி மாதிரியை (LLM) பயன்படுத்தி முழு இடைச் செயல்பாடும் மற்றும் துணை ஏஜென்ட் வெளியீடுகளின் சுருக்கத்தை உருவாக்கி, அதனை இறுதி பதிலாக வழங்குகிறது. இது பயனருக்கு ஒருங்கிணைக்கப்பட்ட, தெளிவான பதிலை வழங்குகிறது. |
| **SCORED** | முறைமை உள்ளக LLM ஐப் பயன்படுத்தி LAST பதிலையும் SUMMARY ஐ மதிப்பீடு செய்து, ஆரம்ப பயனர் கோரிக்கையுடன் ஒப்பிட்டு, அதிக மதிப்பெண் பெற்ற வெளியீட்டை வழங்குகிறது. |

#### வெளியீட்டை புரிந்து கொள்வது

டெமோ இயக்கும்போது, மேற்பார்வையாளர் பல ஏஜென்ட்களை எப்படி ஒருங்கிணைக்கிறது என்பதற்கான கட்டமைக்கப்பட்ட நடைமுறையை காண்வீர்கள். ஒவ்வொரு பகுதியின் பொருள்:

```
======================================================================
  FILE → REPORT WORKFLOW DEMO
======================================================================

This demo shows a clear 2-step workflow: read a file, then generate a report.
The Supervisor orchestrates the agents automatically based on the request.
```

**தலைப்பு** பணியாட்டு கருத்தை அறிமுகப்படுத்துகிறது: கோப்பை வாசித்தல் முதல் அறிக்கை உருவாக்கம் வரை, கவனமாக ஆன ஒரு குழாய்.

```
--- WORKFLOW ---------------------------------------------------------
  ┌─────────────┐      ┌──────────────┐
  │  FileAgent  │ ───▶ │ ReportAgent  │
  │ (MCP tools) │      │  (pure LLM)  │
  └─────────────┘      └──────────────┘
   outputKey:           outputKey:
   'fileContent'        'report'

--- AVAILABLE AGENTS -------------------------------------------------
  [FILE]   FileAgent   - Reads files via MCP → stores in 'fileContent'
  [REPORT] ReportAgent - Generates structured report → stores in 'report'
```

**பணியாட்டு வரைபடம்** ஏஜென்ட்கள் இடையேயான தரவு ஓட்டம் காட்டுகிறது. ஒவ்வொரு ஏஜென்டுக்கும் தனிப்பட்ட பங்கு உள்ளது:
- **FileAgent** MCP கருவிகளைக் கொண்டு கோப்புகளை வாசித்து மூல உள்ளடக்கத்தை `fileContent` இல் சேமிக்கிறது
- **ReportAgent** அந்த உள்ளடக்கத்தை வைத்து `report` என்ற கட்டமைக்கப்பட்ட அறிக்கையை உருவாக்குகிறது

```
--- USER REQUEST -----------------------------------------------------
  "Read the file at .../file.txt and generate a report on its contents"
```

**பயனர் கோரிக்கை** பணியை காட்டுகிறது. மேற்பார்வையாளர் இதை பகுப்பாய்வு செய்து FileAgent → ReportAgent ஐ அழைக்க முடிவு செய்கிறது.

```
--- SUPERVISOR ORCHESTRATION -----------------------------------------
  The Supervisor decides which agents to invoke and passes data between them...

  +-- STEP 1: Supervisor chose -> FileAgent (reading file via MCP)
  |
  |   Input: .../file.txt
  |
  |   Result: LangChain4j is an open-source, provider-agnostic Java framework for building LLM...
  +-- [OK] FileAgent (reading file via MCP) completed

  +-- STEP 2: Supervisor chose -> ReportAgent (generating structured report)
  |
  |   Input: LangChain4j is an open-source, provider-agnostic Java framew...
  |
  |   Result: Executive Summary...
  +-- [OK] ReportAgent (generating structured report) completed
```

**மேற்பார்வையாளர் ஒருங்கமைப்பு** இரு படி செயல்முறையை காட்டுகிறது:
1. **FileAgent** MCP மூலம் கோப்பை வாசித்து உள்ளடக்கம் சேமிக்கிறது
2. **ReportAgent** உள்ளடக்கம் பெற்று கட்டமைக்கப்பட்ட அறிக்கையை உருவாக்குகிறது

மேற்பார்வையாளர் பயனர் கோரிக்கையின் அடிப்படையில் **தன்னியக்கமாக** இந்த முடிவுகளை எடுத்துள்ளது.

```
--- FINAL RESPONSE ---------------------------------------------------
Executive Summary
...

Key Points
...

Recommendations
...

--- AGENTIC SCOPE (Data Flow) ----------------------------------------
  Each agent stores its output for downstream agents to consume:
  * fileContent: LangChain4j is an open-source, provider-agnostic Java framework...
  * report: Executive Summary...
```

#### ஏஜென்டிக் மொத்த அம்சங்கள் விளக்கம்

இந்த உதாரணம் ஏஜென்டிக் மொத்தத்தின் பல மேம்பட்ட அம்சங்களை காட்டுகிறது. Agentic Scope மற்றும் Agent Listeners இனை நெருங்கி பார்க்கலாம்.

**Agentic Scope** என்பது `@Agent(outputKey="...")` பயன்படுத்தி ஏஜென்டுகள் தங்கள் முடிவுகளை சேமிக்கும் பகிர்ந்த நினைவகம் ஆகும். இது:

- பிறகு வரும் ஏஜென்டுகள் முன் ஏஜென்டுகளின் வெளியீடுகளை அணுகலாம்
- மேற்பார்வையாளர் இறுதி பதிலை தொகுக்கலாம்
- நீங்கள் ஒவ்வொரு ஏஜென்ட் தயாரித்ததை பரிசீலிக்கலாம்

<img src="../../../translated_images/ta/agentic-scope.95ef488b6c1d02ef.webp" alt="Agentic Scope பகிர்ந்த நினைவகம்" width="800"/>

*Agentic Scope பகிர்ந்த நினைவகமாக செயல்படுகிறது — FileAgent `fileContent` எழுதுகிறது, ReportAgent அதை வாசித்து `report` எழுதுகிறது, உங்கள் குறியீடு இறுதிப் பெறுமானத்தை வாசிக்கிறது.*

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // FileAgent இலிருந்து அசல் கோப்பு தரவு
String report = scope.readState("report");            // ReportAgent இலிருந்து அமைக்கப்பட்ட அறிக்கை
```

**Agent Listeners** ஏஜென்ட் செயல்பாட்டை கண்காணித்து பிழையற்றவாறு செய்வதற்கு உதவுகிறது. டெமோவில் காணப்படும் படிநிலை-by-படிநிலை வெளியீடு ஒவ்வொரு ஏஜென்ட் அழைப்புக்கும் இணைக்கப்பட்ட AgentListener இல் இருந்து வருகிறது:
- **beforeAgentInvocation** - மேற்பார்வையாளர் ஒரு முகவரைத் தேர்ந்தெடுக்கும் போது அழைக்கப்படுகிறது, எந்த முகவரும் எதற்கு தேர்ந்தெடுக்கப்பட்டது என்பதை உங்களுக்கு காண்பிக்கச் செய்யும்
- **afterAgentInvocation** - ஒரு முகவர் முடிந்த போது அழைக்கப்படுகிறது, அதன் முடிவைக் காட்டுகிறது
- **inheritedBySubagents** - உண்மையாயின், ஞாபகத்தளத்தில் உள்ள அனைத்து முகவர்களையும் சரியாக கண்காணிக்கிறது

<img src="../../../translated_images/ta/agent-listeners.784bfc403c80ea13.webp" alt="Agent Listeners Lifecycle" width="800"/>

*முகவர் শ্রோதேர் செயல்பாட்டின் வாழ்க்கைச் சூழலில் இணைகின்றது — முகவர்கள் எப்போது துவங்குகின்றனர், முடிக்கின்றனர் அல்லது பிழைகளை சந்திக்கின்றனர் என்பதை கண்காணிக்கவும்.*

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // அனைத்து துணை முகவர்களுக்கும் பரவச் செய்க
    }
};
```

மேற்பார்வையாளர் மாதிரிக்கு அப்பால், `langchain4j-agentic` தொகுதி பல சக்திவாய்ந்த வேலைசாலை மாதிரிகள் மற்றும் அம்சங்களை வழங்குகிறது:

<img src="../../../translated_images/ta/workflow-patterns.82b2cc5b0c5edb22.webp" alt="Agent Workflow Patterns" width="800"/>

*முகவர்களை ஒருங்கிணைக்கும் ஐந்து வேலைசாலை மாதிரிகள் — எளிய தொடர் குழாய்கள் முதல் மனித ஏற்றுக்கொள்வுநிலை வரை.*

| மாதிரி | விளக்கம் | பயன்பாட்டு நிலை |
|---------|-------------|----------|
| **தொடர்** | முகவர்கள் வரிசைப்படி இயங்குகிறார்கள், வெளியீடு அடுத்ததாக இயக்கம் பெறுகிறது | குழாய்கள்: ஆராய்ச்சி → பகுப்பாய்வு → அறிக்கை |
| **ஒத்தொரு நேரத்தில்** | முகவர்கள் ஒரே நேரத்தில் இயங்குகிறார்கள் | சுயாதீன பணிகள்: வானிலை + செய்திகள் + பங்கு நிலைகள் |
| **மடக்கு** | நிபந்தனை நிறைவேறும்வரை மீண்டும் மீண்டும் செயல் படுத்தல் | தர மதிப்பீடு: மதிப்பெண் ≥ 0.8 வரை திருத்தம் செய் |
| **நிபந்தனை சார்ந்த** | நிபந்தனைகள் அடிப்படையில் பாதையை மாற்றுக | வகைப்படுத்து → சிறப்பு முகவருக்கு வழி |
| **மனிதம் ஈடுபட்ட** | மனித சரிபார்ப்புகளைச் சேர்க்கவும் | அங்கீகாரம் வேலைசாலைகள், உள்ளடக்க மதிப்பாய்வு |

## முக்கியக் கருத்துக்கள்

இப்போது நீங்கள் MCP மற்றும் agentic தொகுதியின் செயல்பாட்டை ஆராய்ந்துள்ளீர்கள், ஒவ்வொரு முறையிலும் எப்போது பயன்படுத்துவது என்பதை சுருக்கமாக பார்ப்போம்.

<img src="../../../translated_images/ta/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="MCP Ecosystem" width="800"/>

*MCP ஒரு பொதுவான நெறிமுறை சூழலை உருவாக்குகிறது — எந்த MCP-ஊக்கத்துடன் அமைந்த சேவையகம், எந்த MCP-ஊக்கத்துடன் அமைந்த கிளையன்ட் உடன் வேலைசெய்கிறது, செயலிகளுக்கு இடையே கருவிகளின் பகிர்வு சாத்தியமாகிறது.*

**MCP** உங்களுக்கு ஏற்கனவே உள்ள கருவி சூழலைப் பயன்படுத்த வேண்டும், பல செயலிகளால் பகிர்ந்து கொள்ளக்கூடிய கருவிகளை உருவாக்க வேண்டும், மூன்றாம் தரப்பு சேவைகளை தரமான நெறிமுறைகளுடன் இணைக்க வேண்டும் அல்லது குறியீடு மாற்றாமே கருவி அமல்படுத்தல்களை மாற்ற வேண்டும் என்றால் மிக பொருத்தமாகும்.

**Agentic தொகுதி** `@Agent` குறிப்புகளுடன் கூடிய விளக்கவுரை முகவர் வரைவுகளுக்கு சிறந்தது, வேலைசாலை ஒருங்கிணைப்பு (தொடர், மடக்கு, ஒத்தடைகல்) தேவைப்படும்போது, கட்டளையிடும் குறியீட்டிற்குப் பதிலாக இடைமுக அடிப்படையிலான முகவர் வடிவமைப்பை முன்னுரிமை அளிக்கும் போது அல்லது பல முகவர்கள் outputKey மூலம் வெளியீடுகளை பகிரும்போது சிறந்தது.

**மேற்பார்வையாளர் முகவர் மாதிரி** வேலைசாலை முன்கூட்டியே கண்டறிய முடியாத போது மற்றும் LLM முடிவுகளைத் தேர்வு செய்ய வேண்டும் என்பதில் ஒளிர்கிறது, பல சிறப்பு முகவர்கள் மாற்றம் அடிப்படையிலான ஒருங்கிணைப்பை தேவைப்படும்போது, பேச்சு அமைப்புகளை கட்டியமைப்பதில் வேறுபட்ட திறன்களை வழிமொழிய வேண்டும் அல்லது மிகவும் தகுந்த, பொருத்தமான முகவர் நடத்தை வேண்டும் என்றால் சிறந்தது.

<img src="../../../translated_images/ta/custom-vs-mcp-tools.c4f9b6b1cb65d8a1.webp" alt="Custom Tools vs MCP Tools" width="800"/>

*எப்போது தனிப்பயன் @Tool முறைகள் பயன்படுத்துவது மற்றும் எப்போது MCP கருவிகள் பயன்படுத்துவது — செயலிக்கு தனிப்பட்ட லாஜிக்குக்கான தனிப்பயன் கருவிகள் முழு வகை பாதுகாப்புடன், செயலிகளுக்கு மாறாமல் செயற்படக்கூடிய தரநிலையான ஒருங்கிணைப்புகளுக்கு MCP கருவிகள்.*

- தனிப்பயன் கருவிகளுடன் அடிப்படையான AI முகவர்களை உருவாக்குதல் (தொகுதி 04)
- LangChain4j MCP மற்றும் Agentic தொகுதிகளுடன் தரநிலையிட்ட கருவிகளை ஒருங்கிணைத்தல் (தொகுதி 05)
