## మీరు నేర్చుకునేది ఏమిటి

మీరు సంభాషణ యంత్రాన్ని నిర్మించారు, ప్రాంప్ట్‌లను నేర్చుకున్నారు, డాక్యుమెంట్లలో రేపాండ్లను గ్రౌండెడ్ చేసుకున్నారు మరియు పరికరాలతో ఏజెంట్లను సృష్టించారు. కానీ ఆ పరికరాలు మీ నిర్దిష్ట అనువర్తనానికి ప్రత్యేకంగా కస్టమ్-తయారుచేసినవి. మీ AI కు ఎవరైనా సృష్టించి పంచుకునే ఒక ప్రమాణిత పరికరాల పర్యావరణాన్ని యాక్సెస్ చేయడం ఎలా అంటే? ఈ మాడ్యూల్‌లో, మీరు Model Context Protocol (MCP) మరియు LangChain4j యొక్క ఏజెంటిక్ మాడ్యూల్ తో ఎలా ఆ పని చేయాలో నేర్పుకుంటారు. మొదట ఒక సులభమైన MCP ఫైల్ రీడర్‌ని ప్రదర్శించి, తర్వాత సూపర్వైజర్ ఏజెంట్ నమూనాతో అడ్వాన్స్‌డ్ ఏజెంటిక్ వర్క్‌ఫ్లోలో ఎలా సులభంగా అనుసంధానమవుతుందో చూపిస్తాము.

## MCP అంటే ఏమిటి?

Model Context Protocol (MCP) అదే చేస్తుంది - AI అనువర్తనాలు బయట పరికరాలను కనుగొని వాడుకునే ప్రమాణిత మార్గాన్ని అందిస్తుంది. ప్రతి డేటా మూలం లేదా సేవ కోసం కస్టమ్ ఇంటిగ్రేషన్లను రాయడానికి బదులు, మీరు సమానమైన ఆకృతిలో తమ సామర్థ్యాలను ప్రదర్శించే MCP సర్వర్లకు కనెక్ట్ అవుతారు. మీ AI ఏజెంట్ ఆ పరికరాలను ఆటోమేటిక్‌ గా కనుగొని వాడుకోవచ్చు.

<img src="../../../translated_images/te/mcp-comparison.9129a881ecf10ff5.webp" alt="MCP పోలిక" width="800"/>

*MCP క్రితం: సంక్లిష్ట పాయింట్-టు-పాయింట్ ఇంటిగ్రేషన్లు. MCP తర్వాత: ఒక ప్రోటోకాల్, అనంత అవకాశాలు.*

MCP AI అభివృద్ధిలో మూలభూత సమస్యను పరిష్కరిస్తుంది: ప్రతి ఇంటిగ్రేషన్ కస్టమ్‌గా ఉంటుంది. GitHub యాక్సెస్ కావాలంటే? కస్టమ్ కోడ్. ఫైళ్ళు చదవాలంటే? కస్టమ్ కోడ్. డేటాబేస్ నుండి క్వెరీ చేయాలంటే? కస్టమ్ కోడ్. ఈ ఇంటిగ్రేషన్లు ఇతర AI అనువర్తనాలతో పనిచేయవు.

MCP దీనిని ప్రమాణీకరిస్తుంది. ఒక MCP సర్వర్ స్పష్టమైన వివరణలు మరియు స్కీమాలతో పరికరాలను ప్రదర్శిస్తుంది. ఏ MCP క్లయింట్ కనెక్ట్ అవ్వగలదు, అందుబాటులో ఉన్న పరికరాలను కనుగొని వాడుకోవచ్చు. ఒక్కసారి నిర్మించి, ఎక్కడైనా ఉపయోగించుకోండి.

<img src="../../../translated_images/te/mcp-architecture.b3156d787a4ceac9.webp" alt="MCP నిర్మాణం" width="800"/>

*Model Context Protocol నిర్మాణం - ప్రమాణీకృత పరికరాల కనుగొనడం మరియు అమలు*

## MCP ఎలా పని చేస్తుంది

<img src="../../../translated_images/te/mcp-protocol-detail.01204e056f45308b.webp" alt="MCP ప్రోటోకాల్ వివరాలు" width="800"/>

*MCP లో పరిస్థితి ఎలా — క్లయింట్లు పరికరాలను కనుగొంటాయి, JSON-RPC సందేశాలు మార్చుకుంటాయి, మరియు ట్రాన్స్‌పోర్ట్ లేయర్ మార్గంగా ఆపరేషన్లు అమలుపరుస్తాయి.*

**సర్వర్-క్లయింట్ నిర్మాణం**

MCP క్లయింట్-సర్వర్ నమూనా వాడుతుంది. సర్వర్లు పరికరాలను అందిస్తాయి - ఫైళ్ళు చదవడం, డేటాబేస్ అడగడం, APIలను కాల్ చేయడం. క్లయింట్లు (మీ AI అనువర్తనం) సర్వర్లకు కనెక్ట్ అవి వారి పరికరాలను వాడతాయి.

LangChain4j తో MCP వాడాలంటే, ఈ Maven డిపెండెన్సీని చేరండి:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```

**పరికరాల కనుగొనడం**

మీ క్లయింట్ MCP సర్వర్‌కు కనెక్ట్ అవగానే, అది "మీ వద్ద ఏ పరికరాలు ఉన్నాయి?" అని అడుగుతుది. సర్వర్ అందుబాటులో ఉన్న పరికరాల జాబితాను వివరణలు మరియు పారామీటర్ స్కీమాలతో సమాధానం ఇస్తుంది. మీ AI ఏజెంట్ యూజర్ అభ్యర్థనల ఆధారంగా ఏ పరికరాలను ఉపయోగించాలో నిర్ణయించొచ్చు.

<img src="../../../translated_images/te/tool-discovery.07760a8a301a7832.webp" alt="MCP పరికరాల కనుగొనడం" width="800"/>

*AI ప్రారంభంలో అందుబాటులో ఉన్న పరికరాలను కనుగొంటుంది — ఇప్పుడు ఏ సామర్థ్యాలు అందుబాటులో ఉన్నాయో తెలుసుకుని వాటిని ఉపయోగించడానికి నిర్ణయిస్తుంది.*

**ట్రాన్స్‌పోర్ట్ పద్ధతులు**

MCP వేరువేరు ట్రాన్స్‌పోర్ట్ పద్ధతులను మద్దతు ఇస్తుంది. ఈ మాడ్యూల్ లో స్థానిక ప్రక్రియలకు Stdio ట్రాన్స్‌పోర్ట్ ను చూపిస్తాము:

<img src="../../../translated_images/te/transport-mechanisms.2791ba7ee93cf020.webp" alt="ట్రాన్స్‌పోర్ట్ పద్ధతులు" width="800"/>

*MCP ట్రాన్స్‌పోర్ట్ పద్ధతులు: రిమోట్ సర్వర్ల కోసం HTTP, స్థానిక ప్రక్రియలకు Stdio*

**Stdio** - [StdioTransportDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java)

స్థానిక ప్రక్రియల కోసం. మీ అనువర్తనం సబ్ ప్రాసెస్‌గా సర్వర్‌ను స్పాన్ చేసి సాంప్రదాయ ఇన్‌పుట్/అవుట్పుట్ ద్వారా కమ్యూనికేట్ చేస్తుంది. ఫైల్‌సిస్టమ్ యాక్సెస్ లేదా కమాండ్-లైన్ పరికరాలకు ఉపయోగపడుతుంది.

```java
McpTransport stdioTransport = new StdioMcpTransport.Builder()
    .command(List.of(
        npmCmd, "exec",
        "@modelcontextprotocol/server-filesystem@2025.12.18",
        resourcesDir
    ))
    .logEvents(false)
    .build();
```

<img src="../../../translated_images/te/stdio-transport-flow.45eaff4af2d81db4.webp" alt="Stdio ట్రాన్స్‌పోర్ట్ ప్రవాహం" width="800"/>

*Stdio ట్రాన్స్‌పోర్ట్ కార్యంలో — మీ అనువర్తనం MCP సర్వర్‌ను చైల్డ్ ప్రాసెస్‌గా స్పాన్ చేసి stdin/stdout పైప్స్ ద్వారా కమ్యూనికేట్ చేస్తుంది.*

> **🤖 [GitHub Copilot](https://github.com/features/copilot) చాట్ తో ప్రయత్నించండి:** [`StdioTransportDemo.java`](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java) తెరువు మరియు అడుగు:
> - "Stdio ట్రాన్స్‌పోర్ట్ ఎలా పని చేస్తుంది మరియు నేను దానిని HTTP కంటే ఎప్పుడు వాడాలి?"
> - "LangChain4j MCP సర్వర్ ప్రాసెస్‌ల జీవిత చక్రాన్ని ఎలా నిర్వహిస్తుంది?"
> - "AI కి ఫైల్ సిస్టమ్ యాక్సెస్ ఇచ్చేటప్పుడు భద్రతా అంశాలు ఏమిటి?"

## ఏజెంటిక్ మాడ్యూల్

MCP ప్రమాణీకృత పరికరాలను అందించినా, LangChain4j యొక్క **ఏజెంటిక్ మాడ్యూల్** ఆ పరికరాలను ఏర్పాటు చేసే ఏజెంట్లను నిర్మించడానికి ప్రకటనాత్మక మార్గాన్ని అందిస్తుంది. `@Agent` అనోటేషన్ మరియు `AgenticServices` ద్వారా మీరు నియమక కోడ్ కంటే ఇంటర్‌ఫేస్‌ల ద్వారానే ఏజెంట్ పనితీరును నిర్వచించవచ్చు.

ఈ మాడ్యూల్‌లో, మీరు **సూపర్వైజర్ ఏజెంట్** నమూనాను పరిశీలిస్తారు — ఇది ఒక అడ్వాన్స్‌డ్ ఏజెంటిక్ AI విధానం, అక్కడ "సూపర్వైజర్" ఏజెంట్ యూజర్ అభ్యర్థనల ఆధారంగా సబ్-ఏజెంట్లను కాల్ చేసే నిర్ణయాన్ని డైనమిక్‌గా తీసుకుంటుంది. రెండు భావనలను కలిపి, మన సబ్-ఏజెంట్లలో ఒకరికి MCP ఆధారిత ఫైల్ యాక్సెస్ సామర్థ్యాలను ఇస్తాం.

ఏజెంటిక్ మాడ్యూల్ వాడాలంటే, ఈ Maven డిపెండెన్సీని చేరండి:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```

> **⚠️ ప్రయోగాత్మక:** `langchain4j-agentic` మాడ్యూల్ **ప్రయోగాత్మకము** మరియు మార్పులకు లోబులో ఉంటుంది. AI అసిస్టెంట్లను నిర్మించడానికి స్థిరమైన మార్గం ఇంకా `langchain4j-core` కస్టమ్ పరికరాలతో (మాడ్యూల్ 04).

## ఆదర్ణాల నడపడం

### అవసరాలు

- Java 21+, Maven 3.9+
- Node.js 16+ మరియు npm (MCP సర్వర్‌ల కోసం)
- `.env` ఫైల్‌లో వాతావరణ వేరియబుల్‌లు కాన్ఫిగర్ చేయండి (రూట్ డైరెక్టరీ నుండి):
  - `AZURE_OPENAI_ENDPOINT`, `AZURE_OPENAI_API_KEY`, `AZURE_OPENAI_DEPLOYMENT` (మాడ్యూల్ 01-04 తో సారూప్యం)

> **గమనిక:** మీరు ఇంకా వాతావరణ వేరియబుల్స్ సెటప్ చేయకపోతే, [మాడ్యూల్ 00 - త్వరిత ప్రారంభం](../00-quick-start/README.md) చూడండి లేదా `.env.example`ని రూట్‌లో `.env` గా కాపీ చేసి మీ విలువలు నింపండి.

## త్వరిత ప్రారంభం

### సూపర్వైజర్ ఏజెంట్

**సూపర్వైజర్ ఏజెంట్ నమూనా** ఒక **అనువర్తనయోగ్య** ఏజెంటిక్ AI రూపం. సూపర్వైజర్ LLM ఉపయోగించి యూజర్ అభ్యర్థన ఆధారంగా ఏ ఏజెంట్లను కాల్ చేయాలో స్వతంత్రంగా నిర్ణయిస్తాడు. తర్వాతి ఉదాహరణలో, MCP ఆధారిత ఫైల్ యాక్సెస్ మరియు LLM ఏజెంట్ కలిపి ఫైల్ చదవడం → రిపోర్ట్ తయారీ వర్క్‌ఫ్లోను సృష్టిస్తారు.

డెమోలో, `FileAgent` MCP ఫైల్‌సిస్టమ్ పరికరాలతో ఫైల్ చదవడంతో, `ReportAgent` నిర్మిత రిపోర్ట్ (ఒక వాక్యం ఎగ్జిక్యూటివ్ సమ్మరీ, 3 కీలక అంశాలు, సిఫారసులు) తయారు చేస్తుంది. సూపర్వైజర్ ఈ ఫ్లోని ఆటోమేటిక్‌గా సంయోజిస్తుంది:

<img src="../../../translated_images/te/supervisor-agent-pattern.06275a41ae006ac8.webp" alt="సూపర్వైజర్ ఏజెంట్ నమూనా" width="800"/>

*సూపర్వైజర్ తన LLM ఉపయోగించి ఏ ఏజెంట్లను ఎప్పుడు కాల్ చేయాలో నిర్ణయిస్తుంది — హార్డ్‌కోడ్ రౌటింగ్ అవసరం లేదు.*

ఫైల్ నుండి రిపోర్ట్ పైపు‌లైన్‌కు కన్స్రేట్ వర్క్‌ఫ్లో ఇలా ఉంటుంది:

<img src="../../../translated_images/te/file-report-workflow.649bb7a896800de9.webp" alt="ఫైల్ నుండి రిపోర్ట్ వర్క్‌ఫ్లో" width="800"/>

*FileAgent MCP పరికరాలతో ఫైల్ చదివి, ReportAgent అప్రాసెస్ చేసిన కంటెంట్‌ను నిర్మిత రిపోర్టుగా మార్చుతుంది.*

ప్రతి ఏజెంట్ తన ఫలితాన్ని **ఏజెంటిక్ స్కోప్** (పంచుకున్న మెమరీ)లో నిల్వ చేస్తుంది, తద్వారా తర్వాతి ఏజెంట్లు గత ఫలితాలను యాక్సెస్ చేసుకోవచ్చు. ఇది MCP పరికరాలు ఏజెంటిక్ వర్క్‌ఫ్లోలలో సజావుగా ఎలా అనుసంధానమవుతాయో చూపిస్తుంది — సూపర్వైజర్ ఫైల్స్ ఎలా చదవబడుతాయో తెలుసుకోవాల్సిన అవసరం లేదు, కేవలం `FileAgent` చెయ్యగలదనే విషయం.

#### సూపర్వైజర్ ఎలా పని చేస్తుంది

```java
// స్టెప్ 1: ఫైల్ ఏజెంట్ MCP టూల్స్ ఉపయోగించి ఫైళ్లను చదవ Tro
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // ఫైల్ ఆపరేషన్ల కోసం MCP టూల్స్ ఉన్నాయి
        .build();

// స్టెప్ 2: రిపోర్ట్ ఏజెంట్ నిర్మిత రిపోర్టులను ఉత్పత్తి చేస్తుంది
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// సూపర్వైజర్ ఫైల్ → రిపోర్ట్ వర్క్‌ఫ్లోను సమన్వయం చేస్తుంది
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // తుది రిపోర్టును తిరిగి ఇవ్వండి
        .build();

// అభ్యర్థన ఆధారంగా ఏ ఏజెంట్లను పిలవాలో సూపర్వైజర్ నిర్ణయిస్తుంది
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```

#### ప్రతిస్పందన వ్యూహాలు

మీరు `SupervisorAgent`ని కాన్ఫిగర్ చేస్తే, సబ్-ఏజెంట్లు వారి పనులను పూర్తిచేసిన తర్వాత యూజర్‌కు తుది జవాబు ఎలా రూపొందించాలో మీరు నిర్దేశిస్తారు.

<img src="../../../translated_images/te/response-strategies.3d0cea19d096bdf9.webp" alt="ప్రతిస్పందన వ్యూహాలు" width="800"/>

*సూపర్వైజర్ ఎలా తన తుది ప్రతిస్పందన తయారు చేస్తుందో మూడు వ్యూహాలు — చివరి ఏజెంట్ అవుట్పుట్, సారాంశం, లేదా ఉత్తమ స్కోరు పొందిన ఎంపిక.*

అందుబాటులో ఉన్న వ్యూహాలు:

| వ్యూహం | వివరణ |
|----------|-------------|
| **LAST** | సూపర్వైజర్ కాల్ చేసిన చివరి సబ్-ఏజెంట్ లేదా పరికరం ఉత్పత్తిని జవాబుగా ఇస్తుంది. ఇది ముఖ్యంగా వర్క్‌ఫ్లోలో చివరి ఏజెంట్ పూర్తి, తుది జవాబు ఇవ్వడానికి రూపొంది ఉంటే ఉపయోగకరం (ఉదా: రీసెర్చ్ పైపు‌లైన్‌లో "సమ్మరీ ఏజెంట్"). |
| **SUMMARY** | సూపర్వైజర్ తన స్వంత భాషా మోడల్ (LLM) ఉపయోగించి మొత్తం ఇంటరాక్షన్ మరియు సబ్-ఏజెంట్ అవుట్పుట్ల సారాంశాన్ని సృష్టించి, దానిని తుది ప్రతిస్పందనగా ఇస్తుంది. ఇది యూజర్‌కు శుభ్రమైన, ఒక్కటి కూడించిన జవాబును అందిస్తుంది. |
| **SCORED** | వ్యవస్థ అంతర్గత LLM ఉపయోగించి చివరి (LAST) ప్రతిస్పందన మరియు ఇంటరాక్షన్ సారాంశాన్ని మొదటి యూజర్ అభ్యర్థనతో స్కోరు చేస్తుంది, మరింత అధిక స్కోరు గల అవుట్పుట్‌ను ఇవ్వుతుంది. |

#### ఫలితాన్ని అర్థం చేసుకోవడం

డెమో నడిపితే, సూపర్వైజర్ బహు ఏజెంట్లను ఎలా సంయోజిస్తుందో స్ట్రక్చర్డ్ వాక్‌త్రూ చూడవచ్చు. ప్రతి విభాగం అర్థం:

```
======================================================================
  FILE → REPORT WORKFLOW DEMO
======================================================================

This demo shows a clear 2-step workflow: read a file, then generate a report.
The Supervisor orchestrates the agents automatically based on the request.
```

**శీర్షిక** వర్క్‌ఫ్లో భావనను పరిచయపరుస్తుంది: ఫైల్ చదవడం నుండి రిపోర్ట్ తయారీకి ధ్యానమైన పైపు‌లైన్.

```
--- WORKFLOW ---------------------------------------------------------
  ┌─────────────┐      ┌──────────────┐
  │  FileAgent  │ ───▶ │ ReportAgent  │
  │ (MCP tools) │      │  (pure LLM)  │
  └─────────────┘      └──────────────┘
   outputKey:           outputKey:
   'fileContent'        'report'

--- AVAILABLE AGENTS -------------------------------------------------
  [FILE]   FileAgent   - Reads files via MCP → stores in 'fileContent'
  [REPORT] ReportAgent - Generates structured report → stores in 'report'
```

**వర్క్‌ఫ్లో చిత్రం** ఏజెంట్ల మధ్య డేటా ప్రవాహాన్ని చూపిస్తుంది. ప్రతి ఏజెంట్ ప్రత్యేక పాత్ర కల్గింది:
- **FileAgent** MCP పరికరాలతో ఫైళ్ళు చదివి గ.raw‌ను `fileContent` లో నిల్వ చేస్తుంది
- **ReportAgent** ఆ కంటెంట్ వాడి `report`లో నిర్మిత రిపోర్ట్ చేస్తుంది

```
--- USER REQUEST -----------------------------------------------------
  "Read the file at .../file.txt and generate a report on its contents"
```

**యూజర్ అభ్యర్థన** పనిని చూపిస్తుంది. సూపర్వైజర్ దీన్ని విశ్లేషించి FileAgent → ReportAgent కాల్ చేయాలని నిర్ణయిస్తుంది.

```
--- SUPERVISOR ORCHESTRATION -----------------------------------------
  The Supervisor decides which agents to invoke and passes data between them...

  +-- STEP 1: Supervisor chose -> FileAgent (reading file via MCP)
  |
  |   Input: .../file.txt
  |
  |   Result: LangChain4j is an open-source, provider-agnostic Java framework for building LLM...
  +-- [OK] FileAgent (reading file via MCP) completed

  +-- STEP 2: Supervisor chose -> ReportAgent (generating structured report)
  |
  |   Input: LangChain4j is an open-source, provider-agnostic Java framew...
  |
  |   Result: Executive Summary...
  +-- [OK] ReportAgent (generating structured report) completed
```

**సూపర్వైజర్ నిర్వహణ** 2-దశల ప్రవాహాన్ని చూపిస్తుంది:
1. **FileAgent** MCP ద్వారా ఫైల్ చదివి కంటెంట్ నిల్వ చేస్తుంది
2. **ReportAgent** ఆ కంటెంట్ అందుకుని నిర్మిత రిపోర్ట్ ఉత్పత్తి చేస్తుంది

సూపర్వైజర్ యూజర్ అభ్యర్థన ఆధారంగా స్వతంత్రంగా ఈ నిర్ణయాలు తీసుకున్నాడు.

```
--- FINAL RESPONSE ---------------------------------------------------
Executive Summary
...

Key Points
...

Recommendations
...

--- AGENTIC SCOPE (Data Flow) ----------------------------------------
  Each agent stores its output for downstream agents to consume:
  * fileContent: LangChain4j is an open-source, provider-agnostic Java framework...
  * report: Executive Summary...
```

#### ఏజెంటిక్ మాడ్యూల్ లక్షణాల వివరణ

ఈ ఉదాహరణ ఏజెంటిక్ మాడ్యూల్ పరిణత లక్షణాలను చూపిస్తుంది. ఏజెంటిక్ స్కోప్ మరియు ఏజెంట్ లిస్నర్స్‌ను దగ్గరగా చూద్దాం.

** ఏజెంటిక్ స్కోప్** ఏజెంట్లు `@Agent(outputKey="...")` ఉపయోగించి నిల్వ చేసిన ఫలితాలు ఉన్న పంచుకున్న మెమరీ.

ఇది అనుమతిస్తుంది:
- తరువాత ఏజెంట్లు కూర్మ ముందటి ఏజెంట్ల అవుట్పుట్లను యాక్సెస్ చేసుకోవచ్చు
- సూపర్వైజర్ తుది ప్రతిస్పందనను కలుసుకోగలడు
- మీరు ప్రతి ఏజెంట్ ఏమి ఉత్పత్తి చేసిందిలాంటిది పరిశీలించవచ్చు

<img src="../../../translated_images/te/agentic-scope.95ef488b6c1d02ef.webp" alt="ఏజెంటిక్ స్కోప్ పంచుకున్న మెమరీ" width="800"/>

*ఏజెంటిక్ స్కోప్ పంచుకున్న మెమరీగా పనిచేస్తుంది — FileAgent `fileContent` ను వ్రాస్తుంది, ReportAgent దాన్ని చదివి `report` వ్రాస్తుంది, మరియు మీ కోడ్ తుది ఫలితాన్ని చదువుతుంది.*

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // FileAgent నుండి ముడి ఫైల్ డేటా
String report = scope.readState("report");            // ReportAgent నుండి నిర్మిత నివేదిక
```

**ఏజెంట్ లిస్నర్స్** ఏజెంట్ అమలు పర్యవేక్షణ మరియు డిబగ్గింగ్‌కు సహాయపడతాయి. డెమోలో మీరు చూసే స్టెప్-బై-స్టెప్ అవుట్పుట్ ప్రతి ఏజెంట్ కాల్‌కు ఏజెంట్ లిస్నర్ ద్వారా వచ్చినది:
- **beforeAgentInvocation** - సూపర్వైజర్ ఏజెంట్‌ను ఎంచుకున్నపుడు పిలవబడుతుంది, మీరు ఏ ఏజెంట్ ఎంచుకోబడింది మరియు ఎందుకు అనేది చూడగలుగుతారు
- **afterAgentInvocation** - ఏజెంట్ పూర్తి చేసినపుడు పిలవబడుతుంది, దాని ఫలితాన్ని చూపిస్తుంది
- **inheritedBySubagents** - నిజమైతే, ఈ లిసనర్ హయ్యరార్కీలోని అన్ని ఏజెంట్‌లను పర్యవేక్షిస్తుంది

<img src="../../../translated_images/te/agent-listeners.784bfc403c80ea13.webp" alt="Agent Listeners Lifecycle" width="800"/>

*ఏజంట్ లిసనర్లు ఎగ్జిక్యూషన్ జీవ చక్రంలో హుక్ అవుతాయి — ఏజెంట్లు ప్రారంభమయ్యేటప్పుడు, పూర్తి అవుతుండగా లేదా దోషాలను ఎదుర్కొంటున్నప్పుడు పర్యవేక్షణ చేస్తాయి.*

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // అన్ని ఉప-ఏజెంట్లకు వ్యాప్తి చేయండి
    }
};
```

సూపర్వైజర్ ప్యాటర్న్ కంటే మించి, `langchain4j-agentic` మాడ్యూల్ అనేక శక్తివంతమైన వర్క్‌ఫ్లో ప్యాటర్న్లు మరియు ఫీచర్స్ అందిస్తుంది:

<img src="../../../translated_images/te/workflow-patterns.82b2cc5b0c5edb22.webp" alt="Agent Workflow Patterns" width="800"/>

*ఏజెంట్లను నిర్వహించాలని ఐదు వర్క్‌ఫ్లో ప్యాటర్న్లు — సరళమైన వరసగా పైప్‌లైన్ల నుండి హ్యూమన్-ఇన్-ది-లూప్ మంజూరు వర్క్‌ఫ్లోల వరకు.*

| ప్యాటర్న్ | వివరాలు | ఉపయోగం |
|---------|-------------|----------|
| **Sequential** | ఏజెంట్లను క్రమబద్దంగా అమలు చేయండి, అవుట్‌పుట్ తదుపరి దిశగా ప్రవహిస్తుంది | పൈప్‌లైన్లు: పరిశోధనా → విశ్లేషణ → నివేదిక |
| **Parallel** | ఏజెంట్లను సమకాలీనంగా నడపండి | స్వతంత్ర కార్యాలు: వాతావరణం + వార్తలు + స్టాక్స్ |
| **Loop** | షరతు తీరేవరకు పునరావృతి చేయండి | నాణ్యత స్కోరింగ్: స్కోరు 0.8కి పైగా పొందేవరకు మెరుగు పరచడం |
| **Conditional** | షరతుల ఆధారంగా మార్గ నిర్దేశం చేయండి | వర్గీకరణ → నిపుణ ఏజెంట్‌ కు మార్గం |
| **Human-in-the-Loop** | మానవ తనిఖీలు జోడించండి | అనుమతి వర్క్‌ఫ్లోలు, కంటెంట్ సమీక్ష |

## కీలక భావనలు

మీరు ఇప్పుడు MCP మరియు ఏజెంటిక్ మాడ్యూల్ ఆచరణలో చూసిన తర్వాత, ప్రతీరోక పద్ధతిని ఎప్పుడు ఉపయోగించాలో సారాంశం చేద్దాం.

<img src="../../../translated_images/te/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="MCP Ecosystem" width="800"/>

*MCP ఒక విశ్వవ్యాప్త ప్రమాణ వాతావరణాన్ని సృష్టిస్తుంది — ఏ MCP-అనుకూల సర్వర్ ఏ MCP-అనుకూల క్లయింట్తో పని చేస్తుంది, అనువర్తనాల మధ్య సాధనాలను పంచుకునే అవకాశాన్ని ఇస్తుంది.*

**MCP** ఇప్పుడు ఉన్న సాధన వాతావరణాలను ఉపయోగించాలనుకుంటే, పలు అనువర్తనాలకు సాధనాలు పంచుకునేటట్లు నిర్మించాలి, మూడవ పక్ష సేవలను ప్రమాణాలకు అనుగుణంగా ఇంటిగ్రేట్ చేయాలి, లేదా కోడ్ మార్చకుండా సాధన అమలును మార్చాలని ఉన్నప్పుడు ఉత్తమం.

**ఏజెంటిక్ మాడ్యూల్** `@Agent` అనోటేషన్లతో ప్రకటించే ఏజెంట్ నిర్వచనాలు కావాలనుకునే, వర్క్‌ఫ్లో కార్యనిర్వహణ (వరుస, లూప్, సమాంతర) అవసరమైన, కమాండ్ కోడ్ కంటే ఇంటర్‌ఫేస్ ఆధారిత ఏజెంట్ డిజైన్ ఇష్టపడితే లేదా ఒకే అవుట్‌పుట్ కీతో పలు ఏజెంట్లను కలిపితే ఉత్తమం.

**సూపర్వైజర్ ఏజెంట్ ప్యాటర్న్** వర్క్‌ఫ్లో ముందస్తుగా అంచనా వేయలేని, LLM నిర్ణయం తీసుకోవాలని ఉన్నప్పుడు, పలు ప్రత్యేక ఏజెంట్లు డైనమిక్ ఆర్కెస్ట్రేషన్ కావాలనుకునేపుడు, విభిన్న సామర్ధ్యాలకు మార్గనిర్దేశం చేసే సంభాషణ వ్యవస్థల్ని నిర్మించేపుడు లేదా అత్యంత సౌలభ్యవంతమైన, అనుకూల ఏజెంట్ ప్రవర్తన కావాలనుకునేపుడు ఉత్తమం.

<img src="../../../translated_images/te/custom-vs-mcp-tools.c4f9b6b1cb65d8a1.webp" alt="Custom Tools vs MCP Tools" width="800"/>

*ఎప్పుడు కస్టమ్ @Tool పద్ధతులు ఉపయోగించాలి vs MCP సాధనాలు — అనువర్తనానికి ప్రత్యేక లాజిక్ కోసం పూర్తిగా టైప్ సేఫ్టీ కలిగిన కస్టమ్ సాధనాలు, అనువర్తనాల మధ్య పనిచేసే ప్రమాణీకృత ఇంటిగ్రేషన్ల కోసం MCP సాధనాలు.*
