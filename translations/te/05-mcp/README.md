## మీరు నేర్చుకునే విషయాలు

మీరు సంభాషణాత్మక AI నిర్మించారు, ప్రాంప్ట్‌లను మాస్టర్ చేసారు, ప్రతిస్పందనలను డాక్యుమెంట్లలో స్థిరపరిచారు, మరియు టూల్స్‌తో ఏజెంట్లను సృష్టించారు. కానీ ఆ టూల్స్ మీ ప్రత్యేక అనువర్తనానికి అనుగుణంగా సన్నాహకంగా రూపొందించబడినవి. మీరు మీ AIకి ఎవ్వరూ సృష్టించవచ్చు మరియు పంచగల ఒక ప్రమాణీకృత టూల్ వాతావరణాన్ని అందించగలిగితే ఏ ఉంటుందో? ఈ మాడ్యూల్‌లో, మీరు Model Context Protocol (MCP) మరియు LangChain4j యొక్క ఏజెంటిక్ మాడ్యూల్ ద్వారా అలా చేయడం ఎలా అనేది నేర్చుకుంటారు. మొదట ఒక సాదారణ MCP ఫైల్ రీడర్‌ను ప్రదర్శించి, అప్పుడు దీన్ని సులభంగా అధునాతన ఏజెంటిక్ వర్క్‌ఫ్లోలలో ఎలాగూ ఇన్‌టీగ్రేట్ చేయడం చూపిస్తాము, సూపర్‌వైజర్ ఏజెంట్ ప్యాటర్న్ ఉపయోగించి.

## MCP అంటే ఏమిటి?

Model Context Protocol (MCP) అదే అందిస్తుంది - AI అనువర్తనాలు బాహ్య టూల్స్‌ను కనుగొని ఉపయోగించుకోవడానికి ఒక ప్రమాణీకృత మార్గం. ప్రతి డేటా వనరు లేదా సేవ కోసం కస్టమ్ ఇంటిగ్రేషన్లను రాయకుండా, మీరు వారి సామర్థ్యాలను సాదా ఫార్మాట్‌లో ప్రదర్శించే MCP సర్వర్‌లకు కనెక్ట్ అవుతారు. మీ AI ఏజెంట్ ఆ టూల్స్‌ను స్వయంచాలకంగా కనుగొని ఉపయోగించగలదు.

క్రింది రేఖాచిత్రం తేడా చూపుతుంది — MCP లేకుండా, ప్రతి ఇంటిగ్రేషన్ కస్టమ్ పాయింట్-టు-పాయింట్ వార్డింగ్ అవసరం; MCPతో, ఒకే ప్రోటోకాల్ మీ అనువర్తనాన్ని ఏ టూల్‌తో అయినా కనెక్ట్ చేస్తుంది:

<img src="../../../translated_images/te/mcp-comparison.9129a881ecf10ff5.webp" alt="MCP Comparison" width="800"/>

*MCPకు ముందు: సంక్లిష్ట పాయింట్-టు-పాయింట్ ఇంటిగ్రేషన్లు. MCP తర్వాత: ఒక ప్రోటోకాల్, అంతిరాకాయ అవకాశాలు.*

MCP AI అభివృద్ధిలో ఒక ప్రాథమిక సమస్యను పరిష్కరిస్తుంది: ప్రతి ఇంటిగ్రేషన్ కస్టమ్. GitHub ను యాక్సెస్ చేయాలనుకుంటున్నారా? కస్టమ్ కోడ్. ఫైళ్ళను చదవాలనుకుంటున్నారా? కస్టమ్ కోడ్. డేటాబేస్ ప్రశ్నించాలనుకుంటున్నారా? కస్టమ్ కోడ్. ఈ ఇంటిగ్రేషన్లు మరెవరు ఆ AI అనువర్తనాలతో కూడా పని చేయవు.

MCP దీన్ని ప్రమాణీకృతం చేస్తుంది. MCP సర్వర్ స్పష్టమైన వివరణలు మరియు స్కీమాలతో టూల్స్‌ను అందిస్తుంది. ఏ MCP క్లయింట్ కనెక్ట్ అయ్యి, అందుబాటులో ఉన్న టూల్స్‌ను కనుగొని వాటిని ఉపయోగించగలదు. ఒక్కసారి నిర్మించి, ఎక్కడైనా ఉపయోగించండి.

క్రింది చిత్తరువు ఈ శిల్పవిధానాన్ని చూపిస్తుంది — ఒకే MCP క్లయింట్ (మీ AI అనువర్తనం) అనేక MCP సర్వర్‌లకు కనెక్ట్ అవుతుంది, ప్రతి సర్వర్ తన సెట్ టూల్స్‌ను ప్రమాణ ప్రోటోకాల్ ద్వారా అందిస్తుంది:

<img src="../../../translated_images/te/mcp-architecture.b3156d787a4ceac9.webp" alt="MCP Architecture" width="800"/>

*Model Context Protocol శిల్పవిధానం - ప్రమాణీకృత టూల్ కనుగొన్నీ మరియు అమలు*

## MCP ఎలా పనిచేస్తుంది

కింద ముంగిసినట్లు, MCP పొరల శిల్పంలో పనిచేస్తుంది. మీ Java అనువర్తనం (MCP క్లయింట్) అందుబాటులో ఉన్న టూల్‌లను కనుగొంటుంది, ట్రాన్స్పోర్ట్ పొర (Stdio లేదా HTTP) ద్వారా JSON-RPC అభ్యర్థనలు పంపుతుంది మరియు MCP సర్వర్ ఆపరేషన్లను అమలు చేసి ఫలితాలు తిరిగి ఇస్తుంది. క్రింది చిత్రం ఈ ప్రోటోకాల్ యొక్క ప్రతి పొరను విడగొడుతుంది:

<img src="../../../translated_images/te/mcp-protocol-detail.01204e056f45308b.webp" alt="MCP Protocol Detail" width="800"/>

*MCP ఎలా పనిచేస్తుందో - క్లయింట్లు టూల్స్‌ను కనుగొంటాయి, JSON-RPC సందేశాలు ఆదానం/ప్రదానం చేస్తాయి, మరియు ట్రాన్స్పోర్ట్ పొర ద్వారా ఆపరేషన్లు అమలు చేస్తాయి.*

**సర్వర్-క్లయింట్ శిల్పవిధానం**

MCP క్లయింట్-సర్వర్ మోడల్ ఉపయోగిస్తుంది. సర్వర్‌లు టూల్స్ అందిస్తాయి - ఫైళ్ళను చదవడం, డేటాబేస్‌లను ప్రశ్నించడం, APIలను పిలవడం. క్లయింట్లు (మీ AI అనువర్తనం) సర్వర్లకు కనెక్ట్ అవ్వి వారి టూల్స్ ఉపయోగిస్తాయి.

LangChain4jతో MCP ఉపయోగించేందుకు, ఈ Maven ఆధారాన్ని జోడించండి:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```

**టూల్ కనుగొనుట**

మీ క్లయింట్ MCP సర్వర్‌కి కనెక్ట్ అయినప్పుడు, అది "మీ దగ్గర ఏ టూల్స్ ఉన్నాయి?" అని అడుగుతుంది. సర్వర్ అందుబాటులో ఉన్న టూల్స్ జాబితా తో స్పందిస్తుంది, ప్రతి టూల్ వివరణలు మరియు పారామీటర్ స్కీమాలతో ఉంటుంది. మీ AI ఏజెంట్ యూజర్ అభ్యర్థనల ఆధారంగా ఏ టూల్స్ ఉపయోగించాలో నిర్ణయించవచ్చు. క్రింది చిత్రం ఈ చేతి దస్తకం చూపిస్తుంది — క్లయింట్ `tools/list` అభ్యర్థన పంపుతుంది మరియు సర్వర్ తన అందుబాటులో ఉన్న టూల్స్ వివరణలు మరియు స్కీమాలతో తిరిగి పంపుతుంది:

<img src="../../../translated_images/te/tool-discovery.07760a8a301a7832.webp" alt="MCP Tool Discovery" width="800"/>

*AI ప్రారంభంలో అందుబాటులో ఉన్న టూల్స్‌ను కనుగొంటుంది — ఇప్పుడు ఏ సామర్థ్యాలు అందుబాటులో ఉన్నాయో తెలుసుకుని, యే వాటిని ఉపయోగించాలో నిర్ణయించవచ్చు.*

**ట్రాన్స్పోర్ట్ మెకానిజంలు**

MCP వివిధ ట్రాన్స్పోర్ట్ మెకానిజంలను మద్దతు ఇస్తుంది. రెండు ఆప్షన్లు Stdio (స్థానిక ఉపప్రక్రియల కమ్యూనికేషన్ కోసం) మరియు Streamable HTTP (రిమోట్ సర్వర్‌ల కోసం). ఈ మాడ్యూల్ Stdio ట్రాన్స్పోర్ట్ ను ప్రదర్శిస్తుంది:

<img src="../../../translated_images/te/transport-mechanisms.2791ba7ee93cf020.webp" alt="Transport Mechanisms" width="800"/>

*MCP ట్రాన్స్పోర్ట్ మెకానిజంలు: HTTP రిమోట్ సర్వర్‌లకు, Stdio స్థానిక ప్రాసెస్‌లకు*

**Stdio** - [StdioTransportDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java)

స్థానిక ప్రక్రియలకు. మీ అనువర్తనం ఒక సబ్ప్రాసెస్ సర్వర్‌ను ప్రారంభించి, స్టాండర్డ్ ఇన్‌పుట్/ఔట్‌పుట్ ద్వారా కమ్యూనికేట్ చేస్తుంది. ఫైల్ సిస్టమ్ యాక్సెస్ లేదా కమాండ్ లైన్ టూల్స్ కోసం ఉపయోగపడుతుంది.

```java
McpTransport stdioTransport = new StdioMcpTransport.Builder()
    .command(List.of(
        npmCmd, "exec",
        "@modelcontextprotocol/server-filesystem@2025.12.18",
        resourcesDir
    ))
    .logEvents(false)
    .build();
```

`@modelcontextprotocol/server-filesystem` సర్వర్ ఈ క్రింది టూల్స్‌ను అందిస్తుంది, మీ సూచించిన డైరెక్టరీల వరకు సులభమైన సేఫ్ బాక్స్‌లో:

| టూల్ | వివరణ |
|------|-------------|
| `read_file` | ఒకే ఫైల్ యొక్క కంటెంట్ను చదవండి |
| `read_multiple_files` | ఒకేసారి అనేక ఫైల్స్ చదవండి |
| `write_file` | ఫైల్ సృష్టించండి లేదా తిరిగి రాయండి |
| `edit_file` | లక్ష్యంగా తీసుకుని ఫైండ్స్-అండ్-రిప్లేస్ సవరించండి |
| `list_directory` | ఒక దారిలో ఫైల్స్ మరియు డైరెక్టరీస్‌ను జాబితా చేయండి |
| `search_files` | ఆ విధంగా సరిపడే ఫైళ్లను రికర్సివ్‌గా వెతకండి |
| `get_file_info` | ఫైల్ మెటాడేటా పొందండి (పరిమాణం, టైమ్‌స్టాంప్స్, అనుమతులు) |
| `create_directory` | డైరెక్టరీ సృష్టించండి (తల్లి డైరెక్టరీలు సహా) |
| `move_file` | ఫైల్ లేదా డైరెక్టరీని తరలించండి లేదా పేరు మార్చండి |

క్రింది చిత్రం Stdio ట్రాన్స్పోర్ట్ రన్‌టైమ్‌లో ఎలా పనిచేస్తుందో చూపిస్తుంది — మీ Java అనువర్తనం MCP సర్వర్‌ను చైల్డ్ ప్రాసెస్‌గా ప్రారంభించి, stdin/stdout పైప్స్ ద్వారానే కమ్యూనికేట్ చేస్తుంది, నెట్‌వర్క్ లేదా HTTP involved కాదు:

<img src="../../../translated_images/te/stdio-transport-flow.45eaff4af2d81db4.webp" alt="Stdio Transport Flow" width="800"/>

*Stdio ట్రాన్స్పోర్ట్ యాక్షన్ లో — మీ అనువర్తనం MCP సర్వర్‌ను చైల్డ్ ప్రాసెస్‌గా ప్రారంభించి, stdin/stdout పైప్స్ ద్వారా కమ్యూనికేట్ చేస్తుందని.*

> - "Stdio ట్రాన్స్పోర్ట్ ఎలా పనిచేస్తుంది మరియు నేను HTTPతో పోల్చితే ఎప్పుడు దీన్ని ఉపయోగించాలి?"
> - "LangChain4j ఎలా MCP సర్వర్ ప్రాసెస్‌ల జీవిత చక్రాన్ని నిర్వహిస్తుంది?"
> - "AIకి ఫైల్ సిస్టమ్ యాక్సెస్ ఇచ్చినప్పుడు భద్రతా ప్రభావాలు ఏమిటి?"

## ఏజెంటిక్ మాడ్యూల్

MCP ప్రమాణీకృత టూల్స్ అందించినప్పటికీ, LangChain4j యొక్క **ఏజెంటిక్ మాడ్యూల్** ఆ టూల్స్‌ను నిర్వచించడానికి ప్రకటనా పద్ధతిని అందిస్తుంది. `@Agent` అన్‌టేషన్ మరియు `AgenticServices` ద్వారా మీరు ఏజెంటు ప్రవర్తనను ఇంటర్‌ఫేసుల ద్వారానే నిర్వచించవచ్చు, ఆదేశాత్మక కోడ్ కాకుండా.

ఈ మాడ్యూల్‌లో, మీరు **సూపర్‌వైజర్ ఏజెంట్** ప్యాటర్న్‌ను అన్వేషిస్తారు — ఇది ఆధునిక ఏజెంటిక్ AI దృక్పథం, ఇక్కడ “సూపర్‌వైజర్” ఏజెంట్ యూజర్ అభ్యర్థనలు ఆధారంగా సబ్ఐజెంట్లను పిలవడానికి డైనమిక్‌గా నిర్ణయిస్తుంది. మనం ఇరుక్కోవడం ద్వారా ఒక సబ్-ఏజెంట్‌కు MCP ఆధారిత ఫైల్ యాక్సెస్ సామర్థ్యాలను ఇస్తాము.

ఏజెంటిక్ మాడ్యూల్ ఉపయోగించేందుకు, ఈ Maven ఆధారాన్ని జోడించండి:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```

> **గమనిక:** `langchain4j-agentic` మాడ్యూల్ ప్రత్యేక వెర్షన్ ప్రాపర్టీ (`langchain4j.mcp.version`) ఉపయోగిస్తుంది, ఎందుకంటే ఇది కోర్ LangChain4j లైబ్రరీల కంటే వేరుగా విడుదల అవుతుంది.

> **⚠️ ప్రయోగాత్మక:** `langchain4j-agentic` మాడ్యూల్ **ప్రయోగాత్మక**ది మరియు మార్పులకు లోంభోగిస్తుంది. AI అసిస్టెంట్‌లను నిర్మించడానికి స్థిరమైన మార్గం ఇప్పటికీ `langchain4j-core` కస్టమ్ టూల్స్‌తో ఉంటుంది (మాడ్యూల్ 04).

### సూపర్‌వైజర్ ఏజెంట్

**సూపర్‌వైజర్ ఏజెంట్ ప్యాటర్న్** అనేది ఒక **ఫ్లెక్సిబుల్** ఏజెంటిక్ AI రూపం. సూపర్‌వైజర్ LLM ఉపయోగించి యూజర్ అభ్యర్థన ఆధారంగా ఏ ఏజెంట్లను పిలవాలో స్వతంత్రంగా నిర్ణయిస్తుంది. తదుపరి ఉదాహరణలో, MCP ఆధారిత ఫైల్ యాక్సెస్‌ను LLM ఏజెంట్‌తో కలిపి, పర్యవేక్షింపబడిన ఫైల్ పఠనం → రిపోర్ట్ వర్క్‌ఫ్లో సృష్టిస్తాము.

డెమోలో, `FileAgent` MCP ఫైల్‌సిస్టమ్ టూల్స్ ఉపయోగించి ఫైల్ చదువుతుంది, మరియు `ReportAgent` నిర్వహణ సారాంశం (1 వాక్యం), 3 కీలక విషయాలు మరియు సిఫార్సులతో నిర్మిత రిపోర్ట్ రూపొందిస్తుంది. సూపర్‌వైజర్ ఈ ప్రవాహాన్ని ఆటోమేటిక్‌గా నిర్వహిస్తుంది:

<img src="../../../translated_images/te/supervisor-agent-pattern.06275a41ae006ac8.webp" alt="Supervisor Agent Pattern" width="800"/>

*సూపర్‌వైజర్ LLM ద్వారా ఏ ఏజెంట్లను, ఏ క్రమంలో పిలవాలో నిర్ణయిస్తుంది — హార్డ్‌కోడ్ మార్గదర్శనం అవసరం లేదు.*

ఫైల్-టు-రిపోర్ట్ పైప్లైన్ యొక్క స్పష్టం అయిన వర్క్‌ఫ్లో ఇలా ఉంటుంది:

<img src="../../../translated_images/te/file-report-workflow.649bb7a896800de9.webp" alt="File to Report Workflow" width="800"/>

*FileAgent MCP టూల్స్ ద్వారా ఫైల్ చదివి, ReportAgent కాస్త కచ్ఛితమైన కంటెంట్‌ను నిర్మిత రిపోర్ట్‌గా మార్చుతుంది.*

క్రింది సీక్వెన్స్ డయాగ్రామ్ సూపర్‌వైజర్ పూర్తి ఆర్కెస్ట్రేషన్‌ను ట్రేస్ చేస్తుంది — MCP సర్వర్ ప్రారంభించడం, సూపర్‌వైజర్ స్వతంత్ర ఏజెంట్ ఎంపిక, stdio ద్వారా టూల్ కాల్స్ మరియు తుది రిపోర్ట్:

<img src="../../../translated_images/te/supervisor-agent-sequence.1aa389b3bef99956.webp" alt="Supervisor Agent Sequence Diagram" width="800"/>

*సూపర్‌వైజర్ స్వతంత్రంగా FileAgent ను పిలవడం (file చదవడానికి MCP సర్వర్‌కు stdio ద్వారా), తర్వాత ReportAgent కు నిర్మిత రిపోర్ట్ సృష్టించడానికి ఆహ్వానం ఇవ్వడం — ప్రతి ఏజెంట్ తన ఫలితాన్ని Agentic Scopeలో నిల్వ చేస్తుంది.*

ప్రతి ఏజెంట్ తన ఫలితాన్ని **Agentic Scope** (పంచే జ్ఞాపకం) లో నిల్వ చేస్తుంది, తదుపరి ఏజెంట్లు పూర్వ ఫలితాలను యాక్సెస్ చేసుకోవచ్చు. ఇది MCP టూల్స్ ఏజెంటిక్ వర్క్‌ఫ్లోలలో సజావుగా ఎలా లయబద్ధమవుతున్నాయో చూపిస్తుంది — సూపర్‌వైజర్ కి *ఎలా* ఫైల్‌లను చదవాలో తెలియకుండా, కేవలం `FileAgent` దీని కోసం పర్చుకోగలదు.

#### సూపర్‌వైజర్ ఎలా పనిచేస్తుంది

ఏజెంట్లను నిర్మించే ముందే, మీరు MCP ట్రాన్స్పోర్టును క్లయింట్‌కు కనెక్ట్ చేసి, దాన్ని `ToolProvider`గా చుట్టుకోవాలి. ఇలా MCP సర్వర్ టూల్స్ మీ ఏజెంట్లకు అందుబాటులోకి వస్తాయి:

```java
// ట్రాన్స్‌పోర్ట్ నుండి MCP క్లయింట్ సృష్టించండి
McpClient mcpClient = new DefaultMcpClient.Builder()
        .transport(stdioTransport)
        .build();

// క్లయింట్‌ను ToolProvider గా ముడిపర్చండి — ఇది MCP సాధనాలను LangChain4j లోకి అనుసంధానం చేస్తుంది
ToolProvider mcpToolProvider = McpToolProvider.builder()
        .mcpClients(List.of(mcpClient))
        .build();
```

ఇప్పుడు మీరు ఏ ఏజెంట్‌కు అయినా MCP టూల్స్ అవసరం అయితే `mcpToolProvider` ఇంజెక్ట్ చేయవచ్చు:

```java
// దశ 1: FileAgent MCP టూల్స్ ఉపయోగించి ఫైళ్లను చదువుతుంది
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // ఫైల్ ఆపరేషన్స్ కోసం MCP టూల్స్ కలిగి ఉంది
        .build();

// దశ 2: ReportAgent నిర్మిత నివేదికలను తయారు చేస్తుంది
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// Supervisor ఫైల్ → నివేదిక వర్క్‌ఫ్లోను సమన్వయం చేస్తుంది
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // తుది నివేదనను తిరిగివ్వండి
        .build();

// Supervisor అభ్యర్థన ఆధారంగా ఏ ఏజెంట్లను పిలవాలో నిర్ణయిస్తుంది
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```

#### FileAgent కాలంలో MCP టూల్స్‌ను ఎలా కనుగొంటుంది

మీకు ఆశ్చర్యం కలుగవచ్చు: **FileAgent npm ఫైల్‌సిస్టమ్ టూల్స్‌ను ఎలా ఉపయోగించాలో ఎలా తెలుసుకుంది?** సమాధానం అది తెలియదు — **LLM** కాలంలో టూల్ స్కీమాల ద్వారా దీన్ని తెలియజేస్తుంది.

`FileAgent` ఇంటర్‌ఫేస్ కేవలం **ప్రాంప్ట్ నిర్వచనం** మాత్రమే. దీనికి `read_file`, `list_directory` లేదా ఏ ఇతర MCP టూల్ గురించి ఏ హార్డ్‌కోడ్ జ్ఞానం లేదు. ఈ క్రింది విధంగా మొత్తం ప్రక్రియ జరుగుతుంది:
1. **సర్వర్ స్పాంస్:** `StdioMcpTransport` `@modelcontextprotocol/server-filesystem` npm ప్యాకేజ్‌ను చైల్డ్ ప్రాసెస్‌గా ప్రారంభిస్తుంది  
2. **టూల్ కనుగొనుట:** `McpClient` సర్వర్‌కి `tools/list` JSON-RPC అభ్యర్థనను పంపుతుంది, అది టూల్ పేర్లు, వివరణలు మరియు పారామీటర్ స్కీమాలతో (ఉదాహరణకు, `read_file` — *"ఫైల్ మొత్తం కంటెంట్ చదవండి"* — `{ path: string }`) స్పందిస్తుంది  
3. **స్కీమా ఇంజెక్షన్:** `McpToolProvider` ఈ కనుగొనబడిన స్కీమాలను చుట్టుకొని LangChain4j కి అందుబాటులో ఉంచుతుంది  
4. **LLM నిర్ణయం తీసుకోవడం:** `FileAgent.readFile(path)` పిలవబడినప్పుడు, LangChain4j సిస్టమ్ సందేశం, వినియోగదారుని సందేశం, **మరియు టూల్ స్కీమాల జాబితాను** LLM కి పంపుతుంది. LLM టూల్ వివరణలను చదివి టూల్ కాల్ ఉత్పత్తి చేస్తుంది (ఉదా: `read_file(path="/some/file.txt")`)  
5. **అమలు:** LangChain4j టూల్ కాల్‌ని అంతరాయం చేసి, దానిని MCP క్లయింట్ ద్వారా Node.js ఉపప్రాసెస్‌కి పంపుతుంది, ఫలితాన్ని పొందుతుంది మరియు LLM కి తిరిగి అందిస్తుంది  

ఈది పై పేర్కొన్న [టూల్ కనుగొనటం](../../../05-mcp) యంత్రాంగమే, కానీ ముఖ్యంగా ఏజెంట్ వర్క్‌ఫ్లోకి వర్తిస్తుంది. `@SystemMessage` మరియు `@UserMessage` గుర్తింపులు LLM వ్యవహారాన్ని మార్గనిర్దేశం చేస్తాయి, మరియు ఇంజెక్ట్ చేసిన `ToolProvider` దీని **సాధ్యతలు**ని ఇస్తుంది — LLM రెండింటిని రంటైమ్‌లో కలిపి పనిచేస్తుంది.

> - "ఏజెంట్ బిల్డర్ నుండి ToolProvider తీసివేస్తే ఏమవుతుంది?"  
> - "టూల్ స్కీమాలు LLM కి ఎలా పంపబడతాయి?"

#### స్పందన వ్యూహాలు

మీరు ఒక `SupervisorAgent` ని కాన్ఫిగర్ చేసేటప్పుడు, ఉప-ఏజెంట్లు తమ పనులు పూర్తిచేసిన తర్వాత ఇది వినియోగదారునికి తుది సమాధానాన్ని ఎలా రూపకల్పన చేయాలో నిర్దేశిస్తారు. కింద ఒక డైరాగ్రామ్ ఉంది — మూడు అందుబాటులో ఉన్న వ్యూహాలు చూపుతాయి — LAST చివరి ఏజెంట్ అవుట్పుట్‌ను నేరుగా తిరిగి ఇస్తుంది, SUMMARY అన్ని అవుట్పుట్లను LLM ద్వారా సంకలనం చేస్తుంది, మరియు SCORED అసలు అభ్యర్థనతో పోల్చి ఎక్కువ స్కోరు వచ్చిన ఆ అవుట్పుట్‌ను ఎంచుకొంటుంది:

<img src="../../../translated_images/te/response-strategies.3d0cea19d096bdf9.webp" alt="Response Strategies" width="800"/>

*Supervisor తుది స్పందనను ఎలా రూపొందించాలో మూడు వ్యూహాలు — చివరి ఏజెంట్ అవుట్పుట్ కావాలా, సమ్మరీ కావాలా లేదా ఉత్తమ స్కోరు పొందిన ఆప్షన్ కావాలా అని బట్టి ఎంపిక చేసుకోండి.*

| వ్యూహం | వివరణ |
|----------|-------------|
| **LAST** | సూపర్వైజర్ చివరి ఉప-ఏజెంట్ లేదా టూల్ నుండి వచ్చిన అవుట్పుట్‌ను తిరిగి ఇస్తుంది. ఇది చాలా ఉపయోగకరం, ముఖ్యంగా వర్క్‌ఫ్లోలో చివరిది పూర్తి సమాధానం ఇవ్వడానికి ప్రత్యేకంగా రూపొందించబడిన ఏజెంట్ ఉన్నప్పుడు (ఉదా: పరిశోధన పాఠంలో "సారాంశ ఏజెంట్"). |
| **SUMMARY** | సూపర్వైజర్ తన అంతర్గత LLM ఉపయోగించి మొత్తం సంభాషణ మరియు ఉప-ఏజెంట్ల అవుట్పుట్ల సారాంశాన్ని సంకలనం చేసి, దానిని తుది ప్రతిస్పందనగా ఇస్తుంది. ఇది వినియోగదారుకు పరిమిత, సమగ్ర సమాధానాన్ని అందిస్తుంది. |
| **SCORED** | సిస్టమ్ అంతర్గత LLM ఉపయోగించి LAST స్పందన మరియు SUMMARY రెండింటినీ అసలు వినియోగదారుడి అభ్యర్థనతో మోయి స్కోర్లు ఇచ్చి, ఎక్కువ స్కోరు పొందిన అవుట్పుట్‌ను తిరిగి ఇస్తుంది. |

#### అవుట్పుట్ అర్థం చేసుకోవడం

డెమోను నడిపితే, Supervisor ఎలా బహుళ ఏజెంట్లను ఏర్పాటుచేస్తుందో సరైన సూచనతో మీకు చూపబడుతుంది. ప్రతీ భాగం అర్థం:

```
======================================================================
  FILE → REPORT WORKFLOW DEMO
======================================================================

This demo shows a clear 2-step workflow: read a file, then generate a report.
The Supervisor orchestrates the agents automatically based on the request.
```
  
**శీర్షిక** వర్క్‌ఫ్లో కాన్సెప్ట్‌ను పరిచయం చేస్తుంది: ఫైల్ చదవడం నుండి రిపోర్టు తేజీకరణ వరకు ఒక దృష్టి సారించిన పైప్‌లైన్.

```
--- WORKFLOW ---------------------------------------------------------
  ┌─────────────┐      ┌──────────────┐
  │  FileAgent  │ ───▶ │ ReportAgent  │
  │ (MCP tools) │      │  (pure LLM)  │
  └─────────────┘      └──────────────┘
   outputKey:           outputKey:
   'fileContent'        'report'

--- AVAILABLE AGENTS -------------------------------------------------
  [FILE]   FileAgent   - Reads files via MCP → stores in 'fileContent'
  [REPORT] ReportAgent - Generates structured report → stores in 'report'
```
  
**వర్క్‌ఫ్లో డైరాగ్రామ్** ఏజెంట్ల మధ్య డేటా ప్రవాహాన్ని చూపిస్తుంది. ప్రతి ఏజెంట్ యొక్క ప్రత్యేక పాత్ర:  
- **FileAgent** MCP టూల్స్ ఉపయోగించి ఫైళ్లు చదవడం మరియు నార్వ లైన కంటెంట్ `fileContent`లో నిల్వ చేయడం  
- **ReportAgent** ఆ కంటెంట్‌ను ఉపయోగించి నిర్మితం చేసిన రిపోర్టును `report` లో ఉత్పత్తి చేయడం

```
--- USER REQUEST -----------------------------------------------------
  "Read the file at .../file.txt and generate a report on its contents"
```
  
**వినియోగదారి అభ్యర్థన** పని చూపిస్తుంది. Supervisor దీన్ని పార్స్ చేసి FileAgent → ReportAgent ను పిలవాలని నిర్ణయిస్తాడు.

```
--- SUPERVISOR ORCHESTRATION -----------------------------------------
  The Supervisor decides which agents to invoke and passes data between them...

  +-- STEP 1: Supervisor chose -> FileAgent (reading file via MCP)
  |
  |   Input: .../file.txt
  |
  |   Result: LangChain4j is an open-source, provider-agnostic Java framework for building LLM...
  +-- [OK] FileAgent (reading file via MCP) completed

  +-- STEP 2: Supervisor chose -> ReportAgent (generating structured report)
  |
  |   Input: LangChain4j is an open-source, provider-agnostic Java framew...
  |
  |   Result: Executive Summary...
  +-- [OK] ReportAgent (generating structured report) completed
```
  
**Supervisor ఏర్పాటు** రెండు దశల పనితీరును చూపిస్తుంది:  
1. **FileAgent** MCP ద్వారా ఫైల్ చదివి కంటెంట్ నిల్వ చేస్తుంది  
2. **ReportAgent** ఆ కంటెంట్‌ను స్వీకరించి నిర్మిత రిపోర్టును సృష్టిస్తుంది

Supervisor ఈ నిర్ణయాలను వినియోగదారి అభ్యర్థన ఆధారంగా **స్వయాంగతంగా** తీసుకున్నాడు.

```
--- FINAL RESPONSE ---------------------------------------------------
Executive Summary
...

Key Points
...

Recommendations
...

--- AGENTIC SCOPE (Data Flow) ----------------------------------------
  Each agent stores its output for downstream agents to consume:
  * fileContent: LangChain4j is an open-source, provider-agnostic Java framework...
  * report: Executive Summary...
```
  
#### Agentic మాడ్యూల్ లక్షణాల వివరణ

ఉదాహరణ agentic మాడ్యూల్‌లోని కొన్ని ఆధునిక లక్షణాలను ప్రదర్శిస్తుంది. Agentic స్కోప్ మరియు Agent Listeners ని మనం మరింత విడదీసి పరిశీలిద్దాం.

**Agentic స్కోప్** `@Agent(outputKey="...")` ఉపయోగించి ఏజెంట్లు తమ ఫలితాలను నిల్వ చేసిన సంయుక్త మెమరీని చూపిస్తుంది. ఇది వీటిని అనుమతిస్తుంది:  
- తరువాత ఏజెంట్లు ముందు ఏజెంట్ల అవుట్పుట్ లకు యాక్సెస్ చేయగలరు  
- Supervisor తుది స్పందనను సంకలనం చేయగలడు  
- మీరు ప్రతీ ఏజెంట్ ఉత్పత్తిని పరిశీలించగలరు

కింద డైరాగ్రామ్ ఫైల్-నుండి-రిపోర్టు వర్క్‌ఫ్లోలో Agentic స్కోప్ ఎలా సంయుక్త మెమరీగా పనిచేస్తుందో చూపిస్తుంది — FileAgent `fileContent` కీ క్రింద అవుట్పుట్ వ్రాస్తుంది, ReportAgent దానిని చదువుకుని `report` కింద తన సొంత అవుట్పుట్ రాస్తుంది:

<img src="../../../translated_images/te/agentic-scope.95ef488b6c1d02ef.webp" alt="Agentic Scope Shared Memory" width="800"/>

*Agentic స్కోప్ సంయుక్త మెమరీగా పనిచేస్తుంది — FileAgent `fileContent` వ్రాస్తుంది, ReportAgent దాన్ని చదివి `report` రాస్తుంది, మరియు మీ కోడ్ తుది ఫలితాన్ని చదవగలదు.*

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // FileAgent నుండి రా ఫైల్ డేటా
String report = scope.readState("report");            // ReportAgent నుండి స్ట్రక్చర్డ్ రిపోర్ట్
```
  
**Agent Listeners** ఏజెంట్ అమలును పర్యవేక్షణ మరియు డీబగ్గింగ్ చేయడానికి సహాయపడతాయి. మీరు డెమోలో చూస్తున్న దశలవారీ అవుట్పుట్ ఒక AgentListener నుండి వస్తుంది, ఇది ప్రతి ఏజెంట్ పిలుపునందు హుక్ అవుతుంది:  
- **beforeAgentInvocation** - సూపర్వైజర్ ఏ ఏజెంట్ ఎంచుకున్నాడో, ఎందుకు అనేది చూపిస్తుంది  
- **afterAgentInvocation** - ఏజెంట్ పూర్తయ్యాక దాని ఫలితాన్ని చూపిస్తుంది  
- **inheritedBySubagents** - true ఉన్నప్పుడు, ఇది హైరార్కీలోని అన్ని ఏజెంట్లను పర్యవేక్షిస్తుంది

కింద డైరాగ్రామ్ పూర్తి Agent Listener జీవనచక్రాన్ని చూపిస్తుంది, సహా అమలు లోపాలలో `onError` ఎలా నిర్వహిస్తుందో:

<img src="../../../translated_images/te/agent-listeners.784bfc403c80ea13.webp" alt="Agent Listeners Lifecycle" width="800"/>

*Agent Listeners అమలులో హుక్ అవుతాయి — ఏజెంట్లు ప్రారంభం అవటం, పూర్తి అవటం, లోపాలు ఎదుర్కొనటం అన్నింటినీ పర్యవేక్షించడం.*

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // అన్ని ఉప-ఏజెంట్లకు ప్రచారం చేయండి
    }
};
```
  
Supervisor మాదిరి బయట, `langchain4j-agentic` మాడ్యూల్ బలమైన వర్క్‌ఫ్లో ప్యాటర్న్స్ ని అందిస్తుంది. కింద డైరాగ్రామ్ ఆయదు — సాదా వరుస పైప్‌లైన్ల నుంచి మానవ అతిథి పరిశీలన వర్క్‌ఫ్లోలకు:

<img src="../../../translated_images/te/workflow-patterns.82b2cc5b0c5edb22.webp" alt="Agent Workflow Patterns" width="800"/>

*ఏజెంట్ల ఏర్పాటుకు ఐదు వర్క్‌ఫ్లో ప్యాటర్న్లు — సాదా వరుస పైప్‌లైన్‌ల నుండి మానవ అతిథి ఆమోద వర్క్‌ఫ్లో వరకు.*

| ప్యాటర్న్ | వివరణ | ఉపయోగం |
|---------|-------------|----------|
| **Sequential** | ఏజెంట్లను క్రమంలో అమలు చేయడం, అవుట్పుట్ తర్వాతి ఏజెంట్కి సరఫరా అవడం | పైప్‌లైన్లు: పరిశోధన → విశ్లేషణ → రిపోర్టు |
| **Parallel** | ఏజెంట్లను సమకాలీనంగా నడపడం | స్వతంత్రమైన పనులు: వాతావరణం + వార్తలు + స్టాక్‌లు |
| **Loop** | షరతు తీర్చేవరకు పునరావృతం | నాణ్యత స్కోరింగ్: స్కోరు ≥ 0.8 అయ్యేవరకు శుద్ధి |
| **Conditional** | షరతులపై ఆధారపడి మార్గ నిర్ణయం | వర్గీकरण → నిపుణుడు ఏజెంట్‌కి మార్గం |
| **Human-in-the-Loop** | మానవ ఆమోద పాయింట్లు చేర్చడం | ఆమోద వర్క్‌ఫ్లోలు, కంటెంట్ సమీక్ష |

## ముఖ్యమైన భావనలు

మీకేం MCP మరియు agentic మాడ్యూల్ అమలులో ఆసక్తి ఉందో ఇప్పుడు మీరు తెలుసుకున్న తర్వాత, తక్కువగా ఎప్పుడు ఏ విధానాన్ని వాడాలో సంక్షిప్తంగా:

MCP యొక్క పెద్ద లాభం దాని పెరుగుతున్న ఎకోసిస్టమ్. కింద డైరాగ్రామ్ ఒక మరియు సర్వత్ర MCP సర్వర్లతో మీ AI అప్లికేషన్ जुడ్చే ఏకైక విలువైన ప్రోటోకాల్ ని చూపిస్తుంది — ఫైల్‌సిస్టమ్ మరియు డేటాబేస్ యాక్సెస్ నుండి GitHub, ఇమెయిల్, వెబ్ స్క్రాపింగ్ మరియు మరెన్నో వరకు:

<img src="../../../translated_images/te/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="MCP Ecosystem" width="800"/>

*MCP ఏకైక ప్రోటోకాల్ ఎకోసిస్టమ్ సృష్టిస్తుంది — ఏ MCP-అనుకూల సర్వర్ ఏ MCP-అనుకూల క్లయింట్‌తో పని చేస్తుంది, టూల్ షేరింగ్‌ను అనుమతిస్తుంది.*

**MCP** మీరు ఇప్పటికే ఉన్న టూల్ ఎకోసిస్టంలను ఉపయోగించాలనుకున్న ప్రతిసారి, పలు అనువర్తనాలు పంచుకోగల టూల్స్ ను నిర్మించాలనుకున్నప్పుడు, ప్రమాణ ప్రోటోకాలుతో మూడవ పార్టీ సేవలను సమ్మిళితం చేయాలనుకున్నప్పుడు, లేదా కోడ్ మార్చకుండా టూల్ అమలులను మార్చుకోవాలనుకున్నప్పుడు మంచిది.

**Agentic మాడ్యూల్** మీరు `@Agent` గుర్తింపులతో డిక్లరేటివ్ ఏజెంట్ నిర్వచనాలు కోరుకున్నప్పుడు, వర్క్‌ఫ్లో ఏర్పాటును (సీక్వెన్షియల్, లూప్, పారలల్) అవసరమయినప్పుడు, ఆదేశాత్మక కోడ్ కంటే ఇంటర్‌ఫేస్ ఆధారిత ఏజెంట్ రూపకల్పన ఇష్టపడినప్పుడు, లేదా `outputKey` ద్వారా పంచుకునే అవుట్పుట్లు ఉన్న అనేక ఏజెంట్లను కలిపేటప్పుడు ఉత్తమం.

**Supervisor Agent నమూనా** ముందుగానే వర్క్‌ఫ్లో ఊహించలేనప్పుడు, LLM నిర్ణయం తీసుకోవాలనుకునే సమయంలో, అనేక విశేష ఏజెంట్ల గ dynamic ఏర్పాట్లు అవసరమైతే, వివిధ సామర్థ్యాలకు దారి తీసే సంభాషణాత్మక వ్యవస్థలు నిర్మించేటప్పుడు, లేదా అత్యంత సరళమైన, సరిపోయే ఏజెంట్ ప్రవర్తన కావాలనుకోగలిగేటప్పుడు ప్రతిభావంతంగా పనిచేస్తుంది.

మొడ్యూల్ 04 నుంచి ఉన్న స్వంత `@Tool` పద్ధతుల 与 ఈ మాడ్యూల్ నుండి MCP టూల్స్ మధ్య తేడాలు తెలుసుకోవడానికి ఈ కింది కంపేరిజన్ బొమ్మ చూడండి — స్వంత టూల్స్ అప్-స్పెసిఫిక్ లాజిక్ కి కడుపా పట్టు మరియు పూర్తి టైప్ సేఫ్టీ ఇస్తాయి, MCP టూల్స్ ప్రమాణీకృత, పునర్వినియోగించగల ఇంటిగ్రేషన్లు ఇస్తాయి:

<img src="../../../translated_images/te/custom-vs-mcp-tools.c4f9b6b1cb65d8a1.webp" alt="Custom Tools vs MCP Tools" width="800"/>

*ఎప్పుడు స్వంత @Tool పద్ధతులు, ఎప్పుడు MCP టూల్స్ వాడాలో — ఆత్ స్పెసిఫిక్ లాజిక్ కోసం పూర్తి టైప్ సేఫ్టీతో స్వంత టూల్స్, అనువర్తనాల మధ్య పని చేసే ప్రమాణీకృత ఇంటిగ్రేషన్ల కోసం MCP టూల్స్.*
