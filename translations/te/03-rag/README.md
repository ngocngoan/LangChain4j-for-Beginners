## మీరు నేర్చుకోబోయేది

మునుపటి మాడ్యూల్స్ లో, మీరు AIతో సంభాషణలు ఎలా జరపాలో మరియు మీ ప్రాంప్ట్స్ ని సక్రమంగా నిర్మించే విధానాన్ని నేర్చుకున్నారు. కానీ ఒక మౌలిక పరిమితి ఉంది: భాషా మోడల్స్ కేవలం శిక్షణ సమయంలో నేర్చుకున్న విషయాలకే పరిధి కలిగి ఉంటాయి. అవి మీ కంపెనీ పాలసీల, మీ ప్రాజెక్ట్ డాక్యుమెంటేషన్, లేదా శిక్షణ పొందకపోయిన ఏ సమాచారంపై ప్రశ్నలకు జవాబు చెప్పలేవు.

RAG (రిట్రీవల్-ఆగ్మెంటెడ్ జనరేషన్) ఈ సమస్యను పరిష్కరిస్తుంది. మోడల్‌ను మీ సమాచారం నేర్పించే ప్రయత్నం చేసే ಬದಲು (దీని ఖర్చు ఎక్కువ, పనికిరాని), మీరు దానికి మీ డాక్యుమెంట్లలో సెర్చ్ చేసే సామర్థ్యం ఇస్తారు. ఎవరో ప్రశ్న అడిగితే, సిస్టమ్ సంబంధిత సమాచారాన్ని కనుగొని దాన్ని ప్రాంప్ట్‌లో చేరుస్తుంది. ఆపై మోడల్ ఆ పొందిన కాంటెక్స్ట్ ఆధారంగా జవాబు ఇస్తుంది.

RAG ని మోడల్‌కు ఒక సూచన పుస్తకాలశాల ఇవ్వడం అని భావించండి. మీరు ప్రశ్న అడిగినప్పుడు, సిస్టమ్:

1. **వినియోగదారు ప్రశ్న** - మీరు ప్రశ్న అడుగుతారు  
2. **ఎంబెడ్డింగ్** - మీ ప్రశ్నను వెక్టార్‌గా మార్చడం  
3. **వెక్టార్ సెర్చ్** - సంబంధిత డాక్యుమెంట్ చంక్స్ కనుగొనడం  
4. **కాంటెక్స్ట్ అసెంబ్లీ** - సంబంధిత చంక్స్‌ను ప్రాంప్ట్‌లో చేర్చడం  
5. **జవాబు** - కాంటెక్స్ట్ ఆధారంగా LLM జవాబు సృష్టిస్తుంది  

ఇది మోడల్ యొక్క జవాబులను మీ నిజమైన డేటాపై ఆధారపడితుంది, శిక్షణ జ్ఞానంపై లేదా అంచనాలపై కాదు.

## RAG ను అర్థం చేసుకోండి

క్రింది చిత్రంలో ప్రాథమిక భావన చూపబడింది: మోడల్ శిక్షణ డేటాకు మాత్రమే ఆధారపడకుండా, RAG దానికి మీ డాక్యుమెంట్ల సూచన పుస్తకాలశాల ఇస్తుంది, ఒక్కో జవాబు సృష్టించే ముందు దీనిని పరిశీలించడానికి.

ఇక్కడ భాగాలు పూర్తిగా ఎలా కలిసిపోతాయో చూపించారు. వినియోగదారు ప్రశ్న నాలుగు దశల ద్వారా ప్రవహిస్తుంది—ఎంబెడ్డింగ్, వెక్టార్ సెర్చ్, కాంటెక్స్ట్ అసెంబ్లీ, మరియు జవాబు సృష్టించడం—ప్రతి దశ మునుపటి దశపై ఆధారపడి ఉంటుంది:

### డాక్యుమెంట్ ప్రాసెసింగ్

[DocumentService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/DocumentService.java)

మీరు డాక్యుమెంట్ అప్లోడ్ చేసినప్పుడు, సిస్టమ్ దాన్ని పార్స్ చేస్తుంది (PDF లేదా సాదాసీదా టెక్స్ట్), ఫైల్ పేరు వంటి మెటాడేటాను జత చేస్తుంది, తరువాత దాన్ని చంక్‌లు చెయ్యబడతాయి—ఇవి చిన్న భాగాలు, మోడల్ యొక్క కాంటెక్స్ట్ విండోలో సౌకర్యంగా వుంటాయి. ఈ చంక్‌లు కొంత భాగంగా ఒకదానితో మరొకటి ఎరదవి అవుతాయి, ఫలితంగా అంచుల వద్ద కాంటెక్స్ట్ పోవదు.

```java
// అప్‌లోడ్ చేసిన ఫైల్‌ను వర్గీకరించి LangChain4j డాక్యుమెంట్‌లో ముట్టడం
Document document = Document.from(content, metadata);

// 30-టోకెన్ అట్టుకోవడం తో 300-టోకెన్ భాగాలుగా విభజించండి
DocumentSplitter splitter = DocumentSplitters
    .recursive(300, 30);

List<TextSegment> segments = splitter.split(document);
```
  
క్రింది చిత్రంలో దీని విజువల్ ప్రదర్శన ఉంది. ప్రతి చంక్ తన పొరుగువరితో కొంత టోకెన్లు పంచుకుంటుంది—30 టోకెన్ ఎరదవి ముఖ్యమైన కాంటెక్స్ట్ మిస్సవకుండా చూస్తుంది:

### ఎంబెడ్డింగ్స్ సృష్టించడం

[LangChainRagConfig.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/config/LangChainRagConfig.java)

ప్రతి చంక్‌ను గణాంకాత్మక ప్రతీక (embedding) గా మార్చబడుతుంది — ఇది టెక్స్ట్ యొక్క అర్థం ను పట్టు చేసే గణితాత్మక ఫింగర్ప్రింట్. సన్నిహిత టెక్స్ట్‌లు సమాన ఎంబెడ్డింగ్స్‌ను ఉత్పత్తి చేస్తాయి.

```java
@Bean
public EmbeddingModel embeddingModel() {
    return OpenAiOfficialEmbeddingModel.builder()
        .baseUrl(azureOpenAiEndpoint)
        .apiKey(azureOpenAiKey)
        .modelName(azureEmbeddingDeploymentName)
        .build();
}

EmbeddingStore<TextSegment> embeddingStore = 
    new InMemoryEmbeddingStore<>();
```
  
క్లాస్ డియాగ్రామ్ లో ఈ LangChain4j భాగాలు ఎలా కలుసుకున్నాయో చూపబడింది. `OpenAiOfficialEmbeddingModel` టెక్స్ట్‌ను వెక్టార్లుగా మార్చుతుంది, `InMemoryEmbeddingStore` వాటిని వారికి సంబంధించిన `TextSegment` డేటాతో నిల్వచేస్తుంది, మరియు `EmbeddingSearchRequest` షూట్ పరిమాణాలు వంటి స్ధితులను నియంత్రిస్తుంది:

<img src="../../../translated_images/te/rag-langchain4j-classes.bbf3aa9077ab443d.webp" alt="LangChain4j RAG క్లాసులు" width="800"/>

ఎంబెడ్డింగ్స్ నిల్వచేసిన తరువాత, సంబంధిత కంటెంట్ సహజంగానే వెక్టర్ ప్రాంతంలో క్లస్టర్లుగా ఏర్పడతాయి. క్రింది విజువలైజేషన్ డాక్యుమెంట్లు సంబంధిత విషయాల గురించి ఎలా దగ్గరగా పాయింట్లుగా ఉంటాయో చూపుతుంది, ఇది సెమాంటిక్ సెర్చ్ ని సాధ్యంచేస్తుంది:

### సెమాంటిక్ సెర్చ్

[RagService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/RagService.java)

మీరు ప్రశ్న అడిగితే, మీ ప్రశ్న కూడా ఒక ఎంబెడ్డింగ్ గా మారుతుంది. సిస్టమ్ మీరు అడిగిన ప్రశ్నకు సంబంధించిన ఎంబెడ్డింగ్ ను అన్ని డాక్యుమెంట్ చంక్స్ ఎంబెడ్డింగ్స్ తో పోల్చుతుంది. ఎక్కువ సమానార్థకత కలిగిన చంక్స్ కనుగొనబడతాయి—కేవలం కీవర్డ్లతో కాకుండా నిజమైన సెమాంటిక్ సమానత ఆధారంగా.

```java
Embedding queryEmbedding = embeddingModel.embed(question).content();

EmbeddingSearchRequest searchRequest = EmbeddingSearchRequest.builder()
    .queryEmbedding(queryEmbedding)
    .maxResults(5)
    .minScore(0.5)
    .build();

EmbeddingSearchResult<TextSegment> searchResult = embeddingStore.search(searchRequest);
List<EmbeddingMatch<TextSegment>> matches = searchResult.matches();

for (EmbeddingMatch<TextSegment> match : matches) {
    String relevantText = match.embedded().text();
    double score = match.score();
}
```
  
క్రింది చిత్రంలో సెమాంటిక్ సెర్చ్ మరియు సాంప్రదాయ కీవర్డ్ సెర్చ్ మధ్య తేడాలు చూపించారు. "వాహనం" అనే కీవర్డ్ సెర్చ్ "కారు మరియు ట్రక్కులు" అనే చంక్ మిస్సవుతుంది, కానీ సెమాంటిక్ సెర్చ్ అవి ఒకే అర్థం ఇవ్వడం తెలుసుకొని, అధిక స్కోరు గల మ్యాచ్‌గా అందిస్తుంది:

### డాక్యుమెంట్ అప్లోడ్ చేయండి

ముందుగా డాక్యుమెంట్ అప్లోడ్ చేయండి - TXT ఫైల్స్ టెస్టింగ్ కోసం మంచివి. ఈ డైరెక్టరీలో `sample-document.txt` అందుబాటులో ఉంది, ఇందులో LangChain4j ఫీచర్స్, RAG అమలు, మరియు ఉత్తమ పద్ధతులు వివరించబడ్డాయి—సిస్టమ్‌ను పరీక్షించడానికి అనుకూలం.

సిస్టమ్ మీ డాక్యుమెంట్‌ను ప్రాసెస్ చేసి, దానిని చంక్స్‌గా బిభజించి, ప్రతి చంక్‌కు ఎంబెడ్డింగ్స్ సృష్టిస్తుంది. ఇది అప్లోడ్ చేసిన వెంటనే ఆటోమేటిక్ గా జరుగుతుంది.

### ప్రశ్నలు అడగండి

ఇప్పుడు డాక్యుమెంట్ కంటెంట్ గురించి నిర్దిష్ట ప్రశ్నలు అడగండి. డాక్యుమెంట్ స్పష్టం చేసిన వాస్తవాలను ప్రయత్నించండి. సిస్టమ్ సంబంధిత చంక్స్ కోసం వెతుకుతుంది, వాటిని ప్రాంప్ట్‌లో చేర్చుతుంది, మరియు సమాధానం సృష్టిస్తుంది.

### మూల ప్రసూతి సూచనలు తనిఖీ చేయండి

ప్రతి సమాధానంలో మూల సూచనలు ఉంటాయి, వాటికి సామ్య స్కోర్లు కూడా ఉంటాయి. ఈ స్కోర్లు (0 నుండి 1 వరకూ) మీ ప్రశ్నకు సంబంధించి చంక్ ఎంత వరకు సంబంధితమో చూపిస్తాయి. ఎక్కువ స్కోర్లు మెరుగైన సరిపోకలను సూచిస్తాయి. ఇది మీరు సమాధానాన్ని మూల పత్రాలతో సరిచూసుకోవడంలో సహాయపడుతుంది.

ఈ మాడ్యూల్ సొమ్ము కోసం ఇన్-మెమరీ స్టోరేజ్ ఉపయోగిస్తుంది. మీరు అప్లికేషన్ కొత్తగా ప్రారంభించినప్పుడు, అప్లోడ్ చేసిన డాక్యుమెంటులు పోతాయి. ప్రొడక్షన్ సిస్టమ్స్ Qdrant లేదా Azure AI Search వంటి తురిత వెక్టర్ డేటాబేస్ లను ఉపయోగిస్తాయి.

### కాంటెక్స్ట్ విండో నిర్వహణ

ప్రతి మోడల్‌కు గరిష్ట కాంటెక్స్ట్ విండో ఉంటుంది. పెద్ద డాక్యుమెంటు నుంచి అన్ని చంక్స్‌ను ప్రాంప్ట్‌లో చేర్చలేరు. సిస్టమ్ అత్యంత సంబంధిత టాప్ N చంక్స్ (డిఫాల్ట్ 5) ను పరిమితిలో ఉంచి సరిపడా కాంటెక్స్ట్ ఇస్తుంది, తద్వారా సమాధానాలు ఖచ్చితంగా ఉత్పత్తి అవుతాయి.

## ఎప్పుడు RAG ముఖ్యం

RAG ఎప్పుడూ సరైన 접근ం కాదు. క్రింది నిర్ణయ మార్గదర్శిని మీరు ఎప్పుడు RAG విలువను కలిగించుగలదో, ఎప్పుడు సరళమైన పద్ధతులు (ప్రాంప్ట్‌లో కంటెంట్ నేరుగా చేర్చడం లేదా మోడల్ లోపల నిర్మిత జ్ఞానంపై ఆధారపడడం) సరిపోతాయో సహాయపడుతుంది:

<img src="../../../translated_images/te/when-to-use-rag.1016223f6fea26bc.webp" alt="ఎప్పుడు RAG ఉపయోగించాలి" width="800"/>

**RAG ఉపయోగించండి:**
- స్వంత డాక్యుమెంట్ల గురించి ప్రశ్నలకు సమాధానాలు ఇవ్వడం  
- సమాచారం తరచూ మారుతుంది (నీతులు, ధరలు, స్పెసిఫికేషన్స్)  
- ఖచ్చితత్వం కోసం మూల్య సూచన అవసరం  
- కంటెంట్ ఒకే ప్రాంప్ట్‌లో పెట్టడానికి చాలా పెద్దది  
- మీరు నిర్ధారించదగిన, ఆధారపడగలిగే సమాధానాలు కావాలి  

**ఈ సందర్భాల్లో RAG ఉపయోగించకండి:**  
- మోడల్‌కు ఇప్పటికే ఉన్న సాధారణ జ్ఞానం అవసరం ఉన్న ప్రశ్నలు  
- వాస్తవసమయం డేటా అవసరం (RAG అప్‌లోడ్ చేయబడిన డాక్యుమెంట్లపై పనిచేస్తుంది)  
- కంటెంట్ ప్రాంప్ట్‌లలో నేరుగా చేర్చడానికి సరిపడా చిన్నప్పుడు
