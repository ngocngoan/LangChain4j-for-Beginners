## چه چیزی یاد خواهید گرفت

تا کنون آموخته‌اید چگونه با هوش مصنوعی گفتگو کنید، پرامپت‌ها را به‌صورت مؤثر ساختاربندی کنید و پاسخ‌ها را بر اساس اسناد خود استوار سازید. اما هنوز یک محدودیت اساسی وجود دارد: مدل‌های زبانی فقط می‌توانند متن تولید کنند. آنها نمی‌توانند وضعیت آب و هوا را بررسی کنند، محاسبات انجام دهند، پایگاه‌های داده را پرس‌وجو کنند، یا با سیستم‌های خارجی تعامل داشته باشند.

ابزارها این وضعیت را تغییر می‌دهند. با دادن دسترسی به مدل برای فراخوانی توابع، شما آن را از یک تولیدکننده متن به عاملی تبدیل می‌کنید که می‌تواند اقداماتی انجام دهد. مدل تصمیم می‌گیرد که چه زمانی به یک ابزار نیاز دارد، کدام ابزار را استفاده کند و چه پارامترهایی را ارسال نماید. کد شما تابع را اجرا می‌کند و نتیجه را برمی‌گرداند. مدل آن نتیجه را در پاسخ خود وارد می‌کند.

## درک عامل‌های هوش مصنوعی با ابزارها

> **📝 توجه:** اصطلاح «عامل‌ها» در این ماژول به دستیاران هوش مصنوعی‌ای اشاره دارد که با قابلیت فراخوانی ابزارها تقویت شده‌اند. این با الگوهای **Agentic AI** (عامل‌های خودمختار با برنامه‌ریزی، حافظه، و استدلال چندمرحله‌ای) که در [ماژول ۰۵: MCP](../05-mcp/README.md) پوشش می‌دهیم، متفاوت است.

بدون ابزارها، یک مدل زبانی فقط می‌تواند متن تولید کند بر اساس داده‌های آموزشی خود. از آن درباره وضعیت آب و هوا بپرسید، باید حدس بزند. اما با داشتن ابزارها، می‌تواند API وضعیت هوا را فراخوانی کند، محاسبه انجام دهد یا پایگاه داده را پرس‌وجو کند — و سپس آن نتایج واقعی را در پاسخ خود بگنجاند.

<img src="../../../translated_images/fa/what-are-tools.724e468fc4de64da.webp" alt="بدون ابزارها در مقابل با ابزارها" width="800"/>

*بدون ابزارها مدل فقط می‌تواند حدس بزند — با ابزارها می‌تواند APIها را فراخوانی کند، محاسبات انجام دهد، و داده‌های زمان واقعی را برگرداند.*

عامل هوش مصنوعی با ابزارها یک الگوی **استدلال و اقدام (ReAct)** را دنبال می‌کند. مدل فقط پاسخ نمی‌دهد — بلکه می‌اندیشد به آنچه نیاز دارد، اقدام می‌کند با فراخوانی یک ابزار، نتیجه را مشاهده می‌کند و سپس تصمیم می‌گیرد که دوباره اقدام کند یا پاسخ نهایی را ارائه دهد:

۱. **استدلال** — عامل سؤال کاربر را تحلیل می‌کند و مشخص می‌کند چه اطلاعاتی نیاز دارد  
۲. **اقدام** — عامل ابزار مناسب را انتخاب می‌کند، پارامترهای صحیح را ایجاد می‌کند، و آن را فراخوانی می‌کند  
۳. **مشاهده** — عامل خروجی ابزار را دریافت و ارزیابی می‌کند  
۴. **تکرار یا پاسخ** — اگر داده‌های بیشتری لازم است، چرخه برمی‌گردد؛ در غیر این صورت پاسخ نهایی به زبان طبیعی را می‌سازد

<img src="../../../translated_images/fa/react-pattern-detail.96a5efeeb6dd2f61.webp" alt="الگوی ReAct" width="800"/>

*چرخه ReAct — عامل درباره چه کاری باید انجام دهد استدلال می‌کند، با فراخوانی ابزار اقدام می‌کند، نتیجه را مشاهده می‌کند، و تا زمانی که بتواند پاسخ نهایی ارائه دهد تکرار می‌کند.*

این روند به‌صورت خودکار اتفاق می‌افتد. شما ابزارها و توضیحات آنها را تعریف می‌کنید. مدل تصمیم‌گیری درباره اینکه چه زمانی و چگونه از آنها استفاده کند را بر عهده دارد.

## نحوه کار فراخوانی ابزار

### تعاریف ابزار

[WeatherTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) | [TemperatureTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/TemperatureTool.java)

شما توابعی را با توضیحات واضح و مشخصات پارامترها تعریف می‌کنید. مدل این توضیحات را در پرامپت سیستمی خود می‌بیند و می‌فهمد هر ابزار چه کاری انجام می‌دهد.

```java
@Component
public class WeatherTool {
    
    @Tool("Get the current weather for a location")
    public String getCurrentWeather(@P("Location name") String location) {
        // منطق جستجوی آب و هوا شما
        return "Weather in " + location + ": 22°C, cloudy";
    }
}

@AiService
public interface Assistant {
    String chat(@MemoryId String sessionId, @UserMessage String message);
}

// دستیار به طور خودکار توسط Spring Boot متصل شده است با:
// - bean مدل چت
// - تمام متدهای @Tool از کلاس‌های @Component
// - فراهم‌کننده حافظه چت برای مدیریت نشست
```
  
نمودار زیر هر توضیح را جداگانه شکافته و نشان می‌دهد چگونه هر بخش به هوش مصنوعی کمک می‌کند که بفهمد چه زمانی ابزار را فراخوانی کند و چه آرگومان‌هایی ارسال نماید:

<img src="../../../translated_images/fa/tool-definitions-anatomy.f6468546037cf28b.webp" alt="آناتومی تعاریف ابزارها" width="800"/>

*آناتومی یک تعریف ابزار — @Tool به هوش مصنوعی می‌گوید چه زمانی از آن استفاده کند، @P هر پارامتر را توضیح می‌دهد، و @AiService همه چیز را در هنگام راه‌اندازی سیم‌کشی می‌کند.*

> **🤖 امتحان کنید با چت [GitHub Copilot](https://github.com/features/copilot):**  
> فایل [`WeatherTool.java`](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) را باز کنید و بپرسید:  
> - «چگونه می‌توانم به جای داده‌های شبیه‌سازی شده یک API واقعی هوا مانند OpenWeatherMap را یکپارچه کنم؟»  
> - «یک توضیح خوب برای ابزار که به هوش مصنوعی کمک کند آن را درست استفاده کند، چیست؟»  
> - «چگونه خطاهای API و محدودیت نرخ را در پیاده‌سازی ابزار مدیریت کنم؟»

### تصمیم‌گیری

وقتی کاربر «وضعیت هوا در سیاتل چطور است؟» را می‌پرسد، مدل به‌صورت تصادفی یک ابزار انتخاب نمی‌کند. بلکه نیت کاربر را با هر توضیح ابزاری که در دسترس دارد مقایسه می‌کند، هر کدام را بر اساس مرتبط بودن امتیاز می‌دهد و بهترین تطابق را انتخاب می‌کند. سپس تماس تابعی ساخت‌یافته با پارامترهای درست تولید می‌کند — در این حالت، `location` را برابر با `"Seattle"` قرار می‌دهد.

اگر هیچ ابزاری با درخواست کاربر مطابق نباشد، مدل پاسخ را از دانش خود می‌دهد. اگر چند ابزار متناسب باشند، دقیق‌ترین را انتخاب می‌کند.

<img src="../../../translated_images/fa/decision-making.409cd562e5cecc49.webp" alt="چگونه هوش مصنوعی تصمیم می‌گیرد کدام ابزار را استفاده کند" width="800"/>

*مدل هر ابزار موجود را با نیت کاربر ارزیابی و بهترین تطابق را انتخاب می‌کند — به همین دلیل نوشتن توضیحات واضح و خاص ابزار مهم است.*

### تولید پاسخ

مدل داده‌های هوا را دریافت کرده و آنها را به پاسخ زبان طبیعی برای کاربر قالب‌بندی می‌کند.

### معماری: اتو وایرینگ Spring Boot

این ماژول از ادغام LangChain4j با Spring Boot با اینترفیس‌های اعلانی `@AiService` استفاده می‌کند. هنگام راه‌اندازی، Spring Boot هر `@Component` که شامل متدهای `@Tool` است، Bean مدل چت شما و `ChatMemoryProvider` را کشف کرده و همه را در یک اینترفیس واحد `Assistant` بدون کد تکراری سیم‌کشی می‌کند.

<img src="../../../translated_images/fa/spring-boot-wiring.151321795988b04e.webp" alt="معماری اتو وایرینگ Spring Boot" width="800"/>

*اینترفیس @AiService مدل چت، کامپوننت‌های ابزار و ارائه‌دهنده حافظه را کنار هم قرار می‌دهد — Spring Boot تمام سیم‌کشی‌ها را به‌صورت خودکار انجام می‌دهد.*

مزایای کلیدی این رویکرد:

- **اتو وایرینگ Spring Boot** — مدل چت و ابزارها به‌صورت خودکار تزریق می‌شوند  
- **الگوی @MemoryId** — مدیریت حافظه مبتنی بر جلسه به‌صورت خودکار  
- **یک نمونه واحد** — دستیار یک بار ساخته می‌شود و برای کارایی بهتر استفاده مجدد می‌شود  
- **اجرای نوع‌امن** — متدهای جاوا مستقیم فراخوانی می‌شوند با تبدیل نوع  
- **هماهنگی چند مرحله‌ای** — زنجیره‌سازی ابزارها به‌صورت خودکار مدیریت می‌شود  
- **صفر کد تکراری** — هیچ فراخوانی دستی `AiServices.builder()` یا حافظه HashMap لازم نیست  

رویکردهای جایگزین (سازنده دستی `AiServices.builder()`) نیاز به کد بیشتر دارند و مزایای ادغام Spring Boot را ندارند.

## زنجیره‌سازی ابزارها

**زنجیره‌سازی ابزار** — قدرت واقعی عامل‌های مبتنی بر ابزار وقتی ظاهر می‌شود که یک سؤال واحد به چند ابزار نیاز داشته باشد. بپرسید «وضعیت هوا در سیاتل به فارنهایت چقدر است؟» و عامل به‌طور خودکار دو ابزار را زنجیره می‌کند: ابتدا `getCurrentWeather` را برای گرفتن دما به سلسیوس فراخوانی می‌کند، سپس آن مقدار را به `celsiusToFahrenheit` برای تبدیل می‌دهد — همه در یک نوبت مکالمه.

<img src="../../../translated_images/fa/tool-chaining-example.538203e73d09dd82.webp" alt="مثال زنجیره‌سازی ابزار" width="800"/>

*زنجیره‌سازی ابزار در عمل — عامل ابتدا getCurrentWeather را فراخوانی می‌کند، سپس نتیجه سلسیوس را به celsiusToFahrenheit می‌دهد و پاسخ ترکیبی ارائه می‌کند.*

این چیزی است که در برنامه در حال اجرا به این صورت است — عامل دو تماس ابزار را در یک نوبت مکالمه زنجیره می‌کند:

<a href="images/tool-chaining.png"><img src="../../../translated_images/fa/tool-chaining.3b25af01967d6f7b.webp" alt="زنجیره‌سازی ابزار" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*خروجی واقعی برنامه — عامل به‌طور خودکار getCurrentWeather → celsiusToFahrenheit را در یک نوبت زنجیره می‌کند.*

**شکست‌های مؤدبانه** — برای آب و هوای شهری بپرسید که در داده‌های شبیه‌سازی شده نیست. ابزار پیام خطا برمی‌گرداند و هوش مصنوعی توضیح می‌دهد که نمی‌تواند کمک کند به جای سقوط برنامه. ابزارها با ایمنی شکست می‌خورند.

<img src="../../../translated_images/fa/error-handling-flow.9a330ffc8ee0475c.webp" alt="جریان مدیریت خطا" width="800"/>

*وقتی یک ابزار شکست می‌خورد، عامل خطا را می‌گیرد و به جای سقوط با توضیح مفید پاسخ می‌دهد.*

این در یک نوبت مکالمه اتفاق می‌افتد. عامل به صورت خودکار چندین تماس ابزار را هماهنگ می‌کند.

## استفاده از برنامه

برنامه یک رابط وب فراهم می‌کند که می‌توانید با عامل هوش مصنوعی ای تعامل کنید که به ابزارهای هواشناسی و تبدیل دما دسترسی دارد.

<a href="images/tools-homepage.png"><img src="../../../translated_images/fa/tools-homepage.4b4cd8b2717f9621.webp" alt="رابط ابزارهای عامل هوش مصنوعی" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*رابط ابزارهای عامل هوش مصنوعی - نمونه‌های سریع و رابط چت برای تعامل با ابزارها*

### امتحان استفاده ساده از ابزار
با یک درخواست ساده شروع کنید: «100 درجه فارنهایت را به سانتی‌گراد تبدیل کن». عامل تشخیص می‌دهد که به ابزار تبدیل دما نیاز دارد، آن را با پارامترهای مناسب فراخوانی می‌کند و نتیجه را بازمی‌گرداند. توجه کنید چقدر این طبیعی به نظر می‌رسد - شما مشخص نکردید کدام ابزار را استفاده کند یا چگونه آن را فراخوانی کند.

### آزمایش زنجیره ابزارها

حالا چیزی پیچیده‌تر را امتحان کنید: «هوا در سیاتل چگونه است و آن را به فارنهایت تبدیل کن؟» مشاهده کنید که عامل این موضوع را مرحله‌به‌مرحله انجام می‌دهد. ابتدا هوا را دریافت می‌کند (که به سانتی‌گراد بازمی‌گردد)، تشخیص می‌دهد که باید به فارنهایت تبدیل کند، ابزار تبدیل را فراخوانی می‌کند و هر دو نتیجه را در یک پاسخ ترکیب می‌کند.

### مشاهده جریان گفتگو

رابط چت تاریخچه گفتگو را حفظ می‌کند و امکان تعامل چندمرحله‌ای را فراهم می‌کند. شما می‌توانید همه پرسش‌ها و پاسخ‌های قبلی را ببینید، که پیگیری گفتگو و درک اینکه عامل چگونه زمینه را در طول چند تبادل شکل می‌دهد را آسان می‌کند.

<a href="images/tools-conversation-demo.png"><img src="../../../translated_images/fa/tools-conversation-demo.89f2ce9676080f59.webp" alt="گفتگو با چند فراخوانی ابزار" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*گفتگوی چندمرحله‌ای شامل تبدیل‌های ساده، جستجوی آب‌وهوا و زنجیره ابزارها*

### آزمایش با درخواست‌های مختلف

ترکیب‌های مختلف را امتحان کنید:
- جستجوی هوا: «هوای توکیو چگونه است؟»
- تبدیل دما: «25 درجه سانتی‌گراد چند کلوین است؟»
- پرسش‌های ترکیبی: «هوای پاریس را بررسی کن و بگو آیا بالای 20 درجه سانتی‌گراد است؟»

توجه کنید چطور عامل زبان طبیعی را تفسیر می‌کند و آن را به فراخوانی‌های مناسب ابزار‌ها نگاشت می‌کند.

## مفاهیم کلیدی

### الگوی ReAct (استدلال و عمل)

عامل بین استدلال (تصمیم‌گیری درباره انجام چه کاری) و عمل (استفاده از ابزارها) تناوب دارد. این الگو، حل مسئله خودکار را ممکن می‌کند نه فقط پاسخ دادن به دستورالعمل‌ها.

### توضیحات ابزار اهمیت دارد

کیفیت توضیحات ابزار شما مستقیماً روی چگونگی استفاده خوب عامل از آن‌ها تأثیر می‌گذارد. توضیحات روشن و مشخص به مدل کمک می‌کند بفهمد کی و چگونه هر ابزار را فراخوانی کند.

### مدیریت جلسه

حاشیه‌نویسی `@MemoryId` مدیریت حافظه مبتنی بر جلسه را خودکار می‌کند. هر شناسه جلسه یک نمونه `ChatMemory` مختص به خود دریافت می‌کند که توسط bean `ChatMemoryProvider` مدیریت می‌شود، بنابراین چندین کاربر می‌توانند همزمان با عامل تعامل داشته باشند بدون اینکه گفتگوهایشان به هم مخلوط شود.

<img src="../../../translated_images/fa/session-management.91ad819c6c89c400.webp" alt="مدیریت جلسه با @MemoryId" width="800"/>

*هر شناسه جلسه به تاریخچه مکالمه‌ای جداگانه نگاشت می‌شود — کاربران هرگز پیام‌های یکدیگر را نمی‌بینند.*

### مدیریت خطا

ابزارها ممکن است دچار خطا شوند — API ها تایم‌اوت می‌شوند، پارامترها ممکن است نامعتبر باشند، سرویس‌های خارجی ممکن است قطع شوند. عامل‌های تولیدی باید مدیریت خطا داشته باشند تا مدل بتواند مشکلات را توضیح دهد یا جایگزین‌هایی را امتحان کند به جای اینکه کل برنامه از کار بیفتد. هنگامی که یک ابزار خطا پرتاب می‌کند، LangChain4j آن را گرفته و پیام خطا را به مدل می‌دهد، که سپس می‌تواند مشکل را با زبان طبیعی توضیح دهد.

## ابزارهای موجود

نمودار زیر اکوسیستم گسترده ابزارهایی که می‌توانید بسازید را نشان می‌دهد. این ماژول ابزارهای هواشناسی و دما را نشان می‌دهد، اما الگوی یکسان `@Tool` برای هر متد جاوا کار می‌کند — از پرس‌وجوهای پایگاه داده گرفته تا پردازش پرداخت‌ها.

<img src="../../../translated_images/fa/tool-ecosystem.aad3d74eaa14a44f.webp" alt="اکوسیستم ابزار" width="800"/>

*هر متد جاوا با @Tool علامت‌گذاری شده برای هوش مصنوعی در دسترس می‌شود — این الگو به پایگاه‌های داده، API ها، ایمیل، عملیات فایل و موارد بیشتر گسترش می‌یابد.*

## کی باید از عامل‌های مبتنی بر ابزار استفاده کرد

<img src="../../../translated_images/fa/when-to-use-tools.51d1592d9cbdae9c.webp" alt="کی باید از ابزارها استفاده کرد" width="800"/>

*راهنمای تصمیم‌گیری سریع — ابزارها برای داده‌های زمان واقعی، محاسبات و اقدامات هستند؛ دانش عمومی و کارهای خلاقانه به آن‌ها نیاز ندارند.*

**وقتی از ابزار استفاده کنید که:**
- پاسخ نیاز به داده‌های زمان واقعی دارد (هوا، قیمت سهام، موجودی)
- نیاز به انجام محاسبات فراتر از ریاضیات ساده دارید
- دسترسی به پایگاه داده‌ها یا API ها
- انجام اقدامات (ارسال ایمیل، ایجاد تیکت، به‌روزرسانی رکوردها)
- ترکیب چند منبع داده

**وقتی ابزار استفاده نکنید که:**
- سوال‌ها بتوانند از دانش عمومی پاسخ داده شوند
- پاسخ صرفاً گفتگو باشد
- زمان تأخیر ابزار تجربه را خیلی کند کند

## ابزارها در مقابل RAG

ماژول‌های 03 و 04 هر دو قابلیت‌های هوش مصنوعی را گسترش می‌دهند اما به شیوه‌های بنیادی متفاوت. RAG به مدل دسترسی به **دانش** می‌دهد با بازیابی اسناد. ابزارها به مدل توانایی انجام **عملیات** از طریق فراخوانی توابع می‌دهند.

<img src="../../../translated_images/fa/tools-vs-rag.ad55ce10d7e4da87.webp" alt="مقایسه ابزارها و RAG" width="800"/>

*RAG اطلاعات را از اسناد ایستا بازیابی می‌کند — ابزارها عملیات اجرا کرده و داده‌های پویا و زمان واقعی را می‌آورند. بسیاری از سیستم‌های تولیدی هر دو را ترکیب می‌کنند.*

در عمل، بسیاری از سیستم‌های تولیدی هر دو رویکرد را ترکیب می‌کنند: RAG برای پایه‌گذاری پاسخ‌ها در مستندات شما، و ابزارها برای دریافت داده‌های زنده یا انجام عملیات.
