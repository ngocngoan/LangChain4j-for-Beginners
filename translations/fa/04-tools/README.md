## آنچه یاد خواهید گرفت

تا به حال، یاد گرفته‌اید چگونه مکالمه با هوش مصنوعی داشته باشید، پیام‌های ورودی را به شکل موثر ساختاربندی کنید و پاسخ‌ها را مبتنی بر اسناد خود پایه‌گذاری کنید. اما هنوز یک محدودیت اساسی وجود دارد: مدل‌های زبان فقط می‌توانند متن تولید کنند. آنها نمی‌توانند وضعیت هوا را بررسی کنند، محاسبات انجام دهند، پایگاه داده‌ها را جستجو کنند یا با سیستم‌های خارجی تعامل داشته باشند.

ابزارها این وضعیت را تغییر می‌دهند. با دادن دسترسی مدل به توابعی که می‌تواند فراخوانی کند، آن را از یک تولیدکننده متن به عاملی تبدیل می‌کنید که می‌تواند اقدام کند. مدل تصمیم می‌گیرد چه زمانی به ابزاری نیاز دارد، کدام ابزار را استفاده کند و چه پارامترهایی را بفرستد. کد شما تابع را اجرا کرده و نتیجه را بازمی‌گرداند. مدل آن نتیجه را در پاسخ خود وارد می‌کند.

## درک عامل‌های هوش مصنوعی با ابزارها

> **📝 یادداشت:** اصطلاح «عامل‌ها» در این ماژول به دستیارهای هوش مصنوعی اشاره دارد که با قابلیت فراخوانی ابزار بهبود یافته‌اند. این با الگوهای **Agentic AI** (عامل‌های خودمختار با برنامه‌ریزی، حافظه و استدلال چندمرحله‌ای) که در [ماژول ۰۵: MCP](../05-mcp/README.md) پوشش می‌دهیم، متفاوت است.

بدون ابزار، یک مدل زبان فقط می‌تواند از داده‌های آموزشی‌اش متن تولید کند. از آن وضعیت فعلی هوا را بپرسید، مجبور است حدس بزند. اگر ابزارهای لازم به آن داده شود، می‌تواند از API هواشناسی استفاده کند، محاسبات انجام دهد یا پایگاه داده را جستجو کند — و سپس آن نتایج واقعی را در پاسخ خود وارد کند.

<img src="../../../translated_images/fa/what-are-tools.724e468fc4de64da.webp" alt="Without Tools vs With Tools" width="800"/>

*بدون ابزار مدل فقط حدس می‌زند — با ابزارها می‌تواند APIها را فراخوانی کند، محاسبات انجام دهد و داده‌های زمان واقعی بازگرداند.*

یک عامل هوش مصنوعی با ابزارها از الگوی **تفکر و اقدام (ReAct)** پیروی می‌کند. مدل فقط پاسخ نمی‌دهد — فکر می‌کند به چه چیزی نیاز دارد، با فراخوانی ابزار عمل می‌کند، نتیجه را مشاهده می‌کند، و سپس تصمیم می‌گیرد دوباره عمل کند یا پاسخ نهایی را ارائه دهد:

۱. **تفکر** — عامل سوال کاربر را تحلیل می‌کند و تعیین می‌کند به چه اطلاعاتی نیاز دارد  
۲. **اقدام** — عامل ابزار مناسب را انتخاب کرده، پارامترهای صحیح را تولید و آن را فراخوانی می‌کند  
۳. **مشاهده** — عامل خروجی ابزار را دریافت و نتیجه را ارزیابی می‌کند  
۴. **تکرار یا پاسخ‌دهی** — اگر داده بیشتر لازم باشد، حلقه تکرار می‌شود؛ در غیر این صورت پاسخ به زبان طبیعی آماده می‌شود

<img src="../../../translated_images/fa/react-pattern-detail.96a5efeeb6dd2f61.webp" alt="ReAct Pattern" width="800"/>

*چرخه ReAct — عامل درباره کاری که باید انجام دهد فکر می‌کند، با فراخوانی ابزار اقدام می‌کند، نتیجه را مشاهده کرده و تا زمانی که بتواند پاسخ نهایی را بدهد، تکرار می‌کند.*

این فرآیند به‌صورت خودکار اتفاق می‌افتد. شما ابزارها و توضیحات آن‌ها را تعریف می‌کنید. مدل مدیریت تصمیم‌گیری درباره زمان و نحوه استفاده آنها را بر عهده دارد.

## نحوه کار فراخوانی ابزار

### تعاریف ابزار

[WeatherTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) | [TemperatureTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/TemperatureTool.java)

شما توابعی با توضیحات واضح و مشخصات پارامترها تعریف می‌کنید. مدل این توضیحات را در پرامپت سیستمی خود مشاهده می‌کند و متوجه می‌شود هر ابزار چه کاری انجام می‌دهد.

```java
@Component
public class WeatherTool {
    
    @Tool("Get the current weather for a location")
    public String getCurrentWeather(@P("Location name") String location) {
        // منطق جستجوی آب و هوا شما
        return "Weather in " + location + ": 22°C, cloudy";
    }
}

@AiService
public interface Assistant {
    String chat(@MemoryId String sessionId, @UserMessage String message);
}

// دستیار به طور خودکار توسط Spring Boot راه‌اندازی شده است با:
// - Bean مدل چت
// - همه متدهای @Tool از کلاس‌های @Component
// - ChatMemoryProvider برای مدیریت نشست
```
  
نمودار زیر هر نشانه‌گذاری را تجزیه می‌کند و نشان می‌دهد هر قسمت چگونه به هوش مصنوعی کمک می‌کند تا متوجه شود چه زمانی باید ابزار را فراخوانی کند و چه آرگومان‌هایی ارسال کند:

<img src="../../../translated_images/fa/tool-definitions-anatomy.f6468546037cf28b.webp" alt="Anatomy of Tool Definitions" width="800"/>

*نحوه تعریف ابزار — @Tool به هوش مصنوعی می‌گوید چه زمانی باید استفاده شود، @P هر پارامتر را توصیف می‌کند و @AiService همه را در زمان راه‌اندازی به هم متصل می‌کند.*

> **🤖 امتحان با [GitHub Copilot](https://github.com/features/copilot) Chat:** فایل [`WeatherTool.java`](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) را باز کنید و بپرسید:  
> - «چگونه یک API هواشناسی واقعی مانند OpenWeatherMap را به جای داده‌های شبیه‌سازی شده ادغام کنم؟»  
> - «توضیح خوب یک ابزار که به هوش مصنوعی کمک می‌کند درست از آن استفاده کند، چگونه است؟»  
> - «چگونه خطاهای API و محدودیت‌های نرخ را در پیاده‌سازی ابزارها مدیریت کنم؟»

### تصمیم‌گیری

وقتی کاربر می‌پرسد «وضعیت هوا در سیاتل چطور است؟»، مدل به صورت تصادفی ابزاری انتخاب نمی‌کند. بلکه قصد کاربر را با هر توضیح ابزار که در دسترس دارد مقایسه می‌کند، به هر کدام نمره مرتبط بودن می‌دهد و بهترین گزینه را انتخاب می‌کند. سپس یک فراخوانی ساخت‌یافته تابع با پارامترهای صحیح تولید می‌کند — در این مورد `location` را روی `"Seattle"` تنظیم می‌کند.

اگر هیچ ابزاری با درخواست کاربر تطابق نداشته باشد، مدل به پاسخ دادن از دانش خود بازمی‌گردد. اگر چند ابزار تطبیق داشته باشند، دقیق‌ترین را انتخاب می‌کند.

<img src="../../../translated_images/fa/decision-making.409cd562e5cecc49.webp" alt="How the AI Decides Which Tool to Use" width="800"/>

*مدل هر ابزار موجود را با قصد کاربر ارزیابی می‌کند و بهترین تطبیق را انتخاب می‌کند — به همین دلیل نوشتن توضیحات واضح و دقیق ابزار مهم است.*

### تولید پاسخ

مدل داده‌های هواشناسی را دریافت کرده و آن را به یک پاسخ زبان طبیعی برای کاربر قالب‌بندی می‌کند.

### معماری: اتصال خودکار Spring Boot

این ماژول از ادغام LangChain4j با Spring Boot با رابط‌های اعلامی `@AiService` استفاده می‌کند. در زمان راه‌اندازی، Spring Boot هر `@Component` ای که شامل متدهای `@Tool` باشد، Bean مدل چت شما و `ChatMemoryProvider` را کشف کرده و همه را به یک رابط `Assistant` بدون نیاز به کد اضافی متصل می‌کند.

<img src="../../../translated_images/fa/spring-boot-wiring.151321795988b04e.webp" alt="Spring Boot Auto-Wiring Architecture" width="800"/>

*رابط @AiService مدل چت، مولفه‌های ابزار و حافظه را به هم متصل می‌کند — Spring Boot همه اتصال‌ها را به صورت خودکار مدیریت می‌کند.*

مزایای کلیدی این رویکرد:

- **اتصال خودکار Spring Boot** — مدل چت و ابزارها به صورت خودکار تزریق می‌شوند  
- **الگوی @MemoryId** — مدیریت حافظه مبتنی بر جلسه خودکار  
- **نمونه منفرد** — Assistant یکبار ایجاد و برای عملکرد بهتر مجدداً استفاده می‌شود  
- **اجرای ایمن نوع** — متدهای Java به شکل مستقیم با تبدیل نوع فراخوانی می‌شوند  
- **هماهنگی چند مرحله‌ای** — زنجیره‌سازی ابزار را به صورت خودکار مدیریت می‌کند  
- **بدون کد اضافی** — بدون فراخوانی دستی `AiServices.builder()` یا نقشه حافظه

روش‌های جایگزین (ساخت دستی `AiServices.builder()`) به کد بیشتری نیاز دارند و از مزایای ادغام Spring Boot محروم‌اند.

## زنجیره‌سازی ابزار

**زنجیره‌سازی ابزار** — قدرت واقعی عامل‌های مبتنی بر ابزار زمانی نمایان می‌شود که یک سوال به چند ابزار نیاز داشته باشد. بپرسید «وضع هوا در سیاتل به فارنهایت چقدر است؟» و عامل دو ابزار را به طور خودکار زنجیره می‌کند: ابتدا `getCurrentWeather` را برای گرفتن دما به سلسیوس فراخوانی می‌کند، سپس آن مقدار را به `celsiusToFahrenheit` برای تبدیل می‌فرستد — همه در یک نوبت مکالمه.

<img src="../../../translated_images/fa/tool-chaining-example.538203e73d09dd82.webp" alt="Tool Chaining Example" width="800"/>

*زنجیره‌سازی ابزار در عمل — عامل ابتدا getCurrentWeather را فراخوانی می‌کند، سپس نتیجه سلسیوس را به celsiusToFahrenheit می‌فرستد و پاسخ ترکیبی را ارائه می‌دهد.*

**شکست‌های ملایم** — اگر آب و هوا را برای شهری بخواهید که در داده‌های شبیه‌سازی موجود نیست، ابزار پیام خطا برمی‌گرداند و هوش مصنوعی توضیح می‌دهد که نمی‌تواند کمک کند به جای اینکه برنامه کرش کند. ابزارها به شکلی ایمن شکست می‌خورند. نمودار زیر دو رویکرد را مقایسه می‌کند — با مدیریت صحیح خطا، عامل استثنا را گرفته و با توضیح مفید پاسخ می‌دهد، در حالی که بدون آن کل برنامه کرش می‌کند:

<img src="../../../translated_images/fa/error-handling-flow.9a330ffc8ee0475c.webp" alt="Error Handling Flow" width="800"/>

*وقتی ابزار شکست می‌خورد، عامل خطا را گرفته و با توضیح مفید پاسخ می‌دهد به جای اینکه برنامه کرش کند.*

این در یک نوبت مکالمه اتفاق می‌افتد. عامل به صورت مستقل چند فراخوانی ابزار را هماهنگ می‌کند.

## اجرای برنامه

## استفاده از برنامه

برنامه یک رابط وب فراهم می‌کند که می‌توانید با عاملی از هوش مصنوعی تعامل کنید که به ابزارهای هواشناسی و تبدیل دما دسترسی دارد. این هم نمای رابط — شامل مثال‌های شروع سریع و پنل چت برای ارسال درخواست‌ها است:
<a href="images/tools-homepage.png"><img src="../../../translated_images/fa/tools-homepage.4b4cd8b2717f9621.webp" alt="رابط کاربری ابزارهای عامل هوش مصنوعی" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*رابط کاربری ابزارهای عامل هوش مصنوعی - مثال‌های سریع و رابط چت برای تعامل با ابزارها*

### آزمایش استفاده ساده از ابزار

با یک درخواست ساده شروع کنید: «100 درجه فارنهایت را به سلسیوس تبدیل کن». عامل متوجه می‌شود که نیاز به ابزار تبدیل دما دارد، آن را با پارامترهای صحیح فراخوانی می‌کند و نتیجه را بازمی‌گرداند. توجه کنید که این چقدر طبیعی است - شما مشخص نکردید از کدام ابزار استفاده شود یا چگونه آن را فراخوانی کنید.

### آزمایش زنجیره‌ای کردن ابزارها

حالا چیز پیچیده‌تری را امتحان کنید: «هوای سئاتل چگونه است و آن را به فارنهایت تبدیل کن؟» ببینید عامل چگونه این کار را در مراحل انجام می‌دهد. ابتدا هوا را دریافت می‌کند (که در سلسیوس است)، متوجه می‌شود نیاز به تبدیل به فارنهایت دارد، ابزار تبدیل را فراخوانی می‌کند و هر دو نتیجه را در یک پاسخ ترکیب می‌کند.

### مشاهده جریان مکالمه

رابط چت تاریخچه مکالمات را حفظ می‌کند و به شما امکان می‌دهد تعاملات چند مرحله‌ای داشته باشید. می‌توانید همه پرسش‌ها و پاسخ‌های قبلی را ببینید، که ردیابی مکالمه و درک چگونگی ساخت زمینه توسط عامل در مبادلات متعدد را آسان می‌کند.

<a href="images/tools-conversation-demo.png"><img src="../../../translated_images/fa/tools-conversation-demo.89f2ce9676080f59.webp" alt="مکالمه با فراخوانی چند ابزار" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*مکالمه چند مرحله‌ای که تبدیل‌های ساده، جستجوی وضعیت هوا و زنجیره ابزارها را نشان می‌دهد*

### آزمایش با درخواست‌های مختلف

ترکیب‌های مختلفی را امتحان کنید:
- جستجوی وضعیت هوا: «هوای توکیو چگونه است؟»
- تبدیل‌های دمایی: «25 درجه سلسیوس چند کلوین است؟»
- پرسش‌های ترکیبی: «وضعیت هوا در پاریس را چک کن و بگو آیا بالاتر از 20 درجه سلسیوس است»

توجه کنید که چگونه عامل زبان طبیعی را تفسیر می‌کند و آن را به فراخوانی‌های مناسب ابزار نگاشت می‌کند.

## مفاهیم کلیدی

### الگوی ReAct (استدلال و عمل)

عامل بین استدلال (تصمیم‌گیری درباره انجام چه کاری) و عمل (استفاده از ابزارها) جابجا می‌شود. این الگو امکان حل خودکار مسئله را فراهم می‌کند، نه فقط پاسخ به دستورالعمل‌ها.

### اهمیت توضیحات ابزار

کیفیت توضیحات ابزار شما مستقیماً بر استفاده خوب عامل از آنها تأثیر می‌گذارد. توضیحات واضح و مشخص به مدل کمک می‌کند بفهمد چه زمانی و چگونه هر ابزار را فراخوانی کند.

### مدیریت جلسه

نشان `@MemoryId` امکان مدیریت حافظه مبتنی بر جلسه را به صورت خودکار فراهم می‌کند. به هر شناسه جلسه یک نمونه `ChatMemory` اختصاص می‌یابد که توسط مؤلفه `ChatMemoryProvider` مدیریت می‌شود، بنابراین کاربران مختلف می‌توانند به صورت همزمان با عامل تعامل داشته باشند بدون اینکه مکالماتشان با هم مخلوط شود. دیاگرام زیر نشان می‌دهد که چگونه کاربران مختلف بر اساس شناسه جلسه به حافظه‌های مجزا هدایت می‌شوند:

<img src="../../../translated_images/fa/session-management.91ad819c6c89c400.webp" alt="مدیریت جلسه با @MemoryId" width="800"/>

*هر شناسه جلسه به تاریخچه مکالمه جداگانه‌ای نگاشت می‌شود — کاربران هیچ‌گاه پیام‌های یکدیگر را نمی‌بینند.*

### مدیریت خطا

ابزارها ممکن است خطا دهند — پایان زمان API، پارامترهای نامعتبر، خرابی سرویس‌های خارجی. عوامل تولیدی نیازمند مدیریت خطا هستند تا مدل بتواند مشکلات را توضیح دهد یا جایگزین‌هایی امتحان کند، نه اینکه کل برنامه کرش کند. وقتی یک ابزار استثنا پرتاب می‌کند، LangChain4j آن را گرفته و پیام خطا را به مدل بازمی‌گرداند، که سپس می‌تواند مشکل را به زبان طبیعی توضیح دهد.

## ابزارهای موجود

دیاگرام زیر اکوسیستم گسترده ابزارهایی را نشان می‌دهد که می‌توانید بسازید. این ماژول ابزارهای هواشناسی و دما را نشان می‌دهد، اما الگوی `@Tool` روی هر متد جاوا کار می‌کند — از پرس‌وجوهای پایگاه داده تا پرداخت‌ها.

<img src="../../../translated_images/fa/tool-ecosystem.aad3d74eaa14a44f.webp" alt="اکوسیستم ابزار" width="800"/>

*هر متد جاوا که با @Tool نشانه‌گذاری شود برای هوش مصنوعی قابل استفاده می‌شود — این الگو به پایگاه‌های داده، APIها، ایمیل، عملیات فایل و بیشتر توسعه می‌یابد.*

هر درخواستی به ابزار نیاز ندارد. تصمیم بر این است که آیا هوش مصنوعی باید با سیستم‌های خارجی تعامل داشته باشد یا می‌تواند از دانش خود پاسخ دهد. راهنمای زیر خلاصه می‌کند چه زمانی ابزار ارزش افزوده دارد و چه زمانی غیرضروری است:

## ابزارها در مقابل RAG

ماژول‌های 03 و 04 هر دو قابلیت‌های هوش مصنوعی را افزایش می‌دهند، اما به طور بنیادی متفاوت. RAG به مدل دسترسی به **دانش** از طریق بازیابی اسناد را می‌دهد. ابزارها به مدل این امکان را می‌دهند که **عملکرد** انجام دهد با فراخوانی توابع. دیاگرام زیر این دو رویکرد را کنار هم مقایسه می‌کند — از نحوه عملکرد تا مزایا و معایب:

<img src="../../../translated_images/fa/tools-vs-rag.ad55ce10d7e4da87.webp" alt="مقایسه ابزارها و RAG" width="800"/>

*RAG اطلاعات را از اسناد ثابت بازیابی می‌کند — ابزارها اقدامات را اجرا و داده‌های پویا و زنده را می‌گیرند. بسیاری از سیستم‌های تولیدی هر دو را ترکیب می‌کنند.*

در عمل، بسیاری از سیستم‌های تولیدی هر دو رویکرد را ترکیب می‌کنند: RAG برای پایه‌گذاری پاسخ‌ها در مستندات شما و ابزارها برای دریافت داده‌های زنده یا انجام عملیات.
