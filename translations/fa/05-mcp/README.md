## آنچه خواهید آموخت

شما هوش مصنوعی مکالمه‌ای ساختید، در ایجاد پرامپت تسلط یافتید، پاسخ‌ها را در مدارک مبتنی بر زمینه قرار دادید و عاملانی با ابزار ساختید. اما همه آن ابزارها مخصوص برنامه شما ساخته شده بودند. اگر می‌توانستید به هوش مصنوعی خود دسترسی به یک اکوسیستم استاندارد از ابزارها که هر کسی می‌تواند بسازد و به اشتراک بگذارد بدهید، چه؟ در این ماژول، شما یاد می‌گیرید چگونه این کار را با پروتکل زمینه مدل (MCP) و ماژول عاملی LangChain4j انجام دهید. ابتدا یک خواننده فایل ساده MCP را نشان می‌دهیم و سپس چگونگی ادغام آسان آن را در جریان‌های کاری پیشرفته عاملی با استفاده از الگوی عامل ناظر نمایش می‌دهیم.

## MCP چیست؟

پروتکل زمینه مدل (MCP) دقیقاً همین کار را انجام می‌دهد - یک روش استاندارد برای برنامه‌های هوش مصنوعی جهت کشف و استفاده از ابزارهای خارجی. به جای نوشتن ادغام‌های سفارشی برای هر منبع داده یا سرویس، به سرورهای MCP وصل می‌شوید که قابلیت‌های خود را به صورت یک فرمت یکنواخت ارائه می‌دهند. سپس عامل هوش مصنوعی شما می‌تواند به طور خودکار این ابزارها را کشف و استفاده کند.

نمودار زیر تفاوت را نشان می‌دهد — بدون MCP، هر ادغام نیاز به سیم‌کشی نقطه به نقطه سفارشی دارد؛ با MCP، یک پروتکل واحد برنامه شما را به هر ابزاری وصل می‌کند:

<img src="../../../translated_images/fa/mcp-comparison.9129a881ecf10ff5.webp" alt="MCP Comparison" width="800"/>

*قبل از MCP: ادغام‌های پیچیده نقطه به نقطه. بعد از MCP: یک پروتکل، امکانات بی‌پایان.*

MCP یک مشکل بنیادی در توسعه هوش مصنوعی را حل می‌کند: هر ادغام سفارشی است. می‌خواهید به GitHub دسترسی پیدا کنید؟ کد سفارشی. می‌خواهید فایل بخوانید؟ کد سفارشی. می‌خواهید پایگاه داده‌ای را پرس‌وجو کنید؟ کد سفارشی. و هیچ‌کدام از این ادغام‌ها با برنامه‌های دیگر هوش مصنوعی کار نمی‌کند.

MCP این مشکل را استاندارد می‌کند. یک سرور MCP ابزارها را با توضیحات و ساختارهای واضح ارائه می‌دهد. هر مشتری MCP می‌تواند متصل شود، ابزارهای موجود را کشف کند و از آن‌ها استفاده نماید. یک بار بسازید، همه جا استفاده کنید.

نمودار زیر این معماری را نشان می‌دهد — یک مشتری MCP (برنامه هوش مصنوعی شما) به چند سرور MCP متصل می‌شود که هر کدام مجموعه‌ای از ابزارهای خود را از طریق پروتکل استاندارد ارائه می‌دهند:

<img src="../../../translated_images/fa/mcp-architecture.b3156d787a4ceac9.webp" alt="MCP Architecture" width="800"/>

*معماری پروتکل زمینه مدل - کشف و اجرای ابزارهای استاندارد شده*

## نحوه عملکرد MCP

در زیر کاپوت، MCP از معماری لایه‌ای استفاده می‌کند. برنامه جاوای شما (مشتری MCP) ابزارهای موجود را کشف می‌کند، درخواست‌های JSON-RPC را از طریق لایه حمل و نقل (Stdio یا HTTP) ارسال می‌کند، و سرور MCP عملیات را اجرا کرده و نتایج را بازمی‌گرداند. نمودار زیر هر لایه این پروتکل را توضیح می‌دهد:

<img src="../../../translated_images/fa/mcp-protocol-detail.01204e056f45308b.webp" alt="MCP Protocol Detail" width="800"/>

*نحوه عملکرد MCP در پشت صحنه — مشتریان ابزارها را کشف می‌کنند، پیام‌های JSON-RPC را مبادله می‌کنند، و عملیات را از طریق لایه حمل و نقل اجرا می‌کنند.*

**معماری سرور-مشتری**

MCP از مدل سرور-مشتری استفاده می‌کند. سرورها ابزارها را فراهم می‌کنند - خواندن فایل‌ها، پرس‌وجوی پایگاه داده، فراخوانی APIها. مشتریان (برنامه هوش مصنوعی شما) به سرورها متصل شده و از ابزارهای آن‌ها استفاده می‌کنند.

برای استفاده از MCP با LangChain4j، این وابستگی Maven را اضافه کنید:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```

**کشف ابزار**

وقتی مشتری شما به سرور MCP متصل می‌شود، می‌پرسد "چه ابزارهایی دارید؟" سرور با لیستی از ابزارهای موجود پاسخ می‌دهد که هر یک توضیحات و ساختار پارامتر دارند. سپس عامل هوش مصنوعی شما می‌تواند تصمیم بگیرد کدام ابزارها را بر اساس درخواست کاربر استفاده کند. نمودار زیر این دست بدهی را نشان می‌دهد — مشتری درخواست `tools/list` ارسال می‌کند و سرور ابزارهای موجود خود را با توضیحات و ساختار پارامتر برمی‌گرداند:

<img src="../../../translated_images/fa/tool-discovery.07760a8a301a7832.webp" alt="MCP Tool Discovery" width="800"/>

*هوش مصنوعی ابزارهای موجود را در راه‌اندازی کشف می‌کند — اکنون می‌داند چه قابلیت‌هایی موجود است و می‌تواند تصمیم بگیرد کدام را استفاده کند.*

**مکانیزم‌های حمل و نقل**

MCP از مکانیزم‌های حمل و نقل مختلف پشتیبانی می‌کند. دو گزینه وجود دارد: Stdio (برای ارتباط محلی زیر فرایندها) و HTTP قابل جریان (برای سرورهای از راه دور). این ماژول حمل و نقل Stdio را نشان می‌دهد:

<img src="../../../translated_images/fa/transport-mechanisms.2791ba7ee93cf020.webp" alt="Transport Mechanisms" width="800"/>

*مکانیزم‌های حمل و نقل MCP: HTTP برای سرورهای از راه دور، Stdio برای فرآیندهای محلی*

**Stdio** - [StdioTransportDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java)

برای فرآیندهای محلی. برنامه شما یک سرور را به عنوان یک زیر فرایند راه‌اندازی می‌کند و از طریق ورودی/خروجی استاندارد ارتباط برقرار می‌کند. برای دسترسی به سیستم فایل یا ابزارهای خط فرمان مفید است.

```java
McpTransport stdioTransport = new StdioMcpTransport.Builder()
    .command(List.of(
        npmCmd, "exec",
        "@modelcontextprotocol/server-filesystem@2025.12.18",
        resourcesDir
    ))
    .logEvents(false)
    .build();
```

سرور `@modelcontextprotocol/server-filesystem` ابزارهای زیر را ارائه می‌دهد، همه محدود به دایرکتوری‌هایی که مشخص می‌کنید:

| ابزار | توضیحات |
|------|-------------|
| `read_file` | خواندن محتوای یک فایل |
| `read_multiple_files` | خواندن چند فایل در یک فراخوانی |
| `write_file` | ایجاد یا بازنویسی یک فایل |
| `edit_file` | انجام ویرایش‌های مشخص جایگزینی |
| `list_directory` | فهرست فایل‌ها و دایرکتوری‌ها در یک مسیر |
| `search_files` | جستجوی بازگشتی فایل‌ها مطابق یک الگو |
| `get_file_info` | دریافت متادیتای فایل (اندازه، زمان‌ها، مجوزها) |
| `create_directory` | ایجاد یک دایرکتوری (شامل دایرکتوری‌های والد) |
| `move_file` | جابجایی یا تغییر نام یک فایل یا دایرکتوری |

نمودار زیر نشان می‌دهد که حمل و نقل Stdio چگونه در زمان اجرا کار می‌کند — برنامه جاوای شما سرور MCP را به عنوان یک فرایند فرزند راه‌اندازی کرده و از طریق لوله‌های stdin/stdout ارتباط برقرار می‌کنند، بدون دخالت شبکه یا HTTP:

<img src="../../../translated_images/fa/stdio-transport-flow.45eaff4af2d81db4.webp" alt="Stdio Transport Flow" width="800"/>

*حمل و نقل Stdio در عمل — برنامه شما سرور MCP را به عنوان فرایند فرزند راه‌اندازی کرده و از طریق لوله‌های stdin/stdout ارتباط برقرار می‌کند.*

## ماژول عامل

در حالی که MCP ابزارهای استاندارد شده را فراهم می‌کند، ماژول **عامل** LangChain4j روشی بیانی برای ساخت عامل‌هایی که آن ابزارها را هماهنگ می‌کنند، ارائه می‌دهد. انوتیشن `@Agent` و `AgenticServices` به شما اجازه می‌دهند رفتار عامل را از طریق رابط‌ها به جای کد امری تعریف کنید.

در این ماژول، شما الگوی **عامل ناظر** را کاوش خواهید کرد — یک رویکرد پیشرفته هوش مصنوعی عاملی که یک عامل "ناظر" به صورت پویا تصمیم می‌گیرد کدام زیرعامل‌ها را بر اساس درخواست کاربر فراخواند. ما هر دو مفهوم را با دادن قابلیت‌های دسترسی به فایل مبتنی بر MCP به یکی از زیرعامل‌هایمان ترکیب خواهیم کرد.

برای استفاده از ماژول عامل، این وابستگی Maven را اضافه کنید:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```
> **توجه:** ماژول `langchain4j-agentic` از یک خاصیت نسخه جداگانه (`langchain4j.mcp.version`) استفاده می‌کند زیرا به برنامه زمانی متفاوتی نسبت به کتابخانه‌های اصلی LangChain4j منتشر می‌شود.

> **⚠️ آزمایشی:** ماژول `langchain4j-agentic` **آزمایشی** بوده و ممکن است تغییر کند. راه مطمئن برای ساخت دستیارهای هوش مصنوعی همچنان `langchain4j-core` با ابزارهای سفارشی است (ماژول ۰۴).

### عامل ناظر

الگوی **عامل ناظر** یک شکل **انعطاف‌پذیر** از هوش مصنوعی عاملی است. یک ناظر از مدل زبانی بزرگ (LLM) به صورت خودکار تصمیم می‌گیرد کدام عوامل را بر اساس درخواست کاربر فراخواند. در مثال بعدی، ما دسترسی به فایل با قدرت MCP را با یک عامل LLM ترکیب می‌کنیم تا جریان کاری خواندن فایل → گزارش با نظارت ایجاد کنیم.

در دمو، `FileAgent` یک فایل را با استفاده از ابزارهای سیستم فایل MCP می‌خواند و `ReportAgent` گزارشی ساختاریافته با خلاصه اجرایی (1 جمله)، ۳ نکته کلیدی و توصیه‌ها تولید می‌کند. ناظر این جریان را به طور خودکار هماهنگ می‌کند:

<img src="../../../translated_images/fa/supervisor-agent-pattern.06275a41ae006ac8.webp" alt="Supervisor Agent Pattern" width="800"/>

*ناظر با استفاده از LLM خود تصمیم می‌گیرد کدام عوامل را فراخواند و به چه ترتیبی — نیازی به مسیریابی سخت‌کد شده نیست.*

جریان کاری مشخص برای خط لوله فایل به گزارش ما چنین است:

<img src="../../../translated_images/fa/file-report-workflow.649bb7a896800de9.webp" alt="File to Report Workflow" width="800"/>

*FileAgent فایل را از طریق ابزارهای MCP می‌خواند، سپس ReportAgent محتوای خام را به گزارش ساختاریافته تبدیل می‌کند.*

نمودار دنباله‌ای زیر هماهنگی کامل ناظر را ردیابی می‌کند — از راه‌اندازی سرور MCP، انتخاب خودکار زیرعامل توسط ناظر، تا فراخوانی ابزارها از طریق stdio و گزارش نهایی:

<img src="../../../translated_images/fa/supervisor-agent-sequence.1aa389b3bef99956.webp" alt="Supervisor Agent Sequence Diagram" width="800"/>

*ناظر به طور خودکار FileAgent را فراخوانی می‌کند (که سرور MCP را از طریق stdio برای خواندن فایل فراخوانی می‌کند)، سپس ReportAgent را برای تولید گزارش ساختاریافته فرا می‌خواند — هر عامل خروجی خود را در محدوده عاملی مشترک ذخیره می‌کند.*

هر عامل خروجی خود را در **محدوده عاملی** (حافظه مشترک) ذخیره می‌کند تا عوامل بعدی به نتایج قبلی دسترسی داشته باشند. این نشان می‌دهد چگونه ابزارهای MCP به طور یکپارچه در جریان‌های کاری عاملی ادغام می‌شوند — ناظر نیازی ندارد *چگونه* فایل‌ها خوانده می‌شوند بداند، فقط کافی است بفهمد که `FileAgent` می‌تواند این کار را انجام دهد.

#### نحوه کار عامل ناظر

قبل از ساخت عامل‌ها، شما باید حمل و نقل MCP را به یک مشتری متصل کرده و آن را به عنوان `ToolProvider` بسته‌بندی کنید. اینگونه است که ابزارهای سرور MCP در دسترس عوامل شما قرار می‌گیرند:

```java
// ایجاد یک کلاینت MCP از طریق انتقال
McpClient mcpClient = new DefaultMcpClient.Builder()
        .transport(stdioTransport)
        .build();

// پوشش دادن کلاینت به عنوان یک ToolProvider — این پلی است برای اتصال ابزارهای MCP به LangChain4j
ToolProvider mcpToolProvider = McpToolProvider.builder()
        .mcpClients(List.of(mcpClient))
        .build();
```

اکنون می‌توانید `mcpToolProvider` را به هر عاملی که به ابزارهای MCP نیاز دارد تزریق کنید:

```java
// مرحله ۱: FileAgent با استفاده از ابزارهای MCP فایل‌ها را می‌خواند
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // دارای ابزارهای MCP برای عملیات فایل
        .build();

// مرحله ۲: ReportAgent گزارش‌های ساختاریافته تولید می‌کند
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// سوپروایزر جریان کار فایل → گزارش را هماهنگ می‌کند
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // گزارش نهایی را بازگردانید
        .build();

// سوپروایزر تصمیم می‌گیرد کدام عوامل بر اساس درخواست فراخوانی شوند
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```

#### چگونه FileAgent ابزارهای MCP را در زمان اجرا کشف می‌کند

ممکن است بپرسید: **چگونه `FileAgent` می‌داند چگونه از ابزارهای سیستم فایل npm استفاده کند؟** پاسخ این است که نمی‌داند — بلکه **LLM** این موضوع را در زمان اجرا از طریق ساختارهای ابزارها تشخیص می‌دهد.

رابط `FileAgent` صرفاً یک **تعریف پرامپت** است. هیچ دانشی از قبل‌کد زده شده از `read_file`، `list_directory` یا هیچ ابزار MCP دیگر ندارد. این چیزی است که به طور کامل اتفاق می‌افتد:
1. **راه‌اندازی سرور:** `StdioMcpTransport` بسته ‌npm `@modelcontextprotocol/server-filesystem` را به عنوان یک فرآیند فرزند اجرا می‌کند  
2. **کشف ابزار:** `McpClient` درخواست JSON-RPC `tools/list` را به سرور ارسال می‌کند که با نام ابزارها، توضیحات و طرح‌بندی پارامترها (مثلاً `read_file` — *"خواندن محتوای کامل یک فایل"* — `{ path: string }`) پاسخ می‌دهد  
3. **تزریق طرح:** `McpToolProvider` این طرح‌های کشف شده را بسته‌بندی کرده و در اختیار LangChain4j قرار می‌دهد  
4. **تصمیم LLM:** وقتی `FileAgent.readFile(path)` فراخوانی می‌شود، LangChain4j پیام سیستم، پیام کاربر و **فهرست طرح‌های ابزاری** را به LLM می‌فرستد. LLM توضیحات ابزارها را می‌خواند و تماس با ابزار را تولید می‌کند (مثلاً `read_file(path="/some/file.txt")`)  
5. **اجرا:** LangChain4j تماس ابزار را دریافت می‌کند، آن را از طریق کلاینت MCP به فرآیند فرعی Node.js ارسال می‌کند، نتیجه را می‌گیرد و به LLM بازمی‌گرداند  

این همان مکانیزم [کشف ابزار](../../../05-mcp) است که پیش‌تر شرح داده شد، اما به طور خاص برای جریان کاری عامل به کار رفته است. نشانه‌گذاری‌های `@SystemMessage` و `@UserMessage` رفتار LLM را هدایت می‌کند، در حالی که `ToolProvider` تزریق شده به آن **قابلیت‌ها** را می‌دهد — LLM این دو را در زمان اجرا به هم پیوند می‌دهد.

#### استراتژی‌های پاسخ

وقتی `SupervisorAgent` را پیکربندی می‌کنید، مشخص می‌کنید که چگونه باید پاسخ نهایی به کاربر را پس از انجام وظایف زیرعامل‌ها تشکیل دهد. نمودار زیر سه استراتژی موجود را نشان می‌دهد — LAST خروجی نهایی عامل را مستقیماً برمی‌گرداند، SUMMARY تمام خروجی‌ها را از طریق یک LLM ترکیب می‌کند، و SCORED هر کدام را که نمره بالاتری در برابر درخواست اصلی داشت انتخاب می‌کند:

<img src="../../../translated_images/fa/response-strategies.3d0cea19d096bdf9.webp" alt="Response Strategies" width="800"/>

*سه استراتژی برای تشکیل پاسخ نهایی Supervisor — بسته به این که می‌خواهید خروجی آخرین عامل، خلاصه ترکیبی یا گزینه بهترین امتیاز را دریافت کنید، انتخاب کنید.*  

استراتژی‌های موجود عبارتند از:

| استراتژی | توضیح |  
|----------|-------|  
| **LAST** | سوپروایزر خروجی آخرین زیرعامل یا ابزاری که فراخوانی شده را برمی‌گرداند. این گزینه زمانی مفید است که عامل نهایی در جریان کاری به طور خاص برای تولید پاسخ کامل و نهایی طراحی شده باشد (مثلاً «عامل خلاصه» در یک خط تحقیق). |  
| **SUMMARY** | سوپروایزر از مدل زبان داخلی خود (LLM) برای ترکیب خلاصه‌ای از کل تعامل و همه خروجی‌های زیرعامل استفاده می‌کند و سپس آن خلاصه را به عنوان پاسخ نهایی می‌فرستد. این پاسخ ساده و تجمیع‌شده‌ای برای کاربر فراهم می‌آورد. |  
| **SCORED** | سیستم از یک LLM داخلی برای امتیازدهی هر دو پاسخ LAST و خلاصه تعامل (SUMMARY) در برابر درخواست اصلی کاربر استفاده می‌کند و هر خروجی که امتیاز بالاتری گرفت را بازمی‌گرداند. |

وقتی دموی برنامه را اجرا می‌کنید، گام به گام ساختار کار را می‌بینید که سوپروایزر چگونه چندین عامل را هماهنگ می‌کند. هر بخش شامل موارد زیر است:

```
======================================================================

The Supervisor orchestrates the agents automatically based on the request.
```
  
**سرصفحه** مفهوم جریان کاری را معرفی می‌کند: یک خط لوله متمرکز از خواندن فایل تا تولید گزارش.

```
  ┌─────────────┐      ┌──────────────┐
  │  FileAgent  │ ───▶ │ ReportAgent  │
  │ (MCP tools) │      │  (pure LLM)  │
  └─────────────┘      └──────────────┘

  [FILE]   FileAgent   - Reads files via MCP → stores in 'fileContent'
```

**نمودار جریان کاری** جریان داده بین عامل‌ها را نشان می‌دهد. هر عامل نقش خاصی دارد:  
- **FileAgent** فایل‌ها را با ابزارهای MCP می‌خواند و محتوای خام را در `fileContent` ذخیره می‌کند  
- **ReportAgent** آن محتوا را مصرف می‌کند و گزارشی ساختاریافته در `report` تولید می‌کند  

```

  "Read the file at .../file.txt and generate a report on its contents"
```
  
**درخواست کاربر** کار را نشان می‌دهد. سوپروایزر این را تحلیل می‌کند و تصمیم می‌گیرد FileAgent → ReportAgent فراخوانی شود.

1. **FileAgent** فایل را از طریق MCP می‌خواند و محتوا را ذخیره می‌کند  
2. **ReportAgent** محتوا را دریافت و گزارش ساختاریافته تولید می‌کند

#### توضیح ویژگی‌های ماژول عامل

این مثال چندین ویژگی پیشرفته ماژول عامل را نشان می‌دهد. بیایید نگاهی دقیق‌تر به Agentic Scope و Agent Listeners بیندازیم.

**Agentic Scope** حافظه مشترکی را نشان می‌دهد که عامل‌ها نتایج خود را با `@Agent(outputKey="...")` ذخیره کرده‌اند. این امکان را فراهم می‌کند:  
- عامل‌های بعدی به خروجی عامل‌های قبلی دسترسی داشته باشند  
- سوپروایزر پاسخ نهایی را ترکیب کند  
- شما بررسی کنید که هر عامل چه چیزی تولید کرده است  

نمودار زیر نشان می‌دهد که Agentic Scope چگونه به عنوان حافظه مشترک در جریان فایل به گزارش کار می‌کند — FileAgent خروجی خود را زیر کلید `fileContent` می‌نویسد، ReportAgent آن را می‌خواند و خروجی خود را زیر `report` می‌نویسد:

<img src="../../../translated_images/fa/agentic-scope.95ef488b6c1d02ef.webp" alt="Agentic Scope Shared Memory" width="800"/>

*Agentic Scope به عنوان حافظه مشترک عمل می‌کند — FileAgent `fileContent` می‌نویسد، ReportAgent می‌خواند و `report` می‌نویسد، و کد شما نتیجه نهایی را می‌خواند.*

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // داده‌های خام فایل از FileAgent
String report = scope.readState("report");            // گزارش ساختار یافته از ReportAgent
```
  
**Agent Listeners** امکان نظارت و اشکال‌زدایی اجرای عامل‌ها را فراهم می‌کنند. خروجی مرحله به مرحله‌ای که در دمو می‌بینید از یک AgentListener است که به هر فراخوانی عامل متصل می‌شود:  
- **beforeAgentInvocation** - هنگام انتخاب عامل توسط سوپروایزر فراخوانی می‌شود، به شما اجازه می‌دهد ببینید کدام عامل انتخاب شده و چرا  
- **afterAgentInvocation** - وقتی عامل به پایان می‌رسد، نتیجه آن را نمایش می‌دهد  
- **inheritedBySubagents** - وقتی true باشد، این گوش‌دهنده همه عامل‌ها در سلسله‌مراتب را نظارت می‌کند  

نمودار زیر چرخه کامل عمر Agent Listener را نشان می‌دهد، از جمله چگونگی مدیریت `onError` خرابی‌ها در طول اجرای عامل:

<img src="../../../translated_images/fa/agent-listeners.784bfc403c80ea13.webp" alt="Agent Listeners Lifecycle" width="800"/>

*Agent Listeners به چرخه عمر اجرا متصل می‌شوند — هنگام شروع، پایان یا وقوع خطاهای عامل‌ها را نظارت می‌کنند.*

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // به تمام زیرعامل‌ها منتقل شود
    }
};
```
  
فراتر از الگوی Supervisor، ماژول `langchain4j-agentic` چندین الگوی قدرتمند جریان کاری فراهم می‌کند. نمودار زیر همه پنج الگو را نشان می‌دهد — از خطوط لوله ساده ترتیبی تا جریان‌های تأیید انسان در حلقه:

<img src="../../../translated_images/fa/workflow-patterns.82b2cc5b0c5edb22.webp" alt="Agent Workflow Patterns" width="800"/>

*پنج الگوی جریان کاری برای هماهنگی عامل‌ها — از خطوط لوله ترتیبی ساده تا جریان‌های تأیید انسان در حلقه.*

| الگو | توضیح | مورد استفاده |  
|---------|-------------|----------|  
| **Sequential** | عامل‌ها را به ترتیب اجرا می‌کند، خروجی به بعدی می‌رود | خط لوله‌ها: تحقیق → تحلیل → گزارش |  
| **Parallel** | عامل‌ها را همزمان اجرا می‌کند | وظایف مستقل: هواشناسی + اخبار + بورس |  
| **Loop** | تا برآورده شدن شرایط تکرار می‌کند | ارزیابی کیفیت: اصلاح تا نمره ≥ 0.8 |  
| **Conditional** | بر اساس شرایط مسیر‌دهی می‌کند | دسته‌بندی → ارجاع به عامل متخصص |  
| **Human-in-the-Loop** | نقاط کنترل انسانی اضافه می‌کند | جریان‌های تأیید، مرور محتوا |  

## مفاهیم کلیدی

حالا که MCP و ماژول عامل را در عمل دیده‌اید، بیایید خلاصه کنیم هر روش را کی باید استفاده کرد.

یکی از بزرگ‌ترین مزایای MCP اکوسیستم رو به رشد آن است. نمودار زیر نشان می‌دهد چگونه یک پروتکل جهانی برنامه AI شما را به انواع مختلف سرورهای MCP متصل می‌کند — از دسترسی به فایل‌سیستم و پایگاه داده تا GitHub، ایمیل، وب‌اسکریپینگ و بیشتر:

<img src="../../../translated_images/fa/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="MCP Ecosystem" width="800"/>

*MCP یک اکوسیستم پروتکل جهانی ایجاد می‌کند — هر سرور سازگار با MCP با هر کلاینت سازگار با MCP کار می‌کند و امکان اشتراک ابزار بین برنامه‌ها را فراهم می‌آورد.*

**MCP** وقتی ایده‌آل است که می‌خواهید از اکوسیستم‌های ابزار موجود بهره ببرید، ابزارهایی بسازید که چندین برنامه بتوانند به اشتراک بگذارند، سرویس‌های شخص ثالث را با پروتکل‌های استاندارد یکپارچه کنید، یا پیاده‌سازی ابزارها را بدون تغییر کد عوض کنید.

**ماژول Agentic** وقتی بهترین عملکرد را دارد که بخواهید تعاریف اعلانی عامل با نشانه‌گذاری `@Agent` داشته باشید، نیاز به هماهنگی جریان کاری (ترتیبی، تکرار، موازی) دارید، طراحی عامل مبتنی بر رابط ترجیح دهید به جای کد امری، یا چندین عاملی را که خروجی‌ها را با `outputKey` به اشتراک می‌گذارند ترکیب کنید.

**الگوی Supervisor Agent** زمانی می‌درخشد که جریان کاری از قبل قابل پیش‌بینی نباشد و بخواهید LLM تصمیم بگیرد، وقتی چندین عامل تخصصی دارید که نیاز به هماهنگی پویا دارند، وقتی سیستم‌های مکالمه می‌سازید که به قابلیت‌های مختلف هدایت می‌کنند، یا وقتی انعطاف‌پذیرترین و انطباق‌پذیرترین رفتار عامل را می‌خواهید.

برای کمک به انتخاب بین روش‌های سفارشی `@Tool` از ماژول 04 و ابزارهای MCP از این ماژول، مقایسه زیر نکات کلیدی را برجسته می‌کند — ابزارهای سفارشی اتصال نزدیک و ایمنی کامل نوع را برای منطق خاص برنامه می‌دهند، در حالی که ابزارهای MCP یکپارچه‌سازی استاندارد شده و قابل استفاده مجدد را ارائه می‌دهند:

<img src="../../../translated_images/fa/custom-vs-mcp-tools.c4f9b6b1cb65d8a1.webp" alt="Custom Tools vs MCP Tools" width="800"/>

*زمان استفاده از @Tool سفارشی در مقابل ابزارهای MCP — ابزارهای سفارشی برای منطق خاص برنامه با ایمنی کامل نوع، ابزارهای MCP برای یکپارچه‌سازی استاندارد که در برنامه‌ها کار می‌کند.*
