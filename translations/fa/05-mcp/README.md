## آنچه یاد خواهید گرفت

شما یک هوش مصنوعی مکالمه‌ای ساخته‌اید، مهارت‌های پیش‌نویس را فرا گرفته‌اید، پاسخ‌ها را در اسناد جانمایی کرده‌اید و ایجنت‌هایی با ابزارها ایجاد کرده‌اید. اما همه این ابزارها به‌صورت سفارشی برای برنامه خاص شما ساخته شده‌اند. اگر می‌توانستید به هوش مصنوعی خود دسترسی به اکوسیستم استانداردی از ابزارها بدهید که هر کسی بتواند آنها را ایجاد و به اشتراک بگذارد چه؟ در این ماژول، خواهید آموخت که چگونه دقیقاً این کار را با پروتکل زمینه مدل (MCP) و ماژول ایجنتیک LangChain4j انجام دهید. ابتدا یک خواننده فایل ساده MCP را نمایش می‌دهیم و سپس نشان می‌دهیم که چگونه به‌سادگی در جریان‌های کاری پیشرفته ایجنتیک با استفاده از الگوی ایجنت ناظر ادغام می‌شود.

## MCP چیست؟

پروتکل زمینه مدل (MCP) دقیقاً همین را فراهم می‌کند - روشی استاندارد برای برنامه‌های هوش مصنوعی تا ابزارهای خارجی را کشف و استفاده کنند. به جای نوشتن ادغام‌های سفارشی برای هر منبع داده یا سرویس، به سرورهای MCP متصل می‌شوید که قابلیت‌های خود را به شکلی سازگار ارائه می‌دهند. ایجنت هوش مصنوعی شما می‌تواند این ابزارها را به‌صورت خودکار کشف و استفاده کند.

<img src="../../../translated_images/fa/mcp-comparison.9129a881ecf10ff5.webp" alt="MCP Comparison" width="800"/>

*قبل از MCP: ادغام‌های نقطه به نقطه پیچیده. بعد از MCP: یک پروتکل، امکانات بی‌پایان.*

MCP یک مشکل بنیادی در توسعه هوش مصنوعی را حل می‌کند: هر ادغام سفارشی است. می‌خواهید به گیت‌هاب دسترسی داشته باشید؟ کد سفارشی. می‌خواهید فایل بخوانید؟ کد سفارشی. می‌خواهید پرس‌وجو پایگاه داده انجام دهید؟ کد سفارشی. و هیچ یک از این ادغام‌ها با برنامه‌های هوش مصنوعی دیگر کار نمی‌کنند.

MCP این را استاندارد می‌کند. یک سرور MCP ابزارها را با توضیحات و ساختارهای واضح ارائه می‌دهد. هر کلاینت MCP می‌تواند متصل شود، ابزارهای موجود را کشف کند و از آنها استفاده نماید. یک بار بسازید، همه‌جا استفاده کنید.

<img src="../../../translated_images/fa/mcp-architecture.b3156d787a4ceac9.webp" alt="MCP Architecture" width="800"/>

*معماری پروتکل زمینه مدل - کشف و اجرای استاندارد ابزارها*

## نحوه کار MCP

<img src="../../../translated_images/fa/mcp-protocol-detail.01204e056f45308b.webp" alt="MCP Protocol Detail" width="800"/>

*نحوه عملکرد MCP در پشت صحنه — کلاینت‌ها ابزارها را کشف می‌کنند، پیام‌های JSON-RPC رد و بدل می‌شود و عملیات از طریق یک لایه انتقال انجام می‌شود.*

**معماری سرور-کلاینت**

MCP از مدل سرور-کلاینت استفاده می‌کند. سرورها ابزارها را فراهم می‌کنند - خواندن فایل‌ها، پرس‌وجو پایگاه داده‌ها، فراخوانی APIها. کلاینت‌ها (برنامه هوش مصنوعی شما) به سرورها متصل می‌شوند و از ابزارهای آنها استفاده می‌کنند.

برای استفاده از MCP با LangChain4j، این وابستگی Maven را اضافه کنید:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```

**کشف ابزار**

وقتی کلاینت شما به یک سرور MCP متصل می‌شود، می‌پرسد «چه ابزارهایی داری؟» سرور با فهرستی از ابزارهای در دسترس پاسخ می‌دهد، هر کدام با توضیحات و ساختار پارامترها. ایجنت هوش مصنوعی شما سپس می‌تواند تصمیم بگیرد کدام ابزارها را بر اساس درخواست کاربر استفاده کند.

<img src="../../../translated_images/fa/tool-discovery.07760a8a301a7832.webp" alt="MCP Tool Discovery" width="800"/>

*هوش مصنوعی هنگام شروع ابزارهای موجود را کشف می‌کند — اکنون می‌داند چه قابلیت‌هایی موجود است و می‌تواند تصمیم بگیرد کدام را استفاده کند.*

**مکانیزم‌های انتقال**

MCP از مکانیزم‌های انتقال مختلف پشتیبانی می‌کند. این ماژول انتقال Stdio برای پردازش‌های محلی را نشان می‌دهد:

<img src="../../../translated_images/fa/transport-mechanisms.2791ba7ee93cf020.webp" alt="Transport Mechanisms" width="800"/>

*مکانیزم‌های انتقال MCP: HTTP برای سرورهای راه دور، Stdio برای پردازش‌های محلی*

**Stdio** - [StdioTransportDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java)

برای پردازش‌های محلی. برنامه شما سروری را به عنوان یک فرایند فرزند اجرا می‌کند و از طریق ورودی/خروجی استاندارد با آن ارتباط برقرار می‌کند. برای دسترسی به سیستم فایل یا ابزارهای خط فرمان مناسب است.

```java
    .logEvents(false)
    .build();
```

## ماژول ایجنتیک

در حالی که MCP ابزارهای استاندارد فراهم می‌کند، ماژول **ایجنتیک** LangChain4j روش انعطاف‌پذیری برای ساخت ایجنت‌هایی ارائه می‌دهد که آن ابزارها را هماهنگ می‌کنند. صفت `@Agent` و `AgenticServices` به شما اجازه می‌دهند رفتار ایجنت را از طریق رابط‌ها به جای کد امری تعریف کنید.

در این ماژول، الگوی **ایجنت ناظر** را بررسی خواهید کرد — رویکرد پیشرفته‌ای در هوش مصنوعی ایجنتیک که در آن یک ایجنت «ناظر» به‌صورت پویا تصمیم می‌گیرد بر اساس درخواست کاربر کدام زیرایجنت‌ها فراخوانی شوند. هر دو مفهوم را ترکیب می‌کنیم و به یکی از زیرایجنت‌ها قابلیت دسترسی به فایل با MCP می‌دهیم.

برای استفاده از ماژول ایجنتیک، این وابستگی Maven را اضافه کنید:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```

> **⚠️ آزمایشی:** ماژول `langchain4j-agentic` **آزمایشی** بوده و ممکن است تغییر کند. روش پایدار ساخت دستیار هوش مصنوعی همچنان `langchain4j-core` با ابزارهای سفارشی است (ماژول ۰۴).

### ایجنت ناظر

الگوی **ایجنت ناظر** نوعی هوش مصنوعی ایجنتیک **انعطاف‌پذیر** است. یک ناظر با استفاده از مدل زبان بزرگ (LLM) به طور خودکار تصمیم می‌گیرد کدام ایجنت‌ها بر اساس درخواست کاربر فراخوانی شوند. در مثال بعدی، دسترسی به فایل‌های با قدرت MCP را با ایجنت LLM ترکیب می‌کنیم تا جریان کاری خواندن فایل → گزارش نظارت شود.

در دمو، `FileAgent` فایل را با استفاده از ابزارهای MCP سیستم فایل می‌خواند و `ReportAgent` گزارشی ساختار یافته با خلاصه اجرایی (یک جمله)، ۳ نکته کلیدی و توصیه‌ها تولید می‌کند. ناظر این جریان را به‌طور خودکار هماهنگ می‌کند:

<img src="../../../translated_images/fa/supervisor-agent-pattern.06275a41ae006ac8.webp" alt="Supervisor Agent Pattern" width="800"/>

*ناظر با استفاده از LLM خود تصمیم می‌گیرد کدام ایجنت‌ها و به چه ترتیبی فراخوانی شوند — نیازی به مسیر دهی سخت‌کد شده نیست.*

در اینجا جریان کاری ملموس برای خط لوله فایل به گزارش ما است:

<img src="../../../translated_images/fa/file-report-workflow.649bb7a896800de9.webp" alt="File to Report Workflow" width="800"/>

*FileAgent فایل را از طریق ابزارهای MCP می‌خواند، سپس ReportAgent محتوای خام را به گزارش ساختار یافته تبدیل می‌کند.*

هر ایجنت خروجی خود را در **محدوده ایجنتیک** (حافظه مشترک) ذخیره می‌کند، که اجازه می‌دهد ایجنت‌های بعدی به نتایج قبلی دسترسی داشته باشند. این نشان می‌دهد چگونه ابزارهای MCP به‌راحتی در جریان‌های کاری ایجنتیک ادغام می‌شوند — ناظر نیازی ندارد بداند *چگونه* فایل‌ها خوانده می‌شوند، فقط کافی است بداند `FileAgent` این کار را می‌تواند انجام دهد.

#### نحوه کار ناظر

```java
// مرحله ۱: FileAgent فایل‌ها را با استفاده از ابزارهای MCP می‌خواند
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // دارای ابزارهای MCP برای عملیات فایل
        .build();

// مرحله ۲: ReportAgent گزارش‌های ساختارمند تولید می‌کند
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// Supervisor روند کاری فایل → گزارش را هماهنگ می‌کند
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // گزارش نهایی را بازمی‌گرداند
        .build();

// Supervisor بر اساس درخواست تصمیم می‌گیرد کدام عوامل فراخوانی شوند
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```

#### استراتژی‌های پاسخ

وقتی `SupervisorAgent` را پیکربندی می‌کنید، مشخص می‌کنید که چگونه باید پاسخ نهایی خود را پس از پایان کار زیرایجنت‌ها به کاربر ارائه دهد.

<img src="../../../translated_images/fa/response-strategies.3d0cea19d096bdf9.webp" alt="Response Strategies" width="800"/>

*سه استراتژی برای نحوه شکل‌دهی پاسخ نهایی ناظر — براساس اینکه خروجی آخرین ایجنت را می‌خواهید، یا خلاصه‌ای ترکیبی، یا بهترین گزینه دارای امتیاز انتخاب کنید.*

استراتژی‌های موجود عبارتند از:

| استراتژی | توضیح |
|----------|-------------|
| **LAST** | ناظر خروجی آخرین زیرایجنت یا ابزار فراخوانی شده را بازمی‌گرداند. این زمانی مفید است که ایجنت نهایی در جریان کاری به طور خاص برای تولید پاسخ کامل و نهایی طراحی شده باشد (مثلاً «ایجنت خلاصه» در خط لوله تحقیق). |
| **SUMMARY** | ناظر با استفاده از مدل زبان داخلی خود (LLM) خلاصه‌ای از کل تعامل و همه خروجی‌های زیرایجنت‌ها را ترکیب می‌کند و سپس آن خلاصه را به عنوان پاسخ نهایی ارائه می‌دهد. این پاسخ جمع‌بندی شده و تمیزی به کاربر می‌دهد. |
| **SCORED** | سیستم از یک LLM داخلی برای امتیازدهی به هر دو پاسخ LAST و خلاصه SUMMARY استفاده می‌کند و خروجی با امتیاز بالاتر را بازمی‌گرداند. |

#### درک خروجی

وقتی دمو را اجرا می‌کنید، مرور گام‌به‌گام ساختاریافته‌ای خواهید دید که نشان می‌دهد چگونه ناظر چند ایجنت را هماهنگ می‌کند. هر بخش به چه معناست:

```
======================================================================
  FILE → REPORT WORKFLOW DEMO
======================================================================

This demo shows a clear 2-step workflow: read a file, then generate a report.
The Supervisor orchestrates the agents automatically based on the request.
```

**عنوان** مفهوم جریان کاری را معرفی می‌کند: یک خط لوله متمرکز از خواندن فایل تا تولید گزارش.

```
--- WORKFLOW ---------------------------------------------------------
  ┌─────────────┐      ┌──────────────┐
  │  FileAgent  │ ───▶ │ ReportAgent  │
  │ (MCP tools) │      │  (pure LLM)  │
  └─────────────┘      └──────────────┘
   outputKey:           outputKey:
   'fileContent'        'report'

--- AVAILABLE AGENTS -------------------------------------------------
  [FILE]   FileAgent   - Reads files via MCP → stores in 'fileContent'
  [REPORT] ReportAgent - Generates structured report → stores in 'report'
```

**نمودار جریان کاری** جریان داده بین ایجنت‌ها را نشان می‌دهد. هر ایجنت نقش خاصی دارد:
- **FileAgent** فایل‌ها را با ابزارهای MCP می‌خواند و محتوای خام را در `fileContent` ذخیره می‌کند
- **ReportAgent** آن محتوا را مصرف کرده و گزارشی ساختار یافته در `report` تولید می‌کند

```
--- USER REQUEST -----------------------------------------------------
  "Read the file at .../file.txt and generate a report on its contents"
```

**درخواست کاربر** کار را نمایش می‌دهد. ناظر آن را تجزیه می‌کند و تصمیم می‌گیرد FileAgent → ReportAgent فراخوانی شوند.

```
--- SUPERVISOR ORCHESTRATION -----------------------------------------
  The Supervisor decides which agents to invoke and passes data between them...

  +-- STEP 1: Supervisor chose -> FileAgent (reading file via MCP)
  |
  |   Input: .../file.txt
  |
  |   Result: LangChain4j is an open-source, provider-agnostic Java framework for building LLM...
  +-- [OK] FileAgent (reading file via MCP) completed

  +-- STEP 2: Supervisor chose -> ReportAgent (generating structured report)
  |
  |   Input: LangChain4j is an open-source, provider-agnostic Java framew...
  |
  |   Result: Executive Summary...
  +-- [OK] ReportAgent (generating structured report) completed
```

**هماهنگی ناظر** جریان دو مرحله‌ای را در عمل نشان می‌دهد:
1. **FileAgent** فایل را از طریق MCP می‌خواند و محتوا را ذخیره می‌کند
2. **ReportAgent** محتوا را دریافت و گزارش ساختار یافته تولید می‌کند

ناظر این تصمیمات را **به‌صورت خودکار** بر اساس درخواست کاربر گرفته است.

```
--- FINAL RESPONSE ---------------------------------------------------
Executive Summary
...

Key Points
...

Recommendations
...

--- AGENTIC SCOPE (Data Flow) ----------------------------------------
  Each agent stores its output for downstream agents to consume:
  * fileContent: LangChain4j is an open-source, provider-agnostic Java framework...
  * report: Executive Summary...
```

#### توضیح ویژگی‌های ماژول ایجنتیک

این مثال چند ویژگی پیشرفته ماژول ایجنتیک را نشان می‌دهد. بیایید نگاهی دقیق‌تر به محدوده ایجنتیک و شنوندگان ایجنت داشته باشیم.

**محدوده ایجنتیک** حافظه مشترکی را نشان می‌دهد که ایجنت‌ها نتایج خود را با استفاده از `@Agent(outputKey="...")` در آن ذخیره کرده‌اند. این امکان را می‌دهد:
- ایجنت‌های بعدی به خروجی ایجنت‌های قبلی دسترسی داشته باشند
- ناظر پاسخ نهایی را ترکیب کند
- شما بتوانید ببینید هر ایجنت چه تولید کرده است

<img src="../../../translated_images/fa/agentic-scope.95ef488b6c1d02ef.webp" alt="Agentic Scope Shared Memory" width="800"/>

*محدوده ایجنتیک به عنوان حافظه مشترک عمل می‌کند — FileAgent `fileContent` می‌نویسد، ReportAgent آن را می‌خواند و `report` می‌نویسد، و کد شما نتیجه نهایی را می‌خواند.*

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // داده‌های خام فایل از FileAgent
String report = scope.readState("report");            // گزارش ساخت‌یافته از ReportAgent
```

**شنوندگان ایجنت** امکان نظارت و اشکال‌زدایی اجرای ایجنت‌ها را فراهم می‌کنند. خروجی گام‌به‌گام که در دمو می‌بینید از یک AgentListener است که به هر فراخوانی ایجنت متصل می‌شود:
- **beforeAgentInvocation** - زمانی فراخوانی می‌شود که ناظر یک عامل را انتخاب می‌کند و به شما اجازه می‌دهد ببینید کدام عامل انتخاب شده و چرا  
- **afterAgentInvocation** - زمانی فراخوانی می‌شود که یک عامل کامل می‌شود و نتیجه آن را نشان می‌دهد  
- **inheritedBySubagents** - وقتی true باشد، شنونده همه عوامل در سلسله مراتب را نظارت می‌کند  

<img src="../../../translated_images/fa/agent-listeners.784bfc403c80ea13.webp" alt="چرخه عمر شنوندگان عامل" width="800"/>

*شنوندگان عامل به چرخه اجرای برنامه متصل می‌شوند — نظارت می‌کنند بر زمان شروع، اتمام یا مواجهه با خطاهای عوامل.*

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // انتشار به همه زیرعامل‌ها
    }
};
```
  
فراتر از الگوی Supervisor، ماژول `langchain4j-agentic` چندین الگوی کاری قدرتمند و ویژگی ارائه می‌دهد:

<img src="../../../translated_images/fa/workflow-patterns.82b2cc5b0c5edb22.webp" alt="الگوهای جریان کار عامل" width="800"/>

*پنج الگوی جریان کار برای هماهنگی عوامل — از خطوط لوله ساده متوالی تا جریان‌های کاری با تایید انسان در حلقه.*

| الگو | توضیح | کاربرد |
|---------|-------------|----------|
| **متوالی** | اجرای عوامل به ترتیب، خروجی به عامل بعدی منتقل می‌شود | خطوط لوله: تحقیق → تحلیل → گزارش |
| **موازی** | اجرای عوامل به طور همزمان | کارهای مستقل: هواشناسی + اخبار + بورس |
| **حلقه** | تکرار تا رسیدن به شرط | ارزیابی کیفیت: اصلاح تا امتیاز ≥ ۰.۸ |
| **شرطی** | مسیریابی بر اساس شرایط | دسته‌بندی → هدایت به عامل متخصص |
| **انسان در حلقه** | اضافه کردن نقاط کنترل انسانی | جریان‌های کاری تایید، بازبینی محتوا |

## مفاهیم کلیدی

حالا که MCP و ماژول agentic را در عمل بررسی کرده‌اید، بیایید خلاصه کنیم چه زمانی باید از هر رویکرد استفاده کنید.

<img src="../../../translated_images/fa/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="اکوسیستم MCP" width="800"/>

*MCP یک اکوسیستم پروتکل جهانی ایجاد می‌کند — هر سرور سازگار با MCP با هر مشتری سازگار با MCP کار می‌کند، که امکان اشتراک‌گذاری ابزارها بین برنامه‌ها را فراهم می‌کند.*

**MCP** زمانی ایده‌آل است که بخواهید از اکوسیستم‌های ابزار موجود استفاده کنید، ابزارهایی بسازید که چندین برنامه بتوانند به اشتراک بگذارند، خدمات شخص ثالث را با پروتکل‌های استاندارد یکپارچه کنید، یا پیاده‌سازی ابزارها را بدون تغییر کد تعویض کنید.

**ماژول Agentic** بهترین کاربرد را زمانی دارد که بخواهید تعاریف اعلانی عوامل با حاشیه‌نویسی `@Agent` داشته باشید، نیاز به هماهنگی جریان‌های کاری (متوالی، حلقه، موازی) دارید، طراحی عوامل مبتنی بر رابط را به جای کد امری ترجیح می‌دهید، یا چندین عامل را ترکیب می‌کنید که خروجی‌های خود را از طریق `outputKey` به اشتراک می‌گذارند.

**الگوی عامل Supervisor** زمانی برجسته است که جریان کار از پیش قابل پیش‌بینی نباشد و بخواهید LLM تصمیم‌گیری کند، وقتی چندین عامل تخصصی دارید که نیاز به هماهنگی پویا دارند، هنگام ساخت سیستم‌های مکالمه‌ای که به قابلیت‌های مختلف هدایت می‌شوند، یا زمانی که بخواهید منعطف‌ترین و تطبیق‌پذیرترین رفتار عامل را داشته باشید.

<img src="../../../translated_images/fa/custom-vs-mcp-tools.c4f9b6b1cb65d8a1.webp" alt="ابزارهای سفارشی در مقابل ابزارهای MCP" width="800"/>

*چه زمانی از روش‌های سفارشی @Tool در مقابل ابزارهای MCP استفاده کنید — ابزارهای سفارشی برای منطق خاص برنامه با ایمنی کامل نوع، ابزارهای MCP برای یکپارچه‌سازی‌های استاندارد که در برنامه‌های مختلف کار می‌کنند.*
