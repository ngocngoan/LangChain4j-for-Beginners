## آنچه خواهید آموخت

اگر استارت سریع را انجام داده باشید، دیده‌اید چگونه درخواست ارسال کنید و پاسخ دریافت کنید. این پایه است، اما برنامه‌های واقعی به چیزهای بیشتری نیاز دارند. این ماژول به شما می‌آموزد چگونه یک هوش مصنوعی گفتگو ساز بسازید که زمینه را به یاد می‌آورد و حالت را حفظ می‌کند - تفاوت بین یک دمو یک باره و یک برنامه آماده تولید.

در این راهنما از GPT-5.2 سرویس Azure OpenAI استفاده خواهیم کرد، زیرا قابلیت‌های استدلال پیشرفته آن رفتار الگوهای مختلف را برجسته‌تر می‌کند. وقتی حافظه اضافه کنید، تفاوت به وضوح دیده خواهد شد. این درک اینکه هر جزء چه کمکی به برنامه شما می‌کند را آسان‌تر می‌سازد.

شما یک برنامه خواهید ساخت که هر دو الگو را نشان می‌دهد:

**چت بدون حالت** - هر درخواست مستقل است. مدل حافظه‌ای از پیام‌های قبلی ندارد. این الگوی مورد استفاده در شروع سریع است.

**گفتگوی حالت‌دار** - هر درخواست شامل سابقه گفتگو است. مدل زمینه را در چندین نوبت حفظ می‌کند. این نیاز برنامه‌های تولیدی است.

## پیش‌نیازها

- اشتراک Azure با دسترسی به Azure OpenAI
- جاوا ۲۱، Maven 3.9+
- Azure CLI (https://learn.microsoft.com/en-us/cli/azure/install-azure-cli)
- Azure Developer CLI (azd) (https://learn.microsoft.com/en-us/azure/developer/azure-developer-cli/install-azd)

> **توجه:** جاوا، Maven، Azure CLI و Azure Developer CLI (azd) در دِوکانتینر فراهم‌شده از قبل نصب شده‌اند.

> **توجه:** این ماژول از GPT-5.2 در Azure OpenAI استفاده می‌کند. پیاده‌سازی به صورت خودکار از طریق `azd up` پیکربندی شده است - نام مدل را در کد تغییر ندهید.

## درک مسئله اصلی

مدل‌های زبانی بدون حالت هستند. هر فراخوانی API مستقل است. اگر بگویید "اسم من جان است" و سپس بپرسید "اسم من چیست؟"، مدل نمی‌داند که تازه خود را معرفی کرده‌اید. هر درخواست را مانند اولین گفتگوی شما می‌داند.

این برای پرسش و پاسخ ساده خوب است اما برای برنامه‌های واقعی بی‌فایده است. ربات‌های خدمات مشتری باید آنچه گفتید را به یاد داشته باشند. دستیاران شخصی نیاز به زمینه دارند. هر گفتگو چندمرحله‌ای نیاز به حافظه دارد.

<img src="../../../translated_images/fa/stateless-vs-stateful.cc4a4765e649c41a.webp" alt="گفتگوهای بدون حالت در مقابل حالت‌دار" width="800"/>

*تفاوت بین بدون حالت (فراخوانی‌های مستقل) و حالت‌دار (آگاه به زمینه) در گفتگوها*

## درک توکن‌ها

قبل از ورود به گفتگوها، درک توکن‌ها مهم است - واحدهای پایه متن که مدل‌های زبانی پردازش می‌کنند:

<img src="../../../translated_images/fa/token-explanation.c39760d8ec650181.webp" alt="توضیح توکن" width="800"/>

*مثال از نحوه شکستن متن به توکن‌ها - "I love AI!" تبدیل به ۴ واحد پردازش جداگانه می‌شود*

توکن‌ها روشی هستند که مدل‌های هوش مصنوعی متن را اندازه‌گیری و پردازش می‌کنند. کلمات، نشانه‌گذاری و حتی فاصله‌ها می‌توانند توکن باشند. مدل شما محدودیت تعداد توکن‌های قابل پردازش همزمان دارد (۴۰۰,۰۰۰ برای GPT-5.2، با حداکثر ۲۷۲,۰۰۰ توکن ورودی و ۱۲۸,۰۰۰ توکن خروجی). درک توکن‌ها به مدیریت طول گفتگو و هزینه‌ها کمک می‌کند.

## نحوه کار حافظه

حافظه چت مشکل بدون حالت را با حفظ سابقه گفتگو حل می‌کند. پیش از ارسال درخواست به مدل، فریم‌ورک پیام‌های قبلی مرتبط را اضافه می‌کند. وقتی می‌پرسید "اسم من چیست؟"، سیستم در واقع کل سابقه گفتگو را ارسال می‌کند تا مدل ببیند شما قبلاً گفتید "اسم من جان است."

LangChain4j پیاده‌سازی‌هایی برای حافظه فراهم می‌کند که این کار را به صورت خودکار انجام می‌دهد. شما تعیین می‌کنید چند پیام نگه داشته شود و فریم‌ورک پنجره زمینه را مدیریت می‌کند.

<img src="../../../translated_images/fa/memory-window.bbe67f597eadabb3.webp" alt="مفهوم پنجره حافظه" width="800"/>

*MessageWindowChatMemory پنجره‌ای لغزان از پیام‌های اخیر را حفظ می‌کند و به صورت خودکار پیام‌های قدیمی را حذف می‌کند*

## چگونگی استفاده این ماژول از LangChain4j

این ماژول استارت سریع را با افزودن Spring Boot و افزودن حافظه گفتگو گسترش می‌دهد. قطعات به این شکل کنار هم قرار می‌گیرند:

**وابستگی‌ها** - افزودن دو کتابخانه LangChain4j:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j</artifactId> <!-- Inherited from BOM in root pom.xml -->
</dependency>
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-open-ai-official</artifactId> <!-- Inherited from BOM in root pom.xml -->
</dependency>
```

**مدل چت** - پیکربندی Azure OpenAI به عنوان یک bean اسپرینگ ([LangChainConfig.java](../../../01-introduction/src/main/java/com/example/langchain4j/config/LangChainConfig.java)):

```java
@Bean
public OpenAiOfficialChatModel openAiOfficialChatModel() {
    return OpenAiOfficialChatModel.builder()
            .baseUrl(azureEndpoint)
            .apiKey(azureApiKey)
            .modelName(deploymentName)
            .timeout(Duration.ofMinutes(5))
            .maxRetries(3)
            .build();
}
```

سازنده اعتبارنامه‌ها را از متغیرهای محیطی که توسط `azd up` تنظیم شده‌اند می‌خواند. تنظیم `baseUrl` به نقطه پایان Azure شما باعث می‌شود کلاینت OpenAI با Azure OpenAI کار کند.

**حافظه گفتگو** - پیگیری سابقه گفتگو با MessageWindowChatMemory ([ConversationService.java](../../../01-introduction/src/main/java/com/example/langchain4j/service/ConversationService.java)):

```java
ChatMemory memory = MessageWindowChatMemory.withMaxMessages(10);

memory.add(UserMessage.from("My name is John"));
memory.add(AiMessage.from("Nice to meet you, John!"));

memory.add(UserMessage.from("What's my name?"));
AiMessage aiMessage = chatModel.chat(memory.messages()).aiMessage();
memory.add(aiMessage);
```

حافظه با `withMaxMessages(10)` ساخته شده که آخرین ۱۰ پیام را نگه می‌دارد. پیام‌های کاربر و هوش مصنوعی با نوع‌بندی شده `UserMessage.from(text)` و `AiMessage.from(text)` اضافه می‌شوند. سابقه با `memory.messages()` بازیابی و به مدل ارسال می‌گردد. سرویس شناسه گفتگو را به خاطر می‌سپارد تا چندین کاربر بتوانند همزمان چت کنند.

> **🤖 با چت [GitHub Copilot](https://github.com/features/copilot) امتحان کنید:** فایل [`ConversationService.java`](../../../01-introduction/src/main/java/com/example/langchain4j/service/ConversationService.java) را باز کنید و بپرسید:
> - "چگونه MessageWindowChatMemory تصمیم می‌گیرد وقتی پنجره پر است کدام پیام‌ها حذف شوند؟"
> - "آیا می‌توانم خزنۀ حافظه سفارشی با استفاده از پایگاه داده به جای حافظۀ داخل حافظه پیاده کنم؟"
> - "چگونه می‌توانم خلاصه‌سازی اضافه کنم تا سابقۀ قدیمی گفتگو را فشرده کنم؟"

نقطۀ پایانی چت بدون حالت حافظه را کاملا نادیده می‌گیرد - فقط `chatModel.chat(prompt)` مثل استارت سریع. نقطۀ پایانی حالت‌دار پیام‌ها را به حافظه اضافه می‌کند، سابقه را بازیابی کرده و با هر درخواست آن زمینه را ارسال می‌کند. پیکربندی مدل یکسان است، اما الگوها متفاوت‌اند.

## پیاده‌سازی زیرساخت Azure OpenAI

**بش:**
```bash
cd 01-introduction
azd up  # اشتراک و مکان را انتخاب کنید (eastus2 توصیه می‌شود)
```

**PowerShell:**
```powershell
cd 01-introduction
azd up  # اشتراک و مکان را انتخاب کنید (eastus2 توصیه می‌شود)
```

> **توجه:** اگر با خطای timeout مواجه شدید (`RequestConflict: Cannot modify resource ... provisioning state is not terminal`)، به سادگی دوباره `azd up` را اجرا کنید. منابع Azure ممکن است هنوز در حال پیاده‌سازی باشند و تکرار تلاش اجازه می‌دهد پیاده‌سازی پس از رسیدن منابع به حالت نهایی کامل شود.

این کار موارد زیر را انجام می‌دهد:
۱. پیاده‌سازی منبع Azure OpenAI با مدل‌های GPT-5.2 و text-embedding-3-small
۲. خودکار فایل `.env` را در ریشه پروژه با اعتبارنامه تولید می‌کند
۳. تمام متغیرهای محیطی مورد نیاز را تنظیم می‌کند

**مشکلی در پیاده‌سازی دارید؟** به [README زیرساخت](infra/README.md) مراجعه کنید برای راهنمایی دقیق در مورد تعارض نام دامنه فرعی، مراحل پیاده‌سازی دستی در پورتال Azure و راهنمایی پیکربندی مدل.

**بررسی موفقیت پیاده‌سازی:**

**بش:**
```bash
cat ../.env  # باید AZURE_OPENAI_ENDPOINT، API_KEY و غیره را نمایش دهد.
```

**PowerShell:**
```powershell
Get-Content ..\.env  # باید AZURE_OPENAI_ENDPOINT، API_KEY و غیره را نشان دهد.
```

> **توجه:** فرمان `azd up` به صورت خودکار فایل `.env` را تولید می‌کند. اگر بعداً نیاز به به‌روزرسانی داشتید، می‌توانید یا به صورت دستی فایل `.env` را ویرایش کنید یا آن را دوباره تولید کنید با اجرای:
>
> **بش:**
> ```bash
> cd ..
> bash .azd-env.sh
> ```
>
> **PowerShell:**
> ```powershell
> cd ..
> .\.azd-env.ps1
> ```

## اجرای برنامه به صورت محلی

**بررسی پیاده‌سازی:**

اطمینان حاصل کنید که فایل `.env` در دایرکتوری ریشه با اعتبارنامه Azure موجود است:

**بش:**
```bash
cat ../.env  # باید AZURE_OPENAI_ENDPOINT، API_KEY، DEPLOYMENT را نشان دهد
```

**PowerShell:**
```powershell
Get-Content ..\.env  # باید AZURE_OPENAI_ENDPOINT، API_KEY، DEPLOYMENT را نشان دهد
```

**راه‌اندازی برنامه‌ها:**

**گزینه ۱: استفاده از داشبورد Spring Boot (توصیه شده برای کاربران VS Code)**

دِوکانتینر افزونه داشبورد Spring Boot را شامل می‌شود که رابط گرافیکی برای مدیریت همه برنامه‌های Spring Boot فراهم می‌کند. می‌توانید آن را در Activity Bar سمت چپ VS Code بیابید (آیکون Spring Boot).

از داشبورد Spring Boot می‌توانید:
- همه برنامه‌های Spring Boot موجود در فضای کاری را ببینید
- برنامه‌ها را با یک کلیک شروع/توقف کنید
- لاگ‌های برنامه را به صورت زنده مشاهده کنید
- وضعیت برنامه را نظارت کنید

فقط روی دکمه پخش کنار "introduction" کلیک کنید تا این ماژول شروع شود، یا همه ماژول‌ها را همزمان اجرا کنید.

<img src="../../../translated_images/fa/dashboard.69c7479aef09ff6b.webp" alt="داشبورد Spring Boot" width="400"/>

**گزینه ۲: استفاده از اسکریپت‌های شل**

همه برنامه‌های وب (ماژول‌های ۰۱-۰۴) را اجرا کنید:

**بش:**
```bash
cd ..  # از دایرکتوری ریشه
./start-all.sh
```

**PowerShell:**
```powershell
cd ..  # از دایرکتوری ریشه
.\start-all.ps1
```

یا فقط این ماژول را اجرا کنید:

**بش:**
```bash
cd 01-introduction
./start.sh
```

**PowerShell:**
```powershell
cd 01-introduction
.\start.ps1
```

هر دو اسکریپت به صورت خودکار متغیرهای محیطی را از فایل `.env` ریشه بارگذاری می‌کنند و اگر فایل‌های JAR وجود نداشته باشند آنها را می‌سازند.

> **توجه:** اگر ترجیح می‌دهید همه ماژول‌ها را به صورت دستی قبل از اجرا بسازید:
>
> **بش:**
> ```bash
> cd ..  # Go to root directory
> mvn clean package -DskipTests
> ```
>
> **PowerShell:**
> ```powershell
> cd ..  # Go to root directory
> mvn clean package -DskipTests
> ```

به آدرس http://localhost:8080 در مرورگرتان بروید.

**برای توقف:**

**بش:**
```bash
./stop.sh  # فقط این ماژول
# یا
cd .. && ./stop-all.sh  # همه ماژول‌ها
```

**PowerShell:**
```powershell
.\stop.ps1  # این ماژول فقط
# یا
cd ..; .\stop-all.ps1  # همه ماژول‌ها
```

## استفاده از برنامه

برنامه یک رابط وب فراهم می‌کند که دو پیاده‌سازی چت را به صورت کنار هم نشان می‌دهد.

<img src="../../../translated_images/fa/home-screen.121a03206ab910c0.webp" alt="صفحه اصلی برنامه" width="800"/>

*داشبورد که هر دو گزینه چت ساده (بدون حالت) و چت گفتگو (حالت‌دار) را نمایش می‌دهد*

### چت بدون حالت (پنل سمت چپ)

اول این را امتحان کنید. بپرسید "اسم من جان است" و بلافاصله بعد "اسم من چیست؟" مدل چیزی به یاد نمی‌آورد چون هر پیام مستقل است. این مشکل اصلی یکپارچه‌سازی ساده مدل زبانی است - عدم وجود زمینه گفتگو.

<img src="../../../translated_images/fa/simple-chat-stateless-demo.13aeb3978eab3234.webp" alt="دموی چت بدون حالت" width="800"/>

*هوش مصنوعی اسم شما را از پیام قبلی به یاد ندارد*

### چت حالت‌دار (پنل سمت راست)

اکنون همان توالی را اینجا امتحان کنید. بپرسید "اسم من جان است" و بعد "اسم من چیست؟" این بار به یاد می‌آورد. تفاوت MessageWindowChatMemory است - سابقه گفتگو را نگه می‌دارد و با هر درخواست ارسال می‌کند. این همان نحوه عملکرد هوش مصنوعی گفتگویی در تولید است.

<img src="../../../translated_images/fa/conversational-chat-stateful-demo.e5be9822eb23ff59.webp" alt="دموی چت حالت‌دار" width="800"/>

*هوش مصنوعی اسم شما را از ابتدای گفتگو به یاد دارد*

هر دو پنل از مدل GPT-5.2 یکسان استفاده می‌کنند. تنها تفاوت حافظه است. این به وضوح نشان می‌دهد حافظه چه تأثیری به برنامه شما می‌آورد و چرا برای موارد استفاده واقعی ضروری است.
