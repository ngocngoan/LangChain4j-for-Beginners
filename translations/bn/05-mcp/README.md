## আপনি যা শিখবেন

আপনি কথোপকথনমূলক AI তৈরি করেছেন, প্রম্পট দক্ষতা অর্জন করেছেন, উত্তরকে ডকুমেন্টের সাথে সংযুক্ত করেছেন, এবং টুলসসহ এজেন্ট তৈরি করেছেন। কিন্তু সব টুলস আপনার নির্দিষ্ট অ্যাপ্লিকেশনের জন্য কাস্টম তৈরি। যদি আপনি আপনার AI কে এমন একটি স্ট্যান্ডার্ডাইজড টুল ইকোসিস্টেমে প্রবেশাধিকার দিতে পারেন যা যেকেউ তৈরি ও শেয়ার করতে পারে? এই মডিউলে আপনি Model Context Protocol (MCP) এবং LangChain4j এর এজেন্টিক মডিউল ব্যবহার করে সেটি কিভাবে করবেন তা শিখবেন। প্রথমে একটি সহজ MCP ফাইল রিডার দেখানো হবে এবং তারপর দেখানো হবে কিভাবে এটি সুপারভাইজার এজেন্ট প্যাটার্ন ব্যবহার করে উন্নত এজেন্টিক ওয়ার্কফ্লোতে সহজেই একীভূত হয়।

## MCP কি?

Model Context Protocol (MCP) ঠিক সেটাই প্রদান করে — AI অ্যাপ্লিকেশনের জন্য একটি স্ট্যান্ডার্ড উপায় বাইরের টুলস আবিষ্কার ও ব্যবহার করার। প্রতিটি ডেটা সোর্স বা সার্ভিসের জন্য কাস্টম ইন্টিগ্রেশন লেখার পরিবর্তে, আপনি MCP সার্ভারগুলোর সাথে সংযুক্ত হন যারা তাদের সক্ষমতাকে একটি সুশৃঙ্খল ফরম্যাটে প্রকাশ করে। আপনার AI এজেন্ট তখন সেগুলো স্বয়ংক্রিয়ভাবে আবিষ্কার ও ব্যবহার করতে পারে।

নিচের চিত্রে পার্থক্য দেখানো হয়েছে — MCP না থাকলে প্রতিটি ইন্টিগ্রেশন কাস্টম পয়েন্ট-টু-পয়েন্ট ওয়্যারিং প্রয়োজন; MCP থাকলে একটি প্রোটোকল আপনার অ্যাপকে যেকোনো টুলের সাথে যুক্ত করে:

<img src="../../../translated_images/bn/mcp-comparison.9129a881ecf10ff5.webp" alt="MCP Comparison" width="800"/>

*MCP এর আগে: জটিল পয়েন্ট-টু-পয়েন্ট ইন্টিগ্রেশন। MCP এর পরে: এক প্রোটোকল, অসংখ্য সম্ভাবনা।*

MCP একটি মৌলিক সমস্যার সমাধান করে AI ডেভেলপমেন্টে: প্রতিটি ইন্টিগ্রেশন কাস্টম। GitHub অ্যাক্সেস করতে চান? কাস্টম কোড। ফাইল পড়তে চান? কাস্টম কোড। ডাটাবেস অনুসন্ধান করতে চান? কাস্টম কোড। আর এগুলো অন্য AI অ্যাপ্লিকেশনের সাথে কাজ করে না।

MCP এটাকে স্ট্যান্ডার্ডাইজ করে। একটি MCP সার্ভার পরিষ্কার বিবরণ ও স্কিমাসহ টুলস প্রকাশ করে। যেকোন MCP ক্লায়েন্ট সংযুক্ত হয়ে উপলব্ধ টুলস আবিষ্কার ও ব্যবহার করতে পারে। একবার তৈরি করুন, সর্বত্র ব্যবহার করুন।

নিচের চিত্রটি এই স্থাপত্য প্রদর্শন করে — একটি MCP ক্লায়েন্ট (আপনার AI অ্যাপ্লিকেশন) একাধিক MCP সার্ভারের সাথে সংযুক্ত, প্রত্যেকে তাদের নিজস্ব টুলস স্ট্যান্ডার্ড প্রোটোকলের মাধ্যমে প্রকাশ করছে:

*Model Context Protocol স্থাপত্য - স্ট্যান্ডার্ড টুল আবিষ্কার ও কার্যকরকরণ*

## MCP কিভাবে কাজ করে

অন্তর্গতভাবে, MCP একটি স্তরভিত্তিক স্থাপত্য ব্যবহার করে। আপনার Java অ্যাপ্লিকেশন (MCP ক্লায়েন্ট) উপলব্ধ টুলস আবিষ্কার করে, JSON-RPC অনুরোধ প্রেরণ করে ট্রান্সপোর্ট স্তর (Stdio বা HTTP) মাধ্যমে, এবং MCP সার্ভার অপারেশন সম্পাদন করে ফলাফল ফেরত পাঠায়। নিচের চিত্রটি এই প্রোটোকলের প্রতিটি স্তর ভাঙ্গায়:

<img src="../../../translated_images/bn/mcp-protocol-detail.01204e056f45308b.webp" alt="MCP Protocol Detail" width="800"/>

*MCP অন্তর্গত কাজ — ক্লায়েন্ট টুলস আবিষ্কার করে, JSON-RPC মেসেজ বিনিময় করে, এবং ট্রান্সপোর্ট স্তরের মাধ্যমে অপারেশন সম্পাদন করে।*

**সার্ভার-ক্লায়েন্ট স্থাপত্য**

MCP একটি ক্লায়েন্ট-সার্ভার মডেল ব্যবহার করে। সার্ভার টুলস প্রদান করে - ফাইল পড়া, ডাটাবেস অনুসন্ধান, API কল ইত্যাদি। ক্লায়েন্ট (আপনার AI অ্যাপ্লিকেশন) সার্ভারের সাথে সংযুক্ত হয়ে তাদের টুলস ব্যবহার করে।

LangChain4j এর সাথে MCP ব্যবহার করতে, নিচের Maven নির্ভরতাটি যোগ করুন:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```

**টুল আবিষ্কার**

ক্লায়েন্ট যখন MCP সার্ভারের সাথে সংযুক্ত হয়, তখন হয় "আপনার কাছে কোন কোন টুলস আছে?" সার্ভার উপলব্ধ টুলসের তালিকা দেয়, প্রতিটির সাথে বিবরণ এবং প্যারামিটার স্কিমাস। আপনার AI এজেন্ট ব্যবহারকারীর অনুরোধ দেখে কোন টুলস ব্যবহার করা হবে তা নির্ধারণ করে। নিচের চিত্রে এই সংলাপ দেখানো হয়েছে — ক্লায়েন্ট একটি `tools/list` অনুরোধ পাঠায় এবং সার্ভার তার উপলব্ধ টুলস বিবরণসহ পাঠায়:

<img src="../../../translated_images/bn/tool-discovery.07760a8a301a7832.webp" alt="MCP Tool Discovery" width="800"/>

*AI স্টার্টআপে উপলব্ধ টুলস আবিষ্কার করে — এখন জানে কোন সক্ষমতাসমূহ আছে এবং কোনগুলো ব্যবহার করা হবে তা সিদ্ধান্ত নিতে পারে।*

**ট্রান্সপোর্ট প্রক্রিয়া**

MCP বিভিন্ন ট্রান্সপোর্ট প্রক্রিয়া সমর্থন করে। দুটো অপশন হলো Stdio (লোকাল সাবপ্রসেস যোগাযোগের জন্য) এবং Streamable HTTP (রিমোট সার্ভারের জন্য)। এই মডিউলে Stdio ট্রান্সপোর্ট প্রদর্শন করা হয়েছে:

<img src="../../../translated_images/bn/transport-mechanisms.2791ba7ee93cf020.webp" alt="Transport Mechanisms" width="800"/>

*MCP ট্রান্সপোর্ট প্রক্রিয়া: রিমোট সার্ভারের জন্য HTTP, লোকাল প্রসেসের জন্য Stdio*

**Stdio** - [StdioTransportDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java)

লোকাল প্রসেসের জন্য। আপনার অ্যাপ্লিকেশন সাবপ্রসেস হিসেবে একটি সার্ভার স্পন করে এবং স্ট্যান্ডার্ড ইনপুট/আউটপুটের মাধ্যমে যোগাযোগ করে। ফাইলসিস্টেম অ্যাক্সেস বা কমান্ড লাইন টুলসে সহায়ক।

```java
McpTransport stdioTransport = new StdioMcpTransport.Builder()
    .command(List.of(
        npmCmd, "exec",
        "@modelcontextprotocol/server-filesystem@2025.12.18",
        resourcesDir
    ))
    .logEvents(false)
    .build();
```

`@modelcontextprotocol/server-filesystem` সার্ভার নিম্নলিখিত টুলস প্রকাশ করে, সবগুলি আপনি নির্দিষ্ট ডিরেক্টরিতে সীমাবদ্ধ:

| টুল | বর্ণনা |
|------|-------------|
| `read_file` | একটি ফাইলের বিষয়বস্তু পড়া |
| `read_multiple_files` | একবারে একাধিক ফাইল পড়া |
| `write_file` | একটি ফাইল তৈরি বা ওভাররাইট করা |
| `edit_file` | টার্গেটেড ফাইন্ড-এবং-রিপ্লেস সম্পাদনা করা |
| `list_directory` | একটি পাথে ফাইল ও ডিরেক্টরি তালিকা দেখানো |
| `search_files` | প্যাটার্ন মেলে এমন ফাইল রিকার্সিভলি খোঁজা |
| `get_file_info` | ফাইলের মেটাডেটা (সাইজ, টাইমস্ট্যাম্প, পারমিশন) পাওয়া |
| `create_directory` | একটি ডিরেক্টরি তৈরি (প্যারেন্ট ডিরেক্টরি সহ) |
| `move_file` | একটি ফাইল বা ডিরেক্টরি স্থানান্তর বা নাম পরিবর্তন |

নিচের চিত্রটি Stdio ট্রান্সপোর্ট রানটাইমে কিভাবে কাজ করে দেখায় — আপনার Java অ্যাপ MCP সার্ভারকে চাইল্ড প্রসেস হিসেবে স্পন করে এবং তারা stdin/stdout পাইপের মাধ্যমে যোগাযোগ করে, নেটওয়ার্ক বা HTTP জড়িত নয়:

<img src="../../../translated_images/bn/stdio-transport-flow.45eaff4af2d81db4.webp" alt="Stdio Transport Flow" width="800"/>

*Stdio ট্রান্সপোর্ট কার্যক্রম — আপনার অ্যাপ MCP সার্ভারকে চাইল্ড প্রসেস হিসেবে স্পন করে এবং stdin/stdout পাইপ দিয়ে যোগাযোগ করে।*

## এজেন্টিক মডিউল

যদিও MCP স্ট্যান্ডার্ডাইজড টুলস প্রদান করে, LangChain4j এর **agentic module** ডিক্লারেটিভ ভঙ্গিতে এজেন্ট তৈরি করার উপায় দেয় যারা ঐ টুলসগুলোকে ওয়র্কফ্লো হিসেবে পরিচালনা করে। `@Agent` অ্যানোটেশন এবং `AgenticServices` এর মাধ্যমে আপনি ইন্টারফেস ব্যবহার করে এজেন্ট আচরণ নির্ধারণ করতে পারেন, ইম্পেরেটিভ কোডের বদলে।

এই মডিউলে আপনি **Supervisor Agent** প্যাটার্ন অন্বেষণ করবেন — একটি উন্নত এজেন্টিক AI পদ্ধতি যেখানে "সুপারভাইজার" এজেন্ট ব্যবহারকারীর অনুরোধের ভিত্তিতে সাব-এজেন্ট কে ডাকতে হবে তা গতিশীলভাবে নির্ধারণ করে। আমরা দুইটি ধারণা একত্র করব MCP-ক্ষমতাযুক্ত ফাইল অ্যাক্সেস একটি সাব-এজেন্টকে প্রদান করে।

agentic মডিউল ব্যবহার করতে, নিম্ন Maven নির্ভরতাটি যোগ করুন:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```
> **দ্রষ্টব্য:** `langchain4j-agentic` মডিউলটি একটি আলাদা সংস্করণ প্রপার্টি (`langchain4j.mcp.version`) ব্যবহার করে কারণ এটি মূল LangChain4j লাইব্রেরি থেকে আলাদা সময়সূচীতে মুক্তি পায়।

> **⚠️ পরীক্ষামূলক:** `langchain4j-agentic` মডিউল **পরীক্ষামূলক** এবং পরিবর্তনের সম্মুখীন হতে পারে। AI অ্যাসিস্ট্যান্ট তৈরি করার স্থিতিশীল উপায় হলো `langchain4j-core` কাস্টম টুলস সহ (Module 04)।

### সুপারভাইজার এজেন্ট

**সুপারভাইজার এজেন্ট প্যাটার্ন** একটি **নমনীয়** এজেন্টিক AI ফর্ম। একটি সুপারভাইজার ব্যবহারকারী অনুরোধের ভিত্তিতে কোন সাব-এজেন্টদের ডাকতে হবে তা স্বয়ংক্রিয়ভাবে LLM ব্যবহার করে নির্ধারণ করে। পরবর্তী উদাহরণে, আমরা MCP-ক্ষমতাযুক্ত ফাইল অ্যাক্সেস এবং LLM এজেন্টকে মিলিয়ে একটি নিয়ন্ত্রিত ফাইল পড়া → রিপোর্ট ওয়ার্কফ্লো তৈরি করব।

ডেমোতে, `FileAgent` MCP ফাইলসিস্টেম টুল ব্যবহার করে একটি ফাইল পড়ে, এবং `ReportAgent` একটি কাঠামোবদ্ধ রিপোর্ট তৈরি করে যার মধ্যে রয়েছে এক বাক্যের নির্বাহী সারাংশ, ৩টি মূল বিষয় এবং সুপারিশগুলো। সুপারভাইজার স্বয়ংক্রিয়ভাবে এ ওয়ার্কফ্লো পরিচালনা করে:

<img src="../../../translated_images/bn/supervisor-agent-pattern.06275a41ae006ac8.webp" alt="Supervisor Agent Pattern" width="800"/>

*সুপারভাইজার তার LLM ব্যবহার করে নির্ধারণ করে কোন এজেন্টগুলো কখন ডাকা হবে — কোন হার্ডকোডেড রাউটিং প্রয়োজন নেই।*

আমাদের ফাইল থেকে রিপোর্ট পাইপলাইনের স্পষ্ট ওয়ার্কফ্লো এরকম দেখায়:

<img src="../../../translated_images/bn/file-report-workflow.649bb7a896800de9.webp" alt="File to Report Workflow" width="800"/>

*FileAgent MCP টুল ব্যবহার করে ফাইল পড়ে, তারপর ReportAgent কাঁচা তথ্যকে কাঠামোবদ্ধ রিপোর্টে রূপান্তর করে।*

প্রত্যেক এজেন্ট তার আউটপুট **Agentic Scope** এ (শেয়ার্ড মেমোরি) সংরক্ষণ করে, যাতে পরবর্তী এজেন্টরা পূর্বের ফলাফল অ্যাক্সেস করতে পারে। এটি দেখায় MCP টুলস এজেন্টিক ওয়ার্কফ্লোতে নির্বিঘ্নে কিভাবে একীভূত হয় — সুপারভাইজারকে জানার দরকার নেই *কিভাবে* ফাইল পড়া হয়, শুধু জানে `FileAgent` এটা করতে পারে।

#### সুপারভাইজার কিভাবে কাজ করে

এজেন্ট তৈরির আগে, আপনাকে MCP ট্রান্সপোর্টকে একটি ক্লায়েন্টে সংযুক্ত করতে হবে এবং সেটিকে `ToolProvider` হিসেবে র‍্যাপ করতে হবে। এভাবে MCP সার্ভারের টুলস আপনার এজেন্টদের জন্য উপলব্ধ হয়:

```java
// ট্রান্সপোর্ট থেকে একটি MCP ক্লায়েন্ট তৈরি করুন
McpClient mcpClient = new DefaultMcpClient.Builder()
        .transport(stdioTransport)
        .build();

// ক্লায়েন্টটিকে একটি টুলপ্রোভাইডার হিসেবে মোড়া — এটি MCP টুলকে LangChain4j-তে সংযোগ করে
ToolProvider mcpToolProvider = McpToolProvider.builder()
        .mcpClients(List.of(mcpClient))
        .build();
```

এখন যেকোন এজেন্টে `mcpToolProvider` ইনজেক্ট করতে পারেন, যাদের MCP টুলস প্রয়োজন:

```java
// ধাপ ১: ফাইলএজেন্ট MCP সরঞ্জাম ব্যবহার করে ফাইল পড়ে
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // ফাইল অপারেশনের জন্য MCP সরঞ্জাম আছে
        .build();

// ধাপ ২: রিপোর্টএজেন্ট কাঠামোবদ্ধ রিপোর্ট তৈরি করে
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// সুপারভাইজার ফাইল → রিপোর্ট কর্মপ্রবাহ সমন্বয় করে
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // চূড়ান্ত রিপোর্ট ফেরত দিন
        .build();

// সুপারভাইজার অনুরোধের ভিত্তিতে কোন এজেন্টকে আহ্বান করতে হবে তা নির্ধারণ করে
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```

#### প্রতিক্রিয়া কৌশলসমূহ

যখন আপনি একটি `SupervisorAgent` কনফিগার করেন, আপনি নির্দিষ্ট করেন কিভাবে সুপারভাইজার সাব-এজেন্টদের কাজ শেষ হওয়ার পর ব্যবহারকারীর কাছে চূড়ান্ত উত্তর উপস্থাপন করবে। নিচের চিত্রে তিনটি উপলব্ধ কৌশল দেখানো হয়েছে — LAST সরাসরি শেষ এজেন্টের আউটপুট দেয়, SUMMARY সব আউটপুট LLM দিয়ে সংশ্লেষণ করে, আর SCORED মূল অনুরোধের বিরুদ্ধে যারা ভালো স্কোর পায় সেটি বেছে নেয়:

<img src="../../../translated_images/bn/response-strategies.3d0cea19d096bdf9.webp" alt="Response Strategies" width="800"/>

*কিভাবে সুপারভাইজার তার চূড়ান্ত প্রতিক্রিয়া তৈরি করে তার জন্য তিনটি কৌশল — আপনি চাইলে শেষ এজেন্টের আউটপুট, একটি সংশ্লেষিত সারাংশ, বা সেরা স্কোর প্রাপ্ত অপশন বেছে নিতে পারেন।*

উপলব্ধ কৌশলসমূহ:

| কৌশল | বর্ণনা |
|----------|-------------|
| **LAST** | সুপারভাইজার শেষ সাব-এজেন্ট বা টুলের আউটপুট সরাসরি প্রদান করে। এটি কাজে লাগে যখন ওয়ার্কফ্লোর শেষ এজেন্ট পুরো চূড়ান্ত উত্তর তৈরি করে (যেমন গবেষণা পাইপলাইনে একটি "সারাংশ এজেন্ট")। |
| **SUMMARY** | সুপারভাইজার নিজের অভ্যন্তরীণ ভাষা মডেল (LLM) ব্যবহার করে পুরো ইন্টারঅ্যাকশন এবং সব সাব-এজেন্ট আউটপুটের সারাংশ তৈরি করে, তারপর তা চূড়ান্ত প্রতিক্রিয়া হিসেবে ফেরত দেয়। এটি ব্যবহারকারীকে পরিষ্কার ও সম্মিলিত উত্তর দেয়। |
| **SCORED** | সিস্টেম অভ্যন্তরীণ LLM ব্যবহার করে LAST প্রতিক্রিয়া এবং SUMMARY উভয়ের স্কোর নির্ণয় করে, এবং যেটি মূল ব্যবহারকারীর অনুরোধের সাথে বেশি মানানসই তাকে ফেরত দেয়। |

```
This demo shows a clear 2-step workflow: read a file, then generate a report.
The Supervisor orchestrates the agents automatically based on the request.
```

**শিরোনাম** ওয়ার্কফ্লো ধারণা পরিচয় করায়: একটি ফোকাসড পাইপলাইন ফাইল পড়া থেকে রিপোর্ট তৈরির দিকে।

```
--- WORKFLOW ---------------------------------------------------------
  ┌─────────────┐      ┌──────────────┐
  │  FileAgent  │ ───▶ │ ReportAgent  │
  │ (MCP tools) │      │  (pure LLM)  │
  └─────────────┘      └──────────────┘
   outputKey:           outputKey:
   'fileContent'        'report'

--- AVAILABLE AGENTS -------------------------------------------------
  [FILE]   FileAgent   - Reads files via MCP → stores in 'fileContent'
  [REPORT] ReportAgent - Generates structured report → stores in 'report'
```

**ওয়ার্কফ্লো ডায়াগ্রাম** এজেন্টদের মধ্যে ডেটা প্রবাহ দেখায়। প্রতিটি এজেন্টের একটি নির্দিষ্ট ভূমিকা আছে:
- **FileAgent** MCP টুল ব্যবহার করে ফাইল পড়ে এবং কাঁচা বিষয়বস্তু `fileContent` এ সংরক্ষণ করে
- **ReportAgent** ঐ বিষয়বস্তু গ্রহণ করে এবং `report`-এ একটি কাঠামোবদ্ধ রিপোর্ট তৈরি করে

```
--- USER REQUEST -----------------------------------------------------
  "Read the file at .../file.txt and generate a report on its contents"
```

**ব্যবহারকারীর অনুরোধ** কাজটি দেখায়। Supervisor এটিকে বিশ্লেষণ করে FileAgent → ReportAgent আহ্বান করার সিদ্ধান্ত নেয়।

```

  |

  +-- STEP 2: Supervisor chose -> ReportAgent (generating structured report)
  |
  |   Input: LangChain4j is an open-source, provider-agnostic Java framew...
  |
  |   Result: Executive Summary...
  +-- [OK] ReportAgent (generating structured report) completed
```

**Supervisor সংগঠন** ২-ধাপে প্রবাহ প্রদর্শন করে:
1. **FileAgent** MCP এর মাধ্যমে ফাইল পড়ে এবং বিষয়বস্তু সংরক্ষণ করে
2. **ReportAgent** বিষয়বস্তু পেয়ে কাঠামোবদ্ধ রিপোর্ট তৈরি করে

Supervisor এই সিদ্ধান্তগুলি **স্বয়ংক্রিয়ভাবে** ব্যবহারকারীর অনুরোধের ভিত্তিতে গ্রহণ করেছে।

#### Agentic মডিউল ফিচারসমূহের ব্যাখ্যা

উদাহরণটি Agentic মডিউলের কিছু উন্নত ফিচার প্রদর্শন করে। Agentic Scope এবং Agent Listeners সম্পর্কে নিবিড়ভাবে দেখি।

**Agentic Scope** দেখায় শেয়ার্ড মেমোরি যেখানে এজেন্টরা তাদের ফলাফল `@Agent(outputKey="...")` ব্যবহার করে সংরক্ষণ করে। এটি সম্ভব করে:
- পরবর্তী এজেন্টগুলো পূর্ববর্তী এজেন্টের আউটপুট অ্যাক্সেস করতে পারে
- Supervisor চূড়ান্ত প্রতিক্রিয়া তৈরি করতে পারে
- আপনি প্রতিটি এজেন্ট কী তৈরি করেছে দেখার সুযোগ পাবেন

নীচের ডায়াগ্রাম Agentic Scope কিভাবে ফাইল থেকে রিপোর্ট ওয়ার্কফ্লোতে শেয়ার্ড মেমরির মতো কাজ করে তা দেখায় — FileAgent তার আউটপুট `fileContent` কী-র অধীনে লিখে, ReportAgent সেটি পড়ে এবং তার নিজস্ব আউটপুট `report` কী-র অধীনে লেখে:

<img src="../../../translated_images/bn/agentic-scope.95ef488b6c1d02ef.webp" alt="Agentic Scope Shared Memory" width="800"/>

*Agentic Scope শেয়ার্ড মেমোরির মতো কাজ করে — FileAgent `fileContent` লেখে, ReportAgent পড়ে এবং `report` লেখে, এবং আপনার কোড চূড়ান্ত ফলাফল পড়ে।*

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // ফাইলএজেন্ট থেকে র ম ফাইল ডেটা
String report = scope.readState("report");            // রিপোর্টএজেন্ট থেকে গঠনমূলক রিপোর্ট
```

**Agent Listeners** এজেন্ট কার্যকরী তদারকি এবং ডিবাগিং সক্ষম করে। ডেমোতে ধাপে ধাপে যে আউটপুট দেখছেন তা আসে একটি AgentListener থেকে যা প্রতিটি এজেন্ট আহ্বানে যুক্ত হয়:
- **beforeAgentInvocation** - Supervisor যখন একটি এজেন্ট নির্বাচন করে তখন কল হয়, আপনি দেখতে পারেন কোন এজেন্ট নির্বাচন হয়েছে এবং কেন
- **afterAgentInvocation** - একটি এজেন্ট সম্পন্ন হলে কল হয়, তার ফলাফল দেখায়
- **inheritedBySubagents** - যখন true, তখন লিসেনার সম্পূর্ণ হায়ারারকির সব এজেন্টকে মনিটর করে

নীচের ডায়াগ্রাম পুরো Agent Listener লাইফসাইকেল দেখায়, যার মধ্যে `onError` কীভাবে এজেন্ট এক্সিকিউশনে সমস্যা ঘটলে তা পরিচালনা করে:

<img src="../../../translated_images/bn/agent-listeners.784bfc403c80ea13.webp" alt="Agent Listeners Lifecycle" width="800"/>

*Agent Listeners এক্সিকিউশন লাইফসাইকেলে হুক করে — দেখতে পারেন কখন এজেন্ট শুরু হচ্ছে, সম্পন্ন হচ্ছে, বা ত্রুটি হচ্ছে।*

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // সমস্ত সাব-এজেন্টদের কাছে প্রসারিত করুন
    }
};
```

Supervisor প্যাটার্নের বাইরে, `langchain4j-agentic` মডিউল কয়েকটি শক্তিশালী ওয়ার্কফ্লো প্যাটার্ন সরবরাহ করে। নীচের ডায়াগ্রাম পাঁচটি প্যাটার্ন দেখায় — সহজ সিকোয়েন্সিয়াল পাইপলাইন থেকে মানব-ইন-দ্য-লুপ অনুমোদন ওয়ার্কফ্লোগুলোর পর্যায়ক্রম:

<img src="../../../translated_images/bn/workflow-patterns.82b2cc5b0c5edb22.webp" alt="Agent Workflow Patterns" width="800"/>

*এজেন্ট সংগঠনের জন্য পাঁচটি ওয়ার্কফ্লো প্যাটার্ন — সহজ সিকোয়েন্সিয়াল পাইপলাইন থেকে মানব-ইন-দ্য-লুপ অনুমোদন ওয়ার্কফ্লো।*

| প্যাটার্ন | বর্ণনা | ব্যবহারের ঘটনা |
|---------|-------------|----------|
| **Sequential** | এজেন্টগুলো ধারাবাহিকভাবে চালান, আউটপুট পরবর্তী এজেন্টে প্রবাহিত হয় | পাইপলাইন: গবেষণা → বিশ্লেষণ → রিপোর্ট |
| **Parallel** | এজেন্টগুলো একসঙ্গে চালান | স্বতন্ত্র কাজ: আবহাওয়া + সংবাদ + স্টক |
| **Loop** | শর্ত পূরণ না হওয়া পর্যন্ত পুনরাবৃত্তি | মান নির্ধারণ: স্কোর ≥ 0.8 হওয়া পর্যন্ত |
| **Conditional** | শর্ত অনুসারে রুট নির্ধারণ | শ্রেণীবিন্যাস → বিশেষজ্ঞ এজেন্টের কাছে রুট |
| **Human-in-the-Loop** | মানব চেকপয়েন্ট যুক্ত করুন | অনুমোদন ওয়ার্কফ্লো, বিষয়বস্তু পর্যালোচনা |

## মূল ধারণাসমূহ

এখন আপনি MCP এবং agentic মডিউল বাস্তবে অনুসন্ধান করেছেন, চলুন সারাংশ করি কখন কোন পদ্ধতি ব্যবহার করবেন।

MCP এর সবচেয়ে বড় সুবিধাগুলোর মধ্যে একটি হলো এর বৃহত্তর ইকোসিস্টেম। নীচের ডায়াগ্রাম দেখায় একটি সার্বজনীন প্রোটোকল কীভাবে আপনার AI অ্যাপ্লিকেশনকে বিস্তর MCP সার্ভারের সাথে সংযুক্ত করে — ফাইল সিস্টেম ও ডাটাবেস থেকে শুরু করে GitHub, ইমেইল, ওয়েব স্ক্র্যাপিং ইত্যাদি পর্যন্ত:

<img src="../../../translated_images/bn/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="MCP Ecosystem" width="800"/>

*MCP একটি সার্বজনীন প্রোটোকল ইকোসিস্টেম তৈরি করে — যেকোন MCP-সঙ্গত সার্ভার যেকোন MCP-সঙ্গত ক্লায়েন্টের সাথে কাজ করে, অ্যাপ্লিকেশনগুলোর মধ্যে সরঞ্জাম ভাগাভাগির সুযোগ সৃষ্টি করে।*

**MCP** সবচেয়ে ভালো যখন আপনি বিদ্যমান টুল ইকোসিস্টেম ব্যবহার করতে চান, এমন সরঞ্জাম বানাতে চান যেগুলো অনেক অ্যাপ্লিকেশন শেয়ার করতে পারে, তৃতীয়-পক্ষের সার্ভিস স্ট্যান্ডার্ড প্রোটোকল দিয়ে সংহত করতে চান, অথবা সরঞ্জামের বাস্তবায়ন বদল করতে চান কোড না পরিবর্তন করে।

**Agentic মডিউল** সবচেয়ে ভাল কাজ করে যখন আপনি ঘোষণামূলক এজেন্ট সংজ্ঞা চান `@Agent` এনোটেশন সহ, ওয়ার্কফ্লো সংগঠন (সিকোয়েন্সিয়াল, লুপ, প্যারালাল) দরকার, কমান্ড ভিত্তিক কোডের বদলে ইন্টারফেস-ভিত্তিক এজেন্ট ডিজাইন পছন্দ করেন, অথবা একাধিক এজেন্টের আউটপুট শেয়ার করার জন্য `outputKey` ব্যবহার করছেন।

**Supervisor Agent প্যাটার্ন** বিশেষ করে উজ্জ্বল যখন ওয়ার্কফ্লো পূর্বানুমেয় নয় এবং আপনি চান LLM সিদ্ধান্ত নিক, যখন আপনার কাছে একাধিক বিশেষায়িত এজেন্ট আছে যাদের গতিশীল সংগঠন দরকার, কথোপকথন সিস্টেম তৈরি করছেন যা বিভিন্ন ক্ষমতায় রুট করে, অথবা সবচেয়ে নমনীয়, অভিযোজিত এজেন্ট আচরণ আপনি পেতে চান।

মডিউল ০৪ থেকে কাস্টম `@Tool` মেথড এবং এই মডিউল থেকে MCP টুলের মধ্যে সিদ্ধান্ত নিতে সাহায্য করার জন্য নিচের তুলনায় মূল পার্থক্যগুলো তুলে ধরা হয়েছে — কাস্টম টুলগুলো অ্যাপ-নির্দিষ্ট লজিকের জন্য শক্ত সংযুক্তি ও পূর্ণ টাইপ সেফটি দেয়, আর MCP টুলগুলো মানানসই, পুনঃব্যবহারযোগ্য সংহতকরণ অফার করে:

<img src="../../../translated_images/bn/custom-vs-mcp-tools.c4f9b6b1cb65d8a1.webp" alt="Custom Tools vs MCP Tools" width="800"/>

*কখন কাস্টম @Tool মেথড ব্যবহার করবেন vs MCP টুল — কাস্টম টুল অ্যাপ-নির্দিষ্ট লজিকের জন্য পূর্ণ টাইপ সেফটির সাথে, MCP টুল স্ট্যান্ডার্ডাইজড সংহতকরণের জন্য যা বিভিন্ন অ্যাপ্লিকেশনে কাজ করে।*
