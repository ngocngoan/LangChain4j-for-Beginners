## আপনি যা শিখবেন

আপনি কথোপকথনমূলক AI তৈরি করেছেন, প্রম্পট দক্ষতা অর্জন করেছেন, ডকুমেন্টের ওপর ভিত্তি করে উত্তর তৈরি করেছেন, এবং টুলসহ এজেন্ট তৈরি করেছেন। কিন্তু ওই সব টুলগুলো আপনার নির্দিষ্ট অ্যাপ্লিকেশনের জন্য কাস্টম তৈরি। যদি আপনি আপনার AI-কে এমন একটি মানসম্মত টুল ইকোসিস্টেমে প্রবেশাধিকার দিতে পারেন যা কেউই তৈরি ও শেয়ার করতে পারে? এই মডিউলটিতে আপনি এরকমটাই শিখবেন মডেল কনটেক্সট প্রটোকল (MCP) এবং LangChain4j এর এজেন্টিক মডিউল ব্যবহার করে। প্রথমে আমরা একটি সহজ MCP ফাইল রিডার প্রদর্শন করবো এবং তারপর দেখাবো কিভাবে এটি সুপারভাইজার এজেন্ট প্যাটার্ন ব্যবহার করে উন্নত এজেন্টিক ওয়ার্কফ্লো-তে সহজে ইন্টিগ্রেট করা যায়।

## MCP কী?

মডেল কনটেক্সট প্রটোকল (MCP) ঠিক এমনটাই প্রদান করে—AI অ্যাপ্লিকেশনগুলোকে বাইরের টুল আবিষ্কার ও ব্যবহার করার একটি স্ট্যান্ডার্ড পদ্ধতি। প্রতিটি ডেটা সোর্স বা সার্ভিসের জন্য কাস্টম ইন্টিগ্রেশন লেখার বদলে, আপনি MCP সার্ভারে সংযোগ স্থাপন করেন যা তাদের ক্ষমতাগুলো একটি সঙ্গতিপূর্ণ ফর্ম্যাটে প্রকাশ করে। আপনার AI এজেন্ট তখন সেগুলো স্বয়ংক্রিয়ভাবে আবিষ্কার ও ব্যবহার করতে পারে।

নীচের চিত্রটি পার্থক্য দেখায় — MCP ছাড়া প্রতিটি ইন্টিগ্রেশন কাস্টম পয়েন্ট-টু-পয়েন্ট ওয়্যারিং লাগে; MCP-র মাধ্যমে একটি প্রটোকল আপনার অ্যাপকে যেকোন টুলের সাথে সংযুক্ত করে:

<img src="../../../translated_images/bn/mcp-comparison.9129a881ecf10ff5.webp" alt="MCP Comparison" width="800"/>

*MCP এর আগে: জটিল পয়েন্ট-টু-পয়েন্ট ইন্টিগ্রেশন। MCP পরে: একটি প্রটোকল, অসংখ্য সম্ভাবনা।*

MCP AI উন্নয়নের একটি মৌলিক সমস্যা সমাধান করে: প্রতিটি ইন্টিগ্রেশন কাস্টম। GitHub অ্যাক্সেস করতে চান? কাস্টম কোড। ফাইল পড়তে চান? কাস্টম কোড। ডাটাবেজ কুয়েরি করতে চান? কাস্টম কোড। আর এই ইন্টিগ্রেশনগুলো অন্য AI অ্যাপ্লিকেশনের সাথে কাজ করে না।

MCP এটিকে স্ট্যান্ডার্ডাইজ করে। একটি MCP সার্ভার টুলগুলো স্পষ্ট বর্ণনা ও স্কিমাসসহ প্রকাশ করে। যেকোন MCP ক্লায়েন্ট সংযোগ করে উপলব্ধ টুল আবিষ্কার ও ব্যবহার করতে পারে। একবার তৈরি করুন, সর্বত্র ব্যবহার করুন।

নীচের চিত্রটি এই আর্কিটেকচার প্রদর্শন করে—একটি MCP ক্লায়েন্ট (আপনার AI অ্যাপ্লিকেশন) একাধিক MCP সার্ভারের সাথে সংযোগ করে, প্রত্যেকটি তাদের নিজস্ব সেট টুল স্ট্যান্ডার্ড প্রটোকলের মাধ্যমে প্রকাশ করে:

<img src="../../../translated_images/bn/mcp-architecture.b3156d787a4ceac9.webp" alt="MCP Architecture" width="800"/>

*মডেল কনটেক্সট প্রটোকল আর্কিটেকচার — মানসম্মত টুল আবিষ্কার ও এক্সিকিউশন*

## MCP কিভাবে কাজ করে

আন্তর্জালে, MCP একটি স্তরিত আর্কিটেকচার ব্যবহার করে। আপনার Java অ্যাপ্লিকেশন (MCP ক্লায়েন্ট) উপলব্ধ টুল আবিষ্কার করে, JSON-RPC রিকোয়েস্ট ট্রান্সপোর্ট লেয়ার (Stdio বা HTTP) মাধ্যমে পাঠায়, এবং MCP সার্ভার অপারেশন সম্পাদন করে ফলাফল ফেরত দেয়। নিম্নলিখিত চিত্রটি এই প্রটোকলের প্রতিটি স্তর ভাঙিয়ে দেখায়:

<img src="../../../translated_images/bn/mcp-protocol-detail.01204e056f45308b.webp" alt="MCP Protocol Detail" width="800"/>

*MCP কিভাবে কাজ করে ভিতর থেকে—ক্লায়েন্ট টুল আবিষ্কার করে, JSON-RPC বার্তা বিনিময় করে, এবং ট্রান্সপোর্ট লেয়ারের মাধ্যমে অপারেশন সম্পাদন করে।*

**সার্ভার-ক্লায়েন্ট আর্কিটেকচার**

MCP একটি ক্লায়েন্ট-সার্ভার মডেল ব্যবহার করে। সার্ভার টুল সরবরাহ করে - ফাইল পড়া, ডাটাবেজ কুয়েরি, API কল। ক্লায়েন্ট (আপনার AI অ্যাপ্লিকেশন) সার্ভারগুলোর সাথে সংযোগ স্থাপন করে তাদের টুল ব্যবহার করে।

LangChain4j এর সাথে MCP ব্যবহার করতে, এই Maven নির্ভরতা যোগ করুন:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```
  
**টুল আবিষ্কার**

আপনার ক্লায়েন্ট যখন MCP সার্ভারের সাথে সংযোগ স্থাপন করে, তখন প্রশ্ন করে "আপনার কোল টুল আছে?" সার্ভার তার উপলব্ধ টুলের তালিকা ফেরত দেয়, প্রত্যেকটির বর্ণনা ও প্যারামিটার স্কিমাসসহ। আপনার AI এজেন্ট ব্যবহারকারীর অনুরোধ অনুসারে কোন টুল ব্যবহার করবে তা নির্ধারণ করে। নীচের চিত্রে এই হ্যান্ডশেক দেখানো হয়েছে—ক্লায়েন্ট `tools/list` রিকোয়েস্ট পাঠায় এবং সার্ভার তার উপলব্ধ টুলগুলি বর্ণনা ও স্কিমাসসহ ফেরত দেয়:

<img src="../../../translated_images/bn/tool-discovery.07760a8a301a7832.webp" alt="MCP Tool Discovery" width="800"/>

*AI স্টার্টআপে উপলব্ধ টুল আবিষ্কার করে — এখন এটি জানে কোন ক্ষমতাগুলো উপলব্ধ এবং কোনগুলো ব্যবহার করবে তা সিদ্ধান্ত নিতে পারে।*

**ট্রান্সপোর্ট পদ্ধতি**

MCP বিভিন্ন ট্রান্সপোর্ট পদ্ধতি সমর্থন করে। দুটি অপশন হল Stdio (লোকাল সাবপ্রসেস কমিউনিকেশন জন্য) এবং স্ট্রীমেবল HTTP (রিমোট সার্ভারের জন্য)। এই মডিউল Stdio ট্রান্সপোর্ট প্রদর্শন করে:

<img src="../../../translated_images/bn/transport-mechanisms.2791ba7ee93cf020.webp" alt="Transport Mechanisms" width="800"/>

*MCP ট্রান্সপোর্ট পদ্ধতি: রিমোট সার্ভারের জন্য HTTP, লোকাল প্রসেসের জন্য Stdio*

**Stdio** - [StdioTransportDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java)

লোকাল প্রসেসের জন্য। আপনার অ্যাপ্লিকেশন একটি সাবপ্রসেস হিসেবে একটি সার্ভার স্পন করে এবং স্ট্যান্ডার্ড ইনপুট/আউটপুটের মাধ্যমে যোগাযোগ করে। ফাইলসিস্টেম অ্যাক্সেস বা কমান্ড-লাইন টুলের জন্য উপযোগী।

```java
McpTransport stdioTransport = new StdioMcpTransport.Builder()
    .command(List.of(
        npmCmd, "exec",
        "@modelcontextprotocol/server-filesystem@2025.12.18",
        resourcesDir
    ))
    .logEvents(false)
    .build();
```
  
`@modelcontextprotocol/server-filesystem` সার্ভার নিম্নলিখিত টুলগুলো প্রকাশ করে, সবগুলি আপনি নির্ধারিত ডিরেক্টরি সীমিত:

| টুল | বর্ণনা |
|------|--------------|
| `read_file` | একটি ফাইলের বিষয়বস্তু পড়া |
| `read_multiple_files` | একবারে একাধিক ফাইল পড়া |
| `write_file` | ফাইল তৈরি বা ওভাররাইট করা |
| `edit_file` | নির্দিষ্ট ফাইন্ড-অ্যান্ড-রিপ্লেস সম্পাদনা করা |
| `list_directory` | একটি পাথে ফাইল ও ডিরেক্টরি তালিকা দেয়া |
| `search_files` | প্যাটার্ন মিলানোর জন্য রিকার্সিভলি ফাইল অনুসন্ধান |
| `get_file_info` | ফাইল মেটাডেটা (সাইজ, টাইমস্ট্যাম্প, পারমিশন) পাওয়া |
| `create_directory` | ডিরেক্টরি তৈরি (প্যারেন্ট ডিরেক্টরি সহ) |
| `move_file` | ফাইল বা ডিরেক্টরি সরানো বা নাম পরিবর্তন করা |

নীচের চিত্রটি দেখায় কিভাবে Stdio ট্রান্সপোর্ট রানটাইমে কাজ করে—আপনার Java অ্যাপ MCP সার্ভারকে একটি চাইল্ড প্রসেস হিসেবে স্পন করে এবং তারা stdin/stdout পাইপের মাধ্যমে যোগাযোগ করে, কোনো নেটওয়ার্ক বা HTTP জড়িত নয়:

<img src="../../../translated_images/bn/stdio-transport-flow.45eaff4af2d81db4.webp" alt="Stdio Transport Flow" width="800"/>

*Stdio ট্রান্সপোর্ট কর্মরত—আপনার অ্যাপ সার্ভারকে চাইল্ড প্রসেস হিসেবে স্পন করে এবং stdin/stdout পাইপের মাধ্যমে যোগাযোগ করে।*

## এজেন্টিক মডিউল

যেখানে MCP স্ট্যান্ডার্ডাইজড টুল সরবরাহ করে, LangChain4j এর **এজেন্টিক মডিউল** এ টুলগুলোকে ব্যবস্থাপনা করার জন্য ঘোষণামূলক পদ্ধতি দেয়। `@Agent` অ্যানোটেশন এবং `AgenticServices` দিয়ে আপনি ইন্টারফেসের মাধ্যমে এজেন্ট আচরণ সংজ্ঞায়িত করতে পারেন, বাধ্যতামূলক কোডের বদলে।

এই মডিউলটিতে আপনি **সুপারভাইজার এজেন্ট** প্যাটার্ন অন্বেষণ করবেন—একটি উন্নত এজেন্টিক AI পদ্ধতি যেখানে "সুপারভাইজার" এজেন্ট ব্যবহারকারীর অনুরোধের ভিত্তিতে ডাইনামিকভাবে কোন সাবএজেন্ট চালানো হবে তা নির্ধারণ করে। আমরা এই দুই ধারণাকে মিলিয়ে একটি সাবএজেন্টকে MCP-চালিত ফাইল অ্যাক্সেস ক্ষমতা দেবো।

এজেন্টিক মডিউল ব্যবহার করতে, এই Maven নির্ভরতা যোগ করুন:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```
  
> **নোট:** `langchain4j-agentic` মডিউল একটি পৃথক সংস্করণ প্রপার্টি (`langchain4j.mcp.version`) ব্যবহার করে কারণ এটি মূল LangChain4j লাইব্রেরির তুলনায় ভিন্ন সূচি অনুযায়ী প্রকাশিত হয়।

> **⚠️ পরীক্ষামূলক:** `langchain4j-agentic` মডিউল **পরীক্ষামূলক** এবং পরিবর্তনের মধ্যে রয়েছে। AI সহকারী তৈরির স্থিতিশীল উপায় এখনও `langchain4j-core` কাস্টম টুলসহ (মডিউল ০৪)।

### সুপারভাইজার এজেন্ট

**সুপারভাইজার এজেন্ট প্যাটার্ন** একটি **নমনীয়** এজেন্টিক AI রূপ। একটি সুপারভাইজার LLM ব্যবহার করে স্বায়ত্তশাসিতভাবে নির্ধারণ করে কোন এজেন্টকে চালানো হবে ব্যবহারকারীর অনুরোধ অনুসারে। পরবর্তী উদাহরণে, আমরা MCP-চালিত ফাইল অ্যাক্সেসকে LLM এজেন্টের সাথে মিলিয়ে একটি নিয়ন্ত্রিত ফাইল পড়া → রিপোর্ট তৈরি ওয়ার্কফ্লো তৈরি করব।

ডেমোতে, `FileAgent` MCP ফাইলসিস্টেম টুল ব্যবহার করে একটি ফাইল পড়ে, এবং `ReportAgent` একটি রচিত রিপোর্ট তৈরি করে যার মধ্যে একটি নির্বাহী সারাংশ (১ বাক্য), ৩টি মূল পয়েন্ট, এবং সুপারিশ থাকে। সুপারভাইজার স্বয়ংক্রিয়ভাবে এই প্রবাহ পরিচালনা করে:

<img src="../../../translated_images/bn/supervisor-agent-pattern.06275a41ae006ac8.webp" alt="Supervisor Agent Pattern" width="800"/>

*সুপারভাইজার তার LLM ব্যবহার করে কোন এজেন্টকে চালাতে হবে এবং কোন ক্রমে হবে তা ঠিক করে—কোন হার্ডকোডেড রাউটিং দরকার নেই।*

ফাইল-টু-রিপোর্ট পাইপলাইনের সুনির্দিষ্ট ওয়ার্কফ্লো এই রকম দেখায়:

<img src="../../../translated_images/bn/file-report-workflow.649bb7a896800de9.webp" alt="File to Report Workflow" width="800"/>

*FileAgent MCP টুল ব্যবহার করে ফাইল পড়ে, তারপর ReportAgent কাঁচা বিষয়বস্তুকে একটি রচিত রিপোর্টে রুপান্তর করে।*

নিম্নলিখিত সিকোয়েন্স ডায়াগ্রাম পুরো সুপারভাইজার শাসনকালের পথে অনুসরণ করে—MCP সার্ভার স্পন থেকে শুরু করে, সুপারভাইজারের স্বায়ত্তশাসিত এজেন্ট নির্বাচন, স্টেডিওর মাধ্যমে টুল কল এবং চূড়ান্ত রিপোর্ট:

<img src="../../../translated_images/bn/supervisor-agent-sequence.1aa389b3bef99956.webp" alt="Supervisor Agent Sequence Diagram" width="800"/>

*সুপারভাইজার স্বয়ংক্রিয়ভাবে FileAgent ডেকে MCP সার্ভারের সাথে stdin/stdout ব্যবহার করে ফাইল পড়ে, তারপর ReportAgent ডেকে একটি গঠিত রিপোর্ট তৈরি করে—প্রতিটি এজেন্ট তার আউটপুট শেয়ার্ড Agentic Scope-এ সংরক্ষণ করে।*

প্রত্যেক এজেন্ট তার আউটপুট **Agentic Scope** (শেয়ার্ড মেমরি) তে সংরক্ষণ করে, ফলে পরবর্তী এজেন্টগুলি পূর্বের ফলাফল অ্যাক্সেস করতে পারে। এটি দেখায় কিভাবে MCP টুল সহজে এজেন্টিক ওয়ার্কফ্লোতে ইন্টিগ্রেট হয়—সুপারভাইজারকে জানার দরকার নেই *কিভাবে* ফাইল পড়া হয়, শুধু জানতে হয় `FileAgent` পারে।

#### সুপারভাইজার কিভাবে কাজ করে

এজেন্ট তৈরি করার আগে, আপনাকে MCP ট্রান্সপোর্টকে একজন ক্লায়েন্টের সাথে সংযুক্ত করতে হবে এবং এটিকে `ToolProvider` হিসেবে মোড়ানো দরকার। MCP সার্ভারের টুলগুলি এভাবেই আপনার এজেন্টদের উপলব্ধ হয়:

```java
// ট্রান্সপোর্ট থেকে একটি MCP ক্লায়েন্ট তৈরি করুন
McpClient mcpClient = new DefaultMcpClient.Builder()
        .transport(stdioTransport)
        .build();

// ক্লায়েন্টটিকে একটি ToolProvider হিসাবে মোড়া করুন — এটি MCP টুলগুলি LangChain4j তে সংযুক্ত করে
ToolProvider mcpToolProvider = McpToolProvider.builder()
        .mcpClients(List.of(mcpClient))
        .build();
```
  
এখন আপনি `mcpToolProvider` যেকোন এজেন্টের মধ্যে ইনজেক্ট করতে পারেন যাদের MCP টুল দরকার:

```java
// ধাপ ১: FileAgent MCP সরঞ্জাম ব্যবহার করে ফাইল পড়ে
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // ফাইল অপারেশনগুলির জন্য MCP সরঞ্জাম রয়েছে
        .build();

// ধাপ ২: ReportAgent কাঠামোবদ্ধ রিপোর্ট তৈরি করে
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// Supervisor ফাইল → রিপোর্ট কার্যপ্রবাহ পরিচালনা করে
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // চূড়ান্ত রিপোর্ট ফেরত দিন
        .build();

// Supervisor অনুরোধের উপর ভিত্তি করে কোন এজেন্টগুলো আহ্বান করতে হবে তা সিদ্ধান্ত নেয়
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```
  
#### FileAgent কিভাবে রানটাইমে MCP টুল আবিষ্কার করে

আপনি হয়তো ভাবছেন: **FileAgent কিভাবে npm ফাইলসিস্টেম টুলগুলি ব্যবহার করতে জানে?** উত্তর হলো জানে না—**LLM** রানটাইমে টুল স্কিমা দেখে নিজে সিদ্ধান্ত নেয়।

`FileAgent` ইন্টারফেসটি শুধু একটি **প্রম্পট সংজ্ঞা**। এতে `read_file`, `list_directory`, বা অন্যান্য MCP টুলের কোনো হার্ডকোডেড জ্ঞান নেই। এন্ড-টু-এন্ড কি ঘটে তা হলো:
1. **সার্ভার স্পন:** `StdioMcpTransport` `@modelcontextprotocol/server-filesystem` npm প্যাকেজকে একটি চাইল্ড প্রসেস হিসেবে লঞ্চ করে  
2. **টুল আবিষ্কার:** `McpClient` সার্ভারকে একটি `tools/list` JSON-RPC রিকোয়েস্ট পাঠায়, যা টুলের নাম, বর্ণনা এবং প্যারামিটার স্কিমা (যেমন, `read_file` — *"ফাইলের সম্পূর্ণ বিষয়বস্তু পড়ুন"* — `{ path: string }`) ফেরত দেয়  
3. **স্কিমা ইনজেকশন:** `McpToolProvider` আবিষ্কৃত এই স্কিমাগুলো র‍্যাপ করে LangChain4j-কে প্রদান করে  
4. **LLM সিদ্ধান্ত নেয়:** যখন `FileAgent.readFile(path)` কল করা হয়, LangChain4j সিস্টেম মেসেজ, ব্যবহারকারী মেসেজ, **এবং টুল স্কিমাগুলোর লিস্ট** LLM-এ পাঠায়। LLM টুল বর্ণনাগুলো পড়ে একটি টুল কল তৈরি করে (যেমন, `read_file(path="/some/file.txt")`)  
5. **বাস্তবায়ন:** LangChain4j টুল কল ইন্টারসেপ্ট করে, MCP ক্লায়েন্টের মাধ্যমে Nod.js সাবপ্রসেসে রাউট করে, ফলাফল নিয়ে আসে এবং এটি LLM-এ ফেরত দেয়  

এটি উপরে বর্ণিত একই [টুল আবিষ্কার](../../../05-mcp) প্রক্রিয়া, তবে এজেন্ট ওয়ার্কফ্লো-তে বিশেষভাবে প্রয়োগ করা হয়েছে। `@SystemMessage` এবং `@UserMessage` এনোটেশনগুলো LLM-এর আচরণ নির্দেশ করে, আর ইনজেক্ট করা `ToolProvider` এর মাধ্যমে এটি **ক্ষমতাসমূহ** পায় — LLM রানটাইমে উভয়টিকে সংযুক্ত করে।

> - "টুল স্কিমাগুলো কীভাবে LLM-এ পাঠানো হয়?"  

#### প্রতিক্রিয়া কৌশলসমূহ

যখন আপনি একটি `SupervisorAgent` কনফিগার করেন, তখন আপনি স্পষ্ট করেন এটি সাব-এজেন্টরা কাজ শেষ করার পর কিভাবে ব্যবহারকারীকে চূড়ান্ত উত্তর প্রদান করবে। নিচের চিত্রে তিনটি উপলব্ধ কৌশল দেখানো হয়েছে — LAST সরাসরি শেষ এজেন্টের আউটপুট প্রত্যাবর্তন করে, SUMMARY সমস্ত আউটপুট LLM-এর মাধ্যমে সন্নিবেশ করে, আর SCORED যেটার স্কোর বেশি সেটি বেছে নেয়:

<img src="../../../translated_images/bn/response-strategies.3d0cea19d096bdf9.webp" alt="Response Strategies" width="800"/>

*সুপারভাইজার কিভাবে তার চূড়ান্ত প্রতিক্রিয়া তৈরি করে তার তিনটি কৌশল — শেষ এজেন্টের আউটপুট, সম্পৃক্ত সারাংশ, অথবা সর্বোচ্চ স্কোরপ্রাপ্ত অপশন থেকে নির্বাচন করুন।*

উপলব্ধ কৌশলগুলো:

| কৌশল | বিবরণ |
|----------|-------------|
| **LAST** | সুপারভাইজার শেষ সাব-এজেন্ট বা টুলের আউটপুট ফেরত দেয়। এটি কাজে লাগে যখন ওয়ার্কফ্লোর সর্বশেষ এজেন্ট সম্পূর্ণ, চূড়ান্ত উত্তর দেওয়ার জন্য ডিজাইন করা হয় (যেমন, গবেষণা পাইপলাইনের "Summary Agent")। |
| **SUMMARY** | সুপারভাইজার নিজের অভ্যন্তরীণ ল্যাঙ্গুয়েজ মডেল (LLM) ব্যবহার করে পুরো ইন্টারঅ্যাকশন এবং সকল সাব-এজেন্ট আউটপুটের একটি সারাংশ তৈরি করে এবং সেটি চূড়ান্ত উত্তর হিসেবে প্রদান করে। এটি ব্যবহারকারীকে পরিষ্কার, একত্রিত উত্তর দেয়। |
| **SCORED** | সিস্টেম অভ্যন্তরীণ LLM ব্যবহার করে LAST প্রতিক্রিয়া এবং SUMMARY উভয়কেই মূল ব্যবহারকারী অনুরোধের সাথে স্কোর করে, এবং উচ্চ স্কোরপ্রাপ্ত আউটপুট ফেরত দেয়। |

#### আউটপুট বোঝা

ডেমো চালানোর সময় আপনি দেখতে পাবেন সুপারভাইজার কীভাবে একাধিক এজেন্টকে সমন্বয় করে তার একটি কাঠামোবদ্ধ পরিক্রমা। প্রতিটি অংশের অর্থ এখানে দেওয়া হলো:

```

  FILE → REPORT WORKFLOW DEMO
======================================================================

This demo shows a clear 2-step workflow: read a file, then generate a report.
The Supervisor orchestrates the agents automatically based on the request.
```
  
**শিরোনাম** ওয়ার্কফ্লো ধারণাটি পরিচয় করায়: ফাইল পড়া থেকে রিপোর্ট তৈরির একটি লক্ষ্যভিত্তিক পাইপলাইন।

```
--- WORKFLOW ---------------------------------------------------------
  ┌─────────────┐      ┌──────────────┐
  │  FileAgent  │ ───▶ │ ReportAgent  │
  │ (MCP tools) │      │  (pure LLM)  │
  └─────────────┘      └──────────────┘
   outputKey:           outputKey:
   'fileContent'        'report'

--- AVAILABLE AGENTS -------------------------------------------------
  [FILE]   FileAgent   - Reads files via MCP → stores in 'fileContent'
  [REPORT] ReportAgent - Generates structured report → stores in 'report'

**ওয়ার্কফ্লো ডায়াগ্রাম** এজেন্টদের মধ্যে তথ্য প্রবাহ দেখায়। প্রতিটি এজেন্টের নির্দিষ্ট কাজ আছে:  
- **FileAgent** MCP টুল ব্যবহার করে ফাইল পড়ে এবং কাঁচা বিষয়বস্তু `fileContent`-এ রাখে  
- **ReportAgent** ঐ বিষয়বস্তু গ্রহণ করে এবং `report`-এ একটি কাঠামোবদ্ধ রিপোর্ট তৈরি করে  

```
--- USER REQUEST -----------------------------------------------------
  "Read the file at .../file.txt and generate a report on its contents"
```
  
**ব্যবহারকারীর অনুরোধ** কাজ দেখায়। সুপারভাইজার এটি পার্স করে FileAgent → ReportAgent কল করার সিদ্ধান্ত নেয়।

  |

  |   Input: LangChain4j is an open-source, provider-agnostic Java framew...

1. **FileAgent** MCP দিয়ে ফাইল পড়ে এবং তথ্য সংগ্রহ করে  
2. **ReportAgent** তথ্য নিয়ে কাঠামোবদ্ধ রিপোর্ট তৈরি করে

```

  Each agent stores its output for downstream agents to consume:
  * fileContent: LangChain4j is an open-source, provider-agnostic Java framework...

#### এজেন্টিক মডিউলের বৈশিষ্ট্যের ব্যাখ্যা

উদাহরণে এজেন্টিক মডিউলের কয়েকটি উন্নত বৈশিষ্ট্য প্রদর্শিত হয়েছে। চলুন Agentic Scope এবং Agent Listeners কাছাকাছি দেখে নিই।

**Agentic Scope** হলো শেয়ার্ড মেমোরি যেখানে এজেন্টরা `@Agent(outputKey="...")` ব্যবহার করে তাদের ফলাফল সংরক্ষণ করে। এটি:  
- পরবর্তীকালে অন্য এজেন্টদের আগের এজেন্টদের আউটপুট দেখতে দেয়  
- সুপারভাইজারকে চূড়ান্ত প্রতিক্রিয়া তৈরি করতে সহায়তা করে  
- আপনাকে প্রতিটি এজেন্টের উত্পাদিত ফলাফল পর্যালোচনা করতে দেয়  

নিচের ডায়াগ্রাম দেখায় Agentic Scope ফাইল-টু-রিপোর্ট ওয়ার্কফ্লোতে কিভাবে শেয়ার্ড মেমোরি হিসেবে কাজ করে — FileAgent তার আউটপুট `fileContent`-এর অধীনে লেখা হয়, ReportAgent তা পড়ে এবং নিজের আউটপুট `report`-এর অধীনে লেখে:

<img src="../../../translated_images/bn/agentic-scope.95ef488b6c1d02ef.webp" alt="Agentic Scope Shared Memory" width="800"/>

*Agentic Scope শেয়ার্ড মেমোরি হিসেবে কাজ করে — FileAgent `fileContent` লেখে, ReportAgent পরে পড়ে এবং `report` লেখে, আর আপনার কোড চূড়ান্ত ফলাফল পড়ে।*

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // ফাইল এজেন্ট থেকে কাঁচা ফাইল ডেটা
String report = scope.readState("report");            // রিপোর্ট এজেন্ট থেকে গঠিত প্রতিবেদন
```
  
**Agent Listeners** এজেন্টের কার্যক্রম পর্যবেক্ষণ ও ডিবাগিং সক্ষম করে। ডেমোতে ধাপে ধাপে আউটপুটটি একটি AgentListener থেকে আসে যা প্রতিটি এজেন্ট কলের সাথে যুক্ত হয়:  
- **beforeAgentInvocation** - যখন সুপারভাইজার একটি এজেন্ট নির্বাচন করে তখন ডাকা হয়, আপনি দেখতে পান কোন এজেন্ট নির্বাচিত হয়েছে এবং কেন  
- **afterAgentInvocation** - একটি এজেন্ট কাজ শেষ করার পর ডাকা হয়, তার ফলাফল প্রদর্শন করে  
- **inheritedBySubagents** - true হলে, শ্রেণিবিন্যাসের সকল এজেন্ট পর্যবেক্ষণ করে  

নিচের চার্টটি Agent Listener এর সম্পূর্ণ জীবনচক্র দেখায়, যার মধ্যে `onError` কীভাবে এজেন্ট কার্যক্রমে ব্যর্থতা হ্যান্ডেল করে তা অন্তর্ভুক্ত:

<img src="../../../translated_images/bn/agent-listeners.784bfc403c80ea13.webp" alt="Agent Listeners Lifecycle" width="800"/>

*Agent Listeners কার্যক্রমের জীবনচক্রে যুক্ত থাকে — এজেন্ট শুরু, শেষ, বা ত্রুটি ঘটলে পর্যবেক্ষণ করে।*

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // সমস্ত উপ-এজেন্টকে ছড়িয়ে দিন
    }
};
```
  
সুপারভাইজার প্যাটার্নের বাইরে, `langchain4j-agentic` মডিউল অনেক শক্তিশালী ওয়ার্কফ্লো প্যাটার্ন সরবরাহ করে। নিচের ডায়াগ্রামটি দেখায় সব পাঁচটি প্যাটার্ন — সরল ধারাবাহিক পাইপলাইন থেকে মানব-অংশগ্রহণীয় অনুমোদন প্রক্রিয়া পর্যন্ত:

<img src="../../../translated_images/bn/workflow-patterns.82b2cc5b0c5edb22.webp" alt="Agent Workflow Patterns" width="800"/>

*এজেন্ট পরিচালনার জন্য পাঁচটি ওয়ার্কফ্লো প্যাটার্ন — সরল ধারাবাহিক থেকে মানব-অংশগ্রহণীয় অনুমোদন পর্যন্ত।*

| প্যাটার্ন | বর্ণনা | ব্যবহার ক্ষেত্র |
|---------|-------------|----------|
| **Sequential** | এজেন্টদের ধারাবাহিকভাবে চালানো, আউটপুট পরবর্তী এজেন্টে প্রবাহিত হয় | পাইপলাইন: গবেষণা → বিশ্লেষণ → রিপোর্ট |
| **Parallel** | এজেন্টরা সমান্তরালভাবে চালানো হয় | স্বতন্ত্র কাজ: আবহাওয়া + সংবাদ + শেয়ার বাজার |
| **Loop** | শর্ত পূর্ণ না হওয়া পর্যন্ত পুনরাবৃত্তি | গুণগত স্কোরিং: স্কোর ≥ 0.8 হওয়া পর্যন্ত পরিমার্জন |
| **Conditional** | শর্ত অনুযায়ী রাউটিং | শ্রেণীবিভাগ → বিশেষজ্ঞ এজেন্টের কাছে পাঠানো |
| **Human-in-the-Loop** | মানবীয় পর্যায় যোগ করা | অনুমোদন প্রক্রিয়া, বিষয়বস্তু পর্যালোচনা |

## মূল ধারণাসমূহ

এখন MCP ও এজেন্টিক মডিউল প্রয়োগের সময়গুলো সংক্ষেপে দেখি।

MCP এর অন্যতম বড় সুবিধা হল এর প্রসারমান ইকোসিস্টেম। নিচের ডায়াগ্রাম দেখায় কিভাবে একটি সার্বজনীন প্রোটোকল আপনার AI অ্যাপ্লিকেশনকে বিভিন্ন MCP সার্ভারের সাথে সংযুক্ত করে — ফাইলসিস্টেম ও ডাটাবেস থেকে GitHub, ইমেল, ওয়েব স্ক্র্যাপিং এবং আরও অনেক কিছু:

<img src="../../../translated_images/bn/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="MCP Ecosystem" width="800"/>

*MCP একটি সার্বজনীন প্রোটোকল ইকোসিস্টেম তৈরি করে — যেকোন MCP-সঙ্গত সার্ভার যেকোন MCP-সঙ্গত ক্লায়েন্টের সঙ্গে কাজ করে, অ্যাপ্লিকেশন জুড়ে টুল শেয়ারিং সক্ষম করে।*

**MCP** আদর্শ যখন আপনি বিদ্যমান টুল ইকোসিস্টেম ব্যবহার করতে চান, একাধিক অ্যাপ্লিকেশন যে টুল শেয়ার করতে পারে এমন টুল তৈরি করতে চান, তৃতীয় পক্ষের সেবাগুলিকে স্ট্যান্ডার্ড প্রোটোকল দিয়ে একত্রিত করতে চান, অথবা কোড পরিবর্তন ছাড়াই টুল ইমপ্লিমেন্টেশন পরিবর্তন করতে চান।  

**এজেন্টিক মডিউল** শ্রেষ্ঠ যখন আপনি `@Agent` এনোটেশনসহ ঘোষণামূলক এজেন্ট সংজ্ঞা চান, ওয়ার্কফ্লো সমন্বয় (ধারাবাহিক, লুপ, সমান্তরাল) প্রয়োজন, ইন্টারফেসভিত্তিক এজেন্ট ডিজাইন প্রয়োজন যা আদেশমুলক কোডের বদলে ব্যবহৃত হয়, অথবা একাধিক এজেন্ট যাদের আউটপুট `outputKey` দ্বারা ভাগ করা হয়।

**সুপারভাইজার এজেন্ট প্যাটার্ন** তখন কার্যকর যখন ওয়ার্কফ্লো পূর্বে নির্ধারিত নয় এবং আপনি চান LLM সিদ্ধান্ত নিক, যখন আপনার কাছে একাধিক বিশেষায়িত এজেন্ট থাকে যাদের গতিশীল সমন্বয় দরকার, কথোপকথন কেন্দ্রিক সিস্টেম নির্মাণ করতে চান যা বিভিন্ন ক্ষমতায় রাউট করে, অথবা সবচেয়ে নমনীয়, অভিযোজিত এজেন্ট আচরণ চান।

মডিউল 04 এর কাস্টম `@Tool` মেথড এবং এই মডিউলের MCP টুলের মধ্যে সিদ্ধান্ত গ্রহণে সহায়তার জন্য নিম্নলিখিত তুলনাটি প্রধান পার্থক্যগুলো তুলে ধরেছে — কাস্টম টুল অ্যাপ-নির্দিষ্ট লজিকের জন্য সঙ্কীর্ণ সংযোজন এবং পূর্ণ টাইপ সুরক্ষা দেয়, MCP টুলগুলি স্ট্যান্ডার্ড, পুনর্ব্যবহারযোগ্য ইন্টিগ্রেশন প্রদান করে:

<img src="../../../translated_images/bn/custom-vs-mcp-tools.c4f9b6b1cb65d8a1.webp" alt="Custom Tools vs MCP Tools" width="800"/>

*কখন কাস্টম @Tool মেথড ব্যবহার করবেন এবং কখন MCP টুল — অ্যাপ-নির্দিষ্ট লজিকের জন্য কাস্টম টুল, অ্যাপ্লিকেশন ছড়িয়ে ইন্টিগ্রেশনের জন্য MCP টুল।*
