## আপনি যা শিখবেন

এখন পর্যন্ত, আপনি শিখেছেন কিভাবে AI এর সাথে আলাপচারিতা করতে হয়, কীভাবে প্রম্পট গঠন করতে হয় এবং উত্তরগুলোকে আপনার ডকুমেন্টে ভিত্তি করে বানাতে হয়। কিন্তু একটি মৌলিক সীমাবদ্ধতা রয়ে গেছে: ভাষার মডেল কেবলমাত্র টেক্সট তৈরি করতে পারে। তারা আবহাওয়া যাচাই করতে পারে না, গণনা করতে পারে না, ডেটাবেজে অনুসন্ধান করতে পারে না, বা বাহ্যিক সিস্টেমের সাথে যোগাযোগ করতে পারে না।

টুলগুলো এটি পরিবর্তন করে। মডেলকে এমন ফাংশনগুলোর অ্যাক্সেস দেওয়ার মাধ্যমে যা এটি কল করতে পারে, আপনি এটিকে একটি টেক্সট জেনারেটরের পরিবর্তে এমন একটি এজেন্টে রূপান্তর করেন যা কার্য সম্পাদন করতে পারে। মডেল সিদ্ধান্ত নেয় কখন টুল দরকার, কোন টুল ব্যবহার করবে এবং কী প্যারামিটার পাঠাবে। আপনার কোড ফাংশনটি কার্যকর করে এবং ফলাফল ফেরত দেয়। মডেল সেই ফলাফলকে তার উত্তরে অন্তর্ভুক্ত করে।

## টুল সহ AI এজেন্ট বোঝা

> **📝 নোট:** এই মডিউলে "এজেন্ট" শব্দটি টুল-কলিং সক্ষমতা যুক্ত AI সহকারী বোঝাতে ব্যবহৃত হয়েছে। এটি **Agentic AI** প্যাটার্ন থেকে আলাদা (স্বয়ংক্রিয় এজেন্ট যারা পরিকল্পনা, স্মৃতি এবং বহু-ধাপ যুক্তি ব্যবহার করে), যেগুলো আমরা [মডিউল ০৫: MCP](../05-mcp/README.md) এ আলোচনা করব।

টুল না থাকলে, একটি ভাষার মডেল কেবল তার প্রশিক্ষণ ডেটা থেকে টেক্সট তৈরি করতে পারে। "বর্তমান আবহাওয়া কেমন?" প্রশ্ন করলে, এটি কেবল অনুমান করে। টুল দিলে, এটি আবহাওয়ার API কল করতে পারে, গণনা চালাতে পারে, বা ডেটাবেজ অনুসন্ধান করতে পারে — এবং সেই প্রকৃত ফলাফলগুলো তার উত্তরে প্রবাহিত করতে পারে।

<img src="../../../translated_images/bn/what-are-tools.724e468fc4de64da.webp" alt="Without Tools vs With Tools" width="800"/>

*টুল না থাকলে মডেল কেবল অনুমান করে — টুল থাকলে এটি API কল করতে পারে, গণনা চালাতে পারে, এবং বাস্তব সময়ের ডেটা ফেরত দিতে পারে।*

টুলসহ AI এজেন্ট একটি **যুক্তি এবং কর্ম (ReAct)** প্যাটার্ন অনুসরণ করে। মডেল কেবল প্রত্যুত্তর দেয় না — এটি প্রথমে চিন্তা করে কী দরকার, তারপর টুল কল করে, ফলাফল পর্যবেক্ষণ করে, এবং সিদ্ধান্ত নেয় আবার কাজ করবে নাকি চূড়ান্ত উত্তর দেবে:

১. **যুক্তি** — এজেন্ট ব্যবহারকারীর প্রশ্ন বিশ্লেষণ করে এবং প্রয়োজনীয় তথ্য নির্ধারণ করে  
২. **কর্ম** — এজেন্ট সঠিক টুল নির্বাচন করে, সঠিক প্যারামিটার তৈরি করে এবং টুল কল করে  
৩. **পর্যবেক্ষণ** — এজেন্ট টুলের আউটপুট পায় এবং ফলাফল মূল্যায়ন করে  
৪. **পুনরাবৃত্তি অথবা উত্তর** — যদি আরও তথ্য প্রয়োজন হয়, এজেন্ট আবার শুরু করে; নাহলে প্রাকৃতিক ভাষায় উত্তর তৈরি করে

<img src="../../../translated_images/bn/react-pattern-detail.96a5efeeb6dd2f61.webp" alt="ReAct Pattern" width="800"/>

*ReAct চক্র — এজেন্ট সিদ্ধান্ত নেয় কী করতে হবে, টুল কল করে, ফলাফল পর্যবেক্ষণ করে, এবং শেষ পর্যন্ত চূড়ান্ত উত্তর দেয়।*

এটি স্বয়ংক্রিয়ভাবে ঘটে। আপনি টুলগুলো এবং তাদের বিবরণ নির্ধারণ করেন। মডেল সিদ্ধান্ত নেয় কখন এবং কীভাবে টুলগুলো ব্যবহার করতে হবে।

## কিভাবে টুল কল কাজ করে

### টুল সংজ্ঞাসমূহ

[WeatherTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) | [TemperatureTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/TemperatureTool.java)

আপনি পরিষ্কার বিবরণ এবং প্যারামিটার স্পেসিফিকেশন সহ ফাংশন সংজ্ঞায়িত করেন। মডেল এই বিবরণগুলো সিস্টেম প্রম্পটে দেখে এবং বুঝতে পারে প্রতিটি টুল কী কাজ করে।

```java
@Component
public class WeatherTool {
    
    @Tool("Get the current weather for a location")
    public String getCurrentWeather(@P("Location name") String location) {
        // আপনার আবহাওয়া অনুসন্ধান লজিক
        return "Weather in " + location + ": 22°C, cloudy";
    }
}

@AiService
public interface Assistant {
    String chat(@MemoryId String sessionId, @UserMessage String message);
}

// সহকারী স্বয়ংক্রিয়ভাবে Spring Boot দ্বারা যুক্ত হয়েছে:
// - ChatModel বিন
// - @Component শ্রেণীর সব @Tool পদ্ধতি
// - সেশন ব্যবস্থাপনার জন্য ChatMemoryProvider
```
  
নিচের ডায়াগ্রামটি প্রতিটি অ্যানোটেশন বিশ্লেষণ করে এবং দেখায় কিভাবে প্রতিটি অংশ AI কে বুঝতে সাহায্য করে কখন টুল কল করতে হবে এবং কোন আর্গুমেন্টগুলো পাঠাতে হবে:

<img src="../../../translated_images/bn/tool-definitions-anatomy.f6468546037cf28b.webp" alt="Anatomy of Tool Definitions" width="800"/>

*একটি টুল সংজ্ঞার গঠন — @Tool AI কে বলে কখন ব্যবহার করতে হবে, @P প্রতিটি প্যারামিটার বর্ণনা করে, এবং @AiService স্টার্টআপে সব কিছু একত্রিত করে ওয়্যারিং করে।*

### সিদ্ধান্ত গ্রহণ

যখন ব্যবহারকারী "সিয়াটল-এ আবহাওয়া কেমন?" জিজ্ঞাসা করে, মডেল এলোমেলোভাবে কোন টুল বেছে নেয় না। এটি ব্যবহারকারীর উদ্দেশ্য প্রত্যেক টুলের স্পেসিফিকেশনের সাথে মিলিয়ে দেখে, সম্পর্ক বিচার করে এবং শ্রেষ্ঠ মিল বেছে নেয়। তারপর সঠিক প্যারামিটার সহ স্ট্রক্চার্ড ফাংশন কল তৈরি করে — ঐ ক্ষেত্রে, `location` সেট করে `"Seattle"`।

যদি ব্যবহারকারীর অনুরোধের জন্য কোন উপযুক্ত টুল না থাকে, মডেল নিজের জ্ঞানে উত্তর দেয়। যদি একাধিক টুল মেলে, তবে সবচেয়ে নির্দিষ্টটিকে বেছে নেয়।

<img src="../../../translated_images/bn/decision-making.409cd562e5cecc49.webp" alt="How the AI Decides Which Tool to Use" width="800"/>

*মডেল প্রত্যেক উপলব্ধ টুল ব্যবহারকারীর অভিপ্রায়ের সাথে মূল্যায়ন করে এবং সবচেয়ে উপযুক্তটি নির্বাচন করে — এজন্য স্পষ্ট, নির্দিষ্ট টুল বিবরণ লেখা গুরুত্বপূর্ণ।*

> - "যদি টুল কল ব্যর্থ হয় - কিভাবে ত্রুটি সুষ্ঠুভাবে মোকাবিলা করব?"

### প্রতিক্রিয়া সৃজন

মডেল আবহাওয়ার ডেটা পায় এবং ব্যবহারকারীর জন্য প্রাকৃতিক ভাষা উত্তরে রূপান্তর করে।

### স্থাপত্য: স্প্রিং বুট অটো-ওয়্যারিং

এই মডিউল LangChain4j এর স্প্রিং বুট ইন্টিগ্রেশন ব্যবহার করে যা ঘোষণামূলক `@AiService` ইন্টারফেস ব্যবহার করে। স্টার্টআপে, স্প্রিং বুট সব `@Tool` পদ্ধতি যুক্ত `@Component` গুলো, আপনার `ChatModel` bean, এবং `ChatMemoryProvider` আবিষ্কার করে — তারপর সবকিছু একক `Assistant` ইন্টারফেসে জিরো বয়লারপ্লেট সহ ওয়্যার করে।

<img src="../../../translated_images/bn/spring-boot-wiring.151321795988b04e.webp" alt="Spring Boot Auto-Wiring Architecture" width="800"/>

*@AiService ইন্টারফেস ChatModel, টুল কম্পোনেন্টস এবং মেমোরি প্রোভাইডারকে একত্র করে — স্প্রিং বুট সব ওয়্যারিং স্বয়ংক্রিয়ভাবে পরিচালনা করে।*

এই পন্থার প্রধান সুবিধাসমূহ:

- **স্প্রিং বুট অটো-ওয়্যারিং** — ChatModel এবং টুলগুলো স্বয়ংক্রিয়ভাবে ইনজেক্টেড হয়  
- **@MemoryId প্যাটার্ন** — স্বয়ংক্রিয় সেশন-বেসড মেমোরি ব্যবস্থাপনা  
- **একক ইনস্ট্যান্স** — একবার Assistant তৈরি হয় এবং পুনঃব্যবহার করা হয় কর্মক্ষমতা বাড়াতে  
- **টাইপ-সেফ এক্সিকিউশন** — Java পদ্ধতিগুলো সরাসরি টাইপ কনভার্সনের সাথে কল করা হয়  
- **মাল্টি-টার্ন অর্কেস্ট্রেশন** — স্বয়ংক্রিয়ভাবে টুল চেনিং পরিচালনা করে  
- **জিরো বয়লারপ্লেট** — কোন ম্যানুয়াল `AiServices.builder()` কল বা মেমোরি HashMap লাগে না  

বিকল্প পদ্ধতি (ম্যানুয়াল `AiServices.builder()`) বেশি কোড দরকার এবং স্প্রিং বুট ইন্টিগ্রেশনের সুবিধা মিস করে।

## টুল চেনিং

**টুল চেনিং** — টুল-ভিত্তিক এজেন্টের প্রকৃত ক্ষমতা প্রকাশ পায় যখন একটি প্রশ্নের জন্য একাধিক টুলের প্রয়োজন হয়। উদাহরণস্বরূপ, "সিয়াটলের আবহাওয়া ফারেনহাইটে কেমন?" জিজ্ঞাসা করলে এজেন্ট স্বয়ংক্রিয়ভাবে দুইটি টুল চেইন করে: প্রথমে `getCurrentWeather` কল করে সেলসিয়াস তাপমাত্রা নেয়, তারপর সেটি `celsiusToFahrenheit` পাস করে রূপান্তর করে — সব একক আলাপের টার্নে।

<img src="../../../translated_images/bn/tool-chaining-example.538203e73d09dd82.webp" alt="Tool Chaining Example" width="800"/>

*টুল চেনিং কার্যক্রম — এজেন্ট প্রথমে getCurrentWeather কল করে, তারপর সেলসিয়াস ফলাফল celsiusToFahrenheit এ পাস করে এবং সম্মিলিত উত্তর দেয়।*

**দমনযোগ্য ব্যর্থতা** — কোনো মক ডেটা না থাকা শহরের আবহাওয়া চাইলে টুল একটি ত্রুটি বার্তা ফেরত দেয় এবং AI এর ব্যাখ্যা থাকে যে সাহায্য করতে পারে না, অ্যাপ্লিকেশন ক্র্যাশ করে না। টুলগুলো নিরাপদে ব্যর্থ হয়। নিচের ডায়াগ্রাম দুটি পদ্ধতির তুলনা করে — যথাযথ ত্রুটি পরিচালনার সাথে, এজেন্ট এক্সসেপশন ধরে সাহায্যপূর্ণ উত্তর দেয়; অন্যদিকে তা না থাকলে পুরো অ্যাপ্লিকেশন ক্র্যাশ করে:

<img src="../../../translated_images/bn/error-handling-flow.9a330ffc8ee0475c.webp" alt="Error Handling Flow" width="800"/>

*যখন একটি টুল ব্যর্থ হয়, এজেন্ট ত্রুটি ধরে সাহায্যপূর্ণ ব্যাখ্যা দিয়ে উত্তর দেয় ক্র্যাশ না হয়ে।*

এটি একক আলাপের একটি টার্নে ঘটে। এজেন্ট স্বয়ংক্রিয়ভাবে একাধিক টুল কলগুলো পরিচালনা করে।

- অ্যাপ্লিকেশন স্ট্যাটাস মনিটর করতে পারবেন  

"tools" এর পাশে প্লে বাটনে ক্লিক করে এই মডিউল শুরু করুন, অথবা একসাথে সব মডিউল চালু করুন।

## অ্যাপ্লিকেশন ব্যবহার

অ্যাপ্লিকেশন একটি ওয়েব ইন্টারফেস প্রদান করে যেখানে আপনি আবহাওয়া এবং তাপমাত্রা রূপান্তরের টুল অ্যাক্সেসসহ একটি AI এজেন্টের সাথে আলাপ করতে পারেন। ইন্টারফেসটি এইরকম দেখায় — এতে দ্রুত শুরু করার উদাহরণ এবং চ্যাট প্যানেল রয়েছে যেখানে আপনি অনুরোধ পাঠাতে পারবেন:
<a href="images/tools-homepage.png"><img src="../../../translated_images/bn/tools-homepage.4b4cd8b2717f9621.webp" alt="এআই এজেন্ট টুলস ইন্টারফেস" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*এআই এজেন্ট টুলস ইন্টারফেস - টুলসের সাথে ইন্টারঅ্যাক্ট করার দ্রুত উদাহরণ এবং চ্যাট ইন্টারফেস*

### সহজ টুল ব্যবহারের চেষ্টা করুন

একটি সরল অনুরোধ দিয়ে শুরু করুন: "Convert 100 degrees Fahrenheit to Celsius"। এজেন্ট বুঝতে পারে এটি তাপমাত্রা রূপান্তর টুল ব্যবহার করতে হবে, সঠিক প্যারামিটার দিয়ে কল করে এবং ফলাফল ফেরত দেয়। লক্ষ্য করুন এটা কতটা প্রাকৃতিক লাগে - আপনি কোন টুল ব্যবহার করবেন বা কিভাবে কল করবেন তা নির্দিষ্ট করেননি।

### টুল চেইনিং পরীক্ষা করুন

এখন কিছু বেশি জটিল চেষ্টা করুন: "What's the weather in Seattle and convert it to Fahrenheit?" দেখুন এজেন্ট কীভাবে ধাপে ধাপে কাজ করে। প্রথমে এটি আবহাওয়া নেয় (যা সেলসিয়াসে ফিরে আসে), বুঝতে পারে এটি ফারেনহাইটে রূপান্তর করতে হবে, রূপান্তর টুল কল করে, এবং উভয় ফলাফল একত্রিত করে একটি প্রতিক্রিয়া তৈরি করে।

### কথোপকথনের প্রবাহ দেখুন

চ্যাট ইন্টারফেস কথোপকথনের ইতিহাস সংরক্ষণ করে, যা আপনাকে বহু-বারের ইন্টারঅ্যাকশন করতে দেয়। আপনি সব পূর্ববর্তী প্রশ্ন এবং উত্তর দেখতে পারেন, যা কথোপকথন ট্র্যাক করা এবং এজেন্ট কীভাবে বহু বিনিময়ে প্রসঙ্গ গঠন করে তা বুঝতে সহজ করে।

### বিভিন্ন অনুরোধ নিয়ে পরীক্ষা করুন

বিভিন্ন সংমিশ্রণ চেষ্টা করুন:
- আবহাওয়া অনুসন্ধান: "What's the weather in Tokyo?"
- তাপমাত্রার রূপান্তর: "What is 25°C in Kelvin?"
- সংযুক্ত প্রশ্ন: "Check the weather in Paris and tell me if it's above 20°C"

লক্ষ্য করুন এজেন্ট কীভাবে প্রাকৃতিক ভাষা অনুধাবন করে এবং সঠিক টুল কলগুলির সাথে সেটি মানচিত্রিত করে।

## প্রধান ধারণাগুলি

### ReAct প্যাটার্ন (কারণ এবং কার্য)

এজেন্ট যুক্তি (কি করতে হবে তা সিদ্ধান্ত নেওয়া) এবং কার্য (টুল ব্যবহার করা) এর মধ্যে পরিবর্তন করে। এই প্যাটার্নটি স্বয়ংক্রিয় সমস্যার সমাধান সম্ভব করে শুধুমাত্র নির্দেশ মেনে চলার বদলে।

### টুল বর্ণনা গুরুত্বপূর্ণ

আপনার টুল বর্ণনার গুণগত মান সরাসরি প্রভাব ফেলে এজেন্ট কিভাবে এগুলো ব্যবহার করে। পরিষ্কার, নির্দিষ্ট বর্ণনা মডেলকে বুঝতে সাহায্য করে কখন এবং কিভাবে প্রতিটি টুল কল করতে হয়।

### সেশন ব্যবস্থাপনা

`@MemoryId` এনটেশন স্বয়ংক্রিয় সেশন-ভিত্তিক মেমোরি ব্যবস্থাপনা সক্ষম করে। প্রতিটি সেশন আইডি নিজস্ব `ChatMemory` ইনস্ট্যান্স পায় যা `ChatMemoryProvider` বিন দ্বারা পরিচালিত হয়, তাই একাধিক ব্যবহারকারী এজেন্টের সাথে একই সময় ইন্টারঅ্যাক্ট করতে পারে তাদের কথোপকথন মিশে না। নিচের চিত্রটি দেখায় কীভাবে একাধিক ব্যবহারকারী সেশন আইডি অনুযায়ী পৃথক মেমোরি স্টোরে রুটিং পায়:

<img src="../../../translated_images/bn/session-management.91ad819c6c89c400.webp" alt="সেশন ব্যবস্থাপনা @MemoryId সহ" width="800"/>

*প্রতিটি সেশন আইডি আলাদা কথোপকথন ইতিহাসে ম্যাপ করে — ব্যবহারকারীরা একে অপরের বার্তা কখনো দেখতে পায় না।*

### ত্রুটি পরিচালনা

টুলস ব্যর্থ হতে পারে — এপিআই টাইমআউট, প্যারামিটার অবৈধ হতে পারে, বাহ্যিক সার্ভিস ডাউন হতে পারে। প্রোডাকশন এজেন্টদের ত্রুটি পরিচালনার প্রয়োজন যাতে মডেল সমস্যা ব্যাখ্যা করতে পারে বা বিকল্প চেষ্টা করতে পারে পুরো অ্যাপلیکেশন ক্র্যাশ না করে। যখন একটি টুল এক্সসেপশন থ্রো করে, LangChain4j সেটি ধরতে পারে এবং ত্রুটির বার্তাটি মডেলে পাঠায়, যা স্বাভাবিক ভাষায় সমস্যাটি ব্যাখ্যা করতে পারে।

## উপলব্ধ টুলস

নিচের চিত্রটি দেখায় আপনি কোন বিস্তৃত ইকোসিস্টেমের টুল তৈরি করতে পারেন। এই মডিউল আবহাওয়া এবং তাপমাত্রা টুলস প্রদর্শন করে, কিন্তু একই `@Tool` প্যাটার্ন যেকোনো জাভা মেথডের জন্য কাজ করে—ডাটাবেস প্রশ্ন থেকে শুরু করে পেমেন্ট প্রসেসিং পর্যন্ত।

<img src="../../../translated_images/bn/tool-ecosystem.aad3d74eaa14a44f.webp" alt="টুল ইকোসিস্টেম" width="800"/>

*যেকোনো জাভা মেথড যা @Tool দিয়ে এনোটেশন করা হয় তা এআই এর কাছে উপলব্ধ হয়ে যায়—প্যাটার্নটি ডাটাবেস, এপিআই, ইমেল, ফাইল অপারেশন এবং আরও অনেক কিছুর জন্য প্রসারিত হয়।*

## কখন টুল-ভিত্তিক এজেন্ট ব্যবহার করবেন

প্রত্যেক অনুরোধেই টুলের প্রয়োজন হয় না। সিদ্ধান্তটি নির্ভর করে এআইকে বাহ্যিক সিস্টেমের সাথে ইন্টারঅ্যাক্ট করতে হবে কিনা বা নিজের জ্ঞান থেকে উত্তর দিতে পারে কিনা। নিচের গাইড সংক্ষেপে দেখায় কখন টুলস মূল্য যোগ করে এবং কখন তারা অপ্রয়োজনীয়:

<img src="../../../translated_images/bn/when-to-use-tools.51d1592d9cbdae9c.webp" alt="কখন টুলস ব্যবহার করবেন" width="800"/>

*একটি দ্রুত সিদ্ধান্ত নির্দেশিকা — টুলস রিয়েল-টাইম ডাটা, হিসাব-নিকাশ, এবং কার্যাবলীর জন্য; সাধারণ জ্ঞান ও সৃজনশীল কাজের জন্য তাদের প্রয়োজন হয় না।*

## টুলস বনাম RAG

মডিউল ০৩ এবং ০৪ দুটোই এআই কী করতে পারে তা বাড়ায়, কিন্তু মৌলিকভাবে ভিন্নভাবে। RAG মডেলকে **জ্ঞান** অ্যাক্সেস দেয় ডকুমেন্ট রিট্রিভ করার মাধ্যমে। টুলস মডেলকে **কাজ** করার ক্ষমতা দেয় ফাংশন কল করে। নিচের চিত্রে এই দুই পদ্ধতির পার্থক্য পাশাপাশি তুলনা করা হয়েছে—কিভাবে প্রতিটি ওয়ার্কফ্লো কাজ করে এবং তাদের মধ্যে ট্রেড-অফগুলো:

<img src="../../../translated_images/bn/tools-vs-rag.ad55ce10d7e4da87.webp" alt="টুলস বনাম RAG তুলনা" width="800"/>

*RAG স্থির ডকুমেন্ট থেকে তথ্য রিট্রিভ করে — টুলস কাজ সম্পাদন করে এবং ডায়নামিক, রিয়েল-টাইম ডাটা নিয়ে আসে। অনেক প্রোডাকশন সিস্টেম উভয়কে একত্রিত করে।*

প্র্যাকটিসে, অনেক প্রোডাকশন সিস্টেম উভয় পদ্ধতির সংমিশ্রণ ব্যবহার করে: RAG আপনার ডকুমেন্টেশনে উত্তরের ভিত্তি স্থাপনের জন্য, এবং টুলস লাইভ ডাটা আনা বা অপারেশন করার জন্য।
