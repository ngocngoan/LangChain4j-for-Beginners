## आप क्या सीखेंगे

आपने बातचीत योग्य AI बनाया है, प्रॉम्प्ट में महारत हासिल की है, दस्तावेज़ों में उत्तर आधारित किए हैं, और टूल्स वाले एजेंट बनाए हैं। लेकिन ये सभी टूल्स आपके विशिष्ट एप्लिकेशन के लिए कस्टम-निर्मित थे। क्या होगा अगर आप अपने AI को टूल्स के एक मानकीकृत इकोसिस्टम तक पहुंच दे सकें जिसे कोई भी बना और साझा कर सके? इस मॉड्यूल में, आप Model Context Protocol (MCP) और LangChain4j के एजेंटिक मॉड्यूल के साथ ऐसा करना सीखेंगे। हम पहले एक सरल MCP फ़ाइल रीडर दिखाते हैं और फिर दिखाते हैं कि यह कैसे आसानी से सुपरवाइज़र एजेंट पैटर्न के साथ उन्नत एजेंटिक वर्कफ़्लो में एकीकृत होता है।

## MCP क्या है?

Model Context Protocol (MCP) ठीक यही प्रदान करता है - AI एप्लिकेशन के लिए बाहरी टूल्स को खोजने और उपयोग करने का एक मानक तरीका। हर डेटा स्रोत या सेवा के लिए कस्टम इंटीग्रेशन लिखने की बजाय, आप MCP सर्वर से जुड़ते हैं जो अपनी क्षमताओं को एक सुसंगत प्रारूप में प्रदर्शित करता है। आपका AI एजेंट तब इन टूल्स को स्वचालित रूप से खोज और उपयोग कर सकता है।

नीचे दिए गए आरेख से फर्क स्पष्ट होता है — बिना MCP के, हर इंटीग्रेशन के लिए कस्टम पॉइंट-टू-पॉइंट कनेक्शन चाहिए; MCP के साथ, एक प्रोटोकॉल आपके एप्लिकेशन को किसी भी टूल से जोड़ता है:

<img src="../../../translated_images/hi/mcp-comparison.9129a881ecf10ff5.webp" alt="MCP Comparison" width="800"/>

*पहले MCP: जटिल पॉइंट-टू-पॉइंट इंटीग्रेशन। MCP के बाद: एक प्रोटोकॉल, अंतहीन संभावनाएँ।*

MCP AI विकास में एक मौलिक समस्या को हल करता है: हर इंटीग्रेशन कस्टम होता है। GitHub तक पहुंचना चाहते हैं? कस्टम कोड। फ़ाइलें पढ़ना चाहते हैं? कस्टम कोड। डेटाबेस क्वेरी करना चाहते हैं? कस्टम कोड। और ये कोई भी इंटीग्रेशन अन्य AI एप्लिकेशन के साथ काम नहीं करता।

MCP इसे मानकीकृत करता है। एक MCP सर्वर टूल्स को स्पष्ट विवरण और स्कीमाओं के साथ प्रदर्शित करता है। कोई भी MCP क्लाइंट जुड़ सकता है, उपलब्ध टूल्स को खोज सकता है, और उनका उपयोग कर सकता है। एक बार बनाएं, हर जगह उपयोग करें।

नीचे दिया गया आरेख इस आर्किटेक्चर को दर्शाता है — एक MCP क्लाइंट (आपका AI एप्लिकेशन) कई MCP सर्वरों से जुड़ता है, जो अपने-अपने टूल्स को मानक प्रोटोकॉल के माध्यम से प्रदर्शित करते हैं:

<img src="../../../translated_images/hi/mcp-architecture.b3156d787a4ceac9.webp" alt="MCP Architecture" width="800"/>

*Model Context Protocol आर्किटेक्चर - मानकीकृत टूल खोज और निष्पादन*

## MCP कैसे काम करता है

अंदर की सतह के नीचे, MCP एक स्तरित आर्किटेक्चर का उपयोग करता है। आपका जावा एप्लिकेशन (MCP क्लाइंट) उपलब्ध टूल्स को खोजता है, JSON-RPC अनुरोध ट्रांसपोर्ट लेयर (Stdio या HTTP) के जरिए भेजता है, और MCP सर्वर ऑपरेशन को निष्पादित करके परिणाम लौटाता है। निम्नलिखित आरेख इस प्रोटोकॉल के प्रत्येक स्तर को विभाजित करता है:

<img src="../../../translated_images/hi/mcp-protocol-detail.01204e056f45308b.webp" alt="MCP Protocol Detail" width="800"/>

*MCP कैसे काम करता है — क्लाइंट टूल्स खोजते हैं, JSON-RPC संदेशों का आदान-प्रदान करते हैं, और ट्रांसपोर्ट लेयर के माध्यम से ऑपरेशन निष्पादित करते हैं।*

**सर्वर-क्लाइंट आर्किटेक्चर**

MCP क्लाइंट-सर्वर मॉडल का उपयोग करता है। सर्वर टूल्स प्रदान करते हैं - फ़ाइलें पढ़ना, डेटाबेस क्वेरी करना, API कॉल करना। क्लाइंट्स (आपका AI एप्लिकेशन) सर्वरों से जुड़ते हैं और उनके टूल्स का उपयोग करते हैं।

LangChain4j के साथ MCP का उपयोग करने के लिए, इस Maven निर्भरता को जोड़ें:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```


**टूल डिस्कवरी**

जब आपका क्लाइंट किसी MCP सर्वर से जुड़ता है, तो वह पूछता है "आपके पास कौन-कौन से टूल्स हैं?" सर्वर उपलब्ध टूल्स की सूची विवरण और पैरामीटर स्कीमाओं के साथ देता है। आपका AI एजेंट तब उपयोगकर्ता की मांग के आधार पर यह तय कर सकता है कि कौन से टूल्स इस्तेमाल करने हैं। नीचे दिया गया आरेख यह हैंडशेक दर्शाता है — क्लाइंट `tools/list` अनुरोध भेजता है और सर्वर अपने उपलब्ध टूल्स विवरण और पैरामीटर स्कीमाओं के साथ लौटाता है:

<img src="../../../translated_images/hi/tool-discovery.07760a8a301a7832.webp" alt="MCP Tool Discovery" width="800"/>

*AI स्टार्टअप पर उपलब्ध टूल्स खोजता है — अब यह जानता है कि कौन-कौन सी क्षमताएँ उपलब्ध हैं और निर्णय ले सकता है कि कौन से टूल्स का उपयोग करना है।*

**ट्रांसपोर्ट मेकनिज़्म**

MCP विभिन्न ट्रांसपोर्ट मेकनिज़्म का समर्थन करता है। दो विकल्प हैं Stdio (लोकल सबप्रोसेस संचार के लिए) और स्ट्रीमेबल HTTP (रिमोट सर्वरों के लिए)। यह मॉड्यूल Stdio ट्रांसपोर्ट का प्रदर्शन करता है:

<img src="../../../translated_images/hi/transport-mechanisms.2791ba7ee93cf020.webp" alt="Transport Mechanisms" width="800"/>

*MCP ट्रांसपोर्ट मेकनिज़्म: HTTP रिमोट सर्वरों के लिए, Stdio लोकल प्रोसेसेस के लिए*

**Stdio** - [StdioTransportDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java)

लोकल प्रोसेसेस के लिए। आपका एप्लिकेशन एक सर्वर को सबप्रोसेस के रूप में स्पॉन करता है और मानक इनपुट/आउटपुट के माध्यम से संचार करता है। फ़ाइल सिस्टम एक्सेस या कमांड-लाइन टूल्स के लिए उपयोगी।

```java
McpTransport stdioTransport = new StdioMcpTransport.Builder()
    .command(List.of(
        npmCmd, "exec",
        "@modelcontextprotocol/server-filesystem@2025.12.18",
        resourcesDir
    ))
    .logEvents(false)
    .build();
```


`@modelcontextprotocol/server-filesystem` सर्वर निम्नलिखित टूल्स एक्सपोज़ करता है, सभी उस निर्देशिका तक सैंडबॉक्स्ड जिनका आप उल्लेख करते हैं:

| टूल | विवरण |
|------|-------------|
| `read_file` | एक फ़ाइल की सामग्री पढ़े |
| `read_multiple_files` | एक कॉल में कई फ़ाइलें पढ़े |
| `write_file` | एक फ़ाइल बनाए या ओवरराइट करे |
| `edit_file` | फ़ाइंड-एंड-रिप्लेस संपादन करे |
| `list_directory` | किसी पथ पर फ़ाइलें और निर्देशिकाएं सूचीबद्ध करे |
| `search_files` | किसी पैटर्न से मेल खाने वाली फ़ाइलों के लिए पुनरावृत्त खोज |
| `get_file_info` | फ़ाइल मेटाडेटा प्राप्त करें (आकार, टाइमस्टैम्प, अनुमतियाँ) |
| `create_directory` | एक निर्देशिका बनाए (पेरेंट निर्देशिकाओं सहित) |
| `move_file` | फ़ाइल या निर्देशिका को स्थानांतरित या नाम बदलें |

नीचे दिया गया आरेख रUNTIME में Stdio ट्रांसपोर्ट के काम को दिखाता है — आपका जावा एप्लिकेशन MCP सर्वर को एक चाइल्ड प्रोसेस के रूप में स्पॉन करता है और वे stdin/stdout पाइप्स के जरिए संवाद करते हैं, बिना किसी नेटवर्क या HTTP के:

<img src="../../../translated_images/hi/stdio-transport-flow.45eaff4af2d81db4.webp" alt="Stdio Transport Flow" width="800"/>

*Stdio ट्रांसपोर्ट क्रिया में — आपका एप्लिकेशन MCP सर्वर को चाइल्ड प्रोसेस के रूप में स्पॉन करता है और stdin/stdout पाइप्स के जरिए संचार करता है।*

> - "LangChain4j MCP सर्वर प्रक्रियाओं के जीवनचक्र को कैसे प्रबंधित करता है?"
> - "AI को फ़ाइल सिस्टम की पहुंच देने के सुरक्षा निहितार्थ क्या हैं?"

## एजेंटिक मॉड्यूल

जहां MCP मानकीकृत टूल्स प्रदान करता है, LangChain4j का **एजेंटिक मॉड्यूल** उन टूल्स के समन्वय के लिए एक घोषणात्मक तरीका देता है। `@Agent` एनोटेशन और `AgenticServices` आपको एजेंट व्यवहार इंटरफेस के माध्यम से परिभाषित करने देते हैं, न कि अनिवार्य कोड के माध्यम से।

इस मॉड्यूल में, आप **सुपरवाइज़र एजेंट** पैटर्न का अन्वेषण करेंगे — एक उन्नत एजेंटिक AI दृष्टिकोण जिसमें "सुपरवाइज़र" एजेंट उपयोगकर्ता अनुरोधों के आधार पर गतिशील रूप से उप-एजेंट्स को कॉल करने का निर्णय लेता है। हम दोनों अवधारणाओं को मिलाकर अपने उप-एजेंट्स में से एक को MCP संचालित फ़ाइल पहुंच क्षमताएँ देंगे।

एजेंटिक मॉड्यूल का उपयोग करने के लिए, इस Maven निर्भरता को जोड़ें:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```


> **नोट:** `langchain4j-agentic` मॉड्यूल एक अलग संस्करण गुण (`langchain4j.mcp.version`) का उपयोग करता है क्योंकि यह कोर LangChain4j लाइब्रेरीज़ से अलग रिलीज़ योजना पर है।

> **⚠️ प्रायोगिक:** `langchain4j-agentic` मॉड्यूल **प्रायोगिक** है और इसमें परिवर्तन हो सकते हैं। AI असिस्टेंट्स बनाने का स्थिर तरीका `langchain4j-core` के साथ कस्टम टूल्स का उपयोग करना जारी है (मॉड्यूल 04)।

### सुपरवाइज़र एजेंट

**सुपरवाइज़र एजेंट पैटर्न** एजेंटिक AI का एक **लचीला** रूप है। एक सुपरवाइज़र उपयोगकर्ता के अनुरोध के आधार पर स्वायत्त रूप से निर्णय करता है कि कौन से एजेंट्स को बुलाना है। अगले उदाहरण में, हम MCP संचालित फ़ाइल पहुंच को एक LLM एजेंट के साथ मिलाकर एक सुपरवाइज़्ड फ़ाइल पढ़ना → रिपोर्ट वर्कफ़्लो बनाएंगे।

डेमो में, `FileAgent` MCP फ़ाइल सिस्टम टूल्स का उपयोग करके फ़ाइल पढ़ता है, और `ReportAgent` एक संरचित रिपोर्ट (कार्यकारी सारांश - 1 वाक्य, 3 मुख्य बिंदु, और सिफारिशें) बनाता है। सुपरवाइज़र इस फ्लो को स्वचालित रूप से समन्वयित करता है:

<img src="../../../translated_images/hi/supervisor-agent-pattern.06275a41ae006ac8.webp" alt="Supervisor Agent Pattern" width="800"/>

*सुपरवाइज़र अपने LLM का उपयोग करता है जो निष्पादित एजेंट्स और उनका क्रम तय करता है — कोई हार्डकोडेड रूटिंग आवश्यक नहीं।*

यहाँ हमारे फ़ाइल-से-रिपोर्ट पाइपलाइन के लिए ठोस वर्कफ़्लो कैसा दिखता है:

<img src="../../../translated_images/hi/file-report-workflow.649bb7a896800de9.webp" alt="File to Report Workflow" width="800"/>

*FileAgent MCP टूल्स के माध्यम से फ़ाइल पढ़ता है, फिर ReportAgent कच्ची सामग्री को संरचित रिपोर्ट में परिवर्तित करता है।*

निम्न अनुक्रम आरेख पूरा सुपरवाइज़र समन्वय ट्रेस करता है — MCP सर्वर को स्पॉन करने से लेकर, सुपरवाइज़र के स्वायत्त एजेंट चयन, stdio के माध्यम से टूल कॉल्स, और अंतिम रिपोर्ट तक:

<img src="../../../translated_images/hi/supervisor-agent-sequence.1aa389b3bef99956.webp" alt="Supervisor Agent Sequence Diagram" width="800"/>

*सुपरवाइज़र FileAgent को स्वायत्त रूप से बुलाता है (जो stdio पर MCP सर्वर को कॉल करके फ़ाइल पढ़ता है), फिर ReportAgent को संरचित रिपोर्ट बनाने के लिए बुलाता है — प्रत्येक एजेंट अपना आउटपुट साझा Agentic Scope में स्टोर करता है।*

प्रति एजेंट अपना आउटपुट **Agentic Scope** (साझा मेमोरी) में जमा करता है, जिससे डाउनस्ट्रीम एजेंट पिछले परिणामों तक पहुँच सकते हैं। यह दिखाता है कि MCP टूल्स एजेंटिक वर्कफ़्लोज़ में कैसे सहज रूप से एकीकृत होते हैं — सुपरवाइज़र को यह जानने की आवश्यकता नहीं कि फ़ाइलें *कैसे* पढ़ी जाती हैं, केवल इतना कि `FileAgent` इसे कर सकता है।

#### सुपरवाइज़र कैसे काम करता है

एजेंट बनाने से पहले, आपको MCP ट्रांसपोर्ट को क्लाइंट से जोड़ना और इसे `ToolProvider` के रूप में रैप करना होगा। इस तरह MCP सर्वर के टूल्स आपके एजेंट्स के लिए उपलब्ध हो जाते हैं:

```java
// ट्रांसपोर्ट से एक MCP क्लाइंट बनाएं
McpClient mcpClient = new DefaultMcpClient.Builder()
        .transport(stdioTransport)
        .build();

// क्लाइंट को एक ToolProvider के रूप में रैप करें — यह MCP टूल्स को LangChain4j से जोड़ता है
ToolProvider mcpToolProvider = McpToolProvider.builder()
        .mcpClients(List.of(mcpClient))
        .build();
```


अब आप `mcpToolProvider` को किसी भी एजेंट में इंजेक्ट कर सकते हैं जिसे MCP टूल्स की आवश्यकता हो:

```java
// चरण 1: FileAgent MCP उपकरणों का उपयोग करके फाइलें पढ़ता है
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // फाइल ऑपरेशन के लिए MCP उपकरण होते हैं
        .build();

// चरण 2: ReportAgent संरचित रिपोर्टें बनाता है
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// Supervisor फाइल → रिपोर्ट वर्कफ़्लो का समन्वय करता है
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // अंतिम रिपोर्ट लौटाएं
        .build();

// Supervisor अनुरोध के आधार पर कौन से एजेंट्स को बुलाना है, यह तय करता है
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```


#### FileAgent रनटाइम में MCP टूल्स कैसे खोजता है

आप सोच सकते हैं: **FileAgent को कैसे पता चलता है कि npm फ़ाइल सिस्टम टूल्स का उपयोग कैसे करना है?** जवाब है कि उसे पता नहीं होता — **LLM** रनटाइम पर टूल स्कीमाओं के माध्यम से इसे समझता है।
`FileAgent` इंटरफ़ेस केवल एक **प्रॉम्प्ट परिभाषा** है। इसमें `read_file`, `list_directory`, या किसी अन्य MCP टूल का कोई हार्डकोडेड ज्ञान नहीं है। यहाँ एंड-टू-एंड क्या होता है:

1. **सर्वर लॉन्च:** `StdioMcpTransport` `@modelcontextprotocol/server-filesystem` npm पैकेज को एक चाइल्ड प्रोसेस के रूप में लॉन्च करता है  
2. **टूल डिस्कवरी:** `McpClient` सर्वर को `tools/list` JSON-RPC रिक्वेस्ट भेजता है, जो टूल नाम, विवरण और पैरामीटर स्कीमा के साथ प्रतिक्रिया करता है (जैसे, `read_file` — *"एक फ़ाइल की पूरी सामग्री पढ़ें"* — `{ path: string }`)  
3. **स्कीमा इंजेक्शन:** `McpToolProvider` इन खोजे गए स्कीमाओं को लपेटता है और उन्हें LangChain4j को उपलब्ध कराता है  
4. **LLM निर्णय:** जब `FileAgent.readFile(path)` कॉल किया जाता है, LangChain4j सिस्टम संदेश, उपयोगकर्ता संदेश, **और टूल स्कीमा की सूची** LLM को भेजता है। LLM टूल विवरण पढ़ता है और टूल कॉल उत्पन्न करता है (जैसे, `read_file(path="/some/file.txt")`)  
5. **एक्ज़ीक्यूशन:** LangChain4j टूल कॉल को इंटरसेप्ट करता है, इसे MCP क्लाइंट के माध्यम से Node.js subprocess तक रूट करता है, परिणाम प्राप्त करता है, और उसे LLM को वापस भेजता है  

यह वही [टूल डिस्कवरी](../../../05-mcp) मैकेनिज़्म है जो ऊपर वर्णित है, लेकिन विशेष रूप से एजेंट वर्कफ़्लो पर लागू किया गया है। `@SystemMessage` और `@UserMessage` एनोटेशन LLM के व्यवहार को मार्गदर्शित करते हैं, जबकि इंजेक्टेड `ToolProvider` इसे **क्षमताएँ** प्रदान करता है — LLM दोनों को रनटाइम में जोड़ता है।

#### प्रतिक्रिया रणनीतियाँ

जब आप `SupervisorAgent` कॉन्फ़िगर करते हैं, तो आप निर्दिष्ट करते हैं कि सब-एजेंट्स के कार्य पूरा होने के बाद उपयोगकर्ता को अंतिम उत्तर कैसे तैयार किया जाए। नीचे दिया गया डायग्राम तीन उपलब्ध रणनीतियाँ दर्शाता है — LAST सीधे अंतिम एजेंट आउटपुट लौटाता है, SUMMARY सभी आउटपुट को LLM के माध्यम से संश्लेषित करता है, और SCORED मूल अनुरोध के अनुरूप उच्चतम स्कोर वाला विकल्प चुनता है:

<img src="../../../translated_images/hi/response-strategies.3d0cea19d096bdf9.webp" alt="Response Strategies" width="800"/>

*सुपरवाइजर अंतिम प्रतिक्रिया कैसे तैयार करता है इसके तीन विकल्प — अंतिम एजेंट आउटपुट, संश्लेषित सारांश, या सर्वश्रेष्ठ स्कोर वाला विकल्प।*

उपलब्ध रणनीतियाँ हैं:

| रणनीति | विवरण |
|----------|-------------|
| **LAST** | सुपरवाइजर अंतिम सब-एजेंट या टूल के आउटपुट को लौटाता है। यह तब उपयोगी है जब वर्कफ़्लो में अंतिम एजेंट पूरा, अंतिम उत्तर देने के लिए विशेष रूप से डिज़ाइन किया गया हो (जैसे, शोध पाइपलाइन में "Summary Agent")। |
| **SUMMARY** | सुपरवाइजर अपनी आंतरिक भाषा मॉडल (LLM) का उपयोग पूरी बातचीत और सभी सब-एजेंट आउटपुट का सारांश बनाने के लिए करता है, फिर उस सारांश को अंतिम उत्तर के रूप में लौटाता है। इससे उपयोगकर्ता को एक साफ-सुथरा, समेकित उत्तर मिलता है। |
| **SCORED** | सिस्टम एक आंतरिक LLM का उपयोग LAST प्रतिक्रिया और SUMMARY दोनों को मूल उपयोगकर्ता अनुरोध के खिलाफ स्कोर करने के लिए करता है, और जो भी आउटपुट उच्च स्कोर प्राप्त करता है वह वापस करता है। |

#### आउटपुट को समझना

जब आप डेमो चलाते हैं, तो आप देखेंगे कि सुपरवाइजर विभिन्न एजेंटों को कैसे समन्वित करता है इसका संरचित अवलोकन। प्रत्येक अनुभाग का मतलब निम्न है:

```
======================================================================
  FILE → REPORT WORKFLOW DEMO
======================================================================

This demo shows a clear 2-step workflow: read a file, then generate a report.
The Supervisor orchestrates the agents automatically based on the request.
```
  
**हेडर** वर्कफ़्लो कॉन्सेप्ट का परिचय देता है: फ़ाइल पढ़ने से लेकर रिपोर्ट निर्माण तक केंद्रित पाइपलाइन।  

```
--- WORKFLOW ---------------------------------------------------------
  ┌─────────────┐      ┌──────────────┐
  │  FileAgent  │ ───▶ │ ReportAgent  │
  │ (MCP tools) │      │  (pure LLM)  │
  └─────────────┘      └──────────────┘
   outputKey:           outputKey:
   'fileContent'        'report'

--- AVAILABLE AGENTS -------------------------------------------------
  [FILE]   FileAgent   - Reads files via MCP → stores in 'fileContent'
  [REPORT] ReportAgent - Generates structured report → stores in 'report'
```
  
**वर्कफ़्लो डायग्राम** एजेंटों के बीच डेटा प्रवाह दिखाता है। प्रत्येक एजेंट की विशिष्ट भूमिका होती है:  
- **FileAgent** MCP टूल्स के माध्यम से फ़ाइलें पढ़ता है और कच्ची सामग्री को `fileContent` में संग्रहित करता है  
- **ReportAgent** उस सामग्री का उपयोग करता है और संरचित रिपोर्ट `report` में उत्पन्न करता है  

```
--- USER REQUEST -----------------------------------------------------
  "Read the file at .../file.txt and generate a report on its contents"
```
  
**उपयोगकर्ता अनुरोध** कार्य दिखाता है। सुपरवाइजर इसे पार्स करता है और FileAgent → ReportAgent को कॉल करने का निर्णय लेता है।  

```
--- SUPERVISOR ORCHESTRATION -----------------------------------------
  The Supervisor decides which agents to invoke and passes data between them...

  +-- STEP 1: Supervisor chose -> FileAgent (reading file via MCP)
  |
  |   Input: .../file.txt
  |
  |   Result: LangChain4j is an open-source, provider-agnostic Java framework for building LLM...
  +-- [OK] FileAgent (reading file via MCP) completed

  +-- STEP 2: Supervisor chose -> ReportAgent (generating structured report)
  |
  |   Input: LangChain4j is an open-source, provider-agnostic Java framew...
  |
  |   Result: Executive Summary...
  +-- [OK] ReportAgent (generating structured report) completed
```
  
**सुपरवाइजर समन्वय** 2-स्टेप फ्लो दिखाता है:  
1. **FileAgent** MCP के माध्यम से फ़ाइल पढ़ता है और सामग्री संग्रहित करता है  
2. **ReportAgent** सामग्री प्राप्त करता है और संरचित रिपोर्ट बनाता है  

सुपरवाइजर ने ये निर्णय **स्वायत्त रूप से** उपयोगकर्ता के अनुरोध के आधार पर लिए।  

```
--- FINAL RESPONSE ---------------------------------------------------
Executive Summary
...

Key Points
...

Recommendations
...

--- AGENTIC SCOPE (Data Flow) ----------------------------------------
  Each agent stores its output for downstream agents to consume:
  * fileContent: LangChain4j is an open-source, provider-agnostic Java framework...
  * report: Executive Summary...
```
  
#### एजेंटिक मॉड्यूल फीचर्स की व्याख्या

यह उदाहरण एजेंटिक मॉड्यूल की कई उन्नत विशेषताओं को प्रदर्शित करता है। आइए Agentic Scope और Agent Listeners पर करीब से नज़र डालें।

**Agentic Scope** साझा मेमोरी दिखाता है जहाँ एजेंट अपने परिणाम `@Agent(outputKey="...")` के माध्यम से संग्रहित करते हैं। इससे:  
- बाद के एजेंट पूर्व एजेंटों के आउटपुट तक पहुंच सकते हैं  
- सुपरवाइजर अंतिम प्रतिक्रिया को संश्लेषित कर सकता है  
- आप यह जांच सकते हैं कि प्रत्येक एजेंट ने क्या उत्पादित किया  

नीचे दिया गया डायग्राम दिखाता है कि Agentic Scope साझा मेमोरी के रूप में कैसे काम करता है फ़ाइल-से-रिपोर्ट वर्कफ़्लो में — FileAgent `fileContent` के तहत आउटपुट लिखता है, ReportAgent उसे पढ़ता है और अपना आउटपुट `report` के तहत लिखता है:

<img src="../../../translated_images/hi/agentic-scope.95ef488b6c1d02ef.webp" alt="Agentic Scope Shared Memory" width="800"/>

*Agentic Scope साझा मेमोरी के रूप में कार्य करता है — FileAgent `fileContent` लिखता है, ReportAgent पढ़ता है और `report` लिखता है, और आपका कोड अंतिम परिणाम पढ़ता है।*

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // FileAgent से कच्चा फ़ाइल डेटा
String report = scope.readState("report");            // ReportAgent से संरचित रिपोर्ट
```
  
**Agent Listeners** एजेंट निष्पादन की निगरानी और डिबगिंग सक्षम करते हैं। डेमो में जो कदम-दर-कदम आउटपुट आप देख रहे हैं वह एक AgentListener से आता है जो प्रत्येक एजेंट कॉल में हुक करता है:  
- **beforeAgentInvocation** - तब कॉल किया जाता है जब सुपरवाइजर किसी एजेंट का चयन करता है, जिससे आप देख सकें कि कौन सा एजेंट क्यों चुना गया  
- **afterAgentInvocation** - कोई एजेंट पूरा होने पर इसका परिणाम दिखाता है  
- **inheritedBySubagents** - यदि true, तो श्रोता पूरे पदानुक्रम में सभी एजेंटों की निगरानी करता है  

नीचे दिया गया डायग्राम पूर्ण Agent Listener जीवनचक्र दिखाता है, जिसमें `onError` द्वारा एजेंट निष्पादन के दौरान विफलताओं को कैसे संभाला जाता है शामिल है:

<img src="../../../translated_images/hi/agent-listeners.784bfc403c80ea13.webp" alt="Agent Listeners Lifecycle" width="800"/>

*Agent Listeners निष्पादन जीवनचक्र में हुक करते हैं — देखरेख करें कि एजेंट कब शुरू होते हैं, पूरा होते हैं, या त्रुटियों से मिलते हैं।*

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // सभी उप-एजेंटों तक फैलाएं
    }
};
```
  
सुपरवाइजर पैटर्न के अलावा, `langchain4j-agentic` मॉड्यूल कई शक्तिशाली वर्कफ़्लो पैटर्न प्रदान करता है। नीचे दिया गया डायग्राम सभी पांच दिखाता है — सरल क्रमिक पाइपलाइन से लेकर मानव-इन-द-लूप अनुमोदन वर्कफ़्लो तक:

<img src="../../../translated_images/hi/workflow-patterns.82b2cc5b0c5edb22.webp" alt="Agent Workflow Patterns" width="800"/>

*एजेंट समन्वय के लिए पांच वर्कफ़्लो पैटर्न — सरल क्रमिक पाइपलाइन से मानव-इन-द-लूप अनुमोदन वर्कफ़्लो तक।*

| पैटर्न | विवरण | उपयोग का मामला |
|---------|-------------|----------|
| **Sequential** | एजेंटों को क्रम से निष्पादित करें, आउटपुट अगले तक जाए | पाइपलाइन: शोध → विश्लेषण → रिपोर्ट |
| **Parallel** | एजेंटों को एक साथ चलाएं | स्वतंत्र कार्य: मौसम + समाचार + स्टॉक |
| **Loop** | शर्त पूरी होने तक पुनरावृत्ति करें | गुणवत्ता स्कोरिंग: स्कोर ≥ 0.8 तक पुनः परिष्करण |
| **Conditional** | शर्तों के आधार पर मार्ग निर्धारित करें | वर्गीकरण → विशेषज्ञ एजेंट को मार्गदर्शन |
| **Human-in-the-Loop** | मानव जांच जोड़ें | अनुमोदन वर्कफ़्लो, सामग्री समीक्षा |

## मुख्य संकल्पनाएँ

अब जब आपने MCP और एजेंटिक मॉड्यूल को क्रियान्वित होते देखा, तो आइए संक्षेप में समझें कि कब कौन सा दृष्टिकोण उपयोग करना चाहिए।

MCP का एक बड़ा लाभ इसका बढ़ता हुआ इकोसिस्टम है। नीचे दिया गया डायग्राम दिखाता है कि एक सार्वभौमिक प्रोटोकॉल आपका AI एप्लिकेशन विभिन्न MCP सर्वरों से कैसे जोड़ता है — फ़ाइल सिस्टम और डेटाबेस एक्सेस से लेकर GitHub, ईमेल, वेब स्क्रैपिंग, और बहुत कुछ:

<img src="../../../translated_images/hi/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="MCP Ecosystem" width="800"/>

*MCP सार्वभौमिक प्रोटोकॉल इकोसिस्टम बनाता है — कोई भी MCP-संगत सर्वर किसी भी MCP-संगत क्लाइंट के साथ काम करता है, जिससे टूल साझा करना सक्षम होता है।*

**MCP** तब आदर्श होता है जब आप मौजूदा टूल इकोसिस्टम का लाभ उठाना चाहते हैं, टूल बनाना चाहते हैं जिन्हें कई एप्लिकेशन साझा कर सकें, तृतीय-पक्ष सर्विसेज़ को मानक प्रोटोकॉल के साथ इंटीग्रेट करना चाहते हैं, या बिना कोड बदले टूल इम्प्लीमेंटेशन बदलना चाहते हैं।

**एजेंटिक मॉड्यूल** तब सबसे अच्छा काम करता है जब आप `@Agent` एनोटेशन के साथ उद्घोषणात्मक एजेंट परिभाषाएँ चाहते हैं, वर्कफ़्लो समन्वय (क्रमिक, लूप, समानांतर) की जरूरत होती है, इंटरफ़ेस-आधारित एजेंट डिजाइन पसंद करते हैं बजाय आदेशात्मक कोड के, या कई एजेंटों को संयोजित करते हैं जो `outputKey` के माध्यम से आउटपुट साझा करते हैं।

**सुपरवाइजर एजेंट पैटर्न** तब चमकता है जब वर्कफ़्लो पहले से पूर्वानुमेय नहीं होता और आप चाहते हैं कि LLM निर्णय ले, जब आपके पास कई विशेषज्ञ एजेंट होते हैं जिन्हें गतिशील समन्वय की आवश्यकता होती है, संवादात्मक सिस्टम बनाते हैं जो विभिन्न क्षमताओं को मार्गित करते हैं, या जब आप सबसे लचीला, अनुकूल एजेंट व्यवहार चाहते हैं।

कस्टम `@Tool` विधियों (मॉड्यूल 04 से) और MCP टूल्स (इस मॉड्यूल से) के बीच निर्णय लेने में सहायता के लिए, निम्न तुलना मुख्य ट्रेड-ऑफ़ उजागर करती है — कस्टम टूल आपको एप्लिकेशन-विशिष्ट लॉजिक के लिए टाइट कपलिंग और पूर्ण टाइप सेफ़्टी देते हैं, जबकि MCP टूल मानकीकृत, पुन: उपयोग योग्य इंटीग्रेशन प्रदान करते हैं:

<img src="../../../translated_images/hi/custom-vs-mcp-tools.c4f9b6b1cb65d8a1.webp" alt="Custom Tools vs MCP Tools" width="800"/>

*कब कस्टम @Tool विधियाँ इस्तेमाल करें बनाम MCP टूल्स — एप-विशिष्ट लॉजिक के लिए कस्टम टूल्स पूरी टाइप सेफ़्टी के साथ, मानकीकृत इंटीग्रेशन के लिए MCP टूल्स जो एप्लिकेशन के बीच काम करते हैं।*
