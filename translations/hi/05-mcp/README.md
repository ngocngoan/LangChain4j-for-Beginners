## आप क्या सीखेंगे

आपने संवादात्मक AI बनाया है, प्रांप्ट्स में महारत हासिल की है, दस्तावेज़ों में आधारित प्रतिक्रियाएँ दी हैं, और उपकरणों के साथ एजेंट बनाए हैं। लेकिन ये सभी उपकरण आपके विशिष्ट अनुप्रयोग के लिए कस्टम बनाए गए थे। अगर आप अपने AI को ऐसे मानकीकृत टूल्स के पारिस्थितिकी तंत्र का उपयोग करने के लिए सक्षम कर सकें जिन्हें कोई भी बना और साझा कर सके तो? इस मॉड्यूल में, आप मॉडल संदर्भ प्रोटोकॉल (MCP) और LangChain4j के एजेंटिक मॉड्यूल के साथ ठीक ऐसा करना सीखेंगे। हम पहले एक सरल MCP फाइल रीडर दिखाते हैं और फिर दिखाते हैं कि यह कैसे सुपर्वाइजर एजेंट पैटर्न का उपयोग करते हुए उन्नत एजेंटिक वर्कफ़्लोज़ में आसानी से एकीकृत होता है।

## MCP क्या है?

मॉडल संदर्भ प्रोटोकॉल (MCP) बिल्कुल यही प्रदान करता है - AI अनुप्रयोगों के लिए बाह्य टूल्स की खोज करने और उनका उपयोग करने का एक मानक तरीका। हर डेटा स्रोत या सेवा के लिए कस्टम एकीकरण लिखने के बजाय, आप MCP सर्वरों से कनेक्ट करते हैं जो अपनी क्षमताओं को एक सुसंगत प्रारूप में प्रदर्शित करते हैं। आपका AI एजेंट तब इन टूल्स को स्वचालित रूप से खोज और उपयोग कर सकता है।

निम्न आरेख अंतर दिखाता है — MCP के बिना, हर एकीकरण के लिए कस्टम बिंदु से बिंदु कनेक्शन आवश्यक है; MCP के साथ, एक ही प्रोटोकॉल आपका ऐप किसी भी टूल से जोड़ता है:

<img src="../../../translated_images/hi/mcp-comparison.9129a881ecf10ff5.webp" alt="MCP Comparison" width="800"/>

*MCP से पहले: जटिल बिंदु से बिंदु एकीकरण। MCP के बाद: एक प्रोटोकॉल, अंतहीन संभावनाएं।*

MCP AI विकास में एक मौलिक समस्या को हल करता है: हर एकीकरण कस्टम होता है। GitHub तक पहुंचना चाहते हैं? कस्टम कोड। फाइलें पढ़ना चाहते हैं? कस्टम कोड। डेटाबेस से पूछताछ करना चाहते हैं? कस्टम कोड। और ये कोई भी एकीकरण अन्य AI अनुप्रयोगों के साथ काम नहीं करता।

MCP इसे मानकीकृत करता है। एक MCP सर्वर टूल्स को स्पष्ट विवरण और स्कीमाओं के साथ प्रस्तुत करता है। कोई भी MCP क्लाइंट कनेक्ट हो सकता है, उपलब्ध टूल्स खोज सकता है, और उनका उपयोग कर सकता है। एक बार बनाएं, हर जगह उपयोग करें।

निम्न आरेख इस वास्तुकला को दर्शाता है — एक ही MCP क्लाइंट (आपका AI अनुप्रयोग) कई MCP सर्वरों से कनेक्ट होता है, प्रत्येक अपने टूल्स के सेट को मानक प्रोटोकॉल द्वारा प्रदर्शित करता है:

<img src="../../../translated_images/hi/mcp-architecture.b3156d787a4ceac9.webp" alt="MCP Architecture" width="800"/>

*मॉडल संदर्भ प्रोटोकॉल वास्तुकला - मानकीकृत टूल खोज और निष्पादन*

## MCP कैसे काम करता है

दरअसल, MCP एक परतदार वास्तुकला का उपयोग करता है। आपका जावा एप्लिकेशन (MCP क्लाइंट) उपलब्ध टूल्स खोजता है, ट्रांसपोर्ट लेयर (Stdio या HTTP) के माध्यम से JSON-RPC अनुरोध भेजता है, और MCP सर्वर ऑपरेशन्स को निष्पादित करता है और परिणाम लौटाता है। निम्नलिखित आरेख इस प्रोटोकॉल की प्रत्येक परत को विभाजित करता है:

<img src="../../../translated_images/hi/mcp-protocol-detail.01204e056f45308b.webp" alt="MCP Protocol Detail" width="800"/>

*MCP के आंतरिक कार्यप्रणाली — क्लाइंट टूल्स खोजते हैं, JSON-RPC संदेशों का आदान-प्रदान करते हैं, और ट्रांसपोर्ट लेयर के माध्यम से ऑपरेशन निष्पादित करते हैं।*

**सर्वर-क्लाइंट वास्तुकला**

MCP क्लाइंट-सर्वर मॉडल का उपयोग करता है। सर्वर टूल्स प्रदान करते हैं - फाइल पढ़ना, डेटाबेस क्वेरी करना, API कॉल करना। क्लाइंट (आपका AI एप्लिकेशन) सर्वरों से जुड़ता है और उनके टूल्स का उपयोग करता है।

LangChain4j के साथ MCP उपयोग करने के लिए, यह Maven निर्भरता जोड़ें:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```

**टूल डिस्कवरी**

जब आपका क्लाइंट एक MCP सर्वर से जुड़ता है, तो वह पूछता है "आपके पास कौन से टूल्स हैं?" सर्वर उपलब्ध टूल्स की सूची के साथ जवाब देता है, प्रत्येक के विवरण और पैरामीटर स्कीमा के साथ। आपका AI एजेंट फिर उपयोगकर्ता अनुरोधों के आधार पर निर्णय ले सकता है कि कौन से टूल्स का उपयोग करना है। नीचे के आरेख में यह हैंडशेक दिखाया गया है — क्लाइंट `tools/list` अनुरोध भेजता है और सर्वर अपने उपलब्ध टूल्स विवरण और पैरामीटर स्कीमा के साथ लौटता है:

<img src="../../../translated_images/hi/tool-discovery.07760a8a301a7832.webp" alt="MCP Tool Discovery" width="800"/>

*AI स्टार्टअप पर उपलब्ध टूल्स की खोज करता है — अब यह जानता है कि कौन-कौन सी क्षमताएं उपलब्ध हैं और तय कर सकता है कि कौन सा उपयोग करना है।*

**ट्रांसपोर्ट तंत्र**

MCP विभिन्न ट्रांसपोर्ट तंत्रों का समर्थन करता है। दो विकल्प हैं Stdio (स्थानीय सबप्रोसेस संचार के लिए) और Streamable HTTP (दूरस्थ सर्वरों के लिए)। यह मॉड्यूल Stdio ट्रांसपोर्ट का प्रदर्शन करता है:

<img src="../../../translated_images/hi/transport-mechanisms.2791ba7ee93cf020.webp" alt="Transport Mechanisms" width="800"/>

*MCP ट्रांसपोर्ट तंत्र: दूरस्थ सर्वरों के लिए HTTP, स्थानीय प्रक्रियाओं के लिए Stdio*

**Stdio** - [StdioTransportDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java)

स्थानीय प्रक्रियाओं के लिए। आपका एप्लिकेशन एक सर्वर को एक सबप्रोसेस के रूप में स्पॉन करता है और स्टैण्डर्ड इनपुट/आउटपुट के माध्यम से संवाद करता है। फाइल सिस्टम एक्सेस या कमांड-लाइन टूल्स के लिए उपयोगी।

```java
McpTransport stdioTransport = new StdioMcpTransport.Builder()
    .command(List.of(
        npmCmd, "exec",
        "@modelcontextprotocol/server-filesystem@2025.12.18",
        resourcesDir
    ))
    .logEvents(false)
    .build();
```

`@modelcontextprotocol/server-filesystem` सर्वर निम्नलिखित टूल्स प्रदान करता है, जो आपके द्वारा निर्दिष्ट निर्देशिकाओं तक सीमित हैं:

| टूल | विवरण |
|------|-------------|
| `read_file` | एक फ़ाइल की सामग्री पढ़ें |
| `read_multiple_files` | एक कॉल में कई फाइलें पढ़ें |
| `write_file` | एक फ़ाइल बनाएँ या अधिलेखित करें |
| `edit_file` | लक्षित खोज और प्रतिस्थापन संपादन करें |
| `list_directory` | किसी पथ पर फाइलें और निर्देशिकाएं सूचीबद्ध करें |
| `search_files` | पैटर्न से मेल खाने वाली फाइलों के लिए पुनरावर्ती खोज करें |
| `get_file_info` | फ़ाइल मेटाडेटा प्राप्त करें (आकार, टाइमस्टैम्प, अनुमतियाँ) |
| `create_directory` | एक निर्देशिका बनाएँ (मूल निर्देशिकाओं सहित) |
| `move_file` | एक फ़ाइल या निर्देशिका को स्थानांतरित या पुनर्नामित करें |

निम्न आरेख दिखाता है कि Stdio ट्रांसपोर्ट रनटाइम पर कैसे काम करता है — आपका जावा एप्लिकेशन MCP सर्वर को एक चाइल्ड प्रोसेस के रूप में स्पॉन करता है और वे stdin/stdout पाइप के माध्यम से संवाद करते हैं, बिना किसी नेटवर्क या HTTP के:

<img src="../../../translated_images/hi/stdio-transport-flow.45eaff4af2d81db4.webp" alt="Stdio Transport Flow" width="800"/>

*Stdio ट्रांसपोर्ट कार्य में — आपका एप्लिकेशन MCP सर्वर को एक चाइल्ड प्रोसेस के रूप में स्पॉन करता है और stdin/stdout पाइप के माध्यम से संवाद करता है।*

> - "Stdio ट्रांसपोर्ट कैसे काम करता है और मुझे इसे HTTP के मुकाबले कब उपयोग करना चाहिए?"
> - "LangChain4j MCP सर्वर प्रक्रियाओं के जीवनचक्र का प्रबंधन कैसे करता है?"
> - "AI को फाइल सिस्टम एक्सेस देने के क्या सुरक्षा निहितार्थ हैं?"

## एजेंटिक मॉड्यूल

जबकि MCP मानकीकृत टूल्स प्रदान करता है, LangChain4j का **एजेंटिक मॉड्यूल** टूल्स के संयोजन के लिए एक घोषणात्मक तरीका प्रदान करता है। `@Agent` एनोटेशन और `AgenticServices` आपको इंटरफेस के माध्यम से एजेंट व्यवहार निर्धारित करने देते हैं बजाय अनिवार्य कोड के।

इस मॉड्यूल में, आप **सुपरवाइजर एजेंट** पैटर्न का अन्वेषण करेंगे — एक उन्नत एजेंटिक AI दृष्टिकोण जहाँ एक "सुपरवाइजर" एजेंट उपयोगकर्ता अनुरोधों के आधार पर गतिशील रूप से उप-एजेंट्स को सक्रिय करने का निर्णय लेता है। हम दोनों अवधारणाओं को मिलाकर हमारे उप-एजेंट्स में से एक को MCP-संचालित फाइल पहुँच क्षमताएं देंगे।

एजेंटिक मॉड्यूल उपयोग करने के लिए, यह Maven निर्भरता जोड़ें:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```

> **नोट:** `langchain4j-agentic` मॉड्यूल एक अलग संस्करण संपत्ति (`langchain4j.mcp.version`) का उपयोग करता है क्योंकि यह कोर LangChain4j लाइब्रेरीज से अलग समय-सारिणी पर जारी किया जाता है।

> **⚠️ प्रयोगात्मक:** `langchain4j-agentic` मॉड्यूल **प्रयोगात्मक** है और बदल सकता है। AI सहायक बनाने का स्थिर तरीका अभी भी `langchain4j-core` है कस्टम टूल्स के साथ (मॉड्यूल 04)।

### फाइल संचालन (Stdio)

यह स्थानीय सबप्रोसेस-आधारित टूल्स को प्रदर्शित करता है।

**✅ कोई पूर्वापेक्षाएँ आवश्यक नहीं** - MCP सर्वर स्वचालित रूप से स्पॉन होता है।

**स्टार्ट स्क्रिप्ट्स का उपयोग करना (सिफारिश की जाती है):**

स्टार्ट स्क्रिप्ट्स स्वचालित रूप से रूट `.env` फ़ाइल से पर्यावरण चर लोड करती हैं:

**Bash:**

### सुपरवाइजर एजेंट

**सुपरवाइजर एजेंट पैटर्न** एक **लचीला** एजेंटिक AI रूप है। एक सुपरवाइजर LLM का उपयोग उपयोगकर्ता के अनुरोध के आधार पर स्वायत्त रूप से निर्णय लेने के लिए करता है कि कौन से एजेंट्स को सक्रिय करना है। अगले उदाहरण में, हम MCP-संचालित फाइल एक्सेस को एक LLM एजेंट के साथ संयोजित करते हैं ताकि एक नियंत्रित फाइल पढ़ने → रिपोर्ट वर्कफ़्लो बन सके।

डेमो में, `FileAgent` MCP फाइल सिस्टम टूल्स का उपयोग करके एक फ़ाइल पढ़ता है, और `ReportAgent` एक संरचित रिपोर्ट तैयार करता है जिसमें कार्यकारी सारांश (1 वाक्य), 3 मुख्य बिंदु, और सिफारिशें शामिल हैं। सुपरवाइजर इस प्रवाह को स्वचालित रूप से समन्वयित करता है:

<img src="../../../translated_images/hi/supervisor-agent-pattern.06275a41ae006ac8.webp" alt="Supervisor Agent Pattern" width="800"/>

*सुपरवाइजर अपने LLM का उपयोग यह तय करने के लिए करता है कि कौन से एजेंट्स को किस क्रम में सक्रिय करना है — कोई हार्डकोडेड रूटिंग आवश्यक नहीं।*

यहाँ हमारे फाइल-से-रिपोर्ट पाइपलाइन का ठोस वर्कफ़्लो कैसा दिखता है:

<img src="../../../translated_images/hi/file-report-workflow.649bb7a896800de9.webp" alt="File to Report Workflow" width="800"/>

*FileAgent MCP टूल्स के माध्यम से फ़ाइल पढ़ता है, फिर ReportAgent कच्ची सामग्री को संरचित रिपोर्ट में परिवर्तित करता है।*

प्रत्येक एजेंट अपना आउटपुट **एजेंटिक स्कोप** (साझा मेमोरी) में स्टोर करता है, जिससे डाउनस्ट्रीम एजेंट्स पिछली परिणामों तक पहुंच सकते हैं। यह दिखाता है कि MCP टूल्स एजेंटिक वर्कफ़्लोज़ में निर्बाध रूप से कैसे एकीकृत होते हैं — सुपरवाइजर को यह जानने की जरूरत नहीं कि फ़ाइलें कैसे पढ़ी जाती हैं, केवल यह कि `FileAgent` इसे कर सकता है।

#### सुपरवाइजर कैसे काम करता है

एजेंट बनाने से पहले, आपको MCP ट्रांसपोर्ट को क्लाइंट से कनेक्ट करना होगा और इसे `ToolProvider` के रूप में लपेटना होगा। इस प्रकार MCP सर्वर के टूल्स आपके एजेंट्स के लिए उपलब्ध हो जाते हैं:

```java
// ट्रांसपोर्ट से एक MCP क्लाइंट बनाएं
McpClient mcpClient = new DefaultMcpClient.Builder()
        .transport(stdioTransport)
        .build();

// क्लाइंट को एक ToolProvider के रूप में लपेटें — यह LangChain4j में MCP टूल्स को जोड़ता है
ToolProvider mcpToolProvider = McpToolProvider.builder()
        .mcpClients(List.of(mcpClient))
        .build();
```

अब आप `mcpToolProvider` को किसी भी एजेंट में इंजेक्ट कर सकते हैं जिसे MCP टूल्स की आवश्यकता हो:

```java
// चरण 1: FileAgent MCP टूल्स का उपयोग करके फाइलें पढ़ता है
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // फाइल संचालन के लिए MCP टूल्स मौजूद हैं
        .build();

// चरण 2: ReportAgent संरचित रिपोर्ट बनाता है
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// Supervisor फाइल → रिपोर्ट वर्कफ़्लो का समन्वय करता है
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // अंतिम रिपोर्ट लौटाएं
        .build();

// Supervisor अनुरोध के आधार पर कौन से एजेंट्स को बुलाना है यह तय करता है
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```

#### प्रतिक्रिया रणनीतियाँ

जब आप एक `SupervisorAgent` कॉन्फ़िगर करते हैं, तो आप यह निर्दिष्ट करते हैं कि उप-एजेंट्स के कार्य पूर्ण होने के बाद उपयोगकर्ता को अंतिम उत्तर कैसे प्रस्तुत किया जाए। निम्न आरेख तीन उपलब्ध रणनीतियाँ दिखाता है — LAST अंतिम एजेंट का आउटपुट सीधे लौटाता है, SUMMARY सभी आउटपुट को LLM के माध्यम से संश्लेषित करता है, और SCORED मूल अनुरोध के आधार पर उच्चतम स्कोर प्राप्त आउटपुट चुनता है:

<img src="../../../translated_images/hi/response-strategies.3d0cea19d096bdf9.webp" alt="Response Strategies" width="800"/>

*सुपरवाइजर अपनी अंतिम प्रतिक्रिया कैसे बनाता है इसके लिए तीन रणनीतियाँ — अंतिम एजेंट का आउटपुट, एक संश्लेषित सारांश, या सर्वोत्तम स्कोर वाला विकल्प चुनें।*

उपलब्ध रणनीतियाँ हैं:

| रणनीति | विवरण |
|----------|-------------|
| **LAST** | सुपरवाइजर अंतिम उप-एजेंट या टूल द्वारा लौटाए गए आउटपुट को लौटाता है। यह तब उपयोगी होता है जब वर्कफ़्लो में अंतिम एजेंट पूरी, अंतिम उत्तर देने के लिए विशेष रूप से डिज़ाइन किया गया हो (जैसे शोध पाइपलाइन में "सारांश एजेंट")। |
| **SUMMARY** | सुपरवाइजर अपनी आंतरिक भाषा मॉडल (LLM) का उपयोग पूरी बातचीत और सभी उप-एजेंट आउटपुट का सारांश तैयार करने के लिए करता है, फिर उस सारांश को अंतिम प्रतिक्रिया के रूप में लौटाता है। यह उपयोगकर्ता को एक साफ, समेकित उत्तर प्रदान करता है। |
| **SCORED** | सिस्टम आंतरिक LLM का उपयोग LAST प्रतिक्रिया और बातचीत के SUMMARY दोनों को मूल उपयोगकर्ता अनुरोध के खिलाफ स्कोर करने के लिए करता है, और जो आउटपुट उच्चतम स्कोर प्राप्त करता है उसे लौटाता है। |

#### आउटपुट को समझना

जब आप डेमो चलाएंगे, तो आपको यह दिखेगा कि Supervisor कई एजेंट्स को कैसे समन्वित करता है। यहाँ प्रत्येक सेक्शन का अर्थ है:

```
======================================================================
  FILE → REPORT WORKFLOW DEMO
======================================================================

This demo shows a clear 2-step workflow: read a file, then generate a report.
The Supervisor orchestrates the agents automatically based on the request.
```

**हैडर** वर्कफ़्लो अवधारणा को परिचित कराता है: फाइल पढ़ने से लेकर रिपोर्ट जनरेशन तक का केंद्रीकृत पाइपलाइन।

```
--- WORKFLOW ---------------------------------------------------------
  ┌─────────────┐      ┌──────────────┐
  │  FileAgent  │ ───▶ │ ReportAgent  │
  │ (MCP tools) │      │  (pure LLM)  │
  └─────────────┘      └──────────────┘
   outputKey:           outputKey:
   'fileContent'        'report'

--- AVAILABLE AGENTS -------------------------------------------------
  [FILE]   FileAgent   - Reads files via MCP → stores in 'fileContent'
  [REPORT] ReportAgent - Generates structured report → stores in 'report'
```

**वर्कफ़्लो आरेख** एजेंट्स के बीच डेटा प्रवाह दिखाता है। प्रत्येक एजेंट की एक विशिष्ट भूमिका है:
- **FileAgent** MCP टूल्स का उपयोग करके फाइलें पढ़ता है और कच्चा कंटेंट `fileContent` में संग्रहीत करता है
- **ReportAgent** उस कंटेंट का उपयोग करता है और `report` में एक संरचित रिपोर्ट बनाता है

```
--- USER REQUEST -----------------------------------------------------
  "Read the file at .../file.txt and generate a report on its contents"
```

**यूजर रिक्वेस्ट** कार्य दिखाता है। Supervisor इसे पार्स करता है और FileAgent → ReportAgent को लागू करने का निर्णय लेता है।

```
--- SUPERVISOR ORCHESTRATION -----------------------------------------
  The Supervisor decides which agents to invoke and passes data between them...

  +-- STEP 1: Supervisor chose -> FileAgent (reading file via MCP)
  |
  |   Input: .../file.txt
  |
  |   Result: LangChain4j is an open-source, provider-agnostic Java framework for building LLM...
  +-- [OK] FileAgent (reading file via MCP) completed

  +-- STEP 2: Supervisor chose -> ReportAgent (generating structured report)
  |
  |   Input: LangChain4j is an open-source, provider-agnostic Java framew...
  |
  |   Result: Executive Summary...
  +-- [OK] ReportAgent (generating structured report) completed
```

**Supervisor आर्केस्ट्रेशन** 2-चरणीय फ्लो को क्रियान्वित करता है:
1. **FileAgent** MCP के माध्यम से फाइल पढ़ता है और कंटेंट संग्रहीत करता है
2. **ReportAgent** कंटेंट प्राप्त करता है और संरचित रिपोर्ट बनाता है

Supervisor ने ये निर्णय **स्वायत्त रूप से** उपयोगकर्ता के अनुरोध के आधार पर लिए।

```
--- FINAL RESPONSE ---------------------------------------------------
Executive Summary
...

Key Points
...

Recommendations
...

--- AGENTIC SCOPE (Data Flow) ----------------------------------------
  Each agent stores its output for downstream agents to consume:
  * fileContent: LangChain4j is an open-source, provider-agnostic Java framework...
  * report: Executive Summary...
```

#### Agentic Module फीचर्स का स्पष्टीकरण

यह उदाहरण एजेंटिक मॉड्यूल की कई उन्नत सुविधाओं को दर्शाता है। चलिए Agentic Scope और Agent Listeners पर करीब से नज़र डालते हैं।

**Agentic Scope** साझा मेमोरी दिखाता है जहाँ एजेंट्स ने `@Agent(outputKey="...")` का उपयोग कर अपने परिणाम संग्रहीत किए। इससे:
- बाद के एजेंट्स पूर्व एजेंट्स के आउटपुट तक पहुंच सकते हैं
- Supervisor अंतिम प्रतिक्रिया को संश्लेषित कर सकता है
- आप देख सकते हैं कि हर एजेंट ने क्या उत्पादन किया

नीचे आरेख दिखाता है कि Agentic Scope कैसे फाइल-से-रिपोर्ट वर्कफ़्लो में साझा मेमोरी के रूप में काम करता है—FileAgent अपनी आउटपुट `fileContent` कुंजी के अंतर्गत लिखता है, ReportAgent इसे पढ़ता है और अपनी आउटपुट `report` के अंतर्गत लिखता है:

<img src="../../../translated_images/hi/agentic-scope.95ef488b6c1d02ef.webp" alt="Agentic Scope Shared Memory" width="800"/>

*Agentic Scope साझा मेमोरी के रूप में कार्य करता है — FileAgent `fileContent` लिखता है, ReportAgent इसे पढ़ता है और `report` लिखता है, और आपका कोड अंतिम परिणाम पढ़ता है।*

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // FileAgent से कच्चा फ़ाइल डेटा
String report = scope.readState("report");            // ReportAgent से संरचित रिपोर्ट
```

**Agent Listeners** एजेंट निष्पादन की निगरानी और डीबगिंग सक्षम करते हैं। डेमो में दिखने वाला कदम-दर-कदम आउटपुट एक AgentListener से आता है जो प्रत्येक एजेंट कॉल के साथ जुड़ा होता है:
- **beforeAgentInvocation** - जब Supervisor एक एजेंट चुनता है तब कॉल होता है, जिससे आपको पता चलता है कि कौन सा एजेंट क्यों चुना गया
- **afterAgentInvocation** - जब कोई एजेंट पूरा होता है, तो इसका परिणाम दिखाता है
- **inheritedBySubagents** - यदि true है, तो लिस्नर पूरे पदानुक्रम के सभी एजेंट्स की निगरानी करता है

निम्नलिखित आरेख पूरे Agent Listener जीवनचक्र को दिखाता है, जिसमें `onError` कैसे एजेंट निष्पादन के दौरान समस्याओं को संभालता है:

<img src="../../../translated_images/hi/agent-listeners.784bfc403c80ea13.webp" alt="Agent Listeners Lifecycle" width="800"/>

*Agent Listeners निष्पादन जीवनचक्र में जुड़ते हैं — यह तभी मॉनिटर करते हैं जब एजेंट शुरू होता है, पूरा होता है, या त्रुटि आती है।*

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // सभी उप-एजेंट्स तक प्रचारित करें
    }
};
```

Supervisor पैटर्न से परे, `langchain4j-agentic` मॉड्यूल कई शक्तिशाली वर्कफ़्लो पैटर्न प्रदान करता है। नीचे आरेख में सभी पाँच दिखाए गए हैं—सरल क्रमशः पाइपलाइन से लेकर मानव जांच वाली अनुमोदन प्रक्रियाओं तक:

<img src="../../../translated_images/hi/workflow-patterns.82b2cc5b0c5edb22.webp" alt="Agent Workflow Patterns" width="800"/>

*एजेंट्स के समन्वय के लिए पाँच वर्कफ़्लो पैटर्न — सरल क्रमशः पाइपलाइन से लेकर मानव जांच वाली अनुमोदन कार्यप्रवाह तक।*

| पैटर्न | विवरण | उपयोग केस |
|---------|-------------|----------|
| **Sequential** | एजेंट्स को क्रम में चलाए, आउटपुट अगले तक जाता है | पाइपलाइंस: रिसर्च → एनालाइसिस → रिपोर्ट |
| **Parallel** | एजेंट्स को एक साथ चलाएं | स्वतंत्र कार्य: मौसम + समाचार + स्टॉक्स |
| **Loop** | शर्त पूरी होने तक पुनरावृत्ति करें | गुणवत्ता स्कोरिंग: स्कोर ≥ 0.8 तक परिष्कार |
| **Conditional** | शर्तों के आधार पर मार्गदर्शन करें | वर्गीकरण → विशेषज्ञ एजेंट को मार्गदर्शन |
| **Human-in-the-Loop** | मानव जांच पॉइंट जोड़ें | अनुमोदन कार्यप्रवाह, सामग्री समीक्षा |

## मुख्य अवधारणाएँ

अब जब आपने MCP और एजेंटिक मॉड्यूल का प्रयोग देखा है, तो आइए सारांशित करें कि कब कौन सा दृष्टिकोण उपयोग करें।

MCP का सबसे बड़ा लाभ इसका बढ़ता हुआ पारिस्थितिकी तंत्र है। नीचे चित्र दिखाता है कि कैसे एक सार्वभौमिक प्रोटोकॉल आपके AI एप्लिकेशन को विभिन्न MCP सर्वर से जोड़ता है—फाइल सिस्टम और डेटाबेस एक्सेस से लेकर GitHub, ईमेल, वेब स्क्रैपिंग, और अधिक:

<img src="../../../translated_images/hi/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="MCP Ecosystem" width="800"/>

*MCP एक सार्वभौमिक प्रोटोकॉल पारिस्थितिकी तंत्र बनाता है — कोई भी MCP-अनुकूल सर्वर किसी भी MCP-अनुकूल क्लाइंट के साथ काम करता है, जिससे उपकरणों का साझा उपयोग संभव होता है।*

**MCP** तब आदर्श है जब आप मौजूदा टूल पारिस्थितिक तंत्र का लाभ लेना चाहते हैं, ऐसे उपकरण बनाना चाहते हैं जिन्हें कई एप्लिकेशन साझा कर सकें, मानक प्रोटोकॉल के साथ तृतीय पक्ष सेवाओं को एकीकृत करना चाहते हैं, या बिना कोड बदले टूल लागू करना बदलना चाहते हैं।

**Agentic मॉड्यूल** सबसे अच्छी तरह से तब काम करता है जब आप `@Agent` एनोटेशन के साथ घोषणात्मक एजेंट परिभाषा चाहते हैं, वर्कफ़्लो आर्केस्ट्रेशन (क्रमिक, लूप, समांतर) की आवश्यकता हो, इंटरफेस-आधारित एजेंट डिजाइन पसंद करते हैं, या अनेक एजेंट्स को `outputKey` के माध्यम से आउटपुट साझा करने के लिए संयोजित कर रहे हों।

**Supervisor Agent पैटर्न** तब चमकता है जब वर्कफ़्लो पहले से अनुमानित नहीं होता और आप चाहते हैं कि LLM निर्णय ले, जब आपके पास कई विशेषज्ञ एजेंट्स हों जिन्हें गतिशील समन्वय की आवश्यकता हो, जब आप वार्तालाप प्रणाली बनाना चाहते हैं जो विभिन्न क्षमताओं को मार्गदर्शन करती हो, या जब आप सबसे लचीला, अनुकूल एजेंट व्यवहार चाहते हों।

मॉड्यूल 04 के कस्टम `@Tool` मेथड्स और इस मॉड्यूल के MCP टूल्स के बीच निर्णय में मदद के लिए, निम्न तुलना मुख्य व्यापार-बंदियों को उजागर करती है — कस्टम टूल्स आपको तंग जोड़ी और पूर्ण टाइप सुरक्षा देते हैं, जबकि MCP टूल्स मानकीकृत, पुन: प्रयोज्य एकीकरण प्रदान करते हैं:

<img src="../../../translated_images/hi/custom-vs-mcp-tools.c4f9b6b1cb65d8a1.webp" alt="Custom Tools vs MCP Tools" width="800"/>

*कब कस्टम @Tool मेथड्स बनाम MCP टूल्स उपयोग करें — कस्टम टूल्स ऐप-विशिष्ट लॉजिक के लिए पूर्ण टाइप सुरक्षा के साथ, MCP टूल्स मानकीकृत एकीकरण के लिए जो ऐप्स में काम करते हैं।*
