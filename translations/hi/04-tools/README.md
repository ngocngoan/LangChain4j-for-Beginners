## आप क्या सीखेंगे

अब तक, आपने सीखा है कि AI के साथ संवाद कैसे करें, प्रभावी रूप से प्रॉम्प्ट कैसे बनाएं, और उत्तरों को आपके दस्तावेज़ों से कैसे आधारित करें। लेकिन एक मूलभूत सीमा है: भाषा मॉडल केवल टेक्स्ट उत्पन्न कर सकते हैं। वे मौसम की जानकारी जांच नहीं सकते, गणना नहीं कर सकते, डेटाबेस क्वेरी नहीं कर सकते, या बाहरी प्रणालियों के साथ इंटरैक्ट नहीं कर सकते।

टूल इसे बदल देते हैं। मॉडल को फ़ंक्शनों तक पहुंच देकर जिन्हें वह कॉल कर सकता है, आप इसे एक टेक्स्ट जनरेटर से एक एजेंट में बदल देते हैं जो क्रियाएँ कर सकता है। मॉडल तय करता है कि उसे कब टूल की ज़रूरत है, कौन सा टूल इस्तेमाल करना है, और कौन से पैरामीटर पास करने हैं। आपका कोड फ़ंक्शन को निष्पादित करता है और परिणाम लौटाता है। मॉडल उस परिणाम को अपनी प्रतिक्रिया में शामिल करता है।

## टूल वाले AI एजेंट्स को समझना

> **📝 नोट:** इस मॉड्यूल में "एजेंट्स" शब्द का अर्थ ऐसे AI सहायकों से है जिन्हें टूल-कॉलिंग क्षमताओं से संवर्धित किया गया है। यह **Agentic AI** पैटर्न (स्वायत्त एजेंट, योजनाबद्ध, मेमोरी, और बहु-चरण तर्क के साथ) से भिन्न है जिन्हें हम [मॉड्यूल 05: MCP](../05-mcp/README.md) में कवर करेंगे।

टूल के बिना, एक भाषा मॉडल केवल अपने प्रशिक्षण डेटा से टेक्स्ट उत्पन्न कर सकता है। उससे वर्तमान मौसम पूछो, तो उसे अनुमान लगाना पड़ता है। टूल दें, और वह मौसम API कॉल कर सकता है, गणना कर सकता है, या डेटाबेस क्वेरी कर सकता है — फिर उन वास्तविक परिणामों को अपनी प्रतिक्रिया में शामिल कर सकता है।

<img src="../../../translated_images/hi/what-are-tools.724e468fc4de64da.webp" alt="विदाउट टूल्स बनाम विद टूल्स" width="800"/>

*टूल के बिना मॉडल सिर्फ अनुमान लगा सकता है — टूल के साथ वह API कॉल कर सकता है, गणना कर सकता है, और वास्तविक समय का डेटा लौटाता है।*

टूल वाले AI एजेंट एक **Reasoning and Acting (ReAct)** पैटर्न का पालन करते हैं। मॉडल केवल जवाब नहीं देता — बल्कि सोचता है कि उसे क्या चाहिए, टूल कॉल करके कार्य करता है, परिणाम देखता है, फिर निर्णय लेता है कि पुनः कार्य करना है या अंतिम उत्तर देना है:

1. **तर्क करें** — एजेंट उपयोगकर्ता के प्रश्न का विश्लेषण करता है और यह निर्धारित करता है कि उसे किस जानकारी की जरूरत है
2. **कार्य करें** — एजेंट सही टूल चुनता है, सही पैरामीटर बनाता है, और उसे कॉल करता है
3. **निरीक्षण करें** — एजेंट टूल का आउटपुट प्राप्त करता है और परिणाम का मूल्यांकन करता है
4. **दोहराएँ या जवाब दें** — यदि अधिक डेटा चाहिए तो एजेंट पुनः चक्र में जाता है; अन्यथा, एक प्राकृतिक भाषा जवाब तैयार करता है

<img src="../../../translated_images/hi/react-pattern-detail.96a5efeeb6dd2f61.webp" alt="ReAct पैटर्न" width="800"/>

*ReAct चक्र — एजेंट सोचे कि क्या करना है, टूल कॉल करे, परिणाम देखे, और अंतिम उत्तर तक दोहराए।*

यह सब स्वचालित होता है। आप टूल और उनके विवरण परिभाषित करते हैं। मॉडल यह तय करता है कि कब और कैसे उनका उपयोग करना है।

## टूल कॉलिंग कैसे काम करती है

### टूल परिभाषाएँ

[WeatherTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) | [TemperatureTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/TemperatureTool.java)

आप स्पष्ट विवरण और पैरामीटर विनिर्देशों के साथ फ़ंक्शंस परिभाषित करते हैं। मॉडल सिस्टम प्रॉम्प्ट में इन विवरणों को देखता है और समझता है कि प्रत्येक टूल क्या करता है।

```java
@Component
public class WeatherTool {
    
    @Tool("Get the current weather for a location")
    public String getCurrentWeather(@P("Location name") String location) {
        // आपकी मौसम खोज तर्क
        return "Weather in " + location + ": 22°C, cloudy";
    }
}

@AiService
public interface Assistant {
    String chat(@MemoryId String sessionId, @UserMessage String message);
}

// सहायक स्वतः Spring Boot द्वारा कनेक्ट किया गया है:
// - ChatModel बीन
// - @Component वर्गों से सभी @Tool विधियाँ
// - सत्र प्रबंधन के लिए ChatMemoryProvider
```

नीचे दी गई आरेख हर एनोटेशन को तोड़ती है और दिखाती है कि कैसे हर भाग AI को टूल कॉल करने के लिए benodigde वक्त और पास करने वाले आर्गुमेंट्स समझाने में मदद करता है:

<img src="../../../translated_images/hi/tool-definitions-anatomy.f6468546037cf28b.webp" alt="टूल परिभाषाओं की संरचना" width="800"/>

*एक टूल परिभाषा की संरचना — @Tool AI को बताता है कब टूल उपयोग करना है, @P प्रत्येक पैरामीटर का विवरण देता है, और @AiService स्टार्टअप पर सब कुछ वायर्ड करता है।*

> - "एक अच्छा टूल विवरण क्या होता है जो AI को इसे सही तरीके से उपयोग करने में मदद करता है?"
> - "मैं टूल इम्प्लीमेंटेशन में API त्रुटियों और रेट लिमिट्स को कैसे संभालूँ?"

### निर्णय लेना

जब उपयोगकर्ता पूछता है "Seattle में मौसम कैसा है?", तो मॉडल कोई भी टूल यादृच्छिक रूप से नहीं चुनता। यह उपयोगकर्ता की मंशा की तुलना सभी टूल विवरणों से करता है, प्रत्येक को प्रासंगिकता के लिए स्कोर करता है, और सबसे उपयुक्त चुनता है। फिर वह सही पैरामीटर के साथ एक संरचित फ़ंक्शन कॉल बनाता है — इस मामले में, `location` को `"Seattle"` सेट करता है।

यदि कोई टूल उपयोगकर्ता के अनुरोध से मेल नहीं खाता, तो मॉडल अपने ज्ञान से जवाब देता है। यदि कई टूल मेल खाते हैं, तो वह सबसे विशिष्ट टूल चुनता है।

<img src="../../../translated_images/hi/decision-making.409cd562e5cecc49.webp" alt="AI कैसे टूल चुनता है" width="800"/>

*मॉडल उपलब्ध प्रत्येक टूल को उपयोगकर्ता की मंशा के मुकाबले परखता है और सबसे अच्छी मैचिंग का चयन करता है — इसलिए स्पष्ट, विशिष्ट टूल विवरण लिखना महत्वपूर्ण है।*

### प्रतिक्रिया निर्माण

मॉडल मौसम डेटा प्राप्त करता है और इसे उपयोगकर्ता के लिए एक प्राकृतिक भाषा प्रतिक्रिया में स्वरूपित करता है।

### आर्किटेक्चर: स्प्रिंग बूट ऑटो-वायरिंग

यह मॉड्यूल LangChain4j के स्प्रिंग बूट एकीकरण का उपयोग करता है जिसमें डिक्लेरेटिव `@AiService` इंटरफेस होते हैं। स्टार्टअप पर स्प्रिंग बूट हर `@Component` को खोजता है जिसमें `@Tool` मेथड्स होते हैं, आपका `ChatModel` बीन, और `ChatMemoryProvider` — फिर उन्हें एकल `Assistant` इंटरफेस में बिना किसी बोइलरप्लेट के जोड़ देता है।

<img src="../../../translated_images/hi/spring-boot-wiring.151321795988b04e.webp" alt="स्प्रिंग बूट ऑटो-वायरिंग आर्किटेक्चर" width="800"/>

*@AiService इंटरफेस ChatModel, टूल कंपोनेंट्स, और मेमोरी प्रोवाइडर को जोड़ता है — स्प्रिंग बूट सभी वायरिंग को ऑटोमैटिक रूप से संभालता है।*

इस दृष्टिकोण के मुख्य लाभ:

- **स्प्रिंग बूट ऑटो-वायरिंग** — ChatModel और टूल्स अपने आप इंजेक्ट होते हैं
- **@MemoryId पैटर्न** — स्वचालित सत्र आधारित मेमोरी प्रबंधन
- **एकल उदाहरण** — Assistant एक बार बनता है और बेहतर प्रदर्शन के लिए दोबारा उपयोग होता है
- **टाइप-सुरक्षित निष्पादन** — जावा मेथड्स सीधे टाइप कन्वर्शन के साथ कॉल होते हैं
- **मल्टी-टर्न ऑर्केस्ट्रेशन** — टूल चेनिंग को स्वचालित रूप से संभालता है
- **शून्य बोइलरप्लेट** — कोई मैनुअल `AiServices.builder()` कॉल या मेमोरी हैशमैप नहीं

वैकल्पिक तरीके (मैनुअल `AiServices.builder()`) अधिक कोड की जरूरत होती है और स्प्रिंग बूट एकीकरण लाभों से चूक जाते हैं।

## टूल चेनिंग

**टूल चेनिंग** — टूल-आधारित एजेंट्स की असली ताकत तब दिखती है जब एक ही प्रश्न के लिए कई टूल्स की ज़रूरत होती है। पूछें "Seattle में तापमान फ़ारेनहाइट में क्या है?" और एजेंट अपने आप दो टूल्स को श्रृंखला बनाता है: पहले वह `getCurrentWeather` कॉल करता है जो तापमान सेल्सियस में देता है, फिर वह उस मान को `celsiusToFahrenheit` में पास करता है — यह सब एक ही संवाद टर्न में होता है।

<img src="../../../translated_images/hi/tool-chaining-example.538203e73d09dd82.webp" alt="टूल चेनिंग उदाहरण" width="800"/>

*टूल चेनिंग क्रिया में — एजेंट पहले getCurrentWeather कॉल करता है, फिर सेल्सियस परिणाम celsiusToFahrenheit में पास करता है, और एक संयुक्त उत्तर देता है।*

**सौम्य विफलताएँ** — उस शहर के मौसम पूछें जो नकली डेटा में नहीं है। टूल त्रुटि संदेश लौटाता है, और AI बताता है कि वह मदद नहीं कर सकता न कि क्रैश हो। टूल सुरक्षित रूप से विफल होते हैं। नीचे आरेख दो दृष्टिकोणों की तुलना करता है — सही त्रुटि प्रबंधन के साथ, एजेंट अपवाद पकड़ता है और सहायक प्रतिक्रिया देता है, बिना इसके एप्लिकेशन क्रैश हो जाता है:

<img src="../../../translated_images/hi/error-handling-flow.9a330ffc8ee0475c.webp" alt="त्रुटि प्रबंधन प्रवाह" width="800"/>

*जब टूल विफल होता है, एजेंट त्रुटि पकड़ता है और क्रैश होने के बजाय मददगार स्पष्टीकरण के साथ प्रतिक्रिया करता है।*

यह सब एक ही संवाद टर्न में होता है। एजेंट स्वायत्त रूप से कई टूल कॉल का समन्वय करता है।

## एप्लिकेशन का उपयोग

एप्लिकेशन एक वेब इंटरफ़ेस प्रदान करता है जहां आप मौसम और तापमान रूपांतरण टूल्स तक पहुँच वाले AI एजेंट के साथ इंटरैक्ट कर सकते हैं। इंटरफ़ेस इस प्रकार दिखता है — यह त्वरित शुरुआत के उदाहरण और अनुरोध भेजने के लिए चैट पैनल शामिल करता है:
<a href="images/tools-homepage.png"><img src="../../../translated_images/hi/tools-homepage.4b4cd8b2717f9621.webp" alt="एआई एजेंट उपकरण इंटरफ़ेस" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*एआई एजेंट उपकरण इंटरफ़ेस - उपकरणों के साथ बातचीत के लिए त्वरित उदाहरण और चैट इंटरफ़ेस*

### सरल उपकरण उपयोग आज़माएँ

एक सरल अनुरोध से शुरू करें: "100 डिग्री फ़ारेनहाइट को सेल्सियस में कनवर्ट करें"। एजेंट पहचानता है कि इसे तापमान रूपांतरण उपकरण की आवश्यकता है, सही पैरामीटर के साथ इसे कॉल करता है, और परिणाम लौटाता है। ध्यान दें कि यह कितना स्वाभाविक लगता है - आपने यह निर्दिष्ट नहीं किया कि कौन सा उपकरण उपयोग करना है या इसे कैसे कॉल करना है।

### उपकरण श्रृंखला का परीक्षण करें

अब कुछ अधिक जटिल कोशिश करें: "सीएटल में मौसम क्या है और इसे फ़ारेनहाइट में कनवर्ट करें?" देखें कि एजेंट इसे चरणों में कैसे करता है। पहले यह मौसम प्राप्त करता है (जो सेल्सियस में आता है), फिर पहचानता है कि इसे फ़ारेनहाइट में कनवर्ट करना है, रूपांतरण उपकरण को कॉल करता है, और दोनों परिणामों को एक उत्तर में जोड़ता है।

### बातचीत प्रवाह देखें

चैट इंटरफ़ेस बातचीत का इतिहास बनाए रखता है, जिससे आप बहु-चरण इंटरैक्शन कर सकते हैं। आप सभी पिछले प्रश्न और उत्तर देख सकते हैं, जिससे बातचीत का ट्रैक रखना और समझना आसान होता है कि कैसे एजेंट कई एक्सचेंजों में संदर्भ बनाता है।

<a href="images/tools-conversation-demo.png"><img src="../../../translated_images/hi/tools-conversation-demo.89f2ce9676080f59.webp" alt="एकाधिक उपकरण कॉल के साथ बातचीत" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*सरल रूपांतरण, मौसम देखअप, और उपकरण श्रृंखला को दर्शाने वाली बहु-चरण बातचीत*

### विभिन्न अनुरोधों के साथ प्रयोग करें

विभिन्न संयोजन आज़माएँ:
- मौसम जांच: "टोक्यो में मौसम क्या है?"
- तापमान रूपांतरण: "25°C कितने केल्विन है?"
- संयुक्त प्रश्न: "पेरिस में मौसम जांचें और बताएं कि क्या यह 20°C से ऊपर है"

ध्यान दें कि एजेंट प्राकृतिक भाषा को कैसे समझता है और इसे उपयुक्त उपकरण कॉल में कैसे मैप करता है।

## मुख्य अवधारणाएँ

### ReAct पैटर्न (तर्क और क्रिया)

एजेंट तर्क (निर्णय लेना कि क्या करना है) और क्रिया (उपकरणों का उपयोग करना) के बीच वैकल्पिक होता है। यह पैटर्न स्वायत्त समस्या-समाधान को सक्षम बनाता है बजाय केवल निर्देशों के जवाब देने के।

### उपकरण वर्णन महत्वपूर्ण हैं

आपके उपकरण विवरण की गुणवत्ता सीधे प्रभावित करती है कि एजेंट उन्हें कितना अच्छी तरह उपयोग करता है। स्पष्ट, विशिष्ट विवरण मॉडल को समझने में मदद करते हैं कि कब और कैसे प्रत्येक उपकरण को कॉल करना है।

### सत्र प्रबंधन

`@MemoryId` एनोटेशन स्वचालित सत्र-आधारित मेमोरी प्रबंधन सक्षम करता है। प्रत्येक सत्र ID का अपना `ChatMemory` उदाहरण होता है जिसे `ChatMemoryProvider` बीन द्वारा प्रबंधित किया जाता है, इसलिए कई उपयोगकर्ता एक साथ एजेंट से बातचीत कर सकते हैं बिना उनकी बातचीत एक-दूसरे में मिलें। निम्नलिखित आरेख दिखाता है कि कैसे कई उपयोगकर्ताओं को उनके सत्र IDs के आधार पर अलग मेमोरी स्टोर में रूट किया जाता है:

<img src="../../../translated_images/hi/session-management.91ad819c6c89c400.webp" alt="@MemoryId के साथ सत्र प्रबंधन" width="800"/>

*प्रत्येक सत्र ID एक पृथक बातचीत इतिहास से जुड़ा होता है — उपयोगकर्ता कभी भी एक-दूसरे के संदेश नहीं देखते।*

### त्रुटि प्रबंधन

उपकरण विफल हो सकते हैं — API टाइमआउट, पैरामीटर गलत हो सकते हैं, बाहरी सेवाएं बंद हो सकती हैं। उत्पादन एजेंटों को त्रुटि प्रबंधन की आवश्यकता होती है ताकि मॉडल समस्याओं को समझा सके या विकल्प प्रयास कर सके बजाय कि पूरी एप्लिकेशन क्रैश हो जाए। जब कोई उपकरण अपवाद फेंकता है, तो LangChain4j इसे पकड़ता है और त्रुटि संदेश मॉडल को वापस भेजता है, जो फिर प्राकृतिक भाषा में समस्या समझा सकता है।

## उपलब्ध उपकरण

निम्न आरेख दिखाता है कि आप कौन से व्यापक उपकरण पारिस्थितिकी तंत्र बना सकते हैं। यह मॉड्यूल मौसम और तापमान उपकरणों का प्रदर्शन करता है, लेकिन वही `@Tool` पैटर्न किसी भी जावा मेथड के लिए काम करता है — डेटाबेस क्वेरी से लेकर भुगतान प्रसंस्करण तक।

*कोई भी जावा मेथड जिसे @Tool एनोटेट किया गया है वह एआई के लिए उपलब्ध हो जाता है — यह पैटर्न डेटाबेस, API, ईमेल, फाइल ऑपरेशन और अधिक तक बढ़ाया जा सकता है।*

## उपकरण vs RAG

मॉड्यूल 03 और 04 दोनों एआई की क्षमताओं को बढ़ाते हैं, लेकिन मौलिक रूप से अलग तरीकों से। RAG मॉडल को **ज्ञान** तक पहुंच देता है दस्तावेज़ों को पुनः प्राप्त करके। उपकरण मॉडल को **क्रियाएं** लेने की क्षमता देते हैं फ़ंक्शन्स को कॉल करके। नीचे दिया गया आरेख इन दोनों दृष्टिकोणों की तुलना करता है — यह दिखाते हुए कि प्रत्येक कार्यप्रवाह कैसे संचालित होता है और उनके बीच क्या ट्रेड-ऑफ हैं:

<img src="../../../translated_images/hi/tools-vs-rag.ad55ce10d7e4da87.webp" alt="उपकरण बनाम RAG तुलना" width="800"/>

*RAG स्थैतिक दस्तावेज़ों से जानकारी पुनः प्राप्त करता है — उपकरण क्रियाएं निष्पादित करते हैं और गतिशील, वास्तविक समय डेटा लाते हैं। कई उत्पादन सिस्टम दोनों को संयोजित करते हैं।*

व्यवहार में, कई उत्पादन सिस्टम दोनों दृष्टिकोणों को मिलाकर प्रयोग करते हैं: आपके दस्तावेज़ों में उत्तरों को आधार बनाने के लिए RAG, और लाइव डेटा लाने या ऑपरेशन करने के लिए उपकरण।
