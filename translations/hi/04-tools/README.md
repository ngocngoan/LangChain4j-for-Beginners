## आप क्या सीखेंगे

अब तक, आपने सीखा है कि AI के साथ कैसे संवाद करें, प्रभावी ढंग से प्रॉम्प्ट बनाएं, और अपनी दस्तावेज़ों को उत्तरों के आधार के रूप में उपयोग करें। लेकिन एक मौलिक सीमा अभी भी है: भाषा मॉडल केवल टेक्स्ट उत्पन्न कर सकते हैं। वे मौसम की जांच नहीं कर सकते, गणना नहीं कर सकते, डेटाबेस से क्वेरी नहीं कर सकते, या बाहरी प्रणालियों के साथ बातचीत नहीं कर सकते।

उपकरण इसे बदल देते हैं। मॉडल को उन कार्यों का उपयोग करने की अनुमति देकर जिन्हें यह कॉल कर सकता है, आप इसे एक टेक्स्ट जेनरेटर से एक ऐसे एजेंट में बदल देते हैं जो कार्य कर सके। मॉडल तय करता है कि उसे कब कोई उपकरण चाहिए, कौन सा उपकरण उपयोग करना है, और कौन से पैरामीटर देना हैं। आपका कोड उस फ़ंक्शन को निष्पादित करता है और परिणाम लौटाता है। मॉडल उस परिणाम को अपनी प्रतिक्रिया में सम्मिलित करता है।

## उपकरणों के साथ AI एजेंट्स को समझना

> **📝 नोट:** इस मॉड्यूल में "एजेंट्स" से तात्पर्य AI असिस्टेंट्स से है, जिन्हें उपकरण-कॉलिंग क्षमताओं के साथ बढ़ाया गया है। यह [Module 05: MCP](../05-mcp/README.md) में कवर किए जाने वाले **Agentic AI** पैटर्न (स्वायत्त एजेंट जो योजना, मेमोरी, और बहु-चरण तर्क करते हैं) से अलग है।

बिना उपकरणों के, एक भाषा मॉडल केवल अपने प्रशिक्षण डेटा से टेक्स्ट उत्पन्न कर सकता है। यदि आप उससे वर्तमान मौसम पूछते हैं, तो उसे अनुमान लगाना होगा। उपकरण दें, तो वह मौसम API कॉल कर सकता है, गणना कर सकता है, या डेटाबेस से क्वेरी कर सकता है — फिर उन वास्तविक परिणामों को अपनी प्रतिक्रिया में जोड़ सकता है।

<img src="../../../translated_images/hi/what-are-tools.724e468fc4de64da.webp" alt="उपकरणों के बिना बनाम उपकरणों के साथ" width="800"/>

*उपकरणों के बिना मॉडल केवल अनुमान लगा सकता है — उपकरणों के साथ यह API कॉल कर सकता है, गणना चला सकता है, और वास्तविक समय के डेटा लौटा सकता है।*

उपकरणों वाले AI एजेंट एक **तर्क और क्रिया (ReAct)** पैटर्न का पालन करते हैं। मॉडल केवल प्रतिक्रिया नहीं देता — यह सोचता है कि क्या चाहिए, एक उपकरण कॉल करके कार्य करता है, परिणाम देखता है, और फिर तय करता है कि फिर से कार्य करे या अंतिम उत्तर प्रदान करे:

1. **तर्क करें** — एजेंट उपयोगकर्ता के प्रश्न का विश्लेषण करता है और तय करता है उसे कौन-सी जानकारी चाहिए
2. **कार्रवाई करें** — एजेंट सही उपकरण चुनता है, सही पैरामीटर बनाता है, और उसे कॉल करता है
3. **निरीक्षण करें** — एजेंट उपकरण का आउटपुट प्राप्त करता है और परिणाम का मूल्यांकन करता है
4. **दोहराव या प्रतिक्रिया दें** — यदि अधिक डेटा चाहिए, तो एजेंट फिर से लूप में जाता है; अन्यथा, यह प्राकृतिक भाषा उत्तर बनाता है

<img src="../../../translated_images/hi/react-pattern-detail.96a5efeeb6dd2f61.webp" alt="ReAct पैटर्न" width="800"/>

*ReAct चक्र — एजेंट तय करता है कि क्या करना है, उपकरण कॉल करता है, परिणाम देखता है, और जब तक अंतिम उत्तर प्रस्तुत न करे तब तक लूप करता है।*

यह सब स्वचालित होता है। आप उपकरण और उनके विवरण परिभाषित करते हैं। मॉडल यह निर्णय लेता है कि कब और कैसे उन्हें उपयोग करना है।

## उपकरण कॉलिंग कैसे काम करती है

### उपकरण परिभाषाएँ

[WeatherTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) | [TemperatureTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/TemperatureTool.java)

आप स्पष्ट विवरण और पैरामीटर विनिर्देशों के साथ फ़ंक्शन परिभाषित करते हैं। मॉडल इन विवरणों को सिस्टम प्रॉम्प्ट में देखता है और समझता है कि प्रत्येक उपकरण क्या करता है।

```java
@Component
public class WeatherTool {
    
    @Tool("Get the current weather for a location")
    public String getCurrentWeather(@P("Location name") String location) {
        // आपका मौसम खोज तर्क
        return "Weather in " + location + ": 22°C, cloudy";
    }
}

@AiService
public interface Assistant {
    String chat(@MemoryId String sessionId, @UserMessage String message);
}

// सहायक स्वचालित रूप से Spring Boot द्वारा वायर किया गया है:
// - ChatModel बीन
// - @Component क्लासों से सभी @Tool मेथड्स
// - सत्र प्रबंधन के लिए ChatMemoryProvider
```

नीचे दिया गया विवरण हर एनोटेशन को तोड़ता है और दिखाता है कि कैसे हर हिस्सा AI को यह समझाने में मदद करता है कि कब उपकरण कॉल करना है और कौन से तर्क पास करने हैं:

<img src="../../../translated_images/hi/tool-definitions-anatomy.f6468546037cf28b.webp" alt="उपकरण परिभाषाओं की संरचना" width="800"/>

*उपकरण परिभाषा की संरचना — @Tool AI को बताता है कब इसे उपयोग करना है, @P प्रत्येक पैरामीटर का विवरण देता है, और @AiService स्टार्टअप पर सब कुछ जोड़ता है।*

> **🤖 GitHub Copilot चैट के साथ आज़माएँ:** [`WeatherTool.java`](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) खोलें और पूछें:
> - "मैं असली मौसम API जैसे OpenWeatherMap को मॉक डेटा की जगह कैसे एकीकृत करूं?"
> - "एक अच्छा उपकरण विवरण क्या होता है जो AI को इसे सही तरीके से इस्तेमाल करने में मदद करता है?"
> - "मैं उपकरण कार्यान्वयन में API त्रुटियों और दर सीमाओं को कैसे संभालूं?"

### निर्णय लेना

जब उपयोगकर्ता पूछता है "सीएटल में मौसम कैसा है?", तब मॉडल किसी उपकरण को यादृच्छिक रूप से नहीं चुनता। यह उपयोगकर्ता के इरादे की तुलना प्रत्येक उपकरण विवरण से करता है, प्रासंगिकता के लिए प्रत्येक को स्कोर करता है, और सबसे अच्छा मिलान चुनता है। फिर यह सही पैरामीटर के साथ संरचित फ़ंक्शन कॉल बनाता है — इस मामले में `location` को `"Seattle"` सेट करना।

यदि कोई उपकरण उपयोगकर्ता के अनुरोध से मेल नहीं खाता, तो मॉडल अपने ज्ञान से जवाब देता है। यदि कई उपकरण मेल खाते हैं, तो यह सबसे विशिष्ट को चुनता है।

<img src="../../../translated_images/hi/decision-making.409cd562e5cecc49.webp" alt="AI कैसे तय करता है कौन सा उपकरण उपयोग करना है" width="800"/>

*मॉडल उपयोगकर्ता के इरादे के विरुद्ध सभी उपलब्ध उपकरणों का मूल्यांकन करता है और सबसे अच्छा मिलान चुनता है — इसलिए स्पष्ट, विशिष्ट उपकरण विवरण लिखना जरूरी है।*

### प्रतिक्रिया उत्पादन

मॉडल मौसम डेटा प्राप्त करता है और उसे उपयोगकर्ता के लिए प्राकृतिक भाषा प्रतिक्रिया में स्वरूपित करता है।

### आर्किटेक्चर: स्प्रिंग बूट ऑटो-वायरिंग

यह मॉड्यूल LangChain4j के स्प्रिंग बूट एकीकरण का उपयोग करता है जिसमें घोषणा-आधारित `@AiService` इंटरफेस होते हैं। स्टार्टअप पर स्प्रिंग बूट हर `@Component` खोजता है जिसमें `@Tool` मेथड्स होते हैं, आपका `ChatModel` बीन, और `ChatMemoryProvider` — फिर इन्हें ज़ीरो बोयलरप्लेट के साथ एकल `Assistant` इंटरफेस में जोड़ देता है।

<img src="../../../translated_images/hi/spring-boot-wiring.151321795988b04e.webp" alt="स्प्रिंग बूट ऑटो-वायरिंग आर्किटेक्चर" width="800"/>

*@AiService इंटरफेस ChatModel, टूल कंपोनेंट्स, और मेमोरी प्रोवाइडर को जोड़ता है — स्प्रिंग बूट स्वचालित रूप से वायरिंग संभालता है।*

इस दृष्टिकोण के मुख्य लाभ:

- **स्प्रिंग बूट ऑटो-वायरिंग** — ChatModel और उपकरण स्वचालित रूप से इंजेक्ट होते हैं
- **@MemoryId पैटर्न** — स्वचालित सत्र-आधारित मेमोरी प्रबंधन
- **एकल उदाहरण** — असिस्टेंट एक बार बनाया जाता है और बेहतर प्रदर्शन के लिए दोबारा उपयोग किया जाता है
- **टाइप-सुरक्षित निष्पादन** — जावा मेथड्स सीधे टाइप रूपांतरण के साथ कॉल किए जाते हैं
- **मल्टी-टर्न ऑर्केस्ट्रेशन** — उपकरण चेनिंग को स्वचालित रूप से संभालता है
- **जीरो बोयलरप्लेट** — कोई मैनुअल `AiServices.builder()` कॉल या मेमोरी हैशमैप नहीं

वैकल्पिक दृष्टिकोण (मैनुअल `AiServices.builder()`), अधिक कोड की आवश्यकता होती है और स्प्रिंग बूट एकीकरण लाभ नहीं मिलता।

## उपकरण चेनिंग

**उपकरण चेनिंग** — उपकरण-आधारित एजेंट्स की असली ताकत तब दिखती है जब एक सवाल के लिए कई उपकरण चाहिए। पूछें "सीएटल में फैरेनहाइट में मौसम कैसा है?" और एजेंट स्वचालित रूप से दो उपकरण चेन करता है: पहले यह `getCurrentWeather` कॉल करता है ताकि सेल्सियस में तापमान प्राप्त हो, फिर वह मान `celsiusToFahrenheit` को पास करता है ताकि तापमान परिवर्तन हो — वह भी एक ही संवाद टर्न में।

<img src="../../../translated_images/hi/tool-chaining-example.538203e73d09dd82.webp" alt="उपकरण चेनिंग उदाहरण" width="800"/>

*उपकरण चेनिंग क्रिया में — एजेंट सबसे पहले getCurrentWeather कॉल करता है, फिर सेल्सियस परिणाम को celsiusToFahrenheit में पास करता है, और संयुक्त उत्तर देता है।*

यह चल रहे एप्लिकेशन में कुछ इस तरह दिखता है — एजेंट एक ही संवाद टर्न में दो उपकरण कॉल्स को चेन करता है:

<a href="images/tool-chaining.png"><img src="../../../translated_images/hi/tool-chaining.3b25af01967d6f7b.webp" alt="उपकरण चेनिंग" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*असली एप्लिकेशन आउटपुट — एजेंट स्वचालित रूप से getCurrentWeather → celsiusToFahrenheit को एक टर्न में चेन करता है।*

**शालीन विफलताएँ** — यदि आप मौसम को मॉक डेटा में न होने वाले किसी शहर के लिए पूछते हैं, तो उपकरण त्रुटि संदेश लौटाता है, और AI बताता है कि यह मदद नहीं कर सकता बिना क्रैश हुए। उपकरण सुरक्षित रूप से विफल होते हैं।

<img src="../../../translated_images/hi/error-handling-flow.9a330ffc8ee0475c.webp" alt="त्रुटि हैंडलिंग प्रवाह" width="800"/>

*जब कोई उपकरण विफल होता है, एजेंट त्रुटि पकड़ता है और क्रैश होने के बजाय मददगार व्याख्या के साथ जवाब देता है।*

यह एक ही संवाद टर्न में होता है। एजेंट स्वायत्त रूप से कई उपकरण कॉल का प्रबंधन करता है।

**विकल्प 1: स्प्रिंग बूट डैशबोर्ड का उपयोग करना (VS Code उपयोगकर्ताओं के लिए अनुशंसित)**

डेव कंटेनर में स्प्रिंग बूट डैशबोर्ड एक्सटेंशन शामिल है, जो सभी स्प्रिंग बूट एप्लिकेशन प्रबंधित करने के लिए एक विज़ुअल इंटरफ़ेस प्रदान करता है। आप इसे VS Code के बाईं ओर एक्टिविटी बार में स्प्रिंग बूट आइकन के पास पा सकते हैं।

स्प्रिंग बूट डैशबोर्ड से आप:
- कार्यक्षेत्र में सभी उपलब्ध स्प्रिंग बूट एप्लिकेशन देख सकते हैं
- एक क्लिक से एप्लिकेशन स्टार्ट/स्टॉप कर सकते हैं
- एप्लिकेशन लॉग्स रीयल-टाइम में देख सकते हैं
- एप्लिकेशन की स्थिति देख सकते हैं

सिर्फ "tools" के पास प्ले बटन पर क्लिक करें इस मॉड्यूल को शुरू करने के लिए, या सभी मॉड्यूल एक साथ शुरू करें।

## एप्लिकेशन का उपयोग करना

एप्लिकेशन एक वेब इंटरफ़ेस प्रदान करता है जहाँ आप एक AI एजेंट के साथ बातचीत कर सकते हैं जिसके पास मौसम और तापमान रूपांतरण उपकरणों तक पहुंच होती है।

<a href="images/tools-homepage.png"><img src="../../../translated_images/hi/tools-homepage.4b4cd8b2717f9621.webp" alt="AI एजेंट उपकरण इंटरफ़ेस" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*AI एजेंट उपकरण इंटरफ़ेस — उपकरणों के साथ बातचीत के लिए त्वरित उदाहरण और चैट इंटरफ़ेस*

### सरल उपकरण उपयोग आज़माएँ
एक सीधे सरल अनुरोध से शुरुआत करें: "100 डिग्री फैरनहाइट को सेल्सियस में बदलें"। एजेंट पहचानता है कि इसे तापमान रूपांतरण टूल की आवश्यकता है, सही पैरामीटर के साथ इसे कॉल करता है, और परिणाम लौटाता है। देखें कि यह कितना प्राकृतिक लगता है - आपने यह निर्दिष्ट नहीं किया कि कौन सा टूल उपयोग करना है या इसे कैसे कॉल करना है।

### टूल चेनिंग का परीक्षण करें

अब कुछ और जटिल प्रयास करें: "सिएटल में मौसम क्या है और इसे फैरनहाइट में बदलें?" देखें कि एजेंट इसे चरणों में कैसे करता है। वह पहले मौसम प्राप्त करता है (जो सेल्सियस में लौटता है), यह पहचानता है कि इसे फैरनहाइट में रूपांतरण करने की आवश्यकता है, रूपांतरण टूल को कॉल करता है, और दोनों परिणामों को एक उत्तर में संयोजित करता है।

### वार्तालाप प्रवाह देखें

चैट इंटरफ़ेस वार्तालाप इतिहास बनाए रखता है, जिससे आप मल्टी-टर्न बातचीत कर सकते हैं। आप सभी पिछली क्वेरी और प्रतिक्रियाएं देख सकते हैं, जिससे वार्तालाप को ट्रैक करना और समझना आसान हो जाता है कि कैसे एजेंट कई आदान-प्रदानों में संदर्भ बनाता है।

<a href="images/tools-conversation-demo.png"><img src="../../../translated_images/hi/tools-conversation-demo.89f2ce9676080f59.webp" alt="एक से अधिक टूल कॉल के साथ वार्तालाप" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*मल्टी-टर्न वार्तालाप जिसमें सरल रूपांतरण, मौसम खोज और टूल चेनिंग दिखाया गया है*

### विभिन्न अनुरोधों के साथ प्रयोग करें

विभिन्न संयोजनों का प्रयास करें:
- मौसम खोजें: "टोक्यो में मौसम क्या है?"
- तापमान रूपांतरण: "25°C केल्विन में क्या है?"
- संयुक्त प्रश्न: "पेरिस में मौसम देखें और बताएं क्या यह 20°C से ऊपर है"

देखें कि एजेंट प्राकृतिक भाषा को कैसे व्याख्यायित करता है और इसे उपयुक्त टूल कॉल में मैप करता है।

## मुख्य अवधारणाएं

### ReAct पैटर्न (तर्क और क्रियान्वयन)

एजेंट तर्क (क्या करना है निर्णय लेना) और क्रिया (टूल्स का उपयोग करना) के बीच बारी-बारी से चलता है। यह पैटर्न स्वायत्त समस्या समाधान सक्षम करता है बजाय केवल निर्देशों का जवाब देने के।

### टूल विवरण महत्वपूर्ण हैं

आपके टूल विवरण की गुणवत्ता सीधे इस बात को प्रभावित करती है कि एजेंट उन्हें कितनी अच्छी तरह उपयोग करता है। स्पष्ट, विशिष्ट विवरण मॉडल को समझने में मदद करते हैं कि कब और कैसे प्रत्येक टूल को कॉल करना है।

### सत्र प्रबंधन

`@MemoryId` एनोटेशन स्वचालित सत्र-आधारित मेमोरी प्रबंधन सक्षम करता है। प्रत्येक सत्र आईडी को इसका अपना `ChatMemory` उदाहरण मिलता है जिसे `ChatMemoryProvider` बीन द्वारा प्रबंधित किया जाता है, ताकि कई उपयोगकर्ता एक साथ एजेंट के साथ बातचीत कर सकें बिना उनकी वार्तालापें मिलाए।

<img src="../../../translated_images/hi/session-management.91ad819c6c89c400.webp" alt="@MemoryId के साथ सत्र प्रबंधन" width="800"/>

*प्रत्येक सत्र आईडी एक अलग वार्तालाप इतिहास से जुड़ी होती है — उपयोगकर्ता कभी भी एक-दूसरे के संदेश नहीं देखते।*

### त्रुटि हैंडलिंग

टूल्स असफल हो सकते हैं — एपीआई टाइमआउट हो सकते हैं, पैरामीटर अमान्य हो सकते हैं, बाहरी सेवाएं बंद हो सकती हैं। प्रोडक्शन एजेंटों को त्रुटि हैंडलिंग की आवश्यकता होती है ताकि मॉडल समस्याओं को समझा सके या विकल्प आजमा सके बजाय पूरे एप्लिकेशन के क्रैश होने के। जब कोई टूल अपवाद फेंकता है तो LangChain4j उसे पकड़ता है और त्रुटि संदेश मॉडल को वापस भेजता है, जो फिर इसे प्राकृतिक भाषा में समझा सकता है।

## उपलब्ध टूल्स

नीचे दिया गया आरेख वे व्यापक टूल्स दिखाता है जिन्हें आप बना सकते हैं। यह मॉड्यूल मौसम और तापमान टूल्स का प्रदर्शन करता है, लेकिन वही `@Tool` पैटर्न किसी भी जावा मेथड के लिए काम करता है — डेटाबेस क्वेरी से लेकर पेमेंट प्रोसेसिंग तक।

<img src="../../../translated_images/hi/tool-ecosystem.aad3d74eaa14a44f.webp" alt="टूल पारिस्थितिकी तंत्र" width="800"/>

*@Tool से एनोटेट किया गया कोई भी जावा मेथड AI के लिए उपलब्ध हो जाता है — यह पैटर्न डेटाबेस, API, ईमेल, फाइल ऑपरेशंस और अधिक के लिए बढ़ाया जा सकता है।*

## टूल-आधारित एजेंट कब उपयोग करें

<img src="../../../translated_images/hi/when-to-use-tools.51d1592d9cbdae9c.webp" alt="टूल कब उपयोग करें" width="800"/>

*एक त्वरित निर्णय गाइड — टूल वास्तविक समय डेटा, गणनाओं और क्रियाओं के लिए हैं; सामान्य ज्ञान और रचनात्मक कार्यों को उनकी आवश्यकता नहीं होती।*

**टूल्स तब उपयोग करें जब:**
- उत्तर देने के लिए वास्तविक समय डेटा की जरूरत हो (मौसम, स्टॉक की कीमतें, इन्वेंट्री)
- सरल गणित से परे गणना करने की जरूरत हो
- डेटाबेस या APIs तक पहुँच आवश्यक हो
- क्रियाएं करनी हों (ईमेल भेजना, टिकट बनाना, रिकॉर्ड अपडेट करना)
- कई डेटा स्रोतों को संयोजित करना हो

**टूल्स का उपयोग न करें जब:**
- प्रश्न सामान्य ज्ञान से हल हो सकते हों
- प्रतिक्रिया केवल संवादात्मक हो
- टूल विलंबता अनुभव को बहुत धीमा बना दे

## टूल्स बनाम RAG

मॉड्यूल 03 और 04 दोनों AI की क्षमता को बढ़ाते हैं, लेकिन मूल रूप से अलग तरीकों से। RAG मॉडल को दस्तावेज़ों को पुनः प्राप्त कर **ज्ञान** तक पहुँच देता है। टूल्स मॉडल को फ़ंक्शन कॉल करके **कार्य** करने की क्षमता देते हैं।

<img src="../../../translated_images/hi/tools-vs-rag.ad55ce10d7e4da87.webp" alt="टूल्स बनाम RAG तुलना" width="800"/>

*RAG स्थैतिक दस्तावेज़ों से जानकारी पुनः प्राप्त करता है — टूल्स क्रियाएं निष्पादित करते हैं और गतिशील, वास्तविक समय डेटा लाते हैं। कई प्रोडक्शन सिस्टम दोनों का संयोजन करते हैं।*

वास्तव में, कई प्रोडक्शन सिस्टम दोनों दृष्टिकोणों को मिलाते हैं: RAG आपके दस्तावेज़ीकरण में उत्तरों को आधार देने के लिए, और टूल्स लाइव डेटा लाने या संचालन करने के लिए।
