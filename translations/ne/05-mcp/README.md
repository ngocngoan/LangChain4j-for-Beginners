## तपाईंले के सिक्नुहुनेछ

तपाईंले संवादात्मक एआई बनाउनु भयो, प्राँप्टहरूमा महारत हासिल गर्नुभयो, उत्तरहरूलाई कागजातहरूमा आधारित गर्नुभयो, र उपकरणहरू सहित एजेन्टहरू सिर्जना गर्नुभयो। तर ती सबै उपकरणहरू तपाईंको विशिष्ट अनुप्रयोगका लागि कस्टम निर्माण गरिएका थिए। के भए यदि तपाईंले तपाईंको एआईलाई उपकरणहरूको मानकीकृत पारिस्थितिकी तन्त्रको पहुँच दिनुभयो जुन कोही पनि बनाउन र साझा गर्न सक्छ? यस मोड्युलमा, तपाईंले Model Context Protocol (MCP) र LangChain4j को एजेन्टिक मोड्युल प्रयोग गरी त्यो कसरी गर्ने सिक्नेछौं। हामी पहिले एउटा सरल MCP फाइल रिडर देखाउँछौं र त्यसपछि देखाउँछौं यसले कसरी सजिलै सुपरभाइजर एजेन्ट ढाँचामा आधारित उन्नत एजेन्टिक कार्यप्रवाहहरूमा एकीकृत हुन्छ।

## MCP के हो?

Model Context Protocol (MCP) त्यही प्रदान गर्दछ - एआई अनुप्रयोगहरूले बाह्य उपकरणहरू पत्ता लगाउन र प्रयोग गर्नका लागि मानक तरिका। प्रत्येक डेटा स्रोत वा सेवाको लागि कस्टम एकीकरण लेख्नुभन्दा, तपाईं MCP सर्भरहरूमा जडान हुनुहुन्छ जुन आफ्नो क्षमताहरू सुसंगत ढाँचा मा प्रकट गर्छन्। तपाईंको AI एजेन्टले त्यसपछि यी उपकरणहरू स्वतः पत्ता लगाउन र प्रयोग गर्न सक्छ।

तलको चित्रले भिन्नता देखाउँछ—MCP बिना, प्रत्येक एकीकरण कस्टम पाइंट-टु-पाइंट तार जडान चाहिन्छ; MCP सँग, एक मात्र प्रोटोकल तपाईंको अनुप्रयोगलाई कुनै पनि उपकरणमा जडान गर्छ:

<img src="../../../translated_images/ne/mcp-comparison.9129a881ecf10ff5.webp" alt="MCP Comparison" width="800"/>

*MCP अघिको: जटिल पाइंट-टु-पाइंट एकीकरणहरू। MCP पछि: एक प्रोटोकल, अनन्त सम्भावनाहरू।*

MCP एआई विकासमा एक मौलिक समस्या समाधान गर्दछ: हरेक एकीकरण कस्टम हुन्छ। GitHub पहुँच गर्न चाहनुहुन्छ? कस्टम कोड। फाइलहरू पढ्न चाहनुहुन्छ? कस्टम कोड। डेटाबेस क्वेरी गर्न चाहनुहुन्छ? कस्टम कोड। र तीमध्ये कुनै पनि एकीकरण अन्य एआई अनुप्रयोगहरूसँग काम गर्दैन।

MCP यसलाई मानकीकृत गर्छ। एक MCP सर्भरले उपकरणहरू प्रष्ट विवरणहरू र स्किमाहरू सहित प्रकट गर्छ। कुनै पनि MCP क्लाइन्ट जडान गर्न सक्छ, उपलब्ध उपकरणहरू पत्ता लगाउन सक्छ, र प्रयोग गर्न सक्छ। एक पटक बनाउनुहोस्, सबै ठाउँमा प्रयोग गर्नुहोस्।

तलको चित्रले यो संरचना देखाउँछ—एक MCP क्लाइन्ट (तपाईंको AI अनुप्रयोग) धेरै MCP सर्भरहरूमा जडान हुन्छ, प्रत्येकले आफ्नो उपकरणहरूको सेट मानक प्रोटोकलमार्फत प्रकट गर्दछ:

<img src="../../../translated_images/ne/mcp-architecture.b3156d787a4ceac9.webp" alt="MCP Architecture" width="800"/>

*Model Context Protocol वास्तुकला - मानकीकृत उपकरण पत्ता लगाउने र कार्यान्वयन*

## MCP कसरी काम गर्दछ

भित्र MCP ले तहहरूयुक्त वास्तुकला प्रयोग गर्दछ। तपाईंको Java अनुप्रयोग (MCP क्लाइन्ट) ले उपलब्ध उपकरणहरू पत्ता लगाउँछ, ट्रान्सपोर्ट तह (Stdio वा HTTP) मार्फत JSON-RPC अनुरोधहरू पठाउँछ, र MCP सर्भरले अपरेशनहरू कार्यान्वयन गरी नतिजाहरू फर्काउँछ। तलको चित्रले यस प्रोटोकलको हरेक तह तोडेर देखाउँछ:

<img src="../../../translated_images/ne/mcp-protocol-detail.01204e056f45308b.webp" alt="MCP Protocol Detail" width="800"/>

*MCP भित्र कसरी काम गर्दछ—क्लाइन्टहरूले उपकरणहरू पत्ता लगाउँछन्, JSON-RPC सन्देशहरू आदानप्रदान गर्छन्, र ट्रान्सपोर्ट तह मार्फत अपरेशनहरू कार्यान्वयन गर्छन्।*

**सर्भर-क्लाइन्ट वास्तुकला**

MCP ले क्लाइन्ट-सर्भर मोडेल प्रयोग गर्दछ। सर्भरहरूले उपकरणहरू प्रदान गर्दछन् - फाइल पढ्ने, डेटाबेस क्वेरी गर्ने, API कल गर्ने। क्लाइन्टहरू (तपाईंको AI अनुप्रयोग) सर्भरहरूसँग जडान हुन्छन् र उपकरणहरू प्रयोग गर्छन्।

LangChain4j सँग MCP प्रयोग गर्न, यो Maven निर्भरता थप्नुहोस्:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```

**उपकरण पत्ता लगाउने**

जब तपाईंको क्लाइन्टले MCP सर्भरसँग जडान गर्छ, यसले सोध्छ "तपाईंको के उपकरणहरू छन्?" सर्भरले उपलब्ध उपकरणहरूको सूची फिर्ता गर्दछ, प्रत्येकमा विवरणहरू र प्यारामिटर स्किमाहरू सहित। तपाईंको AI एजेन्टले त्यसपछि प्रयोगकर्ताको अनुरोधका आधारमा कुन उपकरण प्रयोग गर्ने छनौट गर्न सक्छ। तलको चित्रले यो हात मिलाउने प्रक्रिया देखाउँछ—क्लाइन्टले `tools/list` अनुरोध पठाउँछ र सर्भरले उपलब्ध उपकरणहरू विवरण र स्किमासहित फिर्ता गर्छ:

<img src="../../../translated_images/ne/tool-discovery.07760a8a301a7832.webp" alt="MCP Tool Discovery" width="800"/>

*AI ले सुरुमा उपलब्ध उपकरणहरू पत्ता लगाउँछ—अब यो थाहा छ कुन क्षमता उपलब्ध छन् र कुन प्रयोग गर्ने निर्णय गर्न सक्छ।*

**ट्रान्सपोर्ट मेकानिज्महरू**

MCP ले विभिन्न ट्रान्सपोर्ट मेकानिज्महरू समर्थन गर्दछ। दुई विकल्पहरू छन् Stdio (स्थानीय उपप्रक्रिया संवादका लागि) र Streamable HTTP (रिमोट सर्भरका लागि)। यो मोड्युलले Stdio ट्रान्सपोर्ट देखाउँछ:

<img src="../../../translated_images/ne/transport-mechanisms.2791ba7ee93cf020.webp" alt="Transport Mechanisms" width="800"/>

*MCP ट्रान्सपोर्ट मेकानिज्महरू: रिमोट सर्भरका लागि HTTP, स्थानीय प्रक्रियाहरूका लागि Stdio*

**Stdio** - [StdioTransportDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java)

स्थानीय प्रक्रियाहरूका लागि। तपाईंको अनुप्रयोग एक सर्भर उपप्रक्रियाको रूपमा सुरु गर्छ र मानक इनपुट/आउटपुट मार्फत संवाद गर्छ। फाइल सिस्टम पहुँच वा कमान्ड-लाइन उपकरणहरूका लागि उपयुक्त।

```java
McpTransport stdioTransport = new StdioMcpTransport.Builder()
    .command(List.of(
        npmCmd, "exec",
        "@modelcontextprotocol/server-filesystem@2025.12.18",
        resourcesDir
    ))
    .logEvents(false)
    .build();
```

`@modelcontextprotocol/server-filesystem` सर्भरले तलका उपकरणहरू प्रकट गर्दछ, सबै तपाईंले निर्दिष्ट गर्नुभएको डाइरेक्टोरीहरूमा सीमित:

| उपकरण | विवरण |
|------|-------------|
| `read_file` | एकल फाइलको सामग्री पढ्नुहोस् |
| `read_multiple_files` | एउटै कलमा धेरै फाइलहरू पढ्नुहोस् |
| `write_file` | फाइल सिर्जना वा पुन:लेखन गर्नुहोस् |
| `edit_file` | लक्षित फेला पार्ने र परिवर्तन गर्ने सम्पादनहरू गर्नुहोस् |
| `list_directory` | पथमा फाइलहरू र डाइरेक्टोरीहरू सूचीबद्ध गर्नुहोस् |
| `search_files` | नमूनासँग मेल खाने फाइलहरूको पुनरावृत्ति खोजी गर्नुहोस् |
| `get_file_info` | फाइल मेटाडेटा (आकार, टाइमस्ट्याम्प, अनुमति) प्राप्त गर्नुहोस् |
| `create_directory` | डाइरेक्टोरी सिर्जना गर्नुहोस् (अभिभावक डाइरेक्टोरीसहित) |
| `move_file` | फाइल वा डाइरेक्टोरी सार्नुहोस् वा नाम परिवर्तन गर्नुहोस् |

तलको चित्रले रनटाइममा Stdio ट्रान्सपोर्ट कसरी काम गर्छ देखाउँछ—तपाईंको Java अनुप्रयोगले MCP सर्भरलाई छोराबाट प्रक्रियाको रूपमा सुरु गर्छ र तिनीहरू stdin/stdout पाइपहरू मार्फत संवाद गर्छन्, कुनै नेटवर्क वा HTTP संलग्न नभएको:

<img src="../../../translated_images/ne/stdio-transport-flow.45eaff4af2d81db4.webp" alt="Stdio Transport Flow" width="800"/>

*Stdio ट्रान्सपोर्ट क्रियाशील—तपाईंको अनुप्रयोगले MCP सर्भरलाई छोराबाट प्रक्रियाको रूपमा सुरु गर्छ र stdin/stdout पाइपमार्फत संवाद गर्छ।*

> - "Stdio ट्रान्सपोर्ट कसरी काम गर्छ र कहिले HTTP को सट्टा प्रयोग गर्ने?"
> - "LangChain4j ले उत्पन्न MCP सर्भर प्रक्रियाहरूको जीवनचक्र कसरी व्यवस्थापन गर्छ?"
> - "एआईलाई फाइल सिस्टम पहुँच दिएर के सुरक्षा प्रभावहरू छन्?"

## एजेन्टिक मोड्युल

MCP ले मानकीकृत उपकरणहरू प्रदान गर्दा, LangChain4j को **एजेन्टिक मोड्युल** ती उपकरणहरूलाई संयोजित गर्ने एजेन्टहरू निर्माण गर्न डिक्लेरेटिभ तरिका प्रदान गर्छ। `@Agent` एनोटेसन र `AgenticServices` ले तपाईंलाई प्रत्यक्ष कोडभन्दा इन्टरफेसमार्फत एजेन्ट व्यवहार परिभाषित गर्न दिन्छ।

यस मोड्युलमा, तपाईं **सुपरभाइजर एजेन्ट** ढाँचाको अन्वेषण गर्नुहुनेछ—एक उन्नत एजेन्टिक AI दृष्टिकोण जहाँ "सुपरभाइजर" एजेन्ट प्रयोगकर्ताको अनुरोधका आधारमा डायनामिक रूपमा कुन सब-एजेन्टलाई कल गर्ने निर्णय गर्छ। हामी यी दुवै अवधारणाहरूलाई मिलाएर हाम्रो सब-एजेन्टहरूलाई MCP-शक्तिसम्पन्न फाइल पहुँच क्षमताहरू दिनेछौं।

एजेन्टिक मोड्युल प्रयोग गर्न, यो Maven निर्भरता थप्नुहोस्:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```
> **ध्यान दिनुहोस्:** `langchain4j-agentic` मोड्युलले अलग संस्करण सम्पत्ति (`langchain4j.mcp.version`) प्रयोग गर्दछ किनकि यो कोर LangChain4j पुस्तकालयहरूको भन्दा भिन्न तालिकामा रिलिज हुन्छ।

> **⚠️ प्रायोगात्मक:** `langchain4j-agentic` मोड्युल **प्रायोगात्मक** छ र परिवर्तन योग्य छ। स्थिर तरिका AI सहायकहरू बनाउन भने `langchain4j-core` कस्टम उपकरणहरूसँग (Module 04) हो।

## उदाहरणहरू चलाउने

### पूर्वापेक्षाहरू

- [Module 04 - Tools](../04-tools/README.md) पूरा गरेको (यो मोड्युलले कस्टम उपकरण अवधारणाहरूमा आधारित छ र MCP उपकरणहरूसँग तुलना गर्छ)

> **ध्यान दिनुहोस्:** यदि तपाईंले अहिलेसम्म वातावरण चरहरू सेट गर्नुभएन भने, [Module 01 - Introduction](../01-introduction/README.md) हेर्नुहोस् स्थापनाका लागि (`azd up` ले स्वतः `.env` फाइल सिर्जना गर्छ), वा `.env.example` लाई रूट डाइरेक्टोरीमा `.env` मा प्रतिलिपि गरी आफ्नो मानहरू भर्नुहोस्।

### सुपरभाइजर एजेन्ट

**सुपरभाइजर एजेन्ट ढाँचा** एक लचिलो एजेन्टिक AI प्रकार हो। सुपरभाइजरले LLM प्रयोग गरेर स्वतः निर्णय गर्छ कुन एजेन्टहरूलाई प्रयोगकर्ताको अनुरोधका आधारमा कल गर्ने। अर्को उदाहरणमा, हामी MCP-शक्तिसम्पन्न फाइल पहुँचलाई LLM एजेन्टसँग मिलाएर सुपरवाइज्ड फाइल पढ्ने → रिपोर्ट प्रवाह सिर्जना गर्नेछौं।

डेमोमा, `FileAgent` ले MCP फाइल सिस्टम उपकरणहरू प्रयोग गरी फाइल पढ्छ, र `ReportAgent` ले कार्यकारी सारांश (१ वाक्य), ३ मुख्य बुँदाहरू, र सिफारिसहरू सहित संरचित रिपोर्ट तयार पार्छ। सुपरभाइजरले यो प्रवाह स्वतः व्यवस्थापन गर्छ:

<img src="../../../translated_images/ne/supervisor-agent-pattern.06275a41ae006ac8.webp" alt="Supervisor Agent Pattern" width="800"/>

*सुपरभाइजरले आफ्नो LLM प्रयोग गरेर कुन एजेन्टहरू कल गर्ने र कुन क्रममा गर्ने निर्णय गर्छ—कुनै हार्डकोडेड मार्गदर्शन आवश्यक छैन।*

हाम्रो फाइलदेखि रिपोर्ट पाइपलाइनको ठोस कार्यप्रवाह यस्तो देखिन्छ:

<img src="../../../translated_images/ne/file-report-workflow.649bb7a896800de9.webp" alt="File to Report Workflow" width="800"/>

*FileAgent ले MCP उपकरणमार्फत फाइल पढ्छ, त्यसपछि ReportAgent कच्चा सामग्रीलाई संरचित रिपोर्टमा परिणत गर्छ।*

प्रत्येक एजेन्टले आफ्नो आउटपुट **Agentic Scope** (साझा मेमोरी) मा भण्डारण गर्छ, जसले तलका एजेन्टहरूले अघिल्ला नतिजाहरूमा पहुँच पाउँछन्। यो देखाउँछ कसरी MCP उपकरणहरू एजेन्टिक कार्यप्रवाहहरूमा सहजै एकीकृत हुन्छन्—सुपरभाइजरलाई थाहा हुनुपर्दैन *कसरी* फाइलहरू पढिन्छन्, केवल थाहा पाउँछ `FileAgent` ले त्यसलाई गर्न सक्छ।

**PowerShell:**

#### सुपरभाइजर कसरी काम गर्दछ

एजेन्टहरू निर्माण गर्नु अघि, तपाईंले MCP ट्रान्सपोर्टलाई क्लाइन्टमा जडान गर्नुपर्छ र यसलाई `ToolProvider` को रूपमा रैप गर्नुपर्छ। यसरी MCP सर्भरका उपकरणहरू तपाईंका एजेन्टहरूलाई उपलब्ध हुन्छन्:

```java
// ट्रान्सपोर्टबाट MCP क्लाइन्ट सिर्जना गर्नुहोस्
McpClient mcpClient = new DefaultMcpClient.Builder()
        .transport(stdioTransport)
        .build();

// क्लाइन्टलाई ToolProvider को रूपमा र्याप गर्नुहोस् — यसले MCP उपकरणहरूलाई LangChain4j मा जोड्दछ
ToolProvider mcpToolProvider = McpToolProvider.builder()
        .mcpClients(List.of(mcpClient))
        .build();
```

अब तपाईं `mcpToolProvider` लाई कुनै पनि एजेन्टमा इन्जेक्ट गर्न सक्नुहुन्छ जसलाई MCP उपकरणहरू चाहिन्छ:

```java
// चरण 1: FileAgent ले MCP उपकरणहरू प्रयोग गरेर फाइलहरू पढ्छ
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // फाइल अपरेसनहरूको लागि MCP उपकरणहरू छन्
        .build();

// चरण 2: ReportAgent ले संरचित रिपोर्टहरू उत्पादन गर्दछ
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// Supervisor ले फाइल → रिपोर्ट कार्यप्रवाह समन्वय गर्दछ
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // अन्तिम रिपोर्ट फिर्ता गर्नुहोस्
        .build();

// Supervisor ले अनुरोधको आधारमा कुन एजेन्टहरूलाई बोलाउने निर्णय गर्दछ
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```

#### प्रतिक्रिया रणनीतिहरू

जब तपाईं `SupervisorAgent` कन्फिगर गर्नुहुन्छ, तपाईंले निर्दिष्ट गर्नुहुन्छ कि उप-एजेन्टहरूले आफ्नो कार्य पूरा गरेपछि अन्तिम उत्तर प्रयोगकर्तालाई कसरी प्रदान गर्ने। तलको चित्रले तीन उपलब्ध रणनीतिहरू देखाउँछ—LAST ले अन्तिम एजेन्टको आउटपुट सिधै फर्काउँछ, SUMMARY ले सबै आउटपुटहरू LLM मार्फत संश्लेषण गर्छ, र SCORED ले मूल अनुरोधसँग तुलना गरेर उच्च स्कोर पाएको विकल्प चुन्छ:

<img src="../../../translated_images/ne/response-strategies.3d0cea19d096bdf9.webp" alt="Response Strategies" width="800"/>

*सुपरभाइजरले अन्तिम प्रतिक्रिया कसरी तयार पार्छ भन्ने तीन रणनीतिहरू—अन्तिम एजेन्टको आउटपुट, संश्लेषित सारांश, वा सर्वश्रेष्ठ स्कोर भएको विकल्प छनौट गर्नुहोस्।*

उपलब्ध रणनीतिहरू:

| रणनीति | विवरण |
|----------|-------------|
| **LAST** | सुपरभाइजरले अन्तिम सब-एजेन्ट वा उपकरणको आउटपुट फर्काउँछ। यो उपयोगी हुन्छ जब कार्यप्रवाहको अन्तिम एजेन्टले पूर्ण, अन्तिम उत्तर उत्पादन गर्न डिजाइन गरिएको हुन्छ (जस्तै अनुसन्धान पाइपलाइनमा "Summary Agent")। |
| **SUMMARY** | सुपरभाइजरले आफ्नै आन्तरिक भाषा मोडेल (LLM) प्रयोग गरी सम्पूर्ण अन्तरक्रिया र सब-एजेन्ट आउटपुटहरूको सारांश संश्लेषण गर्छ, र त्यो सारांशलाई अन्तिम उत्तरको रूपमा फर्काउँछ। यसले प्रयोगकर्तालाई सफा, समग्र उत्तर प्रदान गर्छ। |
| **SCORED** | प्रणालीले आन्तरिक LLM प्रयोग गरी LAST प्रतिक्रिया र SUMMARY लाई आधारभूत प्रयोगकर्ता अनुरोधसँग स्कोर गर्छ, र बढी स्कोर प्राप्त गरेको आउटपुट फर्काउँछ। |

#### आउटपुट बुझ्न

जब तपाईं डेमो चलाउनुहुन्छ, तपाईंले Supervisor ले कसरी धेरै एजेन्टहरूलाई समन्वय गर्छ भन्ने संरचनाबद्ध मार्गदर्शन देख्नुहुनेछ। यहाँ प्रत्येक खण्डले के जनाउँछ:

```
======================================================================
  FILE → REPORT WORKFLOW DEMO
======================================================================

This demo shows a clear 2-step workflow: read a file, then generate a report.
The Supervisor orchestrates the agents automatically based on the request.
```
  
**हेडर** ले workflow अवधारणा परिचय गराउँछ: फाइल पढाइदेखि रिपोर्ट उत्पादनसम्म केन्द्रित पाइपलाइन।

```
--- WORKFLOW ---------------------------------------------------------
  ┌─────────────┐      ┌──────────────┐
  │  FileAgent  │ ───▶ │ ReportAgent  │
  │ (MCP tools) │      │  (pure LLM)  │
  └─────────────┘      └──────────────┘
   outputKey:           outputKey:
   'fileContent'        'report'

--- AVAILABLE AGENTS -------------------------------------------------
  [FILE]   FileAgent   - Reads files via MCP → stores in 'fileContent'
  [REPORT] ReportAgent - Generates structured report → stores in 'report'
```
  
**Workflow डायग्राम** ले एजेन्टहरूबीच डेटा प्रवाह देखाउँछ। प्रत्येक एजेन्टको निश्चित भूमिका छ:  
- **FileAgent** ले MCP उपकरणहरू प्रयोग गरेर फाइलहरू पढ्छ र कच्चा सामग्री `fileContent` मा भण्डारण गर्छ  
- **ReportAgent** त्यो सामग्री उपभोग गर्छ र संरचित रिपोर्ट `report` मा उत्पादन गर्छ

```
--- USER REQUEST -----------------------------------------------------
  "Read the file at .../file.txt and generate a report on its contents"
```
  
**प्रयोगकर्ता अनुरोध** ले कार्य देखाउँछ। Supervisor ले यसलाई पार्स गर्छ र FileAgent → ReportAgent बोलाउने निर्णय गर्छ।

```
--- SUPERVISOR ORCHESTRATION -----------------------------------------
  The Supervisor decides which agents to invoke and passes data between them...

  +-- STEP 1: Supervisor chose -> FileAgent (reading file via MCP)
  |
  |   Input: .../file.txt
  |
  |   Result: LangChain4j is an open-source, provider-agnostic Java framework for building LLM...
  +-- [OK] FileAgent (reading file via MCP) completed

  +-- STEP 2: Supervisor chose -> ReportAgent (generating structured report)
  |
  |   Input: LangChain4j is an open-source, provider-agnostic Java framew...
  |
  |   Result: Executive Summary...
  +-- [OK] ReportAgent (generating structured report) completed
```
  
**Supervisor समन्वय** ले २-चरणीय प्रवाह कार्यमा देखाउँछ:  
1. **FileAgent** ले MCP मार्फत फाइल पढ्छ र सामग्री भण्डारण गर्छ  
2. **ReportAgent** ले सामग्री प्राप्त गरी संरचित रिपोर्ट बनाउँछ

Supervisor ले यी निर्णयहरू **स्वायत्त रूपमा** प्रयोगकर्ताको अनुरोधमा आधारित रूपमा गर्‍यो।

```
--- FINAL RESPONSE ---------------------------------------------------
Executive Summary
...

Key Points
...

Recommendations
...

--- AGENTIC SCOPE (Data Flow) ----------------------------------------
  Each agent stores its output for downstream agents to consume:
  * fileContent: LangChain4j is an open-source, provider-agnostic Java framework...
  * report: Executive Summary...
```
  
#### Agentic मोड्युल सुविधाहरूको व्याख्या

उदाहरणले agentic मोड्युलका धेरै उन्नत विशेषताहरू देखाउँछ। आउनुहोस् Agentic Scope र Agent Listeners लाई नजिकबाट हेरौं।

**Agentic Scope** ले साझा मेमोरी देखाउँछ जहाँ एजेन्टहरूले `@Agent(outputKey="...")` प्रयोग गरी आफ्नो नतिजा राखेका छन्। यसले अनुमति दिन्छ:  
- पछि आउने एजेन्टहरूले पहिलेका एजेन्टहरूको आउटपुट पहुँच गर्न  
- Supervisor लाई अन्तिम जवाफ संश्लेषण गर्न  
- तपाईंलाई हरेक एजेन्टले के उत्पादन गर्‍यो निरीक्षण गर्न

तलको डायग्रामले देखाउँछ Agentic Scope फाइल-देखि-रिपोर्ट workflow मा साझा मेमोरीको काम कसरी गर्छ — FileAgent ले आफ्नो आउटपुट `fileContent` कुञ्जीअन्तर्गत लेख्छ, ReportAgent ले त्यो पढ्छ र आफ्नो आउटपुट `report` कुञ्जीअन्तर्गत लेख्छ:

<img src="../../../translated_images/ne/agentic-scope.95ef488b6c1d02ef.webp" alt="Agentic Scope Shared Memory" width="800"/>

*Agentic Scope साझा मेमोरीको रूपमा काम गर्छ — FileAgent ले `fileContent` लेख्छ, ReportAgent ले यो पढ्छ र `report` लेख्छ, र तपाईंको कोड अन्तिम परिणाम पढ्छ।*

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // FileAgent बाट कच्चा फाइल डेटा
String report = scope.readState("report");            // ReportAgent बाट संरचित प्रतिवेदन
```
  
**Agent Listeners** ले एजेन्ट क्रियान्वयनको निगरानी र डिबगिङ सक्षम पार्छ। डेमोमा देखिने चरण-दर-चरण आउटपुट एउटा AgentListener बाट आउँछ जसले हरेक एजेन्ट आह्वानमा हुक गर्छ:  
- **beforeAgentInvocation** - Supervisor ले एजेन्ट चयन गर्दा कल हुन्छ, तपाईंलाई कुन एजेन्ट किन छानियो देखाउन  
- **afterAgentInvocation** - एजेन्ट पूरा भएपछि कल हुन्छ, यसको नतिजा देखाउँछ  
- **inheritedBySubagents** - जब सत्य हुन्छ, लिस्नरले समग्र पदानुक्रमका सबै एजेन्टहरूलाई अनुगमन गर्छ

तलको डायग्रामले पुरै Agent Listener जीवनचक्र देखाउँछ, जसमा `onError` कसरी एजेन्ट क्रियान्वयन क्रममा असफलता ह्यान्डल गर्छ समावेश छ:

<img src="../../../translated_images/ne/agent-listeners.784bfc403c80ea13.webp" alt="Agent Listeners Lifecycle" width="800"/>

*Agent Listeners कार्यान्वयन जीवनचक्रसँग जोडिन्छन् — एजेन्टहरू सुरु, पूरा, वा त्रुटि हुँदा निगरानी गर्छन्।*

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // सबै उप-एजेन्टहरूमा प्रशारण गर्नुहोस्
    }
};
```
  
Supervisor पैटर्नभन्दा बाहिर, `langchain4j-agentic` मोड्युलले धेरै शक्तिशाली workflow पैटर्नहरू प्रदान गर्छ। तलका डायग्रामले सबै पाँच देखाउँछ — साधारण अनुक्रमिक पाइपलाइनदेखि मानव-इन-द-लूप स्वीकृति workflow सम्म:

<img src="../../../translated_images/ne/workflow-patterns.82b2cc5b0c5edb22.webp" alt="Agent Workflow Patterns" width="800"/>

*एजेन्टहरू समन्वय गर्न पाँच workflow पैटर्नहरू — साधारण अनुक्रमिक पाइपलाइनदेखि मानव-इन-द-लूप स्वीकृति workflows सम्म।*

| पैटर्न | विवरण | प्रयोग केस |
|---------|-------------|----------|
| **Sequential** | एजेन्टहरूलाई आदेशमा चलाउनुहोस्, आउटपुट अर्कोमा जान्छ | पाइपलाइनहरू: अनुसन्धान → विश्लेषण → रिपोर्ट |
| **Parallel** | एजेन्टहरू एकै समयमा चलाउनुहोस् | स्वतन्त्र कार्यहरू: मौसम + समाचार + स्टक |
| **Loop** | सर्त पूरा नहुँदासम्म दोहोर्याउनुहोस् | गुणस्तर स्कोरिङ: स्कोर ≥ ०.८ नभएसम्म सुधार |
| **Conditional** | सर्तहरू अनुसार मार्गनिर्देशन | वर्गीकरण → विशेषज्ञ एजेन्टमा पठाउनुहोस् |
| **Human-in-the-Loop** | मानव चेकपॉइन्ट थप्नुहोस् | स्वीकृति workflows, सामग्री समीक्षा |

## मुख्य अवधारणाहरू

अब तपाईंले MCP र agentic मोड्युल व्यवहारमा अन्वेषण गरिसक्नुभयो, हरेक विधि कहिले प्रयोग गर्ने संक्षेप गरौं।

MCP को सबैभन्दा ठूलो फाइदा यसको बढ्दो पारिस्थितिकी तन्त्र हो। तलको डायग्रामले देखाउँछ कसरी एकल सार्वभौम प्रोटोकलले तपाईंको AI अनुप्रयोगलाई विभिन्न MCP सर्भरहरूसँग जडान गर्छ — फाइलसिस्टम र डेटाबेस पहुँचदेखि GitHub, इमेल, वेब स्क्रापिङ र थपसम्म:

<img src="../../../translated_images/ne/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="MCP Ecosystem" width="800"/>

*MCP ले सार्वभौम प्रोटोकल पारिस्थितिकी तन्त्र बनाउँछ — कुनै पनि MCP-संगत सर्भरले कुनै पनि MCP-संगत ग्राहकसँग काम गर्छ, अनुप्रयोगहरूमा उपकरण साझेदारी सक्षम पार्दै।*

**MCP** तब उपयुक्त छ जब तपाईं विद्यमान उपकरण पारिस्थितिकी तन्त्रलाई उपयोग गर्न चाहनुहुन्छ, यस्तो उपकरणहरू विकास गर्न चाहनुहुन्छ जुन धेरै अनुप्रयोगले साझेदारी गर्न सकून्, मानक प्रोटोकलहरूमार्फत तेस्रो-पक्ष सेवा समाकलन गर्न वा कोड परिवर्तन बिना उपकरण कार्यान्वयन परिवर्तन गर्न।

**Agentic Module** तब उत्तम हो जब तपाईंलाई `@Agent` एनोटेशनहरूसहित घोषणात्मक एजेन्ट परिभाषाहरू चाहिन्छ, workflow समन्वय (अनुक्रम, लूप, समानान्तर) आवश्यक छ, अनिवार्य कोडको सट्टा इन्टरफेस-आधारित एजेन्ट डिजाइन मनपर्छ, वा धेरै एजेन्टहरू जसले `outputKey` मार्फत आउटपुट साझा गर्छन्, एकै साथ प्रयोग गर्न चाहनुहुन्छ।

**Supervisor Agent पैटर्न** तब चम्किन्छ जब workflow पहिलेबाट अनुमान लागेको हुँदैन र तपाईं LLM लाई निर्णय गर्न दिन चाहनुहुन्छ, तपाईंको धेरै विशेष एजेन्टहरू हुन्छन् जुन गतिशील समन्वय आवश्यक पर्छ, वार्तालाप प्रणाली बनाइरहेको छ जुन विभिन्न क्षमता समाहित गर्छ, वा तपाईं सबैभन्दा लचिलो, अनुकूली एजेन्ट व्यवहार चाहनुहुन्छ।

कस्टम `@Tool` विधिहरू (Module 04 बाट) र MCP उपकरणहरू (यस मोड्युलबाट) बीच छनोटमा मद्दत गर्न तल तुलना मुख्य व्यापार-वारहरू देखाउँछ — कस्टम उपकरणहरूले टाइट कप्लिङ र अनुप्रयोग-विशिष्ट तर्कका लागि पूर्ण प्रकार सुरक्षा दिन्छन्, MCP उपकरणहरूले मानकीकृत, पुन: प्रयोग योग्य समाकलनहरू प्रदान गर्छन्:

<img src="../../../translated_images/ne/custom-vs-mcp-tools.c4f9b6b1cb65d8a1.webp" alt="Custom Tools vs MCP Tools" width="800"/>

*कस्टम @Tool विधिहरू र MCP उपकरणहरू कहिले प्रयोग गर्ने — कस्टम उपकरणहरू अनुप्रयोग-विशिष्ट तर्कका लागि पूर्ण प्रकार सुरक्षा दिने, MCP उपकरणहरू मानकीकृत समाकलनका लागि जुन अनुप्रयोगहरू बीच काम गर्छ।*
