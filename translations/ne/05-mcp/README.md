## तपाईंले के सिक्नुहुनेछ

तपाईंले संवादात्मक AI बनाउनु भएको छ, प्रॉम्प्टहरूमा दक्षता हासिल गर्नुभयो, जवाफहरूलाई कागजातहरूमा आधारित गर्नुभयो, र उपकरणहरूसँग एजेण्टहरू सिर्जना गर्नुभयो। तर ती सबै उपकरणहरू तपाईंको विशेष अनुप्रयोगको लागि अनुकूलित थिए। के यदि तपाईंले आफ्नो AI लाई मानकीकृत उपकरणहरूको इकोसिस्टममा पहुँच दिन सक्नुहुन्छ जसलाई कुनै पनि व्यक्ति सिर्जना र साझेदारी गर्न सक्छ? यस मोड्युलमा, तपाईंले मोडेल कन्टेक्स्ट प्रोटोकल (MCP) र LangChain4j को एजेण्टिक मोड्युल प्रयोग गरेर ठीक त्यस्तो गर्ने तरिका सिक्नुहुनेछ। हामी पहिले सरल MCP फाइल रिडर देखाउँछौं र त्यसपछि देखाउँछौं कसरी यो सहजै सुपरभाइजर एजेण्ट ढाँचा प्रयोग गरेर उन्नत एजेण्टिक कार्यप्रवाहहरूमा समाहित हुन्छ।

## MCP के हो?

मोडेल कन्टेक्स्ट प्रोटोकल (MCP) ठीक त्यसै चीज हो - AI अनुप्रयोगहरूले बाह्य उपकरणहरू पत्ता लगाउन र प्रयोग गर्नका लागि मानकीकृत तरिका प्रदान गर्छ। प्रत्येक डेटा स्रोत वा सेवाका लागि अनुकूलन गरिएको एकीकरण लेख्नुको सट्टा, तपाईं MCP सर्भरहरूमा जडान गर्नुहुन्छ जुन एकसमान ढाँचामा आफ्ना क्षमता देखाउँछन्। तपाईंको AI एजेण्ट त्यसपछि यी उपकरणहरू स्वचालित रूपमा पत्ता लगाउन र प्रयोग गर्न सक्छ।

तल्लो चित्रले भिन्नता देखाउँछ — MCP बिना, प्रत्येक एकीकरणका लागि अनुकूलित बिन्दु-देखि-बिन्दु वायरिङ आवश्यक छ; MCP सँग, एकल प्रोटोकलले तपाईंको अनुप्रयोगलाई कुनै पनि उपकरणसँग जडान गर्छ:

<img src="../../../translated_images/ne/mcp-comparison.9129a881ecf10ff5.webp" alt="MCP Comparison" width="800"/>

*पहिले MCP: जटिल बिन्दु-देखि-बिन्दु एकीकरणहरू। पछि MCP: एक प्रोटोकल, अनन्त सम्भावनाहरू।*

MCP AI विकासमा एउटा मौलिक समस्या समाधान गर्दछ: प्रत्येक एकीकरण अनुकूलित हुन्छ। GitHub पहुँच गर्न चाहनुहुन्छ? अनुकूलित कोड। फाइलहरू पढ्न चाहनुहुन्छ? अनुकूलित कोड। डेटाबेस सोध्न चाहनुहुन्छ? अनुकूलित कोड। र यी मध्ये कुनै पनि एकीकरण अरू AI अनुप्रयोगहरूसँग काम गर्दैन।

MCP यसलाई मानकीकृत बनाउँछ। एक MCP सर्भरले उपकरणहरूलाई स्पष्ट विवरण र स्किमासँग प्रदर्शन गर्दछ। कुनै पनि MCP क्लाइन्ट जडान गर्न, उपकरणहरू पत्ता लगाउन र तिनीहरूलाई प्रयोग गर्न सक्छ। एक पटक बनाउनुहोस्, सबै ठाउँमा प्रयोग गर्नुहोस्।

तल्लो चित्रले यस वास्तुकला देखाउँछ — एउटा MCP क्लाइन्ट (तपाईंको AI अनुप्रयोग) एउटाभन्दा बढी MCP सर्भरहरूसँग जडान हुन्छ, प्रत्येकले आफ्ना उपकरणहरू मानक प्रोटोकल मार्फत प्रदर्शन गर्छ:

<img src="../../../translated_images/ne/mcp-architecture.b3156d787a4ceac9.webp" alt="MCP Architecture" width="800"/>

*मोडेल कन्टेक्स्ट प्रोटोकल वास्तुकला - मानकीकृत उपकरण पत्ता लगाउने र सञ्चालन गर्ने तरीका*

## MCP कसरी काम गर्छ

अन्तर्निहित रुपमा, MCP ले तहबद्ध वास्तुकला प्रयोग गर्छ। तपाईंको Java अनुप्रयोग (MCP क्लाइन्ट) उपलब्ध उपकरणहरू पत्ता लगाउँछ, ट्रान्सपोर्ट तह (Stdio वा HTTP) मार्फत JSON-RPC अनुरोधहरू पठाउँछ, र MCP सर्भर अपरेशनहरू सञ्चालन गरी परिणाम फर्काउँछ। तलको चित्रले यस प्रोटोकलका प्रत्येक तहको विवरण देखाउँछ:

<img src="../../../translated_images/ne/mcp-protocol-detail.01204e056f45308b.webp" alt="MCP Protocol Detail" width="800"/>

*MCP कसरी काम गर्छ — क्लाइन्टहरूले उपकरणहरू पत्ता लगाउँछन्, JSON-RPC सन्देशहरू आदानप्रदान गर्छन्, र ट्रान्सपोर्ट तह मार्फत अपरेशनहरू सञ्चालन गर्छन्।*

**सर्भर-क्लाइन्ट वास्तुकला**

MCP ले क्लाइन्ट-सर्भर मोडेल प्रयोग गर्छ। सर्भरहरूले उपकरणहरू उपलब्ध गराउँछन् - फाइलहरू पढ्ने, डेटाबेसहरू सोध्ने, API कल गर्ने। क्लाइन्टहरू (तपाईंको AI अनुप्रयोग) सर्भरसँग जडान हुन्छन् र तिनका उपकरणहरू प्रयोग गर्छन्।

LangChain4j सँग MCP प्रयोग गर्न, निम्न Maven निर्भरता थप्नुहोस्:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```

**उपकरण पत्ता लगाउने**

तपाईंको क्लाइन्टले MCP सर्भरसँग जडान गर्दा, यसले सोध्छ "तिमींसँग कुन-कुन उपकरणहरू छन्?" सर्भरले उपलब्ध उपकरणहरूको सूची फर्काउँछ, प्रत्येकसँग विवरण र प्यारामिटर स्किमा। तपाईंको AI एजेण्टले त्यसपछि प्रयोगकर्ताका अनुरोधको आधारमा कुन उपकरणहरू प्रयोग गर्ने निर्णय गर्न सक्छ। तलको चित्रले यो हातमुडाई देखाउँछ — क्लाइन्टले `tools/list` अनुरोध पठाउँछ र सर्भरले आफ्नो उपलब्ध उपकरणहरू विवरण र स्किमासहित फर्काउँछ:

<img src="../../../translated_images/ne/tool-discovery.07760a8a301a7832.webp" alt="MCP Tool Discovery" width="800"/>

*AI ले सुरुमा उपलब्ध उपकरणहरू पत्ता लगाउँछ — यसले अब के-कसरी क्षमता उपलब्ध छन् बुझ्छ र कुन कुन प्रयोग गर्ने निर्णय गर्न सक्छ।*

**ट्रान्सपोर्ट मेकानिज्महरू**

MCP विभिन्न ट्रान्सपोर्ट मेकानिज्महरू समर्थन गर्छ। दुई विकल्पहरू छन् Stdio (स्थानीय सबप्रोसेस संवादका लागि) र Streamable HTTP (रिमोट सर्भरहरूका लागि)। यस मोड्युलले Stdio ट्रान्सपोर्ट देखाउँछ:

<img src="../../../translated_images/ne/transport-mechanisms.2791ba7ee93cf020.webp" alt="Transport Mechanisms" width="800"/>

*MCP ट्रान्सपोर्ट मेकानिज्महरू: HTTP रिमोट सर्भरहरूका लागि, Stdio स्थानीय प्रक्रिया हरूका लागि*

**Stdio** - [StdioTransportDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java)

स्थानीय प्रक्रियाका लागि। तपाईंको अनुप्रयोगले एउटा सर्भर उत्पन्न गर्छ जुन सबप्रोसेसको रूपमा चल्छ र मानक इनपुट/आउटपुट मार्फत संवाद गर्छ। फाइलसिस्टम पहुँच वा कमाण्ड लाइन उपकरणका लागि उपयोगी।

```java
McpTransport stdioTransport = new StdioMcpTransport.Builder()
    .command(List.of(
        npmCmd, "exec",
        "@modelcontextprotocol/server-filesystem@2025.12.18",
        resourcesDir
    ))
    .logEvents(false)
    .build();
```

`@modelcontextprotocol/server-filesystem` सर्भरले तपाईंले निर्दिष्ट गरेको डाइरेक्टोरीहरूमा सीमित गरिएका तलका उपकरणहरू उपलब्ध गराउँछ:

| उपकरण | विवरण |
|-------|-------|
| `read_file` | एकल फाइलको सामग्री पढ्ने |
| `read_multiple_files` | एकै पटकमा धेरै फाइलहरू पढ्ने |
| `write_file` | फाइल सिर्जना गर्ने वा ओभरराइट गर्ने |
| `edit_file` | लक्षित फेला पार्न र प्रतिस्थापन गर्ने सम्पादन गर्ने |
| `list_directory` | कुनै पथमा फाइलहरू र डाइरेक्टोरीहरू सूचीबद्ध गर्ने |
| `search_files` | कुनै ढाँचासँग मेल खाने फाइलहरू पुनरावृत्त रूपमा खोज्ने |
| `get_file_info` | फाइल मेटाडाटा (आकार, टाइमस्ट्याम्प, अनुमति) प्राप्त गर्ने |
| `create_directory` | डाइरेक्टोरी सिर्जना गर्ने (अभिभावक डाइरेक्टोरीसहित) |
| `move_file` | फाइल वा डाइरेक्टोरी सार्ने वा नाम परिवर्तन गर्ने |

तल्लो चित्रले देखाउँछ Stdio ट्रान्सपोर्ट रनटाइममा कसरी काम गर्छ — तपाईंको Java अनुप्रयोगले MCP सर्भरलाई एक सानो सबप्रोसेसको रूपमा उत्पन्न गर्छ र तिनीहरू stdin/stdout पाइपहरू मार्फत संवाद गर्छन्, कुनै नेटवर्क वा HTTP संलग्न छैन:

<img src="../../../translated_images/ne/stdio-transport-flow.45eaff4af2d81db4.webp" alt="Stdio Transport Flow" width="800"/>

*Stdio ट्रान्सपोर्ट क्रियाशील — तपाईंको अनुप्रयोगले MCP सर्भरलाई एउटा सबप्रोसेसको रूपमा उत्पन्न गर्छ र stdin/stdout पाइपमार्फत संवाद गर्छ।*

## एजेण्टिक मोड्युल

MCP ले मानकीकृत उपकरणहरू प्रदान गर्छ भने LangChain4j को **एजेण्टिक मोड्युल** ती उपकरणहरूलाई संयोजन गर्ने एजेन्टहरू निर्माण गर्न एक घोषणात्मक तरिका दिन्छ। `@Agent` एनोटेसन र `AgenticServices` ले तपाईंलाई एजेन्ट व्यवहार इन्टरफेस मार्फत परिभाषित गर्न दिन्छ, अपरिहार्य कोड लेख्नुभन्दा।

यस मोड्युलमा, तपाईंले **सुपरभाइजर एजेण्ट** ढाँचालाई अन्वेषण गर्नुहुनेछ — एक उन्नत एजेण्टिक AI दृष्टिकोण जहाँ "सुपरभाइजर" एजेन्टले प्रयोगकर्ताको अनुरोधका आधारमा चलिरहेको निर्णय गर्दै कुन उप-एजेन्टहरूलाई कल गर्ने निर्णय गर्दछ। हामी यी दुई अवधारणाहरूलाई संयुक्त पार्छौं, जसमा हाम्रो एक उप-एजेन्टलाई MCP-सञ्चालित फाइल पहुँच क्षमताहरू दिइन्छ।

एजेण्टिक मोड्युल प्रयोग गर्न, निम्न Maven निर्भरता थप्नुहोस्:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```
> **सूचना:** `langchain4j-agentic` मोड्युलले अलग संस्करण गुण (`langchain4j.mcp.version`) प्रयोग गर्छ किनकि यसले मुख्य LangChain4j लाइब्रेरीहरू भन्दा फरक निर्धारित तालिकामा रिलिज गर्छ।

> **⚠️ प्रयोगात्मक:** `langchain4j-agentic` मोड्युल **प्रयोगात्मक** छ र परिवर्तन हुन सक्छ। AI सहायकहरू निर्माण गर्ने स्थिर तरिका `langchain4j-core` र अनुकूलित उपकरणहरूसँगै छ (Module 04)।

### सुपरभाइजर एजेण्ट

**सुपरभाइजर एजेण्ट ढाँचा** एक **लचिलो** रूप हो एजेण्टिक AI को। सुपरभाइजरले LLM प्रयोग गरी स्वतन्त्र रूपमा कुन एजेन्टहरूलाई कल गर्ने निर्णय गर्दछ, प्रयोगकर्ताको अनुरोध अनुसार। अर्को उदाहरणमा, हामी MCP-सञ्चालित फाइल पहुँचलाई LLM एजेन्टसँग मिलाएर एउटा नियन्त्रित फाइल पढ्ने → रिपोर्ट बनाउन कार्यप्रवाह सिर्जना गर्छौं।

डेमोमा, `FileAgent` MCP फाइलसिस्टम उपकरणहरू प्रयोग गरेर फाइल पढ्छ, र `ReportAgent` कार्यकारी सारांश (1 वाक्य), 3 प्रमुख बुँदाहरू, र सिफारिशहरूसहित संरचित रिपोर्ट तयार गर्छ। सुपरभाइजरले यो प्रवाह स्वचालित रूपमा संयोजन गर्छ:

<img src="../../../translated_images/ne/supervisor-agent-pattern.06275a41ae006ac8.webp" alt="Supervisor Agent Pattern" width="800"/>

*सुपरभाइजरले आफ्नो LLM प्रयोग गरेर कुन एजेन्टहरू कल गर्ने र कुन क्रमले गर्ने निर्णय गर्छ — हार्डकोड गरिएको रूटिङ आवश्यक छैन।*

हाम्रो फाइल-देखि-रिपोर्ट पाइपलाइनको ठोस कार्यप्रवाह यस्तै देखिन्छ:

<img src="../../../translated_images/ne/file-report-workflow.649bb7a896800de9.webp" alt="File to Report Workflow" width="800"/>

*FileAgent ले MCP उपकरणहरूमार्फत फाइल पढ्छ, त्यसपछि ReportAgent कच्चा सामग्रीलाई संरचित रिपोर्टमा परिणत गर्दछ।*

तलको अनुक्रम चित्रले पूर्ण सुपरभाइजर संयोजन ट्रयाक गर्छ — MCP सर्भर उत्पन्न गर्नेदेखि सुपरभाइजरको स्वायत्त एजेन्ट चयनसम्म, stdio मार्फत उपकरण कलहरू र अन्तिम रिपोर्टसम्म:

<img src="../../../translated_images/ne/supervisor-agent-sequence.1aa389b3bef99956.webp" alt="Supervisor Agent Sequence Diagram" width="800"/>

*सुपरभाइजरले स्वतन्त्र रूपमा FileAgent कल गर्छ (जसले stdio मा MCP सर्भर कल गरेर फाइल पढ्छ) र त्यसपछि ReportAgent कल गर्छ संरचित रिपोर्ट बनाउन — प्रत्येक एजेन्टले आफ्नो आउटपुट साझा गरिएको Agentic Scope मा भण्डारण गर्छ।*

हरेक एजेन्टले आफ्नो आउटपुट **Agentic Scope** (साझा स्मृति) मा भण्डारण गर्छ, जसले तलका एजेन्टहरूलाई पहिलेका परिणामहरू पहुँच गर्न दिन्छ। यसले कसरी MCP उपकरणहरू सहजै एजेण्टिक कार्यप्रवाहमा समाहित हुने देखाउँछ — सुपरभाइजरले *कसरी* फाइलहरू पढिन्छ भन्ने जान्न आवश्यक छैन, केवल `FileAgent` ले गर्न सक्छ भन्नु पर्छ।

#### सुपरभाइजर कसरी काम गर्छ

एजेन्टहरू बनाउनुअघि, तपाईंले MCP ट्रान्सपोर्टलाई क्लाइन्टसँग जडान गरेर यसलाई `ToolProvider` रूपमा आवरण गर्नुपर्छ। MCP सर्भरका उपकरणहरू यसरी तपाईंका एजेन्टहरूलाई उपलब्ध हुन्छन्:

```java
// ट्रान्सपोर्टबाट एउटा MCP क्लाइєн्ट बनाउनुहोस्
McpClient mcpClient = new DefaultMcpClient.Builder()
        .transport(stdioTransport)
        .build();

// क्लाइएन्टलाई ToolProvider को रूपमा र्याप गर्नुहोस् — यसले MCP उपकरणहरूलाई LangChain4j मा जोड्छ
ToolProvider mcpToolProvider = McpToolProvider.builder()
        .mcpClients(List.of(mcpClient))
        .build();
```

अब तपाईं `mcpToolProvider`लाई कुनै पनि एजेन्टमा इंजेक्ट गर्न सक्नुहुन्छ जसलाई MCP उपकरणहरू चाहिन्छ:

```java
// चरण १: FileAgent ले MCP उपकरणहरू प्रयोग गरेर फाइलहरू पढ्छ
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // फाइल अपरेशनहरूको लागि MCP उपकरणहरू छन्
        .build();

// चरण २: ReportAgent ले संरचित रिपोर्टहरू उत्पादन गर्छ
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// Supervisor ले फाइल → रिपोर्ट कार्यप्रवाह व्यवस्थापन गर्छ
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // अन्तिम रिपोर्ट फर्काउनुहोस्
        .build();

// अनुरोधको आधारमा कुन एजेन्टहरूलाई बोलाउने निर्णय Supervisor ले गर्छ
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```

#### FileAgent ले रनटाइममा कसरी MCP उपकरणहरू पत्ता लगाउँछ

तपाईंलाई लाग्न सक्छ: **`FileAgent` ले कसरी npm फाइलसिस्टम उपकरणहरू कसरी चलाउने थाहा पाउँछ?** जवाफ यो हो कि यो थाहा पाउँदैन — **LLM** ले उपकरण स्किमाहरू मार्फत रनटाइममा यो निर्णय गर्छ।
`FileAgent` इन्टरफेस केवल एक **प्राँप्ट परिभाषा** हो। यसले `read_file`, `list_directory`, वा कुनै पनि अन्य MCP उपकरणको हार्डकोड गरिएको ज्ञान राख्दैन। यहाँ अन्त्य-देखि-अन्त्य के हुन्छ:

1. **सर्भर शुरू हुन्छ:** `StdioMcpTransport` ले `@modelcontextprotocol/server-filesystem` npm प्याकेजलाई चाइल्ड प्रक्रिया रूपमा सुरु गर्छ  
2. **उपकरण खोज:** `McpClient` ले सर्भरमा `tools/list` JSON-RPC अनुरोध पठाउँछ, जसले उपकरण नामहरू, विवरणहरू, र प्यारामिटर स्किमाहरू (जस्तै, `read_file` — *"फाइलको पूर्ण सामग्री पढ्नुहोस्"* — `{ path: string }`) फर्काउँछ  
3. **स्किमा इंजेक्शन:** `McpToolProvider` यी पत्ता लगाइएका स्किमाहरूलाई र्याप गरेर LangChain4j लाई उपलब्ध गराउँछ  
4. **एलएलएम निर्णय लिन्छ:** जब `FileAgent.readFile(path)` कल गरिन्छ, LangChain4j ले प्रणाली सन्देश, प्रयोगकर्ता सन्देश, **र उपकरण स्किमाहरूको सूची** एलएलएममा पठाउँछ। एलएलएमले उपकरण विवरणहरू पढ्छ र उपकरण कल उत्पन्न गर्छ (जस्तै, `read_file(path="/some/file.txt")`)  
5. **कार्यन्वयन:** LangChain4j ले उपकरण कल इंटरसेप्ट गर्छ, यसलाई MCP क्लाइन्ट मार्फत Node.js सबप्रोसेसमा रुट गर्छ, नतिजा प्राप्त गर्छ, र एलएलएमलाई फिडब्याक दिन्छ  

यो माथि वर्णन गरिएको उस्तै [टूल डिस्कवरी](../../../05-mcp) संयन्त्र हो, तर विशेष रूपमा एजेन्ट कार्यप्रवाहमा लागू गरिएको। `@SystemMessage` र `@UserMessage` एनोटेसनहरूले एलएलएमको व्यवहार मार्गदर्शन गर्छन्, जबकि इंजेक्ट गरिएको `ToolProvider` ले यसलाई **क्षमताहरू** दिन्छ — एलएलएमले यी दुईलाई रनटाइममा जोड्दछ।

#### प्रतिक्रिया रणनीतिहरू

जब तपाईं `SupervisorAgent` कन्फिगर गर्नुहुन्छ, तपाईंले निर्दिष्ट गर्नुहुन्छ कि सब-एजेन्टहरूले आफ्नो काम सम्पन्न गरेपछि यसले प्रयोगकर्तालाई अन्तिम उत्तर कसरी तयार गर्ने। तलको डायग्रामले तीन उपलब्ध रणनीतिहरू देखाउँछ — LAST ले अन्तिम एजेन्टको आउटपुट सिधै फिर्ता गर्छ, SUMMARY ले सबै आउटपुटहरू एक एलएलएम मार्फत संश्लेषण गर्छ, र SCORED ले मूल अनुरोधको विरूद्ध उच्च स्कोर पाउनेजस्तो आउटपुट चयन गर्छ:  

<img src="../../../translated_images/ne/response-strategies.3d0cea19d096bdf9.webp" alt="Response Strategies" width="800"/>  

*सुपरवाईजरले आफ्नो अन्तिम प्रतिक्रिया तयार गर्दा प्रयोग गर्न सकिने तीन रणनीतिहरू — अन्तिम एजेन्टको परिणाम, संश्लेषित सारांश, वा सर्वोत्तम स्कोर विकल्प छान्नुहोस्।*  

उपलब्ध रणनीतिहरू:  

| रणनीति      | विवरण                                                                                                                                    |  
|-------------|------------------------------------------------------------------------------------------------------------------------------------------|  
| **LAST**    | सुपरवाईजरले अन्तिम सब-एजेन्ट वा कल गरिएको उपकरणको आउटपुट फर्काउँछ। यो उपयोगी हुन्छ जब कार्यप्रवाहमा अन्तिम एजेन्ट विशेषगरी पूर्ण, अन्तिम उत्तर उत्पादन गर्न डिजाइन गरिएको हुन्छ (जस्तै, अनुसन्धान पाइपलाइनमा "सारांश एजेन्ट")। |  
| **SUMMARY** | सुपरवाईजरले आफ्नै आन्तरिक भाषा मोडेल (एलएलएम) प्रयोग गरी सम्पूर्ण अन्तरक्रिया र सबै सब-एजेन्ट आउटपुटहरूको सारांश संश्लेषण गर्छ र त्यो सारांश अन्तिम जवाफको रूपमा फर्काउँछ। यसले प्रयोगकर्तालाई सफा, कुल उत्तर प्रदान गर्छ। |  
| **SCORED**  | प्रणालीले आन्तरिक एलएलएम प्रयोग गरी अन्तिम प्रतिक्रिया र अन्तरक्रियाको सारांश दुवैलाई मूल प्रयोगकर्ता अनुरोधको विरूद्ध स्कोर गर्छ, र जुन आउटपुटले बढी स्कोर पाउँछ त्यसलाई फर्काउँछ। |

#### आउटपुट बुझाइ

डेमो चलाउँदा, तपाईंले सुपरवाईजरले कसरी धेरै एजेन्टहरूलाई समन्वय गर्छ भन्ने संरचित व्याख्या देख्नुहुनेछ। प्रत्येक सेक्सनको अर्थ के हो:  

```
======================================================================
  FILE → REPORT WORKFLOW DEMO
======================================================================

This demo shows a clear 2-step workflow: read a file, then generate a report.
The Supervisor orchestrates the agents automatically based on the request.
```
  
**हेडर**ले कार्यप्रवाह अवधारणा परिचय गराउँछ: फाइल पढ्नबाट रिपोर्ट उत्पादनसम्म केन्द्रित पाइपलाइन।  

```
--- WORKFLOW ---------------------------------------------------------
  ┌─────────────┐      ┌──────────────┐
  │  FileAgent  │ ───▶ │ ReportAgent  │
  │ (MCP tools) │      │  (pure LLM)  │
  └─────────────┘      └──────────────┘
   outputKey:           outputKey:
   'fileContent'        'report'

--- AVAILABLE AGENTS -------------------------------------------------
  [FILE]   FileAgent   - Reads files via MCP → stores in 'fileContent'
  [REPORT] ReportAgent - Generates structured report → stores in 'report'
```
  
**कार्यप्रवाह डायग्राम**ले एजेन्टहरू बीच डेटा प्रवाह देखाउँछ। प्रत्येक एजेन्टसँग फरक भूमिका छ:  
- **FileAgent** ले MCP उपकरणहरू प्रयोग गरेर फाइलहरू पढ्छ र कच्चा सामग्रीलाई `fileContent` मा भण्डारण गर्छ  
- **ReportAgent** ले त्यो सामग्री प्रयोग गरी संरचित रिपोर्ट `report` मा उत्पादन गर्छ  

```
--- USER REQUEST -----------------------------------------------------
  "Read the file at .../file.txt and generate a report on its contents"
```
  
**प्रयोगकर्ता अनुरोध** कार्य देखाउँछ। सुपरवाईजरले यसलाई पार्स गरी FileAgent → ReportAgent कल गर्ने निर्णय गर्छ।  

```
--- SUPERVISOR ORCHESTRATION -----------------------------------------
  The Supervisor decides which agents to invoke and passes data between them...

  +-- STEP 1: Supervisor chose -> FileAgent (reading file via MCP)
  |
  |   Input: .../file.txt
  |
  |   Result: LangChain4j is an open-source, provider-agnostic Java framework for building LLM...
  +-- [OK] FileAgent (reading file via MCP) completed

  +-- STEP 2: Supervisor chose -> ReportAgent (generating structured report)
  |
  |   Input: LangChain4j is an open-source, provider-agnostic Java framew...
  |
  |   Result: Executive Summary...
  +-- [OK] ReportAgent (generating structured report) completed
```
  
**सुपरवाईजर समन्वय**ले २-चरणको कार्यप्रवाह प्रदर्शन गर्छ:  
1. **FileAgent** ले MCP मार्फत फाइल पढ्छ र सामग्री भण्डारण गर्छ  
2. **ReportAgent** ले सामग्री प्राप्त गर्छ र संरचित रिपोर्ट उत्पादन गर्छ  

सुपरवाईजरले यी निर्णयहरू **स्वचालित रूपमा** प्रयोगकर्ता अनुरोधका आधारमा लियो।  

```
--- FINAL RESPONSE ---------------------------------------------------
Executive Summary
...

Key Points
...

Recommendations
...

--- AGENTIC SCOPE (Data Flow) ----------------------------------------
  Each agent stores its output for downstream agents to consume:
  * fileContent: LangChain4j is an open-source, provider-agnostic Java framework...
  * report: Executive Summary...
```
  
#### एजेन्टिक मोड्युल समावेश सुविधाहरूको व्याख्या  

यो उदाहरणले एजेन्टिक मोड्युलका केही उन्नत सुविधाहरू प्रदर्शन गर्छ। अब एजेन्टिक स्कोप र एजेन्ट लिस्नरहरूमा नजिकबाट हेरौं।  

**एजेन्टिक स्कोप**ले साझा मेमोरी देखाउँछ जहाँ एजेन्टहरूले `@Agent(outputKey="...")` प्रयोग गरी आफ्ना परिणामहरू भण्डारण गर्छन्। यसले अनुमति दिन्छ:  
- पछि आउने एजेन्टहरूले पहिलेका एजेन्टहरूको आउटपुट पहुँच गर्न  
- सुपरवाईजरले अन्तिम प्रतिक्रिया संश्लेषण गर्न  
- तपाईंलाई हरेक एजेन्टले के उत्पादन गर्‍यो निरीक्षण गर्न  

तलको डायग्रामले देखाउँछ कि एजेन्टिक स्कोपले कसरी साझा मेमोरीको रूपमा काम गर्छ फाइल-देखि-रिपोर्ट कार्यप्रवाहमा — FileAgent ले आफ्नो आउटपुट `fileContent` की अन्तर्गत लेख्छ, ReportAgent ले त्यो पढ्छ र आफ्नो आउटपुट `report` अन्तर्गत लेख्छ:  

<img src="../../../translated_images/ne/agentic-scope.95ef488b6c1d02ef.webp" alt="Agentic Scope Shared Memory" width="800"/>  

*एजेन्टिक स्कोपले साझा मेमोरीको रूपमा काम गर्छ — FileAgent ले `fileContent` लेख्छ, ReportAgent ले त्यसलाई पढ्छ र `report` लेख्छ, र तपाईंले अन्तिम परिणाम पढ्न सक्नुहुन्छ।*  

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // FileAgent बाट कच्चा फाइल डेटा
String report = scope.readState("report");            // ReportAgent बाट संरचित प्रतिवेदन
```
  
**एजेन्ट लिस्नरहरू**ले एजेन्ट कार्यान्वयनको अनुगमन र डिबगिंग सक्षम पार्छन्। डेमोमा देखिएको चरण-दर-चरण आउटपुट एउटा AgentListener मार्फत आउँछ जुन प्रत्येक एजेन्ट कलमा हुक छ:  
- **beforeAgentInvocation** - सुपरवाईजरले एजेन्ट छान्दा कल हुन्छ, तपाईंले कुन एजेन्ट किन छानियो हेर्न सक्नुहुन्छ  
- **afterAgentInvocation** - एजेन्टले काम पूरा गर्दा कल हुन्छ, यसको नतिजा देखाउँछ  
- **inheritedBySubagents** - जब सत्य हुन्छ, लिस्नरले सबै एजेन्टहरूलाई भाग लिन्छ  

तलको डायग्रामले पूरा Agent Listener जीवनचक्र देखाउँछ, जसमा `onError` ले एजेन्ट चलिरहेको बेला असफलतामा कसरी ह्यान्डल गर्छ तेस्को जानकारी दिन्छ:  

<img src="../../../translated_images/ne/agent-listeners.784bfc403c80ea13.webp" alt="Agent Listeners Lifecycle" width="800"/>  

*एजेन्ट लिस्नरहरूले कार्यान्वयन जीवनचक्रमा हुक गर्छन् — एजेन्ट सुरु हुँदा, समाप्त हुँदा, वा त्रुटि आए जब अनुगमन गर्छन्।*  

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // सबै सह-एजेन्टहरूमा प्रचार गर्नुहोस्
    }
};
```
  
सुपरवाईजर नमुनासँगै, `langchain4j-agentic` मोड्युलले धेरै शक्तिशाली कार्यप्रवाह नमूनाहरू प्रदान गर्छ। तलको डायग्रामले ती पाँचबाट सबै देखाउँछ — साधारण अनुक्रमिक पाइपलाइनदेखि मान्छेलाई अन्तर्भुक्त अनुमोदन कार्यप्रवाहसम्म:  

<img src="../../../translated_images/ne/workflow-patterns.82b2cc5b0c5edb22.webp" alt="Agent Workflow Patterns" width="800"/>  

*एजेन्टहरूलाई समन्वय गर्नका लागि पाँच कार्यप्रवाह नमूनाहरू — साधारण अनुक्रमिक पाइपलाइनदेखि मान्छेलाई अन्तर्भुक्त अनुमोदन कार्यप्रवाहसम्म।*  

| नमूना             | विवरण                                   | प्रयोग मामला                                       |  
|--------------------|-----------------------------------------|--------------------------------------------------|  
| **Sequential**      | एजेन्टहरूलाई क्रमशः कार्यान्वयन गर्नुहोस्, आउटपुट अर्कोमा फ्लो हुन्छ | पाइपलाइनहरू: अनुसन्धान → विश्लेषण → रिपोर्ट           |  
| **Parallel**        | एजेन्टहरू एकै साथ चलाउनुहोस्               | स्वतन्त्र कार्यहरू: मौसम + समाचार + शेयर              |  
| **Loop**            | सर्त पूरा नभएसम्म दोहोर्याउनुहोस्          | गुणस्तर स्कोरिङ: स्कोर ≥ ०.८ नभएसम्म सुधार गर्नुहोस्     |  
| **Conditional**     | सर्तहरूका आधारमा रुटिङ गर्नुहोस्             | वर्गीकरण → विशेषज्ञ एजेन्टलाई रुट गर्नुहोस्                |  
| **Human-in-the-Loop** | मान्छेको जाँच बिन्दुहरू थप्नुहोस्          | अनुमोदन कार्यप्रवाह, सामग्री समीक्षा                       |  

## मुख्य अवधारणाहरू

अब जब तपाईँले MCP र एजेन्टिक मोड्युललाई कार्यान्वयनमा अन्वेषण गर्नुभएको छ, हरेक दृष्टिकोण कहिले प्रयोग गर्ने भन्ने कुरा सारांश गरौं।  

MCP को सबैभन्दा ठूलो फाइदा यसको बढ्दो पारिसर हो। तलको चित्रले देखाउँछ कसरी एउटै सार्वभौमिक प्रोटोकलले तपाईंको AI अनुप्रयोगलाई विविध MCP सर्भरहरूसँग जडान गर्छ — फाइल सिस्टम र डाटाबेस पहुँचदेखि GitHub, इमेल, वेब स्क्र्यापिङ, र अरू धेरै:  

<img src="../../../translated_images/ne/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="MCP Ecosystem" width="800"/>  

*MCP ले सार्वभौमिक प्रोटोकल पारिसर सिर्जना गर्छ — कुनै पनि MCP-अनुकूल सर्भरले कुनै पनि MCP-अनुकूल क्लाइन्टसँग काम गर्छ, अनुप्रयोगहरूमा उपकरण साझा गर्न सक्षम पार्छ।*  

**MCP** तब उपयुक्त हुन्छ जब तपाईं पहिलेका उपकरण पारिसरलाई उपयोग गर्न चाहनुहुन्छ, बहु अनुप्रयोगहरूले साझा गर्न सक्ने उपकरणहरू बनाउन चाहनुहुन्छ, तेस्रो-पक्ष सेवाहरूलाई मानक प्रोटोकलहरूसँग समायोजन गर्न चाहनुहुन्छ, वा उपकरण कार्यान्वयन परिवर्तन गर्न बिना कोड परिवर्तन गर्न चाहनुहुन्छ।  

**एजेन्टिक मोड्युल** त्यतिबेला राम्रो छ जब तपाईंलाई `@Agent` एनोटेसनहरूसहित घोषणात्मक एजेन्ट परिभाषाहरू चाहिन्छ, कार्यप्रवाह समन्वय (अनुक्रमिक, लूप, समानान्तर) आवश्यक छ, सूचक-आधारित एजेन्ट डिजाइन प्राथमिकता दिनुहुन्छ, वा बहु एजेन्टहरूलाई `outputKey` मार्फत आउटपुट साझा गरेर संयोजन गर्दै हुनुहुन्छ।  

**सुपरवाईजर एजेन्ट नमूना** त्यस बेला उत्कृष्ट हुन्छ जब कार्यप्रवाह पूर्वानुमानित हुँदैन र तपाईं एलएलएमलाई निर्णय गर्न दिन चाहनुहुन्छ, जहाँ विभिन्न विशेषज्ञ एजेन्टहरूलाई गतिशील समन्वय चाहिन्छ, संवादात्मक प्रणाली निर्माण गर्दा जसले विभिन्न क्षमताहरूमा रूट गर्छ, वा सबैभन्दा लचिलो, अनुकुलनशील एजेन्ट व्यवहार चाहनुहुन्छ।  

मोड्युल ०४ का अनुकूल `@Tool` विधिहरूसँग यो मोड्युलका MCP उपकरणहरू बीच निर्णय गर्न, तलको तुलना प्रमुख व्यापार-आफहरूलाई हाइलाइट गर्छ — अनुकूल उपकरणहरूले एप-विशिष्ट तर्कका लागि कडा जोड र पूर्ण प्रकार सुरक्षा दिन्छन्, जबकि MCP उपकरणहरूले मानकीकृत, पुन: प्रयोगयोग्य समायोजनहरू प्रदान गर्छन्:  

<img src="../../../translated_images/ne/custom-vs-mcp-tools.c4f9b6b1cb65d8a1.webp" alt="Custom Tools vs MCP Tools" width="800"/>  

*कहिले अनुकूल @Tool विधिहरू र कहिले MCP उपकरणहरू प्रयोग गर्ने — एप-विशिष्ट तर्कका लागि अनुकूल उपकरणहरू पूर्ण प्रकार सुरक्षा सहित, एप्लिकेसनहरू बीच काम गर्ने मानकीकृत समायोजनहरूको लागि MCP उपकरणहरू।*
