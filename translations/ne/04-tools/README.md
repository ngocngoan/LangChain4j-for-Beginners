## तपाईं के सिक्नुहुनेछ

अहिलेसम्म, तपाईंले AI सँग संवाद कसरी गर्ने, प्रभावकारी रूपमा प्रम्प्टहरू संरचना गर्ने, र जवाफहरूलाई तपाईंका कागजातहरूमा आधारित गर्ने सिक्नुभयो। तर अझै एक मौलिक सीमा छ: भाषा मोडेलहरूले मात्र टेक्स्ट सिर्जना गर्न सक्छन्। तिनीहरूले मौसम जाँच्न सक्दैनन्, गणना गर्न सक्दैनन्, डेटाबेस सोध्न सक्दैनन्, वा बाह्य प्रणालीहरूसँग अन्तरक्रिया गर्न सक्दैनन्।

उपकरणहरूले यो परिवर्तन गर्छन्। मोडेललाई प्रयोग गर्न सक्ने कार्यहरू उपलब्ध गराएर, तपाईं यसलाई मात्र टेक्स्ट उत्पादकबाट कार्य गर्न सक्ने एजेन्टमा परिवर्तन गर्नुहुन्छ। मोडेलले कहिले उपकरण चाहिन्छ, कुन उपकरण प्रयोग गर्ने, र कुन प्यारामिटरहरू पास गर्ने निर्णय गर्छ। तपाईंको कोडले कार्यान्वयन गरी नतिजा फिर्ता गर्छ। मोडेल त्यो नतिजालाई आफ्नो उत्तरमा समावेश गर्छ।

## उपकरणहरूसँग AI एजेन्टहरू बुझ्दै

> **📝 नोट:** यस मोड्युलमा "एजेन्टहरू" भन्नाले टूल-कल्लिंग क्षमता सहित सशक्त AI सहायकहरूलाई जनाउँछ। यो फरक छ **Agentic AI** ढाँचाबाट (स्वायत्त एजेन्टहरू जसमा योजना, सम्झना, र बहु-चरण तर्क हुन्छ) जुन हामी [Module 05: MCP](../05-mcp/README.md) मा समेट्नेछौं।

उपकरणहरू बिना, भाषा मोडेलले मात्र तालिम डाटाबाट टेक्स्ट सिर्जना गर्न सक्छ। अहिलेको मौसम सोध्दा, यसले अनुमान लगाउनुपर्छ। तर उपकरण उपलब्ध गराउँदा, यो मौसम API कल गर्न, गणना गर्न, वा डेटाबेस सोध्न सक्छ — र त्यो वास्तविक नतिजालाई आफ्नो उत्तरमा मिसाउँछ।

<img src="../../../translated_images/ne/what-are-tools.724e468fc4de64da.webp" alt="उपकरण बिना र उपकरण सहित" width="800"/>

*उपकरण बिना मोडेलले अनुमान मात्र लगाउन सक्छ — उपकरण सहित यसले API कल गर्न, गणना गर्न, र वास्तविक-समय डाटा फिर्ता गर्न सक्छ।*

उपकरण भएका AI एजेन्ट **Reasoning and Acting (ReAct)** ढाँचामा चल्दछन्। मोडेलले मात्र जवाफ दिनु हुँदैन — यसले के चाहिन्छ भनेर सोच्दछ, उपकरण कल गरेर कार्य गर्दछ, नतिजा अवलोकन गर्दछ, र फेरि कार्य गर्ने या अन्तिम उत्तर दिने निर्णय गर्दछ:

1. **तर्क गर्नुहोस्** — एजेन्टले प्रयोगकर्ताको प्रश्न विश्लेषण गर्छ र आवश्यक जानकारी के हो निर्धारण गर्छ।
2. **कार्य गर्नुहोस्** — एजेन्ट सही उपकरण चयन गर्छ, उचित प्यारामिटर बनाउँछ, र कल गर्छ।
3. **अवलोकन गर्नुहोस्** — एजेन्टले उपकरणको परिणाम प्राप्त गरी मूल्याङ्कन गर्छ।
4. **पुन: दोहोर्याउनुहोस् या जवाफ दिनुहोस्** — थप डाटा आवश्यक भए फेरि लूप गर्छ; नभए प्राकृतिक भाषामा उत्तर प्रदान गर्छ।

<img src="../../../translated_images/ne/react-pattern-detail.96a5efeeb6dd2f61.webp" alt="ReAct ढाँचा" width="800"/>

*ReAct चक्र — एजेन्टले के गर्ने भनेर तर्क गर्दछ, उपकरण कल गरेर कार्य गर्दछ, नतिजा अवलोकन गर्दछ, र अन्तिम उत्तर दिनसम्म दोहोर्याउँछ।*

यो स्वचालित रूपमा हुन्छ। तपाईंले उपकरणहरू र तिनका विवरणहरू परिभाषित गर्नुहुन्छ। मोडेलले निर्णय गर्छ कहिले र कसरी तिनीहरू प्रयोग गर्ने।

## उपकरण कल कसरी काम गर्दछ

### उपकरण परिभाषाहरू

[WeatherTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) | [TemperatureTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/TemperatureTool.java)

तपाईं कार्यहरू स्पष्ट विवरण र प्यारामिटर विनिर्देशहरूसँग परिभाषित गर्नुहुन्छ। मोडेलले ती विवरणहरू आफ्नो सिस्टम प्रम्प्टमा देख्छ र हरेक उपकरणले के गर्छ बुझ्छ।

```java
@Component
public class WeatherTool {
    
    @Tool("Get the current weather for a location")
    public String getCurrentWeather(@P("Location name") String location) {
        // तपाईँको मौसम खोजी तर्क
        return "Weather in " + location + ": 22°C, cloudy";
    }
}

@AiService
public interface Assistant {
    String chat(@MemoryId String sessionId, @UserMessage String message);
}

// सहायक स्वचालित रूपमा स्प्रिंग बुट द्वारा जडित छ:
// - ChatModel बीन
// - @Component कक्षाहरूबाट सबै @Tool विधिहरू
// - सत्र व्यवस्थापनको लागि ChatMemoryProvider
```

तलको चित्रले हरेक एनोटेसनहरू विश्लेषण गर्छ र देखाउँछ कसरी प्रत्येक भागले AI लाई उपकरण कल गर्ने समय र के आर्गुमेन्टहरू पास गर्ने बुझ्न मद्दत गर्छ:

<img src="../../../translated_images/ne/tool-definitions-anatomy.f6468546037cf28b.webp" alt="उपकरण परिभाषाहरूको संरचना" width="800"/>

*उपकरण परिभाषाको संरचना — @Tool ले AI लाई कहिले प्रयोग गर्ने बताउँछ, @P ले प्रत्येक प्यारामिटर वर्णन गर्छ, र @AiService सुरुमा सबैलाई जोड्छ।*

> - "म कल्पनात्मक डाटा सट्टा वास्तविक मौसम API जस्तै OpenWeatherMap कसरी समावेश गर्ने?"
> - "के एक राम्रो उपकरण विवरण बनाउँछ जुन AI लाई सहि प्रयोग गर्न मद्दत गर्छ?"
> - "API त्रुटिहरू र दर सीमाहरू कसरी ह्यान्डल गर्ने?"

### निर्णय लिनु

जब प्रयोगकर्ताले "Seattle को मौसम कस्तो छ?" भन्छ तब मोडेलले यदृच्छिक रूपमा कुनै उपकरण छनौट गर्दैन। यसले प्रयोगकर्ताको मनसाय सबै उपकरण विवरणसँग तुलना गर्छ, प्रासंगिकताका लागि स्कोर गर्छ, र सबै भन्दा उपयुक्त उपकरण चयन गर्छ। त्यसपछि सही प्यारामिटर साथ संरचित कार्य कल उत्पन्न गर्छ — यो अवस्थामा, `location` लाई `"Seattle"` सेट गर्छ।

यदि कुनै उपकरण प्रयोगकर्ताको अनुरोधसँग मेल खाँदैन भने मोडेलले आफ्नै ज्ञानबाट जवाफ दिन्छ। धेरै उपकरण मेल खान्छन् भने, सबैभन्दा विशिष्ट छनौट हुन्छ।

<img src="../../../translated_images/ne/decision-making.409cd562e5cecc49.webp" alt="AI कसरी निर्णय गर्छ कुन उपकरण प्रयोग गर्ने" width="800"/>

*मोडेलले सबै उपलब्ध उपकरणहरू प्रयोगकर्ताको मनसायसँग मूल्याङ्कन गरिसके पछि सबैभन्दा उपयुक्त छनौट गर्छ — त्यसैले स्पष्ट, विशिष्ट उपकरण विवरण लेख्नु जरुरी छ।*

### प्रतिक्रिया उत्पादन

मोडेलले मौसम डाटा प्राप्त गरेर प्रयोगकर्ताका लागि प्राकृतिक भाषामा जवाफ तयार पार्छ।

### वास्तुकला: Spring Boot Auto-Wiring

यस मोड्युलले LangChain4j को Spring Boot एकीकरणको प्रयोग गर्दछ जसमा घोषणात्मक `@AiService` इन्टरफेसहरू छन्। सुरुमा Spring Boot ले प्रत्येक `@Component` जुन `@Tool` मेथडहरू राख्छ, तपाईंको `ChatModel` बीन, र `ChatMemoryProvider` पत्ता लगाउँछ — र सबैलाई एउटै `Assistant` इन्टरफेसमा बोझमुक्त रूपमा जोड्छ।

<img src="../../../translated_images/ne/spring-boot-wiring.151321795988b04e.webp" alt="Spring Boot Auto-Wiring वास्तुकला" width="800"/>

*@AiService इन्टरफेसले ChatModel, उपकरण कम्पोनेन्टहरू, र मेमोरी प्रोभाइडरलाई एकसाथ बाँध्छ — Spring Boot सबै wiring स्वतः सम्हाल्छ।*

यस दृष्टिकोणका प्रमुख फाइदाहरू:

- **Spring Boot auto-wiring** — ChatModel र उपकरणहरू स्वतः इन्जेक्ट हुन्छन्।
- **@MemoryId ढाँचा** — स्वचालित सत्र-आधारित मेमोरी व्यवस्थापन।
- **एकल उदाहरण** — Assistant एक पटक सिर्जना गरेर पुन: प्रयोग गरिन्छ, प्रदर्शन सुधार्छ।
- **प्रकार-सुरक्षित कार्यान्वयन** — जाभा मेथडहरू सिधै प्रकार रूपान्तरणसहित कल हुन्छ।
- **बहु-पटक समन्वय** — उपकरण चेनिङ स्वचालित रूपमा सम्हाल्छ।
- **कोई बोइलरप्लेट छैन** — कुनै म्यानुअल `AiServices.builder()` कलहरू वा मेमोरी ह्यासम्याप छैन।

वैकल्पिक विधिहरू (म्यानुअल `AiServices.builder()`) मा बढी कोड आवश्यक पर्छ र Spring Boot एकीकरणका फाइदाहरू छुट्छन्।

## उपकरण चेनिङ

**उपकरण चेनिङ** — उपकरण-आधारित एजेन्टहरूको वास्तविक शक्ति तब देखिन्छ जब एउटै प्रश्नले धेरै उपकरणहरू आवश्यक पार्छ। "Seattle को मौसम Fahrenheit मा कस्तो छ?" भन्छ भने एजेन्ट स्वतः दुई उपकरण चेन गर्छ: पहिला `getCurrentWeather` कल गरेर सेल्सियसमा तापक्रम जान्छ, त्यसपछि `celsiusToFahrenheit` कल गरेर रूपान्तरण गर्छ — सबै एकै संवाद चरणमा।

<img src="../../../translated_images/ne/tool-chaining-example.538203e73d09dd82.webp" alt="उपकरण चेनिङ उदाहरण" width="800"/>

*उपकरण चेनिङ क्रियामा — एजेन्टले पहिले getCurrentWeather कल गर्छ,पछि सेल्सियस नतिजा celsiusToFahrenheit मा पठाउँछ, र संयुक्त उत्तर दिन्छ।*

यो चल्दो एप्लिकेशनमा यस्तो देखिन्छ — एजेन्टले एकै संवाद चरणमा दुई उपकरण कलहरू चेन गर्छ:

<a href="images/tool-chaining.png"><img src="../../../translated_images/ne/tool-chaining.3b25af01967d6f7b.webp" alt="उपकरण चेनिङ" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*वास्तविक एप्लिकेशन आउटपुट — एजेन्टले स्वतः getCurrentWeather → celsiusToFahrenheit लाई एकै चरणमा चेन गर्छ।*

**सूलभ असफलताहरू** — कल्पनात्मक डाटामा नभएको शहरको मौसम सोध्दा उपकरणले त्रुटि सन्देश फर्काउँछ, र AI ले सहयोग गर्न नसकिने व्याख्या गरेर क्र्यास नगरी जवाफ दिन्छ। उपकरणहरू सुरक्षित रूपमा असफल हुन्छन्।

<img src="../../../translated_images/ne/error-handling-flow.9a330ffc8ee0475c.webp" alt="त्रुटि ह्यान्डलिङ प्रवाह" width="800"/>

*जब उपकरण असफल हुन्छ, एजेन्टले त्रुटि समात्दछ र क्र्यास नहुँदै सहयोगी व्याख्या दिन्छ।*

यो सबै एक संवाद चरणमा हुन्छ। एजेन्टले स्वतन्त्र रूपमा धेरै उपकरण कलहरू समन्वय गर्दछ।

## एप्लिकेशन प्रयोग गर्दै

एप्लिकेशनले वेब इन्टरफेस दिन्छ जहाँ तपाईं AI एजेन्टसँग अन्तरक्रिया गर्न सक्नुहुन्छ जसलाई मौसम र तापक्रम रूपान्तरण उपकरणहरूमा पहुँच छ।

<a href="images/tools-homepage.png"><img src="../../../translated_images/ne/tools-homepage.4b4cd8b2717f9621.webp" alt="AI एजेन्ट उपकरण अन्तरफलक" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*AI एजेन्ट उपकरण इन्टरफेस - छिटो उदाहरणहरू र उपकरणहरूसँग अन्तरक्रिया गर्ने च्याट इन्टरफेस*

### साधारण उपकरण प्रयोग प्रयास गर्नुहोस्
साधा अनुरोधबाट सुरु गर्नुहोस्: "100 डिग्री फरेनहाइटलाई सेल्सियसमा रूपान्तरण गर्नुहोस्"। एजेन्टले बुझ्छ कि यसलाई तापक्रम रूपान्तरण उपकरणको आवश्यकता छ, सही प्यारामिटरहरूको साथमा यसलाई कल गर्छ र परिणाम फर्काउँछ। कति प्राकृतिक लाग्छ - तपाईंले कुन उपकरण प्रयोग गर्ने वा कसरी कल गर्ने भनेर निर्दिष्ट गर्नुभएको छैन।

### परीक्षण उपकरण श्रृंखला

अब अलि जटिल प्रयास गर्नुहोस्: "सिएटलको मौसम कस्तो छ र यसलाई फरेनहाइटमा रूपान्तरण गर्नुहोस्?" एजेन्टले यसलाई चरणहरूमा कसरी काम गर्छ भनेर हेर्नुहोस्। यसले पहिले मौसम प्राप्त गर्छ (जसले सेल्सियस फर्काउँछ), त्यसपछि रूपान्तरण आवश्यक भएको बुझ्छ, रूपान्तरण उपकरणलाई कल गर्छ, र दुवै परिणामलाई एउटै जवाफमा जोड्छ।

### कुराकानी प्रवाह हेर्नुहोस्

च्याट इन्टरफेसले कुराकानी इतिहास राख्छ, जसले तपाईंलाई बहु-राउन्ड अन्तरक्रियाहरू गर्न अनुमति दिन्छ। तपाईंले सबै अघिल्ला प्रश्नहरू र प्रतिक्रियाहरू देख्न सक्नुहुन्छ, जसले कुराकानी ट्र्याक गर्न र एजेन्टले कसरी धेरै आदानप्रदानमा प्रसंग बनाउँछ बुझ्न सजिलो बनाउँछ।

<a href="images/tools-conversation-demo.png"><img src="../../../translated_images/ne/tools-conversation-demo.89f2ce9676080f59.webp" alt="बहु उपकरण कलसहितको कुराकानी" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*सधैँ रूपान्तरण, मौसम खोजी, र उपकरण श्रृंखलाहरू देखाउने बहु-राउन्ड कुराकानी*

### फरक अनुरोधहरूमा प्रयोग गर्नुहोस्

विभिन्न संयोजन प्रयास गर्नुहोस्:
- मौसम खोजीहरू: "टोकियोको मौसम कस्तो छ?"
- तापक्रम रूपान्तरणहरू: "२५°C कति केल्भिन हुन्छ?"
- संयुक्त प्रश्नहरू: "पेरिसको मौसम जाँच गर्नुहोस् र मलाई बताउनुहोस् यदि यो २०°C भन्दा माथि छ भने"

एजेन्टले प्राकृतिक भाषालाई कसरी बुझ्छ र उपयुक्त उपकरण कलहरूसँग मिलाउँछ भन्ने कुरा नोट गर्नुहोस्।

## प्रमुख अवधारणाहरू

### ReAct ढाँचा (कारण र कार्य)

एजेन्टले कारण (के गर्ने निर्णय गर्ने) र कार्य (उपकरण प्रयोग गर्ने) बीच बारी-बारीले काम गर्दछ। यसले स्वतन्त्र समस्या समाधान सक्षम बनाउँछ, केवल निर्देशनहरूमा जवाफ दिनु भन्दा।

### उपकरण विवरण महत्वपूर्ण छन्

तपाईंको उपकरण विवरणहरूको गुणस्तरले एजेन्टको प्रयोग क्षमतामा सिधै प्रभाव पार्छ। स्पष्ट, विशिष्ट विवरणहरूले मोडेललाई कहिले र कसरी प्रत्येक उपकरण कल गर्ने बुझ्न मद्दत गर्छ।

### सत्र व्यवस्थापन

`@MemoryId` एनोटेशनले स्वतः सत्र-आधारित सम्झना व्यवस्थापन सक्षम बनाउँछ। प्रत्येक सत्र ID लाई यसको आफ्नै `ChatMemory` उदाहरण प्राप्त हुन्छ जुन `ChatMemoryProvider` बीनले व्यवस्थापन गर्छ, जसले धेरै प्रयोगकर्ताहरूलाई एउटै समयमा एजेन्टसँग अन्तरक्रिया गराउन अनुमति दिन्छ उनीहरूको कुराकानीहरू मिस नभई।

<img src="../../../translated_images/ne/session-management.91ad819c6c89c400.webp" alt="Session Management with @MemoryId" width="800"/>

*हरेक सत्र ID अलग-अलग कुराकानी इतिहाससँग जोडिएको छ — प्रयोगकर्ताहरू कहिल्यै एकअर्काका सन्देशहरू देख्दैनन्।*

### त्रुटि व्यवस्थापन

उपकरणहरूले असफल हुन सक्छन् — API हरू समयमै प्रतिक्रिया नदिन सक्छन्, प्यारामिटरहरू अमान्य हुन सक्छन्, बाह्य सेवा बन्द हुन सक्छ। उत्पादन एजेन्टहरूले त्रुटि व्यवस्थापन आवश्यक हुन्छ ताकि मोडेलले समस्या स्पष्ट पार्न सके वा विकल्प प्रयास गर्न सकोस्, सम्पूर्ण एप्लिकेशन क्र्यास नगरी। जब कुनै उपकरणले अपवाद फाल्छ, LangChain4j ले यसलाई समात्छ र त्रुटि सन्देशलाई मोडेललाई फर्काउँछ, जसले त्यसपछि प्राकृतिक भाषामा समस्या स्पष्ट पार्न सक्छ।

## उपलब्ध उपकरणहरू

तलको चित्रले तपाईंले निर्माण गर्न सक्ने उपकरणहरूको व्यापक पारिस्थितिकी तन्त्र देखाउँछ। यो मोड्युलले मौसम र तापक्रम उपकरणहरू प्रदर्शन गर्छ, तर समान `@Tool` ढाँचाले कुनै पनि Java विधिका लागि काम गर्छ — डाटाबेस सोधपुछदेखि शुल्क प्रक्रिया सम्म।

<img src="../../../translated_images/ne/tool-ecosystem.aad3d74eaa14a44f.webp" alt="टूल इकोसिस्टम" width="800"/>

*`@Tool` द्वारा एनोटेट गरिएका कुनै पनि Java विधि AI को पहुँचयोग्य हुन्छ — यो ढाँचा डाटाबेस, API, इमेल, फाइल अपरेसनहरू लगायत अन्यमा फैलिएको छ।*

## उपकरण-आधारित एजेन्टहरू कहिले प्रयोग गर्ने

<img src="../../../translated_images/ne/when-to-use-tools.51d1592d9cbdae9c.webp" alt="कहिले उपकरणहरू प्रयोग गर्ने" width="800"/>

*छिटो निर्णय मार्गदर्शक — उपकरणहरू वास्तविक-समय डेटा, गणना, र कार्यहरूको लागि; सामान्य ज्ञान र सिर्जनात्मक कार्यहरूका लागि आवश्यक पर्दैन।*

**उपकरणहरू प्रयोग गर्नुहोस् जब:**
- जवाफ दिनको लागि वास्तविक-समय डेटा आवश्यक छ (मौसम, स्टक मूल्यहरू, भण्डार)
- सरल गणना भन्दा बढी गणना गर्न आवश्यक छ
- डाटाबेस वा API पहुँच गर्नुपर्ने हो
- कार्यहरू लिनुपर्ने हो (इमेल पठाउने, टिकट बनाउने, रेकर्ड अपडेट गर्ने)
- धेरै डेटा स्रोतहरू संयोजन गर्नुपर्ने हो

**उपकरणहरू नप्रयोग गर्नुहोस् जब:**
- प्रश्नहरू सामान्य ज्ञानबाट जवाफ दिन सकिन्छ
- जवाफ पूर्णतया संवादात्मक छ
- उपकरण ढिलाइले अनुभवलाई धेरै ढिलो बनाउँछ

## उपकरणहरू बनाम RAG

मोड्युल ०३ र ०४ दुवै AI लाई फरक तरिकाले विस्तार गर्छन्। RAG मोडेललाई कागजातहरू प्राप्त गरेर **ज्ञान** मा पहुँच दिन्छ। उपकरणहरूले मोडेललाई कार्यहरू गर्न **कार्यहरू** कल गरेर दिन्छ।

<img src="../../../translated_images/ne/tools-vs-rag.ad55ce10d7e4da87.webp" alt="उपकरणहरू बनाम RAG तुलना" width="800"/>

*RAG स्थिर कागजातहरूबाट जानकारी ल्याउँछ — उपकरणहरूले कार्यहरू सञ्चालन गर्छन् र गतिशील, वास्तविक-समय डेटा ल्याउँछन्। धेरै उत्पादन प्रणालीहरूले दुवै संयोजन गर्छन्।*

व्यावहारिक रूपमा, धेरै उत्पादन प्रणालीहरूले दुवै विधिहरू मिलाउँछन्: RAG ले जवाफहरूलाई तपाईंको कागजातहरूमा आधारित बनाउँछ, र उपकरणहरूले प्रत्यक्ष डेटा ल्याउने वा अपरेसन गर्ने काम गर्छ।
