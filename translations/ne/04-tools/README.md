## तपाईंले के सिक्नुहुनेछ

अहिलेसम्म, तपाईंले AI सँग संवाद कसरी गर्ने, प्रभावकारी रूपमा प्रम्प्ट संरचना गर्ने, र प्रतिक्रियाहरूलाई आफ्नो कागजातहरूमा कसरी आधार बनाउने सिक्नुभयो। तर एउटा आधारभूत सीमितता अझै छ: भाषा मोडेलहरू केवल पाठ सिर्जना गर्न सक्छन्। तिनले मौसम जाँच गर्न, गणना गर्न, डाटाबेस सोध्न, वा बाह्य प्रणालीहरूसँग अन्तरक्रिया गर्न सक्दैनन्।

उपकरणहरूले यसलाई परिवर्तन गर्छन्। मोडेललाई कल गर्न सकिने फङ्शनहरू दिने हो भने, तपाईं यसलाई पाठ उत्पादन गर्नेमोडेलबाट यस्तो एजेन्टमा बदल्नुहुन्छ जुन क्रियाहरू गर्न सक्छ। मोडेलले कहिले उपकरण चाहिन्छ, कुन उपकरण प्रयोग गर्ने, र कुन प्यारामिटरहरू पठाउने निर्णय गर्छ। तपाईंको कोडले फङ्शन कार्यान्वयन गर्छ र नतिजा फर्काउँछ। मोडेलले त्यो नतिजा आफ्नो प्रतिक्रियामा समावेश गर्छ।

## उपकरणहरूसँग AI एजेन्टहरू बुझ्नुहोस्

> **📝 नोट:** यस मोड्युलमा "एजेन्टहरू" भनेको उपकरण कल क्षमतासँग सुधारिएको AI सहायकहरूलाई जनाउँछ। यो फरक छ **Agentic AI** ढाँचाहरू (स्वायत्त एजेन्टहरू जुन योजना, स्मृति, र बहु-चरण तर्क गर्छन्) जुन हामी [Module 05: MCP](../05-mcp/README.md) मा समेट्नेछौं।

उपकरणहरू बिना, भाषा मोडेलले मात्र आफ्नो तालिम डाटाबाट पाठ उत्पादन गर्न सक्छ। यसको वर्तमान मौसम सोध्नुहोस्, र यसले अनुमान लगाउनुपर्ने हुन्छ। उपकरणहरू दिनुहोस् भने, यो मौसम API कल गर्न सक्छ, गणना गर्न सक्छ, वा डाटाबेस सोध्न सक्छ — अनि ती वास्तविक नतिजाहरू आफ्नो प्रतिक्रियामा बुन्छ।

<img src="../../../translated_images/ne/what-are-tools.724e468fc4de64da.webp" alt="Without Tools vs With Tools" width="800"/>

*उपकरणहरू बिना मोडेलले मात्र अनुमान लगाउन सक्छ — उपकरणहरूसँग यो API कल गर्न, गणना चलाउन, र वास्तविक-समय डाटा फर्काउन सक्छ।*

उपकरणहरूसँग AI एजेन्टले **तर्क गर्ने र कार्य गर्ने (ReAct)** ढाँचा अनुसरण गर्छ। मोडेलले केवल प्रतिक्रिया दिँदैन — यो के चाहिन्छ भनी सोच्दछ, उपकरण कल गरेर कार्य गर्दछ, नतिजा निरीक्षण गर्छ, र फेरि कार्य गर्ने वा अन्तिम उत्तर दिने निर्णय गर्छ:

1. **तर्क गर्नुहोस्** — एजेन्टले प्रयोगकर्ताको प्रश्न विश्लेषण गरी कुन जानकारी चाहिन्छ निर्धारण गर्छ
2. **कार्य गर्नुहोस्** — एजेन्ट सही उपकरण चयन गर्छ, सही प्यारामिटरहरू बनाउँछ, र कल गर्छ
3. **निरिक्षण गर्नुहोस्** — एजेन्टले उपकरणको आउटपुट प्राप्त गरी नतिजा मूल्याङ्कन गर्छ
4. **पुनः दोहोर्याउनुहोस् वा प्रतिक्रिया दिनुहोस्** — थप डेटा चाहिन्छ भने पुनः चक्र चलाउँछ; नभए प्राकृतिक भाषामा उत्तर तयार गर्छ

<img src="../../../translated_images/ne/react-pattern-detail.96a5efeeb6dd2f61.webp" alt="ReAct Pattern" width="800"/>

*ReAct चक्र — एजेन्टले के गर्ने भनी तर्क गर्छ, उपकरण कल गरेर कार्य गर्छ, नतिजा हेर्छ, र अन्तिम उत्तर दिनसक्नेवेला सम्म पुनः लूप गर्छ।*

यो स्वतः हुन्छ। तपाईंले उपकरणहरू र तिनका विवरणहरू परिभाषित गर्नुहुन्छ। मोडेलले कहिले र कसरी उपकरणहरू प्रयोग गर्ने निर्णय गर्छ।

## उपकरण कल कसरी काम गर्छ

### उपकरण परिभाषाहरू

[WeatherTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) | [TemperatureTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/TemperatureTool.java)

तपाईंले स्पष्ट विवरण र प्यारामिटर निर्दिष्टिहरू सहित फङ्शनहरू परिभाषित गर्नुहुन्छ। मोडेलले यी विवरणहरू आफ्नो सिस्टम प्रम्प्टमा देख्छ र प्रत्येक उपकरणले के गर्छ बुझ्छ।

```java
@Component
public class WeatherTool {
    
    @Tool("Get the current weather for a location")
    public String getCurrentWeather(@P("Location name") String location) {
        // तपाईंको मौसम खोज्ने तर्क
        return "Weather in " + location + ": 22°C, cloudy";
    }
}

@AiService
public interface Assistant {
    String chat(@MemoryId String sessionId, @UserMessage String message);
}

// सहायक स्वचालित रूपमा Spring Boot द्वारा जडित छ:
// - ChatModel बीन
// - @Component कक्षाहरूबाट सबै @Tool विधिहरू
// - सत्र व्यवस्थापनको लागि ChatMemoryProvider
```


तलको चित्रले प्रत्येक एनोटेसन विबरण गर्छ र कसरी प्रत्येक भागले AI लाई उपकरण कल गर्न र के आर्गुमेन्ट दिन भनेर बुझ्न मद्दत गर्छ देखाउँछ:

<img src="../../../translated_images/ne/tool-definitions-anatomy.f6468546037cf28b.webp" alt="Anatomy of Tool Definitions" width="800"/>

*उपकरण परिभाषाको संरचना — @Tool ले AI लाई कहिले प्रयोग गर्ने भन्छ, @P ले प्रत्येक प्यारामिटर वर्णन गर्छ, र @AiService ले सबैलाई सुरुमा जोड्छ।*

> **🤖 [GitHub Copilot](https://github.com/features/copilot) Chat सँग प्रयास गर्नुहोस्:** [`WeatherTool.java`](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) खोल्नुहोस् र सोध्नुहोस्:
> - "मक डेटा सट्टा वास्तविक मौसम API जस्तै OpenWeatherMap कसरी समावेश गर्ने?"
> - "एउटा राम्रो उपकरण वर्णन के हो जसले AI लाई सही प्रयोग गर्न मद्दत गर्छ?"
> - "API त्रुटि र दर सीमा कसरी ह्यान्डल गर्ने उपकरण निर्माणमा?"

### निर्णय लिने प्रक्रिया

जब प्रयोगकर्ताले "सिएटलमा मौसम कस्तो छ?" सोध्छ, मोडेलले कुनै उपकरण बेतरतीब रूपमा छनोट गर्दैन। यो प्रयोगकर्ताको अभिप्रायलाई सबै उपकरण विवरणहरूसँग तुलना गर्छ, सान्दर्भिकताको आधारमा स्कोर गर्छ, र सबैभन्दा उपयुक्त छानेर ठीक प्यारामिटरहरूसहित सञ्चरित फङ्शन कल बनाउँछ — यस अवस्थामा `location` लाई `"Seattle"` सेट गर्छ।

यदि कुनै उपकरण प्रयोगकर्ताको अनुरोधसँग मेल खाँदैन भने, मोडेल आफ्नो ज्ञानबाट जवाफ दिन्छ। यदि धेरै उपकरण मिल्छन् भने सबैभन्दा विशिष्ट उपकरण छान्छ।

<img src="../../../translated_images/ne/decision-making.409cd562e5cecc49.webp" alt="How the AI Decides Which Tool to Use" width="800"/>

*मोडेलले सबै उपलब्ध उपकरणहरू प्रयोगकर्ताको अभिप्रायसँग मूल्याङ्कन गरेर सबैभन्दा उपयुक्त छनोट गर्छ — त्यसैले स्पष्ट र विशिष्ट उपकरण विवरण लेख्नु महत्त्वपूर्ण छ।*

### प्रतिक्रिया उत्पादन

मोडेलले मौसम डाटा प्राप्त गरी प्रयोगकर्ताका लागि प्राकृतिक भाषा प्रतिक्रिया तयार पार्छ।

### वास्तुकला: Spring Boot अटो-वायरिङ

यो मोड्युलले LangChain4j को Spring Boot इंटिग्रेशन प्रयोग गर्दछ, घोषणात्मक `@AiService` इन्टरफेसहरूसँग। सुरुमा Spring Boot ले सबै `@Component` हरू जसमा `@Tool` मेथडहरू छन्, तपाईंको `ChatModel` बीन, र `ChatMemoryProvider` पत्ता लगाउँछ — अनि तिनीहरू सबैलाई एउटा `Assistant` इन्टरफेसमा जोड्छ बिना कुनै बोइलरप्लेट।

<img src="../../../translated_images/ne/spring-boot-wiring.151321795988b04e.webp" alt="Spring Boot Auto-Wiring Architecture" width="800"/>

*@AiService इन्टरफेसले ChatModel, उपकरण कम्पोनेन्टहरू, र मेमोरी प्रदायकलाई जोड्छ — Spring Boot ले सबै वायरिङ स्वतः सम्हाल्छ।*

पूरा अनुरोध जीवनचक्र तलको अनुक्रम चित्रमा — HTTP अनुरोधबाट कन्ट्रोलर, सेवा, र अटो-वायरड प्रोक्सी हुँदै उपकरण कार्यान्वयन र फर्किनेसम्म:

<img src="../../../translated_images/ne/spring-boot-sequence.f83e3d485aa4a3c6.webp" alt="Spring Boot Tool Calling Sequence" width="800"/>

*पूर्ण Spring Boot अनुरोध जीवनचक्र — HTTP अनुरोध कन्ट्रोलर र सेवामा जान्छ र अटो-वायरड Assistant प्रोक्सी मार्फत LLM र उपकरण कलहरू स्वचालित रूपमा व्यवस्थापन हुन्छ।*

यस तरिकाका मुख्य फाइदाहरू:

- **Spring Boot अटो-वायरिङ** — ChatModel र उपकरणहरू स्वतः इन्जेक्ट हुन्छन्
- **@MemoryId ढाँचा** — स्वतः सत्र-आधारित मेमोरी व्यवस्थापन
- **एकल उदाहरण** — Assistant एकपटक सिर्जना गरी पुनः प्रयोग गरिन्छ प्रदर्शन सुधारका लागि
- **प्रकार-सुरक्षित कार्यान्वयन** — Java मेथडहरू सिधै प्रकार रूपान्तरणसहित कल गरिन्छ
- **बहु-संसार आयोजन** — उपकरण चेनिंग स्वचालितसँग ह्यान्डल गर्छ
- **शून्य बोइलरप्लेट** — कुनै म्यानुअल `AiServices.builder()` कल वा मेमोरी ह्यासम्याप आवश्यक छैन

वैकल्पिक तरिकाहरू (म्यानुअल `AiServices.builder()`) बढी कोड चाहिन्छ र Spring Boot एकीकरणका फाइदा गुम्छ।

## उपकरण चेनिंग

**उपकरण चेनिंग** — उपकरण आधारित एजेन्टहरूको साँच्चिकै शक्ति त्यो बेला देखिन्छ जब एउटा प्रश्नले धेरै उपकरणहरूको आवश्यकता पर्छ। "सिएटलमा तापक्रम फारेनहाइटमा कस्तो छ?" भनेर सोध्दा एजेन्टले दुई उपकरणहरू स्वचालित रूपमा चेन गर्छ: पहिले `getCurrentWeather` कल गर्छ तापक्रम सेल्सियसमा ल्याउन, त्यसपछि त्यो मानलाई `celsiusToFahrenheit` मा पास गरी रूपान्तरण गर्छ — सबै एउटै संवाद चरणमा।

<img src="../../../translated_images/ne/tool-chaining-example.538203e73d09dd82.webp" alt="Tool Chaining Example" width="800"/>

*उपकरण चेनिंग क्रियामा — एजेन्टले पहिले getCurrentWeather कल गर्छ, त्यसपछि सेल्सियस नतिजा celsiusToFahrenheit मा डाइरेक्ट गर्दछ, र संयुक्त उत्तर दिन्छ।*

**सौम्य असफलताहरू** — त्यस्तो शहरको मौसम माग्नुहोस् जुन मोक डाटामा छैन। उपकरणले त्रुटि सन्देश फर्काउँछ, र AI ले बताउँछ कि यसले मद्दत गर्न सक्दैन भनी जम्पिने सट्टा। उपकरणहरूले सुरक्षित रूपमा असफल हुन्छन्। तलको चित्रले दुई तरिकाको तुलना गर्छ — उपयुक्त त्रुटि ह्यान्डलिङ भएमा एजेन्टले एक्सेप्सन समातेर मद्दतपूर्ण प्रतिक्रिया दिन्छ, नभए सम्पूर्ण एप्लिकेसन क्र्यास हुन्छ:

<img src="../../../translated_images/ne/error-handling-flow.9a330ffc8ee0475c.webp" alt="Error Handling Flow" width="800"/>

*जब उपकरण असफल हुन्छ, एजेन्टले त्रुटि समातेर क्र्यास नहुने गरी सहयोगी व्याख्या गर्छ।*

यो सबै एउटै संवाद चरणमा हुन्छ। एजेन्टले धेरै उपकरण कलहरू स्वतः व्यवस्थापन गर्छ।

## अनुप्रयोग प्रयोग

अनुप्रयोगले वेब इन्टरफेस प्रदान गर्दछ जहाँ तपाईं एक AI एजेन्टसँग अन्तरक्रिया गर्न सक्नुहुन्छ जसले मौसम र तापक्रम रूपान्तरण उपकरणहरूसम्म पहुँच राख्छ। इन्टरफेस यसरी देखिन्छ — यसमा छिटो सुरु गर्ने उदाहरणहरू र अनुरोध पठाउनको लागि एक च्याट प्यानल समावेश छ:

<a href="images/tools-homepage.png"><img src="../../../translated_images/ne/tools-homepage.4b4cd8b2717f9621.webp" alt="AI Agent Tools Interface" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*AI एजेन्ट उपकरण इन्टरफेस - छिटो उदाहरणहरू र उपकरणहरूसँग अन्तरक्रिया गर्ने च्याट इन्टरफेस*

### साधारण उपकरण प्रयोग प्रयास गर्नुहोस्

सिधा अनुरोधबाट सुरु गर्नुहोस्: "100 डिग्री फरेनहाइटलाई सेल्सियसमा रूपान्तरण गर्नुहोस्"। एजेन्टले च्यानिन्छ कि यसलाई तापक्रम रूपान्तरण उपकरण चाहिन्छ, सही प्यारामिटरहरू सहित यसलाई कल गर्दछ, र परिणाम फर्काउँछ। यो कति स्वाभाविक महसुस हुन्छ - तपाईंले कुन उपकरण प्रयोग गर्ने वा कसरी कल गर्ने भने निर्दिष्ट गर्नुभएन।

### उपकरण श्रृंखला परीक्षण गर्नुहोस्

अब केही जटिल प्रयास गर्नुहोस्: "सिएटलमा मौसम कस्तो छ र यसलाई फरेनहाइटमा रूपान्तरण गर्नुहोस्?" हेर्नुहोस् एजेन्टले यो चरणहरूमा कसरी काम गर्छ। यसले पहिले मौसम पाउँछ (जसले सेल्सियस फर्काउँछ), रूपान्तरण गर्न फरेनहाइटमा कल गर्नुपर्छ भनी बुझेको, रूपान्तरण उपकरण कल गरेर दुई परिणामहरूलाई एउटै उत्तरमा मिलाउँछ।

### संवाद प्रवाह हेर्नुहोस्

च्याट इन्टरफेसले संवाद इतिहास राख्छ, जसले तपाईंलाई बहु-पटक अन्तरक्रिया गर्न अनुमति दिन्छ। तपाईं सबै अघिल्ला प्रश्नहरू र उत्तरहरू देख्न सक्नुहुन्छ, यसले संवादलाई ट्र्याक गर्न र एजेन्टले कसरी प्रवृत्ति निर्माण गरिरहेको बुझ्न सजिलो बनाउँछ।

<a href="images/tools-conversation-demo.png"><img src="../../../translated_images/ne/tools-conversation-demo.89f2ce9676080f59.webp" alt="Conversation with Multiple Tool Calls" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*धेरै पटकको संवाद जसले साधारण रूपान्तरणहरू, मौसम खोजहरू र उपकरण श्रृंखला देखाउँछ*

### विभिन्न अनुरोधहरूसँग प्रयोग गर्नुहोस्

विभिन्न संयोजनहरू प्रयास गर्नुहोस्:
- मौसम खोजहरू: "टोकियोमा मौसम कस्तो छ?"
- तापक्रम रूपान्तरणहरू: "२५°C कति केल्भिन हुन्छ?"
- संयुक्त प्रश्नहरू: "पेरिसमा मौसम जाँच गर्नुहोस् र २०°C भन्दा माथि छ कि छैन भन्नुहोस्"

हेर्नुहोस् एजेन्टले प्राकृतिक भाषा कसरी व्याख्या गर्छ र उपयुक्त उपकरण कलहरूसँग मिलाउँछ।

## प्रमुख अवधारणाहरू

### ReAct ढाँचा (तर्क र कार्य)

एजेन्टले तर्क (के गर्ने निर्णय) र कार्य (उपकरणहरू प्रयोग) बीच वैकल्पिक गर्दछ। यसले स्वतन्त्र समस्याहरू समाधान गर्न सक्षम बनाउँछ न कि केवल निर्देशनमा प्रतिक्रिया दिन।

### उपकरण विवरण महत्वपूर्ण छन्

तपाईंको उपकरण विवरणहरूको गुणस्तरले एजेन्टले तिनीहरूलाई कति राम्रोसँग प्रयोग गर्ने भन्ने कुरा असर गर्द्छ। स्पष्ट, विशिष्ट विवरणहरूले मोडेललाई कहिले र कसरी प्रत्येक उपकरण कल गर्ने बुझ्न मद्दत गर्दछ।

### सत्र व्यवस्थापन

`@MemoryId` एनोटेसनले स्वचालित सत्र-आधारित स्मृति व्यवस्थापन सक्षम पार्दछ। प्रत्येक सत्र ID लाई `ChatMemoryProvider` बीनले व्यवस्थापन गर्ने आफ्नै `ChatMemory` उदाहरण प्राप्त हुन्छ, जसले गर्दा धेरै प्रयोगकर्ताहरूले एकसाथ एजेन्टसँग अन्तरक्रिया गर्दा तिनीहरूको संवादहरू मिश्रित हुँदैनन्। तलको चित्रले देखाउँछ कसरी धेरै प्रयोगकर्ताहरू सत्र ID हरूको आधारमा अलग-अलग स्मृति भण्डारणमा रूट हुन्छन्:

<img src="../../../translated_images/ne/session-management.91ad819c6c89c400.webp" alt="Session Management with @MemoryId" width="800"/>

*हरेक सत्र ID अलग-अलग संवाद इतिहासमा नक्साङ्कन हुन्छ — प्रयोगकर्ताहरूले कहिल्यै एकअर्काको सन्देशहरू हेर्दैनन्।*

### त्रुटि ह्यान्डलिंग

उपकरणहरू असफल हुन सक्छन् — API समय सकिन्छ, प्यारामिटरहरू अमान्य हुन सक्छन्, बाह्य सेवाहरू बन्द हुन सक्छन्। उत्पादन एजेन्टहरूले त्रुटि ह्यान्डलिंग चाहिन्छ ताकि मोडेलले समस्याहरू व्याख्या गर्न वा विकल्प प्रयास गर्न सकोस् न कि सम्पूर्ण अनुप्रयोग क्र्यास होओस्। जब उपकरणले अपवाद फ्याक्छ, LangChain4j यसलाई समात्छ र त्रुटि सन्देश मोडेलमा फिर्ता पठाउँछ, जसले प्राकृतिक भाषामा समस्याको व्याख्या गर्न सक्छ।

## उपलब्ध उपकरणहरू

तलको चित्रले तपाईंले सिर्जना गर्न सक्ने उपकरणहरूको फराकिलो पारिसर देखाउँछ। यस मोड्युलले मौसम र तापक्रम उपकरणहरू प्रदर्शन गर्दछ, तर एउटै `@Tool` ढाँचाले कुनै पनि Java मेथडमा काम गर्छ — डाटाबेस क्वेरीहरू देखि भुक्तानी प्रक्रिया सम्म।

<img src="../../../translated_images/ne/tool-ecosystem.aad3d74eaa14a44f.webp" alt="Tool Ecosystem" width="800"/>

*कुनै पनि Java मेथड जसमा @Tool एनोटेसन हुन्छ AI को लागि उपलब्ध हुन्छ — यो ढाँचा डाटाबेस, API, इमेल, फाइल अपरेसन र थपमा विस्तार हुन्छ।*

सबै अनुरोधहरूलाई उपकरणहरू चाहिंदैनन्। निर्णय योमा निर्भर छ कि AI ले बाह्य प्रणालीहरूसँग अन्तरक्रिया गर्नुपर्ने छ वा आफ्नै ज्ञानबाट जवाफ दिन सक्नेछ। तलको मार्गदर्शकले सारांश गर्दछ उपकरण कहिले मूल्य थप्छ र कहिले अनावश्यक हुन्छ:

<img src="../../../translated_images/ne/when-to-use-tools.51d1592d9cbdae9c.webp" alt="When to Use Tools" width="800"/>

*छिटो निर्णय मार्गदर्शक — उपकरणहरू वास्तविक-समय डेटा, गणना, र कार्यहरूको लागि हुन्; सामान्य ज्ञान र सिर्जनात्मक कार्यहरूलाई आवश्यक पर्दैन।*

## उपकरणहरू विरुद्ध RAG

मोडुलहरू 03 र 04 दुबैले AI को क्षमता बढाउँछन्, तर मौलिक रूपमा फरक तरिकाले। RAG ले मोडेललाई **ज्ञान** भेट्टाउन दस्तावेजहरू पुनःप्राप्त गर्ने क्षमता दिन्छ। उपकरणहरूले मोडेललाई **कार्यहरू** गर्न अनुमति दिन्छन्, अर्थात् कार्यहरू कल गर्ने। तलको चित्रले यी दुई दृष्टिकोणहरू एकअर्कालाई सँगसँगै तुलना गर्दछ — हरेक कार्यप्रवाह कसरी चल्छ र तिनीहरूको विचमा कस्ता व्यापार-छुटहरू छन्:

<img src="../../../translated_images/ne/tools-vs-rag.ad55ce10d7e4da87.webp" alt="Tools vs RAG Comparison" width="800"/>

*RAG स्थिर दस्तावेजहरूबाट सूचना पुनःप्राप्त गर्छ — उपकरणहरूले कार्यहरू सञ्चालन र गतिशील, वास्तविक-समय डेटा ल्याउँछन्। धेरै उत्पादन प्रणालीहरूले दुवै संयोजन गर्छन्।*

व्यवहारमा, धेरै उत्पादन प्रणालीहरूले दुवै दृष्टिकोण संयोजन गर्दछन्: RAG तपाईंको डकुमेन्टेसनमा आधारभूत जवाफ दिन र उपकरणहरू लाइव डाटा ल्याउन वा अपरेसनहरू गर्ने।
