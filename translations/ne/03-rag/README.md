## तपाईंले के सिक्नेछु

अघिल्लो मोड्युलहरूमा, तपाईंले AI सँग कसरी कुराकानी गर्ने र आफ्ना प्रॉम्प्टहरू प्रभावकारी रूपमा संरचना गर्ने सिक्नुभयो। तर एउटा मौलिक सीमा छ: भाषा मोडेलहरूले केवल तिनीहरूले प्रशिक्षणको क्रममा सिकेका कुरा मात्र जान्दछन्। तिनीहरूले तपाईंको कम्पनीको नीति, तपाईंको परियोजनाको कागजात, वा कुनै जानकारी जुन तिनीहरूले प्रशिक्षणमा प्राप्त गरेका थिएनन् सोधिने प्रश्नहरूको उत्तर दिन सक्दैनन्।

RAG (Retrieval-Augmented Generation) ले यो समस्या समाधान गर्छ। मोडेललाई तपाईंको जानकारी सिकाउन खोज्नुभन्दा (जुन महँगो र असम्भव छ), तपाईंले यसलाई तपाईंको कागजातहरूमा खोजी गर्ने क्षमता दिनुहुन्छ। जब कसैले प्रश्न सोध्छ, प्रणाली सान्दर्भिक जानकारी फेला पार्छ र त्यसलाई प्रॉम्प्टमा समावेश गर्दछ। मोडेल त्यसपछि प्राप्त सन्दर्भको आधारमा उत्तर दिन्छ।

RAG लाई मोडेललाई सन्दर्भ पुस्तकालय दिने कुरा सम्झनुहोस्। जब तपाईं प्रश्न सोध्नुहुन्छ, प्रणालीले:

1. **प्रयोगकर्ता प्रश्न** - तपाईंले प्रश्न सोध्नुहुन्छ
2. **एम्बेडिङ्ग** - तपाईंको प्रश्नलाई भेक्टरमा रूपान्तरण गर्छ
3. **भेक्टर खोज** - समान कागजातका टुक्राहरू पत्ता लगाउँछ
4. **सन्दर्भ संयोजन** - सान्दर्भिक टुक्राहरू प्रॉम्प्टमा थप्दछ
5. **प्रतिक्रिया** - LLM सन्दर्भको आधारमा उत्तर उत्पादन गर्दछ

यसले मोडेलको प्रतिक्रियाहरूलाई यसको प्रशिक्षण ज्ञानमा आधारित नभई तपाईंको वास्तविक डाटामा आधारित बनाउँछ।

## RAG बुझ्दै

तलको चित्रले मुख्य अवधारणालाई देखाउँछ: मोडेलको प्रशिक्षण डाटामा मात्र निर्भर नहुने, RAG ले प्रत्येक उत्तर उत्पादन गर्नुअघि तपाईंका कागजातहरूको सन्दर्भ पुस्तकालय दिन्छ।

<img src="../../../translated_images/ne/what-is-rag.1f9005d44b07f2d8.webp" alt="RAG के हो" width="800"/>

यहाँ भिन्न चरणहरू कसरी जडान हुन्छन् देखाइएको छ। प्रयोगकर्ताको प्रश्न चार चरणहरू — एमबेडिङ्ग, भेक्टर खोज, सन्दर्भ संयोजन, र उत्तर उत्पादन — बाट बग्छ:

## यो कसरी काम गर्छ

### कागजात प्रक्रिया

[DocumentService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/DocumentService.java)

जब तपाईं कागजात अपलोड गर्नुहुन्छ, प्रणाली त्यसलाई पार्स गर्छ (PDF वा सादा पाठ), फाइलनाम जस्ता मेटाडेटा थप्छ र त्यसपछि यसलाई टुक्राहरूमा टुक्र्याउँछ — स-साना भागहरू जुन मोडेलको सन्दर्भ विन्डोमा आरामले फिट हुन्छन्। यी टुक्राहरू सानो ओभरल्याप राख्छन् ताकि सन्दर्भ हराउँदैन।

```java
// अपलोड गरिएको फाइल पार्स गर्नुहोस् र यसलाई LangChain4j Document मा र्याप गर्नुहोस्
Document document = Document.from(content, metadata);

// ३० टोकन ओभरलैप सहित ३००-टोकन भागहरूमा विभाजन गर्नुहोस्
DocumentSplitter splitter = DocumentSplitters
    .recursive(300, 30);

List<TextSegment> segments = splitter.split(document);
```

तलको चित्रले यसलाई दृश्यात्मक रूपमा देखाउँछ। हरेक टुक्राले आफ्नो छिमेकीहरूसँग केही टोकन साझा गर्छ — ३० टोकन ओभरल्यापले कुनै महत्वपूर्ण सन्दर्भ छुट्न नपाओस् भन्ने सुनिश्चित गर्छ:

### एम्बेडिङ्गहरूको सिर्जना

[LangChainRagConfig.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/config/LangChainRagConfig.java)

प्रत्येक टुक्रालाई एक संख्यात्मक प्रतिनिधित्वमा रूपान्तरण गरिन्छ, जसलाई एम्बेडिङ भनिन्छ - यो पाठको अर्थ समेट्ने एक गणितीय फिङ्गरप्रिन्ट जस्तो हुन्छ। समान पाठले समान एम्बेडिङ्गहरू उत्पादन गर्छ।

```java
@Bean
public EmbeddingModel embeddingModel() {
    return OpenAiOfficialEmbeddingModel.builder()
        .baseUrl(azureOpenAiEndpoint)
        .apiKey(azureOpenAiKey)
        .modelName(azureEmbeddingDeploymentName)
        .build();
}

EmbeddingStore<TextSegment> embeddingStore = 
    new InMemoryEmbeddingStore<>();
```

तलको कक्षा चित्रले LangChain4j का यी कम्पोनेन्टहरू कसरी जडान हुन्छन् देखाउँछ। `OpenAiOfficialEmbeddingModel` पाठलाई भेक्टरमा रूपान्तरण गर्छ, `InMemoryEmbeddingStore` भेक्टरहरूलाई तिनीहरूको मूल `TextSegment` डाटासँग राख्दछ, र `EmbeddingSearchRequest` पुनःप्राप्ति प्यारामिटरहरू जस्तै `maxResults` र `minScore` नियन्त्रण गर्छ:

<img src="../../../translated_images/ne/rag-langchain4j-classes.bbf3aa9077ab443d.webp" alt="LangChain4j RAG कक्षाहरू" width="800"/>

एकचोटि एम्बेडिङ्गहरू संग्रहित भएपछि, समान सामग्री नै भेक्टर स्पेसमा समूहबद्ध हुन्छ। तलको दृश्यले देखाउँछ कि सम्बन्धित विषयहरूबारे कागजातहरू कसरी नजिकका बिन्दुहरूमा आउँछन्, जसले semantic search सम्भव बनाउँछ:

### सामान्य खोज

[RagService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/RagService.java)

जब तपाईं प्रश्न सोध्नुहुन्छ, तपाईंको प्रश्न पनि एक एम्बेडिङ हुन्छ। प्रणाली तपाईंको प्रश्नको एम्बेडिङलाई सबै कागजात टुक्राहरूको एम्बेडिङसँग तुलना गर्छ। यो त्यस्ता टुक्राहरू पत्ता लगाउँछ जसका अर्थ धेरै समान छन् - केवल कुञ्जीशब्द मिल्नु मात्र होइन, वास्तविक अर्थगत समानता।

```java
Embedding queryEmbedding = embeddingModel.embed(question).content();

EmbeddingSearchRequest searchRequest = EmbeddingSearchRequest.builder()
    .queryEmbedding(queryEmbedding)
    .maxResults(5)
    .minScore(0.5)
    .build();

EmbeddingSearchResult<TextSegment> searchResult = embeddingStore.search(searchRequest);
List<EmbeddingMatch<TextSegment>> matches = searchResult.matches();

for (EmbeddingMatch<TextSegment> match : matches) {
    String relevantText = match.embedded().text();
    double score = match.score();
}
```

तलको चित्रले semantic search लाई परम्परागत कुञ्जीशब्द खोजसँग तुलना गर्छ। "vehicle" को कुञ्जीशब्द खोजले "cars and trucks" बारेको टुक्रा टिप्दैन, तर semantic search ले बुझ्छ कि तिनीहरू उस्तै कुरा हुन् र उच्च स्कोरको मिल्दा रूपमा फर्काउँछ:

<img src="../../../translated_images/ne/semantic-search.6b790f21c86b849d.webp" alt="सामान्य खोज" width="800"/>

### उत्तर उत्पादन

[RagService.java](../../../03-rag/src/main/java/com/example/langchain4j/rag/service/RagService.java)

सबैभन्दा सान्दर्भिक टुक्राहरूलाई एउटा संरचित प्रॉम्प्टमा मिलाइन्छ जुन स्पष्ट निर्देशनहरू, प्राप्त सन्दर्भ, र प्रयोगकर्ताको प्रश्न समावेश गर्दछ। मोडेलले ती टुक्राहरू पढेर त्यसमा आधारित उत्तर दिन्छ — यो केवल पहिले देखि उपलब्ध जानकारी प्रयोग गर्छ जसले भ्रम उत्पादन हुनबाट रोक्छ।

```java
String context = matches.stream()
    .map(match -> match.embedded().text())
    .collect(Collectors.joining("\n\n"));

String prompt = String.format("""
    Answer the question based on the following context.
    If the answer cannot be found in the context, say so.

    Context:
    %s

    Question: %s

    Answer:""", context, request.question());

String answer = chatModel.chat(prompt);
```

तलको चित्रले यो संयोजन कसरी हुन्छ देखाउँछ — खोज चरणका सबैभन्दा शीर्ष स्कोर प्राप्त टुक्राहरू प्रॉम्प्ट टेम्प्लेटमा राखिन्छन्, र `OpenAiOfficialChatModel` एक आधारभूत उत्तर उत्पन्न गर्छ:

**विकल्प १: Spring Boot ड्यासबोर्ड प्रयोग गर्दै (VS Code प्रयोगकर्ताहरूको लागि सिफारिस गरिएको)**

डेभ कन्टेनरमा Spring Boot ड्यासबोर्ड एक्सटेन्सन समावेश छ, जसले सबै Spring Boot अनुप्रयोगहरूलाई व्यवस्थापन गर्न दृश्यात्मक इन्टरफेस प्रदान गर्छ। यसलाई VS Code को Activity Bar मा Spring Boot आइकनबाट फेला पार्न सकिन्छ।

Spring Boot ड्यासबोर्डबाट तपाईं:

- वर्कस्पेसमा सबै उपलब्ध Spring Boot अनुप्रयोगहरू देख्न सक्नुहुन्छ
- एक क्लिकमा अनुप्रयोगहरू सुरु/रोक्न सक्नुहुन्छ
- वास्तविक समयमा अनुप्रयोगका लगहरू हेर्न सक्नुहुन्छ
- अनुप्रयोग स्थितिमा नजर राख्न सक्नुहुन्छ

"rag" सँग जोडिएको प्ले बटनमा क्लिक गरी यो मोड्युल सुरु गर्नुहोस्, वा सबै मोड्युलहरू एकैपटक सुरु गर्नुहोस्।

## अनुप्रयोग प्रयोग गर्दै

अनुप्रयोगले कागजात अपलोड र प्रश्न सोध्न वेब इन्टरफेस प्रदान गर्छ।

<a href="images/rag-homepage.png"><img src="../../../translated_images/ne/rag-homepage.d90eb5ce1b3caa94.webp" alt="RAG अनुप्रयोग इन्टरफेस" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*RAG अनुप्रयोग इन्टरफेस - कागजातहरू अपलोड गर्नुहोस् र प्रश्नहरू सोध्नुहोस्*

### कागजात अपलोड गर्नुहोस्

पहिले एक कागजात अपलोड गर्नुहोस् - TXT फाइलहरू परीक्षणका लागि राम्रो काम गर्छन्। यस निर्देशिकामा `sample-document.txt` दिइएको छ जसमा LangChain4j सुविधाहरू, RAG कार्यान्वयन, र राम्रो अभ्यासहरूका बारेमा जानकारी छ - प्रणाली परीक्षणका लागि उपयुक्त।

प्रणाली तपाइँको कागजात प्रक्रिया गर्दछ, टुक्र्याउँछ र हरेक टुक्राको लागि एम्बेडिङ्गहरू बनाउँछ। यो तपाईंले अपलोड गर्दा स्वतः हुन्छ।

### प्रश्नहरू सोध्नुहोस्

अब कागजात सामग्रीका बारेमा विशेष प्रश्नहरू सोध्नुहोस्। केहि तथ्यात्मक प्रयास गर्नुहोस् जस स्पष्ट रूपमा कागजातमा उल्लेख छ। प्रणाली सान्दर्भिक टुक्राहरू खोज्छ, तिनीहरूलाई प्रॉम्प्टमा समावेश गर्दछ, र उत्तर उत्पन्न गर्दछ।

### स्रोत सन्दर्भहरू जाँच्नुहोस्

ध्यान दिनुहोस् कि प्रत्येक उत्तरमा स्रोत सन्दर्भहरू र समानता स्कोरहरू हुन्छन्। यी स्कोरहरू (० देखि १ सम्म) देखाउँछन् कि हरेक टुक्रा तपाईंको प्रश्नसँग कति सान्दर्भिक छ। उच्च स्कोरले राम्रो मिलन जनाउँछ। यसले तपाईंलाई स्रोत सामग्रीसँग जवाफलाई प्रमाणित गर्न सक्षम पार्छ।

<a href="images/rag-query-results.png"><img src="../../../translated_images/ne/rag-query-results.6d69fcec5397f355.webp" alt="RAG प्रश्न नतिजाहरू" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*प्रश्न नतिजाहरू - उत्तर स्रोत सन्दर्भ र सान्दर्भिकता स्कोरसहित*

### प्रश्नहरूसँग प्रयोगात्मक गर्नुहोस्

विभिन्न प्रकारका प्रश्नहरू प्रयास गर्नुहोस्:
- विशिष्ट तथ्यहरू: "मुख्य विषय के हो?"
- तुलना: "X र Y बीच के फरक छ?"
- सारांश: "Z को बारेमा मुख्य बुँदाहरू संक्षेप गर्नुहोस्"

हेर्नुहोस् प्रश्न र कागजात सामग्रीबीचको मिलान आधारमा कसरी सान्दर्भिकता स्कोरहरू परिवर्तन हुन्छन्।

## मुख्य अवधारणाहरू

### चंकिङ्ग रणनीति

कागजातहरूलाई ३००-टोकनका टुक्रामा विभाजन गरिन्छ जसमा ३० टोकन ओभरल्याप हुन्छ। यो सन्तुलनले प्रत्येक टुक्रा पर्याप्त सन्दर्भ सहित अर्थपूर्ण रहने सुनिश्चित गर्छ र धेरै टुक्राहरू प्रॉम्प्टमा समावेश गर्न सकिन्छ।

### समानता स्कोरहरू

प्रत्येक प्राप्त टुक्रासँग प्रयोगकर्ताको प्रश्नसँग मिल्दोजुल्दो मापन गर्ने ० देखि १ सम्मको समानता स्कोर हुन्छ। तलको चित्रले स्कोर दायराहरू र प्रणालीले तिनीहरूलाई नतिजाहरू फिल्टर गर्न कसरी प्रयोग गर्छ देखाउँछ:

<img src="../../../translated_images/ne/similarity-scores.b0716aa911abf7f0.webp" alt="समानता स्कोरहरू" width="800"/>

स्कोर दायराहरू:
- 0.7-1.0: अत्यन्त सान्दर्भिक, ठ्याक्कै मिल्दोजुल्दो
- 0.5-0.7: सान्दर्भिक, राम्रो सन्दर्भ
- 0.5 भन्दा मुनि: हटाइन्छ, धेरै फरक

प्रणालीले गुणस्तर सुनिश्चित गर्न न्यूनतम थ्रेसहोल्ड भन्दा माथिका टुक्रा मात्र पुनःप्राप्त गर्छ।

### इन-मेमोरी स्टोरेज

यस मोड्युलले सरलताका लागि इन-मेमोरी स्टोरेज प्रयोग गर्छ। अनुप्रयोग पुनः सुरु गर्दा, अपलोड गरिएका कागजातहरू हराउँछन्। उत्पादन प्रणालीहरूले Qdrant वा Azure AI Search जस्ता स्थायी भेक्टर डाटाबेसहरू प्रयोग गर्छन्।

### सन्दर्भ विन्डो व्यवस्थापन

हरेक मोडेलको अधिकतम सन्दर्भ विन्डो हुन्छ। ठूलो कागजातका सबै टुक्राहरू समावेश गर्न सकिँदैन। प्रणाली सबैभन्दा सान्दर्भिक शीर्ष N (पूर्वनिर्धारित ५) टुक्राहरू प्राप्त गरेर सीमाभित्र रहन्छ र पर्याप्त सन्दर्भ प्रदान गर्छ।

## RAG कहिले महत्त्वपुर्ण छ

RAG सधैं सही विधि होइन। तलको निर्णय मार्गदर्शकले तपाईंलाई RAG कहिले मान थप्छ र कहिले सरल तरिकाहरू — जस्तै सामग्रीलाई प्रत्यक्ष प्रॉम्प्टमा समावेश गर्नु वा मोडेलको अवस्थित ज्ञानमा भर पर्नु — पर्याप्त हुन्छ भनेर निर्धारण गर्न मद्दत गर्दछ:

<img src="../../../translated_images/ne/when-to-use-rag.1016223f6fea26bc.webp" alt="RAG कहिले प्रयोग गर्ने" width="800"/>

**RAG प्रयोग गर्नुहोस् जब:**
- मालिकाना दस्तावेजहरू बारे प्रश्नहरूको उत्तर दिने
- जानकारी बारम्बार परिवर्तन हुन्छ (नीतिहरू, मूल्यहरू, विशिष्टताहरू)
- शुद्धताका लागि स्रोत उल्लेख आवश्यक छ
- सामग्री एकल प्रॉम्प्टमा मिल्न धेरै ठूलो छ
- तपाईंलाई प्रमाणीकरणयोग्य, आधारभूत प्रतिक्रिया आवश्यक छ

**जब RAG प्रयोग नगर्नुहोस्:**
- प्रश्नहरूले मोडेलसँग पहिले देखि भएका सामान्य ज्ञान आवश्यक पर्दछ
- वास्तविक समयमा डेटा आवश्यक छ (RAG अपलोड गरिएका दस्तावेजहरूमा काम गर्छ)
- सामग्री स-साना छ जसलाई सिधै प्रॉम्प्टमा समावेश गर्न सकिन्छ
