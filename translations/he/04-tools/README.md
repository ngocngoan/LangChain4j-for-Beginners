## מה תלמדו

עד כה, למדת כיצד לנהל שיחות עם AI, לבנות פקודות בצורה יעילה, ולהקשר תגובות במסמכים שלך. אבל עדיין יש מגבלה יסודית: מודלים של שפה יכולים רק ליצור טקסט. הם לא יכולים לבדוק את מזג האוויר, לבצע חישובים, לשאול מאגרי נתונים או לתקשר עם מערכות חיצוניות.

כלים משנים זאת. על ידי מתן גישה לפונקציות שהמודל יכול לקרוא להן, אתה הופך אותו מיצרן טקסט לסוכן שיכול לבצע פעולות. המודל מחליט מתי הוא צריך כלי, איזה כלי להשתמש, ואילו פרמטרים להעביר. הקוד שלך מבצע את הפונקציה ומחזיר את התוצאה. המודל משלב את התוצאה בתשובתו.

## הבנת סוכני AI עם כלים

> **📝 הערה:** המונח "סוכנים" במודול זה מתייחס לעוזרי AI משופרים עם יכולות קריאת כלים. זה שונה מתבניות **Agentic AI** (סוכנים אוטונומיים עם תכנון, זיכרון, והיגיון רב-שלבי) שנסקור ב-[מודול 05: MCP](../05-mcp/README.md).

בלי כלים, מודל שפה יכול רק לייצר טקסט מהנתונים שלו. שאל אותו על מזג האוויר הנוכחי, והוא יצטרך לנחש. תן לו כלים, והוא יכול לקרוא ל-API מזג אוויר, לבצע חישובים, או לשאול מאגר נתונים — ואז לשלב את התוצאות האמיתיות בתגובה שלו.

<img src="../../../translated_images/he/what-are-tools.724e468fc4de64da.webp" alt="ללא כלים לעומת עם כלים" width="800"/>

*ללא כלים המודל רק מנחש — עם כלים הוא יכול לקרוא ל-APIs, להריץ חישובים ולהחזיר נתונים בזמן אמת.*

סוכן AI עם כלים פועל לפי תבנית **הגיון ופעולה (ReAct)**. המודל לא רק מגיב — הוא חושב על מה שהוא צריך, פועל בקבלת החלטות על קריאת כלי, צופה בתוצאה, ואז מחליט אם לפעול שוב או לספק תשובה סופית:

1. **הגיון** — הסוכן מנתח את שאלת המשתמש ומחליט אילו מידע הוא צריך
2. **פעולה** — הסוכן בוחר את הכלי הנכון, יוצר את הפרמטרים המתאימים, וקורא לו
3. **תצפית** — הסוכן מקבל את פלט הכלי ומעריך את התוצאה
4. **חזרה או תגובה** — אם נדרש מידע נוסף, הסוכן חוזר על התהליך; אחרת, הוא מנסח תשובה בשפה טבעית

<img src="../../../translated_images/he/react-pattern-detail.96a5efeeb6dd2f61.webp" alt="תבנית ReAct" width="800"/>

*מחזור ReAct — הסוכן מהרהר מה לעשות, מפעיל כלי, מתבונן בתוצאה, וחוזר עד שהוא יכול לתת תשובה סופית.*

הדבר קורה אוטומטית. אתה מגדיר את הכלים ותיאורי הכלים. המודל מנהל את קבלת ההחלטות לגבי מתי ואיך להשתמש בהם.

## איך קריאה לכלים עובדת

### הגדרות כלים

[WeatherTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/WeatherTool.java) | [TemperatureTool.java](../../../04-tools/src/main/java/com/example/langchain4j/agents/tools/TemperatureTool.java)

אתה מגדיר פונקציות עם תיאורים ברורים ומפרט פרמטרים. המודל רואה את התיאורים האלו בפקודת המערכת שלו ומבין מה כל כלי עושה.

```java
@Component
public class WeatherTool {
    
    @Tool("Get the current weather for a location")
    public String getCurrentWeather(@P("Location name") String location) {
        // הלוגיקה שלך לחיפוש מזג אוויר
        return "Weather in " + location + ": 22°C, cloudy";
    }
}

@AiService
public interface Assistant {
    String chat(@MemoryId String sessionId, @UserMessage String message);
}

// העוזר מחובר אוטומטית על ידי Spring Boot עם:
// - רכיב ChatModel
// - כל שיטות ה-@Tool מתוך מחלקות @Component
// - ספק ChatMemory לניהול מושבים
```

התרשים למטה מפצל כל אנוטציה ומראה כיצד כל חלק עוזר ל-AI להבין מתי לקרוא לכלי ואילו ארגומנטים להעביר:

<img src="../../../translated_images/he/tool-definitions-anatomy.f6468546037cf28b.webp" alt="אנטומיית הגדרות כלים" width="800"/>

*אנטומיית הגדרה של כלי — @Tool מודיע ל-AI מתי להשתמש בו, @P מתאר כל פרמטר, ו-@AiService מחבר הכל אוטומטית בהפעלה.*

### קבלת החלטות

כשמשתמש שואל "מה מזג האוויר בסיאטל?", המודל לא בוחר כלי באקראי. הוא משווה את כוונת המשתמש מול כל תיאור כלי שיש לו גישה אליו, מדרג את הרלוונטיות של כל אחד, ובוחר את ההתאמה הטובה ביותר. לאחר מכן הוא יוצר קריאת פונקציה מובנית עם הפרמטרים הנכונים — במקרה זה, מגדיר `location` ל-"Seattle".

אם אף כלי לא מתאים לבקשת המשתמש, המודל חוזר לענות ממידע שברשותו. אם יש כמה כלים מתאימים, הוא בוחר את הספציפי ביותר.

<img src="../../../translated_images/he/decision-making.409cd562e5cecc49.webp" alt="איך ה-AI מחליט איזה כלי להשתמש" width="800"/>

*המודל מעריך כל כלי זמין כנגד כוונת המשתמש ובוחר את ההתאמה הטובה ביותר — לכן חשוב לכתוב תיאורי כלים ברורים וספציפיים.*

המודל מקבל את נתוני מזג האוויר ומעצב אותם לתגובה בשפה טבעית עבור המשתמש.

### ארכיטקטורה: חיבור אוטומטי בספרינג בוט

מודול זה משתמש באינטגרציה של LangChain4j עם ספרינג בוט דרך ממשקי `@AiService` הכריזמתיים. בהפעלה ספרינג בוט מגלה כל `@Component` שמכיל שיטות עם `@Tool`, את רכיב `ChatModel`, ו-`ChatMemoryProvider` — ואז מחבר את כולם לממשק `Assistant` אחד ללא קוד חזרתי.

<img src="../../../translated_images/he/spring-boot-wiring.151321795988b04e.webp" alt="ארכיטקטורת חיבור ספרינג בוט אוטומטי" width="800"/>

*ממשק @AiService מקשר בין ChatModel, רכיבי כלי, וספק הזיכרון — ספרינג בוט מנהל את כל החיבור אוטומטית.*

יתרונות מרכזיים של גישה זו:

- **חיבור אוטומטי בספרינג בוט** — ChatModel והכלים מוזרקים אוטומטית
- **תבנית @MemoryId** — ניהול זיכרון המבוסס על מושבים אוטומטי
- **מופע יחיד** — Assistant נוצר פעם אחת ומשמש מחדש לשיפור ביצועים
- **ביצוע בטוח טיפוס** — קריאה ישירה לשיטות Java עם המרת טיפוסים
- **אורקסטרציה רב-פניות** — מטפל בשרשור כלים אוטומטית
- **אפס קוד חזרתי** — ללא קריאות ידניות ל-AiServices.builder() או HashMap זיכרון

גישות חלופיות (עם `AiServices.builder()` ידני) דורשות יותר קוד ומפעילות פחות את יתרונות האינטגרציה של ספרינג בוט.

## שרשור כלים

**שרשור כלים** — הכוח האמיתי של סוכנים מבוססי כלים מתבטא כאשר שאלה אחת דורשת שימוש במספר כלים. שאל "מה מזג האוויר בסיאטל בפארנהייט?" והסוכן מחבר אוטומטית שני כלים: קודם קורא ל-`getCurrentWeather` לקבלת הטמפרטורה במעלות צלזיוס, ואז מעביר את הערך ל-`celsiusToFahrenheit` להמרה — כל זאת בסיבוב שיחה אחד.

<img src="../../../translated_images/he/tool-chaining-example.538203e73d09dd82.webp" alt="דוגמת שרשור כלים" width="800"/>

*שרשור כלים בפועל — הסוכן קורא ל-getCurrentWeather תחילה, ואז מזין את התוצאה בצלזיוס ל-celsiusToFahrenheit, ומספק תשובה משולבת.*

כך זה נראה באפליקציה הרצה — הסוכן מחבר שתי קריאות כלים בסיבוב שיחה אחד:

<a href="images/tool-chaining.png"><img src="../../../translated_images/he/tool-chaining.3b25af01967d6f7b.webp" alt="שרשור כלים" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*פלט האפליקציה בפועל — הסוכן מחבר אוטומטית getCurrentWeather → celsiusToFahrenheit בסיבוב יחיד.*

**כישלונות מחושבים** — בקש מזג אוויר בעיר שאינה בנתוני הדמה. הכלי מחזיר הודעת שגיאה, וה-AI מסביר שהוא לא יכול לעזור במקום לקרוס. כלים נכשלים באופן בטוח.

<img src="../../../translated_images/he/error-handling-flow.9a330ffc8ee0475c.webp" alt="זרימת טיפול בשגיאות" width="800"/>

*כאשר כלי נכשל, הסוכן לוכד את השגיאה ומגיב בהסבר מועיל במקום לקרוס.*

זה קורה בסיבוב שיחה אחד. הסוכן מארגן קריאות לכלים מרובות באופן עצמאי.

*ממשק הכלים של סוכן ה-AI - דוגמאות מהירות וממשק שיחה לעבוד עם כלים*

### נסה שימוש פשוט בכלים
התחילו עם בקשה פשוטה: "המר 100 מעלות פרנהייט לצלזיוס". הסוכן מזהה כי הוא זקוק לכלי להמרת טמפרטורה, קורא לו עם הפרמטרים המתאימים, ומחזיר את התוצאה. שימו לב כמה זה טבעי - לא ציינתם באיזה כלי להשתמש או איך לקרוא לו.

### בדיקת שרשור כלים

עכשיו נסו משהו מורכב יותר: "מה מזג האוויר בסיאטל והמר אותו לפרנהייט?" צפו בסוכן עובד את זה בשלבים. תחילה הוא מאתר את מזג האוויר (שמחזיר צלזיוס), מזהה כי הוא צריך להמיר לפרנהייט, קורא לכלי ההמרה, ומשלב את שני התוצאות לתגובה אחת.

### ראו את זרימת השיחה

ממשק הצ׳אט שומר היסטוריית שיחה, ומאפשר לכם לקיים אינטראקציות מרובות סבבים. אתם יכולים לראות את כל השאלות והתשובות הקודמות, להקל על מעקב השיחה ולהבין כיצד הסוכן בונה הקשר לאורך מספר החלפות.

<a href="images/tools-conversation-demo.png"><img src="../../../translated_images/he/tools-conversation-demo.89f2ce9676080f59.webp" alt="שיחה עם קריאות כלים מרובות" width="800" style="border: 1px solid #ddd; box-shadow: 0 2px 8px rgba(0,0,0,0.1);"/></a>

*שיחה מרובת סבבים המראה המרות פשוטות, חיפושי מזג אוויר, ושרשור כלים*

### נסו בקשות שונות

נסו שילובים מגוונים:
- חיפושי מזג אוויר: "מה מזג האוויר בטוקיו?"
- המרות טמפרטורה: "כמה זה 25°C בקלווין?"
- שאילתות משולבות: "בדוק את מזג האוויר בפריז ואמר לי אם הוא מעל 20°C"

הבחינו כיצד הסוכן מפרש שפה טבעית וממפה אותה לקריאות כלים מתאימות.

## מונחים מרכזיים

### דפוס ReAct (הסקה ופעולה)

הסוכן מתחלף בין היסק (החלטה מה לעשות) לפעולה (שימוש בכלים). דפוס זה מאפשר פתרון בעיות אוטונומי במקום רק תגובה להוראות.

### תיאורי כלים חשובים

איכות תיאורי הכלים שלכם משפיעה ישירות על כמה טוב הסוכן משתמש בהם. תיאורים ברורים ומדויקים עוזרים למודל להבין מתי ואיך לקרוא לכלי.

### ניהול סשן

האנוטציה `@MemoryId` מאפשרת ניהול זיכרון מבוסס סשן באופן אוטומטי. לכל מזהה סשן יש מופע `ChatMemory` מנוהל על ידי bean בשם `ChatMemoryProvider`, כך שמשתמשים רבים יכולים לתקשר עם הסוכן בלי שהשיחות יתערבבו.

<img src="../../../translated_images/he/session-management.91ad819c6c89c400.webp" alt="ניהול סשן עם @MemoryId" width="800"/>

*לכל מזהה סשן יש היסטוריית שיחה מבודדת — משתמשים לא רואים את הודעות זה של זה.*

### טיפול בשגיאות

כלים עלולים להיכשל — API לא עונה בזמן, פרמטרים לא תקינים, שירותים חיצוניים קורסים. סוכנים בשימוש פרודקשן צריכים טיפול שגיאות כך שהמודל יכול להסביר בעיות או לנסות חלופות במקום לקרוס כולו. כשכלי זורק יוצא דופן, LangChain4j תופס אותו ומחזיר את הודעת השגיאה למודל, שמסוגל להסביר את הבעיה בשפה טבעית.

## כלים זמינים

הדיאגרמה למטה מראה את האקוסיסטם הרחב של כלים שאפשר לבנות. מודול זה מדגים כלים למזג אוויר וטמפרטורות, אך דפוס ה-`@Tool` זהה עובד לכל מתודה בג'אווה — משאילתות מסדי נתונים ועד עיבוד תשלומים.

<img src="../../../translated_images/he/tool-ecosystem.aad3d74eaa14a44f.webp" alt="אקוסיסטם של כלים" width="800"/>

*כל מתודת ג'אווה המסומנת ב-@Tool זמינה ל-AI — הדפוס מתרחב למסדי נתונים, API, דואר אלקטרוני, ניהול קבצים ועוד.*

## מתי להשתמש בסוכנים מבוססי כלים

<img src="../../../translated_images/he/when-to-use-tools.51d1592d9cbdae9c.webp" alt="מתי להשתמש בכלים" width="800"/>

*מדריך החלטה מהיר — כלים משמשים לנתונים בזמן אמת, חישובים ופעולות; ידע כללי ומשימות יצירתיות לא דורשים אותם.*

**השתמשו בכלים כאשר:**
- תידרש נתונים בזמן אמת (מזג אוויר, מחירי מניות, מלאי)
- עליכם לבצע חישובים מעבר למתמטיקה פשוטה
- גישה למסדי נתונים או API
- ביצוע פעולות (שליחת מיילים, יצירת פניות, עדכון רשומות)
- שילוב מקורות נתונים מרובים

**אל תשתמשו בכלים כאשר:**
- שאלות ניתנות למענה מהידע הכללי
- התגובה היא שיחה בלבד
- עיכוב הכלי יהפוך את החוויה לאיטית מדי

## כלים לעומת RAG

מודולים 03 ו-04 מרחיבים את יכולות ה-AI בדרכים שונות באופן מהותי. RAG מעניקה למודל גישה ל**ידע** על ידי משיכת מסמכים. כלים מאפשרים למודל לבצע **פעולות** על ידי קריאות פונקציות.

<img src="../../../translated_images/he/tools-vs-rag.ad55ce10d7e4da87.webp" alt="השוואה בין כלים ל-RAG" width="800"/>

*RAG מושך מידע ממסמכים סטטיים — כלים מבצעים פעולות ומביאים נתונים דינמיים בזמן אמת. מערכות פרודקשן רבות משלבות את שניהם.*

בעבודה שוטפת, מערכות רבות משלבות את שתי הגישות: RAG למתן מענה מבוסס על המסמכים שלכם, וכלים לשליפת נתונים חיים או ביצוע פעולות.
