## Шта ћете научити

Изградили сте конверзацијски AI, савладали промптове, засновали одговоре на документима и креирали агенте са алаткама. Али све те алатке су биле прилагођене вашој одређеној апликацији. Шта ако бисте могли да дате свом AI приступ стандардизованом екосистему алатки које било ко може да направи и дели? У овом модулу ћете научити како управо то да урадите користећи Протокол Контекста Модела (MCP) и LangChain4j агентски модул. Прво представљамо једноставан читач MCP фајлова, а затим показујемо како се лако интегрише у сложене агентске токове рада користећи шаблон Супервизор Агент.

## Шта је MCP?

Протокол Контекста Модела (MCP) пружа управо то — стандардни начин за AI апликације да открију и користе екстерне алатке. Уместо да пишете прилагођене интеграције за сваки извор података или сервис, повезујете се на MCP сервере који своје могућности излажу у доследном формату. Ваш AI агент потом аутоматски открива те алатке и користи их.

Дијаграм испод показује разлику — без MCP-а, свака интеграција захтева прилагођено повезивање од тачке до тачке; са MCP-ом, један протокол повезује вашу апликацију са било којом алатком:

<img src="../../../translated_images/sr/mcp-comparison.9129a881ecf10ff5.webp" alt="MCP Comparison" width="800"/>

*Пре MCP-а: Комплексне интеграције од тачке до тачке. После MCP-а: Један протокол, бескрајне могућности.*

MCP решава фундаментални проблем у развоју AI: свака интеграција је прилагођена. Желите приступ GitHub-у? Прилагођен код. Желите читање фајлова? Прилагођен код. Желите упите према бази података? Прилагођен код. И ниједна од ових интеграција не ради са другим AI апликацијама.

MCP све то стандардизује. MCP сервер излаже алатке са јасним описима и шемама параметара. Било који MCP клијент се може повезати, открити доступне алатке и користити их. Направите једном, користите свуда.

Дијаграм испод илуструје ову архитектуру — један MCP клијент (ваша AI апликација) се повезује на више MCP сервера, од којих сваки излаже свој скуп алатки кроз стандардни протокол:

<img src="../../../translated_images/sr/mcp-architecture.b3156d787a4ceac9.webp" alt="MCP Architecture" width="800"/>

*Архитектура Протокола Контекста Модела - стандардизовано откривање и извршавање алатки*

## Како MCP функционише

Испод хаубе, MCP користи слојевиту архитектуру. Ваша Java апликација (MCP клијент) открива доступне алатке, шаље JSON-RPC захтеве кроз транспортни слој (Stdio или HTTP), а MCP сервер извршава операције и враћа резултате. Следећи дијаграм разлаже сваки слој овог протокола:

<img src="../../../translated_images/sr/mcp-protocol-detail.01204e056f45308b.webp" alt="MCP Protocol Detail" width="800"/>

*Како MCP функционише испод хаубе — клијенти откривају алатке, размењују JSON-RPC поруке и извршавају операције путем транспортног слоја.*

**Архитектура Сервер-Клијент**

MCP користи модел сервер-клијент. Сервери пружају алатке - читање фајлова, упите база података, позиве API-ја. Клијенти (ваша AI апликација) се повезују на сервере и користе њихове алатке.

За коришћење MCP-а са LangChain4j, додајте ову Maven зависност:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```

**Откривање Алатки**

Када се ваш клијент повеже на MCP сервер, он пита "Које алатке имате?" Сервер одговара листом доступних алатки, свака са описима и шемама параметара. Ваш AI агент онда може одлучити које алатке да користи на основу корисничких захтева. Дијаграм испод показује овај руковање — клијент шаље `tools/list` захтев, а сервер враћа своје доступне алатке са описима и шемама параметара:

<img src="../../../translated_images/sr/tool-discovery.07760a8a301a7832.webp" alt="MCP Tool Discovery" width="800"/>

*AI открива доступне алатке при покретању — сада зна које могућности су доступне и може одлучити које ће користити.*

**Транспортни Механизми**

MCP подржава различите транспортне механизме. Две опције су Stdio (за локалну комуникацију између процеса) и Streamable HTTP (за удаљене сервере). Овај модул демонстрира Stdio транспорт:

<img src="../../../translated_images/sr/transport-mechanisms.2791ba7ee93cf020.webp" alt="Transport Mechanisms" width="800"/>

*MCP транспортни механици: HTTP за удаљене сервере, Stdio за локалне процесе*

**Stdio** - [StdioTransportDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java)

За локалне процесе. Ваша апликација покреће сервер као подпроцес и комуницира преко стандардног улаза/излаза. Корисно за приступ фајл системима или алаткама командне линије.

```java
McpTransport stdioTransport = new StdioMcpTransport.Builder()
    .command(List.of(
        npmCmd, "exec",
        "@modelcontextprotocol/server-filesystem@2025.12.18",
        resourcesDir
    ))
    .logEvents(false)
    .build();
```

`@modelcontextprotocol/server-filesystem` сервер излаже следеће алатке, све изоловане на директоријумима које одредите:

| Алатка | Опис |
|--------|-------|
| `read_file` | Чита садржај појединачног фајла |
| `read_multiple_files` | Чита више фајлова у једном позиву |
| `write_file` | Креира или преписује фајл |
| `edit_file` | Прави циљане измене „пронађи и замени“ |
| `list_directory` | Листa фајлова и директоријума на одређеној путањи |
| `search_files` | Рекурзивно тражи фајлове који одговарају шаблону |
| `get_file_info` | Добија метаподатке фајла (величина, временске ознаке, дозволе) |
| `create_directory` | Креира директоријум (укључујући родитељске директоријуме) |
| `move_file` | Помера или преименује фајл или директоријум |

Следећи дијаграм показује како Stdio транспорт функционише у току извршавања — ваша Java апликација покреће MCP сервер као дечји процес и они комуницирају кроз stdin/stdout пипове, без мреже или HTTP укључених:

<img src="../../../translated_images/sr/stdio-transport-flow.45eaff4af2d81db4.webp" alt="Stdio Transport Flow" width="800"/>

*Stdio транспорт у акцији — ваша апликација покреће MCP сервер као дечји процес и комуницира преко stdin/stdout пипова.*

## Агентски модул

Док MCP пружа стандардизоване алатке, LangChain4j-ов **агентски модул** обезбеђује декларативан начин изградње агената који оркестрирају те алатке. @Agent анотација и AgenticServices вам омогућавају да дефинишете понашање агента преко интерфејса уместо императивног кода.

У овом модулу, испитаћете **шаблон Супервизор Агент** — напредни агентски AI приступ где „супервизор“ агент динамички одлучује које под-агенте да позове на основу корисничких захтева. Комбиновапмо оба концепта тако што ћемо једном од наших под-агената дати MCP-погонске могућности приступа фајловима.

За употребу агентског модула, додајте ову Maven зависност:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```
> **Напомена:** `langchain4j-agentic` модул користи посебну верзију и својство (`langchain4j.mcp.version`) јер се објављује по другачијем распореду од главних LangChain4j библиотека.

> **⚠️ Експериментално:** `langchain4j-agentic` модул је **експерименталан** и подложан променама. Стабилан начин за изградњу AI асистената остаје `langchain4j-core` са прилагођеним алаткама (Модул 04).

### Супервизор агент

**Шаблон Супервизор Агент** је **флексибилан** облик агентског AI. Супервизор користи LLM да аутономно одлучи које агенте да позове на основу захтева корисника. У следећем примеру комбинујемо MCP-погонски приступ фајловима са LLM агентом да креирамо надгледани ток рада читања фајла → извештај.

У демо-у, `FileAgent` чита фајл користећи MCP алатке датотечног система, а `ReportAgent` генерише структуриран извештај са извршним резимеом (1 реченица), 3 кључна тачка и препорукама. Супервизор аутоматски оркестрира овај ток:

<img src="../../../translated_images/sr/supervisor-agent-pattern.06275a41ae006ac8.webp" alt="Supervisor Agent Pattern" width="800"/>

*Супервизор користи свој LLM да одлучи које агенте и у ком редоследу позвати — нема потребе за тешко кодираним рутирањем.*

Ево како изгледа конкретан ток рада за наш конвејер од фајла до извештаја:

<img src="../../../translated_images/sr/file-report-workflow.649bb7a896800de9.webp" alt="File to Report Workflow" width="800"/>

*FileAgent чита фајл преко MCP алатки, затим ReportAgent претвара сиров садржај у структуриран извештај.*

Сваки агент чува свој излаз у **Agentic Scope** (заједничка меморија), омогућавајући следећим агенатима приступ претходним резултатима. Ово демонстрира како MCP алатке беспрекорно интегришу у агентске токове — Супервизор не треба да зна *како* се фајлови читају, само да `FileAgent` то може.

#### Како супервизор ради

Пре него што креирате агенте, потребно је да повежете MCP транспорт са клијентом и упакујете га као `ToolProvider`. Ово је начин на који MCP серверске алатке постају доступне вашим агентима:

```java
// Направите MCP клијента из транспортног слоја
McpClient mcpClient = new DefaultMcpClient.Builder()
        .transport(stdioTransport)
        .build();

// Укључите клијента као ToolProvider — ово повезује MCP алате са LangChain4j
ToolProvider mcpToolProvider = McpToolProvider.builder()
        .mcpClients(List.of(mcpClient))
        .build();
```

Сада можете убризгати `mcpToolProvider` у било који агент коме су потребне MCP алатке:

```java
// Корак 1: FileAgent чита фајлове користећи MCP алате
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // Поседује MCP алате за рад са фајловима
        .build();

// Корак 2: ReportAgent генерише структуриране извештаје
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// Supervisor организује радни ток фајл → извештај
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // Вратити коначни извештај
        .build();

// Supervisor одлучује које агенте да позове на основу захтева
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```

#### Стратегије одговора

Када конфигуришете `SupervisorAgent`, наводите како треба да формулише свој коначни одговор кориснику након што под-агенти заврше своје задатке. Дијаграм испод показује три доступне стратегије — LAST враћа излаз последњег агента директно, SUMMARY синтетише све излазе преко LLM-а, а SCORED бира онај који добије виши резултат у односу на оригинални захтев:

<img src="../../../translated_images/sr/response-strategies.3d0cea19d096bdf9.webp" alt="Response Strategies" width="800"/>

*Три стратегије за то како Супервизор формулише коначни одговор — изаберите на основу тога да ли желите излаз последњег агента, сажетак или најбоље оцењену опцију.*

Доступне стратегије су:

| Стратегија | Опис |
|------------|-------|
| **LAST** | Супервизор враћа излаз последњег позваног под-агента или алатке. Корисно када је последњи агент у току рада посебно дизајниран да произведе потпуни, коначни одговор (нпр. „Резиме Агент“ у истраживачком конвејеру). |
| **SUMMARY** | Супервизор користи свој интерни језички модел (LLM) да синтетише резиме целе интеракције и свих излаза под-агената, затим враћа тај резиме као коначни одговор. Ово пружа чист, агрегирани одговор кориснику. |
| **SCORED** | Систем користи интерни LLM да оцењује и LAST одговор и SUMMARY интеракције према оригиналном корисничком захтеву, враћајући онај излаз који добије вишу оцену. |
Погледајте [SupervisorAgentDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/SupervisorAgentDemo.java) за потпуну имплементацију.

#### Разумевање излаза

Када покренете демо, видећете структуриран преглед како Супервизор оркестрира више агената. Ево шта сваки одељак значи:

```
======================================================================
  FILE → REPORT WORKFLOW DEMO
======================================================================

This demo shows a clear 2-step workflow: read a file, then generate a report.
The Supervisor orchestrates the agents automatically based on the request.
```

**Заглавље** уводи концепт радног тока: фокусирану цевовод од читања фајла до генерисања извештаја.

```
--- WORKFLOW ---------------------------------------------------------
  ┌─────────────┐      ┌──────────────┐
  │  FileAgent  │ ───▶ │ ReportAgent  │
  │ (MCP tools) │      │  (pure LLM)  │
  └─────────────┘      └──────────────┘
   outputKey:           outputKey:
   'fileContent'        'report'

--- AVAILABLE AGENTS -------------------------------------------------
  [FILE]   FileAgent   - Reads files via MCP → stores in 'fileContent'
  [REPORT] ReportAgent - Generates structured report → stores in 'report'
```

**Дијаграм радног тока** приказује проток података између агената. Сваком агенту је додељена одређена улога:
- **FileAgent** чита фајлове помоћу MCP алата и чува сиров садржај у `fileContent`
- **ReportAgent** користи тај садржај и производи структурирани извештај у `report`

```
--- USER REQUEST -----------------------------------------------------
  "Read the file at .../file.txt and generate a report on its contents"
```

**Кориснички захтев** приказује задатак. Супервизор га парсира и одлучује да позове FileAgent → ReportAgent.

```
--- SUPERVISOR ORCHESTRATION -----------------------------------------
  The Supervisor decides which agents to invoke and passes data between them...

  +-- STEP 1: Supervisor chose -> FileAgent (reading file via MCP)
  |
  |   Input: .../file.txt
  |
  |   Result: LangChain4j is an open-source, provider-agnostic Java framework for building LLM...
  +-- [OK] FileAgent (reading file via MCP) completed

  +-- STEP 2: Supervisor chose -> ReportAgent (generating structured report)
  |
  |   Input: LangChain4j is an open-source, provider-agnostic Java framew...
  |
  |   Result: Executive Summary...
  +-- [OK] ReportAgent (generating structured report) completed
```

**Оркестрација Супервизора** приказује 2-степени ток у акцији:
1. **FileAgent** чита фајл преко MCP и чува садржај
2. **ReportAgent** прима садржај и генерише структурирани извештај

Супервизор је донео ове одлуке **аутономно** на основу корисничког захтева.

```
--- FINAL RESPONSE ---------------------------------------------------
Executive Summary
...

Key Points
...

Recommendations
...

--- AGENTIC SCOPE (Data Flow) ----------------------------------------
  Each agent stores its output for downstream agents to consume:
  * fileContent: LangChain4j is an open-source, provider-agnostic Java framework...
  * report: Executive Summary...
```

#### Објашњење карактеристика Agentic модула

Пример демонстрира неколико напредних функција agentic модула. Погледајмо ближе Agentic Scope и Agent Listener-е.

**Agentic Scope** приказује заједничку меморију где су агенти чували своје резултате користећи `@Agent(outputKey="...")`. Ово омогућава:
- Далалим агентима приступ излазима претходних агената
- Супервизору да синтетише коначни одговор
- Вама да прегледате шта је који агент произвео

Дијаграм испод показује како Agentic Scope функционише као заједничка меморија у радном току од фајла до извештаја — FileAgent уписује свој излаз под кључем `fileContent`, ReportAgent чита то и уписује свој излаз под `report`:

<img src="../../../translated_images/sr/agentic-scope.95ef488b6c1d02ef.webp" alt="Agentic Scope Shared Memory" width="800"/>

*Agentic Scope делује као заједничка меморија — FileAgent уписује `fileContent`, ReportAgent чита то и уписује `report`, а ваш код чита коначан резултат.*

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // Необрађени подаци из датотеке од FileAgent
String report = scope.readState("report");            // Структурисани извештај од ReportAgent
```

**Agent Listeners** омогућавају праћење и дебаговање извршења агената. Корак по корак излаз који видите у демо приказује AgentListener који се укључује у сваки позив агента:
- **beforeAgentInvocation** - Позива се када Супервизор изабере агента, омогућавајући вам да видите који је агент изабран и зашто
- **afterAgentInvocation** - Позива се кад агент заврши, приказује његов резултат
- **inheritedBySubagents** - Када је истинито, слушалац прати све агенте у хијерархији

Следећи дијаграм приказује цео животни циклус Agent Listener-а, укључујући како `onError` обрађује грешке током извршења агента:

<img src="../../../translated_images/sr/agent-listeners.784bfc403c80ea13.webp" alt="Agent Listeners Lifecycle" width="800"/>

*Agent Listeners се укључују у животни циклус извршења — прате када агенти почињу, завршавају или имају грешке.*

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // Пропагирати на све под-агенте
    }
};
```

Поред шаблона Супервизора, `langchain4j-agentic` модул пружа неколико моћних шаблона радних токова. Дијаграм испод приказује свих пет — од једноставних секвенцијалних цевовода до радних токова са човеком у петљи за одобрење:

<img src="../../../translated_images/sr/workflow-patterns.82b2cc5b0c5edb22.webp" alt="Agent Workflow Patterns" width="800"/>

*Пет шаблона радних токова за оркестрацију агената — од једноставних секвенцијалних цевовода до радних токова са човеком у петљи.*

| Шаблон | Опис | Примена |
|---------|-------------|----------|
| **Секвенцијални** | Извршавање агената по реду, излаз иде следећем | Цевоводи: истраживање → анализа → извештај |
| **Паралелни** | Покретање агената паралелно | Независни задаци: време + вести + берза |
| **Петаљка** | Понављање док се услов не испуни | Оцена квалитета: фино подешавање док ≥ 0.8 |
| **Условни** | Усмеравање на основу услова | Класификација → усмеравање ка стручном агенту |
| **Човек у петљи** | Додавање људских контролних тачака | Радни токови одобрења, преглед садржаја |

## Кључни појмови

Сада када сте истражили MCP и agentic модул у пракси, сумирамо када користити сваки приступ.

Једна од највећих предности MCP-а је његов растући екосистем. Дијаграм испод показује како један универзални протокол повезује вашу AI апликацију са широким спектром MCP сервера — од приступа фајл систему и базама података до GitHub-а, е-поште, веб скрејпинга и других:

<img src="../../../translated_images/sr/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="MCP Ecosystem" width="800"/>

*MCP креира екосистем универзалног протокола — сваки MCP-усаглашени сервер ради са било којим MCP-усаглашеним клијентом, омогућавајући дељење алата између апликација.*

**MCP** је идеалан када желите да користите постојеће екосистеме алата, градите алате које више апликација може делити, интегришете услуге трећих страна са стандардним протоколима, или мењате имплементације алата без промене кода.

**Agentic модул** најбоље функционише када желите декларативне дефиниције агената са `@Agent` анотацијама, потребна вам је оркестрација радног тока (секвенцијално, петаљка, паралелно), више волите дизајн агената заснован на интерфејсима него императивни код, или комбинујете више агената који деле излазе преко `outputKey`.

**Шаблон Supervisor Agent-а** сјаји када радни ток није предвидив унапред и желите да LLM одлучује, када имате више специјализованих агената за динамичку оркестрацију, када градите разговорне системе који усмеравају ка различитим могућностима, или када желите најфлексибилније и најадаптивније понашање агената.

Да бисмо вам помогли да одлучите између прилагођених `@Tool` метода из Модула 04 и MCP алата из овог модула, следеће поређење истиче кључне компромисе — прилагођени алати пружају тесну повезаност и потпуну типску безбедност за логику специфичну за апликацију, док MCP алати нуде стандаризоване, реупотребљиве интеракције:

<img src="../../../translated_images/sr/custom-vs-mcp-tools.c4f9b6b1cb65d8a1.webp" alt="Custom Tools vs MCP Tools" width="800"/>

*Када користити прилагођене @Tool методе у односу на MCP алате — прилагођени алати за апликацијску логику са потпуном типском безбедношћу, MCP алати за стандаризоване интеграције које раде у више апликација.*
