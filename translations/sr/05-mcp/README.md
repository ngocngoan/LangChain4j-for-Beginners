## Шта ћете научити

Изграђивали сте конверзацијски AI, савладали упите, основали одговоре у документима и креирали агенте са алатима. Али сви ти алати били су прилагођени вашој специфичној апликацији. Шта ако бисте могли да дате вашем AI приступ стандарлизованом екосистему алата које свако може да креира и дели? У овом модулу ћете научити како управо то да урадите са Моделским Контекст Протоколом (MCP) и агентским модулом LangChain4j. Прво приказујемо једноставан MCP читач фајлова, а затим показујемо како се он лако интегрише у напредни агентски радни ток користећи образац Супервизор Агент.

## Шта је MCP?

Моделски Контекст Протокол (MCP) обезбеђује управо то - стандардни начин за AI апликације да откривају и користе спољне алате. Уместо да правите прилагођене интеграције за сваки извор података или услугу, повезујете се са MCP серверима који излажу своје могућности у конзистентном формату. Ваш AI агент може аутоматски да открије и користи те алате.

Дијаграм испод приказује разлику — без MCP свака интеграција захтева прилагођено повезивање тачка-на-тачку; са MCP, један протокол повезује вашу апликацију са било којим алатом:

<img src="../../../translated_images/sr/mcp-comparison.9129a881ecf10ff5.webp" alt="MCP Comparison" width="800"/>

*Пре MCP: Комплексне интеграције тачка-на-тачку. После MCP: Један протокол, бескрајне могућности.*

MCP решава фундаментални проблем у развоју AI: свака интеграција је прилагођена. Желите приступ GitHub-у? Прилагођени код. Желите да читате фајлове? Прилагођени код. Желите да упитате базу података? Прилагођени код. И ниједна од ових интеграција не ради са другим AI апликацијама.

MCP то стандардује. MCP сервер излаже алате са јасним описима и шемама. Сваки MCP клијент може да се повеже, открије расположиве алате и користи их. Направите једном, користите свуда.

Дијаграм испод илуструје ову архитектуру — један MCP клијент (ваша AI апликација) повезује се са више MCP сервера, од којих сваки изложи свој скуп алата преко стандардног протокола:

<img src="../../../translated_images/sr/mcp-architecture.b3156d787a4ceac9.webp" alt="MCP Architecture" width="800"/>

*Архитектура Моделског Контекст Протокола - стандарнизовано откривање и извршавање алата*

## Како MCP ради

Испод хаубе, MCP користи слојевиту архитектуру. Ваша Java апликација (MCP клијент) открива расположиве алате, шаље JSON-RPC захтеве преко транспортног слоја (Stdio или HTTP), а MCP сервер извршава операције и враћа резултате. Следећи дијаграм разбија сваки слој овог протокола:

<img src="../../../translated_images/sr/mcp-protocol-detail.01204e056f45308b.webp" alt="MCP Protocol Detail" width="800"/>

*Како MCP ради испод хаубе — клијенти откривају алате, размењују JSON-RPC поруке и извршавају операције преко транспортног слоја.*

**Архитектура сервер-клијент**

MCP користи модел сервер-клијент. Сервери пружају алате - читање фајлова, упити база података, позиви API-ја. Клијенти (ваша AI апликација) се повезују на сервере и користе њихове алате.

Да бисте користили MCP са LangChain4j, додајте ову Maven зависност:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-mcp</artifactId>
    <version>${langchain4j.version}</version>
</dependency>
```

**Откривање алата**

Када се ваш клијент повеже са MCP сервером, он пита „Које алате имате?“ Сервер одговара са списком расположивих алата, сваки са описима и шемама параметара. Ваш AI агент онда може да одлучи које алате да користи на основу корисничких захтева. Дијаграм испод приказује ово руковање — клијент шаље захтев `tools/list`, а сервер враћа своје расположиве алате са описима и шемама параметара:

<img src="../../../translated_images/sr/tool-discovery.07760a8a301a7832.webp" alt="MCP Tool Discovery" width="800"/>

*AI открива расположиве алате при покретању — сада зна које могућности су доступне и може одлучити које ће користити.*

**Транспортни механизми**

MCP подржава разне транспортне механизме. Две опције су Stdio (за локалну комуникацију између процеса) и Streamable HTTP (за удаљене сервере). Овај модул демонстрира Stdio транспорт:

<img src="../../../translated_images/sr/transport-mechanisms.2791ba7ee93cf020.webp" alt="Transport Mechanisms" width="800"/>

*MCP транспортни механизми: HTTP за удаљене сервере, Stdio за локалне процесе*

**Stdio** - [StdioTransportDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/StdioTransportDemo.java)

За локалне процесе. Ваша апликација покреће сервер као под-процес и комуницира преко стандардног улаза/излаза. Корисно за приступ систему фајлова или командно-линијским алатима.

```java
McpTransport stdioTransport = new StdioMcpTransport.Builder()
    .command(List.of(
        npmCmd, "exec",
        "@modelcontextprotocol/server-filesystem@2025.12.18",
        resourcesDir
    ))
    .logEvents(false)
    .build();
```

`@modelcontextprotocol/server-filesystem` сервер излаже следеће алате, сви ограничени на директоријуме које наведете:

| Алат | Опис |
|------|-------|
| `read_file` | Читање садржаја једног фајла |
| `read_multiple_files` | Читање више фајлова у једном позиву |
| `write_file` | Креирање или преписивање фајла |
| `edit_file` | Извођење циљаних операција претраживања и замене |
| `list_directory` | Листинг фајлова и директоријума на путањи |
| `search_files` | Рекурзивно претраживање фајлова који одговарају шаблону |
| `get_file_info` | Добијање метаподатака о фајлу (величина, временски жигови, дозволе) |
| `create_directory` | Креирање директоријума (укључујући родитељске директоријуме) |
| `move_file` | Померање или преименовање фајла или директоријума |

Следећи дијаграм приказује како Stdio транспорт ради током извршавања — ваша Java апликација покреће MCP сервер као дечји процес и они комуницирају преко stdin/stdout цеви, без мреже или HTTP уместо тога:

<img src="../../../translated_images/sr/stdio-transport-flow.45eaff4af2d81db4.webp" alt="Stdio Transport Flow" width="800"/>

*Stdio транспорт на делу — ваша апликација покреће MCP сервер као дечји процес и комуницира преко stdin/stdout цеви.*

## Агентски Модул

Док MCP обезбеђује стандартизоване алате, LangChain4j-ов **агентски модул** пружа декларативан начин за креирање агената који оркестрирају те алате. `@Agent` анотација и `AgenticServices` вам омогућавају да дефинишете понашање агената кроз интерфејсе уместо императивног кода.

У овом модулу, истражићете образац **Супервизор Агент** — напредни агентски AI приступ у којем "супервизор" агент динамички одлучује које под-агенте да позове на основу корисничких захтева. Ујединићемо оба концепта тако што ћемо једном од наших под-агената дати могућности приступа фајловима покретаним MCP-ом.

Да бисте користили агентски модул, додајте ову Maven зависност:

```xml
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-agentic</artifactId>
    <version>${langchain4j.mcp.version}</version>
</dependency>
```
> **Напомена:** Модул `langchain4j-agentic` користи посебну верзијску променљиву (`langchain4j.mcp.version`) јер се објављује по другачијем распореду од основних LangChain4j библиотека.

> **⚠️ Експериментално:** Модул `langchain4j-agentic` је **експерименталан** и подложан променама. Стабилан начин за прављење AI асистената и даље је `langchain4j-core` са прилагођеним алатима (Модул 04).

### Супервизор Агент

Образац **Супервизор Агент** је **флексибилан** облик агентског AI. Супервизор користи LLM да аутономно одлучи које агенте да позове на основу корисничког захтева. У наредном примеру комбинујемо MCP-покретани приступ фајловима са LLM агентом да креирамо надгледани ток фајл-читач → извештај.

У демо-у, `FileAgent` чита фајл користећи MCP алате за фајл систем, а `ReportAgent` генерише структурирани извештај са извршним резимеом (1 реченица), 3 кључна поена и препорукама. Супервизор оркестрира овај ток аутоматски:

<img src="../../../translated_images/sr/supervisor-agent-pattern.06275a41ae006ac8.webp" alt="Supervisor Agent Pattern" width="800"/>

*Супервизор користи свој LLM да одлучи које агенте да позове и којим редоследом — није потребно чврсто кодирање путања.*

Ево како изгледа конкретан радни ток за наш фајл-до-извештаја процес:

<img src="../../../translated_images/sr/file-report-workflow.649bb7a896800de9.webp" alt="File to Report Workflow" width="800"/>

*FileAgent чита фајл преко MCP алата, затим ReportAgent претвара сиров садржај у структурирани извештај.*

Следећи секвенцијални дијаграм прати пуну Супервизор оркестрацију — од покретања MCP сервера, преко аутономног избора агената од стране Супервизора, до позива алата преко stdio и коначног извештаја:

<img src="../../../translated_images/sr/supervisor-agent-sequence.1aa389b3bef99956.webp" alt="Supervisor Agent Sequence Diagram" width="800"/>

*Супервизор аутономно позива FileAgent (који позива MCP сервер преко stdio за читање фајла), а затим позива ReportAgent да генерише структурирани извештај — сваки агент чува свој излаз у заједничком Agentic Scope.*

Сваки агент чува свој излаз у **Agentic Scope** (заједничка меморија), што омогућава следећим агентима да приступе претходним резултатима. Ово показује како се MCP алати беспрекорно интегришу у агентске радне токове — Супервизор не мора да зна *како* се фајлови читају, само да `FileAgent` може то да уради.

#### Како Супервизор ради

Пре него што изградите агенте, морате повезати MCP транспорт са клијентом и омотати га као `ToolProvider`. Овако алати MCP сервера постају доступни вашим агентима:

```java
// Креирај MCP клијента из транспорта
McpClient mcpClient = new DefaultMcpClient.Builder()
        .transport(stdioTransport)
        .build();

// Укрпи клијента као ToolProvider — ово повезује MCP алате са LangChain4j
ToolProvider mcpToolProvider = McpToolProvider.builder()
        .mcpClients(List.of(mcpClient))
        .build();
```

Сада можете убризгати `mcpToolProvider` у било који агент коме су потребни MCP алати:

```java
// Корак 1: FileAgent чита фајлове користећи MCP алате
FileAgent fileAgent = AgenticServices.agentBuilder(FileAgent.class)
        .chatModel(model)
        .toolProvider(mcpToolProvider)  // Има MCP алате за операције са фајловима
        .build();

// Корак 2: ReportAgent генерише структуиране извештаје
ReportAgent reportAgent = AgenticServices.agentBuilder(ReportAgent.class)
        .chatModel(model)
        .build();

// Supervisor координише радни ток фајл → извештај
SupervisorAgent supervisor = AgenticServices.supervisorBuilder()
        .chatModel(model)
        .subAgents(fileAgent, reportAgent)
        .responseStrategy(SupervisorResponseStrategy.LAST)  // Враћа коначни извештај
        .build();

// Supervisor одлучује које агенте позвати на основу захтева
String response = supervisor.invoke("Read the file at /path/file.txt and generate a report");
```

#### Како FileAgent открива MCP алате у време извршавања

Можда се питате: **како `FileAgent` зна како да користи npm алате за фајл систем?** Одговор је да не зна — **LLM** то сазнаје у време извршавања преко шема алата.

`FileAgent` интерфејс је само **дефиниција упита**. Нема унапред кодирано знање о `read_file`, `list_directory` или било ком другом MCP алату. Ево шта се догађа од почетка до краја:
1. **Покретање сервера:** `StdioMcpTransport` покреће npm пакет `@modelcontextprotocol/server-filesystem` као подређени процес  
2. **Откривање алата:** `McpClient` шаље `tools/list` JSON-RPC захтев серверу, који одговара са именима алата, описима и шемама параметара (нпр. `read_file` — *"Прочитај комплетан садржај фајла"* — `{ path: string }`)  
3. **Убацивање шема:** `McpToolProvider` обавија ове откривене шеме и чини их доступним за LangChain4j  
4. **LLM доноси одлуку:** Када се позове `FileAgent.readFile(path)`, LangChain4j шаље системску поруку, корисничку поруку, **и листу шема алата** LLM-у. LLM чита описе алата и генерише позив алата (нпр. `read_file(path="/some/file.txt")`)  
5. **Извршење:** LangChain4j пресреће позив алата, прослеђује га кроз MCP клијент назад на Node.js под-процес, добија резултат и даје га назад LLM-у  

Ово је исти механизам [Откривања алата](../../../05-mcp) описан горе, али примењен специфично на агенатски ток рада. `@SystemMessage` и `@UserMessage` анотације усмеравају понашање LLM-а, док му убацени `ToolProvider` даје **могућности** — LLM повезује ова два у време извршења.

#### Стратегије одговора

Када конфигуришете `SupervisorAgent`, одређујете како треба формулисати коначни одговор кориснику након што потагенти заврше своје задатке. Дијаграм испод приказује три доступне стратегије — LAST директно враћа излаз последњег агента, SUMMARY синтетише све излазе користећи LLM, а SCORED бира онај који боље постиже оцену у односу на оригинални захтев:

<img src="../../../translated_images/sr/response-strategies.3d0cea19d096bdf9.webp" alt="Response Strategies" width="800"/>

*Три стратегије за то како Супервизор формулише коначни одговор — изаберите у зависности од тога да ли желите последњи излаз агента, сажети резиме или најбоље оцењену опцију.*

Доступне стратегије су:

| Стратегија | Опис |
|------------|-------|
| **LAST** | Супервизор враћа излаз последњег позваног потагента или алата. Ово је корисно када је последњи агент у току рада посебно дизајниран да произведе комплетан, коначни одговор (нпр. "агент за сажетак" у истраживачкој пипелини). |
| **SUMMARY** | Супервизор користи свој интерни језички модел (LLM) да синтетише сажетак целокупне интеракције и свих излаза потагената, а затим враћа тај сажетак као коначни одговор. Ово обезбеђује чист и агрегирани одговор кориснику. |
| **SCORED** | Систем користи интерни LLM да оцени и LAST одговор и SUMMARY интеракције у односу на оригинални кориснички захтев, враћајући онај излаз који добије већу оцену. |

Погледајте [SupervisorAgentDemo.java](../../../05-mcp/src/main/java/com/example/langchain4j/mcp/SupervisorAgentDemo.java) за комплетну имплементацију.

#### Разумевање излаза

Када покренете демонстрацију, видећете структуриран приказ како Супервизор оркестрира више агената. Ево шта сваки део значи:

```
======================================================================
  FILE → REPORT WORKFLOW DEMO
======================================================================

This demo shows a clear 2-step workflow: read a file, then generate a report.
The Supervisor orchestrates the agents automatically based on the request.
```
  
**Наслов** уводи концепт ток рада: фокусирана пипелина од читања фајла до генерисања извештаја.

```
--- WORKFLOW ---------------------------------------------------------
  ┌─────────────┐      ┌──────────────┐
  │  FileAgent  │ ───▶ │ ReportAgent  │
  │ (MCP tools) │      │  (pure LLM)  │
  └─────────────┘      └──────────────┘
   outputKey:           outputKey:
   'fileContent'        'report'

--- AVAILABLE AGENTS -------------------------------------------------
  [FILE]   FileAgent   - Reads files via MCP → stores in 'fileContent'
  [REPORT] ReportAgent - Generates structured report → stores in 'report'
```
  
**Дијаграм тока рада** приказује проток података између агената. Сваки агент има специфичну улогу:  
- **FileAgent** чита фајлове помоћу MCP алата и чува сирови садржај у `fileContent`  
- **ReportAgent** користи тај садржај и производи структурисани извештај у `report`

```
--- USER REQUEST -----------------------------------------------------
  "Read the file at .../file.txt and generate a report on its contents"
```
  
**Кориснички захтев** приказује задатак. Супервизор анализира и одлучује да позове FileAgent → ReportAgent.

```
--- SUPERVISOR ORCHESTRATION -----------------------------------------
  The Supervisor decides which agents to invoke and passes data between them...

  +-- STEP 1: Supervisor chose -> FileAgent (reading file via MCP)
  |
  |   Input: .../file.txt
  |
  |   Result: LangChain4j is an open-source, provider-agnostic Java framework for building LLM...
  +-- [OK] FileAgent (reading file via MCP) completed

  +-- STEP 2: Supervisor chose -> ReportAgent (generating structured report)
  |
  |   Input: LangChain4j is an open-source, provider-agnostic Java framew...
  |
  |   Result: Executive Summary...
  +-- [OK] ReportAgent (generating structured report) completed
```
  
**Оркестрација Супервизора** приказује двостепени ток рада у пракси:  
1. **FileAgent** чита фајл преко MCP и чува садржај  
2. **ReportAgent** добија садржај и генерише структурисани извештај

Супервизор је донео ове одлуке **аутономно** на основу корисничког захтева.

```
--- FINAL RESPONSE ---------------------------------------------------
Executive Summary
...

Key Points
...

Recommendations
...

--- AGENTIC SCOPE (Data Flow) ----------------------------------------
  Each agent stores its output for downstream agents to consume:
  * fileContent: LangChain4j is an open-source, provider-agnostic Java framework...
  * report: Executive Summary...
```
  
#### Објашњење функционалности Agentic модула

Пример демонстрира неколико напредних функција agentic модула. Погледајмо ближе Agentic Scope и Agent слушаоце.

**Agentic Scope** приказује дељену меморију у којој агенти чувају своје резултате користећи `@Agent(outputKey="...")`. Ово омогућава:  
- Каснијим агенатима приступ излазима ранијих агената  
- Супервизору да синтетише коначни одговор  
- Вама да проверите шта је сваки агент произвео  

Дијаграм испод показује како Agentic Scope функционише као дељена меморија у току рада од фајла до извештаја — FileAgent пише свој излаз под кључем `fileContent`, ReportAgent чита и пише свој излаз под `report`:

<img src="../../../translated_images/sr/agentic-scope.95ef488b6c1d02ef.webp" alt="Agentic Scope Shared Memory" width="800"/>

*Agentic Scope делује као дељена меморија — FileAgent пише `fileContent`, ReportAgent то чита и пише `report`, а ваш код чита коначни резултат.*

```java
ResultWithAgenticScope<String> result = supervisor.invokeWithAgenticScope(request);
AgenticScope scope = result.agenticScope();
String fileContent = scope.readState("fileContent");  // Рау подаци из ФилеАгент-а
String report = scope.readState("report");            // Структурисани извештај из РепортАгент-а
```
  
**Agent слушаоци** омогућавају праћење и дебаговање извршења агената. Корак-по-корак излаз у демо програму долази од AgentListener-а који се повезује на сваку инвокацију агента:  
- **beforeAgentInvocation** - Позива се када Супервизор изабере агента, омогућавајући вам да видите који агент је изабран и зашто  
- **afterAgentInvocation** - Позива се када агент заврши, приказујући његов резултат  
- **inheritedBySubagents** - Када је true, слушалац прати све агенте у хијерархији  

Следећи дијаграм показује целокупан животни циклус Agent Listener-а, укључујући како `onError` обрађује грешке током извршења агента:

<img src="../../../translated_images/sr/agent-listeners.784bfc403c80ea13.webp" alt="Agent Listeners Lifecycle" width="800"/>

*Agent слушаоци се повезују на животни циклус извршења — прате када се агенти покрећу, завршавају или сусрећу са грешкама.*

```java
AgentListener monitor = new AgentListener() {
    private int step = 0;
    
    @Override
    public void beforeAgentInvocation(AgentRequest request) {
        step++;
        System.out.println("  +-- STEP " + step + ": " + request.agentName());
    }
    
    @Override
    public void afterAgentInvocation(AgentResponse response) {
        System.out.println("  +-- [OK] " + response.agentName() + " completed");
    }
    
    @Override
    public boolean inheritedBySubagents() {
        return true; // Проследи свим подагентима
    }
};
```
  
Осим Supervisor обрасца, `langchain4j-agentic` модул пружа више моћних образаца тока рада. Дијаграм испод приказује све пет — од једноставних секвенцијалних пипелина до одобравања уз учешће људи:

<img src="../../../translated_images/sr/workflow-patterns.82b2cc5b0c5edb22.webp" alt="Agent Workflow Patterns" width="800"/>

*Пет образаца тока рада за оркестрацију агената — од једноставних секвенцијалних пипелина до одобравања уз учешће људи.*

| Образац | Опис | Примена |
|---------|-------|---------|
| **Sequential** | Извршавање агената по реду, излаз иде у следећи | Пипелини: истраживање → анализа → извештај |
| **Parallel** | Паралелно покретање агената | Независни задаци: време + вести + берза |
| **Loop** | Понављање све док се не испуни услов | Оцењивање квалитета: прецизирање док оцена ≥ 0.8 |
| **Conditional** | Усмеравање по условима | Класификација → упућивање ка стручном агенту |
| **Human-in-the-Loop** | Увођење људских провера | Радни токови одобрења, преглед садржаја |

## Кључни концепти

Сада када сте истражили MCP и agentic модул у акцији, сажмимо када користити који приступ.

Једна од највећих предности MCP-а је његов растући екосистем. Дијаграм испод показује како један универзални протокол повезује вашу AI апликацију са широким спектром MCP сервера — од приступа фајл системима и базама података до GitHub-а, е-поште, веб скрепинга и више:

<img src="../../../translated_images/sr/mcp-ecosystem.2783c9cc5cfa07d2.webp" alt="MCP Ecosystem" width="800"/>

*MCP ствара универзални протокол екосистем — сваки MCP-компатибилан сервер ради са било којим MCP-компатибилним клијентом, омогућавајући дељење алата између апликација.*

**MCP** је идеалан када желите да искористите постојеће екосистеме алата, изградите алате које више апликација може делити, интегришете сервисе трећих страна стандардним протоколима или мењате имплементације алата без промене кода.

**Agentic модул** најбоље функционише када желите декларативне дефиниције агената са `@Agent` анотацијама, потребна вам је оркестрација тока (секвенцијални, петља, паралелни), више волите интерфејсни дизајн агената уместо императивног кода, или комбиновање више агената који деле излазе преко `outputKey`.

**Образац Supervisor Agenta** сјаји када ток рада није унапред предвидљив и желите да LLM одлучује, када имате више специјализованих agenata који захтевају динамичку оркестрацију, када градите разговорне системе који усмеравају ка различитим могућностима или када желите најфлексибније, адаптивно понашање агента.

Да бисте лакше одлучили између прилагођених `@Tool` метода из Модула 04 и MCP алата из овог модула, следеће поређење истиче главне компромисе — прилагођени алати пружају чврсту везу и пуну типску безбедност за апликацијску логику, док MCP алати нуде стандардизоване, поновљиве интеграције:

<img src="../../../translated_images/sr/custom-vs-mcp-tools.c4f9b6b1cb65d8a1.webp" alt="Custom Tools vs MCP Tools" width="800"/>

*Када користити прилагођене @Tool методе, а када MCP алате — прилагођени алати за апликацијску логику са потпуном типском безбедношћу, MCP алати за стандардизоване интеграције које раде у више апликација.*
